1. Getting Started

Install EvolutionDB, start the server, connect with a client, and run your first CREATE TABLE, INSERT, and SELECT — in a few minutes, from one binary.

EvolutionDB is a single-process SQL database engine written in C. One server speaks two protocols at once: the PostgreSQL wire protocol on TCP port 5433 (so psql, DBeaver, pgAdmin, and JDBC drivers connect unchanged) and a native EVO text protocol on TCP port 9967 (used by the bundled evosql-cli client). This chapter walks through getting a server running and executing your first statements against it. Later chapters cover the full SQL surface; here we stay deliberately small.

1.1. Installation

You can run EvolutionDB either from a prebuilt Docker image or by building the binaries from source. Docker is the quickest path and the one we use throughout the test suite.

1.1.1. Running with Docker

From the repository root, bring the server up in the background:

docker compose up -d              # PG:5433  EVO:9967

The bundled docker-compose.yml publishes three ports and provisions a persistent named volume evosql-data mounted at /data, so your data survives container restarts. The ports are:

5433
PostgreSQL wire protocol — for psql, DBeaver, pgAdmin, JDBC.
9967
Native EVO text protocol — for the bundled evosql-cli.
9970
CDC (change-data-capture) streaming as JSON Lines. You will not need this for a first session.

The Compose file sets the initial credentials and a few runtime knobs through environment variables:

# from docker-compose.yml
EVOSQL_USER_NAME: admin
EVOSQL_PASSWORD: admin
EVOSQL_CDC_PORT: 9970
EVOSQL_AUTO_RECLAIM_INTERVAL_SEC: 2
EVOSQL_ENCRYPTION_KEY: "…"

So with the default Compose setup the login is admin / admin. To stop the server while keeping your data, run docker compose down; to stop it and delete the data volume, run docker compose down -v.

You can also run the published image directly, overriding the password:

docker run -d -p 5433:5433 -p 9967:9967 \
    -e EVOSQL_PASSWORD=mysecret evolutiondb/evolutiondb:latest
About the default password. Under the Compose file shipped in the repository, EVOSQL_PASSWORD is set to admin. If you remove or comment out that variable, the server generates a random password and prints it once at startup — capture it from the logs, because it is not stored anywhere you can read it back.

1.1.2. Building from source

Building requires GCC, Bison, Flex, and libreadline-dev (for the CLI). TLS support additionally needs libssl-dev. Build all three components — the core engine, the dual-protocol server, and the CLI — with a bare make:

make                  # core engine + adaptor + CLI
make adaptor TLS=1    # build the server with OpenSSL TLS support

This produces the server at adaptor/evosql-server, the CLI at cli/evosql-cli, and the core engine at evolution/evosql. If you ever change the SQL grammar, regenerate the parser with make generate before rebuilding.

1.1.3. Starting a local server

After building, start the server directly. With no arguments it listens on the PostgreSQL port 5433 and the EVO port 9967:

./adaptor/evosql-server                    # PG:5433, EVO:9967
./adaptor/evosql-server --pg-port 5434     # custom PG port

The login user defaults to admin (override with EVOSQL_USER_NAME). As with Docker, if EVOSQL_PASSWORD is unset the server prints a freshly generated password once at startup.

1.2. Connecting to the server

Once a server is listening you can connect with the bundled CLI on the EVO port, or with any PostgreSQL client on the PG port.

1.2.1. The bundled CLI (EVO protocol, port 9967)

evosql-cli connects on the EVO protocol port (default 9967). The simplest interactive session, supplying the password inline:

./cli/evosql-cli -W admin              # connect to localhost:9967

The client recognizes these options (from evosql-cli --help):

-h, --host HOST
Server hostname (default localhost).
-p, --port PORT
Server port (default 9967).
-u, --user USER
Username (default admin).
-W, --password PW
Password — or set the EVOSQL_PASSWORD environment variable. If neither is given on an interactive run, you are prompted (with echo off).
-c, --command SQL
Run a single SQL statement and exit (non-interactive).
-f, --file FILE
Run SQL from a file and exit.
-F, --format FMT
Output format: table (default), csv, or json.
-x, --expanded
Expanded, one-column-per-line display.
--timing
Print query execution time.
--color / --no-color
Force or disable ANSI colors (auto-detected from the TTY by default).

With no -c/-f and a TTY on standard input, the CLI starts an interactive shell. Piping a file in (evosql-cli < file.sql) runs it as a script. End each SQL statement with a semicolon.

Inside the interactive shell, a handful of backslash meta-commands mirror what psql users expect:

CommandAction
\d / \dtList tables
\d NAMEDescribe a table's columns
\lList databases
\duList users
\dnList schemas
\xToggle expanded display
\timingToggle query timing
\format FSet output format: table | csv | json
\i FILERun SQL from a file
\eEdit a statement in $EDITOR, then run it
\! CMDRun a shell command
\qQuit

1.2.2. A PostgreSQL client (PG wire, port 5433)

Because EvolutionDB speaks the PostgreSQL wire protocol, psql connects with no special configuration:

psql -h 127.0.0.1 -p 5433 -U admin testdb

The default test database is testdb. The server emulates enough of pg_catalog and information_schema for GUI tools such as DBeaver and pgAdmin to introspect tables and columns.

Authentication is always cleartext-password. Unlike a stock PostgreSQL install, the server always negotiates CleartextPassword authentication on the wire and then verifies it server-side. Passwords are stored hashed with PBKDF2-SHA256, never in plaintext. Run over TLS (build the server with TLS=1) when connecting across an untrusted network.

1.3. Your first table

The remaining examples assume an interactive evosql-cli session, but they run identically through psql — they are ordinary SQL. We will create a table, insert a few rows, and read them back.

1.3.1. CREATE TABLE

Define a small table. Column types here are drawn from the integer, character, and boolean families that the engine implements:

CREATE TABLE products (
    id        INTEGER PRIMARY KEY,
    name      VARCHAR(50) NOT NULL,
    price     DECIMAL(10, 2),
    in_stock  BOOLEAN DEFAULT TRUE
);

EvolutionDB stores each record in its binary tuple format — a sentinel byte and table id, then a header, a null bitmap, and the column data. A column type code packs both a type family and a size; VARCHAR(50), for instance, is encoded as 130050. See CREATE TABLE for the full column-attribute, constraint, and table-option grammar.

A few of the data types you will reach for first:

TypeUse for
INTEGER / INT, SMALLINT, BIGINTWhole numbers
DECIMAL(p,s) (alias NUMERIC, DEC)Exact fixed-point numbers, e.g. money
REAL, FLOAT, DOUBLEApproximate floating-point numbers
CHAR(n), VARCHAR(n), TEXTCharacter strings
BOOLEAN (alias BOOL)True / false values
DATE, TIMESTAMP, DATETIMEDates and timestamps
UUIDUniversally unique identifiers
JSON (alias JSONB)Semi-structured documents
VECTOR(dim)Embedding vectors for similarity search

1.3.2. INSERT

Add rows. You can name the target columns explicitly and insert several rows in one statement:

INSERT INTO products (id, name, price) VALUES
    (1, 'Mechanical Keyboard', 89.90),
    (2, 'USB-C Cable', 7.50),
    (3, '27-inch Monitor', 249.00);

The in_stock column was omitted, so each row takes its declared DEFAULT TRUE. See INSERT for value lists, default handling, and inserting from a query.

1.3.3. SELECT

Read the rows back. Start with everything:

SELECT * FROM products;

Then narrow and order the result. A WHERE clause filters rows; ORDER BY sorts them:

SELECT name, price
FROM products
WHERE price < 100
ORDER BY price DESC;

Aggregate to summarize. This counts the rows and averages the price across the whole table:

SELECT COUNT(*) AS n, AVG(price) AS avg_price
FROM products;

For the complete projection, filtering, grouping, joining, and subquery grammar, see SELECT.

1.3.4. Cleaning up

When you are finished experimenting, drop the table:

DROP TABLE products;

See DROP TABLE. You can also empty a table without removing it with TRUNCATE.

1.4. Where to go next

You now have a working server, a connection, and a table you created, populated, and queried. From here:

  • Build richer schemas with constraints, indexes, and defaults — see CREATE TABLE and CREATE INDEX.
  • Modify data with UPDATE and DELETE, wrapped in transactions with savepoints.
  • Explore the agent-memory objects that distinguish EvolutionDB — CREATE MEMORY STORE (vector search), CREATE CHECKPOINT STORE, and reactive LISTEN / NOTIFY push.
Notes and limitations.
  • Two ports, one server. The PostgreSQL protocol lives on 5433 and the native EVO protocol on 9967 — they are not interchangeable. Point psql/JDBC at 5433 and evosql-cli at 9967.
  • Authentication mode is fixed. The wire protocol always uses CleartextPassword; there is no MD5/SCRAM negotiation as in stock PostgreSQL. Use TLS on untrusted links.
  • Single storage file, global state. The engine keeps all data in one page-based file and serializes query execution behind a lock for thread safety; it is a single-process server, not a multi-node cluster.
  • Catalog is emulated, not complete. Enough of pg_catalog / information_schema is faked for common GUI tools to introspect schemas; do not assume full PostgreSQL system-catalog fidelity.
  • Type spellings are broad but the implementation is its own. Many MySQL/PostgreSQL type names are accepted as aliases (for example BOOL → BOOLEAN, NUMERIC → DECIMAL), but behavior follows EvolutionDB's engine, not any one upstream database. Verify edge cases against the reference pages rather than against PostgreSQL habits.
