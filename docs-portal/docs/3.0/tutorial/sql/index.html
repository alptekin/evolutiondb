2. The SQL Language

A hands-on tour of the SQL EvolutionDB actually speaks: creating tables, inserting rows, reading them back, filtering with WHERE, and ordering results with ORDER BY: grounded in what the engine implements, not in what the SQL standard merely permits.

This chapter assumes a running server (PostgreSQL wire protocol on port 5433, native EVO protocol on port 9967) and a session connected to a database. Every example below has been kept to features the EvolutionDB engine genuinely executes. Where EvolutionDB deviates from PostgreSQL or the SQL standard, the difference is called out explicitly rather than glossed over.

Conventions. SQL keywords are shown in upper case for readability, but EvolutionDB's lexer is case-insensitive for keywords, SELECT, select, and Select are equivalent. Identifiers (table and column names) are compared case-insensitively as well; the engine resolves columns with strcasecmp.

2.1. Tables and the Storage Model

An EvolutionDB table is an ordered set of typed columns. Internally, every table lives inside a single page-based file (evosql.db, 4 KB pages); its definition, name, columns, types, constraints, is recorded in the system catalog, and its rows are stored as binary tuples in a heap indexed by a B+ tree on the primary key.

Two limits are worth knowing up front, because they are hard caps in the engine rather than soft guidance:

Maximum columns per table
Bounded by CAT_MAX_COLUMNS. CREATE TABLE rejects definitions that exceed it, and the same cap governs inherited columns and system-versioning shadow columns.
Record size
A single row's accumulated value buffer must fit within RECORD_BUF_SIZE. INSERT raises a data_exception if the row's text exceeds it before the tuple is even built.

2.2. Creating a New Table

You create a table with the CREATE TABLE command, naming each column and giving it a data type:

CREATE TABLE employees (
    id        INTEGER PRIMARY KEY,
    name      VARCHAR(50) NOT NULL,
    dept      VARCHAR(30),
    salary    INTEGER,
    active    BOOLEAN DEFAULT true
);

On success the engine reports command(s) completed successfully!... The whitespace and line breaks are not significant, and the statement is terminated by a semicolon.

2.2.1. Data Types

EvolutionDB encodes each column type as a single integer: typeVal / 10000 is the type family and typeVal % 10000 is the family-specific size (for instance the length of a VARCHAR). The grammar assigns these encodings directly. The following are the families you will use in everyday tutorial SQL:

The DATE/TIME/TIMESTAMP and TEXT/BLOB families are accepted without value validation: unlike PostgreSQL, an out-of-range or malformed timestamp literal is stored verbatim rather than rejected. Do not rely on the engine to reject a bad date.

2.2.2. Column Constraints and Defaults

A column definition can carry inline constraints. The ones exercised on the INSERT path are:

PRIMARY KEY
Marks the key column(s). EvolutionDB builds a B+ tree on the primary key and rejects duplicate keys with a unique violation. A table-level PRIMARY KEY (a, b) produces a composite key. PK columns must be supplied a value.
NOT NULL
An INSERT that leaves the column empty or explicitly NULL raises not_null_violation (23502).
UNIQUE
Validated by scanning existing rows. A duplicate non-NULL value raises unique_violation (23505). NULLs do not collide with each other.
DEFAULT value
Used when a column is omitted in a column-list INSERT. Surrounding quotes are stripped from string literals. A handful of function defaults are evaluated at insert time, including CURRENT_TIMESTAMP and gen_random_uuid().
CHECK (expr)
The expression is evaluated against each new row; a falsy result raises check_violation (23514).

See CREATE TABLE for the full constraint surface (foreign keys, composite uniqueness, generated columns, and system versioning).

2.3. Populating a Table With Rows

Use INSERT to add rows. In its simplest form you provide a value for every column, in declaration order:

INSERT INTO employees VALUES (1, 'Ada Lovelace', 'Engineering', 9200, true);

String literals are single-quoted. Booleans accept true/false or 0/1. If the count of supplied values does not match the table's column count, the engine raises an error reporting both numbers, INSERT has N value(s) but table has M column(s).

2.3.1. Naming Columns Explicitly

You can name the target columns, in any order, and omit the rest. Omitted columns take their DEFAULT, or NULL if no default exists:

INSERT INTO employees (id, name) VALUES (2, 'Grace Hopper');

Here dept and salary become NULL, and active takes its declared default of true. EvolutionDB reorders the supplied values into the table's physical column order, validates that every named column exists, and rejects writes that target a generated column. If the number of values does not match the number of named columns, the insert fails with INSERT has N value(s) but N column(s) specified.

2.3.2. Inserting Several Rows at Once

A single INSERT can carry multiple value groups, separated by commas:

INSERT INTO employees VALUES
    (3, 'Alan Turing',   'Research',    8800, true),
    (4, 'Edsger Dijkstra', 'Research',  8800, false),
    (5, 'Barbara Liskov', 'Engineering',9500, true);

Each row is validated and stored independently. Every value is checked against its column's type (§2.2.1), NOT NULL, UNIQUE, and CHECK constraints before the binary tuple is written to the heap and the primary key is inserted into the B+ tree.

Note, control bytes are stripped. EvolutionDB delimits row data internally with control bytes (a field separator and a row separator). To prevent a crafted value from corrupting field boundaries, those bytes are stripped out of user-supplied values during insert. In practice this is invisible for normal text, but it means you cannot store those specific control characters verbatim.

2.4. Querying a Table

The SELECT command retrieves rows. To read an entire table, select all columns with *:

SELECT * FROM employees;

The engine resolves the table's columns from the catalog, prints a header row of column names followed by a separator line, then performs a heap scan, decoding each binary tuple back into its fields. A column whose stored value is NULL is rendered as the literal NULL.

2.4.1. Looking Up a Single Row by Key

When a query targets a row by its primary key, EvolutionDB can answer directly from the B+ tree without scanning the heap: it searches the index for the key, then reads exactly that one tuple. This is the fast path: an O(log n) index probe rather than a full scan. Selecting by other columns falls back to a scan.

2.5. Filtering Rows With WHERE

A WHERE clause restricts the result to rows for which a condition holds. The condition is evaluated per row against that row's decoded column values:

SELECT name, salary FROM employees
WHERE dept = 'Research';

Conditions combine column references, literals, comparison operators, and the boolean connectives AND/OR/NOT:

SELECT * FROM employees
WHERE salary >= 9000 AND active = true;

Expression evaluation is numeric when both sides parse as numbers and lexicographic (string) otherwise, the same dual comparison the engine uses internally. This matters for mixed data: a column compared as text will sort and match by character order, not numeric value, unless both operands are valid numbers.

Honest contrast. EvolutionDB evaluates WHERE on the textual decoding of each row. There is no cost-based optimizer choosing among multiple indexes; the primary key B+ tree is used for direct key lookups, and other predicates are resolved by scanning. Treat WHERE as a row filter over a scan, except when you constrain the primary key to a single value.

2.6. Sorting Results With ORDER BY

By default, rows come back in heap-scan order, which you should treat as unspecified. To impose an order, add ORDER BY with a column name:

SELECT name, salary FROM employees
ORDER BY salary;

The default direction is ascending. Append DESC for descending order (and ASC for explicit ascending):

SELECT name, salary FROM employees
ORDER BY salary DESC;

The sort comparator mirrors the WHERE comparison rule: if both values in a pair parse cleanly as numbers it sorts numerically, otherwise it sorts as text. So an INTEGER column sorts 2 < 10 as expected, while a VARCHAR column holding "10" and "2" sorts "10" before "2".

Note, the 500-row sort cap. The SELECT * path materializes rows into a fixed in-memory buffer before sorting, and that buffer holds at most 500 rows. When ORDER BY is applied to a table larger than 500 rows on this path, only the first 500 scanned rows are collected and sorted; the remainder are not included in the ordered output. This is an engine limitation, not standard SQL behavior, PostgreSQL would sort the entire result set. If a column named in ORDER BY does not exist, the engine reports the error and falls back to an unordered scan rather than aborting.

2.7. Putting It Together

A complete tutorial session, create, populate, filter, and order, looks like this:

-- 1. Define the table
CREATE TABLE employees (
    id     INTEGER PRIMARY KEY,
    name   VARCHAR(50) NOT NULL,
    dept   VARCHAR(30),
    salary INTEGER
);

-- 2. Add rows
INSERT INTO employees VALUES
    (1, 'Ada Lovelace',  'Engineering', 9200),
    (2, 'Alan Turing',   'Research',    8800),
    (3, 'Barbara Liskov','Engineering', 9500);

-- 3. Filter and order
SELECT name, salary FROM employees
WHERE  dept = 'Engineering'
ORDER BY salary DESC;

The final query returns Barbara Liskov (9500) before Ada Lovelace (9200), and excludes Alan Turing because his department does not match the filter.

2.8. Common Errors in This Chapter

The operations above raise SQLSTATE-coded errors when a constraint or type rule is broken. The codes you are most likely to encounter while following this tutorial:

22001, string data right truncation
A value longer than a CHAR(n) or VARCHAR(n) column.
22003, numeric value out of range
A non-numeric value supplied for an integer or float column.
23502, not-null violation
An empty or NULL value in a NOT NULL column.
23505, unique violation
A duplicate primary-key or UNIQUE value.
23514, check violation
A row that fails a CHECK constraint.
42P01, undefined table
A query against a table that does not exist.
42703, undefined column
A column-list INSERT naming a column the table does not have.
