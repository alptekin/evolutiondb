2. The SQL Language

A hands-on tour of the SQL that EvolutionDB speaks. You will create tables, insert rows, and read them back. You will filter rows with WHERE and order results with ORDER BY. Every example here is based on what the engine actually does, not on what the SQL standard allows.

This chapter assumes you have a running server. The PostgreSQL wire protocol runs on port 5433. The native EVO protocol runs on port 9967. You also need a session connected to a database. Every example below uses only features the EvolutionDB engine truly supports. When EvolutionDB works differently from PostgreSQL or the SQL standard, the difference is pointed out clearly.

Conventions. SQL keywords are shown in upper case so they are easy to read. The EvolutionDB lexer does not care about case for keywords. So SELECT, select, and Select all mean the same thing. Table and column names are also compared without regard to case. The engine matches columns with strcasecmp.

2.1. Tables and the Storage Model

An EvolutionDB table is an ordered set of typed columns. Every table lives inside one page-based file. The file is named evosql.db and uses 4 KB pages. The table definition is kept in the system catalog. The catalog stores the name, the columns, the types, and the constraints. Rows are stored as binary tuples in a heap. A B+ tree on the primary key indexes that heap.

Two limits are worth knowing up front. They are hard caps in the engine, not soft advice.

Maximum columns per table
The limit is set by CAT_MAX_COLUMNS. CREATE TABLE rejects any table that has more columns than this. The same limit also covers inherited columns and system-versioning shadow columns.
Record size
One row's value buffer must fit within RECORD_BUF_SIZE. If the row's text is too big, INSERT raises a data_exception. The check happens before the tuple is built.

2.2. Creating a New Table

You create a table with the CREATE TABLE command. You name each column and give it a data type.

CREATE TABLE employees (
    id        INTEGER PRIMARY KEY,
    name      VARCHAR(50) NOT NULL,
    dept      VARCHAR(30),
    salary    INTEGER,
    active    BOOLEAN DEFAULT true
);

When the command works, the engine reports command(s) completed successfully!... Whitespace and line breaks do not matter. The statement ends with a semicolon.

2.2.1. Data Types

EvolutionDB stores each column type as a single integer. The part typeVal / 10000 is the type family. The part typeVal % 10000 is the size for that family. For a VARCHAR, the size is the maximum length. The grammar assigns these codes directly. The table below lists the families you will use in everyday tutorial SQL.

Honest contrast with PostgreSQL. EvolutionDB checks the type of the text value at insert time. These checks are narrow on purpose. The integer and float families check that the text reads as a number. VARCHAR(n) and CHAR(n) check length. BOOLEAN, UUID, and JSON have format checks. The DATE, TIME, TIMESTAMP, TEXT, and BLOB families are accepted without any value check. PostgreSQL rejects a bad timestamp. EvolutionDB stores it as given. Do not count on the engine to reject a bad date.

2.2.2. Column Constraints and Defaults

A column definition can carry inline constraints. These are the ones used on the INSERT path.

PRIMARY KEY
This marks the key column or columns. EvolutionDB builds a B+ tree on the primary key. It rejects duplicate keys with a unique violation. A table-level PRIMARY KEY (a, b) makes a composite key. You must give a value for primary key columns.
NOT NULL
An INSERT that leaves the column empty or sets it to NULL raises not_null_violation (23502).
UNIQUE
The engine checks uniqueness by scanning the existing rows. A duplicate non-NULL value raises unique_violation (23505). Two NULL values do not clash with each other.
DEFAULT value
The default is used when you leave a column out of a column-list INSERT. Surrounding quotes are stripped from string literals. A few function defaults are run at insert time. These include CURRENT_TIMESTAMP and gen_random_uuid().
CHECK (expr)
The expression is tested against each new row. A false result raises check_violation (23514).

See CREATE TABLE for the full set of constraints. That set includes foreign keys, composite uniqueness, generated columns, and system versioning.

2.3. Populating a Table With Rows

Use INSERT to add rows. In the simplest form you give a value for every column. The values follow the order the columns were declared in.

INSERT INTO employees VALUES (1, 'Ada Lovelace', 'Engineering', 9200, true);

String literals use single quotes. Booleans accept true or false, and also 0 or 1. The number of values you give must match the number of columns. If it does not, the engine raises an error and reports both numbers. The error reads INSERT has N value(s) but table has M column(s).

2.3.1. Naming Columns Explicitly

You can name the target columns in any order and leave out the rest. A column you leave out takes its DEFAULT. If it has no default, it takes NULL.

INSERT INTO employees (id, name) VALUES (2, 'Grace Hopper');

Here dept and salary become NULL. The column active takes its declared default of true. EvolutionDB puts the values back into the table's physical column order. It checks that every named column exists. It rejects any write that targets a generated column. If the number of values does not match the number of named columns, the insert fails with INSERT has N value(s) but N column(s) specified.

2.3.2. Inserting Several Rows at Once

A single INSERT can carry several value groups. You separate the groups with commas.

INSERT INTO employees VALUES
    (3, 'Alan Turing',   'Research',    8800, true),
    (4, 'Edsger Dijkstra', 'Research',  8800, false),
    (5, 'Barbara Liskov', 'Engineering',9500, true);

Each row is checked and stored on its own. Every value is checked against its column's type. See §2.2.1 for types. The value is also checked against the NOT NULL, UNIQUE, and CHECK constraints. All of these checks run first. Then the binary tuple is written to the heap and the primary key is added to the B+ tree.

Note on control bytes. Inside the engine, EvolutionDB separates row data with control bytes. One byte marks the end of a field. Another byte marks the end of a row. A crafted value could break these field boundaries. To stop that, the engine removes those bytes from your values during insert. For normal text you never see this. But it means you cannot store those exact control characters.

2.4. Querying a Table

The SELECT command reads rows. To read a whole table, select all columns with *.

SELECT * FROM employees;

The engine looks up the table's columns in the catalog. It prints a header row of column names. Below that it prints a separator line. Then it scans the heap and decodes each binary tuple back into its fields. A column whose stored value is NULL is shown as the word NULL.

2.4.1. Looking Up a Single Row by Key

When a query asks for a row by its primary key, EvolutionDB answers straight from the B+ tree. The heap is not scanned. The engine searches the index for the key and reads exactly that one tuple. The lookup is O(log n). A query on other columns falls back to a full scan.

2.5. Filtering Rows With WHERE

A WHERE clause keeps only the rows where a condition is true. The condition is tested one row at a time. It runs against that row's decoded column values.

SELECT name, salary FROM employees
WHERE dept = 'Research';

A condition can mix column names, literals, and comparison operators. It can also use the boolean words AND, OR, and NOT.

SELECT * FROM employees
WHERE salary >= 9000 AND active = true;

The engine compares numerically when both sides read as numbers. Otherwise it compares as text, character by character. This matters for mixed data. If a value is compared as text, it matches and sorts by character order, not by number value. Numeric comparison only happens when both sides are valid numbers.

Honest contrast. EvolutionDB runs WHERE on the text form of each row. There is no cost-based optimizer that picks among many indexes. The primary key B+ tree is used for direct key lookups. All other conditions are handled by scanning. Think of WHERE as a row filter over a scan. The one exception is when you pin the primary key to a single value.

2.6. Sorting Results With ORDER BY

By default, rows come back in heap-scan order. Treat that order as undefined. To set an order, add ORDER BY with a column name.

SELECT name, salary FROM employees
ORDER BY salary;

The default direction is ascending. Add DESC for descending order. Add ASC if you want to state ascending order explicitly.

SELECT name, salary FROM employees
ORDER BY salary DESC;

The sort uses the same rule as the WHERE comparison. If both values in a pair read cleanly as numbers, the sort is by number. Otherwise it sorts as text. So an INTEGER column sorts 2 < 10 as you expect. A VARCHAR column that holds "10" and "2" sorts "10" before "2".

Note on the 500-row sort cap. The SELECT * path loads rows into a fixed in-memory buffer before it sorts them. That buffer holds at most 500 rows. If you use ORDER BY on a table with more than 500 rows on this path, the engine collects and sorts only the first 500 rows it scans. The rest are left out of the sorted output. This is an engine limit, not standard SQL behavior. PostgreSQL would sort the whole result set. If a column named in ORDER BY does not exist, the engine reports the error and falls back to an unordered scan instead of stopping.

2.7. Putting It Together

Here is a full tutorial session. It creates a table, adds rows, filters them, and orders them.

-- 1. Define the table
CREATE TABLE employees (
    id     INTEGER PRIMARY KEY,
    name   VARCHAR(50) NOT NULL,
    dept   VARCHAR(30),
    salary INTEGER
);

-- 2. Add rows
INSERT INTO employees VALUES
    (1, 'Ada Lovelace',  'Engineering', 9200),
    (2, 'Alan Turing',   'Research',    8800),
    (3, 'Barbara Liskov','Engineering', 9500);

-- 3. Filter and order
SELECT name, salary FROM employees
WHERE  dept = 'Engineering'
ORDER BY salary DESC;

The last query returns Barbara Liskov with 9500 before Ada Lovelace with 9200. Alan Turing is left out because his department does not match the filter.

2.8. Common Errors in This Chapter

The work above raises SQLSTATE error codes when you break a constraint or a type rule. These are the codes you are most likely to see while following this tutorial.

22001, string data right truncation
A value that is longer than a CHAR(n) or VARCHAR(n) column.
22003, numeric value out of range
A non-numeric value given for an integer or float column.
23502, not-null violation
An empty or NULL value in a NOT NULL column.
23505, unique violation
A duplicate primary-key or UNIQUE value.
23514, check violation
A row that fails a CHECK constraint.
42P01, undefined table
A query against a table that does not exist.
42703, undefined column
A column-list INSERT that names a column the table does not have.
