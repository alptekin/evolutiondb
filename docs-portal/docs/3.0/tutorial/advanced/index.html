3. Advanced Features

EvolutionDB does more than plain tables and queries. It has transactional snapshot isolation. It has several kinds of index. It also has agent-memory and vector tools built right into SQL. This chapter shows what those features do in version 3.0.

The earlier chapters covered creating tables and running queries. This chapter is a tutorial for the parts that make EvolutionDB more than a single-file SQL engine. It covers multi-version transactions. It covers the index types you can build on a table. It also covers the vector and memory features made for AI-agent work. Every example below runs against a live server. Connect with the CLI client evosql-cli. You can also use any PostgreSQL wire-protocol client on port 5433. Where the behaviour is different from PostgreSQL or the SQL standard, the text says so plainly. It does not pretend they are the same.

About this chapter. The features here are described as they really work today. This includes their version-1 limits. Some parts have caveats that are written inline. The HNSW vector index and the agent-memory SEARCH path are two of these. Read the "Notes" callouts before you depend on a feature in production.

3.1. Transactions and MVCC

EvolutionDB uses Multi-Version Concurrency Control (MVCC). It does not lock rows for readers. Instead, each changed row stores the transaction that created it and the transaction that deleted it. A query sees one steady snapshot of the database. It skips rows that were not yet committed as of that snapshot. It also skips rows that were already deleted as of that snapshot. This works in the same way PostgreSQL does. The visibility rules are PostgreSQL-style on purpose.

3.1.1. Transaction IDs and the commit log

Every writing transaction gets a 32-bit transaction ID (XID). The number always goes up. The counter is saved in the database file header. So XIDs keep going up even after a restart. Two XIDs are reserved.

XID_FROZEN = 1
A marker that means "committed long ago". If a row has a frozen inserting XID, it is always visible. The engine does not need to check the commit log for it.
XID_NORMAL_MIN = 2
The first XID that a normal transaction can get.

Commit status is kept in a packed on-disk commit log (CLOG). It uses two bits per XID. So each 4 KB page tracks 16,320 transactions. Pages are chained together when one page is full. There are three states.

StateValueMeaning
CLOG_IN_PROGRESS0The transaction has not committed or aborted yet.
CLOG_COMMITTED1The transaction committed. Its rows can now be visible.
CLOG_ABORTED2The transaction rolled back. Its rows are never visible.

Two fast paths keep reads from touching the CLOG most of the time. First, each row header has hint bits. They are XMIN_COMMITTED and XMIN_ABORTED. They get set once the engine knows what happened to the inserting transaction. Second, a Commit Sequence Number (CSN) cache maps recently committed XIDs to a commit order. It uses a 4096-entry ring buffer. This gives an O(1) check for "did this commit before my snapshot?". The CSN cache also stores each commit's wall-clock time. The temporal query feature below uses this time. It limits how far back you can look.

3.1.2. Snapshots and isolation levels

A snapshot stores three things. It stores the highest XID at the moment it is taken, called xmax. It stores the lowest XID that is still active, called xmin. It also stores the list of transactions that were running at that moment. A row is visible when two things are true. Its inserting transaction committed before the snapshot. And its deleting transaction had not committed as of the snapshot. Up to 256 transactions can run at the same time. This limit is MVCC_MAX_ACTIVE.

The session's isolation level controls when the snapshot is taken. Set it with SET TRANSACTION ISOLATION LEVEL. A new connection uses READ COMMITTED by default.

READ UNCOMMITTED
It is accepted, but it works like READ COMMITTED. EvolutionDB never shows uncommitted rows.
READ COMMITTED (default)
A new snapshot is taken at the start of each statement. So a transaction can see rows that others committed between its own statements.
REPEATABLE READ
One snapshot is taken at BEGIN. The same snapshot is reused for every statement in the transaction. So reading the same data again returns the same rows.
SERIALIZABLE
Visibility works the same as REPEATABLE READ. On top of that, it adds a conflict guard. The guard records predicate and gap locks during scans. This detects write skew. To make this safe, the transaction is serialized against other writers. It holds the engine's parse lock until it commits or rolls back.

How this differs from PostgreSQL. PostgreSQL's SERIALIZABLE uses fully optimistic Serializable Snapshot Isolation. It allows a lot of concurrency. It may abort a transaction at commit time. EvolutionDB's SERIALIZABLE is stricter in practice. It holds a writer-exclusive lock the whole time. So it gives up concurrency to get a simpler guarantee. Pick REPEATABLE READ if you want snapshot stability without that serialization cost.

3.1.3. Running a transaction

Transaction control statements are handled at the protocol layer. A transaction starts with BEGIN. You can also use START TRANSACTION. It ends with COMMIT or ROLLBACK. See BEGIN / COMMIT / ROLLBACK for the full reference.

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
  INSERT INTO accounts (id, balance) VALUES (1, 100);
  UPDATE accounts SET balance = balance - 25 WHERE id = 1;
  SELECT balance FROM accounts WHERE id = 1;  -- sees 75
COMMIT;

Two behaviours match PostgreSQL on purpose.

  • Implicit rollback on disconnect. If a connection drops in the middle of a transaction, the open transaction is rolled back for you.
  • COMMIT in an aborted transaction. If a statement fails inside a transaction, a later COMMIT acts like a ROLLBACK.

3.1.4. Savepoints

Inside a transaction you can set a savepoint. Then you do more work. Then you can roll back to that point. You do not lose the whole transaction. See SAVEPOINT.

BEGIN;
  INSERT INTO log (msg) VALUES ('step one');
  SAVEPOINT sp1;
  INSERT INTO log (msg) VALUES ('step two');
  ROLLBACK TO sp1;   -- undoes 'step two' only
COMMIT;          -- 'step one' is durable

3.1.5. Temporal queries (time travel)

Old row versions stay in the heap until they are cleaned up. Because of this, EvolutionDB can rebuild the database as it looked just after a given transaction committed. The grammar borrows the SQL:2011 keyword FOR SYSTEM_TIME AS OF. But the point you give is a transaction ID, not a timestamp.

SELECT * FROM accounts
  FOR SYSTEM_TIME AS OF TRANSACTION 42;

This builds a synthetic snapshot. Its xmax sits one past XID 42. So every transaction up to and including 42 is visible. Anything newer is not visible. The historical view has no active list. It is a clean point-in-time read. It is not a concurrency snapshot.

Notes, transactions.

  • Some rows were written before MVCC existed. They have no MVCC flag in the tuple header. These rows are always visible. This keeps older database files working.
  • The active-transaction array has a fixed size of 256 entries. If it fills up, the engine logs a warning. The extra transaction is not tracked. This can make its visibility wrong. Keep concurrency below that limit.
  • "AS OF TRANSACTION" can only reach back as far as the old row versions still exist. The commit timestamp in the CSN ring sets the retention window. If a CSN slot has been evicted, it reports an unknown commit time.
  • The value you give is a transaction ID, not a clock value. There is no AS OF TIMESTAMP form.

3.2. Indexes

EvolutionDB makes a primary-key index for every table on its own. This section covers the secondary indexes you add with CREATE INDEX. A secondary index is a separate structure. It maps an indexed value back to the heap row. So equality and prefix lookups can skip a full table scan.

3.2.1. Index types

The structure that gets built depends on the USING clause. It also depends on the UNIQUE and fulltext modifiers. Inside, each index stores a single-character type code.

Type codeStructureCreated by
PPrimary-key B+ treeAutomatic, per table (cannot be dropped)
NB+ tree (non-unique)CREATE INDEX (default)
UB+ tree (unique)CREATE UNIQUE INDEX
HHash (non-unique)CREATE INDEX ... USING HASH
VHash (unique)CREATE UNIQUE INDEX ... USING HASH
FFull-text inverted B+ treeCREATE FULLTEXT INDEX
AHNSW vector (ANN), in-memoryCREATE INDEX ... USING HNSW (see §3.3)

B+ tree and hash secondary indexes are stored on disk in the unified page file. A composite index key joins the indexed column values with the row's primary key. An internal Unit Separator byte sits between them. Its value is 0x1F. That separator sorts before any printable character. This keeps each row's key unique. It also lets a prefix scan on "value␟" find every row that has that value.

-- ordinary B+ tree index
CREATE INDEX idx_email ON users (email);

-- unique constraint enforced at build and on insert
CREATE UNIQUE INDEX uq_sku ON products (sku);

-- hash index - equality lookups only
CREATE INDEX idx_status ON orders USING HASH (status);

3.2.2. Composite and expression indexes

An index can cover several columns. It can also index the result of an expression. A composite index joins the column values with a | separator in key order. An expression index stores the serialized expression. It re-evaluates the expression for each row at build time and on later writes.

-- composite: keyed on (last_name, first_name)
CREATE INDEX idx_name ON people (last_name, first_name);

-- expression index
CREATE INDEX idx_lower_email ON users (LOWER(email));

Limitation. You cannot create an index on an ARRAY column in version 1. Doing so raises error 0A000. This means the feature is not supported.

3.2.3. Building an index without blocking writers

Add CONCURRENTLY to build an index while other sessions keep writing to the table. The build runs in three phases. Phase 1 registers the index in the catalog with a building state. Phase 2 scans the existing rows in batches of 100. It releases the engine lock between batches so other queries can run. At the same time, a modification log records the primary keys of any rows that concurrent DML changes. Phase 3 reconciles that log. It removes stale entries. It inserts current values for each touched key. Only then does it mark the index ready.

CREATE INDEX CONCURRENTLY idx_created ON events (created_at);

The query planner ignores an index until it is marked ready. So a concurrent build never returns partial results. The batch-unlock design is safe for two reasons. B+ tree splits only move entries to the right, so the in-progress cursor stays valid. And deletes are lazy markers. They do not move entries.

3.2.4. Full-text indexes

A FULLTEXT index breaks a text column into tokens. It stores one (word, primary-key) pair per token in a B+ tree. This supports word-level MATCH search.

Notes, full-text. In version 1, a full-text index is not kept up to date on INSERT, UPDATE, or DELETE. The MATCH evaluator rescans and re-tokenizes the heap for each row. So the results stay correct. But the index itself can drift after a lot of DML. Re-create the index after a large load to keep it fresh.

To remove any secondary index, use DROP INDEX. The primary-key index has type P. It is protected and cannot be dropped.

DROP INDEX idx_email;

3.3. Vector search with HNSW

EvolutionDB has a native VECTOR(N) column type. It also has an approximate-nearest-neighbour (ANN) index. You build it with CREATE INDEX ... USING HNSW. This is the building block for semantic search over embeddings.

3.3.1. Creating an HNSW index

CREATE INDEX name ON table USING HNSW (vector_column) [WITH (m = N, ef_construction = N)]

CREATE TABLE docs (
  id        INT PRIMARY KEY,
  body      TEXT,
  embedding VECTOR(1024)
);

CREATE INDEX idx_doc_vec ON docs
  USING HNSW (embedding) WITH (m = 16, ef_construction = 64);

Two build parameters tune the graph.

m (default 16)
The number of neighbour links per node. Layer 0 holds up to 2·m links per node. Higher layers hold up to m.
ef_construction (default 64)
The beam width used while inserting nodes during the build. A larger value builds a better graph, but it is slower.

You pick the distance metric with an optional opclass after the column name. If you leave it out, it uses cosine. These metrics are supported.

MetricConstantDistance
Cosine (default)HNSW_DIST_COSINE = 01 − cos(a, b)
Euclidean (L2)HNSW_DIST_L2 = 1‖a − b‖₂
Inner productHNSW_DIST_INNER = 2−(a · b)
Manhattan (L1)HNSW_DIST_L1 = 3Σ|aᵢ − bᵢ|

At CREATE INDEX time the engine scans the table. It reads each vector's float payload straight from the tuple. It loads each one into a dense per-graph store. Then it builds a multi-layer greedy graph over them. The vector column must be a real VECTOR type. The index supports one vector column.

3.3.2. Searching

Queries call the vector functions directly. You can pass a query vector in two ways. One way is a bracketed list [f1, f2, ...]. The other way is the compact b64i8:<base64> form. This form is a float32 scale followed by int8 values. It keeps a 1024-dimension query literal small enough to fit the statement size cap.

-- k-nearest neighbours by the index's configured metric
SELECT id, body
  FROM docs
  WHERE HNSW_KNN(embedding, idx_doc_vec, '[0.12, 0.07, ...]', 10);

For larger datasets the search walks the graph. For small sets it falls back to an exact brute-force top-k scan. It also falls back this way if the graph could not be built. Either way, the results come back in ascending distance order. A search beam width controls the graph walk. It is ef_search and it is 64 by default.

3.3.3. Hybrid (filtered) search

Hybrid search joins an equality filter on a scalar column with vector ranking. The engine picks a strategy on its own. It looks at how selective the filter is.

Pre-filter
Used when fewer than 10% of rows match. The engine computes distances only for the matching rows. Then it ranks them.
Post-filter
Used when the filter is loose. The engine ranks all vectors first. Then it drops the hits that do not match. It takes extra candidates to make up for the dropped ones.
-- restrict to lang='en', then rank by vector distance
SELECT id FROM docs
  WHERE HNSW_FILTER_KNN(embedding, idx_doc_vec,
                       '[...]', 10, 'lang=en');

-- EXPLAIN-style: report which strategy fired and its selectivity
SELECT HNSW_HYBRID_EXPLAIN(embedding, idx_doc_vec,
                          '[...]', 10, 'lang=en');

Notes, HNSW.

  • HNSW graphs live in memory only. The catalog stores the build parameters in the index's metadata. These are m, ef_construction, and distance. At startup the engine rebuilds every HNSW graph from that metadata. If a graph fails to rebuild, it is left empty. Queries then fall back to brute force until you re-create it.
  • An HNSW index covers one VECTOR column. Multi-column ANN indexes are not supported in version 1.
  • Up to 64 HNSW graphs can exist at once across the whole server.
  • This is approximate search. Recall depends on m, ef_construction, and the search beam width. For very small tables the engine uses the exact brute-force path. So results there are exact.

3.4. Agent memory stores

A memory store is a namespace, key, and value store. It can also hold optional vector embeddings. It is made for long-term agent memory. It is a thin, easy layer over the engine. Under the hood, a memory store is a normal table named mem_<store>. So every memory operation gets MVCC, row-level security, triggers, and foreign-key cascade. It works the same way a hand-written query would.

3.4.1. Creating a store

Create a store with CREATE MEMORY STORE. With no options it is a plain JSON key-value store. If you supply embedding_dim, it adds a vector column. This turns on semantic search.

-- KV-only store
CREATE MEMORY STORE notes;

-- store with 1024-d embeddings, cosine distance
CREATE MEMORY STORE recall
  WITH (embedding_dim = 1024, distance = 'cosine');

The backing table has a fixed shape. It has a mem_namespace and a mem_key. It has a composite primary key. The key joins the namespace and key with a Record Separator byte. That byte is 0x1E. It has a JSON mem_value. It has a created_at timestamp and a nullable ttl_at timestamp. It has an embedding vector column too, but only when embedding_dim > 0. The distance option accepts cosine, l2, inner, ip, or l1. The default is cosine.

3.4.2. Storing and reading memories

MEMORY PUT is an upsert. It is keyed on (namespace, key). When there is a collision, it atomically points the primary key at the freshly inserted row. So a failed re-insert can never leave the row missing. The embedding argument is optional. It only matters for stores that were created with an embedding dimension.

-- upsert a value (no embedding)
MEMORY PUT INTO notes
  VALUES ('user:42', 'tone', '{"prefers":"concise"}');

-- upsert with an embedding for semantic recall
MEMORY PUT INTO recall
  VALUES ('user:42', 'fact-1',
          '{"text":"ships to Berlin"}', '[0.01, 0.4, ...]');

-- point lookup by namespace + key
MEMORY GET FROM recall WHERE NS = 'user:42' AND KEY = 'fact-1';

-- delete one memory
MEMORY DELETE FROM recall WHERE NS = 'user:42' AND KEY = 'fact-1';

-- enumerate namespaces, optionally by prefix
MEMORY LIST NAMESPACES IN recall PREFIX 'user:';

MEMORY GET, DELETE, and LIST compile down to ordinary SELECT and DELETE statements against the backing table. So any row-level security policy on that table applies on its own. This is what makes a memory store safe for multi-tenant use. See the per-command pages. MEMORY PUT, MEMORY GET, MEMORY SEARCH, and MEMORY DELETE.

3.4.3. Semantic search

MEMORY SEARCH ranks stored memories by their distance from a query embedding. It uses the store's configured distance metric. It adds a score column. It orders the results so the closest matches come first.

MEMORY SEARCH recall USING VECTOR '[0.02, 0.39, ...]' LIMIT 5;

If you do not give a LIMIT, the search returns up to 10 results by default.

Notes, memory stores.

  • MEMORY SEARCH needs a store created with embedding_dim > 0. Searching a KV-only store raises an error.
  • Search computes the score for each row over the backing table. Then it orders by that score. It does not yet route through the HNSW index. So for very large stores it scales linearly. The result ranking is still correct.
  • ttl_at is stored, and it is NULL by default. But the store layer does not enforce expiry on its own. Treat it as advisory metadata for now.
  • A store is a table. So you can always query mem_<store> directly with standard SQL. Do this when you need joins, aggregates, or filters that the memory verbs do not offer.
