3. Advanced Features

Beyond plain tables and queries, EvolutionDB ships transactional snapshot isolation, several index flavours, and a set of agent-memory and vector primitives built directly into the SQL surface. This chapter walks through what those features actually do in version 3.0.

The earlier chapters covered creating tables and running queries. This chapter is a tutorial for the parts of EvolutionDB that make it more than a single-file SQL engine: multi-version transactions, the index types you can build on a table, and the vector / memory features aimed at AI-agent workloads. Every example below runs against a live server; connect with the CLI client (evosql-cli) or any PostgreSQL wire-protocol client on port 5433. Where behaviour diverges from PostgreSQL or the SQL standard, the text says so plainly rather than implying parity.

About this chapter. The features here are described as implemented, including their version-1 limitations. Several subsystems, notably the HNSW vector index and the agent-memory SEARCH path, carry caveats that are documented inline. Read the "Notes" callouts before depending on a feature in production.

3.1. Transactions and MVCC

EvolutionDB uses Multi-Version Concurrency Control (MVCC). Instead of locking rows for readers, each modified row carries the transaction that created it and the transaction that deleted it; a query sees a consistent snapshot of the database and skips rows that were not yet committed, or were already deleted, as of that snapshot. This is the same family of mechanism PostgreSQL uses, and the visibility rules are deliberately PostgreSQL-style.

3.1.1. Transaction IDs and the commit log

Every writing transaction is assigned a monotonically increasing 32-bit transaction ID (XID). The counter is persisted in the database file header, so XIDs keep climbing across restarts. Two XIDs are reserved:

XID_FROZEN = 1
A sentinel "committed long ago" marker. A row whose inserting XID is frozen is always visible without any commit-log lookup.
XID_NORMAL_MIN = 2
The first XID that an ordinary transaction can receive.

Commit status lives in a packed on-disk commit log (CLOG): two bits per XID, so each 4 KB page tracks 16,320 transactions, and pages are chained for overflow. The three states are:

StateValueMeaning
CLOG_IN_PROGRESS0Transaction has not committed or aborted yet.
CLOG_COMMITTED1Transaction committed; its rows are candidates for visibility.
CLOG_ABORTED2Transaction rolled back; its rows are never visible.

Two fast paths avoid touching the CLOG on the hot read path. Each row header carries hint bits (XMIN_COMMITTED / XMIN_ABORTED) that are stamped once the inserting transaction's fate is known, and a Commit Sequence Number (CSN) cache maps recently committed XIDs to a commit order in a 4096-entry ring buffer, giving O(1) "did this commit before my snapshot?" checks. The CSN cache also records each commit's wall-clock time, which the temporal query feature (below) uses to bound how far back you can look.

3.1.2. Snapshots and isolation levels

A snapshot records the highest XID assigned at the moment it is taken (xmax), the lowest still-active XID (xmin), and the list of transactions that were in flight. A row is visible if its inserting transaction committed before the snapshot and its deleting transaction had not committed as of the snapshot. Up to 256 transactions can be active concurrently (MVCC_MAX_ACTIVE).

The session's isolation level controls when the snapshot is taken. Set it with SET TRANSACTION ISOLATION LEVEL; the default for a fresh connection is READ COMMITTED.

READ UNCOMMITTED
Accepted, but treated like READ COMMITTED, EvolutionDB never exposes uncommitted rows.
READ COMMITTED (default)
A new snapshot is taken at the start of each statement, so a transaction sees rows committed by others between its own statements.
REPEATABLE READ
One snapshot is taken at BEGIN and reused for every statement in the transaction, so repeated reads return the same rows.
SERIALIZABLE
Like REPEATABLE READ for visibility, plus a conflict guard that records predicate / gap locks during scans to detect write skew. To make this safe the transaction is serialized against other writers at the engine's parse lock until it commits or rolls back.

Contrast with PostgreSQL. PostgreSQL's SERIALIZABLE uses fully optimistic Serializable Snapshot Isolation that allows wide concurrency and may abort a transaction at commit. EvolutionDB's SERIALIZABLE is stricter operationally, it holds a writer-exclusive lock for the duration, so it trades concurrency for a simpler guarantee. Choose REPEATABLE READ if you need snapshot stability without that serialization cost.

3.1.3. Running a transaction

Transaction control statements are recognized at the protocol layer. A transaction begins with BEGIN (or START TRANSACTION) and ends with COMMIT or ROLLBACK. See BEGIN / COMMIT / ROLLBACK for the full reference.

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
  INSERT INTO accounts (id, balance) VALUES (1, 100);
  UPDATE accounts SET balance = balance - 25 WHERE id = 1;
  SELECT balance FROM accounts WHERE id = 1;  -- sees 75
COMMIT;

Two behaviours match PostgreSQL semantics intentionally:

  • Implicit rollback on disconnect. If a connection drops mid-transaction, the open transaction is rolled back automatically.
  • COMMIT in an aborted transaction. Once a statement errors inside a transaction, a subsequent COMMIT is treated as an implicit ROLLBACK.

3.1.4. Savepoints

Within a transaction you can mark a savepoint, do more work, then roll back to that point without discarding the whole transaction. See SAVEPOINT.

BEGIN;
  INSERT INTO log (msg) VALUES ('step one');
  SAVEPOINT sp1;
  INSERT INTO log (msg) VALUES ('step two');
  ROLLBACK TO sp1;   -- undoes 'step two' only
COMMIT;          -- 'step one' is durable

3.1.5. Temporal queries (time travel)

Because old row versions remain in the heap until cleaned up, EvolutionDB can reconstruct the database as it was just after a given transaction committed. The grammar borrows the SQL:2011 opener FOR SYSTEM_TIME AS OF, but the as-of point is a transaction ID, not a timestamp:

SELECT * FROM accounts
  FOR SYSTEM_TIME AS OF TRANSACTION 42;

This builds a synthetic snapshot whose xmax sits one past XID 42: every transaction up to and including 42 is visible, and anything newer is invisible. The historical view has no active list: it is a clean point-in-time read, not a concurrency snapshot.

Notes, transactions.

  • Rows written before MVCC existed (no MVCC flag in the tuple header) are unconditionally visible, for backward compatibility with older database files.
  • The active-transaction array is fixed at 256 entries. If it fills, a warning is logged and the overflowing transaction is not tracked, which can make its visibility incorrect, keep concurrency below that bound.
  • Temporal "AS OF TRANSACTION" can only reach back as far as the relevant old row versions still exist; the commit timestamp recorded in the CSN ring is used to bound the retention window, and a CSN slot that has been evicted reports an unknown commit time.
  • The as-of operand is a transaction ID, not a clock value, there is no AS OF TIMESTAMP form.

3.2. Indexes

EvolutionDB creates a primary-key index automatically for every table; this section covers the secondary indexes you add with CREATE INDEX. A secondary index is a separate structure that maps an indexed value back to the heap row, so equality and prefix lookups avoid a full table scan.

3.2.1. Index types

The kind of structure built depends on the USING clause and the UNIQUE / fulltext modifiers. Internally each index records a single-character type code:

Type codeStructureCreated by
PPrimary-key B+ treeAutomatic, per table (cannot be dropped)
NB+ tree (non-unique)CREATE INDEX (default)
UB+ tree (unique)CREATE UNIQUE INDEX
HHash (non-unique)CREATE INDEX ... USING HASH
VHash (unique)CREATE UNIQUE INDEX ... USING HASH
FFull-text inverted B+ treeCREATE FULLTEXT INDEX
AHNSW vector (ANN), in-memoryCREATE INDEX ... USING HNSW (see §3.3)

B+ tree and hash secondary indexes are stored on disk in the unified page file. A composite index key is built by joining the indexed column value(s) with the row's primary key, separated by an internal Unit Separator byte (0x1F). That separator sorts before any printable character, which keeps keys unique per row and lets a prefix scan on "value␟" find every row carrying that value.

-- ordinary B+ tree index
CREATE INDEX idx_email ON users (email);

-- unique constraint enforced at build and on insert
CREATE UNIQUE INDEX uq_sku ON products (sku);

-- hash index - equality lookups only
CREATE INDEX idx_status ON orders USING HASH (status);

3.2.2. Composite and expression indexes

An index can span several columns or index the result of an expression. A composite index joins the column values with a | separator in key order; an expression index stores the serialized expression and re-evaluates it for each row at build time and on subsequent writes.

-- composite: keyed on (last_name, first_name)
CREATE INDEX idx_name ON people (last_name, first_name);

-- expression index
CREATE INDEX idx_lower_email ON users (LOWER(email));

Limitation. Indexes cannot be created on ARRAY columns in version 1; doing so raises 0A000 (feature not supported).

3.2.3. Building an index without blocking writers

Adding CONCURRENTLY lets you build an index while other sessions keep writing to the table. The build runs in three phases. Phase 1 registers the index in the catalog in a building state. Phase 2 scans the existing rows in batches of 100, releasing the engine lock between batches so other queries can proceed, while a modification log captures the primary keys of any rows changed by concurrent DML. Phase 3 reconciles that log, removing stale entries and inserting current values for each touched key, and only then marks the index ready.

CREATE INDEX CONCURRENTLY idx_created ON events (created_at);

Until an index is marked ready, the query planner ignores it, so a concurrent build never returns partial results. The batch-unlock design is safe because B+ tree splits only move entries to the right (the in-progress cursor stays valid) and deletes are lazy markers rather than entry moves.

3.2.4. Full-text indexes

A FULLTEXT index tokenizes a text column and stores one (word, primary-key) pair per token in a B+ tree, supporting word-level MATCH search.

Notes, full-text. In version 1 a full-text index is not auto-maintained on INSERT / UPDATE / DELETE. The MATCH evaluator currently re-scans and re-tokenizes the heap per row, so results stay correct, but the index itself can drift after bulk DML. Re-create the index after a large load to keep it fresh.

To remove any secondary index, use DROP INDEX. The primary-key index (type P) is protected and cannot be dropped.

DROP INDEX idx_email;

3.3. Vector search with HNSW

EvolutionDB has a native VECTOR(N) column type and an approximate-nearest-neighbour (ANN) index built with CREATE INDEX ... USING HNSW. This is the building block for semantic search over embeddings.

3.3.1. Creating an HNSW index

CREATE INDEX name ON table USING HNSW (vector_column) [WITH (m = N, ef_construction = N)]

CREATE TABLE docs (
  id        INT PRIMARY KEY,
  body      TEXT,
  embedding VECTOR(1024)
);

CREATE INDEX idx_doc_vec ON docs
  USING HNSW (embedding) WITH (m = 16, ef_construction = 64);

Two build parameters tune the graph:

m (default 16)
The neighbour fan-out per node. Layer 0 holds up to 2·m links per node; higher layers hold up to m.
ef_construction (default 64)
The beam width used while inserting nodes during the build. Larger values build a higher-quality graph more slowly.

The distance metric is selected by an optional opclass after the column name; if omitted it defaults to cosine. The supported metrics are:

MetricConstantDistance
Cosine (default)HNSW_DIST_COSINE = 01 − cos(a, b)
Euclidean (L2)HNSW_DIST_L2 = 1‖a − b‖₂
Inner productHNSW_DIST_INNER = 2−(a · b)
Manhattan (L1)HNSW_DIST_L1 = 3Σ|aᵢ − bᵢ|

At CREATE INDEX time the engine scans the table, reads each vector's float payload directly from the tuple, loads it into a dense per-graph store, and builds a multi-layer greedy graph over it. The vector column must be a real VECTOR type and the index supports a single vector column.

3.3.2. Searching

Queries call the vector functions directly. A query vector is accepted either as a bracketed list [f1, f2, ...] or as the compact b64i8:<base64> form (a float32 scale followed by int8 values), which keeps a 1024-dimension query literal small enough to fit the statement size cap.

-- k-nearest neighbours by the index's configured metric
SELECT id, body
  FROM docs
  WHERE HNSW_KNN(embedding, idx_doc_vec, '[0.12, 0.07, ...]', 10);

The search uses graph traversal for larger datasets and falls back to an exact brute-force top-k scan for small sets (or if the graph could not be built), the results are returned in ascending distance order either way. A search beam width (ef_search, 64 by default) governs the graph walk.

3.3.3. Hybrid (filtered) search

Hybrid search combines an equality filter on a scalar column with vector ranking. The engine automatically picks a strategy based on how selective the filter is:

Pre-filter
Chosen when fewer than 10% of rows match. Distances are computed only for matching rows, then ranked.
Post-filter
Chosen when the filter is loose. All vectors are ranked, then non-matching hits are dropped; the candidate pool is oversampled to compensate for discards.
-- restrict to lang='en', then rank by vector distance
SELECT id FROM docs
  WHERE HNSW_FILTER_KNN(embedding, idx_doc_vec,
                       '[...]', 10, 'lang=en');

-- EXPLAIN-style: report which strategy fired and its selectivity
SELECT HNSW_HYBRID_EXPLAIN(embedding, idx_doc_vec,
                          '[...]', 10, 'lang=en');

Notes, HNSW.

  • HNSW graphs live in memory only. The catalog records the build parameters (m, ef_construction, distance) in the index's metadata, and the engine rebuilds every HNSW graph from that metadata at startup; a graph that fails to rebuild is left empty and queries fall back to brute force until it is re-created.
  • An HNSW index targets a single VECTOR column; multi-column ANN indexes are not supported in version 1.
  • Up to 64 HNSW graphs can exist at once across the whole server.
  • This is approximate search: recall depends on m, ef_construction, and the search beam width. For very small tables the exact brute-force path is used, so results there are exact.

3.4. Agent memory stores

A memory store is a purpose-built namespace/key/value store with optional vector embeddings, designed for long-term agent memory. It is a thin, ergonomic layer over the engine: under the hood a memory store is an ordinary table named mem_<store>, so every memory operation inherits MVCC, row-level security, triggers, and foreign-key cascade exactly as a hand-written query would.

3.4.1. Creating a store

Create a store with CREATE MEMORY STORE. Without options it is a pure JSON key-value store; supplying embedding_dim adds a vector column and enables semantic search.

-- KV-only store
CREATE MEMORY STORE notes;

-- store with 1024-d embeddings, cosine distance
CREATE MEMORY STORE recall
  WITH (embedding_dim = 1024, distance = 'cosine');

The backing table has a fixed shape: a mem_namespace and mem_key, a composite primary key (namespace and key joined by a Record Separator byte, 0x1E), a JSON mem_value, a created_at and nullable ttl_at timestamp, and, only when embedding_dim > 0, an embedding vector column. The distance option accepts cosine (default), l2, inner / ip, or l1.

3.4.2. Storing and reading memories

MEMORY PUT is an upsert keyed on (namespace, key): a collision atomically repoints the primary key to the freshly inserted row, so a failed re-insert can never leave the row missing. The embedding argument is optional and only meaningful for stores created with an embedding dimension.

-- upsert a value (no embedding)
MEMORY PUT INTO notes
  VALUES ('user:42', 'tone', '{"prefers":"concise"}');

-- upsert with an embedding for semantic recall
MEMORY PUT INTO recall
  VALUES ('user:42', 'fact-1',
          '{"text":"ships to Berlin"}', '[0.01, 0.4, ...]');

-- point lookup by namespace + key
MEMORY GET FROM recall WHERE NS = 'user:42' AND KEY = 'fact-1';

-- delete one memory
MEMORY DELETE FROM recall WHERE NS = 'user:42' AND KEY = 'fact-1';

-- enumerate namespaces, optionally by prefix
MEMORY LIST NAMESPACES IN recall PREFIX 'user:';

Because MEMORY GET, DELETE, and LIST compile down to ordinary SELECT / DELETE statements against the backing table, any row-level security policy attached to that table applies automatically, which is what makes a memory store multi-tenant safe. See the per-command pages: MEMORY PUT, MEMORY GET, MEMORY SEARCH, MEMORY DELETE.

3.4.3. Semantic search

MEMORY SEARCH ranks stored memories by distance from a query embedding, using the store's configured distance metric. It projects a score column and orders results so the closest matches come first.

MEMORY SEARCH recall USING VECTOR '[0.02, 0.39, ...]' LIMIT 5;

If no LIMIT is given the search returns up to 10 results by default.

Notes, memory stores.

  • MEMORY SEARCH requires a store created with embedding_dim > 0; searching a KV-only store raises an error.
  • Search currently computes the score per row over the backing table and orders by it: it does not yet route through the HNSW index, so for very large stores it scales linearly. The result ranking is still correct.
  • ttl_at is stored (NULL by default) but expiry is not auto-enforced by the store layer; treat it as advisory metadata for now.
  • Because a store is a table, you can always query mem_<store> directly with standard SQL when you need joins, aggregates, or filters the memory verbs do not expose.
