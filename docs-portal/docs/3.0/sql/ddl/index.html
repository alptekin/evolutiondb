5. Data Definition

How EvolutionDB defines the objects that hold data — schemas, tables, columns, constraints, sequences, and domains — and how those definitions are altered and dropped.

This chapter describes the data-definition language (DDL) that EvolutionDB actually implements. Every behavior documented here is taken from the engine's CREATE/ALTER/DROP handlers and the SQL grammar; where EvolutionDB diverges from PostgreSQL or the SQL standard, the difference is called out explicitly rather than glossed over. For the full per-statement syntax, follow the links into the command reference under /3.0/ref/.

5.1. The Object Model

EvolutionDB stores all metadata in a system catalog held in B+ trees inside a single page file. Objects are nested four levels deep, addressed internally by the path root/database/schema/table:

Database
The top-level container. The session works inside one current database (see USE).
Schema
A named namespace inside a database. Tables, sequences, and views live in a schema. The session has one current schema; an unqualified object name resolves against it.
Table
A base relation: an ordered list of columns plus its constraints and indexes. Rows are stored as binary tuples in heap pages, keyed by a primary-key B+ tree.
Column
A named, typed field. A table may have at most 256 columns (CAT_MAX_COLUMNS); identifiers are limited to 127 usable characters (CAT_MAX_NAME_LEN is 128 including the terminator).

Note. Unlike PostgreSQL, EvolutionDB has no CREATE SCHEMA ... AUTHORIZATION, no schema search-path list, and no concept of a per-object owner role at the schema level. A name is resolved against exactly one current schema, or against the schema named in a qualified schema.table reference.

5.2. Schemas

A schema is created inside the current database. CREATE SCHEMA takes only a name and an optional existence guard — there is no inline object list and no authorization clause.

CREATE SCHEMA sales;
CREATE SCHEMA IF NOT EXISTS sales;

-- Make it the active schema for unqualified names:
SET SCHEMA sales;
SET SCHEMA DEFAULT;

If the schema already exists, CREATE SCHEMA fails with SQLSTATE 42P06 (duplicate schema) unless IF NOT EXISTS was given, in which case it silently succeeds. SET SCHEMA verifies the schema exists in the current database before switching; a missing schema raises 3F000 (invalid schema name).

DROP SCHEMA is unconditionally cascading in EvolutionDB: it drops every table in the schema before removing the schema itself.

DROP SCHEMA sales;
DROP SCHEMA IF EXISTS sales;

Caveat. There is no RESTRICT / CASCADE keyword on DROP SCHEMA. The behavior is always equivalent to PostgreSQL's CASCADE for the tables it contains: dropping a non-empty schema discards its tables without warning. A non-existent schema raises 3F000 unless IF EXISTS is supplied. See SET SCHEMA.

5.3. Creating Tables

CREATE TABLE defines a base table in the current (or qualified) schema. A column list mixes column definitions with table-level constraints.

CREATE [ TEMPORARY | GLOBAL TEMPORARY ] TABLE [ IF NOT EXISTS ] [ schema . ]table_name (
    { column_name data_type [ column_attribute [ ... ] ]
    | table_constraint }
    [ , ... ]
)
[ table_option [ ... ] ]
[ INHERITS ( parent_table ) ]

A minimal example:

CREATE TABLE employees (
    id        INT PRIMARY KEY AUTO_INCREMENT,
    name      VARCHAR(100) NOT NULL,
    email     VARCHAR(255) UNIQUE,
    salary    DECIMAL(10,2) DEFAULT 0,
    status    VARCHAR(20) DEFAULT 'active',
    hired_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

If a table of the same name already exists in the target schema, CREATE TABLE fails with 42P07 (duplicate table) unless IF NOT EXISTS is present. The clause only suppresses the error; it does not compare the new definition with the existing one.

5.3.1. Temporary tables

The TEMPORARY and GLOBAL TEMPORARY prefixes mark a table as session-local or global-temporary respectively. Row lifetime on transaction boundaries is controlled by the ON COMMIT table option rather than by the prefix alone.

5.3.2. CREATE TABLE AS / SELECT

A table can be populated directly from a query. Both an inferred form (columns derived from the query) and an explicit-column form are accepted, optionally prefixed with IGNORE or REPLACE:

CREATE TABLE active_emps AS
    SELECT id, name FROM employees WHERE status = 'active';

5.3.3. Inheritance

INHERITS ( parent ) prepends the parent's columns to the child's own column list. Duplicate column names between parent and child are rejected with 42701 (duplicate column) — EvolutionDB does not merge same-named columns the way PostgreSQL does. If the child declares its own primary key, the inherited primary-key flag is stripped from the parent columns. The combined column count must stay within the 256-column ceiling.

Dropping an inherited-from parent requires CASCADE; see §5.7.

5.3.4. Range partitioning

A parent table declares PARTITION BY RANGE ( column ) as a table option, then child partitions are attached:

CREATE TABLE events (id INT, ts INT) PARTITION BY RANGE (ts);

CREATE TABLE events_q1 PARTITION OF events
    FOR VALUES FROM (0) TO (100);

Each child copies the parent's columns and registers an integer half-open range [lo, hi). The engine validates that lo < hi and that the new range does not overlap an existing partition; a table may hold at most 1024 partitions (EVO_MAX_PARTITIONS). Range bounds are integers only.

Note. Because the type is a single integer code, ALTER TABLE ... MODIFY can only change a type within the same family (and, for CHAR/VARCHAR, only widen the length). Cross-family conversions are rejected — see §5.6.2.

5.5. Constraints

EvolutionDB records five constraint kinds in the catalog, distinguished by a single type byte: primary key (P), unique (U), check (C), foreign key (F), and domain (D, used internally for domains). Constraints may be written inline on a column or at the table level, and may be explicitly named with CONSTRAINT name.

5.5.1. Primary keys

A single-column primary key is declared inline (PRIMARY KEY or, as a column attribute, the bare keyword KEY); a composite key uses the table-level form:

CREATE TABLE order_items (
    order_id  INT,
    line_no   INT,
    qty       INT NOT NULL,
    PRIMARY KEY (order_id, line_no)
);

The primary key is backed by a B+ tree created automatically with the table; it is the table's clustering index. The composite-key form accepts up to 16 columns.

5.5.2. Unique constraints

A column attribute UNIQUE (or UNIQUE KEY) marks a single column unique; the table-level UNIQUE (cols) form covers one or more columns. Up to 8 table-level unique constraints may be declared per CREATE TABLE. An unnamed unique constraint is auto-named table_cols_key.

5.5.3. Check constraints

A CHECK (expr) constraint stores a serialized boolean expression that each row must satisfy. It may be inline or table-level, and named or auto-named table_check_n:

CREATE TABLE products (
    id     INT PRIMARY KEY,
    price  DECIMAL(10,2) CHECK (price >= 0),
    CONSTRAINT nonempty_name CHECK (name <> '')
);

5.5.4. Foreign keys

A foreign key references the primary key of another table (optionally schema-qualified). Both the single- and composite-column forms are supported, with referential actions:

CREATE TABLE orders (
    id          INT PRIMARY KEY,
    customer_id INT,
    CONSTRAINT fk_cust FOREIGN KEY (customer_id)
        REFERENCES customers (id)
        ON DELETE CASCADE ON UPDATE RESTRICT
);

The referenced table must already exist (otherwise 42P01). Up to 8 foreign keys may be declared per CREATE TABLE; an unnamed one is auto-named table_cols_fkey. The supported clauses and their stored action codes are:

ON DELETE / ON UPDATE
CASCADE (1), SET NULL (2), RESTRICT (3), SET DEFAULT (4), NO ACTION (5). The default when omitted is action 0 (no action).
MATCH
SIMPLE (0, the default), FULL (1), PARTIAL (2).
DEFERRABLE
NOT DEFERRABLE (0), DEFERRABLE / DEFERRABLE INITIALLY IMMEDIATE (1), DEFERRABLE INITIALLY DEFERRED (2). These modes are parsed and stored.

Note. A foreign key always references the parent's primary key B+ tree. Rows whose foreign-key columns are all NULL are accepted regardless of MATCH mode. The MATCH PARTIAL mode is recorded but is not standard-complete.

5.6. Altering Tables

ALTER TABLE supports a broad set of subcommands. Each runs immediately and validates existing data where relevant.

5.6.1. Adding and changing constraints

-- Add constraints (existing rows are validated first):
ALTER TABLE products ADD CONSTRAINT pos_price CHECK (price >= 0);
ALTER TABLE emp ADD CONSTRAINT u_email UNIQUE (email);
ALTER TABLE orders ADD CONSTRAINT fk_c FOREIGN KEY (customer_id) REFERENCES customers (id);
ALTER TABLE t ADD CONSTRAINT pk_t PRIMARY KEY (a, b);

-- Manage existing constraints:
ALTER TABLE t DROP CONSTRAINT u_email;
ALTER TABLE t RENAME CONSTRAINT u_email TO uq_email;
ALTER TABLE t ENABLE CONSTRAINT pos_price;
ALTER TABLE t DISABLE CONSTRAINT pos_price;

When adding a CHECK, the engine scans every existing row and rejects the constraint (with the row number) if any row fails. Adding a UNIQUE constraint runs an O(n²) duplicate scan; adding a PRIMARY KEY additionally rejects NULL values in the key columns and refuses if the table already has a primary key. A duplicate constraint name raises 42710; a missing one raises 42704.

Deferred validation. The NOT VALIDATE suffix on a new CHECK or FOREIGN KEY adds the constraint without scanning existing data, marking it unvalidated. A later VALIDATE CONSTRAINT performs the scan and flips the validated flag:

ALTER TABLE t ADD CONSTRAINT c1 CHECK (x > 0) NOT VALIDATE;
ALTER TABLE t VALIDATE CONSTRAINT c1;

Note. The keyword spelling is NOT VALIDATE (not PostgreSQL's NOT VALID). A DISABLEd CHECK constraint is skipped when reading constraints back, so it is no longer enforced; ENABLE restores it.

5.6.2. Adding, dropping, and changing columns

ALTER TABLE emp ADD COLUMN phone VARCHAR(20);
ALTER TABLE emp ADD COLUMN active BOOLEAN DEFAULT true AFTER name;
ALTER TABLE emp DROP COLUMN phone;
ALTER TABLE emp RENAME COLUMN name TO full_name;
ALTER TABLE emp MODIFY COLUMN full_name VARCHAR(200);
ALTER TABLE emp CHANGE COLUMN full_name display_name VARCHAR(200);
ADD COLUMN
Appends a column (optionally repositioned with FIRST or AFTER col, which eagerly rewrites every tuple). A NOT NULL column without a DEFAULT is rejected if the table already holds rows (23502). A primary-key column cannot be added this way. Adding past 256 columns raises 54011.
DROP COLUMN
Marks the column dropped. Refused (2BP01) if it is part of the primary key, is the table's only column, or is referenced by a secondary index, a foreign key, or a unique constraint.
RENAME COLUMN
Renames a column and updates any secondary index column lists that referenced the old name. A name collision raises 42701.
MODIFY / CHANGE COLUMN
Changes a column's type and attributes (CHANGE also renames it, MySQL-style). The new type must be in the same family as the old one; CHAR/VARCHAR may only widen. A cross-family change raises 42804; changing a primary-key column's type raises 2BP01. Switching to NOT NULL scans for existing NULLs and fails with 23502 if any are found.

5.6.3. Renaming tables

A table is renamed with the standalone RENAME TABLE statement:

RENAME TABLE employees TO staff;

A missing source raises 42P01; a target name already in use raises 42P07.

5.6.4. Table TTL

A table may nominate a column whose value drives row expiry by the background reclaim daemon. The column must exist on the table or the statement fails (42703):

ALTER TABLE sessions SET ttl_column = 'expires_at';
ALTER TABLE sessions DROP TTL;

5.7. Dropping and Truncating Tables

DROP TABLE removes a table and all of its storage — primary-key tree, heap pages, secondary indexes, constraints, RLS policies, and inheritance links:

DROP TABLE employees;
DROP TABLE IF EXISTS employees;
DROP TABLE employees CASCADE;

The drop is refused with 2BP01 if a view references the table, or if the table has inheritance children and CASCADE was not given. With CASCADE, child tables are dropped recursively first. RESTRICT is the default; a missing table raises an error unless IF EXISTS is supplied.

TRUNCATE empties a table (or several) by destroying and recreating its heap and index trees, far faster than a row-by-row delete:

TRUNCATE TABLE logs;
TRUNCATE TABLE t1, t2 CASCADE;
TRUNCATE TABLE logs CONTINUE IDENTITY;

If a non-empty table references the target via a foreign key, TRUNCATE fails with 23503 unless CASCADE is given, in which case the referencing table is truncated first. By default the auto-increment counter is reset to its start; CONTINUE IDENTITY preserves it. Table statistics are reset.

5.8. Sequences

EvolutionDB provides PostgreSQL-style sequence objects, stored per schema. See CREATE SEQUENCE.

CREATE SEQUENCE IF NOT EXISTS order_seq
    START WITH 100
    INCREMENT BY 1
    MINVALUE 1
    MAXVALUE 1000000
    CYCLE;

SELECT nextval('order_seq');
SELECT setval('order_seq', 500);
START [ WITH ] n
The first value nextval returns. Defaults to MINVALUE when omitted.
INCREMENT [ BY ] n
The step between values (default 1). May be negative for a descending sequence.
MINVALUE n / MAXVALUE n
Bounds. Defaults are min 1 and max 2147483647.
CYCLE / NOT CYCLE
Whether the sequence wraps at its bound (to the opposite bound) or raises an error. Default is NOT CYCLE.

nextval is atomic under a dedicated mutex. The first call returns the current (start) value and marks the sequence "called"; subsequent calls add the increment, wrapping or erroring at the bound per the cycle flag. setval sets the stored value directly.

Sequences can be altered, restarted, renamed, and dropped:

ALTER SEQUENCE order_seq INCREMENT BY 2 MAXVALUE 2000000;
ALTER SEQUENCE order_seq RESTART;
ALTER SEQUENCE order_seq RESTART WITH 1;
ALTER SEQUENCE order_seq RENAME TO ord_seq;
DROP SEQUENCE IF EXISTS ord_seq;

Caveat. ALTER SEQUENCE only applies options that differ from their defaults: an option left at its default value (for example INCREMENT 1, MINVALUE 1, or MAXVALUE 2147483647) is treated as "unspecified" and leaves the stored setting unchanged. RENAME is implemented as drop-and-recreate under the same id. There is no CACHE or OWNED BY clause, and column defaults reference generators such as AUTO_INCREMENT/IDENTITY directly rather than via an owned sequence.

5.9. Domains

A domain is a named, reusable type built on a base type, optionally with a default, a NOT NULL marker, and a CHECK. See CREATE DOMAIN.

CREATE DOMAIN positive_int AS INT CHECK (VALUE > 0);
CREATE DOMAIN email AS VARCHAR(255) NOT NULL;
CREATE DOMAIN score AS INT DEFAULT 0;

A domain is stored as a special constraint entry (type D) in a system namespace, packing its base type code, NOT-null flag, and serialized check. Creating a domain whose name already exists raises 42710.

Note. Domain support is intentionally minimal compared to PostgreSQL. The grammar accepts CREATE DOMAIN ... AS type [ DEFAULT expr ] [ NOT NULL ] [ CHECK (expr) ], but the DEFAULT clause in particular is parsed without being persisted in the current build, and there is no DROP DOMAIN or ALTER DOMAIN statement. Treat domains as a lightweight, forward-looking facility rather than a complete type system.
