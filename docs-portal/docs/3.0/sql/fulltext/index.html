12. Full-Text Search

EvolutionDB provides a lightweight, single-column full-text search facility built on a shared tokenizer, an inverted B+ tree index, and the MATCH (col) AGAINST ('query') expression that scores rows by how many query terms they contain.

Full-text search lets you find rows whose text columns contain one or more search words, rather than matching a literal substring or LIKE pattern. The whole facility is intentionally small. It consists of three moving parts that all share a single tokenizer so that the words seen at index-build time are exactly the words seen at query time:

  • A tokenizer that lowercases, splits on non-alphanumeric boundaries, drops very short words, and removes a small list of English stop words.
  • A full-text index (CREATE FULLTEXT INDEX) that stores one inverted-index entry per (word, row) pair in a B+ tree.
  • The MATCH … AGAINST expression that evaluates a query string against a text column and returns a per-row hit count.

How this differs from PostgreSQL. EvolutionDB does not implement PostgreSQL's tsvector/tsquery types, the @@ match operator, GIN/GiST indexes, configurable text-search dictionaries, stemming, weighting, phrase search, or ts_rank/ts_headline. The syntax here is closer to MySQL's MATCH (col) AGAINST ('text'), but the implementation is simpler than either: matching is a plain whole-token equality check with no stemming and no boolean-mode operators. Document only what is described on this page; everything else is unimplemented.

12.1. Tokenization

Tokenization turns a raw text value into a list of normalized words. The same routine is used both when populating a full-text index and when evaluating MATCH … AGAINST, so the two always agree on word boundaries and casing.

The rules applied to the input text are:

Word boundaries
A token runs across consecutive alphanumeric bytes (ASCII letters and digits) plus any byte with a value of 0x80 or greater. Every other byte (spaces, punctuation, ASCII symbols) ends the current token.
Case folding
ASCII letters are folded to lowercase. Bytes at or above 0x80 — that is, the bytes that make up UTF-8 multibyte characters — pass through unchanged. Applying ASCII case folding to UTF-8 bytes is undefined, so non-ASCII letters are deliberately left as-is.
Minimum length
Tokens shorter than 2 bytes are discarded.
Stop words
Tokens that match the built-in English stop-word list (see 12.2) are discarded.
Maximum token length
A token is capped at 63 bytes of content (the 64-byte buffer minus its terminator); longer tokens are silently truncated to that length.
Maximum token count
At most 256 tokens are extracted from a single text value when indexing. When evaluating the AGAINST query string, at most 32 tokens are extracted.

Because the byte-level rule keeps multibyte (for example Turkish or other non-Latin) letters attached to their words but does not case-fold them, non-ASCII text matches case-sensitively in this release. A query term must match the stored term byte-for-byte in its non-ASCII portion.

An example of how a sentence tokenizes:

-- input:  "The Quick, Brown FOX!"
-- output: ["quick", "brown", "fox"]
--   * "The" -> stop word, dropped
--   * "Quick"/"Brown"/"FOX" -> lowercased
--   * punctuation splits and is discarded

Note. There is no stemming or lemmatization. running, ran, and run are three different tokens and will not match each other. Likewise there are no synonyms, thesaurus expansion, or accent folding.

12.2. The stop-word list

The tokenizer removes a fixed, built-in list of common English words. The list is not configurable in this release. The following words are dropped wherever they appear:

Stop words (29 entries)
a, an, and, are, as, at, be, but, by, for, if, in, is, it, no, not, of, on, or, so, that, the, this, to, was, were, will, with

Note that single-character words such as I or a would in any case be dropped by the two-byte minimum-length rule. The stop-word check runs after lowercasing, so it is case-insensitive for ASCII words.

12.3. The MATCH … AGAINST expression

Full-text matching is expressed with the MATCH (column) AGAINST ('query') construct. It is a scalar expression: it names exactly one column and a single quoted query string.

MATCH ( column_name ) AGAINST ( 'query string' )

The grammar accepts a single bare column name inside the MATCH (...) parentheses and a single string literal inside AGAINST (...). The query literal may be quoted with single or double quotes; the surrounding quotes are stripped before tokenization.

12.3.1. Evaluation semantics

When a row is evaluated, EvolutionDB:

  1. reads the named column's value for that row;
  2. tokenizes the column value (up to 256 tokens) and tokenizes the AGAINST string (up to 32 tokens), using the rules in 12.1;
  3. counts how many query terms appear among the row's terms — each query term contributes at most 1 to the score (a term that appears multiple times in the row still counts once);
  4. returns that count as the value of the expression.

This is an any-term (OR) match: a row scores above zero if it contains any of the query words. The expression has no boolean-mode operators (no +, -, quoted phrases, or wildcards) — every word in the query string is treated as an independent OR term.

If the column is missing, NULL, or empty, the expression evaluates to 0. If every query term is filtered out (all stop words or too short), the score is 0 for every row.

12.3.2. MATCH in WHERE — filtering

In a boolean context such as a WHERE clause, the numeric score is coerced to truth in the usual way: a non-zero score is true, a score of zero is false. So WHERE MATCH(col) AGAINST('term') returns exactly the rows that contain at least one of the query terms.

CREATE TABLE articles (
    id    INT PRIMARY KEY,
    body  VARCHAR(2000)
);

INSERT INTO articles VALUES
    (1, 'The quick brown fox jumps over the lazy dog'),
    (2, 'A fast brown rabbit hops past the fence'),
    (3, 'Database engines store rows in pages');

-- rows 1 and 2 both contain "brown"
SELECT id FROM articles
WHERE MATCH(body) AGAINST('brown');

-- any-term match: row 1 ("fox"+"lazy") and row 2 ("rabbit") all qualify
SELECT id FROM articles
WHERE MATCH(body) AGAINST('fox rabbit lazy');

12.3.3. MATCH in the select list and ORDER BY — ranking

Because MATCH … AGAINST returns a hit count rather than just a boolean, you can project it as a relevance score or use it to order results. Higher scores mean more distinct query terms were found in the row.

-- project the score; rank most-relevant first
SELECT id,
       MATCH(body) AGAINST('quick brown fox') AS score
FROM articles
ORDER BY MATCH(body) AGAINST('quick brown fox') DESC;

-- combine relevance ranking with a relevance filter
SELECT id
FROM articles
WHERE    MATCH(body) AGAINST('quick brown fox')
ORDER BY MATCH(body) AGAINST('quick brown fox') DESC;

The projected score is an integer. With the example data, row 1 (containing quick, brown, and fox) scores 3, while row 2 (containing only brown) scores 1.

Note. Unlike PostgreSQL's ts_rank, the score is not normalized for document length or term frequency. It is simply the number of distinct query terms present in the row, so the maximum possible score equals the number of distinct, non-stop, non-trivial words in the query.

12.4. Creating a full-text index

A full-text index is created with the FULLTEXT keyword on CREATE INDEX:

CREATE FULLTEXT INDEX index_name ON table_name ( column_name )
CREATE FULLTEXT INDEX idx_articles_body
    ON articles (body);

Internally this creates a B+ tree whose keys are formed by joining each tokenized word to the row's primary-key value with a unit-separator byte (0x1F): one (word, primary-key) entry is inserted per token of the indexed column, for every existing row. The value stored against each key is the row's heap location. This is a classic inverted index: looking up a word yields the rows that contain it.

At creation time, EvolutionDB scans the table once, tokenizes the indexed column for each row, and inserts the per-token entries. The build runs under the engine's DML lock, so the table is quiesced and no MVCC visibility filtering is applied during the build.

12.4.1. Restrictions

Single text column
A full-text index covers exactly one column, and that column must hold text. Although the index column list is parsed generally, the full-text build path uses a single named column; the named column must exist on the table or the statement fails with a column not found error.
No IF NOT EXISTS, UNIQUE, or USING HASH
The FULLTEXT form is its own grammar production. It does not combine with the unique, hash, or IF NOT EXISTS variants of CREATE INDEX.
Inline table-definition form
A FULLTEXT INDEX (col) / FULLTEXT KEY (col) clause is also accepted inside a CREATE TABLE definition for compatibility, where it is recorded as a text-index column.

12.5. Index maintenance and the query path

Two limitations of this release are important to understand together, because they affect how — and whether — you should rely on a full-text index.

Important: the index is not used by query evaluation. In this release, MATCH … AGAINST always evaluates by reading each candidate row's column value from the heap and tokenizing it on the fly. It does not probe the inverted B+ tree. The full-text index can be created and stored, but it does not yet accelerate MATCH queries. Results are always correct because they come from the live row data; they are simply produced by a per-row scan rather than an index lookup.

Important: the index is not auto-maintained. Incremental maintenance of a full-text index on INSERT, UPDATE, and DELETE is not implemented. The index reflects the rows present at the moment it was built. After bulk data changes, drop and re-create the index with CREATE FULLTEXT INDEX to refresh it. When a full-text index is created, the server prints a note reminding you of this.

The practical consequence is that, for now, you can use MATCH … AGAINST for filtering and ranking without ever creating an index — the expression works on its own. Creating the index does not change query results or speed; it is forward-looking storage for a future index-backed lookup path.

12.6. Reference summary

12.6.1. Keywords and syntax

MATCH ( col ) AGAINST ( 'query' )
Scalar expression returning the integer count of distinct query terms found in col for the current row. Usable in SELECT, WHERE, and ORDER BY.
CREATE FULLTEXT INDEX name ON table ( col )
Builds an inverted B+ tree index over the tokenized words of one text column. See CREATE INDEX.
FULLTEXT INDEX ( col ) / FULLTEXT KEY ( col )
Inline full-text index clause accepted inside CREATE TABLE.

Note — summary of limitations. Full-text search in EvolutionDB 3.0 is whole-token, OR-only matching with ASCII-only case folding. There is no stemming, phrase search, boolean-mode operators, configurable dictionaries, length- or frequency-weighted ranking, or tsvector/tsquery support. A full-text index can be created but is neither consulted by query evaluation nor auto-maintained on DML in this release; rebuild it after bulk changes. MATCH … AGAINST nonetheless returns correct results on its own because it scans live row data directly.
