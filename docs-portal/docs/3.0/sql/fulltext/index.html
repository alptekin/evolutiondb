12. Full-Text Search

EvolutionDB has a small full-text search feature for one text column. It is built on a shared tokenizer, an inverted B+ tree index, and the MATCH (col) AGAINST ('query') expression. The expression scores each row by how many query words it contains.

Full-text search helps you find rows whose text columns contain one or more search words. It does not match a literal substring or a LIKE pattern. The whole feature is small on purpose. It has three parts. They all share one tokenizer. This way the words seen when the index is built are the same words seen at query time. The three parts are listed below.

  • A tokenizer. It lowercases text, splits on non-alphanumeric boundaries, drops very short words, and removes a small list of English stop words.
  • A full-text index made with CREATE FULLTEXT INDEX. It stores one inverted-index entry per (word, row) pair in a B+ tree.
  • The MATCH … AGAINST expression. It checks a query string against a text column and returns a hit count for each row.

How this differs from PostgreSQL. EvolutionDB does not have PostgreSQL's tsvector and tsquery types. It also has no @@ match operator, no GIN or GiST indexes, no configurable text-search dictionaries, no stemming, no weighting, no phrase search, and no ts_rank or ts_headline. The syntax here is closer to MySQL's MATCH (col) AGAINST ('text'). The way it works is simpler than both. Matching is a plain whole-token equality check. There is no stemming and there are no boolean-mode operators. Only the things described on this page work. Everything else is not built yet.

12.1. Tokenization

Tokenization turns a raw text value into a list of normalized words. The same routine runs when a full-text index is filled and when MATCH … AGAINST is evaluated. So the two always agree on word boundaries and on letter case.

These rules apply to the input text.

Word boundaries
A token runs across consecutive alphanumeric bytes. These are ASCII letters and digits. It also runs across any byte with a value of 0x80 or greater. Every other byte ends the current token. This includes spaces, punctuation, and ASCII symbols.
Case folding
ASCII letters are changed to lowercase. Bytes at or above 0x80 pass through unchanged. These are the bytes that make up UTF-8 multibyte characters. ASCII case folding on UTF-8 bytes has no defined result. So non-ASCII letters are left as they are on purpose.
Minimum length
Tokens shorter than 2 bytes are discarded.
Stop words
Tokens that match the built-in English stop-word list are discarded. See 12.2.
Maximum token length
A token holds at most 63 bytes of content. This is the 64-byte buffer minus its terminator. Longer tokens are cut down to that length without warning.
Maximum token count
At most 256 tokens are taken from a single text value when indexing. At most 32 tokens are taken from the AGAINST query string.

The byte-level rule keeps multibyte letters attached to their words. Turkish and other non-Latin letters are examples. But the rule does not change their case. So non-ASCII text matches in a case-sensitive way in this release. A query term must match the stored term byte for byte in its non-ASCII part.

This example shows how a sentence is tokenized.

-- input:  "The Quick, Brown FOX!"
-- output: ["quick", "brown", "fox"]
--   * "The" -> stop word, dropped
--   * "Quick"/"Brown"/"FOX" -> lowercased
--   * punctuation splits and is discarded

Note. There is no stemming and no lemmatization. running, ran, and run are three different tokens. They will not match each other. There are also no synonyms, no thesaurus expansion, and no accent folding.

12.2. The stop-word list

The tokenizer removes a fixed, built-in list of common English words. You cannot change the list in this release. These words are dropped wherever they appear.

Stop words (29 entries)
a, an, and, are, as, at, be, but, by, for, if, in, is, it, no, not, of, on, or, so, that, the, this, to, was, were, will, with

Single-character words like I or a are dropped anyway by the two-byte minimum-length rule. The stop-word check runs after lowercasing. So it is case-insensitive for ASCII words.

12.3. The MATCH … AGAINST expression

Full-text matching is written with the MATCH (column) AGAINST ('query') construct. It is a scalar expression. It names exactly one column and one quoted query string.

MATCH ( column_name ) AGAINST ( 'query string' )

The grammar accepts one bare column name inside the MATCH (...) parentheses. It accepts one string literal inside AGAINST (...). The query literal can use single or double quotes. The outer quotes are removed before tokenization.

12.3.1. Evaluation semantics

When a row is evaluated, EvolutionDB follows these steps.

  1. It reads the named column's value for that row.
  2. It tokenizes the column value, up to 256 tokens. It also tokenizes the AGAINST string, up to 32 tokens. It uses the rules in 12.1.
  3. It counts how many query terms appear among the row's terms. Each query term adds at most 1 to the score. A term that appears many times in the row still counts once.
  4. It returns that count as the value of the expression.

This is an any-term (OR) match. A row scores above zero if it contains any of the query words. The expression has no boolean-mode operators. There is no +, no -, no quoted phrases, and no wildcards. Every word in the query string is treated as its own OR term.

If the column is missing, NULL, or empty, the expression returns 0. The same happens if every query term is filtered out. This happens when the terms are all stop words or all too short. Then the score is 0 for every row.

12.3.2. MATCH in WHERE, filtering

In a boolean context like a WHERE clause, the score is turned into a truth value in the usual way. A non-zero score is true. A score of zero is false. So WHERE MATCH(col) AGAINST('term') returns exactly the rows that contain at least one of the query terms.

CREATE TABLE articles (
    id    INT PRIMARY KEY,
    body  VARCHAR(2000)
);

INSERT INTO articles VALUES
    (1, 'The quick brown fox jumps over the lazy dog'),
    (2, 'A fast brown rabbit hops past the fence'),
    (3, 'Database engines store rows in pages');

-- rows 1 and 2 both contain "brown"
SELECT id FROM articles
WHERE MATCH(body) AGAINST('brown');

-- any-term match: row 1 ("fox"+"lazy") and row 2 ("rabbit") all qualify
SELECT id FROM articles
WHERE MATCH(body) AGAINST('fox rabbit lazy');

12.3.3. MATCH in the select list and ORDER BY, ranking

MATCH … AGAINST returns a hit count, not just a boolean. So you can show it as a relevance score. You can also use it to order results. A higher score means more distinct query terms were found in the row.

-- project the score; rank most-relevant first
SELECT id,
       MATCH(body) AGAINST('quick brown fox') AS score
FROM articles
ORDER BY MATCH(body) AGAINST('quick brown fox') DESC;

-- combine relevance ranking with a relevance filter
SELECT id
FROM articles
WHERE    MATCH(body) AGAINST('quick brown fox')
ORDER BY MATCH(body) AGAINST('quick brown fox') DESC;

The score you get is an integer. Look at the example data. Row 1 contains quick, brown, and fox. It scores 3. Row 2 contains only brown. It scores 1.

Note. PostgreSQL's ts_rank adjusts the score for document length and term frequency. This score does not do that. It is just the number of distinct query terms present in the row. So the highest possible score equals the number of distinct words in the query that are not stop words and not too short.

12.4. Creating a full-text index

A full-text index is created with the FULLTEXT keyword on CREATE INDEX.

CREATE FULLTEXT INDEX index_name ON table_name ( column_name )
CREATE FULLTEXT INDEX idx_articles_body
    ON articles (body);

Inside, this builds a B+ tree. Each key joins a tokenized word to the row's primary-key value with a unit-separator byte. That byte is 0x1F. One (word, primary-key) entry is inserted for each token of the indexed column. This is done for every existing row. The value stored with each key is the row's heap location. This is a classic inverted index. Looking up a word gives you the rows that contain it.

At creation time, EvolutionDB scans the table once. It tokenizes the indexed column for each row. Then it inserts the entries for each token. The build runs under the engine's DML lock. So the table is held still. No MVCC visibility filtering is applied during the build.

12.4.1. Restrictions

Single text column
A full-text index covers exactly one column. That column must hold text. The index column list is parsed in a general way. But the full-text build path uses one named column. The named column must exist on the table. If it does not, the statement fails with a column not found error.
No IF NOT EXISTS, UNIQUE, or USING HASH
The FULLTEXT form is its own grammar production. It does not work with the unique, hash, or IF NOT EXISTS forms of CREATE INDEX.
Inline table-definition form
A FULLTEXT INDEX (col) or FULLTEXT KEY (col) clause is also accepted inside a CREATE TABLE definition. This is for compatibility. There it is recorded as a text-index column.

12.5. Index maintenance and the query path

This release has two limits. You should understand them together. They affect how much you can rely on a full-text index. They also affect whether you should rely on it at all.

Important. The index is not used by query evaluation. In this release, MATCH … AGAINST always works by reading each candidate row's column value from the heap. It tokenizes that value on the fly. It does not read the inverted B+ tree. You can create and store the full-text index. But it does not make MATCH queries faster yet. Results are always correct because they come from the live row data. They are just produced by a per-row scan and not by an index lookup.

Important. The index is not kept up to date on its own. The index is not updated step by step on INSERT, UPDATE, and DELETE. That feature is not built. The index shows the rows that were present when it was built. After large data changes, drop the index and create it again with CREATE FULLTEXT INDEX to refresh it. When a full-text index is created, the server prints a note to remind you of this.

So for now, you can use MATCH … AGAINST for filtering and ranking without creating an index. The expression works on its own. Creating the index does not change query results or speed. It is storage put in place for a future index-backed lookup path.

12.6. Reference summary

12.6.1. Keywords and syntax

MATCH ( col ) AGAINST ( 'query' )
Scalar expression. It returns the integer count of distinct query terms found in col for the current row. You can use it in SELECT, WHERE, and ORDER BY.
CREATE FULLTEXT INDEX name ON table ( col )
Builds an inverted B+ tree index over the tokenized words of one text column. See CREATE INDEX.
FULLTEXT INDEX ( col ) / FULLTEXT KEY ( col )
Inline full-text index clause. It is accepted inside CREATE TABLE.

Raised by CREATE FULLTEXT INDEX when the named column does not exist on the target table. The index is not created in this case.

Note. Summary of limits. Full-text search in EvolutionDB 3.0 is whole-token, OR-only matching. Case folding works for ASCII only. There is no stemming, no phrase search, and no boolean-mode operators. There are no configurable dictionaries. There is no ranking weighted by length or term frequency. There is no tsvector or tsquery support. You can create a full-text index. But in this release it is not read by query evaluation and not kept up to date on DML. Rebuild it after large changes. MATCH … AGAINST still returns correct results on its own because it scans live row data directly.
