11. Transactions & MVCC

How EvolutionDB groups statements into atomic transactions, isolates concurrent connections with multi-version snapshots, and guards rows from conflicting writers.

This chapter describes the concurrency model implemented in EvolutionDB 3.0: explicit transaction blocks, savepoints, the four SQL isolation levels, the snapshot-based visibility rule that lets readers and writers run without blocking each other, and the row- and gap-locking machinery used by writers and by the SERIALIZABLE level. For the exact syntax of individual statements, follow the cross-links to the transaction, savepoint, SET, and SELECT reference pages.

11.1. Overview

Every statement in EvolutionDB runs inside a transaction. If you have not opened an explicit block, each statement runs in its own implicit, auto-committed transaction: the change is applied and made durable as soon as the statement succeeds. Opening a block with BEGIN defers that commit until you say so, so that a group of changes either all take effect together or none of them do.

Concurrency control combines two independent mechanisms:

Undo log (rollback)
A per-connection, thread-local stack of undo entries. Every INSERT, UPDATE, and DELETE inside an open block pushes an entry recording how to reverse it. COMMIT discards the log; ROLLBACK replays it in reverse. This provides per-connection atomicity.
MVCC (isolation)
Each transaction is assigned a 32-bit transaction id (XID). Row versions carry the XID that created them (xmin) and the XID that deleted them (xmax). A transaction reads through a snapshot that decides which versions are visible, so reads never block writes and writes never block reads.

The two layers cooperate: an UPDATE or DELETE does not physically overwrite or remove a row, it sets the old version's xmax to the deleting XID (a "soft delete") and, for an update, inserts a new version. Rollback then simply clears xmax to revive the original version, falling back to a re-insert only for rows written before MVCC existed.

Note. Durability is provided by a write-ahead log: before a COMMIT is acknowledged the transaction's dirty pages are flushed to the WAL, a commit sequence number (CSN) is assigned, and the transaction is marked committed in the commit log (CLOG). If the optional synchronous-commit mode is enabled (EVOSQL_SYNC_COMMIT=1), the commit additionally waits for replica acknowledgement of its WAL position, degrading silently to asynchronous commit on timeout.

11.2. Transaction control statements

Transaction control commands are recognized by prefix at the session layer (they do not go through the SQL parser). The recognized verbs are:

BEGIN
START TRANSACTION

COMMIT
END

ROLLBACK

SAVEPOINT savepoint_name
ROLLBACK TO [ SAVEPOINT ] savepoint_name
RELEASE [ SAVEPOINT ] savepoint_name

11.2.1. BEGIN / START TRANSACTION

BEGIN and its synonym START TRANSACTION open an explicit block. On BEGIN the server creates a fresh undo log, initializes an empty savepoint stack, assigns the transaction's XID, and registers it as active. At REPEATABLE READ and SERIALIZABLE a single snapshot is captured immediately (see 11.4); at READ COMMITTED and READ UNCOMMITTED no transaction-wide snapshot is taken, each statement snapshots on its own.

BEGIN;
INSERT INTO accounts VALUES ('alice', 100);
UPDATE accounts SET balance = balance - 25 WHERE name = 'alice';
COMMIT;
Note. The isolation level is not taken from words typed after BEGIN. Because the verb is matched by prefix, any trailing text (for example BEGIN ISOLATION LEVEL SERIALIZABLE) is accepted but ignored. Use SET TRANSACTION ISOLATION LEVEL first. Issuing BEGIN while already inside a block logs a warning and is otherwise a no-op (the existing block continues), it does not open a nested transaction.

11.2.2. COMMIT / END

COMMIT (synonym: END) finalizes the block. The server flushes dirty pages to the WAL, obtains a CSN, marks the XID committed in the CLOG, releases all row and gap locks held by the transaction, unregisters the XID from the active list, and discards the undo log. From that moment the changes are durable and visible to other connections whose snapshots are taken after the commit.

11.2.3. ROLLBACK

ROLLBACK abandons the block. The undo log is replayed in reverse, restoring every inserted, updated, or deleted row; the XID is marked aborted in the CLOG; and its locks are released. Because deletes and updates are soft, rollback usually just clears the xmax marker on the affected tuples rather than rewriting any data.

DML in the blockUndo entryWhat ROLLBACK does
INSERTTX_OP_INSERTDelete the inserted row and remove its primary-key index entry.
UPDATETX_OP_UPDATEDelete the new version; restore the old version by clearing its xmax (re-insert old data if the old slot was physically gone).
DELETETX_OP_DELETEClear xmax to "un-delete" the tuple (re-insert old data for pre-MVCC rows).
Note. If a statement inside the block raises an error, the transaction enters an aborted state. While aborted, only ROLLBACK, COMMIT/END, SAVEPOINT, and RELEASE are accepted; any other statement is rejected until the block is closed. Issuing COMMIT on an aborted transaction performs an implicit ROLLBACK, all work is discarded, matching PostgreSQL. If a connection disconnects with a block still open, the server automatically rolls it back.

11.3. Savepoints

Savepoints mark a position in the undo-log stack so that you can roll back part of a transaction without abandoning the whole block. They are only valid inside an open transaction. See the SAVEPOINT reference for full syntax.

SAVEPOINT name
Records the current undo-log depth under name and pushes it on the savepoint stack. Re-using an existing name overwrites it in place (PostgreSQL behavior). Names are matched case-insensitively. Up to 64 savepoints may be active at once.
ROLLBACK TO [SAVEPOINT] name
Replays undo entries down to the recorded depth, undoing all work after the savepoint, then destroys any savepoints created after it. The savepoint itself survives and can be rolled back to again. This also clears the aborted state, so you can continue the transaction after a statement error if you had a savepoint before it.
RELEASE [SAVEPOINT] name
Forgets name and all newer savepoints without undoing any work. The data changes remain part of the still-open transaction.
BEGIN;
INSERT INTO log VALUES (1, 'start');
SAVEPOINT sp1;
INSERT INTO log VALUES (2, 'maybe');
ROLLBACK TO sp1;          -- row (2,'maybe') is undone; row (1,'start') stays
INSERT INTO log VALUES (3, 'final');
COMMIT;                    -- rows (1,'start') and (3,'final') persist
Note. Savepoints in EvolutionDB are positions in the per-connection undo log. They roll back data changes recorded since the savepoint, but they do not capture a separate MVCC sub-snapshot: the visibility snapshot in use is the transaction's (or the statement's), not the savepoint's. Issuing SAVEPOINT, ROLLBACK TO, or RELEASE outside a transaction raises SQLSTATE 25P01; referring to an unknown name raises SQLSTATE 3B001 (savepoint does not exist).

11.4. Isolation levels

EvolutionDB recognizes all four standard isolation levels. Set the level for the session (and for the next transaction) before opening a block, using SET:

SET [ SESSION ] TRANSACTION ISOLATION LEVEL
    { READ UNCOMMITTED | READ COMMITTED | REPEATABLE READ | SERIALIZABLE }

The default level for a new connection is READ COMMITTED. Setting the level once a transaction has already issued a query raises SQLSTATE 25001; an unrecognized level raises SQLSTATE 22023.

LevelInternal codeSnapshot strategyExtra machinery
READ UNCOMMITTED0Per-statement snapshot (treated like READ COMMITTED).None.
READ COMMITTED (default)1A fresh snapshot is taken at the start of every statement.None.
REPEATABLE READ2A single snapshot taken at BEGIN is reused for the whole transaction.None.
SERIALIZABLE3A single snapshot taken at BEGIN, like REPEATABLE READ.Conflict guard (write-skew detection) + gap locks; serializer mutex.

Honest comparison with the SQL standard and PostgreSQL:

  • READ UNCOMMITTED. The standard permits dirty reads at this level. EvolutionDB does not implement dirty reads, uncommitted versions are never made visible, so this level behaves exactly like READ COMMITTED. This matches PostgreSQL, which also promotes READ UNCOMMITTED to READ COMMITTED.
  • READ COMMITTED. Each statement sees a snapshot taken when it begins, so two statements in the same transaction can see different committed data (non-repeatable reads and phantoms are possible). This is the standard meaning of the level.
  • REPEATABLE READ. The whole transaction reads through one snapshot, so rows it has already read do not change underneath it. EvolutionDB does not perform first-updater-wins abort detection at this level, so it does not guarantee the stricter "snapshot isolation" anomaly set that PostgreSQL's REPEATABLE READ provides.
  • SERIALIZABLE. Adds a conflict guard that tracks read/write dependencies and rejects a commit that forms a dangerous structure (write skew), plus gap locks to block phantom inserts into a scanned range. To preserve a globally serial order it also takes a process-wide mutex for the life of the transaction, so concurrent DML from other connections blocks until this transaction commits or rolls back. A failed serialization check aborts the commit with SQLSTATE 40001 (could not serialize access due to read/write dependencies among transactions).
Note. Because SERIALIZABLE holds a global DML mutex for the transaction's lifetime, it serializes writes across all connections rather than allowing the optimistic concurrency PostgreSQL's SSI permits. Treat it as a correctness tool for critical sections, not as a high-throughput level. The conflict guard tracks a bounded number of read interests and predicate locks per transaction; it catches common write-skew patterns but is not a complete dependency-graph analysis.

11.5. Snapshots & visibility

A snapshot is a point-in-time view of transaction activity. It records:

xmin
The lowest XID still active when the snapshot was taken. Every XID below this is finished.
xmax
One past the highest XID assigned so far. Every XID at or above this is "in the future" and invisible.
active_xids[]
The list of XIDs that were in progress at snapshot time (up to 256 concurrent transactions are tracked).
my_xid
The XID of the transaction that took the snapshot, used to recognize its own writes.
snapshot_csn
The highest committed CSN at snapshot time, enabling an O(1) "committed before me" fast path.

A heap tuple stores its creator XID (xmin) and deleter XID (xmax) in its MVCC header. The visibility predicate decides whether a tuple is visible to a snapshot roughly as follows:

  1. Tuples without an MVCC header (pre-MVCC rows) are always visible. A frozen xmin (the reserved value 1) is treated as committed long ago and always visible.
  2. If xmin is the snapshot's own XID, the tuple is the reader's own insert: visible unless the same transaction also deleted it.
  3. Otherwise xmin must be committed in the CLOG, below xmax, and not in the active list. Only then is the creating transaction's work visible.
  4. If xmax is 0 the tuple was never deleted and is visible. If the deleter is this transaction, it is invisible. If the deleter has not committed, committed after the snapshot, or was active at snapshot time, the tuple is still visible. Only when the deleter committed and is visible in this snapshot is the tuple considered dead.

To avoid repeating CLOG and CSN lookups, the engine uses hint bits (XMIN_COMMITTED / XMIN_ABORTED stamped on the tuple after the first successful check) and a CSN ring-buffer cache. None of this changes the result of the predicate, it only makes it faster.

11.5.1. Temporal queries (FOR SYSTEM_TIME)

You can read the database as of a past transaction by constructing a synthetic snapshot positioned just after a given XID committed: every committed XID up to and including that value is visible, and anything assigned later is invisible. EvolutionDB exposes this with a SQL:2011-style opener on SELECT:

-- current transaction id of this session
SELECT evo_current_xid();

-- read a table as it stood right after transaction 42 committed
SELECT * FROM orders FOR SYSTEM_TIME AS OF TRANSACTION 42;
Note. The historical view does not track concurrency, it has no active list, so it shows a clean committed-only image at the chosen XID. How far back you can go is bounded by retention: the CSN/commit-timestamp cache is a ring buffer, and a snapshot whose XID has aged out of the retained window is rejected rather than returning a misleading result.

11.6. Row locks & explicit locking

Readers and writers do not block each other through MVCC, but writers to the same row must serialize. EvolutionDB provides a row-level lock manager keyed by (table_id, primary-key value). Locks are held by XID and are released automatically at COMMIT, ROLLBACK, end of an auto-commit statement, or disconnect.

ModeAcquired byCompatibility
EVO_LOCK_EXCLUSIVEUPDATE, DELETE, SELECT ... FOR UPDATESingle holder; conflicts with everything.
EVO_LOCK_SHAREDSELECT ... FOR SHARECompatible with other shared holders; conflicts with exclusive.

Acquisition is re-entrant: the same XID re-locking a row it already holds is a no-op, and a shared lock can be upgraded to exclusive by the same holder. A request that conflicts with another transaction's lock waits on a condition variable until the lock is released or a timeout expires.

BEGIN;
SELECT balance FROM accounts WHERE name = 'alice' FOR UPDATE;
UPDATE accounts SET balance = balance - 10 WHERE name = 'alice';
COMMIT;   -- row lock released here

11.6.1. Lock waits, timeout & deadlocks

The default lock wait timeout is 5000 ms. A per-session override can be set with SET EVO_LOCK_TIMEOUT = ms (0 restores the default). If a waiter is not granted the lock within the timeout, the operation fails with a lock-timeout error (EVO_LOCK_TIMEOUT).

Before sleeping, the lock manager runs a wait-for-graph cycle check: it follows the chain of "who is each holder waiting for" and, if that chain leads back to the requester, reports a deadlock immediately (EVO_LOCK_DEADLOCK) instead of waiting for the timeout. The search is bounded to a fixed depth.

EVO_LOCK_OK (0)
Lock granted (or no lock needed).
EVO_LOCK_TIMEOUT (-1)
Wait exceeded the timeout.
EVO_LOCK_ERROR (-2)
Internal lock-manager error.
EVO_LOCK_DEADLOCK (-3)
Granting the wait would create a wait-for cycle.
Note. Shared locks are tracked with a single holder slot per row (last writer wins for the recorded holder), so FOR SHARE is treated as compatible but not fully multi-holder. SKIP LOCKED is accepted on FOR UPDATE/FOR SHARE for syntax compatibility. Row locks are keyed by primary key, so a table without a usable primary key cannot take a row lock (the request is a no-op).

11.6.2. Gap locks (phantom prevention)

To prevent phantom inserts in SERIALIZABLE transactions, a range scan can take a gap lock covering a key interval (lo, hi] on a table (an empty bound means −∞ or +∞). When another transaction tries to INSERT a key that falls inside an active gap lock held by a different XID, the insert waits, with the same timeout as row locks, until the gap lock is released at the holder's commit or rollback. Gap locks do not conflict with each other; they conflict only with inserts. They are taken automatically during SERIALIZABLE range scans and are not exposed as an explicit statement.

11.7. XID lifecycle & reserved ids

Transaction ids are 32-bit values handed out by a monotonic counter persisted in the database file header, so they survive restart. A handful of values are reserved:

0, pre-MVCC sentinel
A tuple with no MVCC header; always visible. Also used as "no deleter" in xmax.
1, XID_FROZEN
"Committed long ago"; an xmin of 1 is always visible without consulting the CLOG.
2+, normal
The first assignable transaction id; assigned in increasing order.

The CLOG stores two status bits per XID, 00 in-progress, 01 committed, 10 aborted, packed into 4 KB pages, with each page covering 16,320 ids and chained for overflow. At startup the server records the first XID it assigns; any XID below that value still marked in-progress belongs to a session that crashed and is treated as effectively aborted, so its half-written rows never become visible.

Note. XID assignment is lazy for auto-commit DML: a transaction id is not allocated until the first write, so read-only statements do not consume ids. The active transaction array holds up to 256 entries; this caps the number of concurrent transactions that snapshots can track precisely.
