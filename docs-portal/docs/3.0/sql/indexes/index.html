10. Indexes

EvolutionDB implements four index access methods: the default B+ tree, an FNV-1a hash index, an in-memory HNSW graph for vector search, and an inverted full-text index: each with its own on-disk (or in-memory) layout, its own key encoding, and a distinct set of queries it can accelerate.

An index is an auxiliary structure the engine can consult instead of scanning every row of a table. EvolutionDB builds them with CREATE INDEX (and CREATE FULLTEXT INDEX) and removes them with DROP INDEX. Every table also carries one index you do not create explicitly: its primary-key index, a B+ tree whose entries point at the heap location (a RowID of page_no:4 + slot_idx:2) of each live row. Secondary indexes use the same RowID value, so a matched index entry reaches its row directly, without a second primary-key lookup.

Scope of this chapter. Only the access methods EvolutionDB 3.0 actually implements are documented here, and only with the behavior present in the source (btree2.c, hash_idx.c, hnsw.c, FullText.c). Where EvolutionDB borrows PostgreSQL or MySQL surface syntax but diverges: for example, the in-memory-only HNSW graph, or the full-text index that is not auto-maintained on DML, the difference is called out explicitly. Statement-level syntax and option lists live on the CREATE INDEX reference page.

10.1. Choosing an Access Method

The four methods are not interchangeable. Each is keyed differently and answers a different shape of query. The table below summarizes when each applies; the rest of the chapter describes them in detail.

MethodCreated withBest forCannot do
B+ tree (default)CREATE INDEXEquality, range scans (<, <=, >, >=), ordered traversal, prefix lookups on secondary keys,
HashCREATE INDEX … USING HASHSingle-value equality lookups on a secondary columnRange scans, ordered traversal
HNSWCREATE INDEX … USING HNSWApproximate k-nearest-neighbor search over a VECTOR columnExact filtering, persistence across restart (in-memory only in v1)
Full-textCREATE FULLTEXT INDEXWord/token matching in text columns via MATCH … AGAINSTPhrase/ranked search; auto-maintenance on DML

All four store their pages, where they have pages, in the same unified 4 KB page file as table data, allocated through the Page Manager. The HNSW graph is the exception: it lives entirely in process memory.

10.2. B+ Tree Indexes

The B+ tree is EvolutionDB's default and most capable access method. It is what every primary key uses, and what CREATE INDEX builds unless you ask for another method. Internally it is a page-based B+ tree (btree2.c): every node is one 4 KB page, leaf nodes hold the actual (key, RowID) entries, and internal nodes hold only separator keys and child page pointers.

-- single-column B+ tree index
CREATE INDEX idx_users_email ON users (email);

-- enforce uniqueness while indexing
CREATE UNIQUE INDEX idx_users_username ON users (username);

-- composite index over two columns
CREATE INDEX idx_orders_cust_date ON orders (customer_id, order_date);

10.2.1. Node Layout and Capacity

Each node page begins with the standard 16-byte page header, followed by a 4-byte node header (a 16-bit key count plus a reserved 16-bit flags field). That leaves 4076 bytes of node capacity for entries. A leaf entry is [key_len:1][key:N][RowID:6]; an internal node interleaves num_keys separator keys with num_keys + 1 child pointers. Because a single byte prefixes each key length, an index key is capped at 255 bytes (BT2_MAX_KEY_LEN); a longer key is rejected at insert time.

Leaf pages are doubly linked through the page header's prev_page/next_page fields. This sibling chain is what makes ordered scans and ranges cheap: after the tree is descended once to locate the starting leaf, the cursor walks the chain rather than re-descending for each key.

10.2.2. Key Comparison Is Numeric-Aware

Keys are stored as length-prefixed byte strings, but comparison is not plain lexicographic. The comparator first tries to parse both keys as 64-bit integers; if both parse cleanly it compares them numerically. Failing that it tries floating-point. Only if neither side is a number does it fall back to a byte-wise memcmp (with the shorter key ordering first on a common prefix).

Why this matters. Numeric-aware comparison means an integer key like 100 correctly sorts after 20, even though both are stored as text, you do not get the lexicographic surprise of "100" < "20". The trade-off is that values which happen to look numeric are ordered numerically regardless of the column's declared type. Comparison buffers are bounded at 256 bytes, so the numeric fast path only applies to keys shorter than 255 bytes; longer keys always take the lexicographic path.

10.2.3. Secondary Keys and Composite Indexes

A secondary B+ tree does not key on the column value alone. To keep entries unique even when many rows share an indexed value, the engine forms a composite key by joining the indexed value with the row's primary key using the ASCII unit separator \x1F (indexed_value␟PK). A lookup for one indexed value is therefore a prefix scan: position the cursor at the first key whose prefix matches, then walk the leaf chain collecting entries until the prefix changes. A multi-column (composite) index instead joins the participating column values with | into one key.

10.2.4. Inserts, Splits, Deletes, and Compaction

Inserts keep each leaf sorted. When an entry will not fit in a leaf's 4076 bytes, the leaf splits: the upper half of its entries move to a freshly allocated page, the sibling chain is relinked, and the first key of the new page is pushed up to the parent as a separator. A split can cascade up the tree, and a split at the root creates a new root one level taller. The tree therefore grows from the leaves upward and stays balanced.

Deletes are lazy tombstones: bt2_delete finds the entry and overwrites its RowID with {0,0} rather than physically removing it or rebalancing. This deliberately keeps the tree structurally stable, the root never moves and no page is freed, so the system catalog and hot-path callers can hold leaf-page hints and live cursors safely. A subsequent insert of the same key reuses a tombstoned slot. Searches and cursor scans skip tombstones.

Physical space reclamation is a separate, offline pass, bt2_compact, run by RECLAIM. It drops tombstones, then borrows or merges entries across underflowing sibling nodes (a min-fill heuristic of half a page), frees emptied pages, and collapses a single-child root. Because it can move the root, it must run under the global DML lock with no concurrent readers holding page hints or cursors, exactly the conditions the RECLAIM path establishes.

Note. The min-fill target used by compaction (half of NODE_CAPACITY) is a space-usage heuristic, not a correctness invariant. The only hard invariant is that a node's bytes never exceed the 4076-byte capacity, which both the borrow and the merge paths preserve. A node may legitimately sit below min-fill between compactions.

10.3. Hash Indexes

A hash index (USING HASH) trades the B+ tree's ordering for direct equality lookup. It is implemented as static hashing with overflow chains (hash_idx.c): a single directory page holds a fixed array of bucket pointers, and each non-empty bucket is a chain of bucket pages packed with entries.

-- equality-only index for token lookups
CREATE INDEX idx_sessions_token ON sessions USING HASH (token);

10.3.1. Buckets, Hashing, and Overflow

Bucket routing uses the FNV-1a hash function, taken modulo the bucket count. A new index is created with a default of 64 buckets (HASH_DEFAULT_BUCKETS) and is capped at 1017 buckets (HASH_MAX_BUCKETS) so the entire bucket array fits in one 4 KB directory page. Bucket pages are allocated lazily: a bucket holds no page at all until its first entry arrives. An entry is [key_len:2][key:N][RowID:6]; keys are limited to 255 bytes (HASH_MAX_KEY_LEN). When a bucket page fills, a new overflow page is allocated and linked into the chain through the page header.

Like a secondary B+ tree, a hash entry's stored key is the composite indexed_value␟PK. But only the portion before the \x1F separator is hashed for bucket routing, so every row sharing an indexed value lands in the same bucket, and an equality lookup hashes just the search value, walks that one bucket chain, and returns every entry whose key begins with indexed_value␟.

10.3.2. When a Hash Index Applies

A hash index answers exactly one kind of query: equality on the indexed value. Because the bucket layout encodes no ordering, it cannot serve a range predicate, an ORDER BY, or a prefix-of-a-prefix search. Delete removes an entry by shifting the remaining entries in its bucket page down over it (the bucket count is fixed; there is no rehash or split). For anything that needs ordering or ranges, use the default B+ tree instead.

Static, not extendible. The bucket count is chosen at create time and never changes, there is no dynamic resizing or extendible-hashing split in this version. As a table grows well beyond the bucket count, lookups degrade from near-constant time toward a per-bucket chain walk. Size the index for the table, or prefer a B+ tree when the cardinality is unknown or growing.

10.4. HNSW Vector Indexes

An HNSW index (USING HNSW) accelerates approximate nearest-neighbor (ANN) search over a VECTOR(dim) column, finding the rows whose embedding is closest to a query vector under a chosen distance metric. It is built end-to-end through CREATE INDEX but, unlike the other three methods, it does not live in the page file: the graph and its dense vector store are held entirely in process memory (hnsw.c).

-- build an HNSW index, choosing the distance metric via opclass
CREATE INDEX idx_docs_embedding ON docs
    USING HNSW (embedding vector_l2_ops)
    WITH (m = 32, ef_construction = 128);

10.4.1. How the Graph Is Built

At CREATE INDEX time the engine scans the table once, decodes the vector column's float4 payload straight from each live tuple (so the dimension is not bounded by any text-field buffer and 1024-dimension embeddings work), and appends each (primary key, vector) pair to a dense, row-major store. It then builds a multi-layer navigable graph over that store: each node receives a randomly assigned top level, and neighbor lists are populated by greedy beam search using the configured ef_construction candidate width, keeping up to m links per node per level (2 × m at the base layer). Graphs are tracked in a small static table keyed by (table_id, index_name), with a hard cap of 64 concurrent graphs (HNSW_MAX_GRAPHS) and primary keys up to 256 bytes.

In-memory only; rebuild required. The HNSW graph is not persisted in v1 and is not rebuilt automatically at server startup. After a restart, a pre-existing HNSW index must be re-established with a fresh CREATE INDEX (or a reindex). The DDL layer does record enough catalog metadata, column, distance kind, m, ef_construction, for automatic replay to be added in a later iteration. DROP INDEX frees the graph immediately.

10.4.2. Distance Metrics (opclasses)

The opclass written after the column selects the distance function the graph uses for both construction and search. EvolutionDB implements four:

vector_cosine_ops (default)
Cosine distance, 1 − cos(a, b). A zero-norm vector on either side yields distance 1.0. This is the default when no opclass is given, and the fallback for any unrecognized opclass.
vector_l2_ops
Euclidean (L2) distance, the straight-line distance between the two vectors.
vector_ip_ops
Negative inner product (−a·b), so that smaller is nearer, matching the other metrics' ordering. vector_inner_ops is accepted as a synonym.
vector_l1_ops
Manhattan (L1) distance, the sum of absolute per-component differences.

10.4.3. Searching the Index

Vector search is invoked from SQL through helper functions and distance operators evaluated by the expression engine. hnsw_knn returns a comma-separated list of the k nearest primary keys, ordered by ascending distance (k defaults to 10 and is capped at 256):

-- k nearest neighbors of a query vector, by primary key
SELECT hnsw_knn('docs', 'idx_docs_embedding', '[0.12, 0.04, ...]', 5);

-- the distance functions / operators rank directly in a query
SELECT id, l2_distance(embedding, '[0.12, 0.04, ...]') AS dist
    FROM docs ORDER BY embedding <-> '[0.12, 0.04, ...]' LIMIT 5;

The available distance expressions are cosine_distance(a, b), l2_distance(a, b), inner_product(a, b), and l1_distance(a, b), plus the operators <-> (L2), <#> (negative inner product), and <=> (cosine, usable in ORDER BY emb <=> '[…]'). Query vectors may be written as a bracketed literal '[f1, f2, …]' or, for compactness, in the client's b64i8:<base64> quantized form (a float32 scale followed by an int8 array), which keeps a 1024-dimension query under the statement size cap.

Internally, search picks one of two paths under the same interface. For a graph large enough to benefit, it does a multi-layer greedy descent followed by a base-layer beam search of width ef_search (64 by default, widened to at least k). For very small graphs, or if graph construction hit an allocation failure, it transparently falls back to an exact brute-force top-k scan over the dense store, so results are always correct, with the graph only changing how fast they come back.

10.4.4. Hybrid Search (Vector + Filter)

hnsw_filter_knn combines vector ranking with a single equality predicate over another column, supplied as a 'col=val' spec (optionally 'col=val:pre' or ':post' to force a strategy):

SELECT hnsw_filter_knn('docs', 'idx_docs_embedding', '[0.12, ...]', 5, 'lang=en');

-- report which strategy the planner chose and the filter selectivity
SELECT hnsw_hybrid_explain('docs', 'idx_docs_embedding', '[0.12, ...]', 5, 'lang=en');

In auto mode the engine measures the filter's selectivity (matching rows over total rows) and chooses: when fewer than 10% of rows match it pre-filters (rank only the matching rows), otherwise it post-filters (rank a wider, oversampled candidate set, then discard non-matches). An empty filter degrades to plain hnsw_knn. The filter predicate is equality only in this version.

10.5. Full-Text Indexes

A full-text index (CREATE FULLTEXT INDEX) supports word matching inside a text column. It is modeled as an inverted index built on a B+ tree (FullText.h): the text is tokenized and one entry is inserted per token, keyed <word>␟<PK> (the \x1F separator again), with the row's heap RowID as the value. A query tokenizes the search string the same way and looks up each token's word prefix.

-- build the inverted index over a text column
CREATE FULLTEXT INDEX idx_articles_body ON articles (body);

-- query it with MATCH ... AGAINST
SELECT id, title FROM articles
    WHERE MATCH(body) AGAINST('storage engine');

10.5.1. The Tokenizer

The same tokenizer (ft_tokenize) runs at both build time and query time, so the two see an identical token stream. Its rules are:

Word boundaries
A token is a run of ASCII alphanumerics, with any byte ≥ 0x80 treated as part of a word, this keeps multi-byte UTF-8 letters (e.g. Turkish characters) attached to their words. Any other byte ends the current token.
Lowercasing
ASCII letters are lowercased so matching is case-insensitive. Non-ASCII bytes pass through unchanged, because tolower on raw UTF-8 bytes is undefined; as a result, queries against non-ASCII (e.g. Turkish) text match case-sensitively in this version. A proper Unicode case-fold is future work.
Short-token filter
Tokens shorter than 2 characters are dropped.
Stop words
A small built-in English stop-word list (a, an, and, the, of, to, … 28 words) is removed.
Limits
A token is capped at 64 bytes (FT_MAX_TOKEN_LEN, longer tokens are truncated), and at most 256 tokens (FT_MAX_TOKENS) are extracted from a single field.

Note, not auto-maintained. The full-text index is not automatically updated on INSERT, UPDATE, or DELETE in this version. After bulk DML, drop and re-create it (CREATE FULLTEXT INDEX) to refresh the token entries. There is also no stemming, no phrase search, and no relevance ranking beyond token presence; matching is on the shared, normalized token stream described above.

10.6. Compatibility

Indexing is implementation-defined in the SQL standard, so none of these statements are part of standard SQL. EvolutionDB's surface follows PostgreSQL and MySQL closely but differs in concrete ways worth knowing:

  • Access methods. EvolutionDB offers the default B+ tree, USING HASH, and USING HNSW. PostgreSQL's gin/gist/spgist/brin method names, and a generic USING method clause, are not supported.
  • Hash indexes are static. Unlike PostgreSQL's dynamically growing hash index, EvolutionDB's bucket count is fixed at create time (64 by default, max 1017) with overflow chains; there is no rehash or extendible split.
  • HNSW. The opclass names (vector_cosine_ops, vector_l2_ops, vector_ip_ops, vector_l1_ops) and the m/ef_construction build parameters mirror the pgvector extension, but EvolutionDB's graph is in-memory only and is not auto-rebuilt at startup. A single VECTOR column is required; composite HNSW indexes are not supported.
  • Full-text. EvolutionDB uses a dedicated MySQL-style CREATE FULLTEXT INDEX statement and MATCH … AGAINST queries, rather than PostgreSQL's tsvector + GIN model. It is not auto-maintained on DML, and offers no stemming, phrase search, or ranked relevance.
  • Unsupported B+ tree / hash clauses. ASC/DESC, NULLS FIRST/LAST, COLLATE, per-column opclasses, INCLUDE, partial-index WHERE, TABLESPACE, and storage-parameter WITH (…) on non-HNSW indexes are all rejected.
