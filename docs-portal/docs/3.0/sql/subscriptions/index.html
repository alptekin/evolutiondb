18. Subscriptions & CDC

Durable subscriptions turn EvolutionDB's transient LISTEN/NOTIFY channels into persistent, replayable change-data-capture streams: notifications committed on a channel are spooled to an on-disk mailbox that a consumer can drain, acknowledge, and replay after disconnecting.

18.1. Overview

EvolutionDB ships an in-memory LISTEN/NOTIFY registry that broadcasts a payload to every connection currently listening on a channel. That delivery is best-effort: if no client is connected at commit time, the notification is gone. A durable subscription closes that gap. It registers a named, persistent mailbox bound to one channel; whenever a NOTIFY commits on that channel, the payload is appended to the mailbox with a monotonic sequence number. A consumer that was offline can reconnect later, replay everything that arrived in the meantime with RESUME SUBSCRIPTION, and advance an acknowledgement watermark with ACK SUBSCRIPTION so a background daemon can prune what it has already processed.

This is EvolutionDB's change-data-capture (CDC) primitive: application code (or triggers) emits NOTIFY on a channel, and one or more durable subscribers consume that stream at their own pace, with at-least-once delivery and replay across reconnects.

Relationship to in-memory NOTIFY. Durable spooling runs after the in-memory listener fan-out, inside the same commit-time broadcast path. Live listeners are served first; the catalog walk that feeds durable mailboxes only happens once the in-memory broadcast for the channel has been delivered, so the fast path pays nothing when no subscription is enrolled. See NOTIFY and LISTEN.

18.2. The subscription model

A subscription is a catalog object identified by a unique name and bound to exactly one channel. Each subscription owns a private backing table that acts as its mailbox.

18.2.1. Backing table

Creating a subscription named s creates a backing heap table named sub_s in the current schema with this fixed layout:

CREATE TABLE sub_s (
    seq        BIGINT          PRIMARY KEY,   -- monotonic, server-assigned
    payload    VARCHAR(8000),                  -- the NOTIFY payload; '' for payload-less NOTIFY
    posted_at  TIMESTAMP      NOT NULL        -- UTC append time, 'YYYY-MM-DD HH:MM:SS'
);

One row is written per published NOTIFY. The seq is the primary key, so the backing table's B+ tree already orders rows ascending by sequence, no explicit ORDER BY is required to drain in arrival order. The posted_at stamp is generated server-side in UTC at publish time.

18.2.2. Subscription catalog fields

The system catalog stores one descriptor per subscription. SHOW SUBSCRIPTIONS exposes these columns:

name text
Unique subscription name. Acts as the catalog key.
channel text
The single NOTIFY channel this subscription is bound to. Channel matching at publish time is case-insensitive.
last_ack_seq bigint
The acknowledgement watermark. Starts at 0 (nothing acknowledged). RESUME returns only rows with seq > last_ack_seq; the prune daemon deletes rows with seq <= last_ack_seq.
next_seq bigint
The sequence number to assign to the next published row. Starts at 1 and is bumped (and persisted) after each successful append.
ttl_days int
Retention window for unacknowledged rows, in days. Defaults to 7. A value of 0 disables the TTL prune (the acknowledged-row prune still runs).

18.3. Creating and dropping subscriptions

18.3.1. CREATE SUBSCRIPTION

CREATE SUBSCRIPTION name FOR CHANNEL channel
CREATE SUBSCRIPTION IF NOT EXISTS name FOR CHANNEL channel

Registers a durable subscription and provisions its sub_<name> backing table in one step. The watermark starts at 0, the next sequence at 1, and the TTL at the 7-day default. Both name and channel are required; omitting either raises a syntax error (42601).

CREATE SUBSCRIPTION orders_cdc FOR CHANNEL order_events;

If a subscription with that name already exists, the statement fails with 42710 unless IF NOT EXISTS is given, in which case it is a no-op.

18.3.2. DROP SUBSCRIPTION

DROP SUBSCRIPTION name
DROP SUBSCRIPTION IF EXISTS name

Unregisters the subscription and drops its backing table. Without IF EXISTS, dropping a missing subscription fails with 42704; with it, the statement is a no-op.

DROP SUBSCRIPTION IF EXISTS orders_cdc;

18.4. Consuming the stream

18.4.1. RESUME SUBSCRIPTION

RESUME SUBSCRIPTION name

Returns the backlog for the subscription as a result set of (seq, payload, posted_at), including every mailbox row whose seq is strictly greater than the current last_ack_seq, in ascending seq order. RESUME is read-only and idempotent: it never advances the watermark or deletes rows, so the same backlog is returned until you acknowledge it. Resuming an unknown subscription fails with 42704.

RESUME SUBSCRIPTION orders_cdc;
--  seq | payload                  | posted_at
-- -----+--------------------------+---------------------
--    1 | {"id":42,"op":"insert"}  | 2026-06-12 09:14:03
--    2 | {"id":43,"op":"insert"}  | 2026-06-12 09:14:05

18.4.2. ACK SUBSCRIPTION

ACK SUBSCRIPTION name UPTO seq

Advances the acknowledgement watermark to seq. The operation is monotonic and idempotent: it only moves last_ack_seq forward, so ACK ... UPTO 5 followed by ACK ... UPTO 3 leaves the watermark at 5. The acknowledged sequence need not correspond to an actually-stored row: the watermark simply moves, and the prune daemon catches up on its next tick. Acknowledging an unknown subscription fails with 42704.

ACK SUBSCRIPTION orders_cdc UPTO 2;

18.5. End-to-end lifecycle

A complete CDC loop combines the publisher side (NOTIFY) with the durable consumer side:

-- 1. enroll a durable subscriber on a channel
CREATE SUBSCRIPTION orders_cdc FOR CHANNEL order_events;

-- 2. producers emit notifications (e.g. from application code or a trigger)
NOTIFY order_events, '{"id":42,"op":"insert"}';
NOTIFY order_events, '{"id":43,"op":"insert"}';

-- 3. a consumer reconnects later and drains the backlog
RESUME SUBSCRIPTION orders_cdc;        -- rows seq 1, 2

-- 4. after processing, acknowledge progress so the daemon can prune
ACK SUBSCRIPTION orders_cdc UPTO 2;

-- 5. inspect every subscription's watermark and TTL
SHOW SUBSCRIPTIONS;

18.6. Retention & pruning

A background auto-reclaim daemon trims each subscription's backing table on a fixed interval. On every tick it walks the mailbox and deletes a row when either condition holds:

  • Acknowledged: the row's seq is <= last_ack_seq. This prune always runs, regardless of TTL.
  • TTL-expired: ttl_days > 0 and the row's posted_at is older than ttl_days × 86 400 seconds. This catches unacknowledged rows from consumers that never came back. Rows with a NULL timestamp are not TTL-pruned.

The daemon collects victims and bulk-deletes them under the same write lock used by other retention passes, removing each from both the heap and the PK B+ tree. It processes at most 256 rows per subscription per tick, so a very large backlog drains across several ticks rather than in one pass.

Tuning the interval. The reclaim interval defaults to 30 seconds. It can be lowered for tests via the EVOSQL_AUTO_RECLAIM_INTERVAL_SEC environment variable, which accepts a value from 1 to 30 seconds.

18.8. Comparison with PostgreSQL

EvolutionDB's syntax deliberately echoes PostgreSQL's CREATE SUBSCRIPTION keyword but the semantics are entirely different, these are not logical-replication subscriptions.

Source of data
PostgreSQL subscriptions consume a logical replication stream from a publisher's WAL. EvolutionDB subscriptions consume committed NOTIFY payloads on a named channel: there is no upstream node, publication, or replication slot.
Binding
PostgreSQL binds a subscription to a PUBLICATION on a remote CONNECTION. EvolutionDB binds a subscription to a local channel with FOR CHANNEL; there is no publication object and no connection string.
Delivery & replay
EvolutionDB models a pull-based mailbox with an explicit cursor: RESUME replays, ACK ... UPTO advances a watermark, and a daemon prunes. PostgreSQL applies changes automatically via apply workers; there is no equivalent client-driven RESUME/ACK surface.
Payload
An EvolutionDB payload is the free-form text of a NOTIFY (capped at 8000 bytes by the backing column), not a decoded row change. EvolutionDB does not interpret the payload; CDC structure (operation, keys, columns) is whatever the producer chooses to encode, typically JSON.
Standard SQL
Subscriptions, RESUME, and ACK are EvolutionDB extensions. They are not part of the SQL standard, and RESUME/ACK are EvolutionDB-specific verbs with no PostgreSQL counterpart.

Notes & limitations.

  • A subscription binds to exactly one channel, set at creation and not alterable; there is no ALTER SUBSCRIPTION. To rebind, drop and recreate.
  • ttl_days defaults to 7 and is not settable through DDL in this release; it is honored by the prune daemon but exposed read-only via SHOW SUBSCRIPTIONS.
  • The backing table is named sub_<name> in the current schema. It is an internal implementation detail and should be consumed only through RESUME/ACK, not queried or modified directly.
  • Delivery is at-least-once with no automatic deduplication: a consumer that does not ACK will see the same rows again on the next RESUME.
  • Durable spooling occurs at NOTIFY commit time. Notifications emitted in a transaction that rolls back are never spooled.
  • Channel matching is case-insensitive; subscription and channel names follow the catalog name-length limit.
