17. LISTEN / NOTIFY

EvolutionDB has a small, in-memory publish and subscribe channel. A connection signs up for a channel with LISTEN. Any connection can raise an event with NOTIFY. The server then pushes an async message to every listener. This works on both the PostgreSQL wire protocol and the native EVO text protocol.

17.1. Overview

Async notification lets applications signal each other without polling a table. A connection subscribes to a named channel. Later, the same or another connection publishes an event on that channel. The event can carry a short text payload. The server delivers the event to every subscribed connection as an out-of-band message. It arrives on the listener's socket even while the listener is running a query.

EvolutionDB has three statements and a small set of helper functions.

LISTEN
Subscribe the current session to a channel. See LISTEN / NOTIFY.
UNLISTEN
Cancel one subscription, or all of them with UNLISTEN *.
NOTIFY
Publish an event on a channel. A payload string is optional. The event is delivered when the transaction commits.

The registry is a process-local hash map. It is a djb2-hashed bucket array of 1024 slots. Each bucket is a linked list of channel records. Each channel is a linked list of listeners. The registry is not persisted and is not shared across server processes. Subscriptions live and die with the running server and with the connection that created them. For events that must survive a disconnected consumer, see §17.8 Durable subscriptions.

Identity and "PID". PostgreSQL reports the process id of the backend that sent the notification. EvolutionDB reports the sender's internal session id instead. The session id is a per-connection integer. It is not an operating-system PID. On the PG wire protocol it is sent in the 4-byte PID field of the NotificationResponse ('A') message. On the EVO protocol it is the first field of the NOTIFY async frame.

17.2. LISTEN, subscribing to a channel

LISTEN channel
LISTEN channel SELF

A LISTEN has two effects. Both happen right away when the statement runs. They are not deferred to commit. First, the (session, channel) pair is added to the global registry. Second, the channel is recorded in the session's own listen mirror so helper functions can list it. If the session already listens on that channel, the registration is an idempotent upsert. In that case only the self-delivery flag is updated. The flag is explained below.

LISTEN orders;
-- session is now subscribed to channel "orders"

17.2.1. Self-delivery (the SELF keyword)

By default a session does not receive notifications that it raised itself. This matches PostgreSQL's default behavior. EvolutionDB lets you turn on self-delivery with a special keyword instead of a parameter clause.

LISTEN orders SELF;
-- now this session WILL be delivered its own NOTIFY orders events

Honest contrast with PostgreSQL. PostgreSQL has no per-channel "self" switch. Same-session delivery is controlled globally by the client_min_messages and queue mechanics. A backend simply does not see its own notifications. EvolutionDB's LISTEN ... SELF is an EvolutionDB-specific extension. The grammar comment notes one reason for the keyword. It was chosen over a WITH (self = true) clause to avoid a token conflict on = in the LISTEN context.

17.3. UNLISTEN, cancelling subscriptions

UNLISTEN channel
UNLISTEN *

UNLISTEN channel removes this session's subscription to one channel. UNLISTEN * removes every subscription owned by the session. Both forms take effect right away. Both report the command tag UNLISTEN. Cancelling a channel you were not listening on is not an error. It does nothing.

UNLISTEN orders;   -- stop listening on one channel
UNLISTEN *;        -- stop listening on everything

When a channel's last listener is removed, the empty channel record is freed from the registry automatically.

You do not need to call UNLISTEN before disconnecting. When a connection closes, the server runs a disconnect hook. The hook removes all of that session's subscriptions. It also drops any pending notifications the session had queued but not yet committed.

17.4. NOTIFY, publishing an event

NOTIFY channel
NOTIFY channel, 'payload'

A NOTIFY raises an event on a channel. It can take an optional single-quoted payload string. NOTIFY does not fire right away, unlike LISTEN and UNLISTEN. It is enqueued onto the session's pending list. It is delivered when the surrounding transaction commits. See §17.5. The command tag is NOTIFY.

NOTIFY orders;
NOTIFY orders, 'order 4711 shipped';

Payloads are plain text. The parser strips the surrounding quotes and stages the payload for the executor. An empty payload and a missing payload are the same. Both deliver an empty string to listeners.

17.5. Transactional behavior

Notifications respect transaction boundaries. A NOTIFY is buffered until the transaction that issued it commits. If the transaction rolls back, the buffered notifications are discarded. They are never delivered.

  • Autocommit. Outside an explicit BEGIN, a NOTIFY is flushed and delivered right after the statement. A listener sees the message before the client's next response.
  • Explicit transactions. Inside BEGIN … COMMIT, all enqueued notifications are delivered at COMMIT. A ROLLBACK discards every pending notification for the session.
  • Savepoints. Each pending notification records the savepoint depth where it was raised. ROLLBACK TO SAVEPOINT discards only the notifications enqueued after that savepoint. Earlier ones stay queued for the eventual commit.
BEGIN;
NOTIFY orders, 'a';       -- queued (depth 0)
SAVEPOINT sp1;
NOTIFY orders, 'b';       -- queued (depth 1)
ROLLBACK TO sp1;            -- 'b' discarded, 'a' still queued
COMMIT;                      -- only 'a' is delivered

17.5.1. Commit-time de-duplication

At commit, before fan-out, the pending list is collapsed. Duplicate (channel, payload, sender) triples raised in the same transaction are merged into a single delivery. Channel names are compared without case. Payloads are compared exactly, with case. This matches PostgreSQL's rule that identical notifications within a transaction are folded. A tight loop of NOTIFY orders, 'x' delivers one message, not many.

17.6. Delivery model and self-filtering

At commit, the server handles each de-duplicated pending notification in turn. It takes a read-lock on the registry. It copies the current listener list for the channel into a fixed-size snapshot. It releases the lock. Only then does it write to each listener's connection. This snapshot-then-release pattern means network I/O never happens while the registry lock is held. That avoids deadlocks against per-connection write locks.

The self-delivery filter is applied during the snapshot. A listener whose session_id matches the sender's is skipped. The exception is a listener that subscribed with LISTEN ... SELF. See §17.2.1.

Each listener receives the message in its protocol's native async form.

ProtocolFrameFields delivered
PostgreSQL wire (5433)NotificationResponse (message type 'A')sender session id (in the PID field), channel name, payload
EVO text (9967)NOTIFY <session_id> <channel> <payload_len>\n<payload>\nsender session id, channel, payload length, then the payload bytes

Both protocols share the same in-memory registry. A NOTIFY from a psql client can wake an EVO-protocol listener. The reverse also works.

17.7. Helper functions

Two function-call forms are available inside SQL expressions. You can use them in code, for example from a stored procedure body or a trigger. See CALL.

EVO_NOTIFY(channel, payload)
Enqueues a notification. It works the same as the NOTIFY statement. PG_NOTIFY(channel, payload) is a PostgreSQL-compatible alias for the same function. Like the statement form, the event is buffered and delivered at commit.
PG_LISTENING_CHANNELS()
Returns the channels the current session is listening on. The list is read from the session's own listen mirror.
-- raise a notification from inside a procedure / trigger
SELECT EVO_NOTIFY('orders', 'inserted');

-- inspect this session's subscriptions
SELECT PG_LISTENING_CHANNELS();

17.8. Identifiers, limits, and error codes

Channel names are case-insensitive. They are compared with a folded hash and strcasecmp. So orders and ORDERS are the same channel. The fixed limits are listed below.

Channel name length
Maximum 63 characters. The buffer is 64 bytes including the terminator. This mirrors PostgreSQL's NAMEDATALEN. Over-long or empty names are rejected.
Payload length
Maximum 8000 bytes. An over-long payload is rejected before delivery.
Per-session LISTEN cap
A session may listen on at most 32 channels at once. The thirty-third distinct LISTEN is refused.
Snapshot fan-out bound
A single delivery copies up to 1024 listeners per channel into its snapshot.

Errors are reported with these SQLSTATE codes.

SQLSTATECondition
42622Channel name too long (≥ 64 bytes) or invalid; or per-session LISTEN cap exceeded.
22023Payload string too long (over 8000 bytes).

17.9. Durable subscriptions

Plain LISTEN and NOTIFY are fire-and-forget. A notification is delivered only to connections that are subscribed at the moment of commit. A consumer that is offline when the event fires misses it.

EvolutionDB adds a separate durable subscription feature on top of the same channels. It gives you an at-least-once, replayable queue. At commit, after the in-memory fan-out, each notification is also appended to every persistent mailbox bound to that channel. A consumer that reconnects later can drain the backlog. This is described in the Chapter on durable subscriptions and the CREATE SUBSCRIPTION command. The fast path is not affected. When a channel has no durable subscribers, no extra work is done.

Notes and limitations

  • The listener registry is in-memory and per-process. It is not written to evosql.db. It is not recovered after a restart. It is not visible to other server processes. Use durable subscriptions when an event must survive a disconnect or restart.
  • The "PID" reported to listeners is the sender's internal session id. It is not an OS process id.
  • Same-session notifications are filtered out by default. Turn them on per channel with LISTEN ... SELF. This is an EvolutionDB extension, not standard PostgreSQL syntax.
  • NOTIFY is commit-gated. A rolled-back transaction delivers nothing. The same is true for a ROLLBACK TO SAVEPOINT that covers the statement. Identical notifications within one transaction are collapsed to a single delivery.
  • Channel names are compared without case. Payloads are compared with case for de-duplication.
  • NOTIFY currently accepts only a channel name and an optional quoted string literal. The statement form does not yet support arbitrary expression payloads.
  • Disconnecting auto-cancels all of a session's listens. It also discards the session's uncommitted pending notifications.

17.10. End-to-end example

Connection A subscribes and waits. Connection B publishes.

-- connection A
LISTEN orders;
-- A now blocks/polls for async messages on the channel

-- connection B
BEGIN;
INSERT INTO orders (id, status) VALUES (4711, 'shipped');
NOTIFY orders, '4711 shipped';   -- queued
COMMIT;                              -- delivered to A here

-- connection A receives (PG wire): channel "orders", payload "4711 shipped",
-- PID = B's session id. Then it can stop:
UNLISTEN orders;
