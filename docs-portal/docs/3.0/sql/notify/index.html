17. LISTEN / NOTIFY

EvolutionDB ships a lightweight, in-memory publish/subscribe channel: connections register interest with LISTEN, any connection raises an event with NOTIFY, and the server pushes an asynchronous message to every listener — across both the PostgreSQL wire protocol and the native EVO text protocol.

17.1. Overview

Asynchronous notification gives applications a way to signal each other without polling a table. A connection subscribes to a named channel; later, the same or another connection publishes an event on that channel, optionally carrying a short text payload. The server delivers the event to every subscribed connection as an out-of-band message — it arrives on the listener's socket whether or not it is currently running a query.

EvolutionDB implements three statements and a small set of helper functions:

LISTEN
Subscribe the current session to a channel. See LISTEN / NOTIFY.
UNLISTEN
Cancel one subscription, or all of them with UNLISTEN *.
NOTIFY
Publish an event on a channel, optionally with a payload string. Delivery is gated on transaction commit.

The registry is a process-local hash map (a djb2-hashed bucket array of 1024 slots, each bucket a linked list of channel records, each channel a linked list of listeners). It is not persisted and not shared across server processes: subscriptions live and die with the running server and with the connection that created them. For events that must survive a disconnected consumer, see §17.8 Durable subscriptions.

Identity / "PID". Wherever PostgreSQL reports the notifying backend's process id, EvolutionDB reports the sender's internal session id instead. It is a per-connection integer, not an operating-system PID. On the PG wire protocol it is sent in the 4-byte PID field of the NotificationResponse ('A') message; on the EVO protocol it is the first field of the NOTIFY async frame.

17.2. LISTEN — subscribing to a channel

LISTEN channel
LISTEN channel SELF

Registering a listen has two effects, both applied immediately when the statement runs (not deferred to commit): the (session, channel) pair is added to the global registry, and the channel is recorded in the session's own listen mirror so helper functions can enumerate it. If the session is already listening on that channel, the registration is an idempotent upsert — only the self-delivery flag (below) is updated.

LISTEN orders;
-- session is now subscribed to channel "orders"

17.2.1. Self-delivery (the SELF keyword)

By default a session does not receive notifications that it raised itself, matching PostgreSQL's default behavior. EvolutionDB exposes opt-in self-delivery through a dedicated keyword rather than a parameter clause:

LISTEN orders SELF;
-- now this session WILL be delivered its own NOTIFY orders events

Honest contrast with PostgreSQL. PostgreSQL has no per-channel "self" switch — same-session delivery is governed globally by the client_min_messages/queue mechanics, and a backend simply does not see its own notifications. EvolutionDB's LISTEN ... SELF is an EvolutionDB-specific extension. The grammar comment notes it was chosen over a WITH (self = true) clause specifically to avoid a token conflict on = in the LISTEN context.

17.3. UNLISTEN — cancelling subscriptions

UNLISTEN channel
UNLISTEN *

UNLISTEN channel removes this session's subscription to a single channel; UNLISTEN * removes every subscription owned by the session. Both forms take effect immediately and both report the command tag UNLISTEN. Cancelling a channel you were not listening on is not an error — it is silently a no-op.

UNLISTEN orders;   -- stop listening on one channel
UNLISTEN *;        -- stop listening on everything

When a channel's last listener is removed, the empty channel record is freed from the registry automatically.

You do not need to call UNLISTEN before disconnecting: when a connection closes, the server runs a disconnect hook that removes all of that session's subscriptions and drops any pending (un-committed) notifications it had queued.

17.4. NOTIFY — publishing an event

NOTIFY channel
NOTIFY channel, 'payload'

A NOTIFY raises an event on a channel with an optional single-quoted payload string. Unlike LISTEN/UNLISTEN, NOTIFY does not fire immediately — it is enqueued onto the session's pending list and delivered when the surrounding transaction commits (see §17.5). The command tag is NOTIFY.

NOTIFY orders;
NOTIFY orders, 'order 4711 shipped';

Payloads are plain text. Internally the parser strips the surrounding quotes and stages the payload for the executor; an empty payload and an omitted payload are equivalent and deliver an empty string to listeners.

17.5. Transactional behavior

Notifications respect transaction boundaries. A NOTIFY is buffered until the transaction that issued it commits; if the transaction rolls back, the buffered notifications are discarded and never delivered.

  • Autocommit. Outside an explicit BEGIN, a NOTIFY is flushed and delivered immediately after the statement, so a listener sees the message before the client's next response.
  • Explicit transactions. Inside BEGIN … COMMIT, all enqueued notifications are delivered at COMMIT; a ROLLBACK discards every pending notification for the session.
  • Savepoints. Each pending notification records the savepoint depth at which it was raised. ROLLBACK TO SAVEPOINT discards exactly those notifications enqueued after that savepoint, leaving earlier ones queued for the eventual commit.
BEGIN;
NOTIFY orders, 'a';       -- queued (depth 0)
SAVEPOINT sp1;
NOTIFY orders, 'b';       -- queued (depth 1)
ROLLBACK TO sp1;            -- 'b' discarded, 'a' still queued
COMMIT;                      -- only 'a' is delivered

17.5.1. Commit-time de-duplication

At commit, before fan-out, the pending list is collapsed: duplicate (channel, payload, sender) triples raised in the same transaction are merged into a single delivery. Channel names are compared case-insensitively; payloads are compared exactly (case-sensitively). This matches PostgreSQL's "identical notifications within a transaction are folded" rule and means a tight loop of NOTIFY orders, 'x' delivers one message, not many.

17.6. Delivery model and self-filtering

At commit, for each (de-duplicated) pending notification the server takes a read-lock on the registry, copies the current listener list for the channel into a fixed-size snapshot, releases the lock, and only then writes to each listener's connection. This snapshot-then-release pattern means network I/O never happens while the registry lock is held, avoiding deadlocks against per-connection write locks.

The self-delivery filter is applied during the snapshot: a listener whose session_id equals the sender's is skipped unless that listener subscribed with LISTEN ... SELF (see §17.2.1).

Each listener receives the message in its protocol's native asynchronous form:

ProtocolFrameFields delivered
PostgreSQL wire (5433)NotificationResponse (message type 'A')sender session id (in the PID field), channel name, payload
EVO text (9967)NOTIFY <session_id> <channel> <payload_len>\n<payload>\nsender session id, channel, payload length, then the payload bytes

Both protocols share the same in-memory registry, so a NOTIFY from a psql client can wake an EVO-protocol listener and vice versa.

17.7. Helper functions

Two function-call forms are available inside SQL expressions for programmatic use, for example from a stored procedure body or a trigger (see CALL):

EVO_NOTIFY(channel, payload)
Enqueues a notification, equivalent to the NOTIFY statement. PG_NOTIFY(channel, payload) is accepted as a PostgreSQL-compatible alias for the same function. Like the statement form, the event is buffered and delivered at commit.
PG_LISTENING_CHANNELS()
Returns the channels the current session is listening on, read from the session's own listen mirror.
-- raise a notification from inside a procedure / trigger
SELECT EVO_NOTIFY('orders', 'inserted');

-- inspect this session's subscriptions
SELECT PG_LISTENING_CHANNELS();

17.8. Identifiers, limits, and error codes

Channel names are case-insensitive (compared with a folded hash and strcasecmp), so orders and ORDERS are the same channel. The relevant fixed limits are:

Channel name length
Maximum 63 characters (64-byte buffer including the terminator), mirroring PostgreSQL's NAMEDATALEN. Over-long or empty names are rejected.
Payload length
Maximum 8000 bytes. An over-long payload is rejected before delivery.
Per-session LISTEN cap
A session may listen on at most 32 channels concurrently. The thirty-third distinct LISTEN is refused.
Snapshot fan-out bound
A single delivery copies up to 1024 listeners per channel into its snapshot.

Errors are reported with these SQLSTATE codes:

SQLSTATECondition
42622Channel name too long (≥ 64 bytes) or invalid; or per-session LISTEN cap exceeded.
22023Payload string too long (over 8000 bytes).

17.9. Durable subscriptions

Plain LISTEN/NOTIFY is fire-and-forget: a notification is delivered only to connections that are subscribed at the moment of commit. A consumer that is offline when the event fires misses it entirely.

For an at-least-once, replayable queue, EvolutionDB layers a separate durable subscription feature on top of the same channels. At commit, after the in-memory fan-out, each notification is also appended to every persistent mailbox bound to that channel; a consumer that reconnects later can drain the backlog. This is described in Chapter on durable subscriptions and the CREATE SUBSCRIPTION command. The fast path is unaffected: when a channel has no durable subscribers, no extra work is done.

Notes and limitations

  • The listener registry is in-memory and per-process. It is not written to evosql.db, is not recovered after a restart, and is not visible to other server processes. Use durable subscriptions when an event must survive a disconnect or restart.
  • The "PID" reported to listeners is the sender's internal session id, not an OS process id.
  • Same-session notifications are filtered out by default; opt in per channel with LISTEN ... SELF (an EvolutionDB extension, not standard PostgreSQL syntax).
  • NOTIFY is commit-gated; a rolled-back transaction (or ROLLBACK TO SAVEPOINT covering the statement) delivers nothing. Identical notifications within one transaction are collapsed to a single delivery.
  • Channel names are case-insensitive; payloads are compared case-sensitively for de-duplication.
  • NOTIFY currently accepts only a channel name and an optional quoted string literal — arbitrary expression payloads are not yet supported by the statement form.
  • Disconnecting auto-cancels all of a session's listens and discards its uncommitted pending notifications.

17.10. End-to-end example

Connection A subscribes and waits; connection B publishes:

-- connection A
LISTEN orders;
-- A now blocks/polls for async messages on the channel

-- connection B
BEGIN;
INSERT INTO orders (id, status) VALUES (4711, 'shipped');
NOTIFY orders, '4711 shipped';   -- queued
COMMIT;                              -- delivered to A here

-- connection A receives (PG wire): channel "orders", payload "4711 shipped",
-- PID = B's session id. Then it can stop:
UNLISTEN orders;
