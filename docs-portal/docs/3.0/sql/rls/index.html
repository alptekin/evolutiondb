20. Row-Level Security

Row-Level Security (RLS) lets you attach per-table policies that restrict which rows a session user may read, insert, update, or delete — enforced inside the engine on every DML path, not bolted on by the application.

EvolutionDB implements RLS as a small, focused subset of the PostgreSQL model: a table carries an RLS-enabled flag, and any number of named policies attach to that table. A policy pairs an optional row-filter expression (USING) and an optional write-validation expression (WITH CHECK) with a command scope and an optional role list. When the flag is on, the engine rewrites reads to filter rows through the matching USING expressions and validates written rows against the matching WITH CHECK expressions.

The SQL surface and its persistence are handled in evolution/db/Policy.c; runtime enforcement helpers (the read overlay and the write check) live alongside it and are invoked from the SELECT, INSERT, UPDATE, and DELETE paths.

20.1. Enabling and Disabling RLS on a Table

Policies are defined independently, but they only take effect once Row-Level Security is enabled on the owning table. Toggling the flag does not create, drop, or alter any policy — it merely controls whether the engine consults them.

ALTER TABLE table_name ENABLE ROW LEVEL SECURITY;
ALTER TABLE table_name DISABLE ROW LEVEL SECURITY;

The grammar accepts only the exact phrase ROW LEVEL SECURITY; a misspelled middle word raises a syntax error ("expected LEVEL after ROW…"). The toggle sets a single per-table flag (rls_enabled) in the system catalog via cat_update_table_rls_flag().

-- turn enforcement on; existing policies (if any) now apply
ALTER TABLE accounts ENABLE ROW LEVEL SECURITY;

-- turn enforcement off; policies remain in the catalog, just dormant
ALTER TABLE accounts DISABLE ROW LEVEL SECURITY;

Default-deny. When a table has rls_enabled = 1 but no policy matches a non-superuser's command, access is denied: reads return zero rows and writes are rejected. Enabling RLS before defining any policy effectively locks out every non-superuser. The admin user is treated as a superuser and bypasses RLS entirely (see §20.6).

20.2. CREATE POLICY

Creates a named policy attached to a table. See also the command reference page at CREATE POLICY.

CREATE POLICY policy_name ON table_name
    [ AS { PERMISSIVE | RESTRICTIVE } ]
    [ FOR { ALL | SELECT | INSERT | UPDATE | DELETE } ]
    [ TO role [, ...] ]
    [ USING ( using_expression ) ]
    [ WITH CHECK ( check_expression ) ];

Every clause except the name and target table is optional. The clauses must appear in the order shown above (the grammar is positional: AS, then FOR, then TO, then USING, then WITH CHECK).

policy_name
The policy name. Unique per table: policies are keyed by table_id:policy_name in the catalog, so the same name may be reused on different tables. Creating a duplicate name on the same table fails with SQLSTATE 42710.
AS PERMISSIVE / AS RESTRICTIVE
Controls how the policy combines with others. PERMISSIVE policies are OR'd together; RESTRICTIVE policies are AND'd on top. The default is PERMISSIVE. See §20.5.
FOR
The command scope the policy applies to. The default is FOR ALL. Internally each scope maps to a one-character command code (§20.4).
TO role [, ...]
The roles the policy applies to. Up to 8 roles may be listed (CAT_MAX_POLICY_ROLES); excess entries are ignored. When the list is omitted the policy applies to every role. A role named public in the list also matches every role. Role matching is case-insensitive and is compared against the session user name.
USING ( using_expression )
A boolean expression evaluated per existing row. Rows for which it is true are visible to SELECT and eligible for UPDATE/DELETE. The expression AST is serialized into the catalog (hex-encoded, up to 2048 bytes — CAT_MAX_POLICY_EXPR).
WITH CHECK ( check_expression )
A boolean expression that new or updated rows must satisfy on INSERT and UPDATE. If omitted, the USING expression is reused for the write check (matching PostgreSQL).
-- each user only sees and writes their own rows
CREATE POLICY owner_rows ON documents
    FOR ALL
    USING (owner = 'alice')
    WITH CHECK (owner = 'alice');

-- read-only visibility policy scoped to one role
CREATE POLICY public_published ON documents
    AS PERMISSIVE
    FOR SELECT
    TO reader
    USING (status = 'published');

-- a restrictive guard that ANDs on top of any permissive policy
CREATE POLICY not_archived ON documents
    AS RESTRICTIVE
    FOR SELECT
    USING (archived = false);

Memory-store sugar. EvolutionDB also accepts CREATE POLICY name ON MEMORY STORE store (and the matching ALTER MEMORY STORE … ENABLE/DISABLE ROW LEVEL SECURITY). These resolve to the store's backing table (named mem_<store>) and otherwise behave identically. If the store does not exist, the literal name is used and the usual "relation does not exist" error is raised.

20.3. DROP POLICY

Removes a policy from a table. See also DROP POLICY.

DROP POLICY policy_name ON table_name;
DROP POLICY public_published ON documents;

The target table is resolved first; if it does not exist the error is SQLSTATE 42P01 (undefined table). If the table exists but carries no such policy, the error is SQLSTATE 42704 (undefined object). Dropping a policy does not change the table's rls_enabled flag — if you drop the last policy while RLS is still enabled, the table falls back to default-deny for non-superusers.

20.4. Command Scopes

The FOR clause stores a single command-code byte on the policy. A policy participates in enforcement only when its scope matches the operation being run (or when the scope is ALL).

FOR clauseCodeApplies to
FOR ALL (default)'*'SELECT, INSERT, UPDATE, DELETE
FOR SELECT'S'Read path (USING overlay)
FOR INSERT'I'Write check (WITH CHECK)
FOR UPDATE'U'Read overlay for candidates + write check
FOR DELETE'D'Write check on deletion candidates

20.5. How Policies Are Combined and Enforced

Enforcement splits into two distinct mechanisms depending on whether the operation reads or writes rows.

20.5.1. Read overlay (SELECT, and UPDATE/DELETE row selection)

For reads, the engine builds an overlay expression from every policy that matches the command and the session user, and AND's it into the statement's existing WHERE clause. The combination rule is:

  • All matching PERMISSIVE USING expressions are OR'd together.
  • All matching RESTRICTIVE USING expressions are AND'd on top of that OR group.
  • If no permissive policy matches, the overlay is the constant false — i.e. zero rows. (A restrictive policy alone never grants visibility; there must be a permissive grant for any row to pass.)

The result is folded into the same WHERE expression every scan path evaluates, so plain table scans, fast paths, and expression indexes all see the same filtered view. The original WHERE is restored after the statement. The same overlay is applied per joined table in multi-table queries.

-- with the three documents policies above, a non-superuser SELECT
-- effectively runs as:
--   WHERE (<your WHERE>)
--     AND ( owner = 'alice' OR status = 'published' )  -- PERMISSIVE OR-group
--     AND ( archived = false )                          -- RESTRICTIVE AND
SELECT * FROM documents WHERE created > '2026-01-01';

20.5.2. Write check (INSERT, UPDATE, DELETE)

For writes, the engine evaluates the matching policies' WITH CHECK expression (falling back to USING when no WITH CHECK is present) against the candidate row's column values. The acceptance rule is:

  • If any matching RESTRICTIVE check evaluates false, the write is rejected immediately.
  • At least one matching PERMISSIVE check must evaluate true.
  • If no permissive policy matches at all, the write is rejected (default-deny).

A failed check raises SQLSTATE 42501 with the message new row for relation "…" violates row-level security policy. An expression result is treated as truthy when it equals t, true, or 1, or is a non-zero number.

-- rejected: WITH CHECK (owner = 'alice') is false for this row
INSERT INTO documents (id, owner, status) VALUES (1, 'bob', 'draft');
-- ERROR: new row for relation "documents" violates row-level security policy

20.6. Superuser Bypass

The admin user is recognized as a superuser and bypasses both the read overlay and the write check entirely — RLS is never applied to its statements, regardless of the table's flag or any policy. Recognition is by case-insensitive comparison of the session user name against admin. EvolutionDB does not currently expose a per-table "force RLS" mode, so there is no way to subject the superuser to its own policies.

20.7. Error Codes

The RLS surface uses the following SQLSTATE values:

SQLSTATECondition
42601Syntax error — missing policy name or target table, or malformed ENABLE/DISABLE ROW LEVEL SECURITY.
42P01 / 42E01Undefined table — the target relation does not exist.
42704Undefined object — DROP POLICY on a policy (or memory store) that does not exist.
42710Duplicate object — a policy of the same name already exists on the table.
42501Insufficient privilege — a written row violated a WITH CHECK policy.
EX000Internal error — the RLS flag could not be updated.

20.8. Notes and Limitations

EvolutionDB's RLS is a deliberate subset of the SQL/PostgreSQL model. Where it differs from PostgreSQL:

  • No ALTER POLICY. To change a policy, DROP it and CREATE it again.
  • No FORCE ROW LEVEL SECURITY. The owner/superuser (admin) always bypasses policies; there is no opt-in to enforce them on the superuser.
  • Superuser detection is name-based. Only the literal user admin is treated as a superuser for RLS purposes; this is not derived from a role privilege system.
  • Role matching is by session user name. The TO list is compared, case-insensitively, against the current user; public is a special value meaning "all roles." There is no role-membership/inheritance resolution.
  • Fixed limits. At most 8 roles per policy (CAT_MAX_POLICY_ROLES); each serialized USING/WITH CHECK expression is capped at 2048 bytes (CAT_MAX_POLICY_EXPR).
  • Clause order is fixed. CREATE POLICY clauses must appear as AS → FOR → TO → USING → WITH CHECK.
  • Disabling RLS does not drop policies. Policies persist in the catalog and re-activate when RLS is re-enabled.
