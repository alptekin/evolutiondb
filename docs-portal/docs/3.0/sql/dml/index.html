6. Data Manipulation

This chapter shows how rows go into a table, change, and leave it in EvolutionDB. It covers the INSERT, REPLACE, UPDATE, DELETE, COPY, and TRUNCATE commands as the engine's data-manipulation modules really run them.

The earlier chapters showed how to create tables and which types a column can hold. A new table is empty. This chapter covers the commands that put data into a table and change it later. Every command here uses the same engine path. The Bison parser emits an RPN stream. The matching operation module then checks and runs the change. Those modules are Insert.c, Update.c, Delete.c, Copy.c, and Drop.c. The change is written without overwriting old data under MVCC. This way, readers running at the same time still see the version their snapshot allows.

Scope of this chapter. This page only documents behavior that EvolutionDB 3.0 really has. Sometimes EvolutionDB uses MySQL or PostgreSQL syntax but acts in a different way. One example is that ON CONFLICT targets only one column. Another is that REPLACE keys on the primary key alone. Each such difference is pointed out here. Full syntax detail and option lists are on the per-command reference pages linked below.

6.1. Inserting Data

You add a new row with INSERT. You can list one value for every column in order. You can also name some of the columns and give values only for those.

CREATE TABLE products (
    id      INT PRIMARY KEY,
    name    VARCHAR(50) NOT NULL,
    price   DECIMAL(10,2) DEFAULT 0.00
);

-- positional: one value per column, in table order
INSERT INTO products VALUES (1, 'Widget', 9.99);

-- named columns: omitted columns take their DEFAULT (or NULL)
INSERT INTO products (id, name) VALUES (2, 'Gadget');

-- multiple rows in one statement
INSERT INTO products (id, name, price) VALUES
    (3, 'Bolt',  0.10),
    (4, 'Nut',   0.08),
    (5, 'Washer', 0.05);

When you give a column list, the engine puts each row's values back into the real table column order. It fills any missing column from its declared DEFAULT. If a column has neither a value nor a default, the engine uses NULL. You can also write the DEFAULT keyword directly in the VALUES list.

EvolutionDB also accepts the MySQL SET form of INSERT and the INSERT ... SELECT form.

INSERT INTO products SET id = 6, name = 'Spring', price = 0.20;

INSERT INTO products (id, name, price)
    SELECT id + 100, name, price FROM products WHERE price < 0.10;

6.1.1. Default and Generated Columns

Defaults are worked out as the row is built. There are three forms.

Literal defaults
A constant declared with DEFAULT is copied as is.
Function defaults
A small set of generator functions runs once per row. These are gen_random_uuid(), gen_random_uuid_v7(), snowflake_id(), and CURRENT_TIMESTAMP. The first value from a UUID or Snowflake default is also saved for LAST_INSERT_ID().
Expression defaults
An expression default is worked out in a second pass. This pass runs after the literal and function defaults are in place. So an expression default can read other columns of the same row.

Columns declared GENERATED are computed by the engine. You do not supply them. A STORED generated column is worked out and written to disk. A VIRTUAL generated column is stored as a NULL placeholder and worked out again on read. If you try to insert a non-DEFAULT value into a generated column, you get an error.

cannot insert a non-DEFAULT value into column "…" because it is a generated column

6.1.2. AUTO_INCREMENT Columns

Suppose a table has an AUTO_INCREMENT column. A row that gives NULL, 0, or an empty string for that column gets the next counter value. The counter goes up by the table's step. If you give a value larger than the current counter, the counter moves forward. This keeps future generated values unique. The generated value is saved for LAST_INSERT_ID(). The table's stored counter is updated at the end of the statement.

6.1.3. Validation Order

Each row is checked before it is stored. Once per statement, the engine looks at which kinds of constraints the table has. It skips the checks that do not apply. The checks that do apply run in this order.

CheckRaised on failureSQLSTATE
Value count vs. column count, type validationINSERT has N value(s) but table has M column(s) / type error22000
NOT NULLnull value in column "…" violates not-null constraint23502
UNIQUE (column-level and composite)duplicate key value violates unique constraint …23505
CHECKnew row violates check constraint "…"23514
FOREIGN KEYinsert or update on table violates foreign key constraint "…"23503

A primary-key clash is found later, when the row is stored. A duplicate primary key also raises 23505. FOREIGN KEY validation follows the constraint's match type. MATCH SIMPLE is the default. Under it, any NULL in the referencing columns skips the check. Under MATCH FULL, a mix of NULL and non-NULL referencing values is an error by itself. When the referenced columns are the parent's primary key, the engine checks for the row with a direct B+ tree lookup. Otherwise it does a table scan.

Note. A multi-row INSERT checks every row first and stores them after. But the statement is not a transaction on its own. Suppose a later row fails its check after earlier rows were stored. The stored rows stay, unless the statement runs inside an explicit transaction that you roll back. Begin a transaction when you need all-or-nothing behavior across a batch. See Chapter 7 for that. Each row's data also has a limit set by an 8 KB record buffer named RECORD_BUF_SIZE. Tuples above a 3800-byte threshold are spilled to an overflow chain called TOAST. A row that cannot fit on a single 4 KB page at all is rejected.

6.1.4. Upserts: ON CONFLICT and ON DUPLICATE KEY

EvolutionDB supports two ways to write an upsert. Both use the same engine code. The PostgreSQL form is ON CONFLICT. The MySQL form is ON DUPLICATE KEY UPDATE.

-- PostgreSQL form: do nothing on a conflicting key
INSERT INTO products (id, name) VALUES (1, 'Widget v2')
    ON CONFLICT (id) DO NOTHING;

-- PostgreSQL form: update the existing row; EXCLUDED is the would-be row
INSERT INTO products (id, name, price) VALUES (1, 'Widget', 12.50)
    ON CONFLICT (id) DO UPDATE SET price = EXCLUDED.price;

-- MySQL form: equivalent DO UPDATE; column refs read existing row values
INSERT INTO products (id, name) VALUES (1, 'Widget')
    ON DUPLICATE KEY UPDATE price = price + 1;

When a conflicting key is found, DO NOTHING skips the row quietly and does not count it. DO UPDATE and the MySQL UPDATE clause work out the assignment expressions. They then apply them to the existing row through the normal update path. In a DO UPDATE assignment, a plain column name means the existing row's value. EXCLUDED.col means the value the failed INSERT would have written. Conflicts are found against the primary key and against UNIQUE indexes.

Limitation. The ON CONFLICT target takes one column name in parentheses. A multi-column target like ON CONFLICT (a, b) is rejected at parse time with composite ON CONFLICT target (a, b, ...) is not yet supported. You can leave out the target. Then it matches on any unique conflict.

6.1.5. The RETURNING Clause

INSERT, UPDATE, and DELETE all accept a trailing RETURNING clause. It streams back the affected rows. INSERT returns the inserted values. UPDATE returns the new values after SET. DELETE returns the deleted values.

INSERT INTO products (id, name) VALUES (7, 'Clip') RETURNING id;
UPDATE products SET price = price * 1.1 WHERE id = 3 RETURNING id, price;
DELETE FROM products WHERE price = 0 RETURNING *;

Note. The RETURNING result buffer holds at most 256 rows per statement. Rows past that limit are still run, but not returned.

6.2. Replacing Rows with REPLACE

The MySQL-style REPLACE statement is just INSERT with the keyword swapped. It uses the whole insert path. The difference is what happens on a primary-key clash. REPLACE does not fail. Instead it deletes the existing row with the matching primary key and inserts the new row in its place.

-- if id 1 exists, the old row is deleted and this row takes its place
REPLACE INTO products VALUES (1, 'Widget v3', 14.00);

REPLACE INTO products SET id = 1, name = 'Widget v4';

Behavior difference from MySQL. EvolutionDB's REPLACE resolves conflicts on the primary key. The delete-then-insert is done as an MVCC delete of the old version, then a fresh insert. The new row gets a new heap location and a fresh transaction stamp. It does not keep the old row's storage. Like INSERT, REPLACE also accepts the VALUES, SET, and SELECT source forms.

6.3. Updating Data

You change existing rows with UPDATE. The SET clause lists the columns to change. WHERE picks which rows. An optional LIMIT caps how many.

UPDATE products SET price = 19.99 WHERE id = 1;

-- expressions may reference the row's own current values
UPDATE products SET price = price * 0.9 WHERE price > 10;

-- with no WHERE, every row is updated
UPDATE products SET price = 0;

Only the columns named in SET change. Every other column stays the same. After the assignments are applied, the engine works out any STORED generated columns again. Then it checks the new row against CHECK, FOREIGN KEY, and UNIQUE constraints. It only re-checks a foreign-key or unique constraint when a column it covers was in the SET list. As with INSERT, you may not set a generated column to a value other than DEFAULT.

column "…" can only be updated to DEFAULT because it is a generated column

6.3.1. How an Update Is Stored

EvolutionDB does not overwrite rows in place. An UPDATE does not destroy the old data. It works in these steps.

  1. The old tuple is soft-deleted. The engine stamps its MVCC xmax with the current transaction id.
  2. The new version is written to a heap page at a new location.
  3. The primary-key B+ tree entry is pointed at the new row id.

This keeps the old version on its page. So readers running at the same time with an older snapshot still see it. It also means the change can be rolled back. The space used by old versions is reclaimed later. See RECLAIM for that. An exclusive row lock is taken on each affected row. If two transactions fight for the same row, the engine may report a deadlock with 40P01. It may also report that it could not get the lock in the 55P03 family.

6.3.2. Updating a Primary Key, Referential Actions

When an UPDATE changes a primary-key column, the engine runs the ON UPDATE referential action of every foreign key that references the table. It does this for each child row that pointed at the old key value.

RESTRICT / NO ACTION
The update is rejected if any child row references the old key.
CASCADE
The child rows' foreign-key columns are rewritten to the new key value.
SET NULL
The child rows' foreign-key columns are set to NULL.
SET DEFAULT
The child rows' foreign-key columns are set to their declared defaults. If a column has no default, it falls back to NULL.

If the SET list does not really change the primary-key value, no referential action runs.

6.4. Deleting Data

You remove rows with DELETE. A WHERE clause picks the rows to delete. Without one, every row in the table is removed.

DELETE FROM products WHERE price = 0;

-- delete every row (the table itself remains)
DELETE FROM products;

DELETE runs in two phases. First it collects the matching rows. A single-column primary-key test of the form pk = v, pk < v, pk <= v, pk > v, or pk >= v uses a B+ tree cursor walk. That walk visits only the matching keys. Any other test falls back to a full table scan with a per-row WHERE check. An optional LIMIT or OFFSET is then applied to the collected set. Second, it deletes each collected row.

For each row deleted, the engine does several things. First it enforces foreign-key referential integrity before removing the row. Then it runs BEFORE and AFTER DELETE triggers if any exist. It removes the row's entries from secondary indexes. Then it MVCC-soft-deletes the tuple by setting its xmax. The primary-key entry is kept. So readers running at the same time can still find the row and check whether it is visible. The space is reclaimed later.

6.4.1. Foreign-Key Actions on Delete

Deleting a parent row runs the ON DELETE action of every foreign key that references it.

RESTRICT / NO ACTION
The delete is rejected if any child row references the parent. The error is update or delete on table "…" violates foreign key constraint "…" on table "…" with code 23503.
CASCADE
Matching child rows are deleted. Cascades go to grandchildren first. A cascade chain deeper than 32 levels is rejected as a possible circular reference.
SET NULL
The child rows' foreign-key columns are set to NULL.
SET DEFAULT
The child rows' foreign-key columns are set to their declared defaults. If a column has no default, it falls back to NULL.

Note. A DELETE with no WHERE still removes rows one at a time through the full delete path. So foreign-key checks, triggers, and index work all run for every row. To empty a large table fast and reset its identity counter, use TRUNCATE instead. See Section 6.6.

6.5. Bulk Loading and Unloading with COPY

The COPY command moves data in bulk between a table and a file or stream. It uses CSV or TEXT format. COPY ... FROM imports. COPY ... TO exports.

-- import: each CSV line becomes one inserted row
COPY products FROM '/data/products.csv' WITH (FORMAT CSV, HEADER true);

-- export: every row of the table is written out
COPY products TO '/data/dump.csv' WITH (FORMAT CSV, HEADER true);

-- export the result of a query
COPY (SELECT id, name FROM products WHERE price > 0) TO '/data/priced.csv' WITH (FORMAT CSV);

6.5.1. COPY Options and Defaults

The format picks a set of default settings. The individual options then override them.

FORMAT TEXT
This is the default. The delimiter is a tab. The null marker is \N.
FORMAT CSV
The delimiter is a comma. The null marker is the empty string. The quote character is ". The plain keyword CSV also works as a short way to write FORMAT CSV.
DELIMITER 'c'
One field-separator character.
QUOTE 'c'
One quoting character for CSV.
NULL 's'
The string that stands for a SQL NULL.
HEADER [true]
On import, skip the first line. On export, write a header row of column names first.

6.5.2. Import Semantics

COPY ... FROM is not a separate write path. Each parsed line is turned into a single-row INSERT and run through the normal insert code. So every imported row gets the same checks as a normal INSERT. This means the same type validation and the NOT NULL, UNIQUE, CHECK, and FOREIGN KEY checks. It also means the same index and trigger work. File-mode COPY FROM runs in auto-commit mode. On the first row that fails to parse or insert, the import stops and the command reports an error. In file mode, the work done up to the failing row is committed. The command returns the number of rows it inserted.

6.5.3. File Mode and Stream Mode

EvolutionDB supports two ways to move the data.

File mode
This is FROM '/abs/path' and TO '/abs/path'. The path must be absolute. It may not contain a .. part. Relative or traversal paths are rejected.
Stream mode
This is FROM STDIN and TO STDOUT. It is driven by the PostgreSQL wire COPY subprotocol in the dual-protocol server. STDIN means an import. STDOUT means an export.

Security note. Server-side file COPY does not yet require a superuser check. The text .. guard also does not stop symbolic links. In the current release, any user who can read or write a table can use file-mode COPY. Through it they can read or write any file the server process can reach. Until a superuser gate ships, treat file-mode COPY as a privileged operation in practice. Stream mode with STDIN and STDOUT does not touch the server filesystem. It is not affected.

6.6. Emptying a Table with TRUNCATE

TRUNCATE TABLE removes all rows from a table fast. It throws away and rebuilds the table's storage. It does not delete rows one by one. The table definition, its columns, and its constraints stay.

TRUNCATE TABLE products;

-- empty several tables in one statement
TRUNCATE TABLE products, orders, line_items;

-- also empty tables that reference these by foreign key
TRUNCATE TABLE products CASCADE;

-- keep the AUTO_INCREMENT counter instead of resetting it
TRUNCATE TABLE products CONTINUE IDENTITY;

For each truncated table, the engine does several things. It destroys and recreates the primary-key B+ tree and every secondary index as empty trees. It frees all of the table's heap pages. It resets the row-count statistics. Unless you give CONTINUE IDENTITY, it resets the AUTO_INCREMENT counter to zero. The supported options are below.

OptionEffect
CASCADETruncate any table that references the target by a foreign key first. Then truncate the target.
RESTRICTAccepted. This is the default behavior. It refuses to truncate a table that is referenced by foreign keys from non-empty tables.
RESTART IDENTITYAccepted. Identity and auto-increment counters are reset. This is also the default when you leave the option out.
CONTINUE IDENTITYDo not reset the AUTO_INCREMENT counter.

Without CASCADE, you cannot truncate a table that is referenced by a foreign key from a non-empty table. It is rejected.

cannot truncate table "…": referenced by foreign key from "…" (SQLSTATE 23503)

TRUNCATE vs. DELETE. DELETE FROM t removes rows one by one through the MVCC delete path. It fires triggers. It runs ON DELETE referential actions for each row. It supports WHERE and RETURNING. It leaves dead tuples to be reclaimed later. TRUNCATE takes no WHERE. It fires no row triggers. It throws away the storage all at once. It resets identity counters. Use DELETE when you need row-level control or only a subset. Use TRUNCATE to empty a whole table cheaply.
