6. Data Manipulation

How rows enter, change, and leave a table in EvolutionDB, the INSERT, REPLACE, UPDATE, DELETE, COPY, and TRUNCATE commands, as actually implemented by the engine's data-manipulation modules.

The previous chapters described how to create tables and what types a column may hold. A freshly created table is empty. This chapter covers the commands that put data into a table and change it afterward. Every command here goes through the same engine path: the Bison parser emits an RPN stream, the matching operation module (Insert.c, Update.c, Delete.c, Copy.c, Drop.c) validates and executes the change, and the change is written non-destructively under MVCC so concurrent readers keep seeing the version their snapshot entitles them to.

Scope of this chapter. Only behavior that EvolutionDB 3.0 actually implements is documented. Where EvolutionDB borrows MySQL or PostgreSQL surface syntax but diverges in behavior, for example, single-column ON CONFLICT targets only, or REPLACE keyed on the primary key alone, the difference is called out explicitly. Statement-level syntax detail and option lists live on the per-command reference pages linked throughout.

6.1. Inserting Data

A new row is added with INSERT. You may list values positionally for every column, or name a subset of columns and supply values only for those.

CREATE TABLE products (
    id      INT PRIMARY KEY,
    name    VARCHAR(50) NOT NULL,
    price   DECIMAL(10,2) DEFAULT 0.00
);

-- positional: one value per column, in table order
INSERT INTO products VALUES (1, 'Widget', 9.99);

-- named columns: omitted columns take their DEFAULT (or NULL)
INSERT INTO products (id, name) VALUES (2, 'Gadget');

-- multiple rows in one statement
INSERT INTO products (id, name, price) VALUES
    (3, 'Bolt',  0.10),
    (4, 'Nut',   0.08),
    (5, 'Washer', 0.05);

When a column list is given, the engine reorders each row's values into the physical table column order, fills omitted columns from their declared DEFAULT, and substitutes NULL for columns that have neither a value nor a default. A DEFAULT keyword may also be written explicitly in the VALUES list.

EvolutionDB also accepts the MySQL SET form of INSERT and INSERT ... SELECT:

INSERT INTO products SET id = 6, name = 'Spring', price = 0.20;

INSERT INTO products (id, name, price)
    SELECT id + 100, name, price FROM products WHERE price < 0.10;

6.1.1. Default and Generated Columns

Defaults are evaluated as the row is built, in three forms:

Literal defaults
A constant declared with DEFAULT is copied verbatim.
Function defaults
A small set of generator functions is evaluated per row: gen_random_uuid(), gen_random_uuid_v7(), snowflake_id(), and CURRENT_TIMESTAMP. The first value produced by a UUID or Snowflake default is also captured for LAST_INSERT_ID().
Expression defaults
An arbitrary expression default is resolved in a second pass, after the literal and function defaults are in place, so it can reference other columns of the same row.

Columns declared GENERATED are computed by the engine, not supplied by you. A STORED generated column is evaluated and written to disk; a VIRTUAL generated column is stored as a NULL placeholder and recomputed on read. Attempting to insert a non-DEFAULT value into a generated column raises an error:

cannot insert a non-DEFAULT value into column "…" because it is a generated column

6.1.2. AUTO_INCREMENT Columns

If a table has an AUTO_INCREMENT column, a row that supplies NULL, 0, or an empty string for that column receives the next counter value (incremented by the table's step). Supplying an explicit value larger than the current counter advances the counter so future generated values stay unique. The generated value is recorded for LAST_INSERT_ID(), and the table's persisted counter is updated at the end of the statement.

6.1.3. Validation Order

Each row is validated before it is stored. The engine probes which constraint classes the table actually has once per statement and skips the checks that do not apply, but when present they run in this order:

CheckRaised on failureSQLSTATE
Value count vs. column count, type validationINSERT has N value(s) but table has M column(s) / type error22000
NOT NULLnull value in column "…" violates not-null constraint23502
UNIQUE (column-level and composite)duplicate key value violates unique constraint …23505
CHECKnew row violates check constraint "…"23514
FOREIGN KEYinsert or update on table violates foreign key constraint "…"23503

A primary-key collision is detected separately when the row is stored, and a duplicate primary key also raises 23505. FOREIGN KEY validation honors the constraint's match type: under MATCH SIMPLE (the default) any NULL in the referencing columns skips the check; under MATCH FULL a mix of NULL and non-NULL referencing values is itself an error. When the referenced columns are the parent's primary key, the existence check is a direct B+ tree lookup; otherwise it is a table scan.

Note. A multi-row INSERT validates every row first and stores them afterward, but the statement is not transactional on its own: if a later row fails validation after earlier rows were stored, the stored rows remain unless the statement runs inside an explicit transaction that you roll back. Begin a transaction (see Chapter 7) when you need all-or-nothing semantics across a batch. Each row's data is also bounded by an 8 KB record buffer (RECORD_BUF_SIZE); oversized tuples are spilled to an overflow chain (TOAST) above a 3800-byte threshold, and a row that cannot fit on a single 4 KB page at all is rejected.

6.1.4. Upserts: ON CONFLICT and ON DUPLICATE KEY

EvolutionDB supports two upsert spellings against the same underlying machinery. The PostgreSQL form is ON CONFLICT; the MySQL form is ON DUPLICATE KEY UPDATE.

-- PostgreSQL form: do nothing on a conflicting key
INSERT INTO products (id, name) VALUES (1, 'Widget v2')
    ON CONFLICT (id) DO NOTHING;

-- PostgreSQL form: update the existing row; EXCLUDED is the would-be row
INSERT INTO products (id, name, price) VALUES (1, 'Widget', 12.50)
    ON CONFLICT (id) DO UPDATE SET price = EXCLUDED.price;

-- MySQL form: equivalent DO UPDATE; column refs read existing row values
INSERT INTO products (id, name) VALUES (1, 'Widget')
    ON DUPLICATE KEY UPDATE price = price + 1;

When a conflicting key is found, DO NOTHING silently skips the row and does not count it; DO UPDATE (and the MySQL UPDATE clause) evaluates the assignment expressions and applies them to the existing row via the normal update path. In a DO UPDATE assignment, a bare column name refers to the existing row's value, while EXCLUDED.col refers to the value the failed INSERT would have written. Conflicts are detected against the primary key and against UNIQUE indexes.

Limitation. The ON CONFLICT target accepts a single column name in parentheses. A composite target such as ON CONFLICT (a, b) is rejected at parse time with composite ON CONFLICT target (a, b, ...) is not yet supported. Omitting the target entirely is allowed and matches on any unique conflict.

6.1.5. The RETURNING Clause

All three of INSERT, UPDATE, and DELETE accept a trailing RETURNING clause that streams back the affected rows. INSERT returns the inserted values, UPDATE returns the new (post-SET) values, and DELETE returns the deleted values.

INSERT INTO products (id, name) VALUES (7, 'Clip') RETURNING id;
UPDATE products SET price = price * 1.1 WHERE id = 3 RETURNING id, price;
DELETE FROM products WHERE price = 0 RETURNING *;

Note. The RETURNING result buffer captures at most 256 rows per statement; rows beyond that limit are executed but not returned.

6.2. Replacing Rows with REPLACE

The MySQL-style REPLACE statement is syntactically INSERT with the keyword swapped, and it shares the entire insert path. The difference is what happens on a primary-key collision: instead of failing, REPLACE deletes the existing row with the matching primary key and inserts the new row in its place.

-- if id 1 exists, the old row is deleted and this row takes its place
REPLACE INTO products VALUES (1, 'Widget v3', 14.00);

REPLACE INTO products SET id = 1, name = 'Widget v4';

Behavior difference from MySQL. EvolutionDB's REPLACE resolves conflicts on the primary key. The delete-then-insert is performed as an MVCC delete of the old version followed by a fresh insert, the new row gets a new heap location and a fresh transaction stamp; it does not preserve the old row's storage. Like INSERT, REPLACE also accepts the VALUES, SET, and SELECT source forms.

6.3. Updating Data

Existing rows are modified with UPDATE. The SET clause lists the columns to change; WHERE selects which rows, and an optional LIMIT caps how many.

UPDATE products SET price = 19.99 WHERE id = 1;

-- expressions may reference the row's own current values
UPDATE products SET price = price * 0.9 WHERE price > 10;

-- with no WHERE, every row is updated
UPDATE products SET price = 0;

Only the columns named in SET are changed; every other column is preserved. After the assignments are applied, the engine re-evaluates any STORED generated columns, then validates the resulting row against CHECK, FOREIGN KEY, and UNIQUE constraints, but it only re-checks a foreign-key or unique constraint if a column it covers was actually in the SET list. As with INSERT, generated columns may not be assigned a value other than DEFAULT:

column "…" can only be updated to DEFAULT because it is a generated column

6.3.1. How an Update Is Stored

EvolutionDB does not overwrite rows in place. An UPDATE is performed non-destructively:

  1. The old tuple is soft-deleted by stamping its MVCC xmax with the current transaction id.
  2. The new version is written to a heap page (a new location).
  3. The primary-key B+ tree entry is repointed to the new row id.

This preserves the old version on its page so that concurrent readers holding an older snapshot still see it, and so the change can be rolled back. The physical space occupied by superseded versions is reclaimed later (see RECLAIM). An exclusive row lock is taken on each affected row; if two transactions contend, the engine may report a deadlock (40P01) or an inability to obtain the lock (55P03 family).

6.3.2. Updating a Primary Key, Referential Actions

When an UPDATE changes a primary-key column, the engine enforces the ON UPDATE referential action of every foreign key that references the table, for each child row that pointed at the old key value:

RESTRICT / NO ACTION
The update is rejected if any child row references the old key.
CASCADE
The child rows' foreign-key columns are rewritten to the new key value.
SET NULL
The child rows' foreign-key columns are set to NULL.
SET DEFAULT
The child rows' foreign-key columns are set to their declared defaults (falling back to NULL when no default exists).

If the SET list does not actually change the primary-key value, no referential action runs.

6.4. Deleting Data

Rows are removed with DELETE. A WHERE clause selects the rows to delete; without one, every row in the table is removed.

DELETE FROM products WHERE price = 0;

-- delete every row (the table itself remains)
DELETE FROM products;

DELETE runs in two phases. First it collects the matching rows: a single-column primary-key predicate of the form pk = v, pk < v, pk <= v, pk > v, or pk >= v is served by a B+ tree cursor walk that visits only the matching keys; any other predicate falls back to a full table scan with per-row WHERE evaluation. An optional LIMIT/OFFSET is then applied to the collected set. Second, it deletes each collected row.

For each row deleted, the engine enforces foreign-key referential integrity before removing it, runs BEFORE/AFTER DELETE triggers if any exist, removes the row's entries from secondary indexes, and then MVCC-soft-deletes the tuple by setting its xmax, the primary-key entry is kept so concurrent readers can still locate the row and check visibility. The space is reclaimed later.

6.4.1. Foreign-Key Actions on Delete

Deleting a parent row triggers the ON DELETE action of every foreign key that references it:

RESTRICT / NO ACTION
The delete is rejected if any child row references the parent. The error is update or delete on table "…" violates foreign key constraint "…" on table "…" (23503).
CASCADE
Matching child rows are deleted. Cascades recurse to grandchildren first; a cascade chain deeper than 32 levels is rejected as a possible circular reference.
SET NULL
The child rows' foreign-key columns are set to NULL.
SET DEFAULT
The child rows' foreign-key columns are set to their declared defaults (falling back to NULL).

Note. A DELETE with no WHERE still removes rows one at a time through the full delete path, so foreign-key checks, triggers, and index maintenance all run for every row. To empty a large table quickly and reset its identity counter, prefer TRUNCATE (Section 6.6).

6.5. Bulk Loading and Unloading with COPY

The COPY command moves data in bulk between a table and a file or stream, in CSV or TEXT format. COPY ... FROM imports; COPY ... TO exports.

-- import: each CSV line becomes one inserted row
COPY products FROM '/data/products.csv' WITH (FORMAT CSV, HEADER true);

-- export: every row of the table is written out
COPY products TO '/data/dump.csv' WITH (FORMAT CSV, HEADER true);

-- export the result of a query
COPY (SELECT id, name FROM products WHERE price > 0) TO '/data/priced.csv' WITH (FORMAT CSV);

6.5.1. COPY Options and Defaults

The format selects a set of dialect defaults, which the individual options then override:

FORMAT TEXT
Default. Delimiter is a tab, the null marker is \N.
FORMAT CSV
Delimiter is a comma, the null marker is the empty string, quote character is ". The bare keyword CSV is also accepted as a synonym for FORMAT CSV.
DELIMITER 'c'
Single field-separator character.
QUOTE 'c'
Single quoting character (CSV).
NULL 's'
String that represents a SQL NULL.
HEADER [true]
On import, skip the first line; on export, write a header row of column names first.

6.5.2. Import Semantics

COPY ... FROM is not a separate write path, each parsed line is turned into a single-row INSERT and run through the normal insert machinery. This means every imported row is subject to the same type validation, NOT NULL, UNIQUE, CHECK, and FOREIGN KEY checks, and the same index and trigger maintenance, as an ordinary INSERT. File-mode COPY FROM runs auto-commit: on the first row that fails to parse or insert, the import stops and the command reports an error; in file mode the partial work is committed up to the failing row. The command returns the number of rows successfully inserted.

6.5.3. File Mode and Stream Mode

EvolutionDB supports two transports:

File mode
FROM '/abs/path' / TO '/abs/path'. The path must be absolute and may not contain a .. component; relative or traversal paths are rejected.
Stream mode
FROM STDIN / TO STDOUT, driven by the PostgreSQL wire COPY subprotocol in the dual-protocol server. STDIN implies an import and STDOUT an export.

Security note. Server-side file COPY is not yet gated behind a superuser check, and the textual .. guard does not defeat symbolic links. In the current release any user who can read or write a table can, through file-mode COPY, read or write any file the server process can reach. Treat file-mode COPY as a privileged operation operationally until a superuser gate ships. Stream mode (STDIN/STDOUT) does not touch the server filesystem and is not affected.

6.6. Emptying a Table with TRUNCATE

TRUNCATE TABLE removes all rows from a table quickly by discarding and rebuilding the table's storage rather than deleting rows one by one. The table definition, its columns, and its constraints remain.

TRUNCATE TABLE products;

-- empty several tables in one statement
TRUNCATE TABLE products, orders, line_items;

-- also empty tables that reference these by foreign key
TRUNCATE TABLE products CASCADE;

-- keep the AUTO_INCREMENT counter instead of resetting it
TRUNCATE TABLE products CONTINUE IDENTITY;

For each truncated table the engine destroys and recreates the primary-key B+ tree and every secondary index as empty trees, frees all of the table's heap pages, resets the row-count statistics, and, unless CONTINUE IDENTITY is given, resets the AUTO_INCREMENT counter to zero. The supported options are:

OptionEffect
CASCADERecursively truncate any table that references the target by a foreign key, before truncating the target.
RESTRICTAccepted; the default behavior, refuse to truncate a table that is referenced by foreign keys from non-empty tables.
RESTART IDENTITYAccepted; identity/auto-increment counters are reset (this is also the default when the option is omitted).
CONTINUE IDENTITYDo not reset the AUTO_INCREMENT counter.

Without CASCADE, truncating a table that is referenced by a foreign key from a non-empty table is rejected:

cannot truncate table "…": referenced by foreign key from "…" (SQLSTATE 23503)

TRUNCATE vs. DELETE. DELETE FROM t removes rows individually through the MVCC delete path: it fires triggers, enforces ON DELETE referential actions per row, supports WHERE and RETURNING, and leaves dead tuples for later reclamation. TRUNCATE takes no WHERE, fires no row triggers, throws away the storage wholesale, and resets identity counters. Use DELETE when you need row-level control or a subset; use TRUNCATE to empty a whole table cheaply.
