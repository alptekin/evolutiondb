16. Document, Graph & Log Stores

EvolutionDB has four special stores. They are the document store, the graph store, the message log, and the checkpoint store. Each one is a thin set of verbs on top of a normal engine table. MVCC visibility, row-level security, and the standard error path all work the same way.

16.1. Overview

The stores in this chapter are not separate storage engines. Each one is a small layer on top of the same SELECT, INSERT, UPDATE, and DELETE machinery that runs normal SQL. When you create a store, EvolutionDB creates one or more normal backing tables in the current schema. These tables have a fixed column layout. EvolutionDB also records the store in the system catalog. Each store has its own verbs. Examples are DOCUMENT WRITE, GRAPH UPSERT NODE, and LOG APPEND. These verbs turn their arguments into the same internal request structures that a hand-written DML statement would build. Then they call the normal processor.

This design has the same results for all four stores.

  • Documents, nodes, edges, log entries, and checkpoints are normal tuples in the binary tuple format. They are stored in slotted pages and indexed by B+ trees.
  • Transaction behavior, MVCC snapshot visibility, and the setjmp/longjmp error path work the same way for store operations and for plain DML.
  • The backing tables are real tables. Once you know how they are named, you can read them directly with SELECT. This helps when you debug.

The backing-table name for each store comes from the store name in a fixed way.

Store kindCatalog commandBacking table(s)
Document storeCREATE DOCUMENT STOREdoc_<name>
Graph storeCREATE GRAPH STOREkg_<name>_nodes, kg_<name>_edges
Message logCREATE MESSAGE LOGml_<name>
Checkpoint storeCREATE CHECKPOINT STOREck_<name>
Note. None of these store types are in the SQL standard. PostgreSQL has no direct equivalent. In PostgreSQL you would use a jsonb column, the third-party pgvector extension, recursive CTEs for graph traversal, or external services. EvolutionDB puts these patterns into dedicated verbs. The comparisons in each section below are there to help you. They are not claims that the features are equal.

16.2. Document Store

A document store is a collection of JSON documents. It can also have a vector embedding column for similarity search. You create it with CREATE DOCUMENT STORE. This builds a backing table named doc_<name> with a fixed layout.

id
VARCHAR(255), NOT NULL, primary key.
content
VARCHAR(8000), the document body.
meta
VARCHAR(8000), a JSON-shaped metadata string; may be NULL.
embedding
VECTOR(N), created only when embedding_dim > 0.
created_at
TIMESTAMP, NOT NULL, set to the UTC time of each write.

The store also records a distance kind. You choose it when you create the store. It is used later to rank vector searches. The accepted values are not case-sensitive.

WITH distanceInternal kindScoring expression
'cosine' (default)MEMORY_DIST_COSINEcosine distance
'l2'MEMORY_DIST_L2Euclidean (L2) distance
'inner' / 'ip'MEMORY_DIST_INNERinner product
'l1'MEMORY_DIST_L1Manhattan (L1) distance

Any unknown distance value falls back to 'cosine' without warning.

16.2.1. Writing documents

DOCUMENT WRITE runs through INSERT. The verb supplies id and content. It also supplies meta, or a NULL marker for meta. It supplies the embedding when the store has a vector column. It always supplies a server-generated created_at timestamp. An embedding is only sent when two things are true. The store must have an embedding column, and you must supply a non-empty embedding literal.

DOCUMENT WRITE INTO articles
    VALUES ('a1', 'Vector databases explained',
            '{"lang":"en","tags":["db"]}');

For a store with an embedding column, give the vector literal as the fourth value.

DOCUMENT WRITE INTO articles
    VALUES ('a2', 'Nearest-neighbor search',
            '{"lang":"en"}', '[0.12, 0.04, 0.99]');

16.2.2. Filtering documents

DOCUMENT FILTER returns id, content, meta, and created_at. It also returns embedding when that column is present. It runs through SELECT. The WHERE argument is a Mongo-style filter in JSON. The engine parses it into the same whereExpr tree that a normal WHERE clause produces.

DOCUMENT FILTER FROM articles
    WHERE '{"lang":"en"}' LIMIT 10;

The empty filter '{}' is the all-documents wildcard. It parses with no error but makes no clauses. So the SELECT below it runs with no WHERE restriction. A LIMIT is applied only when it is greater than zero.

16.2.3. Vector search

DOCUMENT SEARCH ranks documents by similarity. It only works on a store that has an embedding column. If the store has no embedding column, it fails with SQLSTATE 42P01. The verb returns id, content, meta, and a computed score column. The score uses the store's distance kind. It compares the embedding column with the query vector you supply. You can add a Mongo-style filter to narrow the set of candidates.

DOCUMENT SEARCH articles
    USING VECTOR '[0.10, 0.05, 0.98]'
    WHERE '{"lang":"en"}'
    LIMIT 5;
Note, client-side ordering. In this release DOCUMENT SEARCH returns the score column. It does not run a server-side ORDER BY on it. The caller must sort on the score column on the client side. A server-side ORDER BY on the score expression is planned. So a LIMIT caps the candidate rows that get a score. It does not give you the top-K by distance.

16.2.4. Deleting documents

DOCUMENT DELETE runs through DELETE. It requires a filter argument. If you leave the filter out, it raises SQLSTATE 42601. This is a deliberate guardrail. To wipe a whole store you must pass the wildcard '{}' on purpose. The engine turns it into a 1 = 1 sentinel predicate. This lets the delete path tell "delete all on purpose" apart from "missing WHERE clause".

-- delete one matching document
DOCUMENT DELETE FROM articles WHERE '{"lang":"de"}';

-- intentionally wipe the whole store
DOCUMENT DELETE FROM articles WHERE '{}';

You remove a document store and its backing table with DROP DOCUMENT STORE. This drops the doc_<name> table and removes the store from the registry.

16.3. Graph Store

A graph store is a bitemporal knowledge graph. You create it with CREATE GRAPH STORE. It keeps two backing tables.

kg_<name>_nodes
id VARCHAR(255) PK NOT NULL, type VARCHAR(64), props VARCHAR(8000) (JSON-shaped, may be NULL), valid_from TIMESTAMP NOT NULL.
kg_<name>_edges
id BIGINT PK AUTO_INCREMENT, src / dst VARCHAR(255) NOT NULL, rel VARCHAR(128) NOT NULL, props VARCHAR(8000), valid_from TIMESTAMP NOT NULL, and invalid_at TIMESTAMP where NULL means "currently active".

The invalid_at column is the heart of the temporal model. Edges are never updated in place to change their meaning. Instead, you replace an edge. The old version is closed by stamping its invalid_at. A fresh row then records the new fact. This lets you answer "what did the graph look like at time T".

16.3.1. Upserting nodes

GRAPH UPSERT NODE sets the latest type and props for a node id. First it looks up the id in the nodes table's primary-key B+ tree. If the node exists, it runs an UPDATE in place. This avoids a delete-then-insert round-trip. That round-trip works badly with same-transaction MVCC visibility. If the node does not exist, it runs an INSERT for a new row. When you give no type, the node type defaults to the literal node. The valid_from stamp is set to the current UTC time on every upsert.

GRAPH UPSERT NODE INTO kg
    VALUES ('person:alice', 'Person', '{"name":"Alice"}');

16.3.2. Upserting edges

GRAPH UPSERT EDGE records a directed, typed relationship. It stores it as a bitemporal triple (src, rel, dst). It does two steps.

  1. It runs an UPDATE. This stamps invalid_at = NOW() on any still-live row with the same (src, rel, dst). Still-live means rows where invalid_at IS NULL. This closes the old version of the fact.
  2. Then it runs an INSERT for a fresh edge row with valid_from = NOW() and invalid_at = NULL. The edge id is auto-incremented. You do not supply it.
GRAPH UPSERT EDGE INTO kg
    VALUES ('person:alice', 'WORKS_AT', 'org:acme',
            '{"since":2021}');

Old versions are kept. So the edge table grows as facts change. This is the cost of a graph you can fully audit and travel back in time through.

16.3.3. Querying neighbors

GRAPH NEIGHBORS OF follows outgoing edges from a node. It runs through SELECT. It returns src, rel, dst, props, valid_from, and invalid_at. The OF '<id>' clause is required. If it is missing, you get SQLSTATE 42601. The temporal filter depends on whether you give AS OF.

No AS OF
Only currently-live edges are returned. The filter is src = id AND invalid_at IS NULL.
With AS OF ts
Edges that were live at that moment are returned. The filter is src = id AND valid_from <= ts AND (invalid_at IS NULL OR invalid_at > ts).
-- current neighbors
GRAPH NEIGHBORS OF 'person:alice' IN kg;

-- neighbors as the graph stood at a past moment
GRAPH NEIGHBORS OF 'person:alice' IN kg
    AS OF '2024-01-01 00:00:00';
Note, depth is single-hop in this release. The grammar accepts a DEPTH N clause. The default is 1. For now the engine only runs depth-1 traversal as a single SELECT. Multi-hop BFS expansion is planned. Until then, you build depth-2 traversals by making two NEIGHBORS calls and joining the results on the client side.

You remove a graph store and both of its backing tables with DROP GRAPH STORE.

16.4. Message Log

A message log is an append-only chat-history structure. You create it with CREATE MESSAGE LOG. It is backed by a table named ml_<name>.

seq
BIGINT PRIMARY KEY AUTO_INCREMENT, the monotonic sequence number.
session_id
VARCHAR(255), NOT NULL.
role
VARCHAR(64), for example user or assistant.
content
VARCHAR(8000).
meta
VARCHAR(8000), JSON-shaped, may be NULL.
created_at
TIMESTAMP, NOT NULL, UTC write time.

The store can also record an optional ttl_days retention hint when you create it. The content column is VARCHAR(8000) on purpose. It is not an unbounded type. This keeps each row small enough to fit inline in the slotted-page format. If you need larger payloads, split the content into parts on the client side.

16.4.1. Appending entries

LOG APPEND runs through INSERT with an explicit column list. The columns are session_id, role, content, meta, and created_at. The engine fills the seq column through its auto-increment path. It sets created_at to the current UTC time. So the caller never supplies them.

LOG APPEND INTO chat
    VALUES ('sess-42', 'user', 'Hello there',
            '{"client":"web"}');

16.4.2. Reading entries

LOG READ returns seq, role, content, meta, and created_at. It filters by session_id. It runs through SELECT. The WHERE SESSION = '<sid>' clause is required. If it is missing, you get SQLSTATE 42601. You can add a LAST n form to cap the result to the most recent n rows.

LOG READ FROM chat WHERE SESSION = 'sess-42';

LOG READ FROM chat WHERE SESSION = 'sess-42' LAST 20;

16.4.3. Truncating and counting

LOG TRUNCATE runs through DELETE. It also requires a WHERE SESSION clause. Without a BEFORE clause it deletes every row for the session. With BEFORE n it deletes only rows whose seq < n. This is the natural way to prune old turns and keep recent history.

-- drop the whole session
LOG TRUNCATE FROM chat WHERE SESSION = 'sess-42';

-- keep only seq >= 100
LOG TRUNCATE FROM chat WHERE SESSION = 'sess-42' BEFORE 100;

LOG COUNT returns the number of entries for a session. It uses a COUNT(*) that runs through SELECT.

LOG COUNT FROM chat WHERE SESSION = 'sess-42';
Note, append-only by convention. The log is "append-only" in a simple sense. The verbs only add rows with LOG APPEND or bulk-prune rows with LOG TRUNCATE. There is no in-place update verb. The backing table is a normal table. So the ttl_days retention hint is recorded, but these verbs do not enforce it on their own. You prune on purpose with LOG TRUNCATE.

16.5. Checkpoint Store

A checkpoint store is a native backend for agent state snapshots. Its design follows LangGraph's checkpoint saver. You create it with CREATE CHECKPOINT STORE. It is backed by a table named ck_<name> with a fixed eight-column layout.

thread_id
VARCHAR(255), NOT NULL.
checkpoint_ns
VARCHAR(255), the namespace. It can be NULL. An empty string maps to NULL in this engine.
checkpoint_id
VARCHAR(255), NOT NULL, primary key.
parent_id
VARCHAR(255).
ck_values
JSON, the serialized checkpoint state. It is named ck_values and not values to avoid the SQL reserved word.
metadata
JSON.
parent_config
JSON.
created_at
TIMESTAMP, set by the engine on each PUT.

CREATE CHECKPOINT STORE accepts an optional WITH (retention = '...') clause. The retention string is recorded with the store. DROP CHECKPOINT STORE accepts CASCADE. In this release the drop always removes the backing table and the registry entry.

16.5.1. Writing checkpoints (PUT)

CHECKPOINT PUT writes one eight-column row through INSERT. You supply seven fields. They are thread_id, checkpoint_ns, checkpoint_id, parent_id, ck_values, metadata, and parent_config. The engine stamps the eighth field, created_at. Any field may be sent as NULL. There is also a PUT WRITES variant. In this release the application handles the difference between PUT and PUT WRITES.

CHECKPOINT PUT INTO agent_state
    VALUES ('thread-1', '', 'ckpt-001', NULL,
            '{"step":1}', '{"source":"input"}', NULL);

16.5.2. Reading checkpoints (GET / LIST)

Both read verbs run through SELECT. They use a star projection over the backing table. They use a WHERE thread_id = '...' filter. They use ORDER BY created_at DESC, so the newest row comes first. The two verbs differ as follows.

CHECKPOINT GET
Returns the latest checkpoint for a thread with LIMIT 1. You can add an AT '<checkpoint_id>' clause to narrow the filter to one checkpoint. That picks a single fixed row. So the LIMIT is dropped when you give AT.
CHECKPOINT LIST
Returns the full history for a thread. You can cap it with a positive LIMIT.
CHECKPOINT GET FROM agent_state WHERE THREAD = 'thread-1';

CHECKPOINT GET FROM agent_state WHERE THREAD = 'thread-1'
    AT 'ckpt-001';

CHECKPOINT LIST FROM agent_state WHERE THREAD = 'thread-1'
    LIMIT 50;
Note, direct table access. The DDL commands CREATE and DROP CHECKPOINT STORE and the PUT, GET, and LIST verbs all work on the same ck_<name> table. The backing table uses a ck_ prefix on purpose. It does not use a leading-underscore form. The lexer's identifier pattern is [A-Za-z][A-Za-z0-9_]*, which does not allow a leading underscore. This also means you can use the table directly with plain INSERT and SELECT. See also the CHECKPOINT reference page.

16.6. Common Errors

All four stores raise errors through the standard engine error path. Each error has a SQLSTATE code. You will see these codes most often.

SQLSTATEMeaning in a store context
42601Syntax or usage error. The store name is missing, or a required clause is left out. For example, DOCUMENT DELETE without a filter, GRAPH NEIGHBORS without OF, or LOG READ, COUNT, or TRUNCATE without WHERE SESSION.
42704The store does not exist. Every DML verb can raise this. DROP without IF EXISTS can also raise it.
42710A store with the same name already exists. This happens on CREATE without the idempotent form.
42P01DOCUMENT SEARCH on a store created without an embedding column.
22023Invalid Mongo-style filter JSON passed to DOCUMENT FILTER, DOCUMENT SEARCH, or DOCUMENT DELETE.

16.7. Compatibility

None of the four store types are in the SQL standard. PostgreSQL has no direct counterpart for any of them. They are EvolutionDB extensions. Here are some honest points of contrast.

  • Idempotency keyword. The CREATE ... STORE statements use IF EXISTS as the idempotent form. They do not use PostgreSQL's IF NOT EXISTS. The meaning is the same. It skips creation when the store is already there.
  • Fixed schemas. Every backing table has a fixed column layout. You cannot define it yourself. You cannot add your own columns through the store commands.
  • Documents vs. jsonb / pgvector. PostgreSQL applications join a jsonb column with the third-party pgvector extension. EvolutionDB puts JSON storage and vector search into a single store with DOCUMENT verbs. HNSW build parameters and similar tuning knobs are not exposed.
  • Graph traversal. PostgreSQL writes graph queries with recursive CTEs over user tables. EvolutionDB ships a dedicated bitemporal model. It has invalid_at versioning and an AS OF snapshot filter. Multi-hop traversal is single-hop in this release.
  • Vector ordering. Top-K ranking for DOCUMENT SEARCH is done on the client side for now. Server-side ORDER BY on the score is planned.
