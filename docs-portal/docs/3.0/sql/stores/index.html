16. Document, Graph & Log Stores

EvolutionDB exposes four purpose-built stores — document, graph, message-log, and checkpoint — each a thin verb surface over an ordinary engine table, so MVCC visibility, row-level security, and the standard error path all carry through unchanged.

16.1. Overview

The stores described in this chapter are not separate storage engines. Each one is a small layer of staging and wiring on top of the same SELECT / INSERT / UPDATE / DELETE machinery that powers ordinary SQL. When you create a store, EvolutionDB creates one or more regular backing tables in the current schema, with a fixed column layout, and registers the store in the system catalog. The store-specific verbs (DOCUMENT WRITE, GRAPH UPSERT NODE, LOG APPEND, and so on) translate their arguments into the same internal request structures a hand-written DML statement would build, then call the regular processor.

The practical consequences of this design are uniform across all four stores:

  • Documents, nodes, edges, log entries, and checkpoints are ordinary tuples in the binary tuple format, stored in slotted pages and indexed by B+ trees.
  • Transactional behavior, MVCC snapshot visibility, and the setjmp/longjmp error path apply identically to store operations and to plain DML.
  • The backing tables are real tables. You can inspect them directly with SELECT once you know their naming convention, which is useful for debugging.

The backing-table name for each store is derived mechanically from the store name:

Store kindCatalog commandBacking table(s)
Document storeCREATE DOCUMENT STOREdoc_<name>
Graph storeCREATE GRAPH STOREkg_<name>_nodes, kg_<name>_edges
Message logCREATE MESSAGE LOGml_<name>
Checkpoint storeCREATE CHECKPOINT STOREck_<name>
Note. None of these store types exist in the SQL standard, and PostgreSQL has no direct equivalent. PostgreSQL applications would reach for a jsonb column, the third-party pgvector extension, recursive CTEs for graph traversal, or external services. EvolutionDB folds these patterns into dedicated verbs. The contrasts in each section below are offered as honest orientation, not as claims of feature parity.

16.2. Document Store

A document store is a JSON-document collection with an optional vector embedding column for similarity search. It is created by CREATE DOCUMENT STORE, which builds a backing table named doc_<name> with this fixed layout:

id
VARCHAR(255), NOT NULL, primary key.
content
VARCHAR(8000) — the document body.
meta
VARCHAR(8000) — a JSON-shaped metadata string; may be NULL.
embedding
VECTOR(N) — created only when embedding_dim > 0.
created_at
TIMESTAMP, NOT NULL — set to the UTC time of each write.

The store also records a distance kind chosen at creation time, used later to rank vector searches. The accepted values are case-insensitive:

WITH distanceInternal kindScoring expression
'cosine' (default)MEMORY_DIST_COSINEcosine distance
'l2'MEMORY_DIST_L2Euclidean (L2) distance
'inner' / 'ip'MEMORY_DIST_INNERinner product
'l1'MEMORY_DIST_L1Manhattan (L1) distance

Any unrecognized distance value silently falls back to 'cosine'.

16.2.1. Writing documents

DOCUMENT WRITE routes through INSERT. The verb supplies id, content, and meta (or a NULL marker for meta), optionally the embedding when the store has a vector column, and always a server-generated created_at timestamp. An embedding is only sent when the store was created with an embedding column and a non-empty embedding literal is supplied.

DOCUMENT WRITE INTO articles
    VALUES ('a1', 'Vector databases explained',
            '{"lang":"en","tags":["db"]}');

For a store with an embedding column, supply the vector literal as the fourth value:

DOCUMENT WRITE INTO articles
    VALUES ('a2', 'Nearest-neighbor search',
            '{"lang":"en"}', '[0.12, 0.04, 0.99]');

16.2.2. Filtering documents

DOCUMENT FILTER projects id, content, meta, embedding (when present), and created_at, and routes through SELECT. The WHERE argument is a Mongo-style filter JSON, which the engine parses into the same whereExpr tree an ordinary WHERE clause produces.

DOCUMENT FILTER FROM articles
    WHERE '{"lang":"en"}' LIMIT 10;

The empty filter '{}' is the all-documents wildcard: it parses cleanly but produces no clauses, so the underlying SELECT runs with no WHERE restriction. A LIMIT is applied only when it is greater than zero.

16.2.3. Vector search

DOCUMENT SEARCH performs similarity ranking and is only valid on a store that was created with an embedding column; otherwise it fails with SQLSTATE 42P01. The verb projects id, content, meta, and a computed score column built from the store's distance kind applied to the embedding column against the supplied query vector. An optional Mongo-style filter narrows the candidate set.

DOCUMENT SEARCH articles
    USING VECTOR '[0.10, 0.05, 0.98]'
    WHERE '{"lang":"en"}'
    LIMIT 5;
Note — client-side ordering. In this release DOCUMENT SEARCH emits the score column but does not apply a server-side ORDER BY on it; the caller is expected to sort on the score column client-side. A server-side ORDER BY on the score expression is planned. The LIMIT, if any, therefore caps the candidate rows that are scored, not the top-K by distance.

16.2.4. Deleting documents

DOCUMENT DELETE routes through DELETE and requires a filter argument — omitting it raises SQLSTATE 42601. This is a deliberate guardrail: to wipe an entire store you must explicitly pass the wildcard '{}', which the engine expands into a 1 = 1 sentinel predicate so the delete path can distinguish "delete all (intentional)" from "missing WHERE clause".

-- delete one matching document
DOCUMENT DELETE FROM articles WHERE '{"lang":"de"}';

-- intentionally wipe the whole store
DOCUMENT DELETE FROM articles WHERE '{}';

A document store and its backing table are removed with DROP DOCUMENT STORE, which drops the doc_<name> table and unregisters the store.

16.3. Graph Store

A graph store is a bitemporal knowledge graph created by CREATE GRAPH STORE. It maintains two backing tables:

kg_<name>_nodes
id VARCHAR(255) PK NOT NULL, type VARCHAR(64), props VARCHAR(8000) (JSON-shaped, may be NULL), valid_from TIMESTAMP NOT NULL.
kg_<name>_edges
id BIGINT PK AUTO_INCREMENT, src / dst VARCHAR(255) NOT NULL, rel VARCHAR(128) NOT NULL, props VARCHAR(8000), valid_from TIMESTAMP NOT NULL, and invalid_at TIMESTAMP where NULL means "currently active".

The invalid_at column is the heart of the temporal model: edges are never updated in place to change their meaning. Instead, superseding an edge closes the old version by stamping its invalid_at, while a fresh row records the new fact. This makes "what did the graph look like at time T" answerable.

16.3.1. Upserting nodes

GRAPH UPSERT NODE sets the latest type and props for a node id. The implementation first probes the nodes table's primary-key B+ tree for the id. If the node exists, it issues an UPDATE in place (avoiding a delete-then-insert round-trip that interacts poorly with same-transaction MVCC visibility); otherwise it INSERTs a new row. When no type is supplied, the node type defaults to the literal node. The valid_from stamp is set to the current UTC time on every upsert.

GRAPH UPSERT NODE INTO kg
    VALUES ('person:alice', 'Person', '{"name":"Alice"}');

16.3.2. Upserting edges

GRAPH UPSERT EDGE records a directed, typed relationship as a bitemporal triple (src, rel, dst). It performs two steps:

  1. It runs an UPDATE that stamps invalid_at = NOW() on any still-live row matching the same (src, rel, dst) — that is, rows where invalid_at IS NULL. This closes the prior version of the fact.
  2. It then INSERTs a fresh edge row with valid_from = NOW() and invalid_at = NULL. The edge id is auto-incremented and is not user-supplied.
GRAPH UPSERT EDGE INTO kg
    VALUES ('person:alice', 'WORKS_AT', 'org:acme',
            '{"since":2021}');

Because old versions are retained, the edge table grows as facts change; this is the cost of a fully auditable, time-travelable graph.

16.3.3. Querying neighbors

GRAPH NEIGHBORS OF traverses outgoing edges from a node and routes through SELECT, projecting src, rel, dst, props, valid_from, and invalid_at. The OF '<id>' clause is required (SQLSTATE 42601 if missing). The temporal filter depends on whether AS OF is given:

No AS OF
Only currently-live edges are returned: src = id AND invalid_at IS NULL.
With AS OF ts
Edges live at that instant are returned: src = id AND valid_from <= ts AND (invalid_at IS NULL OR invalid_at > ts).
-- current neighbors
GRAPH NEIGHBORS OF 'person:alice' IN kg;

-- neighbors as the graph stood at a past moment
GRAPH NEIGHBORS OF 'person:alice' IN kg
    AS OF '2024-01-01 00:00:00';
Note — depth is single-hop in this release. The grammar accepts a DEPTH N clause (default 1), but the implementation currently executes only depth-1 traversal as a single SELECT. Multi-hop BFS expansion is planned; until then, depth-2 traversals are built by issuing two NEIGHBORS calls and unioning the results client-side.

A graph store and both of its backing tables are removed with DROP GRAPH STORE.

16.4. Message Log

A message log is an append-only chat-history structure created by CREATE MESSAGE LOG. It is backed by a table named ml_<name>:

seq
BIGINT PRIMARY KEY AUTO_INCREMENT — the monotonic sequence number.
session_id
VARCHAR(255), NOT NULL.
role
VARCHAR(64) — for example user or assistant.
content
VARCHAR(8000).
meta
VARCHAR(8000) — JSON-shaped, may be NULL.
created_at
TIMESTAMP, NOT NULL — UTC write time.

The store also records an optional ttl_days retention hint at creation time. The content column is intentionally VARCHAR(8000) rather than an unbounded type, which keeps each row inline-friendly for the slotted-page format; callers needing larger payloads are expected to shard the content on the client side.

16.4.1. Appending entries

LOG APPEND routes through INSERT with an explicit column list (session_id, role, content, meta, created_at). The seq column is filled by the engine's auto-increment path, and created_at is set to the current UTC time, so the caller never supplies them.

LOG APPEND INTO chat
    VALUES ('sess-42', 'user', 'Hello there',
            '{"client":"web"}');

16.4.2. Reading entries

LOG READ projects seq, role, content, meta, and created_at, filtered by session_id, and routes through SELECT. The WHERE SESSION = '<sid>' clause is required (SQLSTATE 42601 if missing). An optional LAST n form caps the result to the most recent n rows.

LOG READ FROM chat WHERE SESSION = 'sess-42';

LOG READ FROM chat WHERE SESSION = 'sess-42' LAST 20;

16.4.3. Truncating and counting

LOG TRUNCATE routes through DELETE and also requires a WHERE SESSION clause. Without a BEFORE clause it deletes every row for the session; with BEFORE n it deletes only rows whose seq < n, which is the natural way to prune old turns while keeping recent history.

-- drop the whole session
LOG TRUNCATE FROM chat WHERE SESSION = 'sess-42';

-- keep only seq >= 100
LOG TRUNCATE FROM chat WHERE SESSION = 'sess-42' BEFORE 100;

LOG COUNT returns the number of entries for a session via a COUNT(*) that routes through SELECT:

LOG COUNT FROM chat WHERE SESSION = 'sess-42';
Note — append-only by convention. The log is "append-only" in the sense that the verbs only add (LOG APPEND) or bulk-prune (LOG TRUNCATE); there is no in-place update verb. Because the backing table is a regular table, the ttl_days retention hint is recorded but is not enforced automatically by these verbs — pruning is performed explicitly via LOG TRUNCATE.

16.5. Checkpoint Store

A checkpoint store provides a native backend for agent state snapshots (modeled on LangGraph's checkpoint saver). It is created by CREATE CHECKPOINT STORE, backed by a table named ck_<name> with a fixed eight-column layout:

thread_id
VARCHAR(255), NOT NULL.
checkpoint_ns
VARCHAR(255) — namespace (nullable; an empty string maps to NULL in this engine).
checkpoint_id
VARCHAR(255), NOT NULL, primary key.
parent_id
VARCHAR(255).
ck_values
JSON — the serialized checkpoint state. Named ck_values rather than values to avoid the SQL reserved word.
metadata
JSON.
parent_config
JSON.
created_at
TIMESTAMP — set by the engine on each PUT.

CREATE CHECKPOINT STORE accepts an optional WITH (retention = '...') clause; the retention string is recorded with the store. DROP CHECKPOINT STORE accepts CASCADE; in this release the drop always removes the backing table along with the registry entry.

16.5.1. Writing checkpoints (PUT)

CHECKPOINT PUT writes a single eight-column row through INSERT. Seven fields are user-supplied — thread_id, checkpoint_ns, checkpoint_id, parent_id, ck_values, metadata, parent_config — and the eighth, created_at, is stamped by the engine. Any field may be sent as NULL. A PUT WRITES variant exists; in this release the distinction between PUT and PUT WRITES is handled at the application level.

CHECKPOINT PUT INTO agent_state
    VALUES ('thread-1', '', 'ckpt-001', NULL,
            '{"step":1}', '{"source":"input"}', NULL);

16.5.2. Reading checkpoints (GET / LIST)

Both read verbs route through SELECT with a star projection over the backing table, a WHERE thread_id = '...' filter, and ORDER BY created_at DESC (newest first):

CHECKPOINT GET
Returns the latest checkpoint for a thread (LIMIT 1). An optional AT '<checkpoint_id>' clause narrows the filter to a specific checkpoint; because that identifies a single deterministic row, the LIMIT is dropped when AT is given.
CHECKPOINT LIST
Returns the full history for a thread, optionally capped by a positive LIMIT.
CHECKPOINT GET FROM agent_state WHERE THREAD = 'thread-1';

CHECKPOINT GET FROM agent_state WHERE THREAD = 'thread-1'
    AT 'ckpt-001';

CHECKPOINT LIST FROM agent_state WHERE THREAD = 'thread-1'
    LIMIT 50;
Note — direct table access. The DDL surface (CREATE / DROP CHECKPOINT STORE) and the PUT / GET / LIST verbs all operate on the same ck_<name> table. The backing table is deliberately named with a ck_ prefix (rather than a leading-underscore form) because the lexer's identifier pattern is [A-Za-z][A-Za-z0-9_]* and does not permit a leading underscore — which also means you can address the table directly with plain INSERT and SELECT. See also the CHECKPOINT reference page.

16.6. Common Errors

All four stores raise errors through the standard engine error path with SQLSTATE codes. The codes you will encounter most often are:

SQLSTATEMeaning in a store context
42601Syntax / usage error: missing store name, or a required clause omitted (for example DOCUMENT DELETE without a filter, GRAPH NEIGHBORS without OF, LOG READ/COUNT/TRUNCATE without WHERE SESSION).
42704The referenced store does not exist (raised by every DML verb and by DROP without IF EXISTS).
42710A store with the same name already exists (on CREATE without the idempotent form).
42P01DOCUMENT SEARCH on a store created without an embedding column.
22023Invalid Mongo-style filter JSON passed to DOCUMENT FILTER, DOCUMENT SEARCH, or DOCUMENT DELETE.

16.7. Compatibility

None of the four store types appear in the SQL standard, and PostgreSQL has no direct counterpart for any of them; they are EvolutionDB extensions. Honest points of contrast:

  • Idempotency keyword. The CREATE ... STORE statements use IF EXISTS as the idempotent form, not PostgreSQL's IF NOT EXISTS; the meaning is the same (skip creation when present).
  • Fixed schemas. Every backing table has a fixed, non-user-definable column layout. You cannot add arbitrary columns through the store commands.
  • Documents vs. jsonb / pgvector. PostgreSQL applications combine a jsonb column with the third-party pgvector extension; EvolutionDB folds JSON storage and vector search into a single store with DOCUMENT verbs. HNSW build parameters and similar tuning knobs are not exposed.
  • Graph traversal. PostgreSQL expresses graph queries with recursive CTEs over user tables; EvolutionDB ships a dedicated bitemporal model with invalid_at versioning and an AS OF snapshot filter, though multi-hop traversal is single-hop in this release.
  • Vector ordering. Top-K ranking for DOCUMENT SEARCH is currently completed client-side; server-side ORDER BY on the score is planned.
