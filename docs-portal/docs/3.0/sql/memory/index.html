15. Memory & Entity Stores

EvolutionDB exposes two first-class agent-memory primitives — the memory store (a namespaced JSON key/value table with optional per-row embeddings and semantic SEARCH) and the entity store (a per-key summary table that auto-tracks mention counts) — implemented as thin SQL statements that compile down to ordinary tables, B+ tree primary keys, MVCC tuples, and the engine's existing INSERT/SELECT/UPDATE/DELETE machinery.

15.1. Overview

A memory store and an entity store are not new storage engines. Each is a named catalog object bound to one regular backing table; the dedicated MEMORY ... and ENTITY ... statements are convenience verbs that build the right tuple and route it through the same code path a hand-written DML statement would take. Because of that, everything the engine already does to a normal table — primary-key indexing, MVCC visibility, row-level security policies, foreign-key cascades, triggers — applies unchanged.

The two stores answer different questions:

Memory store
"Give me the value I filed under namespace + key," or "find the rows whose embedding is closest to this query vector." A namespaced JSON KV store that can optionally carry one VECTOR(N) column per row for semantic recall. See CREATE MEMORY STORE.
Entity store
"What do I know about this entity, and how often has it come up?" A per-key table holding a summary, an optional facts blob, an auto-incrementing mention_count, and a last_seen timestamp. See CREATE ENTITY STORE.

Relationship to PostgreSQL. These statements are EvolutionDB extensions; they are not part of the SQL standard and have no PostgreSQL equivalent. PostgreSQL users would model the same data with a hand-rolled table plus the pgvector extension and application-side upsert logic. EvolutionDB folds the table layout, the upsert semantics, and the distance ranking into single statements. Vector distance itself reuses the engine's native operators documented in the Vectors chapter.

15.2. Creating a memory store

A memory store is created with CREATE MEMORY STORE. The optional WITH list declares whether the store carries embeddings and which distance metric SEARCH should use.

CREATE MEMORY STORE name
    [ WITH ( embedding_dim = N [, distance = 'cosine' ] ) ]
embedding_dim = N
Dimensionality of the per-row embedding. When N > 0 the backing table gains a VECTOR(N) column and the store supports MEMORY SEARCH. When omitted (or 0, the default) no embedding column is created at all and the store is a pure JSON KV table — SEARCH against it is an error.
distance = '...'
Distance metric recorded in the catalog and used later by SEARCH. Accepted literals are listed in §15.5. The default is cosine; any unrecognized literal silently falls back to cosine.
-- semantic store with 1536-d embeddings, L2 distance
CREATE MEMORY STORE agent_mem WITH (embedding_dim = 1536, distance = 'l2');

-- plain namespaced JSON KV store, no vectors
CREATE MEMORY STORE scratch;

The statement also supports CREATE MEMORY STORE IF NOT EXISTS name, which returns success if a store of that name already exists rather than raising 42710.

15.2.1. The backing table

Creating a memory store named foo creates a regular table named mem_foo in the current schema with the following columns. The embedding column is present only when embedding_dim > 0.

ColumnTypeNotes
mem_namespaceVARCHAR(255)NOT NULL
mem_keyVARCHAR(255)NOT NULL
pkVARCHAR(512)PRIMARY KEY — composite of namespace and key
mem_valueJSONnullable
embeddingVECTOR(N)only if embedding_dim > 0; nullable
created_atTIMESTAMPset at PUT time, UTC
ttl_atTIMESTAMPnullable; written NULL by every PUT (reserved)

The primary key is a single derived column, not a two-column composite. The namespace and key are joined into one string separated by the ASCII Record Separator byte (0x1E):

-- pk is: namespace + 0x1E + key
-- 0x1E avoids the engine's internal field/row separators (0x1F, 0x02)

Because the PK is an indexed VARCHAR, point lookups by namespace+key are O(log n) through the B+ tree, and the mem_namespace / mem_key columns remain queryable directly should you choose to run ordinary SQL against mem_foo.

15.3. Writing memories — MEMORY PUT

MEMORY PUT performs an upsert by primary key. The full reference is at MEMORY PUT.

MEMORY PUT INTO name VALUES (ns, key, value [, embedding])

Three positional values are required — namespace, key, JSON value — and a fourth optional value supplies the embedding (only meaningful for a store created with embedding_dim > 0). Each value may be a quoted string literal or NULL.

-- KV-only write
MEMORY PUT INTO scratch
  VALUES ('user:42', 'lang', '{"locale":"tr-TR"}');

-- write with an embedding (store created WITH embedding_dim = 4)
MEMORY PUT INTO agent_mem
  VALUES ('user:42', 'fact:1',
          '{"text":"prefers dark mode"}',
          '[0.10, 0.22, 0.07, 0.91]');

The embedding literal is a bracketed comma-separated float list (the same textual form the engine accepts for VECTOR columns elsewhere). Omitting the embedding on a semantic store, or passing NULL, leaves that row's embedding null — the row is still stored and retrievable by key, it simply will not participate meaningfully in a SEARCH score.

Upsert is atomic and crash-safe. PUT does not delete-then-insert. The insert is performed first and, on a primary-key collision, the PK is re-pointed to the freshly inserted tuple and the old one is dropped (new-first, swap-last). A failed re-insert can therefore never leave the row in a deleted state. The created_at timestamp is rewritten on every PUT; the value reflects the most recent write, not the first one.

15.4. Reading by key — MEMORY GET

MEMORY GET is an exact point lookup on namespace + key. See MEMORY GET.

MEMORY GET FROM name WHERE NS = '<ns>' AND KEY = '<key>'

The grammar is fixed: both the NS and the KEY predicate are mandatory and joined by AND. Internally this compiles to a single-row SELECT ... WHERE pk = '<ns>0x1E<key>' LIMIT 1. The projected columns are the namespace, key, value, the embedding (only when the store has one), and created_at.

MEMORY GET FROM scratch WHERE NS = 'user:42' AND KEY = 'lang';

15.5. Semantic recall — MEMORY SEARCH

MEMORY SEARCH ranks every row in the store by vector distance from a query embedding, closest first. See MEMORY SEARCH.

MEMORY SEARCH name USING VECTOR '<query>' [ LIMIT n ]

The query is a bracketed float list, identical in form to a stored embedding. The statement projects (mem_namespace, mem_key, mem_value, score), where score is computed per row using the distance function recorded for the store at CREATE time, and orders the result by score ascending so the nearest matches appear first. LIMIT defaults to no engine-imposed cap when omitted; supply a value to bound the result set (the recommended practice is to always pass a small LIMIT).

MEMORY SEARCH agent_mem
  USING VECTOR '[0.11, 0.20, 0.08, 0.90]'
  LIMIT 5;

The distance metric is fixed when the store is created (§15.2) and drives which native operator SEARCH uses for the score column:

distance literalInternal kindOperator used by SEARCH
'cosine' (default)MEMORY_DIST_COSINEcosine distance
'l2'MEMORY_DIST_L2Euclidean (L2) distance
'inner' / 'ip'MEMORY_DIST_INNERinner-product distance
'l1'MEMORY_DIST_L1Manhattan (L1) distance

SEARCH is a full scan, scored per row. It does not use an approximate vector index. The distance = ... metric configured at creation time is what makes the ranking semantically meaningful, but every row's distance is computed during the scan. Calling SEARCH against a store created without an embedding column raises an error (42P01): a non-semantic store has no vector to compare. Ordering is performed inside the engine via the score column's positional ordinal — there is no ORDER BY <expression> exposed on this statement.

15.6. Enumerating namespaces — MEMORY LIST NAMESPACES

To discover which namespaces exist in a store, use MEMORY LIST NAMESPACES. It compiles to SELECT DISTINCT mem_namespace FROM mem_<name>, with an optional prefix filter applied as a LIKE 'prefix%' predicate.

MEMORY LIST NAMESPACES IN name [ PREFIX '<prefix>' ]
MEMORY LIST NAMESPACES IN scratch;
MEMORY LIST NAMESPACES IN scratch PREFIX 'user:';

15.7. Removing memories — MEMORY DELETE

MEMORY DELETE removes the single row matching a namespace + key. See MEMORY DELETE.

MEMORY DELETE FROM name WHERE NS = '<ns>' AND KEY = '<key>'

As with GET, both predicates are mandatory. The statement routes through the engine's standard DELETE path against mem_<name> filtered on the composite pk. That routing is deliberate: any row-level security policy, foreign-key cascade, MVCC versioning, or trigger attached to the backing table applies exactly as it would to a hand-written DELETE FROM mem_<name> WHERE pk = '...'. This is what makes MEMORY DELETE safe to use in a multi-tenant database where each tenant is fenced off by an RLS policy.

MEMORY DELETE FROM scratch WHERE NS = 'user:42' AND KEY = 'lang';

15.8. Dropping a memory store

DROP MEMORY STORE removes both the catalog registration and the backing mem_<name> table.

DROP MEMORY STORE name [ CASCADE ]
DROP MEMORY STORE IF EXISTS name

IF EXISTS suppresses the not-found error (42704) and returns success. The backing table is dropped if present.

15.9. Entity stores

An entity store keeps one row per entity key, tracking a rolling summary plus how many times the entity has been mentioned. It is created and dropped analogously to a memory store but takes no WITH options. See CREATE ENTITY STORE.

CREATE ENTITY STORE name [ IF NOT EXISTS ]
DROP ENTITY STORE name [ IF EXISTS ]

Creating an entity store named foo builds a backing table named ent_foo with this fixed shape:

ColumnTypeNotes
entity_keyVARCHAR(255)PRIMARY KEY, NOT NULL
summaryVARCHAR(8000)nullable
factsVARCHAR(8000)nullable; a JSON-shaped string
mention_countINTNOT NULL; auto-incremented on each PUT
last_seenTIMESTAMPNOT NULL; advanced to the current UTC time on each PUT

15.9.1. ENTITY PUT — upsert with mention tracking

ENTITY PUT is an upsert keyed on entity_key, but unlike MEMORY PUT it carries semantics of its own: a first write inserts the row with mention_count = 1; every subsequent write to the same key updates the summary and facts, increments mention_count by one, and advances last_seen.

ENTITY PUT INTO name VALUES ('key', 'summary' [, 'facts'])
-- first mention: inserts with mention_count = 1
ENTITY PUT INTO people
  VALUES ('ada', 'Works on the storage layer');

-- second mention: updates summary, mention_count -> 2, last_seen bumped
ENTITY PUT INTO people
  VALUES ('ada', 'Leads the B+ tree rewrite',
          '{"team":"core","tz":"UTC+3"}');

The optional third value is the facts blob; when omitted it is stored as NULL. The existence check is a direct primary-key probe of the backing table's B+ tree, and the prior mention_count is read from the existing tuple before being incremented, so the counter is monotonic per key.

15.9.2. ENTITY GET

ENTITY GET fetches a single entity row by key.

ENTITY GET FROM name WHERE KEY = 'k'
ENTITY GET FROM people WHERE KEY = 'ada';

It projects (entity_key, summary, facts, mention_count, last_seen) for the matching key, limited to one row.

15.9.3. ENTITY RANK

ENTITY RANK returns the store's rows ordered by mention_count descending, tie-broken by last_seen descending — the most-discussed, most-recent entities first.

ENTITY RANK FROM name [ LIMIT n ]
-- top 10 entities by how often they've been mentioned
ENTITY RANK FROM people LIMIT 10;

The ordering is performed by the engine's own sort over the result set; LIMIT bounds the output when supplied.

15.10. Status & error codes

The store statements set SQLSTATE codes consistent with the rest of the engine. The codes most relevant to these primitives:

SQLSTATERaised when
42601A required name is missing (e.g. CREATE MEMORY STORE with no store name).
42710Creating a store whose name already exists (without IF NOT EXISTS).
42704Referencing a store (PUT / GET / SEARCH / DELETE / RANK / DROP) that does not exist.
42P01MEMORY SEARCH against a store created without an embedding column.

15.11. Notes & limitations

  • SEARCH is exact, not indexed. MEMORY SEARCH scans and scores every row; it does not consult an approximate-nearest-neighbour vector index. For large stores this is linear in the number of rows.
  • The distance metric is fixed at creation. It cannot be changed after CREATE MEMORY STORE; there is no ALTER for stores. Recreate the store to switch metrics.
  • No ORDER BY <expr> surface. SEARCH sorts internally by its computed score column via a positional ordinal; you cannot supply your own ordering expression on the statement itself.
  • ttl_at is reserved. The memory backing table carries a ttl_at column and the catalog descriptor carries a ttl_days field, but MEMORY PUT always writes ttl_at as NULL and no automatic expiry pass is wired up. Treat TTL as not yet implemented.
  • Fixed predicate shapes. MEMORY GET/DELETE require exactly WHERE NS = '...' AND KEY = '...', and ENTITY GET requires exactly WHERE KEY = '...'. These are not general WHERE clauses. For arbitrary queries, run ordinary SQL against the backing mem_<name> / ent_<name> table directly.
  • Entity facts is a string, not a JSON column. It is stored as VARCHAR(8000) holding a JSON-shaped string; the engine does not parse or validate it as JSON.
  • Stores live in the current schema. The backing table is created in the schema resolved from the current database/schema context; a store name is unique within the catalog's store registry.
  • Timestamps are UTC. created_at and last_seen are formatted as YYYY-MM-DD HH:MM:SS in UTC at write time.
