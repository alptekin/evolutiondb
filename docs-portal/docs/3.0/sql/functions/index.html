9. Functions & Operators

The operators, built-in functions, conditional expressions, type coercion, and NULL handling that EvolutionDB 3.0 evaluates in WHERE, SELECT, HAVING, SET, DEFAULT, and CHECK — exactly as implemented by the engine's single expression evaluator.

Every scalar expression in EvolutionDB is built into an expression tree by the Bison parser and evaluated by one function, expr_evaluate, in evolution/db/expression.c. There is no separate type system threaded through the tree: each node is evaluated to a string (its rendered value), and the evaluator returns a second flag indicating whether that value is NULL. A companion routine, expr_evaluate_num, evaluates the same tree to a C double for arithmetic. Understanding this string-plus-NULL-flag model explains most of the type-coercion and comparison behavior described in this chapter.

Scope of this chapter. Only functions and operators that the expression evaluator actually implements are documented here, with their real argument counts, return shapes, and edge behavior. Where EvolutionDB borrows surface syntax from MySQL or PostgreSQL but evaluates it differently — for example, integer-only %, a non-collating LIKE, or MySQL-style NULL-propagating CONCAT — the difference is called out. Functions that are parsed but only meaningful in other clauses (aggregates resolved by the query executor, window functions, set-returning unnest in FROM) are noted where they touch the evaluator.

9.1. The Evaluation Model

A scalar expression is evaluated against a row presented as parallel arrays of column names and string column values. The evaluator resolves a column reference by name (case-insensitively), returns string literals verbatim, and for any arithmetic or numeric node falls back to the numeric evaluator and re-renders the result as text. Integer results are printed without a decimal point; non-integers use %g formatting.

Two facts follow from the string-valued design and recur throughout this chapter:

Values carry no static type
A column value, a literal, and a function result are all text at evaluation time. Comparisons and arithmetic decide numeric-versus-textual treatment per operation, by inspecting the operand strings (see Section 9.7).
NULL is a separate return channel
The evaluator returns 1 when it produced a value and 0 to signal NULL. A NULL literal, a column whose stored value is the internal null marker, and an unresolved column name all evaluate to NULL. Functions that internally represent NULL in their output use the engine's null marker string.

9.2. Operators

9.2.1. Arithmetic and Bitwise Operators

Arithmetic is evaluated numerically as C double. The following operators are recognized:

OperatorOperationNotes
+Addition
-Subtraction; also unary negation
*Multiplication
/DivisionDivision by zero yields 0, not an error and not infinity.
%ModuloInteger modulo: both operands are truncated to long long first. Modulo by zero yields 0.
<<   >>Bit shift left / rightOperands truncated to long long.
&   |   ^Bitwise AND / OR / XOROperands truncated to long long.
 
SELECT price * 1.1, qty % 12, flags & 4 FROM orders;

NULL in arithmetic. Arithmetic runs through the numeric evaluator, which treats an unresolved or non-numeric operand as 0.0 rather than propagating NULL. So NULL + 1 reached through the numeric path resolves to 1, and a non-numeric string contributes its strtod value (0 for text with no leading number). This differs from the SQL standard, where any NULL arithmetic operand makes the whole expression NULL.

9.2.2. Comparison Operators

The six comparison operators return a boolean rendered as the single characters t or f:

=
Equal
<>
Not equal
<   >
Less than / greater than
<=   >=
Less than or equal / greater than or equal

A comparison first evaluates both sides. If either side is NULL, the comparison result is NULL (the evaluator returns the null channel), in keeping with SQL three-valued logic. Otherwise the evaluator decides how to compare: if both rendered values parse cleanly and entirely as numbers, it compares them numerically; otherwise it compares them as byte strings with strcmp (see Section 9.7).

SELECT * FROM orders WHERE total >= 100 AND status <> 'cancelled';

The <=> operator. EvolutionDB parses <=> as a vector cosine-distance operator (Section 9.6.7). When its operands are not vector literals it falls back to a plain equality test that returns t/f — so SELECT 1 <=> 1 returns t. It is not MySQL's null-safe equals in the general case.

9.2.3. Logical Operators

AND, OR, NOT, and XOR operate on boolean values with full SQL three-valued logic. A value counts as true if it is t, the word true (case-insensitive), or a non-zero number; anything else is false. The truth tables for NULL follow the standard:

ExpressionResult
FALSE AND NULLf
TRUE AND NULLNULL
NULL AND NULLNULL
TRUE OR NULLt
FALSE OR NULLNULL
NOT NULLNULL
NULL XOR xNULL (XOR requires both operands non-null)

9.2.4. Pattern Matching: LIKE and REGEXP

LIKE and NOT LIKE implement the two SQL wildcards: % matches any sequence of characters (including none) and _ matches exactly one character. Matching is performed byte-by-byte by an iterative backtracking matcher.

SELECT name FROM users WHERE name LIKE 'A%' AND email NOT LIKE '%@test._';

If either the subject or the pattern is NULL, the result is NULL.

REGEXP and NOT REGEXP match against a POSIX extended regular expression. On a NULL operand the result is NULL; an invalid pattern that fails to compile evaluates to f (false) rather than raising an error.

Limitations. LIKE is byte-exact and case-sensitive; there is no ESCAPE clause and no ILIKE, so a literal % or _ in the pattern cannot be escaped. REGEXP is backed by the platform POSIX <regex.h> on Linux/macOS and is compiled REG_EXTENDED | REG_NOSUB; on Windows (MinGW) builds, which lack POSIX regex, REGEXP always returns f.

9.2.5. NULL Predicates, BETWEEN, IN

IS NULL and IS NOT NULL test whether their operand is NULL. A column counts as null when its stored value is the engine's null marker or when the named column is not present in the row at all; any literal (integer, float, string, boolean) is never null.

BETWEEN, NOT BETWEEN, IN, and NOT IN are not evaluated as distinct node types — the parser desugars them into combinations of the operators above:

x BETWEEN a AND b
rewritten as (x >= a) AND (x <= b).
x NOT BETWEEN a AND b
rewritten as (x < a) OR (x > b).
x IN (a, b, c)
rewritten as (x = a) OR (x = b) OR (x = c), combined into a balanced OR tree so a long list stays shallow.
x NOT IN (a, b, c)
rewritten as (x <> a) AND (x <> b) AND (x <> c), combined into a balanced AND tree.

Because IN/NOT IN reduce to comparisons and logical operators, they inherit the three-valued NULL behavior of those operators. An empty IN () list evaluates to false and an empty NOT IN () to true.

9.3. Conditional Expressions

9.3.1. CASE

Both forms of CASE are supported. The searched form tests a boolean condition per branch; the simple form compares a single operand against each WHEN value. Internally the simple form is desugared into the searched form by turning each WHEN v into the comparison operand = v, so the two share one evaluator.

-- searched CASE
SELECT CASE
         WHEN score >= 90 THEN 'A'
         WHEN score >= 80 THEN 'B'
         ELSE 'F'
       END FROM grades;

-- simple CASE (desugars to status = 'A', status = 'I', ...)
SELECT CASE status WHEN 'A' THEN 'active'
                   WHEN 'I' THEN 'inactive' END FROM accounts;

Branches are evaluated top to bottom; the first WHEN whose condition is true selects its THEN result. A branch whose condition is NULL is treated as not matched and skipped. If no branch matches, the ELSE result is returned, or NULL if there is no ELSE.

9.3.2. COALESCE, NULLIF, IFNULL, IF

COALESCE(a, b)
Returns a if it is non-null, otherwise b, otherwise NULL. The implemented arity is two arguments.
NULLIF(a, b)
Returns NULL when a and b are equal as strings; otherwise returns a.
IFNULL(a, b)
MySQL-style: returns a if non-null, else b.
IF(cond, t, f)
MySQL-style three-argument conditional: evaluates cond for truthiness (t, true, 1, or any non-zero number) and returns the second or third argument accordingly; a NULL condition takes the false branch.

9.3.3. IS TRUE / IS FALSE

x IS TRUE returns t only when x renders as t, true, or 1; otherwise f (a NULL operand yields f). x IS FALSE returns t only when x renders as f, false, or 0.

9.4. String Functions

String functions evaluate their first argument and, with the noted exceptions, return NULL if it is NULL.

FunctionReturnsBehavior
UPPER(s) / LOWER(s)textASCII case fold via toupper/tolower per byte.
LENGTH(s)integerByte length of the string (strlen), not character count.
SUBSTRING(s, pos [, len])text1-based start position; a position below 1 is clamped to 1. With no length, returns the tail from pos.

REPLACE(s, from, to)textReplaces all non-overlapping occurrences of from with to. An empty from returns s unchanged.

SELECT UPPER(LEFT(name, 3)), LENGTH(bio),
       REPLACE(slug, '-', '_'),
       SUBSTRING(sku, 2, 4)
FROM profiles;

Encoding caveat. All string functions are byte-oriented. LENGTH, SUBSTRING, LEFT/RIGHT, and REVERSE count and slice by byte, so a multi-byte UTF-8 character is several units and can be split. UPPER/LOWER fold only ASCII letters. There is no separate CHAR_LENGTH; intermediate string buffers in the evaluator are bounded (commonly 512 bytes), so very long values may be truncated inside a function.

Math functions evaluate their arguments numerically and render the result with %g (or as an integer where noted). A NULL argument generally yields NULL.

Note. The function-named MOD(a, b) uses floating-point fmod and returns NULL on a zero divisor, whereas the % operator (Section 9.2.1) truncates to integers and returns 0 on a zero divisor. SQRT, LOG, and LOG10 of out-of-domain inputs follow the C library and may render as nan or -inf rather than raising an error.

9.6. Other Built-in Functions

9.6.1. Date and Time

CURRENT_TIMESTAMP / NOW()
Current local date-time as YYYY-MM-DD HH:MM:SS.
CURRENT_DATE
Current local date as YYYY-MM-DD.
CURRENT_TIME
Current local time as HH:MM:SS.
YEAR / MONTH / DAY / HOUR / MINUTE / SECOND(dt)
Extract a single field from a YYYY-MM-DD [HH:MM:SS] string by parsing it with sscanf.
DATEDIFF(d1, d2)

DATE_ADD(dt, n unit) / DATE_SUB(dt, n unit)
Add or subtract n of a unit. Recognized units are YEAR, MONTH, HOUR; any other unit is treated as DAY. The result is rendered as a date (YYYY-MM-DD).

Note. Date functions operate on the textual YYYY-MM-DD form and use the server's local time zone through the C library; there is no time-zone-aware TIMESTAMPTZ arithmetic here. DATE_ADD/DATE_SUB normalize through mktime and emit a date even when an HOUR unit is given.

9.6.5. Array Constructors and Functions

Arrays are represented internally as PostgreSQL-style brace text ({e1,e2,NULL,...}), with quoting applied to elements containing separators or whitespace.

ARRAY[e1, e2, ...]
Construct an array literal. A NULL element is rendered as the unquoted token NULL.
arr[i]
1-based subscript. An out-of-bounds or malformed index yields NULL (PostgreSQL semantics).
array_length(arr)
Element count.
x <op> ANY(arr)
True if the comparison holds for any element; if no element matches but the array contains a NULL, the result is NULL; otherwise false. Element comparison is numeric when both sides parse as numbers, else lexicographic.
unnest(arr)
In scalar context returns the array text as-is; true row-producing expansion requires FROM-clause placement, handled by the join pipeline rather than this evaluator.

9.6.6. Full-Text Match

MATCH(col) AGAINST('query') tokenizes both the row's column value and the query literal and returns the count of query terms that appear in the row (used as a boolean in WHERE and as a relevance score in ORDER BY). It is a simple "any-term" match — a positive count means a match — not a ranked full-text engine.

9.6.7. Vector Distance Functions and Operators

Vectors arrive as their text form, either bracketed ([f1,f2,...] or {f1,f2,...}) or as a compact quantized form (b64i8:<base64>) that is decoded back to floats. The distance operators and their named equivalents:

OperatorDistance
<=>Cosine distance (1 - cosine similarity)
L2 operatorEuclidean distance
inner-product operatorNegative dot product
L1 operatorManhattan distance

If either operand is NULL, the result is NULL. If the dimensions of the two vectors disagree, the result is NULL. When the operands are not vector literals, the cosine operator falls back to a string-equality test (returning t/f); the other operators require vectors and return NULL. Companion functions vector_dim(v) (element count), vector_norm(v) (L2 norm), and vector_normalize(v) (unit vector, rendered as bracketed text) round-trip through the same parser. Approximate nearest-neighbor helpers hnsw_knn(...) and hnsw_filter_knn(...) return comma-separated primary-key lists; see CREATE INDEX.

9.7. Type Coercion and CAST

Because values are untyped text during evaluation, coercion is decided per operation rather than by a declared expression type:

Comparisons
Both operands are compared numerically only when each renders as a complete, valid number (the entire string is consumed by strtod and is non-empty). If either side is not a clean number, the comparison falls back to a byte-wise strcmp. So '10' = 10 and '2' < '10' both behave numerically, but '10x' < '9' compares as text.
Arithmetic
Operands are coerced with strtod; text with no leading number contributes 0.
Boolean context
A value is true if it is t, true (case-insensitive), or a non-zero number; 1 is also accepted as true inside CASE and IF.

Explicit conversion uses CAST(expr AS type). The conversion is driven by the target type's family code (recall the type encoding typeVal / 10000 = family from Chapter 4):

Target familyCAST behavior
Integer typesParse with strtol and render as a base-10 integer.
Float / double typesParse with strtod and render with %g.
BooleanNon-zero numeric value renders as true, otherwise false.
VARCHAR / CHAR / TEXT / DATEString pass-through (no reformatting).
 
SELECT CAST(price AS INT), CAST(active AS BOOLEAN) FROM items;

A CAST of a NULL value is NULL. See Chapter 4, Data Types for the family codes and SELECT for where expressions may appear.

9.8. NULL Handling Summary

The treatment of NULL is not uniform across operations; the table below collects the rules from earlier sections so the differences are visible side by side.

ContextNULL behavior
Comparison operators (=, <, …)Any NULL operand → result NULL (three-valued).
AND / OR / NOT / XORStandard three-valued logic (see Section 9.2.3).
LIKE / REGEXPAny NULL operand → NULL.
String functionsNULL input generally → NULL.
CONCATAny NULL argument → NULL (MySQL style, not PostgreSQL ignore-null).
Most math functionsNULL input → NULL; MOD(a,0) → NULL.
Arithmetic operators (numeric path)NULL / non-numeric operand treated as 0 — does not propagate.
CASE branch with NULL conditionTreated as not matched; falls through.
COALESCE / IFNULLReturn the first non-null argument.
Array subscript out of rangeNULL.
Vector distance, dimension mismatchNULL.

Note — honest divergences from the SQL standard. Three behaviors differ from a standard-conforming engine and are worth remembering: (1) division and modulo by zero return 0 rather than raising a division_by_zero error; (2) arithmetic treats a NULL operand as 0 on the numeric path instead of propagating NULL; (3) CONCAT propagates NULL like MySQL, whereas PostgreSQL's || / concat() ignore nulls. String functions are byte-oriented and ASCII-only for case folding, and LIKE has no ESCAPE clause. Treat these as engine-specific and test edge cases explicitly.

9.9. Expressions in CHECK Constraints

A subset of the expression language can be persisted in a table's metadata for use in CHECK constraints. The expression is serialized to a compact reverse-Polish (postfix) token stream and re-parsed when the constraint is evaluated. The serializable node set is: column references, the four literal kinds plus NULL; the comparison, logical (AND/OR/NOT/XOR), and arithmetic operators; IS NULL/IS NOT NULL; LIKE/NOT LIKE; the functions UPPER, LOWER, LENGTH, TOKEN_LENGTH, CONCAT, COALESCE, the JSON arrow operators; and the session functions CURRENT_USER, CURRENT_TIMESTAMP, CURRENT_DATE, CURRENT_TIME. Node types outside this set are dropped during serialization, so a CHECK built from unsupported functions will not round-trip faithfully — keep CHECK expressions within the list above. See CREATE TABLE for constraint syntax.

CREATE TABLE accounts (
    id       INT PRIMARY KEY,
    email    VARCHAR(120) CHECK (LENGTH(email) > 3),
    balance  DECIMAL(12,2) CHECK (balance >= 0),
    owner    VARCHAR(60) CHECK (owner = CURRENT_USER)
);

For where these expressions are used in practice, see SELECT (projection, WHERE, HAVING), Chapter 6, Data Manipulation (SET assignments, DEFAULT, ON CONFLICT updates), and CREATE SEQUENCE for the sequence functions.
