13. JSON

EvolutionDB stores JSON documents as validated text and exposes a MySQL-compatible function library plus the -> / ->> path operators for reading, transforming, and searching them.

This chapter documents the JSON data type and every JSON operation that EvolutionDB actually implements. The engine's JSON support is a self-contained, allocation-light implementation built around a recursive-descent parser; it follows the MySQL function set and the MySQL $.path expression syntax rather than the PostgreSQL json/jsonb operator set or the SQL/JSON standard. Where the behaviour diverges from those systems, the difference is called out explicitly so you are never surprised.

At a glance. EvolutionDB JSON is text-backed: a JSON column holds the original document bytes, validated on write. There is no binary/decomposed storage form, no path indexing, and no in-place document mutation — the transformation functions return a new string. If you have used MySQL's JSON functions, the surface here will feel familiar; if you have used PostgreSQL jsonb, expect a different operator vocabulary.

13.1. The JSON Data Type

EvolutionDB provides a single JSON type, declared as either JSON or JSONB. Both spellings are accepted by the parser and resolve to the same type — there is no separate binary representation, so JSONB is purely an alias for compatibility with tooling that emits it.

PropertyValue
Type namesJSON, JSONB (alias)
Internal type code230000 (type family 23)
On-disk formOriginal JSON text, stored in the binary tuple format like any variable-length value
ValidationDocument is parsed and rejected if not well-formed JSON (on write)
Maximum nesting64 levels (JSON_MAX_DEPTH)

A value assigned to a JSON column is validated by the same parser used by JSON_VALID. An invalid document is rejected at write time:

CREATE TABLE events (
  id    INT PRIMARY KEY,
  payload JSON
);

-- accepted: well-formed object
INSERT INTO events VALUES (1, '{"user":"ada","tags":["x","y"],"active":true}');

-- rejected: trailing comma is not valid JSON
INSERT INTO events VALUES (2, '{"user":"ada",}');
-- ERROR: invalid JSON: '{"user":"ada",}'   (SQLSTATE data_exception)

See CREATE TABLE for column declaration and INSERT for write semantics.

Validation grammar. The parser accepts standard JSON: objects, arrays, strings (with \" \\ \/ \b \f \n \r \t and \uXXXX escapes), numbers (optional sign, integer/fraction/exponent), and the literals true, false, null. Leading-zero integers, control characters inside strings, and depth beyond 64 levels are rejected. Unlike PostgreSQL jsonb, duplicate object keys are not de-duplicated or reordered — the document is stored verbatim, so key order and whitespace are preserved.

13.2. Path Expressions

Functions and operators that address a position inside a document use MySQL-style path expressions. A path begins with $ (the document root) followed by zero or more steps:

.key
Member access into an object by key name.
[n]
Element access into an array by zero-based integer index.

Steps may be chained to any depth, for example $.user.name, $.tags[0], or $.items[2].sku. A bare $ refers to the entire document.

-- given payload = {"user":"ada","tags":["x","y"],"address":{"city":"berlin"}}
SELECT JSON_EXTRACT(payload, '$.user')        FROM events;  -- "ada"
SELECT JSON_EXTRACT(payload, '$.tags[1]')     FROM events;  -- "y"
SELECT JSON_EXTRACT(payload, '$.address.city') FROM events;  -- "berlin"
SELECT JSON_EXTRACT(payload, '$')             FROM events;  -- whole document
Supported subset. Only single-quoted-string keys via .key and numeric [n] indexes are recognised. Wildcards ($.*, $[*]), the recursive descent operator ($**), quoted member names ($."weird key"), and negative or range indexes are not supported. An index step requires at least one digit, and a key step requires a non-empty name; otherwise the path is rejected. Key names are matched against an internal 256-byte buffer, so extremely long keys are truncated for comparison.

13.3. The -> and ->> Operators

The arrow operators are shorthand for extraction and bind to a left-hand JSON expression and a right-hand path. They are left-associative.

json -> path
Extract the value at path, preserving its JSON form. A string result keeps its surrounding quotes.
json ->> path
Extract the value at path as text. A JSON string result is unquoted; a JSON null becomes SQL NULL.
-- payload = {"user":"ada","age":31}
SELECT payload -> '$.user'  FROM events;  -- "ada"   (quoted JSON string)
SELECT payload ->> '$.user' FROM events;  -- ada     (plain text)
SELECT payload -> '$.age'   FROM events;  -- 31

Both operators evaluate to SQL NULL when either operand is NULL or when the path does not match anything in the document. The right operand follows the same $.path syntax as the functions above; a bare key (no leading $) is also accepted and treated as a single object-key step.

Contrast with PostgreSQL. In PostgreSQL, -> takes a single key or array index (a text or integer), and #> / #>> take a path array; the @>, ?, and jsonb_path_* operators round out the set. EvolutionDB instead overloads -> and ->> to accept a whole MySQL $.path on the right, and provides no #>, @>, or ? operators. The containment and existence checks live in the JSON_CONTAINS / JSON_CONTAINS_PATH functions instead.

13.4. Inspection Functions

These functions read a document without modifying it.

JSON_VALID(doc)
Returns 1 if doc parses as well-formed JSON, otherwise 0. The same check used to validate writes to a JSON column.
JSON_TYPE(doc)
Returns the type of the top-level value: one of OBJECT, ARRAY, STRING, BOOLEAN, NULL, INTEGER, or DOUBLE. Numbers containing ., e, or E report as DOUBLE; otherwise INTEGER.
JSON_LENGTH(doc)
For an object, the number of keys; for an array, the number of elements; for a scalar, 1.
JSON_DEPTH(doc)
Nesting depth. A scalar or empty container has depth 1; each level of nested object/array adds one.
JSON_KEYS(doc)
Returns a JSON array of the top-level object's key names (with their quotes), e.g. ["user","tags"]. Defined only for objects.
JSON_EXTRACT(doc, path)
Returns the value at path in its JSON form, or NULL if the path does not match. Equivalent to the -> operator.
-- payload = {"user":"ada","tags":["x","y"],"meta":{"v":1.5}}
SELECT JSON_VALID(payload)   FROM events;  -- 1
SELECT JSON_TYPE(payload)    FROM events;  -- OBJECT
SELECT JSON_LENGTH(payload)  FROM events;  -- 3
SELECT JSON_DEPTH(payload)   FROM events;  -- 3
SELECT JSON_KEYS(payload)    FROM events;  -- ["user","tags","meta"]
SELECT JSON_TYPE('1.5')        FROM events;  -- DOUBLE
SELECT JSON_TYPE('42')         FROM events;  -- INTEGER

13.5. Construction and Formatting Functions

JSON_OBJECT(k1, v1[, k2, v2])
Builds a JSON object from key/value pairs. Accepts one or two pairs. A value that is itself valid JSON is embedded as-is; any other value is quoted as a JSON string.
JSON_ARRAY(v1[, v2[, v3]])
Builds a JSON array from one to three values, applying the same "valid JSON embeds, otherwise quote as string" rule per element.
JSON_QUOTE(str)
Wraps str in double quotes and escapes special characters (", \, newline, carriage return, tab, and control characters as \uXXXX), producing a JSON string literal.
JSON_UNQUOTE(doc)
Inverse of JSON_QUOTE: if doc is a quoted JSON string it removes the quotes and unescapes; otherwise the input is returned unchanged.
JSON_PRETTY(doc)
Re-renders the document with two-space indentation and newlines. If the input cannot be parsed, the original text is returned unchanged.
SELECT JSON_OBJECT('name', 'ada', 'age', '31');
-- {"name":"ada","age":31}    (31 is valid JSON, embedded; "ada" quoted)

SELECT JSON_ARRAY('1', 'two', 'true');
-- [1,"two",true]

SELECT JSON_QUOTE('a"b');     -- "a\"b"
SELECT JSON_UNQUOTE('"a\nb"'); -- a<newline>b
Argument-count limits. JSON_OBJECT accepts exactly one or two key/value pairs (a four-argument call is internally merged into a single object), and JSON_ARRAY accepts one, two, or three elements. These are fixed grammar arities, not the variadic forms found in MySQL/PostgreSQL. Building larger objects or arrays requires composing or assembling the JSON text another way.

13.6. Modification, Containment, and Search

The modification functions return a new document; they never alter stored data in place. Use them inside UPDATE to persist a change.

JSON_SET(doc, path, val)
Inserts the key if missing, or replaces its value if present.
JSON_INSERT(doc, path, val)
Adds the key only if it does not already exist; existing keys are left untouched.
JSON_REPLACE(doc, path, val)
Updates the key only if it already exists; missing keys are not added.
JSON_REMOVE(doc, path)
Deletes the key at path.
JSON_CONTAINS(target, candidate)
Returns 1 if candidate is contained within target: scalar equality, object key/value subset matching, or array-element membership (recursively). Otherwise 0.
JSON_CONTAINS_PATH(doc, mode, path)
Returns 1 if the path exists in doc. mode is 'all' or 'one' (case-insensitive); any value other than 'all' is treated as 'one'.
JSON_SEARCH(doc, mode, str)
Searches string values for an exact match of str and returns the matching path(s) as JSON path strings (e.g. "$.user"). mode 'all' returns every match (comma-separated); any other mode stops at the first match. Returns NULL when nothing matches.
-- payload = {"user":"ada","age":31}
UPDATE events SET payload = JSON_SET(payload, '$.age', '32') WHERE id = 1;
-- {"user":"ada","age":32}

UPDATE events SET payload = JSON_REMOVE(payload, '$.age')   WHERE id = 1;
-- {"user":"ada"}

SELECT JSON_CONTAINS('{"a":1,"b":2}', '{"a":1}');   -- 1
SELECT JSON_CONTAINS_PATH('{"a":1}', 'one', '$.a');  -- 1
SELECT JSON_SEARCH('{"u":"ada","v":"bob"}', 'one', 'ada');  -- "$.u"
Single-key limitation on modification. JSON_SET, JSON_INSERT, JSON_REPLACE, and JSON_REMOVE operate on top-level object keys only. The document must be an object, and the path must be a single $.key step — nested paths ($.a.b) and array indexes ($.a[0]) are rejected. This is a deliberate simplification; MySQL's equivalents accept arbitrary paths and multiple path/value pairs. JSON_SEARCH matches only string values (numbers, booleans, and null are skipped) and requires an exact, full-string match rather than a LIKE pattern.

13.8. Notes and Limitations

Implementation notes.
  • Text storage, not binary. JSON and JSONB are the same text-backed type. There is no decomposed/indexed binary form, so whitespace, key order, and duplicate keys are preserved exactly as written.
  • No JSON indexing. Queries that filter on ->> or JSON_EXTRACT perform a full scan; there are no expression or path indexes over JSON.
  • Fixed buffers. JSON expression evaluation uses fixed-size buffers (documents up to ~4 KB per evaluation step, keys up to 256 bytes, output bounded by the column/result buffer). Very large documents or values may be truncated.
  • MySQL flavour, not PostgreSQL or SQL/JSON. The function names, the $.path syntax, and the JSON_TYPE labels follow MySQL. There is no jsonb, no @>/?/#> operators, no JSON_TABLE, and no SQL/JSON JSON_VALUE/JSON_QUERY constructs.
  • Modification is shallow. Only single top-level object keys can be set/inserted/replaced/removed.
