21. Triggers & Procedures

EvolutionDB stores procedures, functions, and triggers as raw SQL bodies and runs them through a single text-driven execution engine that substitutes parameters (and OLD./NEW. column references), splits the body into control-flow-aware statements, and re-executes each one against the engine.

Implementation note. The procedure language in EvolutionDB is not a compiled PL like PostgreSQL's PL/pgSQL. A body is kept verbatim between BEGIN and the matching END, and at run time it is split on ; and on control-flow keyword boundaries, then interpreted statement by statement. There is no type system for local variables, no plan caching, and no overloading: a procedure or function is identified by its name alone within a schema.

21.1. Overview

Three persistent object kinds are covered here, all stored in the system catalog of the current database and schema:

Procedure
A named, parameterized block of SQL invoked with CALL. Stored with its parameter list and body. OUT/INOUT parameters and local DECLARE variables are supported by the executor.
Function
Created with CREATE FUNCTION … RETURNS type. Functions share the procedure catalog and the same body engine; the declared return type is recorded but functions are stored alongside procedures (the catalog flag is_function distinguishes them).
Trigger
A BEFORE or AFTER block bound to one table and one of the INSERT/UPDATE/DELETE events, firing once per affected row. A trigger body is executed by wrapping it in a temporary procedure and calling it, so it reuses the full procedure engine (IF/WHILE/SIGNAL, etc.).

21.2. CREATE PROCEDURE / FUNCTION

See the reference page CREATE PROCEDURE for the full grammar. The synopsis below reflects exactly what the parser accepts.

CREATE [ OR REPLACE ] PROCEDURE name ( [ param_list ] ) [ AS ]
BEGIN
    body
END

CREATE [ OR REPLACE ] FUNCTION name ( [ param_list ] ) RETURNS return_type [ AS ]
BEGIN
    body
END

-- param_list element:
[ IN | OUT | INOUT ] param_name data_type

Parameter mode defaults to IN when omitted. The accepted return types for functions are a fixed set: INT/INTEGER, VARCHAR, TEXT, BOOLEAN, FLOAT, DOUBLE, BIGINT, DATE, and TIMESTAMP. Up to 16 parameters are stored (MAX_PROC_PARAMS); additional parameters are silently ignored at parse time.

The body is captured as raw text: the engine scans g_lex_input for the first BEGIN keyword and the matching standalone END, tracking nested BEGIN…END depth and skipping END IF/END WHILE/END LOOP/END FOREACH/END CASE. A body is limited to 8192 bytes; the parameter list to 1024 bytes.

CREATE PROCEDURE add_user(IN uname VARCHAR(50), IN uage INT) AS
BEGIN
    INSERT INTO users(name, age) VALUES (uname, uage);
END;

CREATE OR REPLACE FUNCTION double_it(n INT) RETURNS INT AS
BEGIN
    RETURN n * 2;
END;
OR REPLACE semantics. When OR REPLACE is given and a procedure of the same name already exists, the old definition is dropped and re-created. Without OR REPLACE, creating a duplicate name fails with SQLSTATE 42P07 ("procedure/function already exists"). Because lookup is by name only, CREATE FUNCTION and CREATE PROCEDURE share one namespace per schema.

21.3. Parameter and variable substitution

EvolutionDB does not bind parameters as typed values. Instead, before each statement in the body is executed, the engine performs textual substitution: it walks the SQL and replaces occurrences of a parameter name (or $1, $2, … positional markers) with the corresponding argument text. Substitution skips inside single-quoted string literals so a literal containing a parameter name is left untouched.

This means an argument's surrounding quoting matters. A string argument arrives with its quotes intact (e.g. 'alice'), a numeric argument as bare digits, NULL as the keyword NULL, and a boolean as TRUE/FALSE. The substituted text is what the engine then parses, so the called body must be written to expect substituted SQL, not bound values.

Security caveat. Because arguments are spliced into SQL text, a procedure call is effectively string interpolation. Pass only trusted or already-escaped values; there is no separate bind layer that would neutralise injection. This is a deliberate limitation of the text-driven engine, not the parameterized-statement model the SQL standard describes.

21.4. Control flow inside a body

The body splitter (split_proc_statements) first divides on ;, then recursively splits at control-flow keyword boundaries, and a block map (proc_build_block_map) records matching END markers so the interpreter can jump. The following constructs are recognised:

ConstructForm
ConditionalIF cond THEN … [ ELSEIF cond THEN … ] [ ELSE … ] END IF
Pre-tested loopWHILE cond DO … END WHILE
Unconditional loop[ label: ] LOOP … END LOOP
Counted loopFOR var IN start..end DO … END FOR
Query loopFOREACH vars IN (query) DO … END FOREACH
Multi-branchCASE expr WHEN val THEN … [ ELSE … ] END CASE
Loop controlLEAVE [label], ITERATE [label]
Local variableDECLARE var TYPE [DEFAULT value], then SET var = expr
CursorDECLARE c CURSOR FOR query, OPEN c, FETCH c INTO …, CLOSE c
HandlerDECLARE { CONTINUE | EXIT } HANDLER FOR cond …
Return (functions)RETURN expr
Raise errorSIGNAL … (see §21.7)

Loops support an optional leading label: so LEAVE/ITERATE can target an enclosing loop. Up to 32 local variables, 8 cursors, and 8 handlers are tracked per call. A body is split into at most 128 statement units.

CREATE PROCEDURE classify(score INT) AS
BEGIN
    DECLARE grade VARCHAR(2) DEFAULT 'F';
    IF score >= 90 THEN
        SET grade = 'A';
    ELSEIF score >= 70 THEN
        SET grade = 'C';
    END IF;
    INSERT INTO results(score, grade) VALUES (score, grade);
END;

21.5. CALL

A stored procedure is invoked with CALL; see CALL. The parser merely records the call name and arguments — the actual body execution happens in the adaptor layer after parsing returns, to avoid re-entering the parser mutex.

CALL name ( [ arg [, arg ... ] ] )

Arguments may be integer literals, approximate-number literals, string literals, NULL, booleans, bare names, or session variables (@var). The argument count must match the parameter count exactly, otherwise the call fails with SQLSTATE 42883 ("procedure … expects N arguments, got M"). At most 16 arguments are accepted (MAX_CALL_ARGS). A nonexistent procedure name also raises 42883.

CALL add_user('alice', 30);

OUT / INOUT parameters. When a call argument for an INOUT parameter is a session variable (@var), the engine resolves the variable's current value into the body before execution and can write a result back to the caller's variable. This is how a procedure returns scalar data to the calling session.

21.6. CREATE TRIGGER / DROP TRIGGER / ALTER TRIGGER

Full grammar is on CREATE TRIGGER. A trigger binds a body to one table, one timing, and one event, and always fires FOR EACH ROW.

CREATE TRIGGER name { BEFORE | AFTER } { INSERT | UPDATE | DELETE }
    ON table_name FOR EACH ROW
BEGIN
    body
END

DROP TRIGGER [ IF EXISTS ] name

ALTER TRIGGER name { ENABLE | DISABLE }

The FOR EACH ROW clause is mandatory; there is no statement-level (FOR EACH STATEMENT) trigger, and no WHEN condition or column list on the event. The target table must exist, otherwise creation fails with SQLSTATE 42P01; a duplicate trigger name raises 42P07. A trigger is created enabled by default; ALTER TRIGGER … DISABLE flips the catalog enabled flag so the trigger is skipped at firing time without being dropped.

DROP TRIGGER is name-only. Unlike PostgreSQL, the table is not named in DROP TRIGGER / ALTER TRIGGER. EvolutionDB scans every table in the current schema to find a trigger by name. Trigger names are therefore effectively schema-scoped, and the first match wins.

OLD / NEW row references. Inside a trigger body, OLD.col and NEW.col refer to the row before and after the change. Before the body runs, these references are textually substituted with the corresponding column value for the affected row (numeric values are inserted bare; everything else is single-quoted, with embedded quotes doubled; a missing value becomes NULL). Availability follows the event: INSERT populates NEW, DELETE populates OLD, and UPDATE populates both.

BEFORE-trigger field assignment. A BEFORE trigger may modify the row being written by assigning to a column with SET NEW.col = expr. The engine extracts these assignments from the body, evaluates each expression (after OLD/NEW substitution) via a SELECT, and writes the result back into the row before it is stored. Up to 16 such assignments are honoured per trigger.

CREATE TRIGGER stamp_created BEFORE INSERT ON orders FOR EACH ROW
BEGIN
    SET NEW.created_at = NOW();
END;

CREATE TRIGGER audit_del AFTER DELETE ON accounts FOR EACH ROW
BEGIN
    INSERT INTO deleted_accounts(id, name) VALUES (OLD.id, OLD.name);
END;

Memory-store triggers. A trigger may be attached to a memory store with ON MEMORY STORE name. The parser stages the store name with an internal @mem: sentinel; at creation the store is resolved through the catalog and rewritten to its backing table mem_<name>, after which it fires exactly like a regular table trigger. A PUT event on a memory store is mapped to the INSERT ('I') event internally. A nonexistent store raises SQLSTATE 42704.

21.7. Trigger firing model, errors, and recursion

The engine fires matching triggers from the INSERT/UPDATE/DELETE hook points, once per affected row, filtered by timing, event, and the enabled flag. For each fired trigger the substituted body is wrapped as a temporary procedure, called, then dropped — so triggers and procedures execute through the identical engine:

-- conceptual expansion performed internally per fired trigger
CREATE OR REPLACE PROCEDURE __trig_<id>__() AS BEGIN <substituted body> END;
CALL __trig_<id>__();
DROP PROCEDURE IF EXISTS __trig_<id>__;

Recursion limit. Trigger execution is guarded by a depth counter; nesting beyond 16 levels (MAX_TRIGGER_DEPTH) stops further trigger firing silently rather than erroring, preventing runaway recursion.

Raising errors with SIGNAL. A body — procedure or trigger — can abort with a user-defined error:

SIGNAL [ SQLSTATE 'xxxxx' ] [ SET MESSAGE_TEXT = 'message' ]

If SQLSTATE is omitted it defaults to 45000 (unhandled user-defined exception); the default message is "Unhandled user-defined exception". A DECLARE … HANDLER FOR SQLEXCEPTION can intercept the signal — a CONTINUE handler resumes after the failed statement, an EXIT handler unwinds the body — otherwise the error propagates to the caller. When a trigger body errors, the whole firing fails and the originating DML statement is aborted.

CREATE TRIGGER no_negatives BEFORE INSERT ON ledger FOR EACH ROW
BEGIN
    IF NEW.amount < 0 THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'amount must be non-negative';
    END IF;
END;

21.8. SQLSTATE reference

Error codes raised by the trigger and procedure paths:

42P01
Trigger target table does not exist.
42P07
Duplicate object: procedure/function already exists (without OR REPLACE), or trigger name already exists.
42883
Procedure/function does not exist (on CALL or DROP), trigger does not exist (on DROP without IF EXISTS), or argument-count mismatch on CALL.
42704
Memory store named in ON MEMORY STORE does not exist.
45000
Default SQLSTATE for an unqualified SIGNAL (user-defined exception).
53200
Out of memory while preparing a CALL body for execution.

21.9. Notes & limitations

What EvolutionDB does not implement (and how it differs from PostgreSQL / the SQL standard):
  • No row-vs-statement choice. Triggers are always FOR EACH ROW; there is no statement-level trigger.
  • No WHEN condition, column list, or INSTEAD OF. The event is a single bare INSERT/UPDATE/DELETE with no filter; views cannot carry triggers.
  • One event per trigger. There is no INSERT OR UPDATE multi-event form; create separate triggers.
  • Table is not named on DROP/ALTER TRIGGER — names are resolved by scanning the schema, so trigger names should be unique per schema.
  • Textual parameter substitution, not value binding. Arguments and OLD/NEW references are spliced into SQL text; there is no typed parameter binding and no protection against injection.
  • No procedure/function overloading. Lookup is by name within a schema; functions and procedures share that namespace.
  • Functions are not callable inline. A CREATE FUNCTION body is run through the same engine as a procedure; the return type is recorded but the database does not expose user functions as expression-callable in SELECT the way PostgreSQL does.
  • Fixed capacities. Body ≤ 8192 bytes; ≤ 16 parameters/arguments; ≤ 128 statement units; ≤ 32 local variables, ≤ 8 cursors, ≤ 8 handlers per call; trigger recursion depth ≤ 16.
  • No trigger ordering guarantees by name, and no transition tables (REFERENCING … OLD/NEW TABLE).
