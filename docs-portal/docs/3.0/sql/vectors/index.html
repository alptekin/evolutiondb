14. Vectors & Hybrid Search

EvolutionDB stores fixed-dimension dense embeddings in VECTOR(N) columns, ranks them with four built-in distance metrics, accelerates nearest-neighbor queries with an in-memory HNSW index, and combines an equality filter with vector ranking through a single hybrid-search call.

14.1. Overview

Vector support in EvolutionDB is built from four cooperating pieces:

  • The VECTOR(N) data type, a fixed-dimension array of single-precision floats stored inline in the binary tuple.
  • Distance functions and operators, COSINE_DISTANCE, L2_DISTANCE, INNER_PRODUCT, L1_DISTANCE, plus the <-> and <#> operators.
  • The HNSW index, created with CREATE INDEX … USING HNSW and queried through the HNSW_KNN function or a cosine ORDER BY.
  • Hybrid search, HNSW_FILTER_KNN, which folds a single equality predicate into vector ranking with automatic pre-/post-filter strategy selection.

Relationship to PostgreSQL. The surface syntax deliberately echoes the pgvector extension, VECTOR(N), USING HNSW, the vector_cosine_ops opclass family, the WITH (m=…, ef_construction=…) parameters, and the <-> / <#> operators are all recognized. The implementation, however, is EvolutionDB's own: a brute-force-backed in-memory graph (see Notes). EvolutionDB is not pgvector and does not promise the same recall, persistence, or operator coverage. This chapter documents only what the engine actually implements.

14.2. The VECTOR(N) Type

A VECTOR(N) column holds exactly N single-precision (float4) components. The dimension is declared at CREATE TABLE time and is part of the column's type; rows are not allowed to carry vectors of a different length.

VECTOR ( dimension )

Internally the dimension is folded into the type code as 260000 + N (type family 26). For example VECTOR(3) is encoded as 260003 and VECTOR(1536) as 261536. The payload is stored inline in the tuple as N consecutive 4-byte little-endian floats with no length prefix: the dimension is recovered from the column's type code, not from the stored bytes.

Storage
N × 4 bytes, float4 little-endian, stored inline in the binary tuple. A VECTOR(1536) value occupies 6144 bytes of column data.
Maximum dimension
16384 (VECTOR_MAX_DIM). Declaring or inserting beyond this is rejected.
Element type
32-bit IEEE float. Values are accepted as decimal text and converted to float4 on insert.
NULL
A vector column may be NULL like any other column; the null bitmap marks it and no payload bytes are written.

14.2.1. Vector literals

A vector value is supplied as a string literal in one of three accepted forms:

[f1,f2,…]
Bracketed list of decimal floats. This is the canonical text form and the one used when a vector is rendered back out.
{f1,f2,…}
Brace-delimited list; accepted as an equivalent input form.
b64i8:<base64>
A compact quantized form: a little-endian float32 scale factor followed by an int8[] array, base64-encoded. On decode each byte is dequantized as byte × (scale / 127). This keeps a 1024-dimension query literal around 1.4 KB, comfortably under the statement-size cap, and is intended for machine-generated queries.

Whitespace inside a list is ignored; an embedded NaN is rejected as a parse error.

-- Declare and populate a table of embeddings
CREATE TABLE docs (
    id    INTEGER PRIMARY KEY,
    title VARCHAR(200),
    topic VARCHAR(40),
    emb   VECTOR(3)
);

INSERT INTO docs (id, title, topic, emb) VALUES
  (1, 'intro to storage',  'storage', '[0.10, 0.20, 0.97]'),
  (2, 'btree internals',   'storage', '[0.11, 0.19, 0.98]'),
  (3, 'wire protocol',     'protocol','[0.90, 0.05, 0.10]');

See CREATE TABLE for the full column-definition grammar and INSERT for value syntax.

14.3. Distance Functions and Operators

Four scalar distance functions are available in any expression position (projection, WHERE, ORDER BY). Each takes two vectors of equal length and returns a DOUBLE distance, smaller is more similar.

FunctionOperatorReturnsDefinition
COSINE_DISTANCE(a, b)<=> (in ORDER BY only)double1 − (a·b) / (‖a‖·‖b‖). Returns 1.0 if either vector is all-zero.
L2_DISTANCE(a, b)<->doubleEuclidean distance, √Σ(aᵢ−bᵢ)².
INNER_PRODUCT(a, b)<#>doubleNegated dot product, −Σ(aᵢ·bᵢ). Negated so that "more similar" sorts first, consistent with the other metrics.
L1_DISTANCE(a, b), doubleManhattan distance, Σ|aᵢ−bᵢ|.

Three vector utility functions round out the set:

VECTOR_DIM(v)
Number of components in v.
VECTOR_NORM(v)
The L2 norm (magnitude) of v.
VECTOR_NORMALIZE(v)
The unit vector in the direction of v.
-- Exact (un-indexed) nearest neighbors via a distance function in ORDER BY
SELECT id, title,
       L2_DISTANCE(emb, '[0.10, 0.20, 0.97]') AS dist
FROM   docs
ORDER BY dist ASC
LIMIT 5;

-- The L2 operator form is equivalent to L2_DISTANCE()
SELECT id, emb <-> '[0.10, 0.20, 0.97]' AS dist FROM docs ORDER BY dist;

Operator availability differs by metric. Only two infix operators exist: <-> (L2) and <#> (inner product), usable anywhere an expression is allowed. Cosine has no general infix operator, the <=> spelling is accepted only in the ORDER BY position. For cosine elsewhere, call COSINE_DISTANCE(). There is no L1 operator at all; use L1_DISTANCE(). This is narrower than pgvector, which exposes a full operator set per metric.

14.3.1. Cosine ORDER BY

For convenience, a column may be ranked by cosine distance against a query literal directly in ORDER BY using the <=> operator:

ORDER BY column <=> '[f1,f2,…]' [ ASC | DESC ]
SELECT id, title
FROM   docs
ORDER BY emb <=> '[0.10, 0.20, 0.97]' ASC
LIMIT 3;

Only the cosine sub-operator is honored in this position. Any other comparison operator (=, <, >, …) used with a string literal in ORDER BY is rejected by the grammar with an explicit error, rather than being silently ignored.

14.4. The HNSW Index

For larger collections, an HNSW (Hierarchical Navigable Small World) index provides nearest-neighbor lookup without scanning every row at query time. It is created with the USING HNSW access method on a single VECTOR(N) column.

CREATE INDEX [ IF NOT EXISTS ] name
    ON table USING HNSW ( column [ opclass ] )
    [ WITH ( m = int [, ef_construction = int ] ) ]

14.4.1. Operator classes (metric selection)

The optional opclass names the distance metric the index ranks by. It defaults to cosine when omitted. An unrecognized opclass name falls back to cosine rather than raising an error.

OpclassMetricInternal kind
vector_cosine_ops (default)cosine distanceHNSW_DIST_COSINE (0)
vector_l2_opsL2 / EuclideanHNSW_DIST_L2 (1)
vector_ip_ops / vector_inner_opsinner productHNSW_DIST_INNER (2)
vector_l1_opsL1 / ManhattanHNSW_DIST_L1 (3)

14.4.2. Build parameters

m
Maximum neighbor degree per node above layer 0 (layer 0 nodes get up to 2·m neighbors). Defaults to 16. Larger m increases graph connectivity and recall at the cost of memory and build time.
ef_construction
Beam width used while inserting nodes during the build. Defaults to 64. Larger values build a higher-quality graph at the cost of build time. (Note: a bare CREATE INDEX … USING HNSW with no WITH clause uses ef_construction = 64; the brute-force build helper's standalone default of 200 does not apply through DDL.)

Query-time beam width (ef_search) is fixed at 64 in this release and is not yet user-tunable per query. If the requested k exceeds ef_search, the beam is widened to k for that query.

-- Cosine HNSW index with default m=16, ef_construction=64
CREATE INDEX docs_emb_idx ON docs USING HNSW (emb);

-- L2 metric with explicit build parameters
CREATE INDEX docs_emb_l2 ON docs
    USING HNSW (emb vector_l2_ops)
    WITH (m = 32, ef_construction = 200);

See CREATE INDEX and DROP INDEX for the general index grammar.

14.4.3. How the index is built and stored

At CREATE INDEX time the engine scans the table, decodes the vector column from every live row into a dense in-memory store, then builds a multi-layer navigable graph over that store. Each indexed table/column is identified by (table_id, index_name). The graph and its backing vectors live entirely in process memory, there is currently no on-disk persistence of the graph itself.

Search uses the graph traversal (a sub-linear greedy beam walk) when the graph built successfully and the collection is large enough to benefit. For very small collections (at or below 2·ef_search rows) or if graph construction hit an allocation failure, the engine transparently falls back to an exact brute-force top-k scan over the dense store. The brute-force path is exact by construction; the graph path is approximate.

14.4.4. Querying with HNSW_KNN

Nearest-neighbor lookups against the index are issued through the HNSW_KNN table function:

HNSW_KNN ( table_name, index_name, query_vector, k )
table_name, index_name
The table and the HNSW index to search.
query_vector
A vector literal in any accepted form ([…], {…}, or b64i8:). Its dimension must match the indexed column; a mismatch returns no result.
k
An integer literal: the number of nearest neighbors to return, ordered by ascending distance.

Each result row carries the matching row's primary-key value and its distance to the query vector.

SELECT * FROM HNSW_KNN(
    'docs', 'docs_emb_idx',
    '[0.10, 0.20, 0.97]', 5
);

The index is not auto-maintained or rebuilt. An HNSW graph is a point-in-time snapshot taken at CREATE INDEX. Rows inserted, updated, or deleted afterward are not reflected until the index is rebuilt. Existing HNSW indexes are also not reloaded at server startup in this release. To refresh the graph, re-issue CREATE INDEX (or REINDEX when available). This differs sharply from PostgreSQL indexes, which are transactionally maintained on every write.

14.5. Hybrid (Lexical + Vector) Search

Hybrid search combines a single equality predicate on an ordinary column with vector ranking, in one call. This is the common "filter then rank by similarity" pattern: for example, "find the documents most similar to this embedding, but only among rows whose topic = 'storage'."

HNSW_FILTER_KNN ( table_name, index_name, query_vector, k,
                  'column=value[:mode]' )

The fifth argument is a string literal of the form column=value, with an optional :mode suffix selecting the execution strategy. An empty or absent filter degrades the call to a plain HNSW_KNN.

auto (default)
The engine measures filter selectivity (matching_rows / total_rows) and chooses pre-filter when selectivity is below 10%, otherwise post-filter.
pre
Force pre-filter: restrict to rows where column = value first, then compute distances and take top-k. Best when the filter is tight (few matches).
post
Force post-filter: rank all vectors first, then discard hits whose column value doesn't match, oversampling to compensate. Best when the filter is loose (matches most rows).
-- Most-similar docs, restricted to topic = 'storage', auto strategy
SELECT * FROM HNSW_FILTER_KNN(
    'docs', 'docs_emb_idx',
    '[0.10, 0.20, 0.97]', 5,
    'topic=storage'
);

-- Force the pre-filter strategy
SELECT * FROM HNSW_FILTER_KNN(
    'docs', 'docs_emb_idx',
    '[0.10, 0.20, 0.97]', 5,
    'topic=storage:pre'
);

The filter compares against the column's rendered text value; NULL column values never match. The predicate is a single equality only, ranges, OR, and multi-column predicates are not part of the hybrid call. For richer filtering, combine plain HNSW_KNN with a surrounding SELECT WHERE clause.

14.5.1. Inspecting the chosen strategy

A diagnostic sibling, HNSW_HYBRID_EXPLAIN, takes the same arguments as HNSW_FILTER_KNN but reports which strategy the planner picked along with the selectivity statistics it used: the candidate (total scanned) count, the number of filter matches, and the measured selectivity.

HNSW_HYBRID_EXPLAIN ( table_name, index_name, query_vector, k,
                      'column=value[:mode]' )
SELECT HNSW_HYBRID_EXPLAIN(
    'docs', 'docs_emb_idx',
    '[0.10, 0.20, 0.97]', 5,
    'topic=storage'
);

14.7. Notes and limitations

  • In-memory only. The HNSW graph and its dense vector store live in process memory. There is no on-disk persistence of the graph, and indexes are not reloaded on startup, rebuild with CREATE INDEX after a restart.
  • Snapshot semantics. The graph reflects the table only as of build time; subsequent DML is not auto-maintained into the index. Re-create the index to pick up changes.
  • Approximate vs. exact. The graph path is an approximate nearest-neighbor search; small collections (≤ 2·ef_search rows) and build-failure fallbacks use an exact brute-force scan instead.
  • Fixed query beam width. ef_search is fixed at 64 (widened to k when k is larger). There is no per-query SET ef_search yet.
  • One column per HNSW index. An HNSW index covers a single VECTOR(N) column.
  • Single equality in hybrid search. HNSW_FILTER_KNN supports exactly one column = value predicate; it is not a general WHERE engine.
  • Bounded internal capacity. The active HNSW graph registry holds a fixed number of graphs (64) and primary-key keys up to 256 bytes; very long composite keys are truncated for index bookkeeping.
  • Operator coverage. Only <-> (L2) and <#> (inner) are general operators; cosine via <=> is limited to ORDER BY; L1 has no operator. Unlike PostgreSQL/pgvector, EvolutionDB does not implement a full per-metric operator family or index-condition pushdown for arbitrary distance operators.
