7. Queries

This page shows how EvolutionDB turns a SELECT into rows. It covers projection, filtering with WHERE, joins, subqueries, grouping, aggregates, and finally ORDER BY and LIMIT. It describes how the engine really works.

This chapter describes the read path. The full grammar and a clause-by-clause reference live on the SELECT command page. Here we focus on behavior. We show what each stage computes, in what order it runs, and where EvolutionDB differs from PostgreSQL or the SQL standard. Everything below matches the engine source. The main files are evolution/db/Select.c, evolution/db/expression.c, and the adaptor's join pipeline in adaptor/join.c.

7.1. Overview & Processing Order

A table query runs stage by stage. These are the stages, in order.

  1. FROM / join. The leftmost source is the driving relation. Each next source is joined onto the row so far. Rows are produced one at a time. See §7.3.
  2. WHERE. The boolean condition runs against each candidate row. Rows that do not come out true are dropped. See §7.2.
  3. GROUP BY / aggregates. The rows that are left are collapsed into groups. Aggregate functions are computed for each group. See §7.5.
  4. HAVING. A boolean condition filters whole groups.
  5. Projection. The select list runs to build the output rows. Then DISTINCT removes duplicates.
  6. Window functions. These run after grouping and before sorting.
  7. ORDER BY. Rows are sorted. See §7.6.
  8. LIMIT / OFFSET. The number of returned rows is limited.

Inside the core engine there are two physical shapes. A single-table query finds the table in the catalog. Then it does one of two things. It does a primary-key lookup when the WHERE is a single equality on the PK. Otherwise it does a sequential table scan through the buffer pool and decodes each binary tuple as it goes. A multi-table query goes to the join pipeline. Both paths decode rows with tup_extract_fields() and stream them. The engine does not build a separate query-plan tree. It has no cost-based optimizer.

Note. EvolutionDB runs expressions on the text form of each column value. This is the same form the wire protocol returns. Comparisons try a numeric reading first. If that fails, they fall back to byte-wise string comparison. There is no separate typed expression engine. This keeps the code small. It also means a few behaviors differ from a strictly typed planner. They are called out in each section below.

7.1.1. Projections (the select list)

Each select-list item is an expression node. Its type is ExprNode. The parser builds it. The following forms are supported.

  • * projects every column of the source.
  • Plain column references. These are col or qualified alias.col. Qualified names resolve by exact match first. If that fails, the engine drops the qualifier and matches the bare column name. See expr_evaluate, case EXPR_COLUMN.
  • Literals. These are integer, float, string, boolean, and NULL.
  • Arithmetic and bitwise operators. These are + - * / %, << >>, and & | ^.
  • String functions. These are SUBSTRING, TRIM, UPPER, LOWER, LENGTH, CONCAT, REPLACE, and COALESCE.
  • CASE in both searched and simple forms. The simple form is rewritten into equality comparisons.
  • JSON accessors -> and ->>, plus the JSON_* function family.
  • Aggregate and window functions. See §7.5.

You can rename an item with AS alias. The AS keyword is optional. The alias is stored as the node's display name through expr_store_select(). The same name is used to find the value again in HAVING and in window evaluation.

SELECT name, salary * 12 AS annual, UPPER(dept) FROM employees;
Note. Integer division uses floating-point math inside. Division or modulo by zero returns 0 and does not raise an error. See expr_evaluate_num, cases EXPR_DIV and EXPR_MOD. PostgreSQL would raise division_by_zero. In the scan path, field buffers are fixed at 256 bytes per column. Very long projected values are cut off.

7.2. The WHERE Clause

The WHERE condition is an expression tree. expr_evaluate() runs it for each row. A row is kept only when the result is the boolean true string "t". The join pipeline drops the row when the result is "0", "false", "f", or NULL. A NULL here means the value could not be evaluated. See the end-of-pipeline test in pipeline_recurse.

Supported predicates

Comparisons
=, <>, <, >, <=, >=. Each side is turned into a string. If both sides parse fully and cleanly as numbers, the comparison is numeric. Otherwise it is a strcmp string comparison. See expr_evaluate and EXPR_CMP_*.
Logical
AND, OR, NOT, XOR. NULL operands use SQL three-valued logic. For example, TRUE AND NULL → NULL, and FALSE AND NULL → FALSE.
NULL tests
IS NULL and IS NOT NULL. A column counts as NULL when it is missing from the row or carries the internal null marker.
Pattern match
LIKE and NOT LIKE. Use % for any sequence and _ for any single character. A small in-engine wildcard matcher does the work.
Range / membership
BETWEEN low AND high is rewritten to x >= low AND x <= high. NOT BETWEEN becomes x < low OR x > high. IN (list) and NOT IN (list) are rewritten into a balanced OR/AND tree. This keeps long lists from overflowing the recursive evaluator.
Subqueries
IN (SELECT ...), NOT IN (SELECT ...), EXISTS and NOT EXISTS, and the scalar form (SELECT ...). See §7.4.
Full-text & vector
MATCH(col) AGAINST('terms') matches on any term token and returns a hit count. Vector-distance operators are also supported.

A comparison yields NULL when either side is NULL. NULL is not the same as false. This follows SQL rules, so such a row is not returned.

SELECT * FROM orders
WHERE status = 'open'
  AND total BETWEEN 100 AND 500
  AND note LIKE '%urgent%';

PK lookup & push-down

For a single table, the engine looks for a WHERE that is a lone equality on the primary-key column. A simple comparison on the PK also counts. expr_is_simple_pk_range() detects this case. It is served by a direct B+ tree lookup with bt2_search instead of a full scan. The join pipeline uses the same idea for first-table push-down. An alias.col = literal term that targets the driving table is pulled out by extract_eq_for_alias. It is used as a PK lookup or as an early per-row column filter. This runs before the rest of the WHERE tree is applied to the full merged row.

Note. The choice between numeric and string comparison is made per value at run time. It does not depend on the declared column type. Two operands that both look numeric are compared as numbers, even if the column is text. Mixed cases fall back to strcmp. There is no implicit collation beyond byte order. String comparison is case-sensitive. Column-name resolution, on the other hand, is case-insensitive.

7.3. Joins

Multi-table queries run through the iterator pipeline engine in adaptor/join.c. It uses Volcano-style recursion in pipeline_recurse. No table is fully loaded into memory. Each table gives one row at a time. The merged row grows depth-first until the deepest level emits a complete joined tuple. Memory use is O(number of tables), not O(table size).

Join types

The parser turns each join type into a small integer. The pipeline reads it back.

SyntaxEncoded codeBehavior
JOIN (plain)100Inner join.
INNER JOIN101Inner join.
CROSS JOIN102Cartesian product (no key filter).
STRAIGHT_JOIN200Inner join (MySQL ordering hint; treated as inner).
LEFT [OUTER] JOIN301 / 305Left outer: emit NULL-filled right side when no match.
RIGHT [OUTER] JOIN302 / 306Parsed and accepted; see note below.
NATURAL [LEFT|RIGHT] JOIN400+Parsed and accepted; see note below.

How a join row is produced

Before the pipeline starts, the engine reads each ON condition with extract_eq_columns(). It finds the equality left.col = right.col. It maps this to two keys. The left key is a column index in the row built so far. The right key is a column in the table being joined. For each table after the first, one of the following happens.

  • If the right key is the joined table's primary key, the match is found with an O(log n) B+ tree lookup. The function is bt2_search.
  • If not, the joined table is scanned. Rows are kept where the right-key column equals the left key value. This is a nested-loop scan.
  • If there is no usable key, the joined table is scanned with no filter. CROSS JOIN is one example. This is a full Cartesian expansion.

For a LEFT join with codes 301 or 305, the right side may have no matching row. In that case its columns are filled with NULLs and the merged row is still emitted. Output columns are named alias.column when the source has an alias. MVCC visibility is checked for each scanned row. So joins see the same snapshot as the rest of the statement.

SELECT e.name, d.dept_name
FROM employees AS e
INNER JOIN departments AS d ON e.dept_id = d.id;

LATERAL and UNNEST sources

A LATERAL ( subquery ) source runs again once per outer row. The merged outer row is published into the query context. This lets correlated column references resolve against it. The inner statement is run with query_execute. Each result row is appended. See pipeline_recurse_lateral. A UNNEST(array_expr) source expands an array into one row per element. Both fit in as join slots and follow LEFT rules. A lateral or unnest source that yields no rows still emits NULLs under a left join.

Note on join limits. Only an equality ON condition between two columns is used for key-based matching. A non-equi ON or one with expressions falls back to a full scan of the joined table. The broader WHERE tree still filters the final merged row. RIGHT and NATURAL joins are parsed. But the NULL-fill branch in pipeline_recurse only runs for the LEFT codes 301 and 305. Do not rely on right-outer NULL extension. Do not rely on NATURAL deriving the join key on its own the way PostgreSQL does. FULL OUTER JOIN is not built as a NULL-extending join. Use an explicit LEFT JOIN ... ON for portable outer-join behavior.

7.4. Subqueries

Subqueries are stored as raw SQL text on the expression node. The field is subquery_sql. They are run through a callback g_qctx->subquery_fn. The callback runs the inner statement and returns its values. expr_evaluate handles the following forms.

Scalar subquery, (SELECT ...)
Run and reduced to a single value. You can use it anywhere a scalar expression is allowed.
expr IN (SELECT ...) and NOT IN
The subquery's column is loaded into memory, up to 256 values. Then it is tested for membership.
EXISTS (SELECT ...) and NOT EXISTS
The result depends on whether the subquery returns any row.

Correlated subqueries use the same outer-scope mechanism as LATERAL. A column reference inside the subquery that is not local resolves against the published outer row.

SELECT name FROM employees e
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.emp_id = e.id
);

7.5. Grouping & Aggregates

The executor switches to the grouping path in two cases. The first is when the select list has an aggregate function. The second is when the query has a GROUP BY. Rows are then put into buckets. With GROUP BY, each row's grouping-key value places it in a group. With no GROUP BY but an aggregate present, all rows form one group. Aggregates are computed by walking the rows of each group again.

Aggregate functions

COUNT(*)
Number of rows in the group.
COUNT(expr)
Number of rows where expr is not NULL.
SUM(expr) / AVG(expr)
The total or mean over the non-NULL values. A group with only NULL values returns NULL. AVG shows an integer when the mean is whole. Otherwise it shows four decimal places.
MIN(expr) / MAX(expr)
The lowest or highest value. The comparison is numeric when every value parses as a number. Otherwise it is a string comparison.
GROUP_CONCAT(expr)
Joins the non-NULL values of the group with a separator. The default separator is ,. This works the MySQL way.

Non-aggregate select-list expressions in a grouped query run against the first row of each group. A HAVING condition then runs for each group. Aggregates used in HAVING are matched by display name to the computed select-list aggregates. Any extra aggregates that appear only in HAVING are computed on the fly.

SELECT dept_id, COUNT(*) AS headcount, AVG(salary) AS avg_pay
FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 3;
Note. SQL says every non-aggregated select-list column must appear in GROUP BY. EvolutionDB does not enforce this rule. An ungrouped column just takes its value from the group's first row. This is the permissive MySQL way. PostgreSQL has a strict check instead. COUNT(DISTINCT ...) and other DISTINCT aggregate arguments are not part of this aggregate path.

7.6. ORDER BY, DISTINCT & LIMIT

DISTINCT

SELECT DISTINCT removes duplicate result rows after projection. DISTINCTROW works as a synonym. Duplicate removal compares the rendered output rows.

ORDER BY

Rows are sorted with qsort over the result set. Each sort key may be a column name, a qualified table.column, or a 1-based column position like ORDER BY 2. Each key may end with ASC (the default) or DESC. The comparison is adaptive. When both values parse fully and cleanly as numbers, the order is numeric. Otherwise it is by string. See CompareRecords in Select.c. DESC just flips the comparison result.

There is a special form, ORDER BY vector_col <=> 'literal'. It ranks rows by cosine distance for nearest-neighbor search. The executor can take an approximate-nearest-neighbor fast path. This needs a ready HNSW index on the column and a LIMIT on the query. It also needs no disqualifying clause. Disqualifying clauses are DISTINCT, OFFSET, a join, or extra sort keys. If the fast path does not apply, the engine brute-force sorts by distance.

SELECT id FROM documents
ORDER BY embedding <=> '[0.12, 0.84, 0.05]'
LIMIT 5;

LIMIT / OFFSET

Three forms are accepted. The first is LIMIT count. The second is the MySQL two-argument form LIMIT offset, count. The third is LIMIT count OFFSET offset. LIMIT and OFFSET are applied last. They run after ORDER BY, DISTINCT, and grouping. In the streaming single-table path, OFFSET skips the first N matching rows. LIMIT stops the scan once N rows have been sent.

SELECT name FROM employees
ORDER BY name
LIMIT 20, 10;   -- skip 20, return 10
Note. The engine's built-in text renderer holds rows in fixed-size storage for in-memory ORDER BY sorting. In the SELECT * path this is up to a few hundred rows. Large result sets are sorted and streamed by the adaptor's query executor when clients connect through the server. There is no standalone OFFSET clause without LIMIT. The grammar also has no FETCH FIRST, no set operations UNION, INTERSECT, and EXCEPT, and no CTEs with WITH.

7.7. Common Errors

42P01, relation does not exist
A table named in FROM or a join was not found in the catalog. tapi_resolve failed.
53000, out of memory
The join pipeline could not allocate its per-table or merged-row buffers.
