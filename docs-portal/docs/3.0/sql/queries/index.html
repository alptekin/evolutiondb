7. Queries

How EvolutionDB turns a SELECT into rows, projection, filtering with WHERE, joins, subqueries, grouping and aggregates, and finally ORDER BY / LIMIT, as the engine actually implements it.

This chapter describes the read path. The full grammar and clause-by-clause reference live on the SELECT command page; here we focus on behavior: what each stage computes, in what order, and where EvolutionDB diverges from PostgreSQL or the SQL standard. Everything below reflects the engine source, evolution/db/Select.c, evolution/db/expression.c, and the adaptor's join pipeline in adaptor/join.c.

7.1. Overview & Processing Order

A table query is evaluated stage by stage. Logically, the stages run in this sequence:

  1. FROM / join. The leftmost source is the driving relation; each subsequent source is joined onto the accumulated row (rows are produced one at a time, see §7.3).
  2. WHERE. The boolean condition is evaluated against each candidate row; rows that do not evaluate to true are dropped (§7.2).
  3. GROUP BY / aggregates. Surviving rows are collapsed into groups and aggregate functions are computed per group (§7.5).
  4. HAVING. A boolean condition filters whole groups.
  5. Projection. The select list is evaluated to form output rows; DISTINCT then removes duplicates.
  6. Window functions. Post-processed after grouping and before sorting.
  7. ORDER BY. Rows are sorted (§7.6).
  8. LIMIT / OFFSET. The returned row count is restricted.

Inside the core engine there are two physical shapes. A single-table query resolves the table from the catalog and either does a primary-key lookup (when the WHERE is a single equality on the PK) or a sequential table scan through the buffer pool, decoding each binary tuple as it goes. A multi-table query is handed to the join pipeline. Both paths decode rows with tup_extract_fields() and stream them; the engine does not build a separate query-plan tree or cost-based optimizer.

Note. EvolutionDB evaluates expressions on the text form of each column value (the same rendering the wire protocol returns). Comparisons try a numeric interpretation first and fall back to byte-wise string comparison; there is no separate typed expression engine. This keeps the implementation small but means a few behaviors differ from a strictly typed planner, they are called out per section below.

7.1.1. Projections (the select list)

Each select-list item is an expression node (ExprNode) built by the parser. Supported forms include:

  • *, projects every column of the source.
  • Plain column references (col or qualified alias.col). Qualified names resolve by exact match first, then by stripping the qualifier and matching the bare column name (expr_evaluate, case EXPR_COLUMN).
  • Literals: integer, float, string, boolean, NULL.
  • Arithmetic and bitwise operators: + - * / %, << >>, & | ^.
  • String functions: SUBSTRING, TRIM, UPPER, LOWER, LENGTH, CONCAT, REPLACE, COALESCE.
  • CASE (both searched and simple forms, the simple form is desugared to equality comparisons).
  • JSON accessors (->, ->>) and the JSON_* function family.
  • Aggregate and window functions (see §7.5).

An item may be renamed with AS alias (the AS keyword is optional). The alias is stored into the node's display name via expr_store_select(), which is also the name used to look the value back up in HAVING and window evaluation.

SELECT name, salary * 12 AS annual, UPPER(dept) FROM employees;
Note. Integer division uses floating-point math internally and division/modulo by zero return 0 rather than raising an error (see expr_evaluate_num, cases EXPR_DIV / EXPR_MOD), PostgreSQL would raise division_by_zero. Field rendering buffers are fixed at 256 bytes per column in the scan path, so extremely long projected values are truncated.

7.2. The WHERE Clause

The WHERE condition is an expression tree evaluated per row by expr_evaluate(). A row is kept only when the result is the boolean true string ("t"); the join pipeline treats "0", "false", "f", and a NULL (unevaluable) result as "discard" (see the end-of-pipeline test in pipeline_recurse).

Supported predicates

Comparisons
=, <>, <, >, <=, >=. Each side is evaluated to a string; if both sides parse cleanly and fully as numbers, the comparison is numeric, otherwise it is strcmp-lexical (expr_evaluate, EXPR_CMP_*).
Logical
AND, OR, NOT, XOR, with SQL three-valued logic for NULL operands (e.g. TRUE AND NULL → NULL, FALSE AND NULL → FALSE).
NULL tests
IS NULL / IS NOT NULL. A column is treated as NULL when it is absent from the row or carries the internal null marker.
Pattern match
LIKE / NOT LIKE with % (any sequence) and _ (any single character), matched by a small in-engine wildcard matcher.
Range / membership
BETWEEN low AND high (desugared to x >= low AND x <= high), NOT BETWEEN (→ x < low OR x > high), and IN (list) / NOT IN (list) (desugared into a balanced OR/AND tree so long lists do not overflow the recursive evaluator).
Subqueries
IN (SELECT ...), NOT IN (SELECT ...), EXISTS / NOT EXISTS, and scalar (SELECT ...), see §7.4.
Full-text & vector
MATCH(col) AGAINST('terms') (any-term token match, returning a hit count), and vector-distance operators.

A comparison where either side evaluates to NULL yields NULL (not false), matching SQL semantics, so such a row is not returned.

SELECT * FROM orders
WHERE status = 'open'
  AND total BETWEEN 100 AND 500
  AND note LIKE '%urgent%';

PK lookup & push-down

For a single table, a WHERE that is a lone equality (or simple comparison) on the primary-key column is detected by expr_is_simple_pk_range() and serviced by a direct B+ tree lookup (bt2_search) instead of a full scan. In the join pipeline the same idea drives first-table push-down: an alias.col = literal conjunct that targets the driving table is extracted (extract_eq_for_alias) and used either as a PK lookup or as an early per-row column filter, before the rest of the WHERE tree is applied to the fully merged row.

Note. Numeric-vs-lexical comparison is decided per value at evaluation time, not by declared column type. Two operands that both happen to look numeric are compared numerically even if the column is textual; mixed cases fall back to strcmp. There is no implicit collation beyond byte order, and string comparison is case-sensitive (column-name resolution, by contrast, is case-insensitive).

7.3. Joins

Multi-table queries run through the iterator/pipeline engine in adaptor/join.c. It is a Volcano-style recursion (pipeline_recurse): no table is fully materialized; each table contributes one row at a time, and the merged row is extended depth-first until the deepest level emits a complete joined tuple. Memory is O(number of tables), not O(table size).

Join types

The parser encodes each join type as a small integer the pipeline reads back:

SyntaxEncoded codeBehavior
JOIN (plain)100Inner join.
INNER JOIN101Inner join.
CROSS JOIN102Cartesian product (no key filter).
STRAIGHT_JOIN200Inner join (MySQL ordering hint; treated as inner).
LEFT [OUTER] JOIN301 / 305Left outer: emit NULL-filled right side when no match.
RIGHT [OUTER] JOIN302 / 306Parsed and accepted; see note below.
NATURAL [LEFT|RIGHT] JOIN400+Parsed and accepted; see note below.

How a join row is produced

Before the pipeline starts, the engine analyzes each ON condition with extract_eq_columns() to find the equality left.col = right.col and maps it to a left key (a column index in the already-merged row) and a right key (a column in the table being joined). Then, for each table beyond the first:

  • If the right key is the joined table's primary key, the match is found with an O(log n) B+ tree lookup (bt2_search).
  • Otherwise the joined table is scanned and rows are kept where the right-key column equals the left key value (a nested-loop scan).
  • With no usable key (e.g. CROSS JOIN), the joined table is scanned with no filter, a full Cartesian expansion.

For a LEFT join (codes 301/305), if no inner row matches, the right side's columns are filled with NULLs and the merged row is still emitted. Output columns are named alias.column when the source carries an alias. MVCC visibility is checked per scanned row, so joins observe the same snapshot as the rest of the statement.

SELECT e.name, d.dept_name
FROM employees AS e
INNER JOIN departments AS d ON e.dept_id = d.id;

LATERAL and UNNEST sources

A LATERAL ( subquery ) source is re-executed once per outer row: the merged outer row is published into the query context so correlated column references resolve against it, the inner statement is run via query_execute, and each result row is appended (pipeline_recurse_lateral). A UNNEST(array_expr) source expands an array into one row per element. Both integrate as join slots and respect LEFT semantics (a lateral/unnest that yields no rows still emits NULLs under a left join).

Note, join limitations. Only an equality ON condition between two columns is recognized for key-based matching; a non-equi ON or one with expressions falls back to a full scan of the joined table (the broader WHERE tree still filters the final merged row). RIGHT and NATURAL joins are parsed, but the NULL-fill branch in pipeline_recurse only fires for the LEFT codes (301/305), do not rely on right-outer NULL extension or on NATURAL auto-deriving the join key the way PostgreSQL does. FULL OUTER JOIN is not implemented as a NULL-extending join. Use explicit LEFT JOIN ... ON for portable outer-join behavior.

7.4. Subqueries

Subqueries are stored as their raw SQL text on the expression node (subquery_sql) and dispatched through a callback (g_qctx->subquery_fn) that runs the inner statement and returns its values. The forms handled in expr_evaluate are:

Scalar subquery, (SELECT ...)
Executed and reduced to a single value, usable anywhere a scalar expression is allowed.
expr IN (SELECT ...) / NOT IN
The subquery's column is materialized (up to 256 values) and tested for membership.
EXISTS (SELECT ...) / NOT EXISTS
Truth is decided by whether the subquery returns any row.

Correlated subqueries work through the same outer-scope mechanism used by LATERAL: a column reference inside the subquery that is not local resolves against the published outer row.

SELECT name FROM employees e
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.emp_id = e.id
);

7.5. Grouping & Aggregates

When the select list contains an aggregate function or the query has a GROUP BY, the executor switches to the grouping path. Rows are bucketed: with GROUP BY, each row's grouping-key value(s) assign it to a group; with no GROUP BY but an aggregate present, all rows form a single group. Aggregates are then computed by re-walking the rows of each group.

Aggregate functions

COUNT(*)
Number of rows in the group.
COUNT(expr)
Number of rows where expr evaluates to a non-NULL value.
SUM(expr) / AVG(expr)
Numeric total / mean over non-NULL values; an all-NULL group returns NULL. AVG reports an integer when the mean is whole, otherwise four decimal places.
MIN(expr) / MAX(expr)
Extreme value; numeric when every value parses as a number, lexical otherwise.
GROUP_CONCAT(expr)
Concatenates non-NULL values of the group with a separator (default ,), MySQL-style.

Non-aggregate select-list expressions in a grouped query are evaluated against the first row of each group. A HAVING condition is then evaluated per group; aggregates referenced in HAVING are matched by display name to the computed select-list aggregates, and any extra aggregates appearing only in HAVING are computed on the fly.

SELECT dept_id, COUNT(*) AS headcount, AVG(salary) AS avg_pay
FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 3;
Note. EvolutionDB does not enforce the SQL rule that every non-aggregated select-list column must appear in GROUP BY; an ungrouped column simply takes its value from the group's first row (MySQL-style permissiveness, unlike PostgreSQL's strict check). COUNT(DISTINCT ...) and other DISTINCT-qualified aggregate arguments are not part of this aggregate path.

7.6. ORDER BY, DISTINCT & LIMIT

DISTINCT

SELECT DISTINCT removes duplicate result rows after projection. DISTINCTROW is accepted as a synonym. Duplicate elimination compares the rendered output rows.

ORDER BY

Rows are sorted by qsort over the result set. Each sort key may be a column name, a qualified table.column, or a 1-based column position (ORDER BY 2), each optionally suffixed with ASC (default) or DESC. The comparison is adaptive: when both values being compared parse cleanly and entirely as numbers the order is numeric, otherwise it is lexical (CompareRecords in Select.c); DESC simply negates the comparison result.

A special ORDER BY vector_col <=> 'literal' form ranks rows by cosine distance for nearest-neighbor search. When the column carries a ready HNSW index and the query has a LIMIT (and no disqualifying clause such as DISTINCT, OFFSET, join, or extra sort keys), the executor takes an approximate-nearest-neighbor fast path; otherwise it brute-force sorts by distance.

SELECT id FROM documents
ORDER BY embedding <=> '[0.12, 0.84, 0.05]'
LIMIT 5;

LIMIT / OFFSET

Three forms are accepted: LIMIT count; the MySQL two-argument form LIMIT offset, count; and LIMIT count OFFSET offset. LIMIT / OFFSET is applied last, after ORDER BY, DISTINCT, and grouping. In the streaming single-table path, OFFSET skips the first N matching rows and LIMIT stops the scan once N rows have been sent.

SELECT name FROM employees
ORDER BY name
LIMIT 20, 10;   -- skip 20, return 10
Note. The engine's built-in text renderer buffers rows in fixed-size storage for in-memory ORDER BY sorting (up to a few hundred rows in the SELECT * path); large result sets are sorted/streamed by the adaptor's query executor when clients connect through the server. There is no standalone OFFSET clause without LIMIT, and FETCH FIRST / set operations (UNION/INTERSECT/EXCEPT) / CTEs (WITH) are not part of the query grammar.

7.7. Common Errors

42P01, relation does not exist
A table named in FROM or a join could not be resolved in the catalog (tapi_resolve failed).
53000, out of memory
The join pipeline could not allocate its per-table or merged-row buffers.
