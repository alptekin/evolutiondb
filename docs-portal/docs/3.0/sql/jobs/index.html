19. Scheduled Jobs

EvolutionDB stores named, cron-scheduled SQL statements in the system catalog and evaluates their five-field cron patterns against the clock once per background tick.

A scheduled job is a persistent catalog object that binds a name to a cron expression and a single SQL statement. Jobs are created with CREATE JOB, paused or resumed with ALTER JOB, removed with DROP JOB, and introspected with SHOW JOBS. They are evaluated by the same background worker thread that drives auto-RECLAIM; there is no separate scheduler daemon or external process to configure.

Maturity (3.0). The DDL surface, catalog persistence, cron matching, the ENABLE/DISABLE flag, and SHOW JOBS introspection are complete. Firing a job records the match (it stamps last_run_unix and writes a line to the server log) but does not yet run the stored SQL — side-effect execution is deferred to a future release. See §19.6 for the precise current behavior.

19.1. Overview

Each job carries exactly five pieces of state, all held in the CAT_SYS_SCHEDULED_JOBS system catalog (one B+ tree row per job):

name
The job identifier. Job names are global (not schema-qualified) and must be unique. Maximum length is 127 characters.
schedule (cron expression)
A five-field, space-separated cron pattern stored as a literal string (maximum 63 characters). See §19.5.
sql
The single SQL statement the job is configured to run, stored verbatim (maximum 1023 characters). Surrounding single or double quotes are stripped on ingest.
enabled
A boolean. New jobs are enabled. ALTER JOB … DISABLE pauses evaluation; ALTER JOB … ENABLE resumes it.
last_run_unix
A 64-bit Unix timestamp of the most recent tick at which the job's pattern matched, or 0 if it has never matched. Used both for introspection and to de-duplicate matches within a single wall-clock minute.

Not a PostgreSQL feature. Core PostgreSQL has no CREATE JOB statement and no in-server scheduler; recurring tasks are normally handled by the external pg_cron extension or by the host's cron/systemd. The SQL standard does not define scheduled jobs at all. EvolutionDB's CREATE JOB … ON SCHEDULE … DO syntax is closest in shape to MySQL's CREATE EVENT, but the semantics, cron dialect, and limitations below are EvolutionDB's own.

19.2. Creating a job

See the CREATE JOB reference page for the full command synopsis.

CREATE JOB [ IF EXISTS ] name
    ON SCHEDULE '<min> <hour> <dom> <mon> <dow>'
    DO '<sql>';

Both the cron expression and the SQL body arrive as string literals; the engine strips the surrounding quotes before storing them. A valid statement performs four checks in order: the name must be present, the cron expression must validate (§19.5), the DO body must be non-empty, and the name must not already be in use.

-- Fire every day at 03:00 (UTC — see the note on time zones below)
CREATE JOB nightly_rollup
    ON SCHEDULE '0 3 * * *'
    DO 'INSERT INTO daily_totals SELECT today(), count(*) FROM orders';

-- Fire at the top of every hour
CREATE JOB hourly_ping
    ON SCHEDULE '0 * * * *'
    DO 'UPDATE heartbeat SET ts = now()';

19.2.1. The IF EXISTS guard

The optional IF EXISTS clause makes the create idempotent: if a job with the given name already exists, the statement succeeds as a no-op instead of raising an error. (The keyword spelling is IF EXISTS in this position, not IF NOT EXISTS; it guards against a name that is already present.)

CREATE JOB IF EXISTS nightly_rollup
    ON SCHEDULE '0 3 * * *'
    DO 'INSERT INTO daily_totals SELECT today(), count(*) FROM orders';
-- second run with the same name: no error, no change

Without the guard, creating a job whose name is already taken raises SQLSTATE 42710 (duplicate object).

19.3. Pausing and resuming a job

ALTER JOB toggles the enabled flag. There is no facility to change a job's cron expression or SQL body in place; to redefine those, drop and recreate the job.

ALTER JOB name { ENABLE | DISABLE };
ALTER JOB hourly_ping DISABLE;   -- pause: skipped on every tick
ALTER JOB hourly_ping ENABLE;    -- resume

Altering a job that does not exist raises SQLSTATE 42704 (undefined object). A disabled job remains in the catalog and is still listed by SHOW JOBS; it is simply not considered when the scheduler walks the job list.

19.4. Dropping a job

DROP JOB [ IF EXISTS ] name;
DROP JOB hourly_ping;
DROP JOB IF EXISTS maybe_gone;   -- no error if it isn't there

Dropping a non-existent job raises SQLSTATE 42704 unless IF EXISTS is given, in which case the statement is a no-op.

19.5. Cron expression dialect

EvolutionDB 3.0 implements a deliberately minimal cron dialect. An expression is exactly five space- (or tab-) separated fields:

PositionFieldAllowed rangeNotes
1minute0–59
2hour0–23
3day of month1–31
4month1–12
5day of week0–7both 0 and 7 mean Sunday (POSIX convention)

Each field is one of exactly two forms:

*
A wildcard that matches every value of that field.
a single non-negative integer
A literal that matches only when the corresponding clock field equals it.

That is the entire grammar. The following forms found in standard cron are not supported in 3.0 and cause the expression to be rejected:

  • Ranges, e.g. 1-5
  • Comma lists, e.g. 1,15,30
  • Step values, e.g. */5
  • Named months or weekdays, e.g. JAN, MON

This covers the common every-minute, hourly, and daily-at-a-fixed-time cases without a full cron parser. The catalog stores the complete literal string, so a future release can swap in a richer evaluator with no data migration.

-- Accepted
'* * * * *'      -- every minute
'30 * * * *'     -- at minute 30 of every hour
'0 0 1 * *'      -- midnight on the 1st of every month
'0 9 * * 0'      -- 09:00 on Sundays

-- Rejected → SQLSTATE 22023
'*/5 * * * *'    -- step values unsupported
'0 9-17 * * *'   -- ranges unsupported
'0 0 * * MON'    -- named weekdays unsupported
'0 0 * *'       -- only four fields

Time zone: UTC. The cron pattern is matched against UTC, not the server's local time. A job scheduled for 0 3 * * * fires when the UTC clock reads 03:00, regardless of host time zone. Standard cron implementations match local time; EvolutionDB does not. Account for any offset directly in the hour field.

19.6. How jobs are evaluated and fired

There is no dedicated scheduler process. The background worker that performs auto-RECLAIM also calls the scheduler on each tick. That tick interval defaults to 30 seconds and can be lowered (to as little as 1 second) via the EVOSQL_AUTO_RECLAIM_INTERVAL_SEC environment variable. Because the cron dialect's finest granularity is one minute, sub-minute ticks do not cause extra firings.

On each tick the scheduler walks every registered job and fires a job only when all of the following hold:

  1. the job is enabled;
  2. its cron pattern matches the current UTC minute; and
  3. its last_run_unix is older than the start of the current minute — this de-duplicates a job that would otherwise match on two ticks inside the same minute.

When a job fires, the scheduler stamps last_run_unix with the start of the current minute (persisting it to the catalog) and writes a line to the server log noting the match. In 3.0 it does not execute the stored SQL. The hooks for routing the job's SQL through the engine exist, but running arbitrary SQL from the background thread is gated until that path is made safe; that work is scheduled for a future release.

What this means in practice. You can define, persist, enable/disable, and observe scheduled jobs today, and you can confirm a schedule "ticked" by watching last_run_unix advance in SHOW JOBS. Do not yet rely on a job's DO statement producing data changes — until SQL execution is wired up, the statement is recorded but not run.

19.7. Inspecting jobs

SHOW JOBS returns one row per registered job, enabled or not. See the SHOW reference page.

SHOW JOBS;

The result set has five columns:

ColumnTypeMeaning
nametextJob name
scheduletextThe stored cron expression
sqltextThe configured DO statement
enabledbooleantrue if active, false if disabled
last_run_unixbigintUnix timestamp of the most recent match, or 0 if never matched

A non-zero last_run_unix that advances over time confirms the scheduler is matching the pattern; this is the recommended way to verify a schedule fires.

19.8. Error codes

Job statements report failures via SQLSTATE:

SQLSTATEConditionRaised by
42601Syntax error — missing job name, or empty DO bodyCREATE JOB
22023Invalid cron expression (wrong field count, unsupported syntax, or out-of-range value)CREATE JOB
42710A job with that name already existsCREATE JOB (without IF EXISTS)
42704The named job does not existDROP JOB / ALTER JOB (without IF EXISTS)

Notes and limitations (3.0).

  • A job's DO SQL is recorded but not executed; only the match is logged and last_run_unix is stamped. Treat jobs as schedulable metadata until execution lands.
  • Cron matching uses UTC, not local time.
  • The cron dialect supports only * or a single integer per field — no ranges, lists, steps, or names.
  • Job names are global, not schema-qualified, and must be unique across the server.
  • There is no ALTER JOB form to rewrite the schedule or SQL; drop and recreate to change them.
  • The scheduler walks at most 64 jobs per tick.
  • Evaluation runs on the auto-RECLAIM worker thread; if that worker is not started, no job is ever evaluated.
