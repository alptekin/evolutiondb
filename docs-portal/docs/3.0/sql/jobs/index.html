19. Scheduled Jobs

EvolutionDB keeps named, cron-scheduled SQL statements in the system catalog. It checks each five-field cron pattern against the clock once per background tick.

A scheduled job is a saved catalog object. It links a name to a cron expression and a single SQL statement. You create jobs with CREATE JOB. You pause or resume them with ALTER JOB. You remove them with DROP JOB. You list them with SHOW JOBS. The same background worker thread that runs auto-RECLAIM also checks the jobs. There is no separate scheduler daemon or outside process to set up.

Maturity (3.0). The DDL commands, catalog storage, cron matching, the ENABLE/DISABLE flag, and SHOW JOBS are all finished. When a job fires, the server records the match. It stamps last_run_unix and writes a line to the server log. It does not yet run the stored SQL. Running the SQL is planned for a later release. See §19.6 for the exact current behavior.

19.1. Overview

Each job holds exactly five pieces of state. They all live in the CAT_SYS_SCHEDULED_JOBS system catalog. There is one B+ tree row per job.

name
The job name. Job names are global. They are not schema-qualified, and they must be unique. The maximum length is 127 characters.
schedule (cron expression)
A five-field, space-separated cron pattern. It is stored as a plain string. The maximum length is 63 characters. See §19.5.
sql
The single SQL statement the job will run. It is stored exactly as given. The maximum length is 1023 characters. Any single or double quotes around it are removed when it is stored.
enabled
A boolean. New jobs are enabled. ALTER JOB … DISABLE pauses the job. ALTER JOB … ENABLE starts it again.
last_run_unix
A 64-bit Unix timestamp. It marks the most recent tick where the job's pattern matched. It is 0 if the job has never matched. It is used for introspection. It also stops the same minute from matching twice.

Not a PostgreSQL feature. Core PostgreSQL has no CREATE JOB statement and no built-in scheduler. Recurring tasks are usually handled by the outside pg_cron extension or by the host's cron or systemd. The SQL standard does not define scheduled jobs at all. EvolutionDB's CREATE JOB … ON SCHEDULE … DO syntax looks most like MySQL's CREATE EVENT. The behavior, the cron dialect, and the limits below are all specific to EvolutionDB.

19.2. Creating a job

See the CREATE JOB reference page for the full command synopsis.

CREATE JOB [ IF EXISTS ] name
    ON SCHEDULE '<min> <hour> <dom> <mon> <dow>'
    DO '<sql>';

Both the cron expression and the SQL body come in as string literals. The engine removes the quotes around them before it stores them. A valid statement runs four checks in order. The name must be present. The cron expression must be valid, see §19.5. The DO body must not be empty. The name must not already be in use.

-- Fire every day at 03:00 (UTC - see the note on time zones below)
CREATE JOB nightly_rollup
    ON SCHEDULE '0 3 * * *'
    DO 'INSERT INTO daily_totals SELECT today(), count(*) FROM orders';

-- Fire at the top of every hour
CREATE JOB hourly_ping
    ON SCHEDULE '0 * * * *'
    DO 'UPDATE heartbeat SET ts = now()';

19.2.1. The IF EXISTS guard

The optional IF EXISTS clause makes the create safe to repeat. If a job with that name already exists, the statement does nothing and still succeeds. It does not raise an error. The keyword here is IF EXISTS, not IF NOT EXISTS. It guards against a name that is already there.

CREATE JOB IF EXISTS nightly_rollup
    ON SCHEDULE '0 3 * * *'
    DO 'INSERT INTO daily_totals SELECT today(), count(*) FROM orders';
-- second run with the same name: no error, no change

Without the guard, creating a job with a name that is already taken raises SQLSTATE 42710 (duplicate object).

19.3. Pausing and resuming a job

ALTER JOB flips the enabled flag. There is no way to change a job's cron expression or SQL body in place. To change those, drop the job and create it again.

ALTER JOB name { ENABLE | DISABLE };
ALTER JOB hourly_ping DISABLE;   -- pause: skipped on every tick
ALTER JOB hourly_ping ENABLE;    -- resume

Altering a job that does not exist raises SQLSTATE 42704 (undefined object). A disabled job stays in the catalog. It still shows up in SHOW JOBS. The scheduler just skips it when it walks the job list.

19.4. Dropping a job

DROP JOB [ IF EXISTS ] name;
DROP JOB hourly_ping;
DROP JOB IF EXISTS maybe_gone;   -- no error if it isn't there

Dropping a job that does not exist raises SQLSTATE 42704. If you add IF EXISTS, the statement does nothing instead.

19.5. Cron expression dialect

EvolutionDB 3.0 uses a very small cron dialect on purpose. An expression has exactly five fields. They are separated by spaces or tabs.

PositionFieldAllowed rangeNotes
1minute0–59
2hour0–23
3day of month1–31
4month1–12
5day of week0–7both 0 and 7 mean Sunday, as in POSIX

Each field has one of exactly two forms.

*
A wildcard. It matches every value of that field.
a single non-negative integer
A fixed number. It matches only when the matching clock field is equal to it.

That is the whole grammar. The forms below are common in standard cron. They are not supported in 3.0. They cause the expression to be rejected.

  • Ranges, e.g. 1-5
  • Comma lists, e.g. 1,15,30
  • Step values, e.g. */5
  • Named months or weekdays, e.g. JAN, MON

This handles the common cases. It covers every minute, every hour, and a fixed time each day. It does this without a full cron parser. The catalog stores the whole string as written. A later release can add a richer evaluator with no data migration.

-- Accepted
'* * * * *'      -- every minute
'30 * * * *'     -- at minute 30 of every hour
'0 0 1 * *'      -- midnight on the 1st of every month
'0 9 * * 0'      -- 09:00 on Sundays

-- Rejected → SQLSTATE 22023
'*/5 * * * *'    -- step values unsupported
'0 9-17 * * *'   -- ranges unsupported
'0 0 * * MON'    -- named weekdays unsupported
'0 0 * *'       -- only four fields

Time zone is UTC. The cron pattern is matched against UTC. It is not matched against the server's local time. A job set to 0 3 * * * fires when the UTC clock reads 03:00. The host time zone does not matter. Standard cron matches local time. EvolutionDB does not. Add any offset right into the hour field.

19.6. How jobs are evaluated and fired

There is no separate scheduler process. The background worker that runs auto-RECLAIM also calls the scheduler on each tick. The tick interval is 30 seconds by default. You can lower it to as little as 1 second with the EVOSQL_AUTO_RECLAIM_INTERVAL_SEC environment variable. The cron dialect's smallest step is one minute. So ticks shorter than a minute do not cause extra firings.

On each tick the scheduler walks every job. It fires a job only when all of these are true.

  1. the job is enabled;
  2. its cron pattern matches the current UTC minute; and
  3. its last_run_unix is older than the start of the current minute. This stops a job from firing twice on two ticks inside the same minute.

When a job fires, the scheduler stamps last_run_unix with the start of the current minute. It saves this to the catalog. It also writes a line to the server log to note the match. In 3.0 it does not run the stored SQL. The hooks to send the job's SQL through the engine are in place. Running any SQL from the background thread is blocked until that path is safe. That work is planned for a later release.

What this means in practice. Today you can create, save, enable, disable, and watch scheduled jobs. You can confirm a schedule "ticked" by watching last_run_unix move forward in SHOW JOBS. Do not rely on a job's DO statement to change data yet. The statement is saved but not run until SQL execution is wired up.

19.7. Inspecting jobs

SHOW JOBS returns one row per job. It lists enabled and disabled jobs alike. See the SHOW reference page.

SHOW JOBS;

The result set has five columns:

ColumnTypeMeaning
nametextJob name
scheduletextThe stored cron expression
sqltextThe configured DO statement
enabledbooleantrue if active, false if disabled
last_run_unixbigintUnix timestamp of the most recent match, or 0 if never matched

A non-zero last_run_unix that moves forward over time shows the scheduler is matching the pattern. This is the best way to check that a schedule fires.

19.8. Error codes

Job statements report failures through SQLSTATE.

SQLSTATEConditionRaised by
42601Syntax error. The job name is missing, or the DO body is emptyCREATE JOB
22023Invalid cron expression (wrong field count, unsupported syntax, or out-of-range value)CREATE JOB
42710A job with that name already existsCREATE JOB (without IF EXISTS)
42704The named job does not existDROP JOB / ALTER JOB (without IF EXISTS)

Notes and limitations (3.0).

  • A job's DO SQL is recorded but not run. Only the match is logged, and last_run_unix is stamped. Treat jobs as schedulable metadata until execution lands.
  • Cron matching uses UTC, not local time.
  • Each cron field allows only * or a single integer. There are no ranges, lists, steps, or names.
  • Job names are global. They are not schema-qualified, and they must be unique across the server.
  • There is no ALTER JOB form to rewrite the schedule or the SQL. Drop the job and create it again to change them.
  • The scheduler walks at most 64 jobs per tick.
  • Evaluation runs on the auto-RECLAIM worker thread. If that worker is not started, no job is ever evaluated.
