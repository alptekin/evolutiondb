4. SQL Syntax

How EvolutionDB's scanner and parser read SQL text: the lexical tokens — identifiers, literals, comments, operators, keywords — and how statements are framed and terminated.

This chapter describes the lexical structure of the SQL dialect accepted by EvolutionDB 3.0 — the rules that turn a stream of characters into the tokens the grammar consumes. Everything documented here is taken directly from the Flex scanner (evolution/parser/evolexer.l) and the Bison grammar (evolution/parser/evoparser.y); where the dialect departs from the SQL standard or from PostgreSQL, that is called out explicitly rather than assumed.

Lineage. The scanner is case-insensitive and reentrant, and its keyword set is a blend of MySQL and PostgreSQL spellings (for example both INT8 and BIGINT tokenize to the same type). This is a deliberate compatibility choice — EvolutionDB speaks the PostgreSQL wire protocol but recognizes many MySQL-style constructs at the lexical level. Do not assume that a token existing here implies full PostgreSQL semantics; see the per-command pages under SQL Commands for what each construct actually does.

4.1. Lexical Overview

SQL input is a sequence of tokens separated, where necessary, by whitespace or comments. The scanner recognizes these token classes:

Keywords
Reserved words such as SELECT, FROM, CREATE. Matched case-insensitively. See §4.7.
Identifiers
Names of tables, columns, indexes, and aliases — either bare or backtick-quoted. See §4.2.
Literals (constants)
String, numeric, boolean, NULL, hex, and bit-string constants. See §4.3.
Operators
Arithmetic, comparison, logical, bitwise, JSON, and vector operators. See §4.5.
User variables
Session variables prefixed with @. See §4.4.
Comments and whitespace
Discarded before reaching the grammar. See §4.6.

The scanner advances a byte cursor (g_lex_pos) on every token so the parser can slice out the verbatim text of a subquery — this is how correlated and LATERAL subqueries are captured for re-execution. That mechanism is invisible at the syntax level but explains why subquery text round-trips exactly as written.

4.2. Identifiers and Names

An unquoted identifier (token NAME) must begin with an ASCII letter and may continue with letters, digits, and underscores:

[A-Za-z][A-Za-z0-9_]*

This is the exact pattern in the scanner. Note the consequences, which differ from both the SQL standard and PostgreSQL:

  • An identifier cannot start with an underscore — _tmp is not a valid bare identifier (PostgreSQL allows a leading _).
  • Only ASCII letters are accepted in bare identifiers; non-ASCII / Unicode letters are not part of the NAME pattern.
  • There is no length cap in the lexer itself, but identifiers are copied into fixed-size buffers downstream (qualified table.column references are formatted with a 127-character limit per part — see §4.2.2).

4.2.1. Backtick-Quoted Identifiers

To use a reserved word or a name that does not fit the bare pattern, wrap it in backticks. This is the MySQL convention, not the SQL-standard double-quote:

`any chars except backtick, slash, backslash, dot, newline`

The scanner strips the surrounding backticks and returns the inner text as a NAME. An unterminated backtick run (a backtick with no closing backtick before end-of-line) raises "unterminated quoted name".

CREATE TABLE `orders` (
    `select`  INTEGER,        -- reserved word used as a column name
    `order date` DATE       -- embedded space is legal inside backticks
);

Double quotes are string-like, not identifier-like. Unlike the SQL standard and PostgreSQL — where "foo" is a delimited identifier — EvolutionDB's scanner treats a double-quoted run as a string literal (it shares the same lexer rule as single-quoted strings). If you need to quote an identifier, use backticks. See §4.3.1.

4.2.2. Qualified Names

A name may be qualified with a dotted prefix. The grammar accepts two-part names in expression and table positions:

-- column qualified by table/alias
SELECT o.total FROM orders o;

-- schema-qualified table name
SELECT * FROM public.orders;

In expression position, a.b is folded into a single column reference of the form a.b (each part clamped to 127 characters). The special pseudo-table EXCLUDED is recognized in EXCLUDED.col form for use inside INSERT … ON CONFLICT … DO UPDATE.

4.3. Constants (Literals)

4.3.1. String Constants

A string constant is a character sequence bounded by single quotes or double quotes:

'( \\. | '' | [^'\n] )*'
"( \\. | "" | [^"\n] )*"

Two escaping mechanisms are recognized inside a string:

  • Doubled quote — write '' for a literal single quote (the SQL-standard form), or "" for a literal double quote inside a double-quoted string. The parser collapses the doubled quote to a single character.
  • Backslash escape — a backslash followed by any character (\\.) is accepted by the scanner.

A string may not span a newline: a quote opened with no closing quote before the end of the line raises "Unterminated string".

SELECT 'it''s here';       -- doubled quote -> it's here
SELECT "a double-quoted literal";  -- double quotes = string, not identifier

No dollar-quoting, no E'' prefix. PostgreSQL's dollar-quoted strings ($tag$ … $tag$) and escape-string prefix (E'…') are not part of this lexer. Long literals (for example a multi-kilobyte vector written as text) are supported — the literal buffer is heap-sized to the token — but they must still stay on a single line.

4.3.2. Numeric Constants

The scanner distinguishes exact integers from approximate (floating-point) numbers:

INTNUM — integer
Pattern -?[0-9]+, parsed with atoi. Note that an optional leading minus sign is part of the integer token itself.
APPROXNUM — approximate
A number containing a decimal point and/or an exponent, e.g. 3.14, .5, 1E10, 2.5e-3. Parsed with atof.
SELECT 42, -7, 3.14159, .5, 6.022E23;

Precision caveat. Integers are read through atoi (C int range) and approximate numbers through atof (C double). Values outside those ranges are not promoted to arbitrary precision at the lexical stage; choose column types accordingly (see §4.7.2).

4.3.3. Boolean and NULL Constants

The keywords TRUE, FALSE, and UNKNOWN tokenize to a boolean constant; UNKNOWN carries the internal value -1 (three-valued logic), TRUE is 1, FALSE is 0. The keyword NULL is its own token (NULLX).

SELECT TRUE, FALSE, NULL;
SELECT * FROM t WHERE active IS UNKNOWN;   -- IS UNKNOWN is treated as IS NULL

4.3.4. Hexadecimal and Bit-String Constants

Two non-decimal literal forms are recognized and returned as string tokens for downstream interpretation:

FormPatternExample
Hex stringX'[0-9A-F]+' or 0X[0-9A-F]+X'1F4', 0XFF
Bit stringB'[01]+' or 0B[01]+B'1010', 0B1101

Hex digits are matched in uppercase by the pattern, but because the entire scanner is case-insensitive, lowercase hex digits are accepted as well.

4.3.5. Array Literals

An array constant uses the ARRAY[…] constructor. Elements are comma-separated expressions; an empty array is written ARRAY[]:

SELECT ARRAY[1, 2, 3];
INSERT INTO t (tags) VALUES (ARRAY['a', 'b']);
SELECT ARRAY[];

Array elements may be subscripted with expr[ index ], where the subscript binds tighter than any arithmetic operator. Array column types are declared by appending [] to a base type (see §4.7.2).

4.4. User (Session) Variables

A user variable begins with @ and is a session-scoped value. Four spellings are accepted:

@name
Bare: @[0-9a-z_.$]+ — note the bare form is matched only against lowercase letters, digits, _, ., and $.
@"name" / @`name` / @'name'
Quoted, to allow spaces or mixed case in the variable name.

Variables are assigned with SET (using = or :=) and read inside expressions:

SET @threshold := 100;
SELECT * FROM orders WHERE total > @threshold;

An unterminated quoted variable raises "unterminated quoted user variable". See SET for assignment details.

4.5. Operators

The scanner returns the following operator tokens. Single-character punctuation in the set - + & ~ | ^ / % * ( ) , . ; ! [ ] is returned as itself; multi-character operators have dedicated tokens.

4.5.1. Comparison Operators

All comparison operators collapse to a single COMPARISON token carrying a sub-code; the parser branches on the sub-code. This is why, for example, the = in a WITH (m = 16) index option and the = in a WHERE clause are the same lexical token.

OperatorMeaningSub-code
=equal4
<less than1
>greater than2
!= / <>not equal3
<=less than or equal5
>=greater than or equal6
<=>NULL-safe equal / cosine-distance in ORDER BY12

4.5.2. Logical, Bitwise, JSON, and Vector Operators

TokenSpelling(s)Role
Logical ANDAND, &&boolean conjunction
Logical OROR, ||boolean disjunction (note: || is OR, not string concatenation)
Logical XOR / NOTXOR, NOT, !exclusive-or / negation
Arithmetic+ - * / %, MODadd, subtract, multiply, divide, modulo

MySQL-isms to watch. Two operators behave the way MySQL defines them, not PostgreSQL/SQL-standard: || is logical OR (not string concatenation — use CONCAT(...) for that), and ^ is bitwise XOR (not exponentiation — use POWER(...)). The <-> and <#> vector operators are EvolutionDB/pgvector-style extensions.

4.5.3. Operator Precedence

Precedence is fixed by the grammar, from lowest binding to highest:

  1. := (assignment, right-associative)
  2. OR
  3. XOR
  4. AND
  5. IN, IS, LIKE, REGEXP (non-associative)
  6. NOT, !
  7. BETWEEN
  8. COMPARISON (= <> < > <= >= <=>)
  9. <->, <#> (vector distance)
  10. | then & (bitwise)
  11. <<, >> (shift)
  12. +, -
  13. *, /, %, MOD
  14. ^ (bitwise XOR)
  15. ->, ->> (JSON)
  16. unary minus
  17. [ (array subscript — binds tightest)

Use parentheses where intent matters; (expr) is always valid for grouping.

4.6. Comments and Whitespace

Three comment styles are recognized and discarded by the scanner:

StyleFormNotes
Hash line comment# … (to end of line)MySQL-style
Double-dash line comment-- … (to end of line)Requires a space or tab immediately after --
Block comment/* … */May span multiple lines; an unclosed block raises "unclosed comment"

The -- rule is stricter than PostgreSQL. The scanner pattern is "--"[ \t].* — a dash-dash comment is recognized only when followed by a space or tab. So --comment (no space) is not a comment and will be scanned as operators, whereas -- comment is. PostgreSQL accepts -- with no following space. Block comments are not nestable here (PostgreSQL nests them).

Whitespace — space, tab, newline — separates tokens and is otherwise ignored. Any character that matches no rule raises "mystery character".

4.7. Keywords

Keywords are matched case-insensitively, so SELECT, select, and Select are equivalent. The scanner carries several context-sensitive and disambiguating behaviors worth knowing.

4.7.1. Context-Sensitive Tokenization

BETWEEN … AND
When BETWEEN is scanned, the lexer enters a special mode so the next AND is returned as the range-connector token rather than the logical-AND operator. Outside that mode AND is the boolean operator.
NOT EXISTS / NOT IN
NOT EXISTS is recognized as a single token (an EXISTS with a "negated" sub-flag). A side effect: CREATE … IF NOT EXISTS reaches the grammar as IF EXISTS with the negation flag set — the grammar comments note this explicitly.
Function-vs-keyword lookahead
Names like COUNT, SUM, AVG, MIN, MAX, RANK, IF, MOD, REPLACE, and YEAR peek at the next character: if it is ( they become a function token, otherwise they are returned as an ordinary identifier or keyword. This lets rank be a column name but RANK() a window function.
Trailing-context functions
Many built-ins are matched only when immediately followed by ( using a trailing-context rule (for example CAST/"(", SUBSTRING/"(", L2_DISTANCE/"("). Without the parenthesis they fall through to the identifier rule.

4.7.3. A Note on Reserved Words

The keyword set is large because the dialect spans many features (window functions, vector search, document/graph/memory stores, scheduled jobs, pub/sub). Not every keyword is fully reserved in every position — the grammar deliberately keeps some common words usable as names. For example LEVEL is intentionally not reserved (so … AS level works), and VALUE is scanned but returned as an ordinary NAME. When in doubt, backtick-quote a name that might collide (§4.2.1).

4.8. Statements and Termination

4.8.1. Statement Termination

The grammar's start symbol is a semicolon-terminated statement list:

stmt_list : stmt ';'
          | stmt ';' stmt_list
          ;

Every statement is therefore terminated by a semicolon (;), and multiple statements may be submitted in one batch separated by semicolons. The semicolon is part of the grammar — it is required, not merely a separator the client strips.

CREATE TABLE t (id INTEGER PRIMARY KEY);
INSERT INTO t VALUES (1);
SELECT * FROM t;

4.8.3. Statement Categories

The grammar recognizes statements across these areas — each links to its reference page:

  • Queries: SELECT (with JOIN, GROUP BY, HAVING, ORDER BY, LIMIT/OFFSET, window functions, FOR UPDATE/FOR SHARE, FOR SYSTEM_TIME AS OF).
  • Data modification: INSERT (incl. ON CONFLICT / ON DUPLICATE KEY UPDATE), UPDATE, DELETE, REPLACE, COPY, TRUNCATE.
  • Schema: CREATE TABLE, ALTER TABLE, DROP TABLE, CREATE INDEX (incl. USING HASH and USING HNSW), DROP INDEX, CREATE DOMAIN, CREATE SEQUENCE.
  • Database/schema/session: CREATE DATABASE, USE, SET, SHOW.
  • Programmability: CREATE PROCEDURE, CALL, CREATE TRIGGER.
  • Transactions & durability: BEGIN/COMMIT/ROLLBACK, SAVEPOINT, CHECKPOINT, RECLAIM, ANALYZE.
  • Access control: CREATE USER, GRANT, REVOKE, CREATE POLICY (RLS).
  • Async & specialized stores: LISTEN/NOTIFY, CREATE SUBSCRIPTION, CREATE JOB, CREATE MESSAGE LOG, CREATE DOCUMENT STORE, CREATE GRAPH STORE, CREATE ENTITY STORE, CREATE MEMORY STORE.

Notes & limitations (lexical).

  • Identifiers must start with an ASCII letter — no leading underscore, no Unicode letters in bare names. Use backticks for anything else.
  • Double quotes denote strings, not identifiers. Use backticks to quote identifiers.
  • String and quoted-name literals cannot cross a newline. An unterminated literal is a scan error, not a continuation.
  • -- comments require a following space or tab. --like-this is not a comment.
  • Block comments do not nest.
  • || is logical OR and ^ is bitwise XOR (MySQL semantics) — use CONCAT() and POWER() for concatenation and exponentiation.
  • A leading - is part of a numeric literal at the lexical level; combined with operator precedence this is usually transparent, but it explains why negative numbers tokenize as single constants.
  • No dollar-quoting and no E'…' escape-string prefix.
