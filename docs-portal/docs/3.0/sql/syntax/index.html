4. SQL Syntax

This page shows how EvolutionDB reads SQL text. It covers tokens, identifiers, literals, comments, operators, and keywords. It also shows how statements start and end.

This chapter describes the lexical structure of the SQL dialect that EvolutionDB 3.0 accepts. These are the rules that turn a stream of characters into the tokens the grammar reads. Everything here comes straight from the Flex scanner (evolution/parser/evolexer.l) and the Bison grammar (evolution/parser/evoparser.y). When the dialect differs from the SQL standard or from PostgreSQL, this page says so clearly.

Lineage. The scanner is case-insensitive and reentrant. Its keyword set mixes MySQL and PostgreSQL spellings. For example, both INT8 and BIGINT map to the same type. This mix is on purpose, for better compatibility. EvolutionDB speaks the PostgreSQL wire protocol. It also accepts many MySQL-style forms at the lexical level. A token can exist here without full PostgreSQL meaning. To see what each form really does, read the per-command pages under SQL Commands.

4.1. Lexical Overview

SQL input is a sequence of tokens. Whitespace or comments separate them when needed. The scanner knows these token classes.

Keywords
Reserved words such as SELECT, FROM, CREATE. Matched case-insensitively. See §4.7.
Identifiers
Names of tables, columns, indexes, and aliases, either bare or backtick-quoted. See §4.2.
Literals (constants)
String, numeric, boolean, NULL, hex, and bit-string constants. See §4.3.
Operators
Arithmetic, comparison, logical, bitwise, JSON, and vector operators. See §4.5.
User variables
Session variables prefixed with @. See §4.4.
Comments and whitespace
Discarded before reaching the grammar. See §4.6.

The scanner moves a byte cursor named g_lex_pos forward on every token. This lets the parser copy out the exact text of a subquery. That is how correlated and LATERAL subqueries are saved for re-execution. You do not see this at the syntax level. It is the reason subquery text comes back exactly as you wrote it.

4.2. Identifiers and Names

An unquoted identifier (token NAME) must begin with an ASCII letter and may continue with letters, digits, and underscores:

[A-Za-z][A-Za-z0-9_]*

This is the exact pattern in the scanner. The results differ from both the SQL standard and PostgreSQL. Note the points below.

  • An identifier cannot start with an underscore. So _tmp is not a valid bare identifier. PostgreSQL does allow a leading _.
  • Only ASCII letters work in bare identifiers. Non-ASCII and Unicode letters are not part of the NAME pattern.
  • The lexer sets no length cap. But identifiers go into fixed-size buffers later. Each part of a qualified table.column reference has a 127-character limit. See §4.2.2.

4.2.1. Backtick-Quoted Identifiers

To use a reserved word or a name that does not fit the bare pattern, wrap it in backticks. This follows the MySQL convention. It is not the SQL-standard double quote.

`any chars except backtick, slash, backslash, dot, newline`

The scanner removes the outer backticks. It returns the inner text as a NAME. A backtick with no closing backtick before the end of the line is an error. It raises "unterminated quoted name".

CREATE TABLE `orders` (
    `select`  INTEGER,        -- reserved word used as a column name
    `order date` DATE       -- embedded space is legal inside backticks
);

Double quotes are string-like, not identifier-like. In the SQL standard and PostgreSQL, "foo" is a delimited identifier. EvolutionDB is different. Its scanner treats a double-quoted run as a string literal. It uses the same lexer rule as single-quoted strings. To quote an identifier, use backticks. See §4.3.1.

4.2.2. Qualified Names

A name can have a dotted prefix. The grammar accepts two-part names in expression and table positions.

-- column qualified by table/alias
SELECT o.total FROM orders o;

-- schema-qualified table name
SELECT * FROM public.orders;

In expression position, a.b becomes one column reference of the form a.b. Each part is clamped to 127 characters. The special pseudo-table EXCLUDED works in the EXCLUDED.col form. You use it inside INSERT … ON CONFLICT … DO UPDATE.

4.3. Constants (Literals)

4.3.1. String Constants

A string constant is a character sequence bounded by single quotes or double quotes:

'( \\. | '' | [^'\n] )*'
"( \\. | "" | [^"\n] )*"

A string supports two ways to escape characters.

  • Doubled quote. Write '' for a literal single quote. This is the SQL-standard form. Write "" for a literal double quote inside a double-quoted string. The parser turns the doubled quote into a single character.
  • Backslash escape. The scanner accepts a backslash followed by any character. The pattern is \\..

A string may not span a newline. A quote opened with no closing quote before the end of the line raises "Unterminated string".

SELECT 'it''s here';       -- doubled quote -> it's here
SELECT "a double-quoted literal";  -- double quotes = string, not identifier

No dollar-quoting, no E'' prefix. This lexer does not support PostgreSQL dollar-quoted strings like $tag$ … $tag$. It also does not support the escape-string prefix E'…'. Long literals do work. A multi-kilobyte vector written as text is one example. The literal buffer grows on the heap to fit the token. But the literal must still stay on a single line.

4.3.2. Numeric Constants

The scanner distinguishes exact integers from approximate (floating-point) numbers:

INTNUM, integer
Pattern -?[0-9]+, parsed with atoi. An optional leading minus sign is part of the integer token itself.
APPROXNUM, approximate
A number with a decimal point, an exponent, or both. Examples are 3.14, .5, 1E10, and 2.5e-3. Parsed with atof.
SELECT 42, -7, 3.14159, .5, 6.022E23;

Precision caveat. Integers are read with atoi, so they use the C int range. Approximate numbers are read with atof, so they use the C double range. The lexer does not promote values outside those ranges to arbitrary precision. Pick your column types with this in mind. See §4.7.2.

4.3.3. Boolean and NULL Constants

The keywords TRUE, FALSE, and UNKNOWN become a boolean constant. TRUE has the value 1. FALSE has the value 0. UNKNOWN has the internal value -1 for three-valued logic. The keyword NULL is its own token named NULLX.

SELECT TRUE, FALSE, NULL;
SELECT * FROM t WHERE active IS UNKNOWN;   -- IS UNKNOWN is treated as IS NULL

4.3.4. Hexadecimal and Bit-String Constants

The scanner knows two non-decimal literal forms. It returns them as string tokens. Later stages read them.

FormPatternExample
Hex stringX'[0-9A-F]+' or 0X[0-9A-F]+X'1F4', 0XFF
Bit stringB'[01]+' or 0B[01]+B'1010', 0B1101

The pattern matches hex digits in uppercase. The whole scanner is case-insensitive. So lowercase hex digits work too.

4.3.5. Array Literals

An array constant uses the ARRAY[…] constructor. The elements are expressions separated by commas. You write an empty array as ARRAY[].

SELECT ARRAY[1, 2, 3];
INSERT INTO t (tags) VALUES (ARRAY['a', 'b']);
SELECT ARRAY[];

You can subscript array elements with expr[ index ]. The subscript binds tighter than any arithmetic operator. To declare an array column type, add [] to a base type. See §4.7.2.

4.4. User (Session) Variables

A user variable begins with @. It holds a value for the session. The scanner accepts four spellings.

@name
Bare form with the pattern @[0-9a-z_.$]+. The bare form matches only lowercase letters, digits, _, ., and $.
@"name" / @`name` / @'name'
Quoted form. It allows spaces or mixed case in the variable name.

You assign variables with SET, using = or :=. You read them inside expressions.

SET @threshold := 100;
SELECT * FROM orders WHERE total > @threshold;

A quoted variable with no closing quote raises "unterminated quoted user variable". See SET for assignment details.

4.5. Operators

The scanner returns the operator tokens below. Single-character punctuation in the set - + & ~ | ^ / % * ( ) , . ; ! [ ] is returned as itself. Multi-character operators have their own tokens.

4.5.1. Comparison Operators

All comparison operators collapse to one COMPARISON token. The token carries a sub-code. The parser then branches on that sub-code. This is why the = in a WITH (m = 16) index option and the = in a WHERE clause are the same lexical token.

OperatorMeaningSub-code
=equal4
<less than1
>greater than2
!= / <>not equal3
<=less than or equal5
>=greater than or equal6
<=>NULL-safe equal / cosine-distance in ORDER BY12

4.5.2. Logical, Bitwise, JSON, and Vector Operators

TokenSpelling(s)Role
Logical ANDAND, &&boolean conjunction
Logical OROR, ||boolean disjunction. Here || is OR, not string concatenation
Logical XOR / NOTXOR, NOT, !exclusive-or / negation
Arithmetic+ - * / %, MODadd, subtract, multiply, divide, modulo
Bitwise& | ^and / or / xor
Shift<<, >>left / right bit-shift
JSON->, ->>JSON member access. The ->> form returns text
Vector<->, <#>L2 distance / negative inner product
Assignment:=variable assignment, used in SET and in @v := expr

MySQL-isms to watch. Two operators behave the way MySQL defines them. They do not follow PostgreSQL or the SQL standard. First, || is logical OR. It is not string concatenation. Use CONCAT(...) for that. Second, ^ is bitwise XOR. It is not exponentiation. Use POWER(...) for that. The <-> and <#> vector operators are EvolutionDB and pgvector-style extensions.

4.5.3. Operator Precedence

The grammar fixes the precedence. The list goes from lowest binding to highest.

  1. := (assignment, right-associative)
  2. OR
  3. XOR
  4. AND
  5. IN, IS, LIKE, REGEXP (non-associative)
  6. NOT, !
  7. BETWEEN
  8. COMPARISON (= <> < > <= >= <=>)
  9. <->, <#> (vector distance)
  10. | then & (bitwise)
  11. <<, >> (shift)
  12. +, -
  13. *, /, %, MOD
  14. ^ (bitwise XOR)
  15. ->, ->> (JSON)
  16. unary minus
  17. [ (array subscript, binds tightest)

Use parentheses when your intent matters. The form (expr) is always valid for grouping.

4.6. Comments and Whitespace

The scanner knows three comment styles. It drops all of them.

StyleFormNotes
Hash line comment# … (to end of line)MySQL-style
Double-dash line comment-- … (to end of line)Requires a space or tab immediately after --
Block comment/* … */May span multiple lines; an unclosed block raises "unclosed comment"

The -- rule is stricter than PostgreSQL. The scanner pattern is "--"[ \t].*. A dash-dash comment counts only when a space or tab follows it. So --comment with no space is not a comment. The scanner reads it as operators. But -- comment with a space is a comment. PostgreSQL accepts -- with no space after it. Block comments do not nest here. PostgreSQL does nest them.

Whitespace means a space, a tab, or a newline. It separates tokens. Other than that, it is ignored. Any character that matches no rule raises "mystery character".

4.7. Keywords

Keywords match case-insensitively. So SELECT, select, and Select are the same. The scanner has a few context-sensitive behaviors worth knowing. They help it tell similar tokens apart.

4.7.1. Context-Sensitive Tokenization

BETWEEN … AND
When the scanner reads BETWEEN, the lexer enters a special mode. In that mode the next AND comes back as the range-connector token, not the logical-AND operator. Outside that mode, AND is the boolean operator.
NOT EXISTS / NOT IN
The scanner reads NOT EXISTS as a single token. It is an EXISTS with a "negated" sub-flag. This has a side effect. CREATE … IF NOT EXISTS reaches the grammar as IF EXISTS with the negation flag set. The grammar comments point this out.
Function-vs-keyword lookahead
Names like COUNT, SUM, AVG, MIN, MAX, RANK, IF, MOD, REPLACE, and YEAR peek at the next character. If it is (, they become a function token. If not, they come back as an ordinary identifier or keyword. So rank can be a column name while RANK() is a window function.
Trailing-context functions
Many built-ins match only when ( follows them right away. This uses a trailing-context rule. Examples are CAST/"(", SUBSTRING/"(", and L2_DISTANCE/"(". Without the parenthesis, they fall through to the identifier rule.

4.7.3. A Note on Reserved Words

The keyword set is large because the dialect covers many features. These include window functions, vector search, document, graph, and memory stores, scheduled jobs, and pub/sub. Not every keyword is fully reserved in every position. The grammar keeps some common words usable as names on purpose. For example, LEVEL is not reserved, so … AS level works. The scanner reads VALUE but returns it as an ordinary NAME. When in doubt, backtick-quote a name that might collide. See §4.2.1.

4.8. Statements and Termination

4.8.1. Statement Termination

The grammar's start symbol is a semicolon-terminated statement list:

stmt_list : stmt ';'
          | stmt ';' stmt_list
          ;

So a semicolon (;) ends every statement. You can send many statements in one batch, separated by semicolons. The semicolon is part of the grammar. It is required. It is not just a separator that the client strips.

CREATE TABLE t (id INTEGER PRIMARY KEY);
INSERT INTO t VALUES (1);
SELECT * FROM t;

4.8.3. Statement Categories

The grammar covers statements across these areas. Each one links to its reference page.

  • Queries. SELECT with JOIN, GROUP BY, HAVING, ORDER BY, LIMIT/OFFSET, window functions, FOR UPDATE/FOR SHARE, and FOR SYSTEM_TIME AS OF.
  • Data modification. INSERT with ON CONFLICT / ON DUPLICATE KEY UPDATE, UPDATE, DELETE, REPLACE, COPY, TRUNCATE.
  • Schema. CREATE TABLE, ALTER TABLE, DROP TABLE, CREATE INDEX with USING HASH and USING HNSW, DROP INDEX, CREATE DOMAIN, CREATE SEQUENCE.
  • Database, schema, and session. CREATE DATABASE, USE, SET, SHOW.
  • Programmability. CREATE PROCEDURE, CALL, CREATE TRIGGER.
  • Transactions and durability. BEGIN/COMMIT/ROLLBACK, SAVEPOINT, CHECKPOINT, RECLAIM, ANALYZE.
  • Access control. CREATE USER, GRANT, REVOKE, CREATE POLICY for RLS.
  • Async and specialized stores. LISTEN/NOTIFY, CREATE SUBSCRIPTION, CREATE JOB, CREATE MESSAGE LOG, CREATE DOCUMENT STORE, CREATE GRAPH STORE, CREATE ENTITY STORE, CREATE MEMORY STORE.

Lexical notes and limits.

  • Identifiers must start with an ASCII letter. A bare name cannot start with an underscore and cannot use Unicode letters. Use backticks for anything else.
  • Double quotes mean strings, not identifiers. Use backticks to quote identifiers.
  • String and quoted-name literals cannot cross a newline. A literal with no closing quote is a scan error. It does not continue on the next line.
  • -- comments need a space or tab after them. --like-this is not a comment.
  • Block comments do not nest.
  • || is logical OR and ^ is bitwise XOR. This follows MySQL. Use CONCAT() for concatenation and POWER() for exponentiation.
  • A leading - is part of a numeric literal at the lexical level. With operator precedence this is usually invisible. It is why negative numbers tokenize as single constants.
  • No dollar-quoting and no E'…' escape-string prefix.
