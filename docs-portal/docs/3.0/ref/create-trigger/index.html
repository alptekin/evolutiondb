CREATE TRIGGER

Define a row-level trigger whose procedural body runs automatically before or after an INSERT, UPDATE, or DELETE on a table or memory store.

Synopsis

CREATE TRIGGER name
    { BEFORE | AFTER }
    { INSERT | UPDATE | DELETE | PUT }
    ON { table_name | MEMORY STORE store_name }
    FOR EACH ROW
BEGIN
    trigger_body
END

Description

CREATE TRIGGER registers a new trigger against an existing table (or the backing table of a memory store). Once created, the trigger fires for every row affected by the matching event, executing its procedural body through the same engine that runs stored procedures, so IF, WHILE, SIGNAL, and other procedural statements are available inside the body.

EvolutionDB triggers are always row-level (FOR EACH ROW is mandatory and is the only granularity the grammar accepts; there is no statement-level form). Each trigger fires on exactly one event and one timing. To react to several events you must create one trigger per event.

Within the body, the special qualifiers OLD.column and NEW.column refer to the row values before and after the operation. These references are substituted with literal values immediately before the body is executed: OLD reflects the existing row (available for UPDATE and DELETE), and NEW reflects the incoming row (available for INSERT and UPDATE). A reference whose value is unknown is replaced with NULL.

In a BEFORE trigger, a statement of the form SET NEW.column = expression; is recognized specially: the expression is evaluated and written back into the corresponding column of the row being inserted or updated, letting the trigger transform the row before it is stored. This rewrite is honored only for BEFORE triggers.

Trigger bodies may themselves perform DML that fires other triggers; recursion is bounded by a fixed maximum nesting depth, beyond which further trigger firing is silently suppressed. If the body raises an error, the firing operation fails and the error (with its SQLSTATE, where one is set) propagates to the caller.

Creating a trigger whose name already exists fails (SQLSTATE 42P07). Naming a target table that does not exist fails (SQLSTATE 42P01); naming a memory store that does not exist fails (SQLSTATE 42704).

Parameters

name
The name of the new trigger. Trigger names are looked up across all tables in the current schema, so a name must be unique within the schema.
BEFORE | AFTER
The timing of the trigger relative to the triggering operation. BEFORE runs the body before the row change is applied (and enables SET NEW.column rewrites); AFTER runs the body once the row change has been applied.
INSERT | UPDATE | DELETE | PUT
The single event that fires the trigger. PUT is accepted as an alias and is treated identically to INSERT. Only one event may be specified per trigger; there is no multi-event (INSERT OR UPDATE) form, and UPDATE cannot be restricted to a column list.
ON table_name
The table the trigger is attached to. The table must already exist in the current database and schema.
ON MEMORY STORE store_name
Attach the trigger to a memory store instead of an ordinary table. The store name is resolved through the memory-store catalog and rewritten to its backing table (mem_store_name); the trigger then fires on that backing table exactly as a regular table trigger would.
FOR EACH ROW
Required. Specifies row-level firing, the body runs once per affected row. This is the only granularity supported.
trigger_body
The procedural statements between BEGIN and END, executed for each matching row. The body reuses the stored-procedure execution engine and may reference OLD.column / NEW.column, and (in BEFORE triggers) assign to NEW.column with SET.

Examples

Stamp a value into a new row before it is inserted (BEFORE trigger with SET NEW):

CREATE TRIGGER trg_audit_stamp
    BEFORE INSERT ON orders
    FOR EACH ROW
BEGIN
    SET NEW.status = 'pending';
END;

Mirror deleted rows into an audit table (AFTER trigger using OLD):

CREATE TRIGGER trg_archive_deleted
    AFTER DELETE ON orders
    FOR EACH ROW
BEGIN
    INSERT INTO orders_archive (id, status)
    VALUES (OLD.id, OLD.status);
END;

Reject an update with SIGNAL when a guard condition holds (BEFORE trigger using both OLD and NEW):

CREATE TRIGGER trg_no_downgrade
    BEFORE UPDATE ON orders
    FOR EACH ROW
BEGIN
    IF NEW.total < OLD.total THEN
        SIGNAL SQLSTATE '45000';
    END IF;
END;

Attach a trigger to a memory store (PUT is treated as INSERT):

CREATE TRIGGER trg_cache_touch
    AFTER PUT ON MEMORY STORE sessions
    FOR EACH ROW
BEGIN
    -- runs against the backing table mem_sessions
    INSERT INTO session_log (sid) VALUES (NEW.sid);
END;

Compatibility

EvolutionDB's CREATE TRIGGER resembles the SQL standard and PostgreSQL but is a narrower, more MySQL-flavored form, and several PostgreSQL clauses are not supported:

  • Inline body, not a function. The action is written directly as a BEGIN … END block. There is no EXECUTE FUNCTION / EXECUTE PROCEDURE clause and no separate trigger function; the body runs on the procedure engine.
  • Row-level only. FOR EACH ROW is required. The PostgreSQL FOR EACH STATEMENT granularity is not available.
  • Single event only. Exactly one of INSERT, UPDATE, DELETE, or PUT per trigger. PostgreSQL's INSERT OR UPDATE OR DELETE combinations and UPDATE OF column are not accepted. TRUNCATE triggers are not supported.
  • No WHEN condition. There is no WHEN (condition) clause; gate logic must be written inside the body (for example with IF … SIGNAL).
  • No REFERENCING / transition tables. Old and new values are exposed only through literal OLD.column / NEW.column substitution, not via named transition tables.
  • PUT alias and MEMORY STORE target are EvolutionDB extensions with no equivalent in PostgreSQL or the SQL standard.
  • Row mutation via SET NEW.column in BEFORE triggers follows the MySQL convention rather than PostgreSQL's "assign to NEW and RETURN" model.
