MEMORY PUT

Write one key/value memory item into a memory store. This can insert a new item or replace an old one.

Synopsis

MEMORY PUT INTO store_name VALUES (
    namespace,
    key,
    value [ , embedding ]
)

Description

MEMORY PUT stores one item in a memory store. You make the store first with CREATE MEMORY STORE. Each item has an identity. The identity is the pair of namespace and key. The item holds a value. The item can also hold an embedding vector. The vector is used for semantic search with MEMORY SEARCH.

The command has upsert (PUT) semantics. The pair of namespace and key is the primary key of the store's backing table. So a write to a pair that already exists replaces the old value. It does not raise a duplicate-key error. The replacement is atomic inside the storage layer. The new tuple is inserted first. Then the old one is dropped. This way an item is never missing if the write fails partway through.

Each memory store is built as a normal heap table. The table is named mem_<store_name> in the current schema. MEMORY PUT fills in that table's columns for you. It builds the composite primary key from namespace and key. It sets created_at to the current UTC timestamp. It leaves ttl_at NULL. The write goes through the engine's normal insert path. So MVCC visibility, constraints, and any policies on the backing table all apply.

You can give the embedding argument only for stores that have an embedding dimension. This means embedding_dim > 0. For a store made without embeddings, use the three-value form. The embedding column does not exist, and the four-value form has no slot to fill.

Parameters

store_name
The name of the target memory store. You give it as a bare identifier after INTO. The store must already exist. If it does not, the command fails with SQLSTATE 42704. This means the memory store does not exist.
namespace
A string literal that groups related items. For example, it can be a tenant, an agent, or a conversation id. Together with key it forms the item's identity. It is stored in the mem_namespace column.
key
A string literal that names the item inside its namespace. The pair of namespace and key is unique within the store. Reusing a pair replaces the existing item.
value
The payload. It is written to the store's mem_value JSON column. Each argument is either a string literal or the keyword NULL. If you give NULL here, it stores a NULL value.
embedding
Optional. A string literal that holds the vector for this item in [f1,f2,...] form. It is stored in the embedding column. You can use it only when the store was created with an embedding dimension. Like the other arguments, it may be NULL to store the item without a vector.

The argument syntax is positional. You do not name columns. Every value is a quoted string literal or NULL. Only the two forms shown in the Synopsis are accepted. You give three values without an embedding, or four values with one.

Examples

Store a memory item in a plain key/value store with no embedding.

MEMORY PUT INTO agent_mem VALUES (
    'user:42',
    'favorite_color',
    '{"color": "blue"}'
);

Overwrite the same item with a new value. It uses the same namespace and key.

MEMORY PUT INTO agent_mem VALUES (
    'user:42',
    'favorite_color',
    '{"color": "green"}'
);

Store an item with its embedding vector. The store was created with an embedding dimension.

MEMORY PUT INTO agent_mem VALUES (
    'user:42',
    'note',
    'Prefers concise answers',
    '[0.12, -0.04, 0.88, 0.31]'
);

Store an item with a NULL value.

MEMORY PUT INTO agent_mem VALUES (
    'session:abc',
    'last_seen',
    NULL
);

Compatibility

MEMORY PUT is an EvolutionDB extension. It is not part of the SQL standard. It has no equivalent in PostgreSQL. It is built for long-term agent memory. It gives you a managed key/value-plus-vector layer over a normal backing table.

The upsert behavior is like PostgreSQL's INSERT ... ON CONFLICT DO UPDATE. But the two are not the same in syntax. MEMORY PUT uses positional string-literal arguments and a fixed implicit schema. It does not use an explicit column list and conflict target. Unlike a standard INSERT, values are always string literals or NULL. Bind parameters and expressions are not accepted in the VALUES list. The matching hand-written statement would be an INSERT INTO mem_<store_name> (...) VALUES (...) on the backing table. You would also need to compute the primary key, the timestamp, and the conflict handling by hand.
