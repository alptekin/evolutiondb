MEMORY PUT

Write (insert or replace) a single key/value memory item into a memory store.

Synopsis

MEMORY PUT INTO store_name VALUES (
    namespace,
    key,
    value [ , embedding ]
)

Description

MEMORY PUT stores one item in a memory store previously created with CREATE MEMORY STORE. An item is identified by the pair (namespace, key); its payload is an arbitrary value and, optionally, an embedding vector used for semantic search via MEMORY SEARCH.

The operation has upsert (PUT) semantics: the (namespace, key) pair forms the primary key of the store's backing table, so writing to an existing pair replaces the previous value rather than raising a duplicate-key error. The replacement is performed atomically inside the storage layer, the new tuple is inserted first and the old one is dropped afterward, so an item is never left missing if the write fails partway through.

Under the hood each memory store is materialized as an ordinary heap table named mem_<store_name> in the current schema. MEMORY PUT fills in that table's columns automatically: it derives the composite primary key from namespace and key, stamps created_at with the current UTC timestamp, and leaves ttl_at NULL. Because the write is routed through the engine's normal insert path, MVCC visibility, constraints, and any policies on the backing table apply.

The embedding argument may be supplied only for stores that were created with an embedding dimension (embedding_dim > 0). For a store created without embeddings, use the three-value form; the embedding column does not exist and the four-value form has no slot to fill.

Parameters

store_name
The name of the target memory store, given as a bare identifier (the INTO target). The store must already exist; otherwise the command fails with SQLSTATE 42704 (memory store does not exist).
namespace
A string literal that groups related items (for example a tenant, agent, or conversation id). Together with key it forms the item's identity. Stored in the mem_namespace column.
key
A string literal identifying the item within its namespace. The (namespace, key) pair is unique within the store; reusing a pair replaces the existing item.
value
The payload, written to the store's mem_value (JSON) column. Each argument is either a string literal or the keyword NULL; supplying NULL here stores a NULL value.
embedding
Optional. A string literal holding the vector for this item in [f1,f2,...] form, stored in the embedding column. Permitted only when the store was created with an embedding dimension. As with the other arguments it may be NULL to store the item without a vector.

The argument syntax is positional, not column-named: every value is a quoted string literal (or NULL). Only the two arities shown in the Synopsis are accepted: three values without an embedding, or four values with one.

Examples

Store a memory item in a plain key/value store (no embedding):

MEMORY PUT INTO agent_mem VALUES (
    'user:42',
    'favorite_color',
    '{"color": "blue"}'
);

Overwrite the same item (same namespace and key) with a new value:

MEMORY PUT INTO agent_mem VALUES (
    'user:42',
    'favorite_color',
    '{"color": "green"}'
);

Store an item together with its embedding vector (store created with an embedding dimension):

MEMORY PUT INTO agent_mem VALUES (
    'user:42',
    'note',
    'Prefers concise answers',
    '[0.12, -0.04, 0.88, 0.31]'
);

Store an item with a NULL value:

MEMORY PUT INTO agent_mem VALUES (
    'session:abc',
    'last_seen',
    NULL
);

Compatibility

MEMORY PUT is an EvolutionDB extension. It is not part of the SQL standard and has no equivalent in PostgreSQL. It is purpose-built for long-term agent memory: a managed key/value-plus-vector abstraction over an ordinary backing table.

The upsert behavior resembles PostgreSQL's INSERT ... ON CONFLICT DO UPDATE, but the two are unrelated in syntax: MEMORY PUT uses positional string-literal arguments and a fixed implicit schema rather than an explicit column list and conflict target. Unlike a standard INSERT, values are always string literals (or NULL), bind parameters and expressions are not accepted in the VALUES list. The equivalent hand-written statement would be an INSERT INTO mem_<store_name> (...) VALUES (...) against the backing table with a manually computed primary key, timestamp, and conflict handling.
