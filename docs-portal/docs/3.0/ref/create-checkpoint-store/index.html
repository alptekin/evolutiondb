CREATE CHECKPOINT STORE

Register a named checkpoint store for LangGraph-style agent state and auto-create its backing table.

Synopsis

CREATE CHECKPOINT STORE [ IF NOT EXISTS ] name
    [ WITH ( retention = 'interval_text' ) ]

Description

CREATE CHECKPOINT STORE registers a checkpoint store named name in the system catalog and, in the same statement, auto-creates a backing heap table that holds the actual checkpoint rows. The store is EvolutionDB's native backend for the LangGraph BaseCheckpointSaver contract: it persists per-thread agent snapshots so a graph run can be resumed from where it left off.

The backing table is created in the current session schema with the fixed name ck_name and the following fixed schema:

thread_id VARCHAR(255) NOT NULL
The conversation / graph thread the checkpoint belongs to.
checkpoint_ns VARCHAR(255)
Namespace within the thread. Nullable; an empty string maps to NULL in this engine.
checkpoint_id VARCHAR(255) PRIMARY KEY
The unique identifier of the checkpoint. Indexed via the primary-key B+ tree.
parent_id VARCHAR(255)
The id of the checkpoint this one descends from. Nullable for root checkpoints.
ck_values JSON
The serialized channel values payload. Named ck_values to avoid the SQL reserved word values.
metadata JSON
Application metadata attached to the checkpoint.
parent_config JSON
The serialized configuration of the parent checkpoint.
created_at TIMESTAMP
Insertion time, set automatically by the engine on each write.

Because the store is just a catalog registration over an ordinary table, every standard engine path applies to it unchanged: MVCC visibility, row-level security, triggers, and replication all work on the backing table exactly as on a hand-written table.

If a store with the same name already exists, the statement fails (SQLSTATE 42710) unless IF NOT EXISTS is given, in which case the statement is a no-op and no error is raised. A missing or empty name is a syntax error (SQLSTATE 42601).

Parameters

IF NOT EXISTS
Do not raise an error if a checkpoint store with the same name already exists. When the store exists, the statement returns successfully without creating anything or altering the existing store. Note that this guard checks only for an existing checkpoint store of that name; it does not guarantee the backing table is identical.
name
The name of the checkpoint store to create. It must match the engine's identifier pattern ([A-Za-z][A-Za-z0-9_]*). The backing table is created as ck_name in the current schema.
WITH ( retention = 'interval_text' )
Optional. Stores a free-form retention expression (for example '30 days' or '1 hour') on the store descriptor in the catalog. The value is recorded verbatim with surrounding quotes stripped; it is descriptive metadata only. In this release no automatic retention pruning is performed against it, old checkpoints are not deleted by the engine on the basis of this value. retention is the only key accepted inside the WITH clause.

Examples

Create a checkpoint store named agent_runs:

CREATE CHECKPOINT STORE agent_runs;

Create it only if it does not already exist:

CREATE CHECKPOINT STORE IF NOT EXISTS agent_runs;

Create a store and record a retention expression on it:

CREATE CHECKPOINT STORE agent_runs
    WITH (retention = '30 days');

Because the store is backed by an ordinary table, you can also read and write checkpoints directly against ck_name:

INSERT INTO ck_agent_runs (thread_id, checkpoint_id, ck_values, metadata)
VALUES ('thread-1', 'ckpt-001', '{"step":1}', '{"source":"loop"}');

SELECT checkpoint_id, created_at
FROM   ck_agent_runs
WHERE  thread_id = 'thread-1'
ORDER BY created_at DESC;

The grammar also accepts native checkpoint DML, CHECKPOINT PUT [WRITES] INTO name VALUES (…), CHECKPOINT GET FROM name THREAD 'id' [AT 'checkpoint_id'], and CHECKPOINT LIST FROM name THREAD 'id' [LIMIT n], which compile down to INSERT / SELECT against the same backing table. GET returns the newest matching row; LIST returns history newest-first.

Compatibility

CREATE CHECKPOINT STORE is an EvolutionDB extension. It is not part of the SQL standard and has no equivalent in PostgreSQL, PostgreSQL has no checkpoint-store object (its CHECKPOINT command is an unrelated WAL/buffer flush operation).

The clauses EvolutionDB accepts are exactly IF NOT EXISTS and the single-key WITH (retention = '…') option shown in the Synopsis. There is no TABLESPACE, owner, encoding, or other PostgreSQL CREATE …-style clause here. Although the spelling of the WITH clause resembles a generic storage-parameter list, retention is the only recognized key and it is stored as descriptive metadata rather than enforced in this release.
