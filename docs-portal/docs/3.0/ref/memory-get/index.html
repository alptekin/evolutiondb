MEMORY GET

Fetch a single record from a memory store by its namespace and key.

Synopsis

MEMORY GET FROM store_name
    WHERE NS = 'namespace' AND KEY = 'key'

Description

MEMORY GET retrieves exactly one stored entry from a memory store created with CREATE MEMORY STORE. A memory store is backed by an ordinary table named mem_<store_name>, and each entry is identified by a composite primary key formed from its namespace and key. MEMORY GET resolves the target row by building that composite key and looking it up directly, returning at most one row.

The result set is a projection of the backing table. The following columns are always returned, in order: mem_namespace, mem_key, mem_value, and created_at. If the store was created with a non-zero embedding dimension, the embedding column is additionally projected between mem_value and created_at. The query is internally capped at one row.

Internally the command rewrites to a SELECT against the backing table with a predicate of the form pk = 'namespace' || 0x1E || 'key' (the namespace and key joined by an ASCII Record Separator, 0x1E). Because it runs through the standard SELECT path, row-level security policies, MVCC visibility, and triggers attached to the store apply identically to a hand-written query against mem_<store_name>.

If no entry matches the given namespace and key, the command succeeds and returns an empty result set. If the named store does not exist, an error is raised with SQLSTATE 42704 (undefined_object).

The WHERE clause of MEMORY GET is fixed: it accepts exactly the two conditions NS and KEY joined by AND. The grammar tolerates any comparison operator token in each position, but the lookup is always performed as an exact equality match on the composite key; non-equality operators do not produce a range scan. Always write = for clarity.

Parameters

store_name
The name of the memory store to read from, as given to CREATE MEMORY STORE. This is an identifier (not a quoted string). It resolves to the backing table mem_<store_name>. If the store does not exist, the command fails with SQLSTATE 42704.
NS = 'namespace'
The namespace component of the entry's identity, given as a string literal. The namespace lets a single store hold logically separate partitions of memory (for example, one namespace per agent, session, or tenant). Surrounding quotes are stripped and SQL-standard doubled inner quotes are collapsed before the lookup.
KEY = 'key'
The key component of the entry's identity within the namespace, given as a string literal. Together NS and KEY uniquely identify one stored record. Quote handling is the same as for the namespace.

Examples

Read back a value that was written with MEMORY PUT:

MEMORY PUT INTO agent_mem
    VALUES ('session:42', 'last_intent', 'book_flight');

MEMORY GET FROM agent_mem
    WHERE NS = 'session:42' AND KEY = 'last_intent';

A namespace and key containing a single quote, written with the SQL-standard doubled quote:

MEMORY GET FROM agent_mem
    WHERE NS = 'user''s notes' AND KEY = 'draft';

Fetching from a store created with an embedding dimension also returns the embedding column:

CREATE MEMORY STORE kb WITH (embedding_dim = 3, distance = 'cosine');
MEMORY PUT INTO kb
    VALUES ('docs', 'intro', 'welcome', '[0.1, 0.2, 0.3]');

-- returns mem_namespace, mem_key, mem_value, embedding, created_at
MEMORY GET FROM kb
    WHERE NS = 'docs' AND KEY = 'intro';

Compatibility

MEMORY GET is an EvolutionDB extension and is not part of the SQL standard; there is no corresponding command in PostgreSQL. It exists as ergonomic sugar over a single-row SELECT … WHERE pk = … LIMIT 1 against the store's backing table.

The clause vocabulary is intentionally narrow. Unlike a general SELECT, the WHERE clause cannot be extended: only the fixed NS + KEY equality pair is accepted, there is no column projection list, no ORDER BY, and no user-supplied LIMIT (the result is always capped at one row). To run arbitrary predicates over stored memory, query the backing table mem_<store_name> directly with a standard SELECT.
