MEMORY GET

Get one record from a memory store by its namespace and key.

Synopsis

MEMORY GET FROM store_name
    WHERE NS = 'namespace' AND KEY = 'key'

Description

MEMORY GET gets exactly one stored entry from a memory store. You make the store with CREATE MEMORY STORE. A memory store is held in a normal table named mem_<store_name>. Each entry has a composite primary key. That key is made from its namespace and its key. MEMORY GET builds that composite key and looks it up directly. It returns at most one row.

The result set is a projection of the backing table. These columns always come back, in this order. They are mem_namespace, mem_key, mem_value, and created_at. The store can be made with a non-zero embedding dimension. In that case the embedding column also comes back. It sits between mem_value and created_at. The query is capped at one row inside the engine.

Inside the engine the command turns into a SELECT against the backing table. The predicate has the form pk = 'namespace' || 0x1E || 'key'. The namespace and key are joined by an ASCII Record Separator. That byte is 0x1E. The command runs through the normal SELECT path. So row-level security policies, MVCC visibility, and triggers on the store work the same way as in a query you write by hand against mem_<store_name>.

The command may find no entry for the given namespace and key. Then it still succeeds and returns an empty result set. If the named store does not exist, the command raises an error. The SQLSTATE is 42704. That code means undefined_object.

The WHERE clause of MEMORY GET is fixed. It takes exactly two conditions. They are NS and KEY, joined by AND. The grammar accepts any comparison operator token in each spot. But the lookup is always an exact equality match on the composite key. A non-equality operator does not create a range scan. Always write = to keep things clear.

Parameters

store_name
The name of the memory store you want to read. You give the same name to CREATE MEMORY STORE. This is an identifier, not a quoted string. It points to the backing table mem_<store_name>. If the store does not exist, the command fails with SQLSTATE 42704.
NS = 'namespace'
The namespace part of the entry's identity, given as a string literal. The namespace lets one store hold separate parts of memory. For example, you can use one namespace per agent, session, or tenant. The outer quotes are stripped before the lookup. SQL-standard doubled inner quotes are collapsed into one quote too.
KEY = 'key'
The key part of the entry's identity inside the namespace, given as a string literal. Together NS and KEY point to one stored record. Quotes are handled the same way as for the namespace.

Examples

Read back a value that you wrote with MEMORY PUT.

MEMORY PUT INTO agent_mem
    VALUES ('session:42', 'last_intent', 'book_flight');

MEMORY GET FROM agent_mem
    WHERE NS = 'session:42' AND KEY = 'last_intent';

This example has a namespace and a key with a single quote. You write it with the SQL-standard doubled quote.

MEMORY GET FROM agent_mem
    WHERE NS = 'user''s notes' AND KEY = 'draft';

The store can be made with an embedding dimension. A read from that kind of store also returns the embedding column.

CREATE MEMORY STORE kb WITH (embedding_dim = 3, distance = 'cosine');
MEMORY PUT INTO kb
    VALUES ('docs', 'intro', 'welcome', '[0.1, 0.2, 0.3]');

-- returns mem_namespace, mem_key, mem_value, embedding, created_at
MEMORY GET FROM kb
    WHERE NS = 'docs' AND KEY = 'intro';

Compatibility

MEMORY GET is an EvolutionDB extension. It is not part of the SQL standard. PostgreSQL has no matching command. It is a short, easy form for a single-row SELECT … WHERE pk = … LIMIT 1 against the store's backing table.

The set of clauses is kept small on purpose. A general SELECT is flexible, but here the WHERE clause cannot grow. It takes only the fixed NS + KEY equality pair. There is no column list. There is no ORDER BY. There is no LIMIT that you set yourself. The result is always capped at one row. To run any predicate you like over stored memory, query the backing table mem_<store_name> directly with a standard SELECT.
