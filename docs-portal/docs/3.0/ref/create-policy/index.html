CREATE POLICY

Create a row-level security policy. It limits which rows users can read or write on a table.

Synopsis

CREATE POLICY name ON table_name
    [ AS { PERMISSIVE | RESTRICTIVE } ]
    [ FOR { ALL | SELECT | INSERT | UPDATE | DELETE } ]
    [ TO role_name [, ...] ]
    [ USING ( using_expression ) ]
    [ WITH CHECK ( check_expression ) ]

-- Memory-store variant (sugar; resolves to the backing table)
CREATE POLICY name ON MEMORY STORE store_name
    [ AS { PERMISSIVE | RESTRICTIVE } ]
    [ FOR { ALL | SELECT | INSERT | UPDATE | DELETE } ]
    [ TO role_name [, ...] ]
    [ USING ( using_expression ) ]
    [ WITH CHECK ( check_expression ) ]

Description

CREATE POLICY creates a row-level security (RLS) policy. It stores the policy in the system catalog against the target table. A policy holds a row-filter expression. EvolutionDB applies this expression on its own when ordinary users run SELECT, INSERT, UPDATE, or DELETE on the table.

Creating a policy on its own does not change how queries behave. Row-level security only takes effect after you turn it on for the table with ALTER TABLE table_name ENABLE ROW LEVEL SECURITY. While RLS is on, every policy that applies is checked on each affected row. At other times the policies stay inactive.

The admin account is a superuser. It skips all policies. It always sees and can write every row. This is true no matter how many policies exist or whether RLS is on.

The expressions combine like this for a given statement and user.

  • Permissive policies are the default. They are joined with OR. A row is visible or writable if it passes any one permissive policy that applies to the user and command.
  • Restrictive policies are joined with AND. They are added on top of the permissive result. A row must pass every restrictive policy that applies. It must also pass the permissive set.
  • Sometimes no permissive policy applies to the current user and command. Then the result is "no rows". The table looks empty for reads. Writes are rejected. At least one permissive policy must match before any access is allowed.

USING controls which existing rows a command can see. It is added to the query's WHERE filter for SELECT, UPDATE, and DELETE. WITH CHECK controls which new or changed row values can be written by INSERT and UPDATE. A write is rejected if its new row fails the check. When you leave out WITH CHECK, the USING expression is used as the write check. This matches PostgreSQL.

The ON MEMORY STORE store_name form is a shortcut for protecting a memory store. It resolves the store to its backing mem_store_name table. Then it creates an ordinary table policy there. Policy expressions can use the backing columns like any other column. These columns include mem_namespace, mem_key, and mem_value.

Parameters

name
The name of the new policy. Policy names belong to their table. You can use the same name on different tables. If you create a policy with a name that already exists on the same table, you get a duplicate-object error.
table_name
The existing table the policy applies to. If the relation does not exist, the command fails with relation "table_name" does not exist.
ON MEMORY STORE store_name
Apply the policy to a memory store instead of a regular table. EvolutionDB looks up the store. Then it targets the store's backing mem_store_name table. If the store is not found, the command fails with the same relation-not-found error.
AS PERMISSIVE | AS RESTRICTIVE
Sets how the policy combines with others. PERMISSIVE policies are joined with OR. RESTRICTIVE policies are joined with AND on top. When you leave out AS, the policy is PERMISSIVE.
FOR ALL | FOR SELECT | FOR INSERT | FOR UPDATE | FOR DELETE
Limits the policy to one command type, or to all commands. When you leave out FOR, the policy applies to ALL commands. Only the single keywords shown here are accepted. Comma-separated command lists are not supported.
TO role_name [..]
The roles the policy applies to. Roles are users. Role matching is case-insensitive against the connected username. The special role public matches every user. When you leave out TO, the policy applies to all users.
USING ( using_expression )
A boolean expression. It is checked for each row to decide row visibility for SELECT, UPDATE, and DELETE. Rows where it is false are hidden from the user. The expression can use the table's columns. The parsed expression is serialized and stored in the catalog.
WITH CHECK ( check_expression )
A boolean expression. It is checked against the proposed new row for INSERT and UPDATE. A write is rejected if its resulting row fails the check. If you leave it out, the USING expression is used as the check.

Note. A policy does nothing until you run ALTER TABLE table_name ENABLE ROW LEVEL SECURITY. For a memory store, use ALTER MEMORY STORE store_name ENABLE ROW LEVEL SECURITY.

Examples

This example limits a table so each tenant sees only its own rows. It then turns on enforcement.

CREATE POLICY tenant_isolation ON orders
    USING (tenant_id = current_tenant);

ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

This example creates a read-only policy for one role. It also adds a write check that stops back-dated inserts.

CREATE POLICY analyst_read ON events
    FOR SELECT
    TO analyst, auditor
    USING (region = 'EU');

CREATE POLICY no_backdating ON events
    FOR INSERT
    WITH CHECK (created_at >= '2026-01-01');

This example adds a restrictive policy on top of permissive ones. A row must pass both. It applies to everyone through public.

CREATE POLICY hide_archived ON documents
    AS RESTRICTIVE
    TO public
    USING (archived = false);

This example protects a memory store. The policy is created on its backing table.

CREATE POLICY session_scope ON MEMORY STORE sessions
    USING (mem_namespace = owner_namespace);

ALTER MEMORY STORE sessions ENABLE ROW LEVEL SECURITY;

This example removes a policy.

DROP POLICY tenant_isolation ON orders;

Compatibility

CREATE POLICY is not part of the SQL standard. It follows PostgreSQL's row-level security extension. EvolutionDB supports a useful subset of that extension.

  • Supported. AS PERMISSIVE and RESTRICTIVE with permissive as the default. FOR ALL, SELECT, INSERT, UPDATE, and DELETE with all as the default. TO role lists with the public pseudo-role. USING. WITH CHECK. As in PostgreSQL, a missing WITH CHECK falls back to USING. Permissive policies join with OR. Restrictive policies join with AND on top.
  • Differences. FOR accepts only a single command keyword. There is no ALTER POLICY command. To change a policy, drop it and create it again. The ON MEMORY STORE form is an EvolutionDB extension. PostgreSQL has no equivalent. Role matching is case-insensitive against the connected username. The built-in admin account is always a superuser. It bypasses all policies.
  • As in PostgreSQL, policies have no effect until row-level security is enabled on the table. A table with RLS enabled but no permissive policy that applies denies all access to non-superusers.
