CREATE POLICY

Define a row-level security policy that restricts which rows users may read or write on a table.

Synopsis

CREATE POLICY name ON table_name
    [ AS { PERMISSIVE | RESTRICTIVE } ]
    [ FOR { ALL | SELECT | INSERT | UPDATE | DELETE } ]
    [ TO role_name [, ...] ]
    [ USING ( using_expression ) ]
    [ WITH CHECK ( check_expression ) ]

-- Memory-store variant (sugar; resolves to the backing table)
CREATE POLICY name ON MEMORY STORE store_name
    [ AS { PERMISSIVE | RESTRICTIVE } ]
    [ FOR { ALL | SELECT | INSERT | UPDATE | DELETE } ]
    [ TO role_name [, ...] ]
    [ USING ( using_expression ) ]
    [ WITH CHECK ( check_expression ) ]

Description

CREATE POLICY creates a row-level security (RLS) policy and stores it in the system catalog against the target table. A policy names a row-filter expression that EvolutionDB applies automatically when ordinary users run SELECT, INSERT, UPDATE, or DELETE against the table.

Creating a policy by itself does not change query behavior. Row-level security is only enforced once it has been switched on for the table with ALTER TABLE table_name ENABLE ROW LEVEL SECURITY. While RLS is enabled, every applicable policy is consulted on each affected row; policies are otherwise dormant.

The admin account is treated as a superuser and bypasses all policies entirely — it always sees and may write every row, regardless of how many policies exist or whether RLS is enabled.

How the configured expressions combine for a given statement and user:

  • Permissive policies (the default) are combined with OR. A row is visible / writable if it satisfies any one permissive policy that applies to the user and command.
  • Restrictive policies are combined with AND and are layered on top of the permissive result. A row must satisfy every applicable restrictive policy in addition to passing the permissive set.
  • If no permissive policy applies to the current user and command, the effective result is "no rows" — the table appears empty for reads, and writes are rejected. At least one permissive policy must match for any access to be granted.

USING controls which existing rows are visible to a command (it is folded into the query's WHERE filter for SELECT, UPDATE, and DELETE). WITH CHECK controls which new or modified row values may be written by INSERT and UPDATE; a write whose new row fails the check is rejected. When WITH CHECK is omitted, the USING expression is reused as the write check, matching PostgreSQL.

The ON MEMORY STORE store_name form is syntactic sugar for protecting a memory store: it resolves the store to its backing mem_store_name table and creates an ordinary table policy there. Policy expressions can reference the backing columns (such as mem_namespace, mem_key, mem_value) like any other column.

Parameters

name
The name of the new policy. Policy names are scoped to their table; the same name may be reused on different tables. Creating a policy whose name already exists on the same table raises a duplicate-object error.
table_name
The existing table the policy applies to. If the relation does not exist, the command fails with relation "table_name" does not exist.
ON MEMORY STORE store_name
Apply the policy to a memory store instead of a regular table. EvolutionDB looks up the store and targets its backing mem_store_name table. If the store is not found, the command fails with the same relation-not-found error.
AS PERMISSIVE | AS RESTRICTIVE
Selects how the policy combines with others. PERMISSIVE policies are OR-ed together; RESTRICTIVE policies are AND-ed on top. When AS is omitted, the policy is PERMISSIVE.
FOR ALL | FOR SELECT | FOR INSERT | FOR UPDATE | FOR DELETE
Restricts the policy to a single command type, or to all commands. When FOR is omitted, the policy applies to ALL commands. Only the single keywords shown here are accepted — comma-separated command lists are not supported.
TO role_name [, ...]
The roles (users) the policy applies to. Role matching is case-insensitive against the connected username, and the special role public matches every user. When TO is omitted, the policy applies to all users.
USING ( using_expression )
A boolean expression evaluated per row to decide row visibility for SELECT, UPDATE, and DELETE. Rows for which it is false are hidden from the user. The expression may reference the table's columns. The parsed expression is serialized and stored in the catalog.
WITH CHECK ( check_expression )
A boolean expression evaluated against the proposed new row for INSERT and UPDATE. A write whose resulting row does not satisfy the check is rejected. If omitted, the USING expression is used as the check.

Note. A policy is inert until you run ALTER TABLE table_name ENABLE ROW LEVEL SECURITY. For a memory store, use ALTER MEMORY STORE store_name ENABLE ROW LEVEL SECURITY.

Examples

Restrict a table so each tenant sees only its own rows, then enable enforcement:

CREATE POLICY tenant_isolation ON orders
    USING (tenant_id = current_tenant);

ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

A read-only policy for one role plus a write-check that prevents back-dated inserts:

CREATE POLICY analyst_read ON events
    FOR SELECT
    TO analyst, auditor
    USING (region = 'EU');

CREATE POLICY no_backdating ON events
    FOR INSERT
    WITH CHECK (created_at >= '2026-01-01');

A restrictive policy layered on top of permissive ones — a row must pass both. This one applies to everyone via public:

CREATE POLICY hide_archived ON documents
    AS RESTRICTIVE
    TO public
    USING (archived = false);

Protect a memory store; the policy is created on its backing table:

CREATE POLICY session_scope ON MEMORY STORE sessions
    USING (mem_namespace = owner_namespace);

ALTER MEMORY STORE sessions ENABLE ROW LEVEL SECURITY;

Remove a policy:

DROP POLICY tenant_isolation ON orders;

Compatibility

CREATE POLICY is not part of the SQL standard; it follows PostgreSQL's row-level security extension. EvolutionDB implements a practical subset:

  • Supported: AS PERMISSIVE/RESTRICTIVE (default permissive), FOR ALL/SELECT/INSERT/UPDATE/DELETE (default all), TO role lists with the public pseudo-role, USING, and WITH CHECK. As in PostgreSQL, an absent WITH CHECK falls back to USING, permissive policies OR together, and restrictive policies AND on top.
  • Differences: FOR accepts only a single command keyword. There is no ALTER POLICY command — change a policy by dropping and recreating it. The ON MEMORY STORE form is an EvolutionDB extension with no PostgreSQL equivalent. Role matching is performed case-insensitively against the connected username; the built-in admin account is an unconditional superuser that bypasses all policies.
  • As in PostgreSQL, policies have no effect until row-level security is enabled on the table, and a table with RLS enabled but no applicable permissive policy denies all access to non-superusers.
