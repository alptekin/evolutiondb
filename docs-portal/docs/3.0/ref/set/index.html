SET

Set a session user variable. You can also change the active schema for the current connection.

Synopsis

SET @variable_name { = | := } value_expression [ , @variable_name { = | := } value_expression [ , ... ] ]

SET SCHEMA schema_name
SET SCHEMA DEFAULT

Description

SET has two uses in EvolutionDB. The word after the keyword tells them apart.

User variables. SET @name = expr creates or overwrites a user variable for the session. The right-hand side is read as a full SQL value expression. This can include arithmetic, function calls, and other things the expression engine understands. The result is stored under the given name for the life of the connection. You can then use the variable anywhere an expression is allowed. For example, you can use it in a SELECT list or a WHERE clause by writing @name. A variable that was never set returns NULL instead of raising an error. You can set several variables in one statement. Separate the assignments with commas.

You can use either = or := as the assignment operator. No other comparison operator is accepted. For example, SET @x > 1 is a syntax error.

Schema selection. SET SCHEMA schema_name changes the active schema inside the current database. After this, unqualified table and object names resolve against that schema. The schema must already exist in the current database. If it does not, the statement fails with an undefined schema error. The active schema then stays the same. SET SCHEMA DEFAULT switches back to the schema named default.

Both forms apply only to the connection that runs them. Variables and the chosen schema are not shared between connections. They do not survive after you disconnect.

Note. User variables start with a leading @ sigil. For example, @x and @threshold. This is the only way to read or write them. There is no separate command to inspect them. A connection can hold up to 64 different user variables.

Parameters

@variable_name
The name of the session user variable to assign. Write it with a leading @. If you reuse an existing name, the old value is overwritten.
= / :=
The assignment operator. Both forms mean the same thing. They are the only operators accepted in this position.
value_expression
Any SQL value expression. It is evaluated once when the SET statement runs. The computed value is stored in the variable. The stored value is then returned as is wherever you use @variable_name.
schema_name
The name of an existing schema in the current database. It becomes the active schema for the connection. After this, unqualified object names resolve against this schema.
DEFAULT
In the SET SCHEMA DEFAULT form, this picks the schema named default as the active schema.

Examples

This example assigns an integer and reads it back.

SET @x = 42;
SELECT @x;        -- 42

The right-hand side is a full expression. Arithmetic works too.

SET @x = 10 + 20;
SELECT @x;        -- 30

This example uses the colon-equals form to store a string.

SET @name := 'Alice';
SELECT @name;     -- Alice

This example references a variable inside a query.

SET @threshold = 15;
SELECT * FROM uv_test WHERE val > @threshold;

You can overwrite a variable by assigning to the same name again.

SET @x = 1;
SET @x = 99;
SELECT @x;        -- 99

An unset variable evaluates to NULL.

SELECT @undefined_var_xyz;   -- NULL

This example switches the active schema for the connection, then returns to the default.

SET SCHEMA sales;
SET SCHEMA DEFAULT;

Compatibility

The SQL standard has no SET command for run-time parameters. The standard does define SET SCHEMA as part of session management. EvolutionDB's SET SCHEMA schema_name form follows that intent. It selects the schema used to resolve unqualified names. The SET SCHEMA DEFAULT spelling is an EvolutionDB shortcut. It selects the schema named default.

EvolutionDB's user-variable form is SET @name = expr. You can also use := here. This form follows the MySQL session-variable convention, not PostgreSQL. It is not related to PostgreSQL's SET configuration_parameter TO value or SET ... = ... for run-time configuration. EvolutionDB does not implement that family of configuration parameters. It also does not implement the SET SESSION, SET LOCAL, SET TIME ZONE, or SET TRANSACTION clauses. PostgreSQL also has no equivalent of EvolutionDB's @-sigil user variables. In PostgreSQL, such per-session values are usually handled with customized run-time parameters or PL/pgSQL variables.
