SET

Set a session user variable, or change the active schema for the current connection.

Synopsis

SET @variable_name { = | := } value_expression [ , @variable_name { = | := } value_expression [ , ... ] ]

SET SCHEMA schema_name
SET SCHEMA DEFAULT

Description

SET serves two distinct purposes in EvolutionDB, distinguished by what follows the keyword.

User variables. SET @name = expr defines or overwrites a session-scoped user variable. The right-hand side is evaluated as a full SQL value expression (including arithmetic, function calls and other constructs the expression engine understands), and the resulting value is stored under the given name for the lifetime of the connection. The variable can then be referenced anywhere an expression is allowed, for example in a SELECT list or a WHERE clause, by writing @name. Referencing a variable that was never set yields NULL rather than an error. Several variables may be assigned in one statement by separating the assignments with commas.

Either = or := may be used as the assignment operator. No other comparison operator is accepted; for instance SET @x > 1 is a syntax error.

Schema selection. SET SCHEMA schema_name changes the active schema within the current database, so that subsequently unqualified table and object names resolve against that schema. The schema must already exist in the current database; if it does not, the statement fails with an undefined schema error and the active schema is left unchanged. SET SCHEMA DEFAULT switches back to the schema named default.

The scope of both forms is the connection that issues them. Variables and the chosen schema are not shared between connections, and they do not survive disconnection.

Note. User variables are addressed with a leading @ sigil (for example @x, @threshold). This is the only way to read or write them; there is no separate inspection command. A connection may hold up to 64 distinct user variables.

Parameters

@variable_name
The name of the session user variable to assign, written with a leading @. Reuse of an existing name overwrites its previous value.
= / :=
The assignment operator. Both forms are equivalent. They are the only operators accepted in this position.
value_expression
Any SQL value expression. It is evaluated once at the time the SET statement runs, and its computed value is stored in the variable. The stored value is later returned verbatim wherever @variable_name is referenced.
schema_name
The name of an existing schema in the current database to make active for the connection. Subsequent unqualified object names resolve against this schema.
DEFAULT
In the SET SCHEMA DEFAULT form, selects the schema named default as the active schema.

Examples

Assign an integer and read it back:

SET @x = 42;
SELECT @x;        -- 42

The right-hand side is a full expression, so arithmetic works:

SET @x = 10 + 20;
SELECT @x;        -- 30

Use the colon-equals form, and store a string:

SET @name := 'Alice';
SELECT @name;     -- Alice

Reference a variable inside a query:

SET @threshold = 15;
SELECT * FROM uv_test WHERE val > @threshold;

Overwriting reuses the same name:

SET @x = 1;
SET @x = 99;
SELECT @x;        -- 99

An unset variable evaluates to NULL:

SELECT @undefined_var_xyz;   -- NULL

Switch the active schema for the connection, then return to the default:

SET SCHEMA sales;
SET SCHEMA DEFAULT;

Compatibility

There is no SET command in the SQL standard for run-time parameters; the standard defines SET SCHEMA as part of session management, and EvolutionDB's SET SCHEMA schema_name form follows that intent, it selects the schema used to resolve unqualified names. The SET SCHEMA DEFAULT spelling is an EvolutionDB convenience that selects the schema literally named default.

EvolutionDB's user-variable form, SET @name = expr (with := as an accepted alternative), follows the MySQL session-variable convention rather than PostgreSQL. It is unrelated to PostgreSQL's SET configuration_parameter TO value / SET ... = ... for run-time configuration: EvolutionDB does not implement that family of configuration parameters, nor the SET SESSION, SET LOCAL, SET TIME ZONE, or SET TRANSACTION clauses. Likewise, PostgreSQL has no equivalent of EvolutionDB's @-sigil user variables; in PostgreSQL such per-session values are typically handled with customized run-time parameters or PL/pgSQL variables.
