CREATE DOC STORE

Create a document store, a JSON-document collection with an optional embedding column for vector search.

Synopsis

CREATE DOCUMENT STORE [ IF EXISTS ] store_name
    [ WITH ( option [, ...] ) ]

where option is one of:

    embedding_dim = integer
    distance = 'cosine' | 'l2' | 'inner' | 'ip' | 'l1'

Description

CREATE DOCUMENT STORE defines a new document store named store_name. A document store is a thin abstraction over an ordinary EvolutionDB table: under the hood, the engine creates a backing table named doc_<store_name> in the current schema and registers the store in the system catalog. Because the store is backed by a regular table, MVCC visibility, row-level security, and the standard error path all apply to its documents.

The backing table is created with a fixed layout:

  • id, VARCHAR(255), NOT NULL, primary key
  • content, VARCHAR(8000), the document body
  • meta, VARCHAR(8000), a JSON-shaped metadata string (may be NULL)
  • embedding, VECTOR(N), created only when embedding_dim is greater than 0
  • created_at, TIMESTAMP, NOT NULL, set to the UTC time of each write

If embedding_dim is omitted or 0, no embedding column is created and the store is a pure JSON key/value collection; in that case DOCUMENT SEARCH against the store fails because there is no vector column to score. The distance option records which distance function DOCUMENT SEARCH will use when ranking results; it has no effect on stores created without an embedding column.

If a store with the same name already exists, the command raises an error (SQLSTATE 42710) unless IF EXISTS is given, in which case the command does nothing and succeeds.

Note on IF EXISTS: EvolutionDB's grammar spells the idempotent form CREATE DOCUMENT STORE IF EXISTS, not the PostgreSQL-style IF NOT EXISTS. The semantics are the same as PostgreSQL's IF NOT EXISTS: skip creation when a store of that name is already present.

Parameters

IF EXISTS
Do not raise an error if a document store with the same name already exists. The existing store is left unchanged and the command returns success. Note the keyword spelling is IF EXISTS, not IF NOT EXISTS.
store_name
The name of the document store to create. This name is registered in the catalog and also determines the backing table name (doc_<store_name>) created in the current schema.
WITH ( option [..] )
An optional, comma-separated list of creation options. Two options are accepted; any combination or order is allowed.
embedding_dim = integer
The dimensionality of the vector embedding stored for each document. When greater than 0, the backing table gains a VECTOR(N) column named embedding and the store can be queried with DOCUMENT SEARCH ... USING VECTOR. The default is 0, meaning no embedding column is created.
distance = string
The distance function used to rank results during DOCUMENT SEARCH. Accepted values (case-insensitive): 'cosine', 'l2', 'inner' (alias 'ip'), and 'l1'. The default is 'cosine'. Any unrecognized value falls back to 'cosine'. This setting is recorded with the store and only matters when an embedding column exists.

Examples

Create a pure JSON document store with no vector column:

CREATE DOCUMENT STORE notes;

Create a store with 384-dimensional embeddings, ranked by cosine distance (the default):

CREATE DOCUMENT STORE articles
    WITH (embedding_dim = 384);

Create a store with embeddings ranked by L2 distance:

CREATE DOCUMENT STORE products
    WITH (embedding_dim = 768, distance = 'l2');

Create a store only if one does not already exist:

CREATE DOCUMENT STORE IF EXISTS articles
    WITH (embedding_dim = 384, distance = 'cosine');

Write a document and then query the store with a Mongo-style filter:

DOCUMENT WRITE INTO articles
    VALUES ('a1', 'Vector databases explained', '{"lang":"en","tags":["db"]}');

DOCUMENT FILTER FROM articles
    WHERE '{"lang":"en"}' LIMIT 10;

Compatibility

CREATE DOCUMENT STORE is an EvolutionDB extension. There is no document store concept in the SQL standard, and PostgreSQL has no equivalent statement, PostgreSQL applications instead use a jsonb column on an ordinary table, and vector search is provided by the third-party pgvector extension via CREATE EXTENSION and a vector column type. EvolutionDB folds both ideas into a single command and a small set of DOCUMENT verbs.

Differences worth noting:

  • The idempotency keyword is IF EXISTS, whereas PostgreSQL's CREATE statements use IF NOT EXISTS.
  • The backing table schema (id, content, meta, optional embedding, created_at) is fixed and not user-definable; you cannot add arbitrary columns through this command.
  • Only two WITH options are accepted, embedding_dim and distance. PostgreSQL/pgvector tuning options (for example HNSW build parameters such as m or ef_construction) are not part of this statement.
  • Documents are queried, written, and removed with the EvolutionDB-specific DOCUMENT WRITE, DOCUMENT FILTER, DOCUMENT SEARCH, and DOCUMENT DELETE verbs rather than ordinary INSERT/SELECT/DELETE (although those verbs reuse the same execution machinery internally).
