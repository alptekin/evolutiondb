CREATE DOC STORE

Create a document store. A document store is a collection of JSON documents. It can also have an embedding column for vector search.

Synopsis

CREATE DOCUMENT STORE [ IF EXISTS ] store_name
    [ WITH ( option [, ...] ) ]

where option is one of:

    embedding_dim = integer
    distance = 'cosine' | 'l2' | 'inner' | 'ip' | 'l1'

Description

CREATE DOCUMENT STORE makes a new document store named store_name. A document store is a thin layer on top of a normal EvolutionDB table. The engine creates a backing table named doc_<store_name> in the current schema. It also adds the store to the system catalog. The store sits on a normal table. So MVCC visibility, row-level security, and the standard error path all work on its documents.

The backing table always has the same fixed layout. It has these columns.

  • id, VARCHAR(255), NOT NULL, primary key
  • content, VARCHAR(8000), the document body
  • meta, VARCHAR(8000), a JSON metadata string. It may be NULL.
  • embedding, VECTOR(N), made only when embedding_dim is greater than 0
  • created_at, TIMESTAMP, NOT NULL, set to the UTC time of each write

You can leave out embedding_dim or set it to 0. Then no embedding column is made. The store becomes a plain JSON key/value collection. In that case DOCUMENT SEARCH on the store fails. There is no vector column to score. The distance option saves which distance function DOCUMENT SEARCH uses to rank results. It does nothing for stores made without an embedding column.

The command fails with an error if a store with the same name already exists. The error is SQLSTATE 42710. You can add IF EXISTS to avoid this. Then the command does nothing and succeeds.

Note on IF EXISTS: EvolutionDB's grammar uses CREATE DOCUMENT STORE IF EXISTS for the idempotent form. It does not use the PostgreSQL-style IF NOT EXISTS. The meaning is the same as PostgreSQL's IF NOT EXISTS. It skips creation when a store with that name is already there.

Parameters

IF EXISTS
Do not raise an error if a document store with the same name already exists. The store stays the same and the command returns success. The keyword is IF EXISTS, not IF NOT EXISTS.
store_name
The name of the document store to create. This name goes into the catalog. It also sets the backing table name. The backing table is named doc_<store_name> and is made in the current schema.
WITH ( option [..] )
An optional list of creation options. Separate the options with commas. Two options are accepted. You can use them in any order and in any combination.
embedding_dim = integer
The size of the vector embedding stored for each document. When it is greater than 0, the backing table gets a VECTOR(N) column named embedding. You can then query the store with DOCUMENT SEARCH ... USING VECTOR. The default is 0. That means no embedding column is made.
distance = string
The distance function used to rank results during DOCUMENT SEARCH. The values are case-insensitive. Accepted values are 'cosine', 'l2', 'inner', 'ip', and 'l1'. The value 'ip' is an alias for 'inner'. The default is 'cosine'. Any value that is not known falls back to 'cosine'. This setting is saved with the store. It only matters when an embedding column exists.

Examples

This example creates a plain JSON document store with no vector column.

CREATE DOCUMENT STORE notes;

Create a store with 384-dimensional embeddings. Results are ranked by cosine distance. This is the default.

CREATE DOCUMENT STORE articles
    WITH (embedding_dim = 384);

Create a store with embeddings. Rank results by L2 distance.

CREATE DOCUMENT STORE products
    WITH (embedding_dim = 768, distance = 'l2');

This example creates a store only if one does not already exist.

CREATE DOCUMENT STORE IF EXISTS articles
    WITH (embedding_dim = 384, distance = 'cosine');

Write a document. Then query the store with a Mongo-style filter.

DOCUMENT WRITE INTO articles
    VALUES ('a1', 'Vector databases explained', '{"lang":"en","tags":["db"]}');

DOCUMENT FILTER FROM articles
    WHERE '{"lang":"en"}' LIMIT 10;

Compatibility

CREATE DOCUMENT STORE is an EvolutionDB extension. The SQL standard has no document store. PostgreSQL has no command like this one. In PostgreSQL you use a jsonb column on a normal table instead. PostgreSQL gets vector search from the third-party pgvector extension. You add it with CREATE EXTENSION and a vector column type. EvolutionDB puts both ideas into one command and a small set of DOCUMENT verbs.

These are the main differences.

  • The idempotency keyword is IF EXISTS. PostgreSQL's CREATE statements use IF NOT EXISTS.
  • The backing table schema is fixed. It has id, content, meta, an optional embedding, and created_at. You cannot set it yourself. You cannot add extra columns through this command.
  • Only two WITH options are accepted. They are embedding_dim and distance. PostgreSQL and pgvector tuning options are not part of this statement. For example, HNSW build parameters such as m or ef_construction are not allowed.
  • You query, write, and remove documents with the EvolutionDB verbs DOCUMENT WRITE, DOCUMENT FILTER, DOCUMENT SEARCH, and DOCUMENT DELETE. You do not use the normal INSERT, SELECT, and DELETE. The verbs reuse the same execution machinery inside.
