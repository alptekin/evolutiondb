COPY

COPY moves many rows at once between a table and a file or a client stream. It works in CSV or TEXT format.

Synopsis

COPY table_name [ ( column_name [..] ) ]
    { FROM | TO } { 'filename' | STDIN | STDOUT }
    [ [ WITH ] ( option [..] ) ]

where option is one of:
    FORMAT { CSV | TEXT }
    CSV
    DELIMITER 'delimiter_character'
    HEADER [ boolean ]
    NULL 'null_string'
    QUOTE 'quote_character'

Description

COPY moves data between an EvolutionDB table and a file on the operating system. This is server-side file mode. It can also move data between a table and the connected client. This is stream mode. COPY FROM imports rows into a table. COPY TO exports the rows of a table.

When you name a file, the EvolutionDB server process reads or writes that file directly. So the path is found on the server host. The server's filesystem permissions apply. When you give STDIN or STDOUT instead, the data moves over the open client connection.

COPY FROM reads the input one line at a time. It splits each line into fields using the active dialect. Then it inserts each row through the normal insert path. So all the rules that apply to INSERT apply here too. This includes constraints, primary keys, foreign keys, and indexes. In server-side file mode the import runs as one auto-commit unit. Blank lines in the input are skipped. A row may fail for several reasons. It may have a parse error, break a constraint, or have a type mismatch. When that happens, the operation stops and reports the error.

COPY TO scans the whole table. It writes each row using the active dialect. COPY TO only reads. It does not change the table.

Server-side file paths are restricted. A 'filename' given to COPY must be an absolute path. It must begin with /. It must not contain a .. path component. If it does not follow these rules, the command is rejected. Relative paths are not accepted.

Parameters

table_name
The name of an existing table. You copy into it with FROM or out of it with TO. A target table is always required. COPY (query) TO is not supported in this grammar. Only a table name may follow COPY.
( column_name [..] )
An optional list of columns. On COPY FROM, it says which columns the incoming fields go to. It also sets the order. When you leave it out, all columns of the table are used in their declared order.
FROM
Says that data flows into the table. This is an import. Use it with 'filename' or STDIN.
TO
Says that data flows out of the table. This is an export. Use it with 'filename' or STDOUT.
'filename'
The absolute path of the server-side file to read from or write to. On export the file is opened for writing. It is created if needed, or its old content is removed first. On import the file is opened for reading. The path must be absolute. It must not contain a .. component.
STDIN / STDOUT
Moves data over the client connection instead of a server file. Use STDIN with FROM. Use STDOUT with TO. The connection adaptor drives the stream-mode transfer after the statement is parsed.
FORMAT CSV | FORMAT TEXT
Picks the data format. Choosing CSV also sets the active delimiter to a comma ,. It sets the default null marker to the empty string. Choosing TEXT uses a tab delimiter and the \N null marker. If you give no format option, TEXT is the default.
CSV
A short form that means the same as FORMAT CSV.
DELIMITER 'delimiter_character'
The single character that separates fields within each line. Only the first character of the string is used. The default depends on the format. It is a tab for TEXT and a comma for CSV.
HEADER [ boolean ]
Controls a header line of column names. HEADER on its own turns it on. You may also follow HEADER with a boolean. The boolean can be TRUE, FALSE, or UNKNOWN. On COPY TO, a header row of column names is written first. On COPY FROM, the first input line is skipped. It is off by default.
NULL 'null_string'
The string that stands for a SQL NULL value. On import, a field equal to this string is stored as NULL. On export, NULL values are written as this string. The default is \N in TEXT format. It is the empty string in CSV format.
QUOTE 'quote_character'
The single character used to quote field values. Only the first character of the string is used. The default is the double quote ".
You may write options in three ways. The grammar accepts all of them. You can write a bare list with commas. You can write a list inside parentheses ( ... ). You can use the PostgreSQL-style WITH ( ... ) form.

Examples

Export a whole table to a server-side CSV file with a header row.

COPY employees TO '/var/lib/evosql/export/employees.csv'
    WITH (FORMAT CSV, HEADER);

Import a CSV file into specific columns. Skip its header line.

COPY employees (id, name, dept) FROM '/var/lib/evosql/import/new_hires.csv'
    (FORMAT CSV, HEADER TRUE);

Export in tab-delimited TEXT format. This is the default. It uses the \N null marker.

COPY orders TO '/tmp/orders.tsv';

Import with a custom delimiter and a custom null string.

COPY products FROM '/srv/data/products.psv'
    WITH (DELIMITER '|', NULL '(none)');

Stream rows to the connected client instead of a server file.

COPY audit_log TO STDOUT (FORMAT CSV);

Change the quote character for a CSV import.

COPY notes FROM '/data/notes.csv'
    (CSV, QUOTE '~');

Compatibility

The SQL standard has no COPY statement. It is a PostgreSQL extension. EvolutionDB copies part of it. EvolutionDB supports the FROM and TO directions. It supports server-side 'filename' mode. It supports the STDIN and STDOUT stream forms. It supports an optional column list. It supports these options. The options are FORMAT with CSV or TEXT, the bare CSV short form, DELIMITER, HEADER with an optional boolean, NULL, and QUOTE. You may give options bare, in parentheses, or in a WITH ( ... ) list.

Here are the main differences from PostgreSQL.

  • No COPY (query) TO. The source of a COPY is always a table name. The grammar does not accept the result of any SELECT.
  • Smaller option set. Some PostgreSQL options are not built in and are not recognized. These include ESCAPE, FORCE_QUOTE, FORCE_NOT_NULL, FORCE_NULL, ENCODING, FREEZE, ON_ERROR, and PROGRAM.
  • Path restrictions. Server-side filenames must be absolute. They may not contain a .. component. Relative paths are rejected.
  • Boolean keyword for HEADER. EvolutionDB accepts HEADER TRUE, HEADER FALSE, and HEADER UNKNOWN. The PostgreSQL HEADER MATCH mode is not supported.
