CALL

CALL runs a stored procedure. It runs the body one statement at a time.

Synopsis

CALL procedure_name ( [ argument [, ... ] ] )

Description

CALL runs a procedure that you made with CREATE PROCEDURE. EvolutionDB keeps the procedure body as raw SQL text. When you use CALL, the engine finds the procedure in the system catalog. It looks in the current database and the current schema. Then it binds your arguments to the declared parameters. Then it runs the body.

The body is not compiled ahead of time. When you call the procedure, EvolutionDB does these steps.

  1. It finds procedure_name in the current database and schema. If no such procedure exists, the call fails with SQLSTATE 42883 (procedure "name" does not exist).
  2. It reads the stored parameter definitions. It checks that the number of arguments you passed matches the number of declared parameters exactly. If they do not match, the call raises SQLSTATE 42883 (procedure "name" expects N arguments, got M).
  3. It splits the body into single statements. It keeps string literals whole. It also keeps the control-flow blocks whole. These blocks are IF/ELSEIF/ELSE/END IF, WHILE … DO/END WHILE, LOOP/END LOOP, FOR … DO/END FOR, FOREACH … DO/END FOREACH, and CASE/WHEN/END CASE. Then it runs the statements in order.
  4. Before each statement runs, it replaces each declared parameter with the argument value. This works for positional references like $1, $2, and for references by parameter name.

Argument values are put into the procedure's SQL as text. A string argument keeps its quotes. An integer, float, boolean, or NULL is put in as its literal text. A bare identifier is put in exactly as written. A session variable of the form @var is passed through as a reference.

OUT and INOUT parameters. A parameter can be declared OUT or INOUT. If its argument is a session variable like @var, EvolutionDB writes the parameter's final value back into that variable when the procedure returns. It creates the variable if it does not exist yet. For INOUT, the variable's current value is also read in as the incoming argument. Arguments that are not @-variables do not get write-back.

Limits. A CALL takes at most 16 arguments. This matches the maximum of 16 declared procedure parameters. Extra arguments past the limit are ignored at parse time. Each argument value is stored in a 256-byte buffer. The procedure name is also stored in a 256-byte buffer.

Parameters

procedure_name
The plain name of an existing procedure. It is found in the current database and current schema. The CALL grammar does not allow schema-qualified names.
argument
A value passed by position to the matching parameter. The grammar accepts these argument forms.
  • integer literal, for example 42
  • string literal, for example 'hello'. The quotes are kept when the value is put in.
  • approximate-numeric / float literal, for example 3.14
  • NULL, put in as the literal NULL
  • boolean, TRUE or FALSE
  • identifier, a bare NAME, put in exactly as written
  • session variable, @var, which can take OUT/INOUT write-back
You cannot use expressions, function calls, or arithmetic as arguments. Only the literal forms above work. The number of arguments must match the number of declared parameters.

Examples

Create and call a procedure that inserts a row.

CREATE PROCEDURE add_user(p_id INT, p_name VARCHAR(50))
BEGIN
    INSERT INTO users (id, name) VALUES ($1, $2);
END;

CALL add_user(1, 'Ada');

You can use parameters by name instead of by position.

CREATE PROCEDURE raise_salary(emp_id INT, amount INT)
BEGIN
    UPDATE employees SET salary = salary + amount WHERE id = emp_id;
END;

CALL raise_salary(7, 500);

Call with no arguments. You still need the empty parentheses.

CALL nightly_cleanup();

Use an OUT parameter to read a value back into a session variable.

CREATE PROCEDURE count_users(OUT total INT)
BEGIN
    SET total = (SELECT COUNT(*) FROM users);
END;

CALL count_users(@n);
SELECT @n;

Compatibility

CALL follows the idea of the SQL standard <call statement> and the PostgreSQL CALL statement. But the EvolutionDB form is narrower.

  • Argument forms are limited to literals and variable references. PostgreSQL accepts more. EvolutionDB does not accept free-form expressions, function calls, or named-argument syntax like name => value in the argument list.
  • The argument count must match exactly. Omitted parameters do not get default values.
  • Procedure names are not schema-qualified in the CALL grammar. The procedure is always found in the current database and schema.
  • Parameter binding is text substitution. It puts $N and named placeholders into the stored body. It is not a typed argument-binding system.
  • OUT/INOUT write-back targets session variables like @var. This is an EvolutionDB extension. It is not standard SQL behavior.
  • As in PostgreSQL and the SQL standard, only procedures can be called with CALL. These are procedures made with CREATE PROCEDURE. Functions are called inside expressions, not with CALL.
