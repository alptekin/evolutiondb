CALL

CALL — invoke a stored procedure and execute its body statement by statement.

Synopsis

CALL procedure_name ( [ argument [, ... ] ] )

Description

CALL executes a procedure previously defined with CREATE PROCEDURE. EvolutionDB stores a procedure body as raw SQL text; at CALL time the engine looks the procedure up in the system catalog (in the current database and current schema), binds the supplied arguments to the declared parameters, and runs the body.

The body is not pre-compiled. When the procedure is called, EvolutionDB:

  1. Resolves procedure_name in the current database/schema. If no such procedure exists, the call fails with SQLSTATE 42883 (procedure "name" does not exist).
  2. Parses the stored parameter definitions and checks that the number of arguments passed exactly matches the number of declared parameters. A mismatch raises SQLSTATE 42883 (procedure "name" expects N arguments, got M).
  3. Splits the body into individual statements, honoring string literals and the procedural control-flow constructs (IF/ELSEIF/ELSE/END IF, WHILE … DO/END WHILE, LOOP/END LOOP, FOR … DO/END FOR, FOREACH … DO/END FOREACH, and CASE/WHEN/END CASE), then executes them in order.
  4. Substitutes each declared parameter — both positional $1, $2, … references and references by parameter name — with the argument value before each statement is run.

Argument values are substituted as text into the procedure's SQL. A string argument keeps its surrounding quotes, an integer/float/boolean/NULL is substituted as its literal text, a bare identifier is substituted verbatim, and a session variable of the form @var is passed through as a reference.

OUT and INOUT parameters. If a parameter was declared OUT or INOUT and the corresponding argument is a session variable (@var), EvolutionDB writes the parameter's final value back into that variable when the procedure returns (creating the variable if it does not yet exist). For INOUT, the variable's current value is also read in as the incoming argument. Arguments that are not @-variables do not receive write-back.

Limits. A CALL accepts at most 16 arguments (matching the maximum of 16 declared procedure parameters); extra arguments beyond the limit are ignored at parse time. Each argument value is stored in a 256-byte buffer and the procedure name in a 256-byte buffer.

Parameters

procedure_name
The (unqualified) name of an existing procedure. It is resolved in the current database and current schema. Schema-qualified names are not part of the CALL grammar.
argument
A value passed positionally to the procedure's corresponding parameter. The grammar accepts these argument forms:
  • integer literal — e.g. 42
  • string literal — e.g. 'hello' (quotes are preserved for substitution)
  • approximate-numeric / float literal — e.g. 3.14
  • NULL — substituted as the literal NULL
  • boolean — TRUE or FALSE
  • identifier (a bare NAME) — substituted verbatim
  • session variable — @var, eligible for OUT/INOUT write-back
Expressions, function calls, and arithmetic are not accepted as arguments — only the literal forms listed above. The number of arguments must equal the number of declared parameters.

Examples

Create and call a procedure that inserts a row:

CREATE PROCEDURE add_user(p_id INT, p_name VARCHAR(50))
BEGIN
    INSERT INTO users (id, name) VALUES ($1, $2);
END;

CALL add_user(1, 'Ada');

Reference parameters by name instead of by position:

CREATE PROCEDURE raise_salary(emp_id INT, amount INT)
BEGIN
    UPDATE employees SET salary = salary + amount WHERE id = emp_id;
END;

CALL raise_salary(7, 500);

Call with no arguments (empty parentheses are required):

CALL nightly_cleanup();

Use an OUT parameter to read a value back into a session variable:

CREATE PROCEDURE count_users(OUT total INT)
BEGIN
    SET total = (SELECT COUNT(*) FROM users);
END;

CALL count_users(@n);
SELECT @n;

Compatibility

CALL follows the spirit of the SQL standard <call statement> and the PostgreSQL CALL statement, but EvolutionDB's form is narrower:

  • Argument forms are restricted to literals and variable references. Unlike PostgreSQL, EvolutionDB does not accept arbitrary expressions, function calls, or named-argument syntax (name => value) in the argument list.
  • The argument count must match exactly. There is no defaulting of omitted parameters.
  • Procedure names are not schema-qualified in the CALL grammar. The procedure is always resolved in the current database/schema.
  • Parameter binding is textual substitution of $N and named placeholders into the stored body, not a typed argument-binding mechanism.
  • OUT/INOUT write-back targets session variables (@var), which is an EvolutionDB extension rather than standard SQL behavior.
  • As in PostgreSQL and the SQL standard, only procedures (created with CREATE PROCEDURE) may be invoked with CALL; functions are invoked from within expressions, not via CALL.
