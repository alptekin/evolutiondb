CREATE SUBSCRIPTION

Set up a durable mailbox. It keeps every NOTIFY committed on a channel. A consumer can disconnect, come back later, and read the backlog in order.

Synopsis

CREATE SUBSCRIPTION [ IF NOT EXISTS ] subscription_name
    FOR CHANNEL channel_name

Description

CREATE SUBSCRIPTION creates a durable subscription. It sits on top of EvolutionDB's in-memory LISTEN/NOTIFY mechanism. A plain LISTEN only delivers a notification if a session is connected and listening at the moment a NOTIFY fires. Anything sent while the consumer is offline is lost. A subscription fixes this gap. It binds a named, lasting mailbox to a channel.

Once the subscription exists, every NOTIFY that commits on channel_name adds a row to a private backing table. Each row holds a sequence number that always goes up. This is the seq value. The row also holds the notification payload and a posted_at timestamp. The consumer reads the backlog with RESUME SUBSCRIPTION. That command returns every row whose seq is greater than the acknowledged watermark, sorted by seq from low to high. Then ACK SUBSCRIPTION ... UPTO N moves the watermark forward.

Creating the subscription does two things behind the scenes.

  1. It creates a backing table named sub_<subscription_name> in the current schema. The table has these columns. seq BIGINT PRIMARY KEY. payload VARCHAR(8000), which can be null. And posted_at TIMESTAMP NOT NULL.
  2. It writes a catalog entry. The entry records the channel binding and the backing-table id. It records the acknowledgement watermark, called last_ack_seq, which starts at 0. It records the next sequence number to assign, called next_seq, which starts at 1. It also records a retention window of 7 days. The auto-RECLAIM daemon uses this window when it removes acknowledged and expired rows.

Durable writes happen on the NOTIFY commit path. They run after the in-memory listener fan-out. So live consumers do not pay the cost of the on-disk write when no subscription is set up on the channel.

The channel binding matches the channel named in NOTIFY. The match is case-insensitive. A subscription only keeps notifications committed after it is created. It does not go back and pick up notifications that fired earlier.

Parameters

IF NOT EXISTS
Do nothing if a subscription with the same name already exists. It does not raise an error. This does not promise that the existing subscription is bound to the same channel. It only hides the duplicate-name error. The lexer reads the phrase as one NOT EXISTS token. So the spelling is fixed.
subscription_name
The name of the new subscription. The name must be unique among subscriptions. If you try to create a duplicate without IF NOT EXISTS, the command raises SQLSTATE 42710. This means duplicate object. The name is also used to build the backing table name sub_<subscription_name>.
FOR CHANNEL channel_name
The notification channel this subscription keeps. This clause is required. If you leave it out, the command raises a syntax error with SQLSTATE 42601. Every NOTIFY committed on this channel name adds one row to the subscription's mailbox.

Not supported. EvolutionDB's CREATE SUBSCRIPTION has no CONNECTION, PUBLICATION, or WITH (...) clause. The grammar accepts only the two forms shown in the Synopsis. The retention window has a fixed default of 7 days. You cannot change it through the statement right now.

Examples

Create a durable subscription bound to the orders channel.

CREATE SUBSCRIPTION orders_mailbox FOR CHANNEL orders;

Create one only if it does not already exist. This makes the setup idempotent.

CREATE SUBSCRIPTION IF NOT EXISTS orders_mailbox FOR CHANNEL orders;

A producer sends a message to the channel. The payload is kept durably even if no session is listening.

NOTIFY orders, 'order #4821 created';

Later, a consumer reconnects and reads the backlog. Then it acknowledges the last sequence it processed.

RESUME SUBSCRIPTION orders_mailbox;   -- returns (seq, payload, posted_at) for seq > last_ack
ACK SUBSCRIPTION orders_mailbox UPTO 12;

List the registered subscriptions and remove one when you no longer need it.

SHOW SUBSCRIPTIONS;
DROP SUBSCRIPTION [IF EXISTS] orders_mailbox;

Compatibility

CREATE SUBSCRIPTION is not part of the SQL standard.

PostgreSQL also has a CREATE SUBSCRIPTION statement. It does something very different. It is part of logical replication. It subscribes to a publication on a remote server through a CONNECTION string. EvolutionDB's command shares only the keyword. Here it is a local, single-server feature. It adds durability on top of LISTEN/NOTIFY. There is no remote connection, no publication, and no replication slot. EvolutionDB's grammar does not accept the PostgreSQL CONNECTION, PUBLICATION, and WITH options.

The related verbs RESUME SUBSCRIPTION and ACK SUBSCRIPTION ... UPTO read and acknowledge the mailbox. They are EvolutionDB extensions. PostgreSQL has no equivalent.
