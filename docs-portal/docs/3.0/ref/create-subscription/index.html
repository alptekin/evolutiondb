CREATE SUBSCRIPTION

Register a durable mailbox that captures every NOTIFY committed on a channel, so a consumer can disconnect and later drain the backlog in order.

Synopsis

CREATE SUBSCRIPTION [ IF NOT EXISTS ] subscription_name
    FOR CHANNEL channel_name

Description

CREATE SUBSCRIPTION creates a durable subscription on top of EvolutionDB's in-memory LISTEN/NOTIFY mechanism. A plain LISTEN only delivers notifications to a session that is connected and listening at the moment a NOTIFY fires; anything published while the consumer is offline is lost. A subscription removes that gap: it binds a named, persistent mailbox to a channel.

Once the subscription exists, every NOTIFY that commits on channel_name appends a row to a private backing table. Each row carries a monotonically increasing sequence number (seq), the notification payload, and a posted_at timestamp. The consumer reads the backlog with RESUME SUBSCRIPTION, which returns every row whose seq is greater than the acknowledged watermark, ordered by seq ascending, and then advances the watermark with ACK SUBSCRIPTION ... UPTO N.

Behind the scenes, creating the subscription does two things:

  1. It creates a backing table named sub_<subscription_name> in the current schema with the columns seq BIGINT PRIMARY KEY, payload VARCHAR(8000) (nullable), and posted_at TIMESTAMP NOT NULL.
  2. It registers a catalog entry recording the channel binding, the backing-table id, the acknowledgement watermark (last_ack_seq, starting at 0), the next sequence number to assign (next_seq, starting at 1), and a retention window of 7 days that the auto-RECLAIM daemon honors when pruning acknowledged and expired rows.

Durable writes happen on the NOTIFY commit path after the in-memory listener fan-out, so live consumers do not pay the cost of the on-disk append when no subscription is enrolled on the channel.

The channel binding matches case-insensitively against the channel named in NOTIFY. A subscription captures only notifications committed after it is created; it does not back-fill notifications that fired earlier.

Parameters

IF NOT EXISTS
Do nothing (rather than raising an error) if a subscription with the same name already exists. Note that this does not guarantee the existing subscription is bound to the same channel — it only suppresses the duplicate-name error. The phrase is recognized as a single NOT EXISTS token by the lexer, so the spelling is fixed.
subscription_name
The name of the new subscription. The name must be unique among subscriptions; attempting to create a duplicate without IF NOT EXISTS raises SQLSTATE 42710 (duplicate object). The name is also used to derive the backing table name sub_<subscription_name>.
FOR CHANNEL channel_name
The notification channel this subscription captures. This clause is required; omitting it raises a syntax error (SQLSTATE 42601). Every NOTIFY committed on this channel name appends one row to the subscription's mailbox.

Not supported. EvolutionDB's CREATE SUBSCRIPTION has no CONNECTION, PUBLICATION, or WITH (...) clause. The grammar accepts exactly the two forms shown in the Synopsis. The retention window is a fixed default (7 days) and is not currently configurable through the statement.

Examples

Create a durable subscription bound to the orders channel:

CREATE SUBSCRIPTION orders_mailbox FOR CHANNEL orders;

Create one only if it does not already exist (idempotent setup):

CREATE SUBSCRIPTION IF NOT EXISTS orders_mailbox FOR CHANNEL orders;

A producer publishes to the channel; the payload is captured durably even if no session is listening:

NOTIFY orders, 'order #4821 created';

Later, a consumer reconnects and drains the backlog, then acknowledges through the last sequence it processed:

RESUME SUBSCRIPTION orders_mailbox;   -- returns (seq, payload, posted_at) for seq > last_ack
ACK SUBSCRIPTION orders_mailbox UPTO 12;

Inspect registered subscriptions and remove one when it is no longer needed:

SHOW SUBSCRIPTIONS;
DROP SUBSCRIPTION [IF EXISTS] orders_mailbox;

Compatibility

CREATE SUBSCRIPTION is not part of the SQL standard.

PostgreSQL has a CREATE SUBSCRIPTION statement, but it serves an entirely different purpose: it is part of logical replication and subscribes to a publication on a remote server via a CONNECTION string. EvolutionDB's command shares only the keyword. Here it is a local, single-server feature that layers durability onto LISTEN/NOTIFY — there is no remote connection, no publication, and no replication slot. The PostgreSQL CONNECTION, PUBLICATION, and WITH options are not accepted by EvolutionDB's grammar.

The companion verbs RESUME SUBSCRIPTION and ACK SUBSCRIPTION ... UPTO, used to drain and acknowledge the mailbox, are EvolutionDB extensions with no PostgreSQL equivalent.
