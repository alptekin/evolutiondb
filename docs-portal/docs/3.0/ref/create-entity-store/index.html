CREATE ENTITY STORE

Create a named entity store — a key-addressed, upsert-by-key collection of entity summaries with mention counts and last-seen timestamps, backed by an ordinary EvolutionDB table.

Synopsis

CREATE ENTITY STORE [ IF EXISTS ] name

Description

CREATE ENTITY STORE registers a new entity store named name in the current schema. An entity store is a high-level abstraction over a regular EvolutionDB table: the command creates a backing table and a catalog entry that ties the store name to it, so you can later use the entity DML verbs (ENTITY PUT, ENTITY GET, ENTITY RANK) instead of writing raw SQL.

The backing table is created automatically and named ent_name. It has a fixed five-column shape:

entity_key VARCHAR(255) PRIMARY KEY NOT NULL
The lookup key for the entity. Primary key, so keys are unique and indexed by a B+ tree; ENTITY PUT upserts against this key.
summary VARCHAR(8000)
A free-text summary of the entity. Nullable.
facts VARCHAR(8000)
An optional JSON-shaped string of structured facts. Nullable; stored as SQL NULL when omitted.
mention_count INT NOT NULL
How many times the entity has been written. Set to 1 on first ENTITY PUT and incremented on every subsequent put for the same key.
last_seen TIMESTAMP NOT NULL
UTC ISO timestamp (YYYY-MM-DD HH:MM:SS) of the most recent put.

The store is registered in the system catalog as an entity store entry that records its name and the backing table's id. If a store with the same name already exists, the command raises an error (SQLSTATE 42710) unless the IF EXISTS clause is given, in which case the existing store is left untouched and the command succeeds silently. A missing name is reported as a syntax error (SQLSTATE 42601).

Dropping the store with DROP ENTITY STORE drops the backing ent_name table and removes the catalog entry.

Note on the guard clause. The grammar accepts the literal token sequence IF EXISTS here (not the SQL-conventional IF NOT EXISTS). Its effect, however, is the standard "if-not-exists" semantics: when the store already exists the statement is a no-op rather than an error.

Parameters

IF EXISTS
Do not raise an error if an entity store with the same name already exists; instead leave the existing store in place and return success. Without this clause, attempting to create a store that already exists fails with entity store 'name' already exists (SQLSTATE 42710).
name
The name of the entity store to create. The backing table is created as ent_name in the current schema, and the store is registered under this name in the catalog. Required.

Examples

Create an entity store named people:

CREATE ENTITY STORE people;

Create it only if it does not already exist (idempotent setup):

CREATE ENTITY STORE IF EXISTS people;

Create a store and start populating it with the entity DML verbs:

CREATE ENTITY STORE people;

-- upsert by key: summary only
ENTITY PUT INTO people VALUES ('alice', 'Lead engineer on the storage team');

-- upsert by key: summary + JSON-shaped facts
ENTITY PUT INTO people VALUES ('alice', 'Storage team lead', '{"role":"lead","team":"storage"}');

-- look up one entity by key
ENTITY GET FROM people WHERE KEY = 'alice';

-- rank entities by mention_count DESC (tie-broken by last_seen DESC)
ENTITY RANK FROM people LIMIT 10;

Drop the store and its backing table:

DROP ENTITY STORE people;

Compatibility

CREATE ENTITY STORE is an EvolutionDB extension. There is no CREATE ENTITY STORE statement in the SQL standard, and PostgreSQL has no equivalent command — it is not portable.

The clause spelling differs from common SQL conventions: EvolutionDB's grammar accepts IF EXISTS as the create-time guard, whereas PostgreSQL and the SQL community use IF NOT EXISTS for the same "create-only-if-absent" intent. The runtime behavior matches the conventional if-not-exists meaning.

The store is not a distinct storage engine: it is implemented entirely on top of a normal EvolutionDB table (ent_name) with a B+ tree primary-key index on entity_key, and the entity DML verbs are translated into ordinary INSERT, UPDATE, and SELECT execution paths. The backing table is visible to regular SQL queries.
