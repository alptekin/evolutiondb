CREATE MEMORY STORE

Create a new long-term agent memory store, a namespaced key/value store with optional vector embeddings for semantic search.

Synopsis

CREATE MEMORY STORE [ IF NOT EXISTS ] store_name
    [ WITH ( option [, ... ] ) ]

where option is one of:

    embedding_dim = N
    distance = 'cosine' | 'l2' | 'inner' | 'ip' | 'l1'

Description

CREATE MEMORY STORE registers a new memory store in the current schema and provisions its backing table. A memory store is purpose-built for agent long-term memory: each record is addressed by a namespace plus a key, carries an arbitrary JSON value, and may optionally carry a vector embedding for semantic similarity search.

Internally, the store is backed by an ordinary table named mem_<store_name> created in the current schema. That table holds the following columns:

  • mem_namespace, VARCHAR(255) NOT NULL
  • mem_key, VARCHAR(255) NOT NULL
  • pk, VARCHAR(512) PRIMARY KEY, the composite of namespace and key
  • mem_value, JSON (nullable)
  • embedding, VECTOR(N), present only when embedding_dim is greater than 0
  • created_at, TIMESTAMP
  • ttl_at, TIMESTAMP (nullable)

When embedding_dim is omitted or 0, the embedding column is left out entirely and the store behaves as a pure namespaced JSON key/value table. In that case MEMORY SEARCH against the store fails, because there is no embedding column to score against.

Because the backing table is a regular table, everything that applies to tables applies to a memory store: MVCC, primary-key uniqueness on (namespace, key), row-level security policies, triggers, and foreign-key cascades. The MEMORY PUT, GET, SEARCH, DELETE, and LIST NAMESPACES commands are thin wrappers that compile down to ordinary INSERT/SELECT/DELETE against mem_<store_name>.

Creating a store requires that no memory store of the same name already exists. With IF NOT EXISTS, no error is raised if the store is already present; the existing store is left untouched.

Parameters

IF NOT EXISTS
Do nothing (other than emitting no error) if a memory store with the same name already exists. The pre-existing store is not modified, its embedding_dim and distance settings are kept as-is.
store_name
The name of the memory store to create. The backing table is created as mem_<store_name> in the current schema.
embedding_dim = N
The dimensionality of the vector embedding stored alongside each record. When N > 0, an embedding VECTOR(N) column is added to the backing table and the store supports MEMORY SEARCH ... USING VECTOR. When this option is omitted, the default is 0, no embedding column is created, and the store is a plain JSON key/value store.
distance = 'metric'
The distance/similarity metric used by MEMORY SEARCH to rank results against the query vector. This setting is recorded on the store and selects which vector function scores each row at search time. Accepted values:
  • 'cosine': cosine distance (the default if this option is omitted, or if an unrecognized value is supplied).
  • 'l2', Euclidean (L2) distance.
  • 'inner' or 'ip', inner product.
  • 'l1', Manhattan (L1) distance.
The metric is meaningful only when embedding_dim > 0.

Examples

Create a plain JSON key/value memory store (no semantic search):

CREATE MEMORY STORE agent_notes;

Create a store with 1536-dimension embeddings using the default cosine metric:

CREATE MEMORY STORE agent_memory
    WITH (embedding_dim = 1536);

Create a store with embeddings ranked by L2 distance, only if it does not already exist:

CREATE MEMORY STORE IF NOT EXISTS chat_recall
    WITH (embedding_dim = 768, distance = 'l2');

Populate and query a store after creating it:

-- store a value (upsert on namespace+key)
MEMORY PUT INTO agent_memory
    VALUES ('user:42', 'pref:lang', '{"lang":"tr"}', '[0.1,0.2,...]');

-- read it back
MEMORY GET FROM agent_memory
    WHERE NS = 'user:42' AND KEY = 'pref:lang';

-- semantic search (requires embedding_dim > 0)
MEMORY SEARCH agent_memory USING VECTOR '[0.1,0.2,...]' LIMIT 5;

Compatibility

CREATE MEMORY STORE is an EvolutionDB extension. It is not part of the SQL standard, and PostgreSQL has no equivalent statement. The closest PostgreSQL analogue is creating a table and adding a pgvector column by hand; EvolutionDB's memory store rolls the namespaced key/value schema, the optional embedding column, and the distance metric into a single declarative command.

Notes on the EvolutionDB-specific behavior:

  • The WITH clause accepts only the embedding_dim and distance options. There is no storage-parameter syntax beyond these two; PostgreSQL-style table options are not accepted here.
  • The metric values 'cosine', 'l2', 'inner'/'ip', and 'l1' are the only recognized distances; any other string silently falls back to 'cosine'.
  • The backing table mem_<store_name> is a real table and is fully visible to ordinary SQL, so its rows can be inspected, secured, or triggered exactly like any user table.
