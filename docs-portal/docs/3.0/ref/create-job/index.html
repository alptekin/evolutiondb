CREATE JOB

CREATE JOB — register a scheduled job that fires a SQL statement on a cron-style timetable.

Synopsis

CREATE JOB [ IF EXISTS ] name
    ON SCHEDULE 'min hour dom mon dow'
    DO 'sql_statement'

Description

CREATE JOB registers a named scheduled job in the system catalog. A job couples a five-field cron expression with a single SQL statement; when the scheduler observes a wall-clock minute that matches the cron pattern, the job is marked as due.

The cron expression and the SQL body are both supplied as quoted string literals, so whitespace and special characters in either are handled by the lexer's normal quoting rules rather than the SQL grammar. The surrounding quotes are stripped before the values are stored in the catalog.

Newly created jobs are enabled by default and have a null last-run timestamp. Jobs can be turned on and off with ALTER JOB ... ENABLE / ALTER JOB ... DISABLE, removed with DROP JOB, and inspected with SHOW JOBS, which exposes each job's name, cron expression, SQL body, enabled flag, and last_run_unix timestamp.

Execution status. In this release the scheduler performs cron matching, persistence, ENABLE/DISABLE state, and SHOW JOBS introspection. When a job's cron pattern matches the current minute the scheduler stamps the job's last_run_unix field (so you can confirm a tick fired via SHOW JOBS) and logs a line to stderr. Actually running the job's SQL body as a side effect is deferred to a future release; do not rely on the SQL in DO being executed yet.

Parameters

IF EXISTS
Do not raise an error if a job with the same name is already registered. When the named job exists the statement succeeds silently and the existing job is left unchanged; the cron expression and SQL body in the new statement are ignored. (Note the keyword spelling: EvolutionDB's grammar accepts the bare IF EXISTS form here, not PostgreSQL's IF NOT EXISTS.)
name
The identifier of the job to create. Must be unique among scheduled jobs. Without IF EXISTS, creating a job whose name already exists raises an error (SQLSTATE 42710).
ON SCHEDULE 'min hour dom mon dow'
A five-field cron expression given as a quoted string. The fields are minute, hour, day-of-month, month, and day-of-week, separated by spaces (or tabs). Each field is either * (matches any value) or a single non-negative integer. Ranges (1-5), comma lists (1,15,30), and step values (*/5) are not supported.

Valid integer ranges per field: minute 0–59, hour 0–23, day-of-month 1–31, month 1–12, day-of-week 0–7 where both 0 and 7 denote Sunday. Matching is evaluated against UTC. An expression that does not parse to exactly five valid fields raises an error (SQLSTATE 22023).
DO 'sql_statement'
The SQL statement to associate with the job, supplied as a quoted string literal. It is stored verbatim in the catalog. The DO body must be non-empty.

Examples

Create a job that runs every day at 03:00 UTC:

CREATE JOB nightly_rollup
    ON SCHEDULE '0 3 * * *'
    DO 'INSERT INTO daily_totals SELECT today(), sum(amount) FROM orders';

Create a job that fires at the top of every hour:

CREATE JOB hourly_ping
    ON SCHEDULE '0 * * * *'
    DO 'UPDATE heartbeat SET ticked_at = now()';

Create a job only if one of that name does not already exist:

CREATE JOB IF EXISTS weekly_cleanup
    ON SCHEDULE '30 4 * * 0'   -- 04:30 every Sunday (dow 0)
    DO 'DELETE FROM sessions WHERE expired = TRUE';

Inspect, disable, re-enable, and remove a job:

SHOW JOBS;
ALTER JOB hourly_ping DISABLE;
ALTER JOB hourly_ping ENABLE;
DROP JOB [ IF EXISTS ] hourly_ping;

Compatibility

CREATE JOB is not part of the SQL standard, which has no notion of scheduled jobs.

It is also not a PostgreSQL command — core PostgreSQL has no built-in scheduler. The closest PostgreSQL analog is the third-party pg_cron extension, whose cron.schedule('name', '0 3 * * *', 'SQL') function call uses the same five-field cron convention. EvolutionDB exposes this as native DDL rather than a function, and its cron evaluator is intentionally minimal:

  • Exactly five fields, each either * or a single integer. No ranges, comma lists, or step values (unlike pg_cron / standard crontab).
  • Day-of-week accepts 0 and 7 for Sunday (POSIX cron convention).
  • Matching is done in UTC.
  • The IF EXISTS spelling here differs from PostgreSQL's usual IF NOT EXISTS guard on CREATE statements.
  • In this release the job's SQL body is matched and timestamped but not yet executed; see the note under Description.
