CREATE MESSAGE LOG

Create an append-only message log, a managed store for chat / conversation history keyed by session.

Synopsis

CREATE MESSAGE LOG name
    [ WITH ( ttl = 'N days' ) ]

Description

CREATE MESSAGE LOG registers a new append-only message log. A message log is purpose-built for storing conversational history: agent chat turns, session transcripts, and similar append-heavy workloads, addressed by a session identifier rather than by raw SQL row operations.

Under the hood the command creates an ordinary backing table named ml_name in the current schema and registers an entry in the system catalog that links the log name to that table. The backing table is defined with the following fixed columns:

seq, BIGINT
Primary key, auto-incrementing (start 1, step 1). Establishes the append order of messages.
session_id, VARCHAR(255) NOT NULL
The session the message belongs to. All read, truncate, and count operations filter on this column.
role, VARCHAR(64)
The message author / role (for example user, assistant, system).
content, VARCHAR(8000)
The message body.
meta, VARCHAR(8000)
Optional metadata, stored as a JSON-shaped string. NULL when omitted on append.
created_at, TIMESTAMP NOT NULL
ISO timestamp (UTC) generated by the engine at append time.

Because the log is backed by a regular table, all of EvolutionDB's existing constraint, MVCC, and snapshot machinery continues to apply: appends route through the standard INSERT path (filling seq via auto-increment), reads through SELECT, and truncation through DELETE.

The keyword LOG is not a reserved word in the lexer; it is checked in the parser action. Writing CREATE MESSAGE name name with any word other than LOG in that position raises expected LOG after MESSAGE in CREATE MESSAGE LOG.

Once created, the log is populated and queried with the companion MESSAGE DML statements: MESSAGE APPEND, MESSAGE READ, MESSAGE TRUNCATE, and MESSAGE COUNT.

Parameters

name
The name of the message log to create. The backing table is created as ml_name in the current schema. If a message log with this name already exists, an error is raised (SQLSTATE 42710, message log 'name' already exists).
WITH ( ttl = 'N days' )
Optional. Sets a time-to-live retention hint, in days, recorded with the log. The literal is parsed by taking the first integer it contains, so both '7 days' and a bare '7' resolve to 7. When the clause is omitted the TTL defaults to 0 (no TTL recorded). ttl is the only option accepted inside the WITH (...) list.

Examples

Create a basic message log:

CREATE MESSAGE LOG chat_history;

Create a log with a 7-day retention hint:

CREATE MESSAGE LOG agent_sessions WITH (ttl = '7 days');

Append messages to a session, then read them back:

MESSAGE APPEND INTO chat_history
    VALUES ('sess-42', 'user', 'What is EvolutionDB?');

-- with optional JSON metadata as a 4th value
MESSAGE APPEND INTO chat_history
    VALUES ('sess-42', 'assistant', 'A SQL engine in C.', '{"tokens":7}');

-- read the whole session (ordered by seq)
MESSAGE READ FROM chat_history WHERE SESSION = 'sess-42';

-- read only the most recent 10 turns
MESSAGE READ FROM chat_history WHERE SESSION = 'sess-42' LAST 10;

Count and prune a session:

-- how many messages in the session
MESSAGE COUNT FROM chat_history WHERE SESSION = 'sess-42';

-- delete everything for the session
MESSAGE TRUNCATE FROM chat_history WHERE SESSION = 'sess-42';

-- delete only rows older than seq 1000 (seq < 1000)
MESSAGE TRUNCATE FROM chat_history WHERE SESSION = 'sess-42' BEFORE 1000;

Remove the log (and its backing table):

DROP MESSAGE LOG chat_history;
DROP MESSAGE LOG IF EXISTS chat_history;

Compatibility

CREATE MESSAGE LOG is an EvolutionDB extension. It is not part of the SQL standard, and it has no equivalent in PostgreSQL, PostgreSQL has no MESSAGE LOG object type or MESSAGE DML family. The closest standard approach in any SQL system would be to hand-roll an equivalent table and write your own INSERT/SELECT/DELETE queries; CREATE MESSAGE LOG simply packages that pattern behind a session-oriented command set.

Note that EvolutionDB's WITH clause here is intentionally minimal: the only recognized option is ttl, and unlike PostgreSQL storage-parameter lists there is no comma-separated set of additional options.
