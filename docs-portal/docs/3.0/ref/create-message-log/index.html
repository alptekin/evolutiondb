CREATE MESSAGE LOG

Create an append-only message log. This is a managed store for chat and conversation history. It is keyed by session.

Synopsis

CREATE MESSAGE LOG name
    [ WITH ( ttl = 'N days' ) ]

Description

CREATE MESSAGE LOG registers a new append-only message log. A message log is built for storing conversation history. This means agent chat turns, session transcripts, and similar work where you mostly add new rows. You reach the data by a session identifier and not by raw SQL row operations.

The command creates a normal backing table named ml_name in the current schema. It also adds an entry in the system catalog. That entry links the log name to the table. The backing table has these fixed columns.

seq, BIGINT
Primary key. It auto-increments and starts at 1 with a step of 1. It sets the append order of messages.
session_id, VARCHAR(255) NOT NULL
The session that the message belongs to. All read, truncate, and count operations filter on this column.
role, VARCHAR(64)
The message author or role. For example user, assistant, or system.
content, VARCHAR(8000)
The message body.
meta, VARCHAR(8000)
Optional metadata. It is stored as a JSON-shaped string. It is NULL when you leave it out on append.
created_at, TIMESTAMP NOT NULL
An ISO timestamp in UTC. The engine creates it at append time.

The log is backed by a regular table. So all of EvolutionDB's existing constraint, MVCC, and snapshot machinery still applies. Appends go through the standard INSERT path and fill seq by auto-increment. Reads go through SELECT. Truncation goes through DELETE.

The keyword LOG is not a reserved word in the lexer. The parser action checks it instead. If you write CREATE MESSAGE name name with any word other than LOG in that spot, it raises expected LOG after MESSAGE in CREATE MESSAGE LOG.

After you create the log, you fill it and query it with the companion MESSAGE DML statements. These are MESSAGE APPEND, MESSAGE READ, MESSAGE TRUNCATE, and MESSAGE COUNT.

Parameters

name
The name of the message log to create. The backing table is created as ml_name in the current schema. If a message log with this name already exists, an error is raised. The SQLSTATE is 42710 and the message is message log 'name' already exists.
WITH ( ttl = 'N days' )
Optional. It sets a time-to-live retention hint in days. The hint is recorded with the log. The parser reads the literal by taking the first integer it contains. So both '7 days' and a bare '7' resolve to 7. When you leave out the clause, the TTL defaults to 0 and no TTL is recorded. ttl is the only option accepted inside the WITH (...) list.

Examples

This example creates a basic message log.

CREATE MESSAGE LOG chat_history;

This example creates a log with a retention hint of 7 days.

CREATE MESSAGE LOG agent_sessions WITH (ttl = '7 days');

This example appends messages to a session and then reads them back.

MESSAGE APPEND INTO chat_history
    VALUES ('sess-42', 'user', 'What is EvolutionDB?');

-- with optional JSON metadata as a 4th value
MESSAGE APPEND INTO chat_history
    VALUES ('sess-42', 'assistant', 'A SQL engine in C.', '{"tokens":7}');

-- read the whole session (ordered by seq)
MESSAGE READ FROM chat_history WHERE SESSION = 'sess-42';

-- read only the most recent 10 turns
MESSAGE READ FROM chat_history WHERE SESSION = 'sess-42' LAST 10;

This example counts and prunes a session.

-- how many messages in the session
MESSAGE COUNT FROM chat_history WHERE SESSION = 'sess-42';

-- delete everything for the session
MESSAGE TRUNCATE FROM chat_history WHERE SESSION = 'sess-42';

-- delete only rows older than seq 1000 (seq < 1000)
MESSAGE TRUNCATE FROM chat_history WHERE SESSION = 'sess-42' BEFORE 1000;

This example removes the log and its backing table.

DROP MESSAGE LOG chat_history;
DROP MESSAGE LOG IF EXISTS chat_history;

Compatibility

CREATE MESSAGE LOG is an EvolutionDB extension. It is not part of the SQL standard. It has no equivalent in PostgreSQL. PostgreSQL has no MESSAGE LOG object type and no MESSAGE DML family. In any SQL system, the closest standard way would be to build your own table by hand and write your own INSERT, SELECT, and DELETE queries. CREATE MESSAGE LOG just wraps that pattern in a session-oriented command set.

EvolutionDB's WITH clause here is kept small on purpose. The only option it knows is ttl. Unlike PostgreSQL storage-parameter lists, it has no comma-separated set of extra options.
