BEGIN / COMMIT / ROLLBACK

Open, commit, or abort a multi-statement transaction so that a group of changes either all take effect together or none of them do.

Synopsis

BEGIN
START TRANSACTION

COMMIT
END

ROLLBACK

SAVEPOINT savepoint_name
ROLLBACK TO [ SAVEPOINT ] savepoint_name
RELEASE [ SAVEPOINT ] savepoint_name

Description

In EvolutionDB every statement runs in its own implicit, auto-committed transaction unless you have first opened an explicit transaction block. BEGIN (or its synonym START TRANSACTION) starts such a block: from that point every data-modifying statement is recorded in a per-connection undo log and is held open until you finish the block with COMMIT or ROLLBACK.

COMMIT makes all the changes made since BEGIN durable and visible to other connections. Before acknowledging the commit, the server flushes the transaction's dirty pages to the write-ahead log, assigns a commit sequence number, and marks the transaction committed in the commit log (CLOG). The undo log is then discarded and any locks the transaction held are released. END is accepted as an exact synonym for COMMIT.

ROLLBACK abandons the transaction: the undo log is replayed in reverse, restoring every inserted, updated, or deleted row to its prior state, the transaction is marked aborted in the CLOG, and its locks are released. Under MVCC, rolling back an UPDATE or DELETE works by clearing the soft-delete marker (xmax) on the original tuple rather than rewriting data, with a re-insert fallback for pre-MVCC rows.

Each connection has its own transaction context (the undo log is thread-local), so transactions on different connections are independent. If a connection disconnects with a transaction still open, the server automatically rolls it back.

Note. If a statement inside the block raises an error, the transaction enters an aborted state. While aborted, only ROLLBACK, COMMIT, SAVEPOINT, and RELEASE are accepted; other statements are rejected until the block is closed. Issuing COMMIT on an aborted transaction performs an implicit ROLLBACK (all work is discarded), matching PostgreSQL.

Isolation level

The transaction isolation level is not set on the BEGIN statement itself. EvolutionDB matches BEGIN/START TRANSACTION by prefix, so any words you type after the keyword are accepted but ignored. To choose an isolation level, run SET [SESSION] TRANSACTION ISOLATION LEVEL before opening (or before issuing any query inside) the transaction. The recognized levels are READ UNCOMMITTED, READ COMMITTED (the default), REPEATABLE READ, and SERIALIZABLE.

At READ COMMITTED each statement takes a fresh snapshot. At REPEATABLE READ and SERIALIZABLE, BEGIN takes a single snapshot that the whole transaction sees. A SERIALIZABLE transaction additionally registers with the conflict guard for write-skew detection and may fail at COMMIT with SQLSTATE 40001 (could not serialize access due to read/write dependencies among transactions), in which case the transaction is rolled back and the command reports ROLLBACK.

Savepoints

Within an open transaction you may set named savepoints. SAVEPOINT name marks the current position in the undo log. ROLLBACK TO [SAVEPOINT] name replays the undo log back to that mark — undoing everything done since the savepoint while leaving earlier work and the transaction itself intact — and clears the aborted state so the transaction can continue. RELEASE [SAVEPOINT] name forgets the savepoint (and all newer ones) without undoing any work. Defining a savepoint with the name of an existing one overwrites it. Savepoint commands outside a transaction fail with SQLSTATE 25P01; referencing a savepoint that does not exist fails with 3B001.

Parameters

BEGIN
Start an explicit transaction block on the current connection. Initializes a fresh undo log and savepoint stack and assigns the transaction an MVCC transaction id. The command reports the tag BEGIN. Issuing BEGIN while already inside a transaction logs a warning and is otherwise a no-op (it does not start a nested transaction).
START TRANSACTION
An exact synonym for BEGIN; behaves identically.
COMMIT
End the current transaction, making all of its changes permanent. Flushes WAL, records the commit in the CLOG with a commit sequence number, releases locks, discards the undo log, and delivers any pending notifications. Reports the tag COMMIT. Run outside a transaction it is a harmless no-op.
END
An exact synonym for COMMIT.
ROLLBACK
End the current transaction, undoing all changes made since BEGIN by replaying the undo log in reverse, marking the transaction aborted, and releasing locks. Reports the tag ROLLBACK. Run outside a transaction it is a harmless no-op.
SAVEPOINT savepoint_name
Establish a new savepoint at the current point within the open transaction. Names are case-insensitive; re-using an existing name overwrites that savepoint.
ROLLBACK TO [ SAVEPOINT ] savepoint_name
Undo all changes made after the named savepoint was established, without ending the transaction. The SAVEPOINT keyword is optional. Savepoints created after the target are discarded, and the transaction's aborted flag (if any) is cleared.
RELEASE [ SAVEPOINT ] savepoint_name
Destroy the named savepoint and all savepoints created after it, keeping the work done since then. The SAVEPOINT keyword is optional.

Examples

Transfer money between two accounts atomically:

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

Discard a set of changes instead of keeping them:

BEGIN;
DELETE FROM staging_rows;
-- changed your mind:
ROLLBACK;

Use a savepoint to undo only part of a transaction:

BEGIN;
INSERT INTO orders (id, total) VALUES (1, 50);
SAVEPOINT before_items;
INSERT INTO order_items (order_id, sku) VALUES (1, 'BAD-SKU');
ROLLBACK TO SAVEPOINT before_items;   -- keep the order, drop the bad item
COMMIT;

Choose an isolation level before opening the transaction:

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
SELECT count(*) FROM inventory;
-- the same snapshot is seen for the whole transaction
SELECT sum(qty) FROM inventory;
COMMIT;

END is interchangeable with COMMIT:

BEGIN;
UPDATE settings SET value = 'on' WHERE key = 'feature_x';
END;

Compatibility

EvolutionDB's transaction control follows the spirit of the SQL standard and is broadly compatible with PostgreSQL, with some intentional simplifications.

  • Both BEGIN and the SQL-standard START TRANSACTION are accepted and behave identically. END is accepted as a synonym for COMMIT (a PostgreSQL extension); the standard only defines COMMIT.
  • Optional noise words PostgreSQL allows on these statements (WORK, TRANSACTION, AND CHAIN, etc.) are not parsed as distinct clauses. EvolutionDB matches the leading keyword and ignores any trailing text, so writing COMMIT WORK behaves the same as COMMIT — but the extra words have no effect and AND CHAIN does not start a new transaction.
  • The transaction characteristics that PostgreSQL permits on the BEGIN line (ISOLATION LEVEL ..., READ ONLY/READ WRITE, DEFERRABLE) are not read from the BEGIN statement. Set the isolation level beforehand with SET [SESSION] TRANSACTION ISOLATION LEVEL; there is no read-only or deferrable transaction mode.
  • Nested BEGIN is a no-op that emits a warning rather than an error (matching PostgreSQL), and COMMIT/ROLLBACK outside a transaction are harmless no-ops.
  • SAVEPOINT, ROLLBACK TO [SAVEPOINT], and RELEASE [SAVEPOINT] are supported with PostgreSQL-compatible semantics, including overwriting a savepoint when its name is reused and clearing the aborted state on ROLLBACK TO.
  • A failed statement aborts the whole transaction; subsequent statements are refused until ROLLBACK (or a ROLLBACK TO a prior savepoint). This matches PostgreSQL's failed-transaction behavior, and COMMIT on an aborted transaction performs an implicit rollback.
  • Two-phase commit is available through the separate XA command family (XA START/END/PREPARE/COMMIT/ROLLBACK/RECOVER), not through a PREPARE TRANSACTION statement.
