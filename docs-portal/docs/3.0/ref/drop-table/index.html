DROP TABLE

Remove a table — together with its rows, indexes, constraints and access policies — from the current database.

Synopsis

DROP TABLE [ IF EXISTS ] table_name [ CASCADE | RESTRICT ]

Description

DROP TABLE removes a single table and everything physically attached to it. EvolutionDB resolves table_name through the system catalog, then tears down its storage and metadata in one operation:

  • the primary-key B+ tree is destroyed;
  • all heap pages holding the table's rows are freed back to the page manager;
  • every secondary index — B+ tree as well as hash / unique-hash indexes — is destroyed;
  • the catalog entry is dropped, which also removes the table's columns, indexes and constraints;
  • the table's inheritance link (if it was a child of another table) is scrubbed;
  • any row-level security policies attached to the table are removed.

Only one table may be named per statement. EvolutionDB's grammar accepts a single table_name; there is no comma-separated list (that form is reserved for TRUNCATE).

Before any storage is touched, two dependency checks run:

  • Inheritance children. If the table has child tables that inherit from it, the drop is refused unless CASCADE is given. With CASCADE, each child (and recursively each grandchild) is dropped first, then the parent.
  • Views. If any view in the current database references the table by name, the drop is refused. This check is not overridden by CASCADE — a depended-on view must be dropped explicitly first.

A successful DROP TABLE reports command(s) completed successfully!... A refused drop raises SQLSTATE 2BP01 (dependent objects still exist).

Parameters

IF EXISTS
Do not raise an error if the table does not exist. EvolutionDB silently reports success instead. Without this clause, dropping a missing table is an error (table 'name' not found).
table_name
The name of the table to drop. The name is resolved against the current database and schema via the catalog. Exactly one name is accepted.
CASCADE
Automatically drop tables that inherit from this table, recursing through the whole inheritance subtree, before dropping the named table. Note that CASCADE covers inheritance children only; it does not force the removal of dependent views — a view that references the table still blocks the drop.
RESTRICT
Refuse to drop the table if any other object (an inheritance child) depends on it. This is the default behavior when neither keyword is supplied.

Examples

Drop a table:

DROP TABLE employees;

Drop a table only if it exists, suppressing the error otherwise:

DROP TABLE IF EXISTS staging_import;

Drop a parent table along with every table that inherits from it:

-- 'people' has child tables 'employees' and 'customers'
DROP TABLE people CASCADE;

Explicitly refuse to drop a table that still has children (the default):

DROP TABLE people RESTRICT;
-- ERROR: cannot drop table "people" because other objects depend on it

Compatibility

DROP TABLE is defined by the SQL standard. EvolutionDB implements a focused subset, and its behavior is closest to PostgreSQL with the following honest differences:

  • Single table only. The SQL standard and PostgreSQL allow a comma-separated list of tables (DROP TABLE a, b, c). EvolutionDB's grammar accepts exactly one table_name per statement.
  • CASCADE scope. In PostgreSQL, CASCADE also drops dependent objects such as views and foreign-key constraints in other tables. In EvolutionDB, CASCADE follows the inheritance tree (child tables) only; a referencing view still blocks the drop and must be removed first.
  • RESTRICT is the default, matching the SQL standard and PostgreSQL.
  • IF EXISTS matches PostgreSQL: a missing table is reported as success rather than an error.
  • EvolutionDB does not support the PostgreSQL-only DROP TABLE ... CASCADE CONSTRAINTS, schema-qualified multi-object drops, or the TEMPORARY keyword in this command.
