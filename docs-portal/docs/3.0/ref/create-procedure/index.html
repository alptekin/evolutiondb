CREATE PROCEDURE

Define a new stored procedure whose body is a block of SQL statements that is stored as raw text and replayed statement-by-statement when the procedure is invoked with CALL.

Synopsis

CREATE [ OR REPLACE ] PROCEDURE name ( [ argument [, ...] ] ) [ AS ]
BEGIN
    statements
END

where argument is:

    [ IN | OUT | INOUT ] arg_name arg_type

Description

CREATE PROCEDURE registers a named procedure in the current database and schema. Unlike many engines, EvolutionDB does not compile the body into an intermediate form: the raw text between the opening BEGIN and its matching END is captured verbatim from the original SQL input and stored in the system catalog. At CALL time the stored text is split into individual statements and each is executed through the normal query path.

The procedure is created in the database and schema that are current at definition time, resolved through the system catalog. If a procedure with the same name already exists in that schema, the statement fails with a "procedure already exists" error (SQLSTATE 42P07) unless OR REPLACE is given, in which case the existing definition is dropped first and re-created.

The body is delimited by a balanced BEGIN … END pair. Nested blocks are tracked while scanning for the closing keyword: each inner BEGIN increases the nesting depth and each standalone END decreases it, while compound terminators such as END IF, END WHILE, END LOOP, END FOR, END FOREACH and END CASE are recognized and do not close the procedure. The outer END at depth zero terminates the definition. Text inside single-quoted string literals is ignored during this scan.

Parameters declared in the argument list become substitution targets inside the body. When the procedure is invoked, positional placeholders of the form $1, $2, … and bare references to a parameter's name are replaced with the corresponding argument value supplied to CALL before each statement is executed. References that match nothing are left untouched, and text inside string literals is copied verbatim without substitution.

The body may contain ordinary SQL statements as well as control-flow constructs, IF … END IF, WHILE … END WHILE, LOOP … END LOOP, FOR … END FOR, FOREACH … END FOREACH and CASE … END CASE, which the engine splits and resolves into a block map at call time.

Argument modes (IN, OUT, INOUT) are accepted by the grammar and recorded, but procedures are driven entirely by textual parameter substitution at CALL time; there is no separate output-binding mechanism. A procedure with no parameters still requires the empty parentheses.

Parameters

OR REPLACE
Replace an existing procedure of the same name in the current schema instead of raising an error. When omitted, defining over an existing procedure fails with SQLSTATE 42P07.
name
The identifier of the procedure to create. The procedure is registered under the current database and schema.
IN | OUT | INOUT
Optional mode of the argument that follows. When no mode keyword is given, the argument defaults to IN. The mode is stored with the parameter but does not change the textual-substitution execution model.
arg_name
The name of an argument. Inside the body it may be referenced either by this name or by its positional placeholder ($1 for the first argument, $2 for the second, and so on). Name matching is case-insensitive.
arg_type
The declared data type of the argument (for example INT, VARCHAR(50), TEXT, BOOLEAN). It documents the expected argument and is parsed as a normal column-style type.
AS
Optional noise keyword permitted immediately before BEGIN. The definition is identical whether or not AS is present.
statements
The procedure body: a sequence of SQL statements and control-flow constructs between BEGIN and the matching outer END. Statements are separated by semicolons. The text is stored as-is and executed at CALL time.

Examples

A procedure with no arguments that performs an insert:

CREATE PROCEDURE add_default_user()
BEGIN
    INSERT INTO users (name, active) VALUES ('guest', TRUE);
END;

A procedure that uses named parameters, substituted into the body at call time:

CREATE PROCEDURE raise_salary(dept_id INT, pct INT) AS
BEGIN
    UPDATE employees
       SET salary = salary + (salary * pct / 100)
     WHERE department = dept_id;
END;

-- invoke it
CALL raise_salary(3, 10);

The same body using positional placeholders instead of names:

CREATE PROCEDURE raise_salary(dept_id INT, pct INT)
BEGIN
    UPDATE employees
       SET salary = salary + (salary * $2 / 100)
     WHERE department = $1;
END;

Redefining an existing procedure with OR REPLACE and a control-flow body:

CREATE OR REPLACE PROCEDURE archive_or_skip(min_age INT)
BEGIN
    IF min_age > 0 THEN
        DELETE FROM sessions WHERE age > min_age;
    END IF;
END;

Compatibility

EvolutionDB's CREATE PROCEDURE resembles the SQL/PSM persistent-stored-modules form and MySQL's procedure syntax more than PostgreSQL's. Differences worth noting:

  • No language clause. PostgreSQL requires LANGUAGE and wraps the body in a dollar-quoted or single-quoted string. EvolutionDB has no LANGUAGE clause and no body string-quoting: the body is written directly as a BEGIN … END block, and AS is merely an optional noise word.
  • Textual parameters. Arguments are applied by literal text substitution ($n placeholders or parameter names) rather than as bound, typed values. Argument types and IN/OUT/INOUT modes are recorded but do not establish typed binding or output parameters.
  • Unsupported PostgreSQL options. Clauses such as SECURITY DEFINER/INVOKER, SET configuration, TRANSFORM, EXTERNAL and explicit argmode-defaulted parameters with DEFAULT values are not part of EvolutionDB's grammar and are not accepted.
  • Single-name resolution. Procedures are not overloadable by argument signature; they are identified by name within a database and schema, and DROP PROCEDURE takes only the bare name (no argument list).

The companion CREATE FUNCTION form shares this grammar and adds a RETURNS type clause. Both are stored and executed by the same engine.
