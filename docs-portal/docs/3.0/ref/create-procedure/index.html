CREATE PROCEDURE

Create a new stored procedure. The body is a block of SQL statements. EvolutionDB stores it as raw text. It runs the statements one by one when you call the procedure with CALL.

Synopsis

CREATE [ OR REPLACE ] PROCEDURE name ( [ argument [, ...] ] ) [ AS ]
BEGIN
    statements
END

where argument is:

    [ IN | OUT | INOUT ] arg_name arg_type

Description

CREATE PROCEDURE adds a named procedure to the current database and schema. EvolutionDB does not compile the body. It keeps the raw text between the opening BEGIN and its matching END exactly as you wrote it. It saves this text in the system catalog. When you run CALL, EvolutionDB splits the stored text into single statements. It runs each one through the normal query path.

The procedure is created in the database and schema that are current when you define it. EvolutionDB resolves these through the system catalog. A procedure with the same name may already exist in that schema. In that case the statement fails with a "procedure already exists" error. The SQLSTATE is 42P07. To avoid the error, use OR REPLACE. Then EvolutionDB drops the old definition first and creates it again.

The body sits between a matching BEGIN and END pair. EvolutionDB tracks nested blocks while it looks for the closing keyword. Each inner BEGIN raises the nesting depth. Each plain END lowers it. Some terminators are compound, such as END IF, END WHILE, END LOOP, END FOR, END FOREACH and END CASE. EvolutionDB knows these and they do not close the procedure. The outer END at depth zero ends the definition. EvolutionDB ignores text inside single-quoted string literals during this scan.

Parameters in the argument list become substitution targets inside the body. When you call the procedure, EvolutionDB replaces two things before it runs each statement. It replaces positional placeholders like $1 and $2. It also replaces plain references to a parameter's name. The values come from the arguments you pass to CALL. A reference that matches nothing stays as it is. Text inside string literals is copied as-is with no substitution.

The body may hold ordinary SQL statements. It may also hold control-flow blocks. These include IF … END IF, WHILE … END WHILE, LOOP … END LOOP, FOR … END FOR, FOREACH … END FOREACH and CASE … END CASE. The engine splits these and builds a block map at call time.

The grammar accepts argument modes. These are IN, OUT and INOUT. EvolutionDB records them. But procedures run only by text substitution at CALL time. There is no separate way to bind output values. A procedure with no parameters still needs the empty parentheses.

Parameters

OR REPLACE
Replace a procedure that already has the same name in the current schema. This way you do not get an error. If you leave it out, defining over an existing procedure fails with SQLSTATE 42P07.
name
The name of the procedure to create. EvolutionDB adds it under the current database and schema.
IN | OUT | INOUT
The optional mode of the argument that follows. If you give no mode keyword, the argument is IN by default. EvolutionDB stores the mode with the parameter. It does not change how text substitution runs.
arg_name
The name of an argument. Inside the body you can refer to it by this name. You can also use its positional placeholder. $1 is the first argument. $2 is the second argument, and so on. Name matching is case-insensitive.
arg_type
The data type of the argument. For example INT, VARCHAR(50), TEXT or BOOLEAN. It shows the kind of argument you expect. EvolutionDB parses it like a normal column type.
AS
An optional keyword you may place right before BEGIN. It has no effect. The definition is the same with or without AS.
statements
The procedure body. It is a series of SQL statements and control-flow blocks between BEGIN and the matching outer END. You separate statements with semicolons. EvolutionDB stores the text as-is. It runs the text at CALL time.

Examples

A procedure with no arguments that runs an insert.

CREATE PROCEDURE add_default_user()
BEGIN
    INSERT INTO users (name, active) VALUES ('guest', TRUE);
END;

A procedure that uses named parameters. EvolutionDB puts them into the body at call time.

CREATE PROCEDURE raise_salary(dept_id INT, pct INT) AS
BEGIN
    UPDATE employees
       SET salary = salary + (salary * pct / 100)
     WHERE department = dept_id;
END;

-- invoke it
CALL raise_salary(3, 10);

The same body with positional placeholders instead of names.

CREATE PROCEDURE raise_salary(dept_id INT, pct INT)
BEGIN
    UPDATE employees
       SET salary = salary + (salary * $2 / 100)
     WHERE department = $1;
END;

Redefining a procedure with OR REPLACE and a control-flow body.

CREATE OR REPLACE PROCEDURE archive_or_skip(min_age INT)
BEGIN
    IF min_age > 0 THEN
        DELETE FROM sessions WHERE age > min_age;
    END IF;
END;

Compatibility

The EvolutionDB CREATE PROCEDURE looks more like the SQL/PSM persistent stored modules form and the MySQL procedure syntax than the PostgreSQL one. Here are the main differences.

  • No language clause. PostgreSQL needs LANGUAGE. It also wraps the body in a dollar-quoted or single-quoted string. EvolutionDB has no LANGUAGE clause. It does not quote the body. You write the body straight as a BEGIN … END block. AS is just an optional word with no effect.
  • Textual parameters. EvolutionDB applies arguments by plain text substitution. It uses $n placeholders or parameter names. It does not bind them as typed values. EvolutionDB records argument types and the IN, OUT and INOUT modes. But they do not set up typed binding or output parameters.
  • Unsupported PostgreSQL options. The EvolutionDB grammar does not include some clauses. These include SECURITY DEFINER/INVOKER, SET configuration, TRANSFORM and EXTERNAL. It also does not allow parameters with DEFAULT values. EvolutionDB does not accept any of these.
  • Single-name resolution. You cannot overload procedures by argument signature. EvolutionDB finds a procedure by its name within a database and schema. DROP PROCEDURE takes only the bare name. It needs no argument list.

The related CREATE FUNCTION form uses this same grammar. It adds a RETURNS type clause. The same engine stores and runs both.
