CREATE USER

Make a new database user that can log in with a password.

Synopsis

CREATE USER name PASSWORD 'password'

Description

CREATE USER adds a new login account to the EvolutionDB instance. The account is stored in the system catalog. It lives in the CAT_SYS_USERS B+ tree inside the unified storage file. The account is keyed by user name. The password is kept only as a PBKDF2-SHA256 hash. The plain text password is never written to disk. It is also wiped from memory in a safe way after hashing.

In EvolutionDB the statement always has the same fixed form. You write the user name, then the keyword PASSWORD, then a password value. The name and the password are both required. An empty name is rejected. An empty password is rejected too. There are no role attribute options. For example, you cannot use SUPERUSER, LOGIN, CREATEDB, or VALID UNTIL. These clauses are not part of EvolutionDB's grammar. They are not accepted.

The new account can connect right away. It can use either wire protocol. That means PostgreSQL on port 5433 or the native EVO protocol on port 9967. A new user has no privileges at first. It only gets the access you give it later. Use GRANT to assign access. User lookup at login time is not case sensitive. The authenticator first looks for the lower-case name. If that is not found, it then looks for the name exactly as stored.

Note. Most EvolutionDB statements go through the Flex/Bison SQL parser. CREATE USER does not. The protocol adaptor's catalog layer catches it first. It reads the statement as a plain command string. This is why only the exact CREATE USER name PASSWORD 'password' shape is accepted.

Parameters

name
The name of the new user. It is the first word after CREATE USER. A space ends the word. The name is stored in the system catalog exactly as written. The longest allowed name is 255 characters. Login matches the name in lower case first. So for login, the name is not case sensitive.
PASSWORD 'password'
Required. The login password for the new user. You can write it inside single quotes, like 'secret'. You can also write it as a plain word. A quoted value keeps any spaces up to the closing quote. The value is hashed with PBKDF2-SHA256. Each user gets a random salt, and the hash uses 100 000 iterations. It is stored as pbkdf2:<iterations>:<salt>:<hash>. An empty password is rejected with an invalid-parameter error.

Examples

This example creates a user with a password in single quotes.

CREATE USER reporting PASSWORD 'r3port!ng_2026';

Create a user, then grant it access to a table. Access is a separate step.

CREATE USER analyst PASSWORD 'an4lyst';
GRANT SELECT ON orders TO analyst;

This example connects as the new user with the EVO CLI.

./cli/evosql-cli -U analyst -W   # prompts for the password

Trying to create a user that already exists fails with an error.

CREATE USER analyst PASSWORD 'whatever';
-- ERROR: User 'analyst' already exists  (SQLSTATE 42710)

Omitting the PASSWORD clause is a syntax error.

CREATE USER guest;
-- ERROR: Syntax: CREATE USER <name> PASSWORD '<pass>'  (SQLSTATE 42601)

Compatibility

The SQL standard has no CREATE USER statement. The standard defines roles with CREATE ROLE instead. EvolutionDB's form is a small part of the PostgreSQL extension. This was done on purpose.

  • Password is required. PostgreSQL allows CREATE USER with no password and many optional attributes. EvolutionDB needs exactly PASSWORD 'password'. It accepts nothing else.
  • No role attributes. PostgreSQL has options such as SUPERUSER, CREATEDB, CREATEROLE, LOGIN/NOLOGIN, CONNECTION LIMIT, VALID UNTIL, and IN ROLE. These are not part of EvolutionDB's grammar. They will not parse.
  • No password-encryption keywords. PostgreSQL accepts PASSWORD '...' with optional ENCRYPTED/UNENCRYPTED hints. EvolutionDB always hashes the value with PBKDF2-SHA256. It stores only the hash.
  • Users and roles. EvolutionDB also has a separate CREATE ROLE form. Roles made that way are marked as non-login. They cannot log in. A user made with CREATE USER can log in.
