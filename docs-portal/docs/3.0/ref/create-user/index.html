CREATE USER

Define a new database user (login account) authenticated by a password.

Synopsis

CREATE USER name PASSWORD 'password'

Description

CREATE USER adds a new login account to the EvolutionDB instance. The account is stored in the system catalog (the CAT_SYS_USERS B+ tree inside the unified storage file), keyed by user name, with the supplied password kept only as a PBKDF2-SHA256 hash, the plaintext password is never written to disk and is securely wiped from memory after hashing.

In EvolutionDB the statement is recognized as a fixed two-clause form: the user name, followed by the keyword PASSWORD and a password literal. Both the name and the password are mandatory; supplying an empty name or an empty password is rejected. There are no role attribute options (such as SUPERUSER, LOGIN, CREATEDB, or VALID UNTIL), those clauses are not part of EvolutionDB's grammar and are not accepted.

The created account can immediately connect over either wire protocol (PostgreSQL on port 5433 or the native EVO protocol on port 9967). A freshly created user holds no privileges beyond what it is later granted; use GRANT to assign access. User lookup at authentication time is case-insensitive: the authenticator first searches for the lower-cased name and then falls back to the name exactly as stored.

Note — Unlike most EvolutionDB statements, CREATE USER is not processed by the Flex/Bison SQL parser. It is intercepted by the protocol adaptor's catalog layer and parsed as a literal command string, which is why only the exact CREATE USER name PASSWORD 'password' shape is accepted.

Parameters

name
The name of the new user. Read as the first whitespace-delimited token after CREATE USER and stored verbatim in the system catalog. Maximum length is 255 characters. Because authentication matching lower-cases the name before lookup, user names are effectively case-insensitive for login purposes.
PASSWORD 'password'
Required. The login password for the new user. The literal may be written single-quoted ('secret') or as a bare token; a quoted value preserves spaces up to the closing quote. The value is hashed with PBKDF2-SHA256 (random per-user salt, 100 000 iterations) and stored as pbkdf2:<iterations>:<salt>:<hash>. An empty password is rejected with an invalid-parameter error.

Examples

Create a user with a single-quoted password:

CREATE USER reporting PASSWORD 'r3port!ng_2026';

Create a user, then grant it access to a table (privileges are separate):

CREATE USER analyst PASSWORD 'an4lyst';
GRANT SELECT ON orders TO analyst;

Connecting as the new user with the EVO CLI:

./cli/evosql-cli -U analyst -W   # prompts for the password

Attempting to create a user that already exists fails:

CREATE USER analyst PASSWORD 'whatever';
-- ERROR: User 'analyst' already exists  (SQLSTATE 42710)

Omitting the PASSWORD clause is a syntax error:

CREATE USER guest;
-- ERROR: Syntax: CREATE USER <name> PASSWORD '<pass>'  (SQLSTATE 42601)

Compatibility

There is no CREATE USER statement in the SQL standard; the standard defines roles via CREATE ROLE. EvolutionDB's form is a deliberately narrow subset of the PostgreSQL extension.

  • Password is mandatory. PostgreSQL allows CREATE USER with no password and many optional attributes; EvolutionDB requires exactly PASSWORD 'password' and accepts nothing else.
  • No role attributes. PostgreSQL options such as SUPERUSER, CREATEDB, CREATEROLE, LOGIN/NOLOGIN, CONNECTION LIMIT, VALID UNTIL, and IN ROLE are not part of EvolutionDB's grammar and will not parse.
  • No password-encryption keywords. PostgreSQL accepts PASSWORD '...' with optional ENCRYPTED/UNENCRYPTED hints; EvolutionDB always hashes the supplied value with PBKDF2-SHA256 and stores only the hash.
  • Users vs. roles. EvolutionDB also has a separate CREATE ROLE form; roles created that way are flagged as non-login and cannot authenticate, whereas a user created with CREATE USER can log in.
