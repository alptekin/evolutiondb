RECLAIM

RECLAIM, remove dead row versions and tidy up a table's heap storage

Synopsis

RECLAIM TABLE table_name

Description

RECLAIM frees the storage used by dead row versions. It also reorganizes the heap pages of a single table on disk. EvolutionDB does not overwrite or remove rows right away on DELETE or UPDATE. Instead it keeps the old versions as MVCC tuples. It keeps them until no running transaction can still see them. RECLAIM cleans up those old versions. It also tightens the on-disk layout that they leave behind.

RECLAIM TABLE does the following work on the named table, in this order.

  • Dead tuple removal. EvolutionDB walks the heap page chain and removes every dead MVCC tuple. A tuple is dead in three cases. First, the transaction that inserted the row was aborted. Second, the transaction that inserted the row crashed and never committed, so the row was left in progress. Third, the transaction that deleted or updated the row has committed, and it is older than the oldest transaction that is still active. Each removed row is also purged from the primary-key B-tree and from every secondary index. Any TOAST overflow chain for the row is freed too.
  • XID freezing. Some live rows were inserted by a transaction id that is old enough. It is older than the freeze cutoff. For those rows, the xmin value is rewritten to the frozen marker. This lets the old transaction ids be reused. It also protects the table against transaction-id wraparound. The table's frozen_xid statistic moves forward to match.
  • Index compaction. The primary-key B-tree is compacted. The deletions above leave tombstones, and earlier lazy deletes leave them too. Those tombstones are physically dropped. Nodes that hold too few entries are merged. Index pages that become empty are freed.
  • Page compaction. Each surviving heap page is compacted inside itself. The live records are packed together. The free space in the page is made contiguous.
  • Page merging and freeing. Live records from sparse pages at the end are moved forward into earlier pages that have room. The primary-key entries for the moved rows are repointed to their new location. Pages that become completely empty are returned to the free list. The heap page chain is then rebuilt over the surviving pages.
  • Dropped-column rewrite. A table may have columns that were logically removed by ALTER TABLE ... DROP COLUMN. If so, every row is rewritten without those columns. The dropped-column metadata is also removed from the catalog. This frees the per-row space that the dropped columns still used.

When the operation finishes, the dead-tuple counter for the table is reset. A summary line is then printed. It reports how many dead tuples were removed, how many tuples were frozen, how many bytes were compacted, how many records were moved, how many pages were freed, and how many heap pages remain.

A table may have never received any data, so it has no heap pages. In that case, RECLAIM reports that there is nothing to do and returns successfully.

Automatic RECLAIM. EvolutionDB can run the same procedure on its own. A background path triggers RECLAIM for a table once its dead-tuple count crosses a threshold. When statistics are available, the threshold is more than 20% of the row count. When no statistics have been gathered, the threshold is more than 50 dead tuples. Running RECLAIM TABLE by hand does exactly the same work on demand.

Parameters

table_name
The name of the table to reclaim. Exactly one table name is required. The grammar does not accept a list of tables. It also does not accept any options or a parenthesized argument list. The name may not match an existing table. In that case, the command fails with an "undefined table" error.

Examples

Reclaim a single table after a large batch of deletes and updates.

RECLAIM TABLE orders;

A common maintenance cycle. First delete the old rows. Then reclaim the space they used.

DELETE FROM sessions WHERE last_seen < '2026-01-01';
RECLAIM TABLE sessions;

Reclaim a table after dropping a column. This physically removes the dropped column from every row.

ALTER TABLE products DROP COLUMN legacy_notes;
RECLAIM TABLE products;

Compatibility

The SQL standard has no RECLAIM statement. Storage maintenance of this kind is left fully to the implementation.

RECLAIM TABLE is closest in idea to PostgreSQL's VACUUM. It also matches the freezing behavior of VACUUM FREEZE. It removes dead tuples, freezes old transaction ids, and reclaims index and heap space. It differs from PostgreSQL in some important ways.

  • The keyword is RECLAIM TABLE, not VACUUM. EvolutionDB does not accept the VACUUM keyword.
  • EvolutionDB's grammar accepts only the bare form RECLAIM TABLE table_name. It does not support a parenthesized option list like PostgreSQL's (FULL, ANALYZE, VERBOSE, FREEZE). It does not accept multiple table names. It does not accept a column list.
  • PostgreSQL's ordinary VACUUM reclaims space lazily. It rarely returns pages to the operating system. RECLAIM TABLE works differently. It always compacts and merges heap pages, and it frees emptied pages. This behavior is closer to VACUUM FULL. Freezing happens in the same pass, so a separate FREEZE option is not needed.
  • There is no ANALYZE built into RECLAIM. Statistics collection is a separate concern.
