RECLAIM

RECLAIM — remove dead row versions and defragment a table's heap storage

Synopsis

RECLAIM TABLE table_name

Description

RECLAIM reclaims storage occupied by dead row versions and physically reorganizes the heap pages of a single table. EvolutionDB never overwrites or immediately removes rows on DELETE or UPDATE; instead it keeps old versions (MVCC tuples) until no running transaction can still see them. RECLAIM is the operation that sweeps up those obsolete versions and tightens the on-disk layout that they leave behind.

Running RECLAIM TABLE performs the following work, in order, against the named table:

  • Dead tuple removal. The heap page chain is walked and every dead MVCC tuple is removed: rows inserted by a transaction that aborted, rows left in-progress by a crashed (never-committed) transaction, and rows whose deleting or updating transaction has committed and is older than the oldest still-active transaction. Each removed row is also purged from the primary-key B-tree and from every secondary index, and any associated TOAST overflow chain is freed.
  • XID freezing. Live rows whose inserting transaction id is old enough (older than the freeze cutoff) have their xmin rewritten to the frozen marker. This lets the original transaction ids be recycled and protects the table against transaction-id wraparound. The table's frozen_xid statistic is advanced accordingly.
  • Index compaction. The primary-key B-tree is compacted: tombstones left by the deletions above (and by prior lazy deletes) are physically dropped, underflowing nodes are merged, and emptied index pages are freed.
  • Page compaction. Each surviving heap page is compacted internally so that its live records are packed and the free space within the page is made contiguous.
  • Page merging and freeing. Live records from sparse tail pages are moved forward into earlier pages that have room, the primary-key entries for moved rows are repointed to their new location, completely empty pages are returned to the free list, and the heap page chain is rebuilt over the surviving pages.
  • Dropped-column rewrite. If the table has columns that were logically removed by ALTER TABLE ... DROP COLUMN, every row is rewritten without those columns and the dropped-column metadata is removed from the catalog. This reclaims the per-row space that the dropped columns still occupied.

When the operation completes, the dead-tuple counter for the table is reset and a summary line is printed reporting how many dead tuples were removed, how many tuples were frozen, how many bytes were compacted, how many records were moved, how many pages were freed, and how many heap pages remain.

If the table has never received any data (it has no heap pages), RECLAIM reports that there is nothing to do and returns successfully.

Automatic RECLAIM. EvolutionDB can run the same procedure automatically. A background path triggers RECLAIM for a table once its dead-tuple count crosses a threshold — more than 20% of the row count when statistics are available, or more than 50 dead tuples in absolute terms when no statistics have been gathered. Issuing RECLAIM TABLE manually performs exactly the same work on demand.

Parameters

table_name
The name of the table to reclaim. Exactly one table name is required; the grammar accepts no list of tables and no options or parenthesized argument list. If the name does not resolve to an existing table, the command fails with an "undefined table" error.

Examples

Reclaim a single table after a large batch of deletes and updates:

RECLAIM TABLE orders;

A typical maintenance cycle — delete obsolete rows, then reclaim the space they occupied:

DELETE FROM sessions WHERE last_seen < '2026-01-01';
RECLAIM TABLE sessions;

Reclaim a table after dropping a column, to physically remove the dropped column from every row:

ALTER TABLE products DROP COLUMN legacy_notes;
RECLAIM TABLE products;

Compatibility

There is no RECLAIM statement in the SQL standard; storage maintenance of this kind is left entirely to the implementation.

RECLAIM TABLE is conceptually closest to PostgreSQL's VACUUM (and the freezing behavior of VACUUM FREEZE): it removes dead tuples, freezes old transaction ids, and reclaims index and heap space. It differs from PostgreSQL in important ways:

  • The keyword is RECLAIM TABLE, not VACUUM. EvolutionDB does not accept the VACUUM keyword.
  • EvolutionDB's grammar accepts only the bare form RECLAIM TABLE table_name. It does not support a parenthesized option list (such as PostgreSQL's (FULL, ANALYZE, VERBOSE, FREEZE)), it does not accept multiple table names, and it does not accept a column list.
  • Unlike PostgreSQL's ordinary VACUUM, which reclaims space lazily and rarely returns pages to the operating system, RECLAIM TABLE always compacts and merges heap pages and frees emptied pages — behavior closer to VACUUM FULL. Freezing is performed as part of the same pass rather than requiring a separate FREEZE option.
  • There is no equivalent of ANALYZE built into RECLAIM; statistics collection is a separate concern.
