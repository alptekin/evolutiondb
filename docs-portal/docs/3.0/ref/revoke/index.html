REVOKE

Remove access privileges or role membership previously granted to a database user.

Synopsis

REVOKE { ALL [ PRIVILEGES ] | privilege [, ...] }
    ON { DATABASE | SCHEMA | TABLE } scope_name
    FROM user_name

-- Role membership form
REVOKE role_name FROM user_name

where privilege is one of:

SELECT | INSERT | UPDATE | DELETE | CREATE | DROP

Description

REVOKE removes one or more previously granted privileges from a user, or removes a user's membership in a role. It is the inverse of GRANT.

EvolutionDB privileges are stored in the system catalog as a single grant entry per (user, scope) pair. When you revoke privileges, EvolutionDB locates the matching grant for the named user and scope, removes the requested privilege bits, and then:

  • If privileges remain, the grant entry is rewritten with the reduced privilege set.
  • If no privileges remain, the grant entry is deleted entirely.
  • If ALL (or ALL PRIVILEGES) is revoked, any WITH GRANT OPTION flag on the entry is also cleared.

If no grant entry exists for the named user and scope, the command fails with an error (SQLSTATE 42704, undefined object) reporting that no matching grant was found.

Scopes. A privilege is attached to exactly one of three scope kinds — DATABASE, SCHEMA, or TABLE — identified by scope_name. The revoke must target the same scope kind and name that the original grant used; revoking a TABLE-scoped privilege does not affect a broader DATABASE- or SCHEMA-scoped grant for the same user.

PUBLIC grants are additive. EvolutionDB follows the PostgreSQL additive model: privileges granted to PUBLIC are checked independently from user-specific grants. A user-specific REVOKE removes only that user's own grant entry; it does not mask or override a privilege the user still holds through a PUBLIC grant. To withdraw a privilege from everyone, revoke it from PUBLIC as well.

Role form. When the command contains no ON keyword, it is interpreted as a role-membership revoke: REVOKE role_name FROM user_name removes the user's membership in the named role.

Who may revoke. The admin user may always revoke. A non-admin user may revoke a privilege only if it holds that same privilege WITH GRANT OPTION on the same scope; otherwise the command is denied (SQLSTATE 42501). Role-membership revokes may only be performed by admin.

On success the command reports the tag REVOKE for the privilege form, or REVOKE ROLE for the role-membership form.

Parameters

ALL [ PRIVILEGES ]
Revoke every privilege EvolutionDB tracks for the scope: SELECT, INSERT, UPDATE, DELETE, CREATE, and DROP. The optional keyword PRIVILEGES is accepted and ignored, matching SQL-standard noise-word usage. Revoking ALL also clears the grant's WITH GRANT OPTION flag.
privilege
A specific privilege to remove. Recognized values are SELECT, INSERT, UPDATE, DELETE, CREATE, and DROP. Multiple privileges may be listed separated by commas. Privilege names are case-insensitive. Tokens that are not recognized are silently ignored; if the resulting privilege set is empty, the command fails with an invalid privilege specification error.
ON DATABASE | SCHEMA | TABLE scope_name
Selects the scope kind and names the object the grant applies to. Exactly one of DATABASE, SCHEMA, or TABLE must be given. scope_name may be written unquoted, single-quoted ('name'), or double-quoted ("name"). It must match the scope kind and name used by the original GRANT.
FROM user_name
The user (or PUBLIC) whose privileges or role membership are being removed. May be written unquoted or single-quoted.
role_name
In the role-membership form (no ON clause), the name of the role whose membership is being withdrawn from user_name. If the user is not a member of that role, the command fails.

Not supported. EvolutionDB's REVOKE does not accept the SQL-standard GRANT OPTION FOR clause, the CASCADE / RESTRICT behavior keywords, column-level privilege lists, REVOKE ... FROM PUBLIC shorthand beyond naming PUBLIC as the user, or revoking from multiple users in one statement. Revoking a single privilege from ALL requires it to be present in the existing grant.

Examples

Remove the ability to insert into a table while leaving read access intact:

REVOKE INSERT ON TABLE orders FROM reporting;

Revoke several privileges at once:

REVOKE INSERT, UPDATE, DELETE ON TABLE orders FROM reporting;

Withdraw all privileges on a schema (this also clears any grant option the user held):

REVOKE ALL PRIVILEGES ON SCHEMA sales FROM analyst;

Revoke a database-wide privilege from everyone by targeting PUBLIC:

REVOKE SELECT ON DATABASE testdb FROM PUBLIC;

Use a quoted scope name when it is not a plain identifier:

REVOKE SELECT ON TABLE "Quarterly Report" FROM analyst;

Remove a user's membership in a role (admin only):

REVOKE analysts FROM alice;

Compatibility

EvolutionDB's REVOKE covers the most common case of the SQL standard and of PostgreSQL — removing object privileges from a grantee — but differs in several respects:

  • Scope keyword is mandatory. Unlike the SQL standard / PostgreSQL, which infer the object class, EvolutionDB requires an explicit ON DATABASE, ON SCHEMA, or ON TABLE keyword.
  • No CASCADE / RESTRICT. The standard's dependent-privilege behavior keywords are not accepted. Revoking does not recursively withdraw privileges that the grantee may have re-granted to others.
  • No GRANT OPTION FOR. You cannot revoke only the grant option while keeping the underlying privilege. Revoking ALL clears the grant option; revoking individual privileges leaves the flag unchanged on whatever remains.
  • No column-level privileges and no multi-grantee lists.
  • Privilege set. EvolutionDB tracks only SELECT, INSERT, UPDATE, DELETE, CREATE, and DROP. PostgreSQL privileges such as TRUNCATE, REFERENCES, TRIGGER, CONNECT, TEMPORARY, EXECUTE, and USAGE are not modeled.
  • Additive PUBLIC model. As in PostgreSQL, a user-specific revoke does not override a privilege still held via a grant to PUBLIC.
  • Role revoke is admin-only and uses the same flat REVOKE role FROM user form as PostgreSQL, but without ADMIN OPTION FOR or cascade semantics.
