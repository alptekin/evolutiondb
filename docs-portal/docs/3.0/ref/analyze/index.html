ANALYZE

ANALYZE — scan a table to collect statistics and column histograms used by the query planner.

Synopsis

ANALYZE TABLE table_name
    [ WITH SAMPLE n { PERCENT | ROWS } ]
    [ UPDATE HISTOGRAM ON column [, ...] [ WITH n BUCKETS ]
      | DROP HISTOGRAM ON column [, ...] ]

table_name may be schema-qualified as schema.table.

Description

ANALYZE TABLE performs a sequential scan of table_name and records statistics in the system catalog. Two clauses are optional and independent: the WITH SAMPLE clause controls how many rows are inspected, and the histogram clause controls which per-column histograms are built or removed. The statistics are read by EXPLAIN when it estimates row counts and chooses access paths.

With no optional clause, ANALYZE TABLE t rebuilds every statistic for the table:

  • Table-level statistics — row_count, page_count (heap pages, counted by walking the page chain), and last_analyzed (the wall-clock time of this run).
  • Per-column statistics — null_count, distinct_count, min, max, and avg_width for each column.
  • Per-column histograms — one auto-selected histogram per column that has at least one non-NULL value.
  • Per-index statistics — for every index on the table, the B+ tree depth, leaf-page count, total entries, and average entries per leaf.

Histograms come in two flavors, chosen automatically per column. When the number of distinct values (NDV) is no greater than the bucket budget, a frequency histogram is built (one bucket per value, giving exact equality selectivity). Otherwise an equi-depth histogram is built: distinct values are sorted (numerically when every value parses as a number, otherwise lexicographically) and grouped into buckets holding roughly the same number of rows, each storing an upper bound and a cumulative count. The default bucket budget is 100, and no histogram may exceed 100 buckets.

Distinct-value tracking is capped at 10,000 values per column; columns with more distinct values are summarized from that capped sample. Statistics are stored in the catalog regardless of how many rows were scanned.

On success the server prints a one-line summary of the rows scanned, columns analyzed, and pages counted (and, when sampling, the sampled-row count and keep percentage).

Parameters

table_name
The table to analyze, optionally schema-qualified (schema.table). The table must exist; otherwise the command fails with ERROR: table "…" does not exist.
WITH SAMPLE n PERCENT
Inspect each row with probability n% (Bernoulli sampling). n is clamped to the range 1–100. Sampled per-column counters (null count, width sums) are scaled back up to full-table estimates; min and max are always computed from the full scan because doing so is free.
WITH SAMPLE n ROWS
Target approximately n sampled rows. n has a minimum of 1. The target is converted into a keep percentage using the table's previously recorded row_count; if no prior row count is available (no earlier ANALYZE), the command falls back to a full scan.
(no WITH SAMPLE clause)
Automatic sampling: a full scan is used for tables under 10,000 rows; larger tables (per the previously recorded row_count) are sampled at 10%. Without a prior row count, a full scan is performed.
UPDATE HISTOGRAM ON column [, ...]
Rebuild histograms only for the listed columns and skip rewriting the per-column statistics. The table is still scanned so the listed columns' value distributions can be collected.
WITH n BUCKETS
Used together with UPDATE HISTOGRAM to request n buckets instead of the default of 100. Values above 100 are reduced to 100.
DROP HISTOGRAM ON column [, ...]
Delete the stored histograms for the listed columns. This is a catalog-only operation: no table scan is performed and no other statistics are touched.

Examples

Rebuild all statistics, histograms, and index statistics for a table:

ANALYZE TABLE orders;

Analyze a schema-qualified table:

ANALYZE TABLE sales.orders;

Analyze a large table using a 10% Bernoulli sample:

ANALYZE TABLE events WITH SAMPLE 10 PERCENT;

Aim for roughly 5,000 sampled rows:

ANALYZE TABLE events WITH SAMPLE 5000 ROWS;

Rebuild histograms for two columns with a custom bucket count, without touching other statistics:

ANALYZE TABLE orders
    UPDATE HISTOGRAM ON status, region
    WITH 32 BUCKETS;

Drop the histogram for a single column (no scan):

ANALYZE TABLE orders DROP HISTOGRAM ON notes;

Combine sampling with a targeted histogram rebuild:

ANALYZE TABLE events WITH SAMPLE 20 PERCENT
    UPDATE HISTOGRAM ON event_type;

The WITH SAMPLE clause precedes the histogram clause. When both are present, the histogram is built from the sampled rows and its bucket counts are scaled up to full-table estimates so that histogram-backed selectivity matches the reported row_count.

Compatibility

There is no ANALYZE statement in the SQL standard; statistics collection is implementation-defined.

EvolutionDB's command requires the TABLE keyword (ANALYZE TABLE name) and accepts exactly one table per statement. This differs from PostgreSQL, where the keyword is ANALYZE without TABLE, multiple tables and a per-table column list may be given, and options such as VERBOSE and SKIP_LOCKED appear in a parenthesized option list. PostgreSQL has no WITH SAMPLE, UPDATE HISTOGRAM, or DROP HISTOGRAM clauses; instead it controls sampling globally through the default_statistics_target setting and the per-column ALTER TABLE … SET STATISTICS. None of those PostgreSQL forms are accepted by EvolutionDB.

The WITH SAMPLE … PERCENT form is conceptually similar to the standard TABLESAMPLE BERNOULLI clause used in SELECT, but here it governs only which rows feed the statistics gatherer, not a query result set.
