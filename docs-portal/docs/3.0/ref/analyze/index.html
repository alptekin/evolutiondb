ANALYZE

ANALYZE scans a table to collect statistics and column histograms. The query planner uses them.

Synopsis

ANALYZE TABLE table_name
    [ WITH SAMPLE n { PERCENT | ROWS } ]
    [ UPDATE HISTOGRAM ON column [, ...] [ WITH n BUCKETS ]
      | DROP HISTOGRAM ON column [, ...] ]

You can write table_name with a schema. Use the form schema.table.

Description

ANALYZE TABLE does a sequential scan of table_name. It then records statistics in the system catalog. Two clauses are optional. They work on their own and do not depend on each other. The WITH SAMPLE clause controls how many rows are read. The histogram clause controls which per-column histograms are built or removed. EXPLAIN reads these statistics when it estimates row counts and picks access paths.

If you use no optional clause, ANALYZE TABLE t rebuilds every statistic for the table.

  • Table-level statistics. These are row_count, page_count, and last_analyzed. page_count is the number of heap pages. It is found by walking the page chain. last_analyzed is the wall-clock time of this run.
  • Per-column statistics. These are null_count, distinct_count, min, max, and avg_width for each column.
  • Per-column histograms. One histogram is chosen automatically for each column that has at least one non-NULL value.
  • Per-index statistics. For every index on the table, this records the B+ tree depth, the leaf-page count, the total entries, and the average entries per leaf.

There are two kinds of histogram. The server chooses one for each column. A frequency histogram is built when the number of distinct values, called NDV, is not larger than the bucket budget. It uses one bucket per value. This gives exact equality selectivity. If the NDV is larger, an equi-depth histogram is built instead. The distinct values are sorted first. The sort is numeric when every value parses as a number. Otherwise the sort is lexicographic. The values are then grouped into buckets that hold about the same number of rows. Each bucket stores an upper bound and a cumulative count. The default bucket budget is 100. No histogram may have more than 100 buckets.

Distinct-value tracking is capped at 10,000 values per column. If a column has more distinct values, the statistics are summarized from that capped sample. Statistics are stored in the catalog no matter how many rows were scanned.

On success the server prints a one-line summary. It shows the rows scanned, the columns analyzed, and the pages counted. When sampling is used, it also shows the sampled-row count and the keep percentage.

Parameters

table_name
The table to analyze. You may add a schema with the form schema.table. The table must exist. If it does not, the command fails with ERROR: table "…" does not exist.
WITH SAMPLE n PERCENT
Read each row with a probability of n%. This is Bernoulli sampling. n is clamped to the range 1–100. The sampled per-column counters are the null count and the width sums. They are scaled back up to full-table estimates. min and max are always computed from the full scan. Doing this is free.
WITH SAMPLE n ROWS
Aim for about n sampled rows. n has a minimum of 1. The target is turned into a keep percentage. This uses the table's last recorded row_count. There may be no prior row count because no earlier ANALYZE ran. In that case the command does a full scan instead.
(no WITH SAMPLE clause)
Sampling is automatic. A full scan is used for tables with fewer than 10,000 rows. Larger tables are sampled at 10%. The size is read from the last recorded row_count. If there is no prior row count, a full scan is performed.
UPDATE HISTOGRAM ON column [, ...]
Rebuild histograms only for the listed columns. The per-column statistics are not rewritten. The table is still scanned. This lets the server collect the value distributions of the listed columns.
WITH n BUCKETS
Use this with UPDATE HISTOGRAM to ask for n buckets. The default is 100. Values above 100 are reduced to 100.
DROP HISTOGRAM ON column [, ...]
Delete the stored histograms for the listed columns. This works on the catalog only. No table scan is done. No other statistics are changed.

Examples

Rebuild all statistics, histograms, and index statistics for a table.

ANALYZE TABLE orders;

Analyze a table that has a schema name.

ANALYZE TABLE sales.orders;

Analyze a large table with a 10% Bernoulli sample.

ANALYZE TABLE events WITH SAMPLE 10 PERCENT;

Aim for about 5,000 sampled rows.

ANALYZE TABLE events WITH SAMPLE 5000 ROWS;

Rebuild histograms for two columns with a custom bucket count. Other statistics are not changed.

ANALYZE TABLE orders
    UPDATE HISTOGRAM ON status, region
    WITH 32 BUCKETS;

Drop the histogram for a single column. This does no scan.

ANALYZE TABLE orders DROP HISTOGRAM ON notes;

Combine sampling with a targeted histogram rebuild.

ANALYZE TABLE events WITH SAMPLE 20 PERCENT
    UPDATE HISTOGRAM ON event_type;

The WITH SAMPLE clause comes before the histogram clause. When you use both, the histogram is built from the sampled rows. Its bucket counts are scaled up to full-table estimates. This way the histogram-backed selectivity matches the reported row_count.

Compatibility

The SQL standard has no ANALYZE statement. Each database defines how it collects statistics on its own.

The EvolutionDB command needs the TABLE keyword. The form is ANALYZE TABLE name. It accepts exactly one table per statement. PostgreSQL works in a different way. There the keyword is ANALYZE without TABLE. You may give several tables and a per-table column list. Options such as VERBOSE and SKIP_LOCKED go in an option list inside parentheses. PostgreSQL has no WITH SAMPLE, UPDATE HISTOGRAM, or DROP HISTOGRAM clauses. It sets sampling globally through the default_statistics_target setting. It also sets it per column with ALTER TABLE … SET STATISTICS. EvolutionDB does not accept any of those PostgreSQL forms.

The WITH SAMPLE … PERCENT form is like the standard TABLESAMPLE BERNOULLI clause used in SELECT. But here it only controls which rows feed the statistics gatherer. It does not control a query result set.
