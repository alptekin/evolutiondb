UPDATE

UPDATE, change the values of columns in rows of a table that satisfy a condition.

Synopsis

UPDATE [ LOW_PRIORITY ] [ IGNORE ] table [ AS alias ] [ , table [ AS alias ] ]
    SET [ table. ]column = expression [ , [ table. ]column = expression ] ...
    [ WHERE condition ]
    [ ORDER BY column [ ASC | DESC ] [ , ... ] ]
    [ LIMIT count [ OFFSET start ] ]
    [ LIMIT start , count ]
    [ RETURNING * | output_column [ , ... ] ]

The assignment operator in the SET list must be =; any other comparison operator in that position is rejected by the parser ("bad update assignment").

Description

UPDATE changes the values of the specified columns in all rows of table that satisfy the WHERE condition. Only the columns named in the SET clause are touched; all other columns retain their current values. The number of rows changed is reported back to the client.

EvolutionDB applies each update non-destructively (MVCC): rather than overwriting a row in place, the engine stamps the old version with the current transaction's xmax (soft-delete), writes the new version to a fresh heap location with a fresh xmin, and re-points the primary-key B+ tree entry at the new RowID. This preserves the old version on the page so concurrent readers holding an older snapshot still see the pre-update row, and so the change can be rolled back. An exclusive row lock is acquired for each row before it is modified; if it cannot be granted the statement fails with a lock or deadlock error.

Two execution forms exist. The WHERE-expression form scans the table (or, for a simple primary-key range predicate such as WHERE id <= 5000, walks only the matching B+ tree keys), collects the matching rows, then applies the update to each. The single-key form (no WHERE) addresses one row directly by its primary key. When the FROM-style table list names more than one table, or a join is present, the statement is handled as a multi-table update.

Before a row is written, EvolutionDB validates the new values against the column's declared type and against NOT NULL, UNIQUE, CHECK, and FOREIGN KEY constraints, re-evaluates any STORED generated columns, enforces row-level security USING/WITH CHECK policies, and fires BEFORE/AFTER UPDATE triggers. A SET on a generated column is rejected. When a primary-key column is changed, the configured ON UPDATE referential action (RESTRICT, NO ACTION, CASCADE, SET NULL, SET DEFAULT) is enforced on referencing child rows. For system-versioned tables, the pre-update row is copied to the history table with its valid_to set to the current time, and the live row's validity window is refreshed automatically.

Parameters

LOW_PRIORITY
Optional update modifier accepted by the grammar (MySQL-compatibility keyword). It is parsed and recorded but does not change EvolutionDB's scheduling behavior.
IGNORE
Optional update modifier accepted by the grammar (MySQL-compatibility keyword). It is parsed and recorded as an option flag.
table [ AS alias ]
The target table whose rows are to be updated. An optional alias may be given. More than one table reference may be listed (comma-separated), and join syntax is accepted; when the resolved statement references more than one table it is processed as a multi-table update.
column
The name of a column in the target table. May be written qualified as table.column; the qualified form is what drives multi-table assignments. The column cannot be a generated column, assigning to one raises an error advising that it can only be updated to DEFAULT.
expression
The value to assign to the column, written as a SQL expression after =. The new value must be compatible with the column's declared type; type validation runs before the row is written.
WHERE condition
An expression returning a boolean (truthy) result. Only rows for which it evaluates true are updated. When omitted in the single-key form, the row is located by primary key; when an expression WHERE is present, every row matching the predicate is updated. A simple primary-key range or equality predicate is optimized to a B+ tree cursor walk instead of a full table scan.
ORDER BY column [ ASC | DESC ]
Accepted by the grammar (MySQL-style ordered update). It orders the candidate rows and is meaningful primarily in combination with LIMIT.
LIMIT count [ OFFSET start ] / LIMIT start, count
Restricts how many of the matched rows are actually updated. The matched set is sliced by the effective OFFSET/LIMIT range before the updates are applied; rows outside the range are left unchanged.
RETURNING * | output_column [.. ]
Causes UPDATE to return the affected rows. RETURNING * returns all columns; a column list returns just those columns. The returned values reflect the new (post-update) row.

Examples

Change a single column on one row identified by its primary key:

UPDATE employees SET salary = 75000 WHERE id = 42;

Update several columns at once, computing a new value from an expression:

UPDATE products
   SET price = price * 1.10, updated_by = 'admin'
 WHERE category = 'books';

Range predicate over the primary key (uses the B+ tree cursor fast path):

UPDATE orders SET status = 'archived' WHERE id <= 5000;

Update only the first few matching rows, in order, and return the new values:

UPDATE tickets
   SET priority = 'high'
 WHERE assignee IS NULL
 ORDER BY created_at ASC
 LIMIT 10
RETURNING id, priority;

Set a column to NULL:

UPDATE accounts SET closed_at = NULL WHERE id = 7;

Compatibility

The core form UPDATE table SET col = expr [ , ... ] WHERE condition follows the SQL standard and matches PostgreSQL closely. The optional RETURNING clause is a PostgreSQL extension that EvolutionDB supports, returning post-update column values.

Several clauses are MySQL-flavored extensions rather than SQL-standard or PostgreSQL features: the LOW_PRIORITY and IGNORE modifiers before the table name, and the ORDER BY / LIMIT clauses on UPDATE. The LIMIT start, count two-argument form is MySQL syntax; the LIMIT count OFFSET start form follows PostgreSQL.

Notable differences from PostgreSQL's UPDATE: EvolutionDB does not implement PostgreSQL's UPDATE ... FROM from_list clause for joins: instead, multi-table updates are expressed by listing tables (and joins) directly after the UPDATE keyword in a FROM-style table reference and using qualified SET table.column = ... assignments, in the MySQL multi-table style. EvolutionDB also does not support the WHERE CURRENT OF cursor targeting, table aliases for the sole purpose of a FROM join, or the standard SET (col1, col2) = (...) multi-column row-assignment form; assignments are written one column at a time. The assignment operator must be a plain =.
