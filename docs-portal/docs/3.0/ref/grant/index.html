GRANT

Grant access privileges on a database, schema, or table to a user, or grant membership in a role to a user.

Synopsis

-- Grant privileges on a database object:
GRANT privilege [ , ... ]
    ON { DATABASE | SCHEMA | TABLE } object_name
    TO username
    [ WITH GRANT OPTION ]

-- Grant membership in a role (no ON clause):
GRANT role_name TO username

where privilege is one of:
    SELECT | INSERT | UPDATE | DELETE | CREATE | DROP
    ALL [ PRIVILEGES ]

Description

GRANT gives privileges to a user. EvolutionDB recognizes two distinct forms, distinguished by whether an ON clause is present:

Object privileges. The first form (with ON) grants one or more privileges on a single database object — a DATABASE, a SCHEMA, or a TABLE — to a named user. Each privilege corresponds to a SQL operation: SELECT, INSERT, UPDATE, DELETE, CREATE, and DROP. ALL (also accepted as ALL PRIVILEGES) is shorthand for all six.

Role membership. The second form (no ON) makes username a member of role_name. When privileges are checked, a user inherits every privilege held by the roles it belongs to, in addition to its own.

Additive, merging semantics. Granting privileges that overlap an existing grant for the same user and object does not replace the grant — it merges: the new privilege bits are OR-ed into the bits already recorded, and WITH GRANT OPTION is turned on if requested (an existing grant option is never cleared by a re-grant). A user's own grants and grants made to the special grantee PUBLIC are both additive: an unprivileged user still benefits from any matching PUBLIC grant.

How privileges are resolved. At access-check time EvolutionDB applies a most-specific-first waterfall: it looks for a matching grant at TABLE scope, then SCHEMA scope, then DATABASE scope (a DATABASE grant on the wildcard name * applies to every database). The same waterfall is then repeated for PUBLIC, and finally for each role the user belongs to. If any level yields the requested privilege, access is allowed.

Who may run GRANT. The built-in admin user is a superuser and may grant anything (and is always considered to hold WITH GRANT OPTION). A non-admin user may only delegate privileges it itself holds with the grant option on the same object; otherwise the statement fails with "Permission denied". Only admin may grant role membership.

EvolutionDB ships with one grant pre-installed: admin holds ALL on DATABASE *. New databases are created with no default PUBLIC grants, so non-admin users start with no access until something is granted to them (directly, via PUBLIC, or via a role).

Parameters

privilege
The privilege to grant. One of SELECT, INSERT, UPDATE, DELETE, CREATE, or DROP. Privilege names are case-insensitive. Multiple privileges may be listed, separated by commas. Unrecognized privilege words are silently ignored; if the list ends up empty, the statement fails with "Invalid privilege specification".
ALL [ PRIVILEGES ]
Shorthand for all six privileges (SELECT, INSERT, UPDATE, DELETE, CREATE, DROP). The optional PRIVILEGES keyword is accepted for SQL-standard compatibility and has no additional effect.
ON DATABASE | SCHEMA | TABLE
Selects the scope (kind of object) the grant applies to. Exactly one of these keywords must follow ON. The waterfall resolution treats TABLE as the most specific scope and DATABASE as the broadest.
object_name
The name of the object being granted on. It may be written bare, single-quoted, or double-quoted. For a DATABASE grant the name is a database name (or the wildcard *, meaning all databases). For finer scopes the name is recorded as given; checks build the lookup key from the session's database, schema, and table (for example db.schema.table at table scope), so qualify the name to match the level you intend.
TO username
The user that receives the privileges (or role membership). May be written bare or single-quoted. Use the special name PUBLIC to grant to every user additively. The target user is not required to exist for an object grant; for a role grant the target must be an existing user (not a role).
WITH GRANT OPTION
Allows the recipient to in turn grant the same privileges on the same object to other users. Only valid in the object-privilege form. Without it, the recipient may use the privileges but not delegate them. A non-admin grantor must itself hold the grant option to confer it.
role_name (role-membership form)
In the no-ON form, the name of an existing role whose membership is conferred on username. The role must exist, the target must be a user, and the user must not already be a member; otherwise the statement fails. Only admin may run this form.

Examples

Grant read access on a table to a user:

GRANT SELECT ON TABLE orders TO analyst;

Grant several privileges at once, and let the recipient re-delegate them:

GRANT SELECT, INSERT, UPDATE ON TABLE orders TO app_user WITH GRANT OPTION;

Grant every privilege on a schema to everyone:

GRANT ALL PRIVILEGES ON SCHEMA reporting TO PUBLIC;

Grant create/drop authority across all databases (the wildcard scope):

GRANT CREATE, DROP ON DATABASE '*' TO dba;

Grant database-level read access to a single database:

GRANT SELECT ON DATABASE testdb TO readonly;

Grant role membership (no ON clause — run as admin):

GRANT reporting_role TO analyst;

Inspect the resulting grants:

SHOW GRANTS FOR analyst;

Compatibility

EvolutionDB's GRANT follows the spirit of the SQL standard and PostgreSQL but is intentionally simplified, and the statement is handled by a dedicated command interceptor rather than the main SQL grammar.

  • Consistent with the standard / PostgreSQL: the privilege set SELECT, INSERT, UPDATE, DELETE; the ALL [PRIVILEGES] shorthand; the TO target with the special grantee PUBLIC; WITH GRANT OPTION delegation; and additive PUBLIC semantics all match PostgreSQL's model.
  • EvolutionDB specifics: CREATE and DROP are object privileges in the same grant list as DML privileges. The object scope is one of exactly three keywords — DATABASE, SCHEMA, or TABLE — and a DATABASE grant may use the wildcard name * to cover all databases. Privilege resolution uses a most-specific-first waterfall (TABLE → SCHEMA → DATABASE) plus role inheritance. The admin user is an implicit superuser.
  • Differences and unsupported clauses: there is no ON ALL TABLES IN SCHEMA, no column-level privileges, no privileges on sequences/functions/types/foreign objects, no GRANTED BY clause, and no WITH ADMIN OPTION for role grants. The grant object must be a single named object; lists of objects are not accepted. Privilege checking is by username (and roles); object grants do not require the target user to pre-exist.
