MEMORY SEARCH

MEMORY SEARCH, rank the rows of a memory store by vector distance from a query embedding and return the closest matches.

Synopsis

MEMORY SEARCH store_name USING VECTOR 'query' [ LIMIT count ]

Description

MEMORY SEARCH performs a similarity search over a memory store created with CREATE MEMORY STORE. It compares the supplied query embedding against the embedding column of every row in the store's backing table, computes a distance score for each row using the store's configured distance metric, and returns the rows ordered from nearest (smallest score) to farthest.

Each result row projects four columns:

mem_namespace
The namespace the entry was stored under.
mem_key
The key within that namespace.
mem_value
The stored JSON value.
score
The distance between the row's stored embedding and the query embedding, computed with the metric chosen at CREATE MEMORY STORE time. Results are sorted by this column ascending, so the lowest score is the closest match.

The distance function applied is fixed at store-creation time by the store's distance option and cannot be overridden per query. The mapping is:

cosine
Cosine distance (the default for any store). Used when distance was 'cosine' or unspecified.
l2
Euclidean (L2) distance. Used when distance was 'l2'.
inner / ip
Inner-product distance. Used when distance was 'inner' or 'ip'.
l1
Manhattan (L1) distance. Used when distance was 'l1'.

The search is implemented internally as a SELECT over the backing table, so any row-level security policies, MVCC visibility rules, and other engine-level guarantees that apply to that table apply identically to the search results.

The memory store must have been created with a non-zero embedding_dim. Searching a store that was created without an embedding column raises an error (SQLSTATE 42P01), such a store is a pure JSON key/value table with no semantic-search capability. Searching a store that does not exist raises SQLSTATE 42704.

Parameters

store_name
The name of the memory store to search. The store must exist and must have been created with an embedding dimension greater than zero.
USING VECTOR 'query'
The query embedding, supplied as a string literal. It is interpreted as a vector literal (a bracketed, comma-separated list of floats, e.g. '[0.1, 0.2, 0.3]') and compared against each row's stored embedding. This clause is required; there is no key- or text-based search form.
LIMIT count
Optional. Caps the number of result rows returned. When omitted, the search defaults to returning the 10 closest matches. count is an integer literal.

Examples

Return the 10 closest entries (the default limit) to a query embedding:

MEMORY SEARCH agent_mem USING VECTOR '[0.12, 0.04, 0.91, 0.33]';

Return only the single best match:

MEMORY SEARCH agent_mem USING VECTOR '[0.12, 0.04, 0.91, 0.33]' LIMIT 1;

Retrieve the top five neighbours; each row carries the computed distance in the score column:

-- store created with: CREATE MEMORY STORE notes
--   WITH (embedding_dim = 4, distance = 'l2')
MEMORY SEARCH notes USING VECTOR '[1.0, 0.0, 0.0, 0.0]' LIMIT 5;
-- columns: mem_namespace | mem_key | mem_value | score

Compatibility

MEMORY SEARCH is an EvolutionDB extension. There is no equivalent statement in the SQL standard, and it is not part of PostgreSQL's grammar. It is conceptually similar to a nearest-neighbour ORDER BY query against a pgvector column in PostgreSQL (for example ORDER BY embedding <-> '[...]' LIMIT k), but it is a dedicated single-statement command rather than a general SELECT operator.

Note these EvolutionDB-specific behaviours:

  • The distance metric is bound to the store at creation time and cannot be selected per query; PostgreSQL/pgvector lets you pick an operator (<->, <#>, <=>) per query.
  • Results are always ordered by ascending score internally; there is no user-supplied ORDER BY clause in this command form.
  • When LIMIT is omitted the command applies a built-in default of 10 rows, rather than returning all rows.
  • The projected column set (mem_namespace, mem_key, mem_value, score) is fixed; there is no select-list in the grammar.
