MEMORY SEARCH

MEMORY SEARCH sorts the rows of a memory store by how close they are to a query embedding. It then returns the closest matches.

Synopsis

MEMORY SEARCH store_name USING VECTOR 'query' [ LIMIT count ]

Description

MEMORY SEARCH runs a similarity search over a memory store. You create the store with CREATE MEMORY STORE. The command compares the query embedding you give against the embedding column of every row in the store's backing table. It computes a distance score for each row. The store's distance metric decides how the score is computed. The rows come back sorted from nearest to farthest. The smallest score is the nearest row.

Each result row has four columns.

mem_namespace
The namespace the entry was stored under.
mem_key
The key inside that namespace.
mem_value
The stored JSON value.
score
The distance between the row's stored embedding and the query embedding. The metric you chose at CREATE MEMORY STORE time computes it. Results are sorted by this column from low to high. The lowest score is the closest match.

The distance function is set when you create the store. It comes from the store's distance option. You cannot change it for a single query. The options work like this.

cosine
Cosine distance. This is the default for any store. It is used when distance was 'cosine' or was left out.
l2
Euclidean (L2) distance. It is used when distance was 'l2'.
inner / ip
Inner-product distance. It is used when distance was 'inner' or 'ip'.
l1
Manhattan (L1) distance. It is used when distance was 'l1'.

Inside the engine, the search runs as a SELECT over the backing table. So the same rules apply to the search results as to the table. This includes row-level security policies, MVCC visibility rules, and other engine-level guarantees.

The memory store must have been created with a non-zero embedding_dim. A store made without an embedding column is a plain JSON key/value table. It cannot do semantic search. Searching such a store raises an error with SQLSTATE 42P01. Searching a store that does not exist raises SQLSTATE 42704.

Parameters

store_name
The name of the memory store to search. The store must exist. It must have been created with an embedding dimension greater than zero.
USING VECTOR 'query'
The query embedding. You give it as a string literal. It is read as a vector literal. A vector literal is a comma-separated list of floats inside brackets, for example '[0.1, 0.2, 0.3]'. It is compared against each row's stored embedding. This clause is required. There is no search form based on a key or text.
LIMIT count
Optional. It limits how many result rows come back. When you leave it out, the search returns the 10 closest matches. count is an integer literal.

Examples

Return the 10 closest entries to a query embedding. This uses the default limit.

MEMORY SEARCH agent_mem USING VECTOR '[0.12, 0.04, 0.91, 0.33]';

Return only the single best match.

MEMORY SEARCH agent_mem USING VECTOR '[0.12, 0.04, 0.91, 0.33]' LIMIT 1;

This example gets the top five nearest matches. Each row includes the computed distance in the score column.

-- store created with: CREATE MEMORY STORE notes
--   WITH (embedding_dim = 4, distance = 'l2')
MEMORY SEARCH notes USING VECTOR '[1.0, 0.0, 0.0, 0.0]' LIMIT 5;
-- columns: mem_namespace | mem_key | mem_value | score

Compatibility

MEMORY SEARCH is an EvolutionDB extension. There is no matching statement in the SQL standard. It is not part of PostgreSQL's grammar. It works much like a nearest-neighbour ORDER BY query against a pgvector column in PostgreSQL. One such query is ORDER BY embedding <-> '[...]' LIMIT k. But it is one dedicated command, not a general SELECT operator.

These behaviours are specific to EvolutionDB.

  • The distance metric is set on the store when you create it. You cannot pick it per query. PostgreSQL and pgvector let you pick an operator per query, such as <->, <#>, or <=>.
  • Results are always ordered by score from low to high. There is no user-supplied ORDER BY clause in this command.
  • When you leave out LIMIT, the command uses a built-in default of 10 rows. It does not return all rows.
  • The returned columns are fixed. They are mem_namespace, mem_key, mem_value, and score. There is no select-list in the grammar.
