MEMORY DELETE

Remove a single key/value entry from a memory store, identified by its namespace and key.

Synopsis

MEMORY DELETE FROM store_name
    WHERE NS = 'namespace' AND KEY = 'key'

Description

MEMORY DELETE removes exactly one entry from the memory store named store_name. An entry in a memory store is uniquely identified by the pair (namespace, key); MEMORY DELETE targets the single row whose composite primary key matches the supplied namespace and key.

Internally, a memory store is backed by an ordinary table named mem_<store_name>. The two halves of the entry identity are combined into one composite primary-key value (the namespace and key joined by a record-separator byte). MEMORY DELETE rewrites the request as DELETE FROM mem_<store_name> WHERE pk = '...' and routes it through the engine's standard delete path. As a result, the operation honors exactly the same semantics as a hand-written DELETE: row-level security (RLS) policies attached to the backing table, foreign-key cascade actions, MVCC visibility, and triggers all apply. The RLS pass-through is what makes MEMORY DELETE safe in multi-tenant deployments, a caller can only delete rows its policy permits.

If the named store does not exist, the command fails with SQLSTATE 42704 (undefined object) and does not modify any data. If the store exists but no entry matches the given namespace and key, the command completes successfully having removed zero rows.

Both predicates are required. The grammar accepts only the fixed form WHERE NS = '...' AND KEY = '...'. The comparison operator must be =, the two predicates must be joined by AND, and both the namespace and key values are string literals. There is no way to delete a whole namespace, match a key prefix, or omit either predicate through MEMORY DELETE.

Parameters

store_name
The name of the target memory store, written after FROM. The store must already exist (see CREATE MEMORY STORE); otherwise the command raises SQLSTATE 42704.
NS = 'namespace'
String literal giving the namespace component of the entry to delete. Together with the key it forms the entry's unique identity. An empty string ('') is a valid namespace value.
KEY = 'key'
String literal giving the key component of the entry to delete, within the given namespace.

Examples

Delete a single entry by namespace and key:

MEMORY DELETE FROM chat_memory
    WHERE NS = 'user:42' AND KEY = 'last_topic';

Delete an entry stored under the empty namespace:

MEMORY DELETE FROM kv_store
    WHERE NS = '' AND KEY = 'session_token';

A typical write-then-remove lifecycle for one entry:

-- store an entry
MEMORY PUT INTO chat_memory
    VALUES ('user:42', 'last_topic', '{"text":"billing"}');

-- read it back
MEMORY GET FROM chat_memory WHERE NS = 'user:42' AND KEY = 'last_topic';

-- remove it
MEMORY DELETE FROM chat_memory WHERE NS = 'user:42' AND KEY = 'last_topic';

Compatibility

MEMORY DELETE is an EvolutionDB extension and is not part of the SQL standard; there is no equivalent statement in PostgreSQL. It exists as ergonomic sugar over the standard DELETE statement: the request is translated into DELETE FROM mem_<store_name> WHERE pk = '...' and executed through the ordinary delete engine, so it inherits standard delete behavior including RLS, foreign-key actions, MVCC, and triggers.

Unlike a general-purpose DELETE, the MEMORY DELETE grammar is deliberately restricted: it accepts only equality on both NS and KEY joined by AND, and supports no other WHERE shape, ORDER BY, LIMIT, or RETURNING clause. To delete by any other criterion, target the backing table directly with a standard DELETE statement.
