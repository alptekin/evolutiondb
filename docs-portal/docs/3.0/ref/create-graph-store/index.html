CREATE GRAPH STORE

Create a named bitemporal knowledge-graph store and its backing node and edge tables.

Synopsis

CREATE GRAPH STORE [ IF EXISTS ] store_name

Description

CREATE GRAPH STORE registers a new knowledge-graph store under store_name and provisions the two relational tables that hold its data. It is part of EvolutionDB's bitemporal knowledge-graph surface (Feature 224), alongside GRAPH UPSERT NODE, GRAPH UPSERT EDGE, and GRAPH NEIGHBORS.

For a store named store_name, the command creates:

  • kg_<store_name>_nodes — columns id VARCHAR(255) PRIMARY KEY NOT NULL, type VARCHAR(64), props VARCHAR(8000) (JSON-shaped), and valid_from TIMESTAMP NOT NULL.
  • kg_<store_name>_edges — columns id BIGINT PRIMARY KEY AUTO_INCREMENT, src VARCHAR(255) NOT NULL, rel VARCHAR(128) NOT NULL, dst VARCHAR(255) NOT NULL, props VARCHAR(8000), valid_from TIMESTAMP NOT NULL, and invalid_at TIMESTAMP (where NULL marks a currently-active edge).

Both tables are created in the current schema (the schema resolved from the current database/schema context). The store itself is recorded as one row in the system catalog (CAT_SYS_GRAPH_STORES), holding the store name plus the table ids of its nodes and edges tables. The store name is global to the catalog — it is not schema-qualified.

The invalid_at column on edges is what makes the store bitemporal: an UPSERT EDGE on an existing (src, rel, dst) triple stamps the prior live row's invalid_at to NOW() and appends a fresh row, so a GRAPH NEIGHBORS … AS OF query can reconstruct the graph as it stood at any past timestamp.

Idempotency. Without the optional clause, creating a store whose name already exists raises an error (SQLSTATE 42710, duplicate object). With the clause present, an already-existing store is left untouched and the command succeeds silently.

Grammar note. The optional “create only if absent” clause is spelled IF EXISTS in the accepted grammar (token sequence CREATE GRAPH STORE IF EXISTS store_name). Despite the keyword being EXISTS, the semantics are if-not-exists: when present, an already-existing store is a no-op rather than an error. This differs from the IF NOT EXISTS spelling used by CREATE TABLE and most other EvolutionDB DDL.

Parameters

IF EXISTS
Optional. When present, do not raise an error if a graph store with the same name already exists; instead, do nothing and report success. The backing tables are not recreated or altered. (See the grammar note above: this clause behaves as if-not-exists despite its keyword.)
store_name
The name of the graph store to create. It must be a bare identifier and is required. The same name is used to derive the backing table names kg_<store_name>_nodes and kg_<store_name>_edges, and is recorded verbatim in the system catalog. A missing name raises a syntax error (SQLSTATE 42601).

Examples

Create a knowledge-graph store named kb:

CREATE GRAPH STORE kb;

Create it only if it is not already present:

CREATE GRAPH STORE IF EXISTS kb;

Create a store and populate it. Each VALUES argument is a single-quoted string:

CREATE GRAPH STORE kb;

-- nodes: (id, type, props)
GRAPH UPSERT NODE INTO kb VALUES ('alice', 'person', '{"role":"engineer"}');
GRAPH UPSERT NODE INTO kb VALUES ('acme', 'org', '{"sector":"db"}');

-- edge: (src, rel, dst)
GRAPH UPSERT EDGE INTO kb VALUES ('alice', 'works_at', 'acme');

-- edge with optional props: (src, rel, dst, props)
GRAPH UPSERT EDGE INTO kb VALUES ('alice', 'knows', 'bob', '{"since":2021}');

Walk the graph outward from a node. The optional DEPTH and AS OF clauses control hop count and the temporal snapshot:

GRAPH NEIGHBORS OF 'alice' IN kb;
GRAPH NEIGHBORS OF 'alice' IN kb DEPTH 2;
GRAPH NEIGHBORS OF 'alice' IN kb AS OF '2026-01-01 00:00:00';

Remove the store, dropping both backing tables and the catalog entry:

DROP GRAPH STORE kb;
DROP GRAPH STORE IF EXISTS kb;

Compatibility

CREATE GRAPH STORE is an EvolutionDB extension. There is no such statement in the SQL standard (SQL:2016), and PostgreSQL has no CREATE GRAPH STORE command — graph workloads on PostgreSQL are handled by external extensions (for example Apache AGE's create_graph() function), not by core DDL. EvolutionDB instead exposes a first-class, fixed-shape store: the node and edge schemas are defined by the engine and cannot be customized at creation time, and a store has exactly one nodes table and one edges table.

The optional clause is spelled IF EXISTS (not IF NOT EXISTS) but carries if-not-exists semantics; this is specific to the graph-store and entity-store grammar and is not how the SQL standard or PostgreSQL spell conditional CREATE.

The model is bitemporal in the spirit of Zep Graphiti rather than the SQL standard's system-versioned tables: edges are never deleted on update but are closed by stamping invalid_at, enabling point-in-time AS OF reads. EvolutionDB does not implement SQL standard FOR SYSTEM_TIME syntax for these stores.
