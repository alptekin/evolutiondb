CREATE GRAPH STORE

Create a named bitemporal knowledge-graph store. It comes with its own node table and edge table.

Synopsis

CREATE GRAPH STORE [ IF EXISTS ] store_name

Description

CREATE GRAPH STORE adds a new knowledge-graph store under store_name. It also builds the two tables that hold the store's data. This command is part of EvolutionDB's bitemporal knowledge-graph feature, Feature 224. The related commands are GRAPH UPSERT NODE, GRAPH UPSERT EDGE, and GRAPH NEIGHBORS.

For a store named store_name, the command creates two tables.

  • kg_<store_name>_nodes. It has these columns. id VARCHAR(255) PRIMARY KEY NOT NULL, type VARCHAR(64), props VARCHAR(8000), and valid_from TIMESTAMP NOT NULL. The props column holds JSON-shaped text.
  • kg_<store_name>_edges. It has these columns. id BIGINT PRIMARY KEY AUTO_INCREMENT, src VARCHAR(255) NOT NULL, rel VARCHAR(128) NOT NULL, dst VARCHAR(255) NOT NULL, props VARCHAR(8000), valid_from TIMESTAMP NOT NULL, and invalid_at TIMESTAMP. A NULL value in invalid_at means the edge is still active.

Both tables go into the current schema. The current schema comes from your current database and schema context. The store itself is saved as one row in the system catalog table CAT_SYS_GRAPH_STORES. That row holds the store name and the table ids of its nodes and edges tables. The store name is global to the catalog. It is not schema-qualified.

The invalid_at column on edges is what makes the store bitemporal. Suppose you run UPSERT EDGE on a (src, rel, dst) triple that already exists. The command sets the old live row's invalid_at to NOW(). Then it adds a fresh row. Because of this, a GRAPH NEIGHBORS … AS OF query can rebuild the graph as it looked at any past timestamp.

Idempotency. Without the optional clause, the command fails if the store name already exists. It raises an error with SQLSTATE 42710, which means duplicate object. With the clause present, the command leaves the existing store untouched. It then succeeds without any message.

Grammar note. The optional clause means "create only if the store is not there." In the grammar it is written IF EXISTS. The full token sequence is CREATE GRAPH STORE IF EXISTS store_name. The keyword is EXISTS, but the meaning is if-not-exists. When the clause is present, an existing store is a no-op and not an error. This is different from the IF NOT EXISTS spelling used by CREATE TABLE and most other EvolutionDB DDL.

Parameters

IF EXISTS
Optional. When present, the command does not raise an error if a graph store with the same name already exists. Instead it does nothing and reports success. The backing tables are not rebuilt or changed. See the grammar note above. This clause acts as if-not-exists even though its keyword is EXISTS.
store_name
The name of the graph store to create. It must be a bare identifier. It is required. The same name is used to build the backing table names kg_<store_name>_nodes and kg_<store_name>_edges. The name is saved as is in the system catalog. If the name is missing, the command raises a syntax error with SQLSTATE 42601.

Examples

Create a knowledge-graph store named kb.

CREATE GRAPH STORE kb;

Create it only if it is not there yet.

CREATE GRAPH STORE IF EXISTS kb;

Create a store and add data to it. Each VALUES argument is a single-quoted string.

CREATE GRAPH STORE kb;

-- nodes: (id, type, props)
GRAPH UPSERT NODE INTO kb VALUES ('alice', 'person', '{"role":"engineer"}');
GRAPH UPSERT NODE INTO kb VALUES ('acme', 'org', '{"sector":"db"}');

-- edge: (src, rel, dst)
GRAPH UPSERT EDGE INTO kb VALUES ('alice', 'works_at', 'acme');

-- edge with optional props: (src, rel, dst, props)
GRAPH UPSERT EDGE INTO kb VALUES ('alice', 'knows', 'bob', '{"since":2021}');

Walk the graph outward from a node. The optional DEPTH clause sets how many hops to follow. The optional AS OF clause picks a point in time.

GRAPH NEIGHBORS OF 'alice' IN kb;
GRAPH NEIGHBORS OF 'alice' IN kb DEPTH 2;
GRAPH NEIGHBORS OF 'alice' IN kb AS OF '2026-01-01 00:00:00';

Remove the store. This drops both backing tables and the catalog entry.

DROP GRAPH STORE kb;
DROP GRAPH STORE IF EXISTS kb;

Compatibility

CREATE GRAPH STORE is an EvolutionDB extension. The SQL standard, SQL:2016, has no such statement. PostgreSQL has no CREATE GRAPH STORE command either. On PostgreSQL, graph work is done by outside extensions, not by core DDL. One example is the create_graph() function from Apache AGE. EvolutionDB gives you a built-in store with a fixed shape. The engine defines the node and edge schemas. You cannot change them when you create the store. A store always has exactly one nodes table and one edges table.

The optional clause is written IF EXISTS and not IF NOT EXISTS. Still, it has if-not-exists meaning. This is specific to the graph-store and entity-store grammar. It is not how the SQL standard or PostgreSQL write a conditional CREATE.

The model is bitemporal in the spirit of Zep Graphiti. It does not use the SQL standard's system-versioned tables. On update, edges are never deleted. They are closed by setting invalid_at. This is what makes point-in-time AS OF reads work. EvolutionDB does not support the SQL standard FOR SYSTEM_TIME syntax for these stores.
