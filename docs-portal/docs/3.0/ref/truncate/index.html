TRUNCATE

Remove all rows from one or more tables in a single statement.

Synopsis

TRUNCATE TABLE table_name [ , table_name [, ... ] ]
    [ RESTART IDENTITY | CONTINUE IDENTITY ]
    [ CASCADE | RESTRICT ]

Description

TRUNCATE removes all rows from a table. It does the same job as a DELETE with no WHERE clause. But it does not scan and remove rows one at a time. Instead it throws away the table's storage all at once. This makes it much faster on large tables.

For each named table, EvolutionDB destroys the heap pages, the primary-key B+ tree, and every secondary index. This includes B+ tree, hash, and unique-hash indexes. It then builds them again as empty structures. It also resets the table's row and page statistics to zero. The table definition stays intact. This means the columns, constraints, indexes, and privileges all remain. Only the data is removed.

The TABLE keyword is required. EvolutionDB's grammar accepts TRUNCATE TABLE. It does not accept the bare TRUNCATE form.

You can truncate several tables in one statement. List them after the first table name and separate them with commas. You can name the first table plus up to seven more tables. That is eight tables in total. Any names beyond that limit are ignored without warning. The tables are processed from left to right. If any table fails, the statement stops and reports the error. For example, the table may not exist.

Foreign keys

Before it truncates a table, EvolutionDB checks the other tables. It looks for any other table that has a foreign key pointing to this one and holds at least one row. Say such a referencing table exists and you did not give CASCADE. Then the operation is refused with SQLSTATE 23503. The message names the referencing table. With CASCADE, the referencing tables are truncated first. This happens recursively. So the dependent rows are removed along with the target. Self-references are ignored for this check.

Note. If the named table does not exist, TRUNCATE fails with SQLSTATE 42P01. This means the table does not exist. On success the server reports command(s) completed successfully!...

Parameters

table_name
The name of a table to truncate. The first name is required. You can add more names after it, separated by commas. These truncate further tables in the same statement. You can add up to seven extra tables. Names are resolved through the current database and schema search path.
RESTART IDENTITY
Reset the table's auto-increment counter. The next inserted row then starts again from the beginning. This is EvolutionDB's default behavior. For any table that has an auto-increment column, the counter is reset to zero. The only exception is when you give CONTINUE IDENTITY. You can write this clause directly for SQL and PostgreSQL compatibility.
CONTINUE IDENTITY
Do not reset the auto-increment counter. The sequence continues from where it left off. So values are not reused even though the table is now empty.
CASCADE
Automatically truncate any non-empty tables that hold foreign keys pointing to a table being truncated. This continues recursively down the chain. Without this option, you cannot truncate a table that is still referenced by dependent tables that hold rows. That is an error.
RESTRICT
Refuse to truncate if other tables hold foreign keys pointing to the target. This is the default foreign key behavior. You get it simply by leaving out CASCADE. The grammar accepts this keyword for compatibility. It adds no behavior beyond the default.

Examples

Remove all rows from a single table.

TRUNCATE TABLE orders;

Remove all rows from several tables in one statement.

TRUNCATE TABLE orders, order_items, shipments;

Remove all rows from a table but keep the auto-increment counter moving forward.

TRUNCATE TABLE sessions CONTINUE IDENTITY;

Remove all rows from a table and from all tables that point to it through foreign keys.

-- customers is referenced by orders, which is referenced by order_items
TRUNCATE TABLE customers CASCADE;

Ask for the default identity reset in a direct way.

TRUNCATE TABLE audit_log RESTART IDENTITY;

Compatibility

The SQL standard defines TRUNCATE TABLE with optional RESTART IDENTITY and CONTINUE IDENTITY clauses. EvolutionDB follows that form. It also accepts PostgreSQL's CASCADE and RESTRICT keywords.

Here are the differences to be aware of.

  • EvolutionDB requires the TABLE keyword. PostgreSQL makes it optional and allows the bare TRUNCATE foo form. That bare form is not accepted here.
  • PostgreSQL allows an explicit ONLY qualifier and table inheritance rules. EvolutionDB has no such clause.
  • In EvolutionDB the identity default is RESTART IDENTITY. The auto-increment counter is reset unless you give CONTINUE IDENTITY. This matches PostgreSQL's default.
  • CASCADE recursively truncates referencing tables that hold rows. This matches PostgreSQL. Without CASCADE, you cannot truncate a table that a non-empty table references. This fails with SQLSTATE 23503. RESTRICT simply names this default behavior.
