TRUNCATE

Empty one or more tables of all their rows in a single statement.

Synopsis

TRUNCATE TABLE table_name [ , table_name [, ... ] ]
    [ RESTART IDENTITY | CONTINUE IDENTITY ]
    [ CASCADE | RESTRICT ]

Description

TRUNCATE removes all rows from a table. It is functionally equivalent to an unqualified DELETE on the table, but rather than scanning and removing tuples one at a time it discards the table's underlying storage wholesale, which makes it considerably faster on large tables.

For each named table, EvolutionDB destroys the heap pages, the primary-key B+ tree, and every secondary index (B+ tree, hash, and unique-hash), then recreates them as empty structures and resets the table's row/page statistics to zero. The table definition — columns, constraints, indexes, and privileges — is left intact; only the data is removed.

The TABLE keyword is required: EvolutionDB's grammar accepts TRUNCATE TABLE, not the bare TRUNCATE form.

Multiple tables may be truncated in one statement by listing them, separated by commas, after the first table name. The primary table plus up to seven additional tables (eight in total) may be named; further names beyond that limit are silently ignored. The tables are processed left to right; if any table fails (for example, because it does not exist), the statement stops and reports the error.

Foreign keys

Before truncating a table, EvolutionDB checks whether any other table holds a foreign key that references it and currently contains at least one row. If such a referencing table exists and CASCADE was not given, the operation is refused with SQLSTATE 23503 and a message naming the referencing table. With CASCADE, the referencing tables are truncated first (recursively), so the dependent rows are removed along with the target. Self-references are ignored for this check.

Note. If the named table does not exist, TRUNCATE fails with SQLSTATE 42P01 (table does not exist). On success the server reports command(s) completed successfully!...

Parameters

table_name
The name of a table to truncate. The first name is mandatory; additional comma-separated names truncate further tables in the same statement (up to seven extra tables). Names are resolved through the current database/schema search path.
RESTART IDENTITY
Reset the table's auto-increment counter so that the next inserted row starts again from the beginning. This is EvolutionDB's default behavior: for any table that has an auto-increment column, the counter is reset to zero unless CONTINUE IDENTITY is specified. The clause is accepted explicitly for SQL/PostgreSQL compatibility.
CONTINUE IDENTITY
Do not reset the auto-increment counter. The sequence continues from where it left off, so values are not reused even though the table is now empty.
CASCADE
Automatically truncate any non-empty tables that hold foreign keys referencing a table being truncated (and so on, recursively). Without this option, truncating a table that is still referenced by populated dependent tables is an error.
RESTRICT
Refuse to truncate if other tables hold foreign keys referencing the target. This is the default FK behavior (it is the absence of CASCADE). The keyword is accepted by the grammar for compatibility but adds no behavior beyond the default.

Examples

Empty a single table:

TRUNCATE TABLE orders;

Empty several tables in one statement:

TRUNCATE TABLE orders, order_items, shipments;

Empty a table but keep the auto-increment counter advancing:

TRUNCATE TABLE sessions CONTINUE IDENTITY;

Empty a table and all tables that reference it via foreign keys:

-- customers is referenced by orders, which is referenced by order_items
TRUNCATE TABLE customers CASCADE;

Explicitly request the default identity reset:

TRUNCATE TABLE audit_log RESTART IDENTITY;

Compatibility

The SQL standard defines TRUNCATE TABLE with optional RESTART IDENTITY / CONTINUE IDENTITY; EvolutionDB follows that form and additionally accepts PostgreSQL's CASCADE and RESTRICT keywords.

Differences to be aware of:

  • EvolutionDB requires the TABLE keyword. PostgreSQL makes it optional (TRUNCATE foo); that bare form is not accepted here.
  • PostgreSQL allows an explicit ONLY qualifier and table inheritance semantics; EvolutionDB has no such clause.
  • In EvolutionDB the identity default is RESTART IDENTITY (the auto-increment counter is reset unless CONTINUE IDENTITY is given), which matches PostgreSQL's default.
  • CASCADE recursively truncates referencing tables that contain rows, matching PostgreSQL. Without CASCADE, a table referenced by a non-empty table cannot be truncated (SQLSTATE 23503). RESTRICT simply names this default behavior.
