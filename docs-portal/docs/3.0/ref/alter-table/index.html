ALTER TABLE

Change the definition of an existing table: add, drop, rename, or retype columns; add, drop, rename, enable, disable, or validate constraints; toggle row-level security; and set or drop a TTL column.

Synopsis

ALTER TABLE table_name action

where action is one of:

-- columns
  ADD COLUMN column_name data_type [ column_attribute ... ] [ FIRST | AFTER existing_column ]
  DROP COLUMN column_name
  DROP column_name
  RENAME COLUMN old_column TO new_column
  RENAME old_column TO new_column
  MODIFY COLUMN column_name data_type [ column_attribute ... ] [ FIRST | AFTER existing_column ]
  MODIFY column_name data_type [ column_attribute ... ] [ FIRST | AFTER existing_column ]
  CHANGE COLUMN old_column new_column data_type [ column_attribute ... ] [ FIRST | AFTER existing_column ]
  CHANGE old_column new_column data_type [ column_attribute ... ] [ FIRST | AFTER existing_column ]

-- constraints
  ADD CONSTRAINT constraint_name CHECK ( expression ) [ NOT VALIDATE ]
  ADD CONSTRAINT constraint_name UNIQUE ( column [, ...] )
  ADD CONSTRAINT constraint_name PRIMARY KEY ( column [, ...] )
  ADD CONSTRAINT constraint_name FOREIGN KEY ( column [, ...] )
      REFERENCES ref_table ( ref_column [, ...] ) [ referential_action ... ] [ NOT VALIDATE ]
  DROP CONSTRAINT constraint_name
  RENAME CONSTRAINT old_name TO new_name
  ENABLE CONSTRAINT constraint_name
  DISABLE CONSTRAINT constraint_name
  VALIDATE CONSTRAINT constraint_name

-- row-level security
  ENABLE ROW LEVEL SECURITY
  DISABLE ROW LEVEL SECURITY

-- time-to-live
  SET TTL_COLUMN = 'column_name'
  DROP TTL

and referential_action is one of:

  ON DELETE { CASCADE | SET NULL | SET DEFAULT | RESTRICT | NO ACTION }
  ON UPDATE { CASCADE | SET NULL | SET DEFAULT | RESTRICT | NO ACTION }
  MATCH { FULL | PARTIAL | SIMPLE }
  [ NOT ] DEFERRABLE [ INITIALLY { DEFERRED | IMMEDIATE } ]

Description

ALTER TABLE changes the definition of an existing table. Each statement performs exactly one action, EvolutionDB does not accept a comma-separated list of actions in a single ALTER TABLE command. Issue separate statements to apply several changes.

When the named table does not exist, the command fails with relation "table_name" does not exist (SQLSTATE 42P01). All actions operate against the system catalog, which stores tables, columns, and constraints in B+ trees.

The column actions come in two spelling families. ADD COLUMN, RENAME COLUMN ... TO, DROP COLUMN follow the ANSI/PostgreSQL spelling; MODIFY and CHANGE follow MySQL spelling. The COLUMN keyword is optional for DROP, RENAME, MODIFY, and CHANGE. MODIFY retypes a column in place, while CHANGE renames and retypes it in one statement.

Adding, dropping, and reordering columns may rewrite existing rows. ADD COLUMN with a FIRST or AFTER position reorders the column in the catalog and rebuilds every tuple, filling the new column with NULL. MODIFY/CHANGE with a position clause likewise rewrites all tuples in the new column order. DROP COLUMN marks the column as dropped in the catalog rather than physically deleting it.

Adding a CHECK, PRIMARY KEY, or UNIQUE constraint validates the existing rows first and aborts the command if any row violates the new constraint, unless NOT VALIDATE is given (CHECK and FOREIGN KEY only), which records the constraint as not-validated and skips the scan. A later VALIDATE CONSTRAINT runs the deferred check.

On success, column and TTL/RLS actions report ALTER TABLE; constraint actions report command(s) completed successfully!...

Parameters

table_name
The name of the existing table to alter. Resolved through the current database and schema.
ADD COLUMN column_name data_type [ column_attribute.. ]
Appends a new column to the table. The column is added at the end unless a position clause is supplied. Fails with 42701 if a column of that name already exists, and with 54011 if the table already has the maximum number of columns. Supported attributes are the same as in CREATE TABLE, including NOT NULL, NULL, DEFAULT (literal, CURRENT_TIMESTAMP, gen_random_uuid(), gen_random_uuid_v7(), snowflake_id(), or an expression), UNIQUE, AUTO_INCREMENT / IDENTITY, CHECK (...), COMMENT, and GENERATED ALWAYS AS (...) STORED|VIRTUAL. Note: a column added this way is never made part of the primary key. If the column is NOT NULL with no DEFAULT and the table already holds rows, the command fails with 23502 because existing rows would receive a NULL.
FIRST | AFTER existing_column
Optional position clause for ADD, MODIFY, and CHANGE. FIRST places the column at ordinal 0; AFTER existing_column places it immediately after the named column. Choosing a position triggers an eager rewrite of every row into the new physical column order.
DROP COLUMN column_name  /  DROP column_name
Marks a column as dropped. Fails with 42703 if the column does not exist, and with 2BP01 if it is part of the primary key, is the only remaining active column, is referenced by a secondary index, or is referenced by a FOREIGN KEY or UNIQUE constraint.
RENAME COLUMN old_column TO new_column  /  RENAME old_column TO new_column
Renames a column. The COLUMN keyword is optional.
MODIFY [COLUMN] column_name data_type [ column_attribute.. ]
Changes the type and/or attributes of an existing column. Fails with 42703 if the column does not exist, with 2BP01 if the column is part of the primary key and its type would change, and with 42804 if the old and new types are not automatically convertible. When the new definition adds NOT NULL to a column that was previously nullable, the table is scanned and the command fails with 23502 if any existing row holds a NULL in that column.
CHANGE [COLUMN] old_column new_column data_type [ column_attribute.. ]
MySQL-style combined rename plus retype: renames old_column to new_column (when the names differ) and then applies the new type and attributes, with the same validations as MODIFY. A position clause is honored.
ADD CONSTRAINT constraint_name CHECK ( expression )
Adds a named CHECK constraint. Every existing row is evaluated against expression; if any row fails, the command aborts (23514) and reports the offending row number. Fails with 42710 if a constraint of the same name already exists.
ADD CONSTRAINT constraint_name UNIQUE ( column [..] )
Adds a named UNIQUE constraint over one or more columns. Fails with 42710 on a duplicate constraint name.
ADD CONSTRAINT constraint_name PRIMARY KEY ( column [..] )
Adds a primary key over the listed columns. Fails with 42710 if the table already has a primary key, with 42703 if a named column does not exist, with the not-null violation code if any key column contains NULLs, and with the unique violation code if the existing rows contain duplicate key values.
ADD CONSTRAINT constraint_name FOREIGN KEY ( column [..] ) REFERENCES ref_table ( ref_column [..] ) [ referential_action.. ]
Adds a foreign key referencing ref_table. Fails with 42P01 if the referenced table does not exist and with 42710 on a duplicate constraint name.
referential_action
Zero or more clauses attached to a FOREIGN KEY: ON DELETE and ON UPDATE each accept CASCADE, SET NULL, SET DEFAULT, RESTRICT, or NO ACTION; MATCH accepts FULL, PARTIAL, or SIMPLE; and the constraint may be marked [NOT] DEFERRABLE, optionally with INITIALLY DEFERRED or INITIALLY IMMEDIATE.
NOT VALIDATE
Available on ADD CONSTRAINT ... CHECK and ADD CONSTRAINT ... FOREIGN KEY. Records the constraint as not-validated and skips the scan of existing rows, so the constraint applies only to subsequent writes until a later VALIDATE CONSTRAINT is run.
DROP CONSTRAINT constraint_name
Removes a constraint by name. Fails with 42704 if no such constraint exists on the table.
RENAME CONSTRAINT old_name TO new_name
Renames a constraint. Fails with 42704 if old_name is not found and 42710 if new_name is already in use.
ENABLE CONSTRAINT constraint_name  /  DISABLE CONSTRAINT constraint_name
Toggles the enabled flag of a constraint. A disabled constraint is not enforced on subsequent writes. Fails with 42704 if the constraint does not exist.
VALIDATE CONSTRAINT constraint_name
Validates a constraint previously added with NOT VALIDATE by checking it against the existing rows and marking it validated.
ENABLE / DISABLE ROW LEVEL SECURITY
Sets or clears the row-level-security flag on the table. The keyword sequence must be exactly ROW LEVEL SECURITY; any other word after ROW is a syntax error.
SET TTL_COLUMN = 'column_name'
Designates a timestamp column as the table's time-to-live column. The value is a single-quoted string literal. Fails if the named column does not exist on the table.
DROP TTL
Clears the table's TTL column designation.

Examples

Add a nullable column, then a column with a default placed after an existing one:

ALTER TABLE employees ADD COLUMN nickname VARCHAR(50);
ALTER TABLE employees ADD COLUMN created_at TIMESTAMP
    DEFAULT CURRENT_TIMESTAMP AFTER nickname;

Rename and retype columns (PostgreSQL and MySQL spellings):

ALTER TABLE employees RENAME COLUMN nickname TO display_name;
ALTER TABLE employees MODIFY COLUMN display_name VARCHAR(100) NOT NULL;
ALTER TABLE employees CHANGE display_name full_name VARCHAR(120);

Drop a column:

ALTER TABLE employees DROP COLUMN full_name;

Add constraints:

ALTER TABLE employees ADD CONSTRAINT chk_age CHECK (age >= 18);
ALTER TABLE employees ADD CONSTRAINT uq_email UNIQUE (email);
ALTER TABLE employees ADD CONSTRAINT pk_emp PRIMARY KEY (id);
ALTER TABLE orders ADD CONSTRAINT fk_emp FOREIGN KEY (emp_id)
    REFERENCES employees (id) ON DELETE CASCADE;

Add a foreign key without validating existing data, then validate it later:

ALTER TABLE orders ADD CONSTRAINT fk_emp2 FOREIGN KEY (emp_id)
    REFERENCES employees (id) NOT VALIDATE;
ALTER TABLE orders VALIDATE CONSTRAINT fk_emp2;

Rename, disable, re-enable, and drop a constraint:

ALTER TABLE employees RENAME CONSTRAINT chk_age TO chk_min_age;
ALTER TABLE employees DISABLE CONSTRAINT chk_min_age;
ALTER TABLE employees ENABLE CONSTRAINT chk_min_age;
ALTER TABLE employees DROP CONSTRAINT chk_min_age;

Toggle row-level security and manage a TTL column:

ALTER TABLE sessions ENABLE ROW LEVEL SECURITY;
ALTER TABLE sessions SET TTL_COLUMN = 'expires_at';
ALTER TABLE sessions DROP TTL;
ALTER TABLE sessions DISABLE ROW LEVEL SECURITY;

Compatibility

The SQL standard defines ALTER TABLE with ADD, DROP, and ALTER COLUMN subforms. EvolutionDB deliberately blends PostgreSQL and MySQL spellings:

  • PostgreSQL-style forms supported: ADD COLUMN, DROP COLUMN, RENAME COLUMN ... TO, named ADD CONSTRAINT for CHECK / UNIQUE / PRIMARY KEY / FOREIGN KEY, DROP CONSTRAINT, RENAME CONSTRAINT, VALIDATE CONSTRAINT, the referential MATCH and DEFERRABLE clauses, and ENABLE / DISABLE ROW LEVEL SECURITY.
  • MySQL-style forms supported: MODIFY [COLUMN], CHANGE [COLUMN], the optional COLUMN keyword on DROP/RENAME, and the FIRST / AFTER column-position clauses.
  • Differences from PostgreSQL. PostgreSQL spells deferred validation NOT VALID; EvolutionDB uses NOT VALIDATE. PostgreSQL writes ALTER COLUMN ... TYPE and SET/DROP NOT NULL; EvolutionDB instead retypes through MODIFY/CHANGE. ENABLE / DISABLE CONSTRAINT is an EvolutionDB extension (PostgreSQL has no per-constraint enable flag). EvolutionDB does not accept multiple comma-separated actions in one statement.
  • EvolutionDB extensions. SET TTL_COLUMN = '...' and DROP TTL manage automatic row expiry and have no SQL-standard or PostgreSQL equivalent.
  • Not supported. Clauses such as ALTER COLUMN ... SET DEFAULT, SET/DROP NOT NULL, OWNER TO, SET SCHEMA, SET TABLESPACE, INHERIT, and RENAME TO (table rename, which is a separate statement) are not part of the ALTER TABLE grammar.
