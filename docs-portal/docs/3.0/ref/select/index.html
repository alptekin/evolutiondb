SELECT

Retrieve rows from one or more tables, subqueries, or set-returning sources, with optional filtering, grouping, ordering, and row limiting.

Synopsis

SELECT [ ALL | DISTINCT | DISTINCTROW ] [ HIGH_PRIORITY ] [ STRAIGHT_JOIN ]
       [ SQL_SMALL_RESULT ] [ SQL_BIG_RESULT ] [ SQL_CALC_FOUND_ROWS ]
    { * | select_expr [ [ AS ] output_alias ] [, ...] }
    [ INTO new_table ]
    FROM from_item [, ...]
    [ FOR SYSTEM_TIME AS OF TRANSACTION xid ]
    [ WHERE condition ]
    [ GROUP BY expression [ ASC | DESC ] [, ...] [ WITH ROLLUP ] ]
    [ HAVING condition ]
    [ ORDER BY order_item [, ...] ]
    [ LIMIT { count | offset , count | count OFFSET offset } ]
    [ FOR UPDATE | FOR SHARE [ SKIP LOCKED ] ]
    [ INTO column [, ...] ]

where from_item is one of:

    table_name [ [ AS ] alias ] [ index_hint ]
    ONLY table_name [ [ AS ] alias ]
    schema.table_name [ [ AS ] alias ]
    ( select_stmt ) [ AS ] alias
    LATERAL ( select_stmt ) [ AS ] alias
    UNNEST ( array_expr ) [ [ AS ] alias [ ( col ) ] ]
    from_item [ INNER | CROSS ] JOIN from_item [ join_condition ]
    from_item STRAIGHT_JOIN from_item [ ON condition ]
    from_item { LEFT | RIGHT } [ OUTER ] JOIN from_item join_condition
    from_item NATURAL [ { LEFT | RIGHT } [ OUTER ] ] JOIN from_item

and join_condition is one of:

    ON condition
    USING ( column [, ...] )

and order_item is one of:

    column_name [ ASC | DESC ]
    table.column_name [ ASC | DESC ]
    column_position [ ASC | DESC ]
    vector_column <=> 'vector_literal' [ ASC | DESC ]

and index_hint is one of:

    { USE | IGNORE | FORCE } KEY [ FOR JOIN ] ( index [, ...] )

Description

SELECT reads rows from the database and returns them as a result set. EvolutionDB accepts two principal shapes, both defined by the select_stmt rule in evoparser.y:

  1. A projection-only form, SELECT select_expr_list with no FROM, which evaluates the listed expressions once and returns a single row. (This is how SELECT 1, SELECT NOW(), etc. are handled.)
  2. The full table query form, which adds FROM and the optional WHERE, GROUP BY, HAVING, ORDER BY, LIMIT, and row-locking clauses.

The general processing order of a table query is:

  1. Each from_item in FROM is scanned, then joins are applied (the leftmost item is the driving relation; subsequent items default to INNER JOIN unless an explicit join type is given).
  2. If WHERE is present, its condition is evaluated against each candidate row; rows for which it is not true are discarded.
  3. If GROUP BY is present, surviving rows are collapsed into groups; aggregate functions in the select list are computed per group. WITH ROLLUP adds super-aggregate summary rows.
  4. If HAVING is present, its condition filters the groups.
  5. The select list is evaluated to form the output rows. DISTINCT / DISTINCTROW then removes duplicate rows.
  6. If ORDER BY is present, rows are sorted. For non-numeric values the comparison is lexical; values that parse cleanly as numbers are compared numerically (see CompareRecords in Select.c).
  7. LIMIT / OFFSET restricts how many rows are returned.
  8. If FOR UPDATE / FOR SHARE is present, the returned rows are locked.

The simple single-table path inside the engine resolves the table from the catalog, performs either a primary-key lookup (when a single equality on the PK is given) or a sequential table scan via the buffer pool, decodes the binary tuple format, and prints the matching rows.

Note. The engine's built-in text renderer for the SELECT * path buffers rows for in-memory ORDER BY sorting in fixed-size storage. When driven through the dual-protocol server (the normal way clients connect), result streaming and richer expression handling are managed by the adaptor's query executor.

Parameters

ALL / DISTINCT / DISTINCTROW
Duplicate-row treatment. ALL (the default) returns every row. DISTINCT eliminates duplicate result rows; DISTINCTROW is accepted as a synonym. Specifying the same option twice is a parse error ("duplicate … option").
HIGH_PRIORITY, STRAIGHT_JOIN, SQL_SMALL_RESULT, SQL_BIG_RESULT, SQL_CALC_FOUND_ROWS
MySQL-compatibility select options that are accepted by the grammar as leading keywords. They are parsed without error so MySQL-style statements load; only DISTINCT/DISTINCTROW alter result semantics in this position.
* / select_expr [ [AS] output_alias ]
The select list. * projects all columns of the source. Otherwise each item is an expression (a column reference, literal, arithmetic/string/aggregate/window function call, etc.) optionally renamed with AS alias (the AS keyword is optional).
from_item (FROM)
One or more sources, comma-separated (comma forms an implicit cross/inner join). A source may be a table (table_name or schema.table_name), a parenthesized subquery aliased with a name, a LATERAL subquery, an UNNEST(array_expr) set-returning source, a parenthesized join, or an explicit join (see below).
ONLY table_name
In a table source, ONLY restricts the scan to the named parent table and suppresses rows that would otherwise be drawn from inheriting child tables.
JOIN clauses
EvolutionDB accepts INNER JOIN, CROSS JOIN, plain JOIN (treated as inner), STRAIGHT_JOIN, LEFT [OUTER] JOIN, RIGHT [OUTER] JOIN, and NATURAL joins (optionally LEFT/RIGHT OUTER). The join predicate is supplied with ON condition or USING ( column [, ...] ); outer joins require an explicit condition.
index_hint
An optional per-table hint: USE KEY, IGNORE KEY, or FORCE KEY, each over a parenthesized index list and optionally scoped to joins with FOR JOIN.
FOR SYSTEM_TIME AS OF TRANSACTION xid
A temporal (time-travel) qualifier: evaluate the query against the snapshot of the database as of the given transaction id xid rather than the current snapshot.
WHERE condition
A boolean expression; only rows for which it evaluates to true are passed on. Supports comparisons, AND/OR/NOT, IN (...) and IN (subquery), EXISTS (subquery), scalar/quantified subqueries, and vector-distance operators.
GROUP BY expression [ASC | DESC] [..] [WITH ROLLUP]
Groups rows that share values over the listed expressions so aggregate functions are computed per group. An optional ASC/DESC may follow each grouping expression. WITH ROLLUP appends super-aggregate (subtotal/grand-total) rows.
HAVING condition
A boolean condition applied to grouped rows (typically referencing aggregates), filtering whole groups after GROUP BY.
ORDER BY order_item [..]
Sort key list. Each item is a column name, a qualified table.column, a 1-based column position (e.g. ORDER BY 2), or a vector nearest-neighbor expression column <=> 'vector_literal' using the cosine-distance operator <=>. Each key may be followed by ASC (default) or DESC. Note: in ORDER BY expression position only the <=> operator is permitted; other comparison operators are rejected.
LIMIT { count | offset, count | count OFFSET offset }
Restricts the number of rows returned. Three forms are accepted: LIMIT count; the MySQL two-argument form LIMIT offset, count; and LIMIT count OFFSET offset.
FOR UPDATE | FOR SHARE [ SKIP LOCKED ]
Row-level locking of the returned rows. FOR UPDATE takes an exclusive row lock; FOR SHARE takes a shared lock. The optional SKIP LOCKED modifier skips rows that are already locked rather than waiting on them.
INTO column [..]
The trailing variable-target list form: directs the selected values into the named targets. (Distinct from the table-creating SELECT ... INTO new_table FROM ... form described below.)
SELECT.. INTO new_table FROM..
A PostgreSQL-style table-creating variant: the result of the query is materialized into a brand-new table new_table whose columns are inferred from the select list. (Internally this drives the CREATE-TABLE-AS / inferred-schema path.)

Examples

Select every column from a table:

SELECT * FROM employees;

Project specific columns with an output alias and filter rows:

SELECT name, salary AS pay
FROM employees
WHERE dept_id = 10;

Eliminate duplicates and sort descending:

SELECT DISTINCT dept_id
FROM employees
ORDER BY dept_id DESC;

Aggregate per group, filter the groups, and keep only the top rows:

SELECT dept_id, COUNT(*) AS headcount, AVG(salary) AS avg_pay
FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 3
ORDER BY headcount DESC
LIMIT 5;

Inner join with an ON condition:

SELECT e.name, d.dept_name
FROM employees AS e
INNER JOIN departments AS d ON e.dept_id = d.id;

Left outer join with USING:

SELECT e.name, d.dept_name
FROM employees e
LEFT OUTER JOIN departments d USING (dept_id);

MySQL-style two-argument LIMIT (offset, count) for paging:

SELECT name FROM employees ORDER BY name LIMIT 20, 10;

Vector nearest-neighbor search using the cosine-distance operator:

SELECT id FROM documents
ORDER BY embedding <=> '[0.12, 0.84, 0.05]'
LIMIT 5;

Time-travel query against a past transaction snapshot:

SELECT * FROM accounts
FOR SYSTEM_TIME AS OF TRANSACTION 4200;

Lock the matched rows for update, skipping rows already locked:

SELECT * FROM jobs
WHERE status = 'pending'
FOR UPDATE SKIP LOCKED;

Materialize a query result into a new table (PostgreSQL-style SELECT INTO):

SELECT name, salary
INTO high_earners
FROM employees
WHERE salary > 100000;

Expand an array into rows with UNNEST:

SELECT v FROM UNNEST(ARRAY[1, 2, 3]) AS t(v);

Compatibility

EvolutionDB's SELECT blends SQL-standard, PostgreSQL, and MySQL spellings:

  • SQL standard / PostgreSQL. DISTINCT, the join family (INNER, CROSS, LEFT/RIGHT [OUTER], NATURAL, ON/USING), GROUP BY / HAVING / ORDER BY, LATERAL subqueries, UNNEST, row locking (FOR UPDATE / FOR SHARE with SKIP LOCKED), the SELECT ... INTO new_table table-creating form, and the SQL:2011 temporal opener FOR SYSTEM_TIME AS OF all follow standard / PostgreSQL conventions. Note the temporal clause takes TRANSACTION xid (a transaction id) rather than a timestamp.
  • MySQL extensions. DISTINCTROW, the leading options HIGH_PRIORITY, STRAIGHT_JOIN, SQL_SMALL_RESULT, SQL_BIG_RESULT, SQL_CALC_FOUND_ROWS, the two-argument LIMIT offset, count form, WITH ROLLUP on GROUP BY, and the index hints USE/IGNORE/FORCE KEY are MySQL-style. Several of the leading options are accepted for source compatibility but do not change result semantics.
  • EvolutionDB-specific. The vector-distance operator <=> (cosine distance) is allowed in ORDER BY expression position for nearest-neighbor search; other comparison operators are rejected there.
  • Not part of this command. The grammar's select_stmt rule does not define WITH (common table expressions), set operations (UNION/INTERSECT/EXCEPT), WINDOW-clause named windows, FETCH FIRST, or OFFSET as a standalone clause, only the LIMIT forms shown above. Window functions appear via function expressions in the select list rather than as a separate top-level clause.
