LISTEN / UNLISTEN / NOTIFY

Subscribe to, unsubscribe from, and publish messages on a named asynchronous notification channel.

Synopsis

LISTEN channel [ SELF ]

UNLISTEN { channel | * }

NOTIFY channel [ , 'payload' ]

Description

LISTEN registers the current session as a listener on the notification channel named channel. NOTIFY sends a notification — together with an optional payload string — to every session currently listening on the matching channel. UNLISTEN removes the current session's registration for one channel, or for all channels when * is given.

The notification registry is held entirely in server memory and is shared between both front-end protocols: connections speaking the PostgreSQL wire protocol (port 5433) and connections speaking the native EVO text protocol (port 9967) can listen and notify across each other through the same channel names. A session's listens are automatically dropped when its connection closes.

Channel names are matched case-insensitively (they are folded to lower case for lookup). A channel name may be at most 63 bytes long; longer names are rejected.

Transactional delivery. NOTIFY is commit-gated. A NOTIFY issued inside an explicit transaction block is not delivered immediately — its channel and payload are enqueued on the session and the messages are flushed only when the transaction commits. If the transaction rolls back, the pending notifications are discarded and nothing is delivered. In auto-commit mode (no open BEGIN), the statement commits on its own, so the notification is flushed as soon as the NOTIFY statement finishes. By contrast, LISTEN and UNLISTEN take effect immediately when the statement executes; they are not deferred to commit.

Self-notification. By default a session does not receive notifications it sends on a channel it is itself listening on. To opt in to same-session delivery, register the listen with the SELF keyword (LISTEN channel SELF). Re-running LISTEN for a channel the session is already listening on simply updates the self-delivery flag in place rather than creating a duplicate registration.

Delivery format. On the PostgreSQL wire protocol a notification is delivered as a NotificationResponse ('A') message carrying the sending session id, the channel name, and the payload. On the EVO text protocol it arrives as an asynchronous NOTIFY <sender-session> <channel> <payload-length> line followed by the payload bytes.

Parameters

channel
Name of the notification channel, given as an identifier. Maximum 63 bytes; matching is case-insensitive. A name longer than the limit (or otherwise invalid) raises SQLSTATE 42622 (name too long).
SELF
Only valid with LISTEN. Opts the current session in to receiving its own NOTIFY messages on this channel. Without SELF, a session's own notifications on a channel it listens to are suppressed (the PostgreSQL default behavior). This is EvolutionDB's single-keyword spelling of "deliver to self"; there is no WITH (self = …) clause.
*
Only valid with UNLISTEN. Cancels every channel registration owned by the current session in one statement.
payload
Optional string literal sent along with the notification, separated from the channel by a comma. It is delivered verbatim to every listener. The payload may be at most 8000 bytes; a longer payload raises SQLSTATE 22023 (payload string too long). When no payload is supplied, an empty payload is delivered.

Note. EvolutionDB accepts a quoted string literal as the NOTIFY payload. It does not support passing the payload as an arbitrary expression in the NOTIFY statement form. To build a payload dynamically — for example from query values — use the built-in function evo_notify(channel, payload), which enqueues a notification using the same commit-gated path. The companion function pg_listening_channels() returns the channels the current session is listening on.

Examples

Configure a session to receive event notifications and then publish to it:

LISTEN orders_changed;

NOTIFY orders_changed;
NOTIFY orders_changed, 'order 4711 shipped';

Receive your own notifications on a channel (opt in with SELF):

LISTEN heartbeat SELF;
NOTIFY heartbeat, 'ping';   -- this session also receives 'ping'

Defer delivery to commit: the listener sees the notification only after COMMIT, and never if the transaction is rolled back:

BEGIN;
NOTIFY orders_changed, 'pending';
COMMIT;            -- 'pending' is delivered here

Stop listening on one channel, then on everything:

UNLISTEN orders_changed;
UNLISTEN *;

Build a payload dynamically with the helper function, and inspect current subscriptions:

SELECT evo_notify('orders_changed', 'order ' || id || ' shipped')
FROM orders WHERE id = 4711;

SELECT pg_listening_channels();

Compatibility

There is no LISTEN, UNLISTEN, or NOTIFY command in the SQL standard; these commands follow PostgreSQL.

Differences from PostgreSQL worth noting:

  • EvolutionDB delivers same-session notifications only when the listen was registered with the LISTEN channel SELF form. PostgreSQL controls this with the NOTIFY-time behavior and lacks a SELF keyword; SELF is an EvolutionDB extension and the default (no SELF) matches PostgreSQL's default of not echoing a session's own notifications.
  • The NOTIFY payload must be a string literal. PostgreSQL likewise requires a string literal here; EvolutionDB additionally offers the evo_notify() function for computed payloads.
  • EvolutionDB's payload limit is 8000 bytes and channel names are limited to 63 bytes, consistent with PostgreSQL's limits.
  • Like PostgreSQL, notifications are commit-gated: messages sent inside a transaction are delivered only on COMMIT and dropped on rollback.
  • The notification channel registry is in-memory only and shared across both the PostgreSQL and EVO protocols; it does not persist across a server restart. (For persistent, replayable subscriptions, see CREATE SUBSCRIPTION.)
