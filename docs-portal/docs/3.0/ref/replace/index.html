REPLACE

REPLACE: insert a row, or if a row with the same primary key already exists, delete that row first and then insert the new one.

Synopsis

REPLACE [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY ] [ IGNORE ] [ INTO ] table_name
    [ ( column [, ...] ) ]
    VALUES ( { expression | DEFAULT } [, ...] ) [, ...]

REPLACE [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY ] [ IGNORE ] [ INTO ] table_name
    SET column = { expression | DEFAULT } [, ...]

REPLACE [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY ] [ IGNORE ] [ INTO ] table_name
    [ ( column [, ...] ) ]
    select_statement

Description

REPLACE writes rows into table_name. It behaves exactly like INSERT with one difference: when the new row would collide with an existing row on the table's primary key, the existing row is deleted and the new row is inserted in its place, rather than the statement failing with a duplicate-key error.

Conflict detection is driven entirely by the table's primary key. EvolutionDB builds the primary-key value from the supplied column values, probes the primary-key B+ tree, and:

  • If no matching key is found, the row is inserted normally, identical to INSERT.
  • If a matching key is found, the engine deletes the existing row (an MVCC delete under the current transaction), then inserts the new row.

A REPLACE is therefore a delete-plus-insert, not an in-place update: the replacement row is a brand-new tuple, any columns not provided take their defaults (or NULL), and any BEFORE INSERT triggers fire for the inserted row. The operation runs under the statement's transaction, so it is rolled back together with the surrounding work if the transaction aborts.

Three forms are accepted, all sharing the same conflict semantics:

  • VALUES: one or more parenthesized row constructors, optionally preceded by an explicit column list.
  • SET, MySQL-style assignment form, listing column = value pairs for a single row.
  • SELECT: populate the target from the result of a query, optionally with an explicit column list.
Requires a primary key. REPLACE's conflict resolution operates on the primary-key index. A target table without a usable primary key has nothing to collide on, so REPLACE simply degrades to a plain insert and will raise a duplicate-key error only where the same key value recurs.

Parameters

LOW_PRIORITY | DELAYED | HIGH_PRIORITY
Priority hints carried over from the INSERT grammar. They are accepted for MySQL syntax compatibility but do not change scheduling in EvolutionDB; statements execute under the engine's single serialized write path regardless of the hint.
IGNORE
Accepted modifier shared with INSERT. It is parsed in the same option position as the priority hints. Because REPLACE already resolves primary-key collisions by replacing the existing row, IGNORE has no additional effect on the conflict path.
INTO
Optional noise word before the table name, accepted for compatibility. REPLACE table and REPLACE INTO table are equivalent.
table_name
The table that receives the rows. It must already exist.
column [..]
An optional list of target columns. When given (VALUES and SELECT forms), values are matched positionally to these columns; columns omitted from the list take their default value (or NULL). When omitted, values are matched against the table's columns in definition order.
expression
A value expression producing the column value. Within a VALUES row or a SET assignment, the literal keyword DEFAULT may be used instead of an expression to request the column's declared default.
DEFAULT
Substitutes the column's default value (or NULL when no default is declared) for that position.
select_statement
A query whose result rows are replaced into the target. Each result row is processed with the same primary-key conflict resolution as a literal VALUES row.
SET column = value [..]
The single-row assignment form. Each pair sets one column; columns not listed take their default value (or NULL).

Examples

Create a table keyed by a primary key, then replace into it:

CREATE TABLE products (
    id     INT PRIMARY KEY,
    name   VARCHAR(50),
    price  INT
);

-- No row with id = 1 yet: behaves like INSERT
REPLACE INTO products (id, name, price) VALUES (1, 'Keyboard', 40);

-- id = 1 already exists: old row is deleted, new row inserted in its place
REPLACE INTO products (id, name, price) VALUES (1, 'Keyboard Pro', 65);

Multi-row VALUES, with one new key and one colliding key:

REPLACE INTO products (id, name, price)
VALUES (2, 'Mouse', 25),
       (1, 'Keyboard Max', 80);

The INTO keyword is optional, and a column may be set to its default:

REPLACE products (id, name, price) VALUES (3, 'Cable', DEFAULT);

The SET form replaces a single row:

REPLACE INTO products
SET id = 1, name = 'Keyboard Ultra', price = 95;

The SELECT form copies rows from another table, replacing on key collisions:

REPLACE INTO products (id, name, price)
SELECT id, name, price FROM staging_products;

Compatibility

REPLACE is not part of the SQL standard, and PostgreSQL does not have a REPLACE statement. EvolutionDB's REPLACE follows the MySQL/MariaDB dialect: the same three forms (VALUES, SET, and ... SELECT), the optional INTO keyword, and the priority/IGNORE modifiers are all accepted in the MySQL syntax positions.

Differences and limitations to be aware of:

  • Primary key only. Like MySQL, EvolutionDB resolves the conflict by deleting the existing row and inserting the new one. EvolutionDB performs this resolution against the table's primary key; it does not evaluate additional UNIQUE constraints as separate conflict targets.
  • Priority hints are no-ops. LOW_PRIORITY, DELAYED, and HIGH_PRIORITY are accepted for syntactic compatibility but do not affect scheduling.
  • Portable alternative. For standard-conforming upsert behavior, prefer INSERT ... ON CONFLICT DO UPDATE (PostgreSQL style) or INSERT ... ON DUPLICATE KEY UPDATE (MySQL style), both supported by EvolutionDB's INSERT. Unlike REPLACE, those forms update the existing row in place rather than deleting and re-inserting it.
