REPLACE

REPLACE inserts a row. If a row with the same primary key already exists, it deletes that row first. Then it inserts the new one.

Synopsis

REPLACE [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY ] [ IGNORE ] [ INTO ] table_name
    [ ( column [, ...] ) ]
    VALUES ( { expression | DEFAULT } [, ...] ) [, ...]

REPLACE [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY ] [ IGNORE ] [ INTO ] table_name
    SET column = { expression | DEFAULT } [, ...]

REPLACE [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY ] [ IGNORE ] [ INTO ] table_name
    [ ( column [, ...] ) ]
    select_statement

Description

REPLACE writes rows into table_name. It works just like INSERT, but with one difference. The new row may clash with an existing row on the table's primary key. When this happens, the old row is deleted and the new row takes its place. The statement does not fail with a duplicate-key error.

Conflict detection uses only the table's primary key. EvolutionDB builds the primary-key value from the column values you give. It then checks the primary-key B+ tree.

  • If no matching key is found, the row is inserted in the normal way. This is the same as INSERT.
  • If a matching key is found, the engine deletes the old row. This is an MVCC delete under the current transaction. Then it inserts the new row.

So a REPLACE is a delete plus an insert. It is not an in-place update. The replacement row is a brand-new tuple. Any columns you do not provide take their defaults or NULL. Any BEFORE INSERT triggers fire for the inserted row. The operation runs inside the statement's transaction. If the transaction aborts, it is rolled back with the rest of the work.

Three forms are accepted. They all share the same conflict behavior.

  • VALUES uses one or more rows in parentheses. You can add a column list before them.
  • SET is the MySQL-style assignment form. It lists column = value pairs for a single row.
  • SELECT fills the target from the result of a query. You can add a column list.
Requires a primary key. REPLACE resolves conflicts using the primary-key index. A table without a usable primary key has nothing to collide on. In that case REPLACE acts as a plain insert. It raises a duplicate-key error only when the same key value comes up again.

Parameters

LOW_PRIORITY | DELAYED | HIGH_PRIORITY
Priority hints carried over from the INSERT grammar. They are accepted for MySQL syntax compatibility. They do not change scheduling in EvolutionDB. Statements run on the engine's single serialized write path no matter which hint you use.
IGNORE
A modifier shared with INSERT. It is parsed in the same option spot as the priority hints. REPLACE already resolves primary-key collisions by replacing the old row. So IGNORE has no extra effect on the conflict path.
INTO
An optional word before the table name. It is accepted for compatibility. REPLACE table and REPLACE INTO table mean the same thing.
table_name
The table that receives the rows. It must already exist.
column [..]
An optional list of target columns. You can use it with the VALUES and SELECT forms. When you give it, values are matched to these columns by position. Columns left out of the list take their default value or NULL. When you leave the list out, values are matched to the table's columns in the order they were defined.
expression
A value expression that gives the column value. Inside a VALUES row or a SET assignment, you can use the keyword DEFAULT instead of an expression. This asks for the column's declared default.
DEFAULT
Puts the column's default value in that spot. It uses NULL when no default is declared.
select_statement
A query whose result rows are replaced into the target. Each result row uses the same primary-key conflict handling as a VALUES row.
SET column = value [..]
The single-row assignment form. Each pair sets one column. Columns you do not list take their default value or NULL.

Examples

Create a table with a primary key. Then replace into it.

CREATE TABLE products (
    id     INT PRIMARY KEY,
    name   VARCHAR(50),
    price  INT
);

-- No row with id = 1 yet: behaves like INSERT
REPLACE INTO products (id, name, price) VALUES (1, 'Keyboard', 40);

-- id = 1 already exists: old row is deleted, new row inserted in its place
REPLACE INTO products (id, name, price) VALUES (1, 'Keyboard Pro', 65);

A multi-row VALUES form. One key is new and one key collides.

REPLACE INTO products (id, name, price)
VALUES (2, 'Mouse', 25),
       (1, 'Keyboard Max', 80);

The INTO keyword is optional. A column can be set to its default.

REPLACE products (id, name, price) VALUES (3, 'Cable', DEFAULT);

The SET form replaces a single row.

REPLACE INTO products
SET id = 1, name = 'Keyboard Ultra', price = 95;

The SELECT form copies rows from another table. It replaces rows on key collisions.

REPLACE INTO products (id, name, price)
SELECT id, name, price FROM staging_products;

Compatibility

REPLACE is not part of the SQL standard. PostgreSQL does not have a REPLACE statement. EvolutionDB's REPLACE follows the MySQL/MariaDB dialect. It accepts the same three forms VALUES, SET, and ... SELECT. It also accepts the optional INTO keyword and the priority and IGNORE modifiers in the MySQL syntax spots.

Here are the differences and limits to keep in mind.

  • Primary key only. Like MySQL, EvolutionDB resolves the conflict by deleting the old row and inserting the new one. EvolutionDB does this against the table's primary key. It does not treat extra UNIQUE constraints as separate conflict targets.
  • Priority hints do nothing. LOW_PRIORITY, DELAYED, and HIGH_PRIORITY are accepted for syntax compatibility. They do not affect scheduling.
  • Portable alternative. For standard upsert behavior, use INSERT ... ON CONFLICT DO UPDATE in the PostgreSQL style or INSERT ... ON DUPLICATE KEY UPDATE in the MySQL style. EvolutionDB's INSERT supports both. Unlike REPLACE, those forms update the old row in place. They do not delete and re-insert it.
