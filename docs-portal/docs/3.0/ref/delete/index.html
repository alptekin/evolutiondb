DELETE

Remove rows from a table.

Synopsis

DELETE [ LOW_PRIORITY ] [ QUICK ] [ IGNORE ] FROM table_name
    [ WHERE condition ]
    [ ORDER BY column [ ASC | DESC ] [, ...] ]
    [ LIMIT count [ OFFSET start | , count ] ]
    [ RETURNING * | output_column [, ...] ]

-- Multi-table form 1 (delete from joined targets)
DELETE [ LOW_PRIORITY ] [ QUICK ] [ IGNORE ] target_table [ .* ] [, ...]
    FROM table_references
    [ WHERE condition ]

-- Multi-table form 2 (USING)
DELETE [ LOW_PRIORITY ] [ QUICK ] [ IGNORE ] FROM target_table [ .* ] [, ...]
    USING table_references
    [ WHERE condition ]

Description

DELETE removes rows from table_name. It removes the rows that match the WHERE condition. If you leave out the WHERE clause, the statement deletes all visible rows in the table. The table stays valid but becomes empty.

EvolutionDB accepts three forms of DELETE.

  • A single-table delete. This is the form DELETE FROM table_name ....
  • A multi-table delete. You can name the target tables before FROM with DELETE t1, t2 FROM .... You can also name them after FROM and give the source tables in a separate USING list. Both multi-table forms run a join over table_references. Then they delete the matched rows from each named target.

The single-table form deletes in two steps. First it collects the rows that match the WHERE expression. Then it removes them. The removal is an MVCC soft delete. The row's xmax is set to the current transaction id. The primary-key index entry stays in place. This lets other readers still find the tuple and check whether it is visible to them. Space is freed later by the background reclaim process. This happens once xmax is committed and no snapshot can still see the row. Some tuples were written before MVCC existed and have no MVCC header. For those, the heap slot and index entry are removed right away.

Each delete is transactional. The earlier version of every removed row is written to the undo log. So a ROLLBACK restores the rows. A disconnect also rolls back and restores them. An exclusive row lock is taken for each row. If a lock cannot be acquired, the statement fails with a lock or deadlock error. The whole statement is then rolled back.

Referential integrity. EvolutionDB checks every foreign-key constraint that points to the row's table before it removes the row. The result depends on the constraint's ON DELETE action. The engine may reject the delete for RESTRICT or NO ACTION. It may delete the child rows that reference the row for CASCADE. It may set the child foreign-key columns to NULL for SET NULL. It may set them to their column default for SET DEFAULT. Cascades go down at most 32 levels. A deeper chain raises a foreign-key violation. A circular reference is one example.

Triggers, indexes, and policies. BEFORE DELETE and AFTER DELETE triggers run once for each row when the table defines them. The secondary index entries for the row are removed. Row-level security may be enabled on the table. In that case the session user's DELETE USING policy is checked for each row. Rows that the policy does not expose are skipped without any error. So the deleted count counts only the rows the user is allowed to delete.

ORDER BY is parsed but not applied. The single-table grammar accepts an ORDER BY clause for MySQL syntax compatibility. But EvolutionDB's execution engine does not order the rows it deletes. So ORDER BY has no effect on which rows are removed. Use WHERE to control the row set. Add LIMIT if you need to cap the count.

Parameters

LOW_PRIORITY, QUICK, IGNORE
MySQL-style delete options. The grammar accepts them for compatibility. They are parsed into an option bitmask. They do not change how EvolutionDB deletes rows.
table_name
The name of the table you delete rows from. This is used in the single-table form. The name is looked up in the system catalog in the current database and schema. If the table does not exist, the statement reports an error and removes no rows.
target_table [ .* ]
In the multi-table forms, a table whose matched rows are deleted. You can list one or more targets. Separate them with commas. The .* suffix after a target name is optional and is accepted. In the first form the targets come before FROM. In the second form they come after FROM and the join source comes after USING.
table_references
The join expression that supplies the rows to test in the multi-table forms. It uses the same table-reference syntax as SELECT, including joins. The rows that this join produces decide which rows are removed from each target_table.
WHERE condition
An expression that returns a boolean. Only rows where it is true are deleted. A result of true, t, or 1 counts as a match. Any non-zero numeric value counts as a match too. If you leave it out, every visible row qualifies.
ORDER BY column [ ASC | DESC ]
The single-table grammar accepts it. The executor does not honor it. See the callout above.
LIMIT count [ OFFSET start ]
Caps how many of the matched rows are deleted. This is used in the single-table form. Both the LIMIT count OFFSET start spelling and the MySQL LIMIT start, count spelling are accepted. The limit and offset window is applied to the collected set of matching rows before they are removed.
RETURNING * | output_column
Makes DELETE return values from the rows it removed. This is used in the single-table form. RETURNING * returns all columns of each deleted row. Otherwise it returns the columns you list. The values show the row as it was right before deletion.

Examples

Delete one row by its primary key.

DELETE FROM employees WHERE id = 42;

Delete all rows that match a condition.

DELETE FROM orders WHERE status = 'cancelled';

Empty a table with no WHERE clause.

DELETE FROM session_log;

Delete at most 100 matching rows.

DELETE FROM events WHERE level = 'debug' LIMIT 100;

Delete rows and return the rows you removed.

DELETE FROM products WHERE discontinued = true
    RETURNING id, name;

Multi-table delete with a join. This is the first form. It removes order lines whose parent order no longer exists.

DELETE order_lines
    FROM order_lines LEFT JOIN orders
         ON order_lines.order_id = orders.id
    WHERE orders.id IS NULL;

Multi-table delete with USING. This is the second form.

DELETE FROM order_lines
    USING orders
    WHERE order_lines.order_id = orders.id
      AND orders.status = 'voided';

Compatibility

The single-table DELETE FROM table WHERE ... form follows the SQL standard and PostgreSQL closely. The RETURNING clause is included. EvolutionDB's RETURNING works like PostgreSQL's. You use * or a list of column names. It gives back the row values from before the delete.

  • The LOW_PRIORITY, QUICK, and IGNORE options are MySQL extensions. The multi-table DELETE t1, t2 FROM ... and DELETE FROM ... USING ... forms are MySQL extensions too. They are not part of the SQL standard. The option keywords are accepted for compatibility. They have no effect on execution.
  • The LIMIT clause on DELETE is a non-standard extension. This includes the MySQL LIMIT start, count spelling. PostgreSQL does not allow LIMIT directly on DELETE.
  • ORDER BY on DELETE is accepted for MySQL syntax compatibility. Unlike MySQL, it has no effect in EvolutionDB.
  • EvolutionDB does not support the PostgreSQL WHERE CURRENT OF cursor form. It does not support the ONLY and inheritance qualifiers. It does not support a table alias on the delete target. The PostgreSQL join source keyword is USING. EvolutionDB's second multi-table form uses it too.
  • ON DELETE referential actions are enforced. These are RESTRICT, NO ACTION, CASCADE, SET NULL, and SET DEFAULT. This matches the SQL standard.
