DELETE

Remove rows from a table.

Synopsis

DELETE [ LOW_PRIORITY ] [ QUICK ] [ IGNORE ] FROM table_name
    [ WHERE condition ]
    [ ORDER BY column [ ASC | DESC ] [, ...] ]
    [ LIMIT count [ OFFSET start | , count ] ]
    [ RETURNING * | output_column [, ...] ]

-- Multi-table form 1 (delete from joined targets)
DELETE [ LOW_PRIORITY ] [ QUICK ] [ IGNORE ] target_table [ .* ] [, ...]
    FROM table_references
    [ WHERE condition ]

-- Multi-table form 2 (USING)
DELETE [ LOW_PRIORITY ] [ QUICK ] [ IGNORE ] FROM target_table [ .* ] [, ...]
    USING table_references
    [ WHERE condition ]

Description

DELETE removes rows that satisfy the WHERE condition from table_name. If the WHERE clause is absent, the effect is to delete all visible rows in the table; the statement then leaves a valid but empty table.

EvolutionDB accepts three grammatical forms of DELETE:

  • A single-table delete: DELETE FROM table_name ....
  • A multi-table delete naming the target tables before FROM (DELETE t1, t2 FROM ...), or after FROM with a separate USING source list. Both multi-table forms drive a join over table_references and then delete the matched rows from each named target.

Deletion is performed in two phases for the single-table form: rows matching the WHERE expression are collected first, then removed. Removal is an MVCC soft delete, the row's xmax is stamped with the current transaction id and the primary-key index entry is retained so that concurrent readers can still locate the tuple and evaluate its visibility. Space is reclaimed later by the background reclaim process once xmax is committed and no snapshot can still see the row. Tuples written before MVCC was introduced (no MVCC header) fall back to immediate physical removal of the heap slot and index entry.

Each delete is transactional. The pre-image of every removed row is written to the undo log, so a ROLLBACK (or disconnect, which auto-rolls back) restores the rows. An exclusive row lock is taken per row; if it cannot be acquired the statement fails with a lock or deadlock error and the whole statement is rolled back.

Referential integrity. Before a row is removed, EvolutionDB enforces every foreign-key constraint that references the row's table. Depending on the constraint's ON DELETE action, the engine will reject the delete (RESTRICT / NO ACTION), recursively delete the referencing child rows (CASCADE), or rewrite the child foreign-key columns to NULL (SET NULL) or to their column default (SET DEFAULT). Cascades recurse to a maximum depth of 32; deeper chains (e.g. circular references) raise a foreign-key violation.

Triggers, indexes, and policies. BEFORE DELETE and AFTER DELETE triggers fire per row when the table defines them. Secondary index entries for the row are removed. If row-level security is enabled on the table, the session user's DELETE USING policy gates each row, rows the policy does not expose are silently skipped (so the deleted count reflects only rows the user may delete).

ORDER BY is parsed but not applied. The single-table grammar accepts an ORDER BY clause for MySQL syntax compatibility, but EvolutionDB's execution engine does not order the rows it deletes. ORDER BY therefore has no observable effect on which rows are removed. Use WHERE (and LIMIT if you need to cap the count) to control the row set precisely.

Parameters

LOW_PRIORITY, QUICK, IGNORE
MySQL-style delete options accepted by the grammar for compatibility. They are parsed into an option bitmask but do not change EvolutionDB's delete behavior.
table_name
The name of the table from which rows are deleted (single-table form). It is resolved through the system catalog in the current database and schema. Deleting from a table that does not exist reports an error and removes no rows.
target_table [ .* ]
In the multi-table forms, a table whose matched rows are to be deleted. One or more targets may be listed, comma-separated. The optional .* suffix is accepted after a target name. In the first form the targets precede FROM; in the second they follow FROM and the join source is given after USING.
table_references
The join expression that supplies the rows to test in the multi-table forms (the same table-reference syntax used by SELECT, including joins). Rows produced by this join determine which rows are removed from each target_table.
WHERE condition
An expression returning a boolean. Only rows for which it evaluates to true are deleted. A result of true/t/1, or any non-zero numeric value, is treated as a match. If omitted, every visible row qualifies.
ORDER BY column [ ASC | DESC ]
Accepted by the single-table grammar but not honored by the executor (see callout above).
LIMIT count [ OFFSET start ]
Caps how many of the matched rows are actually deleted (single-table form). Both the LIMIT count OFFSET start and the MySQL LIMIT start, count spellings are accepted. The limit/offset window is applied to the collected set of matching rows before removal.
RETURNING * | output_column
Causes DELETE to return values from the rows it removed (single-table form). RETURNING * returns all columns of each deleted row; otherwise the listed column names are returned. Values reflect the row as it existed immediately before deletion.

Examples

Delete a single row identified by its primary key:

DELETE FROM employees WHERE id = 42;

Delete all rows that match a condition:

DELETE FROM orders WHERE status = 'cancelled';

Empty a table (no WHERE clause):

DELETE FROM session_log;

Delete at most 100 matching rows:

DELETE FROM events WHERE level = 'debug' LIMIT 100;

Delete and return the removed rows:

DELETE FROM products WHERE discontinued = true
    RETURNING id, name;

Multi-table delete using a join (first form): remove orphaned order lines whose parent order no longer exists:

DELETE order_lines
    FROM order_lines LEFT JOIN orders
         ON order_lines.order_id = orders.id
    WHERE orders.id IS NULL;

Multi-table delete with USING (second form):

DELETE FROM order_lines
    USING orders
    WHERE order_lines.order_id = orders.id
      AND orders.status = 'voided';

Compatibility

The single-table DELETE FROM table WHERE ... form, together with RETURNING, follows the SQL standard and PostgreSQL closely. EvolutionDB's RETURNING behaves like PostgreSQL's: * or a list of column names yields the pre-deletion row values.

  • The LOW_PRIORITY, QUICK, and IGNORE options and the multi-table DELETE t1, t2 FROM ... / DELETE FROM ... USING ... forms are MySQL extensions, not part of the SQL standard. The option keywords are accepted for compatibility but have no effect on execution.
  • The LIMIT clause on DELETE (including the MySQL LIMIT start, count spelling) is a non-standard extension. PostgreSQL does not permit LIMIT directly on DELETE.
  • ORDER BY on DELETE is accepted for MySQL syntax compatibility but, unlike MySQL, has no effect in EvolutionDB.
  • EvolutionDB does not support the PostgreSQL WHERE CURRENT OF cursor form, the ONLY / inheritance qualifiers, or table aliasing of the delete target. The PostgreSQL join source keyword is USING, which EvolutionDB's second multi-table form also uses.
  • ON DELETE referential actions (RESTRICT, NO ACTION, CASCADE, SET NULL, SET DEFAULT) are enforced, consistent with the SQL standard.
