CREATE DOMAIN

Register a named, reusable data type built from a base type plus optional NOT NULL and CHECK constraints.

Synopsis

CREATE DOMAIN domain_name AS data_type
    [ DEFAULT default_expr ]
    [ NOT NULL ]
    [ CHECK ( expression ) ]

Description

CREATE DOMAIN creates a domain: a named definition that pairs an underlying base data_type with optional constraints. The domain is recorded in EvolutionDB's system catalog so that it can be referred to by name.

Internally the domain is not given its own catalog entity. EvolutionDB stores it in the shared constraints B+ tree under the system namespace (table_id = 0) with a constraint type of 'D'. The catalog row packs three fields into a single definition string of the form "type_val|not_null|check_serialized": the numeric type code of the base type, a 0/1 flag for NOT NULL, and — when a CHECK clause is given — the serialized check expression. The domain name is unique within that system namespace; creating a domain whose name already exists fails with SQLSTATE 42710 (duplicate object).

The base type is any type accepted by EvolutionDB's data_type grammar — for example INTEGER, BIGINT, DECIMAL(p,s), VARCHAR(n), CHAR(n), TEXT, BOOLEAN, DATE, TIMESTAMP, UUID, JSON, ENUM(...), array types written type[], and so on.

Honest implementation note. Two parts of the syntax are accepted by the parser but have limited effect in the current release, and you should not rely on them:

  • DEFAULT default_expr is parsed but not stored. The grammar accepts the clause, but the value is discarded — the catalog row records only the base type, the NOT NULL flag, and the CHECK expression. No default is persisted with the domain.
  • Domains are recorded, not yet enforced as column types. A domain is written to the catalog, but the engine does not currently resolve a domain name when it appears as a column type in CREATE TABLE; the domain's NOT NULL and CHECK rules are therefore not automatically applied to columns by referencing the domain name.

Parameters

domain_name
The name of the domain to create. It must be unique within the system namespace; a clash returns domain "name" already exists with SQLSTATE 42710.
data_type
The underlying base type for the domain, given after the required AS keyword. Any type from EvolutionDB's data_type grammar is allowed, including parameterized types such as VARCHAR(n) and DECIMAL(p,s) and array types written type[]. The type is stored as its internal numeric code (type_family × 10000 + size).
DEFAULT default_expr
Accepted by the grammar but not persisted in this release. The expression is parsed and then discarded, so it does not become part of the stored domain definition.
NOT NULL
Marks the domain as not accepting null values. When present, the not-null flag is recorded in the domain's catalog definition. May be combined with a CHECK clause.
CHECK ( expression )
A boolean constraint expression associated with the domain. When given, the expression is serialized and stored alongside the base type in the domain's catalog row.

Examples

A simple domain over an integer base type:

CREATE DOMAIN postal_code AS INTEGER;

A text domain that disallows nulls:

CREATE DOMAIN email_addr AS VARCHAR(255) NOT NULL;

A domain with a value constraint:

CREATE DOMAIN percent AS INTEGER
    CHECK (value >= 0 AND value <= 100);

A non-null domain with a check, combined:

CREATE DOMAIN positive_amount AS DECIMAL(12,2) NOT NULL
    CHECK (value > 0);

-- The DEFAULT clause parses, but the value is not stored with the domain:

CREATE DOMAIN country_code AS CHAR(2) DEFAULT 'US';

Compatibility

CREATE DOMAIN exists in the SQL standard and in PostgreSQL. EvolutionDB implements a deliberately narrow subset:

  • EvolutionDB accepts DEFAULT, a single NOT NULL, and a single unnamed CHECK (expression). The standard and PostgreSQL also allow named constraints (CONSTRAINT name CHECK ...), multiple constraints, NULL (explicitly nullable), and COLLATE — none of these are part of EvolutionDB's grammar.
  • Unlike PostgreSQL, EvolutionDB does not require or accept a schema-qualified domain name in this command; the domain is registered in a single system namespace.
  • DEFAULT is syntactically accepted but, in the current release, the default value is not retained. PostgreSQL stores and applies the default.
  • In PostgreSQL a domain becomes a usable column type that enforces its constraints on every column declared with it. In the current EvolutionDB release the domain is recorded in the catalog but is not yet wired in as a referenceable column type, so its constraints are not automatically enforced through the domain name.

Code that depends on default propagation or on domain-typed columns should treat those behaviors as not yet available and use explicit column-level NOT NULL/CHECK constraints in CREATE TABLE instead.
