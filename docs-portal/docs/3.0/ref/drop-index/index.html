DROP INDEX

Remove a secondary index from the database. The storage it used is freed.

Synopsis

DROP INDEX index_name

Description

DROP INDEX removes an existing secondary index. The index is found by its name only. EvolutionDB searches every table for an index with that name. So you do not give the table that owns the index. You also cannot give it.

When the index is found, its backing structure is destroyed. The way it is destroyed depends on the index type.

  • A B+ tree index has its tree pages freed. This is the default for CREATE INDEX and CREATE UNIQUE INDEX.
  • A hash index has its hash directory and bucket pages freed. You create this type with USING HASH.
  • An HNSW approximate-nearest-neighbor index has its in-memory graph released.

After the backing structure is destroyed, the matching entry is removed from the system catalog. On success the server reports Index 'index_name' dropped.

You cannot drop a primary-key index with this command. If you try, you get an error. Drop the table or alter it to remove the primary key constraint instead.

Note. The index name must match an index that really exists. If no index has that name, the command fails with index 'index_name' does not exist. EvolutionDB's grammar does not accept an IF EXISTS clause. So there is no built-in way to hide this error.

Parameters

index_name
The name of the index to remove. The lookup is by name across all tables. So the name finds the index on its own. You do not use a schema qualifier or the owning table. The name must point to an existing secondary index. It must not be a primary-key index.

Examples

This example creates an index and then drops it by name.

CREATE INDEX idx_users_email ON users (email);
DROP INDEX idx_users_email;

This example drops a hash index made with USING HASH. The drop command is the same for every index type.

CREATE INDEX idx_orders_status ON orders (status) USING HASH;
DROP INDEX idx_orders_status;

Trying to drop a primary-key index fails with an error.

-- raises: cannot drop primary key index '...'
DROP INDEX users_pkey;

Dropping an index that does not exist fails with an error.

-- raises: index 'no_such_index' does not exist
DROP INDEX no_such_index;

Compatibility

DROP INDEX is not part of the core SQL standard. The standard leaves index management to each database. EvolutionDB's form is kept very small on purpose.

Next to PostgreSQL, EvolutionDB accepts only DROP INDEX index_name. EvolutionDB's grammar does not support these PostgreSQL clauses. You must leave them out.

  • IF EXISTS is not accepted. A drop of a missing index always raises an error.
  • CONCURRENTLY is not accepted. The drop is not online or concurrent.
  • CASCADE and RESTRICT are not accepted.
  • You cannot drop many indexes in one statement. DROP INDEX a, b is not accepted. Use one index name per statement.
  • Schema-qualified names are not used. schema.index does not apply. The index is found by its bare name across all tables.

Unlike PostgreSQL, EvolutionDB will not drop a primary-key index with this command.
