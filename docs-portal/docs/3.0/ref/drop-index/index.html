DROP INDEX

Remove a secondary index from the database and reclaim the storage it occupied.

Synopsis

DROP INDEX index_name

Description

DROP INDEX removes an existing secondary index. The index is located by its name alone — EvolutionDB searches every table for an index with the given name, so the owning table does not need to be (and cannot be) specified in the command.

When the index is found, its backing structure is destroyed according to the index's type:

  • A B+ tree index has its tree pages freed (the default for CREATE INDEX and CREATE UNIQUE INDEX).
  • A hash index (created with USING HASH) has its hash directory and bucket pages freed.
  • An HNSW approximate-nearest-neighbor index has its in-memory graph released.

After the backing structure is destroyed, the corresponding entry is removed from the system catalog. On success the server reports Index 'index_name' dropped.

A primary-key index cannot be dropped with this command; attempting to do so raises an error. Drop or alter the table to remove a primary key constraint instead.

Note. The index name must match an index that actually exists. If no index by that name is found, the command fails with index 'index_name' does not exist. EvolutionDB's grammar does not accept an IF EXISTS clause, so there is no built-in way to suppress this error.

Parameters

index_name
The name of the index to remove. Because the lookup is by name across all tables, the name identifies the index on its own; no schema qualifier or owning-table reference is used. The name must refer to an existing secondary index that is not a primary-key index.

Examples

Create an index, then drop it by name:

CREATE INDEX idx_users_email ON users (email);
DROP INDEX idx_users_email;

Drop a hash index created with USING HASH. The drop command is the same regardless of how the index was built:

CREATE INDEX idx_orders_status ON orders (status) USING HASH;
DROP INDEX idx_orders_status;

Attempting to drop a primary-key index fails:

-- raises: cannot drop primary key index '...'
DROP INDEX users_pkey;

Dropping an index that does not exist fails:

-- raises: index 'no_such_index' does not exist
DROP INDEX no_such_index;

Compatibility

DROP INDEX is not part of the core SQL standard; the standard leaves index management to implementations. EvolutionDB's form is deliberately minimal.

Compared to PostgreSQL, EvolutionDB accepts only DROP INDEX index_name. The following PostgreSQL clauses are not supported by EvolutionDB's grammar and must be omitted:

  • IF EXISTS — not accepted; dropping a missing index always raises an error.
  • CONCURRENTLY — not accepted; the drop is not online/concurrent.
  • CASCADE / RESTRICT — not accepted.
  • Dropping multiple indexes in one statement (DROP INDEX a, b) — not accepted; one index name per statement.
  • Schema-qualified names (schema.index) — the index is resolved by bare name across all tables.

Unlike PostgreSQL, EvolutionDB explicitly refuses to drop a primary-key index through this command.
