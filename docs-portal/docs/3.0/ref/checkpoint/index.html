CHECKPOINT

Write to and read from a checkpoint store — a native, table-backed key/value log of agent execution state, designed as a server-side backend for the LangGraph BaseCheckpointSaver API.

Not the PostgreSQL CHECKPOINT. In PostgreSQL, CHECKPOINT is a server administration command that forces a write-ahead-log checkpoint to disk. EvolutionDB does not use the CHECKPOINT keyword for that purpose. Here, CHECKPOINT is the data-access verb for checkpoint stores: it inserts (PUT) and retrieves (GET / LIST) rows of saved agent state. The stores themselves are created and destroyed with CREATE CHECKPOINT STORE and DROP CHECKPOINT STORE.

Synopsis

-- Store definition (DDL)
CREATE CHECKPOINT STORE [ IF NOT EXISTS ] name
    [ WITH ( RETENTION = interval ) ]

DROP CHECKPOINT STORE [ IF EXISTS ] name [ CASCADE ]

-- Store access (DML)
CHECKPOINT PUT [ WRITES ] INTO name VALUES (
    thread_id, checkpoint_ns, checkpoint_id, parent_id,
    values, metadata, parent_config )

CHECKPOINT GET FROM name THREAD thread_id [ AT checkpoint_id ]

CHECKPOINT LIST FROM name THREAD thread_id [ LIMIT count ]

Description

A checkpoint store is a named object that persists the intermediate state of a long-running agent run so that it can be paused, resumed, branched, and audited. EvolutionDB implements it as ordinary catalog state on top of the regular storage engine: CREATE CHECKPOINT STORE name registers an entry in the system catalog and auto-creates a backing heap table named ck_name with a fixed schema. Every CHECKPOINT data statement compiles down to a plain INSERT or SELECT against that backing table, so checkpoint writes and reads participate in the same MVCC visibility, row-level security, triggers, and replication as any hand-written DML.

The backing table has eight columns in a fixed order:

thread_id VARCHAR(255) NOT NULL
Identifies the conversation/run a checkpoint belongs to. This is the key all reads filter on.
checkpoint_ns VARCHAR(255)
The checkpoint namespace (used by LangGraph for nested sub-graphs). Nullable; an empty string is stored as NULL.
checkpoint_id VARCHAR(255) PRIMARY KEY
The unique id of this checkpoint. It is the primary key and the value matched by GET … AT.
parent_id VARCHAR(255)
The id of the checkpoint this one descends from, enabling branching/time-travel. Nullable.
ck_values JSON
The serialized channel values / graph state. Exposed in the VALUES list as values; stored under the column name ck_values to avoid the reserved word values.
metadata JSON
Arbitrary metadata attached to the checkpoint (source step, writes summary, etc.).
parent_config JSON
The runnable config of the parent checkpoint.
created_at TIMESTAMP
Set automatically by the server (UTC) on each PUT; it is not supplied in the VALUES list. It defines the newest-first ordering used by GET and LIST.

CHECKPOINT PUT inserts exactly one row. The seven values in the VALUES list map positionally to the first seven columns above; the eighth (created_at) is filled in by the server. Each value slot is either a string literal or the keyword NULL.

CHECKPOINT GET returns the single most recent checkpoint for the given THREAD, ordered by created_at descending (LIMIT 1). When AT checkpoint_id is supplied, it instead returns the one row whose checkpoint_id matches exactly — time-travel to a specific point.

CHECKPOINT LIST returns the full checkpoint history for a THREAD, newest first. An optional LIMIT count caps the number of rows returned.

If the named store does not exist, all four CHECKPOINT data statements raise an error (SQLSTATE 42704, undefined object).

Parameters

IF NOT EXISTS / IF EXISTS
On CREATE CHECKPOINT STORE, IF NOT EXISTS suppresses the "already exists" error (SQLSTATE 42710) and leaves the existing store untouched. On DROP CHECKPOINT STORE, IF EXISTS suppresses the "does not exist" error (SQLSTATE 42704).
name
The store name. It must be a bare identifier (the lexer accepts [A-Za-z][A-Za-z0-9_]*, so no leading underscore). The backing table is created as ck_name in the current schema.
WITH ( RETENTION = interval )
An optional retention hint recorded on the store, given as a string literal such as '30 days'. The value is stored verbatim in the catalog. The = sign is the comparison operator accepted by the grammar.
CASCADE (DROP)
Parsed and accepted on DROP CHECKPOINT STORE. Note that EvolutionDB always drops the backing ck_name table along with the store registration regardless of whether CASCADE is written, because the store is the only object that references that table.
PUT / PUT WRITES
Both insert one checkpoint row. WRITES is an alias kept for parity with the LangGraph put_writes API; in this version it does not change behavior — the row is written the same way either form.
INTO name VALUES ( … ) (PUT)
The target store and the seven positional values: thread_id, checkpoint_ns, checkpoint_id, parent_id, values, metadata, parent_config. Each slot is a string literal or NULL; all seven slots are required by the grammar.
FROM name (GET / LIST)
The store to read from.
THREAD thread_id
A string literal naming the thread to read. This becomes the WHERE thread_id = 'thread_id' filter.
AT checkpoint_id (GET)
A string literal. Narrows the result to the single checkpoint with that checkpoint_id instead of the latest one.
LIMIT count (LIST)
An integer. Caps the number of history rows returned.

Examples

Create a checkpoint store with a retention hint:

CREATE CHECKPOINT STORE agent_runs
    WITH (RETENTION = '30 days');

Create one idempotently:

CREATE CHECKPOINT STORE IF NOT EXISTS agent_runs;

Write a checkpoint. The seven values are thread_id, checkpoint_ns, checkpoint_id, parent_id, values, metadata, parent_config — created_at is filled in by the server:

CHECKPOINT PUT INTO agent_runs VALUES (
    'thread-42',                       -- thread_id
    '',                                -- checkpoint_ns
    'ckpt-0001',                       -- checkpoint_id
    NULL,                              -- parent_id (root)
    '{"messages":["hi"]}',             -- values (JSON)
    '{"step":1}',                      -- metadata (JSON)
    NULL                               -- parent_config
);

Write a follow-up checkpoint that descends from the first (using the cosmetic WRITES alias):

CHECKPOINT PUT WRITES INTO agent_runs VALUES (
    'thread-42', '', 'ckpt-0002', 'ckpt-0001',
    '{"messages":["hi","there"]}', '{"step":2}', NULL);

Fetch the latest checkpoint for a thread:

CHECKPOINT GET FROM agent_runs THREAD 'thread-42';

Time-travel to a specific checkpoint id:

CHECKPOINT GET FROM agent_runs THREAD 'thread-42' AT 'ckpt-0001';

List a thread's history, newest first, capped at five rows:

CHECKPOINT LIST FROM agent_runs THREAD 'thread-42' LIMIT 5;

Because every store is just a table named ck_name, the same data is reachable with ordinary SQL:

SELECT * FROM ck_agent_runs
    WHERE thread_id = 'thread-42'
    ORDER BY created_at DESC;

Remove the store and its backing table:

DROP CHECKPOINT STORE IF EXISTS agent_runs CASCADE;

Compatibility

  • Not in the SQL standard. CHECKPOINT, CREATE CHECKPOINT STORE, and DROP CHECKPOINT STORE are EvolutionDB extensions; there is no equivalent in the SQL standard.
  • Different from PostgreSQL's CHECKPOINT. PostgreSQL's CHECKPOINT forces a WAL checkpoint and takes no operands. EvolutionDB's CHECKPOINT is a data-access verb (PUT / PUT WRITES / GET / LIST) for checkpoint stores and is unrelated to WAL flushing. EvolutionDB does not provide a bare CHECKPOINT; WAL-flush statement.
  • Modeled on the LangGraph BaseCheckpointSaver contract. The fixed eight-column schema (thread_id, checkpoint_ns, checkpoint_id, parent_id, values, metadata, parent config, plus a server-set timestamp) mirrors the LangGraph checkpoint tuple. This is a vendor convenience layer, not a standardized interface.
  • WRITES is accepted but cosmetic in this version — CHECKPOINT PUT and CHECKPOINT PUT WRITES behave identically. Any per-write semantics are handled at the application layer.
  • CASCADE is parsed but the drop always cascades to the backing ck_name table whether or not the keyword is given.
  • The VALUES list is fixed at seven positional slots. Unlike INSERT, you cannot supply a column list, reorder columns, or omit a slot; pass NULL for unused fields. The created_at column is reserved to the server.
  • The store name and backing-table name are not schema-qualified in these statements; objects are created in and resolved against the current schema.
