CHECKPOINT

Write to and read from a checkpoint store. A checkpoint store is a native, table-backed key/value log of agent execution state. It works as a server-side backend for the LangGraph BaseCheckpointSaver API.

Not the PostgreSQL CHECKPOINT. In PostgreSQL, CHECKPOINT is a server admin command. It forces a write-ahead-log checkpoint to disk. EvolutionDB does not use the CHECKPOINT keyword for that. Here, CHECKPOINT is the data-access verb for checkpoint stores. It inserts rows of saved agent state with PUT. It reads them back with GET and LIST. You create and destroy the stores themselves with CREATE CHECKPOINT STORE and DROP CHECKPOINT STORE.

Synopsis

-- Store definition (DDL)
CREATE CHECKPOINT STORE [ IF NOT EXISTS ] name
    [ WITH ( RETENTION = interval ) ]

DROP CHECKPOINT STORE [ IF EXISTS ] name [ CASCADE ]

-- Store access (DML)
CHECKPOINT PUT [ WRITES ] INTO name VALUES (
    thread_id, checkpoint_ns, checkpoint_id, parent_id,
    values, metadata, parent_config )

CHECKPOINT GET FROM name THREAD thread_id [ AT checkpoint_id ]

CHECKPOINT LIST FROM name THREAD thread_id [ LIMIT count ]

Description

A checkpoint store is a named object. It saves the middle state of a long-running agent run. This lets you pause, resume, branch, and audit the run. EvolutionDB builds it as ordinary catalog state on top of the regular storage engine. CREATE CHECKPOINT STORE name adds an entry in the system catalog. It also auto-creates a backing heap table named ck_name with a fixed schema. Every CHECKPOINT data statement turns into a plain INSERT or SELECT against that backing table. So checkpoint writes and reads use the same MVCC visibility, row-level security, triggers, and replication as any hand-written DML.

The backing table has eight columns in a fixed order:

thread_id VARCHAR(255) NOT NULL
Names the conversation or run that a checkpoint belongs to. All reads filter on this key.
checkpoint_ns VARCHAR(255)
The checkpoint namespace. LangGraph uses it for nested sub-graphs. It can be null. An empty string is stored as NULL.
checkpoint_id VARCHAR(255) PRIMARY KEY
The unique id of this checkpoint. It is the primary key. GET … AT matches on this value.
parent_id VARCHAR(255)
The id of the checkpoint that this one comes from. This is what makes branching and time-travel possible. It can be null.
ck_values JSON
The serialized channel values and graph state. It shows up in the VALUES list as values. It is stored under the column name ck_values. This avoids the reserved word values.
metadata JSON
Any metadata attached to the checkpoint. This can be the source step, a writes summary, and so on.
parent_config JSON
The runnable config of the parent checkpoint.
created_at TIMESTAMP
The server sets this automatically in UTC on each PUT. You do not supply it in the VALUES list. It sets the newest-first order used by GET and LIST.

CHECKPOINT PUT inserts exactly one row. The seven values in the VALUES list match the first seven columns above by position. The server fills in the eighth column, created_at. Each value slot is either a string literal or the keyword NULL.

CHECKPOINT GET returns the single most recent checkpoint for the given THREAD. The rows are ordered by created_at descending with LIMIT 1. When you supply AT checkpoint_id, it instead returns the one row whose checkpoint_id matches exactly. This lets you time-travel to a specific point.

CHECKPOINT LIST returns the full checkpoint history for a THREAD, newest first. An optional LIMIT count caps the number of rows returned.

If the named store does not exist, all four CHECKPOINT data statements raise an error. The SQLSTATE is 42704, which means undefined object.

Parameters

IF NOT EXISTS / IF EXISTS
On CREATE CHECKPOINT STORE, IF NOT EXISTS hides the "already exists" error. That error is SQLSTATE 42710. It leaves the existing store as it is. On DROP CHECKPOINT STORE, IF EXISTS hides the "does not exist" error. That error is SQLSTATE 42704.
name
The store name. It must be a bare identifier. The lexer accepts the pattern [A-Za-z][A-Za-z0-9_]*, so it cannot start with an underscore. The backing table is created as ck_name in the current schema.
WITH ( RETENTION = interval )
An optional retention hint kept on the store. You give it as a string literal such as '30 days'. The value is stored as-is in the catalog. The = sign is the comparison operator that the grammar accepts.
CASCADE (DROP)
Parsed and accepted on DROP CHECKPOINT STORE. EvolutionDB always drops the backing ck_name table along with the store. This happens whether or not you write CASCADE. The store is the only object that points to that table.
PUT / PUT WRITES
Both insert one checkpoint row. WRITES is an alias kept to match the LangGraph put_writes API. In this version it does not change behavior. The row is written the same way with either form.
INTO name VALUES ( … ) (PUT)
The target store and the seven positional values. The values are thread_id, checkpoint_ns, checkpoint_id, parent_id, values, metadata, and parent_config. Each slot is a string literal or NULL. The grammar requires all seven slots.
FROM name (GET / LIST)
The store to read from.
THREAD thread_id
A string literal that names the thread to read. This becomes the WHERE thread_id = 'thread_id' filter.
AT checkpoint_id (GET)
A string literal. It narrows the result to the single checkpoint with that checkpoint_id instead of the latest one.
LIMIT count (LIST)
An integer. Caps the number of history rows returned.

Examples

Create a checkpoint store with a retention hint.

CREATE CHECKPOINT STORE agent_runs
    WITH (RETENTION = '30 days');

Create one in an idempotent way. Running it again does no harm.

CREATE CHECKPOINT STORE IF NOT EXISTS agent_runs;

Write a checkpoint. The seven values are thread_id, checkpoint_ns, checkpoint_id, parent_id, values, metadata, and parent_config. The server fills in created_at.

CHECKPOINT PUT INTO agent_runs VALUES (
    'thread-42',                       -- thread_id
    '',                                -- checkpoint_ns
    'ckpt-0001',                       -- checkpoint_id
    NULL,                              -- parent_id (root)
    '{"messages":["hi"]}',             -- values (JSON)
    '{"step":1}',                      -- metadata (JSON)
    NULL                               -- parent_config
);

Write a follow-up checkpoint that comes from the first one. This uses the cosmetic WRITES alias.

CHECKPOINT PUT WRITES INTO agent_runs VALUES (
    'thread-42', '', 'ckpt-0002', 'ckpt-0001',
    '{"messages":["hi","there"]}', '{"step":2}', NULL);

Fetch the latest checkpoint for a thread.

CHECKPOINT GET FROM agent_runs THREAD 'thread-42';

Time-travel to a specific checkpoint id.

CHECKPOINT GET FROM agent_runs THREAD 'thread-42' AT 'ckpt-0001';

List a thread's history, newest first, capped at five rows.

CHECKPOINT LIST FROM agent_runs THREAD 'thread-42' LIMIT 5;

Every store is just a table named ck_name. So you can reach the same data with ordinary SQL.

SELECT * FROM ck_agent_runs
    WHERE thread_id = 'thread-42'
    ORDER BY created_at DESC;

Remove the store and its backing table.

DROP CHECKPOINT STORE IF EXISTS agent_runs CASCADE;

Compatibility

  • Not in the SQL standard. CHECKPOINT, CREATE CHECKPOINT STORE, and DROP CHECKPOINT STORE are EvolutionDB extensions. There is no match for them in the SQL standard.
  • Different from PostgreSQL's CHECKPOINT. PostgreSQL's CHECKPOINT forces a WAL checkpoint and takes no operands. EvolutionDB's CHECKPOINT is a data-access verb for checkpoint stores. It uses PUT, PUT WRITES, GET, and LIST. It has nothing to do with WAL flushing. EvolutionDB does not offer a bare CHECKPOINT; WAL-flush statement.
  • Based on the LangGraph BaseCheckpointSaver contract. The fixed eight-column schema mirrors the LangGraph checkpoint tuple. The columns are thread_id, checkpoint_ns, checkpoint_id, parent_id, values, metadata, parent config, and a server-set timestamp. This is a vendor convenience layer. It is not a standard interface.
  • WRITES is accepted but cosmetic in this version. CHECKPOINT PUT and CHECKPOINT PUT WRITES behave the same way. The application layer handles any per-write meaning.
  • CASCADE is parsed but the drop always cascades. It always drops the backing ck_name table whether or not you give the keyword.
  • The VALUES list is fixed at seven positional slots. Unlike INSERT, you cannot supply a column list, reorder columns, or skip a slot. Pass NULL for unused fields. The created_at column is reserved for the server.
  • The store name and backing-table name are not schema-qualified in these statements. Objects are created in the current schema and resolved against it.
