SHOW

Display the value of a run-time configuration parameter, or list catalog objects such as databases, schemas, tables, users, grants, and stores.

Synopsis

SHOW DATABASES
SHOW SCHEMAS
SHOW TABLES
SHOW VIEWS
SHOW CREATE VIEW view_name
SHOW USERS
SHOW ROLES
SHOW GRANTS [ FOR user_name ]
SHOW JOBS
SHOW SUBSCRIPTIONS
SHOW MEMORY STORES
SHOW CHECKPOINT STORES
SHOW ENTITY STORES
SHOW GRAPH STORES
SHOW DOCUMENT STORES
SHOW MESSAGE LOGS
SHOW EVO_SESSIONS
SHOW EVO_PROCESSES
SHOW REPLICATION STATUS
SHOW REPLICATION LAG
SHOW NODES
SHOW TABLE PLACEMENT
SHOW SHARDS table_name
SHOW configuration_parameter

Description

SHOW reports the value of a run-time configuration parameter or returns a result set listing catalog objects. Unlike most statements in EvolutionDB, SHOW is not defined in the Bison grammar (evolution/parser/evoparser.y). Instead the dual-protocol adaptor intercepts any statement whose first keyword is SHOW in adaptor/catalog.c and dispatches it before the SQL engine ever sees it.

Dispatch works by case-insensitive substring matching against the raw statement text, and the first matching form wins. Three of the forms (SHOW NODES, SHOW TABLE PLACEMENT, SHOW SHARDS) are recognized by the distributed-command handler, which runs ahead of the general SHOW handler. All forms return a result set tagged with the command tag SHOW; the object-listing forms behave like a SELECT and each defines its own columns.

Because matching is substring-based and order-sensitive, the keyword you intend must be the one that matches earliest in the dispatch chain. For example, a statement containing the word tables is treated as SHOW TABLES. Any SHOW statement that matches none of the listed forms falls through to the run-time parameter path, which returns a single setting column (an empty string when the parameter is unknown).

Scope. The object listings resolve against the session's current database and current schema. SHOW TABLES and SHOW VIEWS show only objects in the current schema; temporary tables are listed only to the session that owns them. SHOW DATABASES, SHOW USERS, SHOW ROLES, and the store/job/subscription listings are cluster-wide.

Parameters

DATABASES
Lists every database in the cluster. Returns one column, database.
SCHEMAS
Lists every schema in the current database. Returns one column, schema.
TABLES
Lists tables in the current schema of the current database. Returns one column, table. Temporary tables appear only in the session that created them.
VIEWS
Lists views in the current schema. Returns view_name and view_sql (the stored defining query).
CREATE VIEW view_name
Shows the reconstructed CREATE VIEW statement for the named view. Returns columns View and Create View. The definition is rebuilt as CREATE VIEW name AS <stored sql>. If the view does not exist in the current database/schema, the statement returns SQLSTATE 42P01 (view does not exist).
USERS
Lists all database users. Returns one column, username.
ROLES
Lists all roles. Returns one column, role_name.
GRANTS [ FOR user_name ]
Lists privilege grants. With no FOR clause, all grants are returned; with FOR user_name (the name may be unquoted or single-quoted) the result is filtered to that user. Returns user, scope (TABLE / SCHEMA / DATABASE), object, privileges, and grant_option (YES/NO).
JOBS
Lists registered scheduled (cron) jobs. Returns name, schedule (the cron expression), sql (the body it runs), enabled (boolean), and last_run_unix (Unix timestamp of the last fire; 0 means never run).
SUBSCRIPTIONS
Lists durable subscriptions. Returns name, channel, last_ack_seq, next_seq, and ttl_days.
MEMORY STORES
Lists memory stores created with CREATE MEMORY STORE. Returns name, backing_table_id, embedding_dim, and distance (cosine, l2, inner, or l1). The underscore spelling MEMORY_STORES is also accepted.
CHECKPOINT STORES
Lists checkpoint stores. Returns name, backing_table_id, and retention. The underscore spelling CHECKPOINT_STORES is also accepted.
ENTITY STORES
Lists entity stores. Returns name, backing_table_id, and ttl_days. The underscore spelling ENTITY_STORES is also accepted.
GRAPH STORES
Lists graph stores. Returns name, nodes_table_id, and edges_table_id. The underscore spelling GRAPH_STORES is also accepted.
DOCUMENT STORES
Lists document stores. Returns name, backing_table_id, embedding_dim, and distance_kind. The underscore spelling DOCUMENT_STORES is also accepted.
MESSAGE LOGS
Lists message logs. Returns name, backing_table_id, and ttl_days. The underscore spelling MESSAGE_LOGS is also accepted.
EVO_SESSIONS
Lists currently connected sessions. Returns pid (session id), username, database, state (active or idle), and duration_ms (milliseconds the current query has been running; 0 when idle).
EVO_PROCESSES
Like EVO_SESSIONS but adds a query column with the text of the session's current statement. Columns: pid, username, database, state, duration_ms, query.
REPLICATION STATUS
Per-replica view of the streaming replication slots. Returns role (primary, replica, or witness), replica_id, my_lsn, confirmed_lsn, lag_bytes, active, and last_seen_us. With no replicas a single summary row for the local role is returned. (Matched only when the text does not also contain cluster.)
REPLICATION LAG
Compact per-slot lag view. Returns replica_id and lag_bytes.
NODES
Lists nodes in a distributed cluster. Returns node_id, host, pg_port, and role (SELF or PEER). On a non-distributed server a single STANDALONE row for localhost:5433 is returned. Handled by the distributed-command path.
TABLE PLACEMENT
Lists every non-temporary table with its owning node. Returns table_name, schema, owner_node_id, and location (LOCAL or REMOTE). Handled by the distributed-command path.
SHARDS table_name
Lists the shards of a sharded table. Returns shard_name, ordinal, owner_node_id, range_bound (MAXVALUE for the open upper bound), shard_type (HASH or RANGE), and shard_key. Returns no rows if the table is not sharded. Handled by the distributed-command path.
configuration_parameter
Any SHOW form not matched above returns a single setting column. Recognized parameters include system_time_retention (days of system-versioned history retained; 0 = pruning disabled, returned as system_time_retention_days), max_connections, active_connections, statement_timeout / evo_statement_timeout, evo_lock_timeout, evo_idle_timeout, transaction_isolation, server_version (reports 15.0 (EvoSQL)), max_identifier_length (63), standard_conforming_strings (on), search_path, cluster status (Raft role/term/leader/LSN), buffer_pool_stats (multi-column hit-rate and I/O view), and buffer_pool_size (also accepted as shared_buffers, innodb_buffer_pool_size, or db_cache_size). An unrecognized parameter yields one row with an empty setting value rather than an error.

Examples

List databases, then schemas and tables in the current database:

SHOW DATABASES;
SHOW SCHEMAS;
SHOW TABLES;

Reconstruct the definition of a view:

SHOW CREATE VIEW active_users;

Show all grants, then only those held by one user:

SHOW GRANTS;
SHOW GRANTS FOR 'analyst';

Inspect connected sessions and the queries they are running:

SHOW EVO_PROCESSES;

List the shards of a sharded table on a distributed cluster:

SHOW SHARDS orders;

Read run-time parameters:

SHOW server_version;        -- 15.0 (EvoSQL)
SHOW max_connections;
SHOW buffer_pool_stats;     -- pool_pages, hits, misses, hit_rate, ...
SHOW system_time_retention; -- system_time_retention_days

Compatibility

There is no SHOW statement in the SQL standard. EvolutionDB's run-time-parameter form (SHOW configuration_parameter) follows the PostgreSQL convention and exposes several PostgreSQL-compatible names such as server_version, max_connections, standard_conforming_strings, search_path, and shared_buffers. For interoperability, SHOW server_version reports a PostgreSQL-style string (15.0 (EvoSQL)).

The object-listing and store-listing forms (SHOW DATABASES, SHOW TABLES, SHOW SCHEMAS, SHOW GRANTS, and the various ... STORES forms) are EvolutionDB extensions; their shape is closer to MySQL-style SHOW listings than to PostgreSQL, but the exact keywords and result columns are EvolutionDB-specific. PostgreSQL's SHOW ALL and the SHOW SESSION / SHOW LOCAL qualifiers are not implemented; matching is substring-based, so qualifiers are simply ignored. EvolutionDB does not raise an error for an unknown parameter — it returns an empty setting value — whereas PostgreSQL raises an error.
