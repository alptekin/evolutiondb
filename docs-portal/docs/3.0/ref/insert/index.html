INSERT

Add one or more new rows to a table.

Synopsis

INSERT [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY | IGNORE ] [ INTO ] table_name
    [ ( column [, ...] ) ]
    VALUES ( { expression | DEFAULT } [, ...] ) [, ...]
    [ ON DUPLICATE KEY UPDATE column = expression [, ...] ]
    [ ON CONFLICT [ ( column ) ] { DO NOTHING | DO UPDATE SET column = expression [, ...] } ]
    [ RETURNING { * | column [, ...] } ]

--- INSERT ... SET form ---
INSERT [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY | IGNORE ] [ INTO ] table_name
    SET column = { expression | DEFAULT } [, ...]
    [ ON DUPLICATE KEY UPDATE column = expression [, ...] ]

--- INSERT ... SELECT form ---
INSERT [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY | IGNORE ] [ INTO ] table_name
    [ ( column [, ...] ) ]
    select_statement

Description

INSERT adds new rows to table_name. There are three forms you can use. The VALUES form takes one or more value lists in parentheses. The SET form uses MySQL-style column = value assignments. The INSERT … SELECT form inserts the result of a subquery.

You can give a list of column names in parentheses. Then each value goes into the named column. Columns you leave out get their declared DEFAULT. If there is no default, they get NULL. If you do not give a column list, the values go into the table's columns in their declaration order.

The engine checks the row before it writes it. First it checks the types. Then it checks the NOT NULL, UNIQUE, and CHECK constraints. It only checks a constraint if the table actually declares it. A primary-key clash raises a duplicate-key error. This does not happen if you use an ON DUPLICATE KEY UPDATE or ON CONFLICT clause.

Columns that have a DEFAULT get their value at insert time. Some defaults are function calls. Examples are gen_random_uuid(), gen_random_uuid_v7(), snowflake_id(), and CURRENT_TIMESTAMP. The engine runs these once per row. Expression defaults can use the rest of the row as context. A column with AUTO_INCREMENT gets its value from the table's running counter. The counter moves forward by its step when you do not give a value. Stored generated columns are computed from the inserted row. You cannot write an explicit value into a generated column. Only DEFAULT is allowed there.

A statement may make a value for an AUTO_INCREMENT column. It may also make one with gen_random_uuid*() or snowflake_id(). The engine saves the first such value. You can then read it with LAST_INSERT_ID().

Parameters

LOW_PRIORITY | DELAYED | HIGH_PRIORITY | IGNORE
These keywords come from MySQL. The grammar accepts them. They have no effect when EvolutionDB runs the statement. They are here so that statements copied from MySQL can parse without an error.
INTO
An optional word before the table name. It is accepted for SQL and PostgreSQL compatibility. INSERT t … and INSERT INTO t … mean the same thing.
table_name
The name of an existing table to insert into. The engine removes a .dat suffix if the name has one. This gives the base table name. The table may be a RANGE-partitioned parent. In that case the row goes to the child partition whose bounds hold the partition-key value. In this release every row of a multi-row insert must go to the same partition.
( column [..] )
An optional list of target columns. When you give it, the engine matches the values to these columns by position and puts them in table order. Columns you do not list take their default value. If there is no default, they take NULL. Each VALUES row must have the same number of values as the number of named columns. If it does not, the engine rejects the statement.
VALUES ( … ) [..]
One or more rows of values. Each row is a comma-separated list in parentheses. When you give more than one group, the engine inserts many rows in one statement.
expression
A value expression that makes the column value. It can be a literal, a function call, arithmetic, and so on.
DEFAULT
In a value position, this asks for the column's declared default value. If the column has no default, the value is NULL. For a generated column, DEFAULT is the only value form you can use.
SET column = { expression | DEFAULT } [..]
The MySQL-style assignment form. Each named column is set to the given value or its default. You can only use the = operator. Any other comparison operator is a syntax error.
select_statement
A SELECT whose result rows are inserted. The engine matches its columns to the optional target column list. If there is no list, it matches them to the table's columns in order.
ON DUPLICATE KEY UPDATE column = expression [..]
The MySQL-style upsert. An inserted row may break the primary key or a UNIQUE constraint. When this happens, the engine updates the existing row with the listed assignments. It does not raise a duplicate-key error.
ON CONFLICT [ ( column ) ] DO NOTHING | DO UPDATE SET …
The PostgreSQL-style way to handle conflicts. With DO NOTHING, the engine skips a conflicting row in silence. With DO UPDATE SET, it updates the existing row with the given assignments. You can name an optional conflict target with one column. A target with more than one column such as (a, b) is not supported yet. It raises an error.
RETURNING * | column [..]
Returns data from the rows that were really inserted. RETURNING * returns every column. If you name columns, it returns only those columns. This works on the VALUES form.

Examples

Insert one row with every column set:

INSERT INTO users (id, name, email)
VALUES (1, 'Ada', 'ada@example.com');

Insert several rows in one statement. One column falls back to its default:

INSERT INTO users (id, name, role)
VALUES (2, 'Lin', DEFAULT),
       (3, 'Bo',  'editor');

Leave out the column list. The values map to the columns in declaration order:

INSERT INTO users VALUES (4, 'Mei', 'mei@example.com', DEFAULT);

MySQL-style SET form:

INSERT INTO users
SET id = 5, name = 'Sam', role = DEFAULT;

Insert from a query:

INSERT INTO archived_users (id, name)
SELECT id, name FROM users WHERE active = 0;

Upsert with MySQL ON DUPLICATE KEY UPDATE:

INSERT INTO counters (key, hits)
VALUES ('home', 1)
ON DUPLICATE KEY UPDATE hits = hits + 1;

Upsert with PostgreSQL ON CONFLICT and return the result:

INSERT INTO counters (key, hits)
VALUES ('home', 1)
ON CONFLICT (key) DO UPDATE SET hits = hits + 1
RETURNING *;

Skip a conflicting row silently:

INSERT INTO counters (key, hits)
VALUES ('home', 1)
ON CONFLICT DO NOTHING;

Compatibility

The INSERT in EvolutionDB mixes the SQL standard, PostgreSQL, and MySQL.

  • SQL standard and PostgreSQL. These parts follow PostgreSQL rules. They are the INSERT INTO … VALUES … and INSERT INTO … SELECT … forms, multi-row VALUES, the DEFAULT keyword in value positions, the ON CONFLICT … DO NOTHING / DO UPDATE SET clause, and RETURNING.
  • MySQL extensions. These parts come from MySQL. They are the optional priority and IGNORE modifiers, the INTO keyword that you can leave out, the SET column = value assignment form, and ON DUPLICATE KEY UPDATE. The engine parses the priority and IGNORE modifiers but they have no effect when it runs.
  • Differences from PostgreSQL. Some features are not supported. These are the standard OVERRIDING { SYSTEM | USER } VALUE clause, a DEFAULT VALUES clause, and naming a constraint as the conflict target with ON CONFLICT ON CONSTRAINT …. The WHERE predicate on ON CONFLICT … DO UPDATE is also not supported. Common table expressions with WITH … INSERT are not supported either. An ON CONFLICT target with more than one column is rejected. RETURNING works on the VALUES form.
