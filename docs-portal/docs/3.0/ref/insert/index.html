INSERT

Add one or more new rows to a table.

Synopsis

INSERT [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY | IGNORE ] [ INTO ] table_name
    [ ( column [, ...] ) ]
    VALUES ( { expression | DEFAULT } [, ...] ) [, ...]
    [ ON DUPLICATE KEY UPDATE column = expression [, ...] ]
    [ ON CONFLICT [ ( column ) ] { DO NOTHING | DO UPDATE SET column = expression [, ...] } ]
    [ RETURNING { * | column [, ...] } ]

--- INSERT ... SET form ---
INSERT [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY | IGNORE ] [ INTO ] table_name
    SET column = { expression | DEFAULT } [, ...]
    [ ON DUPLICATE KEY UPDATE column = expression [, ...] ]

--- INSERT ... SELECT form ---
INSERT [ LOW_PRIORITY | DELAYED | HIGH_PRIORITY | IGNORE ] [ INTO ] table_name
    [ ( column [, ...] ) ]
    select_statement

Description

INSERT adds new rows to table_name. There are three accepted forms: the VALUES form (one or more parenthesized value lists), the SET form (MySQL-style column = value assignments), and the INSERT … SELECT form that inserts the result of a subquery.

If a parenthesized column list is supplied, each value is matched to the named column; columns that are omitted receive their declared DEFAULT, or NULL when no default exists. If no column list is supplied, the values are assigned to the table's columns in their catalog (declaration) order.

Before a row is written, the engine validates types, then, only when the table actually declares such constraints, NOT NULL and UNIQUE constraints, plus any CHECK constraints. A primary-key collision raises a duplicate-key error unless an ON DUPLICATE KEY UPDATE or ON CONFLICT clause is present.

Columns declared DEFAULT are resolved at insert time. Function-style defaults such as gen_random_uuid(), gen_random_uuid_v7(), snowflake_id() and CURRENT_TIMESTAMP are evaluated per row; expression defaults are evaluated with the rest of the row available as context. A column with AUTO_INCREMENT is filled from the table's running counter (advanced by its step) when no explicit value is given. Stored generated columns are computed from the inserted row; writing an explicit non-DEFAULT value into a generated column is rejected.

The first value produced for an AUTO_INCREMENT column or by gen_random_uuid*() / snowflake_id() in a statement is captured and is afterwards available through LAST_INSERT_ID().

Parameters

LOW_PRIORITY | DELAYED | HIGH_PRIORITY | IGNORE
MySQL-compatibility option keywords. They are accepted by the grammar but have no effect on execution in EvolutionDB; they exist so that statements copied from MySQL parse without error.
INTO
Optional noise word before the table name, accepted for SQL/PostgreSQL compatibility. INSERT t … and INSERT INTO t … are equivalent.
table_name
The name of an existing table to insert into. A .dat suffix, if present, is stripped to obtain the base table name. If the table is a RANGE-partitioned parent, the row is routed to the child partition whose bounds contain the partition-key value (in this release every row of a multi-row insert must route to the same partition).
( column [..] )
An optional target column list. When given, supplied values are matched positionally to these columns and reordered into table order; columns not listed take their default (or NULL). The number of values in each VALUES row must equal the number of named columns, otherwise the statement is rejected.
VALUES ( … ) [..]
One or more rows of values, each a parenthesized, comma-separated list. Multiple parenthesized groups perform a multi-row insert in a single statement.
expression
A value expression producing the column value (literal, function call, arithmetic, etc.).
DEFAULT
In a value position, requests the column's declared default value (or NULL if the column has no default). For a generated column, DEFAULT is the only value form allowed.
SET column = { expression | DEFAULT } [..]
MySQL-style assignment form: each named column is set to the given value (or its default). Only the = operator is permitted; any other comparison operator is a syntax error.
select_statement
A SELECT whose result rows are inserted. The projected columns are matched against the optional target column list, or against the table's columns in order when no list is given.
ON DUPLICATE KEY UPDATE column = expression [..]
MySQL-style upsert. When an inserted row would violate the primary key or a UNIQUE constraint, the existing row is updated with the listed assignments instead of raising a duplicate-key error.
ON CONFLICT [ ( column ) ] DO NOTHING | DO UPDATE SET …
PostgreSQL-style conflict handling. With DO NOTHING a conflicting row is silently skipped; with DO UPDATE SET the existing row is updated with the given assignments. An optional single-column conflict target may be named; composite targets such as (a, b) are not yet supported and raise an error.
RETURNING * | column [..]
Returns data from the rows actually inserted. RETURNING * returns every column; otherwise the named columns are returned. Available on the VALUES form.

Examples

Insert a single fully-specified row:

INSERT INTO users (id, name, email)
VALUES (1, 'Ada', 'ada@example.com');

Insert several rows in one statement, letting a column fall back to its default:

INSERT INTO users (id, name, role)
VALUES (2, 'Lin', DEFAULT),
       (3, 'Bo',  'editor');

Omit the column list, values map to columns in declaration order:

INSERT INTO users VALUES (4, 'Mei', 'mei@example.com', DEFAULT);

MySQL-style SET form:

INSERT INTO users
SET id = 5, name = 'Sam', role = DEFAULT;

Insert from a query:

INSERT INTO archived_users (id, name)
SELECT id, name FROM users WHERE active = 0;

Upsert with MySQL ON DUPLICATE KEY UPDATE:

INSERT INTO counters (key, hits)
VALUES ('home', 1)
ON DUPLICATE KEY UPDATE hits = hits + 1;

Upsert with PostgreSQL ON CONFLICT, returning the result:

INSERT INTO counters (key, hits)
VALUES ('home', 1)
ON CONFLICT (key) DO UPDATE SET hits = hits + 1
RETURNING *;

Skip a conflicting row silently:

INSERT INTO counters (key, hits)
VALUES ('home', 1)
ON CONFLICT DO NOTHING;

Compatibility

EvolutionDB's INSERT is a hybrid of the SQL standard, PostgreSQL, and MySQL:

  • SQL standard / PostgreSQL: the INSERT INTO … VALUES … and INSERT INTO … SELECT … forms, multi-row VALUES, the DEFAULT keyword in value positions, the ON CONFLICT … DO NOTHING / DO UPDATE SET clause, and RETURNING follow PostgreSQL conventions.
  • MySQL extensions: the optional priority/IGNORE modifiers, the omittable INTO keyword, the SET column = value assignment form, and ON DUPLICATE KEY UPDATE come from MySQL. The priority and IGNORE modifiers are parsed but have no runtime effect.
  • Differences from PostgreSQL: the standard OVERRIDING { SYSTEM | USER } VALUE clause, a DEFAULT VALUES clause, naming a constraint as the conflict target (ON CONFLICT ON CONSTRAINT …), the WHERE predicate on ON CONFLICT … DO UPDATE, and common table expressions (WITH … INSERT) are not supported. A multi-column ON CONFLICT target is rejected. RETURNING is supported on the VALUES form.
