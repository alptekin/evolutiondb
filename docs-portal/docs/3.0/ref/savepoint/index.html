SAVEPOINT

Define a new savepoint within the current transaction block, marking a point you can later roll back to without aborting the whole transaction.

Synopsis

SAVEPOINT savepoint_name

Description

SAVEPOINT establishes a named marker inside the currently open transaction. Once a savepoint is set, you can use ROLLBACK TO SAVEPOINT to undo every change made after the savepoint was created — leaving the savepoint itself and any work done before it intact — or RELEASE SAVEPOINT to discard the marker and merge its work into the surrounding transaction.

In EvolutionDB, transaction control statements are interpreted by the protocol adaptor rather than by the SQL grammar parser. The adaptor maintains a per-session undo log of the changes made since the transaction began. Creating a savepoint records the current length of that undo log; rolling back to the savepoint replays the undo entries in reverse until the log is back at the recorded length. This means a savepoint is essentially a saved position in the undo stack, not a nested sub-transaction with its own transaction id.

A savepoint can only be defined while a transaction is open. If no transaction is in progress, the command fails with SQLSTATE 25P01 ("SAVEPOINT can only be used in a transaction"). A savepoint name is required; issuing SAVEPOINT with no name fails with SQLSTATE 42601.

If you create a savepoint with the same name as an existing savepoint in the current transaction, EvolutionDB overwrites the existing one in place, re-pointing it at the current position in the undo log. (See Compatibility — this differs from the SQL standard and PostgreSQL, which keep both and shadow the older one.)

Savepoint names are compared case-insensitively and are truncated to 127 characters. A single transaction can hold up to 64 active savepoints; attempting to create more fails with SQLSTATE 53000 ("too many savepoints").

Note. The command keyword and name are parsed positionally: the adaptor reads the word SAVEPOINT, then takes the next whitespace-delimited token (terminated by a space, a semicolon, or end of statement) as the savepoint name. There is no quoting or parenthesized option syntax.

Parameters

savepoint_name
The name to give the new savepoint. Required. The name is a single whitespace-delimited token; it is matched case-insensitively when later referenced by ROLLBACK TO SAVEPOINT or RELEASE SAVEPOINT, and is stored truncated to 127 characters. Reusing a name that already exists in the current transaction overwrites that savepoint rather than creating a second one.

Examples

Establish a savepoint, then roll back to it to discard only the most recent change:

BEGIN;

INSERT INTO accounts VALUES (1, 'opening');

SAVEPOINT after_open;

INSERT INTO accounts VALUES (2, 'oops');

-- undo the second insert, keep the first
ROLLBACK TO SAVEPOINT after_open;

INSERT INTO accounts VALUES (3, 'correct');

COMMIT;
-- rows 1 and 3 are committed; row 2 is gone

Release a savepoint once you are sure the work it guards is good. After release the marker is gone, but the underlying changes remain part of the transaction:

BEGIN;

SAVEPOINT sp1;

UPDATE inventory SET qty = qty - 1 WHERE id = 42;

RELEASE SAVEPOINT sp1;

COMMIT;

Reusing a savepoint name moves the marker forward instead of stacking a new one:

BEGIN;

INSERT INTO log VALUES ('a');
SAVEPOINT mark;

INSERT INTO log VALUES ('b');
-- re-pointing 'mark' here: a later ROLLBACK TO mark
-- will only undo work done after this point
SAVEPOINT mark;

INSERT INTO log VALUES ('c');
ROLLBACK TO SAVEPOINT mark;   -- undoes only 'c'

COMMIT;

Compatibility

EvolutionDB's SAVEPOINT follows the spirit of the SQL standard's <savepoint statement> and is wire-compatible with the PostgreSQL form for the common case of SAVEPOINT name. The companion commands ROLLBACK TO [SAVEPOINT] name and RELEASE [SAVEPOINT] name are accepted with the SAVEPOINT keyword optional, matching PostgreSQL.

Differences to be aware of:

Duplicate names overwrite
The SQL standard and PostgreSQL allow multiple savepoints with the same name, where the newest shadows the older ones until released. EvolutionDB instead overwrites a same-named savepoint in place, so there is never more than one savepoint per name in a transaction.
No nested sub-transactions
Savepoints are implemented as positions in a per-session undo log, not as nested transactions with their own ids. There is no separate sub-transaction visibility or per-savepoint error isolation beyond undoing the recorded changes.
Fixed savepoint limit
A transaction may hold at most 64 active savepoints. Exceeding this fails with SQLSTATE 53000.
Plain identifier names only
Names are read as a single unquoted, whitespace-delimited token and compared case-insensitively. Quoted or delimited identifiers and any parenthesized option syntax are not supported.
Statement handled by the adaptor
Transaction control statements (including SAVEPOINT) are recognized by the protocol adaptor, not the SQL grammar. They must appear as standalone statements rather than embedded within other SQL.
