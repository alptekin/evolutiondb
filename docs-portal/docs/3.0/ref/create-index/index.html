CREATE INDEX

Build a secondary index over one or more columns (or a supported expression) of a table to speed up lookups, equality scans, full-text matching, and approximate nearest-neighbor search.

Synopsis

CREATE [ UNIQUE ] INDEX [ CONCURRENTLY ] [ IF EXISTS ] name ON table
    [ USING HASH ]
    ( { column_name | expression } [, ...] )

-- Full-text inverted index
CREATE FULLTEXT INDEX name ON table ( text_column )

-- HNSW approximate nearest-neighbor (ANN) index on a VECTOR column
CREATE INDEX [ IF EXISTS ] name ON table USING HNSW
    ( vector_column [ opclass ] )
    [ WITH ( m = N [, ef_construction = N ] ) ]

where expression is one of:
    UPPER(col) | LOWER(col) | LENGTH(col) | CONCAT(col1, col2)

where opclass is one of:
    vector_cosine_ops | vector_l2_ops | vector_ip_ops | vector_l1_ops

Description

CREATE INDEX builds a secondary index on the named table. The default access method is a B+ tree stored in the unified page file; each entry is a composite key of the form indexed_value␟PK (the \x1F unit separator joins the indexed value with the row's primary key), and the entry points directly at the heap row location, so a matched index entry reaches its row without a second primary-key lookup.

The index is populated immediately by scanning every existing row in the table, building the key for each row, and inserting it. The command runs while holding the engine's write lock, so the table is quiesced during the build (except when CONCURRENTLY is used; see below).

An index may cover a single column, several columns (a composite index, whose key is the column values joined with |), or a single supported expression. Indexes on ARRAY columns are rejected in this version.

EvolutionDB supports four index shapes, selected by the modifiers and the access-method clause:

  • B+ tree (default), ordered index supporting equality and the prefix scans used by the planner.
  • Hash (USING HASH), equality-only index backed by an extendible-hash directory.
  • Full-text (CREATE FULLTEXT INDEX), an inverted index: the text column is tokenized and one (token, PK) pair is inserted per token.
  • HNSW (USING HNSW), an in-memory graph for approximate nearest-neighbor search over a VECTOR column.

If an index already exists on the same column list (and it is not the primary-key index), CREATE INDEX fails with index already exists on column '…', unless IF EXISTS was given, in which case the statement is a no-op.

Parameters

UNIQUE
Enforces that no two rows have the same indexed value. During the build, each key is checked for duplicates; the first collision aborts the command with could not create unique index: duplicate key '…' and the partially built index is destroyed. May be combined with USING HASH. Not accepted together with FULLTEXT or USING HNSW.
CONCURRENTLY
Builds the index without holding the write lock for the whole scan. The index is first registered in a building state, then populated in batches; the lock is released between batches so other queries can proceed. Concurrent writes during the build are recorded in a modification log and reconciled in a final pass before the index is marked ready. Only the building session pays the cost; readers ignore an index that is still building.
IF EXISTS
Makes the statement a no-op (instead of an error) when an index already exists on the same column list. Note the spelling: EvolutionDB's grammar accepts the IF EXISTS token sequence here, and it carries the standard "if not exists" meaning, skip creation if a matching index is already present.
name
The name of the index to create. Index names are looked up globally by DROP INDEX, so choose a name unique across the database.
table
The table on which to create the index. Must already exist, or the command fails with table '…' does not exist.
USING HASH
Selects an extendible-hash index instead of the default B+ tree. Hash indexes are created with the engine's default bucket count and support equality lookups only.
column_name [..]
One or more columns to index. Each named column must exist in the table, or the command fails with column '…' does not exist in table '…'. More than one column produces a composite index keyed on the concatenated column values.
expression
An expression index over a single supported function of a column. The accepted forms are exactly UPPER(col), LOWER(col), LENGTH(col), and CONCAT(col1, col2). The expression is serialized and stored with the index; at build and maintenance time the engine evaluates it per row to produce the key. Arbitrary SQL expressions are not supported.
FULLTEXT (in CREATE FULLTEXT INDEX)
Builds an inverted full-text index on a single text column. The column is tokenized at build time and one entry per token is inserted, supporting MATCH queries.

The full-text index is not automatically maintained on INSERT/UPDATE/DELETE in this version. After bulk DML, drop and re-create the index (CREATE FULLTEXT INDEX) to refresh it.

USING HNSW
Builds a Hierarchical Navigable Small World graph for approximate nearest-neighbor search. The indexed column must be a single VECTOR column (composite HNSW indexes are not supported in this version), otherwise the command fails with HNSW index requires a VECTOR column… or HNSW index supports a single vector column…. The graph lives in memory only; build parameters are persisted in the catalog so the graph can be rebuilt at server startup.
opclass (HNSW)
The distance function for the HNSW graph. One of vector_cosine_ops (default), vector_l2_ops, vector_ip_ops (inner product; vector_inner_ops is also accepted), or vector_l1_ops. An unrecognized opclass falls back to cosine.
WITH ( m = N, ef_construction = N ) (HNSW)
Tunes graph construction. m is the number of bidirectional links per node and ef_construction is the build-time candidate-list size. Defaults are m = 16 and ef_construction = 64. Later entries in the list override earlier ones for the same key.

Examples

Create a plain B+ tree index on a single column:

CREATE INDEX idx_users_email ON users (email);

Create a unique index:

CREATE UNIQUE INDEX idx_users_username ON users (username);

Create a composite index over two columns:

CREATE INDEX idx_orders_cust_date ON orders (customer_id, order_date);

Skip creation if a matching index is already present:

CREATE INDEX IF EXISTS idx_users_email ON users (email);

Create a hash index for equality lookups:

CREATE INDEX idx_sessions_token ON sessions USING HASH (token);

Create an expression index (case-insensitive lookups on a lowercased column):

CREATE INDEX idx_users_lower_email ON users (LOWER(email));

Build an index without locking the table for the whole scan:

CREATE INDEX CONCURRENTLY idx_events_type ON events (event_type);

Build a full-text index, then query it:

CREATE FULLTEXT INDEX idx_articles_body ON articles (body);

Build an HNSW vector index with custom build parameters:

CREATE INDEX idx_docs_embedding ON docs
    USING HNSW (embedding vector_l2_ops)
    WITH (m = 32, ef_construction = 128);

Compatibility

CREATE INDEX is not part of the SQL standard; the standard leaves indexing to the implementation. EvolutionDB's syntax follows PostgreSQL closely but differs in several concrete ways:

  • IF EXISTS vs IF NOT EXISTS, PostgreSQL spells the skip-if-present option IF NOT EXISTS. EvolutionDB's grammar accepts the token sequence IF EXISTS in this position, and it carries the same "skip if a matching index already exists" semantics.
  • Access methods: EvolutionDB supports the default B+ tree, USING HASH, and USING HNSW. PostgreSQL's btree/gin/gist/spgist/brin method names and a generic USING method clause are not supported.
  • Full-text, EvolutionDB uses a dedicated CREATE FULLTEXT INDEX statement (MySQL-style) rather than PostgreSQL's tsvector + GIN approach. The full-text index is not auto-maintained on DML.
  • Expression indexes, limited to UPPER, LOWER, LENGTH, and CONCAT(a, b). PostgreSQL allows arbitrary immutable expressions and parenthesized expression lists.
  • Unsupported clauses, EvolutionDB does not accept PostgreSQL's ASC/DESC, NULLS FIRST/LAST, COLLATE, per-column opclasses for B+ tree/hash indexes, INCLUDE, WHERE (partial indexes), TABLESPACE, or storage-parameter WITH (...) on non-HNSW indexes. The HNSW WITH clause accepts only m and ef_construction.
  • HNSW, the opclass names (vector_cosine_ops, vector_l2_ops, vector_ip_ops, vector_l1_ops) and WITH parameters mirror the pgvector extension. The HNSW graph is held in memory and rebuilt at startup from catalog-stored parameters.
