28. Routine Maintenance

The two housekeeping operations every EvolutionDB table needs over its lifetime, ANALYZE to keep planner statistics current and RECLAIM to sweep dead row versions and defragment the heap, plus the background daemon that runs reclamation and time-based pruning for you.

Because EvolutionDB uses multi-version concurrency control (see MVCC), a DELETE or UPDATE never removes a row in place: the old version stays on its heap page until no running transaction could still read it. Likewise, the cost-based planner cannot estimate row counts or selectivity without sampled statistics. Routine maintenance is what keeps both of these honest: statistics fresh, and obsolete tuples swept out of the heap and the indexes. Both operations are explicit SQL commands, and reclamation additionally runs on a background timer.

Terminology. EvolutionDB calls the storage-reclamation command RECLAIM, not VACUUM; there is no VACUUM keyword. The statistics command is ANALYZE TABLE, not bare ANALYZE. Neither command appears in the SQL standard, both are implementation-defined maintenance utilities.

28.1. Collecting Statistics with ANALYZE

ANALYZE TABLE performs a sequential scan of one table and records statistics in the system catalog. The planner reads these statistics, and EXPLAIN reports them, when it estimates how many rows a scan or predicate will produce. Without an ANALYZE, a table has no recorded row_count, and several maintenance heuristics described below fall back to conservative defaults.

ANALYZE TABLE table_name
    [ WITH SAMPLE n { PERCENT | ROWS } ]
    [ UPDATE HISTOGRAM ON column [, ...] [ WITH n BUCKETS ]
      | DROP HISTOGRAM ON column [, ...] ]

A bare ANALYZE TABLE t rebuilds every statistic for the table. The full reference, including every clause and its edge cases, lives on the ANALYZE command page; this section summarizes what is collected and how the numbers are derived.

28.1.1. What is collected

A single full ANALYZE TABLE writes four kinds of catalog records:

Table-level statistics
row_count (rows seen during the scan), page_count (heap pages, counted by walking the page chain), and last_analyzed (the wall-clock time of this run).
Per-column statistics
For each column: null_count, distinct_count, min, max, and avg_width (average value width in bytes).
Per-column histograms
One auto-selected histogram per column that has at least one non-NULL value (see 28.1.2).
Per-index statistics
For every index on the table, the B+ tree depth, leaf-page count, total entries, and average entries per leaf, read directly from the index tree.

On success the server prints a one-line summary, for example:

# Full scan
ANALYZE: 50000 rows, 4 columns, 412 pages

# Sampled scan
ANALYZE: 5012 rows sampled from 50000 (keep 10%), 4 columns, 412 pages

28.1.2. Histograms

Histograms give the planner a distribution rather than a single distinct-count, so it can estimate equality and range selectivity. EvolutionDB builds one of two flavors per column, chosen automatically:

  • Frequency histogram, built when the number of distinct values (NDV) is no greater than the bucket budget. It stores one bucket per distinct value with its exact count, giving exact equality selectivity rather than an estimate.
  • Equi-depth histogram, built when the NDV exceeds the bucket budget. Distinct values are sorted (numerically when every value parses as a number, otherwise lexicographically) and grouped into buckets that each hold roughly the same number of rows. Each bucket stores its upper bound and a cumulative count.

The default bucket budget is 100, and no histogram exceeds 100 buckets; UPDATE HISTOGRAM … WITH n BUCKETS overrides the default but is still capped at 100. Distinct-value tracking is capped at 10,000 values per column: columns with more distinct values are summarized from that capped sample, which is also what keeps equi-depth bounds representative on very-high-cardinality columns.

-- Rebuild histograms for two columns only, with a smaller bucket budget,
-- leaving the per-column statistics untouched.
ANALYZE TABLE orders
    UPDATE HISTOGRAM ON status, region
    WITH 32 BUCKETS;

-- Forget a histogram (catalog-only, no scan).
ANALYZE TABLE orders DROP HISTOGRAM ON notes;

28.1.3. Sampling

For large tables a full scan is expensive, so ANALYZE can inspect a Bernoulli sample. Each row is kept with a fixed probability; min/max always track the full scan (their accuracy is free), while null counts and width sums are scaled back up to full-table estimates, and histogram bucket counts are scaled so that histogram-backed selectivity matches the reported row_count.

WITH SAMPLE n PERCENT
Keep each row with probability n%, clamped to 1–100.
WITH SAMPLE n ROWS
Target approximately n sampled rows. The target is converted to a keep percentage using the table's previously recorded row_count; with no prior ANALYZE, this falls back to a full scan.
(no clause)
Automatic: a full scan for tables under 10,000 rows (per the previously recorded row_count), otherwise a 10% sample. Without a prior row count, a full scan is performed.
-- Re-analyze a large fact table with a 10% sample.
ANALYZE TABLE events WITH SAMPLE 10 PERCENT;

Contrast with PostgreSQL. EvolutionDB requires the TABLE keyword and accepts exactly one table per statement; there is no parenthesized option list (VERBOSE, SKIP_LOCKED) and no autovacuum-style automatic statistics refresh. Sampling is controlled per-statement with WITH SAMPLE rather than globally via default_statistics_target. Histogram bucket counts are set inline with WITH n BUCKETS rather than per-column with ALTER TABLE … SET STATISTICS.

28.2. Reclaiming Storage with RECLAIM

RECLAIM TABLE sweeps obsolete MVCC row versions out of one table and physically reorganizes its heap pages and indexes. Because EvolutionDB never overwrites a row in place, the space consumed by deleted and superseded versions accumulates until a RECLAIM (manual or automatic) frees it.

RECLAIM TABLE table_name

The command takes no options. Its full behavior and exit messages are documented on the RECLAIM command page; the phases below explain what actually happens to the heap.

28.2.1. What RECLAIM does

For a non-empty table, RECLAIM TABLE works in order:

  1. Dead-tuple removal and XID freeze. It walks every heap page and removes dead tuples: those whose inserting transaction aborted or crashed, and those whose deleting/updating transaction committed and is older than the oldest active transaction (the removal horizon). For each removed tuple it also cleans the primary-key B+ tree, every secondary index, and any TOAST overflow chain. In the same pass it freezes the transaction id of old committed live tuples, replacing an xmin older than the freeze cutoff with XID_FROZEN, so the original id can be recycled without wraparound risk.
  2. Per-page compaction. Each surviving heap page is compacted internally so its live records become contiguous and its free space is reclaimable.
  3. Sparse-page merge. Live records are moved from tail pages into earlier pages that have room. A record is relocated only when the primary-key entry currently points at that exact version, so stale or duplicate MVCC versions are never mis-repointed.
  4. Empty-page free + chain rebuild. Pages that end up with no live records are freed back to the shared page free list, and the heap page chain is rebuilt over only the surviving pages. Surviving pages are then marked all-visible in the visibility map.
  5. Index fix-up. The primary-key tree is updated to the new row locations and compacted (tombstones dropped, underflowing nodes merged); secondary indexes are kept consistent throughout.
  6. Bookkeeping. The table's dead_tuple_count is reset to zero and its frozen_xid is advanced.

If the table also has columns that were logically dropped with ALTER TABLE … DROP COLUMN, a final pass rewrites every tuple without the dropped columns and removes the dropped-column metadata from the catalog. (See ALTER TABLE.) On completion the server prints a one-line summary:

RECLAIM: 1240 dead tuples removed, 83 tuples frozen, compacted 38114 bytes, moved 512 records, freed 17 pages (228 pages remain)

An empty table prints RECLAIM: table is empty, nothing to do and returns immediately.

28.2.2. When to run it manually

The background daemon (28.3) reclaims most tables for you, so explicit RECLAIM is mostly needed after bulk operations that the daemon's ratio test would not catch promptly:

-- After a large bulk DELETE or a heavy UPDATE batch.
DELETE FROM events WHERE created_at < '2025-01-01';
RECLAIM TABLE events;

-- To physically finish an ALTER TABLE ... DROP COLUMN.
ALTER TABLE orders DROP COLUMN legacy_note;
RECLAIM TABLE orders;

Because dead-tuple counters are only an estimate, it is good practice to run ANALYZE TABLE after a RECLAIM that moved many rows, so the planner sees the new page_count and row distribution.

Locking. RECLAIM rewrites heap pages from an in-memory snapshot and returns freed pages to the single shared page free list. It must therefore run with exclusive access to the engine, it cannot overlap concurrent writers or readers of the table. When invoked by the background daemon it takes both the DML mutex and the parser write lock for the duration of the reclamation (see Server Configuration §23.3 for the lock model).

28.3. Automatic Reclamation and Time-Based Pruning

A background thread (auto-RECLAIM) starts with the server and stops at shutdown. On each tick it does two things: it triggers RECLAIM TABLE on tables that have accumulated too many dead tuples, and it runs a set of time-based pruning passes for retention, TTL, and durable subscriptions.

28.3.1. The dead-tuple trigger

For every non-temporary table in every database and schema, the daemon decides whether to reclaim using the table's recorded statistics:

With a recorded row_count (table has been analyzed)
Reclaim when dead tuples exceed 20% of the row count.
Without a recorded row_count (no prior ANALYZE)
Reclaim when the absolute dead-tuple count exceeds 50.
No dead tuples
Skip the table entirely.

Temporary tables are always skipped. When a table qualifies, the daemon logs and reclaims it under full mutual exclusion:

[auto-RECLAIM] Triggered for table mydb.public.events

28.3.2. Interval and tuning

The daemon ticks every 30 seconds by default. The interval can be lowered for test runs via an environment variable; the threshold and the 30-second ceiling are otherwise fixed in code.

EVOSQL_AUTO_RECLAIM_INTERVAL_SEC (env)
Lowers the tick interval. Accepted values are 1–30 seconds; values outside that range are ignored and the 30-second default stands. Intended for tests that need a prune or reclaim to fire within a single iteration, not for production tuning.

At startup the daemon logs its configuration, for example:

[auto-RECLAIM] Background thread started (interval=30s, threshold=20%, stack=16MB)

No autovacuum knobs. Unlike PostgreSQL's autovacuum, there is no per-table reloadable configuration: the 20% / 50-tuple thresholds and the freeze cutoff are compiled in, the only runtime knob is the test-only interval override above, and there is no cost-based delay. The daemon serves both as a vacuum equivalent and as the scheduler tick for the time-based prunes below.

28.3.3. Time-based pruning passes

On the same tick, and under the parser write lock, the daemon runs three pruning passes that delete expired rows. These are distinct from dead-tuple reclamation, they delete live rows that have aged out of a retention window, but they share the maintenance tick.

PassApplies toDeletes rows where…Window source
Temporal retention System-versioned tables' <name>_history shadow valid_to is older than the cutoff SYSTEM_TIME_RETENTION (days; 0 disables)
Subscription prune Each durable subscription's backing table seq ≤ the consumer's last acknowledged seq, or posted_at older than the subscription TTL Per-subscription ttl_days
TTL column expire Any table with a configured TTL column the TTL column's timestamp is older than NOW() Per-table TTL column; NULL values never expire

Each pass collects at most 256 victim rows per table per tick (to avoid invalidating its B+ tree cursor mid-traversal), then deletes them from the heap and the primary-key tree before moving on; a table with more expired rows than that is pruned over successive ticks. Rows whose retention/TTL timestamp is NULL are always preserved, on the contract that NULL means "no expiry yet." When a pass removes rows it logs a one-line count, for example:

[temporal-retention] Pruned 256 row(s) from orders_history
[ttl-prune] Expired 14 row(s) from session_token.sessions

The temporal-retention window is the only maintenance setting that is runtime-adjustable. It is seeded at boot by EVOSQL_SYSTEM_TIME_RETENTION_DAYS (default 7 days; 0 disables pruning) and can be changed for the running process with:

SET SYSTEM_TIME_RETENTION = 30;   -- keep history rows for 30 days

See Server Configuration §23.8 for the boot-time form, SET for the runtime form, and CREATE SUBSCRIPTION for durable-subscription TTLs.

28.4. A Maintenance Routine

For most deployments the background daemon handles reclamation and pruning automatically; the only routine task left to you is keeping statistics current. A workable cadence:

  1. Run ANALYZE TABLE on tables whose data distribution has shifted significantly (large loads, bulk deletes, or new value ranges), using WITH SAMPLE on large tables.
  2. After a bulk DELETE/UPDATE or an ALTER TABLE … DROP COLUMN, run RECLAIM TABLE explicitly rather than waiting for the daemon, then re-ANALYZE.
  3. Set SYSTEM_TIME_RETENTION and any per-table TTL columns to match your data-retention policy, and let the daemon prune.

Notes & limitations

  • No VACUUM, no autovacuum knobs. The command is RECLAIM TABLE; its thresholds (20% / 50 dead tuples), 30-second ceiling, and freeze cutoff are compiled in. The only runtime override is the test-only EVOSQL_AUTO_RECLAIM_INTERVAL_SEC (1–30 s).
  • One table per statement. Both ANALYZE TABLE and RECLAIM TABLE act on a single, optionally schema-qualified table; there is no database-wide form.
  • Statistics are not auto-refreshed. Reclamation runs automatically, but ANALYZE does not, stale statistics persist until you re-run it.
  • RECLAIM needs exclusive access. It cannot overlap concurrent readers or writers of the engine; the daemon enforces this with the DML mutex plus the parser write lock.
  • Per-tick prune cap. Each time-based pruning pass removes at most 256 rows per table per tick; large backlogs drain over multiple ticks.
  • Distinct-value cap. ANALYZE tracks at most 10,000 distinct values per column and at most 100 histogram buckets; both are summarized beyond those caps.
