22. Installation

How to obtain a running EvolutionDB 3.0 server — either by building the three executables from source with a C toolchain, or by pulling the prebuilt container image and starting it with docker compose.

EvolutionDB ships as three independent executables built from one source tree:

evolution/evosql
The standalone core SQL engine (parser, storage, executor). Built by make evolution. Useful for embedded or single-user experimentation; it has no network listener.
adaptor/evosql-server
The dual-protocol server. It links the engine object files directly (it does not link a shared library) and exposes the PostgreSQL wire protocol and the native EVO text protocol over TCP. This is what you run in production. Built by make adaptor.
cli/evosql-cli
The interactive command-line client (GNU readline based). Built by make cli.

Compared with PostgreSQL. PostgreSQL distributes a configure/make build with dozens of optional contrib modules and a separate initdb step to lay out a data directory. EvolutionDB has neither: there is no configure script, no initdb, and no cluster directory. The server creates and manages a single page file (evosql.db, 4 KB pages) on first start in its working directory.

22.1. Requirements

The build toolchain and library dependencies are deliberately small. The exact set depends on which component you are building and whether you enable TLS.

ComponentBuild toolsLibraries
Core engine (evosql)GCC, Makeliblz4
Server (evosql-server)GCC, Make; Bison + Flex only to regenerate the parserliblz4; libssl/libcrypto when TLS=1
CLI client (evosql-cli)GCC, Makelibreadline; libssl/libcrypto when TLS=1

Notes on individual libraries, drawn from the actual Makefiles:

liblz4
Always linked into the engine. It backs the TOAST overflow store that keeps large text rows from monopolising disk space. It is a small (~150 KB) dependency available as liblz4-dev (Debian/Ubuntu), lz4 (Homebrew, Alpine apk), or lz4-devel (RHEL/Fedora).
libreadline
Required only by the CLI client for line editing and history. The history file is ~/.evosql_history (capped at 500 lines).
OpenSSL (libssl / libcrypto)
Optional. Needed only when building with TLS=1 (server) or TLS=1 (CLI). The server uses it for the TLS transport and AES-256 Transparent Data Encryption.
Bison + Flex
Needed only to run make generate. The generated parser/lexer (evoparser.tab.c/h, lex.yy.c) are committed to the repository, so an ordinary build does not require Bison or Flex.

22.2. Building from source

From the repository root, the top-level Makefile drives the three sub-project Makefiles. Order matters: the engine is built first because the server and CLI link against its object files.

# Build all three executables (evolution, adaptor, cli)
make

# Or build a single component
make evolution
make adaptor
make cli

# Clean every object file and executable
make clean

A successful make prints the paths of the three binaries it produced:

====================================
  Build complete!
    evolution/evosql
    adaptor/evosql-server
    cli/evosql-cli
====================================

22.2.1. Enabling TLS and encryption

TLS support is off by default. Pass TLS=1 to compile the server with the OpenSSL transport and the page-encryption code path (the EVOSQL_TLS and EVO_ENCRYPTION compile-time flags):

# Server with OpenSSL TLS + Transparent Data Encryption
make adaptor TLS=1

# CLI with TLS so it can connect to a TLS-enabled server
make -C cli TLS=1

TLS is a build-time decision. Unlike PostgreSQL, where ssl = on is a runtime postgresql.conf setting on a single binary, EvolutionDB compiles the TLS code in or out. A server built without TLS=1 cannot serve TLS at all, regardless of environment variables. See 23. Server Configuration for the runtime certificate variables (EVOSQL_TLS_CERT, EVOSQL_TLS_KEY).

22.2.2. Regenerating the parser

You only need this if you have modified the grammar (evolution/parser/evoparser.y) or lexer (evolution/parser/evolexer.l). It requires Bison and Flex:

# Regenerate evoparser.tab.c/h and lex.yy.c, then rebuild
make generate
make

22.2.3. Release binaries

make release builds everything, copies the three executables into release/, and strips their symbol tables:

make release   # stripped binaries land in release/

22.3. Platform notes

The Makefiles detect the platform with uname -s and adjust link flags and the executable suffix accordingly.

Linux
The reference platform. Executables have no suffix. Server links -lpthread -lm -llz4. Install dev packages with your package manager, e.g. apt-get install gcc make bison flex liblz4-dev libssl-dev libreadline-dev.
macOS (Darwin)
Builds native binaries with no suffix. Because Homebrew keeps liblz4, readline, and openssl off the default search path, the Makefiles add the Homebrew cellar include/lib paths automatically (e.g. -I/opt/homebrew/include, and brew --prefix readline / brew --prefix openssl for the keg-only formulas). Install with brew install lz4 readline openssl bison flex.
Windows (MinGW / MSYS2)
Produces .exe files (evosql.exe, evosql-server.exe, evosql-cli.exe). The Makefiles redirect linker temp files to /tmp to work around a permission issue, and link the Winsock library -lws2_32 instead of pthreads.

Honest scope. The container image and CI run on Linux; macOS and MinGW are supported by the Makefiles but are best-effort targets. There are no prebuilt OS packages (no .deb, .rpm, Homebrew tap, or Windows installer) — you build from source or use Docker.

22.4. Installing with Docker

The shipped docker-compose.yml builds the server from a multi-stage Dockerfile (gcc:13 builder → debian:bookworm-slim runtime, built with TLS=1) and starts it. This is the recommended path for testing and for clients such as DBeaver, pgAdmin, or psql.

# Start the server (builds the image on first run)
docker compose up -d              # PG:5433  EVO:9967

# Stop the server, keeping the data volume
docker compose down

# Stop the server and DELETE all data
docker compose down -v

To run the published image directly without the compose file:

docker run -d -p 5433:5433 -p 9967:9967 \
    -e EVOSQL_PASSWORD=mysecret evolutiondb/evolutiondb:latest

The container entrypoint generates a self-signed TLS certificate under /data/certs/ on first start (10-year validity, CN evosql-server) if none exists, exports EVOSQL_TLS_CERT/EVOSQL_TLS_KEY, and then execs the server. The database file and certificates live in the /data volume, so data survives docker compose down but is destroyed by down -v.

22.4.1. Ports

The server is dual-protocol. The defaults below come directly from the source (DEFAULT_PG_PORT, DEFAULT_EVO_PORT, REPL_DEFAULT_PORT) and the compose file.

PortProtocolDefaultHow to change
PostgreSQL wirePG frontend/backend protocol — psql, DBeaver, JDBC5433--pg-port
EVO nativeEvoSQL text protocol — evosql-cli, C/Python SDK9967--evo-port
ReplicationWAL sender for replicas9968 (disabled unless configured)--replication-port
CDC streamChange-data-capture, JSON Linesdisabled (compose maps 9970)--cdc-port / EVOSQL_CDC_PORT

Note: port 5433, not 5432. EvolutionDB defaults its PostgreSQL-compatible listener to 5433 so it does not collide with a real PostgreSQL on the same host. Point your client at -p 5433.

22.4.2. First-start credentials

The default administrative user name is admin (EVOSQL_USER_NAME). The password is controlled by EVOSQL_PASSWORD:

  • If EVOSQL_PASSWORD is set, that password is used. The bundled docker-compose.yml sets it to admin for convenience.
  • If it is unset, a random password is generated and printed to stdout once on first start — check docker logs to retrieve it.

A few other environment variables seen in the compose file shape first start; the complete reference is in 23. Server Configuration:

EVOSQL_ENCRYPTION_KEY
Master passphrase for AES-256 Transparent Data Encryption. Set it to an empty string to disable encryption. Page 0 (the file header) is always stored in plaintext regardless.
EVOSQL_CDC_PORT
Enables the change-data-capture streaming server on the given port (JSON Lines).
EVOSQL_AUTO_RECLAIM_INTERVAL_SEC
How often the background reclaim/TTL tick runs. The compose file lowers it to 2 so TTL-driven row expiry is visible in tests; production leaves the default (30 s).
EVOSQL_BUFFER_POOL_SIZE
Clock-sweep buffer pool size, accepting a raw page count or a unit suffix (K/M/G) interpreted against the 4 KB page size; also settable via --buffer-pool-size.

22.5. Running a local build

After make, start the server and connect. The server prints its version and a banner showing the active ports and whether TLS is compiled in.

# Start the server (defaults: PG 5433, EVO 9967)
./adaptor/evosql-server

# Use a different PostgreSQL-wire port
./adaptor/evosql-server --pg-port 5434

# Connect with the native CLI (EVO port 9967)
./cli/evosql-cli
./cli/evosql-cli -W admin        # prompt for password

# Or connect any PostgreSQL client over the wire protocol
psql -h 127.0.0.1 -p 5433 -U admin testdb

Once connected you can create objects immediately — there is no initdb or template database to bootstrap. See CREATE DATABASE, CREATE TABLE, and the agent-memory DDL such as CREATE MEMORY STORE and CREATE CHECKPOINT STORE.

22.6. Verifying the installation

The test suite is a set of Python scripts that speak the raw PG wire protocol (no psycopg2 dependency). They require a running server and connect to 127.0.0.1:5433 (PG) or :9967 (EVO) with the default credentials admin/admin against database testdb.

# Start the server first
docker compose up -d

# Run individual test files
python tests/test_memory_store.py
python tests/test_checkpoint_store.py
python tests/test_evo_protocol.py

Notes & limitations.

  • No configure script and no initdb: the engine creates its single 4 KB-page file (evosql.db) in the working directory on first run.
  • The server links the engine's object files directly rather than a shared library; there is no installable libevolution. (The separate C client SDK, client/libevosql-memory/, is built with its own make -C client/libevosql-memory.)
  • TLS and page encryption are compile-time options (TLS=1); a non-TLS build cannot be switched to TLS at runtime.
  • Bison and Flex are needed only for make generate — never for an ordinary build, because the generated sources are committed.
  • No OS-native packages are published; distribution is source build or container image only.
