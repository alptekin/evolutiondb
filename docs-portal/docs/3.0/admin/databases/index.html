24. Managing Databases

A database in EvolutionDB is a named, catalog-registered namespace that owns one or more schemas; this chapter covers creating databases, switching between them with USE, and how the multi-database layout maps onto EvolutionDB's single unified storage file.

24.1. Overview

Unlike systems that give each database its own files or its own physical cluster directory, EvolutionDB stores every database in one unified, page-based file: root/evosql.db. The root/ directory contains nothing else, there are no per-database subdirectories on disk. Databases, schemas, tables, and the rest of the metadata all live inside that file's system catalog, which is itself a set of B+trees.

This means that "creating a database" is a pure catalog operation: it inserts a row into the catalog's database list and does not allocate or initialize any separate file. A database is therefore best understood as a logical grouping rather than a physical storage boundary.

layers The naming hierarchy is three levels deep: database → schema → table. Every database created by EvolutionDB is given an initial schema named default, and every session starts out pointed at some database and some schema. Schema management is covered in detail in Chapter 25, Managing Schemas; this chapter focuses on the database level.

24.2. The Default Database and Schema

On the very first start of a fresh storage file, EvolutionDB bootstraps the catalog and creates a single database named evosql containing a single schema named default. These are the values a new session inherits if nothing else is specified: the current database defaults to evosql and the current schema to default.

Default database
evosql, created automatically when the catalog is initialized for the first time. It always has internal database id 1.
Default schema
default, created inside every database (including evosql) at creation time.
Storage root
The directory root/, holding the single file root/evosql.db. The directory name is fixed; it is not per-database.

info PostgreSQL contrast. PostgreSQL ships with postgres, template0, and template1, and CREATE DATABASE physically clones a template directory. EvolutionDB has no template databases and no per-database directory, there is nothing to clone. A new database is simply a catalog entry plus its own default schema.

24.3. Creating a Database

Use the CREATE DATABASE command. The only argument is the database name; EvolutionDB does not accept the per-database options found in standard SQL or PostgreSQL (owner, encoding, locale, template, tablespace, connection limit, and so on). 

CREATE DATABASE name
CREATE DATABASE IF NOT EXISTS name

Creating a database registers it in the catalog and immediately gives it a schema named default:

CREATE DATABASE shop;
-- server responds: CREATE DATABASE shop

-- The new database already contains schema "default".
USE shop;
CREATE TABLE products (
    id    INT PRIMARY KEY,
    name  VARCHAR(100)
);  -- created in shop.default.products

24.3.1. IF NOT EXISTS

Without IF NOT EXISTS, creating a database whose name is already registered fails with SQLSTATE 42E04 (duplicate database). With IF NOT EXISTS, the command is a no-op and succeeds silently when the database already exists. 

CREATE DATABASE shop;                  -- ok
CREATE DATABASE shop;                  -- ERROR 42E04: database "shop" already exists
CREATE DATABASE IF NOT EXISTS shop;    -- ok, no change

warning Note. CREATE DATABASE does not switch your session to the new database. The current database is unchanged after the command returns; you must issue USE separately to start working inside it.

24.4. Switching Databases with USE

The USE command changes which database subsequent statements operate against. EvolutionDB verifies the name against the catalog, and if found, sets it as the session's current database. As a side effect, the current schema is reset to default, switching databases does not preserve the schema you had selected before. 

USE database_name
USE shop;
-- server responds: USE DATABASE shop
-- current database = shop, current schema reset to "default"

If the named database is not present in the catalog, USE fails with SQLSTATE 3D000 (undefined database) and the session's current database is left untouched: 

USE nope;   -- ERROR 3D000: database "nope" does not exist

24.4.1. USE Is Per-Session

The current database and current schema are stored in each connection's own query context, not as a global server setting. Two clients connected at the same time can therefore be pointed at two different databases simultaneously, and one connection's USE never affects another. The selection lasts only for the life of the connection.

info PostgreSQL / standard-SQL contrast. In PostgreSQL there is no USE statement; you select a database by reconnecting (a connection is bound to one database for its lifetime). EvolutionDB's USE behaves more like the MySQL USE statement, an in-session switch, but it additionally resets the active schema to default, which MySQL does not, because MySQL conflates "database" and "schema".

24.4.2. The Database Chosen at Connect Time

When a client connects over the PostgreSQL wire protocol, the database name requested in the startup message becomes the session's initial current database, so a properly configured PostgreSQL client (or driver, such as the one used by DBeaver or psql) lands directly in the database it asked for without an explicit USE. A session that does not specify a database falls back to the default, evosql.

24.5. The Multi-Database Layout

Because all databases share root/evosql.db, qualifying an object name is purely logical. A fully qualified table reference has the form: 

database.schema.table

At runtime, the engine resolves unqualified names against the session's current database and current schema. The legacy path form root/<database>/<schema>/<table> still appears internally for backward compatibility when extracting bare table names, but it does not correspond to real directories on disk: it is a naming convention, not a filesystem layout.

LevelCreated byNotes
DatabaseCREATE DATABASECatalog entry only. Always seeded with a default schema.
SchemaCREATE SCHEMACreated inside the current database. See Chapter 25.
TableCREATE TABLECreated inside the current schema of the current database.

24.5.1. Practical Implications

One file, one buffer pool
Every database is paged through the same 4 KB buffer pool and the same write-ahead log. Databases are an organizational construct; they do not isolate storage, caching, or I/O from one another.
No per-database tuning
There are no per-database encodings, collations, owners, or tablespaces to set, because CREATE DATABASE accepts only a name.
Names are case-sensitive in the catalog
Database names are stored and looked up as given. Note, however, that the current-database guard compares names case-insensitively.

24.6. Dropping a Database

The DROP DATABASE command removes a database and cascades to all of its schemas and the tables within them. 

DROP DATABASE name
DROP DATABASE IF EXISTS name
DROP DATABASE shop;             -- drops shop and everything inside it
DROP DATABASE IF EXISTS shop;   -- no error if shop is already gone

You cannot drop the database your session is currently using. Switch away first (or connect elsewhere):

USE shop;
DROP DATABASE shop;   -- ERROR 55006: cannot drop the currently open database "shop"
USE evosql;
DROP DATABASE shop;   -- ok

warning Note — the drop is unconditionally cascading. DROP DATABASE always removes every schema and every table the database contains. There is no RESTRICT mode that refuses to drop a non-empty database, so this is a destructive, irreversible operation. Without IF EXISTS, dropping a database that does not exist fails with SQLSTATE 3D000 (undefined database).
