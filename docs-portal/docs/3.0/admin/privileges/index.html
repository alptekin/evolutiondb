25. Roles & Privileges

EvolutionDB controls who may connect and what they may do through three catalog-backed concepts: login users, non-login roles, and object privileges checked by a most-specific-first TABLE → SCHEMA → DATABASE waterfall.

This chapter describes the access-control model exactly as it is implemented in the engine (evolution/db/UserMgmt.c and evolution/db/GrantMgmt.c). All users, roles, and grants are stored as ordinary rows in the system catalog B+ trees inside the unified evosql.db file; there is no separate authentication file. Where EvolutionDB diverges from the SQL standard or from PostgreSQL, the difference is called out explicitly rather than assumed.

25.1. Overview of the access-control model

Three catalog tables underpin everything in this chapter:

CAT_SYS_USERS
One row per principal. A principal is either a login user (is_role = 0) or a role (is_role = 1). The key is the principal name; the record carries the PBKDF2-SHA256 password hash (empty for roles).
CAT_SYS_GRANTS
One row per grant. The composite key is username:scope_type:scope_name, and the record stores a comma-separated privilege list and a single grant_option flag. Role memberships are stored in this same tree using scope_type = ROLE.

Two non-table principals are special and exist implicitly:

  • admin — the built-in superuser. §25.6.
  • PUBLIC — a synthetic principal whose grants apply to every user additively. §25.5.

Not the SQL-standard separation. In the SQL standard and in PostgreSQL, authorization identifiers, roles, and privileges are distinct catalog concepts. In EvolutionDB users and roles share a single catalog table distinguished only by the is_role flag, and a role is simply a principal that cannot log in but can hold grants. There is no privilege inheritance graph beyond direct role membership (§25.4).

25.2. Users

A user is a principal that can authenticate and open a session. Users are created, altered, dropped, and listed through SQL that the adaptor intercepts and dispatches to UserMgmt.c.

25.2.1. Creating users

See the CREATE USER reference page for full syntax.

CREATE USER name PASSWORD 'password';

Both the name and a non-empty password are mandatory; either being empty raises an error (SQLSTATE 22023, invalid parameter value). The password is hashed with PBKDF2-SHA256 before the catalog write lock is taken, then the user row is created if and only if no principal of that name already exists. A duplicate name raises SQLSTATE 42710 (duplicate object).

-- create an application login
CREATE USER app_rw PASSWORD 's3cret-pw';

25.2.2. Changing a password

ALTER USER name PASSWORD 'newpassword';

ALTER USER ... PASSWORD re-hashes the new password and overwrites the stored hash. EvolutionDB currently supports only the password change form of ALTER USER; there are no renames, attribute changes, or privilege attributes (SUPERUSER, CREATEDB, etc.) as found in PostgreSQL. Altering a name that does not exist raises SQLSTATE 42704 (undefined object).

ALTER USER app_rw PASSWORD 'rotated-pw';

25.2.3. Dropping users

DROP USER name;

Dropping a user removes the catalog row. The engine also exposes DropUserGrants(), which deletes every grant whose key begins with that username, so a dropped user leaves no orphaned grants behind. Dropping a non-existent user raises SQLSTATE 42704.

25.2.4. Listing users

SHOW USERS enumerates the principals in CAT_SYS_USERS and filters out anything with is_role = 1, so it returns login users only. See SHOW.

SHOW USERS;

25.3. Authentication

When a session presents credentials, AuthenticateUser() performs the following, in order:

  1. The supplied username is lower-cased and looked up in the catalog. If that fails, the original (unmodified) case is tried as a fallback. This makes login effectively case-insensitive for the common case while still tolerating a name that was stored with mixed case.
  2. If the matched principal is a role (is_role = 1), authentication fails immediately — roles cannot log in.
  3. The presented password is verified against the stored hash with a constant-time comparison to resist timing attacks.

25.3.1. Password hashing

Passwords are never stored in plaintext. The stored value is a self-describing PBKDF2-SHA256 string:

pbkdf2:100000:<32-hex-salt>:<64-hex-hash>
Algorithm
PBKDF2 with HMAC-SHA256.
Iterations
100000 (compiled constant PBKDF2_ITERATIONS).
Salt
16 random bytes per password (PBKDF2_SALT_LEN), stored as hex in the hash string.
Derived key
32 bytes (PBKDF2_KEY_LEN).

Plaintext password buffers and derived hashes are wiped from memory with evo_secure_wipe() after use.

25.3.2. The initial admin password

On first start, the engine establishes the admin login from environment variables:

EVOSQL_USER_NAME
The initial login name. Defaults to admin when unset or empty.
EVOSQL_PASSWORD
The initial password. If unset or empty, a random 16-character password is generated, hashed, stored, and printed to stdout exactly once (MySQL --initialize style). It is never shown again; rotate it later with ALTER USER.
# first boot with a generated password (printed once on stdout)
./adaptor/evosql-server

# or pin the credentials up front
EVOSQL_USER_NAME=admin EVOSQL_PASSWORD=changeme ./adaptor/evosql-server

Documentation vs. source note. The project security guideline recommends a PBKDF2 iteration count of at least 600,000. The shipped 3.0 engine compiles PBKDF2_ITERATIONS = 100000. This page documents the value the code actually uses; treat the higher figure as a hardening target, not current behavior.

25.4. Roles

A role is a non-login principal that can hold privileges and be granted to users. Roles let you assemble a set of privileges once and attach it to many users.

25.4.1. Creating and dropping roles

CREATE ROLE name;
DROP ROLE name;

CREATE ROLE fails if any user or role of that name already exists — users and roles share one namespace. DROP ROLE verifies the target really is a role (is_role = 1) and then performs a full cleanup: it removes the role's catalog row, deletes all grants held by the role, and removes every membership edge that points to the role (so no user is left a member of a role that no longer exists).

25.4.2. Granting roles to users

GRANT role_name TO user_name;
REVOKE role_name FROM user_name;

Membership is itself a catalog grant: it is stored with scope_type = ROLE, scope name equal to the role, and the literal privilege text MEMBER. Granting a role enforces three checks: the role must exist, the target user must exist, and the target must be a user (not another role). Granting a role to a principal that is itself a role is rejected — roles cannot be nested. A duplicate membership is also rejected.

CREATE ROLE reporting;
GRANT SELECT ON SCHEMA testdb.public TO reporting;
GRANT reporting TO app_rw;   -- app_rw now inherits reporting's grants

25.4.3. How role privileges are inherited

During a privilege check, after a user's own grants and PUBLIC grants are exhausted, the engine looks up the user's direct role memberships and re-runs the same waterfall (§25.7) for each role. Membership is resolved one level deep only — a role granted to a role would not be followed, and in any case such nesting is disallowed at grant time. At most 16 role memberships per user are consulted during a check.

25.5. The PUBLIC pseudo-principal

Grants made to PUBLIC apply to every user. EvolutionDB treats PUBLIC grants as purely additive: during a check the engine first evaluates the user's own grants, then — if still undecided — evaluates PUBLIC grants over the identical waterfall. A user-specific REVOKE therefore does not cancel a privilege that was granted to PUBLIC; to remove it you must revoke from PUBLIC directly.

No default PUBLIC grants. Unlike PostgreSQL, which grants some default privileges to PUBLIC on new objects, a freshly created EvolutionDB database carries no implicit PUBLIC grants (a MySQL-style closed default). New non-admin users start with no access until a grant is made.

25.6. The admin superuser

The principal named admin (matched case-insensitively) is a hard-coded superuser. Every privilege check for admin returns allowed immediately, with no catalog I/O, and HasGrantOption() likewise always returns true for admin. To make the rule self-consistent, on first initialization the engine seeds one grant — admin on DATABASE * with ALL and grant option — but the in-memory fast path means admin is authorized even before any grant row is consulted.

Caution. Superuser status is fixed to the literal name admin. There is no ALTER USER ... SUPERUSER and no way to promote or demote superuser status at runtime. Treat the admin credential as the master key to the instance.

25.7. Privileges and scopes

EvolutionDB recognizes exactly six object privileges plus the ALL shorthand. They are stored per grant as a comma-separated text list and parsed into a bitmask (parse_privs()) for checking. Privilege keywords are case-insensitive; ALL and ALL PRIVILEGES are equivalent.

KeywordInternal bitGoverns
SELECT0x01Reading rows
INSERT0x02Adding rows
UPDATE0x04Modifying rows
DELETE0x08Removing rows
CREATE0x10Creating objects
DROP0x20Dropping objects
ALL0x3FAll six of the above

An unrecognized token contributes nothing to the mask. A privilege string that parses to an empty mask is rejected by both GRANT and REVOKE as an invalid privilege specification (SQLSTATE 42501).

25.7.1. Scope levels

A grant attaches a privilege set to one of three object scopes. The scope name is canonicalized to a dotted path:

ScopeType codeScope-name form
DATABASE1database (or * for the wildcard)
SCHEMA2database.schema
TABLE3database.schema.table
ROLE4rolename (membership edge; see §25.4.2)

The * wildcard is only honored at DATABASE scope and is the mechanism behind the seeded admin grant. There is no schema- or table-level wildcard, and there are no column-, sequence-, or routine-level privileges.

25.8. GRANT and REVOKE

Full syntax is on the GRANT and REVOKE reference pages.

GRANT priv[, priv ...] ON {DATABASE | SCHEMA | TABLE} object TO user [WITH GRANT OPTION];
REVOKE priv[, priv ...] ON {DATABASE | SCHEMA | TABLE} object FROM user;

25.8.1. GRANT semantics

A grant is keyed by (user, scope_type, scope_name). If no grant exists for that triple, a new row is created with the parsed privilege mask. If a grant already exists, the new privileges are merged (bitwise OR) into the existing set rather than replacing it, and WITH GRANT OPTION is sticky — specifying it sets the option to true, while omitting it leaves an already-set option untouched. GRANT never silently drops a privilege the principal already held at that scope.

CREATE USER analyst PASSWORD 'pw';
GRANT SELECT ON TABLE testdb.public.orders TO analyst;
GRANT INSERT, UPDATE ON TABLE testdb.public.orders TO analyst;
-- analyst now holds SELECT,INSERT,UPDATE on that table

25.8.2. WITH GRANT OPTION

The grant option is a single boolean stored on the grant row — it is not tracked per individual privilege. HasGrantOption() answers whether a principal may re-grant a privilege: it returns true if the principal holds the requested privilege at the exact scope and the row's grant-option flag is set. For DATABASE scope the wildcard * grant is also consulted. (admin always passes.)

25.8.3. REVOKE semantics

REVOKE targets the same (user, scope_type, scope_name) triple. If no matching grant exists, it errors with SQLSTATE 42704. Otherwise the requested privileges are cleared from the mask: if some privileges remain, the grant row is rewritten with the reduced set; if none remain, the row is deleted entirely. Revoking ALL additionally clears the grant option on the way out.

REVOKE INSERT, UPDATE ON TABLE testdb.public.orders FROM analyst;
-- analyst retains only SELECT on that table
REVOKE ALL ON TABLE testdb.public.orders FROM analyst;
-- grant row removed entirely

No cascading revoke. REVOKE affects only the named principal's own grant row. It does not cascade to principals who received the privilege from that principal via WITH GRANT OPTION, and there is no RESTRICT/CASCADE clause. There is also no separate "revoke grant option only" form — to drop the option you must revoke and re-grant.

25.8.4. Inspecting grants

SHOW GRANTS lists grants for the current user; SHOW GRANTS FOR user lists them for a named principal. With no user specified, the engine enumerates every user's grants and then appends the PUBLIC grants. See SHOW.

SHOW GRANTS FOR analyst;

25.9. The TABLE → SCHEMA → DATABASE waterfall

Authorization for a data operation is decided by CheckPrivilege(username, database, schema, table, privilege). The check is most-specific-first and short-circuits the moment any source grants the needed bit. The order is fixed:

  1. admin fast path — if the principal is admin, return allowed at once.
  2. User grants, by scope, narrowest first:
    1. TABLE — database.schema.table (only attempted when all three are present)
    2. SCHEMA — database.schema
    3. DATABASE — the named database, then the * wildcard
  3. PUBLIC grants — identical TABLE→SCHEMA→DATABASE→* sequence for the PUBLIC principal.
  4. Role-inherited grants — for each of the user's roles, the same waterfall again.
  5. If nothing matched, return denied.

"Most specific wins first" means a privilege found at TABLE scope ends the search before SCHEMA or DATABASE rows are even read. Because the check stops at the first granting source, the waterfall can only ever add authority as it widens — it never withholds a broad grant because a narrower scope is silent. There is no deny rule: a grant either exists and permits, or it is absent.

-- broad read at the database level
GRANT SELECT ON DATABASE testdb TO analyst;
-- a SELECT on testdb.public.orders is now allowed:
-- TABLE and SCHEMA rows are absent, the DATABASE grant matches.

-- a narrower write grant, layered on top
GRANT INSERT ON TABLE testdb.public.orders TO analyst;
-- INSERT on that one table is allowed via the TABLE grant;
-- INSERT elsewhere in testdb remains denied.

25.11. Notes and limitations

What EvolutionDB 3.0 does not implement. The model is deliberately narrower than the SQL standard or PostgreSQL. Notably:

  • No column-, sequence-, function/procedure-, or large-object-level privileges. Only the six object privileges in §25.7 exist, at DATABASE/SCHEMA/TABLE scope only.
  • No object OWNER concept and therefore no implicit owner privileges. Authorization comes entirely from explicit grants (plus the admin fast path and PUBLIC).
  • No role inheritance graph beyond one level of direct membership, and roles cannot be nested. At most 16 memberships per user are consulted per check.
  • No cascading REVOKE, no RESTRICT/CASCADE, and no standalone way to drop only a grant option.
  • No USAGE, TEMPORARY, CONNECT, or other PostgreSQL privilege keywords.
  • ALTER USER changes only the password; there are no login attributes or role flags.
  • Superuser is fixed to the name admin and cannot be reassigned.
  • The shipped PBKDF2 iteration count is 100000, below the project's stated 600,000 hardening target.
