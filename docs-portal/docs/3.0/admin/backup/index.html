27. Backup, WAL & Recovery

This page explains how EvolutionDB keeps your data safe when a crash happens. It covers the full-page-image write-ahead log, the checkpoint that recycles it, two-pass replay on restart, point-in-time recovery from the WAL archive, and the base backup that clones a live database.

Durability in EvolutionDB rests on one mechanism. It is a full-page-image (FPI) write-ahead log built in evolution/db/wal.c. Before any changed 4 KB page reaches the data file evosql.db, a full image of that page is added to the log file evosql.wal and flushed. If the process dies between the log write and the data write, the next start replays the logged images. No page is ever left torn. Everything else in this chapter is built on top of that one log. That includes checkpoints, recovery, the time-travel archive, and base backups.

A note on terminology. The SQL keyword CHECKPOINT in EvolutionDB belongs to the LangGraph checkpoint store feature. See CHECKPOINT and CREATE CHECKPOINT STORE. It has nothing to do with the WAL checkpoint described here. The WAL checkpoint is an internal engine operation. No SQL statement triggers it. When this chapter says “checkpoint” on its own, it means the WAL checkpoint.

27.1. The Write-Ahead Log

The WAL lives in one sequential file named evosql.wal. It is created next to the data file. The page manager and buffer pool manage it fully. It is not a directory of segments. It begins with a fixed 16-byte header. After the header comes a stream of variable-length records.

FieldSizeMeaning
magic4 B0x57414C45, which means “WALE”. A header without it is treated as corrupt and reset.
version2 BThe WAL format version. It is 1 right now.
reserved2 BPadding.
checkpoint_lsn4 BEvery page up to this LSN is known to be on disk in the data file. Replay skips records at or below it.
next_lsn4 BThe next log sequence number to assign. It starts at 1 for a fresh log.

Each record after the header is a full page image. It is wrapped in an 18-byte header and a CRC at the end.

[ lsn:4B ][ page_no:4B ][ page_len:2B ][ timestamp:8B ][ page image:page_len ][ crc32:4B ]
\________________ 18-byte record header _________________/
lsn
A log sequence number that always increases. It is assigned under the WAL mutex when the record is written.
page_no
The data-file page this image belongs to. Values 0xFFFFFFFC–0xFFFFFFFE are reserved sentinels for distributed-transaction records. They are not real pages. See §27.7.
page_len
The number of bytes in the page payload. It is always 1–4096. A record that claims 0 or more than EVO_PAGE_SIZE ends replay.
timestamp
Microseconds since the Unix epoch. It is captured with gettimeofday() at log time. This is what makes the WAL archive replayable to a point in time. See §27.5.
crc32
An IEEE CRC-32 over the 18-byte header plus the page image. It uses the zlib and gzip polynomial. Replay recomputes it and compares it. If it does not match, replay stops at that record. A partly written tail record is never applied.

EvolutionDB logs whole pages, not logical row operations. There is no redo or undo record type. There is no per-tuple delta. There is no separate logical record stream in the WAL itself. The unit of recovery is the 4 KB page. This makes replay safe to repeat. Re-applying a page image does no harm. The cost is a larger log than a delta-based design would produce.

Contrast with PostgreSQL. PostgreSQL writes typed redo records. It only writes a full-page image for the first change to a page after a checkpoint. The rest are deltas. EvolutionDB always logs the full page. PostgreSQL stores the WAL as a directory of 16 MB segment files. Their names come from the LSN. EvolutionDB uses one growing evosql.wal file. That file is cut back to its 16-byte header at each checkpoint.

27.2. The Write-Ahead Rule & Commit Path

The core rule is simple. A page's WAL record must be durable before that page is allowed to reach the data file. EvolutionDB applies this rule at commit time, not on every page write. It does this by holding back both the data-page flush and the WAL fsync until a transaction commits.

An auto-commit statement or an explicit COMMIT may succeed. When it does, the server calls bp_wal_flush_dirty() in the buffer pool. This happens before the transaction is marked committed in the commit log. The steps are below.

  1. Walk the set of pages this transaction made dirty. A small array tracks them. If that array overflows, it falls back to a full buffer-pool scan. This is rare.
  2. For each dirty page, add its current image to the WAL with wal_log_page().
  3. Run a single wal_fsync() that flushes all of those records at once.
  4. Only then assign a commit sequence number and record the commit.

Batching the fsync to one flush per commit instead of one per page is the key performance choice. The source notes it is roughly 100x faster than a per-page sync. The trade-off is that the durability boundary is the commit. If the server crashes after COMMIT returns, the WAL holds the images and recovery restores them. If it crashes mid-transaction, the un-flushed dirty pages are lost and the partial work never reaches disk.

Durability is at commit, not at statement. Because the WAL fsync is deferred to commit, a statement inside an open transaction is not durable until you COMMIT (see transaction control). This matches the SQL standard's notion of transactional durability and PostgreSQL's default behavior. EvolutionDB has no per-statement or asynchronous-commit knob to change it.

27.3. WAL Checkpoints

A checkpoint is the operation that lets the WAL be recycled. It declares that everything logged so far is now safely in the data file, so the log can be thrown away. wal_checkpoint() performs three steps under the WAL lock.

  1. Archive. If the active WAL holds any records, they are appended to evosql.wal.archive before being discarded. This archive is what powers point-in-time recovery (§27.5). Without it, checkpointing would permanently erase history.
  2. Advance. checkpoint_lsn is moved up to the current write position.
  3. Truncate. The active evosql.wal is truncated back to its 16-byte header, then the header is rewritten and fsync'd.

Any distributed-transaction PREPARE records that are still live are re-logged into the freshly truncated WAL so prepared transactions survive the checkpoint (see §27.7).

The caller is responsible for flushing dirty pages to the data file before calling wal_checkpoint(). The WAL layer itself does not flush the buffer pool. The only place this contract is honored is wal_shutdown(). It runs after pgm_shutdown() has flushed all pages.

When does a checkpoint actually happen? In the current engine, a WAL checkpoint runs only at clean shutdown. When the server receives SIGTERM/SIGINT and drains, it flushes the buffer pool and then calls wal_shutdown(). This function checkpoints and closes the log. There is no periodic background checkpointer, no time- or size-triggered checkpoint, and no SQL command to force one. As a consequence, evosql.wal grows for the entire lifetime of a running process and is reclaimed only on a graceful stop. Plan log-disk headroom accordingly for long-lived servers, and prefer clean shutdowns over SIGKILL.

Contrast with PostgreSQL. PostgreSQL has a dedicated checkpointer process driven by checkpoint_timeout and max_wal_size, plus a manual CHECKPOINT SQL command. EvolutionDB has none of these tuning knobs. Its checkpoint is an implicit, shutdown-time event.

27.4. Crash Recovery

Recovery is automatic and happens during pgm_init() when the data file is opened. EvolutionDB supports transparent data encryption (TDE). Page 0 (the file header) is always plaintext but data pages are encrypted. Because of this, replay is split into two passes.

Pass 1, page 0 only (wal_replay_page0(), inside wal_init())
Replays only logged images of page 0. Page 0 is the file header and is never encrypted. The page manager then re-reads the recovered header. If the database is encrypted, it initializes TDE from the header. This must come first because the keys needed to decrypt data pages live in the header.
Pass 2, data pages (wal_replay_remaining() → wal_replay_data())
Runs after TDE is live. Replays every non-page-0 record, encrypting each page on the way to disk when TDE is enabled. On completion it performs a checkpoint (advancing checkpoint_lsn and truncating the WAL).

Within each pass the replay loop reads records sequentially from just past the header to end-of-file. For every record it does the following.

  1. Reads the record header and validates page_len is in range. If it is out of range, replay stops. This is how a torn tail record is detected.
  2. Recomputes the CRC-32 over header+payload and compares it to the stored CRC. On mismatch, replay stops.
  3. Skips any record whose lsn ≤ checkpoint_lsn (already on disk).
  4. pwrite()s the page image to page_no × 4096 in the data file, encrypting first if TDE is on and the page is not page 0.

A clean start (empty WAL) recovers zero pages and proceeds normally. When recovery does apply pages, the server logs the count.

[WAL] Initialized (checkpoint_lsn=0, next_lsn=1)
[WAL] Pass 2: replayed 12 data page(s)
[WAL] Crash recovery: 12 page(s) restored

Recovery is redo-only and idempotent. There is no undo phase and no concept of replaying a transaction log up to the last commit. Replay simply re-applies page images that were logged but might not have landed in the data file. Because uncommitted work is never flushed (its WAL records are never fsync'd), there is nothing to roll back. Recovery also cannot be made to stop just before a bad transaction. For selective restore you need the time-travel archive below.

27.5. Point-in-Time / Time-Travel Recovery

Every checkpoint appends the records it is about to discard to evosql.wal.archive. Each record carries a microsecond timestamp. The archive can be replayed up to a chosen instant, restoring the database to how it looked at that time. This is implemented by wal_restore_to_timestamp().

The procedure restores a base snapshot taken before the target time, then replays archived records in order, applying each page image (TDE-encrypting as needed) and stopping at the first record whose timestamp exceeds the target.

# Conceptual workflow (engine-internal; see the caveat below)
# 1. Put a base copy of evosql.db in place - taken before the target time.
# 2. Replay evosql.wal.archive up to target epoch-microseconds.
#    Records with timestamp <= target are applied; the first one past it halts replay.

wal_archive_range() reports the timestamp span available in the archive. It returns the first and last record timestamps. A recovery tool can use this to confirm that the requested target falls inside the retained window before attempting a restore.

Limitation: no SQL/operator surface yet. wal_restore_to_timestamp() and wal_archive_range() are implemented engine functions, but in this release nothing calls them. There is no SQL statement, server flag, or CLI command that exposes point-in-time recovery to an operator. The plumbing (timestamped records, the growing evosql.wal.archive) is in place and correct, but driving a restore requires invoking the API directly. Treat PITR as a building block, not a turnkey feature. Also note the archive only ever grows. The RETENTION clause you may see on checkpoint stores does not prune it, and there is no archive-cleanup job.

Contrast with PostgreSQL. PostgreSQL drives PITR through recovery_target_time/recovery_target_lsn in the recovery configuration plus a restore_command that fetches archived segments. It can also recover to a transaction id or named restore point. EvolutionDB's archive is timestamp-keyed only, replay is whole-page, and the restore is invoked programmatically rather than through a config-driven recovery mode.

27.6. Base Backups

A base backup is a consistent physical copy of the database. It is produced by a one-shot server mode. Starting the server with --base-backup PATH initializes the engine, clones the data directory to PATH, and exits without binding any listener sockets. It never serves clients (see Server Configuration §23.7). It is the supported way to bootstrap a fresh replica from a primary.

# Clone a consistent copy into a directory and exit (0 = success)
evosql-server --base-backup /backups/evosql-2026-06-12

Internally repl_create_backup() first pins a consistent point. It flushes the page manager and buffer pool, then fsyncs the WAL, then records the current LSN as the backup's start LSN. It then writes one of two layouts depending on PATH.

Directory mode (PATH is or becomes a directory)
Produces a small bundle:
  • evosql.db, a byte-for-byte copy of the data file.
  • evosql.wal, a copy of the live WAL when present (best-effort; if it cannot be found the backup still succeeds).
  • BACKUP_LABEL, a tiny text manifest: start_lsn=…, timestamp=… (epoch seconds), and format=v1. A replica uses start_lsn to know where to begin streaming WAL to catch up.
Single-file mode (PATH is a plain file)
A legacy path that copies just evosql.db to that file, no WAL, no label.
# On success the server logs the bundle, its LSN, and the data size:
[REPL] Base backup bundle created at /backups/evosql-2026-06-12 (lsn=437, 8388608 data bytes)

Backup is physical, file-level, and offline-to-clients. EvolutionDB has no pg_dump-style logical/SQL backup and no streaming pg_basebackup protocol. The backup mechanism is this one-shot copy. The simplest manual alternative is to stop the server cleanly (which checkpoints and truncates the WAL) and copy evosql.db. If you want point-in-time material, also copy evosql.wal.archive using ordinary filesystem tools. Copying evosql.db from a running server without using --base-backup is unsafe, because there is no externally-triggerable checkpoint to quiesce it.

27.7. WAL & Distributed Transactions

The WAL is also where two-phase (XA) commit records are persisted so prepared transactions survive a crash. These are not page images. They reuse the record format with reserved sentinel page_no values and a small payload instead of a 4 KB page.

Recordpage_no sentinelPayload
XA PREPARE0xFFFFFFFExa_xid + the internal MVCC transaction id
XA COMMIT0xFFFFFFFDxa_xid
XA ROLLBACK0xFFFFFFFCxa_xid

During replay, records with a page_no at or above 0xFFFFFFFC are recognized as XA records and are not written to the data file. They are counted as replayed but handled in memory by the XA subsystem after recovery, reconstructing the set of prepared-but-unresolved transactions. As noted in §27.3, live PREPARE records are also re-logged across a checkpoint so they are not lost when the active WAL is truncated.

Notes & limitations

  • Whole-page log. The WAL stores full 4 KB page images, not logical deltas. Recovery is redo-only and idempotent. There is no undo log in the WAL and no transaction-level replay target.
  • Checkpoint only at clean shutdown. No background, timed, or size-triggered checkpointer exists, and there is no SQL command to force one. The active WAL grows for the life of the process and is reclaimed only on graceful stop. Avoid hard kills.
  • Durability is per-commit. The WAL fsync is batched to commit time. Work in an open transaction is not durable until COMMIT.
  • PITR has no operator surface. Time-travel restore is implemented but unexposed. No statement, flag, or tool invokes it in this release. The archive is also never pruned.
  • Backups are physical. Use --base-backup on a running server, or copy files after a clean shutdown. There is no logical (pg_dump-style) export and no streaming basebackup protocol.
  • TDE-aware replay. Recovery and restore re-encrypt data pages on write when the database is encrypted. Page 0 is always handled in plaintext first so keys can be loaded.
