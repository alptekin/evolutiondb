26. Encryption (TDE)

EvolutionDB transparently encrypts every data page on disk with AES-256, protecting the database file at rest while leaving page contents in cleartext inside the running server's buffer pool.

Transparent Data Encryption (TDE) operates entirely below the SQL layer. Tables, indexes, the system catalog, and the write-ahead log are all stored as 4 KB pages; when TDE is active, those pages are encrypted on their way to disk and decrypted on their way back, with no change to the SQL you write or the rows you read. The only artifact visible from SQL is that the on-disk file (evosql.db) and its WAL are unreadable without the passphrase.

This chapter documents exactly what EvolutionDB implements: a two-layer key hierarchy (MEK/DEK), AES-256 page encryption in CTR mode, key wrapping with AES-256-GCM, and passphrase-derived key material via PBKDF2-SHA256. Where it helps, it contrasts EvolutionDB's choices with PostgreSQL and the broader TDE landscape: honestly, without claiming features that do not exist.

Build requirement. TDE is a compile-time feature guarded by the EVO_ENCRYPTION macro. The server's TLS=1 build sets it automatically (it pulls in OpenSSL's EVP API for AES). A server built without EVO_ENCRYPTION compiles the entire crypto path out to no-op stubs, the database is always plaintext and the encryption controls in this chapter have no effect.

26.1. Overview

EvolutionDB's TDE is page-level and whole-database. There is no per-table, per-column, or per-tablespace encryption: when a database is created with a passphrase set, every page except page 0 is encrypted, and that decision is permanent for the life of the file. This mirrors the granularity of cluster-wide TDE designs and differs from column-level encryption such as PostgreSQL's pgcrypto, which encrypts individual values inside otherwise-plaintext pages.

What is encrypted
All pages numbered 1 and above: heap pages, B+ tree index pages, slotted-page records, the system catalog, the free list, and WAL page images written during replay.
What is not encrypted
Page 0 (the FileHeader) is always plaintext. It holds the file magic, version, page size, catalog roots, and the encryption metadata itself (salt, wrapped DEK, IV prefix) needed to bootstrap decryption.
Where plaintext lives
In the buffer pool. Pages are held decrypted in memory; encryption happens only as a page is flushed to disk, and decryption only as it is read from disk. The query engine never sees ciphertext.
Threat model
Protection at rest. TDE defends against theft of the database file, the WAL, or the physical disk. It does not protect against an attacker who can read the server's memory or who holds the passphrase.

Because the buffer pool stores plaintext and the engine operates on plaintext, TDE is invisible to queries and adds no semantic restrictions. A SELECT over an encrypted database is byte-for-byte identical to one over a plaintext database.

26.2. Key hierarchy: MEK and DEK

EvolutionDB uses a two-layer key architecture in the style of Oracle TDE:

KeySizeDerived / generatedStored where
MEK, Master Encryption Key256 bitsDerived from the passphrase + a random 16-byte salt via PBKDF2-SHA256Never stored. Recomputed in memory at startup; wiped immediately after use.
DEK, Data Encryption Key256 bitsGenerated once from the OS CSPRNG when the database is first createdStored wrapped (encrypted by the MEK) in the plaintext FileHeader on page 0.

The DEK is the key that actually encrypts pages. The MEK exists only to protect the DEK. This indirection is what makes rekeying cheap: changing the passphrase changes the MEK, which re-wraps the same DEK, so the gigabytes of already-encrypted pages never need rewriting.

The passphrase is supplied through the EVOSQL_ENCRYPTION_KEY environment variable. It is read with getenv() at startup and is never persisted. If the variable is unset or empty, encryption is not initialized.

# Key flow at a glance
EVOSQL_ENCRYPTION_KEY  ──PBKDF2-SHA256(salt,100000)──▶  MEK (256-bit)
MEK  ──AES-256-GCM unwrap──▶  DEK (256-bit, from FileHeader)
DEK  ──AES-256-CTR──▶  page ciphertext  (pages 1..N)

26.2.1. MEK derivation (PBKDF2-SHA256)

The MEK is derived by stretching the passphrase with PBKDF2-SHA256. EvolutionDB ships a pure-C implementation of SHA-256, HMAC-SHA256, and PBKDF2 (no external dependency for the KDF itself), conforming to FIPS 180-4, RFC 2104, and RFC 2898.

PRF
HMAC-SHA256
Iterations
PBKDF2_ITERATIONS = 100000 (fixed, compiled-in)
Salt
16 random bytes, generated per database, stored in the FileHeader as encryption_salt[16]
Derived key length
32 bytes (256 bits)

The same PBKDF2 implementation also backs user-password hashing (see Authentication), where hashes are stored in the format pbkdf2:100000:<hex-salt>:<hex-hash>. For TDE the derived bytes are used directly as the MEK and are never written to disk.

Honest comparison. 100,000 PBKDF2-SHA256 iterations is a fixed, moderate work factor. It is below the project's own user-auth guidance (the security standard calls for 600,000+ iterations for password storage) and well below modern memory-hard KDFs such as Argon2 or scrypt, which PBKDF2 cannot match against GPU/ASIC attackers. The practical mitigation is to use a long, high-entropy passphrase: against a strong passphrase the KDF iteration count is not the limiting factor.

26.2.2. DEK wrapping (AES-256-GCM)

The 32-byte DEK is wrapped (encrypted) under the MEK with AES-256-GCM, producing a 48-byte blob: 32 bytes of ciphertext followed by a 16-byte GCM authentication tag. This blob is stored in the FileHeader as wrapped_dek[48].

The GCM tag does double duty as a passphrase verifier. On startup the server derives the MEK from the supplied passphrase and attempts to unwrap the DEK; if the passphrase is wrong, the derived MEK is wrong, and GCM tag verification fails. The server treats that failure as a fatal "wrong encryption key" error rather than silently producing garbage:

FATAL: Wrong encryption key for this database.
The EVOSQL_ENCRYPTION_KEY does not match.

The wrapping IV is the fixed 12-byte string EVO_WRAP_KEY. Reusing a fixed IV with GCM is normally dangerous, but it is safe here because each database has a unique random salt, which produces a unique MEK, so the (key, IV) pair is never reused across databases with the same DEK.

26.3. Page encryption (AES-256-CTR)

Pages are encrypted with AES-256 in CTR mode using the DEK. CTR is a stream-cipher mode, chosen deliberately for two properties that matter to a page store:

Zero storage overhead
CTR ciphertext is the same length as the plaintext. A 4096-byte page encrypts to exactly 4096 bytes, so the on-disk layout, page numbering, and offsets are identical whether or not TDE is enabled. There is no per-page authentication tag.
Symmetric operation
In CTR mode encryption and decryption are the same operation. EvolutionDB uses a single routine for both directions.

26.3.1. Per-page IV construction

Each page gets a distinct 16-byte CTR IV built from a per-database random prefix and the page number:

IV (16 bytes) = iv_prefix[8] || page_no_LE[4] || zeros[4]
               └ random, per-DB ┘  └ little-endian ┘ └ counter ┘

The 8-byte iv_prefix is generated once with the OS CSPRNG when the database is created and stored in the FileHeader as page_iv_prefix[8]. Critically, the prefix never changes, even when the passphrase is rotated. Because the page number is part of the IV, no two pages share an IV, which is the requirement CTR mode places on the caller.

Limitation: CTR offers confidentiality, not integrity. Unlike GCM (used for the DEK wrap), CTR has no authentication tag, so EvolutionDB does not detect tampering or bit-rot of encrypted pages, a flipped ciphertext bit silently flips the corresponding plaintext bit. TDE here defends against reading the file, not against an adversary who can modify it. This is the same trade-off most zero-overhead page-level TDE schemes make; use filesystem/volume integrity (or checksums at a higher layer) if tamper-detection is required.

26.3.2. Page 0 stays plaintext

Page 0, the FileHeader, is never encrypted. Both the encrypt and decrypt entry points short-circuit when page_no == 0. This is necessary because page 0 holds the bootstrap metadata (magic, salt, wrapped DEK, IV prefix) that the server must read before it can derive any key. The header therefore exposes the fact that a database is encrypted, the salt, the wrapped (encrypted) DEK, and the IV prefix, but none of these reveal the DEK without the passphrase.

EvolutionDB does not encrypt the page_no == 0 region for the same architectural reason PostgreSQL's at-rest encryption proposals keep certain control data accessible: you cannot decrypt the thing that tells you how to decrypt.

26.3.3. Where encryption happens in the I/O path

All page I/O flows through the Clock-Sweep buffer pool (see Buffer Pool). Encryption and decryption are applied at exactly the disk boundary:

PathAction
Buffer pool flush (dirty page → disk)Page is copied into a scratch buffer, encrypted in place, then written. The in-memory page stays plaintext.
Buffer pool fault (disk → buffer)Page is read, then decrypted in place if encryption is enabled and page_no > 0.
WAL replay (recovery)Page images written to the data file during replay are encrypted before pwrite, page 0 excepted.
Replication shippingPages are encrypted on the same code path before being sent downstream.

Every one of these guards on both pcrypt_is_enabled() and page_no > 0, so a plaintext database and page 0 always bypass the cipher.

26.4. Enabling encryption

TDE is decided at database creation time and is controlled entirely by the EVOSQL_ENCRYPTION_KEY environment variable. There is no CREATE DATABASE ... ENCRYPTED clause and no SQL statement that turns encryption on for an existing plaintext file.

26.4.1. A new (empty) database

When the page manager creates a fresh evosql.db and EVOSQL_ENCRYPTION_KEY is set, it initializes encryption: it generates a random DEK, a random salt, and a random IV prefix; derives the MEK; wraps the DEK; and records all of that in the FileHeader with encryption_enabled = 1.

# docker-compose.yml - set a strong custom key BEFORE first start
services:
  evosql:
    environment:
      EVOSQL_ENCRYPTION_KEY: "<a long, random, high-entropy passphrase>"
# Or locally, when the .db file does not yet exist:
export EVOSQL_ENCRYPTION_KEY="<a long, random, high-entropy passphrase>"
./adaptor/evosql-server

On success the server logs [TDE] Encryption initialized for new database. From that point the file is encrypted permanently, there is no in-place "decrypt back to plaintext" operation.

26.4.2. An existing encrypted database

For an existing file, the server reads encryption_enabled from page 0 and, if set, derives the MEK from the supplied passphrase and unwraps the DEK. The startup matrix is:

File encryption_enabledEVOSQL_ENCRYPTION_KEYResult
1 (encrypted)correct passphraseDatabase unlocked, logs [TDE] Encryption key verified, database unlocked
1 (encrypted)wrong passphraseFATAL, GCM tag verification fails; server refuses to start
1 (encrypted)unset / emptyFATAL, "Database is encrypted but EVOSQL_ENCRYPTION_KEY.. not set"

Default-key warning. The shipped docker-compose.yml sets a built-in fallback passphrase so the demo "just works." That key is publicly known. When the server detects it, it prints a prominent warning to stderr. Always override EVOSQL_ENCRYPTION_KEY with your own secret before creating a production database, a database created under the default key is no more protected than a plaintext one.

26.5. Rekeying (passphrase rotation)

Because the passphrase only protects the MEK, and the MEK only wraps the DEK, rotating the passphrase is an O(1) metadata operation: a new salt is generated, a new MEK is derived from the new passphrase, and the same DEK is re-wrapped under the new MEK. The IV prefix is left unchanged, so no data pages are re-encrypted. Only encryption_salt and wrapped_dek in the FileHeader change.

Implementation status, be precise. The rekey primitive (pcrypt_rekey) exists in the engine, but in EvolutionDB 3.0 it is not exposed through SQL. There is no ALTER SYSTEM ... REKEY statement, and no command-level surface in the parser or protocol layer drives it. Passphrase rotation is therefore an internal capability, not a user-facing operation in this release. This is a deliberately honest limitation rather than an advertised feature.

26.6. Key material in memory

Sensitive buffers are scrubbed after use rather than left on the stack or heap. The MEK is wiped immediately after it wraps or unwraps the DEK; SHA-256 contexts, HMAC pads, and PBKDF2 intermediates are wiped at the end of each operation; and on shutdown the DEK and IV prefix are wiped from the module's static state.

Scrubbing uses evo_secure_wipe(), which is hardened against the compiler optimizing away a "dead" memset (CWE-14). It prefers platform primitives where available, SecureZeroMemory() on Windows, memset_s() when the C11 Annex K bounds-checking interface is present, and otherwise falls back to a volatile function-pointer indirection that the optimizer cannot prove is side-effect-free.

Password verification additionally uses a constant-time comparison to avoid leaking information through timing. Note these protections cover the process: while the server runs, the DEK and decrypted pages are necessarily resident in memory, which is the boundary of the at-rest threat model.

26.7. Randomness source

All key material: the DEK, the MEK salt, and the IV prefix, comes from the operating system CSPRNG via crypto_random_bytes():

Linux / macOS
Reads from /dev/urandom
Windows
CryptGenRandom via CryptAcquireContext

A failure to obtain randomness aborts encryption initialization rather than falling back to a weak source.

26.9. On-disk metadata layout

The encryption metadata lives in the plaintext FileHeader (page 0), 73 bytes total:

encryption_enabled (1 byte)
0 = plaintext database, 1 = AES-256 TDE active
encryption_salt[16]
PBKDF2 salt for MEK derivation
wrapped_dek[48]
DEK encrypted by the MEK: 32 bytes ciphertext + 16-byte GCM tag
page_iv_prefix[8]
Fixed CTR IV prefix; unchanged across rekeys

None of these fields reveals the DEK or any plaintext: without the correct passphrase the wrapped DEK cannot be unwrapped (GCM verification fails) and the pages cannot be decrypted.

26.10. Comparison with PostgreSQL and the SQL standard

For readers coming from PostgreSQL, the honest mapping is:

Cluster-wide TDE
Core PostgreSQL has no built-in transparent cluster encryption (it has been a long-standing community proposal); at-rest encryption is typically delegated to filesystem/volume encryption or to forks/extensions. EvolutionDB ships whole-database page TDE in the engine itself.
Column-level encryption
PostgreSQL's pgcrypto encrypts individual column values, leaving pages otherwise readable and requiring SQL changes. EvolutionDB has no column-level encryption, it encrypts whole pages and is invisible to SQL.
Key management
EvolutionDB's MEK/DEK split resembles Oracle TDE's master-key / table-key model. The passphrase comes from an environment variable; there is no external KMS, HSM, or keystore integration.
SQL standard
The SQL standard does not define transparent storage encryption. TDE here is an implementation-level feature with no standard syntax, which is why it is configured by environment variable rather than by DDL.

Notes & limitations.

  • Compile-time gated. Requires a build with EVO_ENCRYPTION (the TLS=1 build). Otherwise the crypto path is a no-op and the database is plaintext.
  • Permanent at creation. Encryption is set when the database file is first created. You cannot encrypt an existing plaintext file in place, nor decrypt an encrypted file back to plaintext, through any supported operation.
  • No SQL rekey. The rekey primitive exists internally but has no SQL/command surface in 3.0; passphrase rotation is not a user-facing feature.
  • Confidentiality only. Page encryption uses CTR with no per-page authentication tag, there is no tamper detection on data pages.
  • KDF work factor. PBKDF2 is fixed at 100,000 SHA-256 iterations; rely on a strong passphrase, not the iteration count.
  • Default key is public. The shipped compose default passphrase is well known. Override it before creating any database you care about.
  • Memory is the boundary. While the server runs, the DEK and decrypted pages live in memory; TDE protects the file at rest, not a live-memory attacker who already holds the key.
