23. Server Configuration

How to start and tune evosql-server: the listener ports it opens, the command-line flags and environment variables it reads at boot, and the limits that govern how many clients it will serve at once.

EvolutionDB has a single server executable, evosql-server (built from adaptor/). It is a dual-protocol server: one process spawns two TCP listeners, the PostgreSQL wire protocol and the native EVO text protocol, backed by one shared SQL engine and one shared data file (evosql.db). All configuration is supplied at process start through command-line flags and environment variables. Unlike PostgreSQL, there is no postgresql.conf file and no SET-able GUC reload: the knobs documented here are read once in main() and apply for the lifetime of the process.

Precedence. For every setting that has both a flag and an environment variable, the server follows a consistent rule: environment variables are read first to seed defaults, then command-line flags override them. The flag always wins.

23.1. Invoking the Server

The synopsis below lists every flag the argument parser in adaptor/main.c recognizes.

evosql-server
    [--pg-port PORT]
    [--evo-port PORT]
    [--buffer-pool-size SIZE]
    [--replication-port PORT]
    [--cdc-port PORT]
    [--replica HOST:PORT]
    [--cluster node1:port,node2:port,...]
    [--node-id N]
    [--dist-port PORT]
    [--role primary|replica|witness]
    [--base-backup PATH]
    [PORT]

A bare positional number is accepted for backward compatibility and is interpreted as the PG port:

# These two are equivalent
evosql-server --pg-port 5544
evosql-server 5544

The argument parser is positional-tolerant but not strict: an unrecognized flag is silently ignored (there is no usage/error exit for unknown options). Flags that require a value are only consumed when a following argument exists.

23.1.1. Startup Output

On a normal boot the server prints its version, the buffer-pool size, TLS status, the two protocol ports, the resolved maximum connection count, and an ASCII banner, then begins accepting connections. A typical session looks like:

EvolutionDB v3.0.0
[BufferPool] 32768 pages (128 MB)
[TLS] No EVOSQL_TLS_CERT / EVOSQL_TLS_KEY set - TLS disabled
==============================================
  EvoSQL Adaptor Server (dual-protocol)
  PG  protocol on port 5433
  EVO protocol on port 9967
  TLS: disabled
  Max connections: 100
==============================================
[EVO] Listening on port 9967
[PG] Listening on port 5433

The PG listener runs on the main thread and blocks forever; the EVO listener (and, when configured, the distributed-query listener) runs in a background thread.

23.2. Ports and Listeners

The server can open up to four kinds of TCP listeners. Only the two protocol listeners are opened unconditionally; the rest are opened only when their feature is configured.

ListenerDefault portHow to setOpened when
PG wire protocol5433--pg-port (or bare positional number)Always
EVO text protocol9967--evo-portAlways
Replication / WAL sender9968 *--replication-port / EVOSQL_REPLICATION_PORTPort > 0
CDC streaming (JSON Lines), (off)--cdc-port / EVOSQL_CDC_PORTPort > 0
Distributed query engine9969--dist-port--cluster + --node-id given

* 9968 (REPL_DEFAULT_PORT) is the default upstream port assumed for a replica when only a host is supplied; the WAL sender itself only starts when --replication-port/EVOSQL_REPLICATION_PORT is explicitly > 0.

The two protocol ports are the ones you will configure most often. The default 5433 deliberately avoids PostgreSQL's 5432 so the two can run side by side on one host. Each listener binds with a fixed listen() backlog of 5.

# Run two engines on one box without colliding
evosql-server --pg-port 5433 --evo-port 9967
evosql-server --pg-port 5443 --evo-port 9977

No bind-address control. EvolutionDB has no listen_addresses equivalent. The bind address is fixed in the network layer; you cannot, via configuration, restrict the server to localhost or pick a specific interface. Use OS firewall rules or container networking to limit reach.

23.3. Connection Limits and the Worker Pool

EvolutionDB enforces a hard ceiling on concurrent client connections. The accept loop in server_listen() checks the live count under a mutex before dispatching each connection; if the limit is already reached, the new socket is closed immediately and a rejection line is logged.

[PG] Connection rejected: limit 100 reached
max_connections
Maximum number of concurrent client connections across both protocols combined. The compiled-in default is 100. The active/limit pair is printed on every connect and disconnect (active: 7/100). There is no startup flag for this value; it is exposed through the catalog accessors get_max_connections() / set_max_connections() and reported by the server banner as Max connections.
EVOSQL_POOL_SIZE (env)
Size of the pre-spawned worker thread pool that services accepted sockets. Default 20. Setting it to 0 disables the pool and falls back to the legacy one-thread-per-connection path (useful under Valgrind / ThreadSanitizer). The pool's internal task queue capacity tracks max_connections so a connect storm cannot exhaust memory.
MAX_SESSIONS (compile-time)
The session registry, the array that backs session introspection and statement cancellation, has a fixed capacity of 256 slots. When all slots are in use a session still receives a valid ID but is not tracked in the registry.

Each connection runs in its own thread (pooled or spawned). Query execution across all threads is serialized through a parser read/write lock plus a DML mutex (see Concurrency & Locking), so the engine remains single-writer even though the network layer is fully multi-threaded.

Contrast with PostgreSQL. PostgreSQL's max_connections is a reloadable GUC and connections each get a backend process; EvolutionDB's limit is a fixed default of 100, set in code, and connections are threads sharing one address space. There is no per-role or per-database connection limit, and no reserved superuser slots.

23.4. Buffer Pool Size

The Clock-Sweep buffer pool (see Storage & the Buffer Pool) caches 4 KB pages. Its size is the single most impactful memory tuning knob and is the only one parsed with unit suffixes.

--buffer-pool-size SIZE / EVOSQL_BUFFER_POOL_SIZE
Sets the number of 4 KB pages held in the buffer pool. The default is 32768 pages (128 MB) (BP_DEFAULT_PAGES). The flag takes precedence; if the flag is absent the environment variable is consulted.

The size string accepts a unit suffix or a raw page count, parsed as follows:

InputInterpreted asResulting pages
1G / 1gGigabytesval × 1024 × 1024 / 4 = 262144
128M / 128mMegabytesval × 1024 / 4 = 32768
512K / 512kKilobytesval / 4 = 128
32768Raw page count32768
 
# All three select a 256 MB pool
evosql-server --buffer-pool-size 256M
EVOSQL_BUFFER_POOL_SIZE=256M evosql-server
evosql-server --buffer-pool-size 65536   # raw pages

A value that parses to zero or less is treated as "unset" and the 128 MB default is used. The resolved size is echoed at startup as a [BufferPool] line in pages and MB (or KB for very small pools).

23.5. Authentication Bootstrap

The initial superuser is provisioned at first boot from two environment variables read by the user-management layer (evolution/db/UserMgmt.c):

EVOSQL_USER_NAME
Name of the bootstrap superuser. Default admin.
EVOSQL_PASSWORD
Password for the bootstrap superuser. If unset or empty, the server generates a random 16-character password and prints it once at startup. Capture it from the logs, because it is not stored in plaintext or shown again. The stored credential uses PBKDF2-SHA256 (see CREATE USER and Security).
# Deterministic credentials for local development
EVOSQL_USER_NAME=admin EVOSQL_PASSWORD=admin evosql-server

Both protocols force cleartext-password auth. The PG wire handler always issues a CleartextPassword challenge, the actual hashing happens server-side. Run behind TLS (next section) so the cleartext password is never exposed on the wire.

23.6. Transport Security (TLS)

TLS is opt-in and is enabled purely through the environment. If both of the following point to readable PEM files, the server initializes an OpenSSL context for the PG-wire transport at startup; otherwise it prints TLS disabled and continues in cleartext. (TLS support also requires a build with TLS=1.)

EVOSQL_TLS_CERT
Path to the server certificate (PEM).
EVOSQL_TLS_KEY
Path to the matching private key (PEM).
EVOSQL_TLS_CERT=/etc/evosql/server.crt \
EVOSQL_TLS_KEY=/etc/evosql/server.key \
evosql-server

A failed TLS init is non-fatal: the server logs [TLS] Failed to initialise - continuing without TLS and runs in cleartext rather than refusing to start. The replication transport can reuse the same certificate/key pair when EVOSQL_REPLICATION_TLS is truthy (see below).

23.7. Replication, CDC and Cluster Boot Settings

Replication topology can be primed entirely from the environment (handy for containers), then refined with flags. The environment is read first; matching flags override it. Full operational detail lives in High Availability & Replication, this section only documents the boot-time configuration surface.

EVOSQL_REPLICATION_ROLE / --role
One of primary, replica, or witness. Pins the role reported by SHOW REPLICATION STATUS before any peer connects. An unrecognized value logs a warning and leaves the role unset.
EVOSQL_PRIMARY_HOST / EVOSQL_PRIMARY_PORT
Upstream the server should follow when role=replica. When a host is given without a port, the replica targets REPL_DEFAULT_PORT (9968). Equivalent to the flag --replica HOST:PORT.
EVOSQL_REPLICATION_PORT / --replication-port
Port on which a primary runs its WAL sender. The sender starts only when this resolves to a value > 0.
EVOSQL_REPLICATION_USER / EVOSQL_REPLICATION_PASSWORD
Credentials pre-registered for the replication auth handshake. Both must be set together to take effect.
EVOSQL_REPLICATION_TLS
Truthy (1, t, T, y, Y) enables TLS on the replication transport using the same cert/key as the PG-wire context.
--cdc-port / EVOSQL_CDC_PORT
Starts a Change-Data-Capture streaming listener that pushes WAL-decoded changes as JSON Lines. Off unless > 0.
--cluster + --node-id
A comma-separated host:port node list plus this node's zero-based index. Supplying both starts the Raft consensus module and the distributed query engine; per-node PG and dist ports are derived by offsetting from this node's ports.
--base-backup PATH
One-shot mode: clones the data directory to PATH and exits without binding any listener sockets. Used to bootstrap a fresh replica from a primary. Exit code 0 on success, 1 on failure.
# Primary with WAL sender on 9968
evosql-server --role primary --replication-port 9968

# Replica following the primary above (env-driven)
EVOSQL_REPLICATION_ROLE=replica \
EVOSQL_PRIMARY_HOST=10.0.0.1 EVOSQL_PRIMARY_PORT=9968 \
evosql-server

23.8. Temporal Retention

EVOSQL_SYSTEM_TIME_RETENTION_DAYS
Seeds the retention window for system-versioned (temporal) table history. Default 7 days; 0 disables retention pruning. Any non-negative integer is accepted at boot. Unlike most settings here this one can be changed at runtime with SET SYSTEM_TIME_RETENTION = N, which overrides the boot value for the running process.

Notes & limitations

  • No config file. There is no postgresql.conf/evosql.conf. All configuration is flags + environment, resolved once at process start.
  • Most settings are not runtime-reloadable. Changing ports, buffer-pool size, pool size, TLS, or the connection limit requires a restart. The only documented exception is SYSTEM_TIME_RETENTION, changeable via SET.
  • Connection limit is compiled-in (100). It is not exposed as a startup flag; over-limit connections are dropped at accept time with a logged rejection, not a protocol-level error message.
  • No bind-address / interface selection. Restrict exposure with firewall or container networking.
  • Unknown flags are ignored silently: there is no usage banner or non-zero exit for a typo'd option, so double-check spelling.
  • Graceful shutdown on SIGTERM/SIGINT drains the worker pool, checkpoints the WAL, flushes the buffer pool, and closes the data file before exiting (POSIX only).
