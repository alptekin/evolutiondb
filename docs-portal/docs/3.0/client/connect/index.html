30. Connecting (PG wire & EVO)

EvolutionDB accepts client connections on two protocols at once: the PostgreSQL frontend/backend protocol version 3.0 (default TCP port 5433) and the native line-oriented EVO text protocol (default TCP port 9967). Both authenticate against the same user catalog before any query is accepted.

A single evosql-server process spawns one listener thread per protocol. The PostgreSQL wire listener lets you connect with psql, JDBC/ODBC drivers, DBeaver, pgAdmin, and any PostgreSQL client library. The EVO listener is the channel used by evosql-cli and by the EvoSQL language drivers; it is simpler to parse and exposes server-side prepared statements and batch execution that the simple SQL path does not.

This chapter documents only what EvolutionDB implements. Where a familiar PostgreSQL feature is absent or behaves differently, the difference is called out explicitly rather than implied to work.

30.1. Listeners and ports

The server binds two ports at startup. The defaults and the flags that override them are:

ProtocolDefault portOverride flagTypical clients
PostgreSQL wire v3.05433--pg-port PORTpsql, JDBC, ODBC, DBeaver, pgAdmin, psycopg
EVO native text9967--evo-port PORTevosql-cli, EvoSQL drivers

Both listeners are started by the same process. The PG listener runs on the main thread and the EVO listener runs on a background thread; each accepted connection is handled in its own thread.

# Start the dual-protocol server with default ports
./adaptor/evosql-server

# Move the PG listener to 5434, leave EVO on 9967
./adaptor/evosql-server --pg-port 5434

# A bare numeric argument is accepted as the PG port (backward compat)
./adaptor/evosql-server 5433

Note. The standard PostgreSQL port is 5432. EvolutionDB defaults to 5433 so it can coexist with a real PostgreSQL instance on the same host. Point your client's port at 5433 (or whatever you set with --pg-port) — connecting to 5432 reaches PostgreSQL, not EvolutionDB.

30.2. Credentials

Both protocols verify the supplied username and password against the database user catalog via the same internal routine (AuthenticateUser). Passwords are stored using PBKDF2-SHA256; see CREATE USER and Authentication & users for how accounts are provisioned and hashed.

EVOSQL_USER_NAME
Bootstrap superuser name. Defaults to admin.
EVOSQL_PASSWORD
Bootstrap superuser password. If unset, a random password is generated and printed to the server log once at first start. Capture it from that line, or set this variable before starting the server to pin a known value.

The default test database is testdb; the test and example credentials throughout this manual are admin / admin against a server started with EVOSQL_PASSWORD=admin.

Note. The user value in the PG startup packet selects the account to authenticate. There is no anonymous or trust path: a connection that does not present a valid password is rejected before the query loop begins. Authentication failures are reported with SQLSTATE 28P01 on both protocols.

30.3. PostgreSQL wire protocol (port 5433)

EvolutionDB speaks the PostgreSQL frontend/backend protocol version 3.0 (protocol number 196608). A startup message announcing any other version is rejected. The handshake proceeds as follows.

30.3.1. Startup and negotiation

Before the regular startup packet, a client may send one of the special request codes. EvolutionDB handles them as:

RequestCodeServer response
SSLRequest80877103S then a TLS handshake if TLS is built in and available; otherwise N and the connection continues in plaintext. After negotiation the server re-reads the startup packet.
GSSENCRequest80877104Always declined with N; the startup packet is re-read. GSSAPI encryption is not supported.
CancelRequest80877102Looks up the target session by its PID + secret and sets that session's cancel flag, then closes this connection (per the PG spec). See Query cancellation.
StartupMessage196608Parses startup parameters, extracts user, and begins authentication.

30.3.2. Authentication

After a valid v3.0 StartupMessage, the server always issues an AuthenticationCleartextPassword request and expects a PasswordMessage ('p') in reply. The password is verified with AuthenticateUser; on success the server sends AuthenticationOk, on failure a FATAL error with SQLSTATE 28P01 (password authentication failed).

Note. The PG path uses cleartext password authentication only — MD5 and SCRAM-SHA-256 are not offered on the wire here (real PostgreSQL negotiates scram-sha-256 by default). Because the password crosses the wire in clear, enable TLS via SSLRequest so the credential is encrypted in transit. The native EVO protocol (below) does offer SCRAM and does not transmit the password.

The server then sends BackendKeyData — carrying the session-registry ID (used as the PID) and a randomly generated 32-bit cancel key — followed by ReadyForQuery. The transaction-status byte in ReadyForQuery is I (idle), T (in a transaction block), or E (failed transaction awaiting ROLLBACK).

30.3.4. Simple Query protocol

A Query message ('Q') may carry several statements separated by semicolons. The handler splits on semicolons while respecting single quotes, double quotes, SQL line comments (-- ...), and BEGIN ... END nesting (so a procedure body is not cut at an internal semicolon). One response is sent per statement, followed by a single ReadyForQuery. SELECT results are streamed row-by-row during the scan.

psql "host=127.0.0.1 port=5433 user=admin dbname=testdb sslmode=disable"

testdb=> CREATE TABLE t (id INT, name VARCHAR(50));
testdb=> INSERT INTO t VALUES (1, 'alice'), (2, 'bob');
testdb=> SELECT id, name FROM t WHERE id = 1;

30.3.5. Extended Query protocol

The Parse / Bind / Describe / Execute / Sync / Close sequence is supported, which is what JDBC and psycopg's auto-prepare rely on. Key behaviors specific to EvolutionDB:

  • Named statements are retained. A Parse stores the query text keyed by statement name (up to 64 named statements per connection). A later Bind+Execute that re-uses the name with no preceding Parse rebuilds the query from the retained template. Unnamed statements use the empty name and are freely replaced.
  • Parameter substitution is textual. Bound $1, $2, … placeholders are interpolated into the SQL string: NULL parameters become the literal NULL; all other values are single-quoted with embedded quotes doubled. Up to 64 parameters per Bind are handled.
  • Binary parameter formats are decoded for the common fixed widths — 4-byte int32, 2-byte int16, 8-byte int64, and 1-byte boolean — and converted to text. Other binary inputs are copied through as-is.
  • Describe of a portal/statement returns a RowDescription for a SELECT with columns, or NoData otherwise; Describe('S') first emits a ParameterDescription reporting zero parameter type OIDs.
  • Close('S', name) deallocates the named prepared statement and drops its retained template; Close('P', …) for a portal is acknowledged but retains nothing.

Note. Parameters are bound by string interpolation, not as separate typed values. Quoting and escaping are handled for you, but server-side typed parameter binding (and the type negotiation that implies) is not performed. RowDescription reports 0 parameter OIDs in ParameterDescription, so drivers must not depend on the server to declare parameter types.

30.3.6. COPY over the wire

Both COPY ... FROM STDIN and COPY ... TO STDOUT are implemented with the standard CopyIn/CopyOut subprotocol (CopyInResponse / CopyData / CopyDone / CopyFail). A failed CopyIn surfaces SQLSTATE 57014; an unexpected message during CopyIn yields 08P01. See COPY.

30.4. EVO native protocol (port 9967)

The EVO protocol is a newline-delimited text protocol designed for the EvoSQL CLI and drivers. It is request/response with an explicit READY marker after every command result.

30.4.1. Greeting, TLS, and authentication

The exchange opens with a greeting and may upgrade to TLS before any credential is sent:

  1. The client sends the literal line EVO. Anything else is rejected with ERR 08000.
  2. The server replies HELLO EvoSQL 1.0.
  3. If TLS is available the server sends STARTTLS. If the client answers STARTTLS the TLS handshake runs; if it answers anything else the session continues in plaintext (backward compatibility).
  4. The server sends AUTH_SCRAM to request SCRAM-SHA-256.

The client then chooses one of two auth flows:

SCRAM-SHA-256 (preferred)
Client sends SCRAM-CLIENT-FIRST <data>; server replies with the server-first message; client sends SCRAM-CLIENT-FINAL <data>; server verifies the proof and returns the server-final (ServerSignature) for mutual authentication. The password is never transmitted, so this is safe even over plaintext.
Legacy cleartext (backward compat)
Client sends AUTH <username> <password>. The password is verified directly. Prefer SCRAM, or wrap the session in TLS.

On success the server sends AUTH_OK and enters the command loop. Authentication errors use SQLSTATE 28P01 (bad credentials) or 28000 (unexpected auth message).

# Connect with the bundled CLI (EVO, port 9967)
./cli/evosql-cli                # defaults to localhost:9967
./cli/evosql-cli -W admin        # prompt for a password

30.4.2. Running SQL: the SQL command

A query is sent as a length header line followed by the SQL body line:

SQL <byte_length>
<sql text>

The length must be in the range 1..65000; out of range yields ERR 42000. A SELECT result is returned as a structured block:

RESULT
COLS <n>
COL <name> <type_oid> <type_modifier>     # one per column
ROW
FIELD <value>                            # or: FIELD NULL
...
END
TAG <command_tag>
READY

A non-SELECT statement returns OK then TAG <command_tag> (for example TAG INSERT 0 2), and errors return ERR <sqlstate> <message>. Every command — success or error — is terminated by a READY line. Sending QUIT ends the session with BYE.

30.4.3. Server-side prepared statements and batches

EVO exposes named prepared statements directly on the protocol — there is no SQL-level PREPARE required. Up to 32 statements may be held per connection. Placeholders are $1…$9+; the parameter count is derived from the highest placeholder seen.

CommandPurpose
PREPARE <name> <byte_len> + bodyStore a parameterized statement under name. Replies OK / TAG PREPARE.
EXECUTE <name> <n> + n param linesSubstitute the parameters and run once. Use the line \N for a SQL NULL; other values are single-quoted with internal quotes doubled.
EXECUTE_BATCH <name> <rows> <cols> + param linesRun the statement once per row. Replies BATCH_OK <total_affected> or BATCH_ERR <row_index> <sqlstate> <msg>.
EXECUTE_BATCH_BINARY <name> <rows> <cols> + binary payloadAs above, but each parameter is sent as a typed binary value. Same BATCH_OK / BATCH_ERR reply.
DEALLOCATE <name>Free a prepared statement. Replies OK / TAG DEALLOCATE, or ERR 26000 if not found.

The binary batch payload encodes each parameter as a one-byte type code followed by its big-endian value:

CodeTypePayload
NNULLnone — emitted as SQL NULL
iint44 bytes, big-endian
lint88 bytes, big-endian
dfloat88 bytes, big-endian IEEE-754
bbool1 byte (0 = false)
sstring4-byte big-endian length + UTF-8 bytes (no NUL), max 16 MB

Affected-row counts for batches are accumulated from the INSERT/UPDATE/DELETE command tags. A batch stops at the first failing row and reports its index in BATCH_ERR.

Note. Like the PG extended path, EVO prepared statements substitute parameters as quoted SQL text rather than binding typed values server-side. The binary batch path still decodes typed values, but converts them to text before substitution. There is a hard limit of 32 prepared statements per EVO connection (ERR 53000 when exceeded) versus 64 named statements on the PG path.

30.4.4. COPY over EVO

A COPY ... FROM STDIN issued through the SQL command makes the server reply COPY_READY; the client then streams one CSV line per row and ends with \. on its own line (the PostgreSQL convention). For COPY ... TO STDOUT the server sends COPY_READY, then one D:<line> per row, then a \. terminator. Both finish with OK / TAG COPY <n>. See COPY.

30.5. Session lifetime and limits

Sessions on both protocols share the same lifecycle handling:

  • Default isolation is READ COMMITTED. See SET TRANSACTION and MVCC & isolation.
  • Idle timeout: if a session idle timeout is configured, a connection that sends no data within the window is terminated — SQLSTATE 57P05 (terminating connection due to idle timeout on PG; idle timeout on EVO).
  • Auto-rollback on disconnect: if a client drops while a transaction is open, the server rolls it back, releases any SERIALIZABLE lock, and frees the undo log.
  • Temporary objects: session-scoped temporary tables and global-temp-table contents are dropped automatically when the session ends.
  • LISTEN/NOTIFY: asynchronous notifications are delivered between replies — as standard NotificationResponse on PG, and as a NOTIFY <sender_id> <channel> <len> framed line on EVO. See LISTEN / NOTIFY.

30.6. Query cancellation

On the PG protocol, BackendKeyData hands the client a PID (the session-registry ID) and a random 32-bit secret. To cancel an in-flight query the client opens a second connection and sends a CancelRequest (code 80877102) carrying that PID and secret. The server matches the session by PID + secret, sets its cancel flag, and immediately closes the cancel connection — exactly as the PostgreSQL spec prescribes.

Note. The cancel key is generated fresh per connection from a cryptographic random source (and forced non-zero), so a leaked key only affects the one session it was issued for. The EVO protocol does not expose an out-of-band cancel channel; cancellation there relies on the idle timeout and connection teardown.
