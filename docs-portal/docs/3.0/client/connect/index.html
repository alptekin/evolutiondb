30. Connecting (PG wire & EVO)

EvolutionDB accepts client connections on two protocols at the same time. The first is the PostgreSQL frontend/backend protocol version 3.0 on default TCP port 5433. The second is the native line-oriented EVO text protocol on default TCP port 9967. Both check the same user catalog before any query is accepted.

A single evosql-server process starts one listener thread for each protocol. The PostgreSQL wire listener lets you connect with psql, JDBC/ODBC drivers, DBeaver, pgAdmin, and any PostgreSQL client library. The EVO listener is used by evosql-cli and by the EvoSQL language drivers. It is easier to parse. It also offers server-side prepared statements and batch execution that the simple SQL path does not have.

This chapter shows only what EvolutionDB actually does. Some PostgreSQL features are missing or work in a different way. We point out each difference clearly. We do not pretend a feature works when it does not.

30.1. Listeners and ports

The server binds two ports at startup. The defaults and the flags that change them are shown below.

ProtocolDefault portOverride flagTypical clients
PostgreSQL wire v3.05433--pg-port PORTpsql, JDBC, ODBC, DBeaver, pgAdmin, psycopg
EVO native text9967--evo-port PORTevosql-cli, EvoSQL drivers

The same process starts both listeners. The PG listener runs on the main thread. The EVO listener runs on a background thread. Each new connection is handled in its own thread.

# Start the dual-protocol server with default ports
./adaptor/evosql-server

# Move the PG listener to 5434, leave EVO on 9967
./adaptor/evosql-server --pg-port 5434

# A bare numeric argument is accepted as the PG port (backward compat)
./adaptor/evosql-server 5433

Note. The standard PostgreSQL port is 5432. EvolutionDB uses 5433 by default. This way it can run next to a real PostgreSQL instance on the same host. Set your client to port 5433. You can also use the port you chose with --pg-port. If you connect to 5432, you reach PostgreSQL and not EvolutionDB.

30.2. Credentials

Both protocols check the username and password against the database user catalog. They both use the same internal routine called AuthenticateUser. Passwords are stored with PBKDF2-SHA256. To learn how accounts are created and hashed, see CREATE USER and Authentication & users.

EVOSQL_USER_NAME
Bootstrap superuser name. Defaults to admin.
EVOSQL_PASSWORD
Bootstrap superuser password. If you do not set it, the server makes a random password. It prints this password to the server log one time at first start. Copy it from that line. You can also set this variable before you start the server to use a known value.

The default test database is testdb. The test and example logins in this manual use admin / admin. They work with a server started with EVOSQL_PASSWORD=admin.

Note. The user value in the PG startup packet picks the account to log in. There is no anonymous path and no trust path. A connection without a valid password is rejected before the query loop starts. Both protocols report login failures with SQLSTATE 28P01.

30.3. PostgreSQL wire protocol (port 5433)

EvolutionDB uses the PostgreSQL frontend/backend protocol version 3.0. The protocol number is 196608. A startup message that asks for any other version is rejected. The handshake works as follows.

30.3.1. Startup and negotiation

Before the regular startup packet, a client can send one of the special request codes. EvolutionDB handles them like this.

RequestCodeServer response
SSLRequest80877103The server sends S and then runs a TLS handshake if TLS is built in and available. If not, it sends N and the connection stays in plaintext. After this step the server reads the startup packet again.
GSSENCRequest80877104Always declined with N. The startup packet is read again. GSSAPI encryption is not supported.
CancelRequest80877102Finds the target session by its PID and secret. Sets that session's cancel flag. Then closes this connection. This follows the PG spec. See Query cancellation.
StartupMessage196608Parses startup parameters, extracts user, and begins authentication.

30.3.2. Authentication

After a valid v3.0 StartupMessage, the server always sends an AuthenticationCleartextPassword request. It waits for a PasswordMessage ('p') in reply. The password is checked with AuthenticateUser. On success the server sends AuthenticationOk. On failure it sends a FATAL error with SQLSTATE 28P01 (password authentication failed).

Note. The PG path uses cleartext password login only. MD5 and SCRAM-SHA-256 are not offered on the wire here. Real PostgreSQL uses scram-sha-256 by default. The password travels over the wire in clear text. So you should turn on TLS with SSLRequest. This keeps the password encrypted while it moves. The native EVO protocol does offer SCRAM and does not send the password. See below.

Next the server sends BackendKeyData. This message carries the session-registry ID, which is used as the PID, and a random 32-bit cancel key. After that the server sends ReadyForQuery. The transaction-status byte in ReadyForQuery can be three values. I means idle. T means inside a transaction block. E means a failed transaction that waits for ROLLBACK.

30.3.4. Simple Query protocol

A Query message ('Q') can carry several statements split by semicolons. The handler splits on semicolons but is careful with some cases. It respects single quotes, double quotes, and SQL line comments (-- ...). It also respects BEGIN ... END nesting. This way a procedure body is not cut at a semicolon inside it. The server sends one response for each statement. Then it sends a single ReadyForQuery. SELECT results are streamed one row at a time during the scan.

psql "host=127.0.0.1 port=5433 user=admin dbname=testdb sslmode=disable"

testdb=> CREATE TABLE t (id INT, name VARCHAR(50));
testdb=> INSERT INTO t VALUES (1, 'alice'), (2, 'bob');
testdb=> SELECT id, name FROM t WHERE id = 1;

30.3.5. Extended Query protocol

The Parse / Bind / Describe / Execute / Sync / Close sequence is supported. JDBC and psycopg's auto-prepare use this sequence. Here are the main behaviors that are special to EvolutionDB.

  • Named statements are kept. A Parse stores the query text under a statement name. A connection can hold up to 64 named statements. A later Bind and Execute can reuse the name with no Parse before it. In that case the server rebuilds the query from the stored template. Unnamed statements use the empty name and can be replaced freely.
  • Parameter substitution is done as text. Bound $1, $2, and so on are placed into the SQL string. A NULL parameter becomes the literal NULL. All other values are single-quoted, and quotes inside them are doubled. The server handles up to 64 parameters per Bind.
  • Binary parameter formats are decoded for the common fixed widths. These are 4-byte int32, 2-byte int16, 8-byte int64, and 1-byte boolean. They are then turned into text. Other binary inputs are copied through as they are.
  • Describe of a portal or statement returns a RowDescription for a SELECT with columns. Otherwise it returns NoData. Describe('S') first sends a ParameterDescription that reports zero parameter type OIDs.
  • Close('S', name) frees the named prepared statement and drops its stored template. Close('P', …) for a portal is acknowledged but keeps nothing.

Note. Parameters are bound by placing them into the string. They are not bound as separate typed values. The server handles quoting and escaping for you. But the server does not do typed parameter binding. So it does not do the type negotiation that comes with it. RowDescription reports 0 parameter OIDs in ParameterDescription. So drivers must not rely on the server to declare parameter types.

30.3.6. COPY over the wire

Both COPY ... FROM STDIN and COPY ... TO STDOUT work. They use the standard CopyIn/CopyOut subprotocol. This means CopyInResponse, CopyData, CopyDone, and CopyFail. A failed CopyIn returns SQLSTATE 57014. An unexpected message during CopyIn returns 08P01. See COPY.

30.4. EVO native protocol (port 9967)

The EVO protocol is a text protocol that uses newlines to split lines. It is made for the EvoSQL CLI and drivers. It works as request and response. The server sends a clear READY marker after every command result.

30.4.1. Greeting, TLS, and authentication

The exchange starts with a greeting. It can upgrade to TLS before any password is sent. The steps are below.

  1. The client sends the line EVO and nothing more. Anything else is rejected with ERR 08000.
  2. The server replies HELLO EvoSQL 1.0.
  3. If TLS is available, the server sends STARTTLS. If the client answers STARTTLS, the TLS handshake runs. If it answers anything else, the session stays in plaintext for backward compatibility.
  4. The server sends AUTH_SCRAM to ask for SCRAM-SHA-256.

The client then picks one of two login flows.

SCRAM-SHA-256 (preferred)
The client sends SCRAM-CLIENT-FIRST <data>. The server replies with the server-first message. The client sends SCRAM-CLIENT-FINAL <data>. The server checks the proof and returns the server-final message, which holds the ServerSignature for mutual authentication. The password is never sent. So this is safe even over plaintext.
Legacy cleartext (backward compat)
The client sends AUTH <username> <password>. The password is checked directly. Use SCRAM instead, or wrap the session in TLS.

On success the server sends AUTH_OK and starts the command loop. Login errors use SQLSTATE 28P01 for bad credentials or 28000 for an unexpected auth message.

# Connect with the bundled CLI (EVO, port 9967)
./cli/evosql-cli                # defaults to localhost:9967
./cli/evosql-cli -W admin        # prompt for a password

30.4.2. Running SQL with the SQL command

A query is sent as a length header line followed by the SQL body line:

SQL <byte_length>
<sql text>

The length must be in the range 1..65000. A value out of this range gives ERR 42000. A SELECT result is returned as a structured block:

RESULT
COLS <n>
COL <name> <type_oid> <type_modifier>     # one per column
ROW
FIELD <value>                            # or: FIELD NULL
...
END
TAG <command_tag>
READY

A non-SELECT statement returns OK and then TAG <command_tag>. One example is TAG INSERT 0 2. Errors return ERR <sqlstate> <message>. Every command ends with a READY line, whether it succeeds or fails. Sending QUIT ends the session with BYE.

30.4.3. Server-side prepared statements and batches

EVO offers named prepared statements right on the protocol. You do not need a SQL-level PREPARE. A connection can hold up to 32 statements. Placeholders are $1…$9+. The parameter count comes from the highest placeholder that is seen.

CommandPurpose
PREPARE <name> <byte_len> + bodyStores a parameterized statement under name. Replies OK / TAG PREPARE.
EXECUTE <name> <n> + n param linesFills in the parameters and runs once. Use the line \N for a SQL NULL. Other values are single-quoted, and quotes inside them are doubled.
EXECUTE_BATCH <name> <rows> <cols> + param linesRuns the statement once for each row. Replies BATCH_OK <total_affected> or BATCH_ERR <row_index> <sqlstate> <msg>.
EXECUTE_BATCH_BINARY <name> <rows> <cols> + binary payloadSame as above. But each parameter is sent as a typed binary value. The reply is the same BATCH_OK / BATCH_ERR.
DEALLOCATE <name>Frees a prepared statement. Replies OK / TAG DEALLOCATE. Replies ERR 26000 if it is not found.

The binary batch payload encodes each parameter as a one-byte type code followed by its big-endian value:

CodeTypePayload
NNULLno bytes. Sent as SQL NULL
iint44 bytes, big-endian
lint88 bytes, big-endian
dfloat88 bytes, big-endian IEEE-754
bbool1 byte. A value of 0 means false
sstring4-byte big-endian length, then UTF-8 bytes with no NUL. Max 16 MB

For batches, the affected-row counts are added up from the INSERT, UPDATE, and DELETE command tags. A batch stops at the first row that fails. It reports that row's index in BATCH_ERR.

Note. EVO prepared statements work like the PG extended path. They place parameters as quoted SQL text. They do not bind typed values on the server. The binary batch path still decodes typed values. But it turns them into text before it places them. There is a hard limit of 32 prepared statements per EVO connection. If you go over it, you get ERR 53000. The PG path allows 64 named statements.

30.4.4. COPY over EVO

Run a COPY ... FROM STDIN through the SQL command and the server replies COPY_READY. The client then streams one CSV line per row. It ends with \. on its own line. This follows the PostgreSQL convention. For COPY ... TO STDOUT the server sends COPY_READY. Then it sends one D:<line> per row. Then it sends a \. terminator. Both finish with OK / TAG COPY <n>. See COPY.

30.5. Session lifetime and limits

Sessions on both protocols share the same lifecycle handling.

  • Default isolation is READ COMMITTED. See SET TRANSACTION and MVCC & isolation.
  • Idle timeout. You can set a session idle timeout. If you do, a connection that sends no data within the time window is closed. The SQLSTATE is 57P05. The message is terminating connection due to idle timeout on PG and idle timeout on EVO.
  • Auto-rollback on disconnect. If a client drops while a transaction is open, the server rolls it back. It releases any SERIALIZABLE lock. It frees the undo log.
  • Temporary objects. Session-scoped temporary tables and global-temp-table contents are dropped on their own when the session ends.
  • LISTEN/NOTIFY. Asynchronous notifications are delivered between replies. On PG they come as a standard NotificationResponse. On EVO they come as a NOTIFY <sender_id> <channel> <len> framed line. See LISTEN / NOTIFY.

30.6. Query cancellation

On the PG protocol, BackendKeyData gives the client a PID and a random 32-bit secret. The PID is the session-registry ID. To cancel a running query, the client opens a second connection. It sends a CancelRequest with code 80877102 that carries that PID and secret. The server finds the session by PID and secret. It sets the cancel flag. Then it closes the cancel connection right away. This is exactly what the PostgreSQL spec says.

Note. The cancel key is made fresh for each connection. It comes from a cryptographic random source and is forced to be non-zero. So a leaked key only affects the one session it was given to. The EVO protocol has no separate cancel channel. To cancel there, you rely on the idle timeout and on closing the connection.
