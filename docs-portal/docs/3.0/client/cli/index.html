31. evosql-cli

evosql-cli is EvolutionDB's interactive terminal client: it speaks the native EVO text protocol, renders result sets as tables, CSV or JSON, supports backslash meta-commands, and runs equally well as an interactive shell or a non-interactive script runner.

Unlike PostgreSQL's psql, which talks the binary PostgreSQL wire protocol on port 5432, evosql-cli connects over EvolutionDB's line-oriented EVO protocol on TCP port 9967. The two are not interchangeable: psql (and other PostgreSQL clients such as DBeaver) connect to the server's PostgreSQL-compatible port 5433, while evosql-cli is the purpose-built client for the EVO port. Everything documented on this page reflects what the client actually implements; where a familiar psql feature is absent or behaves differently, that difference is called out honestly.

31.1. Invocation

The client takes connection parameters and an optional one-shot command or script file. With no command or file and a terminal attached to standard input, it starts an interactive shell; otherwise it runs in non-interactive (script) mode.

evosql-cli [OPTIONS] [-c SQL | -f FILE]
-h, --host HOST
Server hostname. Default: localhost. Resolved via gethostbyname(), falling back to inet_addr() for a literal IPv4 address.
-p, --port PORT
Server EVO port. Default: 9967. (This is the EVO port, not the PostgreSQL port 5433.)
-u, --user USER
Username. Default: admin.
-W, --password PW
Password. If omitted, the client reads the EVOSQL_PASSWORD environment variable; if that is also unset and the server requires authentication, it prompts interactively with echo disabled.
-c, --command SQL
Run the given SQL string and exit. Forces non-interactive mode.
-f, --file FILE
Run SQL read from FILE and exit. Forces non-interactive mode.
-F, --format FMT
Output format: table (default), csv, or json. An unknown value is rejected with exit code 2.
-x, --expanded
Expanded display: print one column | value line per field instead of an aligned grid. Equivalent to the \x meta-command. Applies to table format only.
--timing
Print per-statement client-side wall time after each statement. Equivalent to \timing.
--color / --no-color
Force ANSI colors on / off. The default is automatic: colors are emitted only when the format is table and standard output is a TTY (on Windows, colors are always off).
--help
Print usage and exit 0.

Example sessions:

# interactive shell, default localhost:9967 as admin
evosql-cli

# connect to a remote host, prompt for password
evosql-cli -h db.example.net -p 9967 -u alice

# one-shot query, machine-readable output
evosql-cli -F json -c "SELECT id, name FROM users"

# run a script file
evosql-cli -f schema.sql

# pipe a script in on stdin (runs non-interactively)
evosql-cli < schema.sql
Password handling. Precedence is -W → EVOSQL_PASSWORD → interactive prompt. Passing a secret on the command line exposes it to other users via the process table; prefer the environment variable or the prompt for anything sensitive.

31.2. Connecting and authentication

On startup the client opens a TCP socket, disables nothing on the line, and performs the EVO handshake:

  1. The client sends the literal line EVO as protocol identification.
  2. The server replies with a line beginning HELLO. Any other greeting is a fatal error.
  3. The server then sends either STARTTLS, AUTH_REQUIRED, or AUTH_SCRAM.
  4. If STARTTLS is offered and the client was built with TLS support (see below), the connection is upgraded to TLS before any credentials are sent. The next message (the auth request) is then read over the encrypted channel.
  5. For AUTH_REQUIRED or AUTH_SCRAM, the client sends AUTH username password and expects AUTH_OK; anything else is an authentication failure and the client exits.

The password buffer and the assembled AUTH message are securely wiped from memory immediately after the line is sent, using a volatile memset indirection that the compiler cannot optimize away.

SCRAM is advertised but not performed by this client. Even when the server offers AUTH_SCRAM (SCRAM-SHA-256), evosql-cli answers with the cleartext AUTH form. The server accepts both. If the connection is not protected by TLS, credentials travel in cleartext over the wire.

31.2.1. TLS

TLS is a compile-time option. Build the client with -DEVOSQL_CLI_TLS (linking -lssl -lcrypto), or use make TLS=1. Behavior at the STARTTLS step depends on whether that flag was set:

Built with TLS
The client acknowledges STARTTLS, performs an OpenSSL handshake using TLS_client_method(), and continues authentication over the encrypted channel. On success it prints [TLS] Encrypted connection established to standard error, and the interactive banner shows (TLS).
Built without TLS
The client declines by sending NOTLS and continues in plaintext, printing a warning that credentials will be sent in cleartext and suggesting make TLS=1.
No certificate verification. The TLS client creates a context and connects, but does not verify the server certificate or hostname. This protects against passive eavesdropping but not against an active man-in-the-middle. There is no equivalent of psql's sslmode=verify-full.

31.3. The interactive shell

In interactive mode the client prints a banner, loads command history, primes the tab-completion cache, installs its SIGINT handler, and enters the read loop. The primary prompt is evosql>; while a statement is still being accumulated the continuation prompt is ...>.

Connected to EvoSQL at localhost:9967
Type SQL (end with ';'), or \? for help, \q to quit.

evosql> SELECT 1 AS n;
n
-
1
(1 row)
evosql>

31.3.1. Statement assembly and the semicolon

SQL statements are terminated by a semicolon, not by pressing Enter. Lines are accumulated into a buffer until the buffer's last significant (non-whitespace, non-comment) character is a ; that lies outside any quoted string. A semicolon inside a string literal keeps the continuation prompt rather than submitting a half-typed statement:

evosql> INSERT INTO t(msg)
   ...> VALUES ('a; b; c');
INSERT 0 1
(1 row)

Because the EVO protocol frames each query as a single line, embedded newlines (and carriage returns) inside a multi-line statement are folded to spaces before the statement is sent. Quote tracking distinguishes single quotes (') from double quotes ("), and a -- line comment outside quotes is consumed to end of line.

31.3.2. Quitting

You can leave the shell with the \q meta-command, with the bare words quit or exit (case-insensitive, kept for familiarity), or with end-of-file (Ctrl-D on an empty line). On a non-empty multi-line buffer, Ctrl-D discards the pending buffer and re-prompts rather than exiting.

31.3.3. Ctrl-C, cancel, not quit

evosql-cli never terminates on Ctrl-C. Its behavior depends on what the client is doing:

At the prompt (typing)
Ctrl-C abandons the current line and any pending multi-line buffer and re-draws a fresh prompt, shell-like. (On POSIX this is implemented by long-jumping out of readline after cleaning up the terminal; rl_catch_signals is disabled so readline never re-raises the signal.)
During a running query
The EVO protocol has no in-band query-cancel message. The client therefore sets an interrupt flag, abandons the now-desynchronized socket, and transparently reconnects a fresh session, printing query canceled. Connection parameters are stashed at startup precisely so this reconnect can happen silently. If the schema may have changed (the canceled statement contained CREATE/DROP/ALTER), the completion cache is refreshed afterward.
This is a real difference from psql. In PostgreSQL, Ctrl-C sends a protocol-level cancel request on a side channel and the same session continues. In EvolutionDB the only way to abandon a running EVO query from the prompt is to drop and reopen the socket, so any uncommitted transaction state on the old connection is lost. See transaction control.

31.3.4. Command history

History is persisted to ~/.evosql_history (on Windows, %USERPROFILE%\.evosql_history), capped at 500 entries. On POSIX the client uses GNU readline's history facilities (with arrow-key recall and in-line editing); on Windows it provides a built-in readline replacement using the Win32 console API with the same 500-entry cap, up/down recall, and basic line editing.

Secrets stay out of history. Any completed statement that contains a bare PASSWORD keyword is excluded from the on-disk history file, so cleartext passwords from e.g. CREATE USER never reach disk. The keyword boundary is checked against any non-identifier character, so forms like PASSWORD'secret' are caught too.

31.3.5. Tab completion

On POSIX (GNU readline) only, pressing Tab completes against a fixed list of SQL keywords and a cached list of table names. Matching is case-insensitive on the typed prefix; filename fallback is disabled. The table-name cache is populated at startup via SHOW TABLES and refreshed after any statement containing CREATE, DROP, or ALTER (and after a canceled DDL reconnect), so newly created or dropped tables appear or disappear without restarting the client. The completed keyword set is fixed and includes the common DML/DDL words, join keywords, aggregate names, and a few type names:

SELECT FROM WHERE INSERT INTO VALUES UPDATE SET DELETE CREATE TABLE
DROP ALTER INDEX JOIN LEFT RIGHT INNER OUTER ON GROUP BY ORDER HAVING
LIMIT OFFSET DISTINCT AS AND OR NOT NULL LIKE IN BETWEEN PRIMARY KEY
FOREIGN REFERENCES DEFAULT UNIQUE CHECK BEGIN COMMIT ROLLBACK DATABASE
SCHEMA USER GRANT REVOKE SHOW TABLES DATABASES USERS COUNT SUM AVG MIN
MAX VARCHAR INT INTEGER TIMESTAMP BOOLEAN TEXT

Tab completion is not available in the Windows console build.

31.4. Output formats

The result format is set with -F/--format at startup or the \format meta-command at runtime. Three formats are supported.

31.4.1. table (default)

An aligned grid: a bold header row, a dimmed -+- rule, then the rows, finishing with a dimmed (N row[s]) count. Column widths are computed from the widest value (or header). NULL values render as the dimmed literal NULL. Status tags and the row count are written to standard output in this format. When \x (expanded) is on, each record is printed as a -[ RECORD n ]- banner followed by one column | value line per field, handy for wide rows.

31.4.2. csv

RFC 4180-style CSV: a header row of column names, then one line per row. A field is quoted only if it contains a comma, a double quote, CR, or LF; embedded double quotes are doubled. A NULL value is emitted as an empty field (indistinguishable from an empty string). In CSV (and JSON) mode only the data goes to standard output; status tags, the row count, errors, and timing go to standard error, so a redirected stream contains clean data.

evosql-cli -F csv -c "SELECT id, name FROM users" > users.csv

31.4.3. json

A JSON array of objects, one object per row, keyed by column name. Strings use minimal escaping (\", \\, \n, \r, \t, and \u00XX for other control characters). A NULL value is emitted as JSON null. A value is emitted as an unquoted JSON number only when both conditions hold: the column's PostgreSQL type OID is numeric, and the value's text is a strict JSON number; otherwise it is quoted. The recognised numeric OIDs are:

OIDType
20int8 (bigint)
21int2 (smallint)
23int4 (integer)
26oid
700float4 (real)
701float8 (double precision)
1700numeric

The OID for each column is recovered from the EVO COL line, whose form is <name> <pg_type_oid> <type_modifier>; the client drops the two trailing integer fields by count (the column name itself may contain spaces) and reads the OID from the second-from-last token.

evosql-cli -F json -c "SELECT id, name FROM users LIMIT 1"
[
 {"id": 1, "name": "alice"}
]
Limitations to know. NULL and empty string are indistinguishable in CSV. A result set is buffered fully in memory before printing: there is no streaming or paging, and the internal row buffer starts at 64 rows and doubles as needed (there is no fixed truncation in the EVO read path). The JSON emitter performs no pretty-printing beyond one object per line.

31.5. Meta-commands

A line beginning with a backslash is a meta-command. Unlike SQL statements, a meta-command is terminated by the newline (not by a semicolon), though a trailing semicolon is tolerated. Run \? for the built-in summary. The full set the client implements:

\?, \h, \help
Print the meta-command reference.
\d, \dt
With no argument, list tables (runs SHOW TABLES). With a name, describe that table's columns by querying information_schema.columns for column_name, data_type, is_nullable. See SHOW.
\l
List databases (SHOW DATABASES). See CREATE DATABASE.
\du
List users (SHOW USERS). See CREATE USER.
\dn
List schemas (SHOW SCHEMAS).
\x
Toggle expanded display (one column per line). Prints the new state.
\timing
Toggle per-statement timing. Prints the new state.
\format F
Set the output format to table, csv, or json (case-insensitive). Any other argument prints a usage line.
\i FILE
Read FILE and run it as a script (the same splitter used for -f). The file may itself contain meta-commands, including a nested \i.
\e
Open $EDITOR (default vi) on a private temporary file created with mkstemp; on a clean editor exit, the file's contents are run as a script. POSIX only, on Windows it prints that it is unsupported.
\! CMD
Run CMD in the system shell.
\q, \quit
Quit the client.
Naming notes vs. psql. The verbs are deliberately psql-flavored, but the set is small and a few behave slightly differently: \d and \dt are aliases (both list tables, and \d NAME describes columns); \l, \du, \dn are thin wrappers over the corresponding SHOW command. There is no \copy, no \g, no variable system (\set), and no \connect, the connection is fixed for the life of the process (apart from the transparent reconnect after a Ctrl-C cancel).

31.6. Scripting and non-interactive use

The client runs non-interactively whenever -c or -f is given, or when standard input is not a terminal (piped or redirected). In that mode it runs the supplied SQL, sends a QUIT line, closes the connection, and exits.

31.6.1. The statement splitter

Script text, from -c, -f, piped stdin, \i, or \e, is split into statements by the same quote- and comment-aware scanner used interactively:

  • The kind of each statement is fixed at its first non-blank character: a line starting with \ is a meta-command (terminated by newline); anything else is SQL (terminated by an unquoted ;).
  • -- line comments outside quotes run to end of line and are collapsed to a single separating space inside a statement; a standalone comment line is dropped.
  • Single and double quotes are tracked so semicolons and comment markers inside string literals are not treated as syntax.
  • Newlines and carriage returns inside a SQL statement are folded to spaces (the EVO protocol expects a single-line query).
  • A trailing statement with no final ; or newline is still run.
-- schema.sql
CREATE TABLE t (id INT PRIMARY KEY, name VARCHAR(50));
INSERT INTO t VALUES (1, 'alice'), (2, 'bob');
SELECT count(*) FROM t;

31.6.2. Exit status

Non-interactive exit status is designed so CI and shell scripts can detect a failed query:

0
All statements ran and the server reported no error.
1
A statement failed, the run hit a non-zero internal code (an unreadable -f file, a connection loss, or a Ctrl-C cancel) or the server returned an ERR for any statement (tracked by an error-seen flag, even if later statements succeed).
2
An invalid --format argument was given (the client exits before connecting).
evosql-cli -c "SELECT 1" && echo ok || echo failed
Errors don't stop a script. In non-interactive mode a server ERR sets the error flag and the run continues to the next statement; the process still exits 1 at the end. A connection loss (-1) or Ctrl-C cancel (-2), however, stops the script. This differs from psql's default and from its ON_ERROR_STOP, there is no equivalent setting here.

31.7. Status tags, errors, and timing

The EVO protocol reports the outcome of each statement as a command tag (e.g. INSERT 0 1, SELECT, CREATE TABLE), an error line, or a result set. The client surfaces these as follows:

Command tag
Printed verbatim. In table format it goes to standard output; in csv/json it goes to standard error so it does not pollute the data stream.
Error
Printed as ERROR: message to standard error (in red when color is on). The error-seen flag is set so non-interactive runs can exit non-zero.
Row count
A (N row[s]) footer follows each result set, routed like the command tag (stdout for table, stderr otherwise).
Timing
When \timing / --timing is on, a Time: X.X ms line is printed to standard error after each statement. This is the client-side wall time around the request/response (measured with clock()), not a server-reported execution time.

31.8. Type fidelity

The client itself is type-agnostic for display: every value arrives as text on a FIELD line and is printed as-is in table and csv formats. The only place type information is consulted is JSON output, where the column's PostgreSQL type OID (carried on the EVO COL line) decides whether a numeric-looking value is emitted unquoted (see §31.4.3). For the full type system and how values are stored, see the SQL reference and the data-type chapters; for query syntax see SELECT, INSERT, UPDATE, and DELETE.

31.9. Notes and limitations

  • EVO port only. evosql-cli connects to the EVO text protocol on port 9967. To use the PostgreSQL wire protocol (port 5433) use psql or any PostgreSQL-compatible client; evosql-cli cannot talk to that port.
  • No in-band query cancel. Canceling a running query drops and reopens the connection, losing any uncommitted transaction state on that session.
  • SCRAM not implemented client-side. The client always authenticates with the cleartext AUTH form even when the server advertises SCRAM-SHA-256.
  • TLS does not verify certificates. When built with TLS, the handshake protects against passive eavesdropping only.
  • Results are fully buffered. Large result sets are read into memory before display; there is no paging or streaming output.
  • Single fixed connection. No \connect, no connection switching, no variables. The host/port/user/password are bound for the life of the process.
  • NULL vs. empty string in CSV. Both render as an empty field; they are distinguishable only in json (where NULL is null) and in table (where NULL prints as NULL).
  • Platform parity. Tab completion and \e are POSIX-only; the Windows console build provides history and basic line editing but not these features.
