C. Keywords

The complete list of words the EvolutionDB lexer recognizes as keywords — what they mean, how they are matched (case-insensitive, with several context-sensitive special cases), and how to use ordinary identifiers that happen to collide with them.

C.1. Overview

Every SQL statement EvolutionDB accepts is first broken into tokens by the Flex scanner defined in evolution/parser/evolexer.l. A keyword is any word the scanner maps to a dedicated grammar token (for example SELECT → SELECT, BIGINT → BIGINT) rather than to the generic identifier token NAME. The grammar that consumes those tokens lives in evoparser.y; this appendix documents only the scanner side — which words are reserved and exactly how they are matched.

This list is the authoritative, source-derived set: it is enumerated by hand-written rules in the lexer, not generated from a standard. As a result EvolutionDB's keyword set deliberately mixes spellings drawn from the SQL standard, PostgreSQL, and MySQL. Where a word exists only to support one dialect's syntax, that is noted below.

Note. Unlike PostgreSQL, EvolutionDB's lexer does not partition keywords into the SQL-standard categories reserved, non-reserved, reserved (can be function or type name), etc. Any word listed here is recognized as a keyword token wherever it appears, unless it is written as a quoted identifier (see C.5) or falls under one of the context-sensitive rules in C.4. The categorization used in this appendix is descriptive (grouped by purpose), not a formal reservation class.

C.2. Case sensitivity

The scanner is declared with Flex's case-insensitive option:

%option noyywrap nodefault yylineno case-insensitive reentrant bison-bridge

Consequently keywords may be written in any mix of upper- and lower-case. SELECT, select, and SeLeCt are all the same token. The same applies to type names, function names, and every other word in this appendix.

Case-insensitivity applies only to keyword recognition. Unquoted identifiers are captured verbatim by the NAME rule ([A-Za-z][A-Za-z0-9_]*) and string literals preserve their bytes exactly, so the case of identifiers and data values is retained as typed.

-- all three statements parse identically
SELECT * FROM employees;
select * from employees;
Select * From employees;

C.4. Context-sensitive and special-cased words

Most keywords are matched by a plain word rule. A handful are handled specially because the same spelling must behave differently depending on its surroundings. Understanding these cases explains parser behavior that otherwise looks surprising.

C.4.1. BETWEEN ... AND ...

The word AND is overloaded: it is both the boolean connective and the separator inside a BETWEEN range. The lexer resolves this with a start condition. Seeing BETWEEN switches the scanner into a BTWMODE state; the next AND is then returned as the dedicated range token (and the scanner returns to its initial state), while every other AND is the ordinary boolean connective token.

-- the AND here is the BETWEEN separator, not a boolean connective
SELECT * FROM orders
WHERE total BETWEEN 10 AND 100;

C.4.2. Multi-word tokens

A few constructs are matched as a single token even though they are written as two words separated by whitespace. These are not two independent keywords but one lexical unit (the patterns allow runs of spaces, tabs, and — for two of them — newlines between the words):

NOT EXISTS
Matched as one EXISTS token carrying a "negated" flag, distinct from a bare EXISTS. (A bare NOT followed by something other than EXISTS is still the ordinary NOT keyword.)
ON DUPLICATE
Matched as a single token introducing the MySQL ON DUPLICATE KEY UPDATE upsert clause — see INSERT.
NO ACTION
Matched as one referential-action token (the foreign-key default behavior).

C.4.3. Function-or-identifier words (trailing ( lookahead)

Several words are keywords only when they immediately precede a (; otherwise they are ordinary identifiers. The lexer peeks at the next character: if it is ( the word becomes the corresponding function token, otherwise it is returned as a plain NAME (column/alias) so these names remain usable as identifiers.

This is why count, sum, min, max, avg and rank can all be used as column names, but writing them immediately before a parenthesis invokes the function:

-- "count" used as a column name (no following parenthesis)
SELECT count FROM inventory;

-- COUNT used as an aggregate (immediately followed by "(")
SELECT COUNT(*) FROM inventory;

C.4.4. VALUE

The word VALUE is unusual: rather than returning a keyword token, its rule returns an identifier token whose text is the string "VALUE". In effect VALUE is always treated as an ordinary name, never reserved. (Its plural, VALUES, is a reserved keyword used by INSERT.)

C.4.5. TRUE, FALSE, UNKNOWN, NULL

The boolean literals return a single BOOL token distinguished by an attached integer value: TRUE carries 1, FALSE carries 0, and UNKNOWN carries -1 (three-valued logic). NULL returns the separate NULLX token. Because UNKNOWN appears in two rules, it is recognized both as a standalone keyword token and as a boolean literal depending on grammar position.

C.5. Using keywords as identifiers

If you need a table, column, or alias whose name collides with a reserved word, wrap it in backticks. The lexer's quoted-identifier rule accepts `...` and strips the surrounding backticks, returning the inner text as a plain NAME regardless of whether that text would otherwise be a keyword:

`any text except backtick / backslash / dot / newline`  →  identifier
-- "order" and "select" are keywords; backticks make them column names
CREATE TABLE t (`order` INT, `select` VARCHAR(20));
SELECT `order`, `select` FROM t;
Note. A backtick-quoted identifier may not contain a backtick, backslash, dot, or newline — those characters are excluded by the rule's character class. An opening backtick with no closing backtick before end-of-line raises "unterminated quoted name". EvolutionDB uses backticks (MySQL style) for quoted identifiers here; the double-quote character is consumed by the string-literal rule, not the identifier rule, so it is not an interchangeable identifier quote.

C.8. Compatibility

EvolutionDB's keyword set is intentionally polyglot. A few honest contrasts with the systems it borrows spellings from:

  • vs. the SQL standard. EvolutionDB has no formal reservation classes. The standard distinguishes reserved words from non-reserved words that may double as identifiers; EvolutionDB instead relies on (a) every keyword being usable as a quoted (backtick) identifier and (b) the trailing-( trick of C.4.3 for a small set of words. There is no concept here of a word that is reserved "except as a function or type name" the way the standard defines it.
  • vs. PostgreSQL. PostgreSQL uses double quotes for identifiers and a categorized keyword table; EvolutionDB uses MySQL-style backticks (C.5) and a flat, hand-maintained list. Many PostgreSQL spellings are present nonetheless — LATERAL, RETURNING, JSONB (folded to the JSON token), UUID, MATERIALIZED, CONCURRENTLY, NEXTVAL/CURRVAL, GEN_RANDOM_UUID, and PG_NOTIFY as an alias.
  • vs. MySQL. Many MySQL spellings are accepted: integer aliases (INT8, INT4, INT2, INT1, MIDDLEINT), AUTO_INCREMENT, DISTINCTROW, STRAIGHT_JOIN, HIGH_PRIORITY/LOW_PRIORITY, SQL_BIG_RESULT and friends, the ON DUPLICATE KEY upsert, REGEXP/RLIKE, GROUP_CONCAT, and LAST_INSERT_ID.
  • EvolutionDB-only. The extension keywords in C.3.6 (pub/sub, subscriptions, the document/graph/memory/message/checkpoint/entity stores, scheduled jobs, sharding, system-versioning) and the vector operators <->, <#>, <=> are specific to EvolutionDB.
Note — caveats.
  • This appendix reflects exactly the rules in evolution/parser/evolexer.l as compiled into the parser. Because the keyword list is hand-written, it can drift between releases; regenerate the parser with make generate after any lexer change and re-check this list.
  • Being recognized as a keyword token does not guarantee the surrounding statement is accepted by the grammar, nor that the feature is fully implemented by the engine. The scanner only classifies words; semantics live on the individual command reference pages.
  • A word's "reserved-ness" is effectively global: there is no per-statement reservation. If a word here collides with a name you want, quote it with backticks (C.5).
  • The duplicate appearance of some words across multiple rules (e.g. UNKNOWN, NODE, OF, STORE, RANK, CHECKPOINT) is intentional — Flex applies the first matching rule, so order in the source file determines which token wins in ambiguous positions.
