C. Keywords

This is the full list of words that the EvolutionDB lexer treats as keywords. It tells you what they mean. It tells you how they are matched. Matching is case-insensitive. A few words have special context-sensitive rules. It also shows how to use a normal identifier when its name is the same as a keyword.

C.1. Overview

EvolutionDB first breaks every SQL statement into tokens. The Flex scanner does this work. It is defined in evolution/parser/evolexer.l. A keyword is a word that the scanner maps to its own grammar token. For example, SELECT maps to SELECT, and BIGINT maps to BIGINT. Other words map to the generic identifier token NAME. The grammar that reads those tokens lives in evoparser.y. This appendix covers only the scanner side. It shows which words are reserved and how they are matched.

This list comes straight from the source code. Hand-written rules in the lexer build it. It is not generated from a standard. So the keyword set mixes spellings from the SQL standard, PostgreSQL, and MySQL on purpose. Some words exist only to support one dialect. The notes below point those out.

Note. PostgreSQL sorts keywords into SQL-standard groups. These groups are reserved, non-reserved, and reserved but usable as a function or type name. EvolutionDB's lexer does not do this. Any word in this list is a keyword token everywhere it appears. There are two exceptions. The first is when you write it as a quoted identifier. See C.5. The second is when one of the context-sensitive rules in C.4 applies. The groups in this appendix only sort words by purpose. They are not a formal reservation class.

C.2. Case sensitivity

The scanner is declared with Flex's case-insensitive option:

%option noyywrap nodefault yylineno case-insensitive reentrant bison-bridge

So you can write keywords in any mix of upper- and lower-case. SELECT, select, and SeLeCt are all the same token. This is also true for type names, function names, and every other word in this appendix.

Case does not matter only when the scanner recognizes a keyword. Unquoted identifiers are kept exactly as typed by the NAME rule. That rule is [A-Za-z][A-Za-z0-9_]*. String literals also keep their bytes exactly. So the case of your identifiers and data values stays the way you typed it.

-- all three statements parse identically
SELECT * FROM employees;
select * from employees;
Select * From employees;

C.4. Context-sensitive and special-cased words

A plain word rule matches most keywords. A few words need special handling. The same spelling has to act in different ways based on what is around it. These cases explain parser behavior that can look surprising at first.

C.4.1. BETWEEN ... AND ...

The word AND has two jobs. It is the boolean connective. It is also the separator inside a BETWEEN range. The lexer sorts this out with a start condition. When the scanner sees BETWEEN, it switches into a BTWMODE state. The next AND is then returned as the special range token. After that the scanner goes back to its starting state. Every other AND is the normal boolean connective token.

-- the AND here is the BETWEEN separator, not a boolean connective
SELECT * FROM orders
WHERE total BETWEEN 10 AND 100;

C.4.2. Multi-word tokens

A few things are matched as a single token even though you write them as two words with a space between them. They are not two separate keywords. They are one lexical unit. The patterns allow runs of spaces and tabs between the words. Two of them also allow newlines.

NOT EXISTS
Matched as one EXISTS token with a "negated" flag set. It is not the same as a bare EXISTS. A bare NOT followed by anything other than EXISTS is still the normal NOT keyword.
ON DUPLICATE
Matched as a single token. It starts the MySQL ON DUPLICATE KEY UPDATE upsert clause, see INSERT.
NO ACTION
Matched as one referential-action token. This is the default behavior for a foreign key.

C.4.3. Function-or-identifier words (trailing ( lookahead)

Several words are keywords only when a ( comes right after them. In any other case they are normal identifiers. The lexer looks at the next character. If it is (, the word becomes the matching function token. If it is not, the word is returned as a plain NAME for a column or alias. This way you can still use these names as identifiers.

WordFollowed by (Otherwise
COUNTaggregate COUNT(...)identifier NAME

This is why you can use count, sum, min, max, avg and rank as column names. But if you write them right before a parenthesis, you call the function instead.

-- "count" used as a column name (no following parenthesis)
SELECT count FROM inventory;

-- COUNT used as an aggregate (immediately followed by "(")
SELECT COUNT(*) FROM inventory;

C.4.4. VALUE

The word VALUE works in an unusual way. Its rule does not return a keyword token. Instead it returns an identifier token whose text is the string "VALUE". So VALUE is always treated as a normal name. It is never reserved. Its plural is VALUES. That one is a reserved keyword used by INSERT.

C.4.5. TRUE, FALSE, UNKNOWN, NULL

The boolean literals return a single BOOL token. An integer value is attached to tell them apart. TRUE carries 1. FALSE carries 0. UNKNOWN carries -1 for three-valued logic. NULL returns the separate NULLX token. UNKNOWN appears in two rules. So it can be a standalone keyword token or a boolean literal. The grammar position decides which one.

C.5. Using keywords as identifiers

Sometimes you need a table, column, or alias whose name is the same as a reserved word. To do this, wrap the name in backticks. The lexer's quoted-identifier rule accepts `...`. It strips the backticks and returns the inner text as a plain NAME. This works even if that text would normally be a keyword.

`any text except backtick / backslash / dot / newline`  →  identifier
-- "order" and "select" are keywords; backticks make them column names
CREATE TABLE t (`order` INT, `select` VARCHAR(20));
SELECT `order`, `select` FROM t;
Note. A backtick-quoted identifier may not contain a backtick, backslash, dot, or newline. The rule's character class leaves those characters out. An opening backtick with no closing backtick before the end of the line raises "unterminated quoted name". EvolutionDB uses MySQL-style backticks for quoted identifiers here. The double-quote character belongs to the string-literal rule, not the identifier rule. So you cannot use it in place of a backtick to quote an identifier.

C.8. Compatibility

EvolutionDB's keyword set mixes many dialects on purpose. Here are a few honest contrasts with the systems it borrows spellings from.

  • vs. the SQL standard. EvolutionDB has no formal reservation classes. The standard splits words into reserved words and non-reserved words. Non-reserved words can also be used as identifiers. EvolutionDB works in a different way. First, you can use any keyword as a backtick-quoted identifier. Second, a small set of words use the trailing-( trick of C.4.3. There is no idea here of a word that is reserved "except as a function or type name" the way the standard defines it.
  • vs. PostgreSQL. PostgreSQL uses double quotes for identifiers and a keyword table sorted into groups. EvolutionDB uses MySQL-style backticks (C.5) and a flat, hand-maintained list. Many PostgreSQL spellings are still here. These are LATERAL, RETURNING, JSONB folded to the JSON token, UUID, MATERIALIZED, CONCURRENTLY, NEXTVAL/CURRVAL, GEN_RANDOM_UUID, and PG_NOTIFY as an alias.
  • vs. MySQL. Many MySQL spellings are accepted. These are the integer aliases INT8, INT4, INT2, INT1, and MIDDLEINT, plus AUTO_INCREMENT, DISTINCTROW, STRAIGHT_JOIN, HIGH_PRIORITY/LOW_PRIORITY, SQL_BIG_RESULT and friends, the ON DUPLICATE KEY upsert, REGEXP/RLIKE, GROUP_CONCAT, and LAST_INSERT_ID.
  • EvolutionDB-only. The extension keywords in C.3.6 are specific to EvolutionDB. They cover pub/sub, subscriptions, the document, graph, memory, message, checkpoint, and entity stores, scheduled jobs, sharding, and system-versioning. The vector operators <->, <#>, and <=> are also specific to EvolutionDB.
Note, caveats.
  • This appendix matches the rules in evolution/parser/evolexer.l exactly as they are compiled into the parser. The keyword list is hand-written. So it can drift between releases. After any lexer change, regenerate the parser with make generate and check this list again.
  • A word being a keyword token does not mean the grammar accepts the whole statement. It also does not mean the engine fully implements the feature. The scanner only sorts words. The meaning lives on the individual command reference pages.
  • A word is reserved everywhere. There is no per-statement reservation. If a word here is the same as a name you want, quote it with backticks (C.5).
  • Some words show up in more than one rule on purpose. Examples are UNKNOWN, NODE, OF, STORE, RANK, and CHECKPOINT. Flex uses the first rule that matches. So the order in the source file decides which token wins when the position is unclear.
