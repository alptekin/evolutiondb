A. Error Codes (SQLSTATE)

Every error EvolutionDB raises carries a five-character SQLSTATE code; this appendix lists the codes the engine actually emits, what each one means, and how EvolutionDB's vendor numbering differs from PostgreSQL and the SQL standard.

A.1. Overview

EvolutionDB follows the SQLSTATE convention defined by the SQL standard: a code is exactly five characters, split into a two-character class and a three-character subclass. Class 00 means success, class 01 a warning, class 02 "no data", and every other class denotes an error. Codes are returned to clients regardless of protocol:

  • PostgreSQL wire protocol (port 5433): the code is placed in the C (Code) field of the ErrorResponse message, alongside the human-readable M (Message) field.
  • EVO text protocol (port 9967): the code is the first token of an error line, e.g. ERR 23505 duplicate key value violates unique constraint.

Internally, the active code lives in a six-byte field g_err.sqlstate (five characters plus a NUL terminator). It is cleared to an empty string at the start of every statement and set by the operation module that detects the error. The adaptor copies it verbatim into the result set that is serialized back to the client.

How a code is set in the engine. Operation modules use a single convenience macro defined in evolution/db/evosql_errcodes.h: 
/* sets g_err.sqlstate to the 5-char literal, NUL-terminated */
EVOSQL_SET_SQLSTATE(EVOSQL_ERRCODE_UNIQUE_VIOLATION);
The macro copies the five characters of the named code and writes the trailing NUL. There is no message catalog: the message text is supplied separately by the operation that raised the error.

A.2. The EvolutionDB numbering scheme

Where a code is mandated by the SQL standard, EvolutionDB uses the standard value unchanged. Where the standard leaves a slot open for the implementation, vendors invent their own subclasses, and each vendor reserves a distinct letter so codes never collide. EvolutionDB reserves the letter E:

VendorSubclass markerExample
SQL standard (ISO)numeric subclass23505 unique violation
PostgreSQLP42P01 undefined table
MySQLSHY000 / S subclasses
EvolutionDBE42E01 undefined table

EvolutionDB applies the marker in two ways:

E vendor subclass within a standard class
For conditions the standard does not assign a subclass to, EvolutionDB places an E in the third position of a recognized class. For example, "undefined table" is 42E01 (class 42, the standard's "syntax error or access rule violation"), which is the EvolutionDB analogue of PostgreSQL's 42P01.
E0–EZ vendor classes
For conditions with no standard class at all — storage engine, buffer pool, garbage collection, ID generation — EvolutionDB owns the entire class range E0 through EZ. These codes have no SQL-standard or PostgreSQL equivalent.
Honest caveat: legacy literals. A few code paths that predate the central header still emit hard-coded literals that follow PostgreSQL's P convention rather than the E scheme — notably 42P01 (undefined table) in some CREATE paths and the PG protocol fallback, 08P01 (protocol violation) in the connection handler, and 45000 (the SQL-standard "unhandled user-defined exception" class 45) as the default for the SIGNAL statement. These are intentionally PostgreSQL-compatible so existing drivers recognize them, and they coexist with the canonical E-scheme codes in evosql_errcodes.h.

A.3. Default and fallback codes

If an operation raises an error but does not set a specific SQLSTATE, the adaptor substitutes a class-level default rather than sending an empty code:

42000 — Syntax Error or Access Rule Violation (generic)
The general fallback used by the query executor and the EVO protocol when g_err.sqlstate is empty. Many EVO protocol framing errors (bad command, missing arguments, invalid SQL length) report 42000 directly.
22000 — Data Exception (generic)
The fallback used inside COPY row parsing when no more specific data-exception code is available.
00000 — Successful Completion
The initial value of a result set's SQLSTATE before any error occurs.

The internal error handlers. evolution/db/error.c provides the family of err_* functions. In server (GUI) mode they do not call exit(); instead they set a SQLSTATE and longjmp back to the executor so the connection survives. System-call failures (err_sys, err_dump) set 58000 (SYSTEM_ERROR); generic fatal errors (err_exit, err_quit) set EX000 (INTERNAL_ERROR). If you see EX000 in a client, it indicates a condition the engine considers a bug — please report it.

A.6. Differences from PostgreSQL and the SQL standard

EvolutionDB is deliberately compatible with PostgreSQL clients but does not duplicate PostgreSQL's full error-code catalog. The honest differences:

  • Vendor marker is E, not P. Where PostgreSQL would emit 42P01, 22P02, 28P01, or 25P01, EvolutionDB's canonical codes are 42E01, 22E02, 28E01, and 25E01. A handful of legacy paths still emit the PostgreSQL P literals (notably 42P01, 08P01, 40P01, 55P03) for maximum driver compatibility.
  • Far fewer codes than PostgreSQL. EvolutionDB defines only the codes it actually needs; large swaths of PostgreSQL's catalog (PL/pgSQL classes, configuration-file errors, FDW errors, etc.) are not present.
  • Vendor classes E0–EX are unique to EvolutionDB. They describe the page-based storage engine, buffer pool, RECLAIM, and ID generation; no portable application should depend on these beyond logging.
  • No message catalog. Unlike systems that map an SQLSTATE to a fixed message, EvolutionDB sets the code and message independently at each error site. Match on the SQLSTATE code for programmatic handling, never on the message text.
Notes and limitations.
  • Several codes in evosql_errcodes.h are reserved — defined for completeness and forward compatibility but not yet set anywhere in the engine. Treat the "emitted" annotations above as the practical, current behavior; the reserved codes may begin appearing in future releases.
  • The active code is stored in a single global (g_err.sqlstate) guarded by the executor mutex, and is cleared at the start of each statement. It reflects only the most recent error on that connection.
  • When no specific code is set, clients see 42000 (general) or, within COPY, 22000 — not an empty string.
  • The PostgreSQL wire protocol always includes the code in the C field; the EVO protocol always prefixes it as the first token of ERR … / BATCH_ERR … lines.
