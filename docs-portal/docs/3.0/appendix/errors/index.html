A. Error Codes (SQLSTATE)

Every error EvolutionDB raises carries a five-character SQLSTATE code. This appendix lists the codes the engine actually emits. It tells you what each one means. It also shows how EvolutionDB's vendor numbering differs from PostgreSQL and the SQL standard.

A.1. Overview

EvolutionDB follows the SQLSTATE convention from the SQL standard. A code is exactly five characters. It is split into a two-character class and a three-character subclass. Class 00 means success. Class 01 is a warning. Class 02 means "no data". Every other class means an error. Codes go to clients on both protocols.

  • PostgreSQL wire protocol on port 5433. The code goes into the C field of the ErrorResponse message. The M field next to it holds the readable message.
  • EVO text protocol on port 9967. The code is the first token of an error line. For example, ERR 23505 duplicate key value violates unique constraint.

Inside the engine, the active code lives in a six-byte field g_err.sqlstate. That is five characters plus a NUL terminator. The code is cleared to an empty string at the start of every statement. The operation module that finds the error sets it. The adaptor copies it as is into the result set sent back to the client.

How a code is set in the engine. Operation modules use one helper macro. It is defined in evolution/db/evosql_errcodes.h. 
/* sets g_err.sqlstate to the 5-char literal, NUL-terminated */
EVOSQL_SET_SQLSTATE(EVOSQL_ERRCODE_UNIQUE_VIOLATION);
The macro copies the five characters of the named code. Then it writes the trailing NUL. There is no message catalog. The operation that raised the error supplies the message text on its own.

A.2. The EvolutionDB numbering scheme

When the SQL standard fixes a code, EvolutionDB uses that value as is. When the standard leaves a slot open, each vendor makes its own subclasses. Each vendor keeps its own letter so codes never clash. EvolutionDB keeps the letter E.

VendorSubclass markerExample
SQL standard (ISO)numeric subclass23505 unique violation
PostgreSQLP42P01 undefined table
MySQLSHY000 / S subclasses
EvolutionDBE42E01 undefined table

EvolutionDB uses the marker in two ways.

E vendor subclass within a standard class
Some conditions have no subclass in the standard. For those, EvolutionDB puts an E in the third spot of a known class. Take "undefined table" as an example. Its code is 42E01. Class 42 is the standard's "syntax error or access rule violation" class. This code is the EvolutionDB version of PostgreSQL's 42P01.
E0 to EZ vendor classes
Some conditions have no standard class at all. These include the storage engine, the buffer pool, garbage collection, and ID generation. EvolutionDB owns the whole class range E0 through EZ for them. These codes have no SQL-standard or PostgreSQL equivalent.
Honest note about legacy literals. A few code paths are older than the central header. They still emit hard-coded literals in PostgreSQL's P style instead of the E scheme. One is 42P01 for undefined table. It shows up in some CREATE paths and in the PG protocol fallback. Another is 08P01 for protocol violation in the connection handler. A third is 45000. That is the SQL-standard "unhandled user-defined exception" class 45. It is the default for the SIGNAL statement. These literals are PostgreSQL-compatible on purpose so existing drivers know them. They live next to the canonical E-scheme codes in evosql_errcodes.h.

A.3. Default and fallback codes

Sometimes an operation raises an error but sets no specific SQLSTATE. In that case the adaptor uses a class-level default. It never sends an empty code.

42000, Syntax Error or Access Rule Violation (generic)
The query executor and the EVO protocol use this as the general fallback. They use it when g_err.sqlstate is empty. Many EVO protocol framing errors report 42000 directly. These include a bad command, missing arguments, and an invalid SQL length.
22000, Data Exception (generic)
COPY row parsing uses this fallback. It uses it when no more specific data-exception code is available.
00000, Successful Completion
The start value of a result set's SQLSTATE. It holds before any error happens.

The internal error handlers. evolution/db/error.c provides the family of err_* functions. In server mode they do not call exit(). Instead they set a SQLSTATE and run longjmp back to the executor. This way the connection stays alive. System-call failures set 58000 for SYSTEM_ERROR. Those are err_sys and err_dump. Generic fatal errors set EX000 for INTERNAL_ERROR. Those are err_exit and err_quit. If you see EX000 in a client, the engine treats it as a bug. Please report it.

A.6. Differences from PostgreSQL and the SQL standard

EvolutionDB works with PostgreSQL clients on purpose. But it does not copy PostgreSQL's full error-code catalog. Here are the honest differences.

  • The vendor marker is E, not P. PostgreSQL emits 42P01, 22P02, 28P01, or 25P01. EvolutionDB's canonical codes are 42E01, 22E02, 28E01, and 25E01. A few legacy paths still emit the PostgreSQL P literals for the best driver compatibility. These are 42P01, 08P01, 40P01, and 55P03.
  • Far fewer codes than PostgreSQL. EvolutionDB defines only the codes it needs. Large parts of PostgreSQL's catalog are missing. These include PL/pgSQL classes, configuration-file errors, and FDW errors.
  • Vendor classes E0 to EX are unique to EvolutionDB. They describe the page-based storage engine, the buffer pool, RECLAIM, and ID generation. A portable application should not rely on these except for logging.
  • No message catalog. Some systems map an SQLSTATE to a fixed message. EvolutionDB does not. It sets the code and the message on their own at each error site. For code that handles errors, match on the SQLSTATE code. Never match on the message text.
Notes and limitations.
  • Several codes in evosql_errcodes.h are reserved. They are defined for completeness and future use. The engine does not set them yet. Treat the "emitted" notes above as the current, real behavior. The reserved codes may start to appear in future releases.
  • The active code is stored in one global field g_err.sqlstate. The executor mutex guards it. It is cleared at the start of each statement. It shows only the most recent error on that connection.
  • When no specific code is set, clients see 42000 as the general code. Inside COPY they see 22000. They never see an empty string.
  • The PostgreSQL wire protocol always puts the code in the C field. The EVO protocol always puts it first. It is the first token of ERR … and BATCH_ERR … lines.
