Connect (PostgreSQL wire & EVO)

A single EvolutionDB server speaks two protocols at once: the PostgreSQL wire protocol on :5433 (so psql, DBeaver and any PG driver just work) and the native EVO text protocol on :9967 (used by evosql-cli). This page covers how to reach the server on each, the default credentials and the environment variables that control them, and how to switch the active database with USE.

Two listeners, one server

On startup the adaptor (adaptor/evosql-server) spawns two listener threads and prints a banner with the live ports. The PG listener runs on the main thread; the EVO listener runs in a background thread.

# start the server (local build)
./adaptor/evosql-server
#   PG  protocol on port 5433
#   EVO protocol on port 9967

Both default ports are configurable on the command line. A bare numeric argument is treated as the PG port for backward compatibility.

evosql-server [--pg-port PORT] [--evo-port PORT]
# move the PG listener to 5434, keep EVO on 9967
./adaptor/evosql-server --pg-port 5434

# backward-compatible: a bare number sets the PG port
./adaptor/evosql-server 5500
info
The defaults are DEFAULT_PG_PORT = 5433 and DEFAULT_EVO_PORT = 9967. The startup banner echoes the resolved ports and whether TLS is enabled, so check the log if you are unsure which port a given instance is on.

Default credentials & environment

The initial administrator user is created from two environment variables, read once when the database is first initialized:

  • EVOSQL_USER_NAME — the admin username. Defaults to admin when unset or empty.
  • EVOSQL_PASSWORD — the admin password. If unset or empty, the server generates a random 16-character password and prints it to stdout once (MySQL --initialize style).

When no password is supplied, the server prints a one-time banner like this and never shows it again:

============================================================
  GENERATED PASSWORD for 'admin': <random-16-chars>
  Save this password! It will NOT be shown again.
  Change it with: ALTER USER admin PASSWORD '<newpass>'
============================================================

To get predictable credentials for development, set both variables before the first run:

export EVOSQL_USER_NAME=admin
export EVOSQL_PASSWORD=admin
./adaptor/evosql-server
warning
These variables seed the admin user only on first initialization. Once the user exists in the catalog, changing the env vars has no effect — rotate the password with ALTER USER <name> PASSWORD '<newpass>' instead.

PostgreSQL wire — psql & DBeaver (:5433)

The PG listener implements the PostgreSQL v3 wire protocol — startup handshake, simple query, and the extended query protocol (Parse / Bind / Describe / Execute / Sync). That means standard PostgreSQL clients connect without any EvolutionDB-specific driver. Authentication is cleartext-password based, so always supply a username and password.

# psql — note the explicit port
psql -h 127.0.0.1 -p 5433 -U admin -d testdb

# or as a connection URI
psql "postgresql://admin@127.0.0.1:5433/testdb"

For DBeaver (or pgAdmin, or any JDBC tool), create a new PostgreSQL connection and fill in the fields:

  • Host: 127.0.0.1
  • Port: 5433
  • Username: admin (or your EVOSQL_USER_NAME)
  • Password: your EVOSQL_PASSWORD

The adaptor fakes enough of pg_catalog and information_schema for GUI tools to introspect the schema tree, and named prepared statements are retained across Bind/Execute re-use (so driver auto-prepare and JDBC work). If the client opens a transaction and then disconnects, the server auto-rolls it back.

info
TLS on the PG wire is enabled only when the server is started with both EVOSQL_TLS_CERT and EVOSQL_TLS_KEY set; otherwise the banner reports TLS: disabled.

Native EVO text protocol (:9967)

EVO is a simple, line-oriented text protocol — easy to script over a raw socket and the transport that evosql-cli uses. A session is a short greeting handshake followed by line-framed queries and multi-line replies.

client → "EVO" · server → "HELLO EvoSQL 1.0" · [optional STARTTLS] · server → "AUTH_SCRAM"

After the greeting the server offers authentication. It advertises AUTH_SCRAM to request SCRAM-SHA-256, but it also accepts a legacy cleartext AUTH <user> <pass> line for backward compatibility; on success it replies AUTH_OK. Queries are then sent as a two-line frame:

# wire frame for one statement
SQL <byte-length>
<the SQL text on one line>

The server streams the reply as keyword lines — OK, TAG <text> for command tags, RESULT / COLS n / COL … / ROW / FIELD … / END for result sets, ERR <sqlstate> <message> for errors — terminated by a final READY line. You rarely write this by hand: use the CLI below.

The evosql-cli client

cli/evosql-cli is the interactive shell for the EVO protocol. With no arguments it connects to localhost:9967 as user admin and prompts for a password.

evosql-cli [-h HOST] [-p PORT] [-u USER] [-W PASSWORD] [-c SQL | -f FILE]
# interactive shell, prompts for the password
./cli/evosql-cli

# pass the password on the command line
./cli/evosql-cli -W admin

# or via the environment (same var the server reads)
export EVOSQL_PASSWORD=admin
./cli/evosql-cli

# a different host / port / user
./cli/evosql-cli -h db.internal -p 9967 -u admin

The CLI defaults are host=localhost, port=9967, user=admin. The password comes from -W/--password, else the EVOSQL_PASSWORD environment variable, else an interactive prompt with echo disabled. Command history is kept in ~/.evosql_history (last 500 lines); statements containing a PASSWORD keyword are deliberately excluded from the on-disk history.

It is also scriptable for one-off commands and pipelines:

# run one statement and exit
./cli/evosql-cli -c "SELECT 1"

# run a SQL file
./cli/evosql-cli -f schema.sql

# pipe SQL on stdin
./cli/evosql-cli < seed.sql

# machine-readable output
./cli/evosql-cli --format json -c "SELECT id, name FROM users"

Inside the interactive shell, end statements with ; and use backslash meta-commands — for example \dt (list tables), \l (list databases), \du (list users), \dn (list schemas), \d NAME (describe a table), \format table|csv|json, \timing, \x, \i FILE, and \q to quit. Output formats are table (default), csv, and json, selectable with --format or \format.

info
The EVO protocol has no in-band query cancel. Pressing Ctrl-C while a query is running abandons the socket and the CLI transparently reconnects, returning you to the prompt — the client itself never exits.

Selecting a database with USE

A fresh session starts on a default database with the schema set to default. To switch the active database for the rest of the session, run USE:

USE <database>;
USE testdb;
SELECT * FROM users;

Switching the active database also resets the active schema back to default. If the named database does not exist, the server raises an undefined database error and the active database is unchanged.

warning
On the PostgreSQL wire you typically pick the database at connect time (the -d flag in psql, or the Database field in DBeaver). USE works on both protocols, but for PG clients selecting the database in the connection string is the more conventional path.
