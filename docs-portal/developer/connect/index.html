Connect (PostgreSQL wire & EVO)

One EvolutionDB server speaks two protocols at the same time. It speaks the PostgreSQL wire protocol on :5433. So psql, DBeaver and any PG driver just work. It also speaks the native EVO text protocol on :9967. The evosql-cli tool uses this one. This page shows how to reach the server on each port. It shows the default login details and the environment variables that control them. It also shows how to switch the active database with USE.

Two listeners, one server

When it starts, the adaptor adaptor/evosql-server opens two listener threads. It prints a banner with the live ports. The PG listener runs on the main thread. The EVO listener runs in a background thread.

# start the server (local build)
./adaptor/evosql-server
#   PG  protocol on port 5433
#   EVO protocol on port 9967

You can change both default ports on the command line. A plain number on its own is read as the PG port. This keeps old commands working.

evosql-server [--pg-port PORT] [--evo-port PORT]
# move the PG listener to 5434, keep EVO on 9967
./adaptor/evosql-server --pg-port 5434

# backward-compatible: a bare number sets the PG port
./adaptor/evosql-server 5500
info
The defaults are DEFAULT_PG_PORT = 5433 and DEFAULT_EVO_PORT = 9967. The startup banner prints the chosen ports and whether TLS is on. Check the log if you are not sure which port an instance uses.

Default credentials & environment

The first admin user is built from two environment variables. The server reads them once, when the database is first set up.

  • EVOSQL_USER_NAME is the admin username. It defaults to admin when it is not set or is empty.
  • EVOSQL_PASSWORD is the admin password. If it is not set or is empty, the server makes a random 16-character password. It prints this password to stdout once. This works like MySQL --initialize.

When you give no password, the server prints a one-time banner like this. It never shows it again.

============================================================
  GENERATED PASSWORD for 'admin': <random-16-chars>
  Save this password! It will NOT be shown again.
  Change it with: ALTER USER admin PASSWORD '<newpass>'
============================================================

To get fixed login details for development, set both variables before the first run.

export EVOSQL_USER_NAME=admin
export EVOSQL_PASSWORD=admin
./adaptor/evosql-server
warning
These variables set up the admin user only on first init. Once the user is in the catalog, changing the env vars does nothing. Change the password with ALTER USER <name> PASSWORD '<newpass>' instead.

PostgreSQL wire, psql & DBeaver (:5433)

The PG listener supports the PostgreSQL v3 wire protocol. This covers the startup handshake, the simple query, and the extended query protocol with Parse, Bind, Describe, Execute and Sync. So normal PostgreSQL clients connect with no EvolutionDB driver. Login uses a cleartext password. Always send a username and password.

# psql - note the explicit port
psql -h 127.0.0.1 -p 5433 -U admin -d testdb

# or as a connection URI
psql "postgresql://admin@127.0.0.1:5433/testdb"

For DBeaver, pgAdmin, or any JDBC tool, make a new PostgreSQL connection. Fill in the fields.

  • Host: 127.0.0.1
  • Port: 5433
  • Username: admin (or your EVOSQL_USER_NAME)
  • Password: your EVOSQL_PASSWORD

The adaptor fakes enough of pg_catalog and information_schema for GUI tools to read the schema tree. Named prepared statements are kept across Bind and Execute re-use. So driver auto-prepare and JDBC work. If the client opens a transaction and then disconnects, the server rolls it back on its own.

info
TLS on the PG wire is on only when the server starts with both EVOSQL_TLS_CERT and EVOSQL_TLS_KEY set. If not, the banner reports TLS: disabled.

Native EVO text protocol (:9967)

EVO is a simple text protocol that works line by line. It is easy to script over a raw socket. It is the transport that evosql-cli uses. A session starts with a short greeting handshake. After that come line-framed queries and multi-line replies.

client → "EVO" · server → "HELLO EvoSQL 1.0" · [optional STARTTLS] · server → "AUTH_SCRAM"

After the greeting the server offers login. It sends AUTH_SCRAM to ask for SCRAM-SHA-256. It also accepts an old cleartext AUTH <user> <pass> line to keep older clients working. On success it replies AUTH_OK. You then send queries as a two-line frame.

# wire frame for one statement
SQL <byte-length>
<the SQL text on one line>

The server streams the reply as keyword lines. It sends OK for success. It sends TAG <text> for command tags. It sends RESULT / COLS n / COL … / ROW / FIELD … / END for result sets. It sends ERR <sqlstate> <message> for errors. A final READY line ends the reply. You rarely write this by hand. Use the CLI below.

The evosql-cli client

cli/evosql-cli is the interactive shell for the EVO protocol. With no arguments it connects to localhost:9967 as user admin. It then asks you for a password.

evosql-cli [-h HOST] [-p PORT] [-u USER] [-W PASSWORD] [-c SQL | -f FILE]
# interactive shell, prompts for the password
./cli/evosql-cli

# pass the password on the command line
./cli/evosql-cli -W admin

# or via the environment (same var the server reads)
export EVOSQL_PASSWORD=admin
./cli/evosql-cli

# a different host / port / user
./cli/evosql-cli -h db.internal -p 9967 -u admin

The CLI defaults are host=localhost, port=9967 and user=admin. The password comes from -W or --password first. If that is missing, it comes from the EVOSQL_PASSWORD environment variable. If that is also missing, the CLI asks you with echo turned off. Command history is kept in ~/.evosql_history and holds the last 500 lines. Statements that contain the PASSWORD keyword are kept out of the on-disk history on purpose.

You can also script it for one-off commands and pipelines.

# run one statement and exit
./cli/evosql-cli -c "SELECT 1"

# run a SQL file
./cli/evosql-cli -f schema.sql

# pipe SQL on stdin
./cli/evosql-cli < seed.sql

# machine-readable output
./cli/evosql-cli --format json -c "SELECT id, name FROM users"

Inside the interactive shell, end each statement with ;. You can also use backslash meta-commands. For example, \dt lists tables. \l lists databases. \du lists users. \dn lists schemas. \d NAME describes a table. There are also \format table|csv|json, \timing, \x and \i FILE. Use \q to quit. The output formats are table, csv and json. The default is table. Pick a format with --format or \format.

info
The EVO protocol has no in-band query cancel. If you press Ctrl-C while a query runs, the CLI drops the socket. It then reconnects on its own and brings you back to the prompt. The client itself never exits.

Selecting a database with USE

A new session starts on a default database. The schema is set to default. To switch the active database for the rest of the session, run USE.

USE <database>;
USE testdb;
SELECT * FROM users;

Switching the active database also resets the active schema back to default. If the named database does not exist, the server raises an undefined database error. The active database stays the same.

warning
On the PostgreSQL wire you usually pick the database when you connect. You use the -d flag in psql or the Database field in DBeaver. USE works on both protocols. But for PG clients, picking the database in the connection string is the more common way.
