PII guard

This tool finds, masks, and encrypts sensitive data. It runs before the data is saved to EvolutionDB long-term memory. It also runs before any data leaves the host through a public LLM. This is the evolutiondb-pii Python package. Every *-sync connector calls it before a MEMORY PUT.

What it does

For each record, the library makes two versions of the same text.

  • a masked copy. This is what the row stores by default. It is also what the MCP server search path gives to LLMs (***@example.com, 4111 1111 **** 1111, Me***et Yı***ız, …);
  • a cipher_tokens list. This holds the original sensitive text pieces. Each piece is encrypted with AES-256-GCM. Each piece uses a fresh key. That key is then wrapped with RSA-OAEP (SHA-256), 2048-bit. To recover the data you need the private key. By default the private key never leaves ~/.evosql/pii_private.pem.

No plaintext is stored next to the masked record. No PII reaches an outside LLM through the normal search path. A small CLI command restores the originals when the operator has the private key.

Here is why the library uses hybrid encryption instead of plain RSA. Raw RSA-2048 can only hold about 245 bytes of plaintext. That is too small for an email body or a Notes entry. The library uses the same shape that TLS, S/MIME, and PGP use. It picks a random 256-bit AES key for each token. It wraps that key with RSA-OAEP-SHA256. AES-256-GCM then encrypts the bulk of the data. The triple (wrapped_key, iv, ciphertext+tag) goes into the row.

Install

# from PyPI
pip install evolutiondb-pii

# or editable, from the monorepo
pip install -e client/evolutiondb-pii

The install name is evolutiondb-pii. The import package is evolutiondb_pii. The runtime dependencies come from pyproject.toml. They are cryptography>=42 and psycopg[binary]>=3.1. The second one is only used by the DB-backed keystore, the DB rule store, and the row-decrypt/backfill commands. These all import it only when needed. It requires Python 3.9+. The console script evolutiondb-pii maps to evolutiondb_pii.__main__:main. The current version is 0.3.0 (Beta).

Configure (keys & env vars)

The keypair can live on the filesystem by default. It can also live in a DB row that is wrapped with a passphrase. Pick one with a single keygen run.

File-backed (default)

evolutiondb-pii keygen
# {"ok": true, "store": "file",
#  "public": "~/.evosql/pii_public.pem",
#  "private": "~/.evosql/pii_private.pem", "bits": "2048"}

The private file is written with mode 0600. Pass --passphrase to encrypt the PEM on disk. It then asks you for the passphrase. If you do not type one, it uses EVOSQL_PII_PASSPHRASE. keygen will not overwrite an existing keypair unless you pass --force. A new keypair makes every token that was encrypted under the old public key unreadable.

DB-backed (multi-host / shared backup)

evolutiondb-pii keygen --store=db
# Private key passphrase: ********
# {"ok": true, "store": "db", "namespace": "...",
#  "key_id": "<uuid4>", "created_at": "...Z"}

The keypair is wrapped before it is saved. First PBKDF2-SHA256 (100,000 iterations) makes a 32-byte key. Then AES-256-GCM encrypts the keypair. It is saved as a single row in the __mem_pii_keystore memory store. The store name is pii_keystore and the key is default. The row holds public_pem in the clear plus the wrapped private key. Only the passphrase can unwrap it. So a DB snapshot on its own is useless. The row's namespace is the MCP_USER_ID value. The default is default_user. This keeps each tenant's keypair separate.

You can move between storage modes without rotating. Every existing cipher token stays readable.

evolutiondb-pii keystore-import   # ~/.evosql/pii_*.pem  →  DB
evolutiondb-pii keystore-export   # DB  →  ~/.evosql/pii_*.pem

Custom rules in the DB

Rules can also live in the DB. There is one row per rule in the __mem_pii_rules store. The store name is pii_rules. Each row is keyed by the rule name.

evolutiondb-pii rules-push my_rules.json --replace
evolutiondb-pii rules-list
evolutiondb-pii rules-pull > /tmp/dump.json
export EVOSQL_PII_RULES=db

DB rules work the same way as file overrides. They merge on top of the bundled defaults by name.

Environment matrix

VariableMeaning
EVOSQL_PII_KEYSTORETells the library where the keypair lives. The values are auto (default), file, or db. auto uses a DB row when one exists. If there is none, it uses the filesystem PEM.
EVOSQL_PII_RULESA path to a JSON override file. Or the value db to read rules from __mem_pii_rules. If it is not set, the library uses ~/.evosql/pii_rules.json when that file exists.
EVOSQL_PII_PASSPHRASEThe private-key passphrase for non-interactive use, such as CI or Vault flows. When it is set, CLI passphrase prompts read it instead of calling getpass.
EVOSQL_PII_MASKon (default) turns on the integration engine. It fails loudly if no key is available. auto turns it on only when a key exists. off turns masking off and passes data through unchanged.
EVOSQL_PII_PUBLIC_KEYAn override path to the public PEM. It is used by encrypt and by the integration layer's file fallback.
MCP_USER_IDThe namespace for the DB keystore and rules rows. The default is default_user.

The row-decrypt and backfill commands open a psycopg connection. They read the connection settings from these variables. EVOSQL_HOST (127.0.0.1), EVOSQL_PORT (5433), EVOSQL_USER (admin), EVOSQL_PASSWORD (admin), and EVOSQL_DATABASE (evosql).

Usage

Python, quick tour

from evolutiondb_pii import encrypt_record, decrypt_record, mask

# Just see the mask (runs detection inline)
mask("Kart 4111 1111 1111 1111 sahibi mehmet@example.com")
# → 'Kart 4111 1111 **** 1111 sahibi ***@example.com'

# Full pipeline - what a sync client does
record = {"text": "Mehmet Yıldız, kart 4111 1111 1111 1111"}
masked, tokens = encrypt_record(record, fields=["text"])
# masked["text"] → "Me***et Yı***ız, kart 4111 1111 **** 1111"
# tokens         → [{"id":"t000001","tag":"credit_card", ...}, ...]

# Round-trip (private key needed)
restored = decrypt_record(masked, tokens)
# restored["text"] == record["text"]

encrypt_record makes a shallow copy of the record. So the caller's original dict is not changed. If you give no public_key argument, it loads ~/.evosql/pii_public.pem. If you give no rules argument, it loads the merged default and user rule set.

Python, sync-client integration

This is the one call a connector makes. It resolves the engine once. The engine is controlled by env vars and cached per process. It masks the named fields. It splits the cipher tokens into companion rows. This keeps a large record from going past the per-statement wire limit.

from evolutiondb_pii.integration import protect_record

rec, companions = protect_record(rec, fields=_PII_FIELDS,
                                 key_prefix=f"outlook_pii_{msg_id}")
store.put_record(main_key, rec)
for ck, cv in companions:
    store.put_record(ck, cv)

The masked record gets a _pii_refs list. This list points at its companion keys (f"{key_prefix}_{i}"). Each companion value is {"version":1, "of":source, "chunk":i, "tokens":[...]}. When EVOSQL_PII_MASK=off, protect_record does nothing and returns (rec, []). The same happens in auto mode when no key can be found. So callers do not need a feature-flag branch.

CLI

Use - for FILE to read from stdin. Commands that touch the DB read their connection settings from the EVOSQL_* env vars above.

evolutiondb-pii keygen [--store=file|db] [--passphrase] [--force]
evolutiondb-pii keystore-import        # filesystem keypair → DB
evolutiondb-pii keystore-export        # DB keypair → filesystem PEMs
evolutiondb-pii rules-push FILE [--replace]
evolutiondb-pii rules-pull             # dump DB rules as JSON
evolutiondb-pii rules-list [--json]    # name/tag/priority/enabled table
evolutiondb-pii scan FILE [--rules-file FILE]      # masked rendering
evolutiondb-pii encrypt FILE           # emit {masked, cipher_tokens} JSON
evolutiondb-pii decrypt FILE [--keystore=file|db]  # restore plaintext
evolutiondb-pii row-decrypt --namespace=NS --key=K [--store=...] [--keystore=...]
evolutiondb-pii backfill --namespace=NS [--store=...] [--key-prefix=...] [--dry-run]

All subcommands share a --namespace flag. It defaults to MCP_USER_ID. For row-decrypt and backfill, --store defaults to mcp_mem. On any uncaught error the CLI prints {"ok": false, "error": "..."} to stderr and exits with a non-zero code.

Reference

These names are re-exported from evolutiondb_pii/__init__.py. They are detect, mask, encrypt_record, decrypt_record, generate_keypair, load_rules, and protect_record. The Match and Rule dataclasses are also exported.

detect(text: str, rules: list[Rule] | None = None) -> list[Match]

Scan text and return Match records that do not overlap. They come back in start order. It runs every loaded rule's regex. It keeps the spans that the validator approves. Then it resolves overlaps by priority, from high to low. When two have the same priority, it picks the earliest start, then the longest match. It defaults to the merged default and user rule set. It is bounded by the engine limits. max_field_chars_scanned defaults to 65536. max_matches_per_field defaults to 200.

mask(text: str, matches=None, rules=None) -> str

Render text. Every detected match is replaced by its rule's masked form. Pass matches that you already detected to save work. Leave them out to detect inline. An unknown mask kind falls back to full-star. So a typo in the config never leaks data.

encrypt_record(record: dict, fields: list[str], public_key=None, rules=None) -> tuple[dict, list[dict]]

Walk through fields. Mask and encrypt every detected span. Return (masked_record, cipher_tokens). Token offsets are measured against the masked string. It loads the public key from disk when public_key is None.

decrypt_record(masked_record: dict, cipher_tokens: list[dict], private_key=None, passphrase: bytes | None = None) -> dict

This is the reverse of encrypt_record. It returns a new dict. The targeted fields hold the original strings, rebuilt from cipher_tokens. It loads the private PEM when private_key is None. It can decrypt that PEM with passphrase.

protect_record(rec, fields, key_prefix, *, source=None, env_var="EVOSQL_PII_MASK") -> tuple[dict, list[tuple[str, dict]]]

This is the high-level connector helper in evolutiondb_pii.integration. It masks the named fields. It splits cipher tokens into companion chunks of about 5,500 bytes. It stamps the masked record with _pii_refs. It returns (masked_rec, [(row_key, row_value), ...]). When protection is off, it does nothing and returns (rec, []).

generate_keypair(public_path="~/.evosql/pii_public.pem", private_path="~/.evosql/pii_private.pem", bits=2048, passphrase=None) -> dict

Generate a fresh RSA keypair and save both halves. The keys use PKCS#8 PEM and public exponent 65537. The private file is set to mode 0600. It rejects bits < 2048. With a passphrase, the PEM is encrypted with the library's best available algorithm. It returns the paths it wrote.

load_rules(user_path: str | None = None) -> list[Rule]

Return the merged list of enabled rules, sorted by priority. The bundled data/default_rules.json comes first. Then the user override is laid on top by name. The override comes from an explicit path, from EVOSQL_PII_RULES, or from ~/.evosql/pii_rules.json. Matching names replace fields one by one. New names are added to the end. load_limits() is the sibling function that returns the scan guardrails.

@dataclass Match(start, end, text, rule, tag)

This is one detected PII fragment. It holds the half-open span inside the source string. It holds the covered substring. It holds the matching rule name. It holds the semantic tag.

@dataclass Rule(name, enabled, priority, tag, pattern, validate, mask: MaskSpec)

This is a single PII rule. __post_init__ compiles pattern into compiled. This keeps detection fast on the hot path. mask is a MaskSpec. It selects one of the rendering kinds below.

Mask kinds

KindKey fieldsWhat it does
positionalkeep_first, keep_last, preserve_sepStars the middle. Keeps the head and tail. preserve_sep leaves spaces and dashes visible. This suits cards, IBANs, and plates.
keep_afterseparator, head_starsStars the head. Keeps everything from separator onward. This suits the email shape.
keep_beforeseparatorThe mirror of keep_after.
name_smartshort_thresholdWorks on each whitespace token. Long tokens keep the first 2 and last 2 characters. Short ones keep the first and last. It keeps the whitespace.
full_star, Stars that keep the length.
templatetemplate, extractRenders a template. url_userinfo strips user:pass from a URL. literal echoes the template with no change.

Validators

These are checks that run after detection. They live in validators.py. They drop false positives that the regex cannot catch. An unknown validator name accepts the span. So a typo never quietly drops a span.

NameUse
luhnChecks the credit or debit card check digit. The card has 13 to 19 digits.
tckn_checksumChecks the Turkish citizen ID (TC Kimlik). It has 11 digits with two check digits at the end.
vkn_checksumChecks the Turkish tax number. It has 10 digits.
iban_mod97Checks an IBAN by ISO 13616.
entropy_gte_4_0Rejects long strings with low entropy. The runtime threshold is 3.8 bits per symbol. The 4.0 in the name is from an older version.
person_name_heuristicTrims runs of title-cased words that are clearly not names. This is best-effort only.

Default rule set (v1)

The bundled default_rules.json ships these rules. They are email, credit_card, tckn, iban, phone_tr, phone_intl, vkn, passport_generic, plate_tr, api_token_long, high_entropy_blob, jwt, and url_with_credentials. date_of_birth and person_name_tr are off by default. You turn them on yourself. A rule looks like this.

{
  "name":     "credit_card",
  "enabled":  true,
  "priority": 95,
  "tag":      "credit_card",
  "pattern":  "\\b(?:\\d[ \\-]?){12,18}\\d\\b",
  "validate": "luhn",
  "mask": { "kind": "positional", "keep_first": 8,
             "keep_last": 4, "preserve_sep": true }
}

Notes & limitations

  • Names need real NER. The default person_name_tr rule uses heuristics only. It ships turned off. Reliable name detection needs an LLM or a proper NER pass. That comes in a later phase.
  • Validators are conservative. They prefer to drop a span they are unsure about rather than leak it. A clearly valid PII value that fails a checksum will look like normal text.
  • No in-place re-encrypt or rotation today. A fresh keygen makes a new public key. But cipher tokens that used the old public key must be decrypted and then encrypted again under the new one. The README roadmap lists keygen --rotate, a rekey command, and a decrypt_only legacy-key mode as future work in Phase 6. These are not in 0.3.0.
  • The DB-backed keystore needs the passphrase only at decrypt time. Sync workers run on their own with just the public key. They never need the private one.
  • SQL string handling. store_io writes rows by escaping quotes in the values inside MEMORY PUT and DELETE statements. It does not use bind parameters. It does a DELETE then a PUT for an upsert. This avoids an old duplicate-key issue with large values. Memory stores set themselves up. A missing store is created on the first write.
  • The masked text is also the only PII a search query sees. So the quality of the masking sets the limit on what an LLM can ever recover from the normal search path. The matching engine and the SQL surface behind MEMORY PUT are documented in the code wiki.
