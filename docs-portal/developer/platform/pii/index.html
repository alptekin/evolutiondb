PII guard

Detect, mask, and hybrid-encrypt sensitive data before it lands in EvolutionDB long-term memory or leaves the host via a public LLM. This is the evolutiondb-pii Python package — the bridge every *-sync connector calls before a MEMORY PUT.

What it does

For a given record, the library produces two parallel renderings of the same text:

  • a masked copy — what the row stores by default and what the MCP server's search path hands to LLMs (***@example.com, 4111 1111 **** 1111, Me***et Yı***ız, …);
  • a cipher_tokens list — the original sensitive substrings, each encrypted with AES-256-GCM under a fresh per-token key that is itself wrapped with RSA-OAEP (SHA-256), 2048-bit. Recovery needs the private key, which by default never leaves ~/.evosql/pii_private.pem.

No plaintext is stored alongside the masked record, and no PII reaches an external LLM through the standard search path. A one-shot CLI restores the originals when the operator holds the private key.

Why hybrid rather than plain RSA: raw RSA-2048 caps a single plaintext at roughly 245 bytes, too small for an email body or a Notes entry. The library follows the same shape TLS / S/MIME / PGP use — a random 256-bit AES key per token, wrapped with RSA-OAEP-SHA256, with AES-256-GCM doing the bulk encryption. The triple (wrapped_key, iv, ciphertext+tag) lands in the row.

Install

# from PyPI
pip install evolutiondb-pii

# or editable, from the monorepo
pip install -e client/evolutiondb-pii

Install name is evolutiondb-pii; the import package is evolutiondb_pii. Runtime dependencies (from pyproject.toml) are cryptography>=42 and psycopg[binary]>=3.1 — the latter is only exercised by the DB-backed keystore, DB rule store, and the row-decrypt/backfill commands, all of which import it lazily. Requires Python 3.9+. The console script evolutiondb-pii maps to evolutiondb_pii.__main__:main. Current version: 0.3.0 (Beta).

Configure (keys & env vars)

The keypair can live on the filesystem (default) or in a passphrase-wrapped DB row. Pick one with a one-time keygen.

File-backed (default)

evolutiondb-pii keygen
# {"ok": true, "store": "file",
#  "public": "~/.evosql/pii_public.pem",
#  "private": "~/.evosql/pii_private.pem", "bits": "2048"}

The private file is written with mode 0600. Pass --passphrase to encrypt the PEM at rest (it then prompts, falling back to EVOSQL_PII_PASSPHRASE). keygen refuses to overwrite an existing keypair unless you pass --force — and a fresh keypair makes every token encrypted under the old public key unreadable.

DB-backed (multi-host / shared backup)

evolutiondb-pii keygen --store=db
# Private key passphrase: ********
# {"ok": true, "store": "db", "namespace": "...",
#  "key_id": "<uuid4>", "created_at": "...Z"}

The keypair is wrapped with PBKDF2-SHA256 (100,000 iterations) deriving a 32-byte key, then AES-256-GCM, before it lands as a single row in the __mem_pii_keystore memory store (store name pii_keystore, key default). The row carries public_pem in the clear plus the wrapped private key; the passphrase is the only thing that can unwrap it, so a DB snapshot alone is useless. The row's namespace is the MCP_USER_ID value (default default_user), which isolates keypairs per tenant.

To move between storage modes without rotating (every existing cipher token stays decryptable):

evolutiondb-pii keystore-import   # ~/.evosql/pii_*.pem  →  DB
evolutiondb-pii keystore-export   # DB  →  ~/.evosql/pii_*.pem

Custom rules in the DB

Rules can also live in the DB — one row per rule in the __mem_pii_rules store (store name pii_rules), keyed by rule name:

evolutiondb-pii rules-push my_rules.json --replace
evolutiondb-pii rules-list
evolutiondb-pii rules-pull > /tmp/dump.json
export EVOSQL_PII_RULES=db

DB rules layer the same way the file overrides do — they merge over the bundled defaults by name.

Environment matrix

VariableMeaning
EVOSQL_PII_KEYSTOREauto (default), file, or db — tells the library where the keypair lives. auto prefers a DB row when one exists and falls back to the filesystem PEM.
EVOSQL_PII_RULESA path to a JSON override file, or the literal db to read rules from __mem_pii_rules. Unset uses ~/.evosql/pii_rules.json if present.
EVOSQL_PII_PASSPHRASENon-interactive private-key passphrase (CI / Vault flows). When set, CLI passphrase prompts read this instead of getpass.
EVOSQL_PII_MASKon (default) forces the integration engine and errors loudly if no key is available; auto enables only when a key exists; off disables masking entirely (passthrough).
EVOSQL_PII_PUBLIC_KEYOverride path to the public PEM (used by encrypt and the integration layer's file fallback).
MCP_USER_IDNamespace for the DB keystore + rules rows (default default_user).

The row-decrypt and backfill commands open a psycopg connection from the standard EVOSQL_HOST (127.0.0.1), EVOSQL_PORT (5433), EVOSQL_USER (admin), EVOSQL_PASSWORD (admin), EVOSQL_DATABASE (evosql) variables.

Usage

Python — quick tour

from evolutiondb_pii import encrypt_record, decrypt_record, mask

# Just see the mask (runs detection inline)
mask("Kart 4111 1111 1111 1111 sahibi mehmet@example.com")
# → 'Kart 4111 1111 **** 1111 sahibi ***@example.com'

# Full pipeline — what a sync client does
record = {"text": "Mehmet Yıldız, kart 4111 1111 1111 1111"}
masked, tokens = encrypt_record(record, fields=["text"])
# masked["text"] → "Me***et Yı***ız, kart 4111 1111 **** 1111"
# tokens         → [{"id":"t000001","tag":"credit_card", ...}, ...]

# Round-trip (private key needed)
restored = decrypt_record(masked, tokens)
# restored["text"] == record["text"]

encrypt_record shallow-copies the record, so the caller's original dict stays intact. With no public_key argument it loads ~/.evosql/pii_public.pem; with no rules argument it loads the merged default + user rule set.

Python — sync-client integration

The one call a connector makes; it resolves the engine once (env-gated, cached per process), masks the named fields, and splits the cipher tokens into companion rows so a single oversize record never blows past the per-statement wire limit:

from evolutiondb_pii.integration import protect_record

rec, companions = protect_record(rec, fields=_PII_FIELDS,
                                 key_prefix=f"outlook_pii_{msg_id}")
store.put_record(main_key, rec)
for ck, cv in companions:
    store.put_record(ck, cv)

The masked record gains a _pii_refs list pointing at its companion keys (f"{key_prefix}_{i}"). Each companion value is {"version":1, "of":source, "chunk":i, "tokens":[...]}. When EVOSQL_PII_MASK=off (or no key is resolvable in auto mode), protect_record is a no-op returning (rec, []) — callers need no feature-flag branch.

CLI

FILE is - for stdin. Commands that touch the DB read connection settings from the EVOSQL_* env vars above.

evolutiondb-pii keygen [--store=file|db] [--passphrase] [--force]
evolutiondb-pii keystore-import        # filesystem keypair → DB
evolutiondb-pii keystore-export        # DB keypair → filesystem PEMs
evolutiondb-pii rules-push FILE [--replace]
evolutiondb-pii rules-pull             # dump DB rules as JSON
evolutiondb-pii rules-list [--json]    # name/tag/priority/enabled table
evolutiondb-pii scan FILE [--rules-file FILE]      # masked rendering
evolutiondb-pii encrypt FILE           # emit {masked, cipher_tokens} JSON
evolutiondb-pii decrypt FILE [--keystore=file|db]  # restore plaintext
evolutiondb-pii row-decrypt --namespace=NS --key=K [--store=...] [--keystore=...]
evolutiondb-pii backfill --namespace=NS [--store=...] [--key-prefix=...] [--dry-run]

All subcommands share a --namespace flag (defaults to MCP_USER_ID). --store defaults to mcp_mem for row-decrypt / backfill. On any uncaught error the CLI prints {"ok": false, "error": "..."} to stderr and exits non-zero.

Reference

Public surface re-exported from evolutiondb_pii/__init__.py: detect, mask, encrypt_record, decrypt_record, generate_keypair, load_rules, protect_record, plus the Match and Rule dataclasses.

detect(text: str, rules: list[Rule] | None = None) -> list[Match]

Scan text and return non-overlapping Match records in start order. Runs every loaded rule's regex, keeps validator-approved spans, then resolves overlaps by descending priority (ties: earliest start, then longest match). Defaults to the merged default + user rule set. Bounded by the engine limits (max_field_chars_scanned, default 65536; max_matches_per_field, default 200).

mask(text: str, matches=None, rules=None) -> str

Render text with every detected match replaced by its rule's masked form. Pass pre-detected matches to share work; omit them to detect inline. An unknown mask kind falls back to full-star so a config typo never leaks data.

encrypt_record(record: dict, fields: list[str], public_key=None, rules=None) -> tuple[dict, list[dict]]

Walk fields, mask + encrypt every detected span, and return (masked_record, cipher_tokens). Token offsets are computed against the masked string. Loads the public key from disk when public_key is None.

decrypt_record(masked_record: dict, cipher_tokens: list[dict], private_key=None, passphrase: bytes | None = None) -> dict

Reverse of encrypt_record. Returns a new dict whose targeted fields hold the original strings reconstructed from cipher_tokens. Loads the private PEM (optionally decrypting it with passphrase) when private_key is None.

protect_record(rec, fields, key_prefix, *, source=None, env_var="EVOSQL_PII_MASK") -> tuple[dict, list[tuple[str, dict]]]

High-level connector helper in evolutiondb_pii.integration. Masks the named fields, splits cipher tokens into ~5,500-byte companion chunks, stamps the masked record with _pii_refs, and returns (masked_rec, [(row_key, row_value), ...]). No-op ((rec, [])) when protection is off.

generate_keypair(public_path="~/.evosql/pii_public.pem", private_path="~/.evosql/pii_private.pem", bits=2048, passphrase=None) -> dict

Generate a fresh RSA keypair (PKCS#8 PEM, public exponent 65537) and persist both halves; the private file is chmod'd 0600. Rejects bits < 2048. With a passphrase the PEM is encrypted with the library's best-available algorithm. Returns the written paths.

load_rules(user_path: str | None = None) -> list[Rule]

Return the merged, priority-sorted list of enabled rules: bundled data/default_rules.json first, then the user override (explicit path, EVOSQL_PII_RULES, or ~/.evosql/pii_rules.json) overlaid by name — matching names replace field-by-field, novel names append. load_limits() is the sibling that returns the scan guardrails.

@dataclass Match(start, end, text, rule, tag)

One detected PII fragment: the half-open span inside the source string, the covered substring, the matching rule name, and the semantic tag.

@dataclass Rule(name, enabled, priority, tag, pattern, validate, mask: MaskSpec)

A single PII rule. __post_init__ compiles pattern into compiled so detection stays hot-path cheap. mask is a MaskSpec selecting one of the rendering kinds below.

Validators

Detection-time post-checks (in validators.py) that drop false positives the regex can't catch. An unknown validator name accepts silently, so a typo never silently drops a span.

NameUse
luhnCredit / debit card check digit (13–19 digits).
tckn_checksumTurkish citizen ID (TC Kimlik), 11 digits, two trailing check digits.
vkn_checksumTurkish tax number, 10 digits.
iban_mod97IBAN per ISO 13616.
entropy_gte_4_0Reject low-entropy long strings (the runtime threshold is 3.8 bits/symbol — the 4.0 in the name is historical).
person_name_heuristicTrim obviously-not-name title-cased word runs. Best-effort only.

Default rule set (v1)

The bundled default_rules.json ships: email, credit_card, tckn, iban, phone_tr, phone_intl, vkn, passport_generic, plate_tr, api_token_long, high_entropy_blob, jwt, and url_with_credentials. date_of_birth and person_name_tr are off by default — opt-in. A rule's shape:

{
  "name":     "credit_card",
  "enabled":  true,
  "priority": 95,
  "tag":      "credit_card",
  "pattern":  "\\b(?:\\d[ \\-]?){12,18}\\d\\b",
  "validate": "luhn",
  "mask": { "kind": "positional", "keep_first": 8,
             "keep_last": 4, "preserve_sep": true }
}

Notes & limitations

  • Names need real NER. The default person_name_tr rule is heuristic-only and ships disabled; reliable name detection needs an LLM or a proper NER pass (a later phase).
  • Validators are conservative. They err on the side of dropping ambiguous spans rather than leaking — a clearly valid PII value that happens to flunk a checksum will look like normal text.
  • No in-place re-encrypt / rotation today. A fresh keygen emits a new public key, but cipher tokens encrypted under the old public key must be decrypted then re-encrypted under the new one. The README's roadmap lists keygen --rotate, a rekey command, and decrypt_only legacy-key mode as future work (Phase 6) — these are not implemented in 0.3.0.
  • DB-backed keystore needs the passphrase only at decrypt time. Sync workers run unattended with just the public key; they never need the private one.
  • SQL string handling. store_io writes rows by quote-escaping values into MEMORY PUT / DELETE statements (no bind parameters), and uses a DELETE-then-PUT upsert to dodge a historical large-value duplicate-key issue. Memory stores are self-bootstrapping — a missing store is created on first write.
  • The masked rendering is also the only PII surface a search query sees, so the masking quality directly bounds what an LLM can ever recover from the standard search path. The matching engine bits and the SQL surface that backs MEMORY PUT are documented in the code wiki.
