evolution-agent

A standalone agent loop over EvolutionDB's MCP tools. It drives the model itself in a plan → act → observe → repeat cycle. So it runs without a host like Claude Desktop. Every tool call still flows through the MCP server. The human-approved send rule stays intact.

What it does

mcp-server-evolutiondb is an MCP server. It runs tools that a host model decides to call. The host model is Claude Desktop or Claude Code. The server has no will of its own. evolution-agent adds that missing will. It calls a model directly. It lets the model choose tools. It runs them. It feeds the results back. It repeats until the model is done.

"Standalone" means host-independent, not model-independent. The reasoning still comes from an LLM. Anthropic is the default. You can inject any provider that exposes a compatible messages.create(...) client. The change is that this process owns the loop instead of Claude Desktop.

Every tool call goes through the MCP server's own _call_tool. So you get RBAC, the append-only audit log, the PII gate, multi-tenant isolation, and the embedded EvolutionDB for free. This package does not re-implement any of it.

info
The send rule (ADR-004) is intact. The agent never delivers anything to the outside world. It may draft and queue a reply with queue_reply. Then a human approves it with approve_send. Delivery stays dry-run unless the operator turns it on. The system prompt states this rule to the model. The outbox enforces the rule in code.

Install

The distribution name on PyPI is evolutiondb-agent. The importable package is evolution_agent. It depends on mcp-server-evolutiondb>=1.16.0. The default model runs through the Anthropic SDK. The SDK is an optional extra.

# the package + the Anthropic SDK extra
pip install evolutiondb-agent[anthropic]
export ANTHROPIC_API_KEY=sk-...

Two console scripts are installed:

  • evolution-agent → evolution_agent.__main__:main is the terminal loop
  • evolution-agent-web → evolution_agent.web:main is the browser chat

It needs Python 3.9 or newer. The license is MIT.

Configure (env vars / auth)

The agent connects to EvolutionDB the same way the MCP server does. It can use the PostgreSQL wire to reach a running instance. Or it can start a local zero-Docker instance with EVOSQL_EMBEDDED=1.

mcp-server-evolutiondb reads the EvolutionDB connection variables and MCP_USER_ID. This package does not read them. evolution-agent just builds MCPServer() and lets the server read the environment.

Usage

Terminal

Pass the request as arguments. Or pipe it on stdin. Tool calls stream to stderr. The final answer goes to stdout.

evolution-agent "what meetings do I have today and who owes me a reply?"
echo "summarize my open loops" | evolution-agent
evolution-agent -v "draft a reply to Alice about the Q3 report"   # show tool I/O

The CLI accepts these flags. They are parsed by hand in __main__.py. Everything else is treated as prompt text.

FlagEffect
--model <id>Set the model for this run. Without it, the agent uses EVOSQL_AGENT_MODEL or the default.
--max-turns <n>Cap the plan→act→observe iterations. Default 12.
-v, --verboseAlso stream each tool's result to stderr, not just the call.
--onceRun one prompt, then exit. This is the default mode. The flag is accepted to make it clear.
-h, --helpPrint the module docstring and exit.

Here are the exit codes. 0 means success. 2 means no prompt was given. 1 means mcp_server_evosql or the Anthropic SDK is missing, or the agent hit an error. If the loop reaches --max-turns without a final answer, it writes a note to stderr.

Browser chat

evolution-agent-web                       # http://127.0.0.1:8800
EVOSQL_EMBEDDED=1 evolution-agent-web

This is a dependency-free single-page chat. It uses the stdlib http.server and Server-Sent Events. There is no web framework. The agent's tool timeline streams live. A "pending approvals" panel shows every draft the agent queued. Clicking Approve calls approve_send. That stays dry-run unless the operator turned on real delivery. The chat needs the anthropic SDK and ANTHROPIC_API_KEY. Without them, the chat endpoint returns a clear, streamed error. There are four HTTP routes. GET / serves the UI. GET /api/chat?q=… streams the run over SSE. GET /api/pending returns queued drafts through list_pending_replies. POST /api/approve takes a JSON {"id": …} body and calls approve_send.

Library

You can inject both the model client and the tool server. So you can unit-test the loop with no network, no API key, and no database. See tests/test_loop.py.

from evolution_agent import AgentLoop
from mcp_server_evosql.server import MCPServer

agent = AgentLoop(MCPServer())             # defaults: Anthropic + the server's tools
result = agent.run("who am I waiting on?")
print(result["final_text"])
for step in result["steps"]:          # every tool call + its result
    print(step["tool"], step["input"])

The repo also ships examples/demo.py. It is a self-contained morning-assistant task. It starts a throwaway local evosql-server if none is running. Then it runs a real multi-step task. The task recalls and saves memory and drafts and queues a reply. If ANTHROPIC_API_KEY is set and anthropic is installed, it uses the real model. If not, it falls back to a scripted model. The tools, memory, and database stay real, but the demo runs offline.

Reference

evolution_agent.__init__ exports the public surface. It exports AgentLoop, DEFAULT_MODEL, DEFAULT_SYSTEM, and build_anthropic_tools.

AgentLoop

class AgentLoop(server=None, *, client=None, model=None, system=None, tools=None, identity=None, max_turns=12, max_tokens=4096, token_budget=None, on_event=None)

This is a plan → act → observe → repeat loop over the MCP server's tools. server only needs a _call_tool(name, args, identity=None) -> dict method. MCPServer provides it, and tests pass a fake. client only needs messages.create(...) -> response. The result must expose .content blocks, .stop_reason, and an optional .usage. The Anthropic SDK matches this. If you leave out server, client, or tools, they are built lazily from mcp_server_evosql and anthropic. So importing the package never needs either one. token_budget is a hard ceiling on total input plus output tokens for a single run. None means unlimited. The turn cap still bounds the call count. on_event(kind, payload) is an optional observer. If it raises an exception, the loop swallows it, so the observer can never break the loop.

AgentLoop.run(user_message: str) -> dict

This drives the conversation to the end and returns a transcript dict. The dict has final_text as a str. It has steps, a list of {tool, input, result} with one entry per executed tool call. It has turns as an int. It has stop_reason as a str. It has usage as {input_tokens, output_tokens, total_tokens}. If one model turn makes parallel tool calls, the loop runs them all. It feeds them back as a single user turn of tool_result blocks. A pause_turn stop reason is resumed, not treated as final. stop_reason is "max_turns" if the turn cap is hit without a final answer. It is "budget" if token_budget runs out before the next model call.

build_anthropic_tools

build_anthropic_tools(tools: list[dict]) -> list[dict]

This maps MCP tool descriptors to the Anthropic Messages tools= shape. The MCP shape is {name, description, inputSchema}. The Anthropic shape is {name, description, input_schema}. The only real difference is the schema key name. A missing schema defaults to {"type": "object", "properties": {}}. A missing description defaults to "".

DEFAULT_MODEL

DEFAULT_MODEL = os.environ.get("EVOSQL_AGENT_MODEL", "claude-opus-4-8")

This is the model id used when you pass none. It is read from the environment at import time.

DEFAULT_SYSTEM

DEFAULT_SYSTEM: str

This holds the agent's persona and the firm send rule. It tells the model it is a personal and work assistant backed by EvolutionDB long-term memory. It states the HARD RULE that the model must never deliver anything itself. The model drafts and calls queue_reply. Then a human calls approve_send. Sending stays dry-run. The model must never claim a message was sent. The prompt also tells the model to recall existing memory first, stay concise, answer in the user's own language, and adjust when a tool returns an error.

Notes & limitations

  • Model-dependent. The reasoning is an LLM call. "Standalone" only means it does not need a host application. The default path needs the anthropic SDK and ANTHROPIC_API_KEY. You can inject any client with a compatible messages.create(...) through the client= argument.
  • Single-shot CLI. The terminal entry point runs one prompt, then exits. There is no interactive REPL. It does not stream the model's text to stdout. It prints only the final answer. Text and tool I/O stream to stderr.
  • Tool set comes from the MCP server. This package does not define tools. It bridges whatever the installed mcp-server-evolutiondb exposes. That includes recall and save memory, daily and meeting summaries, open-loop tracking, PR review, drafting replies, and the approval gate.
  • Cost guards. max_turns bounds the iteration count everywhere. token_budget also caps spend, but only the web server wires it up. The default is 60000 through EVOSQL_AGENT_TOKEN_BUDGET. The CLI runs with no token budget.
  • Web server is minimal. The browser chat uses the stdlib only and has no authentication. The default bind is loopback. Do not expose it publicly. First put your own auth and an enforced token budget in front of it.
  • Send rule. The loop itself never delivers anything. The MCP server's outbox enforces this in code, not just in the prompt. See the code wiki for the ADR-004 internals.
