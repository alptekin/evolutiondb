evolution-agent

A standalone agent loop over EvolutionDB's MCP tools: it drives the model itself (plan → act → observe → repeat) so it runs without a host like Claude Desktop, while every tool call still flows through the MCP server and the human-approved send invariant stays intact.

What it does

mcp-server-evolutiondb is an MCP server: it executes tools that a host model (Claude Desktop / Claude Code) decides to call, it has no will of its own. evolution-agent adds that missing will. It calls a model directly, lets it choose tools, executes them, feeds the results back, and repeats until the model is done.

"Standalone" means host-independent, not model-independent. The reasoning still comes from an LLM (Anthropic by default; any provider exposing a compatible messages.create(...) client can be injected). What changes is that this process owns the loop instead of Claude Desktop.

Every tool call goes through the MCP server's own _call_tool, so RBAC, the append-only audit log, the PII gate, multi-tenant isolation, and the embedded EvolutionDB all come for free, none of it is re-implemented in this package.

info
The send invariant (ADR-004) is intact. Nothing is ever delivered to the outside world by the agent. It may draft and queue a reply (queue_reply), but a human approves it (approve_send), and delivery stays dry-run unless the operator explicitly opted in. The system prompt states this rule to the model, and the underlying outbox enforces it structurally.

Install

The distribution name on PyPI is evolutiondb-agent; the importable package is evolution_agent. It depends on mcp-server-evolutiondb>=1.16.0. The default model runs through the Anthropic SDK, which is an optional extra.

# the package + the Anthropic SDK extra
pip install evolutiondb-agent[anthropic]
export ANTHROPIC_API_KEY=sk-...

Two console scripts are installed:

  • evolution-agent → evolution_agent.__main__:main (the terminal loop)
  • evolution-agent-web → evolution_agent.web:main (the browser chat)

Requires Python 3.9+. License is MIT.

Configure (env vars / auth)

The agent connects to EvolutionDB exactly the way the MCP server does: either over the PostgreSQL wire to a running instance, or by spawning a local zero-Docker instance with EVOSQL_EMBEDDED=1.

The EvolutionDB connection variables (EVOSQL_*) and MCP_USER_ID are interpreted by mcp-server-evolutiondb, not by this package, evolution-agent simply constructs MCPServer() and lets it read the environment.

Usage

Terminal

Pass the request as arguments or pipe it on stdin. Tool calls stream to stderr; the final answer goes to stdout.

evolution-agent "what meetings do I have today and who owes me a reply?"
echo "summarize my open loops" | evolution-agent
evolution-agent -v "draft a reply to Alice about the Q3 report"   # show tool I/O

The CLI accepts these flags (parsed by hand in __main__.py; everything else is treated as prompt text):

FlagEffect
--model <id>Override the model for this run (otherwise EVOSQL_AGENT_MODEL / the default).
--max-turns <n>Cap the plan→act→observe iterations. Default 12.
-v, --verboseAlso stream each tool's result to stderr, not just the call.
--onceSingle prompt then exit. This is the default mode; the flag is accepted for clarity.
-h, --helpPrint the module docstring and exit.

Exit codes: 0 on success, 2 when no prompt was supplied, 1 when mcp_server_evosql or the Anthropic SDK is missing, or on any agent error. If the loop hits --max-turns without a final answer, a note is written to stderr.

Browser chat

evolution-agent-web                       # http://127.0.0.1:8800
EVOSQL_EMBEDDED=1 evolution-agent-web

A dependency-free single-page chat (stdlib http.server + Server-Sent Events, no web framework). The agent's tool timeline streams live, and a "pending approvals" panel surfaces every draft the agent queued, clicking Approve calls approve_send, which stays dry-run unless the operator enabled real delivery. It needs the anthropic SDK + ANTHROPIC_API_KEY; without them the chat endpoint returns a clear, streamed error. The HTTP routes are: GET / (the UI), GET /api/chat?q=… (SSE stream of the run), GET /api/pending (queued drafts via list_pending_replies), and POST /api/approve with a JSON {"id": …} body (calls approve_send).

Library

Both the model client and the tool server are injectable, so the loop is unit-testable with no network, API key, or database (see tests/test_loop.py).

from evolution_agent import AgentLoop
from mcp_server_evosql.server import MCPServer

agent = AgentLoop(MCPServer())             # defaults: Anthropic + the server's tools
result = agent.run("who am I waiting on?")
print(result["final_text"])
for step in result["steps"]:          # every tool call + its result
    print(step["tool"], step["input"])

The repo also ships examples/demo.py, a self-contained morning-assistant task. It auto-starts a throwaway local evosql-server if none is running, then drives a real multi-step task (recall/save memory, draft + queue a reply). If ANTHROPIC_API_KEY is set and anthropic is installed it uses the real model; otherwise it falls back to a scripted model so the tools, memory, and database stay real but the demo runs offline.

Reference

The public surface is exported from evolution_agent.__init__: AgentLoop, DEFAULT_MODEL, DEFAULT_SYSTEM, and build_anthropic_tools.

AgentLoop

class AgentLoop(server=None, *, client=None, model=None, system=None, tools=None, identity=None, max_turns=12, max_tokens=4096, token_budget=None, on_event=None)

A plan → act → observe → repeat loop over the MCP server's tools. server only needs a _call_tool(name, args, identity=None) -> dict method (MCPServer provides it; tests pass a fake). client only needs messages.create(...) -> response whose result exposes .content (blocks), .stop_reason, and optional .usage (the Anthropic SDK matches). When server, client, or tools are omitted they are built lazily from mcp_server_evosql and anthropic, so importing the package never requires either. token_budget is a hard ceiling on cumulative input+output tokens for a single run (None = unlimited; the turn cap still bounds the call count). on_event(kind, payload) is an optional observer, an exception it raises is swallowed so it can never break the loop.

AgentLoop.run(user_message: str) -> dict

Drives the conversation to completion and returns a transcript dict: final_text (str), steps (a list of {tool, input, result}, one per executed tool call), turns (int), stop_reason (str), and usage ({input_tokens, output_tokens, total_tokens}). Parallel tool calls in one model turn are all executed and fed back as a single user turn of tool_result blocks. A pause_turn stop reason is resumed rather than treated as final. stop_reason is "max_turns" if the turn cap is hit without a final answer, or "budget" if token_budget is exhausted before the next model call.

build_anthropic_tools

build_anthropic_tools(tools: list[dict]) -> list[dict]

Maps MCP tool descriptors ({name, description, inputSchema}) to the Anthropic Messages tools= shape ({name, description, input_schema}). The only real difference is the schema key name; a missing schema defaults to {"type": "object", "properties": {}} and a missing description to "".

DEFAULT_MODEL

DEFAULT_MODEL = os.environ.get("EVOSQL_AGENT_MODEL", "claude-opus-4-8")

The model id used when none is passed. Resolved from the environment at import time.

DEFAULT_SYSTEM

DEFAULT_SYSTEM: str

The agent's persona plus the non-negotiable send rule. It tells the model it is a personal/work assistant backed by EvolutionDB long-term memory, states the HARD RULE that it must never deliver anything itself (draft + queue_reply, then human approve_send; sending stays dry-run; never claim a message was sent), and instructs it to recall existing memory first, stay concise, answer in the user's own language, and adjust when a tool returns an error.

Notes & limitations

  • Model-dependent. The reasoning is an LLM call; "standalone" only means it does not need a host application. The default path requires the anthropic SDK and ANTHROPIC_API_KEY. Any client with a compatible messages.create(...) can be injected via the client= argument.
  • Single-shot CLI. The terminal entry point runs one prompt then exits; there is no interactive REPL. There is no streaming of the model's text to stdout, only the final answer is printed (text and tool I/O stream to stderr).
  • Tool set comes from the MCP server. This package does not define tools. It bridges whatever the installed mcp-server-evolutiondb exposes (recall/save memory, daily/meeting summaries, open-loop tracking, PR review, drafting replies, and the approval gate).
  • Cost guards. max_turns bounds the iteration count everywhere; token_budget additionally caps spend, but it is only wired up in the web server (default 60000 via EVOSQL_AGENT_TOKEN_BUDGET). The CLI runs with no token budget.
  • Web server is minimal. The browser chat is stdlib-only with no authentication; the default bind is loopback. Do not expose it publicly without putting your own auth and an enforced token budget in front of it.
  • Send invariant. The loop itself never delivers anything; the guarantee is enforced structurally by the MCP server's outbox, not just by the prompt. See the code wiki for the ADR-004 internals.
