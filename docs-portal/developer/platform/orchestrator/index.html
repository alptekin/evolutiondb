Orchestrator (hub)

One local control panel — a web UI and a terminal that share the same controls — for supervising every EvolutionDB *-sync connector and wiring the same evolutiondb-memory MCP server into each AI host. The PyPI package is evolutiondb-hub; the import package is orchestrator.

What it does

The hub is a connector supervisor. It does not sync anything itself and it stores no secrets — it starts, stops, schedules, installs and authenticates the per-source connector packages, and tracks which ones are running. Concretely it:

  • Lists every connector on one page with current state (running / stopped / external), last-start time, the install status of the connector's binary, an OS-scheduled (launchd / systemd) flag, and a free-form agent-owner label.
  • Starts and stops connectors as detached background processes (start_new_session=True). It also detects connectors already running under launchd / systemd by scanning ps, marks them managed_by: "external", and refuses to double-spawn or stop them.
  • Schedules connectors at the OS level so they survive a reboot — a launchd .plist on macOS or a systemd --user service + timer on Linux. Windows scheduling is not implemented.
  • Installs / upgrades a connector's PyPI package into the hub's own Python, and runs its --auth flow (or, for token-based connectors, reports the .env path to edit).
  • Wires the evolutiondb-memory MCP block into AI-host config files (Claude Desktop, Claude Code, ChatGPT Desktop / Codex, Gemini CLI / Code Assist) or removes it — delegating to mcp_server_evosql.setup. See the MCP server page.
  • Manages an MCP HTTP transport + OAuth proxy + optional Cloudflare quick tunnel so web-based AI hosts can reach the same memory server the desktop clients use.
  • Runs the evolutiondb-inspector memory browser as a hub-owned process.

Everything the web UI does is also available from the CLI, so headless / shell-script setups need nothing else.

Install

# the hub itself, plus the MCP server it drives for client/tunnel/inspector commands
pip install evolutiondb-hub mcp-server-evolutiondb

The hub has no runtime dependencies of its own (dependencies = [] in pyproject.toml; stdlib http.server powers the web UI). It requires Python >=3.9 and exposes a single console script, evolutiondb-hub (entry point orchestrator.cli:main). The client …, tunnel … and inspector … subcommands additionally need mcp-server-evolutiondb on PATH; install/start of a given source needs that source's own package:

pip install evolutiondb-gmail-sync evolutiondb-slack-sync   # …etc, per source you use

Configure (env vars)

There is no config file you edit by hand — behaviour is driven by a handful of environment variables, all optional with sane defaults. The hub binds the web UI and every spawned listener to 127.0.0.1 only and has no auth; it is a personal local panel.

State on disk

The hub keeps one JSON state file so pids, labels and intervals survive restarts. There are no secrets in it — connector auth tokens live in their own per-connector caches under ~/.evosql/.

~/.evosql/orchestrator.json          # sources{}, tunnel{}, inspector{}, user_id, updated_at — mode 0600
~/.evosql/logs/<source>.log           # stdout+stderr of each hub-spawned connector
~/.evosql/logs/install-<source>.log   # pip install output
~/.evosql/logs/auth-<source>.log      # --auth flow output
~/.evosql/logs/{mcp-http,mcp-oauth,cloudflared,inspector}.log

Saves are atomic (write to .tmp, chmod 0600, then replace). The README notes the single-user race window is tiny.

Usage

Launch the web UI (binds 127.0.0.1 only):

evolutiondb-hub web                      # → http://127.0.0.1:8771
evolutiondb-hub web --port 9000 --host 127.0.0.1

Drive the same operations from the terminal:

# inspect every connector (add --json for the machine-readable form the UI uses)
evolutiondb-hub status
evolutiondb-hub status --json

# spawn / kill one connector (interval overrides the per-source poll_default)
evolutiondb-hub start gmail --interval 600
evolutiondb-hub stop  gmail

# install / upgrade its PyPI package, then run its auth flow
evolutiondb-hub install gmail
evolutiondb-hub auth    gmail

# keep it running across reboots (launchd on macOS, systemd --user on Linux)
evolutiondb-hub schedule   gmail --interval 900
evolutiondb-hub unschedule gmail

# label a source with the agent you mentally assigned to it (or 'none' to clear)
evolutiondb-hub set-agent gmail gemini

# wire / unwire one AI host's evolutiondb-memory MCP block
evolutiondb-hub client claude-desktop  enable
evolutiondb-hub client chatgpt-desktop disable
evolutiondb-hub client gemini-cli      status

# publish the MCP server over HTTP (+ OAuth proxy) and a Cloudflare quick tunnel
evolutiondb-hub tunnel start
evolutiondb-hub tunnel start --no-public --port 8970 --token MYTOKEN
evolutiondb-hub tunnel status
evolutiondb-hub tunnel stop

# open the embedded memory inspector
evolutiondb-hub inspector start --port 8765
evolutiondb-hub inspector status

Every subcommand except web and the human status table prints a JSON result object and exits non-zero when ok is false. The set-agent label is a purely local annotation — connectors do not read it, and every record still lands in the same memory store searchable from any AI host.

The agent-owner label

The label answers "which AI assistant am I supposed to point at my Gmail this week?" — it is metadata on the panel, not data the connector acts on. Accepting none, clear or - clears it.

Reference

Public surface, by module. All operation functions return a JSON-serialisable dict; the convention is {"ok": bool, …} (the client enable path instead returns {"status": "wrote"|"unchanged"|…} from mcp_setup.configure_client).

supervisor.py — process supervision

supervisor.status_all() -> dict[str, dict]

Returns one status dict per built-in connector, keyed by name. Each carries state (running / stopped / unknown), managed_by (hub / external / None), pid, binary (resolved via shutil.which), agent_owner and scheduled.

supervisor.status(name) -> dict

Status for one connector. Live hub pid wins; a stale pid in the state file is cleared; otherwise it scans ps -Ao pid,command for the console-script or module name and reports an external run if found.

supervisor.start(name, *, interval=None) -> dict

Spawns <binary> --interval <N> detached, waits ~0.3s, and if the process exited immediately returns the last 5 log lines instead of recording a dead pid. Refuses to start if already running. interval defaults to the spec's poll_default.

supervisor.stop(name, *, timeout_s=5.0) -> dict

Sends SIGTERM to the hub-managed pid, waits up to timeout_s, then SIGKILL. Refuses to touch an externally-managed pid (telling you to use launchctl / systemctl).

supervisor.set_agent_owner(name, agent) -> dict

Persists the agent-owner label into the state file. Also exposed: pid_alive(pid) and find_external_pid(*needles).

config.py — registry

config.CONNECTORS: list[ConnectorSpec]  ·  config.AGENTS: list[AgentSpec]

The static registries. config.get(name) returns a ConnectorSpec or None. add_custom_connector(dict) / add_custom_agent(dict) validate, persist to ~/.evosql/custom_*.json and append to the live lists.

schedule.py — OS-level scheduling

schedule.schedule(name, *, interval=None, user_id=None) -> dict

On macOS writes ~/Library/LaunchAgents/com.evolutiondb.<name>-sync.plist and loads it idempotently with launchctl bootout then bootstrap gui/$UID. On Linux writes ~/.config/systemd/user/evolutiondb-<name>-sync.{service,timer} and runs systemctl --user enable --now. Scheduled runs invoke the binary with --once on a StartInterval / OnUnitActiveSec cadence. Returns an error on any other platform.

schedule.unschedule(name) -> dict  ·  schedule.is_scheduled(name) -> bool

render_plist(...) and render_systemd_units(...) are pure helpers (no side effects) that produce the unit text.

install.py — package + auth lifecycle

install.install(name) -> dict

Runs {sys.executable} -m pip install --upgrade <pip_package> (3-minute timeout) into the hub's own environment, streams output to install-<name>.log, and returns the resolved binary plus a 20-line log tail.

install.authenticate(name) -> dict

For connectors with an auth_subcommand (default --auth): spawns <binary> --auth detached and returns auth_kind: "interactive" with a pid and log path (the connector opens its own browser / device-code flow). For token connectors (slack, github, notion): returns auth_kind: "env" with the best-guess .env path to edit and the spec's hint.

tunnel.py — MCP HTTP transport + Cloudflare

tunnel.start(*, port=None, token=None, tunnel=True) -> dict

Spawns evolutiondb-mcp-http --host 127.0.0.1 --port <port> with the bearer token in its env, then layers an OAuth 2.1 proxy (evolutiondb-mcp-oauth) in front so ChatGPT-class clients can connect, and — unless tunnel=False — a cloudflared tunnel --url … quick tunnel. It parses the https://*.trycloudflare.com URL out of the cloudflared log (20s timeout). All three binaries come from mcp-server-evolutiondb (cloudflared is optional / skipped if absent).

tunnel.stop() -> dict  ·  tunnel.status() -> dict

stop() SIGTERMs cloudflared, the OAuth proxy and the HTTP listener (2s grace, then SIGKILL) and clears state["tunnel"]. status() reports liveness of each half plus the public URL.

inspector_mgr.py — memory browser

inspector_mgr.start(*, port=None) -> dict  ·  stop() -> dict  ·  status() -> dict

Spawns evolutiondb-inspector --host 127.0.0.1 --port <port> (default 8765) with MCP_USER_ID resolved from the env, falling back to the value persisted in the state file, then "default_user".

state.py — persisted state

state.load() -> dict  ·  state.save(state) -> None

Read/write ~/.evosql/orchestrator.json (or $EVOSQL_HUB_STATE). Helpers: get_source, set_source, remove_source_pid, get_user_id, set_user_id.

web.py — web UI

web.serve(*, host="127.0.0.1", port=8771) -> int

A no-framework ThreadingHTTPServer. Pages: /, /connectors, /clients, /tunnel, /inspector. JSON API: GET /api/{sources,clients,tunnel,inspector,health} and POST /api/sources/<name>/{start|stop|schedule|unschedule|install|auth|agent}, POST /api/clients/<name>/{enable|disable|install}, POST /api/{tunnel,inspector}/{start|stop}.

Notes & limitations

  • No auth, localhost only. The web UI and every spawned listener bind to 127.0.0.1. Public exposure only ever happens through the explicit tunnel command, which sits behind a bearer token plus the OAuth proxy.
  • The hub stores no secrets. The state file is mode 0600 and contains pids, intervals, labels and (only while a tunnel is live) the generated bearer token. Connector OAuth/refresh tokens stay in their own per-connector caches.
  • Windows scheduling is not implemented — schedule() / unschedule() return an error there. Foreground start/stop still work cross-platform.
  • The README under-documents the CLI. The shipped README.md mentions five connectors and six subcommands; the code in this version (evolutiondb-hub 0.5.1) registers eleven built-in connectors and also exposes schedule, unschedule, install, auth, tunnel and inspector. (orchestrator.__init__.__version__ still reads 0.5.0 while pyproject.toml is 0.5.1.)
  • External-run detection is best-effort — it relies on string-matching the console-script or module name in ps output; an unusual launch command may not be recognised.
  • The client commands import mcp_server_evosql.setup lazily and fail with a clear message if mcp-server-evolutiondb is not installed; the same is true for tunnel / inspector, which need the evolutiondb-mcp-http, evolutiondb-mcp-oauth and evolutiondb-inspector binaries.
