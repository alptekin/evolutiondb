Orchestrator (hub)

The hub is one local control panel. It has a web UI and a terminal that share the same controls. You use it to watch every EvolutionDB *-sync connector. You also use it to wire the same evolutiondb-memory MCP server into each AI host. The PyPI package is evolutiondb-hub. The import package is orchestrator.

What it does

The hub watches your connectors. It does not sync anything itself. It stores no secrets. It starts, stops, schedules, installs and authenticates the per-source connector packages. It also tracks which ones are running. Here is what it does.

  • Lists every connector on one page. For each one it shows the current state, which is running, stopped or external. It also shows the last start time, the install status of the connector's binary, an OS-scheduled flag for launchd or systemd, and a free-form agent-owner label.
  • Starts and stops connectors as detached background processes (start_new_session=True). It also finds connectors that already run under launchd or systemd by scanning ps. It marks those as managed_by: "external". It will not start them twice or stop them.
  • Schedules connectors at the OS level so they keep running after a reboot. On macOS it uses a launchd .plist. On Linux it uses a systemd --user service plus a timer. Windows scheduling is not implemented.
  • Installs or upgrades a connector's PyPI package into the hub's own Python. It also runs its --auth flow. For token-based connectors it reports the .env path to edit instead.
  • Wires the evolutiondb-memory MCP block into AI-host config files. The hosts are Claude Desktop, Claude Code, ChatGPT Desktop / Codex, and Gemini CLI / Code Assist. It can also remove the block. This work is handled by mcp_server_evosql.setup. See the MCP server page.
  • Manages an MCP HTTP transport, an OAuth proxy and an optional Cloudflare quick tunnel so web-based AI hosts can reach the same memory server the desktop clients use.
  • Runs the evolutiondb-inspector memory browser as a hub-owned process.

Everything the web UI does is also in the CLI. So headless and shell-script setups need nothing else.

You can add your own without code changes. Drop entries in ~/.evosql/custom_connectors.json or ~/.evosql/custom_agents.json. You can also call config.add_custom_connector() or add_custom_agent(). The hub merges them into the in-process lists at import time.

Install

# the hub itself, plus the MCP server it drives for client/tunnel/inspector commands
pip install evolutiondb-hub mcp-server-evolutiondb

The hub has no runtime dependencies of its own. The line dependencies = [] in pyproject.toml shows this. The web UI runs on the stdlib http.server. The hub needs Python >=3.9. It exposes one console script, evolutiondb-hub. The entry point is orchestrator.cli:main. The client …, tunnel … and inspector … subcommands also need mcp-server-evolutiondb on PATH. To run install or start for a given source, you need that source's own package.

pip install evolutiondb-gmail-sync evolutiondb-slack-sync   # …etc, per source you use

Configure (env vars)

There is no config file you edit by hand. A few environment variables drive the behaviour. They are all optional and have good defaults. The hub binds the web UI and every spawned listener to 127.0.0.1 only. It has no auth. It is a personal local panel.

State on disk

The hub keeps one JSON state file. This way pids, labels and intervals survive restarts. There are no secrets in it. Connector auth tokens live in their own per-connector caches under ~/.evosql/.

~/.evosql/orchestrator.json          # sources{}, tunnel{}, inspector{}, user_id, updated_at - mode 0600
~/.evosql/logs/<source>.log           # stdout+stderr of each hub-spawned connector
~/.evosql/logs/install-<source>.log   # pip install output
~/.evosql/logs/auth-<source>.log      # --auth flow output
~/.evosql/logs/{mcp-http,mcp-oauth,cloudflared,inspector}.log

Saves are atomic. The hub writes to .tmp, runs chmod 0600, then runs replace. The README notes that the single-user race window is tiny.

Usage

Launch the web UI. It binds to 127.0.0.1 only.

evolutiondb-hub web                      # → http://127.0.0.1:8771
evolutiondb-hub web --port 9000 --host 127.0.0.1

Drive the same operations from the terminal:

# inspect every connector (add --json for the machine-readable form the UI uses)
evolutiondb-hub status
evolutiondb-hub status --json

# spawn / kill one connector (interval overrides the per-source poll_default)
evolutiondb-hub start gmail --interval 600
evolutiondb-hub stop  gmail

# install / upgrade its PyPI package, then run its auth flow
evolutiondb-hub install gmail
evolutiondb-hub auth    gmail

# keep it running across reboots (launchd on macOS, systemd --user on Linux)
evolutiondb-hub schedule   gmail --interval 900
evolutiondb-hub unschedule gmail

# label a source with the agent you mentally assigned to it (or 'none' to clear)
evolutiondb-hub set-agent gmail gemini

# wire / unwire one AI host's evolutiondb-memory MCP block
evolutiondb-hub client claude-desktop  enable
evolutiondb-hub client chatgpt-desktop disable
evolutiondb-hub client gemini-cli      status

# publish the MCP server over HTTP (+ OAuth proxy) and a Cloudflare quick tunnel
evolutiondb-hub tunnel start
evolutiondb-hub tunnel start --no-public --port 8970 --token MYTOKEN
evolutiondb-hub tunnel status
evolutiondb-hub tunnel stop

# open the embedded memory inspector
evolutiondb-hub inspector start --port 8765
evolutiondb-hub inspector status

Every subcommand prints a JSON result object. The two exceptions are web and the human status table. A subcommand exits non-zero when ok is false. The set-agent label is only a local note. Connectors do not read it. Every record still lands in the same memory store. You can search that store from any AI host.

The agent-owner label

The label answers a simple question. Which AI assistant am I supposed to point at my Gmail this week? It is metadata on the panel. The connector does not act on it. Pass none, clear or - to clear it.

Reference

This is the public surface, listed by module. All operation functions return a JSON-serialisable dict. The usual shape is {"ok": bool, …}. The client enable path is different. It returns {"status": "wrote"|"unchanged"|…} from mcp_setup.configure_client.

supervisor.py, process supervision

supervisor.status_all() -> dict[str, dict]

Returns one status dict per built-in connector, keyed by name. Each one carries state, which is running, stopped or unknown. It carries managed_by, which is hub, external or None. It also carries pid, binary, agent_owner and scheduled. The binary value is resolved with shutil.which.

supervisor.status(name) -> dict

Status for one connector. A live hub pid wins. A stale pid in the state file is cleared. If neither applies, it scans ps -Ao pid,command for the console-script name or the module name. It reports an external run if it finds one.

supervisor.start(name, *, interval=None) -> dict

Spawns <binary> --interval <N> detached. It then waits about 0.3s. If the process exited right away, it returns the last 5 log lines instead of recording a dead pid. It will not start a connector that is already running. The interval defaults to the spec's poll_default.

supervisor.stop(name, *, timeout_s=5.0) -> dict

Sends SIGTERM to the hub-managed pid. It waits up to timeout_s, then sends SIGKILL. It will not touch an externally-managed pid. It tells you to use launchctl or systemctl instead.

supervisor.set_agent_owner(name, agent) -> dict

Saves the agent-owner label into the state file. The module also exposes pid_alive(pid) and find_external_pid(*needles).

config.py, registry

config.CONNECTORS: list[ConnectorSpec]  ·  config.AGENTS: list[AgentSpec]

The static registries. config.get(name) returns a ConnectorSpec or None. The functions add_custom_connector(dict) and add_custom_agent(dict) validate the input, save it to ~/.evosql/custom_*.json and append it to the live lists.

schedule.py, OS-level scheduling

schedule.schedule(name, *, interval=None, user_id=None) -> dict

On macOS it writes ~/Library/LaunchAgents/com.evolutiondb.<name>-sync.plist. It loads the file idempotently. It runs launchctl bootout and then bootstrap gui/$UID. On Linux it writes ~/.config/systemd/user/evolutiondb-<name>-sync.{service,timer} and runs systemctl --user enable --now. Each scheduled run calls the binary with --once. The cadence comes from StartInterval on macOS or OnUnitActiveSec on Linux. On any other platform it returns an error.

schedule.unschedule(name) -> dict  ·  schedule.is_scheduled(name) -> bool

render_plist(...) and render_systemd_units(...) are pure helpers. They produce the unit text and have no side effects.

install.py, package + auth lifecycle

install.install(name) -> dict

Runs {sys.executable} -m pip install --upgrade <pip_package> into the hub's own environment. It uses a 3-minute timeout. It streams output to install-<name>.log. It returns the resolved binary plus a 20-line log tail.

install.authenticate(name) -> dict

Some connectors have an auth_subcommand, which defaults to --auth. For those, it spawns <binary> --auth detached. It returns auth_kind: "interactive" with a pid and log path. The connector then opens its own browser or device-code flow. Other connectors use tokens. These are slack, github and notion. For those it returns auth_kind: "env" with the best-guess .env path to edit and the spec's hint.

tunnel.py, MCP HTTP transport + Cloudflare

tunnel.start(*, port=None, token=None, tunnel=True) -> dict

Spawns evolutiondb-mcp-http --host 127.0.0.1 --port <port> with the bearer token in its env. It then puts an OAuth 2.1 proxy in front, the evolutiondb-mcp-oauth binary, so ChatGPT-class clients can connect. Unless tunnel=False, it also starts a cloudflared tunnel --url … quick tunnel. It reads the https://*.trycloudflare.com URL out of the cloudflared log, with a 20s timeout. All three binaries come from mcp-server-evolutiondb. cloudflared is optional. It is skipped if it is absent.

tunnel.stop() -> dict  ·  tunnel.status() -> dict

stop() sends SIGTERM to cloudflared, the OAuth proxy and the HTTP listener. It gives them 2s, then sends SIGKILL. It then clears state["tunnel"]. status() reports whether each part is alive. It also reports the public URL.

inspector_mgr.py, memory browser

inspector_mgr.start(*, port=None) -> dict  ·  stop() -> dict  ·  status() -> dict

Spawns evolutiondb-inspector --host 127.0.0.1 --port <port>. The port defaults to 8765. It sets MCP_USER_ID from the env first. If that is not set, it uses the value saved in the state file. If that is also missing, it uses "default_user".

state.py, persisted state

state.load() -> dict  ·  state.save(state) -> None

Read and write ~/.evosql/orchestrator.json. You can point it at another file with $EVOSQL_HUB_STATE. The helper functions are get_source, set_source, remove_source_pid, get_user_id and set_user_id.

web.py, web UI

web.serve(*, host="127.0.0.1", port=8771) -> int

A no-framework ThreadingHTTPServer. The pages are /, /connectors, /clients, /tunnel and /inspector. The JSON API has GET /api/{sources,clients,tunnel,inspector,health}. It also has POST /api/sources/<name>/{start|stop|schedule|unschedule|install|auth|agent}, POST /api/clients/<name>/{enable|disable|install} and POST /api/{tunnel,inspector}/{start|stop}.

Notes & limitations

  • No auth, localhost only. The web UI and every spawned listener bind to 127.0.0.1. The hub only goes public through the explicit tunnel command. That command sits behind a bearer token and the OAuth proxy.
  • The hub stores no secrets. The state file is mode 0600. It contains pids, intervals and labels. It also holds the generated bearer token, but only while a tunnel is live. Connector OAuth and refresh tokens stay in their own per-connector caches.
  • Windows scheduling is not implemented. There, schedule() and unschedule() return an error. Foreground start and stop still work on every platform.
  • The README under-documents the CLI. The shipped README.md mentions five connectors and six subcommands. The code in this version, evolutiondb-hub 0.5.1, registers eleven built-in connectors. It also exposes schedule, unschedule, install, auth, tunnel and inspector. Note that orchestrator.__init__.__version__ still reads 0.5.0 while pyproject.toml is 0.5.1.
  • External-run detection is best-effort. It matches the console-script name or the module name in ps output. An unusual launch command may not be recognised.
  • The client commands import mcp_server_evosql.setup lazily. They fail with a clear message if mcp-server-evolutiondb is not installed. The same is true for tunnel and inspector. Those need the evolutiondb-mcp-http, evolutiondb-mcp-oauth and evolutiondb-inspector binaries.
