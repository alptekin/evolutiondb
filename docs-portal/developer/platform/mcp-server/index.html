MCP Server

This is a Model Context Protocol server. It is written in pure Python. It gives Claude Desktop and Claude Code long-term memory. The memory is stored in EvolutionDB. It also adds a personal assistant action loop over your synced mail and chat. The action loop is opt-in. A human approves every action.

The PyPI distribution name is mcp-server-evolutiondb. The package module is mcp_server_evosql. The source lives in the main repo under client/mcp-server-evosql/. The MCP server tells hosts that its name is evolutiondb-memory and its version is 1.16.0. It speaks MCP protocol version 2024-11-05.

What it does

The server is a JSON-RPC 2.0 process. A host launches it over stdio. The host can be Claude Desktop, Claude Code, or any MCP host. The server implements the MCP methods initialize, tools/list, tools/call, and ping. It also offers a set of tools. These tools read and write a real EvolutionDB database. The server connects to EvolutionDB over the PostgreSQL wire protocol with psycopg. So the install is pure Python. You do not need a C toolchain. You do not need the compiled libevosql-memory shared library at runtime.

The server offers two groups of tools.

  • Long-term memory (core). The core tools are save_memory, search_memory, recent_memories, forget, and list_tags. There are also retrieval and profile extras. These are show_profile, expand_episode, feedback, restore_memory, and set_language.
  • Personal assistant action loop. The action tools are daily_brief, suggest_reply, queue_reply, list_pending_replies, approve_send, reject_reply, outbox_audit, and send_scheduled. There are dev and team helpers. These are meeting_brief, review_pr, release_status, and draft_changelog. There are template and rule tools too. These are create_template, apply_template, list_templates, delete_template, create_rule, list_rules, delete_rule, and run_rules.

The server sets the user_id on every tool call. It takes the value from the MCP_USER_ID environment variable. The model never sees this value and never sets it. So the model cannot split the namespace by making up IDs across sessions.

For more background, see the MCP Server platform overview and the project wiki.

Install

# pure-Python install - no compiler needed
pipx install mcp-server-evolutiondb
# or
pip install --user mcp-server-evolutiondb

The only runtime dependency is psycopg[binary]>=3.1. Optional extras add more features.

  • mcp-server-evolutiondb[embeddings-local] adds on-device semantic search through sentence-transformers.
  • mcp-server-evolutiondb[turkish] adds Turkish word stemming for the gist channel through snowballstemmer.
  • mcp-server-evolutiondb[send] adds the Teams send transport through msal. Gmail and Outlook reuse their sibling sync packages. iMessage and Slack need no extra package.

The package declares these console entry points. They come from [project.scripts] in pyproject.toml.

  • mcp-server-evolutiondb and mcp-server-evosql run the stdio MCP server. Both map to mcp_server_evosql.__main__:main.
  • evolutiondb-mcp-setup finds installed MCP hosts and merges the server into their configs.
  • evolutiondb-send-setup prints what a send channel needs. It writes nothing.
  • evolutiondb-brief and evolutiondb-outbox drive the action loop from the CLI.
  • evolutiondb-embedded-fetch fetches the embedded EvolutionDB binary ahead of time.
  • evolutiondb-mcp-http runs the server as a streamable-HTTP listener.
  • evolutiondb-mcp-oauth, evolutiondb-identity, evolutiondb-inspector, and evolutiondb-tags are the OAuth proxy, the identity CLI, the DB inspector, and the tagger CLI.

EvolutionDB must be reachable. The simplest way is to run docker compose up -d in the main repo. You can also set EVOSQL_EMBEDDED=1. Then the server starts a local EvolutionDB on first connect. If no binary is found, the server fetches a prebuilt one. To turn this fetch off, set EVOSQL_EMBEDDED_AUTOFETCH=0. If an instance is already running, the server uses that one instead.

Configure (env vars / auth)

The server reads its connection and namespace settings from the environment. See MCPServer.__init__. Here are the core variables and their real defaults.

Env varDefaultPurpose
EVOSQL_HOST127.0.0.1EvolutionDB host.
EVOSQL_PORT5433PostgreSQL-wire port. psycopg needs the PG-wire port 5433. It does not work with the native EVO port 9967.
EVOSQL_USERadminDB user.
EVOSQL_PASSWORDadminDB password.
EVOSQL_DATABASEevosqlDatabase name.
MCP_USER_IDdefault_userThe namespace used on every tool call. It overrides any value the model supplies.
MCP_STORE_PREFIXmcpThe prefix for the catalog stores. For example, mcp_mem and mcp_ents.

These variables control retrieval, embedding, and context layers. They are all opt-in. They are off unless you set them.

Env varDefaultPurpose
EVOSQL_EMBEDDING_PROVIDERnoneSet to openai, local, or none. It makes search_memory use both semantic and keyword search.
OPENAI_API_KEY, Needed when the provider is openai.
EVOSQL_SALIENCE_BOOST / EVOSQL_GRAPH_BOOST / EVOSQL_PROFILE_BOOST0Ranking weights for salience, knowledge-graph activation, and the interest profile.
EVOSQL_GRAPH_BUILD / EVOSQL_ENTITY_EXTRACT1Build graph edges and extract entities while saving. Set to 0 to turn this off.
EVOSQL_EMBEDDED / EVOSQL_EMBEDDED_AUTOFETCHunset / onStart a local EvolutionDB and fetch its binary.

Sending is off by default. The approve_send tool only delivers when two things are set. You must set EVOSQL_SEND_ENABLED=1 and a per-channel send scope. See ADR-004. If they are not set, the outbox stays in dry-run mode. Multi-tenant token resolution turns on only when EVOSQL_TENANT_SECRET is set. For the HTTP transport, set EVOSQL_MCP_AUTH_TOKEN. The bearer token is required when you are not on loopback. You can also set EVOSQL_MCP_ALLOWED_ORIGINS.

Usage

Host config (stdio). Every supported host accepts the same mcpServers shape.

{
  "mcpServers": {
    "evolutiondb-memory": {
      "command": "uvx",
      "args":    ["mcp-server-evolutiondb"],
      "env": {
        "EVOSQL_HOST":     "127.0.0.1",
        "EVOSQL_PORT":     "5433",
        "EVOSQL_USER":     "admin",
        "EVOSQL_PASSWORD": "admin",
        "EVOSQL_DATABASE": "evosql",
        "MCP_USER_ID":     "your-handle",
        "MCP_STORE_PREFIX": "mcp"
      }
    }
  }
}

Or merge it into every detected host with one command.

# idempotent; --dry-run shows changes without writing
evolutiondb-mcp-setup
evolutiondb-mcp-setup --dry-run
evolutiondb-mcp-setup --client claude-desktop --user-id your-handle

Wire format. The format is newline-delimited JSON-RPC 2.0 over stdio. There are no Content-Length headers. The stdio loop in main() reads one JSON object per line from stdin. It sends each one through MCPServer.handle. Then it writes one JSON object per line to stdout.

{"jsonrpc":"2.0","id":1,"method":"initialize",
 "params":{"protocolVersion":"2024-11-05","capabilities":{}}}

{"jsonrpc":"2.0","id":2,"method":"tools/list"}

{"jsonrpc":"2.0","id":3,"method":"tools/call",
 "params":{"name":"save_memory",
           "arguments":{"fact":"loves jazz","tags":["preference"]}}}

A tools/call result returns its payload as a single text content block. Failures come back with isError: true. On an internal exception, you only get an opaque correlation ref. The raw engine error is logged on the server. It is never sent to the client.

Action-loop CLI. By default it reads and suggests. Nothing is delivered without an explicit, gated approval.

evolutiondb-brief                 # who's waiting on you, what you owe
evolutiondb-brief --queue 3       # draft + queue replies (nothing sent)
evolutiondb-outbox list           # review what's queued
evolutiondb-outbox show <id>       # preview exactly what would go out
evolutiondb-brief --approve       # deliver (dry-run unless sending is enabled)

Inspecting stored data. Memory rows are plain records you can query. They are not opaque blobs.

SELECT mem_namespace, mem_key, mem_value FROM __mem_mcp_mem;

Reference

The host sees the tool catalog from the module-level TOOLS list. MCPServer._dispatch_tool routes each call into a MemoryBackend method. Here are the key entry points.

def main() -> int # mcp_server_evosql.server (also mcp_server_evosql.__main__:main)

This is the stdio entry point. The mcp-server-evolutiondb and mcp-server-evosql console scripts call it. It builds an MCPServer. It prints a one-line readiness banner to stderr. Then it loops over stdin and reads newline-delimited JSON-RPC. It dispatches each message and writes the response to stdout.

class MCPServer

It owns the configuration. The configuration is read from the environment in __init__. It also owns the per-tenant MemoryBackend cache. handle(msg, identity=None) implements initialize, tools/list, tools/call, and ping. Notifications without an id get no response. _dispatch_tool(...) maps each tool name to a backend method. An in-tenant RBAC layer blocks read-only viewers from the mutating tools listed in _MUTATING_TOOLS.

class MemoryBackend(host, port, user, password, database, prefix..)

It wraps a single psycopg connection. A reentrant lock serializes access to it. It exposes the memory operations. If the connection dies, it reconnects and retries. All generated SQL is inlined. So keys are validated with _valid_key and string values are escaped with _e before use.

def save(self, user_id, fact, tags=None, derived_from=None, source_class=None) -> str

It saves a fact. It generates a time-ordered key in the form mem_<ms>_<hex>. It merges any auto-derived topic tags into tags. It records a source_class. The value is asserted, observed, or inferred. It checks the derived_from provenance keys like foreign keys. It returns the new key.

def search(self, user_id, query, limit=5, tag=None, sender=None, mode='flat', include_archived=False, as_of=None) -> list

This is hybrid retrieval. When an embedding provider is set, it ranks results by a weighted mix of cosine similarity and keyword overlap. Without a provider, it uses plain substring matching. It supports tag and sender filters, hierarchical episode mode, and archived rows. The as_of option does time-travel against the validity gate. It backs the search_memory tool. That tool also returns a query_id for the feedback loop.

def recent(self, user_id, limit=10) -> list

It returns the most recently saved facts for the namespace. The newest fact comes first. It sorts by the stored created timestamp. It backs recent_memories.

def forget(self, user_id, key) -> bool

It deletes a fact by key with MEMORY DELETE. It rejects malformed or injection keys first. It returns False on a bad key or a failure. It backs forget.

def list_tags(self, user_id) -> list

It collects all distinct tags in the namespace with their counts. It sorts them by count from high to low. It backs list_tags.

Notes & limitations

  • PG-wire port only. Connections go through psycopg. It needs the PostgreSQL-wire port. The default is 5433. The native EVO port 9967 will not work. Point EVOSQL_PORT at the PG-wire port. The bundled README config table lists 9967 in one place. But the server's real default and the working config example both use 5433.
  • Single connection per backend. Each MemoryBackend holds one psycopg connection. A reentrant lock serializes it. stdio is single-threaded, so there is no contention here. The HTTP transport uses the lock for thread safety.
  • No streaming responses. Tool results come back as a single JSON content block. The protocol could stream, but the host UI does not show partial responses.
  • Sending is gated on purpose. approve_send is the only tool that can deliver. It stays a dry-run unless you set both EVOSQL_SEND_ENABLED=1 and a per-channel scope. See the action-loop send and approval design in ADR-004 in the main repo.
  • Auth via env vars. Credentials come from the environment. The server does not rotate secrets. It is meant to run locally over stdio. For remote or HTTP use, use the bearer-token HTTP transport.
  • Background jobs. python -m mcp_server_evosql.scheduler run maintains salience, profile, decay, episode summaries, and inline embedding upkeep. The matching ranking boosts have no effect until it runs.
