MCP Server

A Model Context Protocol server, written in pure Python, that gives Claude Desktop / Claude Code persistent long-term memory backed by EvolutionDB: plus an opt-in, human-approved personal-assistant action loop over your synced mail and chat.

Distribution name on PyPI: mcp-server-evolutiondb. Package module: mcp_server_evosql. Source lives in the main repo under client/mcp-server-evosql/. The MCP server identifies itself to hosts as evolutiondb-memory (version 1.16.0) and speaks MCP protocol version 2024-11-05.

What it does

The server is a JSON-RPC 2.0 process that a host (Claude Desktop, Claude Code, or any MCP host) launches over stdio. It implements the MCP surface, initialize, tools/list, tools/call, and ping, and exposes a set of tools that read and write a real EvolutionDB database. It connects to EvolutionDB over the PostgreSQL wire protocol using psycopg, so the install is pure-Python: no C toolchain and no compiled libevosql-memory shared library are required at runtime.

Two capability groups are exposed:

  • Long-term memory (core): save_memory, search_memory, recent_memories, forget, list_tags, plus retrieval/profile extras show_profile, expand_episode, feedback, restore_memory, set_language.
  • Personal-assistant action loop: daily_brief, suggest_reply, queue_reply, list_pending_replies, approve_send, reject_reply, outbox_audit, send_scheduled, plus dev/team helpers (meeting_brief, review_pr, release_status, draft_changelog) and template/rule management (create_template, apply_template, list_templates, delete_template, create_rule, list_rules, delete_rule, run_rules).

Every tool call has its user_id overridden server-side from the MCP_USER_ID environment variable. The model never sees or sets it, so it cannot fragment the namespace by inventing IDs across sessions.

See the MCP Server platform overview and the project wiki for broader context.

Install

# pure-Python install - no compiler needed
pipx install mcp-server-evolutiondb
# or
pip install --user mcp-server-evolutiondb

The only runtime dependency is psycopg[binary]>=3.1. Optional extras add capabilities:

  • mcp-server-evolutiondb[embeddings-local], on-device semantic search via sentence-transformers.
  • mcp-server-evolutiondb[turkish], Turkish morphological stemming for the gist channel (snowballstemmer).
  • mcp-server-evolutiondb[send], the Teams send transport (msal). Gmail/Outlook reuse their sibling sync packages; iMessage/Slack need no extra package.

The package declares the following console entry points (from pyproject.toml [project.scripts]):

  • mcp-server-evolutiondb and mcp-server-evosql, the stdio MCP server (both map to mcp_server_evosql.__main__:main).
  • evolutiondb-mcp-setup, autodetect installed MCP hosts and merge the server into their configs.
  • evolutiondb-send-setup, print what a send channel needs (writes nothing).
  • evolutiondb-brief / evolutiondb-outbox, drive the action loop from the CLI.
  • evolutiondb-embedded-fetch, pre-fetch the embedded EvolutionDB binary.
  • evolutiondb-mcp-http, run the server as a streamable-HTTP listener.
  • evolutiondb-mcp-oauth, evolutiondb-identity, evolutiondb-inspector, evolutiondb-tags: OAuth proxy, identity CLI, DB inspector, and tagger CLI.

EvolutionDB itself must be reachable. The simplest path is docker compose up -d in the main repo. Alternatively set EVOSQL_EMBEDDED=1 and the server spawns a local EvolutionDB on first connect (auto-fetching a prebuilt binary if none is found, unless EVOSQL_EMBEDDED_AUTOFETCH=0); an already-running instance always wins.

Configure (env vars / auth)

The server reads its connection and namespace settings from the environment (see MCPServer.__init__). Core variables and their real defaults:

Env varDefaultPurpose
EVOSQL_HOST127.0.0.1EvolutionDB host.
EVOSQL_PORT5433PostgreSQL-wire port. Note: psycopg requires the PG-wire port (5433), not the native EVO port (9967).
EVOSQL_USERadminDB user.
EVOSQL_PASSWORDadminDB password.
EVOSQL_DATABASEevosqlDatabase name.
MCP_USER_IDdefault_userSticky namespace applied to every tool call; overrides anything the model supplies.
MCP_STORE_PREFIXmcpPrefix for the catalog stores (e.g. mcp_mem, mcp_ents).

Retrieval/embedding and context-layer toggles (all opt-in, off unless set):

Env varDefaultPurpose
EVOSQL_EMBEDDING_PROVIDERnoneopenai, local, or none. Turns search_memory into hybrid semantic + keyword.
OPENAI_API_KEY, Required when the provider is openai.
EVOSQL_SALIENCE_BOOST / EVOSQL_GRAPH_BOOST / EVOSQL_PROFILE_BOOST0Ranking-blend weights for salience, knowledge-graph activation, and the interest profile.
EVOSQL_GRAPH_BUILD / EVOSQL_ENTITY_EXTRACT1Build graph edges / extract entities inline on save. Set to 0 to disable.
EVOSQL_EMBEDDED / EVOSQL_EMBEDDED_AUTOFETCHunset / onSpawn a local EvolutionDB and auto-fetch its binary.

Sending is off by default. Delivery via approve_send only happens when both EVOSQL_SEND_ENABLED=1 and a per-channel send scope are set (see ADR-004); otherwise the outbox stays dry-run. Multi-tenant token resolution activates only when EVOSQL_TENANT_SECRET is set. For the HTTP transport, set EVOSQL_MCP_AUTH_TOKEN (bearer token required off-loopback) and optionally EVOSQL_MCP_ALLOWED_ORIGINS.

Usage

Host config (stdio). Every supported host accepts the same mcpServers shape:

{
  "mcpServers": {
    "evolutiondb-memory": {
      "command": "uvx",
      "args":    ["mcp-server-evolutiondb"],
      "env": {
        "EVOSQL_HOST":     "127.0.0.1",
        "EVOSQL_PORT":     "5433",
        "EVOSQL_USER":     "admin",
        "EVOSQL_PASSWORD": "admin",
        "EVOSQL_DATABASE": "evosql",
        "MCP_USER_ID":     "your-handle",
        "MCP_STORE_PREFIX": "mcp"
      }
    }
  }
}

Or merge it into every detected host in one command:

# idempotent; --dry-run shows changes without writing
evolutiondb-mcp-setup
evolutiondb-mcp-setup --dry-run
evolutiondb-mcp-setup --client claude-desktop --user-id your-handle

Wire format. Newline-delimited JSON-RPC 2.0 over stdio (no Content-Length headers). The stdio loop in main() reads one JSON object per line from stdin, dispatches through MCPServer.handle, and writes one JSON object per line to stdout:

{"jsonrpc":"2.0","id":1,"method":"initialize",
 "params":{"protocolVersion":"2024-11-05","capabilities":{}}}

{"jsonrpc":"2.0","id":2,"method":"tools/list"}

{"jsonrpc":"2.0","id":3,"method":"tools/call",
 "params":{"name":"save_memory",
           "arguments":{"fact":"loves jazz","tags":["preference"]}}}

A tools/call result returns its payload as a single text content block; failures come back with isError: true (and, on an internal exception, only an opaque correlation ref: the raw engine error is logged server-side, never echoed to the client).

Action-loop CLI. Read + suggest by default; nothing is delivered without an explicit, gated approval:

evolutiondb-brief                 # who's waiting on you, what you owe
evolutiondb-brief --queue 3       # draft + queue replies (nothing sent)
evolutiondb-outbox list           # review what's queued
evolutiondb-outbox show <id>       # preview exactly what would go out
evolutiondb-brief --approve       # deliver (dry-run unless sending is enabled)

Inspecting stored data. Memory rows are plain queryable records, not opaque blobs:

SELECT mem_namespace, mem_key, mem_value FROM __mem_mcp_mem;

Reference

The tool catalog the host sees is the module-level TOOLS list; MCPServer._dispatch_tool routes each call into a MemoryBackend method. Key entry points:

def main() -> int # mcp_server_evosql.server (also mcp_server_evosql.__main__:main)

The stdio entry point invoked by the mcp-server-evolutiondb / mcp-server-evosql console scripts. Constructs an MCPServer, prints a one-line readiness banner to stderr, then loops over stdin reading newline-delimited JSON-RPC, dispatching each message and writing the response to stdout.

class MCPServer

Owns configuration (read from the environment in __init__) and the per-tenant MemoryBackend cache. handle(msg, identity=None) implements initialize, tools/list, tools/call, and ping; notifications without an id get no response. _dispatch_tool(...) maps each tool name to a backend method. An in-tenant RBAC layer denies read-only viewers the mutating tools listed in _MUTATING_TOOLS.

class MemoryBackend(host, port, user, password, database, prefix..)

Wraps a single psycopg connection (serialized by a reentrant lock) and exposes the memory operations. Reconnects with retry on dead connections; all generated SQL is inlined, so keys are validated (_valid_key) and string values escaped (_e) before use.

def save(self, user_id, fact, tags=None, derived_from=None, source_class=None) -> str

Persists a fact. Generates a time-ordered key (mem_<ms>_<hex>), merges any auto-derived topic tags into tags, records a source_class (asserted / observed / inferred), and validates derived_from provenance keys foreign-key style. Returns the new key.

def search(self, user_id, query, limit=5, tag=None, sender=None, mode='flat', include_archived=False, as_of=None) -> list

Hybrid retrieval. With an embedding provider configured it ranks by a weighted mix of cosine similarity and keyword overlap, otherwise pure substring matching. Supports tag/sender filters, hierarchical episode mode, archived inclusion, and as_of time-travel against the validity gate. Backs the search_memory tool, which also returns a query_id for the feedback loop.

def recent(self, user_id, limit=10) -> list

Returns the most recently saved facts for the namespace, most-recent-first (sorted by the stored created timestamp). Backs recent_memories.

def forget(self, user_id, key) -> bool

Deletes a fact by key via MEMORY DELETE. Rejects malformed/injection keys up front; returns False on a bad key or failure. Backs forget.

def list_tags(self, user_id) -> list

Aggregates all distinct tags in the namespace with counts, sorted by count descending. Backs list_tags.

Notes & limitations

  • PG-wire port only. Connections go through psycopg, which requires the PostgreSQL-wire port (default 5433). The native EVO port (9967) will not work; point EVOSQL_PORT at the PG-wire port. (The bundled README's configuration table lists 9967 in one place, but the server's actual default and the working config example both use 5433.)
  • Single connection per backend. Each MemoryBackend holds one psycopg connection serialized by a reentrant lock. stdio is single-threaded so this is uncontended; the HTTP transport relies on the lock for thread safety.
  • No streaming responses. Tool results return as a single JSON content block; the protocol could stream but the host UI does not render partial responses.
  • Sending is deliberately gated. approve_send is the only tool that can deliver, and it stays a dry-run unless both EVOSQL_SEND_ENABLED=1 and a per-channel scope are configured. See the action-loop send/approval design (ADR-004) in the main repo.
  • Auth via env vars. Credentials come from the environment; the server does not rotate secrets. Intended to run locally over stdio. For remote/HTTP exposure use the bearer-token HTTP transport.
  • Background jobs. Salience, profile, decay, episode summaries, and inline embedding upkeep are maintained by python -m mcp_server_evosql.scheduler run; the corresponding ranking boosts have no effect until it runs.
