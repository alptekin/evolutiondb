Install & Run EvolutionDB

EvolutionDB is a single-process database server. There are three ways to get one running: Docker Compose (fastest), building from source with make, and the zero-Docker embedded mode where the MCP server spawns its own instance. Every path exposes the same two ports: the PostgreSQL wire protocol on 5433 and the native EVO text protocol on 9967.

info
Default credentials in all examples are admin / admin. The user name comes from EVOSQL_USER_NAME (default admin); if you don't set EVOSQL_PASSWORD a random password is generated and printed once at startup.

Option 1 — Docker Compose

The repository ships a docker-compose.yml that builds the server image and starts it with sensible defaults. This is the recommended path for testing.

# start the server in the background — PG:5433  EVO:9967
docker compose up -d

# stop the server, keeping the data volume
docker compose down

# stop and delete the data volume too
docker compose down -v

The Compose service publishes three ports: 5433 (PostgreSQL wire), 9967 (native EVO protocol), and 9970 (CDC streaming, JSON Lines). Data is persisted in a named volume (evosql-data, mounted at /data), so it survives docker compose down and is only erased with -v.

The bundled docker-compose.yml sets these environment variables on the server:

# var                              value     meaning
EVOSQL_USER_NAME                   admin     login user name
EVOSQL_PASSWORD                    admin     login password (remove for a random one)
EVOSQL_CDC_PORT                    9970      CDC streaming port
EVOSQL_AUTO_RECLAIM_INTERVAL_SEC   2         auto-RECLAIM tick (default is 30; 2 is for tests)
EVOSQL_ENCRYPTION_KEY              (set)     TDE passphrase; set empty to disable encryption
warning
The Compose file drops EVOSQL_AUTO_RECLAIM_INTERVAL_SEC to 2 so TTL-driven row expiry runs visibly inside a test iteration. Production leaves the default 30s — set it back if you are running benchmarks.

To skip Compose and run the published image directly, mapping both protocol ports and supplying your own password:

docker run -d -p 5433:5433 -p 9967:9967 \
    -e EVOSQL_PASSWORD=mysecret evolutiondb/evolutiondb:latest

The Compose stack also defines a cli service that builds the CLI client image and depends on the server, so you can exec into it to reach the running database.

Option 2 — Build from source

EvolutionDB builds into three executables: the core SQL engine (evolution/evosql), the dual-protocol server (adaptor/evosql-server), and the CLI client (cli/evosql-cli).

Requirements: GCC, Bison, Flex, and libreadline-dev (for the CLI). For TLS support you also need libssl-dev (OpenSSL).

# build all three components
make

# build individual components
make evolution        # core SQL engine -> evolution/evosql
make adaptor          # dual-protocol server -> adaptor/evosql-server
make cli              # CLI client -> cli/evosql-cli

# build the server with OpenSSL TLS support
make adaptor TLS=1

# build the C memory SDK
make -C client/libevosql-memory

# stripped binaries into release/
make release

# clean all build artifacts
make clean
info
If you change the SQL grammar (evolution/parser/evoparser.y or evolution/parser/evolexer.l), run make generate to regenerate the Flex/Bison parser. The generated files are committed to the repo, so a normal build does not require Bison or Flex to be installed.

Once built, start the server and connect with the CLI:

# start the server — PG:5433  EVO:9967
./adaptor/evosql-server

# start on a custom PostgreSQL port
./adaptor/evosql-server --pg-port 5434

# connect with the CLI (defaults to localhost:9967)
./cli/evosql-cli

# connect with a password prompt
./cli/evosql-cli -W admin

The server accepts --pg-port and --evo-port to override the default ports. On Linux/macOS the build produces plain executables; on Windows (MinGW/MSYS2) they carry a .exe suffix.

Option 3 — Zero-Docker embedded mode

The Python MCP server can run EvolutionDB without Docker and without a manual build. With EVOSQL_EMBEDDED set, on first connect the MCP server checks whether anything is already listening on EVOSQL_HOST:PORT. If the port is free it spawns a local EvolutionDB against a per-user data directory and connects to it; if something is already listening, it leaves that instance alone — an external or shared server still wins.

# turn on embedded mode (any of 1/true/yes/on)
export EVOSQL_EMBEDDED=1

The spawned process lives until the MCP server exits. A signal reaper handles SIGTERM and SIGINT so that docker stop, systemctl stop, or a plain kill tears down the spawned server rather than orphaning it.

How the binary is found

Embedded mode resolves the evosql-server binary in this precedence order:

# 1. EVOSQL_SERVER_BINARY                an explicit path you set
# 2. bundled in the package              the [embedded] wheel vendors one under bin/
# 3. evosql-server on PATH               whatever shutil.which finds
# 4. a previously auto-fetched binary    cached in the per-user data dir
# 5. the repo's adaptor/evosql-server    dev checkout fallback

If none of those exist, auto-fetch downloads the prebuilt binary for your platform. Auto-fetch is on by default under embedded mode; disable it with EVOSQL_EMBEDDED_AUTOFETCH=0, in which case you must set EVOSQL_SERVER_BINARY, put evosql-server on PATH, or the server raises at startup.

You can also pre-fetch the binary explicitly with the bundled console script, which downloads the prebuilt evosql-server for your platform into the per-user cache:

# pre-download the prebuilt server binary (default version 3.0.0)
evolutiondb-embedded-fetch

# or a specific server release version
evolutiondb-embedded-fetch --version 3.0.0
warning
Auto-fetch downloads and executes a binary, so the source is pinned: it only fetches over HTTPS from github.com / objects.githubusercontent.com, and the version must match N.N.N. The published .sha256 is verified on download. To use a binary from anywhere else, set EVOSQL_SERVER_BINARY to a local path instead of widening the release source.

Ports and data directory

The embedded launcher defaults to the PostgreSQL port 5433 and derives the EVO port as port + 4534 (so the default pair is 5433 / 9967, matching every other mode). It starts the server with --pg-port and --evo-port, runs it inside the data directory (the binary uses its working directory for storage — there is no --data-dir flag), and waits up to 20 seconds for the port to open.

The per-user data directory is OS-appropriate, and can be overridden with EVOSQL_EMBEDDED_DATA_DIR:

# macOS
~/Library/Application Support/EvolutionDB/data

# Windows
%APPDATA%\EvolutionDB\data

# Linux (or $XDG_DATA_HOME)
~/.local/share/evolutiondb/data

Embedded environment variables

# var                          default   meaning
EVOSQL_EMBEDDED                (off)     turn embedded mode on (1/true/yes/on)
EVOSQL_EMBEDDED_AUTOFETCH      1         auto-download the binary when none is found
EVOSQL_SERVER_BINARY           (none)    explicit path to an evosql-server binary
EVOSQL_SERVER_VERSION          3.0.0     prebuilt server release to fetch
EVOSQL_SERVER_RELEASE_BASE     github    release download base (pinned to github.com)
EVOSQL_EMBEDDED_DATA_DIR       (per-OS)  override the per-user data directory

Verify it's running

Whichever path you took, the server listens on the same two ports. Connect with a PostgreSQL client over the wire protocol on 5433, or with the native CLI over EVO on 9967:

psql -h 127.0.0.1 -p 5433 -U admin
./cli/evosql-cli -W admin

The default database used throughout the test suite is testdb, with credentials admin / admin. From here, see Connect (PG wire & EVO) for client details and the MCP server docs for the embedded-mode integration.
