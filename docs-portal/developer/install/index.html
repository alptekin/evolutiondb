Install & Run EvolutionDB

EvolutionDB is a single-process database server. There are three ways to start one. You can use Docker Compose, which is the fastest. You can build from source with make. You can use the zero-Docker embedded mode, where the MCP server starts its own instance. Every way opens the same two ports. The PostgreSQL wire protocol runs on 5433. The native EVO text protocol runs on 9967.

info
The default login in all examples is admin / admin. The user name comes from EVOSQL_USER_NAME. Its default is admin. If you do not set EVOSQL_PASSWORD, the server makes a random password. It prints that password once at startup.

Option 1, Docker Compose

The repository comes with a docker-compose.yml file. It builds the server image and starts it with good defaults. This is the best way to test.

# start the server in the background - PG:5433  EVO:9967
docker compose up -d

# stop the server, keeping the data volume
docker compose down

# stop and delete the data volume too
docker compose down -v

The Compose service opens three ports. Port 5433 is the PostgreSQL wire protocol. Port 9967 is the native EVO protocol. Port 9970 is CDC streaming in JSON Lines. Data is stored in a named volume called evosql-data. The volume is mounted at /data. The data stays after docker compose down. It is only deleted when you use -v.

The bundled docker-compose.yml sets these environment variables on the server.

# var                              value     meaning
EVOSQL_USER_NAME                   admin     login user name
EVOSQL_PASSWORD                    admin     login password (remove for a random one)
EVOSQL_CDC_PORT                    9970      CDC streaming port
EVOSQL_AUTO_RECLAIM_INTERVAL_SEC   2         auto-RECLAIM tick (default is 30; 2 is for tests)
EVOSQL_ENCRYPTION_KEY              (set)     TDE passphrase; set empty to disable encryption
warning
The Compose file lowers EVOSQL_AUTO_RECLAIM_INTERVAL_SEC to 2. This makes TTL row expiry happen fast enough to see during a test. The default is 30 seconds for production. Set it back if you run benchmarks.

You can skip Compose and run the published image yourself. This command maps both protocol ports and sets your own password.

docker run -d -p 5433:5433 -p 9967:9967 \
    -e EVOSQL_PASSWORD=mysecret evolutiondb/evolutiondb:latest

The Compose stack also has a cli service. It builds the CLI client image and depends on the server. You can exec into it to reach the running database.

Option 2, Build from source

EvolutionDB builds into three programs. The core SQL engine is evolution/evosql. The dual-protocol server is adaptor/evosql-server. The CLI client is cli/evosql-cli.

Requirements. You need GCC, Bison, and Flex. You need libreadline-dev for the CLI. For TLS support you also need libssl-dev from OpenSSL.

# build all three components
make

# build individual components
make evolution        # core SQL engine -> evolution/evosql
make adaptor          # dual-protocol server -> adaptor/evosql-server
make cli              # CLI client -> cli/evosql-cli

# build the server with OpenSSL TLS support
make adaptor TLS=1

# build the C memory SDK
make -C client/libevosql-memory

# stripped binaries into release/
make release

# clean all build artifacts
make clean
info
You may change the SQL grammar in evolution/parser/evoparser.y or evolution/parser/evolexer.l. If you do, run make generate to rebuild the Flex/Bison parser. The generated files are committed to the repo. So a normal build does not need Bison or Flex installed.

After the build, start the server and connect with the CLI.

# start the server - PG:5433  EVO:9967
./adaptor/evosql-server

# start on a custom PostgreSQL port
./adaptor/evosql-server --pg-port 5434

# connect with the CLI (defaults to localhost:9967)
./cli/evosql-cli

# connect with a password prompt
./cli/evosql-cli -W admin

The server takes --pg-port and --evo-port to change the default ports. On Linux and macOS the build makes plain programs. On Windows with MinGW or MSYS2 they have a .exe suffix.

Option 3, Zero-Docker embedded mode

The Python MCP server can run EvolutionDB without Docker and without a manual build. When EVOSQL_EMBEDDED is set, the MCP server checks the first time you connect. It looks at whether anything is already listening on EVOSQL_HOST:PORT. If the port is free, it starts a local EvolutionDB with a per-user data directory and connects to it. If something is already listening, it leaves that instance alone. An external or shared server still wins.

# turn on embedded mode (any of 1/true/yes/on)
export EVOSQL_EMBEDDED=1

The started process runs until the MCP server exits. A signal reaper handles SIGTERM and SIGINT. So docker stop, systemctl stop, or a plain kill shuts down the started server. It does not leave it orphaned.

How the binary is found

Embedded mode looks for the evosql-server binary in this order.

# 1. EVOSQL_SERVER_BINARY                an explicit path you set
# 2. bundled in the package              the [embedded] wheel vendors one under bin/
# 3. evosql-server on PATH               whatever shutil.which finds
# 4. a previously auto-fetched binary    cached in the per-user data dir
# 5. the repo's adaptor/evosql-server    dev checkout fallback

If none of those exist, auto-fetch downloads the prebuilt binary for your platform. Auto-fetch is on by default in embedded mode. You can turn it off with EVOSQL_EMBEDDED_AUTOFETCH=0. If you turn it off, you must set EVOSQL_SERVER_BINARY or put evosql-server on PATH. If you do not, the server fails at startup.

You can also fetch the binary ahead of time with the bundled console script. It downloads the prebuilt evosql-server for your platform into the per-user cache.

# pre-download the prebuilt server binary (default version 3.0.0)
evolutiondb-embedded-fetch

# or a specific server release version
evolutiondb-embedded-fetch --version 3.0.0
warning
Auto-fetch downloads and runs a binary. For this reason the source is pinned. It only fetches over HTTPS from github.com and objects.githubusercontent.com. The version must match N.N.N. The published .sha256 is checked on download. To use a binary from somewhere else, set EVOSQL_SERVER_BINARY to a local path. Do not widen the release source.

Ports and data directory

The embedded launcher uses the PostgreSQL port 5433 by default. It works out the EVO port as port + 4534. So the default pair is 5433 / 9967. This matches every other mode. It starts the server with --pg-port and --evo-port. It runs the server inside the data directory. The binary uses its working directory for storage. There is no --data-dir flag. The launcher waits up to 20 seconds for the port to open.

The per-user data directory fits each operating system. You can change it with EVOSQL_EMBEDDED_DATA_DIR.

# macOS
~/Library/Application Support/EvolutionDB/data

# Windows
%APPDATA%\EvolutionDB\data

# Linux (or $XDG_DATA_HOME)
~/.local/share/evolutiondb/data

Embedded environment variables

# var                          default   meaning
EVOSQL_EMBEDDED                (off)     turn embedded mode on (1/true/yes/on)
EVOSQL_EMBEDDED_AUTOFETCH      1         auto-download the binary when none is found
EVOSQL_SERVER_BINARY           (none)    explicit path to an evosql-server binary
EVOSQL_SERVER_VERSION          3.0.0     prebuilt server release to fetch
EVOSQL_SERVER_RELEASE_BASE     github    release download base (pinned to github.com)
EVOSQL_EMBEDDED_DATA_DIR       (per-OS)  override the per-user data directory

Verify it's running

No matter which way you chose, the server listens on the same two ports. Connect with a PostgreSQL client over the wire protocol on 5433. Or connect with the native CLI over EVO on 9967.

psql -h 127.0.0.1 -p 5433 -U admin
./cli/evosql-cli -W admin

The test suite uses testdb as its default database. The login is admin / admin. For client details, see Connect (PG wire & EVO). For the embedded-mode integration, see the MCP server docs.
