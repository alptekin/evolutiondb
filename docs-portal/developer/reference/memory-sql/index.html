Memory & Entity SQL

EvolutionDB ships two purpose-built SQL surfaces for agent memory: MEMORY stores, a namespaced JSON key/value store with optional vector embeddings for semantic search, and ENTITY stores, a key/summary/facts store that tracks how often each entity is mentioned. Both are thin SQL skins over ordinary tables: every statement compiles down to the engine's regular INSERT / SELECT / DELETE path, so RLS policies, MVCC, triggers, and FK cascade all apply unchanged.

How it works

A MEMORY store named foo is backed by a regular table named mem_foo; an ENTITY store named bar is backed by ent_bar. The store itself is registered in the system catalog, and the backing table is created in the current schema. Because the backing object is a normal table, you can attach policies and triggers to it through the dedicated ON MEMORY STORE grammar, and everything a hand-written DML statement supports works here too.

Both surfaces are case-insensitive on keywords, like the rest of EvolutionDB SQL. Store names are bare identifiers (the NAME token). Namespace, key, value, summary, and facts arguments are single-quoted string literals (SQL-standard doubled quotes '' are collapsed to a single ' on write, so values round-trip).

CREATE MEMORY STORE

Creates a memory store and its mem_<name> backing table. The optional WITH list configures the embedding dimension and the distance metric used by MEMORY SEARCH.

CREATE MEMORY STORE name [ IF NOT EXISTS ]   [ WITH ( embedding_dim = N [, distance = 'metric'] ) ]

The backing table columns are:

  • mem_namespace, VARCHAR(255) NOT NULL
  • mem_key, VARCHAR(255) NOT NULL
  • pk, VARCHAR(512) PRIMARY KEY (composite namespace + separator + key)
  • mem_value, JSON
  • embedding, VECTOR(N) (only present when embedding_dim > 0)
  • created_at, TIMESTAMP
  • ttl_at: TIMESTAMP (nullable, NULL by default)

When embedding_dim is 0 (the default), the embedding column is omitted entirely and the store is a pure JSON KV table. Valid distance values are 'cosine' (default), 'l2', 'inner' / 'ip', and 'l1'; any unrecognized value falls back to cosine.

-- pure JSON key/value store
CREATE MEMORY STORE agent_notes;

-- store with 1536-dim embeddings, L2 distance for SEARCH
CREATE MEMORY STORE recall WITH (embedding_dim = 1536, distance = 'l2');

CREATE MEMORY STORE agent_notes IF NOT EXISTS;

The grammar token sequence is literally CREATE MEMORY STORE IF EXISTS name, but it is emitted as IF NOT EXISTS semantics, when the store already exists the statement is a silent no-op rather than an error.

DROP MEMORY STORE

Removes the store and drops its backing table.

DROP MEMORY STORE name [ CASCADE ] DROP MEMORY STORE IF EXISTS name 
DROP MEMORY STORE recall;
DROP MEMORY STORE recall CASCADE;
DROP MEMORY STORE IF EXISTS recall;

Without IF EXISTS, dropping a missing store raises an error (SQLSTATE 42704). The drop locates the backing table in the current schema and removes it, then unregisters the store from the catalog.

MEMORY PUT

Upserts a single row keyed on (namespace, key). PUT semantics are an atomic upsert: on a primary-key collision the engine repoints the PK to the freshly-inserted tuple and drops the old one, so a failed re-insert can never leave the row deleted.

MEMORY PUT INTO name VALUES ( 'ns', 'key', value [, 'embedding'] )

The value position accepts either a string literal or the bare keyword NULL (the mem_val rule is STRING | NULL). When present, the optional fourth argument is the embedding vector as a literal of the form '[f1,f2,...]'; if the store has an embedding column and you omit the vector, the embedding is stored as NULL. created_at is set to the current UTC timestamp; ttl_at defaults to NULL.

-- JSON value, no embedding
MEMORY PUT INTO agent_notes
  VALUES ('user:42', 'fav_color', '{"color":"teal"}');

-- explicit NULL value
MEMORY PUT INTO agent_notes
  VALUES ('user:42', 'cleared', NULL);

-- with embedding vector (store created WITH embedding_dim)
MEMORY PUT INTO recall
  VALUES ('session:7', 'greeting', '{"text":"hi there"}', '[0.12,0.04,-0.91]');

PUT into a store that does not exist raises SQLSTATE 42704.

MEMORY GET

Fetches the single row for an exact (namespace, key) pair. The WHERE clause is fixed: it must be NS = '...' AND KEY = '...'.

MEMORY GET FROM name WHERE NS = 'ns' AND KEY = 'key' 
MEMORY GET FROM agent_notes
  WHERE NS = 'user:42' AND KEY = 'fav_color';

The result projects mem_namespace, mem_key, mem_value, then embedding (only when the store has one), and created_at. The lookup matches on the composite pk column and is limited to one row.

The NS and KEY comparisons must use =. The grammar accepts the generic COMPARISON token in that position, but GET resolves the pair to the exact composite primary key, so only equality is meaningful.

MEMORY SEARCH

Semantic search over a store's embeddings. Scans the store and projects a per-row score computed with the store's configured distance metric, ordered closest-first.

MEMORY SEARCH name USING VECTOR 'query' [ LIMIT k ] 
MEMORY SEARCH recall USING VECTOR '[0.10,0.05,-0.88]';
MEMORY SEARCH recall USING VECTOR '[0.10,0.05,-0.88]' LIMIT 5;

The projection is mem_namespace, mem_key, mem_value, and score. Results are ordered ascending by the score column so the closest matches appear first. When LIMIT is omitted the default is 10.

SEARCH requires the store to have been created with embedding_dim > 0. Running it against a store with no embedding column raises SQLSTATE 42P01.

MEMORY DELETE

Deletes the row for an exact (namespace, key) pair. Like GET, the WHERE clause is fixed to NS = '...' AND KEY = '...'.

MEMORY DELETE FROM name WHERE NS = 'ns' AND KEY = 'key' 
MEMORY DELETE FROM agent_notes
  WHERE NS = 'user:42' AND KEY = 'fav_color';

Internally this emits DELETE FROM mem_<name> WHERE pk = '<ns>\x1E<key>' and runs it through the standard delete path. That means any RLS policy attached to the backing table is enforced, this is what makes MEMORY DELETE multi-tenant safe. The composite key uses Record Separator (0x1E) as the namespace/key joiner internally; you never write that byte yourself.

Both NS and KEY are mandatory in the DELETE grammar. There is no "delete a whole namespace" form on this surface, DELETE always targets exactly one (namespace, key) row. To clear a namespace, list its keys and delete them one by one, or operate on the mem_<name> backing table directly with ordinary SQL.

MEMORY LIST NAMESPACES

Returns the distinct namespaces present in a store, optionally filtered by a prefix.

MEMORY LIST NAMESPACES IN name [ PREFIX 'prefix' ] 
MEMORY LIST NAMESPACES IN agent_notes;
MEMORY LIST NAMESPACES IN agent_notes PREFIX 'user:';

This compiles to SELECT DISTINCT mem_namespace FROM mem_<name>, and when a prefix is given it adds WHERE mem_namespace LIKE 'prefix%'.

ENTITY STORE

An entity store is a key/summary/facts store that also tracks a mention count and a last-seen timestamp, intended for "what does the agent know about this person/thing" memory. It is backed by an ent_<name> table.

CREATE ENTITY STORE name [ IF NOT EXISTS ] DROP ENTITY STORE name [ IF EXISTS ]

The backing table columns are:

  • entity_key, VARCHAR(255) PRIMARY KEY NOT NULL
  • summary, VARCHAR(8000)
  • facts, VARCHAR(8000) (JSON-shaped string)
  • mention_count, INT NOT NULL
  • last_seen, TIMESTAMP NOT NULL
CREATE ENTITY STORE people;
CREATE ENTITY STORE people IF NOT EXISTS;
DROP ENTITY STORE IF EXISTS people;

ENTITY PUT

Upserts an entity by key. On an existing key it updates summary / facts and bumps mention_count and last_seen; otherwise it inserts a new row. The facts argument is optional.

ENTITY PUT INTO name VALUES ( 'key', 'summary' [, 'facts'] ) 
ENTITY PUT INTO people
  VALUES ('alice', 'Lead engineer on the storage team');

ENTITY PUT INTO people
  VALUES ('alice', 'Lead engineer', '{"team":"storage","tz":"UTC+3"}');

Unlike MEMORY PUT, all three positional arguments are plain string literals, there is no NULL keyword form on the entity surface (omit the third argument to leave facts NULL).

ENTITY GET

Fetches a single entity by exact key.

ENTITY GET FROM name WHERE KEY = 'key' 
ENTITY GET FROM people WHERE KEY = 'alice';

Projects entity_key, summary, facts, mention_count, and last_seen, limited to one row.

ENTITY RANK

Returns entities ordered by how often they have been mentioned.

ENTITY RANK FROM name [ LIMIT n ] 
ENTITY RANK FROM people;
ENTITY RANK FROM people LIMIT 10;

Rows come back sorted by mention_count descending, with last_seen descending as the tie-breaker. The projection is the same five columns as ENTITY GET.

Policies and triggers on stores

Because a store's backing table is a regular table, EvolutionDB exposes dedicated ON MEMORY STORE grammar that resolves a store name to its backing table for security and automation:

-- attach an RLS policy to a memory store's backing table
CREATE POLICY tenant_isolation ON MEMORY STORE recall /* ... */;
DROP POLICY tenant_isolation ON MEMORY STORE recall;

-- toggle row-level security on a memory store
ALTER MEMORY STORE recall ENABLE ROW level SECURITY;
ALTER MEMORY STORE recall DISABLE ROW level SECURITY;

-- row-level trigger on a memory store
CREATE TRIGGER audit_writes BEFORE INSERT ON MEMORY STORE recall
  FOR EACH ROW BEGIN /* ... */ END;

Since MEMORY DELETE routes through the engine's standard delete path, any policy you attach this way is enforced on those deletes as well as on direct DML against the mem_<name> table.
