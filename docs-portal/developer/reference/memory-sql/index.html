Memory & Entity SQL

EvolutionDB has two SQL surfaces built for agent memory. MEMORY stores are a namespaced JSON key/value store. They can also hold vector embeddings for semantic search. ENTITY stores hold a key, a summary, and facts. They also track how often each entity is mentioned. Both surfaces are thin SQL skins over normal tables. Every statement compiles down to the engine's regular INSERT, SELECT, and DELETE path. RLS policies, MVCC, triggers, and FK cascade all work the same way.

How it works

A MEMORY store named foo uses a regular table named mem_foo. An ENTITY store named bar uses a table named ent_bar. The store is registered in the system catalog. The backing table is created in the current schema. The backing object is a normal table. You can attach policies and triggers to it through the ON MEMORY STORE grammar. Everything that a hand-written DML statement supports works here too.

Both surfaces treat keywords as case-insensitive. This is the same as the rest of EvolutionDB SQL. Store names are bare identifiers. They use the NAME token. The namespace, key, value, summary, and facts arguments are single-quoted string literals. A doubled quote '' is turned into a single ' on write. Values round-trip correctly.

CREATE MEMORY STORE

Creates a memory store and its mem_<name> backing table. The optional WITH list sets the embedding dimension. It also sets the distance metric that MEMORY SEARCH uses.

CREATE MEMORY STORE name [ IF NOT EXISTS ]   [ WITH ( embedding_dim = N [, distance = 'metric'] ) ]

The backing table has these columns.

  • mem_namespace, VARCHAR(255) NOT NULL
  • mem_key, VARCHAR(255) NOT NULL
  • pk, VARCHAR(512) PRIMARY KEY. It is namespace plus a separator plus key.
  • mem_value, JSON
  • embedding, VECTOR(N). This column exists only when embedding_dim > 0.
  • created_at, TIMESTAMP
  • ttl_at, TIMESTAMP. It can be NULL. It is NULL by default.

By default embedding_dim is 0. When it is 0, the embedding column is left out. The store is then a pure JSON key/value table. The valid distance values are 'cosine', 'l2', 'inner', 'ip', and 'l1'. The default is 'cosine'. Any value that is not in this list falls back to cosine.

-- pure JSON key/value store
CREATE MEMORY STORE agent_notes;

-- store with 1536-dim embeddings, L2 distance for SEARCH
CREATE MEMORY STORE recall WITH (embedding_dim = 1536, distance = 'l2');

CREATE MEMORY STORE agent_notes IF NOT EXISTS;

The grammar token sequence is written as CREATE MEMORY STORE IF EXISTS name. It works with IF NOT EXISTS semantics. When the store already exists, the statement does nothing. It does not raise an error.

DROP MEMORY STORE

Removes the store and drops its backing table.

DROP MEMORY STORE name [ CASCADE ] DROP MEMORY STORE IF EXISTS name 
DROP MEMORY STORE recall;
DROP MEMORY STORE recall CASCADE;
DROP MEMORY STORE IF EXISTS recall;

Without IF EXISTS, dropping a store that does not exist raises an error. The SQLSTATE is 42704. The drop finds the backing table in the current schema and removes it. Then it removes the store from the catalog.

MEMORY PUT

Upserts a single row. The row is keyed on (namespace, key). PUT is an atomic upsert. When the primary key already exists, the engine points the PK to the new tuple and drops the old one. A failed re-insert can never leave the row deleted.

MEMORY PUT INTO name VALUES ( 'ns', 'key', value [, 'embedding'] )

The value position takes a string literal or the bare keyword NULL. The mem_val rule is STRING | NULL. The fourth argument is optional. It is the embedding vector. It uses a literal of the form '[f1,f2,...]'. If the store has an embedding column and you leave out the vector, the embedding is stored as NULL. created_at is set to the current UTC timestamp. ttl_at defaults to NULL.

-- JSON value, no embedding
MEMORY PUT INTO agent_notes
  VALUES ('user:42', 'fav_color', '{"color":"teal"}');

-- explicit NULL value
MEMORY PUT INTO agent_notes
  VALUES ('user:42', 'cleared', NULL);

-- with embedding vector (store created WITH embedding_dim)
MEMORY PUT INTO recall
  VALUES ('session:7', 'greeting', '{"text":"hi there"}', '[0.12,0.04,-0.91]');

A PUT into a store that does not exist raises SQLSTATE 42704.

MEMORY GET

Fetches the single row for an exact (namespace, key) pair. The WHERE clause is fixed. It must be NS = '...' AND KEY = '...'.

MEMORY GET FROM name WHERE NS = 'ns' AND KEY = 'key' 
MEMORY GET FROM agent_notes
  WHERE NS = 'user:42' AND KEY = 'fav_color';

The result returns mem_namespace, mem_key, mem_value, then embedding, and created_at. The embedding column appears only when the store has one. The lookup matches on the composite pk column. It returns at most one row.

The NS and KEY comparisons must use =. The grammar accepts the generic COMPARISON token in that spot. GET turns the pair into the exact composite primary key. Only equality has any meaning here.

MEMORY SEARCH

Runs a semantic search over a store's embeddings. It scans the store and returns a score for each row. The score uses the store's distance metric. The closest matches come first.

MEMORY SEARCH name USING VECTOR 'query' [ LIMIT k ] 
MEMORY SEARCH recall USING VECTOR '[0.10,0.05,-0.88]';
MEMORY SEARCH recall USING VECTOR '[0.10,0.05,-0.88]' LIMIT 5;

The result returns mem_namespace, mem_key, mem_value, and score. Results are sorted by the score column in ascending order. The closest matches appear first. When you leave out LIMIT, the default is 10.

SEARCH needs the store to have been created with embedding_dim > 0. Running it on a store with no embedding column raises SQLSTATE 42P01.

MEMORY DELETE

Deletes the row for an exact (namespace, key) pair. The WHERE clause is fixed to NS = '...' AND KEY = '...'. This is the same form that GET uses.

MEMORY DELETE FROM name WHERE NS = 'ns' AND KEY = 'key' 
MEMORY DELETE FROM agent_notes
  WHERE NS = 'user:42' AND KEY = 'fav_color';

Inside the engine, this emits DELETE FROM mem_<name> WHERE pk = '<ns>\x1E<key>'. It runs through the standard delete path. Any RLS policy attached to the backing table is enforced. This is what makes MEMORY DELETE safe for multiple tenants. The composite key joins the namespace and key with the Record Separator byte 0x1E. You never write that byte yourself.

Both NS and KEY are required in the DELETE grammar. There is no "delete a whole namespace" form here. DELETE always targets exactly one (namespace, key) row. To clear a namespace, list its keys and delete them one by one. You can also work on the mem_<name> backing table directly with normal SQL.

MEMORY LIST NAMESPACES

Returns the distinct namespaces in a store. You can filter them by a prefix.

MEMORY LIST NAMESPACES IN name [ PREFIX 'prefix' ] 
MEMORY LIST NAMESPACES IN agent_notes;
MEMORY LIST NAMESPACES IN agent_notes PREFIX 'user:';

This compiles to SELECT DISTINCT mem_namespace FROM mem_<name>. When you give a prefix, it adds WHERE mem_namespace LIKE 'prefix%'.

ENTITY STORE

An entity store holds a key, a summary, and facts. It also tracks a mention count and a last-seen timestamp. It is made for "what does the agent know about this person or thing" memory. It uses an ent_<name> table.

CREATE ENTITY STORE name [ IF NOT EXISTS ] DROP ENTITY STORE name [ IF EXISTS ]

The backing table has these columns.

  • entity_key, VARCHAR(255) PRIMARY KEY NOT NULL
  • summary, VARCHAR(8000)
  • facts, VARCHAR(8000). It holds a JSON-shaped string.
  • mention_count, INT NOT NULL
  • last_seen, TIMESTAMP NOT NULL
CREATE ENTITY STORE people;
CREATE ENTITY STORE people IF NOT EXISTS;
DROP ENTITY STORE IF EXISTS people;

ENTITY PUT

Upserts an entity by key. When the key already exists, it updates summary and facts. It also raises mention_count and updates last_seen. When the key is new, it inserts a new row. The facts argument is optional.

ENTITY PUT INTO name VALUES ( 'key', 'summary' [, 'facts'] ) 
ENTITY PUT INTO people
  VALUES ('alice', 'Lead engineer on the storage team');

ENTITY PUT INTO people
  VALUES ('alice', 'Lead engineer', '{"team":"storage","tz":"UTC+3"}');

Unlike MEMORY PUT, all three arguments are plain string literals. The entity surface has no NULL keyword form. To leave facts NULL, just omit the third argument.

ENTITY GET

Fetches a single entity by exact key.

ENTITY GET FROM name WHERE KEY = 'key' 
ENTITY GET FROM people WHERE KEY = 'alice';

Returns entity_key, summary, facts, mention_count, and last_seen. It returns at most one row.

ENTITY RANK

Returns entities ordered by how often they have been mentioned.

ENTITY RANK FROM name [ LIMIT n ] 
ENTITY RANK FROM people;
ENTITY RANK FROM people LIMIT 10;

Rows come back sorted by mention_count in descending order. When two rows tie, last_seen descending breaks the tie. The result has the same five columns as ENTITY GET.

Policies and triggers on stores

A store's backing table is a regular table. EvolutionDB gives you the ON MEMORY STORE grammar. It maps a store name to its backing table. You use it for security and automation.

-- attach an RLS policy to a memory store's backing table
CREATE POLICY tenant_isolation ON MEMORY STORE recall /* ... */;
DROP POLICY tenant_isolation ON MEMORY STORE recall;

-- toggle row-level security on a memory store
ALTER MEMORY STORE recall ENABLE ROW level SECURITY;
ALTER MEMORY STORE recall DISABLE ROW level SECURITY;

-- row-level trigger on a memory store
CREATE TRIGGER audit_writes BEFORE INSERT ON MEMORY STORE recall
  FOR EACH ROW BEGIN /* ... */ END;

MEMORY DELETE routes through the engine's standard delete path. Any policy you attach this way is enforced on those deletes. It is also enforced on direct DML against the mem_<name> table.
