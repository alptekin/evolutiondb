EVO text protocol

EVO is EvolutionDB's native, line-oriented wire protocol served on TCP port 9967. Every message is a UTF-8 text line terminated by \n. A session walks through a fixed handshake, greeting, optional STARTTLS, then authentication, before entering a request/response command loop where each reply is bracketed by a trailing READY marker. This page documents the exact tokens emitted and accepted by adaptor/evo_protocol.c.

All framing below is taken verbatim from the server implementation. Tokens are case-sensitive on the server's output side; client commands are parsed with strncasecmp, so command keywords (SQL, QUIT, PREPARE, …) are accepted case-insensitively.

Overview

A complete session has two phases:

  1. Handshake, client greeting, server HELLO, optional STARTTLS upgrade, then AUTH_SCRAM/AUTH exchange ending in AUTH_OK.
  2. Command loop, the client issues commands (SQL, PREPARE, EXECUTE, EXECUTE_BATCH, EXECUTE_BATCH_BINARY, DEALLOCATE, QUIT); the server answers and emits a final READY line per command.

Lines are read with the connection's line reader; the server's main read buffer is 65536 bytes. All I/O passes through a conn_t wrapper so the same code works transparently over plaintext or TLS.

Handshake

The handshake is strictly ordered. Each step below is a single line unless noted.

Greeting

The client opens by sending the literal greeting line:

EVO

If the first line is anything other than EVO, the server replies with an error and closes the connection:

ERR 08000 Expected EVO greeting

On a valid greeting the server responds with its version banner:

HELLO EvoSQL 1.0

STARTTLS upgrade

If TLS is compiled in and available, the server offers an upgrade before authentication so credentials are never sent in cleartext:

// server, only when TLS is available
STARTTLS

The client then either accepts or declines:

  • Client sends STARTTLS → the server performs the TLS handshake (conn_tls_accept) and the rest of the session is encrypted. A failed handshake closes the connection.
  • Client sends anything else → the server logs the decline and continues in plaintext for backward compatibility.

The STARTTLS offer is only emitted when the server build has TLS support. A client must not assume it will always appear; read the line after HELLO and branch on whether it is STARTTLS or the AUTH_SCRAM challenge.

Authentication

The server advertises SCRAM-SHA-256 by sending:

AUTH_SCRAM

The client replies with one of two formats. The next line's prefix selects the path:

SCRAM-SHA-256 (preferred)

SCRAM never transmits the password, so it protects the secret even without TLS. The exchange is three client/server message pairs:

SCRAM-CLIENT-FIRST <client-first-message>

The server processes the client-first message and replies with the server-first message (the raw SCRAM line produced by scram_server_first). The client then sends:

SCRAM-CLIENT-FINAL <client-final-message>

The server verifies the client proof and replies with the server-final message (containing the ServerSignature for mutual authentication). Failures at any step emit a SQLSTATE 28P01 error and close the connection, for example:

ERR 28P01 SCRAM: invalid client-first message
ERR 28P01 Expected SCRAM-CLIENT-FINAL
ERR 28P01 SCRAM authentication failed

Legacy cleartext AUTH

For backward compatibility a client may instead send a single space-separated line:

AUTH <username> <password>

The username runs up to the first whitespace; everything after the separating space is taken as the password. On failure:

ERR 28P01 Authentication failed

If the line is neither a SCRAM client-first nor an AUTH line:

ERR 28000 Expected AUTH or SCRAM-CLIENT-FIRST

Success

After either path authenticates, the server confirms and the session enters the command loop:

AUTH_OK

The default credentials in development are admin/admin. The server's default session context is database default with isolation level READ COMMITTED.

Command loop

After AUTH_OK the server loops reading one command line at a time. If an idle timeout is configured for the session, the server uses select() to wait; on timeout it emits ERR 57P05 idle timeout and closes.

Every command's response ends with a standalone:

READY

The one exception is QUIT, which ends the session.

QUIT

QUIT

The server replies BYE and ends the loop. (Any open transaction is auto-rolled back during teardown, see below.)

BYE

SQL request

A query is sent as a header line declaring the byte length, immediately followed by the SQL body on its own line:

SQL <length>\n<sql-text>\n

The declared <length> must be greater than 0 and at most 65000, otherwise:

ERR 42000 Invalid SQL length
READY

The body is read as the next line. The server then executes the statement and produces one of four response shapes (all followed by READY):

  • Error, ERR <sqlstate> <message>
  • Result set (SELECT), the RESULT framing described below
  • COPY stream, COPY_READY followed by a streamed body
  • Command (non-SELECT), OK then TAG <command_tag>

RESULT framing (SELECT)

A SELECT result set is framed as a fixed sequence of tokens. Column metadata first, then rows, then the end marker and command tag:

RESULT
COLS <num_cols>
COL <name> <pg_type_oid> <type_modifier>
... one COL line per column ...
ROW
FIELD <value>
... one FIELD line per column ...
... one ROW block per row ...
END
TAG <command_tag>

Token semantics, exactly as emitted by evo_send_result:

  • RESULT, opens the result set.
  • COLS <num_cols>, the column count.
  • COL <name> <pg_type_oid> <type_modifier>: one line per column, carrying the column name, its PostgreSQL type OID, and the type modifier.
  • ROW, opens a row; followed by exactly num_cols FIELD lines.
  • FIELD <value>, one field value. A SQL NULL (or a missing field pointer) is sent as the literal token FIELD NULL.
  • END, closes the row stream.
  • TAG <command_tag>, the command tag (e.g. SELECT).

The complete reply is then closed with READY.

FIELD NULL is indistinguishable at the token level from a non-null value whose text happens to be the four characters NULL. The server sends FIELD NULL only when the field is actually a SQL NULL, but clients that need to round-trip a string literal "NULL" should be aware of the ambiguity in the text framing.

OK / TAG (non-SELECT)

For statements that are not SELECT and not a COPY stream, the server replies with:

OK
TAG <command_tag>
READY

The command tag is the engine's result tag, e.g. INSERT 0 1, UPDATE 3, DELETE 2.

ERR responses

Errors are a single line carrying the SQLSTATE and message:

ERR <sqlstate> <message>

For query-execution errors the SQLSTATE and message come from the result set (rs->error_sqlstate, rs->error_message). Protocol-level errors use fixed SQLSTATEs seen in the source, including 08000 (bad greeting), 28000/28P01 (auth), 42000 (bad command / invalid length / unknown command), 57P05 (idle timeout), 53000 (resource limits), 26000 (prepared statement not found), and 22000 (COPY failure). An unrecognized command produces:

ERR 42000 Unknown command: <line>
READY

COPY streaming

When a SQL request is a COPY statement, the server enters a streaming sub-protocol instead of the normal result framing. It first signals readiness:

COPY_READY

COPY FROM STDIN

After COPY_READY, the client sends one CSV-formatted row per line, each \n-terminated, and signals end-of-data with the PostgreSQL sentinel line:

\.

On success the server replies with the row count as a COPY tag; on failure a SQLSTATE 22000 error:

OK
TAG COPY <n>
READY

// on failure
ERR 22000 <message>
READY

COPY TO STDOUT

After COPY_READY, the server streams one row per line, each prefixed with D:, then emits the terminator sentinel, then the COPY tag:

COPY_READY
D:<csv-line>
... one D: line per row ...
\.
OK
TAG COPY <n>
READY

The \. terminator is always emitted (even on error) so the client knows the stream has ended.

Prepared statements

EVO supports server-side prepared statements with positional $N placeholders (1-based). The session can hold at most 32 prepared statements.

PREPARE

PREPARE <stmt_name> <byte_length>\n<sql-body>\n

The body length must be 1.65000. The server counts $1.$9-style placeholders to derive the parameter count, stores the template, and replies:

OK
TAG PREPARE
READY

Error cases include a missing name (42000), invalid length (42000), more than 32 statements (53000 Too many prepared statements (max 32)), and out-of-memory (53000). Re-preparing an existing name overwrites it.

EXECUTE

EXECUTE <stmt_name> <param_count>\n<param1>\n<param2>\n...

The header is followed by param_count lines, one per parameter value. The server substitutes each $N in the template: the special value \N becomes the SQL literal NULL; every other value is wrapped in single quotes with internal quotes doubled. The substituted SQL is then executed and answered with either the RESULT framing (SELECT) or OK/TAG (otherwise), followed by READY.

If the named statement does not exist, the server still drains the declared parameter lines to stay in sync, then:

ERR 26000 Prepared statement '<name>' not found
READY

EXECUTE_BATCH

EXECUTE_BATCH <stmt_name> <row_count> <param_count>\n<row1_param1>\n...<rowN_paramM>\n

Sends row_count × param_count parameter lines in row-major order. Each row reuses the same substitution rules as EXECUTE (\N → NULL, otherwise quoted). The server sums affected rows from each command tag (INSERT/UPDATE/DELETE) and replies with one of:

BATCH_OK <total_affected>
READY

// on the first failing row
BATCH_ERR <row_index> <sqlstate> <msg>
READY

On a row error the server drains the remaining parameter lines so the protocol stays aligned. Invalid arguments yield ERR 42000 EXECUTE_BATCH: invalid arguments.

EXECUTE_BATCH_BINARY

A binary variant that avoids text re-encoding of parameters. The header is identical, but the payload that follows is a raw binary stream rather than text lines:

EXECUTE_BATCH_BINARY <stmt_name> <row_count> <param_count>\n<binary payload>

For each row, for each parameter in order, the payload is a 1-byte type code optionally followed by its value:

CodeTypePayload
NNULLnone
iint44 bytes, big-endian
lint88 bytes, big-endian
dfloat88 bytes, big-endian IEEE 754
bbool1 byte (0 = false, non-zero = true)
sstring4-byte big-endian length + UTF-8 bytes (no NUL); max 16 MB

The server decodes each binary parameter into its text form and reuses the same substitution and execution path as EXECUTE_BATCH. The response is identical to EXECUTE_BATCH (BATCH_OK <total_affected> or BATCH_ERR <row_index> <sqlstate> <msg>, each followed by READY). Here param_count must be 0.32; out-of-range arguments yield ERR 42000 EXECUTE_BATCH_BINARY: invalid arguments.

Unlike the text batch, the binary batch has no fixed-size tail after an error. If the server replies BATCH_ERR, the client MUST stop sending payload bytes, the next read begins a fresh command. If the statement name is not found, the server drains the entire payload (decoding it field-by-field) and then replies ERR 26000.

DEALLOCATE

DEALLOCATE <stmt_name>

Frees a prepared statement and compacts the slot table. Success:

OK
TAG DEALLOCATE
READY

A missing name yields ERR 42000 DEALLOCATE: missing statement name; an unknown statement yields ERR 26000 Prepared statement '<name>' not found.

Async NOTIFY frames

LISTEN/NOTIFY delivery is pushed asynchronously between replies. A notification is a two-part, line-oriented frame the client can read between two READY markers:

NOTIFY <sender_session_id> <channel> <payload_len>
<payload>

The header gives the originating session id, the channel name, and the payload byte length; the payload bytes follow on the next line. The frame is written as a unit under the connection's write lock so it cannot interleave with a multi-line RESULT being streamed.

Session teardown

When the loop ends (client QUIT, disconnect, or idle timeout), the server:

  • Frees all prepared statements for the session.
  • Auto-rolls back any open transaction (the undo log is replayed and freed). For a SERIALIZABLE transaction it also unregisters the transaction from the conflict graph.
  • Drops LISTEN/NOTIFY state and marks the connection invalid so in-flight async senders abort their writes.
  • Unregisters from the session registry and drops session-scoped temporary tables.
  • Shuts down TLS (if active). The socket itself is closed by server.c, not the protocol handler.

Auto-rollback on disconnect means an EVO client that drops mid-transaction never leaves partial work committed, the same guarantee the PG wire handler provides.
