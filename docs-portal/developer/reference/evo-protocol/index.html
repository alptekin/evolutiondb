EVO text protocol

EVO is EvolutionDB's own wire protocol. It works on lines of text. It runs on TCP port 9967. Every message is one line of UTF-8 text. Each line ends with \n. A session follows a fixed order. First it does a handshake. Then it sends a greeting. Then it may upgrade with STARTTLS. Then it does authentication. After that it enters a command loop. The client sends a command and the server sends back a reply. Each reply ends with a READY marker. This page lists the exact tokens that adaptor/evo_protocol.c sends and accepts.

All framing below comes straight from the server code. The tokens the server sends are case-sensitive. Client commands are parsed with strncasecmp. So command keywords like SQL, QUIT, and PREPARE work in any case.

Overview

A full session has two phases.

  1. Handshake. The client sends a greeting. The server sends HELLO. The session may upgrade with STARTTLS. Then comes the AUTH_SCRAM or AUTH exchange. It ends with AUTH_OK.
  2. Command loop. The client sends commands like SQL, PREPARE, EXECUTE, EXECUTE_BATCH, EXECUTE_BATCH_BINARY, DEALLOCATE, and QUIT. The server answers. It sends a final READY line for each command.

The server reads lines with the connection's line reader. The main read buffer is 65536 bytes. All I/O goes through a conn_t wrapper. The same code works the same way over plaintext or TLS.

Handshake

The handshake follows a strict order. Each step below is a single line unless noted.

Greeting

The client starts by sending the greeting line.

EVO

The first line must be EVO. If it is anything else, the server sends an error and closes the connection.

ERR 08000 Expected EVO greeting

When the greeting is valid, the server sends back its version banner.

HELLO EvoSQL 1.0

STARTTLS upgrade

The server offers an upgrade when TLS is built in and ready. It offers this before authentication. The password is never sent in cleartext.

// server, only when TLS is available
STARTTLS

The client can accept or decline.

  • The client sends STARTTLS. The server runs the TLS handshake with conn_tls_accept. The rest of the session is then encrypted. A failed handshake closes the connection.
  • The client sends anything else. The server logs the decline and keeps going in plaintext. This keeps old clients working.

The server only sends the STARTTLS offer when the build has TLS support. A client must not assume it always appears. Read the line after HELLO. Then check if it is STARTTLS or the AUTH_SCRAM challenge.

Authentication

The server offers SCRAM-SHA-256. It sends this challenge.

AUTH_SCRAM

The client replies with one of two formats. The prefix of the next line picks the path.

SCRAM-SHA-256 (preferred)

SCRAM never sends the password. It keeps the secret safe even without TLS. The exchange has three client and server message pairs.

SCRAM-CLIENT-FIRST <client-first-message>

The server reads the client-first message. It replies with the server-first message. This is the raw SCRAM line that scram_server_first makes. The client then sends the final message.

SCRAM-CLIENT-FINAL <client-final-message>

The server checks the client proof. It replies with the server-final message. This message holds the ServerSignature for mutual authentication. A failure at any step sends a SQLSTATE 28P01 error and closes the connection. Here are some examples:

ERR 28P01 SCRAM: invalid client-first message
ERR 28P01 Expected SCRAM-CLIENT-FINAL
ERR 28P01 SCRAM authentication failed

Legacy cleartext AUTH

To keep old clients working, a client may instead send one line with spaces.

AUTH <username> <password>

The username runs up to the first whitespace. Everything after the space is the password. If the login fails, the server sends this error.

ERR 28P01 Authentication failed

The line must be a SCRAM client-first line or an AUTH line. If it is neither, the server sends:

ERR 28000 Expected AUTH or SCRAM-CLIENT-FIRST

Success

After either path passes, the server confirms it. The session then enters the command loop.

AUTH_OK

The default login for development is admin with the password admin. The default session uses the database default. The isolation level is READ COMMITTED.

Command loop

After AUTH_OK the server loops. It reads one command line at a time. If the session has an idle timeout, the server uses select() to wait. On timeout it sends ERR 57P05 idle timeout and closes.

Every command response ends with a line on its own.

READY

The one exception is QUIT. It ends the session.

QUIT

QUIT

The server replies BYE and ends the loop. Any open transaction is rolled back during teardown.

BYE

SQL request

A query is sent in two parts. First comes a header line that gives the byte length. Right after it comes the SQL body on its own line.

SQL <length>\n<sql-text>\n

The <length> must be greater than 0 and at most 65000. If it is not, the server sends:

ERR 42000 Invalid SQL length
READY

The body is read as the next line. The server then runs the statement. It gives back one of four kinds of response. Each one is followed by READY.

  • Error. This is ERR <sqlstate> <message>.
  • Result set for a SELECT. This uses the RESULT framing shown below.
  • COPY stream. This is COPY_READY followed by a streamed body.
  • Command for a non-SELECT. This is OK then TAG <command_tag>.

RESULT framing (SELECT)

A SELECT result set uses a fixed sequence of tokens. The column metadata comes first. Then come the rows. Then come the end marker and the command tag.

RESULT
COLS <num_cols>
COL <name> <pg_type_oid> <type_modifier>
... one COL line per column ...
ROW
FIELD <value>
... one FIELD line per column ...
... one ROW block per row ...
END
TAG <command_tag>

Here is what each token means. This is exactly what evo_send_result sends.

  • RESULT opens the result set.
  • COLS <num_cols> gives the column count.
  • COL <name> <pg_type_oid> <type_modifier> is one line per column. It carries the column name, its PostgreSQL type OID, and the type modifier.
  • ROW opens a row. It is followed by exactly num_cols FIELD lines.
  • FIELD <value> is one field value. A SQL NULL or a missing field pointer is sent as the token FIELD NULL.
  • END closes the row stream.
  • TAG <command_tag> is the command tag, for example SELECT.

The full reply then ends with READY.

At the token level, FIELD NULL looks the same as a real value whose text is the four characters NULL. The server sends FIELD NULL only when the field is a true SQL NULL. Some clients need to send back a string value of "NULL". Those clients should know that the text framing cannot tell the two apart.

OK / TAG (non-SELECT)

Some statements are not SELECT and not a COPY stream. For those, the server sends this reply.

OK
TAG <command_tag>
READY

The command tag is the engine's result tag. For example, INSERT 0 1, UPDATE 3, or DELETE 2.

ERR responses

An error is a single line. It carries the SQLSTATE and the message.

ERR <sqlstate> <message>

For query errors the SQLSTATE and message come from the result set. These are rs->error_sqlstate and rs->error_message. Protocol errors use fixed SQLSTATEs from the source code. The codes are listed below.

  • 08000 for a bad greeting.
  • 28000 and 28P01 for auth.
  • 42000 for a bad command, an invalid length, or an unknown command.
  • 57P05 for an idle timeout.
  • 53000 for resource limits.
  • 26000 for a prepared statement that is not found.
  • 22000 for a COPY failure.

An unknown command returns this error.

ERR 42000 Unknown command: <line>
READY

COPY streaming

A SQL request can be a COPY statement. The server then uses a streaming sub-protocol instead of the normal result framing. First it signals that it is ready.

COPY_READY

COPY FROM STDIN

After COPY_READY, the client sends the data. It sends one CSV row per line. Each line ends with \n. The client marks the end of the data with the PostgreSQL sentinel line.

\.

On success the server replies with the row count as a COPY tag. On failure it sends a SQLSTATE 22000 error.

OK
TAG COPY <n>
READY

// on failure
ERR 22000 <message>
READY

COPY TO STDOUT

After COPY_READY, the server streams the data. It sends one row per line. Each line starts with D:. Then it sends the terminator sentinel. Then it sends the COPY tag.

COPY_READY
D:<csv-line>
... one D: line per row ...
\.
OK
TAG COPY <n>
READY

The server always sends the \. terminator. It sends it even on error. This way the client knows the stream has ended.

Prepared statements

EVO supports server-side prepared statements. They use positional $N placeholders. The placeholders start at 1. A session can hold at most 32 prepared statements.

PREPARE

PREPARE <stmt_name> <byte_length>\n<sql-body>\n

The body length must be 1 to 65000. The server counts the $1 to $9 placeholders to find the parameter count. It stores the template and replies.

OK
TAG PREPARE
READY

Several things can cause an error. A missing name gives 42000. An invalid length gives 42000. More than 32 statements gives 53000 Too many prepared statements (max 32). Out of memory gives 53000. If you prepare a name that already exists, the new one replaces the old one.

EXECUTE

EXECUTE <stmt_name> <param_count>\n<param1>\n<param2>\n...

The header is followed by param_count lines. There is one line per parameter value. The server puts each value into the matching $N in the template. The special value \N becomes the SQL value NULL. Every other value is wrapped in single quotes. Any quotes inside the value are doubled. The server then runs the filled-in SQL. It answers with the RESULT framing for a SELECT, or with OK and TAG for anything else. Then it sends READY.

The named statement may not exist. In that case the server still reads the declared parameter lines so the protocol stays in sync. Then it sends this error.

ERR 26000 Prepared statement '<name>' not found
READY

EXECUTE_BATCH

EXECUTE_BATCH <stmt_name> <row_count> <param_count>\n<row1_param1>\n...<rowN_paramM>\n

This sends row_count × param_count parameter lines. They are in row-major order. Each row uses the same rules as EXECUTE. So \N becomes NULL, and every other value is quoted. The server adds up the affected rows from each command tag. The tags are INSERT, UPDATE, and DELETE. It then replies with one of these:

BATCH_OK <total_affected>
READY

// on the first failing row
BATCH_ERR <row_index> <sqlstate> <msg>
READY

On a row error the server reads the rest of the parameter lines so the protocol stays aligned. Invalid arguments give ERR 42000 EXECUTE_BATCH: invalid arguments.

EXECUTE_BATCH_BINARY

This is a binary version. It avoids turning the parameters back into text. The header is the same. But the payload after it is a raw binary stream, not text lines.

EXECUTE_BATCH_BINARY <stmt_name> <row_count> <param_count>\n<binary payload>

The payload holds the rows in order. Each row holds its parameters in order. Each parameter starts with a 1-byte type code. The value may follow the code.

CodeTypePayload
NNULLnone
iint44 bytes, big-endian
lint88 bytes, big-endian
dfloat88 bytes, big-endian IEEE 754
bbool1 byte. 0 is false. Non-zero is true.
sstring4-byte big-endian length, then UTF-8 bytes with no NUL. Max 16 MB.

The server turns each binary parameter into its text form. It then uses the same substitution and run path as EXECUTE_BATCH. The response is the same as EXECUTE_BATCH. It is BATCH_OK <total_affected> or BATCH_ERR <row_index> <sqlstate> <msg>. Each one is followed by READY. Here param_count must be 0 to 32. An out-of-range argument gives ERR 42000 EXECUTE_BATCH_BINARY: invalid arguments.

The binary batch has no fixed-size tail after an error. The text batch works differently here. If the server replies BATCH_ERR, the client MUST stop sending payload bytes. The next read starts a fresh command. If the statement name is not found, the server reads the whole payload. It decodes it field by field. Then it replies ERR 26000.

DEALLOCATE

DEALLOCATE <stmt_name>

This frees a prepared statement. It also compacts the slot table. On success the server sends this response.

OK
TAG DEALLOCATE
READY

A missing name gives ERR 42000 DEALLOCATE: missing statement name. An unknown statement gives ERR 26000 Prepared statement '<name>' not found.

Async NOTIFY frames

The server pushes LISTEN/NOTIFY messages on its own time. It sends them between replies. A notification is a frame made of lines. The client can read it between two READY markers.

NOTIFY <sender_session_id> <channel> <payload_len>
<payload>

The header gives the sender session id, the channel name, and the payload byte length. The payload bytes follow on the next line. The server writes the frame as one unit under the connection's write lock. It cannot mix with a multi-line RESULT that is being streamed.

Session teardown

The loop can end in three ways. The client can send QUIT. The client can disconnect. The session can hit an idle timeout. When the loop ends, the server does these things.

  • It frees all prepared statements for the session.
  • It rolls back any open transaction. The undo log is replayed and freed. For a SERIALIZABLE transaction it also removes the transaction from the conflict graph.
  • It drops the LISTEN/NOTIFY state. It marks the connection invalid. Any in-flight async senders stop their writes.
  • It removes the session from the session registry. It drops session-scoped temporary tables.
  • It shuts down TLS if TLS is active. The socket itself is closed by server.c, not by the protocol handler.

Auto-rollback on disconnect protects your data. An EVO client may drop in the middle of a transaction. It never leaves partial work committed. The PG wire handler gives the same guarantee.
