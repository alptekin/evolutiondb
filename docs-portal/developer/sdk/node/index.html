Node.js SDK

evosql-memory is a Node.js and TypeScript binding. It sits on top of the native libevosql-memory C library. It gives you a Connection class for raw SQL. It also has helpers for agent memory and LangGraph checkpoints.

What it does

The node-evosql-memory package is a thin ffi-napi binding over the C SDK libevosql-memory. It loads the shared library when you call require. It maps the C surface with ffi-napi and ref-napi. It gives you a single Connection class. On top of plain exec and query it adds helper methods. The agent memory store uses memoryPut, memoryGet, and memoryDelete. LangGraph checkpoints use checkpointPut and checkpointGetLatest. It connects to a running EvolutionDB server over the network. It is a client, not an embedded engine.

It uses FFI over a native .dylib or .so file. The shared library must be present on the host. See Configure for more. It supports Node 18 and above.

The MCP server opens the same memory store to AI agents. You can read more about the main ideas on the project wiki.

Install

The package name is evosql-memory. In the monorepo it lives at client/node-evosql-memory. It depends on ffi-napi at ^4.0.3 and ref-napi at ^3.0.3. You also need the native shared library. You can build it from source or download a pre-built binary.

# Build the C SDK first, then install the Node deps
make -C ../libevosql-memory                 # build libevosql-memory
cd client/node-evosql-memory
npm install                                  # ffi-napi + ref-napi
npm run example                              # runs examples/put_get.js

You can also use a pre-built binary. Every GitHub Release has binaries for Linux, macOS, and Windows. Download the one for your platform and unpack it. Then point EVOSQL_MEMORY_LIB at the shared library. This lets you skip the make step.

# https://github.com/alptekin/evolutiondb/releases/latest
export EVOSQL_MEMORY_LIB=/path/to/libevosql-memory.dylib

Configure

There is no config file. You pass connection settings to the Connection constructor in order. The settings are host, port, user, and password. The examples use 127.0.0.1, port 9967, and admin for both the user and the password.

The SDK reads only one environment variable. It is the location of the shared library. At load time the binding looks for libevosql-memory.{dylib,so} in this exact order:

  1. process.env.EVOSQL_MEMORY_LIB. This is an explicit path. It is used only if the file exists.
  2. ../../libevosql-memory/build/libevosql-memory.{dylib,so}. This is the in-tree SDK build directory. The path is relative to lib/index.js.
  3. The system loader path, such as /usr/local/lib or /usr/lib. Here ffi.Library gets the bare name evosql-memory.

The file extension comes from process.platform. It is .dylib on macOS, which is darwin. On other systems it is .so.

The EvolutionDB server controls authentication. The SDK forwards the user and password you pass to evo_connect. The binding itself does not store credentials or handle tokens.

Usage

Open a connection and run statements. Always close the connection in a finally block.

const { Connection } = require('evosql-memory');

const c = new Connection('127.0.0.1', 9967, 'admin', 'admin');
try {
    c.exec('CREATE MEMORY STORE demo');
    c.memoryPut('demo', 'users', 'alice', '{"role":"pm"}');
    console.log(c.memoryGet('demo', 'users', 'alice'));
} finally {
    c.close();
}

The smoke example in examples/put_get.js shows the full cycle. It creates a store, puts a value, gets it, and drops the store.

const { Connection } = require('..');

const c = new Connection('127.0.0.1', 9967, 'admin', 'admin');
try {
    c.exec('DROP MEMORY STORE IF EXISTS node_demo');
    c.exec('CREATE MEMORY STORE node_demo');

    c.memoryPut('node_demo', 'users', 'alice',
                 '{"role":"pm","team":"infra"}');
    const value = c.memoryGet('node_demo', 'users', 'alice');
    console.log('alice ->', value);

    c.exec('DROP MEMORY STORE node_demo');
} finally {
    c.close();
}

Use query for any read. It returns an array of rows. Each row is an array of column strings.

const rows = c.query('SELECT key, value FROM ...');
for (const [key, value] of rows) {
    console.log(key, value);
}

To add an embedding, pass a numeric array as the optional fifth argument to memoryPut. The array is turned into a Float32 vector literal by formatVector before it is sent.

c.memoryPut('demo', 'docs', 'doc-1',
             '{"title":"intro"}', [0.12, 0.98, -0.4]);

Reference

The module exports four names. They are Connection, EvoError, EvoNotFoundError, and formatVector.

Connection

new Connection(host, port, user, password)

Opens a connection with evo_connect. It throws the last C error if the handle is null. A Connection is not safe to share across worker threads. Open one connection per worker.

close()

Calls evo_close and sets the handle to null. It is safe to call more than once. You can call it on a connection that is already closed.

exec(sql) → int

Runs a statement with evo_exec. It returns the non-negative return code. For example, this can be the number of affected rows as reported by the C layer. It throws EvoError if the code is negative.

query(sql, opts = {}) → string[][]

Runs a query with evo_query. It decodes the flat result buffer into rows of column strings. The opts object sets the buffer sizes. maxRows has a default of 256. maxCols has a default of 32. colBufSize sets the bytes per cell and has a default of 4096. It throws EvoError on a negative return code.

memoryPut(store, namespace, key, valueJson, embedding = null)

Inserts or updates a memory row with evo_memory_put. valueJson is a JSON string. If embedding is a non-empty array, it is turned into a vector literal. If it is empty, the embedding is left out. It throws EvoError if the return code is not zero.

memoryGet(store, namespace, key, bufSize = 32 * 1024)

Reads a memory row with evo_memory_get. It reads into a buffer of bufSize bytes. The default is 32 KiB. It returns the NUL-terminated value as a UTF-8 string. It throws EvoNotFoundError when the row is missing. The missing-row return code is -7. It throws EvoError for any other non-zero code.

memoryDelete(store, namespace, key)

Deletes a memory row with evo_memory_delete. It throws EvoError on a non-zero return code.

checkpointPut(store, threadId, ns, checkpointId, valuesJson, metadataJson = '{}')

Writes a LangGraph checkpoint with evo_checkpoint_put. valuesJson and metadataJson are JSON strings. metadataJson defaults to '{}'. It throws EvoError on a non-zero return code.

checkpointGetLatest(store, threadId, bufSize = 64 * 1024)

Gets the latest checkpoint for a thread with evo_checkpoint_get_latest. It reads into a buffer of bufSize bytes. The default is 64 KiB. It returns the UTF-8 string. It throws EvoNotFoundError when no checkpoint exists. The missing-checkpoint return code is -7. It throws EvoError otherwise.

formatVector(vec) → string

formatVector(vec)

This helper turns a numeric JavaScript array into a vector literal. It returns '' for an empty input or for input that is not an array. It packs the values as Float32 and calls evo_vector_format. If that returns a positive length, the formatted literal is used. If not, it falls back to a JS-side [v0,v1,...] string. Each value has 6 decimal places.

EvoError / EvoNotFoundError

class EvoError extends Error

EvoError is the base error for all failures. It carries code, which is a number with a default of -1. It also carries sqlstate, a 5-char Postgres-style state. The message and SQLSTATE come from the C SDK's thread-local last-error storage. They are read with evo_strerror and evo_sqlstate.

class EvoNotFoundError extends EvoError

EvoNotFoundError means the row is not present. The code is -7. It is thrown by memoryGet and checkpointGetLatest.

Notes & limitations

  • FFI dependency. The package is an ffi-napi binding. The native libevosql-memory shared library must be present and found. See Configure for more. There is no pure-JS fallback.
  • Node 18+. This is set with engines.node in package.json.
  • Not thread-safe. Do not share a Connection across worker threads. Open one connection per worker. The C SDK's last-error storage is thread-local. An error in one worker does not leak into another worker's reading.
  • String results only. query returns columns as decoded UTF-8 strings. There is no automatic type coercion. colBufSize sets the limit for cells that are too large.
  • Fixed buffers. memoryGet and checkpointGetLatest read into a buffer that you size. The defaults are 32 KiB and 64 KiB. Raise bufSize for larger values.
  • No TypeScript typings shipped. The package is written in plain JavaScript. It uses CommonJS with main: lib/index.js. You can use it from TypeScript, but no .d.ts file is bundled.
