Node.js SDK

A Node.js / TypeScript-friendly binding (evosql-memory) over the native libevosql-memory C library, exposing a Connection class for raw SQL plus agent-memory and LangGraph-checkpoint helpers.

What it does

The node-evosql-memory package is a thin ffi-napi binding over the C SDK libevosql-memory. It loads the shared library at require time, maps the C surface with ffi-napi / ref-napi, and gives you a single Connection class. On top of plain exec/query it provides convenience methods for the agent-memory store (memoryPut / memoryGet / memoryDelete) and LangGraph-style checkpoints (checkpointPut / checkpointGetLatest). It connects to a running EvolutionDB server over the wire: it is a client, not an embedded engine.

Because it is FFI over a native .dylib / .so, the shared library must be present on the host (see Configure). Supports Node 18+.

Related: the MCP server exposes the same memory store to AI agents, and the broader concepts are documented on the project wiki.

Install

The package name is evosql-memory (in the monorepo at client/node-evosql-memory). It depends on ffi-napi (^4.0.3) and ref-napi (^3.0.3). You also need the native shared library, which you can either build from source or download as a pre-built binary.

# Build the C SDK first, then install the Node deps
make -C ../libevosql-memory                 # build libevosql-memory
cd client/node-evosql-memory
npm install                                  # ffi-napi + ref-napi
npm run example                              # runs examples/put_get.js

Alternatively, pre-built binaries for Linux / macOS / Windows are attached to every GitHub Release. Download the one for your platform, unpack it, and point EVOSQL_MEMORY_LIB at the shared library to skip the make step:

# https://github.com/alptekin/evolutiondb/releases/latest
export EVOSQL_MEMORY_LIB=/path/to/libevosql-memory.dylib

Configure

There is no config file. Connection parameters are passed positionally to the Connection constructor (host, port, user, password), the examples use 127.0.0.1, port 9967, and admin / admin.

The only environment variable the SDK reads is the shared-library location. At load time the binding resolves libevosql-memory.{dylib,so} in this exact order:

  1. process.env.EVOSQL_MEMORY_LIB, an explicit path (used only if the file exists).
  2. ../../libevosql-memory/build/libevosql-memory.{dylib,so}: the in-tree SDK build directory, relative to lib/index.js.
  3. The system loader path (e.g. /usr/local/lib, /usr/lib), ffi.Library is given the bare name evosql-memory.

The extension is chosen from process.platform: .dylib on macOS (darwin), otherwise .so.

Authentication is whatever the EvolutionDB server enforces, the SDK simply forwards the user / password you pass to evo_connect. There is no credential storage or token handling in the binding itself.

Usage

Open a connection, run statements, and always close it in a finally block:

const { Connection } = require('evosql-memory');

const c = new Connection('127.0.0.1', 9967, 'admin', 'admin');
try {
    c.exec('CREATE MEMORY STORE demo');
    c.memoryPut('demo', 'users', 'alice', '{"role":"pm"}');
    console.log(c.memoryGet('demo', 'users', 'alice'));
} finally {
    c.close();
}

The smoke example shipped in examples/put_get.js shows the full create / put / get / drop cycle:

const { Connection } = require('..');

const c = new Connection('127.0.0.1', 9967, 'admin', 'admin');
try {
    c.exec('DROP MEMORY STORE IF EXISTS node_demo');
    c.exec('CREATE MEMORY STORE node_demo');

    c.memoryPut('node_demo', 'users', 'alice',
                 '{"role":"pm","team":"infra"}');
    const value = c.memoryGet('node_demo', 'users', 'alice');
    console.log('alice ->', value);

    c.exec('DROP MEMORY STORE node_demo');
} finally {
    c.close();
}

For arbitrary reads, query returns an array of rows, each row being an array of column strings:

const rows = c.query('SELECT key, value FROM ...');
for (const [key, value] of rows) {
    console.log(key, value);
}

To attach an embedding, pass a numeric array as the optional fifth argument to memoryPut; it is converted to a Float32 vector literal via formatVector before being sent:

c.memoryPut('demo', 'docs', 'doc-1',
             '{"title":"intro"}', [0.12, 0.98, -0.4]);

Reference

The module exports four names: Connection, EvoError, EvoNotFoundError, and formatVector.

Connection

new Connection(host, port, user, password)

Opens a connection via evo_connect. Throws the last C error if the handle is null. A Connection is not safe to share across worker threads, open one per worker.

close()

Calls evo_close and nulls the handle. Idempotent, safe to call on an already-closed connection.

exec(sql) → int

Runs a statement via evo_exec. Returns the non-negative return code (e.g. affected rows as reported by the C layer); throws EvoError if the code is negative.

query(sql, opts = {}) → string[][]

Runs a query via evo_query and decodes the flat result buffer into rows of column strings. opts tunes the buffer sizing: maxRows (default 256), maxCols (default 32), and colBufSize bytes per cell (default 4096). Throws EvoError on a negative return code.

memoryPut(store, namespace, key, valueJson, embedding = null)

Upserts a memory row via evo_memory_put. valueJson is a JSON string. If embedding is a non-empty array it is formatted to a vector literal; otherwise the embedding is omitted. Throws EvoError if the return code is non-zero.

memoryGet(store, namespace, key, bufSize = 32 * 1024)

Reads a memory row via evo_memory_get into a buffer of bufSize bytes (default 32 KiB) and returns the NUL-terminated value as a UTF-8 string. Throws EvoNotFoundError when the row is absent (return code -7), or EvoError for any other non-zero code.

memoryDelete(store, namespace, key)

Deletes a memory row via evo_memory_delete. Throws EvoError on a non-zero return code.

checkpointPut(store, threadId, ns, checkpointId, valuesJson, metadataJson = '{}')

Writes a LangGraph-style checkpoint via evo_checkpoint_put. valuesJson and metadataJson are JSON strings; metadataJson defaults to '{}'. Throws EvoError on a non-zero return code.

checkpointGetLatest(store, threadId, bufSize = 64 * 1024)

Fetches the latest checkpoint for a thread via evo_checkpoint_get_latest into a buffer of bufSize bytes (default 64 KiB) and returns the UTF-8 string. Throws EvoNotFoundError when no checkpoint exists (return code -7), or EvoError otherwise.

formatVector(vec) → string

formatVector(vec)

Helper that turns a numeric JavaScript array into a vector literal. Returns '' for an empty / non-array input. It packs the values as Float32 and calls evo_vector_format; if that returns a positive length the formatted literal is used, otherwise it falls back to a JS-side [v0,v1,...] string with each value at 6 decimal places.

EvoError / EvoNotFoundError

class EvoError extends Error

Base error for all failures. Carries code (numeric, default -1) and sqlstate (5-char Postgres-style state). The message and SQLSTATE are read from the C SDK's thread-local last-error storage via evo_strerror / evo_sqlstate.

class EvoNotFoundError extends EvoError

Subclass signalling "row not present" (code -7), thrown by memoryGet and checkpointGetLatest.

Notes & limitations

  • FFI dependency. The package is an ffi-napi binding, so the native libevosql-memory shared library must be present and resolvable (see Configure). There is no pure-JS fallback.
  • Node 18+. Declared via engines.node in package.json.
  • Not thread-safe. Do not share a Connection across worker threads, open one per worker. The C SDK's last-error storage is thread-local, so an error in one worker does not leak into another's reading.
  • String results only. query returns columns as decoded UTF-8 strings; there is no automatic type coercion. Oversized cells are bounded by colBufSize.
  • Fixed buffers. memoryGet / checkpointGetLatest read into a caller-sized buffer (defaults 32 KiB / 64 KiB); raise bufSize for larger values.
  • No TypeScript typings shipped. The package is written in plain JavaScript (CommonJS, main: lib/index.js); it is usable from TypeScript but no .d.ts is bundled.
