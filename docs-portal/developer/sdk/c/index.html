C library (libevosql-memory)

A thin C client over the EVO text protocol with high-level wrappers for the agent-memory primitives that ship with EvolutionDB. It is the FFI seam every higher-level binding (Python ctypes, Go cgo, Rust bindgen, Node node-ffi-napi) links against.

info
This is libevosql-memory 1.0.0, shipping with EvolutionDB 2.x. The header and ABI are stable, new API is added by appending functions to the bottom of evosql_memory.h, so existing call sites never need to recompile.

What it does

The library wraps the EVO text protocol (TCP port 9967) and exposes three layers, all declared in a single header, include/evosql_memory.h:

  • Typed agent-memory wrappers, MEMORY STORE (namespaced KV with optional vector embedding) and CHECKPOINT STORE (LangGraph-compatible checkpoint records).
  • A generic escape hatch, evo_exec / evo_query for any SQL the typed wrappers don't cover.
  • Streaming surfaces, evo_subscribe (LISTEN/NOTIFY) and evo_cdc_subscribe (the CDC JSON-lines stream), each running on its own socket and worker thread.

The build produces a shared library (.dylib/.so/.dll), a static archive (libevosql-memory.a), and a pkg-config file. The source is two C files: src/conn.c (connection, query, typed wrappers, errors) and src/subscribe.c (the streaming workers).

Install

The fastest path is to download a pre-built artifact for your platform from the project's GitHub Releases page. Five platforms ship per release:

  • libevosql-memory-linux-x64-vX.Y.Z.tar.gz, Linux glibc x86_64
  • libevosql-memory-linux-arm64-vX.Y.Z.tar.gz, Linux glibc aarch64
  • libevosql-memory-macos-arm64-vX.Y.Z.tar.gz, macOS 14+ Apple Silicon
  • libevosql-memory-macos-x64-vX.Y.Z.tar.gz, macOS 13+ Intel
  • libevosql-memory-windows-x64-vX.Y.Z.zip, Windows 10/11 x86_64

Each archive untars to a directory holding the shared library, the static archive, and evosql_memory.h. Verify the asset's SHA-256 against the SHA256SUMS file attached to the same release.

# download + unpack a pre-built artifact
curl -L https://github.com/alptekin/evolutiondb/releases/latest/download/libevosql-memory-linux-x64-v3.0.1.tar.gz \
    | tar -xz
cp libevosql-memory-linux-x64/libevosql-memory.so /usr/local/lib/
cp libevosql-memory-linux-x64/evosql_memory.h     /usr/local/include/

Build from source

From client/libevosql-memory, the Makefile detects the host (Darwin / Linux / MinGW / MSYS) and picks the right shared-library extension and link flags. Windows builds use MinGW-w64 and link ws2_32 + pthread.

cd client/libevosql-memory
make                       # build/ shared lib + .a + pkg-config
make test                  # runs against 127.0.0.1:9967 (server must be up)
make install PREFIX=/usr/local

# override toolchain / prefix the usual way
make CC=clang
make PREFIX=/opt/evosql install

Artifacts land in build/: libevosql-memory.{dylib,so,dll}, the static archive libevosql-memory.a, the evosql-memory.pc pkg-config file, and the test binaries test_sdk and test_subscribe.

Compiling your program against it

# link the static archive (no install needed)
cc -Iclient/libevosql-memory/include demo.c \
   client/libevosql-memory/build/libevosql-memory.a -o demo

# …or the shared library after `make install`
cc demo.c -levosql-memory -o demo

The streaming functions need -lpthread on POSIX; on Windows link ws2_32 and pthread as the Makefile does.

Configure (connection & auth)

There are no environment variables or config files, connection details are passed as function arguments. evo_connect() takes host, port, user, and password directly. The default credentials for a stock server are admin/admin, and the default EVO port is 9967.

Authentication uses the EVO protocol's plaintext path. The client sends the greeting, declines STARTTLS, and authenticates with the legacy cleartext AUTH <user> <pass> line, which the server accepts as a fallback:

client → "EVO\n"
server → "HELLO EvoSQL 1.0\n"
server → "STARTTLS\n"           (optional, server-driven)
client → "NOTLS\n"              (the SDK declines TLS in v1)
server → "AUTH_SCRAM\n"
client → "AUTH <user> <pass>\n"
server → "AUTH_OK\n"
warning
v1 is plaintext only: no TLS client, no SCRAM, no extended-query / batch. Credentials cross the wire in cleartext, keep the connection on a trusted network or behind a tunnel.

Usage

Put / get memory

From examples/put_get.c: open a connection, create a store, write JSON values, read them back:

#include <stdio.h>
#include <evosql_memory.h>

int main(void)
{
    evo_conn_t *c = evo_connect("127.0.0.1", 9967, "admin", "admin");
    if (!c) {
        fprintf(stderr, "connect failed: %s\n", evo_strerror(NULL));
        return 1;
    }

    evo_exec(c, "DROP MEMORY STORE IF EXISTS demo");
    evo_exec(c, "CREATE MEMORY STORE demo");

    evo_memory_put(c, "demo", "users", "alice",
                   "{\"role\":\"pm\"}", NULL);

    char value[1024];
    if (evo_memory_get(c, "demo", "users", "alice",
                       value, sizeof(value)) == EVO_OK) {
        printf("alice -> %s\n", value);
    }

    evo_close(c);
    return 0;
}

Reactive subscription

From examples/reactive_agent.c, evo_subscribe() opens its own connection, runs LISTEN <channel>, and dispatches each NOTIFY to your callback on a worker thread:

static void on_event(const char *channel, const char *payload,
                     uint64_t lsn, void *ctx)
{
    (void)lsn; (void)ctx;
    printf("[event] %s -> %s\n", channel, payload);
}

evo_subscription_t *sub = evo_subscribe(
    "127.0.0.1", 9967, "admin", "admin",
    "agent_events", on_event, NULL);
if (!sub) {
    fprintf(stderr, "subscribe failed: %s\n", evo_strerror(NULL));
    return 1;
}

/* …main loop runs as long as the agent is alive… */

evo_unsubscribe(sub);   /* deterministic: stops the worker, joins, frees */

To produce events, another session runs NOTIFY agent_events, 'hello world';, for example via evosql-cli.

Formatting a vector

evo_vector_format() turns a float[] into the [v1,v2,...] literal EvoSQL accepts as a VECTOR(N) value, which you then pass as the optional embedding argument to evo_memory_put():

float emb[] = { 0.1f, 0.2f, 0.3f };
char lit[256];
evo_vector_format(emb, 3, lit, sizeof(lit));   /* lit = "[0.100000,0.200000,0.300000]" */
evo_memory_put(c, "demo", "users", "alice", "{\"role\":\"pm\"}", lit);

Reference

The canonical signatures and documentation live in include/evosql_memory.h. Every evo_* call returns an evo_error_t code (or a row count / NULL where noted). The error codes are: EVO_OK (0), EVO_ERR (-1), EVO_ERR_CONNECT (-2), EVO_ERR_AUTH (-3), EVO_ERR_PROTOCOL (-4), EVO_ERR_SQL (-5), EVO_ERR_BUFFER (-6), EVO_ERR_NOT_FOUND (-7).

Lifecycle

evo_conn_t *evo_connect(const char *host, int port, const char *user, const char *password)

Opens a connection and runs the EVO handshake + auth. Returns NULL on failure, call evo_strerror(NULL) for the reason. The returned handle is opaque and must be freed with evo_close().

void evo_close(evo_conn_t *conn)

Sends a best-effort QUIT, closes the socket, and frees the handle. NULL-tolerant.

Generic SQL

int evo_exec(evo_conn_t *conn, const char *sql)

Runs a non-SELECT statement. Returns the number of affected rows (parsed from the server's TAG line, 0 if none) or -1 on error.

int evo_query(evo_conn_t *conn, const char *sql, char *out, int max_rows, int max_cols, int col_buf_size, int *cols_out)

SELECT helper. The caller supplies one flat char buffer of max_rows × max_cols × col_buf_size bytes; the field at row r, column c lives at out + (r * max_cols + c) * col_buf_size. NULL fields are stored as the empty string. Returns the row count, or 0 for a non-SELECT, or -1 on error. cols_out (optional) receives the column count.

Memory store

int evo_memory_put(evo_conn_t *conn, const char *store, const char *ns, const char *key, const char *value_json, const char *embedding_or_null)

Emits MEMORY PUT INTO <store> VALUES (ns, key, json [, embedding]). embedding_or_null may be NULL; otherwise pass a [0.1,0.2,...] literal (see evo_vector_format). Single quotes in ns/key/value_json are escaped by the library.

int evo_memory_get(evo_conn_t *conn, const char *store, const char *ns, const char *key, char *out_value, int out_size)

Runs MEMORY GET FROM <store> WHERE NS = ? AND KEY = ?, writes the value JSON (NUL-terminated) into out_value. Returns EVO_OK, or EVO_ERR_NOT_FOUND when no row matches.

int evo_memory_delete(evo_conn_t *conn, const char *store, const char *ns, const char *key)

Runs MEMORY DELETE FROM <store> WHERE NS = ? AND KEY = ?. Returns EVO_OK on success.

Checkpoint store

int evo_checkpoint_put(evo_conn_t *conn, const char *store, const char *thread_id, const char *checkpoint_ns, const char *checkpoint_id, const char *values_json, const char *metadata_json)

Emits CHECKPOINT PUT INTO <store> VALUES (...). The backing table has 7 fields; the SDK exposes the 5 the agent layer cares about and supplies empty defaults for parent linkage. checkpoint_ns and metadata_json may be NULL (defaulting to empty / {}).

int evo_checkpoint_get_latest(evo_conn_t *conn, const char *store, const char *thread_id, char *out_value, int out_size)

Runs CHECKPOINT GET FROM <store> THREAD ? and writes the latest checkpoint's values JSON into out_value. Returns EVO_OK or EVO_ERR_NOT_FOUND.

Vectors

int evo_vector_format(const float *vec, int n, char *out, int out_size)

Formats n floats as a [v1,v2,...] literal (six decimal places). Returns the number of bytes written (excluding NUL) or -1 on overflow.

Streaming

evo_subscription_t *evo_subscribe(const char *host, int port, const char *user, const char *password, const char *channel, evo_event_callback_t callback, void *ctx)

Opens its own authenticated connection, runs LISTEN <channel>, and dispatches every NOTIFY to callback. The LISTEN response is drained synchronously so startup errors surface before the worker starts. Returns NULL on connect / listen failure. For NOTIFY events, lsn is always 0.

evo_subscription_t *evo_cdc_subscribe(const char *host, int cdc_port, evo_event_callback_t callback, void *ctx, uint64_t from_lsn)

Connects to the CDC JSON-lines stream (default port 9970, a plain TCP port with no EVO handshake or auth) and emits one callback per WAL change. The callback's channel is the table name extracted from the JSON, payload is the full JSON line, and lsn carries the xid when present. from_lsn is reserved for v2, pass 0.

int evo_ack_up_to(evo_subscription_t *sub, uint64_t lsn)

Records the highest acked LSN. In v1 this is stored locally only; the server-side ack channel is wired in v2 once durable CDC subscription queues land. NOTIFY subscriptions ignore the call. Always returns EVO_OK.

void evo_unsubscribe(evo_subscription_t *sub)

Deterministic teardown: flags the worker, calls shutdown() so the blocked recv() returns, joins the thread, then frees. The callback will not fire after this returns. NULL-tolerant.

typedef void (*evo_event_callback_t)(const char *channel, const char *payload, uint64_t lsn, void *ctx)

Your callback runs on the subscription's worker thread. Treat the arguments as borrowed pointers: copy what you need, then return.

Error inspection

const char *evo_strerror(const evo_conn_t *conn)

Returns the last error message on the calling thread (thread-local). Never NULL; returns "(no error)" when clean. The conn argument is accepted for completeness but ignored, pass NULL after a failed evo_connect().

const char *evo_sqlstate(const evo_conn_t *conn)

Returns the SQLSTATE of the last error, or the empty string if there was none.

Notes & limitations

  • Threading. A single evo_conn_t * is not thread-safe, open one connection per thread. The last-error string is in thread-local storage, so errors don't leak across threads. Subscriptions own their own socket and worker; a single subscription never re-enters its own callback (it holds an internal mutex), but two distinct subscriptions can run their callbacks in parallel.
  • Plaintext only (v1). No TLS client, no SCRAM, no extended-query / batch protocol. The SDK declines STARTTLS and uses the legacy cleartext AUTH fallback.
  • CDC ack is local-only in v1. evo_ack_up_to() records the LSN client-side; durable server-side acknowledgement is a v2 item.
  • Fixed-size buffers. The typed wrappers stage SQL and results in fixed stack buffers (e.g. value JSON up to ~16 KB for puts). Oversized arguments return EVO_ERR_BUFFER. For arbitrarily large or unusual statements, use evo_query / evo_exec directly.
  • SQL escaping. The wrappers double single quotes in ns, key, and JSON values, but assume store/identifier names are already sane. Don't feed untrusted store names into the typed helpers.
