Python SDK

evosql_memory, a Pythonic ctypes wrapper over the libevosql-memory shared library, plus thin duck-typed adapters for six agent frameworks (LangGraph, LangChain, LlamaIndex, CrewAI, AutoGen, Mem0).

What it does

The package gives Python apps and agents direct access to EvolutionDB's memory primitives, MEMORY STORE, CHECKPOINT STORE, MESSAGE LOG, ENTITY STORE, DOCUMENT STORE, over the native EVO wire protocol (default port 9967). It is a binding, not a reimplementation: the C SDK (see the platform packages and the C SDK in the monorepo) already owns connection state, error handling, and the wire protocol; evosql_memory loads that shared library and exposes:

  • Connection, generic SQL (exec_, query) plus typed helpers for the memory and checkpoint stores.
  • Subscription, a background LISTEN/event stream with a Python callback bridged to the C callback, running on a C-owned worker thread.
  • cdc_subscribe, a CDC stream subscriber (same Subscription wrapper, different server-side endpoint).
  • format_vector, turns a Python float sequence into the EVO vector literal.
  • Six adapter modules under evosql_memory.adapters.* that map each framework's memory interface onto these primitives.

The package version (evosql_memory.__version__) is 1.0.0.

Install

There is no packaging file (no pyproject.toml, setup.py, or package.json) and no PyPI distribution in the repo today, the package is consumed straight from the source tree. The only hard dependency is the C SDK shared library, libevosql-memory.{dylib,so}.

Build the shared library once, then put the package directory on PYTHONPATH:

# 1. build the C SDK shared library
cd client/libevosql-memory
make                         # produces build/libevosql-memory.{dylib,so}

# 2. add the Python package to PYTHONPATH (no pip install needed)
export PYTHONPATH=$PWD/client/python-evosql-memory:$PYTHONPATH
python3 client/python-evosql-memory/examples/react_agent_demo.py

Pre-built shared libraries for Linux x64/arm64, macOS arm64/x64, and Windows x64 are attached to GitHub Releases (github.com/alptekin/evolutiondb/releases/latest); point EVOSQL_MEMORY_LIB at a downloaded library to skip the make step.

Configure (library location & auth)

At import time, evosql_memory.core searches for the shared library in this exact order and loads the first one that resolves:

  1. $EVOSQL_MEMORY_LIB, an explicit path to the library (the override).
  2. ../../libevosql-memory/build/libevosql-memory.{dylib,so}: relative to the package directory, so a source checkout works with no env setup.
  3. libevosql-memory.{dylib,so}, resolved through the system loader path (DYLD_LIBRARY_PATH / LD_LIBRARY_PATH, or after a system install).

The extension is dylib on macOS (platform.system() == "Darwin") and so elsewhere. If none load, import raises OSError telling you to set EVOSQL_MEMORY_LIB or build the SDK.

Auth. There is no config file or auth env var, credentials are passed positionally to connect() / Connection / Subscription as (host, port, user, password). The examples and tests use 127.0.0.1, port 9967, user admin, password admin.

Usage

Quickstart: connect, create a store, put/get. Connection is a context manager:

from evosql_memory import connect

with connect("127.0.0.1", 9967, "admin", "admin") as conn:
    conn.exec_("DROP MEMORY STORE IF EXISTS demo")
    conn.exec_("CREATE MEMORY STORE demo")
    conn.memory_put("demo", "users", "alice", '{"role":"pm"}')
    print(conn.memory_get("demo", "users", "alice"))

Generic SQL. query() returns rows as lists of strings (the wire format is text; NULL comes back as ""):

rows = conn.query("SHOW MEMORY STORES")   # -> List[List[str]]
n = conn.exec_("NOTIFY my_chan, 'hello'")   # -> affected-row count (or 0)

Streaming subscription. The callback signature is (channel, payload, lsn) and runs on a dedicated C-owned worker thread:

from evosql_memory import Subscription

def on_event(channel, payload, lsn):
    print(f"{channel}: {payload}")

with Subscription("127.0.0.1", 9967, "admin", "admin",
                  "agent_events", on_event) as sub:
    ...   # run your loop; events flow into on_event

Vector embeddings. Pass an optional float sequence to memory_put; format_vector renders the literal the engine expects:

conn.memory_put("demo", "docs", "d1", '{"text":"hi"}',
                embedding=[0.1, 0.2, 0.3])

Framework adapters. Each adapter is a thin (50–200 line) shim that mirrors a framework's interface. The package never imports the framework itself: duck-typed objects slot in because frameworks check for methods, not classes. Import only the one your stack needs:

# LangGraph in a few lines
from evosql_memory import connect
from evosql_memory.adapters.langgraph_evosql import EvoCheckpointSaver, EvoBaseStore

conn = connect("127.0.0.1", 9967, "admin", "admin")
checkpointer = EvoCheckpointSaver(conn, "graph_ck")
store        = EvoBaseStore(conn, "graph_mem", embedding_dim=384)
graph = workflow.compile(checkpointer=checkpointer, store=store)
# Mem0 drop-in: keep "from mem0 import Memory" code working
from evosql_memory.adapters.mem0_compat import Memory

with Memory(host="127.0.0.1", port=9967,
            user="admin", password="admin",
            store_name="my_app") as m:
    m.add({"role": "user", "content": "I love jazz"}, user_id="u1")
    print(m.search("jazz", user_id="u1"))

Reference

Top-level exports from evosql_memory: connect, Connection, Subscription, cdc_subscribe, format_vector, and the error classes EvoError, EvoNotFound, EvoSqlError.

Top-level functions

connect(host: str, port: int, user: str, password: str) -> Connection

Convenience factory (mirrors sqlite3.connect / psycopg2.connect) that returns a new Connection.

format_vector(vec: Iterable[float]) -> str

Renders a float sequence into the EVO vector literal (e.g. [0.100000,0.200000]). Calls the C formatter; falls back to a pure-Python formatter only if the C side rejects the (already over-sized) buffer.

cdc_subscribe(host, port, callback, from_lsn: int = 0) -> Subscription

CDC stream subscriber. Same Subscription wrapper as Subscription(...) but targets the server's CDC endpoint, starting from from_lsn. The callback signature is (channel, payload, lsn).

class Connection

Wraps a single libevosql-memory connection. Not thread-safe, use one Connection per thread. Supports the context-manager protocol; __del__ closes the handle if you forget.

Connection(host: str, port: int, user: str, password: str)

Opens the connection; raises EvoError / EvoSqlError if evo_connect fails.

close() -> None

Closes the underlying handle (idempotent).

exec_(sql: str) -> int

Runs DML/DDL. Returns the affected-row count (or 0). Raises EvoSqlError on a server-side failure. (Note the trailing underscore, exec avoids the Python builtin.)

query(sql: str, max_rows: int = 256, max_cols: int = 32, col_buf_size: int = 4096) -> List[List[str]]

Runs a SELECT and returns rows; every cell is a string, NULL is "". The result buffer is allocated Python-side (max_rows * max_cols * col_buf_size bytes), so bump the sizes for larger results, over-sizing only costs RAM.

memory_put(store, ns, key, value_json, embedding: Optional[Sequence[float]] = None) -> None

Upserts a memory row. If embedding is supplied it is rendered with format_vector and stored alongside.

memory_get(store, ns, key, buf_size: int = 32768) -> str

Returns the stored JSON string. Raises EvoNotFound when no row matches (the SDK maps the EVO_ERR_NOT_FOUND code, -7).

memory_delete(store, ns, key) -> None

Deletes a memory row.

checkpoint_put(store, thread_id, ns, checkpoint_id, values_json, metadata_json: str = "{}") -> None

Persists a checkpoint into a CHECKPOINT STORE.

checkpoint_get_latest(store, thread_id, buf_size: int = 65536) -> str

Returns the latest checkpoint JSON for a thread; raises EvoNotFound if none.

class Subscription

Background subscription. The C SDK owns the worker thread; the Python callback is invoked under a private lock that keeps the trampoline closure alive. Supports the context-manager protocol.

Subscription(host, port, user, password, channel: str, callback: Callable[[str, str, int], None])

Subscribes to channel; the callback receives (channel, payload, lsn).

ack(lsn: int) -> None

Acknowledges delivery up to lsn (calls evo_ack_up_to).

close() -> None

Unsubscribes and releases the worker thread (idempotent, thread-safe).

Errors

EvoError(message, code: int = -1, sqlstate: str = "")

Generic SDK error; code is the EVO_ERR_* enum, sqlstate is the 5-char Postgres-style state when known.

EvoNotFound(EvoError)

Raised by memory_get / checkpoint_get_latest when no row matches.

EvoSqlError(EvoError)

Raised when the server returns ERR <sqlstate> ...; sqlstate is populated.

Framework adapters

Each module lives under evosql_memory.adapters and is independent, import the one you need. Constructors take a live Connection (and usually a store name); most accept auto_create=True, which DROPs and re-creates the backing store on construction.

Examples & tests

examples/react_agent_demo.py runs a minimal LangGraph-style ReAct loop entirely on EvolutionDB: checkpoint per step, chat history per session, long-term memory keyed by question, and a real-time NOTIFY channel broadcasting each step. It does not require langgraph to be installed; start the server first (docker compose up -d), then python3 client/python-evosql-memory/examples/react_agent_demo.py.

python_tests/test_adapters.py is a self-contained runner (plain assert + a run() harness, no pytest) with eleven cases: connection, streaming, and one happy path per adapter. Each test drops and recreates its store on entry so the suite is re-runnable. It needs a live evosql-server on 127.0.0.1:9967.

python3 client/python-evosql-memory/python_tests/test_adapters.py

Notes & limitations

  • Source-tree only. No pip install, no pyproject.toml / setup.py, no PyPI package, use PYTHONPATH. (The package landing page's pip install python-evosql-memory / MemoryClient snippet describes an aspirational API; the real entry points are connect() / Connection, with no MemoryClient class in the code today.)
  • Shared library required. Import fails with OSError if libevosql-memory.{dylib,so} cannot be located. Set EVOSQL_MEMORY_LIB or build the C SDK.
  • Threading. One Connection per thread (the C SDK contract). A single Subscription never re-enters its own callback (internal per-subscription mutex), but two subscriptions can fire callbacks in parallel, make callbacks safe for that.
  • Results are text. query() returns strings; callers parse JSON / numbers themselves. Buffers are pre-allocated, so very large rows need larger col_buf_size / buf_size.
  • SQL is string-built. Adapters interpolate identifiers and values into SQL and escape single quotes (' → ''); store names and other identifiers are not parameterized, so treat them as trusted.
  • Adapters are duck-typed shims against framework contracts, not the frameworks themselves, verify against your installed framework version. Several reference engine features tied to specific server tasks (MESSAGE LOG, DOCUMENT FILTER, ENTITY STORE); they require server support for those statements.

See also the MCP server for an agent-facing wrapper over the same primitives, and the EvolutionDB wiki for the EVO protocol and store reference.
