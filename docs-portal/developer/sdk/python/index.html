Python SDK

evosql_memory is a Pythonic ctypes wrapper over the libevosql-memory shared library. It also ships thin duck-typed adapters for six agent frameworks. These are LangGraph, LangChain, LlamaIndex, CrewAI, AutoGen, and Mem0.

What it does

The package gives Python apps and agents direct access to EvolutionDB's memory primitives. These are MEMORY STORE, CHECKPOINT STORE, MESSAGE LOG, ENTITY STORE, and DOCUMENT STORE. Access runs over the native EVO wire protocol. The default port is 9967. The package is a binding, not a rewrite. The C SDK already owns connection state, error handling, and the wire protocol. You can see the platform packages and the C SDK in the monorepo. evosql_memory loads that shared library and exposes the following:

  • Connection runs generic SQL with exec_ and query. It also has typed helpers for the memory and checkpoint stores.
  • Subscription is a background LISTEN and event stream. A Python callback is bridged to the C callback. It runs on a C-owned worker thread.
  • cdc_subscribe is a CDC stream subscriber. It uses the same Subscription wrapper but a different server-side endpoint.
  • format_vector turns a Python float sequence into the EVO vector literal.
  • Six adapter modules live under evosql_memory.adapters.*. They map each framework's memory interface onto these primitives.

The package version evosql_memory.__version__ is 1.0.0.

Install

There is no packaging file in the repo today. There is no pyproject.toml, setup.py, or package.json. There is also no PyPI distribution. You use the package straight from the source tree. The only hard dependency is the C SDK shared library libevosql-memory.{dylib,so}.

Build the shared library once. Then put the package directory on PYTHONPATH.

# 1. build the C SDK shared library
cd client/libevosql-memory
make                         # produces build/libevosql-memory.{dylib,so}

# 2. add the Python package to PYTHONPATH (no pip install needed)
export PYTHONPATH=$PWD/client/python-evosql-memory:$PYTHONPATH
python3 client/python-evosql-memory/examples/react_agent_demo.py

Pre-built shared libraries are attached to GitHub Releases. They cover Linux x64/arm64, macOS arm64/x64, and Windows x64. You can find them at github.com/alptekin/evolutiondb/releases/latest. Point EVOSQL_MEMORY_LIB at a downloaded library to skip the make step.

Configure library location and auth

At import time, evosql_memory.core searches for the shared library in this exact order. It loads the first one that resolves.

  1. $EVOSQL_MEMORY_LIB is an explicit path to the library. This is the override.
  2. ../../libevosql-memory/build/libevosql-memory.{dylib,so} is relative to the package directory. A source checkout works with no env setup.
  3. libevosql-memory.{dylib,so} is resolved through the system loader path. This means DYLD_LIBRARY_PATH or LD_LIBRARY_PATH, or a system install.

The extension is dylib on macOS when platform.system() == "Darwin". It is so elsewhere. If none load, import raises OSError. The error tells you to set EVOSQL_MEMORY_LIB or build the SDK.

Auth. There is no config file or auth env var. You pass credentials positionally to connect(), Connection, or Subscription as (host, port, user, password). The examples and tests use 127.0.0.1, port 9967, user admin, and password admin.

Usage

Quickstart. Connect, create a store, put and get. Connection is a context manager.

from evosql_memory import connect

with connect("127.0.0.1", 9967, "admin", "admin") as conn:
    conn.exec_("DROP MEMORY STORE IF EXISTS demo")
    conn.exec_("CREATE MEMORY STORE demo")
    conn.memory_put("demo", "users", "alice", '{"role":"pm"}')
    print(conn.memory_get("demo", "users", "alice"))

Generic SQL. query() returns rows as lists of strings. The wire format is text. NULL comes back as "".

rows = conn.query("SHOW MEMORY STORES")   # -> List[List[str]]
n = conn.exec_("NOTIFY my_chan, 'hello'")   # -> affected-row count (or 0)

Streaming subscription. The callback signature is (channel, payload, lsn). It runs on a dedicated C-owned worker thread.

from evosql_memory import Subscription

def on_event(channel, payload, lsn):
    print(f"{channel}: {payload}")

with Subscription("127.0.0.1", 9967, "admin", "admin",
                  "agent_events", on_event) as sub:
    ...   # run your loop; events flow into on_event

Vector embeddings. Pass an optional float sequence to memory_put. format_vector renders the literal the engine expects.

conn.memory_put("demo", "docs", "d1", '{"text":"hi"}',
                embedding=[0.1, 0.2, 0.3])

Framework adapters. Each adapter is a thin shim of 50 to 200 lines. It mirrors a framework's interface. The package never imports the framework itself. Duck-typed objects slot in because frameworks check for methods, not classes. Import only the one your stack needs.

# LangGraph in a few lines
from evosql_memory import connect
from evosql_memory.adapters.langgraph_evosql import EvoCheckpointSaver, EvoBaseStore

conn = connect("127.0.0.1", 9967, "admin", "admin")
checkpointer = EvoCheckpointSaver(conn, "graph_ck")
store        = EvoBaseStore(conn, "graph_mem", embedding_dim=384)
graph = workflow.compile(checkpointer=checkpointer, store=store)
# Mem0 drop-in: keep "from mem0 import Memory" code working
from evosql_memory.adapters.mem0_compat import Memory

with Memory(host="127.0.0.1", port=9967,
            user="admin", password="admin",
            store_name="my_app") as m:
    m.add({"role": "user", "content": "I love jazz"}, user_id="u1")
    print(m.search("jazz", user_id="u1"))

Reference

evosql_memory has these top-level exports. They are connect, Connection, Subscription, cdc_subscribe, and format_vector. It also exports the error classes EvoError, EvoNotFound, and EvoSqlError.

Top-level functions

connect(host: str, port: int, user: str, password: str) -> Connection

A convenience factory that returns a new Connection. It mirrors sqlite3.connect and psycopg2.connect.

format_vector(vec: Iterable[float]) -> str

Renders a float sequence into the EVO vector literal. An example is [0.100000,0.200000]. It calls the C formatter. It falls back to a pure-Python formatter only if the C side rejects the buffer. That buffer is already over-sized.

cdc_subscribe(host, port, callback, from_lsn: int = 0) -> Subscription

A CDC stream subscriber. It uses the same Subscription wrapper as Subscription(...). It targets the server's CDC endpoint. It starts from from_lsn. The callback signature is (channel, payload, lsn).

class Connection

Wraps a single libevosql-memory connection. It is not thread-safe. Use one Connection per thread. It supports the context-manager protocol. __del__ closes the handle if you forget.

Connection(host: str, port: int, user: str, password: str)

Opens the connection. It raises EvoError or EvoSqlError if evo_connect fails.

close() -> None

Closes the underlying handle. It is idempotent.

exec_(sql: str) -> int

Runs DML and DDL. It returns the affected-row count, or 0. It raises EvoSqlError on a server-side failure. Note the trailing underscore. The name exec avoids the Python builtin.

query(sql: str, max_rows: int = 256, max_cols: int = 32, col_buf_size: int = 4096) -> List[List[str]]

Runs a SELECT and returns rows. Every cell is a string. NULL is "". The result buffer is allocated on the Python side. Its size is max_rows * max_cols * col_buf_size bytes. Bump the sizes for larger results. Over-sizing only costs RAM.

memory_put(store, ns, key, value_json, embedding: Optional[Sequence[float]] = None) -> None

Upserts a memory row. If you supply embedding, it is rendered with format_vector and stored alongside.

memory_get(store, ns, key, buf_size: int = 32768) -> str

Returns the stored JSON string. It raises EvoNotFound when no row matches. The SDK maps the EVO_ERR_NOT_FOUND code -7.

memory_delete(store, ns, key) -> None

Deletes a memory row.

checkpoint_put(store, thread_id, ns, checkpoint_id, values_json, metadata_json: str = "{}") -> None

Persists a checkpoint into a CHECKPOINT STORE.

checkpoint_get_latest(store, thread_id, buf_size: int = 65536) -> str

Returns the latest checkpoint JSON for a thread. It raises EvoNotFound if there is none.

class Subscription

A background subscription. The C SDK owns the worker thread. The Python callback is invoked under a private lock. That lock keeps the trampoline closure alive. It supports the context-manager protocol.

Subscription(host, port, user, password, channel: str, callback: Callable[[str, str, int], None])

Subscribes to channel. The callback receives (channel, payload, lsn).

ack(lsn: int) -> None

Acknowledges delivery up to lsn. It calls evo_ack_up_to.

close() -> None

Unsubscribes and releases the worker thread. It is idempotent and thread-safe.

Errors

EvoError(message, code: int = -1, sqlstate: str = "")

A generic SDK error. code is the EVO_ERR_* enum. sqlstate is the 5-char Postgres-style state when known.

EvoNotFound(EvoError)

Raised by memory_get and checkpoint_get_latest when no row matches.

EvoSqlError(EvoError)

Raised when the server returns ERR <sqlstate> .... sqlstate is populated.

Framework adapters

Each module lives under evosql_memory.adapters. Each one is independent. Import the one you need. Constructors take a live Connection. They usually take a store name too. Most accept auto_create=True. This option DROPs and re-creates the backing store on construction.

Examples & tests

examples/react_agent_demo.py runs a minimal LangGraph-style ReAct loop. It runs entirely on EvolutionDB. It uses a checkpoint per step, chat history per session, and long-term memory keyed by question. It also uses a real-time NOTIFY channel that broadcasts each step. It does not require langgraph to be installed. Start the server first with docker compose up -d. Then run python3 client/python-evosql-memory/examples/react_agent_demo.py.

python_tests/test_adapters.py is a self-contained runner. It uses plain assert and a run() harness. It does not use pytest. It has eleven cases. These cover connection, streaming, and one happy path per adapter. Each test drops and recreates its store on entry. This makes the suite re-runnable. It needs a live evosql-server on 127.0.0.1:9967.

python3 client/python-evosql-memory/python_tests/test_adapters.py

Notes & limitations

  • Source-tree only. There is no pip install, no pyproject.toml, no setup.py, and no PyPI package. Use PYTHONPATH. The package landing page shows a pip install python-evosql-memory and MemoryClient snippet. That describes a planned API. The real entry points are connect() and Connection. There is no MemoryClient class in the code today.
  • Shared library required. Import fails with OSError if libevosql-memory.{dylib,so} cannot be located. Set EVOSQL_MEMORY_LIB or build the C SDK.
  • Threading. Use one Connection per thread. This is the C SDK contract. A single Subscription never re-enters its own callback. It uses an internal per-subscription mutex. But two subscriptions can fire callbacks in parallel. Make your callbacks safe for that.
  • Results are text. query() returns strings. Callers parse JSON and numbers themselves. Buffers are pre-allocated. Very large rows need a larger col_buf_size or buf_size.
  • SQL is string-built. Adapters interpolate identifiers and values into SQL. They escape single quotes from ' to ''. Store names and other identifiers are not parameterized. Treat them as trusted.
  • Adapters are duck-typed shims against framework contracts. They are not the frameworks themselves. Verify against your installed framework version. Several reference engine features are tied to specific server tasks. These are MESSAGE LOG, DOCUMENT FILTER, and ENTITY STORE. They require server support for those statements.

See also the MCP server for an agent-facing wrapper over the same primitives. See the EvolutionDB wiki for the EVO protocol and store reference.
