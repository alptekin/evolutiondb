SDK · LANGUAGE FAMILY

SDKs — language roadmap

EvolutionDB ships an agent-memory client family under client/*-evosql-memory. One C library, libevosql-memory, owns the wire work; every other language is a thin binding over it through C FFI. This page is the honest status of the family — what is implemented and exercised, what is implemented but not yet test-covered in-tree, and what each binding still leans on. Where a binding has no test suite, it is labelled exactly that — never "works".

The shape of the family

Every binding talks to evosql-server through one common foundation. The heavy work — TCP, the EVO handshake, AUTH, RESULT/ROW parsing, NOTIFY framing, CDC NDJSON, vector formatting — lives in the C library. The language layers add idiomatic ergonomics (generics, exceptions, IDisposable, finalizers) without re-implementing the protocol.

            ┌──────────────────────────────────────────┐
            │            evosql-server                  │
            │   PG wire (5433)   │   EVO native (9967)   │
            └──────────────────────────────────────────┘
                          ▲
                          │ EVO text protocol
            ┌──────────────────────────────────────────┐
            │   libevosql-memory  (C — the FFI seam)    │
            └──────────────────────────────────────────┘
      Python  Node  Go  C++  .NET  Java  Ruby  Rust  Swift

The C contract is small and stable: roughly 16 public functions in include/evosql_memory.h — connect/close, exec/query, the memory and checkpoint verbs, vector formatting, the subscribe/CDC pair, and the two last-error accessors. Every binding mirrors that same surface.

warning
Read this page, not the directory listing. Each binding directory contains real source files of meaningful size, so "is there code?" is the wrong question — all nine have code. The honest distinction is maturity: which bindings ship their own tests, which ship only a smoke example, and what each one depends on to run at all. That gradient is documented below.

The shared contract (read this first)

No binding stands alone. Each one resolves the C artifact — libevosql-memory.{so,dylib,dll} or the static .a — at build or load time, then surfaces the C error path (evo_strerror / evo_sqlstate) as a language-native exception. The C error codes every binding maps from:

EVO_OK = 0 · EVO_ERR_CONNECT = -2 · EVO_ERR_AUTH = -3 · EVO_ERR_PROTOCOL = -4 · EVO_ERR_SQL = -5 · EVO_ERR_BUFFER = -6 · EVO_ERR_NOT_FOUND = -7

Build the C SDK once; every other binding consumes the artifact in client/libevosql-memory/build/:

# produces libevosql-memory.{dylib,so,a} + a pkg-config file
make -C client/libevosql-memory

This dependency is the single most important fact about the non-C, non-Python-pure bindings: until that artifact exists (or a pre-built release binary is pointed at via an env override), they have nothing to link against.

Maturity tiers

Bindings group into three tiers by how much of the binding itself is verified in-tree — independent of whether the underlying C library works (it does; it has its own tests and compiled artifacts).

Tier 0 — Foundation (C): implemented & tested

The C library libevosql-memory is the canonical client and the only one that owns wire logic. It is the most complete member of the family.

  • Source: src/conn.c (~590 lines) and src/subscribe.c (~496 lines) — connection/query plus the live subscription and CDC paths.
  • Build: a Makefile producing both the static archive (libevosql-memory.a) and a shared library (libevosql-memory.dylib/.so), plus an evosql-memory.pc pkg-config file. Compiled artifacts are present in build/.
  • Tests: tests/test_sdk.c and tests/test_subscribe.c ship in-tree.

Status: implemented and test-covered. This is the one to trust without caveats.

info
"Test-covered" here means a test file exists in the binding's own tree. Like every test in this project, these are runtime tests — they assume a reachable evosql-server. They are not a substitute for confirming your own deployment connects.

Tier 2 — Implemented bindings, smoke example only

These bindings have a complete, real wrapper over the C surface and a runnable smoke example, but no test suite of their own in-tree. The code is there and reviewable; the verification is the single put/get example, not an assertion suite.

Tier 3 — Implemented bindings, no tests, binary-dependent

Six bindings exist as complete, idiomatic wrappers — full FFI/P-Invoke/JNA/system-module surfaces with native error types — but each one ships only source plus a smoke example, no test suite, and each is hard-dependent on the C artifact being built or a pre-built release binary being pointed at. They are real code; they are not independently verified in-tree. Do not assume any of these is production-proven from this repo alone.

Each of these mirrors the same put/get/checkpoint/subscribe contract and maps evo_strerror/evo_sqlstate to a native error (evosql::Error/NotFound, EvoException/EvoNotFoundException, Java NotFoundException, Ruby NotFound, a Rust Error enum, Swift EvoError/EvoNotFound). The wrappers are written and idiomatic. What is missing is in-tree test coverage and, until you build it, the C artifact they bind against.

warning
What this page will not tell you: that any Tier-3 binding is "shipped and working." The source is present and the smoke example compiles against a built C library — but there is no test suite in these directories to back a stronger claim, so none is made here. Build the C SDK, run the binding's put_get example against your own server, and verify before depending on it.

What "completing" a binding means

The contract for promoting any Tier-2 / Tier-3 binding toward first-class is the same five steps every existing binding followed:

  1. Resolve the libevosql-memory artifact by env override or relative path to ../libevosql-memory/build/.
  2. Bind the ~16-function public API from include/evosql_memory.h.
  3. Wrap the opaque evo_conn_t* and evo_subscription_t* handles in an idiomatic type with explicit close + finalizer.
  4. Surface evo_strerror() / evo_sqlstate() as native exceptions.
  5. Ship a smoke example that drops + creates + writes + reads a memory store.

The honest gap for the Tier-2 and Tier-3 set is step beyond (5): an in-tree test suite that asserts each verb, the way the C library and Python binding already do. Until that lands for a given binding, this page keeps it labelled by what is actually present, not by aspiration.

Picking a binding today

  • Want the safest path? Use the C library or the Python binding — the two with in-tree tests.
  • On Node or Go? The binding is real; confirm it against your server with the bundled put_get example first.
  • On C++ / .NET / Java / Ruby / Rust / Swift? The wrapper exists and is idiomatic, but build the C SDK, run the smoke example, and validate before production use — there is no test suite proving it here.
  • Every binding needs the C artifact built once (make -C client/libevosql-memory) or a pre-built release library pointed at via the binding's env override.
