SDK · LANGUAGE FAMILY

SDKs, language roadmap

EvolutionDB ships a family of agent-memory clients under client/*-evosql-memory. One C library, libevosql-memory, does the wire work. Every other language is a thin binding over it through C FFI. This page gives the honest status of the family. It shows what is built and tested. It shows what is built but not yet tested in the repo. It shows what each binding still depends on. When a binding has no test suite, it says so. It never says "works".

The shape of the family

Every binding talks to evosql-server through one shared base. The heavy work lives in the C library. This includes TCP, the EVO handshake, AUTH, RESULT/ROW parsing, NOTIFY framing, CDC NDJSON, and vector formatting. The language layers add idiomatic features on top. These are things like generics, exceptions, IDisposable, and finalizers. The language layers do not rewrite the protocol.

The C contract is small and stable. It has about 16 public functions in include/evosql_memory.h. These cover connect/close, exec/query, the memory and checkpoint verbs, vector formatting, the subscribe/CDC pair, and the two last-error accessors. Every binding mirrors that same surface.

warning
Read this page, not the directory listing. Each binding directory holds real source files of meaningful size. So "is there code?" is the wrong question. All nine have code. The real difference is maturity. Some bindings ship their own tests. Some ship only a smoke example. Each one has its own things it needs to run at all. The sections below cover all of this.

The shared contract (read this first)

No binding stands alone. Each one finds the C artifact at build or load time. This is libevosql-memory.{so,dylib,dll} or the static .a. Each one then takes the C error path and turns it into a language-native exception. The C error path uses evo_strerror and evo_sqlstate. Here are the C error codes every binding maps from.

EVO_OK = 0 · EVO_ERR_CONNECT = -2 · EVO_ERR_AUTH = -3 · EVO_ERR_PROTOCOL = -4 · EVO_ERR_SQL = -5 · EVO_ERR_BUFFER = -6 · EVO_ERR_NOT_FOUND = -7

Build the C SDK once. Every other binding then uses the artifact in client/libevosql-memory/build/.

# produces libevosql-memory.{dylib,so,a} + a pkg-config file
make -C client/libevosql-memory

This is the most important fact about the bindings that are not C and not pure Python. The artifact must exist first. You can also point them at a pre-built release binary through an env override. Until then, they have nothing to link against.

Maturity tiers

Bindings fall into three tiers. The tier shows how much of the binding itself is tested in the repo. This is separate from the C library. The C library works. It has its own tests and compiled artifacts.

Tier 0, Foundation (C): implemented & tested

The C library libevosql-memory is the main client. It is the only one that owns the wire logic. It is the most complete member of the family.

  • Source: src/conn.c (~590 lines) and src/subscribe.c (~496 lines). These hold connection and query work plus the live subscription and CDC paths.
  • Build: a Makefile that produces the static archive (libevosql-memory.a) and a shared library (libevosql-memory.dylib/.so). It also produces an evosql-memory.pc pkg-config file. Compiled artifacts are present in build/.
  • Tests: tests/test_sdk.c and tests/test_subscribe.c ship in-tree.

Status: implemented and test-covered. This is the one to trust without caveats.

Tier 2: Implemented bindings, smoke example only

These bindings have a complete, real wrapper over the C surface. They also have a smoke example you can run. But they have no test suite of their own in-tree. The code is there and you can review it. The only check is the single put/get example. There is no assertion suite.

Tier 3: Implemented bindings, no tests, binary-dependent

Six bindings exist as complete, idiomatic wrappers. They have full FFI, P-Invoke, JNA, or system-module surfaces with native error types. But each one ships only source plus a smoke example, no test suite. Each one also needs the C artifact to be built first. You can also point it at a pre-built release binary. They are real code. They are not checked on their own in the repo. Do not assume any of these is production-proven from this repo alone.

Each of these mirrors the same put/get/checkpoint/subscribe contract. Each one maps evo_strerror and evo_sqlstate to a native error. C++ uses evosql::Error and NotFound. .NET uses EvoException and EvoNotFoundException. Java uses NotFoundException. Ruby uses NotFound. Rust uses an Error enum. Swift uses EvoError and EvoNotFound. The wrappers are written and idiomatic. Two things are missing. There is no test coverage in the repo. And the C artifact they bind against is not there until you build it.

warning
This page will not tell you that any Tier-3 binding is "shipped and working." The source is present. The smoke example compiles against a built C library. But there is no test suite in these directories to back a stronger claim. So no stronger claim is made here. Build the C SDK. Run the binding's put_get example against your own server. Verify it before you depend on it.

What "completing" a binding means

You move a Tier-2 or Tier-3 binding toward first-class with the same five steps every existing binding followed.

  1. Find the libevosql-memory artifact by env override or relative path to ../libevosql-memory/build/.
  2. Bind the ~16-function public API from include/evosql_memory.h.
  3. Wrap the opaque evo_conn_t* and evo_subscription_t* handles in an idiomatic type with an explicit close and a finalizer.
  4. Surface evo_strerror() and evo_sqlstate() as native exceptions.
  5. Ship a smoke example that drops, creates, writes, and reads a memory store.

The honest gap for the Tier-2 and Tier-3 set is the step after (5). That step is a test suite in the repo that asserts each verb. The C library and Python binding already do this. Until that lands for a given binding, this page labels it by what is really there, not by hope.

Picking a binding today

  • Want the safest path? Use the C library or the Python binding. These are the two with tests in the repo.
  • On Node or Go? The binding is real. Confirm it against your server with the bundled put_get example first.
  • On C++ /.NET / Java / Ruby / Rust / Swift? The wrapper exists and is idiomatic. Build the C SDK, run the smoke example, and validate it before production use. There is no test suite here to prove it.
  • Every binding needs the C artifact built once with make -C client/libevosql-memory. You can also point it at a pre-built release library through the binding's env override.
