CONNECTOR

YouTube connector

This is a *-sync connector. It pulls your YouTube subscriptions, playlists and liked videos into EvolutionDB long-term memory. These are the public signals that represent your watch activity. An AI assistant can then recall which channels you follow about Rust, or which video you liked last month.

What it does

The package is called evolutiondb-youtube-sync. The version is 0.1.1 in pyproject.toml. It talks to the YouTube Data API v3. It walks three resources for the signed-in account. It writes one memory record per item into a MEMORY STORE. It uses the PostgreSQL wire protocol on :5433. You can run it once or as a polling loop.

It covers the three signals the public API still exposes. Watch history has not been in the public API since 2016. So the connector captures what channels you follow, what playlists you own, and what videos you liked. One pass reads these resources.

  • Subscriptions are the channels you follow. It reads /subscriptions?mine=true with part=snippet.
  • Playlists are the playlists you own. It reads /playlists?mine=true with part=snippet,contentDetails.
  • Liked videos are read through /playlistItems against the synthetic playlist id LL. This is YouTube's "Liked videos" list.

What goes into memory per item depends on the kind.

  • Subscription. fact = Subscribed to YouTube channel "<title>". It also stores channel_id, title, description clipped at 400 chars, url as https://www.youtube.com/channel/<id>, modified_at, and tags = ["youtube","subscription","channel"]. The key is youtube_sub_<channel_id>.
  • Playlist. fact = YouTube playlist "<title>" (<itemCount> items). It also stores playlist_id, title, item_count, url as …/playlist?list=<id>, modified_at, and tags = ["youtube","playlist"]. The key is youtube_playlist_<playlist_id>.
  • Liked video. fact = Liked YouTube video "<title>" by <channel>. It also stores video_id, title, channel, description clipped at 400 chars, url as …/watch?v=<id>, modified_at, published_at, and tags = ["youtube","video","liked"]. The key is youtube_liked_<video_id>.

Every record also carries source = "youtube" and a kind. The kind is one of subscription, playlist, or video. Each record is keyed by its stable YouTube id. So a re-sync UPSERTs the record instead of making a duplicate. After a successful pass the connector writes a watermark record named youtube_state_last_sync with the pass timestamp.

Install

# from PyPI
pip install evolutiondb-youtube-sync

# or editable, from the monorepo
pip install -e client/youtube-sync

It needs Python >=3.9. The runtime dependencies are psycopg[binary]>=3.1 and evolutiondb-pii>=0.1.1. The first one is the EvolutionDB and PostgreSQL wire client. The PII package is used only when it is present. See Notes & limitations. Installing the package adds one console script named evolutiondb-youtube-sync. Its entry point is youtube_sync.__main__:main. That entry point calls youtube_sync.sync:main.

Configure

Configuration is read from environment variables. The CLI also loads a dotenv file on its own. The default file is .env. You can change it with --env-file. Existing environment variables win over file values. The loader uses os.environ.setdefault and skips lines with empty values.

Google OAuth

The connector reuses the same Google OAuth installed-app client you already set up for gmail-sync. It only asks for the youtube.readonly scope. You must add that scope to your OAuth consent screen. Then set these values.

GOOGLE_CLIENT_ID=xxxxxxxxxx.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=xxxxxxxxxxxxxxxxxxxxxxxx

# refresh-token cache; written 0600 on POSIX, kept separate
# from the Gmail / Calendar caches so they do not collide
YOUTUBE_TOKEN_CACHE=~/.evosql/youtube_token.json

The requested scope is read-only. The full scope is https://www.googleapis.com/auth/youtube.readonly. The connector cannot change subscriptions, playlists or likes.

First-time auth

The first run with --auth opens a browser tab for Google's consent screen. The package opens a one-shot HTTP listener on a random 127.0.0.1 port. This is the OAuth loopback redirect for installed apps. It checks the state parameter against CSRF. It swaps the auth code for a refresh token. It stores the result at $YOUTUBE_TOKEN_CACHE with mode 0600 on POSIX. Later runs are silent. The cached refresh token mints fresh access tokens when they are needed.

evolutiondb-youtube-sync --auth

On success it prints {"ok": true, "token_cache": "~/.evosql/youtube_token.json"} and exits with code 0. Sometimes Google returns no refresh_token. This usually means the app was already approved. In that case the flow raises an error. It tells you to revoke the app at https://myaccount.google.com/permissions and try again.

Usage

The CLI accepts these flags from youtube_sync/sync.py. They are --auth, --once, --interval SECONDS, --dry-run, and --env-file. The default for --env-file is .env.

# One-time consent; caches a refresh token, then exits.
evolutiondb-youtube-sync --auth

# A single sync pass (also the default when --interval is omitted).
evolutiondb-youtube-sync --once

# Daemon mode: re-poll every 900s (minimum enforced is 60s).
evolutiondb-youtube-sync --interval 900

# Authenticate + fetch without writing anything to the database.
evolutiondb-youtube-sync --once --dry-run

Each pass prints a one-line JSON summary on stdout. It has one counter per resource.

{"ok": true, "subscriptions": 37, "playlists": 4, "liked_videos": 218, "errors": 0}

On an auth failure the summary is {"ok": false, "error": "…", "hint": "run with --auth"} and the exit code is 3. Any other failure prints {"ok": false, "error": "…"} and the exit code is 4. A fetch failure on one resource is caught for that resource alone. It bumps the errors counter and is logged to stderr. The other resources still run. In daemon mode, SIGTERM and SIGINT ask for a graceful stop after the current pass finishes.

You can also drive the engine from your own code.

from youtube_sync.sync import Config, sync_once

cfg = Config()  # reads env vars described above
counts = sync_once(cfg, dry_run=False, interactive=False)
print(counts)  # {"subscriptions": .., "playlists": .., "liked_videos": .., "errors": ..}

Reference

The package has four modules under youtube_sync/. sync.py runs the orchestration and the CLI. auth.py handles OAuth. api.py is the YouTube Data API client. state.py is the memory backend.

youtube_sync.sync

def main(argv: list | None = None) -> int

The CLI entry point and the console script. It parses the arguments, loads the dotenv file, and builds a Config. Then it runs the --auth flow, a single pass, or the polling loop. It returns a process exit code.

def sync_once(cfg: Config, *, dry_run: bool = False, interactive: bool = False) -> dict

Runs one sync pass. It builds a YouTubeAuth and a YouTubeClient. It opens a MemoryStore unless dry_run is set. Then it reads subscriptions, playlists and liked videos. For each item it builds one record, masks PII if available, and UPSERTs it. It writes the last_sync watermark at the end. It returns a counter dict with keys subscriptions, playlists, liked_videos, and errors. It raises RuntimeError if GOOGLE_CLIENT_ID or GOOGLE_CLIENT_SECRET is missing.

class Config

A snapshot of all configuration read from environment variables. Fields include the Google client id and secret, the token cache path, the EvolutionDB connection, the MCP_USER_ID namespace, and the store name. All fields are filled in __init__ from os.environ with the defaults listed above.

youtube_sync.auth

class YouTubeAuth(client_id: str, client_secret: str, cache_path: str)

Runs the Google OAuth 2.0 loopback-redirect flow for an installed app. It uses only the standard library. It uses urllib and http.server. It asks for the youtube.readonly scope with access_type=offline and prompt=consent. It raises AuthError if the client id or secret is missing.

def YouTubeAuth.ensure_token(interactive: bool = False) -> str

Returns a valid access token. It uses the cached access token if it still has more than 60s of life. Otherwise it refreshes the token with the stored refresh token. When interactive=True and no token works, it runs the browser consent flow. It also runs that flow again if a refresh fails. When interactive=False it raises AuthError and tells you to run --auth.

class TokenCache(path: Path)

Reads and writes the JSON refresh-token cache. save() writes atomically through a .tmp file and chmod 0600 on POSIX. load() returns {} when the file is missing or corrupt.

class AuthError(Exception)

Raised for any OAuth problem. This covers missing credentials, token-endpoint errors, a state mismatch, a timeout while waiting for the redirect, or a missing refresh_token in Google's response. A state mismatch is treated as a possible MITM.

youtube_sync.api

class YouTubeClient(token_provider: Callable[[], str], user_agent: str = "evolutiondb-youtube-sync")

A small YouTube Data API v3 client for https://youtube.googleapis.com/youtube/v3. It is built on urllib and pins no SDK. The token_provider callable gives a bearer token for each request. Token refresh is invisible to the caller. It retries once on 401 by minting a new token. It also retries once on 429. On 429 it honors Retry-After before it sleeps and retries. The default wait is 30s.

def YouTubeClient.subscriptions() -> Iterator[dict]

Yields the signed-in user's subscription items. It uses part=snippet, mine=true, and maxResults=50. It follows nextPageToken pagination on its own.

def YouTubeClient.playlists() -> Iterator[dict]

Yields the playlists the user owns. It uses part=snippet,contentDetails, mine=true, and maxResults=50. It handles pagination.

def YouTubeClient.playlist_items(playlist_id: str) -> Iterator[dict]

Yields the items of a given playlist. It uses part=snippet,contentDetails and maxResults=50. It handles pagination.

def YouTubeClient.liked_videos() -> Iterator[dict]

A convenience wrapper over playlist_items("LL"). LL is YouTube's synthetic playlist id for the user's liked videos. The API accepts it on /playlistItems.

class YouTubeError(status: int, msg: str, url: str)

Raised for HTTP responses from the API that are not retried. It carries the HTTP status. The constructor reads the human message from the API's error.message field when it is present.

youtube_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

The EvolutionDB memory backend. It mirrors the backend used by the sibling connectors. It connects with psycopg in autocommit mode. It runs CREATE MEMORY STORE when needed and ignores an "already exists" error. It reconnects on its own when a connection drops. It tries 3 times with a growing backoff. It raises RuntimeError if psycopg is not installed.

def MemoryStore.put_record(key: str, record: dict) -> None

UPSERTs one JSON record into the store under (namespace, key). It uses MEMORY PUT INTO <store>.

def MemoryStore.get_watermark_iso(resource: str) -> str | None / set_watermark_iso(resource: str, iso: str) -> None

Read or write a per-resource sync watermark. It is stored as a youtube_state_<resource> record. The value carries synced_at and a saved_at epoch. The driver writes last_sync at the end of each pass. The read side exists, but the current pass does not use it to limit fetches. See Notes.

Notes & limitations

  • No watch history. Watch history is not in the public YouTube API. It was removed in 2016. So the connector covers subscriptions, owned playlists and liked videos on purpose. These are the closest public signals.
  • Read-only. Only the youtube.readonly scope is requested. There is no write or send path. No EVOSQL_*_SEND-style opt-in exists for this connector.
  • Full re-sync each pass. state.py exposes get_watermark_iso() and the driver records a last_sync watermark. But sync_once() does not yet use the watermark to limit what it fetches. Every pass re-reads all three resources. It relies on key-based UPSERT to avoid duplicates. Incremental fetch is wired up but not yet used.
  • Playlist contents are not stored. YouTubeClient.playlist_items() exists and is used for liked videos through LL. But the driver does not expand the items of your own playlists. It stores only the playlist record. That record contains the title and the item count.
  • Optional PII masking. If evolutiondb-pii can be imported, the fields fact, title, description, channel and url are run through protect_record before storage. This emits companion records keyed <main_key>_pii. Without the package, records are stored as they are. See the PII guard.
  • Version mismatch. pyproject.toml declares 0.1.1. youtube_sync/__init__.__version__ still reads 0.1.0. Treat the pyproject value as the correct one.
  • Daemon floor. --interval has a minimum of 60 seconds. With no --interval, or with --once, it runs a single pass and exits.
  • Privacy. Everything read with this token can be searched by anyone with the matching MCP_USER_ID namespace. Use a separate Google account if your subscriptions and likes are sensitive.
