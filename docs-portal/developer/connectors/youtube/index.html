CONNECTOR

YouTube connector

A *-sync connector that pulls your YouTube subscriptions, playlists and liked videos into EvolutionDB long-term memory: the public signals that stand in for a watch history, so an AI assistant can recall "which channels do I follow about Rust?" or "what was that video I liked last month?".

What it does

The package (evolutiondb-youtube-sync, version 0.1.1 per pyproject.toml) talks to the YouTube Data API v3, walks three resources for the authenticated account, and writes one memory record per item into a MEMORY STORE via the PostgreSQL wire protocol on :5433. It runs as a one-shot pass or as a polling loop.

It covers the three signals the public API still exposes. As the source notes, watch history itself has not been in the public API since 2016, so the connector captures what channels you follow, what playlists you own, and what videos you liked instead. Concretely, one pass drains:

  • Subscriptions, your followed channels (/subscriptions?mine=true, part=snippet).
  • Playlists, playlists you own (/playlists?mine=true, part=snippet,contentDetails).
  • Liked videos, read via /playlistItems against the synthetic playlist id LL (YouTube's "Liked videos" list).

What lands in memory per item depends on the kind:

  • Subscription, fact = Subscribed to YouTube channel "<title>", plus channel_id, title, description (clipped at 400 chars), url (https://www.youtube.com/channel/<id>), modified_at, and tags = ["youtube","subscription","channel"]. Keyed youtube_sub_<channel_id>.
  • Playlist, fact = YouTube playlist "<title>" (<itemCount> items), plus playlist_id, title, item_count, url (…/playlist?list=<id>), modified_at, and tags = ["youtube","playlist"]. Keyed youtube_playlist_<playlist_id>.
  • Liked video, fact = Liked YouTube video "<title>" by <channel>, plus video_id, title, channel, description (clipped at 400 chars), url (…/watch?v=<id>), modified_at, published_at, and tags = ["youtube","video","liked"]. Keyed youtube_liked_<video_id>.

Every record also carries source = "youtube" and a kind (subscription / playlist / video). Because each record is keyed by its stable YouTube id, a re-sync UPSERTs rather than duplicates. After a successful pass the connector writes a watermark record youtube_state_last_sync with the pass timestamp.

Install

# from PyPI
pip install evolutiondb-youtube-sync

# or editable, from the monorepo
pip install -e client/youtube-sync

Requires Python >=3.9. Runtime dependencies are psycopg[binary]>=3.1 (the EvolutionDB / PostgreSQL wire client) and evolutiondb-pii>=0.1.1. The PII package is used opportunistically, see Notes & limitations. Installing exposes one console script, evolutiondb-youtube-sync (entry point youtube_sync.__main__:main, which delegates to youtube_sync.sync:main).

Configure

Configuration is read from environment variables. The CLI also auto-loads a dotenv file (.env by default, override with --env-file); existing environment variables win over file values (the loader uses os.environ.setdefault, and skips lines with empty values).

Google OAuth

The connector reuses the same Google OAuth installed-app client you already configured for gmail-sync, it only requests the youtube.readonly scope. You must add that scope to your OAuth consent screen, then set:

GOOGLE_CLIENT_ID=xxxxxxxxxx.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=xxxxxxxxxxxxxxxxxxxxxxxx

# refresh-token cache; written 0600 on POSIX, kept separate
# from the Gmail / Calendar caches so they do not collide
YOUTUBE_TOKEN_CACHE=~/.evosql/youtube_token.json

The requested scope is read-only (https://www.googleapis.com/auth/youtube.readonly); the connector cannot modify subscriptions, playlists or likes.

First-time auth

The first run with --auth opens a browser tab for Google's consent screen. The package binds a one-shot HTTP listener on a random 127.0.0.1 port (the OAuth loopback redirect for installed apps), validates the state parameter against CSRF, exchanges the auth code for a refresh token, and stores the result at $YOUTUBE_TOKEN_CACHE (mode 0600 on POSIX). Subsequent runs are silent, the cached refresh token mints fresh access tokens on demand.

evolutiondb-youtube-sync --auth

On success it prints {"ok": true, "token_cache": "~/.evosql/youtube_token.json"} (exit 0). If Google returns no refresh_token (typically because the app was already consented), the flow raises and tells you to revoke the app at https://myaccount.google.com/permissions and retry.

Usage

The CLI accepts these flags (from youtube_sync/sync.py): --auth, --once, --interval SECONDS, --dry-run, and --env-file (default .env).

# One-time consent; caches a refresh token, then exits.
evolutiondb-youtube-sync --auth

# A single sync pass (also the default when --interval is omitted).
evolutiondb-youtube-sync --once

# Daemon mode: re-poll every 900s (minimum enforced is 60s).
evolutiondb-youtube-sync --interval 900

# Authenticate + fetch without writing anything to the database.
evolutiondb-youtube-sync --once --dry-run

Each pass prints a one-line JSON summary on stdout, with a counter per resource:

{"ok": true, "subscriptions": 37, "playlists": 4, "liked_videos": 218, "errors": 0}

On an auth failure the summary is {"ok": false, "error": "…", "hint": "run with --auth"} (exit 3); any other failure prints {"ok": false, "error": "…"} (exit 4). A fetch failure on a single resource is caught per-resource, it bumps the errors counter and is logged to stderr, but the other resources still run. In daemon mode, SIGTERM/SIGINT requests a graceful stop after the current pass finishes.

You can also drive the engine programmatically:

from youtube_sync.sync import Config, sync_once

cfg = Config()  # reads env vars described above
counts = sync_once(cfg, dry_run=False, interactive=False)
print(counts)  # {"subscriptions": .., "playlists": .., "liked_videos": .., "errors": ..}

Reference

The package is four modules under youtube_sync/: sync.py (orchestration + CLI), auth.py (OAuth), api.py (YouTube Data API client), and state.py (memory backend).

youtube_sync.sync

def main(argv: list | None = None) -> int

CLI entry point (the console script). Parses arguments, loads the dotenv file, builds a Config, and runs either the --auth flow, a single pass, or the polling loop. Returns a process exit code.

def sync_once(cfg: Config, *, dry_run: bool = False, interactive: bool = False) -> dict

Runs one sync pass. Builds a YouTubeAuth + YouTubeClient, opens a MemoryStore (unless dry_run), then drains subscriptions, playlists and liked videos: building one record per item, optionally PII-protecting it, and UPSERTing it. Writes the last_sync watermark at the end. Returns a counter dict {"subscriptions", "playlists", "liked_videos", "errors"}. Raises RuntimeError if GOOGLE_CLIENT_ID/GOOGLE_CLIENT_SECRET are missing.

class Config

Snapshot of all configuration read from environment variables (Google client id/secret, token cache path, EvolutionDB connection, MCP_USER_ID namespace, and store name). All fields are populated in __init__ from os.environ with the defaults listed above.

youtube_sync.auth

class YouTubeAuth(client_id: str, client_secret: str, cache_path: str)

Implements the Google OAuth 2.0 loopback-redirect flow for an installed app, using only the standard library (urllib + http.server). Requests the youtube.readonly scope with access_type=offline and prompt=consent. Raises AuthError if client id/secret are missing.

def YouTubeAuth.ensure_token(interactive: bool = False) -> str

Returns a valid access token. Uses the cached access token if it still has >60s of life, otherwise refreshes via the stored refresh token. When interactive=True and no token works, runs the browser consent flow (and re-runs it if a refresh fails); when interactive=False it raises AuthError telling you to run --auth.

class TokenCache(path: Path)

Reads/writes the JSON refresh-token cache. save() writes atomically via a .tmp file and chmod 0600 on POSIX; load() returns {} on a missing or corrupt file.

class AuthError(Exception)

Raised for any OAuth problem, missing credentials, token-endpoint errors, state mismatch (treated as a possible MITM), a timeout waiting for the redirect, or a missing refresh_token in Google's response.

youtube_sync.api

class YouTubeClient(token_provider: Callable[[], str], user_agent: str = "evolutiondb-youtube-sync")

Thin YouTube Data API v3 client against https://youtube.googleapis.com/youtube/v3, built on urllib (no SDK pin). The token_provider callable supplies a bearer token per request (so token refresh is transparent). It retries once on 401 (token re-mint) and once on 429, honoring Retry-After (default 30s) before sleeping and retrying.

def YouTubeClient.subscriptions() -> Iterator[dict]

Yields the authenticated user's subscription items (part=snippet, mine=true, maxResults=50), transparently following nextPageToken pagination.

def YouTubeClient.playlists() -> Iterator[dict]

Yields the user's owned playlists (part=snippet,contentDetails, mine=true, maxResults=50), with pagination.

def YouTubeClient.playlist_items(playlist_id: str) -> Iterator[dict]

Yields the items of a given playlist (part=snippet,contentDetails, maxResults=50), with pagination.

def YouTubeClient.liked_videos() -> Iterator[dict]

Convenience wrapper over playlist_items("LL"), LL is YouTube's synthetic playlist id for the user's liked videos, which the API accepts on /playlistItems.

class YouTubeError(status: int, msg: str, url: str)

Raised for non-retried HTTP responses from the API; carries the HTTP status. The constructor pulls the human message out of the API's error.message field when present.

youtube_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

The EvolutionDB memory backend (a mirror of the sibling connectors' backend). Connects with psycopg (autocommit), issues CREATE MEMORY STORE if needed (ignoring "already exists"), and auto-reconnects (3 attempts, growing backoff) on dropped connections. Raises RuntimeError if psycopg is not installed.

def MemoryStore.put_record(key: str, record: dict) -> None

UPSERTs one JSON record into the store under (namespace, key) via MEMORY PUT INTO <store>.

def MemoryStore.get_watermark_iso(resource: str) -> str | None / set_watermark_iso(resource: str, iso: str) -> None

Read/write a per-resource sync watermark, stored as a youtube_state_<resource> record whose value carries synced_at (and a saved_at epoch). The driver writes last_sync at the end of each pass; the read side exists but the current pass does not gate fetches on it (see Notes).

Notes & limitations

  • No watch history. Watch history is not in the public YouTube API (removed in 2016); the connector intentionally covers subscriptions, owned playlists and liked videos as the closest public signals.
  • Read-only. Only the youtube.readonly scope is requested, there is no write/send path. (No EVOSQL_*_SEND-style opt-in exists for this connector.)
  • Full re-sync each pass. Although state.py exposes get_watermark_iso() and the driver records a last_sync watermark, sync_once() does not currently use the watermark to limit what it fetches, every pass re-walks all three resources and relies on key-based UPSERT for idempotency. Incremental fetch is wired but not yet applied.
  • Playlist contents are not stored. YouTubeClient.playlist_items() exists and is used for liked videos via LL, but the driver does not expand the items of your own playlists, only the playlist record (title + item count) is stored.
  • Optional PII masking. If evolutiondb-pii is importable, the fields fact, title, description, channel and url are run through protect_record before storage, emitting companion records (keyed <main_key>_pii). Without it, records are stored as-is (transparent passthrough). See the PII guard.
  • Version mismatch. pyproject.toml declares 0.1.1 while youtube_sync/__init__.__version__ still reads 0.1.0; treat the pyproject value as authoritative.
  • Daemon floor. --interval is clamped to a minimum of 60 seconds. With no --interval (or --once), it runs a single pass and exits.
  • Privacy. Everything read with this token becomes searchable by anyone with the matching MCP_USER_ID namespace. Use a dedicated Google account if your subscriptions/likes are sensitive.

See also: the MCP server that surfaces these records to an assistant, the sibling Gmail connector (shares the Google OAuth client), and the wider EvolutionDB wiki.
