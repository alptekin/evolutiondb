CONNECTOR

Gmail connector

A *-sync connector that pulls Gmail message metadata into EvolutionDB long-term memory. It saves the subject, sender, recipients, snippet, labels and timestamps. An AI assistant can then recall "did Ahmet ever reply to that procurement thread?" weeks later.

What it does

The package is evolutiondb-gmail-sync, version 0.1.2. It talks to the Gmail REST API. It walks your mailbox with a Gmail search query. It writes one memory record per message into a MEMORY STORE. The write goes over the PostgreSQL wire protocol on :5433. It runs as a one-shot pass or as a polling daemon.

The connector stays in metadata mode on purpose. Message bodies are never fetched. Each Gmail message is fetched with format=metadata. The only preview kept is Gmail's pre-computed snippet, which is about 150 characters. The README explains why. The subject, sender, recipient, date and snippet cover most recall use-cases. There is no need to walk the multipart MIME tree.

Here is what lands in memory for each message.

  • fact is a one-line human string. The format is Email from <sender>: <subject> - <snippet>
  • subject, from, to, cc, taken from the matching headers
  • snippet is Gmail's preview. It is clipped at 1000 chars by _short()
  • labels are label IDs resolved to names against users/me/labels
  • message_id, thread_id, rfc_message_id are the Gmail ids plus the RFC Message-ID header
  • internal_date_ms and sent_at hold the server-side receipt time. One is unix millis. The other is an ISO-8601 rendering.
  • source="gmail", kind="email", and tags = ["gmail", "email", <labels…>]

Each record is keyed by the Gmail message id as gmail_msg_<id>. A re-sync UPSERTs the record instead of making a duplicate. A separate watermark record gmail_state_latest tracks the highest internalDate seen. The next pass only asks Gmail for newer mail.

Install

# from PyPI
pip install evolutiondb-gmail-sync

# or editable, from the monorepo
pip install -e client/gmail-sync

It requires Python >=3.9. There are two runtime dependencies. One is psycopg[binary]>=3.1, the EvolutionDB and PostgreSQL wire client. The other is evolutiondb-pii>=0.1.1. The PII package is used only when it is available. See Notes & limitations. Installing adds one console script named evolutiondb-gmail-sync. Its entry point is gmail_sync.__main__:main.

Configure

Configuration is read from environment variables. The CLI also auto-loads a dotenv file. The default file is .env. You can override it with --env-file. Existing environment variables win over file values. This is done with os.environ.setdefault. A starter .env.example ships with the package.

Google OAuth

The Gmail API does not accept simple bearer tokens. You must supply an OAuth client from your own Google Cloud project. First enable the Gmail API. Then create an OAuth client ID of type Desktop app. Then set these values.

GOOGLE_CLIENT_ID=xxxxxxxxxx.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=xxxxxxxxxxxxxxxxxxxxxxxx

# refresh-token cache; written 0600 on POSIX
GMAIL_TOKEN_CACHE=~/.evosql/gmail_token.json

By default the connector requests only the gmail.readonly scope. It cannot send mail, mark messages, or modify labels. Sending is opt-in. Set EVOSQL_GMAIL_SEND to one of 1/true/yes/on to add the gmail.send scope at consent time. The outbox action loop uses this for its approve-send step, per ADR-004. The default is read-only.

First-time auth

The first run opens a browser tab for Google's consent screen. The package starts a one-shot HTTP listener on a random 127.0.0.1 port. This is the OAuth loopback redirect for installed apps. It validates the state parameter against CSRF. It exchanges the auth code for a refresh token. Then it stores the result at $GMAIL_TOKEN_CACHE. The file mode is 0600 on POSIX. Later runs are silent. The refresh token mints fresh access tokens when needed.

evolutiondb-gmail-sync --auth

Usage

The CLI accepts these flags from gmail_sync/sync.py. The flags are --auth, --once, --interval SECONDS, --since with a default of 7d, --query STR, --dry-run, and --env-file with a default of .env.

# One pass; first-time bootstrap pulls the last 7 days.
evolutiondb-gmail-sync --once --since 7d

# Daemon mode: re-poll every 900s (minimum enforced is 60s).
evolutiondb-gmail-sync --interval 900

# Extra Gmail search syntax appended to the built-in after: filter.
evolutiondb-gmail-sync --once --since 30d \
  --query "-category:promotions -category:social"

# Verify auth + fetch without writing anything to the database.
evolutiondb-gmail-sync --once --dry-run

--since takes a compact duration like 1h, 7d, or 2w. The units are s/m/h/d/w. It is only used as the first-run floor when the store has no watermark yet. Once a watermark exists, the pass derives after:<epoch> from it. If neither gives a floor, the query falls back to newer_than:30d.

Each pass prints a one-line JSON summary on stdout.

{"ok": true, "messages": 42, "skipped": 8, "errors": 0}

On auth failure the summary is {"ok": false, "error": "…", "hint": "run with --auth"} and the exit code is 3. Any other failure prints {"ok": false, "error": "…"} and the exit code is 4. In daemon mode, SIGTERM/SIGINT asks for a graceful stop after the current pass finishes.

You can also drive the engine from your own code.

from gmail_sync.sync import Config, sync_once

cfg = Config()  # reads env vars described above
counts = sync_once(cfg, since_seconds=None,
                   extra_query="-category:promotions",
                   dry_run=False)
print(counts)  # {"messages": .., "skipped": .., "errors": ..}

Reference

The package has four modules under gmail_sync/. sync.py handles orchestration and the CLI. auth.py handles OAuth. api.py is the Gmail REST client. state.py is the memory backend.

gmail_sync.sync

def main(argv: list | None = None) -> int

The CLI entry point. It is the console script. It parses arguments, loads the dotenv file, and builds a Config. Then it runs the --auth flow, a single pass, or the daemon loop. It returns a process exit code.

def sync_once(cfg: Config, *, since_seconds: int | None, extra_query: str = "", dry_run: bool = False) -> dict

Runs one sync pass. It authenticates without a prompt and assumes --auth already ran. It resolves labels. It builds the Gmail query from the stored watermark or from since_seconds. Then it goes through the messages one by one. It skips ones that are already stored. For the rest it fetches metadata, builds a record, and writes it. It returns a counter dict {"messages", "skipped", "errors"}.

class Config

A snapshot of all configuration read from environment variables. It covers the Google client id and secret, the token cache path, the EvolutionDB connection, the MCP_USER_ID namespace, the store name, and the poll interval.

def parse_since(text: str) -> int

Parses a compact duration like "7d" or "1h" into a unix-seconds floor. The floor suits Gmail's after: operator. It raises ValueError on bad input.

gmail_sync.auth

class GmailAuth(client_id: str, client_secret: str, cache_path: str)

Runs the Google OAuth 2.0 loopback-redirect flow for an installed app. It uses only the standard library, including urllib and http.server. It raises AuthError if the client id or secret is missing.

def GmailAuth.ensure_token(interactive: bool = False) -> str

Returns a valid access token. It uses the cached access token if it still has more than 60s of life. Otherwise it refreshes the token with the stored refresh token. When interactive=True and no token works, it runs the browser consent flow. When interactive=False it raises AuthError and tells you to run --auth.

class AuthError(Exception)

Raised for any OAuth problem. Problems include missing credentials, token-endpoint errors, a state mismatch, a timeout waiting for the redirect, or a missing refresh_token in Google's response.

gmail_sync.api

class GmailClient(token_provider: Callable[[], str], user_agent: str = "evolutiondb-gmail-sync")

A small Gmail REST client for https://gmail.googleapis.com/gmail/v1. It is built on urllib. The token_provider callable supplies a bearer token for each request. Token refresh is transparent. It retries transient transport errors with exponential backoff (1,2,4,8,16). It retries once on 401 to re-mint the token. It retries once on 429 and honors Retry-After.

def GmailClient.list_messages(q: str, max_results: int = 100) -> Iterator[dict]

Yields message stubs of the form {id, threadId} for a Gmail search query. It follows nextPageToken pagination automatically. q is ordinary Gmail search syntax. It is the same thing you type in the web UI.

def GmailClient.get_message_metadata(msg_id: str) -> dict

Fetches one message with format=metadata. It requests only the From, To, Cc, Subject, Date and Message-ID headers. To capture full bodies, switch to format=full and add a MIME body walker. That code does not exist in the package today.

def GmailClient.list_labels() -> list[dict]

Returns the mailbox's labels from users/me/labels. It is used to resolve label IDs to display names.

def headers_to_dict(headers: list[dict]) -> dict[str, str]

Flattens Gmail's payload.headers array into a dict. It keeps only the headers in KEEP_HEADERS. It comma-joins repeated values, such as multiple Cc lines.

class GmailError(status: int, msg: str, url: str)

Raised for non-transient HTTP responses from Gmail. It carries the HTTP status.

gmail_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

The EvolutionDB memory backend. It connects with psycopg in autocommit mode. It issues CREATE MEMORY STORE if needed. It auto-reconnects on dropped connections. It tries 3 times with growing backoff.

def MemoryStore.put_record(key: str, record: dict) -> None

UPSERTs one JSON record into the store under (namespace, key). It uses MEMORY PUT INTO <store>.

def MemoryStore.has_message(msg_id: str) -> bool

Returns whether gmail_msg_<msg_id> already exists in the namespace. This is the de-dup check that powers the skipped counter.

def MemoryStore.get_watermark_ms() -> int | None / set_watermark_ms(internal_date_ms: int) -> None

Read and write the incremental sync watermark. The watermark is the highest internalDate in millis. It is stored as the special gmail_state_latest record.

Notes & limitations

  • Metadata only. Message bodies and attachments are never fetched. The README and api.py document how to extend it to full bodies with format=full and a MIME walker. That code does not exist in the package today.
  • Read-only by default. Only gmail.readonly is requested. To add gmail.send you must set EVOSQL_GMAIL_SEND yourself. That scope is for the outbox action loop in ADR-004.
  • Optional PII masking. If evolutiondb-pii can be imported, some fields are run through protect_record before storage. The fields are fact, subject, from, to, cc, snippet and rfc_message_id. This emits companion records. Without the package, records are stored as-is. See the PII guard.
  • Version note. pyproject.toml declares 0.1.2. But gmail_sync/__init__.__version__ still reads 0.1.0. Treat the pyproject value as the correct one.
  • Rate limits. Per the README, Gmail's per-user quota is 250 units/second. A metadata fetch costs 5 units. The client retries once on 429 using Retry-After.
  • Privacy. Everything read with this token becomes searchable by anyone with the matching MCP_USER_ID. Use a dedicated Google account for noisy or work-confidential mail. Do not use your primary inbox.

See also the MCP server that surfaces these records to an assistant. See also the wider EvolutionDB wiki.
