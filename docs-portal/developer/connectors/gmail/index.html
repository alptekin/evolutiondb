CONNECTOR

Gmail connector

A *-sync connector that pulls Gmail message metadata — subject, sender, recipients, snippet, labels and timestamps — into EvolutionDB long-term memory, so an AI assistant can recall "did Ahmet ever reply to that procurement thread?" weeks later.

What it does

The package (evolutiondb-gmail-sync, version 0.1.2) talks to the Gmail REST API, walks your mailbox with a Gmail search query, and writes one memory record per message into a MEMORY STORE via the PostgreSQL wire protocol on :5433. It runs as a one-shot pass or as a polling daemon.

It deliberately stays in metadata mode: message bodies are never fetched. Each Gmail message is fetched with format=metadata, and the only preview kept is Gmail's pre-computed snippet (~150 characters). The README's rationale: subject + sender + recipient + date + snippet covers most recall use-cases without walking the multipart MIME tree.

What lands in memory per message:

  • fact — a one-line human string: Email from <sender>: <subject> — <snippet>
  • subject, from, to, cc — from the corresponding headers
  • snippet — Gmail's preview (clipped at 1000 chars by _short())
  • labels — label IDs resolved to names against users/me/labels
  • message_id, thread_id, rfc_message_id — Gmail ids plus the RFC Message-ID header
  • internal_date_ms and sent_at — server-side receipt time (unix millis, plus an ISO-8601 rendering)
  • source="gmail", kind="email", and tags = ["gmail", "email", <labels…>]

Each record is keyed by the Gmail message id as gmail_msg_<id>, so a re-sync UPSERTs rather than duplicates. A separate watermark record (gmail_state_latest) tracks the highest internalDate seen, so the next pass only asks Gmail for newer mail.

Install

# from PyPI
pip install evolutiondb-gmail-sync

# or editable, from the monorepo
pip install -e client/gmail-sync

Requires Python >=3.9. Runtime dependencies are psycopg[binary]>=3.1 (the EvolutionDB / PostgreSQL wire client) and evolutiondb-pii>=0.1.1. The PII package is used opportunistically — see Notes & limitations. Installing exposes one console script, evolutiondb-gmail-sync (entry point gmail_sync.__main__:main).

Configure

Configuration is read from environment variables. The CLI also auto-loads a dotenv file (.env by default, override with --env-file); existing environment variables win over file values (os.environ.setdefault). A starter .env.example ships with the package.

Google OAuth

The Gmail API does not accept simple bearer tokens — you supply an OAuth client from your own Google Cloud project (enable the Gmail API, create an OAuth client ID of type Desktop app), then set:

GOOGLE_CLIENT_ID=xxxxxxxxxx.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=xxxxxxxxxxxxxxxxxxxxxxxx

# refresh-token cache; written 0600 on POSIX
GMAIL_TOKEN_CACHE=~/.evosql/gmail_token.json

By default the connector requests only the gmail.readonly scope — it cannot send mail, mark messages, or modify labels. Sending is opt-in: setting EVOSQL_GMAIL_SEND to one of 1/true/yes/on adds the gmail.send scope at consent time (used by the outbox action loop's approve-send per ADR-004). The default is read-only.

First-time auth

The first run opens a browser tab for Google's consent screen. The package spins up a one-shot HTTP listener on a random 127.0.0.1 port (the OAuth loopback redirect for installed apps), validates the state parameter against CSRF, exchanges the auth code for a refresh token, and stores the result at $GMAIL_TOKEN_CACHE (mode 0600 on POSIX). Subsequent runs are silent — the refresh token mints fresh access tokens on demand.

evolutiondb-gmail-sync --auth

Usage

The CLI accepts these flags (from gmail_sync/sync.py): --auth, --once, --interval SECONDS, --since (default 7d), --query STR, --dry-run, and --env-file (default .env).

# One pass; first-time bootstrap pulls the last 7 days.
evolutiondb-gmail-sync --once --since 7d

# Daemon mode: re-poll every 900s (minimum enforced is 60s).
evolutiondb-gmail-sync --interval 900

# Extra Gmail search syntax appended to the built-in after: filter.
evolutiondb-gmail-sync --once --since 30d \
  --query "-category:promotions -category:social"

# Verify auth + fetch without writing anything to the database.
evolutiondb-gmail-sync --once --dry-run

--since takes a compact duration like 1h, 7d, 2w (units s/m/h/d/w); it is only used as the first-run floor when the store has no watermark yet. Once a watermark exists, the pass derives after:<epoch> from it. If neither yields a floor, the query falls back to newer_than:30d.

Each pass prints a one-line JSON summary on stdout:

{"ok": true, "messages": 42, "skipped": 8, "errors": 0}

On auth failure the summary is {"ok": false, "error": "…", "hint": "run with --auth"} (exit 3); any other failure prints {"ok": false, "error": "…"} (exit 4). In daemon mode, SIGTERM/SIGINT requests a graceful stop after the current pass finishes.

You can also drive the engine programmatically:

from gmail_sync.sync import Config, sync_once

cfg = Config()  # reads env vars described above
counts = sync_once(cfg, since_seconds=None,
                   extra_query="-category:promotions",
                   dry_run=False)
print(counts)  # {"messages": .., "skipped": .., "errors": ..}

Reference

The package is four modules under gmail_sync/: sync.py (orchestration + CLI), auth.py (OAuth), api.py (Gmail REST client), and state.py (memory backend).

gmail_sync.sync

def main(argv: list | None = None) -> int

CLI entry point (the console script). Parses arguments, loads the dotenv file, builds a Config, and runs either the --auth flow, a single pass, or the daemon loop. Returns a process exit code.

def sync_once(cfg: Config, *, since_seconds: int | None, extra_query: str = "", dry_run: bool = False) -> dict

Runs one sync pass. Authenticates non-interactively (assumes --auth ran already), resolves labels, builds the Gmail query from the stored watermark or since_seconds, then iterates messages — skipping those already stored, fetching metadata, building a record, and writing it. Returns a counter dict {"messages", "skipped", "errors"}.

class Config

Snapshot of all configuration read from environment variables (Google client id/secret, token cache path, EvolutionDB connection, MCP_USER_ID namespace, store name, poll interval).

def parse_since(text: str) -> int

Parses a compact duration ("7d", "1h", …) into a unix-seconds floor suitable for Gmail's after: operator. Raises ValueError on malformed input.

gmail_sync.auth

class GmailAuth(client_id: str, client_secret: str, cache_path: str)

Implements the Google OAuth 2.0 loopback-redirect flow for an installed app, using only the standard library (urllib + http.server). Raises AuthError if client id/secret are missing.

def GmailAuth.ensure_token(interactive: bool = False) -> str

Returns a valid access token. Uses the cached access token if still valid (>60s of life), otherwise refreshes via the stored refresh token. When interactive=True and no token works, runs the browser consent flow; when interactive=False it raises AuthError telling you to run --auth.

class AuthError(Exception)

Raised for any OAuth problem — missing credentials, token-endpoint errors, state mismatch, timeout waiting for the redirect, or a missing refresh_token in Google's response.

gmail_sync.api

class GmailClient(token_provider: Callable[[], str], user_agent: str = "evolutiondb-gmail-sync")

Thin Gmail REST client against https://gmail.googleapis.com/gmail/v1, built on urllib. The token_provider callable supplies a bearer token per request (so token refresh is transparent). It retries transient transport errors with exponential backoff (1,2,4,8,16), retries once on 401 (token re-mint) and once on 429 honoring Retry-After.

def GmailClient.list_messages(q: str, max_results: int = 100) -> Iterator[dict]

Yields message stubs ({id, threadId}) for a Gmail search query, transparently following nextPageToken pagination. q is ordinary Gmail search syntax — the same thing you type in the web UI.

def GmailClient.get_message_metadata(msg_id: str) -> dict

Fetches one message with format=metadata, requesting only the From, To, Cc, Subject, Date and Message-ID headers. (To capture bodies instead, the source suggests switching to format=full and adding a MIME body walker.)

def GmailClient.list_labels() -> list[dict]

Returns the mailbox's labels from users/me/labels, used to resolve label IDs to display names.

def headers_to_dict(headers: list[dict]) -> dict[str, str]

Flattens Gmail's payload.headers array into a dict, keeping only the headers in KEEP_HEADERS and comma-joining repeated values (e.g. multiple Cc lines).

class GmailError(status: int, msg: str, url: str)

Raised for non-transient HTTP responses from Gmail; carries the HTTP status.

gmail_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

The EvolutionDB memory backend. Connects with psycopg (autocommit), issues CREATE MEMORY STORE if needed, and auto-reconnects (3 attempts, growing backoff) on dropped connections.

def MemoryStore.put_record(key: str, record: dict) -> None

UPSERTs one JSON record into the store under (namespace, key) via MEMORY PUT INTO <store>.

def MemoryStore.has_message(msg_id: str) -> bool

Returns whether gmail_msg_<msg_id> already exists in the namespace — the de-dup check that powers the skipped counter.

def MemoryStore.get_watermark_ms() -> int | None / set_watermark_ms(internal_date_ms: int) -> None

Read/write the incremental sync watermark (the highest internalDate in millis), stored as the special gmail_state_latest record.

Notes & limitations

  • Metadata only. Message bodies and attachments are never fetched. The README and api.py document how to extend to full bodies (format=full + a MIME walker) if you need them — that code does not exist in the package today.
  • Read-only by default. Only gmail.readonly is requested unless you explicitly set EVOSQL_GMAIL_SEND, which adds gmail.send for the outbox action loop (ADR-004).
  • Optional PII masking. If evolutiondb-pii is importable, the fields fact, subject, from, to, cc, snippet and rfc_message_id are run through protect_record before storage, emitting companion records. Without it, records are stored as-is (transparent passthrough). See the PII guard.
  • Version note. pyproject.toml declares 0.1.2 while gmail_sync/__init__.__version__ still reads 0.1.0; treat the pyproject value as authoritative.
  • Rate limits. Per the README, Gmail's per-user quota is 250 units/second and a metadata fetch costs 5 units; the client retries once on 429 using Retry-After.
  • Privacy. Everything read with this token becomes searchable by anyone with the matching MCP_USER_ID. Use a dedicated Google account for noisy or work-confidential mail rather than your primary inbox.

See also: the MCP server that surfaces these records to an assistant, and the wider EvolutionDB wiki.
