GitHub connector

The GitHub connector pulls every GitHub PR and issue thread you take part in into EvolutionDB long-term memory. The MCP server can then show them through search_memory.

What it does

The evolutiondb-github-sync package walks GitHub's /search/issues endpoint. It finds everything that matches involves:<you> updated:><since>. This covers issues and PRs you wrote, were assigned to, reviewed, were mentioned in, or commented on. It then writes each thread and its comments into an EvolutionDB memory store. It uses the PostgreSQL wire protocol. An AI assistant connected through the MCP server can then recall what you decided on a past PR without reading the diff again.

For every matching thread the connector writes one record per object. Each record uses a fixed key. A re-sync UPSERTs the record instead of making a duplicate. The keys follow this pattern.

# Record                  Key shape
The thread itself          gh_thread_<repo>#<number>
Diff-anchored PR comments   gh_pr_comment_<comment_id>
Issue-thread comments       gh_issue_comment_<comment_id>
PR reviews with a body      gh_pr_review_<review_id>

Each issue gets a per-thread watermark on its updated_at. The watermark is stored under github_state_<repo>#<number>. It lets a re-run skip threads that have not changed since the last pass. Issue and comment bodies are cut to 4 kB before storage. This keeps them small enough for an LLM context window.

info
Records are stored as JSON values. Each one holds a one-line fact string. An example is "alice opened pr o/r#42: …". Each record also holds structured fields. These are repo, number, author, url, state, labels, assignees, timestamps, and tags. PR comments also hold path and commit. PR reviews hold state such as APPROVED and submitted_at.

Install

The PyPI distribution name is evolutiondb-github-sync. The module you import is github_sync. It needs Python 3.9+. Its only dependency is psycopg[binary]>=3.1. The GitHub client is built on the stdlib urllib. It does not use requests or PyGithub.

# from PyPI
pip install evolutiondb-github-sync

# or editable, from the EvolutionDB repo
pip install -e client/github-sync

Installation adds one console script. It is named evolutiondb-github-sync. It maps to github_sync.__main__:main. You can also run the module directly with python -m github_sync.

Configure

Configuration comes from environment variables. At startup the CLI loads a dotenv file. The default is .env in the working directory. You can change it with --env-file. A missing file is ignored. Existing environment variables win. The loader uses setdefault, so it never overwrites a value that is already set in the environment. Copy .env.example to .env and fill it in.

# --- GitHub ---
GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx   # classic PAT: repo (or public_repo) + read:user
GITHUB_USERNAME=your-handle                          # the handle to follow (involves:<user>)

# --- EvolutionDB (PG wire protocol) ---
EVOSQL_HOST=127.0.0.1
EVOSQL_PORT=5433
EVOSQL_USER=admin
EVOSQL_PASSWORD=admin
EVOSQL_DATABASE=evosql

# --- MCP correlation ---
MCP_USER_ID=default_user      # namespace shared with the MCP server
GITHUB_MEMORY_STORE=mcp_mem   # memory store to write into
GITHUB_POLL_INTERVAL=900     # daemon-mode poll interval, seconds

The table below shows each variable and what the Config class reads from it.

# var                  default        meaning
GITHUB_TOKEN          (none, required)  GitHub PAT - Bearer auth on every API call
GITHUB_USERNAME       (none, required)  handle plugged into involves:<user>
EVOSQL_HOST           127.0.0.1         EvolutionDB host
EVOSQL_PORT           5433              EvolutionDB PG-wire port
EVOSQL_USER           admin             EvolutionDB user
EVOSQL_PASSWORD       admin             EvolutionDB password
EVOSQL_DATABASE       evosql            target database
MCP_USER_ID           default_user      memory namespace (mem_namespace column)
GITHUB_MEMORY_STORE   mcp_mem           memory store name (CREATE MEMORY STORE)
GITHUB_POLL_INTERVAL  900               daemon poll interval (informational; --interval wins)
warning
GITHUB_TOKEN and GITHUB_USERNAME are required. If either one is empty after the env is loaded, the CLI prints an error and exits with code 2. It does this before it contacts GitHub or the database.

Usage

Three flags set the run mode. There is also a window flag and a dry-run switch.

# One pass, then exit. Bootstrap from the last 7 days.
evolutiondb-github-sync --once --since 7d

# Daemon mode - re-run every N seconds (floored at 60).
evolutiondb-github-sync --interval 900

# Verify token + scopes and fetch from GitHub, but write nothing to the DB.
evolutiondb-github-sync --once --dry-run

# Point at an explicit dotenv file.
evolutiondb-github-sync --once --env-file ~/secrets/github.env

The list below shows all flags that main() accepts.

# --once              run a single pass and exit
# --interval SECONDS  daemon mode; loops until SIGINT/SIGTERM (min 60s)
# --since SPEC        floor for the first pass; SPEC is N + s|m|h|d|w (default 7d)
# --dry-run           fetch from GitHub but do not open or write the DB
# --env-file PATH     dotenv file to load (default .env; ignored if missing)

If you give neither --once nor --interval, the connector runs a single pass. This is the same as --once. Each pass prints a one-line JSON summary to stdout.

{"ok": true, "threads": 12, "comments": 84, "reviews": 6, "errors": 0}

On a fatal error the pass prints {"ok": false, "error": "<message>"} instead. In daemon mode, SIGINT and SIGTERM ask for a clean stop. The current pass finishes before the loop exits.

The exit codes are simple. 0 means success. 2 means bad arguments or missing required config. 4 means a pass raised an exception.

Reference

github_sync.sync, CLI and sync loop

main(argv: list | None = None) -> int

The console-script entry point is registered as evolutiondb-github-sync. It parses arguments, loads the dotenv file, and builds a Config. Then it runs once or loops on the interval. It returns the process exit code.

sync_once(cfg: Config, *, since_default: str | None, dry_run: bool = False) -> dict[str, int]

Runs a single pass. It searches GitHub for involved threads, builds records, and applies watermark skipping. Unless dry_run is set, it writes to the store. It returns the counters dict {"threads", "comments", "reviews", "errors"}. It raises RuntimeError if the token or username is missing.

parse_since(text: str) -> str

Converts a window spec like "1h", "7d", or "2w" into a UTC ISO-8601 timestamp that ends in Z. The units are s|m|h|d|w. The timestamp is used as the updated:> floor. It raises ValueError on anything it cannot parse.

class Config

Reads all environment variables into typed attributes. These are token, username, evosql_host/port/user/pass/db, user_id, store, and poll_secs.

github_sync.api, GitHub REST client

A thin client built on the standard library urllib. It has no third-party HTTP dependency. You build it as GitHubClient(token). It raises GitHubError(401, …) if the token is empty.

GitHubClient.search_involves(user: str, since_iso: str | None = None) -> Iterator[dict]

Yields the issues and PRs the user is involved in. It uses /search/issues with involves:<user> and an optional updated:><since_iso>. The results are sorted by updated in ascending order. They are paginated 100 per page.

GitHubClient.get_pr_review_comments(owner, repo, number) -> list[dict]
GitHubClient.get_issue_comments(owner, repo, number) -> list[dict]
GitHubClient.get_pr_reviews(owner, repo, number) -> list[dict]

These three methods fetch the data for one thread and fully paginate it. The first gets the diff-anchored review comments. The second gets the general issue-thread comments. The third gets the PR reviews.

parse_repository_url(url: str) -> dict | None

Extracts {"owner", "repo"} from a search result's repository_url. It returns None if the shape is not what it expects.

class GitHubError(Exception)

Holds the HTTP status, the message, and the url. Per-thread GitHubErrors are caught in sync_once. They are logged to stderr and counted as an error. They do not abort the whole pass.

github_sync.state, EvolutionDB memory backend

Writes records and watermarks over the PG wire protocol with psycopg. When it is built, it runs CREATE MEMORY STORE <store>. It ignores the error if the store already exists. If the connection drops, it reconnects up to three times with backoff.

class MemoryStore(host, port, user, password, database, store, namespace)

Opens an autocommit connection. It makes sure the target memory store exists.

MemoryStore.put_record(key: str, record: dict) -> None

Turns record into JSON and runs MEMORY PUT INTO <store> VALUES (<namespace>, <key>, <value>).

MemoryStore.get_watermark(repo, number) -> str | None
MemoryStore.set_watermark(repo, number, updated_at: str) -> None

These read and write the per-thread updated_at watermark under github_state_<repo>#<number>. get_watermark queries the underlying __mem_<store> table directly.

Rate limits

A classic PAT gives 5,000 REST calls per hour. One pass spends about one search call plus three calls per active thread. The client reads X-RateLimit-Remaining and X-RateLimit-Reset. When the budget runs out, it sleeps until reset. It also respects the secondary-rate-limit Retry-After header. It waits at most 5 times before it raises. Transient transport failures are retried with exponential backoff of 1, 2, 4, 8, and 16 s.

info
/search/issues returns at most 1,000 results per query. A narrow --since window matters for very active accounts.

Notes & limitations

  • Not indexed. The connector skips a few things. It skips commit messages, because you can already query them in local git. It skips workflow and check runs, because they are high volume and have low recall value. It skips notifications, because they overlap with the issue and comment streams.
  • PR reviews without a body are skipped. Only reviews that carry a summary body are recorded.
  • Privacy. Records land in the same store the MCP server reads from. Anyone with access to that database and the matching MCP_USER_ID can recall the content through search_memory. Do not use a GITHUB_TOKEN with broader scope than you would hand the assistant directly.
  • State backend. The source says state.py is a copy of the teams-sync backend. It is copied into each connector. Each package works after a single pip install and does not need a shared library.
  • Version. 0.1.0 (Development Status: 4 - Beta).
