GitHub connector

Pull every GitHub PR and issue thread you're involved in into EvolutionDB long-term memory, so the MCP server can surface them through search_memory.

What it does

The evolutiondb-github-sync package walks GitHub's /search/issues endpoint for everything that matches involves:<you> updated:><since>: issues and PRs you authored, were assigned, reviewed, mentioned in, or commented on, and writes each thread and its comments into an EvolutionDB memory store over the PostgreSQL wire protocol. An AI assistant connected through the MCP server can then recall "what did we decide on PR #142 last week?" without re-reading the diff.

For every matching thread the connector writes one record per object, each under a deterministic key so a re-sync UPSERTs rather than duplicating:

# Record                  Key shape
The thread itself          gh_thread_<repo>#<number>
Diff-anchored PR comments   gh_pr_comment_<comment_id>
Issue-thread comments       gh_issue_comment_<comment_id>
PR reviews with a body      gh_pr_review_<review_id>

A per-thread watermark on each issue's updated_at (stored under github_state_<repo>#<number>) lets a re-run skip threads that haven't moved since the last pass. Issue and comment bodies are clipped to 4 kB before storage so they fit comfortably in an LLM context window.

info
Records are stored as JSON values. Each carries a one-line fact string (e.g. "alice opened pr o/r#42: …") plus structured fields, repo, number, author, url, state, labels, assignees, timestamps, and tags. PR comments also carry path + commit; PR reviews carry state (e.g. APPROVED) + submitted_at.

Install

The PyPI distribution name is evolutiondb-github-sync; the importable module is github_sync. It requires Python 3.9+ and depends only on psycopg[binary]>=3.1 (the GitHub client is built on the stdlib urllib, with no requests / PyGithub dependency).

# from PyPI
pip install evolutiondb-github-sync

# or editable, from the EvolutionDB repo
pip install -e client/github-sync

Installation registers a single console script, evolutiondb-github-sync, which maps to github_sync.__main__:main. You can also run the module directly with python -m github_sync.

Configure

Configuration comes from environment variables. On startup the CLI loads a dotenv file (default .env in the working directory, overridable with --env-file; a missing file is ignored). Existing environment variables take precedence, the loader uses setdefault, so it never overwrites a value already set in the environment. Copy .env.example to .env and fill in:

# --- GitHub ---
GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx   # classic PAT: repo (or public_repo) + read:user
GITHUB_USERNAME=your-handle                          # the handle to follow (involves:<user>)

# --- EvolutionDB (PG wire protocol) ---
EVOSQL_HOST=127.0.0.1
EVOSQL_PORT=5433
EVOSQL_USER=admin
EVOSQL_PASSWORD=admin
EVOSQL_DATABASE=evosql

# --- MCP correlation ---
MCP_USER_ID=default_user      # namespace shared with the MCP server
GITHUB_MEMORY_STORE=mcp_mem   # memory store to write into
GITHUB_POLL_INTERVAL=900     # daemon-mode poll interval, seconds

Variable reference, as read by the Config class:

# var                  default        meaning
GITHUB_TOKEN          (none, required)  GitHub PAT - Bearer auth on every API call
GITHUB_USERNAME       (none, required)  handle plugged into involves:<user>
EVOSQL_HOST           127.0.0.1         EvolutionDB host
EVOSQL_PORT           5433              EvolutionDB PG-wire port
EVOSQL_USER           admin             EvolutionDB user
EVOSQL_PASSWORD       admin             EvolutionDB password
EVOSQL_DATABASE       evosql            target database
MCP_USER_ID           default_user      memory namespace (mem_namespace column)
GITHUB_MEMORY_STORE   mcp_mem           memory store name (CREATE MEMORY STORE)
GITHUB_POLL_INTERVAL  900               daemon poll interval (informational; --interval wins)
warning
GITHUB_TOKEN and GITHUB_USERNAME are mandatory. If either is empty after loading the env, the CLI prints an error and exits with code 2 before contacting GitHub or the database.

Usage

Three flags drive the run mode, plus a window and a dry-run switch:

# One pass, then exit. Bootstrap from the last 7 days.
evolutiondb-github-sync --once --since 7d

# Daemon mode - re-run every N seconds (floored at 60).
evolutiondb-github-sync --interval 900

# Verify token + scopes and fetch from GitHub, but write nothing to the DB.
evolutiondb-github-sync --once --dry-run

# Point at an explicit dotenv file.
evolutiondb-github-sync --once --env-file ~/secrets/github.env

Flags accepted by main():

# --once              run a single pass and exit
# --interval SECONDS  daemon mode; loops until SIGINT/SIGTERM (min 60s)
# --since SPEC        floor for the first pass; SPEC is N + s|m|h|d|w (default 7d)
# --dry-run           fetch from GitHub but do not open or write the DB
# --env-file PATH     dotenv file to load (default .env; ignored if missing)

With neither --once nor --interval, the connector runs a single pass (the same as --once). Each pass prints a one-line JSON summary to stdout:

{"ok": true, "threads": 12, "comments": 84, "reviews": 6, "errors": 0}

On a fatal error the pass instead prints {"ok": false, "error": "<message>"}. In daemon mode, SIGINT and SIGTERM request a graceful stop, the current pass finishes before the loop exits.

Exit codes: 0 success, 2 bad arguments or missing required config, 4 a pass raised an exception.

Reference

github_sync.sync, CLI and sync loop

main(argv: list | None = None) -> int

Console-script entry point (registered as evolutiondb-github-sync). Parses arguments, loads the dotenv file, builds a Config, then runs once or loops on the interval. Returns the process exit code.

sync_once(cfg: Config, *, since_default: str | None, dry_run: bool = False) -> dict[str, int]

Runs a single pass: searches GitHub for involved threads, builds records, applies watermark skipping, and (unless dry_run) writes to the store. Returns the counters dict {"threads", "comments", "reviews", "errors"}. Raises RuntimeError if the token or username is missing.

parse_since(text: str) -> str

Converts a window spec like "1h", "7d", or "2w" (units s|m|h|d|w) into a UTC ISO-8601 timestamp ending in Z, used as the updated:> floor. Raises ValueError on anything it can't parse.

class Config

Reads all environment variables into typed attributes (token, username, evosql_host/port/user/pass/db, user_id, store, poll_secs).

github_sync.api, GitHub REST client

A thin client over the standard library urllib, no third-party HTTP dependency. Constructed as GitHubClient(token); raises GitHubError(401, …) if the token is empty.

GitHubClient.search_involves(user: str, since_iso: str | None = None) -> Iterator[dict]

Yields issues and PRs the user is involved in (via /search/issues with involves:<user> plus an optional updated:><since_iso>), sorted by updated ascending, paginated 100 per page.

GitHubClient.get_pr_review_comments(owner, repo, number) -> list[dict]
GitHubClient.get_issue_comments(owner, repo, number) -> list[dict]
GitHubClient.get_pr_reviews(owner, repo, number) -> list[dict]

Fetch (and fully paginate) the diff-anchored review comments, the general issue-thread comments, and the PR reviews for one thread, respectively.

parse_repository_url(url: str) -> dict | None

Extracts {"owner", "repo"} from a search result's repository_url. Returns None for an unexpected shape.

class GitHubError(Exception)

Carries the HTTP status, message, and url. Per-thread GitHubErrors are caught in sync_once, logged to stderr, and counted as an error rather than aborting the whole pass.

github_sync.state, EvolutionDB memory backend

Writes records and watermarks over the PG wire protocol via psycopg. On construction it issues CREATE MEMORY STORE <store> (ignoring the error if it already exists) and reconnects up to three times with backoff on a dropped connection.

class MemoryStore(host, port, user, password, database, store, namespace)

Opens an autocommit connection and ensures the target memory store exists.

MemoryStore.put_record(key: str, record: dict) -> None

Serializes record to JSON and runs MEMORY PUT INTO <store> VALUES (<namespace>, <key>, <value>).

MemoryStore.get_watermark(repo, number) -> str | None
MemoryStore.set_watermark(repo, number, updated_at: str) -> None

Read and write the per-thread updated_at watermark under github_state_<repo>#<number>; get_watermark queries the underlying __mem_<store> table directly.

Rate limits

A classic PAT gives 5,000 REST calls per hour. One pass spends roughly one search call plus three calls per active thread. The client reads X-RateLimit-Remaining / X-RateLimit-Reset and sleeps until reset when the budget is exhausted; it also honours the secondary-rate-limit Retry-After header (bounded to 5 waits before raising). Transient transport failures are retried with exponential backoff (1, 2, 4, 8, 16 s).

info
/search/issues caps at 1,000 results per query, so a narrow --since window matters for very active accounts.

Notes & limitations

  • Not indexed: commit messages (already queryable in local git), workflow / check runs (high volume, low recall value), and notifications (they overlap with the issue / comment streams).
  • PR reviews without a body are skipped, only reviews carrying a summary body are recorded.
  • Privacy: records land in the same store the MCP server reads from. Anyone with access to that database and the matching MCP_USER_ID can recall the content through search_memory. Don't use a GITHUB_TOKEN with broader scope than you'd hand the assistant directly.
  • State backend: state.py is described in-source as a copy of the teams-sync backend, duplicated per connector so each package stays a single pip install away from working rather than sharing a library.
  • Version: 0.1.0 (Development Status: 4 - Beta).
