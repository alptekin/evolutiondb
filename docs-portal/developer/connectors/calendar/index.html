Google Calendar connector

A one-way *-sync connector that pulls Google Calendar events into EvolutionDB long-term memory, so an assistant can recall meetings weeks later alongside the chat and email memory the sibling connectors write.

What it does

The package evolutiondb-calendar-sync reads events from every calendar your Google account can see and UPSERTs one record per event into an EvolutionDB memory store. It is strictly read-only against Google: the only OAuth scope it requests is https://www.googleapis.com/auth/calendar.readonly, so it cannot create, edit, or delete events on your calendar.

Sync is incremental. Google hands back a nextSyncToken on each event-list call; the connector persists that token per calendar (in the same memory store, under a gcal_state_<calendar_id> key) and passes it on the next run, so only changed events come over the wire. A cancelled event is treated as a delete — the matching memory row is removed. When Google answers 410 Gone (the token is older than 30 days) the connector clears the stale token and re-bootstraps from the --since floor.

Each event is shaped into a record with these fields, written as JSON into the store:

  • fact — a one-line natural-language sentence built from the title, time, calendar, and organizer (used for search recall)
  • summary — event title (falls back to (no title))
  • description — event body, clipped to the first 1000 characters (see Notes)
  • location, organizer (email or display name), attendees (list of names)
  • start / end — RFC 3339 timestamp for timed events, or a bare date for all-day events; all_day is a boolean
  • status (confirmed / tentative / cancelled), html_link, created_at, updated_at
  • calendar_id, calendar_name, source (gcal), kind (event), and tags (["gcal", "event", <calendar_name>])

Records are keyed gcal_event_<calendar_id>_<event_id>, so a re-sync of the same event overwrites the previous row rather than duplicating it.

The memory store is reached over the PostgreSQL wire protocol (default port 5433) using the EvolutionDB MEMORY SQL surface. See the MCP server for how an assistant later reads these records back via search_memory.

Install

# From PyPI
pip install evolutiondb-calendar-sync

# Or editable, from this repo
pip install -e client/calendar-sync

Requires Python 3.9 or newer. Runtime dependencies are psycopg[binary]>=3.1 (the EvolutionDB connection) and evolutiondb-pii>=0.1.1 (optional PII redaction — see Notes). The Google API access uses only the Python standard library (urllib, http.server); there is no Google SDK dependency.

Configure

Configuration is read from environment variables, optionally seeded from a .env file. A .env.example ships with the package. The loader only fills variables that are not already set in the environment (os.environ.setdefault), so real environment values win over the file.

Google OAuth setup

If you already configured evolutiondb-gmail-sync you can reuse the same OAuth Desktop client — just enable the Calendar API on the same Google Cloud project.

  1. Enable the Calendar API: calendar-json.googleapis.com → Enable.
  2. APIs & Services → Credentials → reuse an existing Desktop OAuth client, or create one of type Desktop app.
  3. Copy the Client ID and Client secret into your environment as GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET.
  4. Make sure your Google account is listed under Test Users on the OAuth consent screen.

Usage

First-time auth

Run the consent flow once. It opens a browser tab to Google's consent screen, captures the redirect on a loopback port, and stores the refresh token at the cache path. Subsequent runs are silent.

evolutiondb-calendar-sync --auth

Running a sync

# One pass; bootstrap from a wall-clock floor (default --since is 29d).
evolutiondb-calendar-sync --once --since 30d

# Daemon mode: poll every 15 minutes (900 seconds).
evolutiondb-calendar-sync --interval 900

# Auth + scope check, no writes.
evolutiondb-calendar-sync --once --dry-run

Each pass prints a one-line JSON summary to stdout, for example:

{"ok": true, "calendars": 6, "events": 142, "deleted": 3, "skipped": 1, "errors": 0}

On failure it prints {"ok": false, "error": "…"} (with a "hint": "run with --auth" when the token is missing).

CLI flags

FlagDefaultMeaning
--auth—Run the consent flow, cache the token, and exit
--once—Run a single sync pass and exit
--interval N—Daemon mode: repeat a pass every N seconds (floored at 60)
--since DUR29dFirst-run wall-clock floor for updatedMin; format like 30s, 5m, 1h, 3d, 2w

If neither --once nor --interval is given, the tool runs exactly one pass and exits. Exit codes: 0 success, 2 bad --since, 3 auth error (run --auth), 4 any other failure.

Reference

calendar_sync.sync

main(argv: list | None = None) -> int

CLI entry point (wired as the evolutiondb-calendar-sync console script via calendar_sync.__main__:main). Parses arguments, loads the dotenv file, builds Config, and either runs the consent flow (--auth), a single pass, or the daemon loop. Installs SIGTERM/SIGINT handlers so the daemon finishes the current pass before exiting.

sync_once(cfg: Config, *, since_iso: str | None, dry_run: bool = False) -> dict

Runs one full pass over every calendar. For each calendar it reads the persisted sync token, calls the API (incrementally if a token exists, otherwise bootstrapping from since_iso), writes or deletes records, and persists the new nextSyncToken. On SyncTokenExpired (410) it clears the token and re-bootstraps. Returns the counter dict {"calendars", "events", "deleted", "skipped", "errors"}. With dry_run=True no store is opened and nothing is written.

parse_since(text: str) -> str

Converts a duration string (30s, 5m, 1h, 3d, 2w) into an RFC 3339 UTC timestamp (…Z) by subtracting it from now. Raises ValueError on anything it cannot parse.

Config()

Reads all environment variables listed under Environment variables into attributes (client_id, client_secret, token_cache, evosql_host, evosql_port, evosql_user, evosql_pass, evosql_db, user_id, store, poll_secs).

Two module-internal helpers are also covered by the test suite: _event_when(event) renders the start time as a human string (handling all-day date vs timed dateTime), and _build_event_record(event, calendar) shapes one event into the record dict, returning None for cancelled events.

calendar_sync.api

class CalendarClient(token_provider: Callable[[], str], user_agent="evolutiondb-calendar-sync")

Thin Google Calendar v3 REST client built on urllib. The bearer token is fetched lazily via token_provider on each request.

CalendarClient.list_calendars() -> list[dict]

Returns every calendar in the user's calendar list (owned, subscribed, holiday feeds, …), following nextPageToken pagination.

CalendarClient.list_events(calendar_id, *, sync_token=None, updated_min_iso=None) -> dict

Walks all events on one calendar (paginated, singleEvents=true, maxResults=250). With sync_token it does an incremental delta; otherwise updated_min_iso bootstraps a wall-clock floor (and sets showDeleted=true). Returns {"events": [...], "next_sync_token": str | None}.

The client retries transient transport errors (timeouts, SSL/connection resets, dropped keep-alives) with exponential backoff 1,2,4,8,16s, honours 429 Retry-After, transparently retries one 401 (token refresh), and retries Google 5xx (500/502/503/504) with backoff 1,2,4,8s. Per-request read timeout is 90s.

exception CalendarError(status, msg, url)

Raised for an HTTP error response the client could not recover from; carries .status.

exception SyncTokenExpired

Raised on 410 Gone — the sync token is older than 30 days and a full re-sync is required. The sync loop catches this to clear the token and re-bootstrap.

calendar_sync.auth

class GmailAuth(client_id, client_secret, cache_path)

OAuth 2.0 loopback-redirect ("Installed app") flow, shared verbatim with gmail-sync — only the requested scope differs (calendar.readonly). The class name and some log prefixes still read gmail-sync (see Notes).

GmailAuth.ensure_token(interactive: bool = False) -> str

Returns a valid access token. Uses the cached access token if still valid, else exchanges the cached refresh token, else (only when interactive=True, i.e. --auth) spins up a one-shot loopback HTTP server, opens the consent URL in a browser, and exchanges the auth code. Raises AuthError when no token is cached and interactive is false.

class TokenCache(path) · exception AuthError

TokenCache loads/saves the JSON token file ({refresh_token, access_token, expires_at, scope}) atomically, chmod 0600. AuthError wraps any consent or token-endpoint failure.

calendar_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

EvolutionDB backend over psycopg (autocommit). On construction it issues CREATE MEMORY STORE (ignored if it already exists) and auto-reconnects up to 3 times on a dropped connection. Records and per-calendar sync tokens are read/written through the EvolutionDB MEMORY SQL surface (MEMORY PUT INTO, MEMORY DELETE FROM, and a SELECT against __mem_<store>).

put_record(key, record) · delete_record(key)

UPSERT a JSON record under the namespace, or delete one (used when Google reports an event as cancelled/deleted).

get_sync_token(calendar_id) · set_sync_token(calendar_id, token) · clear_sync_token(calendar_id)

Read, persist, and drop the per-calendar nextSyncToken (stored under key gcal_state_<calendar_id>). clear_sync_token is called on a 410.

Notes & limitations

  • --since default is 29d, not 30d. The README's run examples show --since 30d, but the code defaults to 29d on purpose: 30d hits Google's exact-30-day ceiling for updatedMin + showDeleted=true and the API answers 410. If a re-sync bootstrap itself hits that ceiling, the connector reports an error per calendar and asks you to narrow --since below 30 days rather than crashing.
  • Description clip. The README describes the description as clipped to "1 kB"; the actual clip is the first 1000 characters (the _short helper), with a trailing " …[clipped]" marker.
  • Daemon interval source. Daemon mode is driven by the --interval flag (floored at 60s), not by the CALENDAR_POLL_INTERVAL env var; the env var is read into Config.poll_secs but the loop does not currently consult it.
  • Version string mismatch. pyproject.toml declares version 0.1.2 while calendar_sync.__version__ is "0.1.0".
  • Shared auth module. auth.py is reused from gmail-sync unchanged: the OAuth class is named GmailAuth and several stderr log lines / the callback page still say gmail-sync. This is cosmetic — the scope requested is Calendar's calendar.readonly.
  • Optional PII redaction. If evolutiondb-pii is importable, the fields fact, summary, description, location, organizer, and attendees are passed through protect_record before being stored, and any companion redaction records are written too. If the package is not installed, records are stored as-is.
  • SQL value escaping. The store builds SQL by string interpolation with single-quote doubling and control-character stripping (the _e helper); it does not use parameterized queries.
  • Privacy. The token can read events you were invited to on other people's calendars. Once an event lands in EvolutionDB, anyone with the matching MCP_USER_ID can recall it through search_memory. Use a dedicated Google account for sensitive calendars. See the EvolutionDB wiki for the wider memory/privacy model.
