Google Calendar connector

A one-way *-sync connector. It pulls Google Calendar events into EvolutionDB long-term memory. An assistant can then recall meetings weeks later. The events sit next to the chat and email memory that the sibling connectors write.

What it does

The package evolutiondb-calendar-sync reads events from every calendar your Google account can see. It writes one record per event into an EvolutionDB memory store with an UPSERT. It is strictly read-only against Google. The only OAuth scope it asks for is https://www.googleapis.com/auth/calendar.readonly. So it cannot create, edit, or delete events on your calendar.

Sync is incremental. Google sends back a nextSyncToken on each event-list call. The connector saves that token per calendar and sends it on the next run. The token is kept in the same memory store under a gcal_state_<calendar_id> key. This way only changed events come over the wire. A cancelled event is treated as a delete. The matching memory row is removed. Sometimes Google answers 410 Gone because the token is older than 30 days. Then the connector clears the stale token and starts over from the --since floor.

Each event is shaped into a record. The record holds these fields. It is written as JSON into the store.

  • fact is a one-line sentence in plain language. It is built from the title, time, calendar, and organizer. It is used for search recall.
  • summary is the event title. It falls back to (no title).
  • description is the event body. It is clipped to the first 1000 characters. See Notes.
  • location, then organizer as an email or display name, then attendees as a list of names
  • start and end hold an RFC 3339 timestamp for timed events. They hold a bare date for all-day events. all_day is a boolean.
  • status is one of confirmed, tentative, or cancelled. There are also html_link, created_at, and updated_at.
  • calendar_id, calendar_name, then source set to gcal, then kind set to event, and tags set to ["gcal", "event", <calendar_name>]

Each record is keyed gcal_event_<calendar_id>_<event_id>. So a re-sync of the same event overwrites the old row. It does not make a duplicate.

The memory store is reached over the PostgreSQL wire protocol. The default port is 5433. It uses the EvolutionDB MEMORY SQL surface. See the MCP server to learn how an assistant later reads these records back with search_memory.

Install

# From PyPI
pip install evolutiondb-calendar-sync

# Or editable, from this repo
pip install -e client/calendar-sync

It needs Python 3.9 or newer. It has two runtime dependencies. psycopg[binary]>=3.1 handles the EvolutionDB connection. evolutiondb-pii>=0.1.1 is optional and does PII redaction. See Notes. The Google API access uses only the Python standard library, such as urllib and http.server. There is no Google SDK dependency.

Configure

The connector reads its settings from environment variables. You can seed them from a .env file. A .env.example ships with the package. The loader only fills variables that are not already set in the environment. It uses os.environ.setdefault. So real environment values win over the file.

Google OAuth setup

Maybe you already set up evolutiondb-gmail-sync. Then you can reuse the same OAuth Desktop client. Just enable the Calendar API on the same Google Cloud project.

  1. Enable the Calendar API. Go to calendar-json.googleapis.com → Enable.
  2. APIs & Services → Credentials → reuse an existing Desktop OAuth client, or create one of type Desktop app.
  3. Copy the Client ID and Client secret into your environment as GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET.
  4. Make sure your Google account is listed under Test Users on the OAuth consent screen.

Usage

First-time auth

Run the consent flow once. It opens a browser tab to Google's consent screen. It catches the redirect on a loopback port. It stores the refresh token at the cache path. Later runs are silent.

evolutiondb-calendar-sync --auth

Running a sync

# One pass; bootstrap from a wall-clock floor (default --since is 29d).
evolutiondb-calendar-sync --once --since 30d

# Daemon mode: poll every 15 minutes (900 seconds).
evolutiondb-calendar-sync --interval 900

# Auth + scope check, no writes.
evolutiondb-calendar-sync --once --dry-run

Each pass prints a one-line JSON summary to stdout, for example:

{"ok": true, "calendars": 6, "events": 142, "deleted": 3, "skipped": 1, "errors": 0}

On failure it prints {"ok": false, "error": "…"}. It adds a "hint": "run with --auth" when the token is missing.

CLI flags

FlagDefaultMeaning
--auth, Run the consent flow, cache the token, and exit
--once, Run a single sync pass and exit
--interval N, Daemon mode. Repeat a pass every N seconds. The value is floored at 60.
--since DUR29dFirst-run wall-clock floor for updatedMin. The format looks like 30s, 5m, 1h, 3d, 2w.
--dry-run, Fetch and authenticate but write nothing to EvolutionDB
--env-file PATH.envPath to the dotenv file to load before reading config

If you give neither --once nor --interval, the tool runs exactly one pass and exits. The exit codes are simple. 0 means success. 2 means a bad --since. 3 means an auth error, so run --auth. 4 means any other failure.

Reference

calendar_sync.sync

main(argv: list | None = None) -> int

This is the CLI entry point. It is wired as the evolutiondb-calendar-sync console script through calendar_sync.__main__:main. It parses arguments, loads the dotenv file, and builds Config. Then it does one of three things. It runs the consent flow for --auth, runs a single pass, or runs the daemon loop. It installs SIGTERM and SIGINT handlers. So the daemon finishes the current pass before it exits.

sync_once(cfg: Config, *, since_iso: str | None, dry_run: bool = False) -> dict

This runs one full pass over every calendar. For each calendar it reads the saved sync token. Then it calls the API. It runs incrementally if a token exists. Otherwise it bootstraps from since_iso. It writes or deletes records and saves the new nextSyncToken. On a SyncTokenExpired error, which is a 410, it clears the token and starts over. It returns the counter dict {"calendars", "events", "deleted", "skipped", "errors"}. With dry_run=True no store is opened and nothing is written.

parse_since(text: str) -> str

This takes a duration string like 30s, 5m, 1h, 3d, or 2w. It turns the string into an RFC 3339 UTC timestamp that ends in …Z. It does this by subtracting the duration from now. It raises ValueError on anything it cannot parse.

Config()

This reads all the environment variables listed under Environment variables into attributes. The attributes are client_id, client_secret, token_cache, evosql_host, evosql_port, evosql_user, evosql_pass, evosql_db, user_id, store, and poll_secs.

The test suite also covers two helpers that live inside the module. _event_when(event) renders the start time as a human string. It handles an all-day date and a timed dateTime. _build_event_record(event, calendar) shapes one event into the record dict. It returns None for cancelled events.

calendar_sync.api

class CalendarClient(token_provider: Callable[[], str], user_agent="evolutiondb-calendar-sync")

This is a small Google Calendar v3 REST client built on urllib. The bearer token is fetched lazily through token_provider on each request.

CalendarClient.list_calendars() -> list[dict]

This returns every calendar in the user's calendar list. That includes owned calendars, subscribed calendars, holiday feeds, and more. It follows nextPageToken pagination.

CalendarClient.list_events(calendar_id, *, sync_token=None, updated_min_iso=None) -> dict

This walks all events on one calendar. It is paginated and uses singleEvents=true and maxResults=250. With sync_token it does an incremental delta. Otherwise updated_min_iso bootstraps a wall-clock floor and sets showDeleted=true. It returns {"events": [...], "next_sync_token": str | None}.

The client retries transient transport errors. These are timeouts, SSL or connection resets, and dropped keep-alives. It retries them with exponential backoff of 1,2,4,8,16s. It honours the 429 Retry-After header. It quietly retries one 401 by refreshing the token. It retries a Google 5xx response, such as 500/502/503/504, with backoff of 1,2,4,8s. The read timeout per request is 90s.

exception CalendarError(status, msg, url)

This is raised for an HTTP error response that the client could not recover from. It carries .status.

exception SyncTokenExpired

This is raised on 410 Gone. The sync token is older than 30 days. A full re-sync is needed. The sync loop catches this. It then clears the token and starts over.

calendar_sync.auth

class GmailAuth(client_id, client_secret, cache_path)

This is the OAuth 2.0 loopback-redirect flow, also called the "Installed app" flow. It is shared word for word with gmail-sync. Only the requested scope differs, which is calendar.readonly. The class name and some log prefixes still read gmail-sync. See Notes.

GmailAuth.ensure_token(interactive: bool = False) -> str

This returns a valid access token. It uses the cached access token if it is still valid. If not, it exchanges the cached refresh token. If that fails too, it does one more thing, but only when interactive=True, which means --auth. It spins up a one-shot loopback HTTP server, opens the consent URL in a browser, and exchanges the auth code. It raises AuthError when no token is cached and interactive is false.

class TokenCache(path) · exception AuthError

TokenCache loads and saves the JSON token file in an atomic way, with chmod 0600. The file holds {refresh_token, access_token, expires_at, scope}. AuthError wraps any consent or token-endpoint failure.

calendar_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

This is the EvolutionDB backend over psycopg in autocommit mode. On construction it issues CREATE MEMORY STORE. That is ignored if the store already exists. It auto-reconnects up to 3 times on a dropped connection. Records and per-calendar sync tokens are read and written through the EvolutionDB MEMORY SQL surface. That surface uses MEMORY PUT INTO, MEMORY DELETE FROM, and a SELECT against __mem_<store>.

put_record(key, record) · delete_record(key)

UPSERT a JSON record under the namespace, or delete one. The delete is used when Google reports an event as cancelled or deleted.

get_sync_token(calendar_id) · set_sync_token(calendar_id, token) · clear_sync_token(calendar_id)

Read, save, and drop the per-calendar nextSyncToken. It is stored under the key gcal_state_<calendar_id>. clear_sync_token is called on a 410.

Notes & limitations

  • The --since default is 29d, not 30d. The README run examples show --since 30d. But the code defaults to 29d on purpose. 30d hits Google's exact-30-day ceiling for updatedMin with showDeleted=true, and the API answers 410. If a re-sync bootstrap itself hits that ceiling, the connector does not crash. It reports an error per calendar and asks you to lower --since below 30 days.
  • Description clip. The README says the description is clipped to "1 kB". The real clip is the first 1000 characters. The _short helper does this and adds a trailing " …[clipped]" marker.
  • Daemon interval source. Daemon mode is driven by the --interval flag, which is floored at 60s. It is not driven by the CALENDAR_POLL_INTERVAL env var. The env var is read into Config.poll_secs, but the loop does not use it right now.
  • Version string mismatch. pyproject.toml declares version 0.1.2. But calendar_sync.__version__ is "0.1.0".
  • Shared auth module. auth.py is reused from gmail-sync without changes. The OAuth class is named GmailAuth. Several stderr log lines and the callback page still say gmail-sync. This is only cosmetic. The scope it asks for is Calendar's calendar.readonly.
  • Optional PII redaction. Maybe evolutiondb-pii can be imported. Then the fields fact, summary, description, location, organizer, and attendees are passed through protect_record before they are stored. Any companion redaction records are written too. If the package is not installed, records are stored as they are.
  • SQL value escaping. The store builds SQL by string interpolation. It doubles single quotes and strips control characters with the _e helper. It does not use parameterized queries.
  • Privacy. The token can read events you were invited to on other people's calendars. Once an event lands in EvolutionDB, anyone with the matching MCP_USER_ID can recall it through search_memory. Use a dedicated Google account for sensitive calendars. See the EvolutionDB wiki for the wider memory and privacy model.
