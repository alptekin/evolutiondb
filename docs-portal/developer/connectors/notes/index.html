Apple Notes connector

A macOS-only *-sync connector that reads your Apple Notes through the JavaScript-for-Automation bridge and stores each note in EvolutionDB long-term memory, so an AI assistant can recall it later through search_memory.

What it does

The package evolutiondb-notes-sync talks to Notes.app over macOS's JavaScript-for-Automation (JXA) bridge, converts each note's HTML body to plain text, and persists it into an EvolutionDB MCP memory store. The MCP server then surfaces those notes through search_memory, so an assistant can answer questions about something you jotted down weeks ago.

It deliberately reads notes through JXA rather than parsing NoteStore.sqlite directly. As the source comments explain, Apple encodes each note body as gzip plus a private "NotesBody" protobuf inside ZICCLOUDSYNCINGOBJECT.ZDATA, and that format shifts across macOS releases. JXA returns the already-rendered HTML body and is stable across versions; the trade-off is that the user must grant Automation permission (System Settings → Privacy & Security → Automation → Terminal/iTerm → Notes), which macOS prompts for on first run.

Concretely, each pass:

  • Spawns /usr/bin/osascript -l JavaScript with an embedded JXA script that iterates Application('Notes').notes() and prints one JSON object per note (id, name, body, folder, modified, created).
  • Converts each note's HTML body to plain text (strips tags, decodes entities, collapses whitespace).
  • Skips empty notes (no text and an untitled name), counted as skipped.
  • Writes one record per note, keyed note_<note-id> (slashes in the id replaced with underscores). The key is deterministic, so re-syncing the same note overwrites rather than duplicates.
  • Advances a single ISO-8601 watermark (the highest modified_at seen) so the next pass only imports notes modified more recently.

Records carry the note title, folder, note id, and created/modified timestamps so downstream search_memory queries have context.

What is not stored

  • Notes whose body is empty and whose name is blank (the record builder returns None; counted as skipped).
  • Attachments, images, tables, and rich formatting, the body is reduced to plain text by a regex-based _html_to_text that only understands simple tags (h1–h6, p, div, br, li).
  • Note text beyond ~3000 characters, the body is clipped and the record is tagged truncated (see Record shape).

Install

The distribution name is evolutiondb-notes-sync; the importable package is notes_sync. It requires Python 3.9+ (macOS only) and depends on psycopg[binary]>=3.1 and evolutiondb-pii>=0.1.1.

# From PyPI
pip install evolutiondb-notes-sync

# Or editable, from this repo
pip install -e client/notes-sync

Installing exposes a single console script, evolutiondb-notes-sync (entry point notes_sync.__main__:main).

There is no bundled README in the package directory; this page is written from the source itself (notes_sync/sync.py, notes_sync/state.py) and the packaging metadata.

Configure (env vars / auth)

Configuration is entirely through environment variables. The CLI loads a dotenv file before reading them, by default .env in the working directory, overridable with --env-file. Existing process environment values win: the loader (_load_dotenv) uses os.environ.setdefault, so it never overwrites what is already set, and it skips blank values, comments, and lines without an =.

macOS Automation permission

There is no API key or OAuth flow. The only "auth" is the macOS Automation permission that lets the running terminal control Notes.app. On the first invocation macOS shows the Automation consent prompt; once granted (System Settings → Privacy & Security → Automation → your terminal → Notes), subsequent runs are silent. If permission is denied or unavailable, individual notes surface as per-note errors and are logged to stderr.

If /usr/bin/osascript is not present (i.e. you are not on macOS), the connector prints [notes-sync] /usr/bin/osascript not found - Notes sync is macOS-only. and yields nothing.

Usage

The CLI has five flags: --once, --interval <seconds>, --since <iso> (default empty), --dry-run, and --env-file <path> (default .env).

# First run - import everything, then exit.
evolutiondb-notes-sync --once

# Only import notes modified on or after a given ISO timestamp.
evolutiondb-notes-sync --once --since 2026-01-01T00:00:00Z

# Daemon mode: pass, then sleep --interval seconds, repeat.
evolutiondb-notes-sync --interval 600

# Read Notes and build records, but make no DB writes.
evolutiondb-notes-sync --once --dry-run

If --once is given, or no --interval is supplied, exactly one pass runs. In daemon mode the interval is clamped to a minimum of 60 seconds, and SIGTERM/SIGINT request a graceful stop after the current pass finishes (the loop checks a _stop flag once per second of sleep).

Each pass prints a one-line JSON summary to stdout. On success:

{"ok": true, "notes": 137, "skipped": 4, "errors": 0}

On failure the line is {"ok": false, "error": "<message>"} and the process exits with code 4. (The figures above are illustrative, not guaranteed counts.)

--since semantics

--since is an optional ISO-8601 floor used only as the lower bound for a first run (e.g. 2026-01-01T00:00:00Z). It is not a relative duration, the value is passed through verbatim and compared as a string against each note's modified_at. An empty value (the default) means "import everything". Once the store holds a watermark, the watermark takes precedence over --since: the effective floor is the stored watermark if present, otherwise the --since value.

Record shape

Each note is stored as a JSON record under the key note_<note-id>. Fields written by _build_record:

{
  "fact":        "Note \"Groceries\": milk, eggs, bread …",  // title + first 240 text chars
  "source":      "notes",
  "kind":        "note",
  "title":       "Groceries",             // "(untitled)" when the note has no name
  "text":        "milk, eggs, bread",     // plain text, clipped to 3000 chars
  "folder":      "Notes",
  "note_id":     "x-coredata://…/ICNote/p42",
  "modified_at": "2026-05-13T08:33:20Z",  // falls back to created when modified is absent
  "created_at":  "2026-05-01T11:00:00Z",
  "truncated":   false,
  "tags":        ["notes", "apple-notes"]  // + "truncated" when text was clipped
}

The note text is clipped to _MAX_TEXT_CHARS = 3000. The source explains the headroom: EvolutionDB caps a single SQL statement at 8KB, and the MEMORY PUT wrapper plus JSON envelope plus single-quote doubling can roughly double the on-wire byte count, so 3000 chars leaves room for a note full of quotes or multi-byte UTF-8. When text is clipped, truncated is set true and a "truncated" tag is appended.

PII protection (optional)

If evolutiondb-pii exposes protect_record, the connector runs each record through it before writing, masking/encrypting the fact, title, and text fields (key prefix notes_pii_<id>) and optionally emitting companion records that are written alongside the main one. If the import fails, _protect is a transparent passthrough, the record is stored unchanged. Structural metadata (folder, id, timestamps) is always stored in the clear.

Reference

The package is four small modules: notes_sync.__init__ (version, docstring), notes_sync.__main__ (the script shim that calls main), notes_sync.sync (config, JXA, record building, sync loop, CLI), and notes_sync.state (the EvolutionDB memory backend). There is no separate API-client module, the "client" is the embedded JXA script driven by osascript.

notes_sync.sync

def main(argv: Optional[List[str]] = None) -> int

CLI entry point (the evolutiondb-notes-sync console script). Parses arguments, loads the dotenv file, builds Config, installs signal handlers, then runs one pass or loops on --interval. Returns a process exit code: 4 if a pass raises, otherwise 0.

def sync_once(cfg: Config, *, since_iso: Optional[str], dry_run: bool = False) -> Dict[str, int]

Runs a single sync pass. Unless dry_run, constructs a MemoryStore and reads its watermark; the effective floor is watermark or since_iso. Iterates every note via iter_notes, skips empty notes and notes at or below the floor, writes the rest (plus any PII companion records), advances the watermark to the highest modified_at seen, and returns the counter dict (notes, skipped, errors). A single failing note is logged to stderr and counted as errors without aborting the pass.

def iter_notes() -> Iterable[Dict]

Spawns /usr/bin/osascript -l JavaScript running the embedded JXA, and streams one JSON note per line. Because JXA's console.log writes to stderr, stdout and stderr are merged (stderr=STDOUT), otherwise the stderr pipe would fill (~64KB) on the first oversized note and osascript would block forever. Non-JSON lines are ignored; lines with an error key are logged and skipped. On a missing osascript (non-macOS) it prints a notice and returns nothing.

def _html_to_text(body_html: str) -> str

Cheap regex HTML→text conversion: turns block tags (br/p/div/h1–h6/li) into newlines, strips all other tags, decodes HTML entities, and collapses whitespace. Not a full HTML parser, adequate for the simple markup Apple Notes emits.

class Config

Reads all environment variables into typed attributes (evosql_host, evosql_port, evosql_user, evosql_pass, evosql_db, user_id, store). Constructed after the dotenv file is loaded.

Internal helpers: _build_record(n) assembles the record dict shown above (returns None for an empty note); _note_key(n) builds the deterministic note_<id> key; _protect(rec, main_key) applies optional PII masking and returns (record, companions); _load_dotenv(path) populates the environment without overwriting existing values; _install_signal_handlers() wires SIGTERM/SIGINT to a graceful-stop flag.

notes_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

The EvolutionDB memory backend. Connects via psycopg (autocommit) and best-effort issues CREATE MEMORY STORE <store> on init. All operations run through a reconnect-with-backoff wrapper (_retry, 3 attempts, 0.5s × attempt backoff) that recovers from dropped connections (OperationalError/InterfaceError). The module docstring notes it is a mirror of the sibling connector packages, so each connector stays dependency-free of the others.

MemoryStore.get_watermark_iso() -> Optional[str]

Reads the stored ISO watermark (key notes_state_latest, the modified_at field of its JSON value) for the configured namespace, or None if not yet seen.

MemoryStore.set_watermark_iso(iso: str) -> None

Writes the watermark with MEMORY PUT INTO <store>, storing {"modified_at": iso, "saved_at": <epoch>}.

MemoryStore.put_record(key: str, record: Dict) -> None

Serializes the record to JSON (ensure_ascii=False) and writes it under the namespace. It first issues a DELETE for the same key, then MEMORY PUT INTO <store>: per the source comment, MEMORY PUT upserts small payloads but reports a spurious "duplicate key" error for larger ones, and the cheap pre-delete sidesteps that path. On failure it logs the key and value length to stderr and re-raises.

Notes and limitations

  • macOS only. The connector shells out to /usr/bin/osascript and the Notes.app JXA bridge. On any other platform it prints a notice and does nothing.
  • Version mismatch in metadata. The packaging metadata (pyproject.toml and PKG-INFO) declares version 0.1.2, while the in-package __version__ in notes_sync/__init__.py is still 0.1.0. Marked Development Status 4 - Beta.
  • Read-only. There is no code path that writes back to Notes.app, the connector only reads.
  • Plain text only. Bodies are flattened by a regex, not a real HTML parser; attachments, images, tables, and checklists are not preserved. Long notes are clipped at 3000 characters and tagged truncated.
  • String-comparison watermark. The incremental floor compares modified_at values as strings, which is only correct because Notes timestamps are emitted as fixed-width ISO-8601 UTC (NSISO8601DateFormatWithInternetDateTime). A note with no modification date falls back to its creation date.
  • SQL built by string interpolation. state.py escapes values with a local _e() helper (newline/tab stripping + single-quote doubling) and interpolates them into the MEMORY PUT/SELECT/DELETE statements rather than using parameter binding.
  • Privacy. Notes can hold highly personal data. The optional evolutiondb-pii integration masks fact/title/text, but if that package or a public key is absent the records are stored in the clear, and anyone with the matching MCP_USER_ID namespace can recall them via search_memory.
  • No tests. The package ships no test suite (no tests/ directory in the distribution).
