Outlook connector

Pull your Microsoft Outlook / Microsoft 365 mail, full body, not just metadata, into EvolutionDB long-term memory, so the MCP server can recall it weeks later through search_memory.

What it does

The evolutiondb-outlook-sync package authenticates against the Microsoft identity platform with the OAuth 2.0 device-code flow, then walks Microsoft Graph's /me/messages (and the well-known SentItems folder) and writes each email into an EvolutionDB memory store over the PostgreSQL wire protocol. An assistant connected through the MCP server can then answer questions like "did the procurement team ever close the loop on that Q3 quote?" without the message still being in the inbox.

Each email becomes one record, keyed outlook_msg_<graph-id>, stored as a JSON value. The connector captures subject, from / to / cc, a short snippet (bodyPreview), the full message body stripped of HTML, the resolved folder name, categories, importance, read state, attachment flag, the received timestamp, and the Outlook web link. Bodies are clipped at 3 000 characters.

# Record field        Source
fact                   one-line "Email from <sender>: <subject> - <preview>"
subject                Graph subject (falls back to "(no subject)")
from / to / cc         flattened "Name <addr>" header strings
snippet                bodyPreview, clipped to ~240 chars
body                   full text, HTML-stripped, capped at 3 000 chars
folder                 parentFolderId resolved to its display name
categories             Graph categories
importance             "low" | "normal" | "high"
is_read                Graph isRead
has_attachments        Graph hasAttachments
received_at            receivedDateTime (ISO 8601, also the watermark)
web_link               direct link to the Outlook web UI
tags                   outlook, email, folder + categories, plus
                       important / unread / attachment / truncated
info
Tag-based recall lets a downstream search_memory(tag="important") or search_memory(tag="unread") answer "what should I read first" without scanning every row. A high-importance message gets the important tag; an unread one gets unread; one carrying an attachment gets attachment; a body that hit the 3 000-char cap gets truncated.

A single watermark on the highest receivedDateTime seen (stored under the key outlook_state_latest) lets the next pass filter Graph with $filter=receivedDateTime gt <watermark> and skip everything older.

Install

The PyPI distribution name is evolutiondb-outlook-sync; the importable module is outlook_sync. It requires Python 3.9+ and depends on psycopg[binary]>=3.1 and evolutiondb-pii>=0.1.1. The Microsoft Graph and OAuth code is built entirely on the standard-library urllib, there is no MSAL or requests dependency.

# from PyPI
pip install evolutiondb-outlook-sync

# or editable, from the EvolutionDB repo
pip install -e client/outlook-sync

Installation registers a single console script, evolutiondb-outlook-sync, which maps to outlook_sync.__main__:main. You can also run the module directly with python -m outlook_sync. Both resolve to the same main() in outlook_sync.sync.

Configure

Microsoft Graph requires an Entra ID app registration in your own Azure tenant. Create a registration in Azure Portal → Microsoft Entra ID → App registrations, enable "Allow public client flows" under Authentication (the device-code flow uses no redirect URI), and add the delegated Microsoft Graph permissions Mail.Read, User.Read, and offline_access. Copy the Application (client) ID into AZURE_CLIENT_ID; use the Directory (tenant) ID for AZURE_TENANT_ID, or leave it common to allow personal Microsoft accounts.

Configuration comes from environment variables. On startup the CLI loads a dotenv file (default .env in the working directory, overridable with --env-file; a missing file is ignored). The loader uses setdefault, so a value already set in the environment is never overwritten, and blank values in the file are skipped.

# --- Microsoft / Azure ---
AZURE_CLIENT_ID=00000000-0000-0000-0000-000000000000   # Application (client) ID - required
AZURE_TENANT_ID=common                                 # tenant ID, or "common" for personal accounts

# --- EvolutionDB (PG wire protocol) ---
EVOSQL_HOST=127.0.0.1
EVOSQL_PORT=5433
EVOSQL_USER=admin
EVOSQL_PASSWORD=admin
EVOSQL_DATABASE=evosql

# --- MCP correlation / runtime ---
MCP_USER_ID=default_user          # namespace shared with the MCP server
OUTLOOK_MEMORY_STORE=mcp_mem      # memory store to write into
OUTLOOK_TOKEN_CACHE=~/.evosql/outlook_token.json   # refresh-token cache (mode 0600)
OUTLOOK_POLL_INTERVAL=900        # default --interval seconds

Variable reference, as read by the Config class and the auth module:

# var                   default                          meaning
AZURE_CLIENT_ID          (none, required)                 Application (client) ID from Azure
AZURE_TENANT_ID          common                           Directory (tenant) ID; "common" = personal accounts
OUTLOOK_TOKEN_CACHE      ~/.evosql/outlook_token.json     refresh-token cache location
OUTLOOK_MEMORY_STORE     mcp_mem                          EvolutionDB memory store name
OUTLOOK_POLL_INTERVAL    900                              default daemon poll interval, seconds
MCP_USER_ID              default_user                     memory namespace (mem_namespace column)
EVOSQL_HOST              127.0.0.1                        EvolutionDB host
EVOSQL_PORT              5433                             EvolutionDB PG-wire port
EVOSQL_USER              admin                            EvolutionDB user
EVOSQL_PASSWORD          admin                            EvolutionDB password
EVOSQL_DATABASE          evosql                           target database
EVOSQL_OUTLOOK_SEND      (unset = read-only)              opt-in: 1/true/yes/on adds the Mail.Send scope
warning
AZURE_CLIENT_ID is mandatory. sync_once raises a RuntimeError if it is empty, and the OutlookAuth constructor raises AuthError. By default the connector requests only Mail.Read, User.Read, and offline_access, it cannot send mail or modify the mailbox. Setting EVOSQL_OUTLOOK_SEND to a truthy value adds the Mail.Send scope at consent time, used only by the outbox action loop's approve_send (ADR-004); the default stays read-only.

First-time auth

The first run prints a short device code and the https://microsoft.com/devicelogin URL to stderr. Open the URL in any browser: including on a different machine, which is handy for SSH / headless hosts: sign in, type the code, and approve consent. The token is then cached at OUTLOOK_TOKEN_CACHE (default ~/.evosql/outlook_token.json, written with mode 0600); subsequent runs use the refresh token silently. Microsoft rotates refresh tokens on each refresh, so the new one is persisted every time.

evolutiondb-outlook-sync --auth
# [outlook-sync] open https://microsoft.com/devicelogin and enter the code:
#
#     ABCD-EFGH
#

Usage

One flag runs the consent flow; the rest drive the sync, with a lookback window, an optional folder filter, and a dry-run switch:

# Run the device-code consent flow, cache the refresh token, exit.
evolutiondb-outlook-sync --auth

# One pass; bootstrap from the last 7 days, then exit.
evolutiondb-outlook-sync --once --since 7d

# Restrict the pull to a single folder by its display name.
evolutiondb-outlook-sync --once --since 30d --folder Inbox

# Daemon mode - re-run every N seconds (floored at 60).
evolutiondb-outlook-sync --interval 900

# Fetch from Graph but write nothing to the database.
evolutiondb-outlook-sync --once --since 1d --dry-run

# Point at an explicit dotenv file.
evolutiondb-outlook-sync --once --env-file ~/secrets/outlook.env

Flags accepted by main():

# --auth              run the device-code consent flow, persist the refresh token, exit
# --once              run a single pass and exit
# --interval SECONDS  daemon mode; loops until SIGINT/SIGTERM (floored at 60s)
# --since SPEC        first-run lookback; SPEC is N + s|m|h|d|w (default 7d)
# --folder NAME       restrict to a single mail folder by display name
# --dry-run           fetch from Graph but do not open or write the DB
# --env-file PATH     dotenv file to load (default .env; ignored if missing)

With neither --once nor --interval, the connector runs a single pass (the same as --once). Each pass prints a one-line JSON summary to stdout:

{"ok": true, "messages": 42, "skipped": 9, "errors": 0}

On a fatal error the pass instead prints {"ok": false, "error": "<message>"}; an auth failure also includes "hint": "run with --auth". In daemon mode, SIGINT and SIGTERM request a graceful stop, the current pass finishes before the loop exits.

Exit codes: 0 success, 2 a bad --since spec, 3 an auth error, 4 a pass raised an exception.

Reference

outlook_sync.sync, CLI and sync loop

main(argv: list | None = None) -> int

Console-script entry point (registered as evolutiondb-outlook-sync). Loads the dotenv file, builds a Config, and either runs the --auth consent flow, runs a single pass, or loops on the interval. Returns the process exit code.

sync_once(cfg: Config, *, since_iso: str | None, folder_name: str | None = None, dry_run: bool = False) -> dict[str, int]

Runs a single pass. Builds an OutlookAuth and an OutlookClient, resolves folder IDs to display names once, then makes two passes: a cross-folder /me/messages pass and an explicit SentItems pass (skipped when --folder is given). An in-process seen-set plus the store's has_message check deduplicate. Each surviving message is built into a record, optionally PII-protected, and written unless dry_run. Returns the counters dict {"messages", "skipped", "errors"}. Raises RuntimeError if AZURE_CLIENT_ID is unset or the named folder is not found.

parse_since(text: str) -> str

Converts a window spec like "1h", "7d", or "2w" (units s|m|h|d|w) into a UTC ISO-8601 timestamp ending in Z, used as the first-run receivedDateTime gt floor. Raises ValueError on anything it can't parse.

class Config

Reads all environment variables into typed attributes (client_id, tenant_id, token_cache, evosql_host/port/user/pass/db, user_id, store, poll_secs).

outlook_sync.auth, device-code OAuth

Implements the Microsoft identity platform device-code flow over urllib only. Tokens are cached as JSON at the configured path with mode 0600, written atomically via a .tmp + rename.

class OutlookAuth(client_id: str, tenant: str, cache_path: str)

Holds the client ID, tenant (defaults to common), and a TokenCache. Raises AuthError immediately if client_id is empty.

OutlookAuth.ensure_token(interactive: bool = False) -> str

Returns a valid access token: a cached one if it is still valid for >60s, else a silent refresh-token exchange. With interactive=True it falls back to the device-code flow on a failed refresh; with interactive=False (the sync path) it raises AuthError instead, prompting "run with --auth".

class TokenCache(path: Path)

load() reads the cache JSON (returning {} on missing or corrupt files); save(data) writes it atomically with 0600 permissions.

class AuthError(Exception)

Raised on any consent / token failure. The Microsoft device-code endpoint's normal authorization_pending and slow_down responses are caught inside the poll loop and not surfaced as errors.

outlook_sync.api, Microsoft Graph client

A thin Graph client over urllib (base https://graph.microsoft.com/v1.0). Every request sends Prefer: outlook.body-content-type="text" so Graph strips HTML for us, and requests an explicit $select projection per message. A 401 triggers one silent token re-fetch; a 429 honours Retry-After; transient transport failures (read timeout, SSL EOF, reset, server hang-up) are retried with exponential backoff (1, 2, 4, 8, 16 s). Per-request socket timeout is 90 s.

class OutlookClient(token_provider: Callable[[], str], user_agent="evolutiondb-outlook-sync")

Takes a callable that yields a bearer token (wired to OutlookAuth.ensure_token).

OutlookClient.list_messages(*, received_after_iso=None, top=50, folder_filter=None) -> Iterator[dict]

Yields message dicts from /me/messages in ascending receivedDateTime order, optionally filtered by an after-timestamp and / or a parentFolderId, paginated 50 per page.

OutlookClient.list_sent_messages(*, received_after_iso=None, top=50) -> Iterator[dict]

Yields from the well-known /me/mailFolders/sentitems/messages endpoint. Cross-folder /me/messages under-reports Sent Items, so this explicit pass recovers the full outbound history for "did I reply?" recall.

OutlookClient.list_folders() -> list[dict]

Returns a flat list of mail folders from /me/mailFolders, used to resolve parentFolderId to a display name.

OutlookClient.get_user() -> dict

Returns the signed-in user resource from /me.

addr_text(addr: dict | None) -> str
recipient_list(items: list[dict] | None) -> str

Flatten Graph's {"emailAddress": {"name", "address"}} shape into a single "Name <addr>" string, and join a recipient array into a comma-separated string, respectively.

class GraphError(Exception)

Carries the HTTP status, message, and URL. Folder-fetch and SentItems-pass GraphErrors are caught in sync_once, logged to stderr, and counted rather than aborting the pass.

outlook_sync.state, EvolutionDB memory backend

Writes records and the watermark over the PG wire protocol via psycopg. On construction it opens an autocommit connection and issues CREATE MEMORY STORE <store> (ignoring the error if it already exists), and reconnects up to three times with backoff on a dropped connection.

class MemoryStore(host, port, user, password, database, store, namespace)

Opens the connection and ensures the target memory store exists.

MemoryStore.put_record(key: str, record: dict) -> None

Serializes record to JSON and UPSERTs it: a DELETE on the key first (a no-op when nothing matches), then MEMORY PUT INTO <store> VALUES (<namespace>, <key>, <value>).

MemoryStore.has_message(msg_id: str) -> bool

Returns whether outlook_msg_<msg_id> already exists in the store, used to skip messages already written on an earlier pass.

MemoryStore.get_watermark_iso() -> str | None
MemoryStore.set_watermark_iso(iso: str) -> None

Read and write the single highest-receivedDateTime watermark, stored under the key outlook_state_latest; get_watermark_iso queries the underlying __mem_<store> table directly.

Notes & limitations

  • Read-only by default: the requested scopes are Mail.Read, User.Read, and offline_access. The connector cannot send mail, mark messages, or modify the mailbox unless EVOSQL_OUTLOOK_SEND is set to add Mail.Send for the outbox action loop (ADR-004).
  • Body cap: bodies are clipped at 3 000 characters to stay under EvolutionDB's current per-row size limit; clipped rows carry the truncated tag. The in-source comment notes the cap will lift once Task #233 (TOAST overflow) lands.
  • Append-only: messages are never updated after first write, changes to read state or folder membership on the Outlook side are not back-propagated. A row that fails to write is logged and skipped; one bad row never aborts the pass.
  • Watermark blind spot: the watermark is receivedDateTime, so a message backdated by a sender to a time earlier than the current watermark won't be picked up on the next pass. An occasional --since 30d re-run catches stragglers via the has_message dedup check.
  • PII protection: if evolutiondb-pii is importable, the fact, subject, from, to, cc, snippet, and body fields are passed through its protect_record() before storage; otherwise it is a transparent passthrough.
  • Privacy: records land in the same store the MCP server reads from. Anyone with access to that database and the matching MCP_USER_ID can recall the content through search_memory.
  • Version: the packaged distribution is 0.1.3 (Development Status: 4 - Beta); note outlook_sync.__version__ in the module is "0.1.0".

See also the MCP server docs, and the project wiki at wiki.evolutiondb.org.
