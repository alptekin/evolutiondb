Outlook connector

Pull your Microsoft Outlook and Microsoft 365 mail into EvolutionDB long-term memory. It stores the full body, not just metadata. The MCP server can then recall it weeks later through search_memory.

What it does

The evolutiondb-outlook-sync package signs in to the Microsoft identity platform with the OAuth 2.0 device-code flow. It then walks Microsoft Graph's /me/messages and the well-known SentItems folder. It writes each email into an EvolutionDB memory store over the PostgreSQL wire protocol. An assistant connected through the MCP server can then answer questions like "did the procurement team ever close the loop on that Q3 quote?". The message no longer needs to be in the inbox.

Each email becomes one record. The record key is outlook_msg_<graph-id> and the value is JSON. The connector captures the subject, the from, to and cc headers, a short snippet from bodyPreview, the full message body with HTML stripped, the resolved folder name, categories, importance, read state, the attachment flag, the received timestamp, and the Outlook web link. Bodies are clipped at 3 000 characters.

# Record field        Source
fact                   one-line "Email from <sender>: <subject> - <preview>"
subject                Graph subject (falls back to "(no subject)")
from / to / cc         flattened "Name <addr>" header strings
snippet                bodyPreview, clipped to ~240 chars
body                   full text, HTML-stripped, capped at 3 000 chars
folder                 parentFolderId resolved to its display name
categories             Graph categories
importance             "low" | "normal" | "high"
is_read                Graph isRead
has_attachments        Graph hasAttachments
received_at            receivedDateTime (ISO 8601, also the watermark)
web_link               direct link to the Outlook web UI
tags                   outlook, email, folder + categories, plus
                       important / unread / attachment / truncated
info
Tags make recall faster. A downstream search_memory(tag="important") or search_memory(tag="unread") can answer "what should I read first" without scanning every row. A high-importance message gets the important tag. An unread message gets unread. A message with an attachment gets attachment. A body that hit the 3 000-char cap gets truncated.

The connector keeps one watermark. It holds the highest receivedDateTime seen so far. It is stored under the key outlook_state_latest. The next pass uses it to filter Graph with $filter=receivedDateTime gt <watermark> and skip everything older.

Install

The PyPI distribution name is evolutiondb-outlook-sync. The importable module is outlook_sync. It needs Python 3.9+ and depends on psycopg[binary]>=3.1 and evolutiondb-pii>=0.1.1. The Microsoft Graph and OAuth code is built entirely on the standard-library urllib. There is no MSAL or requests dependency.

# from PyPI
pip install evolutiondb-outlook-sync

# or editable, from the EvolutionDB repo
pip install -e client/outlook-sync

Installation registers a single console script named evolutiondb-outlook-sync. It maps to outlook_sync.__main__:main. You can also run the module directly with python -m outlook_sync. Both resolve to the same main() in outlook_sync.sync.

Configure

Microsoft Graph needs an Entra ID app registration in your own Azure tenant. Create a registration in Azure Portal → Microsoft Entra ID → App registrations. Enable "Allow public client flows" under Authentication. The device-code flow uses no redirect URI. Then add the delegated Microsoft Graph permissions Mail.Read, User.Read, and offline_access. Copy the Application (client) ID into AZURE_CLIENT_ID. Use the Directory (tenant) ID for AZURE_TENANT_ID, or leave it common to allow personal Microsoft accounts.

Configuration comes from environment variables. On startup the CLI loads a dotenv file. The default is .env in the working directory. You can change it with --env-file. A missing file is ignored. The loader uses setdefault. A value already set in the environment is never overwritten. Blank values in the file are skipped.

# --- Microsoft / Azure ---
AZURE_CLIENT_ID=00000000-0000-0000-0000-000000000000   # Application (client) ID - required
AZURE_TENANT_ID=common                                 # tenant ID, or "common" for personal accounts

# --- EvolutionDB (PG wire protocol) ---
EVOSQL_HOST=127.0.0.1
EVOSQL_PORT=5433
EVOSQL_USER=admin
EVOSQL_PASSWORD=admin
EVOSQL_DATABASE=evosql

# --- MCP correlation / runtime ---
MCP_USER_ID=default_user          # namespace shared with the MCP server
OUTLOOK_MEMORY_STORE=mcp_mem      # memory store to write into
OUTLOOK_TOKEN_CACHE=~/.evosql/outlook_token.json   # refresh-token cache (mode 0600)
OUTLOOK_POLL_INTERVAL=900        # default --interval seconds

The table below lists all variables as read by the Config class and the auth module.

# var                   default                          meaning
AZURE_CLIENT_ID          (none, required)                 Application (client) ID from Azure
AZURE_TENANT_ID          common                           Directory (tenant) ID; "common" = personal accounts
OUTLOOK_TOKEN_CACHE      ~/.evosql/outlook_token.json     refresh-token cache location
OUTLOOK_MEMORY_STORE     mcp_mem                          EvolutionDB memory store name
OUTLOOK_POLL_INTERVAL    900                              default daemon poll interval, seconds
MCP_USER_ID              default_user                     memory namespace (mem_namespace column)
EVOSQL_HOST              127.0.0.1                        EvolutionDB host
EVOSQL_PORT              5433                             EvolutionDB PG-wire port
EVOSQL_USER              admin                            EvolutionDB user
EVOSQL_PASSWORD          admin                            EvolutionDB password
EVOSQL_DATABASE          evosql                           target database
EVOSQL_OUTLOOK_SEND      (unset = read-only)              opt-in: 1/true/yes/on adds the Mail.Send scope
warning
AZURE_CLIENT_ID is required. sync_once raises a RuntimeError if it is empty. The OutlookAuth constructor raises AuthError for the same reason. By default the connector requests only Mail.Read, User.Read, and offline_access. It cannot send mail or change the mailbox. Set EVOSQL_OUTLOOK_SEND to a truthy value to add the Mail.Send scope at consent time. Only the outbox action loop's approve_send uses it (ADR-004). The default stays read-only.

First-time auth

The first run prints a short device code and the https://microsoft.com/devicelogin URL to stderr. Open the URL in any browser. You can use a different machine. This helps with SSH and headless hosts. Sign in, type the code, and approve consent. The token is then cached at OUTLOOK_TOKEN_CACHE. The default path is ~/.evosql/outlook_token.json. The file is written with mode 0600. Later runs use the refresh token silently. Microsoft rotates the refresh token on each refresh, so the new one is saved every time.

evolutiondb-outlook-sync --auth
# [outlook-sync] open https://microsoft.com/devicelogin and enter the code:
#
#     ABCD-EFGH
#

Usage

One flag runs the consent flow. The rest drive the sync. You can set a lookback window, an optional folder filter, and a dry-run switch.

# Run the device-code consent flow, cache the refresh token, exit.
evolutiondb-outlook-sync --auth

# One pass; bootstrap from the last 7 days, then exit.
evolutiondb-outlook-sync --once --since 7d

# Restrict the pull to a single folder by its display name.
evolutiondb-outlook-sync --once --since 30d --folder Inbox

# Daemon mode - re-run every N seconds (floored at 60).
evolutiondb-outlook-sync --interval 900

# Fetch from Graph but write nothing to the database.
evolutiondb-outlook-sync --once --since 1d --dry-run

# Point at an explicit dotenv file.
evolutiondb-outlook-sync --once --env-file ~/secrets/outlook.env

The table below lists the flags that main() accepts.

# --auth              run the device-code consent flow, persist the refresh token, exit
# --once              run a single pass and exit
# --interval SECONDS  daemon mode; loops until SIGINT/SIGTERM (floored at 60s)
# --since SPEC        first-run lookback; SPEC is N + s|m|h|d|w (default 7d)
# --folder NAME       restrict to a single mail folder by display name
# --dry-run           fetch from Graph but do not open or write the DB
# --env-file PATH     dotenv file to load (default .env; ignored if missing)

If you give neither --once nor --interval, the connector runs a single pass. This is the same as --once. Each pass prints a one-line JSON summary to stdout.

{"ok": true, "messages": 42, "skipped": 9, "errors": 0}

On a fatal error the pass prints {"ok": false, "error": "<message>"} instead. An auth failure also includes "hint": "run with --auth". In daemon mode, SIGINT and SIGTERM request a graceful stop. The current pass finishes before the loop exits.

The exit codes are simple. 0 means success. 2 means a bad --since spec. 3 means an auth error. 4 means a pass raised an exception.

Reference

outlook_sync.sync, CLI and sync loop

main(argv: list | None = None) -> int

The console-script entry point is registered as evolutiondb-outlook-sync. It loads the dotenv file and builds a Config. It then either runs the --auth consent flow, runs a single pass, or loops on the interval. It returns the process exit code.

sync_once(cfg: Config, *, since_iso: str | None, folder_name: str | None = None, dry_run: bool = False) -> dict[str, int]

Runs a single pass. It builds an OutlookAuth and an OutlookClient. It resolves folder IDs to display names once. It then makes two passes. The first is a cross-folder /me/messages pass. The second is an explicit SentItems pass. The second pass is skipped when --folder is given. An in-process seen-set and the store's has_message check remove duplicates. Each surviving message is built into a record and optionally PII-protected. It is written unless dry_run is set. The function returns the counters dict {"messages", "skipped", "errors"}. It raises RuntimeError if AZURE_CLIENT_ID is unset or the named folder is not found.

parse_since(text: str) -> str

Converts a window spec like "1h", "7d", or "2w" into a UTC ISO-8601 timestamp ending in Z. The units are s|m|h|d|w. This timestamp is the first-run receivedDateTime gt floor. It raises ValueError on anything it cannot parse.

class Config

Reads all environment variables into typed attributes. These are client_id, tenant_id, token_cache, evosql_host/port/user/pass/db, user_id, store, and poll_secs.

outlook_sync.auth, device-code OAuth

Implements the Microsoft identity platform device-code flow over urllib only. Tokens are cached as JSON at the configured path with mode 0600. The write is atomic. It uses a .tmp file and then a rename.

class OutlookAuth(client_id: str, tenant: str, cache_path: str)

Holds the client ID, the tenant, and a TokenCache. The tenant defaults to common. It raises AuthError right away if client_id is empty.

OutlookAuth.ensure_token(interactive: bool = False) -> str

Returns a valid access token. It uses the cached token if it is still valid for more than 60s. Otherwise it does a silent refresh-token exchange. With interactive=True it falls back to the device-code flow on a failed refresh. With interactive=False it raises AuthError instead and prompts "run with --auth". The sync path uses interactive=False.

class TokenCache(path: Path)

load() reads the cache JSON. It returns {} on missing or corrupt files. save(data) writes it atomically with 0600 permissions.

class AuthError(Exception)

Raised on any consent or token failure. The Microsoft device-code endpoint's normal authorization_pending and slow_down responses are caught inside the poll loop. They are not surfaced as errors.

outlook_sync.api, Microsoft Graph client

A thin Graph client over urllib. The base URL is https://graph.microsoft.com/v1.0. Every request sends Prefer: outlook.body-content-type="text" so Graph strips HTML for us. It also requests an explicit $select projection per message. A 401 triggers one silent token re-fetch. A 429 honours Retry-After. Transient transport failures are retried with exponential backoff at 1, 2, 4, 8 and 16 s. These failures include a read timeout, an SSL EOF, a reset, and a server hang-up. The per-request socket timeout is 90 s.

class OutlookClient(token_provider: Callable[[], str], user_agent="evolutiondb-outlook-sync")

Takes a callable that yields a bearer token. It is wired to OutlookAuth.ensure_token.

OutlookClient.list_messages(*, received_after_iso=None, top=50, folder_filter=None) -> Iterator[dict]

Yields message dicts from /me/messages in ascending receivedDateTime order. You can filter by an after-timestamp or a parentFolderId or both. The results are paginated 50 per page.

OutlookClient.list_sent_messages(*, received_after_iso=None, top=50) -> Iterator[dict]

Yields from the well-known /me/mailFolders/sentitems/messages endpoint. The cross-folder /me/messages pass under-reports Sent Items. This explicit pass recovers the full outbound history for "did I reply?" recall.

OutlookClient.list_folders() -> list[dict]

Returns a flat list of mail folders from /me/mailFolders. It is used to resolve parentFolderId to a display name.

OutlookClient.get_user() -> dict

Returns the signed-in user resource from /me.

addr_text(addr: dict | None) -> str
recipient_list(items: list[dict] | None) -> str

addr_text flattens Graph's {"emailAddress": {"name", "address"}} shape into a single "Name <addr>" string. recipient_list joins a recipient array into a comma-separated string.

class GraphError(Exception)

Carries the HTTP status, the message, and the URL. Folder-fetch and SentItems-pass GraphErrors are caught in sync_once. They are logged to stderr and counted. They do not abort the pass.

outlook_sync.state, EvolutionDB memory backend

Writes records and the watermark over the PG wire protocol via psycopg. On construction it opens an autocommit connection and issues CREATE MEMORY STORE <store>. It ignores the error if the store already exists. On a dropped connection it reconnects up to three times with backoff.

class MemoryStore(host, port, user, password, database, store, namespace)

Opens the connection and ensures the target memory store exists.

MemoryStore.put_record(key: str, record: dict) -> None

Serializes record to JSON and UPSERTs it. It runs a DELETE on the key first. The delete is a no-op when nothing matches. It then runs MEMORY PUT INTO <store> VALUES (<namespace>, <key>, <value>).

MemoryStore.has_message(msg_id: str) -> bool

Returns whether outlook_msg_<msg_id> already exists in the store. It is used to skip messages already written on an earlier pass.

MemoryStore.get_watermark_iso() -> str | None
MemoryStore.set_watermark_iso(iso: str) -> None

Read and write the single highest-receivedDateTime watermark. It is stored under the key outlook_state_latest. get_watermark_iso queries the underlying __mem_<store> table directly.

Notes & limitations

  • Read-only by default. The requested scopes are Mail.Read, User.Read, and offline_access. The connector cannot send mail, mark messages, or change the mailbox. To allow sending, set EVOSQL_OUTLOOK_SEND to add Mail.Send for the outbox action loop. See ADR-004 for details.
  • Body cap. Bodies are clipped at 3 000 characters to stay under EvolutionDB's current per-row size limit. Clipped rows carry the truncated tag. The in-source comment notes the cap will lift once Task #233 lands. That task is the TOAST overflow work.
  • Append-only. Messages are never updated after the first write. Changes to read state or folder membership on the Outlook side are not copied back. A row that fails to write is logged and skipped. One bad row never aborts the pass.
  • Watermark blind spot. The watermark is receivedDateTime. A message backdated by a sender to a time earlier than the current watermark will not be picked up on the next pass. An occasional --since 30d re-run catches stragglers via the has_message dedup check.
  • PII protection. If evolutiondb-pii can be imported, the fact, subject, from, to, cc, snippet, and body fields are passed through its protect_record() before storage. Otherwise it is a transparent passthrough.
  • Privacy. Records land in the same store the MCP server reads from. Anyone with access to that database and the matching MCP_USER_ID can recall the content through search_memory.
  • Version. The packaged distribution is 0.1.3. Its PyPI status is Development Status 4, Beta. The outlook_sync.__version__ attribute in the module is "0.1.0".

See also the MCP server docs. The project wiki is at wiki.evolutiondb.org.
