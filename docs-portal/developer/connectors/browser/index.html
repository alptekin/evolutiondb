Browser history connector

Reads your local browser history straight out of each browser's on-disk SQLite store and pours the visit records into EvolutionDB long-term memory: no OAuth, no extension, no remote API.

What it does

evolutiondb-browser-sync is a *-sync connector that discovers every browser profile the current user owns, opens each browser's history file read-only, normalises the visit rows into a common record shape, and writes them into an EvolutionDB MEMORY STORE over the PostgreSQL wire protocol. It can run once or as a polling daemon, and it keeps a per-profile watermark so repeat passes only import new visits.

The supported browsers fall into two on-disk families:

  • Chromium family, Chrome (incl. Beta / Chromium), Edge, Brave, Arc, Vivaldi, Opera. History lives in a History SQLite file with a urls(url, title, visit_count, last_visit_time) table; timestamps are Chrome-epoch microseconds since 1601.
  • Mozilla family, Firefox (incl. Developer Edition / Nightly). History lives in places.sqlite with a moz_places(url, title, visit_count, last_visit_date) table; timestamps are Unix microseconds.

Each visit becomes one memory record keyed by (browser, profile, sha1(url)), so re-syncing the same URL upserts the existing row rather than multiplying it. Records are auto-tagged by URL host (e.g. youtube/video, github/code, search, ai, news, banking) so the catalog can answer topical questions without scanning every URL string.

There is no README in the package; this page is written from the source itself. The pyproject.toml reports version 0.1.2, while browser_sync/__init__.py still hard-codes __version__ = "0.1.0", that mismatch is in the real package.

Install

The distribution name is evolutiondb-browser-sync and it requires Python >=3.9.

# From the package directory (client/browser-sync)
pip install evolutiondb-browser-sync

Two runtime dependencies are pulled in:

  • psycopg[binary]>=3.1, the PostgreSQL driver used to talk to EvolutionDB.
  • evolutiondb-pii>=0.1.1, optional PII masking. The sync imports evolutiondb_pii.integration.protect_record; if it cannot be imported, masking is simply skipped (records are written unmasked).

Installing registers one console script (see [project.scripts]):

evolutiondb-browser-sync = browser_sync.__main__:main

You can also invoke the module directly with python -m browser_sync.

Configure (env vars / auth)

There is no auth flow, no --auth subcommand and no OAuth. The browser's history lives in a file the user already owns, so the connector just opens it read-only. Connection and behaviour are driven entirely by environment variables (read in Config.__init__), with the defaults shown below: 

# EvolutionDB connection (PostgreSQL wire protocol, port 5433)
EVOSQL_HOST=127.0.0.1
EVOSQL_PORT=5433
EVOSQL_USER=admin
EVOSQL_PASSWORD=admin
EVOSQL_DATABASE=evosql

# Memory store + namespace
MCP_USER_ID=default_user          # memory namespace
BROWSER_MEMORY_STORE=mcp_mem      # target MEMORY STORE name
BROWSER_POLL_INTERVAL=1800        # read into Config but see note below

Variables are loaded from a .env file (path configurable with --env-file, default .env) via a small built-in loader. Note the loader uses os.environ.setdefault, so a value already present in the real environment wins over the .env file, and blank values in .env are ignored.

info
BROWSER_POLL_INTERVAL is parsed into Config.poll_secs but the CLI daemon loop actually uses the --interval argument, not this field. The daemon interval is floored at 60 seconds.

Usage

The CLI is built with argparse (prog="evolutiondb-browser-sync"). Real flags:

  • --once: run a single sync pass over every discovered profile, then exit.
  • --interval SECONDS, daemon mode; repeat passes on a timer (minimum 60s). Each pass is incremental.
  • --since, wall-clock floor used on a profile's first run, before it has a stored watermark. Default 7d. Accepts <n><unit> where unit is one of s m h d w (e.g. 1h, 30d).
  • --browser, comma list to restrict browsers (chrome,edge,firefox,brave,arc,vivaldi,opera). Empty means every detected browser.
  • --list: print every detected profile (browser, profile id, history path) and exit.
  • --dry-run, do everything except write to memory (no connection is opened).
  • --env-file, path to the .env file (default .env).
# See which profiles this machine exposes (no DB needed)
evolutiondb-browser-sync --list

# Single pass over everything, importing the last 30 days on first run
evolutiondb-browser-sync --once --since 30d

# Chrome + Firefox only, no writes, just count what would import
evolutiondb-browser-sync --once --dry-run --browser chrome,firefox

# Daemon: re-scan every 30 minutes (incremental via watermark)
evolutiondb-browser-sync --interval 1800

A pass prints one JSON line to stdout. On success it carries the counters returned by sync_once; on failure it reports the error: 

# success
{"ok": true, "profiles": 3, "visits": 1280, "skipped": 0, "errors": 0}
# failure
{"ok": false, "error": "..."}

Exit codes: 0 normal, 2 on a bad --since value, 4 when a pass raises. SIGINT/SIGTERM request a graceful stop, the daemon finishes the current pass before exiting.

The connector can also be driven as a library:

from browser_sync import scanner, extract
from browser_sync.sync import Config, sync_once

# 1. discover profiles, optionally filtered
profiles = scanner.discover_profiles(["chrome"])
for p in profiles:
    for rec in extract.iter_visits(p, since_iso=None):
        print(rec["url"], rec["last_visited_at"], rec["tags"])

# 2. or run a full pass (dry-run = no DB connection)
counts = sync_once(Config(), since_iso=None,
                   browser_filter=None, dry_run=True)

Reference

browser_sync.scanner

@dataclass(frozen=True) Profile(family, browser, profile_id, history_path)

One discovered profile. family is "chromium" or "firefox"; browser is an id like "chrome"/"edge"/"firefox"; profile_id is a human label such as "Chrome/Default" or "Firefox/<random-id>"; history_path is the Path to the SQLite history file.

discover_profiles(browser_filter: List[str] = None) -> List[Profile]

Enumerates every Chromium and Firefox profile visible on the current OS (macOS, Linux incl. snap/flatpak, Windows). For Chromium it walks each vendor's user-data root and keeps Default / Profile* directories that contain a History file. For Firefox it reads profiles.ini (and falls back to scanning a Profiles/ directory) and keeps those with a places.sqlite. The optional browser_filter keeps only matching browser ids (lower-cased).

browser_sync.extract

iter_visits(profile: Profile, since_iso: Optional[str] = None) -> Iterator[Dict]

Opens the profile's history store read-only and yields normalised visit records, dispatching to the Chromium or Firefox SELECT based on profile.family. When since_iso (a UTC ISO 8601 string) is given, rows whose latest visit is older are filtered out. URLs with internal/non-web schemes (chrome://, edge://, moz-extension://, about:, file://, data:, javascript:, blob:, view-source:, …) are skipped.

To avoid the exclusive WAL lock a running browser holds, the store is first copied to a temp file (shutil.copy, including any -wal/-shm sidecars). If that copy hits PermissionError (typically Windows + busy browser), it falls back to opening the live file via the SQLite URI file:<path>?mode=ro&immutable=1.

Each record is a dict with these keys: fact (a human-readable sentence), source ("browser"), kind ("visit"), browser, browser_family, profile, url, title, visit_count, last_visited_at (ISO string), url_hash (first 16 hex chars of sha1(url)), and tags (always ["browser", "history", <browser>] plus any topical tags derived from the host).

browser_sync.sync

main(argv: Optional[List[str]] = None) -> int

CLI entry point (the registered console script). Parses arguments, loads the .env file, installs signal handlers, and runs one pass (--once or no --interval) or loops on --interval. Returns the process exit code.

sync_once(cfg, *, since_iso, browser_filter, dry_run=False) -> Dict[str, int]

Runs a single pass. For each discovered profile it reads the stored watermark (or uses since_iso as the floor), iterates new visits, optionally masks them, writes each record, advances the watermark to the newest visit seen, and returns the counters {"profiles", "visits", "skipped", "errors"}. Per-profile failures are caught, logged to stderr, counted in errors, and the pass continues. When dry_run is true no MemoryStore is created.

parse_since(text: str) -> str

Converts a relative duration like "1h" or "7d" into an absolute UTC ISO 8601 floor (…Z). Raises ValueError on bad input. Units: s m h d w.

class Config

Reads all eight environment variables listed under Configure into typed fields (evosql_host, evosql_port, evosql_user, evosql_pass, evosql_db, user_id, store, poll_secs).

browser_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

The memory backend. Connects with psycopg (autocommit) and best-effort issues CREATE MEMORY STORE <store> on init. It transparently reconnects up to 3 times with a 0.5s linear backoff on OperationalError/InterfaceError. Key methods:

  • put_record(key, record), JSON-encodes the record and runs MEMORY PUT INTO <store> VALUES ('<namespace>','<key>','<json>').
  • get_watermark_iso(profile_id), reads the per-profile watermark row (key browser_state_<profile>) directly from __mem_<store> and returns its last_visited_at.
  • set_watermark_iso(profile_id, last_iso), upserts that watermark row.
warning
Values are escaped by hand (the module-level _e helper doubles single quotes and strips newlines/tabs) and interpolated into the SQL string, these calls are not parameterised. Treat the connector as trusted-local tooling, not a general-purpose query path.

Notes & limitations

  • Local-only, read-only. Nothing is fetched remotely and no browser data is modified, the connector opens a snapshot copy (or an immutable read-only handle) of files the user already owns.
  • No --auth / OAuth. This is intentional and called out in the source; do not expect a credentials flow.
  • One row per unique URL per profile. Records are keyed by the SHA-1 of the URL, so a URL visited many times is a single upserted row carrying its latest visit_count/last_visited_at, not one row per visit.
  • Incremental via watermark. After the first run the per-profile floor is the stored watermark, so --since only matters before a watermark exists.
  • PII masking is optional. If evolutiondb-pii is not importable, the fact/title/url fields are stored unmasked.
  • Version field mismatch. __init__.__version__ (0.1.0) lags the packaged version (0.1.2).
  • No tests / no README shipped in the package directory.

The records this connector writes are queryable through the same memory store the MCP server exposes. For protocol and store internals see the EvolutionDB wiki.
