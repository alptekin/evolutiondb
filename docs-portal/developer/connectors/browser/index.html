Browser history connector

This connector reads your local browser history. It reads the visit records from each browser's SQLite file on disk. It writes those records into EvolutionDB long-term memory. There is no OAuth, no extension, and no remote API.

What it does

evolutiondb-browser-sync is a *-sync connector. It finds every browser profile the current user owns. It opens each browser's history file read-only. It turns the visit rows into one common record shape. Then it writes them into an EvolutionDB MEMORY STORE over the PostgreSQL wire protocol. It can run once or as a polling daemon. It keeps a watermark for each profile, so repeat passes only import new visits.

The supported browsers fall into two families on disk.

  • Chromium family. This includes Chrome (and Beta / Chromium), Edge, Brave, Arc, Vivaldi, and Opera. History lives in a History SQLite file with a urls(url, title, visit_count, last_visit_time) table. Timestamps are Chrome-epoch microseconds since 1601.
  • Mozilla family. This includes Firefox (and Developer Edition / Nightly). History lives in places.sqlite with a moz_places(url, title, visit_count, last_visit_date) table. Timestamps are Unix microseconds.

Each visit becomes one memory record. The key is (browser, profile, sha1(url)). So if you sync the same URL again, it updates the existing row instead of adding a new one. Records get automatic tags from the URL host. For example youtube/video, github/code, search, ai, news, and banking. This way the catalog can answer topic questions without reading every URL string.

info
There is no README in the package. This page is written from the source itself. The pyproject.toml reports version 0.1.2. But browser_sync/__init__.py still hard-codes __version__ = "0.1.0". That mismatch is in the real package.

Install

The distribution name is evolutiondb-browser-sync. It needs Python >=3.9.

# From the package directory (client/browser-sync)
pip install evolutiondb-browser-sync

It pulls in two runtime dependencies.

  • psycopg[binary]>=3.1 is the PostgreSQL driver used to talk to EvolutionDB.
  • evolutiondb-pii>=0.1.1 is optional PII masking. The sync imports evolutiondb_pii.integration.protect_record. If it cannot be imported, masking is skipped. The records are then written unmasked.

Installing registers one console script. See [project.scripts].

evolutiondb-browser-sync = browser_sync.__main__:main

You can also run the module directly with python -m browser_sync.

Configure (env vars / auth)

There is no auth flow. There is no --auth subcommand and no OAuth. The browser's history lives in a file the user already owns. So the connector just opens it read-only. Connection and behavior come entirely from environment variables. These are read in Config.__init__. The defaults are shown below. 

# EvolutionDB connection (PostgreSQL wire protocol, port 5433)
EVOSQL_HOST=127.0.0.1
EVOSQL_PORT=5433
EVOSQL_USER=admin
EVOSQL_PASSWORD=admin
EVOSQL_DATABASE=evosql

# Memory store + namespace
MCP_USER_ID=default_user          # memory namespace
BROWSER_MEMORY_STORE=mcp_mem      # target MEMORY STORE name
BROWSER_POLL_INTERVAL=1800        # read into Config but see note below

A small built-in loader reads the variables from a .env file. You can set the path with --env-file. The default path is .env. The loader uses os.environ.setdefault. So a value that is already in the real environment wins over the .env file. Blank values in .env are ignored.

info
BROWSER_POLL_INTERVAL is parsed into Config.poll_secs. But the CLI daemon loop uses the --interval argument instead of this field. The daemon interval has a floor of 60 seconds.

Usage

The CLI is built with argparse. The program name is prog="evolutiondb-browser-sync". These are the real flags.

  • --once runs a single sync pass over every found profile, then exits.
  • --interval SECONDS turns on daemon mode. It repeats passes on a timer. The minimum is 60 seconds. Each pass is incremental.
  • --since sets the time floor used on a profile's first run, before it has a stored watermark. The default is 7d. It accepts <n><unit> where the unit is one of s m h d w. For example 1h or 30d.
  • --browser takes a comma list to limit the browsers. The options are chrome,edge,firefox,brave,arc,vivaldi,opera. Empty means every browser that is found.
  • --list prints every found profile, then exits. It shows the browser, profile id, and history path.
  • --dry-run does everything except write to memory. No connection is opened.
  • --env-file sets the path to the .env file. The default is .env.
# See which profiles this machine exposes (no DB needed)
evolutiondb-browser-sync --list

# Single pass over everything, importing the last 30 days on first run
evolutiondb-browser-sync --once --since 30d

# Chrome + Firefox only, no writes, just count what would import
evolutiondb-browser-sync --once --dry-run --browser chrome,firefox

# Daemon: re-scan every 30 minutes (incremental via watermark)
evolutiondb-browser-sync --interval 1800

A pass prints one JSON line to stdout. On success the line carries the counters returned by sync_once. On failure it reports the error. 

# success
{"ok": true, "profiles": 3, "visits": 1280, "skipped": 0, "errors": 0}
# failure
{"ok": false, "error": "..."}

There are three exit codes. 0 means normal. 2 means a bad --since value. 4 means a pass raised an error. SIGINT and SIGTERM ask for a clean stop. The daemon finishes the current pass before it exits.

You can also use the connector as a library.

from browser_sync import scanner, extract
from browser_sync.sync import Config, sync_once

# 1. discover profiles, optionally filtered
profiles = scanner.discover_profiles(["chrome"])
for p in profiles:
    for rec in extract.iter_visits(p, since_iso=None):
        print(rec["url"], rec["last_visited_at"], rec["tags"])

# 2. or run a full pass (dry-run = no DB connection)
counts = sync_once(Config(), since_iso=None,
                   browser_filter=None, dry_run=True)

Reference

browser_sync.scanner

@dataclass(frozen=True) Profile(family, browser, profile_id, history_path)

A Profile represents one found browser profile. family is "chromium" or "firefox". browser is an id like "chrome"/"edge"/"firefox". profile_id is a human label such as "Chrome/Default" or "Firefox/<random-id>". history_path is the Path to the SQLite history file.

discover_profiles(browser_filter: List[str] = None) -> List[Profile]

discover_profiles lists every Chromium and Firefox profile on the current OS. It works on macOS, on Linux (including snap and flatpak), and on Windows. For Chromium it walks each vendor's user-data root. It keeps Default and Profile* directories that hold a History file. For Firefox it reads profiles.ini. If that is missing, it scans a Profiles/ directory instead. It keeps the profiles that have a places.sqlite. The optional browser_filter keeps only the matching browser ids. The ids are lower-cased.

browser_sync.extract

iter_visits(profile: Profile, since_iso: Optional[str] = None) -> Iterator[Dict]

iter_visits opens the profile's history store read-only. It yields the visit records in a common shape. It picks the Chromium or Firefox SELECT based on profile.family. You can pass since_iso as a UTC ISO 8601 string to filter out rows whose latest visit is older. Some URLs use internal or non-web schemes and are skipped. The skipped schemes are chrome://, edge://, moz-extension://, about:, file://, data:, javascript:, blob:, view-source:, and more.

A running browser holds an exclusive WAL lock. To get around it, the store is first copied to a temp file with shutil.copy. The copy includes any -wal and -shm sidecars. The copy can hit a PermissionError. This usually happens on Windows when the browser is busy. In that case it falls back to opening the live file using the SQLite URI file:<path>?mode=ro&immutable=1.

Each record is a dict. The fields are: fact is a sentence a person can read. source is "browser". kind is "visit". Then there are browser, browser_family, profile, url, title, and visit_count. last_visited_at is an ISO string. url_hash is the first 16 hex chars of sha1(url). tags is always ["browser", "history", <browser>], plus any topic tags taken from the host.

browser_sync.sync

main(argv: Optional[List[str]] = None) -> int

main is the CLI entry point and the registered console script. It parses arguments, loads the .env file, and installs signal handlers. Then it runs one pass for --once or when there is no --interval. With --interval it loops instead. It returns the process exit code.

sync_once(cfg, *, since_iso, browser_filter, dry_run=False) -> Dict[str, int]

sync_once runs a single pass. For each found profile it reads the stored watermark. If there is none, it uses since_iso as the floor. It then goes through the new visits, masks them if masking is on, and writes each record. It moves the watermark forward to the newest visit it saw. It returns the counters {"profiles", "visits", "skipped", "errors"}. If one profile fails, the error is caught and logged to stderr. It is counted in errors, and the pass keeps going. When dry_run is true, no MemoryStore is created.

parse_since(text: str) -> str

parse_since turns a relative duration like "1h" or "7d" into an absolute UTC ISO 8601 floor that ends in …Z. It raises ValueError on bad input. The units are s m h d w.

class Config

Config reads all eight environment variables listed under Configure into typed fields. The fields are evosql_host, evosql_port, evosql_user, evosql_pass, evosql_db, user_id, store, and poll_secs.

browser_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

MemoryStore is the memory backend. It connects with psycopg in autocommit mode. On init it tries to run CREATE MEMORY STORE <store> as a best effort. It reconnects on its own up to 3 times. It waits a bit longer each time in steps of 0.5s. This happens on OperationalError and InterfaceError. The main methods are listed below.

  • put_record(key, record) turns the record into JSON. It then runs MEMORY PUT INTO <store> VALUES ('<namespace>','<key>','<json>').
  • get_watermark_iso(profile_id) reads the watermark row for one profile. The key is browser_state_<profile>. It reads it straight from __mem_<store> and returns its last_visited_at.
  • set_watermark_iso(profile_id, last_iso) upserts that watermark row.
warning
Values are escaped by hand. The module-level _e helper doubles single quotes and strips newlines and tabs. The values are put straight into the SQL string. These calls are not parameterised. Treat the connector as trusted local tooling. It is not a general-purpose query path.

Notes & limitations

  • Local-only and read-only. Nothing is fetched from a remote place. No browser data is changed. The connector opens a snapshot copy of files the user already owns. Or it opens an immutable read-only handle to them.
  • No --auth or OAuth. This is on purpose and is noted in the source. Do not expect a credentials flow.
  • One row per unique URL per profile. Records are keyed by the SHA-1 of the URL. So a URL visited many times is a single upserted row. That row carries its latest visit_count and last_visited_at. It is not one row per visit.
  • Incremental via watermark. After the first run the floor for each profile is the stored watermark. So --since only matters before a watermark exists.
  • PII masking is optional. If evolutiondb-pii cannot be imported, the fact, title, and url fields are stored unmasked.
  • Version field mismatch. __init__.__version__ is 0.1.0. It lags the packaged version 0.1.2.
  • No tests and no README ship in the package directory.

The records this connector writes can be queried through the same memory store the MCP server exposes. For protocol and store internals, see the EvolutionDB wiki.
