CONNECTOR · SYNC

Azure DevOps connector

A *-sync app that pulls Azure DevOps pull requests and pipeline runs you take part in. It fetches the ones you created and the ones waiting for your review. It writes them into EvolutionDB long-term memory. From there the MCP server can use them through search_memory, the morning brief, and the open-loops layer.

What it does

The package is evolutiondb-azure-devops-sync. It is version 0.1.0 and the status is Development Status :: 4 - Beta. Each pass walks the Azure DevOps REST API version 7.1 for the projects you can access. It writes stable records into a memory store over the PostgreSQL wire protocol. The keys come from stable IDs. A re-sync updates the same row instead of making a copy.

It writes three record kinds plus one watermark per PR.

mem_keycontent
ado_thread_<project/repo>#<prId>one row per pull request
ado_pr_comment_<threadId>_<commentId>PR thread comments
ado_build_<buildId>pipeline runs (CI context)
azdo_state_<project/repo>#<prId>per-PR sync watermark

"Me" is the PAT owner. The connector reads the owner's identity GUID from /_apis/connectionData. It then runs two exact PR queries per project against that GUID. One query is created-by-me. The other is review-requested-from-me. There is no fuzzy text search. The connector always fetches active PRs. It also fetches recently completed and abandoned PRs. Open rows turn into closed rows and the open loops downstream resolve on their own.

The ADO PR object carries reviewers[] with real vote values. The GitHub connector does not have these. The connector stores the votes as they come from the API.

votemeaning
10approved
5approved with suggestions
0no vote
-5waiting for author
-10rejected

Next to the vote, each reviewer keeps isRequired. The PR keeps isDraft and status. Downstream code can work out "review requested from me and I haven't voted" without guessing.

info
Two honest notes about the data. The PR list object has no updatedDate. So updated_at uses the best available value. That value is closedDate, or creationDate if there is no close date. This makes the per-PR watermarks rough. Pipeline runs are searchable context. They are not open loops.

Install

Run an editable install from the monorepo.

pip install -e client/azure-devops-sync

The only runtime dependency is psycopg[binary]>=3.1. The REST client uses the Python standard library through urllib. It does not use requests or the Azure SDK. You need Python 3.9 or newer. Installing the package adds one console script.

evolutiondb-azure-devops-sync

Configure (env vars / auth)

Copy .env.example to .env and fill it in. The .env file is gitignored. You must set AZURE_DEVOPS_ORG_URL and AZURE_DEVOPS_PAT. The PAT needs the Code (Read) and Build (Read) scopes. The dotenv loader uses setdefault. A value already set in the real environment wins over the file.

variabledefaultpurpose
AZURE_DEVOPS_ORG_URL, org URL. It must start with https://. For example https://dev.azure.com/your-org
AZURE_DEVOPS_PAT, Personal Access Token. Used with Basic auth. The username is empty and the password is the PAT
AZURE_DEVOPS_PROJECTS(empty)comma-separated project filter. Empty means every project in the org
EVOSQL_HOST127.0.0.1EvolutionDB host
EVOSQL_PORT5433PG wire port
EVOSQL_USERadminDB user
EVOSQL_PASSWORDadminDB password
EVOSQL_DATABASEevosqldatabase name
MCP_USER_IDdefault_usermemory namespace. Use the same value as the MCP server's MCP_USER_ID
ADO_MEMORY_STOREmcp_memmemory store the records land in. This is the MCP server's main store
ADO_POLL_INTERVAL900daemon poll seconds. The --interval flag wins when you pass it

The store is created for you on the first connect. The connector runs CREATE MEMORY STORE. If the store already exists, the error is ignored. You do not need a manual schema step.

Usage

The CLI takes one set of flags. It has no subcommands. Here is a single pass.

# one pass, then exit
evolutiondb-azure-devops-sync --once

# fetch everything but persist nothing (no DB writes)
evolutiondb-azure-devops-sync --once --dry-run

# run as a daemon, one pass every 900 seconds
evolutiondb-azure-devops-sync --interval 900

These are the flags it supports. They come from argparse.

flageffect
--oncerun a single pass and exit
--interval Ndaemon mode, sleep N seconds between passes (floored to 60)
--sincestart floor when no watermark exists. Default is 7d. The format is <n><unit> and the unit is one of s m h d w. For example 1h or 7d
--dry-runfetch but do not save
--env-filedotenv path. Default is .env. Ignored if the file is missing

If you give neither --once nor --interval, the connector runs exactly one pass. This is the same as --once.

Output & exit codes

Each pass prints one JSON line to stdout. This is the line on success.

{"ok": true, "prs": N, "comments": N, "builds": N, "errors": N}

On a failed pass it prints {"ok": false, "error": "<message>"}. A fetch failure for one project does not stop the pass. The connector logs it to stderr and counts it in errors.

The exit codes are simple. 0 means ok. 2 means a config error. A bad --since value or a missing org URL or PAT are examples. 4 means the pass failed. SIGTERM and SIGINT ask for a clean stop. The current pass finishes before the daemon exits.

Record shape

A PR record is the value stored under ado_thread_…. It looks like this.

{
  "fact": "jane.dev@example.com opened pr widgets-project/widgets#42: Add retry/backoff - …",
  "source": "azure_devops",
  "kind": "pr",
  "repo": "widgets-project/widgets",
  "number": 42,
  "status": "active",           // raw ADO status: active|completed|abandoned
  "state": "open",              // derived; "closed" once status != active
  "reviewers": [{"name": "sam.dev@example.com", "display": "Sam", "vote": -5, "is_required": true}],
  "is_draft": false,
  "source_branch": "topic/backoff",
  "target_branch": "main",
  "updated_at": "2026-06-08T09:00:00Z",  // closedDate, else creationDate
  "tags": ["azure_devops", "pr", "widgets-project/widgets"]
}

The connector clips the fact, title, and body text to about 4 KB before it stores them. Comment records have kind set to pr_comment. They skip system comments and empty bodies. They add a path field when the thread is tied to a file. Build records have kind set to pipeline_run. They carry pipeline, result, branch, commit, and the run's web URL.

Reference

The package has three modules. api is the REST client. sync holds the record builders and the CLI. state is the memory backend. The entry point is azure_devops_sync.__main__:main. It re-exports sync.main.

azure_devops_sync.api

class AzdoClient(org_url: str, pat: str, user_agent: str = "evolutiondb-azure-devops-sync")

This is a small Azure DevOps REST 7.1 client for one organization. It checks that org_url starts with https:// and that a PAT is present. It raises AzdoError if either check fails. It builds the Basic auth header from :<pat>.

AzdoClient.connection_data() -> dict

Returns the identity of the PAT owner from /_apis/connectionData. The result is the {'id': <guid>, …} object that drives the "me" queries.

AzdoClient.projects() -> List[dict]

Returns all projects in the org. It pages through them with $top and $skip.

AzdoClient.pull_requests(project, *, creator_id=None, reviewer_id=None, status="active", min_time=None) -> List[dict]

Returns pull requests for one project. It filters by creator identity GUID, reviewer identity GUID, or both. When min_time is set, it filters on close time with queryTimeRangeType=closed. This only applies to completed or abandoned PRs.

AzdoClient.pr_threads(project, repository_id, pr_id) -> List[dict]

Returns the comment threads on one PR. The connector uses these to build pr_comment records.

AzdoClient.builds(project, *, min_time=None, top=200) -> List[dict]

Returns pipeline runs for one project, sorted newest first using queueTimeDescending. You can floor them by min_time.

The client handles 429 and 503 responses that include a Retry-After header. It retries up to 5 times. It also retries short-lived socket, SSL, and connection errors with an exponential backoff of (1, 2, 4, 8, 16). A non-JSON response is treated as a bad or expired PAT. An HTML login page is one such case. Errors come back as AzdoError with .status and .url.

azure_devops_sync.sync

class Config()

This class reads all settings from environment variables into typed attributes. See Configure.

sync_once(cfg: Config, *, since_default: Optional[str], dry_run: bool = False) -> Dict[str, int]

Runs one full pass. It resolves the PAT owner. It opens the memory store unless dry_run is set. It loops over the projects. It fetches PRs, threads, and builds. It applies the per-PR watermark check and writes the records. Then it returns the {prs, comments, builds, errors} counters.

build_pr_record(pr: Dict, project: str) -> Optional[Dict]

Turns a raw ADO PR object into the stored record. The shape is shown above. Returns None when there is no pullRequestId.

build_comment_record(comment, thread, repo, number) -> Optional[Dict]

Builds a pr_comment record. Returns None for system comments or empty bodies. It adds path when the thread has a filePath context.

build_build_record(build: Dict, project: str) -> Optional[Dict]

Builds a pipeline_run record. It holds the pipeline name, status, result, branch, commit, requester, and web URL. Returns None when there is no build id.

parse_since(text: str) -> str

Parses a relative duration like 1h or 7d into an ISO-8601 UTC timestamp such as 2026-06-05T…Z. Raises ValueError on anything that is not <n><s|m|h|d|w>.

main() -> int

The console-script entry point. It parses the flags. It loads the dotenv file and checks the config. It installs signal handlers. Then it runs one pass or the daemon loop. It returns the process exit code.

azure_devops_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

The memory backend. It connects to EvolutionDB with psycopg in autocommit mode. It makes sure the store exists. If the connection drops, it reconnects up to 3 times with backoff. It writes all records and watermarks with MEMORY PUT INTO <store> VALUES (namespace, key, value). The values are JSON.

MemoryStore.put_record(key: str, record: Dict) -> None

Upserts one JSON record under key in the set namespace.

MemoryStore.get_watermark(repo, number) -> Optional[str]
MemoryStore.set_watermark(repo, number, updated_at) -> None

Read and write the per-PR watermark. It is stored under the azdo_state_ prefix. sync_once skips a PR whose updated_at is not newer than the stored watermark.

info
The backend escapes values and puts them straight into the SQL string. It doubles single quotes and flattens newlines and tabs. It does not use bound parameters. It is meant for internal, trusted PR and build data on a local EvolutionDB instance.

Notes & limitations

  • Rough PR freshness. The PR list object has no updatedDate. So updated_at falls back to closedDate or creationDate. A new comment on an unchanged PR will not move the PR watermark. The comment rows are still re-fetched and upserted each pass.
  • Pipeline runs are context, not loops. Build records are written for search and brief context only. They are not tracked as open loops.
  • "Me" is the PAT owner. There is no username option. The connector always scopes PRs to the PAT owner's identity GUID.
  • Optional PII masking. If evolutiondb-pii is installed, the fields fact, title, body, and author are protected with encrypted companion rows through the PII guard. Without it, masking does nothing.
  • Cleartext by default. Records store logins, titles, and up to about 4 KB of PR and comment text in cleartext. Anyone with DB access and the namespace can read them. Keep real org names and tokens in .env only. That file is gitignored.
  • Tests use pure functions. tests/test_sync.py runs the record builders and the parse_since parser with fictional fixtures. There is no live API and no DB.
