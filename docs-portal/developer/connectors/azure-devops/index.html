CONNECTOR · SYNC

Azure DevOps connector

A *-sync app that pulls the Azure DevOps pull requests and pipeline runs you're involved in, created by you or awaiting your review: into EvolutionDB long-term memory, where the MCP server's search_memory, morning brief, and open-loops layer can use them.

What it does

The package is evolutiondb-azure-devops-sync (version 0.1.0, Development Status :: 4 - Beta). Each pass walks the Azure DevOps REST API (version 7.1) for the projects you have access to and writes deterministic records into a memory store over the PostgreSQL wire protocol. Because the keys are derived from stable IDs, a re-sync upserts the same row instead of duplicating it.

Three record kinds plus a per-PR watermark are written:

mem_keycontent
ado_thread_<project/repo>#<prId>one row per pull request
ado_pr_comment_<threadId>_<commentId>PR thread comments
ado_build_<buildId>pipeline runs (CI context)
azdo_state_<project/repo>#<prId>per-PR sync watermark

"Me" is the PAT owner. The connector resolves the owner's identity GUID from /_apis/connectionData, then runs two precise per-project PR queries against that GUID, created-by-me and review-requested-from-me, so there is no fuzzy text search. Active PRs are always fetched; recently completed / abandoned PRs are fetched too, so open rows flip to closed and downstream open-loops auto-resolve.

Unlike the GitHub connector, the ADO PR object carries reviewers[] with first-class vote values, which are stored verbatim:

votemeaning
10approved
5approved with suggestions
0no vote
-5waiting for author
-10rejected

Alongside the vote, each reviewer keeps isRequired, and the PR keeps isDraft and status, so "review requested from me and I haven't voted" is derivable downstream without guessing.

info
Honesty notes baked into the data: the PR list object has no updatedDate, so updated_at is the best available (closedDate, else creationDate) and per-PR watermarks are correspondingly coarse. Pipeline runs are searchable context, not open loops.

Install

Editable install from the monorepo:

pip install -e client/azure-devops-sync

The only runtime dependency is psycopg[binary]>=3.1, the REST client uses the Python standard library (urllib), no requests or Azure SDK. Python 3.9+ is required. Installing the package exposes one console script:

evolutiondb-azure-devops-sync

Configure (env vars / auth)

Copy .env.example to .env (gitignored) and fill it in. At minimum set AZURE_DEVOPS_ORG_URL and AZURE_DEVOPS_PAT. The PAT needs Code (Read) + Build (Read) scopes. The dotenv loader uses setdefault, so a value already present in the real environment wins over the file.

variabledefaultpurpose
AZURE_DEVOPS_ORG_URL, org URL, must start with https:// (e.g. https://dev.azure.com/your-org)
AZURE_DEVOPS_PAT, Personal Access Token; Basic auth, username empty, password = PAT
AZURE_DEVOPS_PROJECTS(empty)comma-separated project filter; empty = every project in the org
EVOSQL_HOST127.0.0.1EvolutionDB host
EVOSQL_PORT5433PG wire port
EVOSQL_USERadminDB user
EVOSQL_PASSWORDadminDB password
EVOSQL_DATABASEevosqldatabase name
MCP_USER_IDdefault_usermemory namespace, share it with the MCP server's MCP_USER_ID
ADO_MEMORY_STOREmcp_memmemory store the records land in (the MCP server's primary store)
ADO_POLL_INTERVAL900daemon poll seconds; the --interval flag wins when passed

On first connect the store is auto-created (a CREATE MEMORY STORE attempt whose error is swallowed if it already exists), so no manual schema step is needed.

Usage

The CLI takes a single subcommand-free set of flags. A one-shot pass:

# one pass, then exit
evolutiondb-azure-devops-sync --once

# fetch everything but persist nothing (no DB writes)
evolutiondb-azure-devops-sync --once --dry-run

# run as a daemon, one pass every 900 seconds
evolutiondb-azure-devops-sync --interval 900

The supported flags (from argparse):

flageffect
--oncerun a single pass and exit
--interval Ndaemon mode, sleep N seconds between passes (floored to 60)
--sinceinitial floor when no watermark exists; default 7d. Format <n><unit> with unit one of s m h d w (e.g. 1h, 7d)
--dry-runfetch but don't persist
--env-filedotenv path; default .env, ignored if missing

If neither --once nor --interval is given, the connector runs exactly one pass (same as --once).

Output & exit codes

Each pass prints one JSON line to stdout. On success:

{"ok": true, "prs": N, "comments": N, "builds": N, "errors": N}

On a failed pass: {"ok": false, "error": "<message>"}. Per-project fetch failures don't abort the pass, they're logged to stderr and counted in errors.

Exit codes: 0 ok · 2 config error (bad --since, or missing org URL / PAT) · 4 pass failed. SIGTERM/SIGINT requests a graceful stop, the current pass finishes before the daemon exits.

Record shape

A PR record (the value stored under ado_thread_…) looks like:

{
  "fact": "jane.dev@example.com opened pr widgets-project/widgets#42: Add retry/backoff - …",
  "source": "azure_devops",
  "kind": "pr",
  "repo": "widgets-project/widgets",
  "number": 42,
  "status": "active",           // raw ADO status: active|completed|abandoned
  "state": "open",              // derived; "closed" once status != active
  "reviewers": [{"name": "sam.dev@example.com", "display": "Sam", "vote": -5, "is_required": true}],
  "is_draft": false,
  "source_branch": "topic/backoff",
  "target_branch": "main",
  "updated_at": "2026-06-08T09:00:00Z",  // closedDate, else creationDate
  "tags": ["azure_devops", "pr", "widgets-project/widgets"]
}

fact/title/body text is clipped to ~4 KB before storing. Comment records (kind: pr_comment) skip system comments and empty bodies, and add a path field when the thread is anchored to a file. Build records carry kind: pipeline_run with pipeline, result, branch, commit, and the run's web URL.

Reference

The package is three modules: api (REST client), sync (record builders + CLI), and state (memory backend). The entry point is azure_devops_sync.__main__:main, which simply re-exports sync.main.

azure_devops_sync.api

class AzdoClient(org_url: str, pat: str, user_agent: str = "evolutiondb-azure-devops-sync")

Minimal Azure DevOps REST 7.1 client bound to one organization. Validates that org_url is https:// and that a PAT is present (raising AzdoError otherwise), and builds the Basic auth header from :<pat>.

AzdoClient.connection_data() -> dict

Returns the authenticated (PAT-owner) identity from /_apis/connectionData, the {'id': <guid>, …} object that drives the "me" queries.

AzdoClient.projects() -> List[dict]

All projects in the org (paginated via $top/$skip).

AzdoClient.pull_requests(project, *, creator_id=None, reviewer_id=None, status="active", min_time=None) -> List[dict]

Pull requests for one project, filtered by creator and/or reviewer identity GUID. When min_time is set it filters on close time (queryTimeRangeType=closed), which is only meaningful for completed/abandoned PRs.

AzdoClient.pr_threads(project, repository_id, pr_id) -> List[dict]

The comment threads on one PR (used to build pr_comment records).

AzdoClient.builds(project, *, min_time=None, top=200) -> List[dict]

Pipeline runs for one project, newest first (queueTimeDescending), optionally floored by min_time.

The client honors 429/503 with Retry-After (bounded to 5 retries) and retries transient socket/SSL/connection errors with an exponential backoff (1, 2, 4, 8, 16); a non-JSON response (e.g. an HTML login page) is treated as a bad/expired PAT. Errors surface as AzdoError with .status and .url.

azure_devops_sync.sync

class Config()

Reads all configuration from environment variables (see Configure) into typed attributes.

sync_once(cfg: Config, *, since_default: Optional[str], dry_run: bool = False) -> Dict[str, int]

Runs one full pass: resolves the PAT owner, opens the memory store (unless dry_run), iterates the projects, fetches PRs / threads / builds, applies the per-PR watermark check, writes records, and returns the {prs, comments, builds, errors} counters.

build_pr_record(pr: Dict, project: str) -> Optional[Dict]

Turns a raw ADO PR object into the stored record (the shape shown above). Returns None when there's no pullRequestId.

build_comment_record(comment, thread, repo, number) -> Optional[Dict]

Builds a pr_comment record. Returns None for system comments or empty bodies; adds path when the thread has a filePath context.

build_build_record(build: Dict, project: str) -> Optional[Dict]

Builds a pipeline_run record (pipeline name, status/result, branch, commit, requester, web URL). Returns None when there's no build id.

parse_since(text: str) -> str

Parses a relative duration like 1h / 7d into an ISO-8601 UTC timestamp (e.g. 2026-06-05T…Z). Raises ValueError on anything that isn't <n><s|m|h|d|w>.

main() -> int

The console-script entry point: parses flags, loads the dotenv file, validates config, installs signal handlers, and runs one pass or the daemon loop. Returns the process exit code.

azure_devops_sync.state

class MemoryStore(host, port, user, password, database, store, namespace)

The memory backend. Connects to EvolutionDB with psycopg (autocommit), ensures the store exists, and reconnects up to 3 times with backoff on a dropped connection. All records and watermarks are written with MEMORY PUT INTO <store> VALUES (namespace, key, value); values are JSON.

MemoryStore.put_record(key: str, record: Dict) -> None

Upserts one JSON record under key in the configured namespace.

MemoryStore.get_watermark(repo, number) -> Optional[str]
MemoryStore.set_watermark(repo, number, updated_at) -> None

Read / write the per-PR watermark (stored under the azdo_state_ prefix). sync_once skips a PR whose updated_at isn't newer than the stored watermark.

info
The backend escapes values and inlines them into the SQL string (single quotes doubled, newlines/tabs flattened) rather than using bound parameters. It's intended for internal, trusted PR/build data on a local EvolutionDB instance.

Notes & limitations

  • Coarse PR freshness. The PR list object has no updatedDate, so updated_at falls back to closedDate or creationDate. A new comment on an unchanged PR won't move the PR watermark; comment rows are still re-fetched and upserted each pass.
  • Pipeline runs are context, not loops. Build records are written for search/brief context only and are not tracked as open loops.
  • "Me" = PAT owner. There's no username option; the connector always scopes PRs to the PAT owner's identity GUID.
  • Optional PII masking. If evolutiondb-pii is installed, the fields fact, title, body, author are protected with encrypted companion rows via the PII guard; without it, masking is a no-op.
  • Cleartext by default. Records store logins, titles, and up to ~4 KB of PR/comment text in cleartext, anyone with DB access and the namespace can recall them. Keep real org names and tokens in .env (gitignored) only.
  • Tests are pure-function. tests/test_sync.py exercises the record builders and the parse_since parser with fictional fixtures: no live API, no DB.
