Microsoft Teams connector

Stream your Microsoft Teams 1:1 and group chat messages into EvolutionDB long-term memory, so the MCP server can surface them through search_memory from any MCP-aware client (Claude, ChatGPT, Gemini, Copilot).

What it does

The evolutiondb-teams-sync package walks Microsoft Graph's /me/chats listing, reads the messages in each chat, and writes each message into an EvolutionDB memory store over the PostgreSQL wire protocol. An AI assistant connected through the MCP server can then answer "what did Ahmet say last week about the release plan?" as a normal search_memory call.

It walks every chat returned by GET /me/chats (expanded with members and the last message preview), and for each chat pulls /chats/{id}/messages newest-first, stopping once it crosses the chat's watermark. Each message becomes one record under a deterministic key, so a re-sync UPSERTs rather than duplicating:

# Record                 Key shape
A chat message            teams_chat_<chat_id>_<message_id>
Per-chat watermark        teams_state_<chat_id>

The per-chat watermark stores the highest lastModifiedDateTime seen so far (keyed teams_state_<chat_id>). On later runs the watermark is read from EvolutionDB and used as the floor, so reruns never re-ingest the same message. Watermark rows carry no fact field, which is how the MCP search_memory tool filters them out of results.

info
Records are stored as JSON values. Each message carries a one-line fact string ("<sender>: <text>") plus structured fields, source ("teams"), kind ("chat"), chat_id, chat_name, chat_type, message_id, sender, sender_id, text, created_at, modified_at, and tags (["teams", "chat", <chat_name>]).

Channel messages, meeting transcripts, and call recordings are explicitly out of scope: they need higher-privilege Graph permissions that require tenant admin consent. The connector deliberately stays inside delegated scopes a user can self-consent to.

Install

The PyPI distribution name is evolutiondb-teams-sync; the importable module is teams_sync. It requires Python 3.9+ and depends on msal>=1.28 (device-code auth), httpx>=0.27 (Graph HTTP client), psycopg[binary]>=3.1 (PG-wire writes), evolutiondb-pii>=0.1.1 (optional PII masking), and mcp-server-evolutiondb>=1.3.0 (so a single install brings the MCP integration along).

# from PyPI
pip install evolutiondb-teams-sync

# or editable, from the EvolutionDB repo
pip install -e client/teams-sync

Installation registers a single console script, evosql-teams-sync, which maps to teams_sync.sync:main. You can also run the module directly with python -m teams_sync.sync.

info
The package version differs across metadata: pyproject.toml declares 0.1.3 while the importable module's teams_sync.__version__ reads 0.1.2. Development Status is 4 - Beta.

Configure

Before configuring, register a public-client app under Microsoft Entra ID (App registrations → New registration), enable Allow public client flows, and add the delegated Graph permissions Chat.Read, User.Read, and offline_access (and ChatMessage.Send only if you opt into sending, see Notes). None of these require tenant admin consent. Copy the app's Application (client) ID and Directory (tenant) ID.

Configuration then comes from environment variables. On startup the CLI loads a dotenv file (default .env in the working directory, overridable with --env-file; a missing file is ignored). The real shell environment always wins over the file, and within the file the last non-empty assignment for a key wins. Copy .env.example to .env and fill in:

# --- Microsoft Entra ID app ---
AZURE_TENANT_ID=common            # or your tenant GUID for single-tenant apps
AZURE_CLIENT_ID=<application-client-id>

# --- EvolutionDB (PG wire protocol) ---
EVOSQL_HOST=127.0.0.1
EVOSQL_PORT=5433
EVOSQL_USER=admin
EVOSQL_PASSWORD=admin
EVOSQL_DATABASE=evosql

# --- MCP correlation ---
MCP_USER_ID=alptekin_topal      # namespace shared with the MCP server
TEAMS_MEMORY_STORE=mcp_mem       # memory store to write into
TEAMS_POLL_INTERVAL=600         # daemon poll interval, seconds
TEAMS_TOKEN_CACHE=~/.evosql/teams_token_cache.json

Variable reference, as read by the Config class (plus TEAMS_TOKEN_CACHE read in Config.token_cache, and EVOSQL_TEAMS_SEND read by the auth module):

# var                  default                              meaning
AZURE_TENANT_ID        common                               Entra ID tenant (authority)
AZURE_CLIENT_ID        (none, required for sync)            app Application (client) ID
TEAMS_TOKEN_CACHE      ~/.evosql/teams_token_cache.json     MSAL refresh-token cache path
EVOSQL_HOST            127.0.0.1                            EvolutionDB host
EVOSQL_PORT            5433                                 EvolutionDB PG-wire port
EVOSQL_USER            admin                                EvolutionDB user
EVOSQL_PASSWORD        admin                                EvolutionDB password
EVOSQL_DATABASE        evosql                               target database
MCP_USER_ID            default_user                         memory namespace (mem_namespace)
TEAMS_MEMORY_STORE     mcp_mem                              memory store (CREATE MEMORY STORE)
TEAMS_POLL_INTERVAL    600                                  poll interval in Config.poll_secs
EVOSQL_TEAMS_SEND      (unset)                              opt-in: adds ChatMessage.Send scope
warning
AZURE_CLIENT_ID is mandatory for any sync mode. If it is empty after loading the env, the CLI prints an error and exits with code 2 before contacting Graph or the database. (--auth also constructs TeamsAuth, which raises if the client ID is missing.)

The token cache stores the MSAL refresh token at TEAMS_TOKEN_CACHE (default ~/.evosql/teams_token_cache.json), chmod'd to 0600 after each write. After the first interactive login, subsequent runs refresh silently.

Usage

The first step after install is a one-time login; then you run one-shot or daemon syncs. Exactly one mode flag (--setup, --auth, --once, or --interval) is required per invocation, they form a mutually-exclusive group.

# One-time bootstrap: download the evosql-server binary and wire up MCP host configs.
evosql-teams-sync --setup

# Log in once via device code; caches the refresh token and exits.
evosql-teams-sync --auth

# One pass, then exit. --since bounds how far back the FIRST run on a chat goes.
python -m teams_sync.sync --once --since 24h

# Daemon mode - re-run every N seconds (floored at 60).
python -m teams_sync.sync --interval 600

# Verify auth + scopes, fetch from Graph, but write nothing to the DB.
python -m teams_sync.sync --once --dry-run

Flags accepted by main():

# --setup            one-time: download embedded evosql-server + write MCP host configs
# --auth             run device-code login, cache refresh token, exit
# --once             run a single sync pass and exit
# --interval SECONDS daemon mode; loops until SIGINT/SIGTERM (min 60s)
# --since SPEC       first-run watermark; SPEC is N + s|m|h|d|w (default 24h)
# --dry-run          talk to Graph but do not write to EvolutionDB
# --embedded         bring up a self-contained evosql-server before syncing
# --env-file PATH    dotenv file to load (default .env)
info
Daemon mode is driven by --interval SECONDS, there is no --daemon flag, despite the comment in .env.example. TEAMS_POLL_INTERVAL populates Config.poll_secs but the sync loop's interval comes from the --interval argument (floored at 60s). On the first run of a given chat, --since sets the floor; after that the per-chat watermark in EvolutionDB takes over.

The device-code login surfaces the Microsoft URL and short code on stderr, and, best-effort, through native affordances on the host: on macOS it copies the code to the clipboard, opens the login page, posts a Notification Center banner, and shows a modal dialog; on Linux it tries xdg-open + clipboard tools + notify-send; on Windows it copies the code and opens the browser. All of these are silent no-ops if the tools are missing.

Each pass prints a one-line summary to stderr, e.g.:

[teams-sync] wrote 37 messages across 12 chats (skipped 4 system / empty, 1 chats inaccessible (403/404 - orphan or deleted))

Exit codes: 0 success, 2 configuration error (e.g. missing AZURE_CLIENT_ID or a bad --since), 3 authentication failure, 4 a sync pass raised an exception, 5 embedded-server failure. In daemon mode, SIGINT and SIGTERM request a graceful stop, the current pass finishes before the loop exits.

Running in Docker

The repo's demo compose file ships a teams-sync service behind a profile so it doesn't start with the rest of the demo:

# First-run device-code login (runs once, cleans up after itself):
docker compose -f docker-compose.demo.yml --profile teams \
    run --rm teams-sync --auth

# Bring the daemon up:
docker compose -f docker-compose.demo.yml --profile teams up -d teams-sync

The token cache lives on a named volume (teams_sync_state), so container recreates don't force a fresh login.

Reference

teams_sync.sync, CLI and sync loop

main(argv: list | None = None) -> int

Console-script entry point (registered as evosql-teams-sync). Parses arguments, loads the dotenv file, builds a Config, then dispatches to --setup / --auth / a single pass / the daemon loop. Returns the process exit code.

sync_once(cfg: Config, *, since_default: str | None, dry_run: bool = False) -> dict[str, int]

Runs a single pass: authenticates, fetches the signed-in user via get_me(), walks every chat, builds one record per message, applies watermark skipping, and (unless dry_run) writes to the store and advances the watermark. Returns the counters dict {"chats", "messages", "skipped", "inaccessible_chats"}. Chats that 403/404 on the messages endpoint (orphaned meeting chats, deleted chats) are counted as inaccessible_chats and skipped; any other Graph error propagates.

parse_since(text: str) -> str

Converts a window spec like "1h", "24h", "7d", or "30m" (units s|m|h|d|w) into a UTC ISO-8601 timestamp ending in Z, used as the first-run floor. Raises ValueError on anything it can't parse.

class Config

Reads all environment variables into typed attributes (tenant_id, client_id, token_cache, evosql_host/port/user/pass/db, user_id, store, poll_secs).

teams_sync.auth, MSAL device-code login

Wraps msal.PublicClientApplication with a persistent on-disk token cache. The requested scopes are Chat.Read and User.Read (MSAL adds offline_access automatically); setting EVOSQL_TEAMS_SEND to a truthy value also requests ChatMessage.Send.

class TeamsAuth(tenant_id: str, client_id: str, cache_path: str)

Builds the MSAL public-client app against https://login.microsoftonline.com/<tenant_id>. Raises RuntimeError immediately if client_id is empty.

TeamsAuth.login_interactive() -> str

Runs the device-code flow, surfaces the user code via stderr plus the platform's native affordances, blocks until the user completes login (or MSAL times out), flushes the token cache, and returns the access token.

TeamsAuth.get_token() -> str

Token provider passed to the Graph client; called per request. Tries silent refresh from the cached account first, falling back to login_interactive() if no account is cached or the refresh token was revoked.

class TokenStore(cache_path: str)

Thin wrapper around msal.SerializableTokenCache: deserializes the cache file on construction, and on flush() writes it back and chmods it to 0600.

teams_sync.graph, Microsoft Graph client

A thin client over httpx against graph.microsoft.com/v1.0, with a 90s read timeout. Constructed as GraphClient(token_provider=…).

GraphClient.get_me() -> dict

Returns the signed-in user's profile (id, displayName, userPrincipalName), cached for the client's lifetime. The id is used so 1:1 chats render as the other participant's name.

GraphClient.list_chats() -> list[dict]

Returns every chat the user is in, ordered by recent activity, via GET /me/chats?$top=50&$orderby=lastMessagePreview/createdDateTime desc&$expand=members,lastMessagePreview. Fully materialized (chat counts are small).

GraphClient.get_chat_messages(chat_id: str, since_iso: str | None = None) -> Iterator[dict]

Yields messages from one chat newest-first via GET /chats/{chat_id}/messages?$top=50, stopping as soon as a message at or older than since_iso is seen (client-side cutoff, since Graph rejects the $filter form on some tenants).

chat_display_name(chat: dict, me_id: str | None = None) -> str

Best-effort chat label: the explicit topic for group chats, the other member's display name for 1:1 chats (when me_id is passed), a member-list summary otherwise, falling back to the chat id.

message_text(msg: dict) -> str
message_sender(msg: dict) -> dict[str, str]

message_text returns the plain-text body (stripping HTML via an internal regex stripper) and returns an empty string for system messages (joins, leaves), which the caller counts as skipped. message_sender extracts a {"name", "id"} pair from from.user or from.application.

strip_sender_prefix(text: str, sender_name: str) -> str

Removes a duplicated leading sender display name (some tenants prefix every message body with it) before the searchable fact field re-prepends the sender.

class GraphError(RuntimeError)

Carries the HTTP status. Retryable statuses (429, 502, 503, 504) are retried inside the client, 429/503 honour Retry-After; 502/504 use exponential backoff. Transient transport failures (timeouts, SSL EOF, resets) get a 1/2/4/8/16s backoff before propagating.

teams_sync.state, EvolutionDB memory backend

Writes messages and watermarks over the PG wire protocol via psycopg. On construction it issues CREATE MEMORY STORE <store> (ignoring the error if it already exists) and reconnects up to three times with backoff on a dropped connection. The source notes this backend mirrors the reconnect pattern in mcp-server-evosql.

class MemoryStore(host, port, user, password, database, store, namespace)

Opens an autocommit connection and ensures the target memory store exists. Raises RuntimeError if psycopg is not installed.

MemoryStore.put_message(key: str, value: dict) -> None

Encodes value to JSON (with size-budget truncation, see below) and runs MEMORY PUT INTO <store> VALUES (<namespace>, <key>, <value>). MEMORY DML does not bind parameters, so string values are escaped inline.

MemoryStore.get_watermark(chat_id: str) -> str | None
MemoryStore.set_watermark(chat_id: str, watermark_iso: str) -> None

Read and write the per-chat watermark under teams_state_<chat_id>. get_watermark queries the underlying __mem_<store> table directly and filters by key client-side (MEMORY tables don't expose mem_key filtering over the PG wire).

MemoryStore.count_messages() -> int

Counts total memory rows (messages + watermarks) for this namespace; used by smoke tests to confirm writes land.

Notes & limitations

  • Out of scope: channel messages, meeting transcripts, and call recordings, they require tenant-admin-consent Graph scopes the connector deliberately avoids.
  • System / empty messages are skipped: joins, leaves, and bodies that strip to empty text are not indexed; they're counted under skipped.
  • Inaccessible chats are tolerated: /me/chats can list chats that 403/404 on the messages endpoint (orphaned meeting chats, shared-channel proxies, chats deleted between list and read). These are counted and skipped rather than aborting the pass.
  • Value size budget: EvolutionDB's MEMORY PUT path misbehaves on large payloads, so state.py caps a stored value at ~2500 bytes. Oversized messages are progressively trimmed (drop an html mirror, then clip text and fact with a head/tail marker), and as a last resort persisted as a stub carrying "truncated": true so the message's existence is still recorded.
  • Optional PII masking: if evolutiondb-pii is importable, message records are passed through evolutiondb_pii.integration.protect_record over the fact/text/sender/chat fields before storage; if not installed, records are stored unmasked.
  • Sending is opt-in and off by default: the connector is read-only unless EVOSQL_TEAMS_SEND is set, which adds the ChatMessage.Send scope at consent time (ADR-004). There is no send code path exercised by these modules, the flag only widens the requested OAuth scopes.
  • Privacy & policy: a 1:1 or group chat captures everyone's messages, not just yours; some workplaces treat external archiving of colleagues' messages as a policy violation, and every Graph call is logged in the tenant audit log. Records land in the same store the MCP server reads from, so anyone with that database and the matching MCP_USER_ID can recall the content through search_memory. You can revoke the token at any time from https://myaccount.microsoft.com/applications, the next call gets a 401 and stops.
  • Embedded server reach: --setup and --embedded download a prebuilt evosql-server from the latest server-v* GitHub release for Linux x86_64, Linux arm64, and macOS arm64 only. Windows and macOS Intel are not shipped as native binaries, those hosts fall back to Docker.
  • Version: 0.1.3 per pyproject.toml (Development Status: 4 - Beta); note teams_sync.__version__ still reads 0.1.2.
