Microsoft Teams connector

Copy your Microsoft Teams 1:1 and group chat messages into EvolutionDB long-term memory. The MCP server can then show them through search_memory from any MCP-aware client. This works with Claude, ChatGPT, Gemini, and Copilot.

What it does

The evolutiondb-teams-sync package reads Microsoft Graph's /me/chats list. It reads the messages in each chat. Then it writes each message into an EvolutionDB memory store over the PostgreSQL wire protocol. An AI assistant connected through the MCP server can then answer a question like "what did Ahmet say last week about the release plan?". That is a normal search_memory call.

It reads every chat returned by GET /me/chats. The list includes the members and the last message preview. For each chat it pulls /chats/{id}/messages newest-first. It stops once it passes the chat's watermark. Each message becomes one record under a fixed key. A re-sync then UPSERTs instead of making duplicates.

# Record                 Key shape
A chat message            teams_chat_<chat_id>_<message_id>
Per-chat watermark        teams_state_<chat_id>

The per-chat watermark stores the highest lastModifiedDateTime seen so far. It is keyed teams_state_<chat_id>. On later runs the watermark is read from EvolutionDB and used as the floor. This way reruns never read the same message again. Watermark rows carry no fact field. That is how the MCP search_memory tool keeps them out of results.

info
Records are stored as JSON values. Each message carries a one-line fact string ("<sender>: <text>") plus structured fields, source ("teams"), kind ("chat"), chat_id, chat_name, chat_type, message_id, sender, sender_id, text, created_at, modified_at, and tags (["teams", "chat", <chat_name>]).

Channel messages, meeting transcripts, and call recordings are out of scope. They need higher-privilege Graph permissions. Those permissions need tenant admin consent. The connector stays inside the delegated scopes that a user can grant on their own.

Install

The PyPI distribution name is evolutiondb-teams-sync. The importable module is teams_sync. It needs Python 3.9 or later. It depends on a few packages. msal>=1.28 handles device-code auth. httpx>=0.27 is the Graph HTTP client. psycopg[binary]>=3.1 handles PG-wire writes. evolutiondb-pii>=0.1.1 does optional PII masking. mcp-server-evolutiondb>=1.3.0 is also included, so a single install brings the MCP integration as well.

# from PyPI
pip install evolutiondb-teams-sync

# or editable, from the EvolutionDB repo
pip install -e client/teams-sync

Installation registers a single console script named evosql-teams-sync. It maps to teams_sync.sync:main. You can also run the module directly with python -m teams_sync.sync.

info
The package version is not the same in every place. pyproject.toml declares 0.1.3. The importable module's teams_sync.__version__ reads 0.1.2. Development Status is 4 - Beta.

Configure

Before you configure, register a public-client app under Microsoft Entra ID. Go to App registrations and then New registration. Enable Allow public client flows. Add the delegated Graph permissions Chat.Read, User.Read, and offline_access. Add ChatMessage.Send only if you opt into sending. See the Notes section for that. None of these need tenant admin consent. Copy the app's Application (client) ID and Directory (tenant) ID.

Configuration comes from environment variables. On startup the CLI loads a dotenv file. The default file is .env in the working directory. You can change it with --env-file. A missing file is ignored. The real shell environment always wins over the file. Inside the file, the last non-empty value for a key wins. Copy .env.example to .env and fill it in.

# --- Microsoft Entra ID app ---
AZURE_TENANT_ID=common            # or your tenant GUID for single-tenant apps
AZURE_CLIENT_ID=<application-client-id>

# --- EvolutionDB (PG wire protocol) ---
EVOSQL_HOST=127.0.0.1
EVOSQL_PORT=5433
EVOSQL_USER=admin
EVOSQL_PASSWORD=admin
EVOSQL_DATABASE=evosql

# --- MCP correlation ---
MCP_USER_ID=alptekin_topal      # namespace shared with the MCP server
TEAMS_MEMORY_STORE=mcp_mem       # memory store to write into
TEAMS_POLL_INTERVAL=600         # daemon poll interval, seconds
TEAMS_TOKEN_CACHE=~/.evosql/teams_token_cache.json

The table below shows the variable reference as read by the Config class. TEAMS_TOKEN_CACHE is read in Config.token_cache. EVOSQL_TEAMS_SEND is read by the auth module.

# var                  default                              meaning
AZURE_TENANT_ID        common                               Entra ID tenant (authority)
AZURE_CLIENT_ID        (none, required for sync)            app Application (client) ID
TEAMS_TOKEN_CACHE      ~/.evosql/teams_token_cache.json     MSAL refresh-token cache path
EVOSQL_HOST            127.0.0.1                            EvolutionDB host
EVOSQL_PORT            5433                                 EvolutionDB PG-wire port
EVOSQL_USER            admin                                EvolutionDB user
EVOSQL_PASSWORD        admin                                EvolutionDB password
EVOSQL_DATABASE        evosql                               target database
MCP_USER_ID            default_user                         memory namespace (mem_namespace)
TEAMS_MEMORY_STORE     mcp_mem                              memory store (CREATE MEMORY STORE)
TEAMS_POLL_INTERVAL    600                                  poll interval in Config.poll_secs
EVOSQL_TEAMS_SEND      (unset)                              opt-in: adds ChatMessage.Send scope
warning
AZURE_CLIENT_ID is required for any sync mode. If it is empty after the env loads, the CLI prints an error and exits with code 2. This happens before it contacts Graph or the database. --auth also builds TeamsAuth, which raises an error if the client ID is missing.

The token cache stores the MSAL refresh token at the path set by TEAMS_TOKEN_CACHE. The default path is ~/.evosql/teams_token_cache.json. The file is chmod'd to 0600 after each write. After the first interactive login, later runs refresh silently.

Usage

The first step after install is a one-time login. After that you run one-shot or daemon syncs. Each run needs exactly one mode flag. The choices are --setup, --auth, --once, and --interval. You can only pick one of them at a time.

# One-time bootstrap: download the evosql-server binary and wire up MCP host configs.
evosql-teams-sync --setup

# Log in once via device code; caches the refresh token and exits.
evosql-teams-sync --auth

# One pass, then exit. --since bounds how far back the FIRST run on a chat goes.
python -m teams_sync.sync --once --since 24h

# Daemon mode - re-run every N seconds (floored at 60).
python -m teams_sync.sync --interval 600

# Verify auth + scopes, fetch from Graph, but write nothing to the DB.
python -m teams_sync.sync --once --dry-run

These are the flags accepted by main().

# --setup            one-time: download embedded evosql-server + write MCP host configs
# --auth             run device-code login, cache refresh token, exit
# --once             run a single sync pass and exit
# --interval SECONDS daemon mode; loops until SIGINT/SIGTERM (min 60s)
# --since SPEC       first-run watermark; SPEC is N + s|m|h|d|w (default 24h)
# --dry-run          talk to Graph but do not write to EvolutionDB
# --embedded         bring up a self-contained evosql-server before syncing
# --env-file PATH    dotenv file to load (default .env)
info
Daemon mode is driven by --interval SECONDS. There is no --daemon flag, even though the comment in .env.example mentions one. TEAMS_POLL_INTERVAL fills Config.poll_secs. The sync loop's interval comes from the --interval argument, with a floor of 60s. On the first run of a given chat, --since sets the floor. After that the per-chat watermark in EvolutionDB takes over.

The device-code login shows the Microsoft URL and short code on stderr. It also tries to use native features on the host when it can. On macOS it copies the code to the clipboard, opens the login page, posts a Notification Center banner, and shows a modal dialog. On Linux it tries xdg-open plus clipboard tools plus notify-send. On Windows it copies the code and opens the browser. All of these do nothing if the tools are missing.

Each pass prints a one-line summary to stderr. Here is an example.

[teams-sync] wrote 37 messages across 12 chats (skipped 4 system / empty, 1 chats inaccessible (403/404 - orphan or deleted))

The exit codes have these meanings. 0 means success. 2 means a configuration error, such as a missing AZURE_CLIENT_ID or a bad --since. 3 means an authentication failure. 4 means a sync pass raised an exception. 5 means an embedded-server failure. In daemon mode, SIGINT and SIGTERM ask for a graceful stop. The current pass finishes before the loop exits.

Running in Docker

The repo's demo compose file ships a teams-sync service behind a profile. This way it does not start with the rest of the demo.

# First-run device-code login (runs once, cleans up after itself):
docker compose -f docker-compose.demo.yml --profile teams \
    run --rm teams-sync --auth

# Bring the daemon up:
docker compose -f docker-compose.demo.yml --profile teams up -d teams-sync

The token cache lives on a named volume called teams_sync_state. This means recreating the container does not force a fresh login.

Reference

teams_sync.sync, CLI and sync loop

main(argv: list | None = None) -> int

The console-script entry point is registered as evosql-teams-sync. It parses the arguments, loads the dotenv file, and builds a Config. Then it dispatches to --setup, --auth, a single pass, or the daemon loop. It returns the process exit code.

sync_once(cfg: Config, *, since_default: str | None, dry_run: bool = False) -> dict[str, int]

sync_once runs a single pass. It authenticates, fetches the signed-in user via get_me(), reads every chat, and builds one record per message. It applies watermark skipping. Unless dry_run is set, it writes to the store and advances the watermark. It returns the counters dict {"chats", "messages", "skipped", "inaccessible_chats"}. Some chats return 403 or 404 on the messages endpoint. Orphaned meeting chats and deleted chats are examples of this. These are counted as inaccessible_chats and skipped. Any other Graph error is raised.

parse_since(text: str) -> str

parse_since converts a window spec like "1h", "24h", "7d", or "30m" into a UTC ISO-8601 timestamp ending in Z. The units are s|m|h|d|w. The result is used as the first-run floor. It raises ValueError on anything it cannot parse.

class Config

Config reads all environment variables into typed attributes. These are tenant_id, client_id, token_cache, evosql_host/port/user/pass/db, user_id, store, and poll_secs.

teams_sync.auth, MSAL device-code login

teams_sync.auth wraps msal.PublicClientApplication with a token cache that is saved on disk. The requested scopes are Chat.Read and User.Read. MSAL adds offline_access on its own. If you set EVOSQL_TEAMS_SEND to a truthy value, it also requests ChatMessage.Send.

class TeamsAuth(tenant_id: str, client_id: str, cache_path: str)

TeamsAuth builds the MSAL public-client app against https://login.microsoftonline.com/<tenant_id>. It raises RuntimeError right away if client_id is empty.

TeamsAuth.login_interactive() -> str

login_interactive runs the device-code flow. It shows the user code on stderr and through the platform's native features. It blocks until the user finishes login or MSAL times out. Then it flushes the token cache and returns the access token.

TeamsAuth.get_token() -> str

get_token is the token provider passed to the Graph client. It is called for each request. It tries a silent refresh from the cached account first. It falls back to login_interactive() if no account is cached or the refresh token was revoked.

class TokenStore(cache_path: str)

TokenStore is a thin wrapper around msal.SerializableTokenCache. It reads the cache file when it is created. On flush() it writes the file back and chmods it to 0600.

teams_sync.graph, Microsoft Graph client

teams_sync.graph is a thin client over httpx against graph.microsoft.com/v1.0. It has a 90s read timeout. You build it as GraphClient(token_provider=…).

GraphClient.get_me() -> dict

get_me returns the signed-in user's profile, with id, displayName, and userPrincipalName. The result is cached for the client's lifetime. The id is used so 1:1 chats show the other participant's name.

GraphClient.list_chats() -> list[dict]

list_chats returns every chat the user is in, ordered by recent activity, via GET /me/chats?$top=50&$orderby=lastMessagePreview/createdDateTime desc&$expand=members,lastMessagePreview. It loads the full list, since chat counts are small.

GraphClient.get_chat_messages(chat_id: str, since_iso: str | None = None) -> Iterator[dict]

get_chat_messages yields messages from one chat newest-first via GET /chats/{chat_id}/messages?$top=50. It stops as soon as it sees a message at or older than since_iso. The cutoff is done on the client side because Graph rejects the $filter form on some tenants.

chat_display_name(chat: dict, me_id: str | None = None) -> str

chat_display_name makes a best-effort chat label. It uses the explicit topic for group chats. It uses the other member's display name for 1:1 chats when me_id is passed. Otherwise it uses a member-list summary. As a last resort it uses the chat id.

message_text(msg: dict) -> str
message_sender(msg: dict) -> dict[str, str]

message_text returns the plain-text body. It strips HTML with an internal regex stripper. It returns an empty string for system messages like joins and leaves. The caller counts those as skipped. message_sender extracts a {"name", "id"} pair from from.user or from.application.

strip_sender_prefix(text: str, sender_name: str) -> str

strip_sender_prefix removes a duplicated leading sender display name. Some tenants put the name at the start of every message body. This runs before the searchable fact field adds the sender back.

class GraphError(RuntimeError)

GraphError carries the HTTP status. Some statuses are retryable. The retryable ones are 429, 502, 503, and 504. They are retried inside the client. 429 and 503 honour Retry-After. 502 and 504 use exponential backoff. Transient transport failures like timeouts, SSL EOF, and resets get a 1/2/4/8/16s backoff before the error is raised.

teams_sync.state, EvolutionDB memory backend

teams_sync.state writes messages and watermarks over the PG wire protocol via psycopg. When it is created, it issues CREATE MEMORY STORE <store> and ignores the error if the store already exists. It reconnects up to three times with backoff if the connection drops. The source notes that this backend follows the same reconnect pattern as mcp-server-evosql.

class MemoryStore(host, port, user, password, database, store, namespace)

MemoryStore opens an autocommit connection and makes sure the target memory store exists. It raises RuntimeError if psycopg is not installed.

MemoryStore.put_message(key: str, value: dict) -> None

put_message encodes value to JSON with size-budget truncation as described below. Then it runs MEMORY PUT INTO <store> VALUES (<namespace>, <key>, <value>). MEMORY DML does not bind parameters, so string values are escaped inline.

MemoryStore.get_watermark(chat_id: str) -> str | None
MemoryStore.set_watermark(chat_id: str, watermark_iso: str) -> None

These read and write the per-chat watermark under teams_state_<chat_id>. get_watermark queries the underlying __mem_<store> table directly and filters by key on the client side. MEMORY tables do not expose mem_key filtering over the PG wire.

MemoryStore.count_messages() -> int

count_messages counts the total memory rows for this namespace. That includes messages and watermarks. Smoke tests use it to confirm that writes land.

Notes & limitations

  • Out of scope. Channel messages, meeting transcripts, and call recordings are not included. They need Graph scopes that require tenant admin consent, and the connector stays away from those.
  • System and empty messages are skipped. Joins, leaves, and bodies that strip to empty text are not indexed. They are counted under skipped.
  • Inaccessible chats are tolerated. /me/chats can list chats that return 403 or 404 on the messages endpoint. This includes orphaned meeting chats, shared-channel proxies, and chats deleted between the list and the read. These are counted and skipped. The pass does not stop.
  • Value size budget. EvolutionDB's MEMORY PUT path behaves badly with large payloads. So state.py caps a stored value at about 2500 bytes. Oversized messages are trimmed step by step. First it drops an html mirror. Then it clips text and fact with a head/tail marker. As a last resort it stores a stub that carries "truncated": true, so the message is still recorded as existing.
  • Optional PII masking. If evolutiondb-pii can be imported, message records are passed through evolutiondb_pii.integration.protect_record over the fact, text, sender, and chat fields before storage. If it is not installed, records are stored unmasked.
  • Sending is opt-in and off by default. The connector is read-only unless EVOSQL_TEAMS_SEND is set. Setting it adds the ChatMessage.Send scope at consent time. See ADR-004. There is no send code path in these modules. The flag only widens the requested OAuth scopes.
  • Privacy and policy. A 1:1 or group chat captures everyone's messages, not just yours. Some workplaces treat external archiving of colleagues' messages as a policy violation. Every Graph call is also logged in the tenant audit log. Records land in the same store the MCP server reads from. So anyone with that database and the matching MCP_USER_ID can recall the content through search_memory. You can revoke the token at any time from https://myaccount.microsoft.com/applications. The next call then gets a 401 and stops.
  • Embedded server reach. --setup and --embedded download a prebuilt evosql-server from the latest server-v* GitHub release. The binary is built for Linux x86_64, Linux arm64, and macOS arm64 only. Windows and macOS Intel do not have native binaries. Those hosts fall back to Docker.
  • Version. pyproject.toml says 0.1.3, with Development Status 4 - Beta. Note that teams_sync.__version__ still reads 0.1.2.
