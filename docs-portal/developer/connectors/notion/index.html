Notion connector

Pull every Notion page that is shared with an integration token into EvolutionDB long-term memory. It writes one memory record for each page.

What it does

evolutiondb-notion-sync is a one-way connector. It walks every page a Notion integration can see. It flattens each page's body text. It writes one record per page into an EvolutionDB memory store. This is the same record-into-memory pattern that the other *-sync connectors use. The MCP server reads these records back as agent long-term memory.

Here is the flow for each sync pass.

  • Call the Notion /search endpoint to list pages, newest first.
  • Fetch each page's top-level child blocks with /blocks/{id}/children and join their plain text.
  • Build a JSON record with fact, title, text, page_id, url, timestamps, and tags. Store it under a key for that page.
  • Track a watermark. This is the newest last_edited_time seen so far. The next pass stops as soon as it reaches pages that are already synced. This makes repeated runs incremental.
info
The connector only fetches the top-level child blocks of each page. It does not expand nested children like toggles, sub-lists, and columns. Non-text blocks like images, embeds, and tables add no body text.

Install

The distribution name on PyPI is evolutiondb-notion-sync. The import package is notion_sync. It needs Python 3.9+.

# installs the console script `evolutiondb-notion-sync`
pip install evolutiondb-notion-sync

The runtime dependencies are declared in pyproject.toml.

  • psycopg[binary]>=3.1. It connects to EvolutionDB over the PostgreSQL wire protocol.
  • evolutiondb-pii>=0.1.1. This is optional PII masking. The connector still works if it cannot import this. See the Notes.

The Notion HTTP client uses only the Python standard library urllib. It does not use a third-party Notion SDK.

Configure (env vars / auth)

The Config class reads all settings from environment variables when it is built. It loads a .env file first with os.environ.setdefault. The --env-file flag sets the path. The default is .env in the working directory. Real environment variables always win over .env entries.

VariableDefaultPurpose
NOTION_TOKEN(required)Notion internal-integration secret. An empty value raises an error.
EVOSQL_HOST127.0.0.1EvolutionDB host.
EVOSQL_PORT5433EvolutionDB PG-wire port.
EVOSQL_USERadminDatabase user.
EVOSQL_PASSWORDadminDatabase password.
EVOSQL_DATABASEevosqlDatabase name.
MCP_USER_IDdefault_userThe memory namespace that records are written under.
NOTION_MEMORY_STOREmcp_memTarget memory store name.

Follow these steps to set up Notion access.

  1. Create an internal integration at https://www.notion.so/my-integrations and copy its secret.
  2. Export it as NOTION_TOKEN in .env or the calling shell.
  3. Open the share menu on each Notion page or database you want to index. Add the integration there. The sync only sees pages that you share with it.
# .env
NOTION_TOKEN=secret_xxxxxxxxxxxxxxxxxxxx
EVOSQL_HOST=127.0.0.1
EVOSQL_PORT=5433
EVOSQL_DATABASE=evosql
MCP_USER_ID=default_user
NOTION_MEMORY_STORE=mcp_mem

Usage

The console script is the main entry point. It supports four flags. The main() function parses them.

FlagEffect
--onceRun a single sync pass and exit.
--interval NLoop forever. Sleep N seconds between passes. The minimum is 60.
--dry-runWalk Notion and count pages. Do not open a DB connection or write any records.
--env-file PATHThe .env path to load. The default is .env.

If you give neither --once nor --interval, a single pass runs. This is the same as --once.

# one pass, write to memory
evolutiondb-notion-sync --once

# walk Notion but write nothing - useful to verify the token works
evolutiondb-notion-sync --dry-run

# continuous daemon: a pass every 5 minutes
evolutiondb-notion-sync --interval 300

# load a non-default env file
evolutiondb-notion-sync --once --env-file /etc/evosql/notion.env

Each pass prints a single JSON line to stdout. On success it reports the counters pages, skipped, and errors. On failure it reports the error. The process then returns exit code 4. 

# success
{"ok": true, "pages": 12, "skipped": 0, "errors": 0}

# failure (e.g. NOTION_TOKEN unset) - exit code 4
{"ok": false, "error": "NOTION_TOKEN is not set. ..."}

The connector handles SIGINT and SIGTERM. In interval mode they set a stop flag. This interrupts the current sleep. The loop then exits cleanly after the pass that is in flight.

As a library

You can run a pass directly without the CLI.

from notion_sync.sync import Config, sync_once

cfg = Config()                  # reads env vars
counts = sync_once(cfg, dry_run=False)
print(counts)                    # {"pages": .., "skipped": .., "errors": ..}

Reference

sync module

class Config()

Reads all settings from environment variables when it is built. The attributes are token, evosql_host, evosql_port, evosql_user, evosql_pass, evosql_db, user_id, and store.

sync_once(cfg: Config, *, dry_run: bool = False) -> Dict[str, int]

Runs a single sync pass. It raises RuntimeError if cfg.token is empty. It iterates NotionClient.search_pages() and stops once it reaches the stored watermark. For each page it fetches the blocks, builds a record, optionally masks PII, and writes the record to the memory store. It moves the watermark to the newest last_edited_time seen. It returns the counters {"pages", "skipped", "errors"}. With dry_run=True it does not open a DB connection and writes nothing.

main(argv: Optional[List[str]] = None) -> int

The CLI entry point. This is the console script evolutiondb-notion-sync. It parses flags, loads the env file, installs signal handlers, and runs one pass or an interval loop. It returns the process exit code.

The module also has these internal helpers. _load_dotenv(path) is a small .env reader. _build_record(page, body_text) assembles the stored record. It returns None if the page has no last_edited_time. _page_key(page_id) builds the record key notion_page_<id-without-dashes>. _short(s, n=240) shortens the fact preview. _protect(rec, main_key) masks PII. See the Notes.

api module

class NotionClient(token: str, user_agent: str = "evolutiondb-notion-sync")

A small Notion REST client built on urllib. It talks to https://api.notion.com/v1 with the header Notion-Version: 2022-06-28. It retries once on HTTP 429 and respects the Retry-After header.

NotionClient.search_pages() -> Iterator[Dict]

Yields every page the integration can see, newest first. It pages through POST /search and follows next_cursor. The search is filtered to object = page, sorted by last_edited_time in descending order, with a page size of 100.

NotionClient.page_blocks(page_id: str, page_size: int = 100) -> List[Dict]

Returns the top-level child blocks of a page with GET /blocks/{page_id}/children. It follows pagination cursors.

page_title(page: Dict) -> str

Reads a readable title from a page object. It finds the property of type title. It returns "(untitled)" when there is none.

block_plain(block: Dict) -> str

Returns the plain text of a single block. It supports these rich-text block types: paragraph, heading_1/2/3, bulleted_list_item, numbered_list_item, to_do, quote, callout, toggle, and code. Any other block type gives an empty string.

rich_text_plain(items: Optional[List[Dict]]) -> str

Joins the plain_text of a Notion rich-text array.

NotionError(status, msg, url) is raised for HTTP errors that cannot be retried. It carries the HTTP status.

state module

class MemoryStore(host, port, user, password, database, store, namespace)

The EvolutionDB memory backend. It mirrors the backend used by the sibling connectors. It connects with psycopg in autocommit mode. On init it tries to run CREATE MEMORY STORE. All operations go through a small retry with backoff. It tries up to 3 times. It rebuilds the connection on OperationalError and InterfaceError.

MemoryStore.get_watermark_iso() -> Optional[str]

Reads the stored last_edited_time watermark for this namespace. The key is notion_state_latest.

MemoryStore.set_watermark_iso(iso: str) -> None

Persists the watermark using MEMORY PUT INTO <store>.

MemoryStore.put_record(key: str, record: Dict) -> None

Serializes record to JSON. It writes it under (namespace, key) with MEMORY PUT INTO <store>.

Notes & limitations

  • One-way, page-level only. The connector reads from Notion and writes to EvolutionDB. It never writes back to Notion. It indexes pages. These are objects of type page. It does not list databases as records. It only flattens top-level page blocks into the body text.
  • Incremental via watermark. /search returns newest pages first. The loop stops at the stored watermark. An interrupted or repeated run starts again from the last synced page. It does not walk everything again. The first run against a full workspace walks all visible pages.
  • PII masking is optional. If evolutiondb-pii is installed, the fields fact, title, and text pass through protect_record before storage. Any companion records are written too. If the import fails, records are stored without masking. There is no warning. The masking step is just skipped.
  • SQL value escaping. state.py builds MEMORY PUT and SELECT statements by string interpolation. It uses a local _e() helper. This helper strips control characters and doubles single quotes. The namespace, store, and key values flow through it. Treat NOTION_MEMORY_STORE and MCP_USER_ID as trusted configuration, not as untrusted input.
  • Interval floor. --interval values below 60 are raised to 60 seconds.
  • Version note. pyproject.toml declares version 0.1.1. notion_sync/__init__.py still reports __version__ = "0.1.0".
warning
The integration only sees pages that you share with it. A sync pass may report "pages": 0 while your token is valid. The most common cause is that no pages have been shared with the integration in Notion's share menu.

See also the MCP server that reads these memory records. See the wider EvolutionDB wiki too.
