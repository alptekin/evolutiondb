26. Encryption (TDE)

EvolutionDB encrypts every data page on disk with AES-256. This protects the database file when it sits on disk. Inside the running server's buffer pool, the page contents stay in cleartext.

Transparent Data Encryption (TDE) works below the SQL layer. Tables, indexes, the system catalog, and the write-ahead log are all stored as 4 KB pages. When TDE is active, those pages are encrypted as they go to disk. They are decrypted as they come back. The SQL you write and the rows you read do not change. From SQL the only visible effect is that the on-disk file evosql.db and its WAL cannot be read without the passphrase.

This chapter describes exactly what EvolutionDB implements. EvolutionDB uses a two-layer key hierarchy called MEK and DEK. It uses AES-256 page encryption in CTR mode. It wraps keys with AES-256-GCM. It derives key material from the passphrase with PBKDF2-SHA256. This chapter also compares EvolutionDB with PostgreSQL and other TDE designs where that is helpful. The comparison is honest. It does not claim features that do not exist.

Build requirement. TDE is a compile-time feature. It is guarded by the EVO_ENCRYPTION macro. The server's TLS=1 build sets this macro for you. That build also pulls in OpenSSL's EVP API for AES. A server built without EVO_ENCRYPTION turns the whole crypto path into no-op stubs. In that case the database is always plaintext. The encryption controls in this chapter then have no effect.

26.1. Overview

EvolutionDB's TDE works at the page level and covers the whole database. There is no per-table, per-column, or per-tablespace encryption. When a database is created with a passphrase set, every page except page 0 is encrypted. That choice is permanent for the life of the file. This matches the granularity of cluster-wide TDE designs. It differs from column-level encryption such as PostgreSQL's pgcrypto. That tool encrypts single values inside pages that stay otherwise plaintext.

What is encrypted
All pages numbered 1 and above. This covers heap pages, B+ tree index pages, slotted-page records, the system catalog, the free list, and WAL page images written during replay.
What is not encrypted
Page 0 is the FileHeader. It is always plaintext. It holds the file magic, version, page size, and catalog roots. It also holds the encryption metadata itself, such as the salt, the wrapped DEK, and the IV prefix. The server needs this metadata to start decryption.
Where plaintext lives
In the buffer pool. Pages are held decrypted in memory. Encryption happens only when a page is flushed to disk. Decryption happens only when a page is read from disk. The query engine never sees ciphertext.
Threat model
Protection at rest. TDE defends against theft of the database file, the WAL, or the physical disk. It does not protect against an attacker who can read the server's memory. It does not protect against an attacker who holds the passphrase.

The buffer pool stores plaintext and the engine works on plaintext. So TDE is invisible to queries and adds no rules to your SQL. A SELECT over an encrypted database returns the same bytes as one over a plaintext database.

26.2. Key hierarchy: MEK and DEK

EvolutionDB uses a two-layer key design in the style of Oracle TDE.

KeySizeDerived / generatedStored where
MEK, Master Encryption Key256 bitsDerived from the passphrase and a random 16-byte salt with PBKDF2-SHA256Never stored. It is recomputed in memory at startup. It is wiped right after use.
DEK, Data Encryption Key256 bitsGenerated once from the OS CSPRNG when the database is first createdStored wrapped in the plaintext FileHeader on page 0. The MEK encrypts it.

The DEK is the key that encrypts pages. The MEK exists only to protect the DEK. This extra layer makes rekeying cheap. Changing the passphrase changes the MEK. The new MEK re-wraps the same DEK. The gigabytes of already-encrypted pages never need rewriting.

The passphrase comes from the EVOSQL_ENCRYPTION_KEY environment variable. The server reads it with getenv() at startup and never saves it. If the variable is unset or empty, encryption is not initialized.

# Key flow at a glance
EVOSQL_ENCRYPTION_KEY  ──PBKDF2-SHA256(salt,100000)──▶  MEK (256-bit)
MEK  ──AES-256-GCM unwrap──▶  DEK (256-bit, from FileHeader)
DEK  ──AES-256-CTR──▶  page ciphertext  (pages 1..N)

26.2.1. MEK derivation (PBKDF2-SHA256)

The MEK is derived by stretching the passphrase with PBKDF2-SHA256. EvolutionDB ships its own pure-C code for SHA-256, HMAC-SHA256, and PBKDF2. The KDF needs no external library. The code follows FIPS 180-4, RFC 2104, and RFC 2898.

PRF
HMAC-SHA256
Iterations
PBKDF2_ITERATIONS = 100000. This value is fixed and compiled in.
Salt
16 random bytes. Each database gets its own salt. It is stored in the FileHeader as encryption_salt[16].
Derived key length
32 bytes (256 bits)

The same PBKDF2 code also backs user-password hashing. See Authentication. There the hashes are stored in the format pbkdf2:100000:<hex-salt>:<hex-hash>. For TDE the derived bytes are used straight as the MEK. They are never written to disk.

Honest comparison. 100,000 PBKDF2-SHA256 iterations is a fixed, moderate work factor. It is below the project's own user-auth guidance. The security standard asks for 600,000 or more iterations for password storage. It is also well below modern memory-hard KDFs such as Argon2 or scrypt. PBKDF2 cannot match those against GPU or ASIC attackers. The best fix is a long passphrase with high entropy. With a strong passphrase the iteration count is not the weak point.

26.2.2. DEK wrapping (AES-256-GCM)

The 32-byte DEK is wrapped under the MEK with AES-256-GCM. To wrap means to encrypt. The result is a 48-byte blob. It holds 32 bytes of ciphertext and then a 16-byte GCM authentication tag. This blob is stored in the FileHeader as wrapped_dek[48].

The GCM tag also works as a passphrase verifier. At startup the server derives the MEK from the passphrase and tries to unwrap the DEK. If the passphrase is wrong, the MEK is wrong. The GCM tag check fails. The server treats that failure as a fatal "wrong encryption key" error. It does not quietly produce garbage.

FATAL: Wrong encryption key for this database.
The EVOSQL_ENCRYPTION_KEY does not match.

The wrapping IV is the fixed 12-byte string EVO_WRAP_KEY. Reusing a fixed IV with GCM is normally dangerous. Here it is safe. Each database has its own random salt. That salt produces a unique MEK. The key and IV pair is therefore never reused across databases that share the same DEK.

26.3. Page encryption (AES-256-CTR)

Pages are encrypted with AES-256 in CTR mode using the DEK. CTR is a stream-cipher mode. It was picked on purpose for two reasons that matter to a page store.

Zero storage overhead
CTR ciphertext has the same length as the plaintext. A 4096-byte page encrypts to exactly 4096 bytes. So the on-disk layout, page numbering, and offsets are the same with or without TDE. There is no per-page authentication tag.
Symmetric operation
In CTR mode encryption and decryption are the same operation. EvolutionDB uses one routine for both directions.

26.3.1. Per-page IV construction

Each page gets its own 16-byte CTR IV. The IV is built from a random per-database prefix and the page number.

IV (16 bytes) = iv_prefix[8] || page_no_LE[4] || zeros[4]
               └ random, per-DB ┘  └ little-endian ┘ └ counter ┘

The 8-byte iv_prefix is generated once with the OS CSPRNG when the database is created. It is stored in the FileHeader as page_iv_prefix[8]. The prefix never changes, even when the passphrase is rotated. The page number is also part of the IV. This means no two pages share an IV. CTR mode requires exactly that.

Limitation. CTR gives confidentiality, not integrity. GCM is used for the DEK wrap and has an authentication tag. CTR does not. So EvolutionDB does not detect tampering or bit-rot of encrypted pages. A flipped ciphertext bit quietly flips the matching plaintext bit. TDE here defends against reading the file. It does not defend against an attacker who can change it. Most zero-overhead page-level TDE schemes make the same trade-off. Use filesystem or volume integrity if you need tamper detection. You can also add checksums at a higher layer.

26.3.2. Page 0 stays plaintext

Page 0 is the FileHeader. It is never encrypted. Both the encrypt and decrypt entry points return early when page_no == 0. Page 0 holds the startup metadata: the magic, the salt, the wrapped DEK, and the IV prefix. The server must read this data before it can derive any key. The header does show that a database is encrypted and it shows the salt, the wrapped DEK, and the IV prefix. None of these reveal the DEK without the passphrase.

EvolutionDB does not encrypt the page_no == 0 region. PostgreSQL's at-rest encryption proposals keep some control data readable for the same reason. You cannot decrypt the thing that tells you how to decrypt.

26.3.3. Where encryption happens in the I/O path

All page I/O flows through the Clock-Sweep buffer pool. See Buffer Pool. Encryption and decryption happen right at the disk boundary.

PathAction
Buffer pool flush (dirty page → disk)The page is copied into a scratch buffer. It is encrypted in place. Then it is written. The in-memory page stays plaintext.
Buffer pool fault (disk → buffer)The page is read. Then it is decrypted in place if encryption is enabled and page_no > 0.
WAL replay (recovery)Page images written to the data file during replay are encrypted before pwrite. Page 0 is the exception.
Replication shippingPages are encrypted on the same code path before they are sent downstream.

Every one of these checks both pcrypt_is_enabled() and page_no > 0. So a plaintext database and page 0 always skip the cipher.

26.4. Enabling encryption

TDE is decided when the database is created. It is controlled only by the EVOSQL_ENCRYPTION_KEY environment variable. There is no CREATE DATABASE ... ENCRYPTED clause. There is no SQL statement that turns encryption on for an existing plaintext file.

26.4.1. A new (empty) database

When the page manager creates a fresh evosql.db and EVOSQL_ENCRYPTION_KEY is set, it initializes encryption. It generates a random DEK, a random salt, and a random IV prefix. It derives the MEK, wraps the DEK, and records all of that in the FileHeader with encryption_enabled = 1.

# docker-compose.yml - set a strong custom key BEFORE first start
services:
  evosql:
    environment:
      EVOSQL_ENCRYPTION_KEY: "<a long, random, high-entropy passphrase>"
# Or locally, when the .db file does not yet exist:
export EVOSQL_ENCRYPTION_KEY="<a long, random, high-entropy passphrase>"
./adaptor/evosql-server

On success the server logs [TDE] Encryption initialized for new database. From that point the file is encrypted permanently. There is no in-place "decrypt back to plaintext" operation.

26.4.2. An existing encrypted database

For an existing file, the server reads encryption_enabled from page 0. If it is set, the server derives the MEK from the supplied passphrase and unwraps the DEK. The table below shows all startup outcomes.

File encryption_enabledEVOSQL_ENCRYPTION_KEYResult
1 (encrypted)correct passphraseDatabase unlocked, logs [TDE] Encryption key verified, database unlocked
1 (encrypted)wrong passphraseFATAL, GCM tag verification fails; server refuses to start
1 (encrypted)unset / emptyFATAL, "Database is encrypted but EVOSQL_ENCRYPTION_KEY.. not set"
0 (plaintext)setWarning, key ignored; existing plaintext files are not retroactively encrypted; continues without encryption
0 (plaintext)unsetNormal plaintext operation

Default-key warning. The shipped docker-compose.yml sets a built-in fallback passphrase so the demo "just works." That key is publicly known. When the server detects it, it prints a prominent warning to stderr. Always override EVOSQL_ENCRYPTION_KEY with your own secret before creating a production database. A database created under the default key is no more protected than a plaintext one.

26.5. Rekeying (passphrase rotation)

The passphrase only protects the MEK. The MEK only wraps the DEK. Rotating the passphrase is therefore a fast metadata operation. A new salt is generated. A new MEK is derived from the new passphrase. The same DEK is re-wrapped under the new MEK. The IV prefix is left unchanged. No data pages are re-encrypted. Only encryption_salt and wrapped_dek in the FileHeader change.

Implementation status. The rekey primitive (pcrypt_rekey) exists in the engine. In EvolutionDB 3.0 it is not exposed through SQL. There is no ALTER SYSTEM ... REKEY statement. No command-level surface in the parser or protocol layer drives it. Passphrase rotation is therefore an internal capability. It is not a user-facing operation in this release. This is a deliberate limitation, not an advertised feature.

26.6. Key material in memory

Sensitive buffers are scrubbed after use. They are not left on the stack or heap. The MEK is wiped immediately after it wraps or unwraps the DEK. SHA-256 contexts, HMAC pads, and PBKDF2 intermediates are wiped at the end of each operation. On shutdown the DEK and IV prefix are wiped from the module's static state.

Scrubbing uses evo_secure_wipe(). It is hardened against the compiler optimizing away a "dead" memset (CWE-14). It prefers platform primitives where available. On Windows it uses SecureZeroMemory(). When the C11 Annex K bounds-checking interface is present, it uses memset_s(). Otherwise it falls back to a volatile function-pointer indirection. The optimizer cannot prove that call is side-effect-free, so it keeps it.

Password verification also uses a constant-time comparison to avoid leaking information through timing. These protections cover the running process. While the server runs, the DEK and decrypted pages must stay in memory. That is the boundary of the at-rest threat model.

26.7. Randomness source

All key material comes from the operating system CSPRNG via crypto_random_bytes(). This includes the DEK, the MEK salt, and the IV prefix.

Linux / macOS
Reads from /dev/urandom
Windows
CryptGenRandom via CryptAcquireContext

If obtaining randomness fails, encryption initialization is aborted. The server does not fall back to a weak source.

26.9. On-disk metadata layout

The encryption metadata lives in the plaintext FileHeader (page 0). It is 73 bytes total.

encryption_enabled (1 byte)
0 = plaintext database, 1 = AES-256 TDE active
encryption_salt[16]
PBKDF2 salt for MEK derivation
wrapped_dek[48]
DEK encrypted by the MEK. It is 32 bytes of ciphertext followed by a 16-byte GCM tag.
page_iv_prefix[8]
Fixed CTR IV prefix; unchanged across rekeys

None of these fields reveals the DEK or any plaintext. Without the correct passphrase the wrapped DEK cannot be unwrapped. The GCM verification fails and the pages cannot be decrypted.

26.10. Comparison with PostgreSQL and the SQL standard

These notes give an honest mapping for readers coming from PostgreSQL.

Cluster-wide TDE
Core PostgreSQL has no built-in transparent cluster encryption. At-rest encryption is typically delegated to filesystem or volume encryption, or to forks and extensions. EvolutionDB ships whole-database page TDE in the engine itself.
Column-level encryption
PostgreSQL's pgcrypto encrypts individual column values. Pages stay otherwise readable and SQL changes are required. EvolutionDB has no column-level encryption. It encrypts whole pages and is invisible to SQL.
Key management
EvolutionDB's MEK/DEK split resembles Oracle TDE's master-key / table-key model. The passphrase comes from an environment variable. There is no external KMS, HSM, or keystore integration.
SQL standard
The SQL standard does not define transparent storage encryption. TDE here is an implementation-level feature with no standard syntax. That is why it is configured by environment variable rather than by DDL.

Notes & limitations.

  • Compile-time gated. Requires a build with EVO_ENCRYPTION (the TLS=1 build). Otherwise the crypto path is a no-op and the database is plaintext.
  • Permanent at creation. Encryption is set when the database file is first created. You cannot encrypt an existing plaintext file in place, nor decrypt an encrypted file back to plaintext, through any supported operation.
  • No SQL rekey. The rekey primitive exists internally but has no SQL/command surface in 3.0; passphrase rotation is not a user-facing feature.
  • Confidentiality only. Page encryption uses CTR with no per-page authentication tag. There is no tamper detection on data pages.
  • KDF work factor. PBKDF2 is fixed at 100,000 SHA-256 iterations; rely on a strong passphrase, not the iteration count.
  • Default key is public. The shipped compose default passphrase is well known. Override it before creating any database you care about.
  • Memory is the boundary. While the server runs, the DEK and decrypted pages live in memory. TDE protects the file at rest, not a live-memory attacker who already holds the key.
