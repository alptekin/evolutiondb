29. Replication & Sharding

How EvolutionDB copies a database across nodes. It uses physical WAL streaming. It picks a leader with a Raft consensus core. It can also spread one table over several machines with hash or range sharding and automatic query routing.

EvolutionDB has three clustering parts that work together. They all live in the adaptor layer. They are all opt-in at runtime. The first part is physical replication in adaptor/replication.c. The second is Raft leader election in adaptor/raft.c. The third is a distributed query engine with sharding in adaptor/distributed.c and adaptor/shard.c. A server started without the right flags runs as a normal single node. None of these paths run in that case. They also need POSIX sockets and pthreads. On Windows with MinGW these files compile to stubs. Only the default embedded single-node flow works there.

Scope of this chapter. This page describes only what the engine does today. Where EvolutionDB differs from PostgreSQL or the SQL standard, we say so clearly. Several of these parts are kept simple on purpose. Raft is heartbeat-only. The WAL ships page images. SQL key extraction uses no regex. The limits are listed in the Notes callout.

29.1. Overview & topology

A node takes one of three replication roles. The role is set at startup. You can see it with SHOW REPLICATION STATUS.

primary (REPL_ROLE_PRIMARY = 0)
Accepts DML. It runs the WAL sender. It streams WAL records to the replicas that connect. This is the default role.
replica (REPL_ROLE_REPLICA = 1)
Connects to a primary. It replays the streamed WAL into its own data file. It serves read traffic. repl_is_replica() blocks the write path.
witness (REPL_ROLE_WITNESS = 2)
Participates in Raft voting but stores no data and rejects SELECTs. It breaks ties in an even-sized cluster without needing a full data copy.

The role is chosen by the --role flag or the EVOSQL_REPLICATION_ROLE environment variable. A late-arriving replica start always wins over a stored pin. repl_get_role() returns replica as soon as the receiver is running. A Raft election can flip the role at runtime (see 29.3).

PostgreSQL uses separate primary_conninfo, standby.signal, and a recovery.conf file. EvolutionDB configures everything through a small set of CLI flags and environment variables. They are all read at boot in adaptor/main.c.

29.2. Physical (WAL streaming) replication

Physical replication ships raw WAL page images from a primary to one or more replicas. The unit of replication is a WAL record. Its layout is [lsn:4][page_no:4][page_len:2][timestamp:8]…[page bytes][CRC]. EvolutionDB's WAL stores full page images (4 KB pages). A replica reapplies a record by writing the page image to the same byte offset (page_no × 4096) in its local data file. It also invalidates any cached copy in its buffer pool. There is no logical redo or operator replay. This approach is closer to PostgreSQL's full-page-write streaming than to its record-level redo.

29.2.1. The streaming protocol

Replication uses a private line protocol over TCP. It is separate from the PostgreSQL wire protocol. The replica drives the handshake.

# Replica → Primary, on connect:
REPLICATE <last_lsn>\n
# optional, only when the primary has auth configured:
AUTH <user> <password>\n
# Primary → Replica reply to AUTH:
OK   | ERR <reason>

After the handshake the primary streams framed messages. Each message begins with a single type byte.

'W', REPL_MSG_WAL_DATA
Followed by a 4-byte length and the full WAL record. The replica parses the embedded record header, writes the page, and advances its slot LSN. Every 100 records it also persists the slot file and sends an ACK.
'H', REPL_MSG_HEARTBEAT
Keepalive, sent on a 5-second interval when no WAL is flowing. The replica answers with an ACK carrying its current confirmed LSN. This keeps synchronous-commit waiters and status views fresh.
'A', REPL_MSG_ACK
Replica to primary. A 5-byte frame ('A' + uint32 confirmed LSN). The primary's sender drains ACKs between batches and on every heartbeat tick. It performs byte-level resync if it sees an unexpected byte.

The sender first streams everything in evosql.wal after the requested start_lsn (catch-up). Then it blocks on a condition variable waiting for repl_notify_new_wal() to fire as new records are flushed.

29.2.2. Setting up a replica

Bring up a primary with the WAL sender listening, then bootstrap a replica from a base backup and point it at the primary.

# 1. Primary - start with the WAL sender on port 9968
./evosql-server --role primary --replication-port 9968

# 2. Take a base backup into a fresh data directory (runs and exits)
./evosql-server --base-backup /var/lib/evosql/standby1

# 3. Replica - replay the backup, then stream from the primary
EVOSQL_PRIMARY_HOST=primary.internal \
EVOSQL_PRIMARY_PORT=9968 \
./evosql-server --role replica --replica primary.internal:9968

The replica auto-reconnects with exponential backoff (1 s, doubling up to a 30 s cap). It resumes from the LSN persisted in its slot file root/evosql.slot. A dropped link or a restarted primary recovers without operator action.

29.2.3. Base backups

--base-backup PATH flushes the buffer pool, fsyncs the WAL to pin a consistent LSN, and writes a directory bundle.

evosql.db
Full data file, consistent at WAL-fsync time.
evosql.wal
Current WAL, copied best-effort for replay past the pinned LSN.
BACKUP_LABEL
Metadata: start_lsn, timestamp, format=v1.

If PATH names a regular file rather than a directory, the engine falls back to a legacy single-file copy of evosql.db for backward compatibility. This is far simpler than PostgreSQL's pg_basebackup. There is no tablespace mapping, no streamed WAL during the copy, and no manifest checksums.

29.2.4. Authentication & TLS

Replication auth is opt-in. When EVOSQL_REPLICATION_USER/_PASSWORD are set, the primary requires the AUTH line. It validates the password against the catalog using crypto_verify_password with PBKDF2-SHA256. This is the same path as client login. A missing or malformed AUTH, or bad credentials, is rejected before any WAL is streamed. Setting EVOSQL_REPLICATION_TLS to a truthy value upgrades the transport with the server's TLS context. It reuses the same cert/key as PG-wire TLS. The replica must enable TLS to match.

29.2.5. Synchronous commit

By default replication is asynchronous. A commit returns as soon as the primary has flushed its own WAL. Setting EVOSQL_SYNC_COMMIT to a truthy value makes the commit path call repl_sync_commit(lsn, timeout_ms). That call blocks until a majority of active replica slots have ACK'd confirmed_lsn ≥ lsn.

  • Majority is ceil(active / 2). 1 replica needs 1 ACK. 2 need 1. 3 need 2.
  • With no active replicas the call returns success immediately. Sync degrades gracefully to async rather than stalling the only writer.
  • The default wait is REPL_SYNC_COMMIT_DEFAULT_MS = 2000 ms. On timeout the caller falls back to async commit (returns -1).

Compare this with PostgreSQL. There is no per-transaction synchronous_commit GUC, no named synchronous_standby_names quorum sets, and no remote_apply level. EvolutionDB offers a single server-wide majority-ACK switch.

29.2.6. Monitoring

Status is surfaced through SHOW and through emulated PostgreSQL catalog views. Existing tooling like DBeaver and psql can read it.

SHOW REPLICATION STATUS;
-- role | replica_id | my_lsn | confirmed_lsn | lag_bytes | active | last_seen_us

SHOW REPLICATION LAG;
-- replica_id | lag_bytes  (one row per active slot)

SELECT * FROM pg_stat_replication;
-- one row per active replica slot (Task 97 v1 emulation)

The engine tracks up to REPL_MAX_SLOTS = 16 replica slots. Each slot keeps a replica_id (the peer's ip:port), its highest confirmed_lsn, an active flag, and a last_seen microsecond timestamp. On disconnect a slot is marked inactive but keeps its LSN. This way catch-up resumes cleanly on reconnect. lag_bytes is reported as my_lsn − confirmed_lsn.

29.2.7. Logical change capture (CDC)

Beyond physical streaming, EvolutionDB can emit row-level change events by decoding WAL page images. It compares each tuple's MVCC header inside the slotted page.

Condition on tupleEvent op
xmin > 0, xmax = 0, not HEAP_ONLY'I', INSERT
xmin > 0, xmax = 0, HEAP_ONLY flag (0x08)'U', UPDATE (new version)
xmax > 0, not HOT_UPDATED (0x04)'D', DELETE

On the primary, decoding is filtered by the committing transaction's XID. This prevents a page that comes back through the flush path from re-emitting unchanged rows as duplicate INSERTs. Consumers connect to the CDC server (--cdc-port) and receive one JSON object per line (NDJSON, Debezium-style op field).

{"event":"ready"}
{"op":"I","table":"orders","table_id":42,"xid":1007,"pk":"","ts":1718200000000000}

Up to CDC_MAX_SUBSCRIBERS = 32 consumers fan out from the same decode pass. This is page-image decode, not a published logical-decoding plugin API. There is no REPLICA IDENTITY and no column-value payload. Events carry only op, table, XID, and a best-effort primary-key string. The stream is one-way once established. The related declarative CREATE SUBSCRIPTION surface is documented separately.

29.3. Raft consensus

Starting a node with --cluster and --node-id activates the Raft core. It provides automatic leader election and failover. EvolutionDB's Raft is a leader-election + heartbeat implementation. Actual log shipping reuses the WAL streaming infrastructure of 29.2 rather than carrying entries inside Raft messages.

./evosql-server --cluster node1:9968,node2:9968,node3:9968 --node-id 0

29.3.1. Roles, terms & election

FOLLOWER (0)
Default. Resets its randomized election timer on every valid heartbeat.
CANDIDATE (1)
Election timeout elapsed. It increments the term, votes for itself, and requests votes from peers.
LEADER (2)
Won a majority vote. Sends periodic heartbeats and accepts writes.

Timing constants are defined in adaptor/raft.h. Heartbeat fires every 150 ms. The election timeout is randomized in [300, 500) ms to avoid split votes. A candidate wins with g_num_nodes / 2 + 1 votes. A vote is granted only if the candidate's term is at least as high and the follower has not already voted this term. The candidate's last_lsn must also be at least as high as the follower's. This up-to-dateness check prevents a stale node from becoming leader. The maximum cluster size is RAFT_MAX_NODES = 7.

Persistent Raft state includes current_term and voted_for. It is written atomically to root/raft.state via temp-file-plus-rename with an fsync. A restart cannot grant a second vote in the same term. Every 10000 AppendEntries the node snapshots its high-water LSN and term to root/raft.snapshot. There is no in-memory log to truncate in this design. The snapshot is mainly an operator artifact and a placeholder for future log-based compaction.

29.3.2. Failover & the replication glue

A role transition fires a callback registered by repl_install_raft_glue(). When Raft elects this node LEADER, the glue clears replica state and starts the WAL sender on the bound replication port (idempotent). When Raft moves it to FOLLOWER, the role is set back to replica. This is how a follower is promoted automatically on primary failure. A manual repl_promote() performs the same follower-to-standalone-master transition outside Raft.

29.3.3. Split-brain fencing

Fencing is term-based. Any node that receives a message with a higher term immediately steps down to follower, resets its vote, and persists the new term. A leader needs a current-term majority. A stale leader's term is superseded the moment a new election completes. This means two leaders cannot both retain write authority. repl_check_leader_lease() exposes a lease hook. In the current build it is simplified. Raft term step-down is the real fencing mechanism.

29.3.4. Online membership & cluster status

Members can be added or removed without a restart through repl_add_member() / repl_remove_member() (delegating to raft_add_member / raft_remove_member). Cluster health is visible from SQL.

SHOW CLUSTER STATUS;
-- role | term | leader_id | my_lsn

Honesty note on Raft. This is a pragmatic subset of the Raft paper. AppendEntries carries only a heartbeat and a last-LSN advertisement. There is no replicated command log, no log-index/commit-index matching, and no per-entry conflict resolution. Durable agreement comes from the separate WAL stream and the majority-ACK sync-commit path. Treat it as automated election + failover, not a general-purpose replicated state machine.

29.4. Distributed query engine

The distributed engine makes any node able to serve any query. The system catalog is replicated to every node via the WAL stream, but data pages live only on the table's owner node. The catalog column owner_node_id records placement. A query touching a remote table is forwarded to its owner over the standard PostgreSQL wire protocol.

Routing happens in dist_try_route() before local execution. It extracts the target table name from the statement. It understands SELECT … FROM, INSERT INTO, UPDATE, DELETE FROM, DROP TABLE, TRUNCATE, and RECLAIM TABLE. It resolves the table name in the catalog and then does one of three things.

  • if the table is sharded, hands off to the shard router (29.5);
  • if owner_node_id is 0 or this node, executes locally;
  • otherwise forwards the raw SQL to the owner and relays the result set back.

Forwarding opens a pooled connection per peer (one persistent socket each). It performs a trusted internal PG startup as user _dist_internal with no password. It sends the query as a Simple Query ('Q') and reassembles RowDescription/DataRow/CommandComplete into a local ResultSet. Connection failures surface as SQLSTATE 08001 (cannot connect) or 08006 (connection lost).

29.4.1. Placing and moving tables

A table's home node is chosen at creation with ON NODE. The catalog layer records the placement. You can change it afterward with MOVE TABLE.

CREATE TABLE events (id INT, payload TEXT) ON NODE 2;

MOVE TABLE events TO NODE 1;

MOVE TABLE (dist_move_table) scans every row on the current owner. It replays them as INSERT statements forwarded to the destination. Then it destroys the local primary-key tree, secondary indexes and heap pages. It also updates the catalog fields owner_node_id, PK root, and heap page. If the mover is not the current owner, it forwards a MOVE TABLE command to the owner instead. You can inspect placement with these commands.

SHOW NODES;            -- node_id | host | pg_port | role (SELF/PEER)
SHOW TABLE PLACEMENT;  -- table_name | schema | owner_node_id | location (LOCAL/REMOTE)

29.5. Sharding

Sharding splits a single logical table across several owner nodes. A shard is declared in CREATE TABLE via a table option. Each shard records its shard_ordinal and owner_node_id in the catalog. Range shards also record a range_bound.

29.5.1. Hash sharding

CREATE TABLE sessions (
    sid   VARCHAR(64) PRIMARY KEY,
    data  TEXT
) SHARD BY HASH(sid) SHARDS 4;

A key is routed by hash_fnv1a(key) % shard_count. This uses the FNV-1a hash also used by the engine's hash indexes. Hash routing always resolves to exactly one shard ordinal.

29.5.2. Range sharding

CREATE TABLE ledger (
    acct  VARCHAR(32),
    amt   NUMERIC
) SHARD BY RANGE(acct) (
    SHARD s_lo VALUES LESS THAN 'M'        ON NODE 0,
    SHARD s_hi VALUES LESS THAN MAXVALUE   ON NODE 1
);

Range routing walks the shard descriptors and picks the first shard whose range_bound is greater than the key (string comparison). A catch-all bound declared with MAXVALUE is stored as an empty bound that always matches. Bounds are compared as strings. There is no numeric or collation-aware range partitioning. Use the query below to inspect shards.

SHOW SHARDS ledger;
-- shard_name | ordinal | owner_node_id | range_bound | shard_type | shard_key

29.5.3. Shard pruning & scatter-gather

When the distributed engine is enabled, shard_try_route() inspects the statement for the shard key.

  • Pruning. For an INSERT it reads the shard-key value from the column/VALUES list. For SELECT/UPDATE/DELETE it reads a shard_key = value equality from the WHERE clause. The matching shard is resolved. If the shard is remote, the whole statement is forwarded to that shard's owner. This is a single-shard query.
  • Scatter-gather. If no shard key is present, the statement is broadcast to every shard. Each local shard runs in-process. Each remote shard is forwarded over PG wire. SELECT results are merged (column metadata taken from the first non-empty result, rows appended) and the command tag is rewritten as SELECT <total_rows>. UPDATE/DELETE without a shard key also scatter to all shards.

In standalone (non-distributed) mode every shard is local and routing is skipped. The ordinary executor handles the statement.

Notes & limitations.

  • Platform. Replication, Raft, distributed routing and sharding are POSIX-only. On Windows (MinGW) these files compile to stubs. Only single-node embedded mode is available.
  • Page-image WAL. Physical replication ships 4 KB page images, not logical redo. Primary and replica must share the same page layout and TDE settings. The replica re-encrypts pages on write if TDE is enabled.
  • Shard-key extraction is textual. shard.c parses the shard key from raw SQL with a hand-written scanner. It handles single quotes and simple WHERE col = value equality only. Parameterized queries, expressions, functions, IN lists, joins, or multi-row INSERT defeat pruning and fall back to scatter-gather or local execution. Cross-shard JOINs, aggregates and ORDER BY are not reconciled. Scatter-gather concatenates rows without re-sorting or re-aggregating.
  • Raft is election-only. No replicated command log, no log-matching, and a leader lease that is effectively term-based. Durability of writes comes from the WAL stream and sync-commit, not from Raft log agreement.
  • Sync commit is server-wide. A single EVOSQL_SYNC_COMMIT switch with a 2 s timeout and majority-of-active-slots quorum. There is no per-transaction control and no named quorum sets like PostgreSQL's synchronous_standby_names.
  • CDC payload is minimal. Events carry op, table, xid, a best-effort PK string and a timestamp, not full row images. The stream is one-way NDJSON.
  • Internal forwarding is trusted. Cross-node query forwarding authenticates as _dist_internal with no password. The distributed port must not be exposed to untrusted networks.
  • Cluster ceilings. RAFT_MAX_NODES = 7, DIST_MAX_NODES = 7, REPL_MAX_SLOTS = 16, CDC_MAX_SUBSCRIBERS = 32.
