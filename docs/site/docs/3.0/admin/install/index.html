22. Installation

This page shows how to get a running EvolutionDB 3.0 server. You can build the three executables from source with a C toolchain. You can also pull the prebuilt container image and start it with docker compose.

EvolutionDB ships as three separate executables. They are all built from one source tree.

evolution/evosql
The standalone core SQL engine. It holds the parser, storage, and executor. You build it with make evolution. It is useful for embedded or single-user tests. It has no network listener.
adaptor/evosql-server
The dual-protocol server. It links the engine object files directly. It does not link a shared library. It serves the PostgreSQL wire protocol and the native EVO text protocol over TCP. Run this in production. You build it with make adaptor.
cli/evosql-cli
The interactive command-line client. It is based on GNU readline. You build it with make cli.

Compared with PostgreSQL. PostgreSQL ships a configure/make build with many optional contrib modules. It also has a separate initdb step to lay out a data directory. EvolutionDB has neither of these. There is no configure script, no initdb, and no cluster directory. The server creates and manages a single page file on first start in its working directory. That file is evosql.db and it uses 4 KB pages.

22.1. Requirements

The build tools and library needs are kept small on purpose. The exact set depends on which component you build. It also depends on whether you turn on TLS.

ComponentBuild toolsLibraries
Core engine (evosql)GCC, Makeliblz4
Server (evosql-server)GCC, Make; Bison + Flex only to regenerate the parserliblz4; libssl/libcrypto when TLS=1
CLI client (evosql-cli)GCC, Makelibreadline; libssl/libcrypto when TLS=1

Here are notes on each library. They come from the real Makefiles.

liblz4
It is always linked into the engine. It backs the TOAST overflow store. The store keeps large text rows from taking over the disk. It is a small dependency of about 150 KB. You can get it as liblz4-dev on Debian and Ubuntu. You can get it as lz4 on Homebrew and Alpine apk. You can get it as lz4-devel on RHEL and Fedora.
libreadline
Only the CLI client needs it. It is used for line editing and history. The history file is ~/.evosql_history. It holds up to 500 lines.
OpenSSL (libssl / libcrypto)
It is optional. You need it only when you build with TLS=1 for the server or TLS=1 for the CLI. The server uses it for the TLS transport and for AES-256 Transparent Data Encryption.
Bison + Flex
You need them only to run make generate. The generated parser and lexer files are committed to the repository. These are evoparser.tab.c/h and lex.yy.c. So an ordinary build does not need Bison or Flex.

22.2. Building from source

The top-level Makefile sits in the repository root. It drives the three sub-project Makefiles. The order matters. The engine is built first. The server and CLI link against its object files.

# Build all three executables (evolution, adaptor, cli)
make

# Or build a single component
make evolution
make adaptor
make cli

# Clean every object file and executable
make clean

When make works, it prints the paths of the three binaries it built.

====================================
  Build complete!
    evolution/evosql
    adaptor/evosql-server
    cli/evosql-cli
====================================

22.2.1. Enabling TLS and encryption

TLS support is off by default. Pass TLS=1 to compile the server with the OpenSSL transport and the page-encryption code path. This sets the EVOSQL_TLS and EVO_ENCRYPTION compile-time flags.

# Server with OpenSSL TLS + Transparent Data Encryption
make adaptor TLS=1

# CLI with TLS so it can connect to a TLS-enabled server
make -C cli TLS=1

TLS is a build-time decision. In PostgreSQL, ssl = on is a runtime setting in postgresql.conf on a single binary. EvolutionDB works differently. It compiles the TLS code in or out. A server built without TLS=1 cannot serve TLS at all. Environment variables do not change this. See 23. Server Configuration for the runtime certificate variables EVOSQL_TLS_CERT and EVOSQL_TLS_KEY.

22.2.2. Regenerating the parser

You only need this if you have changed the grammar or the lexer. The grammar is evolution/parser/evoparser.y. The lexer is evolution/parser/evolexer.l. This step needs Bison and Flex.

# Regenerate evoparser.tab.c/h and lex.yy.c, then rebuild
make generate
make

22.2.3. Release binaries

make release builds everything. It copies the three executables into release/. It also strips their symbol tables.

make release   # stripped binaries land in release/

22.3. Platform notes

The Makefiles find the platform with uname -s. They then adjust the link flags and the executable suffix to match.

Linux
Linux is the reference platform. Executables have no suffix. The server links -lpthread -lm -llz4. Install the dev packages with your package manager. For example, run apt-get install gcc make bison flex liblz4-dev libssl-dev libreadline-dev.
macOS (Darwin)
macOS builds native binaries with no suffix. Homebrew keeps liblz4, readline, and openssl off the default search path. The Makefiles add the Homebrew cellar include and lib paths for you. For example, they add -I/opt/homebrew/include. They also use brew --prefix readline and brew --prefix openssl for the keg-only formulas. Install with brew install lz4 readline openssl bison flex.
Windows (MinGW / MSYS2)
Windows builds .exe files. These are evosql.exe, evosql-server.exe, and evosql-cli.exe. The Makefiles send linker temp files to /tmp to work around a permission issue. They link the Winsock library -lws2_32 instead of pthreads.

Honest scope. The container image and CI run on Linux. The Makefiles support macOS and MinGW too. But those are best-effort targets. There are no prebuilt OS packages. There is no .deb, no .rpm, no Homebrew tap, and no Windows installer. You build from source or use Docker.

22.4. Installing with Docker

The shipped docker-compose.yml builds the server from a multi-stage Dockerfile and starts it. The build uses a gcc:13 builder and a debian:bookworm-slim runtime. It is built with TLS=1. This is the recommended path for testing. It also works well with clients such as DBeaver, pgAdmin, or psql.

# Start the server (builds the image on first run)
docker compose up -d              # PG:5433  EVO:9967

# Stop the server, keeping the data volume
docker compose down

# Stop the server and DELETE all data
docker compose down -v

You can also run the published image directly without the compose file.

docker run -d -p 5433:5433 -p 9967:9967 \
    -e EVOSQL_PASSWORD=mysecret evolutiondb/evolutiondb:latest

On first start the container entrypoint makes a self-signed TLS certificate under /data/certs/ if none exists. The certificate is valid for 10 years and its CN is evosql-server. The entrypoint then exports EVOSQL_TLS_CERT and EVOSQL_TLS_KEY. After that it starts the server. The database file and certificates live in the /data volume. The data survives docker compose down. Running down -v destroys it.

22.4.1. Ports

The server is dual-protocol. The defaults below come straight from the source and the compose file. The source values are DEFAULT_PG_PORT, DEFAULT_EVO_PORT, and REPL_DEFAULT_PORT.

PortProtocolDefaultHow to change
PostgreSQL wirePG frontend/backend protocol, psql, DBeaver, JDBC5433--pg-port
EVO nativeEvoSQL text protocol, evosql-cli, C/Python SDK9967--evo-port
ReplicationWAL sender for replicas9968 (disabled unless configured)--replication-port
CDC streamChange-data-capture, JSON Linesdisabled (compose maps 9970)--cdc-port / EVOSQL_CDC_PORT

Use port 5433, not 5432. EvolutionDB sets its PostgreSQL-compatible listener to 5433 by default. This way it does not clash with a real PostgreSQL on the same host. Point your client at -p 5433.

22.4.2. First-start credentials

The default admin user name is admin. You set it with EVOSQL_USER_NAME. The password is controlled by EVOSQL_PASSWORD.

  • If EVOSQL_PASSWORD is set, that password is used. The bundled docker-compose.yml sets it to admin to keep things simple.
  • If it is not set, a random password is made and printed to stdout once on first start. Check docker logs to find it.

A few other environment variables in the compose file shape the first start. The full list is in 23. Server Configuration.

EVOSQL_ENCRYPTION_KEY
The master passphrase for AES-256 Transparent Data Encryption. Set it to an empty string to turn encryption off. Page 0 is the file header. It is always stored in plaintext no matter what.
EVOSQL_CDC_PORT
Turns on the change-data-capture streaming server on the given port. The stream uses JSON Lines.
EVOSQL_AUTO_RECLAIM_INTERVAL_SEC
Sets how often the background reclaim and TTL tick runs. The compose file lowers it to 2 so TTL-driven row expiry shows up fast in tests. Production keeps the default of 30 s.
EVOSQL_BUFFER_POOL_SIZE
Sets the clock-sweep buffer pool size. It takes a raw page count or a unit suffix of K, M, or G. The suffix is read against the 4 KB page size. You can also set it with --buffer-pool-size.

22.5. Running a local build

After make, start the server and connect. The server prints its version and a banner. The banner shows the active ports. It also shows whether TLS is compiled in.

# Start the server (defaults: PG 5433, EVO 9967)
./adaptor/evosql-server

# Use a different PostgreSQL-wire port
./adaptor/evosql-server --pg-port 5434

# Connect with the native CLI (EVO port 9967)
./cli/evosql-cli
./cli/evosql-cli -W admin        # prompt for password

# Or connect any PostgreSQL client over the wire protocol
psql -h 127.0.0.1 -p 5433 -U admin testdb

Once you are connected you can create objects right away. There is no initdb step and no template database to bootstrap. See CREATE DATABASE and CREATE TABLE. There is also agent-memory DDL such as CREATE MEMORY STORE and CREATE CHECKPOINT STORE.

22.6. Verifying the installation

The test suite is a set of Python scripts. They speak the raw PG wire protocol. They do not need psycopg2. They need a running server. They connect to 127.0.0.1:5433 for PG or :9967 for EVO. They use the default credentials admin/admin against database testdb.

# Start the server first
docker compose up -d

# Run individual test files
python tests/test_memory_store.py
python tests/test_checkpoint_store.py
python tests/test_evo_protocol.py

Notes and limitations.

  • There is no configure script and no initdb. The engine creates its single 4 KB-page file evosql.db in the working directory on first run.
  • The server links the engine's object files directly. It does not link a shared library. There is no installable libevolution. The separate C client SDK is client/libevosql-memory/. You build it with its own make -C client/libevosql-memory.
  • TLS and page encryption are compile-time options set with TLS=1. A non-TLS build cannot be switched to TLS at runtime.
  • Bison and Flex are needed only for make generate. They are never needed for an ordinary build. This is because the generated sources are committed.
  • No OS-native packages are published. You get it by source build or container image only.
