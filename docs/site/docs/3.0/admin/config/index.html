23. Server Configuration

This page shows how to start and tune evosql-server. It covers the listener ports it opens. It covers the command-line flags and environment variables it reads at boot. It also covers the limits on how many clients it will serve at once.

EvolutionDB has one server executable, evosql-server. It is built from adaptor/. It is a dual-protocol server. One process starts two TCP listeners. One is the PostgreSQL wire protocol. The other is the native EVO text protocol. Both share one SQL engine and one data file, evosql.db. You set all configuration when the process starts. You use command-line flags and environment variables. PostgreSQL uses a postgresql.conf file and a reloadable GUC. EvolutionDB has neither. The settings on this page are read once in main(). They apply for the whole life of the process.

Precedence. Some settings have both a flag and an environment variable. The server uses one rule for all of them. It reads environment variables first to set defaults. Then command-line flags override them. The flag always wins.

23.1. Invoking the Server

The synopsis below lists every flag the argument parser in adaptor/main.c knows about.

evosql-server
    [--pg-port PORT]
    [--evo-port PORT]
    [--buffer-pool-size SIZE]
    [--replication-port PORT]
    [--cdc-port PORT]
    [--replica HOST:PORT]
    [--cluster node1:port,node2:port,...]
    [--node-id N]
    [--dist-port PORT]
    [--role primary|replica|witness]
    [--base-backup PATH]
    [PORT]

You can also pass a plain number with no flag. The server reads it as the PG port. This is kept for backward compatibility.

# These two are equivalent
evosql-server --pg-port 5544
evosql-server 5544

The argument parser is not strict. It ignores any flag it does not know. It does not print a usage message or exit with an error for unknown options. A flag that needs a value is only used when another argument follows it.

23.1.1. Startup Output

On a normal boot the server prints some information. It prints its version, the buffer-pool size, and the TLS status. It prints the two protocol ports and the maximum connection count. It also prints an ASCII banner. Then it starts to accept connections. A typical boot looks like this.

EvolutionDB v
[BufferPool] 32768 pages (128 MB)
[TLS] No EVOSQL_TLS_CERT / EVOSQL_TLS_KEY set - TLS disabled
==============================================
  EvoSQL Adaptor Server (dual-protocol)
  PG  protocol on port 5433
  EVO protocol on port 9967
  TLS: disabled
  Max connections: 100
==============================================
[EVO] Listening on port 9967
[PG] Listening on port 5433

The PG listener runs on the main thread. It blocks forever. The EVO listener runs in a background thread. The distributed-query listener also runs in a background thread when it is configured.

23.2. Ports and Listeners

The server can open up to four kinds of TCP listeners. It always opens the two protocol listeners. It opens the other listeners only when you turn their feature on.

ListenerDefault portHow to setOpened when
PG wire protocol5433--pg-port (or bare positional number)Always
EVO text protocol9967--evo-portAlways
Replication / WAL sender9968 *--replication-port / EVOSQL_REPLICATION_PORTPort > 0
CDC streaming (JSON Lines), (off)--cdc-port / EVOSQL_CDC_PORTPort > 0
Distributed query engine9969--dist-port--cluster + --node-id given

* 9968 is REPL_DEFAULT_PORT. A replica uses it as the default upstream port when you give only a host. The WAL sender itself starts only when --replication-port or EVOSQL_REPLICATION_PORT is set to a value > 0.

You will set the two protocol ports most often. The default PG port is 5433. This is on purpose. It is not PostgreSQL's 5432, so both can run on one host at the same time. Each listener binds with a fixed listen() backlog of 5.

# Run two engines on one box without colliding
evosql-server --pg-port 5433 --evo-port 9967
evosql-server --pg-port 5443 --evo-port 9977

No bind-address control. EvolutionDB has nothing like listen_addresses. The bind address is fixed in the network layer. You cannot use configuration to limit the server to localhost. You also cannot use it to pick one interface. Use OS firewall rules or container networking to limit who can reach it.

23.3. Connection Limits and the Worker Pool

EvolutionDB sets a hard limit on the number of client connections at one time. The accept loop in server_listen() checks the live count under a mutex before it handles each connection. When the limit is already reached, the new socket is closed right away. The server also logs a rejection line.

[PG] Connection rejected: limit 100 reached
max_connections
The most client connections allowed at one time. This count covers both protocols together. The default is built into the code and is 100. The server prints the active count and the limit on every connect and disconnect. It looks like active: 7/100. There is no startup flag for this value. You reach it through the catalog accessors get_max_connections() and set_max_connections(). The server banner shows it as Max connections.
EVOSQL_POOL_SIZE (env)
The size of the worker thread pool. The server starts these threads in advance to handle accepted sockets. The default is 20. Set it to 0 to turn the pool off. The server then falls back to the old one-thread-per-connection path. This is useful under Valgrind or ThreadSanitizer. The pool's task queue size follows max_connections. So a burst of connections cannot use up all the memory.
MAX_SESSIONS (compile-time)
The session registry has a fixed size of 256 slots. This array backs session introspection and statement cancellation. A new session still gets a valid ID when all slots are in use. But the registry does not track it.

Each connection runs in its own thread. The thread comes from the pool or is spawned. The server runs queries one at a time across all threads. It uses a parser read/write lock and a DML mutex to do this. You can read more in Concurrency & Locking. So the engine still has a single writer, even though the network layer uses many threads.

How this differs from PostgreSQL. In PostgreSQL max_connections is a reloadable GUC. Each connection gets its own backend process. EvolutionDB has a fixed limit of 100, set in code. Its connections are threads that share one address space. There is no limit per role or per database. There are no reserved superuser slots.

23.4. Buffer Pool Size

The Clock-Sweep buffer pool caches 4 KB pages. You can read more in Storage & the Buffer Pool. Its size has the biggest effect on memory of any setting here. It is also the only setting that accepts unit suffixes.

--buffer-pool-size SIZE / EVOSQL_BUFFER_POOL_SIZE
Sets how many 4 KB pages the buffer pool holds. The default is 32768 pages, which is 128 MB. This default is BP_DEFAULT_PAGES. The flag wins over the environment variable. The server reads the environment variable only when the flag is absent.

The size string can use a unit suffix or a plain page count. The server reads it as follows.

InputInterpreted asResulting pages
1G / 1gGigabytesval × 1024 × 1024 / 4 = 262144
128M / 128mMegabytesval × 1024 / 4 = 32768
512K / 512kKilobytesval / 4 = 128
32768Raw page count32768
 
# All three select a 256 MB pool
evosql-server --buffer-pool-size 256M
EVOSQL_BUFFER_POOL_SIZE=256M evosql-server
evosql-server --buffer-pool-size 65536   # raw pages

A value that comes out as zero or less counts as "unset". The server then uses the 128 MB default. It prints the final size at startup on a [BufferPool] line. The line shows pages and MB. For very small pools it shows KB.

23.5. Authentication Bootstrap

The server creates the first superuser on the first boot. It reads two environment variables to do this. The user-management layer reads them in evolution/db/UserMgmt.c.

EVOSQL_USER_NAME
The name of the bootstrap superuser. The default is admin.
EVOSQL_PASSWORD
The password for the bootstrap superuser. When this is unset or empty, the server creates a random 16-character password and prints it once at startup. Save it from the logs. The server does not store it in plaintext and does not show it again. The stored credential uses PBKDF2-SHA256. See CREATE USER and Security.
# Deterministic credentials for local development
EVOSQL_USER_NAME=admin EVOSQL_PASSWORD=admin evosql-server

Both protocols force cleartext-password auth. The PG wire handler always sends a CleartextPassword challenge. The hashing happens on the server side. Run the server behind TLS so the cleartext password is never seen on the wire. The next section covers TLS.

23.6. Transport Security (TLS)

TLS is off by default. You turn it on only through the environment. The server sets up an OpenSSL context for the PG-wire transport at startup when both variables below point to readable PEM files. If they do not, it prints TLS disabled and runs in cleartext. TLS also needs a build made with TLS=1.

EVOSQL_TLS_CERT
Path to the server certificate (PEM).
EVOSQL_TLS_KEY
Path to the matching private key (PEM).
EVOSQL_TLS_CERT=/etc/evosql/server.crt \
EVOSQL_TLS_KEY=/etc/evosql/server.key \
evosql-server

A failed TLS init does not stop the server. It logs [TLS] Failed to initialise - continuing without TLS and runs in cleartext. It does not refuse to start. The replication transport can reuse the same certificate and key pair. This happens when EVOSQL_REPLICATION_TLS is truthy. See below.

23.7. Replication, CDC and Cluster Boot Settings

You can set up the replication topology fully from the environment. This is handy for containers. You can then refine it with flags. The server reads the environment first. Matching flags override it. High Availability & Replication has the full operational detail. This section only covers the boot-time settings.

EVOSQL_REPLICATION_ROLE / --role
One of primary, replica, or witness. This sets the role that SHOW REPLICATION STATUS reports before any peer connects. An unknown value logs a warning and leaves the role unset.
EVOSQL_PRIMARY_HOST / EVOSQL_PRIMARY_PORT
The upstream the server should follow when role=replica. The replica uses REPL_DEFAULT_PORT, which is 9968, when you give a host without a port. This is the same as the flag --replica HOST:PORT.
EVOSQL_REPLICATION_PORT / --replication-port
The port where a primary runs its WAL sender. The sender starts only when this comes out as a value > 0.
EVOSQL_REPLICATION_USER / EVOSQL_REPLICATION_PASSWORD
The credentials for the replication auth handshake. You must set both for them to take effect.
EVOSQL_REPLICATION_TLS
A truthy value turns on TLS for the replication transport. The truthy values are 1, t, T, y, and Y. It uses the same cert and key as the PG-wire context.
--cdc-port / EVOSQL_CDC_PORT
Starts a Change-Data-Capture streaming listener. It pushes WAL-decoded changes as JSON Lines. It is off unless the value is > 0.
--cluster + --node-id
A comma-separated host:port node list plus this node's zero-based index. Give both to start the Raft consensus module and the distributed query engine. The server works out each node's PG and dist ports by offsetting from this node's ports.
--base-backup PATH
A one-shot mode. It copies the data directory to PATH and exits without binding any listener sockets. Use it to bootstrap a fresh replica from a primary. The exit code is 0 on success and 1 on failure.
# Primary with WAL sender on 9968
evosql-server --role primary --replication-port 9968

# Replica following the primary above (env-driven)
EVOSQL_REPLICATION_ROLE=replica \
EVOSQL_PRIMARY_HOST=10.0.0.1 EVOSQL_PRIMARY_PORT=9968 \
evosql-server

23.8. Temporal Retention

EVOSQL_SYSTEM_TIME_RETENTION_DAYS
Sets the retention window for system-versioned table history. These are temporal tables. The default is 7 days. A value of 0 turns retention pruning off. The server accepts any non-negative integer at boot. Most settings here are fixed at boot, but this one can change at runtime. Use SET SYSTEM_TIME_RETENTION = N. This overrides the boot value for the running process.

Notes & limitations

  • No config file. There is no postgresql.conf or evosql.conf. All configuration comes from flags and the environment. The server resolves it once at process start.
  • Most settings cannot reload at runtime. You must restart to change ports, buffer-pool size, pool size, TLS, or the connection limit. The only documented exception is SYSTEM_TIME_RETENTION. You can change it with SET.
  • The connection limit is built into the code and is 100. There is no startup flag for it. The server drops over-limit connections at accept time and logs a rejection. It does not send a protocol-level error message.
  • No bind-address or interface selection. Use a firewall or container networking to limit who can reach it.
  • Unknown flags are ignored without any message. There is no usage banner and no non-zero exit for a misspelled option. So check your spelling.
  • Graceful shutdown happens on SIGTERM or SIGINT. The server drains the worker pool, checkpoints the WAL, flushes the buffer pool, and closes the data file before it exits. This is POSIX only.
