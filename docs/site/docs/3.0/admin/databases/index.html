24. Managing Databases

A database in EvolutionDB is a named namespace. It is registered in the catalog and owns one or more schemas. This chapter shows how to create databases. It also shows how to switch between them with USE. It explains how the multi-database layout maps onto EvolutionDB's single unified storage file.

24.1. Overview

Some systems give each database its own files. Some give each database its own cluster directory. EvolutionDB is different. It stores every database in one unified, page-based file. That file is root/evosql.db. The root/ directory contains nothing else. There are no per-database subdirectories on disk. Databases, schemas, tables, and all other metadata live inside that file's system catalog. The catalog is itself a set of B+trees.

Creating a database is a pure catalog operation. It inserts a row into the catalog's database list. It does not allocate or set up any separate file. A database is best understood as a logical grouping. It is not a physical storage boundary.

layers The naming hierarchy has three levels. The levels are database → schema → table. Every new database gets a starting schema named default. Every session starts out pointed at some database and some schema. Schema management is covered in detail in Chapter 25, Managing Schemas. This chapter focuses on the database level.

24.2. The Default Database and Schema

On the very first start of a fresh storage file, EvolutionDB sets up the catalog. It creates a single database named evosql. That database holds a single schema named default. A new session inherits these values if you do not specify anything else. The current database defaults to evosql. The current schema defaults to default.

Default database
evosql. It is created automatically the first time the catalog is set up. It always has internal database id 1.
Default schema
default. It is created inside every database at creation time. This includes evosql.
Storage root
The directory root/. It holds the single file root/evosql.db. The directory name is fixed. It is not per-database.

info PostgreSQL contrast. PostgreSQL ships with postgres, template0, and template1. In PostgreSQL, CREATE DATABASE physically clones a template directory. EvolutionDB has no template databases. It has no per-database directory. So there is nothing to clone. A new database is just a catalog entry plus its own default schema.

24.3. Creating a Database

Use the CREATE DATABASE command. The only argument is the database name. EvolutionDB does not accept the per-database options found in standard SQL or PostgreSQL. Those options include owner, encoding, locale, template, tablespace, and connection limit. 

CREATE DATABASE name
CREATE DATABASE IF NOT EXISTS name

Creating a database registers it in the catalog. It also gives the database a schema named default right away.

CREATE DATABASE shop;
-- server responds: CREATE DATABASE shop

-- The new database already contains schema "default".
USE shop;
CREATE TABLE products (
    id    INT PRIMARY KEY,
    name  VARCHAR(100)
);  -- created in shop.default.products

24.3.1. IF NOT EXISTS

Suppose you create a database whose name is already registered. Without IF NOT EXISTS, the command fails with SQLSTATE 42E04. That code means a duplicate database. With IF NOT EXISTS, the command does nothing and succeeds quietly when the database already exists. 

CREATE DATABASE shop;                  -- ok
CREATE DATABASE shop;                  -- ERROR 42E04: database "shop" already exists
CREATE DATABASE IF NOT EXISTS shop;    -- ok, no change

warning Note. CREATE DATABASE does not switch your session to the new database. The current database stays the same after the command returns. You must run USE on its own to start working inside the new database.

24.4. Switching Databases with USE

The USE command changes which database your next statements work on. EvolutionDB checks the name against the catalog. If the name is found, it sets that database as the session's current database. There is one side effect. The current schema is reset to default. Switching databases does not keep the schema you had selected before. 

USE database_name
USE shop;
-- server responds: USE DATABASE shop
-- current database = shop, current schema reset to "default"

Suppose the named database is not in the catalog. Then USE fails with SQLSTATE 3D000. That code means an undefined database. The session's current database stays the same. 

USE nope;   -- ERROR 3D000: database "nope" does not exist

24.4.1. USE Is Per-Session

The current database and current schema are stored in each connection's own query context. They are not a global server setting. Two clients connected at the same time can point at two different databases. One connection's USE never affects another. The selection lasts only for the life of the connection.

info PostgreSQL and standard-SQL contrast. PostgreSQL has no USE statement. You select a database by reconnecting. A connection is bound to one database for its whole life. EvolutionDB's USE works more like the MySQL USE statement. It is an in-session switch. But it also resets the active schema to default. MySQL does not do this. MySQL treats "database" and "schema" as the same thing.

24.4.2. The Database Chosen at Connect Time

A client can connect over the PostgreSQL wire protocol. The database name in the startup message becomes the session's first current database. A well-configured PostgreSQL client lands right in the database it asked for. It does not need an explicit USE. The same applies to a driver such as the one used by DBeaver or psql. A session that does not name a database falls back to the default. The default is evosql.

24.5. The Multi-Database Layout

All databases share root/evosql.db. Qualifying an object name is purely logical. A fully qualified table reference has this form. 

database.schema.table

At runtime, the engine resolves unqualified names against the session's current database and current schema. The old path form root/<database>/<schema>/<table> still shows up inside the engine for backward compatibility. It is used when extracting bare table names. But it does not map to real directories on disk. It is a naming convention. It is not a filesystem layout.

LevelCreated byNotes
DatabaseCREATE DATABASECatalog entry only. Always starts with a default schema.
SchemaCREATE SCHEMACreated inside the current database. See Chapter 25.
TableCREATE TABLECreated inside the current schema of the current database.

24.5.1. Practical Implications

One file, one buffer pool
Every database is paged through the same 4 KB buffer pool. They all use the same write-ahead log. Databases are a way to organize your data. They do not isolate storage, caching, or I/O from one another.
No per-database tuning
CREATE DATABASE accepts only a name. So there are no per-database encodings, collations, owners, or tablespaces to set.
Names are case-sensitive in the catalog
Database names are stored and looked up as given. However, the current-database guard compares names without case sensitivity.

24.6. Dropping a Database

The DROP DATABASE command removes a database. The removal cascades to all of its schemas and the tables inside them. 

DROP DATABASE name
DROP DATABASE IF EXISTS name
DROP DATABASE shop;             -- drops shop and everything inside it
DROP DATABASE IF EXISTS shop;   -- no error if shop is already gone

You cannot drop the database your session is currently using. Switch away first. You can also connect somewhere else.

USE shop;
DROP DATABASE shop;   -- ERROR 55006: cannot drop the currently open database "shop"
USE evosql;
DROP DATABASE shop;   -- ok

warning Note. The drop always cascades. DROP DATABASE always removes every schema and every table the database holds. There is no RESTRICT mode that refuses to drop a non-empty database. This operation destroys data and cannot be undone. If you drop a database that does not exist, the command fails with SQLSTATE 3D000 unless you use IF EXISTS. That code means an undefined database.
