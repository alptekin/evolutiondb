25. Roles & Privileges

EvolutionDB controls who may connect and what they may do. It uses three things stored in the catalog. There are login users, non-login roles, and object privileges. Privileges are checked with a waterfall that tries the most specific scope first. The waterfall order is TABLE → SCHEMA → DATABASE.

This chapter describes the access-control model. It matches how the engine works in evolution/db/UserMgmt.c and evolution/db/GrantMgmt.c. All users, roles, and grants are stored as normal rows. The rows live in the system catalog B+ trees inside the evosql.db file. There is no separate authentication file. EvolutionDB sometimes works differently from the SQL standard or PostgreSQL. When it does, this page points out the difference clearly.

25.1. Overview of the access-control model

Three catalog tables support everything in this chapter.

CAT_SYS_USERS
One row per principal. A principal is a login user with is_role = 0. Or it is a role with is_role = 1. The key is the principal name. The record holds the PBKDF2-SHA256 password hash. The hash is empty for roles.
CAT_SYS_GRANTS
One row per grant. The composite key is username:scope_type:scope_name. The record stores a comma-separated privilege list and a single grant_option flag. Role memberships are stored in this same tree. They use scope_type = ROLE.

Two principals are special. They are not stored in a table. They exist on their own.

  • admin, the built-in superuser. §25.6.
  • PUBLIC, a special principal. Its grants apply to every user. They add up with the user's own grants. §25.5.

Not the SQL-standard separation. The SQL standard and PostgreSQL keep three things apart in the catalog. These are authorization identifiers, roles, and privileges. EvolutionDB does it differently. Users and roles share one catalog table. The is_role flag is the only thing that tells them apart. A role is just a principal that cannot log in but can hold grants. There is no privilege inheritance graph. The only inheritance comes from direct role membership. See §25.4.

25.2. Users

A user is a principal that can log in and open a session. You create, change, drop, and list users with SQL. The adaptor catches this SQL and sends it to UserMgmt.c.

25.2.1. Creating users

See the CREATE USER reference page for full syntax.

CREATE USER name PASSWORD 'password';

You must give both a name and a password. The password cannot be empty. If either one is empty, you get an error. The SQLSTATE is 22023, which means invalid parameter value. The engine hashes the password with PBKDF2-SHA256. It does this before it takes the catalog write lock. Then it creates the user row. It only creates the row if no principal of that name already exists. A duplicate name raises SQLSTATE 42710, which means duplicate object.

-- create an application login
CREATE USER app_rw PASSWORD 's3cret-pw';

25.2.2. Changing a password

ALTER USER name PASSWORD 'newpassword';

ALTER USER ... PASSWORD hashes the new password again. Then it writes over the stored hash. EvolutionDB only supports the password change form of ALTER USER. You cannot rename a user. You cannot change other attributes. PostgreSQL has privilege attributes like SUPERUSER and CREATEDB. EvolutionDB does not have these. If you alter a name that does not exist, you get SQLSTATE 42704, which means undefined object.

ALTER USER app_rw PASSWORD 'rotated-pw';

25.2.3. Dropping users

DROP USER name;

Dropping a user removes the catalog row. The engine also has DropUserGrants(). It deletes every grant whose key starts with that username. So a dropped user leaves no orphaned grants behind. Dropping a user that does not exist raises SQLSTATE 42704.

25.2.4. Listing users

SHOW USERS lists the principals in CAT_SYS_USERS. It leaves out anything with is_role = 1. So it returns login users only. See SHOW.

SHOW USERS;

25.3. Authentication

When a session sends credentials, AuthenticateUser() runs the following steps in order.

  1. The engine makes the username lower case and looks it up in the catalog. If that fails, it tries the original case as a fallback. This makes login case-insensitive in most cases. It still works for a name stored with mixed case.
  2. If the matched principal is a role with is_role = 1, authentication fails right away. Roles cannot log in.
  3. The engine checks the password against the stored hash. It uses a constant-time comparison. This helps stop timing attacks.

25.3.1. Password hashing

Passwords are never stored in plaintext. The stored value is a PBKDF2-SHA256 string. It describes itself. It looks like this.

pbkdf2:100000:<32-hex-salt>:<64-hex-hash>
Algorithm
PBKDF2 with HMAC-SHA256.
Iterations
100000. This is the compiled constant PBKDF2_ITERATIONS.
Salt
16 random bytes per password. This is PBKDF2_SALT_LEN. It is stored as hex in the hash string.
Derived key
32 bytes. This is PBKDF2_KEY_LEN.

The engine wipes plaintext password buffers and derived hashes from memory after use. It uses evo_secure_wipe() for this.

25.3.2. The initial admin password

On first start, the engine sets up the admin login from environment variables.

EVOSQL_USER_NAME
The first login name. It defaults to admin when it is unset or empty.
EVOSQL_PASSWORD
The first password. If it is unset or empty, the engine makes a random 16-character password. It hashes the password, stores it, and prints it to stdout one time only. This works like MySQL --initialize. The password is never shown again. You can change it later with ALTER USER.
# first boot with a generated password (printed once on stdout)
./adaptor/evosql-server

# or pin the credentials up front
EVOSQL_USER_NAME=admin EVOSQL_PASSWORD=changeme ./adaptor/evosql-server

Documentation vs. source note. The project security guideline recommends a PBKDF2 iteration count of at least 600,000. The shipped 3.0 engine compiles PBKDF2_ITERATIONS = 100000. This page documents the value the code really uses. Treat the higher figure as a hardening target. It is not the current behavior.

25.4. Roles

A role is a principal that cannot log in. It can hold privileges. You can grant it to users. Roles let you build a set of privileges one time. Then you can attach that set to many users.

25.4.1. Creating and dropping roles

CREATE ROLE name;
DROP ROLE name;

CREATE ROLE fails if any user or role of that name already exists. Users and roles share one namespace. DROP ROLE first checks that the target really is a role with is_role = 1. Then it does a full cleanup. It removes the role's catalog row. It deletes all grants held by the role. It removes every membership edge that points to the role. So no user stays a member of a role that no longer exists.

25.4.2. Granting roles to users

GRANT role_name TO user_name;
REVOKE role_name FROM user_name;

Membership is itself a catalog grant. It is stored with scope_type = ROLE. The scope name is equal to the role. The privilege text is the literal word MEMBER. Granting a role runs three checks. The role must exist. The target user must exist. The target must be a user and not another role. You cannot grant a role to a principal that is itself a role. Roles cannot be nested. A duplicate membership is also rejected.

CREATE ROLE reporting;
GRANT SELECT ON SCHEMA testdb.public TO reporting;
GRANT reporting TO app_rw;   -- app_rw now inherits reporting's grants

25.4.3. How role privileges are inherited

The engine runs a privilege check in steps. First it checks the user's own grants and the PUBLIC grants. If none of those decide the result, it looks up the user's direct role memberships. Then it runs the same waterfall again for each role. See §25.7. Membership goes only one level deep. The engine does not follow a role granted to another role. In any case, that kind of nesting is not allowed at grant time. A check looks at no more than 16 role memberships per user.

25.5. The PUBLIC pseudo-principal

Grants made to PUBLIC apply to every user. EvolutionDB treats PUBLIC grants as additive only. During a check, the engine first looks at the user's own grants. If the result is still undecided, it looks at PUBLIC grants. It uses the same waterfall for both. So a REVOKE for one user does not cancel a privilege granted to PUBLIC. To remove that privilege, you must revoke it from PUBLIC directly.

No default PUBLIC grants. PostgreSQL grants some default privileges to PUBLIC on new objects. EvolutionDB does not. A new EvolutionDB database has no implicit PUBLIC grants. This is a closed default like MySQL. New non-admin users start with no access. They get access only after you make a grant.

25.6. The admin superuser

The principal named admin is a hard-coded superuser. The name is matched without case. Every privilege check for admin returns allowed right away. It does no catalog I/O. HasGrantOption() also always returns true for admin. The engine seeds one grant on first start to keep the rule consistent. This grant gives admin the ALL privilege on DATABASE * with grant option. The in-memory fast path means admin is authorized even before any grant row is read.

Caution. Superuser status is fixed to the exact name admin. There is no ALTER USER ... SUPERUSER. You cannot promote or demote superuser status at runtime. Treat the admin credential as the master key to the instance.

25.7. Privileges and scopes

EvolutionDB has six object privileges. It also has the ALL shorthand. Each grant stores them as a comma-separated text list. The engine parses this list into a bitmask with parse_privs() when it checks a privilege. Privilege keywords are case-insensitive. ALL and ALL PRIVILEGES mean the same thing.

KeywordInternal bitGoverns
SELECT0x01Reading rows
INSERT0x02Adding rows
UPDATE0x04Modifying rows
DELETE0x08Removing rows
CREATE0x10Creating objects
DROP0x20Dropping objects
ALL0x3FAll six of the above

An unknown token adds nothing to the mask. A privilege string that parses to an empty mask is rejected. Both GRANT and REVOKE reject it as an invalid privilege. The SQLSTATE is 42501.

25.7.1. Scope levels

A grant attaches a privilege set to one of three object scopes. The scope name is turned into a standard dotted path.

ScopeType codeScope-name form
DATABASE1database, or * for the wildcard
SCHEMA2database.schema
TABLE3database.schema.table
ROLE4rolename. This is a membership edge. See §25.4.2.

The * wildcard is only honored at DATABASE scope. It is the mechanism behind the seeded admin grant. There is no schema-level or table-level wildcard. There are also no column-, sequence-, or routine-level privileges.

25.8. GRANT and REVOKE

Full syntax is on the GRANT and REVOKE reference pages.

GRANT priv[, priv ...] ON {DATABASE | SCHEMA | TABLE} object TO user [WITH GRANT OPTION];
REVOKE priv[, priv ...] ON {DATABASE | SCHEMA | TABLE} object FROM user;

25.8.1. GRANT semantics

A grant is keyed by (user, scope_type, scope_name). If no grant exists for that triple, a new row is created with the parsed privilege mask. If a grant already exists, the new privileges are merged (bitwise OR) into the existing set rather than replacing it. WITH GRANT OPTION is sticky. Specifying it sets the option to true. Omitting it leaves an already-set option untouched. GRANT never silently drops a privilege the principal already held at that scope.

CREATE USER analyst PASSWORD 'pw';
GRANT SELECT ON TABLE testdb.public.orders TO analyst;
GRANT INSERT, UPDATE ON TABLE testdb.public.orders TO analyst;
-- analyst now holds SELECT,INSERT,UPDATE on that table

25.8.2. WITH GRANT OPTION

The grant option is a single boolean stored on the grant row. It is not tracked per individual privilege. HasGrantOption() answers whether a principal may re-grant a privilege. It returns true if the principal holds the requested privilege at the exact scope and the row's grant-option flag is set. For DATABASE scope the wildcard * grant is also consulted. admin always passes.

25.8.3. REVOKE semantics

REVOKE targets the same (user, scope_type, scope_name) triple. If no matching grant exists, it errors with SQLSTATE 42704. Otherwise the requested privileges are cleared from the mask. If some privileges remain, the grant row is rewritten with the reduced set. If none remain, the row is deleted entirely. Revoking ALL additionally clears the grant option.

REVOKE INSERT, UPDATE ON TABLE testdb.public.orders FROM analyst;
-- analyst retains only SELECT on that table
REVOKE ALL ON TABLE testdb.public.orders FROM analyst;
-- grant row removed entirely

No cascading revoke. REVOKE affects only the named principal's own grant row. It does not cascade to principals who received the privilege from that principal via WITH GRANT OPTION. There is no RESTRICT/CASCADE clause. There is also no separate form to revoke only the grant option. To drop the option, you must revoke and re-grant.

25.8.4. Inspecting grants

SHOW GRANTS lists grants for the current user. SHOW GRANTS FOR user lists them for a named principal. With no user specified, the engine enumerates every user's grants and then appends the PUBLIC grants. See SHOW.

SHOW GRANTS FOR analyst;

25.9. The TABLE → SCHEMA → DATABASE waterfall

Authorization for a data operation is decided by CheckPrivilege(username, database, schema, table, privilege). The check is most-specific-first and short-circuits the moment any source grants the needed bit. The order is fixed.

  1. admin fast path. If the principal is admin, return allowed at once.
  2. User grants, by scope, narrowest first.
    1. TABLE, database.schema.table (only attempted when all three are present)
    2. SCHEMA, database.schema
    3. DATABASE, the named database, then the * wildcard
  3. PUBLIC grants. The same TABLE→SCHEMA→DATABASE→* sequence runs for the PUBLIC principal.
  4. Role-inherited grants. For each of the user's roles, the same waterfall runs again.
  5. If nothing matched, return denied.

Most specific wins first. A privilege found at TABLE scope ends the search before SCHEMA or DATABASE rows are even read. The check stops at the first granting source. The waterfall can only add authority as it widens. It never withholds a broad grant because a narrower scope is silent. There is no deny rule. A grant either exists and permits, or it is absent.

-- broad read at the database level
GRANT SELECT ON DATABASE testdb TO analyst;
-- a SELECT on testdb.public.orders is now allowed:
-- TABLE and SCHEMA rows are absent, the DATABASE grant matches.

-- a narrower write grant, layered on top
GRANT INSERT ON TABLE testdb.public.orders TO analyst;
-- INSERT on that one table is allowed via the TABLE grant;
-- INSERT elsewhere in testdb remains denied.

25.11. Notes and limitations

What EvolutionDB 3.0 does not implement. The model is deliberately narrower than the SQL standard or PostgreSQL. These are the notable gaps.

  • No column-, sequence-, function/procedure-, or large-object-level privileges. Only the six object privileges in §25.7 exist, at DATABASE/SCHEMA/TABLE scope only.
  • No object OWNER concept and therefore no implicit owner privileges. Authorization comes entirely from explicit grants (plus the admin fast path and PUBLIC).
  • No role inheritance graph beyond one level of direct membership, and roles cannot be nested. At most 16 memberships per user are consulted per check.
  • No cascading REVOKE, no RESTRICT/CASCADE, and no standalone way to drop only a grant option.
  • No USAGE, TEMPORARY, CONNECT, or other PostgreSQL privilege keywords.
  • ALTER USER changes only the password; there are no login attributes or role flags.
  • Superuser is fixed to the name admin and cannot be reassigned.
  • The shipped PBKDF2 iteration count is 100000, below the project's stated 600,000 hardening target.
