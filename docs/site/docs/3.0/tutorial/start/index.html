1. Getting Started

Install EvolutionDB, start the server, and connect with a client. Then run your first CREATE TABLE, INSERT, and SELECT. This takes a few minutes and uses one binary.

EvolutionDB is a single-process SQL database engine written in C. One server speaks two protocols at the same time. The first is the PostgreSQL wire protocol on TCP port 5433. With it, psql, DBeaver, pgAdmin, and JDBC drivers connect with no changes. The second is a native EVO text protocol on TCP port 9967. The bundled evosql-cli client uses it. This chapter shows you how to get a server running and run your first statements. Later chapters cover the full SQL surface. Here we stay small on purpose.

1.1. Installation

You can run EvolutionDB in two ways. You can use a prebuilt Docker image, or you can build the binaries from source. Docker is the fastest way. We use it throughout the test suite.

1.1.1. Running with Docker

From the repository root, start the server in the background.

docker compose up -d              # PG:5433  EVO:9967

The bundled docker-compose.yml publishes three ports. It also creates a named volume evosql-data mounted at /data. This volume keeps your data, so it survives container restarts. Here are the ports.

5433
PostgreSQL wire protocol, for psql, DBeaver, pgAdmin, JDBC.
9967
Native EVO text protocol, for the bundled evosql-cli.
9970
CDC change-data-capture streaming as JSON Lines. You will not need this for a first session.

The Compose file sets the first login details and a few runtime options. It does this through environment variables.

# from docker-compose.yml
EVOSQL_USER_NAME: admin
EVOSQL_PASSWORD: admin
EVOSQL_CDC_PORT: 9970
EVOSQL_AUTO_RECLAIM_INTERVAL_SEC: 2
EVOSQL_ENCRYPTION_KEY: "…"

With the default Compose setup, the login is admin / admin. To stop the server and keep your data, run docker compose down. To stop it and delete the data volume, run docker compose down -v.

You can also run the published image directly. This lets you set your own password.

docker run -d -p 5433:5433 -p 9967:9967 \
    -e EVOSQL_PASSWORD=mysecret evolutiondb/evolutiondb:latest
About the default password. The Compose file shipped in the repository sets EVOSQL_PASSWORD to admin. If you remove or comment out that variable, the server makes a random password. It prints this password once at startup. Copy it from the logs. The server does not store it anywhere you can read it back.

1.1.2. Building from source

To build, you need GCC, Bison, Flex, and libreadline-dev. The CLI needs libreadline-dev. TLS support also needs libssl-dev. A bare make builds all three components. These are the core engine, the dual-protocol server, and the CLI.

make                  # core engine + adaptor + CLI
make adaptor TLS=1    # build the server with OpenSSL TLS support

This builds the server at adaptor/evosql-server, the CLI at cli/evosql-cli, and the core engine at evolution/evosql. If you ever change the SQL grammar, run make generate to rebuild the parser first. Then rebuild.

1.1.3. Starting a local server

After building, start the server directly. With no arguments it listens on the PostgreSQL port 5433 and the EVO port 9967.

./adaptor/evosql-server                    # PG:5433, EVO:9967
./adaptor/evosql-server --pg-port 5434     # custom PG port

The login user is admin by default. You can change it with EVOSQL_USER_NAME. This works the same as Docker. If EVOSQL_PASSWORD is not set, the server prints a new password once at startup.

1.2. Connecting to the server

Once a server is listening, you can connect to it. Use the bundled CLI on the EVO port, or use any PostgreSQL client on the PG port.

1.2.1. The bundled CLI (EVO protocol, port 9967)

evosql-cli connects on the EVO protocol port. The default port is 9967. Here is the simplest interactive session. It passes the password on the command line.

./cli/evosql-cli -W admin              # connect to localhost:9967

The client knows these options. You can also see them with evosql-cli --help.

-h, --host HOST
Server hostname (default localhost).
-p, --port PORT
Server port (default 9967).
-u, --user USER
Username (default admin).
-W, --password PW
Password. You can also set the EVOSQL_PASSWORD environment variable. If you give neither on an interactive run, the client asks you for it. The password does not show on screen.
-c, --command SQL
Run a single SQL statement and exit. This is non-interactive.
-f, --file FILE
Run SQL from a file and exit.
-F, --format FMT
Output format. Choose table, csv, or json. The default is table.
-x, --expanded
Expanded, one-column-per-line display.
--timing
Print query execution time.
--color / --no-color
Turn ANSI colors on or off. By default the client detects this from the TTY.

If you give no -c or -f and standard input is a TTY, the CLI starts an interactive shell. You can also pipe a file in. Run evosql-cli < file.sql to run it as a script. End each SQL statement with a semicolon.

Inside the interactive shell, a few backslash meta-commands work the way psql users expect.

CommandAction
\d / \dtList tables
\d NAMEDescribe a table's columns
\lList databases
\duList users
\dnList schemas
\xToggle expanded display
\timingToggle query timing
\format FSet output format to table, csv, or json
\i FILERun SQL from a file
\eEdit a statement in $EDITOR, then run it
\! CMDRun a shell command
\qQuit

1.2.2. A PostgreSQL client (PG wire, port 5433)

EvolutionDB speaks the PostgreSQL wire protocol. So psql connects with no special setup.

psql -h 127.0.0.1 -p 5433 -U admin testdb

The default test database is testdb. The server fakes enough of pg_catalog and information_schema for GUI tools to work. Tools like DBeaver and pgAdmin can then read tables and columns.

Authentication is always cleartext-password. A stock PostgreSQL install works in other ways. This server always uses CleartextPassword authentication on the wire. It then checks the password on the server side. Passwords are stored hashed with PBKDF2-SHA256. They are never stored in plaintext. Use TLS when you connect across an untrusted network. To do this, build the server with TLS=1.

1.3. Your first table

The next examples assume an interactive evosql-cli session. They run the same way through psql. They are ordinary SQL. We will create a table, insert a few rows, and read them back.

1.3.1. CREATE TABLE

Define a small table. The column types here come from the integer, character, and boolean families that the engine supports.

CREATE TABLE products (
    id        INTEGER PRIMARY KEY,
    name      VARCHAR(50) NOT NULL,
    price     DECIMAL(10, 2),
    in_stock  BOOLEAN DEFAULT TRUE
);

EvolutionDB stores each record in its binary tuple format. First comes a sentinel byte and a table id. Then comes a header, a null bitmap, and the column data. A column type code packs both a type family and a size. For example, VARCHAR(50) is encoded as 130050. See CREATE TABLE for the full grammar of column attributes, constraints, and table options.

Here are a few data types you will use first.

TypeUse for
INTEGER / INT, SMALLINT, BIGINTWhole numbers
DECIMAL(p,s), alias NUMERIC or DECExact fixed-point numbers, such as money
REAL, FLOAT, DOUBLEApproximate floating-point numbers
CHAR(n), VARCHAR(n), TEXTCharacter strings
BOOLEAN, alias BOOLTrue / false values
DATE, TIMESTAMP, DATETIMEDates and timestamps
UUIDUniversally unique identifiers
JSON, alias JSONBSemi-structured documents
VECTOR(dim)Embedding vectors for similarity search

1.3.2. INSERT

Add rows. You can name the target columns. You can also insert several rows in one statement.

INSERT INTO products (id, name, price) VALUES
    (1, 'Mechanical Keyboard', 89.90),
    (2, 'USB-C Cable', 7.50),
    (3, '27-inch Monitor', 249.00);

We left out the in_stock column. So each row uses its declared DEFAULT TRUE. See INSERT for value lists, default handling, and inserting from a query.

1.3.3. SELECT

Read the rows back. Start with everything.

SELECT * FROM products;

Then narrow and order the result. A WHERE clause filters rows. ORDER BY sorts them.

SELECT name, price
FROM products
WHERE price < 100
ORDER BY price DESC;

Use aggregates to summarize. This counts the rows. It also finds the average price across the whole table.

SELECT COUNT(*) AS n, AVG(price) AS avg_price
FROM products;

For the full grammar of projection, filtering, grouping, joining, and subqueries, see SELECT.

1.3.4. Cleaning up

When you are done experimenting, drop the table.

DROP TABLE products;

See DROP TABLE. You can also empty a table and keep it. Use TRUNCATE for that.

1.4. Where to go next

You now have a working server, a connection, and a table. You created the table, filled it, and queried it. Here is where to go next.

  • Build richer schemas with constraints, indexes, and defaults. See CREATE TABLE and CREATE INDEX.
  • Change data with UPDATE and DELETE. You can wrap these in transactions with savepoints.
  • Explore the agent-memory objects that make EvolutionDB special. These are CREATE MEMORY STORE for vector search, CREATE CHECKPOINT STORE, and reactive LISTEN / NOTIFY push.
Notes and limitations.
  • Two ports, one server. The PostgreSQL protocol uses port 5433. The native EVO protocol uses port 9967. They are not interchangeable. Point psql and JDBC at 5433. Point evosql-cli at 9967.
  • Authentication mode is fixed. The wire protocol always uses CleartextPassword. There is no MD5 or SCRAM negotiation like in stock PostgreSQL. Use TLS on untrusted links.
  • Single storage file, global state. The engine keeps all data in one page-based file. It runs queries one at a time behind a lock for thread safety. It is a single-process server, not a multi-node cluster.
  • Catalog is emulated, not complete. The server fakes enough of pg_catalog and information_schema for common GUI tools to read schemas. Do not expect full PostgreSQL system-catalog fidelity.
  • Type aliases are accepted but behavior is EvolutionDB's own. The engine accepts many MySQL and PostgreSQL type names as aliases. For example, BOOL maps to BOOLEAN, and NUMERIC maps to DECIMAL. Behavior follows EvolutionDB's engine, not any one upstream database. Check edge cases against the reference pages, not against PostgreSQL habits.
