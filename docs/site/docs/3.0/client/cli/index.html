31. evosql-cli

evosql-cli is the interactive terminal client for EvolutionDB. It speaks the native EVO text protocol. It shows result sets as tables, CSV, or JSON. It supports backslash meta-commands. You can use it as an interactive shell or as a script runner.

PostgreSQL's psql talks the binary PostgreSQL wire protocol on port 5432. evosql-cli is different. It connects over EvolutionDB's line-based EVO protocol on TCP port 9967. The two are not interchangeable. psql and other PostgreSQL clients like DBeaver connect to the server's PostgreSQL-compatible port 5433. evosql-cli is built for the EVO port. This page describes what the client really does. When a familiar psql feature is missing or works differently, the page says so.

31.1. Invocation

The client takes connection parameters. It also takes an optional one-shot command or a script file. If you give no command and no file, and a terminal is attached to standard input, the client starts an interactive shell. In every other case it runs in script mode.

evosql-cli [OPTIONS] [-c SQL | -f FILE]
-h, --host HOST
The server hostname. The default is localhost. The name is resolved with gethostbyname(). If that fails, the client tries inet_addr() for a literal IPv4 address.
-p, --port PORT
The server EVO port. The default is 9967. This is the EVO port. It is not the PostgreSQL port 5433.
-u, --user USER
The username. The default is admin.
-W, --password PW
The password. If you leave it out, the client reads the EVOSQL_PASSWORD environment variable. If that is not set either and the server needs authentication, the client asks you for it. The password is not shown on screen while you type.
-c, --command SQL
Run the given SQL string and then exit. This forces script mode.
-f, --file FILE
Read SQL from FILE, run it, and then exit. This forces script mode.
-F, --format FMT
The output format. The choices are table, csv, and json. The default is table. An unknown value is rejected with exit code 2.
-x, --expanded
Use the expanded display. The client prints one column | value line for each field. It does not use an aligned grid. This is the same as the \x meta-command. It works with the table format only.
--timing
Print the client-side wall time after each statement. This is the same as \timing.
--color / --no-color
Turn ANSI colors on or off by force. The default is automatic. With the automatic default, colors appear only when the format is table and standard output is a TTY. On Windows, colors are always off.
--help
Print the usage text and exit with code 0.

These examples show common ways to start the client.

# interactive shell, default localhost:9967 as admin
evosql-cli

# connect to a remote host, prompt for password
evosql-cli -h db.example.net -p 9967 -u alice

# one-shot query, machine-readable output
evosql-cli -F json -c "SELECT id, name FROM users"

# run a script file
evosql-cli -f schema.sql

# pipe a script in on stdin (runs non-interactively)
evosql-cli < schema.sql
Password handling. The client checks for the password in this order. First it checks -W. Then it checks EVOSQL_PASSWORD. Then it shows an interactive prompt. A secret on the command line is visible to other users in the process table. For anything sensitive, use the environment variable or the prompt instead.

31.2. Connecting and authentication

When it starts, the client opens a TCP socket. Then it runs the EVO handshake.

  1. The client sends the line EVO. This says which protocol it uses.
  2. The server replies with a line that starts with HELLO. Any other greeting is a fatal error.
  3. The server then sends one of STARTTLS, AUTH_REQUIRED, or AUTH_SCRAM.
  4. TLS support is a compile-time option. If STARTTLS is offered and the client has TLS support, the connection is upgraded to TLS before any credentials are sent. The next message is the auth request. It is read over the encrypted channel.
  5. For AUTH_REQUIRED or AUTH_SCRAM, the client sends AUTH username password and waits for AUTH_OK. Any other reply means the login failed and the client exits.

The password buffer and the AUTH message are wiped from memory right after the line is sent. This uses a volatile memset trick that the compiler cannot optimize away.

This client advertises SCRAM but does not use it. The server may offer AUTH_SCRAM, which is SCRAM-SHA-256. Even then, evosql-cli answers with the cleartext AUTH form. The server accepts both forms. If the connection is not protected by TLS, the credentials travel over the wire in cleartext.

31.2.1. TLS

TLS is a compile-time option. Build the client with -DEVOSQL_CLI_TLS and link -lssl -lcrypto. You can also use make TLS=1. What happens at the STARTTLS step depends on whether that flag was set.

Built with TLS
The client accepts STARTTLS. It runs an OpenSSL handshake with TLS_client_method(). Then it does the login over the encrypted channel. On success it prints [TLS] Encrypted connection established to standard error. The interactive banner shows (TLS).
Built without TLS
The client sends NOTLS to decline. It continues in plaintext. It prints a warning that the credentials will be sent in cleartext. It also suggests make TLS=1.
No certificate check. The TLS client creates a context and connects. It does not check the server certificate or hostname. This stops passive eavesdropping. It does not stop an active man-in-the-middle attack. There is no setting like psql's sslmode=verify-full.

31.3. The interactive shell

In interactive mode the client does several things at start. It prints a banner. It loads the command history. It fills the tab-completion cache. It installs its SIGINT handler. Then it enters the read loop. The main prompt is evosql>. While a statement is still being built up, the continuation prompt is ...>.

Connected to EvoSQL at localhost:9967
Type SQL (end with ';'), or \? for help, \q to quit.

evosql> SELECT 1 AS n;
n
-
1
(1 row)
evosql>

31.3.1. Statement assembly and the semicolon

A SQL statement ends with a semicolon. It does not end when you press Enter. Lines are added to a buffer. The client keeps reading until the last meaningful character in the buffer is a ; that sits outside any quoted string. The last meaningful character is the last one that is not whitespace and not part of a comment. A semicolon inside a string literal does not submit the statement. It keeps the continuation prompt instead.

evosql> INSERT INTO t(msg)
   ...> VALUES ('a; b; c');
INSERT 0 1
(1 row)

The EVO protocol sends each query as a single line. So newlines and carriage returns inside a multi-line statement are turned into spaces before the statement is sent. The client tracks quotes. It tells single quotes ' apart from double quotes ". A -- line comment outside quotes runs to the end of the line.

31.3.2. Quitting

There are a few ways to leave the shell. You can use the \q meta-command. You can type the word quit or exit on its own. Case does not matter for these words. They are kept because they feel familiar. You can also use end-of-file, which is Ctrl-D on an empty line. If the multi-line buffer is not empty, Ctrl-D throws away the pending buffer and shows the prompt again. It does not exit.

31.3.3. Ctrl-C, cancel, not quit

evosql-cli never quits on Ctrl-C. What it does depends on what the client is doing at the time.

At the prompt while typing
Ctrl-C drops the current line and any pending multi-line buffer. It draws a fresh prompt, like a shell does. On POSIX this works by long-jumping out of readline after the terminal is cleaned up. rl_catch_signals is turned off, so readline never raises the signal again.
While a query is running
The EVO protocol has no in-band query-cancel message. So the client sets an interrupt flag. It drops the socket, which is now out of sync. Then it reconnects a fresh session on its own and prints query canceled. The connection parameters are saved at startup so this reconnect can happen quietly. The canceled statement may have changed the schema. If it contained CREATE, DROP, or ALTER, the completion cache is refreshed afterward.
This is a real difference from psql. In PostgreSQL, Ctrl-C sends a cancel request on a side channel. The same session keeps going. In EvolutionDB the only way to stop a running EVO query from the prompt is to drop and reopen the socket. So any uncommitted transaction state on the old connection is lost. See transaction control.

31.3.4. Command history

History is saved to ~/.evosql_history. On Windows it is saved to %USERPROFILE%\.evosql_history. The file holds at most 500 entries. On POSIX the client uses GNU readline for history. This gives you arrow-key recall and in-line editing. On Windows the client has its own readline replacement built on the Win32 console API. It has the same 500-entry limit, up and down recall, and basic line editing.

Secrets stay out of history. A finished statement may contain the PASSWORD keyword on its own. Such a statement is kept out of the history file on disk. So a cleartext password from a command like CREATE USER never reaches the disk. The keyword check looks at any non-identifier character around it. So a form like PASSWORD'secret' is caught too.

31.3.5. Tab completion

Tab completion works on POSIX only, where the client uses GNU readline. When you press Tab, the client completes against a fixed list of SQL keywords and a cached list of table names. Matching ignores the case of the prefix you typed. There is no fallback to filenames. The table-name cache is filled at startup with SHOW TABLES. It is refreshed after any statement that contains CREATE, DROP, or ALTER. It is also refreshed after a reconnect that follows a canceled DDL statement. So new tables appear and dropped tables disappear without a restart. The keyword set is fixed. It includes the common DML and DDL words, join keywords, aggregate names, and a few type names.

SELECT FROM WHERE INSERT INTO VALUES UPDATE SET DELETE CREATE TABLE
DROP ALTER INDEX JOIN LEFT RIGHT INNER OUTER ON GROUP BY ORDER HAVING
LIMIT OFFSET DISTINCT AS AND OR NOT NULL LIKE IN BETWEEN PRIMARY KEY
FOREIGN REFERENCES DEFAULT UNIQUE CHECK BEGIN COMMIT ROLLBACK DATABASE
SCHEMA USER GRANT REVOKE SHOW TABLES DATABASES USERS COUNT SUM AVG MIN
MAX VARCHAR INT INTEGER TIMESTAMP BOOLEAN TEXT

Tab completion is not available in the Windows console build.

31.4. Output formats

You set the result format with -F or --format at startup. You can also change it with the \format meta-command while the client runs. The client supports three formats.

31.4.1. table (default)

This is an aligned grid. It has a bold header row. Then comes a dimmed -+- rule. Then the rows. At the end there is a dimmed (N row[s]) count. Column widths come from the widest value or header. A NULL value shows as the dimmed word NULL. In this format, status tags and the row count go to standard output. When \x expanded mode is on, each record gets a -[ RECORD n ]- banner. After it comes one column | value line for each field. This is handy for wide rows.

31.4.2. csv

This is CSV in the RFC 4180 style. It starts with a header row of column names. Then there is one line per row. A field is quoted only if it contains a comma, a double quote, CR, or LF. A double quote inside a field is doubled. A NULL value comes out as an empty field. You cannot tell it apart from an empty string. In CSV mode and JSON mode, only the data goes to standard output. Status tags, the row count, errors, and timing go to standard error. So a redirected stream holds clean data.

evosql-cli -F csv -c "SELECT id, name FROM users" > users.csv

31.4.3. json

This is a JSON array of objects. There is one object per row. Each object is keyed by column name. Strings use only the needed escapes. These are \", \\, \n, \r, \t, and \u00XX for other control characters. A NULL value comes out as JSON null. A value comes out as an unquoted JSON number only when two things are both true. First, the column's PostgreSQL type OID is numeric. Second, the value's text is a strict JSON number. If either is not true, the value is quoted. These are the numeric OIDs the client knows.

OIDType
20int8 (bigint)
21int2 (smallint)
23int4 (integer)
26oid
700float4 (real)
701float8 (double precision)
1700numeric

The OID for each column comes from the EVO COL line. Its form is <name> <pg_type_oid> <type_modifier>. The column name itself may contain spaces. So the client counts in from the end and drops the two trailing integer fields. It reads the OID from the second-from-last token.

evosql-cli -F json -c "SELECT id, name FROM users LIMIT 1"
[
 {"id": 1, "name": "alice"}
]
Limits to know. In CSV you cannot tell NULL from an empty string. The whole result set is held in memory before it is printed. There is no streaming and no paging. The internal row buffer starts at 64 rows and doubles when it needs more. There is no fixed cut-off in the EVO read path. The JSON output is not pretty-printed beyond one object per line.

31.5. Meta-commands

A line that starts with a backslash is a meta-command. A SQL statement ends with a semicolon. A meta-command is different. It ends with the newline. A trailing semicolon is allowed but not needed. Run \? for the built-in summary. Here is the full set the client supports.

\?, \h, \help
Print the meta-command reference.
\d, \dt
With no argument, this lists tables. It runs SHOW TABLES. With a name, it describes that table's columns. It queries information_schema.columns for column_name, data_type, is_nullable. See SHOW.
\l
List databases. This runs SHOW DATABASES. See CREATE DATABASE.
\du
List users. This runs SHOW USERS. See CREATE USER.
\dn
List schemas. This runs SHOW SCHEMAS.
\x
Turn the expanded display on or off. It shows one column per line. It prints the new state.
\timing
Turn per-statement timing on or off. It prints the new state.
\format F
Set the output format to table, csv, or json. Case does not matter. Any other argument prints a usage line.
\i FILE
Read FILE and run it as a script. It uses the same splitter as -f. The file may contain meta-commands too. It may even contain a nested \i.
\e
Open $EDITOR on a private temporary file made with mkstemp. The default editor is vi. When the editor exits cleanly, the file's contents are run as a script. This works on POSIX only. On Windows it prints that it is not supported.
\! CMD
Run CMD in the system shell.
\q, \quit
Quit the client.
Naming notes next to psql. The verbs are made to feel like psql. But the set is small and a few work a bit differently. \d and \dt are aliases. Both list tables. \d NAME describes columns. \l, \du, and \dn are thin wrappers over the matching SHOW command. There is no \copy. There is no \g. There is no variable system like \set. There is no \connect either. The connection is fixed for the life of the process. The one exception is the quiet reconnect after a Ctrl-C cancel.

31.6. Scripting and non-interactive use

The client runs in script mode in a few cases. It does so when you pass -c or -f. It also does so when standard input is not a terminal, for example when it is piped or redirected. In this mode it runs the SQL you gave. Then it sends a QUIT line, closes the connection, and exits.

31.6.1. The statement splitter

Script text can come from -c, -f, piped stdin, \i, or \e. It is split into statements by the same scanner used in interactive mode. That scanner is aware of quotes and comments.

  • The kind of each statement is fixed by its first non-blank character. A line that starts with \ is a meta-command. It ends at the newline. Anything else is SQL. It ends at an unquoted ;.
  • A -- line comment outside quotes runs to the end of the line. Inside a statement it becomes a single separating space. A comment line on its own is dropped.
  • Single and double quotes are tracked. So semicolons and comment markers inside string literals are not treated as syntax.
  • Newlines and carriage returns inside a SQL statement become spaces. The EVO protocol expects a single-line query.
  • A last statement with no final ; or newline is still run.
-- schema.sql
CREATE TABLE t (id INT PRIMARY KEY, name VARCHAR(50));
INSERT INTO t VALUES (1, 'alice'), (2, 'bob');
SELECT count(*) FROM t;

31.6.2. Exit status

The script-mode exit status is built so that CI jobs and shell scripts can detect a failed query.

0
All statements ran. The server reported no error.
1
Something went wrong. A statement failed and the run hit a non-zero internal code. This happens with an unreadable -f file, a lost connection, or a Ctrl-C cancel. It also happens when the server returned an ERR for any statement. An error-seen flag tracks this. The code stays 1 even if later statements succeed.
2
An invalid --format argument was given. The client exits before it connects.
evosql-cli -c "SELECT 1" && echo ok || echo failed
Errors do not stop a script. In script mode a server ERR sets the error flag. The run keeps going to the next statement. At the end the process still exits 1. A lost connection with code -1 does stop the script. A Ctrl-C cancel with code -2 also stops it. This is different from psql's default and from its ON_ERROR_STOP. There is no equivalent setting here.

31.7. Status tags, errors, and timing

The EVO protocol reports the result of each statement in one of three ways. It can send a command tag, for example INSERT 0 1, SELECT, or CREATE TABLE. It can send an error line. Or it can send a result set. The client shows these as follows.

Command tag
Printed as is. In table format it goes to standard output. In csv and json it goes to standard error. This keeps the data stream clean.
Error
Printed as ERROR: message to standard error. It is red when color is on. The error-seen flag is set. This lets script-mode runs exit with a non-zero code.
Row count
A (N row[s]) footer follows each result set. It goes to the same place as the command tag. That is stdout for table and stderr otherwise.
Timing
When \timing or --timing is on, a Time: X.X ms line is printed to standard error after each statement. This is the client-side wall time around the request and response. It is measured with clock(). It is not an execution time reported by the server.

31.8. Type fidelity

For display, the client does not care about types. Every value arrives as text on a FIELD line. It is printed as is in the table and csv formats. There is only one place where type info is used. That place is JSON output. There the column's PostgreSQL type OID decides whether a number-like value is printed without quotes. The OID is carried on the EVO COL line. See §31.4.3. For the full type system and how values are stored, see the SQL reference and the data-type chapters. For query syntax see SELECT, INSERT, UPDATE, and DELETE.

31.9. Notes and limitations

  • EVO port only. evosql-cli connects to the EVO text protocol on port 9967. To use the PostgreSQL wire protocol on port 5433, use psql or any PostgreSQL-compatible client. evosql-cli cannot talk to that port.
  • No in-band query cancel. When you cancel a running query, the client drops and reopens the connection. Any uncommitted transaction state on that session is lost.
  • SCRAM not done on the client side. The client always logs in with the cleartext AUTH form. It does so even when the server advertises SCRAM-SHA-256.
  • TLS does not check certificates. When built with TLS, the handshake stops passive eavesdropping only.
  • Results are fully buffered. A large result set is read into memory before it is shown. There is no paging and no streaming output.
  • One fixed connection. There is no \connect, no way to switch connections, and no variables. The host, port, user, and password are fixed for the life of the process.
  • NULL and empty string in CSV. Both show as an empty field. You can tell them apart only in json, where NULL is null, and in table, where NULL prints as NULL.
  • Platform differences. Tab completion and \e work on POSIX only. The Windows console build has history and basic line editing, but not these features.
