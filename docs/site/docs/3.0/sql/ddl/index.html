5. Data Definition

This chapter shows how EvolutionDB defines the objects that hold data. These objects are schemas, tables, columns, constraints, sequences, and domains. It also shows how you change and drop those definitions.

This chapter describes the data-definition language (DDL) that EvolutionDB really implements. Every behavior here comes from the engine's CREATE/ALTER/DROP handlers and the SQL grammar. When EvolutionDB differs from PostgreSQL or the SQL standard, the difference is stated clearly. For the full syntax of each statement, follow the links into the command reference under /3.0/ref/.

5.1. The Object Model

EvolutionDB stores all metadata in a system catalog. The catalog is held in B+ trees inside a single page file. Objects are nested four levels deep. The engine addresses them by the path root/database/schema/table.

Database
The top-level container. The session works inside one current database. See USE.
Schema
A named namespace inside a database. Tables, sequences, and views live in a schema. The session has one current schema. An object name without a schema is resolved against it.
Table
A base relation. It is an ordered list of columns plus its constraints and indexes. Rows are stored as binary tuples in heap pages. They are keyed by a primary-key B+ tree.
Column
A named, typed field. A table may have at most 256 columns (CAT_MAX_COLUMNS). Identifiers are limited to 127 usable characters. CAT_MAX_NAME_LEN is 128 and that includes the terminator.

Note. EvolutionDB is not like PostgreSQL here. It has no CREATE SCHEMA ... AUTHORIZATION. It has no schema search-path list. It has no owner role per object at the schema level. A name is resolved against exactly one current schema. Or it is resolved against the schema named in a schema.table reference.

5.2. Schemas

A schema is created inside the current database. CREATE SCHEMA takes only a name and an optional existence guard. There is no inline object list. There is no authorization clause.

CREATE SCHEMA sales;
CREATE SCHEMA IF NOT EXISTS sales;

-- Make it the active schema for unqualified names:
SET SCHEMA sales;
SET SCHEMA DEFAULT;

If the schema already exists, CREATE SCHEMA fails with SQLSTATE 42P06 for duplicate schema. The one exception is when you give IF NOT EXISTS. In that case it succeeds without a message. SET SCHEMA checks that the schema exists in the current database before it switches. A missing schema raises 3F000 for invalid schema name.

DROP SCHEMA always cascades in EvolutionDB. It drops every table in the schema first. Then it removes the schema itself.

DROP SCHEMA sales;
DROP SCHEMA IF EXISTS sales;

Caveat. There is no RESTRICT or CASCADE keyword on DROP SCHEMA. The behavior always matches PostgreSQL's CASCADE for the tables it contains. Dropping a schema that is not empty discards its tables with no warning. A schema that does not exist raises 3F000 unless you give IF EXISTS. See SET SCHEMA.

5.3. Creating Tables

CREATE TABLE defines a base table. The table goes in the current schema, or in a schema you name. A column list mixes column definitions with table-level constraints.

CREATE [ TEMPORARY | GLOBAL TEMPORARY ] TABLE [ IF NOT EXISTS ] [ schema . ]table_name (
    { column_name data_type [ column_attribute [ ... ] ]
    | table_constraint }
    [ , ... ]
)
[ table_option [ ... ] ]
[ INHERITS ( parent_table ) ]

Here is a small example.

CREATE TABLE employees (
    id        INT PRIMARY KEY AUTO_INCREMENT,
    name      VARCHAR(100) NOT NULL,
    email     VARCHAR(255) UNIQUE,
    salary    DECIMAL(10,2) DEFAULT 0,
    status    VARCHAR(20) DEFAULT 'active',
    hired_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

A table with the same name may already exist in the target schema. In that case CREATE TABLE fails with 42P07 for duplicate table. To avoid the error, add IF NOT EXISTS. The clause only hides the error. It does not compare the new definition with the existing one.

5.3.1. Temporary tables

The TEMPORARY prefix marks a table as session-local. The GLOBAL TEMPORARY prefix marks it as global-temporary. The ON COMMIT table option controls how long rows live across transaction boundaries. The prefix alone does not control this.

5.3.2. CREATE TABLE AS / SELECT

A table can be filled straight from a query. Two forms are accepted. One form takes the columns from the query. The other form lists the columns directly. You may add IGNORE or REPLACE in front of either form.

CREATE TABLE active_emps AS
    SELECT id, name FROM employees WHERE status = 'active';

5.3.3. Inheritance

INHERITS ( parent ) puts the parent's columns at the start of the child's own column list. A column name that appears in both parent and child is rejected with 42701 for duplicate column. EvolutionDB does not merge same-named columns the way PostgreSQL does. If the child declares its own primary key, the engine removes the inherited primary-key flag from the parent columns. The total column count must stay within the 256-column limit.

To drop a parent that is inherited from, you need CASCADE. See §5.7.

5.3.4. Range partitioning

A parent table declares PARTITION BY RANGE ( column ) as a table option. Then you attach child partitions to it.

CREATE TABLE events (id INT, ts INT) PARTITION BY RANGE (ts);

CREATE TABLE events_q1 PARTITION OF events
    FOR VALUES FROM (0) TO (100);

Each child copies the parent's columns. Each child registers an integer half-open range [lo, hi). The engine checks that lo < hi. It also checks that the new range does not overlap an existing partition. A table may hold at most 1024 partitions (EVO_MAX_PARTITIONS). Range bounds are integers only.

Note. The type is a single integer code. So ALTER TABLE ... MODIFY can only change a type within the same family. For CHAR/VARCHAR it can only make the length larger. Changes across families are rejected. See §5.6.2.

5.5. Constraints

EvolutionDB records five constraint kinds in the catalog. A single type byte tells them apart. The kinds are primary key (P), unique (U), check (C), foreign key (F), and domain (D). The domain kind is used inside the engine for domains. You can write a constraint inline on a column or at the table level. You can also give it a name with CONSTRAINT name.

5.5.1. Primary keys

A single-column primary key is declared inline. You write PRIMARY KEY, or you write the bare keyword KEY as a column attribute. A composite key uses the table-level form.

CREATE TABLE order_items (
    order_id  INT,
    line_no   INT,
    qty       INT NOT NULL,
    PRIMARY KEY (order_id, line_no)
);

The primary key is backed by a B+ tree. The engine creates this tree with the table. It is the table's clustering index. The composite-key form accepts up to 16 columns.

5.5.2. Unique constraints

The column attribute UNIQUE or UNIQUE KEY marks a single column unique. The table-level UNIQUE (cols) form covers one or more columns. You may declare up to 8 table-level unique constraints per CREATE TABLE. A unique constraint with no name gets the auto name table_cols_key.

5.5.3. Check constraints

A CHECK (expr) constraint stores a serialized boolean expression. Each row must satisfy it. It may be inline or table-level. It may have a name, or it gets the auto name table_check_n.

CREATE TABLE products (
    id     INT PRIMARY KEY,
    price  DECIMAL(10,2) CHECK (price >= 0),
    CONSTRAINT nonempty_name CHECK (name <> '')
);

5.5.4. Foreign keys

A foreign key references the primary key of another table. You may add a schema name to that table. Both the single-column and composite-column forms are supported. You can also set referential actions.

CREATE TABLE orders (
    id          INT PRIMARY KEY,
    customer_id INT,
    CONSTRAINT fk_cust FOREIGN KEY (customer_id)
        REFERENCES customers (id)
        ON DELETE CASCADE ON UPDATE RESTRICT
);

The referenced table must already exist. If it does not, you get 42P01. You may declare up to 8 foreign keys per CREATE TABLE. A foreign key with no name gets the auto name table_cols_fkey. The supported clauses and their stored action codes are listed below.

ON DELETE / ON UPDATE
CASCADE (1), SET NULL (2), RESTRICT (3), SET DEFAULT (4), NO ACTION (5). When you omit the clause, the default is action 0, which is no action.
MATCH
SIMPLE (0, the default), FULL (1), PARTIAL (2).
DEFERRABLE
NOT DEFERRABLE (0), DEFERRABLE / DEFERRABLE INITIALLY IMMEDIATE (1), DEFERRABLE INITIALLY DEFERRED (2). These modes are parsed and stored.

Note. A foreign key always references the parent's primary key B+ tree. A row is accepted when all of its foreign-key columns are NULL. This is true for any MATCH mode. The MATCH PARTIAL mode is recorded, but it does not fully match the standard.

5.6. Altering Tables

ALTER TABLE supports a wide set of subcommands. Each one runs right away. Each one checks existing data when that matters.

5.6.1. Adding and changing constraints

-- Add constraints (existing rows are validated first):
ALTER TABLE products ADD CONSTRAINT pos_price CHECK (price >= 0);
ALTER TABLE emp ADD CONSTRAINT u_email UNIQUE (email);
ALTER TABLE orders ADD CONSTRAINT fk_c FOREIGN KEY (customer_id) REFERENCES customers (id);
ALTER TABLE t ADD CONSTRAINT pk_t PRIMARY KEY (a, b);

-- Manage existing constraints:
ALTER TABLE t DROP CONSTRAINT u_email;
ALTER TABLE t RENAME CONSTRAINT u_email TO uq_email;
ALTER TABLE t ENABLE CONSTRAINT pos_price;
ALTER TABLE t DISABLE CONSTRAINT pos_price;

When you add a CHECK, the engine scans every existing row. If any row fails, it rejects the constraint and reports the row number. Adding a UNIQUE constraint runs an O(n²) duplicate scan. Adding a PRIMARY KEY also rejects NULL values in the key columns. The engine refuses if the table already has a primary key. A duplicate constraint name raises 42710. A missing one raises 42704.

Deferred validation. The NOT VALIDATE suffix on a new CHECK or FOREIGN KEY adds the constraint without scanning existing data. The constraint is marked as not validated. A later VALIDATE CONSTRAINT runs the scan and sets the validated flag.

ALTER TABLE t ADD CONSTRAINT c1 CHECK (x > 0) NOT VALIDATE;
ALTER TABLE t VALIDATE CONSTRAINT c1;

Note. The keyword spelling is NOT VALIDATE. It is not PostgreSQL's NOT VALID. A DISABLEd CHECK constraint is skipped when the engine reads constraints back. So it is no longer enforced. ENABLE brings it back.

5.6.2. Adding, dropping, and changing columns

ALTER TABLE emp ADD COLUMN phone VARCHAR(20);
ALTER TABLE emp ADD COLUMN active BOOLEAN DEFAULT true AFTER name;
ALTER TABLE emp DROP COLUMN phone;
ALTER TABLE emp RENAME COLUMN name TO full_name;
ALTER TABLE emp MODIFY COLUMN full_name VARCHAR(200);
ALTER TABLE emp CHANGE COLUMN full_name display_name VARCHAR(200);
ADD COLUMN
Adds a column at the end. You can move it with FIRST or AFTER col. That move rewrites every tuple right away. A NOT NULL column without a DEFAULT is rejected if the table already holds rows. The code is 23502. You cannot add a primary-key column this way. Going past 256 columns raises 54011.
DROP COLUMN
Marks the column dropped. It is refused with 2BP01 in some cases. These are when the column is part of the primary key, when it is the table's only column, or when it is referenced by a secondary index, a foreign key, or a unique constraint.
RENAME COLUMN
Renames a column. It also updates any secondary index column lists that used the old name. A name clash raises 42701.
MODIFY / CHANGE COLUMN
Changes a column's type and attributes. CHANGE also renames it, MySQL-style. The new type must be in the same family as the old one. CHAR/VARCHAR may only get wider. A change across families raises 42804. Changing a primary-key column's type raises 2BP01. Switching to NOT NULL scans for existing NULLs. It fails with 23502 if it finds any.

5.6.3. Renaming tables

You rename a table with the standalone RENAME TABLE statement.

RENAME TABLE employees TO staff;

A missing source raises 42P01. A target name that is already in use raises 42P07.

5.6.4. Table TTL

A table may pick a column whose value decides when rows expire. The background reclaim daemon uses that value. The column must exist on the table. If it does not, the statement fails with 42703.

ALTER TABLE sessions SET ttl_column = 'expires_at';
ALTER TABLE sessions DROP TTL;

5.7. Dropping and Truncating Tables

DROP TABLE removes a table and all of its storage. This includes the primary-key tree, heap pages, secondary indexes, constraints, RLS policies, and inheritance links.

DROP TABLE employees;
DROP TABLE IF EXISTS employees;
DROP TABLE employees CASCADE;

The drop is refused with 2BP01 in two cases. The first is when a view references the table. The second is when the table has inheritance children and you did not give CASCADE. With CASCADE, child tables are dropped first, in turn. RESTRICT is the default. A missing table raises an error unless you give IF EXISTS.

TRUNCATE empties one or more tables. It destroys and recreates each heap and index tree. This is much faster than a row-by-row delete.

TRUNCATE TABLE logs;
TRUNCATE TABLE t1, t2 CASCADE;
TRUNCATE TABLE logs CONTINUE IDENTITY;

Another table may reference the target through a foreign key and still have rows. In that case TRUNCATE fails with 23503. To allow it, give CASCADE. Then the referencing table is truncated first. By default the auto-increment counter is reset to its start. CONTINUE IDENTITY keeps it. Table statistics are reset.

5.8. Sequences

EvolutionDB provides PostgreSQL-style sequence objects. They are stored per schema. See CREATE SEQUENCE.

CREATE SEQUENCE IF NOT EXISTS order_seq
    START WITH 100
    INCREMENT BY 1
    MINVALUE 1
    MAXVALUE 1000000
    CYCLE;

SELECT nextval('order_seq');
SELECT setval('order_seq', 500);
START [ WITH ] n
The first value nextval returns. When you omit it, it defaults to MINVALUE.
INCREMENT [ BY ] n
The step between values. The default is 1. It may be negative for a sequence that counts down.
MINVALUE n / MAXVALUE n
The bounds. The defaults are min 1 and max 2147483647.
CYCLE / NOT CYCLE
This sets what happens at a bound. The sequence either wraps to the opposite bound or raises an error. The default is NOT CYCLE.

nextval is atomic. It runs under its own mutex. The first call returns the current value. That value is the start value. It also marks the sequence as called. Later calls add the increment. At a bound they wrap or raise an error, based on the cycle flag. setval sets the stored value directly.

You can alter, restart, rename, and drop sequences.

ALTER SEQUENCE order_seq INCREMENT BY 2 MAXVALUE 2000000;
ALTER SEQUENCE order_seq RESTART;
ALTER SEQUENCE order_seq RESTART WITH 1;
ALTER SEQUENCE order_seq RENAME TO ord_seq;
DROP SEQUENCE IF EXISTS ord_seq;

Caveat. ALTER SEQUENCE only applies options that differ from their defaults. An option left at its default value is treated as unspecified. For example INCREMENT 1, MINVALUE 1, or MAXVALUE 2147483647 all count as default. Such an option leaves the stored setting unchanged. RENAME works as a drop and recreate under the same id. There is no CACHE or OWNED BY clause. Column defaults reference generators like AUTO_INCREMENT/IDENTITY directly. They do not use an owned sequence.

5.9. Domains

A domain is a named, reusable type built on a base type. You may also add a default, a NOT NULL marker, and a CHECK. See CREATE DOMAIN.

CREATE DOMAIN positive_int AS INT CHECK (VALUE > 0);
CREATE DOMAIN email AS VARCHAR(255) NOT NULL;
CREATE DOMAIN score AS INT DEFAULT 0;

A domain is stored as a special constraint entry of type D in a system namespace. The entry packs its base type code, its NOT-null flag, and its serialized check. Creating a domain whose name already exists raises 42710.

Note. Domain support is small on purpose. It is less than PostgreSQL. The grammar accepts CREATE DOMAIN ... AS type [ DEFAULT expr ] [ NOT NULL ] [ CHECK (expr) ]. But the DEFAULT clause is parsed and then not saved in the current build. There is no DROP DOMAIN or ALTER DOMAIN statement. Treat domains as a light, early feature. Do not treat them as a full type system.
