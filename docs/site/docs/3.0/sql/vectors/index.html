14. Vectors & Hybrid Search

EvolutionDB stores dense embeddings in VECTOR(N) columns. Each vector has a fixed number of dimensions. You can rank vectors with four built-in distance metrics. An in-memory HNSW index makes nearest-neighbor queries faster. One hybrid-search call lets you mix an equality filter with vector ranking.

14.1. Overview

Vector support in EvolutionDB is built from four parts that work together.

  • The VECTOR(N) data type. It is a fixed-size array of single-precision floats. It is stored inline in the binary tuple.
  • Distance functions and operators. These are COSINE_DISTANCE, L2_DISTANCE, INNER_PRODUCT, and L1_DISTANCE. There are also the <-> and <#> operators.
  • The HNSW index. You create it with CREATE INDEX … USING HNSW. You query it through the HNSW_KNN function or a cosine ORDER BY.
  • Hybrid search with HNSW_FILTER_KNN. It folds a single equality predicate into vector ranking. It picks a pre-filter or post-filter strategy on its own.

Relationship to PostgreSQL. The syntax is made to look like the pgvector extension. The engine recognizes VECTOR(N), USING HNSW, the vector_cosine_ops opclass family, the WITH (m=…, ef_construction=…) parameters, and the <-> and <#> operators. But the implementation is EvolutionDB's own. It uses an in-memory graph backed by brute force. See the Notes for details. EvolutionDB is not pgvector. It does not promise the same recall, persistence, or operator coverage. This chapter only describes what the engine really does.

14.2. The VECTOR(N) Type

A VECTOR(N) column holds exactly N single-precision float4 components. You set the dimension when you run CREATE TABLE. The dimension is part of the column's type. Rows cannot store vectors of a different length.

VECTOR ( dimension )

Inside the engine, the dimension is folded into the type code as 260000 + N. The type family is 26. For example, VECTOR(3) is encoded as 260003 and VECTOR(1536) as 261536. The payload is stored inline in the tuple. It is N 4-byte little-endian floats in a row with no length prefix. The dimension comes from the column's type code, not from the stored bytes.

Storage
N × 4 bytes. The values are float4 little-endian. They are stored inline in the binary tuple. A VECTOR(1536) value takes 6144 bytes of column data.
Maximum dimension
16384. This limit is VECTOR_MAX_DIM. The engine rejects any dimension above it, whether declared or inserted.
Element type
32-bit IEEE float. You give values as decimal text. The engine converts them to float4 on insert.
NULL
A vector column can be NULL like any other column. The null bitmap marks it and no payload bytes are written.

14.2.1. Vector literals

You give a vector value as a string literal. There are three accepted forms.

[f1,f2,…]
A list of decimal floats in brackets. This is the standard text form. The engine uses it when it prints a vector back out.
{f1,f2,…}
A list in braces. The engine accepts it as the same kind of input.
b64i8:<base64>
A small quantized form. It is a little-endian float32 scale factor, then an int8[] array, all base64-encoded. On decode, each byte is dequantized as byte × (scale / 127). This keeps a 1024-dimension query literal near 1.4 KB. That stays well under the statement-size cap. It is meant for queries built by a machine.

Whitespace inside a list is ignored. A NaN inside a list is rejected as a parse error.

-- Declare and populate a table of embeddings
CREATE TABLE docs (
    id    INTEGER PRIMARY KEY,
    title VARCHAR(200),
    topic VARCHAR(40),
    emb   VECTOR(3)
);

INSERT INTO docs (id, title, topic, emb) VALUES
  (1, 'intro to storage',  'storage', '[0.10, 0.20, 0.97]'),
  (2, 'btree internals',   'storage', '[0.11, 0.19, 0.98]'),
  (3, 'wire protocol',     'protocol','[0.90, 0.05, 0.10]');

See CREATE TABLE for the full column grammar. See INSERT for value syntax.

14.3. Distance Functions and Operators

Four scalar distance functions are available. You can use them anywhere an expression is allowed. That includes a projection, a WHERE clause, and an ORDER BY clause. Each one takes two vectors of equal length. Each one returns a DOUBLE distance. A smaller distance means more similar.

FunctionOperatorReturnsDefinition
COSINE_DISTANCE(a, b)<=> (in ORDER BY only)double1 − (a·b) / (‖a‖·‖b‖). Returns 1.0 if either vector is all-zero.
L2_DISTANCE(a, b)<->doubleEuclidean distance, √Σ(aᵢ−bᵢ)².
INNER_PRODUCT(a, b)<#>doubleNegated dot product, −Σ(aᵢ·bᵢ). It is negated so the most similar rows sort first. This matches the other metrics.
L1_DISTANCE(a, b), doubleManhattan distance, Σ|aᵢ−bᵢ|.

Three more vector helper functions complete the set.

VECTOR_DIM(v)
The number of components in v.
VECTOR_NORM(v)
The L2 norm of v. This is its magnitude.
VECTOR_NORMALIZE(v)
The unit vector that points the same way as v.
-- Exact (un-indexed) nearest neighbors via a distance function in ORDER BY
SELECT id, title,
       L2_DISTANCE(emb, '[0.10, 0.20, 0.97]') AS dist
FROM   docs
ORDER BY dist ASC
LIMIT 5;

-- The L2 operator form is equivalent to L2_DISTANCE()
SELECT id, emb <-> '[0.10, 0.20, 0.97]' AS dist FROM docs ORDER BY dist;

Operator availability differs by metric. Only two infix operators exist. They are <-> for L2 and <#> for inner product. You can use both anywhere an expression is allowed. Cosine has no general infix operator. The <=> spelling works only in the ORDER BY position. For cosine anywhere else, call COSINE_DISTANCE(). There is no L1 operator at all. Use L1_DISTANCE() instead. This is narrower than pgvector. pgvector gives a full set of operators for each metric.

14.3.1. Cosine ORDER BY

You can rank a column by cosine distance against a query literal right inside ORDER BY. Use the <=> operator as shown below.

ORDER BY column <=> '[f1,f2,…]' [ ASC | DESC ]
SELECT id, title
FROM   docs
ORDER BY emb <=> '[0.10, 0.20, 0.97]' ASC
LIMIT 3;

Only the cosine sub-operator works in this position. If you use another comparison operator with a string literal in ORDER BY, such as =, <, or >, the grammar rejects it with a clear error. It is not ignored without warning.

14.4. The HNSW Index

For larger collections, use an HNSW index. HNSW stands for Hierarchical Navigable Small World. It finds nearest neighbors without scanning every row at query time. You create it with the USING HNSW access method on a single VECTOR(N) column.

CREATE INDEX [ IF NOT EXISTS ] name
    ON table USING HNSW ( column [ opclass ] )
    [ WITH ( m = int [, ef_construction = int ] ) ]

14.4.1. Operator classes (metric selection)

The opclass is optional. It names the distance metric the index ranks by. If you leave it out, the index uses cosine. If you give an opclass name the engine does not know, it falls back to cosine. It does not raise an error.

OpclassMetricInternal kind
vector_cosine_ops (default)cosine distanceHNSW_DIST_COSINE (0)
vector_l2_opsL2 / EuclideanHNSW_DIST_L2 (1)
vector_ip_ops / vector_inner_opsinner productHNSW_DIST_INNER (2)
vector_l1_opsL1 / ManhattanHNSW_DIST_L1 (3)

14.4.2. Build parameters

m
The most neighbors a node can have above layer 0. Nodes on layer 0 can have up to 2·m neighbors. The default is 16. A larger m connects the graph more and raises recall. It also uses more memory and more build time.
ef_construction
The beam width used while the engine inserts nodes during the build. The default is 64. Larger values build a better graph but take more build time. A bare CREATE INDEX … USING HNSW with no WITH clause uses ef_construction = 64. The brute-force build helper has its own default of 200. That default does not apply through DDL.

The query-time beam width is ef_search. It is fixed at 64 in this release. You cannot yet set it per query. If your k is larger than ef_search, the engine widens the beam to k for that query.

-- Cosine HNSW index with default m=16, ef_construction=64
CREATE INDEX docs_emb_idx ON docs USING HNSW (emb);

-- L2 metric with explicit build parameters
CREATE INDEX docs_emb_l2 ON docs
    USING HNSW (emb vector_l2_ops)
    WITH (m = 32, ef_construction = 200);

See CREATE INDEX and DROP INDEX for the general index grammar.

14.4.3. How the index is built and stored

When you run CREATE INDEX, the engine scans the table. It decodes the vector column from every live row into a dense in-memory store. Then it builds a multi-layer navigable graph over that store. Each indexed table and column pair is identified by (table_id, index_name). The graph and its vectors live only in process memory. There is no on-disk copy of the graph right now.

Search walks the graph when two things are true. The graph must have built without errors. The collection must be large enough to gain from it. The walk is a sub-linear greedy beam walk. Some collections have 2·ef_search rows or fewer. For those collections, or when graph construction hit an allocation failure, the engine falls back on its own to an exact brute-force top-k scan over the dense store. The brute-force path is always exact. The graph path is approximate.

14.4.4. Querying with HNSW_KNN

You run nearest-neighbor lookups against the index through the HNSW_KNN table function.

HNSW_KNN ( table_name, index_name, query_vector, k )
table_name, index_name
The table and the HNSW index to search.
query_vector
A vector literal in any accepted form. The accepted forms are […], {…}, and b64i8:. Its dimension must match the indexed column. A mismatch returns no result.
k
An integer literal. It is the number of nearest neighbors to return. They come back ordered by ascending distance.

Each result row holds the matching row's primary-key value and its distance to the query vector.

SELECT * FROM HNSW_KNN(
    'docs', 'docs_emb_idx',
    '[0.10, 0.20, 0.97]', 5
);

The index is not kept up to date or rebuilt on its own. An HNSW graph is a snapshot from the moment you ran CREATE INDEX. Rows you insert, update, or delete after that do not show up until you rebuild the index. This release also does not reload existing HNSW indexes when the server starts. To refresh the graph, run CREATE INDEX again. You can use REINDEX once it is available. This is very different from PostgreSQL indexes. PostgreSQL indexes are kept up to date inside each transaction on every write.

14.5. Hybrid (Lexical + Vector) Search

Hybrid search joins two things in one call. It uses a single equality predicate on a normal column. It also ranks by vector. This is the common "filter then rank by similarity" pattern. For example, you might want the documents most similar to an embedding. But you only want them among rows whose topic = 'storage'.

HNSW_FILTER_KNN ( table_name, index_name, query_vector, k,
                  'column=value[:mode]' )

The fifth argument is a string literal. It has the form column=value. You can add a :mode suffix to pick the execution strategy. If the filter is empty or missing, the call drops to a plain HNSW_KNN.

auto (default)
The engine measures filter selectivity. Selectivity is matching_rows / total_rows. It uses pre-filter when selectivity is below 10%. Otherwise it uses post-filter.
pre
Force pre-filter. The engine first keeps only rows where column = value. Then it computes distances and takes top-k. This is best when the filter is tight and has few matches.
post
Force post-filter. The engine ranks all vectors first. Then it drops hits whose column value does not match. It oversamples to make up for the dropped rows. This is best when the filter is loose and matches most rows.
-- Most-similar docs, restricted to topic = 'storage', auto strategy
SELECT * FROM HNSW_FILTER_KNN(
    'docs', 'docs_emb_idx',
    '[0.10, 0.20, 0.97]', 5,
    'topic=storage'
);

-- Force the pre-filter strategy
SELECT * FROM HNSW_FILTER_KNN(
    'docs', 'docs_emb_idx',
    '[0.10, 0.20, 0.97]', 5,
    'topic=storage:pre'
);

The filter compares against the column's text value. NULL column values never match. The predicate is a single equality only. Ranges, OR, and multi-column predicates are not part of the hybrid call. For richer filtering, use plain HNSW_KNN with a WHERE clause in a surrounding SELECT.

14.5.1. Inspecting the chosen strategy

There is a diagnostic sibling called HNSW_HYBRID_EXPLAIN. It takes the same arguments as HNSW_FILTER_KNN. Instead of results, it tells you which strategy the planner picked. It also reports the selectivity stats it used. Those stats are the candidate count, the number of filter matches, and the measured selectivity. The candidate count is the total number of rows scanned.

HNSW_HYBRID_EXPLAIN ( table_name, index_name, query_vector, k,
                      'column=value[:mode]' )
SELECT HNSW_HYBRID_EXPLAIN(
    'docs', 'docs_emb_idx',
    '[0.10, 0.20, 0.97]', 5,
    'topic=storage'
);

14.7. Notes and limitations

  • In-memory only. The HNSW graph and its dense vector store live in process memory. There is no on-disk copy of the graph. Indexes are not reloaded on startup. Rebuild with CREATE INDEX after a restart.
  • Snapshot semantics. The graph reflects the table only as of build time. Later DML is not kept up to date in the index. Re-create the index to pick up changes.
  • Approximate vs. exact. The graph path is an approximate nearest-neighbor search. Small collections with 2·ef_search rows or fewer use an exact brute-force scan instead. Build-failure fallbacks use the same exact scan.
  • Fixed query beam width. ef_search is fixed at 64. It is widened to k when k is larger. There is no per-query SET ef_search yet.
  • One column per HNSW index. An HNSW index covers a single VECTOR(N) column.
  • Single equality in hybrid search. HNSW_FILTER_KNN supports exactly one column = value predicate. It is not a general WHERE engine.
  • Bounded internal capacity. The active HNSW graph registry holds a fixed number of graphs. That number is 64. It holds primary-key keys up to 256 bytes. Very long composite keys are truncated for index bookkeeping.
  • Operator coverage. Only <-> for L2 and <#> for inner product are general operators. Cosine through <=> works only in ORDER BY. L1 has no operator. PostgreSQL and pgvector give a full operator family per metric and push distance operators into the index. EvolutionDB does not do either.
