13. JSON

EvolutionDB stores JSON documents as text. It checks each document when you write it. It gives you a MySQL-compatible set of functions. It also gives you the -> and ->> path operators to read, change, and search documents.

This chapter describes the JSON data type. It also describes every JSON operation that EvolutionDB supports. The JSON support is small and self-contained. It uses little memory and is built on a recursive-descent parser. It follows the MySQL function set and the MySQL $.path expression syntax. It does not follow the PostgreSQL json/jsonb operator set. It does not follow the SQL/JSON standard. When the behaviour differs from those systems, this chapter says so clearly.

At a glance. EvolutionDB JSON is text-backed. A JSON column holds the original document bytes. EvolutionDB checks the bytes when you write them. There is no binary or decomposed storage form. There is no path indexing. You cannot change a document in place. The transformation functions return a new string. If you have used MySQL's JSON functions, this will feel familiar. If you have used PostgreSQL jsonb, the operators here are different.

13.1. The JSON Data Type

EvolutionDB has one JSON type. You can write it as JSON or JSONB. The parser accepts both spellings. Both map to the same type. There is no separate binary form. So JSONB is only an alias. It exists so tools that emit JSONB still work.

PropertyValue
Type namesJSON, JSONB (alias)
Internal type code230000 (type family 23)
On-disk formOriginal JSON text, stored in the binary tuple format like any variable-length value
ValidationDocument is parsed and rejected if not well-formed JSON (on write)
Maximum nesting64 levels (JSON_MAX_DEPTH)

EvolutionDB checks a value before it goes into a JSON column. It uses the same parser as JSON_VALID. An invalid document is rejected at write time.

CREATE TABLE events (
  id    INT PRIMARY KEY,
  payload JSON
);

-- accepted: well-formed object
INSERT INTO events VALUES (1, '{"user":"ada","tags":["x","y"],"active":true}');

-- rejected: trailing comma is not valid JSON
INSERT INTO events VALUES (2, '{"user":"ada",}');
-- ERROR: invalid JSON: '{"user":"ada",}'   (SQLSTATE data_exception)

See CREATE TABLE for column declaration and INSERT for write semantics.

Validation grammar. The parser accepts standard JSON. This means objects, arrays, and strings. Strings may use the escapes \" \\ \/ \b \f \n \r \t and \uXXXX. It also accepts numbers. A number may have an optional sign, an integer part, a fraction, and an exponent. It accepts the literals true, false, and null. It rejects integers with a leading zero. It rejects control characters inside strings. It rejects documents deeper than 64 levels. PostgreSQL jsonb removes duplicate object keys and reorders keys. EvolutionDB does not do this. It stores the document exactly as written. So key order and whitespace are kept.

13.2. Path Expressions

Some functions and operators point to a place inside a document. They use MySQL-style path expressions. A path starts with $. This is the document root. After it come zero or more steps.

.key
Reads a member of an object by its key name.
[n]
Reads an element of an array by its index. The index starts at zero.

You can chain steps to any depth. For example $.user.name, $.tags[0], or $.items[2].sku. A bare $ means the whole document.

-- given payload = {"user":"ada","tags":["x","y"],"address":{"city":"berlin"}}
SELECT JSON_EXTRACT(payload, '$.user')        FROM events;  -- "ada"
SELECT JSON_EXTRACT(payload, '$.tags[1]')     FROM events;  -- "y"
SELECT JSON_EXTRACT(payload, '$.address.city') FROM events;  -- "berlin"
SELECT JSON_EXTRACT(payload, '$')             FROM events;  -- whole document
Supported subset. Only two kinds of step work here. The first is a key step with .key. The second is a numeric index step with [n]. Some features are not supported. Wildcards like $.* and $[*] do not work. The recursive descent operator $** does not work. Quoted member names like $."weird key" do not work. Negative indexes and range indexes do not work. An index step needs at least one digit. A key step needs a name that is not empty. If not, the path is rejected. Key names are matched against an internal 256-byte buffer. Very long keys are cut short for the comparison.

13.3. The -> and ->> Operators

The arrow operators are a short way to extract a value. The left side is a JSON expression. The right side is a path. They are left-associative.

json -> path
Extracts the value at path and keeps its JSON form. A string result keeps its quotes.
json ->> path
Extracts the value at path as text. A JSON string result loses its quotes. A JSON null becomes SQL NULL.
-- payload = {"user":"ada","age":31}
SELECT payload -> '$.user'  FROM events;  -- "ada"   (quoted JSON string)
SELECT payload ->> '$.user' FROM events;  -- ada     (plain text)
SELECT payload -> '$.age'   FROM events;  -- 31

Both operators return SQL NULL in two cases. The first is when either side is NULL. The second is when the path matches nothing in the document. The right side uses the same $.path syntax as the functions above. A bare key with no leading $ is also accepted. It is treated as a single object-key step.

Contrast with PostgreSQL. In PostgreSQL, -> takes a single key or array index. That is one text value or one integer. The #> and #>> operators take a path array. The @>, ?, and jsonb_path_* operators complete the set. EvolutionDB is different. Its -> and ->> accept a whole MySQL $.path on the right. It has no #>, @>, or ? operators. For containment and existence checks, use the JSON_CONTAINS and JSON_CONTAINS_PATH functions instead.

13.4. Inspection Functions

These functions read a document. They do not change it.

JSON_VALID(doc)
Returns 1 if doc is well-formed JSON. Returns 0 if it is not. This is the same check used when you write to a JSON column.
JSON_TYPE(doc)
Returns the type of the top-level value. It is one of OBJECT, ARRAY, STRING, BOOLEAN, NULL, INTEGER, or DOUBLE. A number with ., e, or E is reported as DOUBLE. Any other number is reported as INTEGER.
JSON_LENGTH(doc)
For an object, this is the number of keys. For an array, it is the number of elements. For a scalar, it is 1.
JSON_DEPTH(doc)
The nesting depth. A scalar or an empty container has depth 1. Each nested object or array adds one level.
JSON_KEYS(doc)
Returns a JSON array of the top-level object's key names. The names keep their quotes. For example ["user","tags"]. This works only for objects.
JSON_EXTRACT(doc, path)
Returns the value at path in its JSON form. Returns NULL if the path matches nothing. This is the same as the -> operator.
-- payload = {"user":"ada","tags":["x","y"],"meta":{"v":1.5}}
SELECT JSON_VALID(payload)   FROM events;  -- 1
SELECT JSON_TYPE(payload)    FROM events;  -- OBJECT
SELECT JSON_LENGTH(payload)  FROM events;  -- 3
SELECT JSON_DEPTH(payload)   FROM events;  -- 3
SELECT JSON_KEYS(payload)    FROM events;  -- ["user","tags","meta"]
SELECT JSON_TYPE('1.5')        FROM events;  -- DOUBLE
SELECT JSON_TYPE('42')         FROM events;  -- INTEGER

13.5. Construction and Formatting Functions

JSON_OBJECT(k1, v1[, k2, v2])
Builds a JSON object from key and value pairs. It takes one or two pairs. A value that is itself valid JSON is kept as-is. Any other value is quoted as a JSON string.
JSON_ARRAY(v1[, v2[, v3]])
Builds a JSON array from one to three values. It uses the same rule for each element. Valid JSON is kept as-is. Anything else is quoted as a string.
JSON_QUOTE(str)
Wraps str in double quotes. It escapes special characters. These are ", \, newline, carriage return, tab, and control characters as \uXXXX. The result is a JSON string literal.
JSON_UNQUOTE(doc)
The reverse of JSON_QUOTE. If doc is a quoted JSON string, it removes the quotes and unescapes the text. If not, it returns the input unchanged.
JSON_PRETTY(doc)
Renders the document again with two-space indentation and newlines. If the input cannot be parsed, it returns the original text unchanged.
SELECT JSON_OBJECT('name', 'ada', 'age', '31');
-- {"name":"ada","age":31}    (31 is valid JSON, embedded; "ada" quoted)

SELECT JSON_ARRAY('1', 'two', 'true');
-- [1,"two",true]

SELECT JSON_QUOTE('a"b');     -- "a\"b"
SELECT JSON_UNQUOTE('"a\nb"'); -- a<newline>b
Argument-count limits. JSON_OBJECT takes one or two key and value pairs. A four-argument call is merged into one object. JSON_ARRAY takes one, two, or three elements. These argument counts are fixed in the grammar. They are not the variadic forms that MySQL and PostgreSQL have. To build larger objects or arrays, you must put the JSON text together in another way.

13.6. Modification, Containment, and Search

The modification functions return a new document. They never change stored data in place. Use them inside an UPDATE to save a change.

JSON_SET(doc, path, val)
Adds the key if it is missing. Replaces its value if the key is there.
JSON_INSERT(doc, path, val)
Adds the key only if it does not already exist. It leaves existing keys alone.
JSON_REPLACE(doc, path, val)
Updates the key only if it already exists. It does not add missing keys.
JSON_REMOVE(doc, path)
Deletes the key at path.
JSON_CONTAINS(target, candidate)
Returns 1 if candidate is inside target. The match can be scalar equality. It can be a subset of object keys and values. It can be array-element membership, checked recursively. Otherwise it returns 0.
JSON_CONTAINS_PATH(doc, mode, path)
Returns 1 if the path exists in doc. mode is 'all' or 'one' and is case-insensitive. Any value other than 'all' is treated as 'one'.
JSON_SEARCH(doc, mode, str)
Searches string values for an exact match of str. It returns the matching paths as JSON path strings. For example "$.user". With mode 'all' it returns every match, separated by commas. Any other mode stops at the first match. It returns NULL when nothing matches.
-- payload = {"user":"ada","age":31}
UPDATE events SET payload = JSON_SET(payload, '$.age', '32') WHERE id = 1;
-- {"user":"ada","age":32}

UPDATE events SET payload = JSON_REMOVE(payload, '$.age')   WHERE id = 1;
-- {"user":"ada"}

SELECT JSON_CONTAINS('{"a":1,"b":2}', '{"a":1}');   -- 1
SELECT JSON_CONTAINS_PATH('{"a":1}', 'one', '$.a');  -- 1
SELECT JSON_SEARCH('{"u":"ada","v":"bob"}', 'one', 'ada');  -- "$.u"
Single-key limitation on modification. JSON_SET, JSON_INSERT, JSON_REPLACE, and JSON_REMOVE work on top-level object keys only. The document must be an object. The path must be a single $.key step. Nested paths like $.a.b are rejected. Array indexes like $.a[0] are rejected. This is a planned simplification. The MySQL versions accept any path and many path and value pairs. JSON_SEARCH matches only string values. It skips numbers, booleans, and null. It needs an exact, full-string match. It does not use a LIKE pattern.

13.8. Notes and Limitations

Implementation notes.
  • Text storage, not binary. JSON and JSONB are the same text-backed type. There is no decomposed or indexed binary form. So whitespace, key order, and duplicate keys are kept exactly as written.
  • No JSON indexing. A query that filters on ->> or JSON_EXTRACT does a full scan. There are no expression or path indexes over JSON.
  • Fixed buffers. JSON expression evaluation uses fixed-size buffers. A document can be up to about 4 KB per evaluation step. A key can be up to 256 bytes. The output is limited by the column or result buffer. Very large documents or values may be cut short.
  • MySQL flavour, not PostgreSQL or SQL/JSON. The function names, the $.path syntax, and the JSON_TYPE labels follow MySQL. There is no jsonb. There are no @>, ?, or #> operators. There is no JSON_TABLE. There are no SQL/JSON JSON_VALUE or JSON_QUERY constructs.
  • Modification is shallow. You can only set, insert, replace, or remove single top-level object keys.
