11. Transactions & MVCC

How EvolutionDB groups statements into atomic transactions. How it keeps connections apart with multi-version snapshots. How it guards rows from writers that conflict.

This chapter explains how EvolutionDB 3.0 handles many things at once. It covers explicit transaction blocks, savepoints, and the four SQL isolation levels. It also covers the snapshot rule that lets readers and writers run at the same time without blocking each other. And it covers the row locks and gap locks that writers use, and that the SERIALIZABLE level uses. For the exact syntax of each statement, follow the links to the transaction, savepoint, SET, and SELECT reference pages.

11.1. Overview

Every statement in EvolutionDB runs inside a transaction. If you have not opened an explicit block, each statement runs in its own auto-committed transaction. The change is applied and made durable as soon as the statement succeeds. Opening a block with BEGIN holds back that commit until you say so. A group of changes either all take effect together or none of them do.

Concurrency control uses two separate parts.

Undo log (rollback)
The undo log is a stack of undo entries. Each connection has its own. Every INSERT, UPDATE, and DELETE inside an open block pushes an entry. The entry records how to reverse the change. COMMIT throws the log away. ROLLBACK replays it in reverse. This gives atomicity per connection.
MVCC (isolation)
Each transaction gets a 32-bit transaction id (XID). Row versions carry the XID that created them (xmin) and the XID that deleted them (xmax). A transaction reads through a snapshot. The snapshot decides which versions are visible. Reads never block writes and writes never block reads.

The two layers work together. An UPDATE or DELETE does not overwrite or remove a row on disk. It sets the old version's xmax to the deleting XID. This is a "soft delete". For an update, it also inserts a new version. Rollback then just clears xmax to bring the original version back. It only falls back to a re-insert for rows written before MVCC existed.

Note. A write-ahead log gives durability. Before a COMMIT is acknowledged, the transaction's dirty pages are flushed to the WAL. Then a commit sequence number (CSN) is assigned. Then the transaction is marked committed in the commit log (CLOG). You can turn on the optional synchronous-commit mode with EVOSQL_SYNC_COMMIT=1. When it is on, the commit also waits for a replica to acknowledge its WAL position. If that wait times out, the commit quietly falls back to asynchronous commit.

11.2. Transaction control statements

The session layer recognizes transaction control commands by their prefix. They do not go through the SQL parser. These are the verbs it knows.

BEGIN
START TRANSACTION

COMMIT
END

ROLLBACK

SAVEPOINT savepoint_name
ROLLBACK TO [ SAVEPOINT ] savepoint_name
RELEASE [ SAVEPOINT ] savepoint_name

11.2.1. BEGIN / START TRANSACTION

BEGIN and its synonym START TRANSACTION open an explicit block. On BEGIN the server creates a new undo log. It sets up an empty savepoint stack. It assigns the transaction's XID. And it marks the transaction as active. At REPEATABLE READ and SERIALIZABLE the server takes one snapshot right away. See 11.4 for more. At READ COMMITTED and READ UNCOMMITTED no snapshot is taken for the whole transaction. Each statement takes its own snapshot.

BEGIN;
INSERT INTO accounts VALUES ('alice', 100);
UPDATE accounts SET balance = balance - 25 WHERE name = 'alice';
COMMIT;
Note. The isolation level is not read from words typed after BEGIN. The verb is matched by its prefix. Any extra text is accepted but ignored. For example, BEGIN ISOLATION LEVEL SERIALIZABLE ignores the extra words. Use SET TRANSACTION ISOLATION LEVEL first instead. If you run BEGIN while already inside a block, the server logs a warning and does nothing else. The existing block continues. It does not open a nested transaction.

11.2.2. COMMIT / END

COMMIT finishes the block. Its synonym is END. The server flushes dirty pages to the WAL. It gets a CSN. It marks the XID committed in the CLOG. It releases all row and gap locks the transaction held. It removes the XID from the active list. And it throws away the undo log. From that moment the changes are durable. They are visible to other connections that take their snapshots after the commit.

11.2.3. ROLLBACK

ROLLBACK drops the block. The undo log is replayed in reverse. This restores every inserted, updated, or deleted row. The XID is marked aborted in the CLOG. And its locks are released. Deletes and updates are soft. Rollback usually just clears the xmax marker on the affected tuples. It does not rewrite any data.

DML in the blockUndo entryWhat ROLLBACK does
INSERTTX_OP_INSERTDelete the inserted row and remove its primary-key index entry.
UPDATETX_OP_UPDATEDelete the new version. Restore the old version by clearing its xmax. Re-insert the old data if the old slot was already gone.
DELETETX_OP_DELETEClear xmax to bring the tuple back. Re-insert the old data for pre-MVCC rows.
Note. If a statement inside the block raises an error, the transaction enters an aborted state. While aborted, the server only accepts ROLLBACK, COMMIT, END, SAVEPOINT, and RELEASE. It rejects any other statement until the block is closed. If you run COMMIT on an aborted transaction, the server does a ROLLBACK instead. All work is discarded. This matches PostgreSQL. If a connection disconnects with a block still open, the server rolls it back on its own.

11.3. Savepoints

Savepoints mark a position in the undo-log stack. They let you roll back part of a transaction without dropping the whole block. They only work inside an open transaction. See the SAVEPOINT reference for the full syntax.

SAVEPOINT name
Records the current undo-log depth under name and pushes it on the savepoint stack. If you re-use an existing name, it is overwritten in place. This matches PostgreSQL. Names are matched without case. Up to 64 savepoints may be active at once.
ROLLBACK TO [SAVEPOINT] name
Replays undo entries down to the recorded depth. This undoes all work done after the savepoint. Then it destroys any savepoints made after it. The savepoint itself stays and you can roll back to it again. This also clears the aborted state. You can keep going after a statement error as long as you set a savepoint before it.
RELEASE [SAVEPOINT] name
Forgets name and all newer savepoints. It does not undo any work. The data changes stay part of the still-open transaction.
BEGIN;
INSERT INTO log VALUES (1, 'start');
SAVEPOINT sp1;
INSERT INTO log VALUES (2, 'maybe');
ROLLBACK TO sp1;          -- row (2,'maybe') is undone; row (1,'start') stays
INSERT INTO log VALUES (3, 'final');
COMMIT;                    -- rows (1,'start') and (3,'final') persist
Note. In EvolutionDB a savepoint is a position in the undo log of one connection. It rolls back data changes made since the savepoint. It does not take a separate MVCC sub-snapshot. The visibility snapshot in use stays the transaction's snapshot or the statement's snapshot. It is not tied to the savepoint. If you run SAVEPOINT, ROLLBACK TO, or RELEASE outside a transaction, the server raises SQLSTATE 25P01. If you name a savepoint that does not exist, it raises SQLSTATE 3B001 (savepoint does not exist).

11.4. Isolation levels

EvolutionDB supports all four standard isolation levels. Set the level for the session and for the next transaction before you open a block. Use SET to do this.

SET [ SESSION ] TRANSACTION ISOLATION LEVEL
    { READ UNCOMMITTED | READ COMMITTED | REPEATABLE READ | SERIALIZABLE }

The default level for a new connection is READ COMMITTED. If you set the level after a transaction has already run a query, the server raises SQLSTATE 25001. If the level is not one it knows, it raises SQLSTATE 22023.

LevelInternal codeSnapshot strategyExtra machinery
READ UNCOMMITTED0One snapshot per statement. It is treated like READ COMMITTED.None.
READ COMMITTED (default)1A new snapshot is taken at the start of every statement.None.
REPEATABLE READ2One snapshot is taken at BEGIN and used for the whole transaction.None.
SERIALIZABLE3One snapshot is taken at BEGIN, like REPEATABLE READ.Conflict guard for write-skew detection, plus gap locks and a serializer mutex.

Here is an honest comparison with the SQL standard and PostgreSQL.

  • READ UNCOMMITTED. The standard allows dirty reads at this level. EvolutionDB does not do dirty reads. Uncommitted versions are never made visible. This level acts exactly like READ COMMITTED. This matches PostgreSQL. PostgreSQL also turns READ UNCOMMITTED into READ COMMITTED.
  • READ COMMITTED. Each statement sees a snapshot taken when it starts. Two statements in the same transaction can see different committed data. Non-repeatable reads and phantoms are possible. This is the standard meaning of the level.
  • REPEATABLE READ. The whole transaction reads through one snapshot. Rows it has already read do not change under it. EvolutionDB does not do first-updater-wins abort detection at this level. It does not give the stricter "snapshot isolation" anomaly set that PostgreSQL's REPEATABLE READ gives.
  • SERIALIZABLE. This adds a conflict guard. The guard tracks read and write dependencies. It rejects a commit that forms a dangerous shape known as write skew. It also adds gap locks to block phantom inserts into a scanned range. To keep a single serial order across the whole system, it takes a process-wide mutex for the life of the transaction. DML from other connections waits until this transaction commits or rolls back. A failed serialization check aborts the commit with SQLSTATE 40001 (could not serialize access due to read/write dependencies among transactions).
Note. SERIALIZABLE holds a global DML mutex for the whole life of the transaction. It serializes writes across all connections. It does not allow the optimistic concurrency that PostgreSQL's SSI allows. Use it as a correctness tool for critical sections. Do not use it as a high-throughput level. The conflict guard tracks a limited number of read interests and predicate locks per transaction. It catches common write-skew patterns. But it is not a full dependency-graph analysis.

11.5. Snapshots & visibility

A snapshot is a view of transaction activity at one moment in time. It records the following fields.

xmin
The lowest XID still active when the snapshot was taken. Every XID below this is finished.
xmax
One past the highest XID assigned so far. Every XID at or above this is "in the future" and invisible.
active_xids[]
The list of XIDs that were in progress at snapshot time. Up to 256 transactions running at once are tracked.
my_xid
The XID of the transaction that took the snapshot. It is used to recognize its own writes.
snapshot_csn
The highest committed CSN at snapshot time. It gives an O(1) "committed before me" fast path.

A heap tuple stores its creator XID (xmin) and deleter XID (xmax) in its MVCC header. The visibility rule decides whether a tuple is visible to a snapshot. It works roughly like this.

  1. Tuples without an MVCC header are pre-MVCC rows. They are always visible. A frozen xmin is the reserved value 1. It is treated as committed long ago and is always visible.
  2. If xmin is the snapshot's own XID, the tuple is the reader's own insert. It is visible unless the same transaction also deleted it.
  3. In other cases xmin must be committed in the CLOG, below xmax, and not in the active list. Only then is the creating transaction's work visible.
  4. If xmax is 0, the tuple was never deleted and is visible. If the deleter is this transaction, it is invisible. The tuple is still visible if the deleter has not committed, or committed after the snapshot, or was active at snapshot time. The tuple counts as dead only when the deleter committed and is visible in this snapshot.

The engine avoids repeating CLOG and CSN lookups. It uses hint bits and a CSN ring-buffer cache. The hint bits are XMIN_COMMITTED and XMIN_ABORTED. They are stamped on the tuple after the first successful check. None of this changes the result of the rule. It only makes it faster.

11.5.1. Temporal queries (FOR SYSTEM_TIME)

You can read the database as it was at a past transaction. The engine builds a snapshot placed just after a given XID committed. Every committed XID up to and including that value is visible. Anything assigned later is invisible. EvolutionDB offers this with a SQL:2011-style clause on SELECT.

-- current transaction id of this session
SELECT evo_current_xid();

-- read a table as it stood right after transaction 42 committed
SELECT * FROM orders FOR SYSTEM_TIME AS OF TRANSACTION 42;
Note. The historical view does not track concurrency. It has no active list. It shows a clean image with only committed data at the chosen XID. How far back you can go is limited by retention. The CSN cache and commit-timestamp cache use a ring buffer. If a snapshot's XID has dropped out of the kept window, the server rejects it. It does not return a misleading result.

11.6. Row locks & explicit locking

MVCC means readers and writers do not block each other. But writers to the same row must run one at a time. EvolutionDB has a row-level lock manager. It is keyed by the table_id and the primary-key value. Locks are held by XID. They are released on their own at COMMIT, at ROLLBACK, at the end of an auto-commit statement, or at disconnect.

ModeAcquired byCompatibility
EVO_LOCK_EXCLUSIVEUPDATE, DELETE, SELECT ... FOR UPDATEOne holder only. It conflicts with everything.
EVO_LOCK_SHAREDSELECT ... FOR SHAREIt works with other shared holders. It conflicts with exclusive.

Locking is re-entrant. If the same XID locks a row it already holds, nothing happens. The same holder can also upgrade a shared lock to exclusive. If a request conflicts with another transaction's lock, it waits on a condition variable. It waits until the lock is released or the timeout runs out.

BEGIN;
SELECT balance FROM accounts WHERE name = 'alice' FOR UPDATE;
UPDATE accounts SET balance = balance - 10 WHERE name = 'alice';
COMMIT;   -- row lock released here

11.6.1. Lock waits, timeout & deadlocks

The default lock wait timeout is 5000 ms. You can override it per session with SET EVO_LOCK_TIMEOUT = ms. A value of 0 restores the default. If a waiter does not get the lock within the timeout, the operation fails with a lock-timeout error (EVO_LOCK_TIMEOUT).

Before it sleeps, the lock manager runs a wait-for-graph cycle check. It follows the chain of who each holder is waiting for. If that chain leads back to the requester, it reports a deadlock right away with EVO_LOCK_DEADLOCK. It does not wait for the timeout. The search has a fixed depth limit.

EVO_LOCK_OK (0)
Lock granted (or no lock needed).
EVO_LOCK_TIMEOUT (-1)
Wait exceeded the timeout.
EVO_LOCK_ERROR (-2)
Internal lock-manager error.
EVO_LOCK_DEADLOCK (-3)
Granting the wait would create a wait-for cycle.
Note. Shared locks use a single holder slot per row. The last writer wins for the recorded holder. FOR SHARE is treated as compatible, but it is not fully multi-holder. SKIP LOCKED is accepted on FOR UPDATE and FOR SHARE for syntax compatibility. Row locks are keyed by primary key. A table without a usable primary key cannot take a row lock. In that case the request does nothing.

11.6.2. Gap locks (phantom prevention)

Gap locks stop phantom inserts in SERIALIZABLE transactions. A range scan can take a gap lock. The lock covers a key interval (lo, hi] on a table. An empty bound means −∞ or +∞. If another transaction tries to INSERT a key that falls inside an active gap lock held by a different XID, the insert waits. It uses the same timeout as row locks. It waits until the gap lock is released at the holder's commit or rollback. Gap locks do not conflict with each other. They conflict only with inserts. They are taken on their own during SERIALIZABLE range scans. There is no explicit statement for them.

11.7. XID lifecycle & reserved ids

Transaction ids are 32-bit values. A counter that only goes up hands them out. The counter is kept in the database file header. The ids survive a restart. A few values are reserved.

0, pre-MVCC sentinel
A tuple with no MVCC header. It is always visible. It is also used as "no deleter" in xmax.
1, XID_FROZEN
It means "committed long ago". An xmin of 1 is always visible. The server does not check the CLOG for it.
2+, normal
The first transaction id that can be assigned. Ids are assigned in increasing order.

The CLOG stores two status bits per XID. 00 means in-progress. 01 means committed. 10 means aborted. The bits are packed into 4 KB pages. Each page covers 16,320 ids and is chained for overflow. At startup the server records the first XID it assigns. Any XID below that value still marked in-progress belongs to a session that crashed. It is treated as aborted. Its half-written rows never become visible.

Note. For auto-commit DML, XID assignment is lazy. A transaction id is not given out until the first write. Read-only statements do not use up ids. The active transaction array holds up to 256 entries. This sets the limit on how many transactions running at once a snapshot can track exactly.
