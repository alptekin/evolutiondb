Aggregate Functions

Aggregate functions reduce a set of rows to a single value. You use them in SELECT, HAVING, and window function frames. EvolutionDB supports COUNT, SUM, AVG, MIN, MAX, and GROUP_CONCAT.

An aggregate function runs after the WHERE filter. It runs before ORDER BY. When you add a GROUP BY clause, rows are split into groups first. The aggregate then runs once per group. Without GROUP BY, all rows form one group and the aggregate gives a single result row.

The main implementation lives in evolution/db/Select.c. The join pipeline in adaptor/join.c handles aggregates for multi-table queries.

Aggregate Functions

COUNT

COUNT(*)
COUNT(expr)

COUNT(*) returns the number of rows in the group. It includes rows where columns are NULL.

COUNT(expr) counts only the rows where expr is not NULL.

SELECT COUNT(*), COUNT(email)
FROM users;

SUM

SUM(expr)

SUM adds up all non-NULL values of expr in the group. When every value in the group is NULL, the result is NULL. Values are treated as numbers. Non-numeric strings contribute 0.

SELECT dept_id, SUM(salary)
FROM employees
GROUP BY dept_id;

AVG

AVG(expr)

AVG returns the mean of all non-NULL values. When every value is NULL, the result is NULL. When the mean is a whole number, it is shown as an integer. Otherwise it is shown with four decimal places.

SELECT dept_id, AVG(salary) AS avg_pay
FROM employees
GROUP BY dept_id;

MIN and MAX

MIN(expr)
MAX(expr)

MIN returns the lowest value in the group. MAX returns the highest. Both skip NULL values. When all values are NULL, the result is NULL.

The comparison follows the same rule as the rest of the engine. When every value in the group parses cleanly as a number, the comparison is numeric. Otherwise it is a byte-by-byte string comparison.

SELECT MIN(price), MAX(price)
FROM products
WHERE category = 'electronics';

GROUP_CONCAT

GROUP_CONCAT(expr)
GROUP_CONCAT(expr SEPARATOR sep)

GROUP_CONCAT joins the non-NULL values of the group into one string. The default separator is a comma. You can change it with the SEPARATOR keyword. This works the same way as the MySQL GROUP_CONCAT function.

SELECT dept_id, GROUP_CONCAT(name SEPARATOR ', ') AS members
FROM employees
GROUP BY dept_id;

Using GROUP BY

Add a GROUP BY clause to split rows into groups before the aggregate runs. Each unique combination of grouping column values forms one group. The aggregate returns one row per group.

SELECT dept_id, COUNT(*) AS headcount, AVG(salary) AS avg_pay
FROM employees
GROUP BY dept_id;

You can group by more than one column. Each unique combination of those columns forms a separate group.

SELECT dept_id, job_title, COUNT(*) AS n
FROM employees
GROUP BY dept_id, job_title;

Non-grouped columns. The SQL standard requires every non-aggregated column in the select list to appear in GROUP BY. EvolutionDB does not enforce this rule. When you select a column that is not in GROUP BY, the engine uses the value from the first row of the group. This is the MySQL permissive behavior. PostgreSQL raises an error in the same situation.

Filtering Groups with HAVING

HAVING filters groups after the aggregate is computed. It works like WHERE but runs on the grouped result. You can reference aggregate functions directly in a HAVING condition.

SELECT dept_id, COUNT(*) AS headcount
FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 3;

You can also reference the alias from the select list in HAVING. The engine matches the alias name to the already-computed aggregate value.

SELECT dept_id, AVG(salary) AS avg_pay
FROM employees
GROUP BY dept_id
HAVING avg_pay > 60000;

When HAVING references an aggregate that is not in the select list, the engine computes it separately for the filter step.

WITH ROLLUP

GROUP BY ... WITH ROLLUP adds summary rows to the result. After each group, a super-aggregate row appears. It shows the aggregate over the whole set for that grouping level. The grouping column in a summary row is shown as NULL.

SELECT dept_id, job_title, COUNT(*) AS n
FROM employees
GROUP BY dept_id, job_title WITH ROLLUP;

The above query produces one row per group. Then it adds a subtotal row for each dept_id. Finally it adds a grand total row where both dept_id and job_title are NULL.

NULL Handling

All aggregate functions ignore NULL values. The only exception is COUNT(*), which counts every row regardless of NULL.

Aggregates as Window Functions

You can use aggregate functions as window functions by adding an OVER clause. This computes the aggregate over a sliding frame rather than collapsing rows into one. The result row is kept. Each row receives its own aggregate value.

SELECT name, salary,
       SUM(salary) OVER (PARTITION BY dept_id) AS dept_total
FROM employees;

See Chapter 7, Grouping and Aggregates for how the grouping path works inside the engine. See SELECT for the full syntax of GROUP BY, HAVING, and OVER.

Examples

This example counts all rows and also counts only the rows that have a non-null email.

SELECT COUNT(*) AS total, COUNT(email) AS with_email
FROM users;

This example finds the total and average salary per department. It returns only departments with more than three employees.

SELECT dept_id,
       COUNT(*)       AS headcount,
       SUM(salary)    AS total_pay,
       AVG(salary)    AS avg_pay,
       MIN(salary)    AS lowest,
       MAX(salary)    AS highest
FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 3
ORDER BY avg_pay DESC;

This example builds a comma-separated list of product names for each category.

SELECT category, GROUP_CONCAT(name ORDER BY name SEPARATOR ', ') AS products
FROM items
GROUP BY category;

This example shows a running department total alongside each employee row using a window function.

SELECT name, dept_id, salary,
       SUM(salary) OVER (PARTITION BY dept_id) AS dept_total
FROM employees
ORDER BY dept_id, name;
