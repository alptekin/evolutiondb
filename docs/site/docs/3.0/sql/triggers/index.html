21. Triggers & Procedures

EvolutionDB stores procedures, functions, and triggers as raw SQL bodies. One text-driven engine runs all of them. The engine replaces parameters and OLD./NEW. column references with text. It splits the body into statements that follow the control flow. Then it runs each statement against the engine.

Implementation note. The procedure language in EvolutionDB is not a compiled PL like PostgreSQL's PL/pgSQL. A body is kept word for word between BEGIN and the matching END. At run time the engine splits it on ; and on control-flow keyword boundaries. Then it runs each statement one by one. There is no type system for local variables. There is no plan caching. There is no overloading. A procedure or function is found by its name alone within a schema.

21.1. Overview

This page covers three kinds of stored objects. The system catalog of the current database and schema holds all of them.

Procedure
A named block of SQL with parameters. You run it with CALL. It is stored with its parameter list and body. The executor supports OUT/INOUT parameters and local DECLARE variables.
Function
You create it with CREATE FUNCTION … RETURNS type. Functions use the same catalog and the same body engine as procedures. The return type you declare is saved. Functions are stored next to procedures. The catalog flag is_function tells them apart.
Trigger
A BEFORE or AFTER block tied to one table and one of the INSERT/UPDATE/DELETE events. It fires once for each affected row. To run a trigger body, the engine wraps it in a temporary procedure and calls it. So the trigger reuses the full procedure engine. This includes IF, WHILE, SIGNAL, and more.

21.2. CREATE PROCEDURE / FUNCTION

See the reference page CREATE PROCEDURE for the full grammar. The synopsis below shows exactly what the parser accepts.

CREATE [ OR REPLACE ] PROCEDURE name ( [ param_list ] ) [ AS ]
BEGIN
    body
END

CREATE [ OR REPLACE ] FUNCTION name ( [ param_list ] ) RETURNS return_type [ AS ]
BEGIN
    body
END

-- param_list element:
[ IN | OUT | INOUT ] param_name data_type

If you leave out the parameter mode, it defaults to IN. Functions accept a fixed set of return types. These are INT/INTEGER, VARCHAR, TEXT, BOOLEAN, FLOAT, DOUBLE, BIGINT, DATE, and TIMESTAMP. The engine stores up to 16 parameters (MAX_PROC_PARAMS). It quietly ignores extra parameters at parse time.

The body is captured as raw text. The engine scans g_lex_input for the first BEGIN keyword and the matching standalone END. It tracks the depth of nested BEGIN…END blocks. It skips END IF, END WHILE, END LOOP, END FOREACH, and END CASE. A body can be at most 8192 bytes. The parameter list can be at most 1024 bytes.

CREATE PROCEDURE add_user(IN uname VARCHAR(50), IN uage INT) AS
BEGIN
    INSERT INTO users(name, age) VALUES (uname, uage);
END;

CREATE OR REPLACE FUNCTION double_it(n INT) RETURNS INT AS
BEGIN
    RETURN n * 2;
END;
OR REPLACE semantics. When you give OR REPLACE and a procedure of the same name already exists, the engine drops the old definition and creates it again. Without OR REPLACE, a duplicate name fails with SQLSTATE 42P07 ("procedure/function already exists"). Lookup is by name only. So CREATE FUNCTION and CREATE PROCEDURE share one namespace per schema.

21.3. Parameter and variable substitution

EvolutionDB does not bind parameters as typed values. Instead it uses textual substitution before it runs each statement in the body. The engine walks the SQL. It replaces each parameter name with the matching argument text. It also replaces the positional markers $1, $2, and so on. The substitution skips text inside single-quoted string literals. So a literal that holds a parameter name is left alone.

The quoting around an argument matters. A string argument keeps its quotes. For example, it arrives as 'alice'. A numeric argument arrives as bare digits. NULL arrives as the keyword NULL. A boolean arrives as TRUE/FALSE. The engine then parses the substituted text. So you must write the body to expect substituted SQL, not bound values.

Security caveat. Arguments are spliced into SQL text. So a procedure call is really string interpolation. Pass only trusted or already-escaped values. There is no separate bind layer to stop injection. This is a chosen limit of the text-driven engine. It is not the parameterized-statement model that the SQL standard describes.

21.4. Control flow inside a body

The body splitter divides the body on ;. Then it splits again at control-flow keyword boundaries. A block map records the matching END markers. This lets the interpreter jump. The engine recognises the following constructs.

ConstructForm
ConditionalIF cond THEN … [ ELSEIF cond THEN … ] [ ELSE … ] END IF
Pre-tested loopWHILE cond DO … END WHILE
Unconditional loop[ label: ] LOOP … END LOOP
Counted loopFOR var IN start..end DO … END FOR
Query loopFOREACH vars IN (query) DO … END FOREACH
Multi-branchCASE expr WHEN val THEN … [ ELSE … ] END CASE
Loop controlLEAVE [label], ITERATE [label]
Local variableDECLARE var TYPE [DEFAULT value], then SET var = expr
CursorDECLARE c CURSOR FOR query, OPEN c, FETCH c INTO …, CLOSE c
HandlerDECLARE { CONTINUE | EXIT } HANDLER FOR cond …
Return (functions)RETURN expr
Raise errorSIGNAL … (see §21.7)

A loop can take an optional leading label:. This lets LEAVE/ITERATE target a loop that wraps it. The engine tracks up to 32 local variables, 8 cursors, and 8 handlers per call. A body is split into at most 128 statement units.

CREATE PROCEDURE classify(score INT) AS
BEGIN
    DECLARE grade VARCHAR(2) DEFAULT 'F';
    IF score >= 90 THEN
        SET grade = 'A';
    ELSEIF score >= 70 THEN
        SET grade = 'C';
    END IF;
    INSERT INTO results(score, grade) VALUES (score, grade);
END;

21.5. CALL

You run a stored procedure with CALL. See CALL. The parser only records the call name and arguments. The body runs in the adaptor layer after parsing returns. This avoids re-entering the parser mutex.

CALL name ( [ arg [, arg ... ] ] )

An argument can be an integer literal, an approximate-number literal, a string literal, NULL, a boolean, a bare name, or a session variable (@var). The argument count must match the parameter count exactly. If it does not, the call fails with SQLSTATE 42883 ("procedure … expects N arguments, got M"). The engine accepts at most 16 arguments (MAX_CALL_ARGS). A procedure name that does not exist also raises 42883.

CALL add_user('alice', 30);

OUT / INOUT parameters. A call argument for an INOUT parameter can be a session variable (@var). In that case the engine reads the current value of the variable into the body before it runs. It can also write a result back to the caller's variable. This is how a procedure returns scalar data to the calling session.

21.6. CREATE TRIGGER / DROP TRIGGER / ALTER TRIGGER

The full grammar is on CREATE TRIGGER. A trigger ties a body to one table, one timing, and one event. It always fires FOR EACH ROW.

CREATE TRIGGER name { BEFORE | AFTER } { INSERT | UPDATE | DELETE }
    ON table_name FOR EACH ROW
BEGIN
    body
END

DROP TRIGGER [ IF EXISTS ] name

ALTER TRIGGER name { ENABLE | DISABLE }

The FOR EACH ROW clause is required. There is no statement-level (FOR EACH STATEMENT) trigger. There is no WHEN condition or column list on the event. The target table must exist. If it does not, creation fails with SQLSTATE 42P01. A duplicate trigger name raises 42P07. A new trigger is enabled by default. ALTER TRIGGER … DISABLE flips the catalog enabled flag. The trigger is then skipped at firing time, but it is not dropped.

DROP TRIGGER is name-only. Unlike PostgreSQL, you do not name the table in DROP TRIGGER / ALTER TRIGGER. EvolutionDB scans every table in the current schema to find a trigger by name. So trigger names are scoped to the schema. The first match wins.

OLD / NEW row references. Inside a trigger body, OLD.col and NEW.col refer to the row before and after the change. Before the body runs, the engine replaces these references with text. It uses the matching column value for the affected row. A numeric value is inserted bare. Everything else is single-quoted, and any embedded quotes are doubled. A missing value becomes NULL. Which values you get depends on the event. INSERT fills NEW. DELETE fills OLD. UPDATE fills both.

BEFORE-trigger field assignment. A BEFORE trigger can change the row that is being written. It does this by assigning to a column with SET NEW.col = expr. The engine pulls these assignments out of the body. It first does the OLD/NEW substitution. Then it evaluates each expression with a SELECT. It writes the result back into the row before the row is stored. The engine honours up to 16 such assignments per trigger.

CREATE TRIGGER stamp_created BEFORE INSERT ON orders FOR EACH ROW
BEGIN
    SET NEW.created_at = NOW();
END;

CREATE TRIGGER audit_del AFTER DELETE ON accounts FOR EACH ROW
BEGIN
    INSERT INTO deleted_accounts(id, name) VALUES (OLD.id, OLD.name);
END;

Memory-store triggers. You can attach a trigger to a memory store with ON MEMORY STORE name. The parser stages the store name with an internal @mem: sentinel. At creation, the engine resolves the store through the catalog. It rewrites the store to its backing table mem_<name>. After that, the trigger fires just like a normal table trigger. A PUT event on a memory store is mapped to the INSERT ('I') event inside the engine. A store that does not exist raises SQLSTATE 42704.

21.7. Trigger firing model, errors, and recursion

The engine fires matching triggers from the INSERT/UPDATE/DELETE hook points. It fires them once per affected row. It filters them by timing, event, and the enabled flag. For each fired trigger, the engine wraps the substituted body as a temporary procedure. It calls the procedure, then drops it. So triggers and procedures run through the same engine. The steps are shown below.

-- conceptual expansion performed internally per fired trigger
CREATE OR REPLACE PROCEDURE __trig_<id>__() AS BEGIN <substituted body> END;
CALL __trig_<id>__();
DROP PROCEDURE IF EXISTS __trig_<id>__;

Recursion limit. A depth counter guards trigger execution. Nesting beyond 16 levels (MAX_TRIGGER_DEPTH) quietly stops further trigger firing. No error is raised. This prevents runaway recursion.

Raising errors with SIGNAL. A body can stop with a user-defined error. This works for both procedures and triggers.

SIGNAL [ SQLSTATE 'xxxxx' ] [ SET MESSAGE_TEXT = 'message' ]

If you leave out SQLSTATE, it defaults to 45000 (unhandled user-defined exception). The default message is "Unhandled user-defined exception". A DECLARE … HANDLER FOR SQLEXCEPTION can catch the signal. A CONTINUE handler resumes after the failed statement. An EXIT handler unwinds the body. If there is no handler, the error passes to the caller. When a trigger body errors, the whole firing fails. The DML statement that started it is aborted.

CREATE TRIGGER no_negatives BEFORE INSERT ON ledger FOR EACH ROW
BEGIN
    IF NEW.amount < 0 THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'amount must be non-negative';
    END IF;
END;

21.8. SQLSTATE reference

These are the error codes that the trigger and procedure paths raise.

42P01
Trigger target table does not exist.
42P07
Duplicate object. A procedure or function already exists (without OR REPLACE), or a trigger name already exists.
42883
Procedure or function does not exist on CALL or DROP. The trigger does not exist on DROP without IF EXISTS. The argument count does not match on CALL.
42704
Memory store named in ON MEMORY STORE does not exist.
45000
Default SQLSTATE for an unqualified SIGNAL (user-defined exception).
53200
Out of memory while preparing a CALL body for execution.

21.9. Notes & limitations

What EvolutionDB does not implement. Here is how it differs from PostgreSQL and the SQL standard.
  • No row-vs-statement choice. Triggers are always FOR EACH ROW. There is no statement-level trigger.
  • No WHEN condition, column list, or INSTEAD OF. The event is a single bare INSERT/UPDATE/DELETE with no filter. Views cannot carry triggers.
  • One event per trigger. There is no INSERT OR UPDATE multi-event form. Create separate triggers.
  • Table is not named on DROP/ALTER TRIGGER. Names are resolved by scanning the schema. So trigger names should be unique per schema.
  • Textual parameter substitution, not value binding. Arguments and OLD/NEW references are spliced into SQL text. There is no typed parameter binding and no protection against injection.
  • No procedure/function overloading. Lookup is by name within a schema. Functions and procedures share that namespace.
  • Functions are not callable inline. A CREATE FUNCTION body runs through the same engine as a procedure. The return type is recorded. But the database does not let you call user functions inside a SELECT the way PostgreSQL does.
  • Fixed capacities. Body ≤ 8192 bytes. ≤ 16 parameters/arguments. ≤ 128 statement units. ≤ 32 local variables, ≤ 8 cursors, ≤ 8 handlers per call. Trigger recursion depth ≤ 16.
  • No trigger ordering guarantees by name. There are also no transition tables (REFERENCING … OLD/NEW TABLE).
