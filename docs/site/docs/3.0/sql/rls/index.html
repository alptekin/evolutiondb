20. Row-Level Security

Row-Level Security (RLS) lets you add per-table policies. These policies limit which rows a session user can read, insert, update, or delete. The engine enforces them on every DML path. Your application does not have to do this work.

EvolutionDB supports RLS as a small subset of the PostgreSQL model. A table carries an RLS-enabled flag. You can attach any number of named policies to that table. A policy has an optional row-filter expression with USING. It also has an optional write-validation expression with WITH CHECK. Each policy has a command scope and an optional role list. When the flag is on, the engine filters reads through the matching USING expressions. It also checks written rows against the matching WITH CHECK expressions.

The SQL commands and their storage are handled in evolution/db/Policy.c. The runtime helpers live in the same place. These are the read overlay and the write check. The SELECT, INSERT, UPDATE, and DELETE paths call them.

20.1. Enabling and Disabling RLS on a Table

You define policies on their own. But they only start to work once Row-Level Security is enabled on the table that owns them. The flag does not create, drop, or change any policy. It only controls whether the engine uses them.

ALTER TABLE table_name ENABLE ROW LEVEL SECURITY;
ALTER TABLE table_name DISABLE ROW LEVEL SECURITY;

The grammar accepts only the exact phrase ROW LEVEL SECURITY. If you misspell the middle word, you get a syntax error. The message is "expected LEVEL after ROW…". The flag is a single per-table value named rls_enabled. It is stored in the system catalog. The function cat_update_table_rls_flag() sets it.

-- turn enforcement on; existing policies (if any) now apply
ALTER TABLE accounts ENABLE ROW LEVEL SECURITY;

-- turn enforcement off; policies remain in the catalog, just dormant
ALTER TABLE accounts DISABLE ROW LEVEL SECURITY;

Default-deny. A table can have rls_enabled = 1 while no policy matches a non-superuser's command. In that case access is denied. Reads return zero rows. Writes are rejected. If you enable RLS before you define any policy, every non-superuser is locked out. The admin user is a superuser. It skips RLS fully. See §20.6.

20.2. CREATE POLICY

Creates a named policy and attaches it to a table. See also the command reference page at CREATE POLICY.

CREATE POLICY policy_name ON table_name
    [ AS { PERMISSIVE | RESTRICTIVE } ]
    [ FOR { ALL | SELECT | INSERT | UPDATE | DELETE } ]
    [ TO role [, ...] ]
    [ USING ( using_expression ) ]
    [ WITH CHECK ( check_expression ) ];

Only the name and the target table are required. Every other clause is optional. The clauses must come in the order shown above. The order is AS, then FOR, then TO, then USING, then WITH CHECK.

policy_name
The policy name. It must be unique within one table. In the catalog, policies are keyed by table_id:policy_name. So you can reuse the same name on different tables. If you create a duplicate name on the same table, it fails with SQLSTATE 42710.
AS PERMISSIVE / AS RESTRICTIVE
Sets how the policy combines with other policies. PERMISSIVE policies are OR'd together. RESTRICTIVE policies are AND'd on top. The default is PERMISSIVE. See §20.5.
FOR
The command scope that the policy applies to. The default is FOR ALL. Inside the engine, each scope maps to a one-character command code. See §20.4.
TO role [..]
The roles that the policy applies to. You can list up to 8 roles. The limit is CAT_MAX_POLICY_ROLES. Any extra entries are ignored. If you leave the list out, the policy applies to every role. A role named public in the list also matches every role. Role matching is case-insensitive. It is compared against the session user name.
USING ( using_expression )
A boolean expression that is checked on each existing row. A row is visible to SELECT when this expression is true. The same row can then be used by UPDATE and DELETE. The expression AST is stored in the catalog. It is hex-encoded and can be up to 2048 bytes. The limit is CAT_MAX_POLICY_EXPR.
WITH CHECK ( check_expression )
A boolean expression that new or updated rows must pass on INSERT and UPDATE. If you leave it out, the engine reuses the USING expression for the write check. This matches PostgreSQL.
-- each user only sees and writes their own rows
CREATE POLICY owner_rows ON documents
    FOR ALL
    USING (owner = 'alice')
    WITH CHECK (owner = 'alice');

-- read-only visibility policy scoped to one role
CREATE POLICY public_published ON documents
    AS PERMISSIVE
    FOR SELECT
    TO reader
    USING (status = 'published');

-- a restrictive guard that ANDs on top of any permissive policy
CREATE POLICY not_archived ON documents
    AS RESTRICTIVE
    FOR SELECT
    USING (archived = false);

Memory-store sugar. EvolutionDB also accepts CREATE POLICY name ON MEMORY STORE store. It also accepts the matching ALTER MEMORY STORE … ENABLE/DISABLE ROW LEVEL SECURITY. These point at the store's backing table. That table is named mem_<store>. In every other way they work the same. If the store does not exist, the engine uses the literal name. It then raises the usual "relation does not exist" error.

20.3. DROP POLICY

Removes a policy from a table. See also DROP POLICY.

DROP POLICY policy_name ON table_name;
DROP POLICY public_published ON documents;

The engine resolves the target table first. If the table does not exist, the error is SQLSTATE 42P01. This means undefined table. If the table exists but has no such policy, the error is SQLSTATE 42704. This means undefined object. Dropping a policy does not change the table's rls_enabled flag. So if you drop the last policy while RLS is still on, the table goes back to default-deny for non-superusers.

20.4. Command Scopes

The FOR clause stores a single command-code byte on the policy. A policy takes part in enforcement only when its scope matches the current operation. A scope of ALL always matches.

FOR clauseCodeApplies to
FOR ALL (default)'*'SELECT, INSERT, UPDATE, DELETE
FOR SELECT'S'Read path (USING overlay)
FOR INSERT'I'Write check (WITH CHECK)
FOR UPDATE'U'Read overlay for candidates + write check
FOR DELETE'D'Write check on deletion candidates

20.5. How Policies Are Combined and Enforced

Enforcement works in two different ways. The way depends on whether the operation reads rows or writes rows.

20.5.1. Read overlay (SELECT, and UPDATE/DELETE row selection)

For reads, the engine builds an overlay expression. It uses every policy that matches the command and the session user. It then AND's this overlay into the statement's existing WHERE clause. The combination rule works like this.

  • All matching PERMISSIVE USING expressions are OR'd together.
  • All matching RESTRICTIVE USING expressions are AND'd on top of that OR group.
  • If no permissive policy matches, the overlay is the constant false. That means zero rows. A restrictive policy on its own never makes any row visible. There must be a permissive grant for any row to pass.

The result is folded into the same WHERE expression that every scan path uses. So plain table scans, fast paths, and expression indexes all see the same filtered view. The original WHERE is put back after the statement runs. In multi-table queries, the same overlay is applied to each joined table.

-- with the three documents policies above, a non-superuser SELECT
-- effectively runs as:
--   WHERE (<your WHERE>)
--     AND ( owner = 'alice' OR status = 'published' )  -- PERMISSIVE OR-group
--     AND ( archived = false )                          -- RESTRICTIVE AND
SELECT * FROM documents WHERE created > '2026-01-01';

20.5.2. Write check (INSERT, UPDATE, DELETE)

For writes, the engine checks the matching policies' WITH CHECK expression. When no WITH CHECK is present, it uses the USING expression instead. It runs this check against the candidate row's column values. The acceptance rule works like this.

  • If any matching RESTRICTIVE check is false, the write is rejected right away.
  • At least one matching PERMISSIVE check must be true.
  • If no permissive policy matches at all, the write is rejected. This is default-deny.

A failed check raises SQLSTATE 42501. The message is new row for relation "…" violates row-level security policy. An expression result counts as true when it equals t, true, or 1. It also counts as true when it is a non-zero number.

-- rejected: WITH CHECK (owner = 'alice') is false for this row
INSERT INTO documents (id, owner, status) VALUES (1, 'bob', 'draft');
-- ERROR: new row for relation "documents" violates row-level security policy

20.6. Superuser Bypass

The admin user counts as a superuser. It skips both the read overlay and the write check fully. RLS is never applied to its statements. This is true no matter what the table's flag is or what policies exist. The engine identifies the superuser by a case-insensitive match of the session user name against admin. EvolutionDB has no per-table "force RLS" mode right now. So there is no way to make the superuser follow its own policies.

20.7. Error Codes

RLS uses the following SQLSTATE values.

SQLSTATECondition
42601Syntax error. The policy name or target table is missing. Or the ENABLE/DISABLE ROW LEVEL SECURITY phrase is wrong.
42P01 / 42E01Undefined table. The target relation does not exist.
42704Undefined object. DROP POLICY ran on a policy or memory store that does not exist.
42710Duplicate object. A policy with the same name already exists on the table.
42501Insufficient privilege. A written row broke a WITH CHECK policy.
EX000Internal error. The RLS flag could not be updated.

20.8. Notes and Limitations

EvolutionDB's RLS is a chosen subset of the SQL and PostgreSQL model. These are the main differences from PostgreSQL.

  • No ALTER POLICY. To change a policy, DROP it and CREATE it again.
  • No FORCE ROW LEVEL SECURITY. The owner and superuser admin always skip policies. You cannot turn on enforcement for the superuser.
  • Superuser detection is name-based. Only the literal user admin counts as a superuser for RLS. This does not come from a role privilege system.
  • Role matching is by session user name. The TO list is compared against the current user. The match is case-insensitive. The value public is special and means all roles. There is no role-membership or inheritance resolution.
  • Fixed limits. A policy can have at most 8 roles. The limit is CAT_MAX_POLICY_ROLES. Each serialized USING or WITH CHECK expression is capped at 2048 bytes. The limit is CAT_MAX_POLICY_EXPR.
  • Clause order is fixed. CREATE POLICY clauses must come as AS → FOR → TO → USING → WITH CHECK.
  • Disabling RLS does not drop policies. Policies stay in the catalog. They become active again when you re-enable RLS.
