18. Subscriptions & CDC

Durable subscriptions turn EvolutionDB's short-lived LISTEN/NOTIFY channels into change-data-capture streams that last and can be replayed. When a notification commits on a channel, it is saved to a mailbox on disk. A consumer can read that mailbox, acknowledge it, and replay it after reconnecting.

18.1. Overview

EvolutionDB has an in-memory LISTEN/NOTIFY registry. It sends a payload to every connection that is listening on a channel right now. This delivery is best-effort. If no client is connected when the notification commits, the notification is lost. A durable subscription fixes this gap. It registers a named mailbox that is saved and bound to one channel. Each time a NOTIFY commits on that channel, the payload is added to the mailbox with a sequence number that always goes up. A consumer that was offline can connect again later. It can replay everything that arrived in the meantime with RESUME SUBSCRIPTION. It can also move an acknowledgement watermark forward with ACK SUBSCRIPTION. A background daemon then prunes what the consumer has already processed.

Durable subscriptions are EvolutionDB's change-data-capture (CDC) building block. Application code or triggers emit NOTIFY on a channel. One or more durable subscribers read that stream at their own pace. Delivery is at-least-once. The stream can be replayed after a reconnect.

How this relates to in-memory NOTIFY. Durable saving runs after the in-memory listener fan-out. It runs inside the same broadcast that happens at commit time. Live listeners are served first. The catalog walk that fills durable mailboxes only happens after the in-memory broadcast for the channel is delivered. So the fast path costs nothing when no subscription is set up. See NOTIFY and LISTEN.

18.2. The subscription model

A subscription is a catalog object. It has a unique name and is bound to exactly one channel. Each subscription owns a private backing table. This table acts as its mailbox.

18.2.1. Backing table

When you create a subscription named s, EvolutionDB creates a backing heap table named sub_s in the current schema. The table always has this fixed layout.

CREATE TABLE sub_s (
    seq        BIGINT          PRIMARY KEY,   -- monotonic, server-assigned
    payload    VARCHAR(8000),                  -- the NOTIFY payload; '' for payload-less NOTIFY
    posted_at  TIMESTAMP      NOT NULL        -- UTC append time, 'YYYY-MM-DD HH:MM:SS'
);

One row is written for each published NOTIFY. The seq column is the primary key. The backing table's B+ tree keeps rows in ascending order by sequence. You do not need an ORDER BY to read them in arrival order. The posted_at stamp is set on the server in UTC at publish time.

18.2.2. Subscription catalog fields

The system catalog stores one descriptor for each subscription. SHOW SUBSCRIPTIONS returns these columns.

name text
Unique subscription name. Acts as the catalog key.
channel text
The single NOTIFY channel that this subscription is bound to. Channel matching at publish time is case-insensitive.
last_ack_seq bigint
The acknowledgement watermark. It starts at 0, which means nothing is acknowledged yet. RESUME returns only rows with seq > last_ack_seq. The prune daemon deletes rows with seq <= last_ack_seq.
next_seq bigint
The sequence number to give to the next published row. It starts at 1. After each successful append, it goes up by one and is saved.
ttl_days int
How long unacknowledged rows are kept, in days. The default is 7. A value of 0 turns off the TTL prune. The acknowledged-row prune still runs.

18.3. Creating and dropping subscriptions

18.3.1. CREATE SUBSCRIPTION

CREATE SUBSCRIPTION name FOR CHANNEL channel
CREATE SUBSCRIPTION IF NOT EXISTS name FOR CHANNEL channel

This registers a durable subscription. It also creates its sub_<name> backing table in the same step. The watermark starts at 0. The next sequence starts at 1. The TTL starts at the 7-day default. Both name and channel are required. If you leave out either one, you get a syntax error. The error code is 42601.

CREATE SUBSCRIPTION orders_cdc FOR CHANNEL order_events;

If a subscription with that name already exists, the statement fails with 42710. If you add IF NOT EXISTS, the statement does nothing instead.

18.3.2. DROP SUBSCRIPTION

DROP SUBSCRIPTION name
DROP SUBSCRIPTION IF EXISTS name

This unregisters the subscription and drops its backing table. Without IF EXISTS, dropping a subscription that does not exist fails with 42704. With IF EXISTS, the statement does nothing instead.

DROP SUBSCRIPTION IF EXISTS orders_cdc;

18.4. Consuming the stream

18.4.1. RESUME SUBSCRIPTION

RESUME SUBSCRIPTION name

This returns the backlog for the subscription. The result set has the columns (seq, payload, posted_at). It includes every mailbox row whose seq is strictly greater than the current last_ack_seq. The rows come back in ascending seq order. RESUME is read-only and idempotent. It never moves the watermark forward and never deletes rows. You get the same backlog until you acknowledge it. Resuming a subscription that does not exist fails with 42704.

RESUME SUBSCRIPTION orders_cdc;
--  seq | payload                  | posted_at
-- -----+--------------------------+---------------------
--    1 | {"id":42,"op":"insert"}  | 2026-06-12 09:14:03
--    2 | {"id":43,"op":"insert"}  | 2026-06-12 09:14:05

18.4.2. ACK SUBSCRIPTION

ACK SUBSCRIPTION name UPTO seq

This moves the acknowledgement watermark to seq. The operation only moves last_ack_seq forward. It is idempotent. ACK ... UPTO 5 followed by ACK ... UPTO 3 leaves the watermark at 5. The acknowledged sequence does not have to match a row that is actually stored. The watermark just moves, and the prune daemon catches up on its next tick. Acknowledging a subscription that does not exist fails with 42704.

ACK SUBSCRIPTION orders_cdc UPTO 2;

18.5. End-to-end lifecycle

A full CDC loop joins the publisher side with the durable consumer side. The publisher side is NOTIFY.

-- 1. enroll a durable subscriber on a channel
CREATE SUBSCRIPTION orders_cdc FOR CHANNEL order_events;

-- 2. producers emit notifications (e.g. from application code or a trigger)
NOTIFY order_events, '{"id":42,"op":"insert"}';
NOTIFY order_events, '{"id":43,"op":"insert"}';

-- 3. a consumer reconnects later and drains the backlog
RESUME SUBSCRIPTION orders_cdc;        -- rows seq 1, 2

-- 4. after processing, acknowledge progress so the daemon can prune
ACK SUBSCRIPTION orders_cdc UPTO 2;

-- 5. inspect every subscription's watermark and TTL
SHOW SUBSCRIPTIONS;

18.6. Retention & pruning

A background auto-reclaim daemon trims each subscription's backing table on a fixed interval. On every tick it walks the mailbox. It deletes a row when one of these conditions is true.

  • Acknowledged. The row's seq is <= last_ack_seq. This prune always runs, no matter what the TTL is.
  • TTL-expired. ttl_days > 0 and the row's posted_at is older than ttl_days × 86 400 seconds. This catches unacknowledged rows from consumers that never came back. Rows with a NULL timestamp are not TTL-pruned.

The daemon collects the rows to remove and deletes them in bulk. It uses the same write lock as the other retention passes. It removes each row from both the heap and the PK B+ tree. It handles at most 256 rows per subscription per tick. A very large backlog is cleared over several ticks, not in one pass.

Tuning the interval. The reclaim interval is 30 seconds by default. You can lower it for tests with the EVOSQL_AUTO_RECLAIM_INTERVAL_SEC environment variable. It accepts a value from 1 to 30 seconds.

18.8. Comparison with PostgreSQL

EvolutionDB's syntax copies PostgreSQL's CREATE SUBSCRIPTION keyword on purpose. But the meaning is completely different. These are not logical-replication subscriptions.

Source of data
PostgreSQL subscriptions read a logical replication stream from a publisher's WAL. EvolutionDB subscriptions read committed NOTIFY payloads on a named channel. There is no upstream node, publication, or replication slot.
Binding
PostgreSQL binds a subscription to a PUBLICATION on a remote CONNECTION. EvolutionDB binds a subscription to a local channel with FOR CHANNEL. There is no publication object and no connection string.
Delivery & replay
EvolutionDB uses a pull-based mailbox with a cursor you control. RESUME replays. ACK ... UPTO moves a watermark forward. A daemon prunes. PostgreSQL applies changes on its own with apply workers. It has no client-driven RESUME/ACK like this.
Payload
An EvolutionDB payload is the free-form text of a NOTIFY. The backing column caps it at 8000 bytes. It is not a decoded row change. EvolutionDB does not read the payload. The CDC structure is whatever the producer chooses to encode. This is usually JSON. It can include the operation, keys, and columns.
Standard SQL
Subscriptions, RESUME, and ACK are EvolutionDB extensions. They are not part of the SQL standard. RESUME and ACK are EvolutionDB-specific verbs with no PostgreSQL equivalent.

Notes & limitations.

  • A subscription binds to exactly one channel. The channel is set at creation and cannot be changed. There is no ALTER SUBSCRIPTION. To rebind, drop the subscription and create it again.
  • ttl_days defaults to 7. You cannot set it through DDL in this release. The prune daemon honors it, but SHOW SUBSCRIPTIONS only shows it read-only.
  • The backing table is named sub_<name> in the current schema. It is an internal detail. Use only RESUME and ACK to read it. Do not query or change it directly.
  • Delivery is at-least-once. There is no automatic deduplication. A consumer that does not ACK will see the same rows again on the next RESUME.
  • Durable saving happens at NOTIFY commit time. Notifications sent in a transaction that rolls back are never saved.
  • Channel matching is case-insensitive. Subscription and channel names follow the catalog name-length limit.
