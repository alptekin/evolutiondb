10. Indexes

EvolutionDB has four index access methods. They are the default B+ tree, an FNV-1a hash index, an in-memory HNSW graph for vector search, and an inverted full-text index. Each one has its own layout on disk or in memory. Each one encodes keys in its own way. Each one speeds up a different kind of query.

An index is an extra structure. The engine can read it instead of scanning every row of a table. EvolutionDB builds indexes with CREATE INDEX and CREATE FULLTEXT INDEX. It removes them with DROP INDEX. Every table also has one index that you do not create yourself. This is its primary-key index. It is a B+ tree. Its entries point at the heap location of each live row. The location is a RowID, which is page_no:4 + slot_idx:2. Secondary indexes use the same RowID value. So a matched index entry reaches its row directly. It does not need a second primary-key lookup.

Scope of this chapter. This chapter only covers the access methods that EvolutionDB 3.0 really has. It only describes the behavior found in the source code. The source files are btree2.c, hash_idx.c, hnsw.c, and FullText.c. EvolutionDB borrows some syntax from PostgreSQL and MySQL, but it does not always behave the same way. The HNSW graph lives only in memory. The full-text index is not kept up to date on DML. We point out these differences clearly. The full statement syntax and the option lists are on the CREATE INDEX reference page.

10.1. Choosing an Access Method

The four methods are not the same. Each one is keyed in a different way. Each one answers a different kind of query. The table below shows when to use each one. The rest of the chapter describes them in detail.

MethodCreated withBest forCannot do
B+ tree (default)CREATE INDEXEquality, range scans (<, <=, >, >=), ordered traversal, prefix lookups on secondary keys,
HashCREATE INDEX … USING HASHSingle-value equality lookups on a secondary columnRange scans, ordered traversal
HNSWCREATE INDEX … USING HNSWApproximate k-nearest-neighbor search over a VECTOR columnExact filtering, persistence across restart. In-memory only in v1
Full-textCREATE FULLTEXT INDEXWord/token matching in text columns via MATCH … AGAINSTPhrase/ranked search; auto-maintenance on DML

Three of the four keep their pages in the same unified 4 KB page file as the table data. The Page Manager allocates these pages. The HNSW graph is different. It lives only in process memory.

10.2. B+ Tree Indexes

The B+ tree is EvolutionDB's default access method. It is also the most capable one. Every primary key uses it. CREATE INDEX builds it too, unless you ask for another method. Inside, it is a page-based B+ tree in btree2.c. Every node is one 4 KB page. Leaf nodes hold the real (key, RowID) entries. Internal nodes hold only separator keys and child page pointers.

-- single-column B+ tree index
CREATE INDEX idx_users_email ON users (email);

-- enforce uniqueness while indexing
CREATE UNIQUE INDEX idx_users_username ON users (username);

-- composite index over two columns
CREATE INDEX idx_orders_cust_date ON orders (customer_id, order_date);

10.2.1. Node Layout and Capacity

Each node page starts with the standard 16-byte page header. Next comes a 4-byte node header. The node header holds a 16-bit key count and a reserved 16-bit flags field. That leaves 4076 bytes of node space for entries. A leaf entry is [key_len:1][key:N][RowID:6]. An internal node mixes num_keys separator keys with num_keys + 1 child pointers. One byte holds each key length. An index key can be at most 255 bytes long. This limit is set by BT2_MAX_KEY_LEN. A longer key is rejected at insert time.

Leaf pages are linked in both directions through the page header's prev_page and next_page fields. This sibling chain makes ordered scans and ranges cheap. The engine descends the tree once to find the starting leaf. After that, the cursor walks the chain. It does not descend again for each key.

10.2.2. Key Comparison Is Numeric-Aware

Keys are stored as length-prefixed byte strings. The comparison is not plain lexicographic. First the comparator tries to read both keys as 64-bit integers. If both read cleanly, it compares them as numbers. If that fails, it tries floating-point. If neither side is a number, it falls back to a byte-wise memcmp. On a common prefix, the shorter key comes first.

Why this matters. Numeric-aware comparison means an integer key like 100 sorts after 20. That is the correct order. Both are stored as text. You do not get the lexicographic surprise of "100" < "20". There is a trade-off. Any value that looks like a number is ordered as a number. This happens no matter what type the column was declared as. The comparison buffers are limited to 256 bytes. So the numeric fast path only works on keys shorter than 255 bytes. Longer keys always use the lexicographic path.

10.2.3. Secondary Keys and Composite Indexes

A secondary B+ tree does not key on the column value alone. Many rows can share the same indexed value. To keep the entries unique, the engine builds a composite key. It joins the indexed value with the row's primary key. The join uses the ASCII unit separator \x1F. The key looks like indexed_value␟PK. So a lookup for one indexed value is a prefix scan. The engine sets the cursor at the first key whose prefix matches. Then it walks the leaf chain and collects entries until the prefix changes. A multi-column index works differently. It joins the column values with | into one key.

10.2.4. Inserts, Splits, Deletes, and Compaction

Inserts keep each leaf sorted. Sometimes an entry will not fit in a leaf's 4076 bytes. Then the leaf splits. The upper half of its entries move to a new page. The sibling chain is relinked. The first key of the new page is pushed up to the parent as a separator. A split can cascade up the tree. A split at the root creates a new root one level taller. So the tree grows from the leaves upward and stays balanced.

Deletes use lazy tombstones. bt2_delete finds the entry and overwrites its RowID with {0,0}. It does not remove the entry from the page and it does not rebalance. This keeps the tree stable on purpose. The root never moves and no page is freed. So the system catalog and hot-path callers can safely hold leaf-page hints and live cursors. A later insert of the same key reuses a tombstoned slot. Searches and cursor scans skip tombstones.

Reclaiming the physical space is a separate offline pass called bt2_compact. RECLAIM runs it. It drops tombstones. Then it borrows or merges entries across underflowing sibling nodes. The min-fill heuristic is half a page. It frees empty pages and collapses a single-child root. It can move the root. So it must run under the global DML lock. No concurrent reader may hold page hints or cursors. The RECLAIM path sets up exactly these conditions.

Note. Compaction uses a min-fill target of half of NODE_CAPACITY. This is only a space-usage heuristic. It is not a correctness rule. There is just one hard rule. A node's bytes must never go over the 4076-byte capacity. Both the borrow path and the merge path keep this rule. Between compactions, a node may sit below min-fill, and that is fine.

10.3. Hash Indexes

A hash index uses USING HASH. It gives up the B+ tree's ordering and gives you direct equality lookup instead. It is built as static hashing with overflow chains in hash_idx.c. A single directory page holds a fixed array of bucket pointers. Each non-empty bucket is a chain of bucket pages packed with entries.

-- equality-only index for token lookups
CREATE INDEX idx_sessions_token ON sessions USING HASH (token);

10.3.1. Buckets, Hashing, and Overflow

Bucket routing uses the FNV-1a hash function. The hash is taken modulo the bucket count. A new index starts with 64 buckets by default. This default is set by HASH_DEFAULT_BUCKETS. The bucket count is capped at 1017 buckets. This cap is set by HASH_MAX_BUCKETS. It lets the whole bucket array fit in one 4 KB directory page. Bucket pages are allocated lazily. A bucket holds no page at all until its first entry arrives. An entry is [key_len:2][key:N][RowID:6]. Keys are limited to 255 bytes. This limit is set by HASH_MAX_KEY_LEN. When a bucket page fills up, a new overflow page is allocated. It is linked into the chain through the page header.

Like a secondary B+ tree, a hash entry stores the composite key indexed_value␟PK. Only the part before the \x1F separator is hashed for bucket routing. So every row that shares an indexed value lands in the same bucket. An equality lookup hashes just the search value. It walks that one bucket chain. It returns every entry whose key begins with indexed_value␟.

10.3.2. When a Hash Index Applies

A hash index answers exactly one kind of query. That query is equality on the indexed value. The bucket layout has no ordering in it. So it cannot serve a range predicate, an ORDER BY, or a prefix-of-a-prefix search. To delete an entry, the engine shifts the other entries in its bucket page down over it. The bucket count is fixed. There is no rehash and no split. If you need ordering or ranges, use the default B+ tree instead.

Static, not extendible. The bucket count is chosen at create time. It never changes after that. This version has no dynamic resizing and no extendible-hashing split. When a table grows far past the bucket count, lookups slow down. They move from near-constant time toward a per-bucket chain walk. Size the index for the table. If the cardinality is unknown or growing, use a B+ tree instead.

10.4. HNSW Vector Indexes

An HNSW index uses USING HNSW. It speeds up approximate nearest-neighbor search, also called ANN search, over a VECTOR(dim) column. It finds the rows whose embedding is closest to a query vector under a chosen distance metric. You build it with CREATE INDEX. It is not like the other three methods. It does not live in the page file. The graph and its dense vector store are held only in process memory, in hnsw.c.

-- build an HNSW index, choosing the distance metric via opclass
CREATE INDEX idx_docs_embedding ON docs
    USING HNSW (embedding vector_l2_ops)
    WITH (m = 32, ef_construction = 128);

10.4.1. How the Graph Is Built

At CREATE INDEX time the engine scans the table once. It decodes the vector column's float4 payload straight from each live tuple. No text-field buffer limits the dimension, so 1024-dimension embeddings work. It appends each (primary key, vector) pair to a dense, row-major store. Then it builds a multi-layer navigable graph over that store. Each node gets a random top level. Neighbor lists are filled by greedy beam search. The search uses the configured ef_construction candidate width. It keeps up to m links per node per level. At the base layer it keeps 2 × m links. Graphs are tracked in a small static table keyed by (table_id, index_name). There is a hard cap of 64 concurrent graphs. This cap is set by HNSW_MAX_GRAPHS. Primary keys can be up to 256 bytes.

In-memory only. Rebuild required. In v1 the HNSW graph is not persisted. It is also not rebuilt automatically at server startup. After a restart, an existing HNSW index must be rebuilt with a fresh CREATE INDEX or a reindex. The DDL layer does record enough catalog metadata for automatic replay later. This metadata covers the column, the distance kind, m, and ef_construction. DROP INDEX frees the graph right away.

10.4.2. Distance Metrics (opclasses)

You write the opclass after the column. It picks the distance function the graph uses. The graph uses it for both build and search. EvolutionDB has four opclasses.

vector_cosine_ops (default)
Cosine distance, 1 − cos(a, b). A zero-norm vector on either side gives distance 1.0. This is the default when you give no opclass. It is also the fallback for any opclass the engine does not know.
vector_l2_ops
Euclidean (L2) distance. This is the straight-line distance between the two vectors.
vector_ip_ops
Negative inner product, −a·b. Smaller means nearer. This ordering matches the other metrics. vector_inner_ops works as a synonym.
vector_l1_ops
Manhattan (L1) distance. This is the sum of the absolute per-component differences.

10.4.3. Searching the Index

You run vector search from SQL using helper functions and distance operators. The expression engine evaluates them. hnsw_knn returns a comma-separated list of the k nearest primary keys. They are ordered by distance from low to high. The value k defaults to 10 and is capped at 256.

-- k nearest neighbors of a query vector, by primary key
SELECT hnsw_knn('docs', 'idx_docs_embedding', '[0.12, 0.04, ...]', 5);

-- the distance functions / operators rank directly in a query
SELECT id, l2_distance(embedding, '[0.12, 0.04, ...]') AS dist
    FROM docs ORDER BY embedding <-> '[0.12, 0.04, ...]' LIMIT 5;

The distance expressions are cosine_distance(a, b), l2_distance(a, b), inner_product(a, b), and l1_distance(a, b). There are also three operators. <-> is L2. <#> is negative inner product. <=> is cosine, and you can use it in ORDER BY emb <=> '[…]'. You can write a query vector in two ways. One way is a bracketed literal '[f1, f2, …]'. The other way is the client's b64i8:<base64> quantized form. This form is a float32 scale followed by an int8 array. It is more compact. It keeps a 1024-dimension query under the statement size cap.

Inside, search picks one of two paths under the same interface. For a graph large enough to help, it does a multi-layer greedy descent. Then it does a base-layer beam search of width ef_search. The default width is 64. It is widened to at least k when needed. For very small graphs the engine takes the other path. It also takes it if graph construction hit an allocation failure. That path is an exact brute-force top-k scan over the dense store. The switch is silent. So results are always correct. The graph only changes how fast they come back.

10.4.4. Hybrid Search (Vector + Filter)

hnsw_filter_knn joins vector ranking with one equality predicate over another column. You supply the predicate as a 'col=val' spec. You can also write 'col=val:pre' or ':post' to force a strategy.

SELECT hnsw_filter_knn('docs', 'idx_docs_embedding', '[0.12, ...]', 5, 'lang=en');

-- report which strategy the planner chose and the filter selectivity
SELECT hnsw_hybrid_explain('docs', 'idx_docs_embedding', '[0.12, ...]', 5, 'lang=en');

In auto mode the engine measures the filter's selectivity. Selectivity is the number of matching rows divided by the total rows. Then it chooses a strategy. When fewer than 10% of rows match, it pre-filters. It ranks only the matching rows. Otherwise it post-filters. It ranks a wider, oversampled candidate set and then drops the non-matches. An empty filter falls back to plain hnsw_knn. In this version the filter predicate can only be equality.

10.5. Full-Text Indexes

A full-text index uses CREATE FULLTEXT INDEX. It supports word matching inside a text column. It is built as an inverted index on top of a B+ tree, in FullText.h. The text is tokenized. One entry is inserted per token. Each entry is keyed <word>␟<PK>, using the \x1F separator again. The value is the row's heap RowID. A query tokenizes the search string in the same way. Then it looks up each token's word prefix.

-- build the inverted index over a text column
CREATE FULLTEXT INDEX idx_articles_body ON articles (body);

-- query it with MATCH ... AGAINST
SELECT id, title FROM articles
    WHERE MATCH(body) AGAINST('storage engine');

10.5.1. The Tokenizer

The same tokenizer, ft_tokenize, runs at both build time and query time. So the two see the same token stream. Its rules are listed below.

Word boundaries
A token is a run of ASCII alphanumerics. Any byte ≥ 0x80 counts as part of a word. This keeps multi-byte UTF-8 letters attached to their words, such as Turkish characters. Any other byte ends the current token.
Lowercasing
ASCII letters are lowercased, so matching is case-insensitive. Non-ASCII bytes pass through unchanged. This is because tolower on raw UTF-8 bytes is undefined. So in this version, queries against non-ASCII text match case-sensitively. Turkish text is one example. A proper Unicode case-fold is future work.
Short-token filter
Tokens shorter than 2 characters are dropped.
Stop words
A small built-in English stop-word list is removed. It holds 28 words, such as a, an, and, the, of, and to.
Limits
A token is capped at 64 bytes. This limit is set by FT_MAX_TOKEN_LEN. Longer tokens are truncated. At most 256 tokens are taken from a single field. This limit is set by FT_MAX_TOKENS.

Note. Not auto-maintained. In this version the full-text index is not updated automatically on INSERT, UPDATE, or DELETE. After bulk DML, drop it and re-create it with CREATE FULLTEXT INDEX to refresh the token entries. There is no stemming and no phrase search. There is no relevance ranking beyond whether a token is present. Matching uses the shared, normalized token stream described above.

10.6. Compatibility

In the SQL standard, indexing is implementation-defined. So none of these statements are part of standard SQL. EvolutionDB's surface follows PostgreSQL and MySQL closely. There are some clear differences that are worth knowing.

  • Access methods. EvolutionDB offers the default B+ tree, USING HASH, and USING HNSW. It does not support PostgreSQL's gin, gist, spgist, and brin method names. It also does not support a generic USING method clause.
  • Hash indexes are static. PostgreSQL's hash index grows on its own. EvolutionDB's does not. The bucket count is fixed at create time. The default is 64 and the max is 1017. It uses overflow chains. There is no rehash and no extendible split.
  • HNSW. The opclass names vector_cosine_ops, vector_l2_ops, vector_ip_ops, and vector_l1_ops, plus the m and ef_construction build parameters, match the pgvector extension. EvolutionDB's graph is in-memory only. It is not auto-rebuilt at startup. You must use a single VECTOR column. Composite HNSW indexes are not supported.
  • Full-text. EvolutionDB uses a dedicated MySQL-style CREATE FULLTEXT INDEX statement with MATCH … AGAINST queries. It does not use PostgreSQL's tsvector and GIN model. It is not auto-maintained on DML. It offers no stemming, no phrase search, and no ranked relevance.
  • Unsupported B+ tree and hash clauses. On non-HNSW indexes, these clauses are all rejected. They are ASC/DESC, NULLS FIRST/LAST, COLLATE, per-column opclasses, INCLUDE, partial-index WHERE, TABLESPACE, and storage-parameter WITH (…).
