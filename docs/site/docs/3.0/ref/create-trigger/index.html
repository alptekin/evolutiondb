CREATE TRIGGER

Create a row-level trigger. Its body runs on its own before or after an INSERT, UPDATE, or DELETE on a table or memory store.

Synopsis

CREATE TRIGGER name
    { BEFORE | AFTER }
    { INSERT | UPDATE | DELETE | PUT }
    ON { table_name | MEMORY STORE store_name }
    FOR EACH ROW
BEGIN
    trigger_body
END

Description

CREATE TRIGGER adds a new trigger to a table that already exists. It also works on the backing table of a memory store. After you create it, the trigger fires for every row that the matching event touches. The body runs on the same engine that runs stored procedures. So you can use IF, WHILE, SIGNAL, and other procedural statements inside the body.

EvolutionDB triggers are always row-level. FOR EACH ROW is required. It is the only form the grammar accepts. There is no statement-level form. Each trigger fires on one event and one timing only. To react to several events, you must create one trigger for each event.

Inside the body, the special names OLD.column and NEW.column hold the row values before and after the operation. The engine replaces these names with real values just before it runs the body. OLD is the existing row. It is available for UPDATE and DELETE. NEW is the incoming row. It is available for INSERT and UPDATE. If a value is not known, the engine replaces the reference with NULL.

In a BEFORE trigger, the engine gives special meaning to a statement like SET NEW.column = expression;. It works out the expression and writes the result back into that column of the row. The row is the one being inserted or updated. This lets the trigger change the row before it is stored. The engine does this rewrite for BEFORE triggers only.

A trigger body can run its own DML. That DML can fire other triggers. There is a fixed limit on how deep this nesting can go. Past that limit, the engine quietly stops firing more triggers. If the body raises an error, the operation that fired it fails. The error then passes back to the caller. It carries its SQLSTATE when one is set.

It fails if you create a trigger with a name that already exists. The error is SQLSTATE 42P07. It fails if you name a target table that does not exist. The error is SQLSTATE 42P01. It fails if you name a memory store that does not exist. The error is SQLSTATE 42704.

Parameters

name
The name of the new trigger. The engine looks up trigger names across all tables in the current schema. So a name must be unique within the schema.
BEFORE | AFTER
When the trigger runs in relation to the operation. BEFORE runs the body before the row change is applied. It also turns on SET NEW.column rewrites. AFTER runs the body after the row change is applied.
INSERT | UPDATE | DELETE | PUT
The one event that fires the trigger. PUT is an alias. It works the same as INSERT. You can give only one event per trigger. There is no multi-event form like INSERT OR UPDATE. You cannot limit UPDATE to a list of columns.
ON table_name
The table that the trigger is attached to. The table must already exist in the current database and schema.
ON MEMORY STORE store_name
Attach the trigger to a memory store instead of a normal table. The engine looks up the store name in the memory-store catalog. It then maps the name to its backing table named mem_store_name. The trigger fires on that backing table just like a normal table trigger.
FOR EACH ROW
Required. It sets row-level firing. The body runs once for each row that is touched. This is the only form supported.
trigger_body
The procedural statements between BEGIN and END. They run for each matching row. The body uses the same stored-procedure engine. It can read OLD.column and NEW.column. In BEFORE triggers it can also assign to NEW.column with SET.

Examples

Set a value in a new row before it is inserted. This is a BEFORE trigger with SET NEW.

CREATE TRIGGER trg_audit_stamp
    BEFORE INSERT ON orders
    FOR EACH ROW
BEGIN
    SET NEW.status = 'pending';
END;

Copy deleted rows into an audit table. This is an AFTER trigger that uses OLD.

CREATE TRIGGER trg_archive_deleted
    AFTER DELETE ON orders
    FOR EACH ROW
BEGIN
    INSERT INTO orders_archive (id, status)
    VALUES (OLD.id, OLD.status);
END;

Block an update with SIGNAL when a check is true. This is a BEFORE trigger that uses both OLD and NEW.

CREATE TRIGGER trg_no_downgrade
    BEFORE UPDATE ON orders
    FOR EACH ROW
BEGIN
    IF NEW.total < OLD.total THEN
        SIGNAL SQLSTATE '45000';
    END IF;
END;

Attach a trigger to a memory store. Here PUT works the same as INSERT.

CREATE TRIGGER trg_cache_touch
    AFTER PUT ON MEMORY STORE sessions
    FOR EACH ROW
BEGIN
    -- runs against the backing table mem_sessions
    INSERT INTO session_log (sid) VALUES (NEW.sid);
END;

Compatibility

EvolutionDB's CREATE TRIGGER is close to the SQL standard and PostgreSQL. It is narrower and feels more like MySQL. Several PostgreSQL clauses are not supported.

  • Inline body, not a function. You write the action straight into a BEGIN … END block. There is no EXECUTE FUNCTION or EXECUTE PROCEDURE clause. There is no separate trigger function. The body runs on the procedure engine.
  • Row-level only. FOR EACH ROW is required. The PostgreSQL FOR EACH STATEMENT form is not available.
  • Single event only. You give one of INSERT, UPDATE, DELETE, or PUT per trigger. PostgreSQL's INSERT OR UPDATE OR DELETE combinations are not accepted. UPDATE OF column is not accepted. TRUNCATE triggers are not supported.
  • No WHEN condition. There is no WHEN (condition) clause. You must write the gate logic inside the body. For example, use IF … SIGNAL.
  • No REFERENCING or transition tables. Old and new values come only through the OLD.column and NEW.column substitution. There are no named transition tables.
  • PUT alias and MEMORY STORE target are EvolutionDB extensions. PostgreSQL and the SQL standard have no match for them.
  • Row change with SET NEW.column in BEFORE triggers follows the MySQL way. It does not use PostgreSQL's "assign to NEW and RETURN" model.
