REVOKE

Take away access privileges or role membership that you gave to a database user before.

Synopsis

REVOKE { ALL [ PRIVILEGES ] | privilege [, ...] }
    ON { DATABASE | SCHEMA | TABLE } scope_name
    FROM user_name

-- Role membership form
REVOKE role_name FROM user_name

where privilege is one of:

SELECT | INSERT | UPDATE | DELETE | CREATE | DROP

Description

REVOKE takes away one or more privileges that a user got before. It can also take away a user's membership in a role. It does the opposite of GRANT.

EvolutionDB keeps privileges in the system catalog. There is one grant entry for each user and scope. When you revoke privileges, EvolutionDB finds the matching grant for the user and scope. It removes the privilege bits you asked for. Then it does the following.

  • If some privileges are left, EvolutionDB saves the grant entry again with the smaller set.
  • If no privileges are left, EvolutionDB deletes the grant entry.
  • If you revoke ALL or ALL PRIVILEGES, EvolutionDB also clears any WITH GRANT OPTION flag on the entry.

The user and scope must have a grant entry. If there is no grant entry, the command fails with an error. The SQLSTATE is 42704, which means undefined object. The error says that no matching grant was found.

Scopes. A privilege belongs to one of three scope kinds. These are DATABASE, SCHEMA, and TABLE. The scope_name names the object. The revoke must use the same scope kind and name that the original grant used. A revoke on a TABLE scope does not change a wider DATABASE or SCHEMA grant for the same user.

PUBLIC grants are additive. EvolutionDB uses the PostgreSQL additive model. EvolutionDB checks privileges granted to PUBLIC on their own, apart from user grants. A revoke for one user removes only that user's own grant entry. It does not hide or override a privilege the user still has through a PUBLIC grant. To take a privilege away from everyone, revoke it from PUBLIC too.

Role form. The command has no ON keyword in this form. EvolutionDB then reads it as a role-membership revoke. REVOKE role_name FROM user_name removes the user's membership in the named role.

Who may revoke. The admin user can always revoke. A non-admin user can revoke a privilege only when that user holds the same privilege WITH GRANT OPTION on the same scope. If not, the command is denied with SQLSTATE 42501. Only admin can do role-membership revokes.

On success the command reports a tag. It reports REVOKE for the privilege form. It reports REVOKE ROLE for the role-membership form.

Parameters

ALL [ PRIVILEGES ]
Revoke every privilege EvolutionDB tracks for the scope. These are SELECT, INSERT, UPDATE, DELETE, CREATE, and DROP. The keyword PRIVILEGES is optional. EvolutionDB accepts it but ignores it. This matches the noise-word usage in the SQL standard. Revoking ALL also clears the grant's WITH GRANT OPTION flag.
privilege
One privilege to remove. The allowed values are SELECT, INSERT, UPDATE, DELETE, CREATE, and DROP. You can list more than one privilege. Separate them with commas. Privilege names are case-insensitive. EvolutionDB quietly ignores tokens it does not know. If the final privilege set is empty, the command fails with an invalid privilege specification error.
ON DATABASE | SCHEMA | TABLE scope_name
Picks the scope kind and names the object the grant applies to. You must give exactly one of DATABASE, SCHEMA, or TABLE. You can write scope_name with no quotes. You can also use single quotes like 'name' or double quotes like "name". It must match the scope kind and name used by the original GRANT.
FROM user_name
The user whose privileges or role membership you are removing. You can also name PUBLIC here. You can write it with no quotes or with single quotes.
role_name
This is used in the role-membership form, which has no ON clause. It is the name of the role you take away from user_name. If the user is not a member of that role, the command fails.

Not supported. EvolutionDB's REVOKE does not support some SQL-standard features. It does not accept the GRANT OPTION FOR clause. It does not accept the CASCADE or RESTRICT behavior keywords. It does not accept column-level privilege lists. There is no REVOKE ... FROM PUBLIC shorthand beyond naming PUBLIC as the user. You cannot revoke from more than one user in one statement. To revoke a single privilege from a grant that holds ALL, that privilege must be in the existing grant.

Examples

Take away the right to insert into a table. The read access stays.

REVOKE INSERT ON TABLE orders FROM reporting;

Revoke several privileges at the same time.

REVOKE INSERT, UPDATE, DELETE ON TABLE orders FROM reporting;

Take away all privileges on a schema. This also clears any grant option the user had.

REVOKE ALL PRIVILEGES ON SCHEMA sales FROM analyst;

Revoke a database-wide privilege from everyone. Target PUBLIC to do this.

REVOKE SELECT ON DATABASE testdb FROM PUBLIC;

Use a quoted scope name when the name is not a plain identifier.

REVOKE SELECT ON TABLE "Quarterly Report" FROM analyst;

Remove a user's membership in a role. Only admin can do this.

REVOKE analysts FROM alice;

Compatibility

EvolutionDB's REVOKE handles the most common case in the SQL standard and in PostgreSQL. It removes object privileges from a user. But it differs in a few ways.

  • The scope keyword is required. The SQL standard and PostgreSQL work out the object class on their own. EvolutionDB does not. It needs an explicit ON DATABASE, ON SCHEMA, or ON TABLE keyword.
  • No CASCADE or RESTRICT. EvolutionDB does not accept the standard's behavior keywords for dependent privileges. A revoke does not also take away privileges that the user may have re-granted to others.
  • No GRANT OPTION FOR. You cannot revoke only the grant option and keep the privilege itself. Revoking ALL clears the grant option. Revoking single privileges leaves the flag the same on whatever is left.
  • No column-level privileges and no lists of more than one user.
  • Privilege set. EvolutionDB tracks only SELECT, INSERT, UPDATE, DELETE, CREATE, and DROP. It does not model PostgreSQL privileges such as TRUNCATE, REFERENCES, TRIGGER, CONNECT, TEMPORARY, EXECUTE, and USAGE.
  • Additive PUBLIC model. As in PostgreSQL, a revoke for one user does not override a privilege the user still has through a grant to PUBLIC.
  • Role revoke is admin-only. It uses the same flat REVOKE role FROM user form as PostgreSQL. But it has no ADMIN OPTION FOR and no cascade behavior.
