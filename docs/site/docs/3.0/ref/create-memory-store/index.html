CREATE MEMORY STORE

Create a new long-term agent memory store. It is a namespaced key/value store. It can also hold vector embeddings for semantic search.

Synopsis

CREATE MEMORY STORE [ IF NOT EXISTS ] store_name
    [ WITH ( option [, ... ] ) ]

where option is one of:

    embedding_dim = N
    distance = 'cosine' | 'l2' | 'inner' | 'ip' | 'l1'

Description

CREATE MEMORY STORE adds a new memory store in the current schema. It also creates the table behind it. A memory store is made for agent long-term memory. Each record has a namespace and a key. Each record holds a JSON value. Each record can also hold a vector embedding for semantic similarity search.

The store is backed by a normal table. The table is named mem_<store_name> and is created in the current schema. The table has these columns.

  • mem_namespace, VARCHAR(255) NOT NULL
  • mem_key, VARCHAR(255) NOT NULL
  • pk, VARCHAR(512) PRIMARY KEY. It joins the namespace and the key.
  • mem_value, JSON. It can be null.
  • embedding, VECTOR(N). It exists only when embedding_dim is greater than 0.
  • created_at, TIMESTAMP
  • ttl_at, TIMESTAMP. It can be null.

When you leave out embedding_dim or set it to 0, there is no embedding column. The store then works as a plain namespaced JSON key/value table. In that case MEMORY SEARCH on the store fails. There is no embedding column to score against.

The backing table is a normal table. So everything that works for tables works for a memory store. This includes MVCC, primary-key uniqueness on (namespace, key), row-level security policies, triggers, and foreign-key cascades. The MEMORY PUT, GET, SEARCH, DELETE, and LIST NAMESPACES commands are simple wrappers. They turn into normal INSERT/SELECT/DELETE statements on mem_<store_name>.

To create a store, no memory store with the same name may exist yet. With IF NOT EXISTS, no error is raised if the store already exists. The existing store stays the same.

Parameters

IF NOT EXISTS
Do nothing if a memory store with the same name already exists. No error is raised. The existing store is not changed. Its embedding_dim and distance settings stay the same.
store_name
The name of the memory store to create. The backing table is named mem_<store_name> and is created in the current schema.
embedding_dim = N
The number of dimensions in the vector embedding kept with each record. When N > 0, an embedding VECTOR(N) column is added to the backing table. The store then supports MEMORY SEARCH ... USING VECTOR. When you leave out this option, the default is 0. No embedding column is created. The store is a plain JSON key/value store.
distance = 'metric'
The distance metric used by MEMORY SEARCH to rank results against the query vector. This setting is saved on the store. It picks which vector function scores each row at search time. These are the accepted values.
  • 'cosine' is cosine distance. This is the default. It is also used when you leave out the option or give a value that is not known.
  • 'l2' is Euclidean distance. It is also called L2.
  • 'inner' or 'ip' is inner product.
  • 'l1' is Manhattan distance. It is also called L1.
The metric matters only when embedding_dim > 0.

Examples

Create a plain JSON key/value memory store. It has no semantic search.

CREATE MEMORY STORE agent_notes;

Create a store with 1536-dimension embeddings. It uses the default cosine metric.

CREATE MEMORY STORE agent_memory
    WITH (embedding_dim = 1536);

Create a store with embeddings ranked by L2 distance. Create it only if it does not already exist.

CREATE MEMORY STORE IF NOT EXISTS chat_recall
    WITH (embedding_dim = 768, distance = 'l2');

This example populates a store and queries it after creating it.

-- store a value (upsert on namespace+key)
MEMORY PUT INTO agent_memory
    VALUES ('user:42', 'pref:lang', '{"lang":"tr"}', '[0.1,0.2,...]');

-- read it back
MEMORY GET FROM agent_memory
    WHERE NS = 'user:42' AND KEY = 'pref:lang';

-- semantic search (requires embedding_dim > 0)
MEMORY SEARCH agent_memory USING VECTOR '[0.1,0.2,...]' LIMIT 5;

Compatibility

CREATE MEMORY STORE is an EvolutionDB extension. It is not part of the SQL standard. PostgreSQL has no matching statement. The closest thing in PostgreSQL is to create a table and add a pgvector column by hand. The EvolutionDB memory store puts the namespaced key/value schema, the optional embedding column, and the distance metric into one simple command.

Here are some notes on the EvolutionDB behavior.

  • The WITH clause accepts only the embedding_dim and distance options. There is no other storage-parameter syntax. PostgreSQL-style table options are not accepted here.
  • The only known distances are 'cosine', 'l2', 'inner'/'ip', and 'l1'. Any other string quietly falls back to 'cosine'.
  • The backing table mem_<store_name> is a real table. It is fully visible to normal SQL. You can read, secure, or trigger its rows just like any user table.
