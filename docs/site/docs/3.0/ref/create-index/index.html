CREATE INDEX

Build a secondary index on a table. An index can cover one or more columns. It can also cover one supported expression. An index speeds up lookups, equality scans, full-text matching, and approximate nearest-neighbor search.

Synopsis

CREATE [ UNIQUE ] INDEX [ CONCURRENTLY ] [ IF EXISTS ] name ON table
    [ USING HASH ]
    ( { column_name | expression } [, ...] )

-- Full-text inverted index
CREATE FULLTEXT INDEX name ON table ( text_column )

-- HNSW approximate nearest-neighbor (ANN) index on a VECTOR column
CREATE INDEX [ IF EXISTS ] name ON table USING HNSW
    ( vector_column [ opclass ] )
    [ WITH ( m = N [, ef_construction = N ] ) ]

where expression is one of:
    UPPER(col) | LOWER(col) | LENGTH(col) | CONCAT(col1, col2)

where opclass is one of:
    vector_cosine_ops | vector_l2_ops | vector_ip_ops | vector_l1_ops

Description

CREATE INDEX builds a secondary index on the named table. The default access method is a B+ tree. The tree is stored in the unified page file. Each entry is a composite key of the form indexed_value␟PK. The \x1F unit separator joins the indexed value with the row's primary key. The entry points directly at the heap row location. So a matched index entry reaches its row without a second primary-key lookup.

The index is filled right away. The engine scans every existing row in the table. It builds the key for each row and inserts it. The command runs while it holds the engine's write lock. So the table is paused during the build. The one exception is when you use CONCURRENTLY. See below.

An index can cover a single column. It can cover several columns. This is a composite index. Its key is the column values joined with |. An index can also cover one supported expression. Indexes on ARRAY columns are rejected in this version.

EvolutionDB supports four index shapes. You pick the shape with the modifiers and the access-method clause.

  • B+ tree. This is the default. It is an ordered index. It supports equality and the prefix scans used by the planner.
  • Hash with USING HASH. This is an equality-only index. It is backed by an extendible-hash directory.
  • Full-text with CREATE FULLTEXT INDEX. This is an inverted index. The text column is split into tokens. One (token, PK) pair is inserted per token.
  • HNSW with USING HNSW. This is an in-memory graph. It is used for approximate nearest-neighbor search over a VECTOR column.

An index may already exist on the same column list. If that index is not the primary-key index, CREATE INDEX fails with index already exists on column '…'. You can avoid the error with IF EXISTS. Then the statement does nothing.

Parameters

UNIQUE
Makes sure no two rows have the same indexed value. During the build, each key is checked for duplicates. The first duplicate stops the command with could not create unique index: duplicate key '…'. The partly built index is then destroyed. You can combine this with USING HASH. You cannot use it with FULLTEXT or USING HNSW.
CONCURRENTLY
Builds the index without holding the write lock for the whole scan. First the index is registered in a building state. Then it is filled in batches. The lock is released between batches. So other queries can keep running. Writes that happen during the build are saved in a modification log. They are applied in a final pass before the index is marked ready. Only the building session pays the cost. Readers ignore an index that is still building.
IF EXISTS
An index may already exist on the same column list. With this option, the statement does nothing instead of failing. Watch the spelling. EvolutionDB's grammar accepts the IF EXISTS token sequence here. It carries the standard "if not exists" meaning. It skips creation if a matching index is already there.
name
The name of the index to create. DROP INDEX looks up index names across the whole database. So pick a name that is unique across the database.
table
The table to create the index on. It must already exist. If it does not, the command fails with table '…' does not exist.
USING HASH
Picks an extendible-hash index instead of the default B+ tree. Hash indexes are created with the engine's default bucket count. They support equality lookups only.
column_name [..]
One or more columns to index. Each named column must exist in the table. If a column does not exist, the command fails with column '…' does not exist in table '…'. More than one column makes a composite index. Its key is the column values joined together.
expression
An expression index over one supported function of a column. The accepted forms are exactly UPPER(col), LOWER(col), LENGTH(col), and CONCAT(col1, col2). The expression is stored with the index. At build time and during maintenance, the engine runs it for each row to make the key. Other SQL expressions are not supported.
FULLTEXT (in CREATE FULLTEXT INDEX)
Builds an inverted full-text index on one text column. The column is split into tokens at build time. One entry per token is inserted. This supports MATCH queries.

The full-text index is not kept up to date on INSERT, UPDATE, or DELETE in this version. After bulk DML, drop the index and create it again with CREATE FULLTEXT INDEX to refresh it.

USING HNSW
Builds a Hierarchical Navigable Small World graph for approximate nearest-neighbor search. The indexed column must be one VECTOR column. Composite HNSW indexes are not supported in this version. If the column is wrong, the command fails with HNSW index requires a VECTOR column… or HNSW index supports a single vector column…. The graph lives in memory only. The build parameters are saved in the catalog. So the graph can be rebuilt at server startup.
opclass (HNSW)
The distance function for the HNSW graph. The choices are vector_cosine_ops, vector_l2_ops, vector_ip_ops, and vector_l1_ops. The default is vector_cosine_ops. vector_ip_ops means inner product. The name vector_inner_ops is also accepted for it. An unknown opclass falls back to cosine.
WITH ( m = N, ef_construction = N ) (HNSW)
Tunes how the graph is built. m is the number of two-way links per node. ef_construction is the size of the candidate list at build time. The defaults are m = 16 and ef_construction = 64. For the same key, later entries in the list replace earlier ones.

Examples

Create a plain B+ tree index on one column.

CREATE INDEX idx_users_email ON users (email);

Create a unique index.

CREATE UNIQUE INDEX idx_users_username ON users (username);

Create a composite index over two columns.

CREATE INDEX idx_orders_cust_date ON orders (customer_id, order_date);

Skip creation if a matching index is already there.

CREATE INDEX IF EXISTS idx_users_email ON users (email);

Create a hash index for equality lookups.

CREATE INDEX idx_sessions_token ON sessions USING HASH (token);

Create an expression index. This gives case-insensitive lookups on a lowercased column.

CREATE INDEX idx_users_lower_email ON users (LOWER(email));

Build an index without locking the table for the whole scan.

CREATE INDEX CONCURRENTLY idx_events_type ON events (event_type);

Build a full-text index. Then query it.

CREATE FULLTEXT INDEX idx_articles_body ON articles (body);

Build an HNSW vector index with your own build parameters.

CREATE INDEX idx_docs_embedding ON docs
    USING HNSW (embedding vector_l2_ops)
    WITH (m = 32, ef_construction = 128);

Compatibility

CREATE INDEX is not part of the SQL standard. The standard leaves indexing to each database. EvolutionDB's syntax follows PostgreSQL closely. But it differs in a few clear ways.

  • IF EXISTS vs IF NOT EXISTS. PostgreSQL spells the skip-if-present option IF NOT EXISTS. EvolutionDB's grammar accepts the token sequence IF EXISTS in this spot. It has the same "skip if a matching index already exists" meaning.
  • Access methods. EvolutionDB supports the default B+ tree, USING HASH, and USING HNSW. PostgreSQL's btree, gin, gist, spgist, and brin method names are not supported. A generic USING method clause is also not supported.
  • Full-text. EvolutionDB uses a dedicated CREATE FULLTEXT INDEX statement in the MySQL style. It does not use PostgreSQL's tsvector and GIN approach. The full-text index is not kept up to date on DML.
  • Expression indexes. These are limited to UPPER, LOWER, LENGTH, and CONCAT(a, b). PostgreSQL allows any immutable expression and parenthesized expression lists.
  • Unsupported clauses. EvolutionDB does not accept PostgreSQL's ASC, DESC, NULLS FIRST/LAST, COLLATE, per-column opclasses for B+ tree and hash indexes, INCLUDE, WHERE for partial indexes, TABLESPACE, or a storage-parameter WITH (...) on non-HNSW indexes. The HNSW WITH clause accepts only m and ef_construction.
  • HNSW. The opclass names vector_cosine_ops, vector_l2_ops, vector_ip_ops, and vector_l1_ops mirror the pgvector extension. The WITH parameters mirror it too. The HNSW graph is held in memory. It is rebuilt at startup from parameters stored in the catalog.
