CREATE ENTITY STORE

Create a named entity store. An entity store keeps short summaries of things, looked up by a key. It tracks how many times you wrote each entity and when you last saw it. It is built on top of a normal EvolutionDB table.

Synopsis

CREATE ENTITY STORE [ IF EXISTS ] name

Description

CREATE ENTITY STORE adds a new entity store named name in the current schema. An entity store sits on top of a regular EvolutionDB table. The command makes a backing table and a catalog entry. The catalog entry links the store name to that table. After this, you can use the entity DML verbs ENTITY PUT, ENTITY GET, and ENTITY RANK instead of writing raw SQL.

The backing table is created for you. It is named ent_name. It always has the same five columns.

entity_key VARCHAR(255) PRIMARY KEY NOT NULL
The lookup key for the entity. It is the primary key. Keys are unique and indexed by a B+ tree. ENTITY PUT upserts against this key.
summary VARCHAR(8000)
A free-text summary of the entity. It can be null.
facts VARCHAR(8000)
An optional JSON-shaped string of structured facts. It can be null. It is stored as SQL NULL when you leave it out.
mention_count INT NOT NULL
How many times the entity has been written. It is set to 1 on the first ENTITY PUT. It goes up by one on every later put for the same key.
last_seen TIMESTAMP NOT NULL
The UTC ISO timestamp of the most recent put. It uses the form YYYY-MM-DD HH:MM:SS.

The store is saved in the system catalog as an entity store entry. The entry holds its name and the id of the backing table. If a store with the same name already exists, the command raises an error with SQLSTATE 42710. You can stop that error with the IF EXISTS clause. In that case the existing store is left alone and the command quietly succeeds. A missing name is reported as a syntax error with SQLSTATE 42601.

You can drop the store with DROP ENTITY STORE. This drops the backing ent_name table and removes the catalog entry.

Note on the guard clause. The grammar accepts the exact tokens IF EXISTS here. It does not use the usual SQL form IF NOT EXISTS. The behavior still matches the normal if-not-exists meaning. When the store already exists, the statement does nothing instead of raising an error.

Parameters

IF EXISTS
Do not raise an error if an entity store with the same name already exists. The existing store stays in place and the command returns success. Without this clause, trying to create a store that already exists fails with entity store 'name' already exists and SQLSTATE 42710.
name
The name of the entity store to create. The backing table is made as ent_name in the current schema. The store is saved under this name in the catalog. This is required.

Examples

Create an entity store named people.

CREATE ENTITY STORE people;

Create it only if it does not already exist. This makes the setup safe to run again.

CREATE ENTITY STORE IF EXISTS people;

Create a store and start filling it with the entity DML verbs.

CREATE ENTITY STORE people;

-- upsert by key: summary only
ENTITY PUT INTO people VALUES ('alice', 'Lead engineer on the storage team');

-- upsert by key: summary + JSON-shaped facts
ENTITY PUT INTO people VALUES ('alice', 'Storage team lead', '{"role":"lead","team":"storage"}');

-- look up one entity by key
ENTITY GET FROM people WHERE KEY = 'alice';

-- rank entities by mention_count DESC (tie-broken by last_seen DESC)
ENTITY RANK FROM people LIMIT 10;

Drop the store and its backing table.

DROP ENTITY STORE people;

Compatibility

CREATE ENTITY STORE is an EvolutionDB extension. The SQL standard has no CREATE ENTITY STORE statement. PostgreSQL has no matching command. So this is not portable.

The clause spelling is different from common SQL. EvolutionDB's grammar accepts IF EXISTS as the guard at create time. PostgreSQL and the wider SQL community use IF NOT EXISTS for the same "create only if absent" intent. The runtime behavior still matches the usual if-not-exists meaning.

The store is not a separate storage engine. It is built fully on top of a normal EvolutionDB table named ent_name. That table has a B+ tree primary-key index on entity_key. The entity DML verbs turn into ordinary INSERT, UPDATE, and SELECT execution paths. The backing table is visible to regular SQL queries.
