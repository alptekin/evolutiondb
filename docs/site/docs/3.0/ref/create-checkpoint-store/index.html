CREATE CHECKPOINT STORE

Register a named checkpoint store for LangGraph-style agent state. It also creates the backing table for you.

Synopsis

CREATE CHECKPOINT STORE [ IF NOT EXISTS ] name
    [ WITH ( retention = 'interval_text' ) ]

Description

CREATE CHECKPOINT STORE registers a checkpoint store named name in the system catalog. The same statement also creates a backing heap table. This table holds the real checkpoint rows. The store is EvolutionDB's native backend for the LangGraph BaseCheckpointSaver contract. It saves per-thread agent snapshots. This lets a graph run start again from where it stopped.

The backing table is created in the current session schema. It has the fixed name ck_name. It uses this fixed schema.

thread_id VARCHAR(255) NOT NULL
The conversation or graph thread that owns the checkpoint.
checkpoint_ns VARCHAR(255)
The namespace inside the thread. This can be NULL. An empty string becomes NULL in this engine.
checkpoint_id VARCHAR(255) PRIMARY KEY
The unique id of the checkpoint. The primary-key B+ tree indexes it.
parent_id VARCHAR(255)
The id of the checkpoint that this one comes from. This is NULL for root checkpoints.
ck_values JSON
The serialized channel values payload. It is named ck_values so it does not clash with the SQL reserved word values.
metadata JSON
Application metadata that is attached to the checkpoint.
parent_config JSON
The serialized configuration of the parent checkpoint.
created_at TIMESTAMP
The time the row was inserted. The engine sets it for you on each write.

The store is just a catalog entry over a normal table. So every standard engine feature works on it without any change. MVCC visibility, row-level security, triggers, and replication all work on the backing table. They work the same as on a table you write by hand.

The statement fails if a store with the same name already exists. It returns SQLSTATE 42710. You can avoid this with IF NOT EXISTS. Then the statement does nothing and raises no error. A missing or empty name is a syntax error. It returns SQLSTATE 42601.

Parameters

IF NOT EXISTS
Do not raise an error if a checkpoint store with the same name already exists. When the store exists, the statement just returns. It creates nothing and changes nothing. This guard only checks for a checkpoint store with that name. It does not make sure the backing table is the same.
name
The name of the checkpoint store to create. It must match the engine's identifier pattern [A-Za-z][A-Za-z0-9_]*. The backing table is created as ck_name in the current schema.
WITH ( retention = 'interval_text' )
This is optional. It stores a free-form retention text on the store descriptor in the catalog. For example, you can use '30 days' or '1 hour'. The value is stored as is. The outer quotes are removed. It is only descriptive metadata. This release does no automatic retention pruning. The engine does not delete old checkpoints based on this value. retention is the only key allowed inside the WITH clause.

Examples

Create a checkpoint store named agent_runs.

CREATE CHECKPOINT STORE agent_runs;

Create it only if it does not already exist.

CREATE CHECKPOINT STORE IF NOT EXISTS agent_runs;

Create a store and save a retention text on it.

CREATE CHECKPOINT STORE agent_runs
    WITH (retention = '30 days');

A normal table backs the store. So you can also read and write checkpoints straight against ck_name.

INSERT INTO ck_agent_runs (thread_id, checkpoint_id, ck_values, metadata)
VALUES ('thread-1', 'ckpt-001', '{"step":1}', '{"source":"loop"}');

SELECT checkpoint_id, created_at
FROM   ck_agent_runs
WHERE  thread_id = 'thread-1'
ORDER BY created_at DESC;

The grammar also accepts native checkpoint DML. These are CHECKPOINT PUT [WRITES] INTO name VALUES (…), CHECKPOINT GET FROM name THREAD 'id' [AT 'checkpoint_id'], and CHECKPOINT LIST FROM name THREAD 'id' [LIMIT n]. They compile down to INSERT or SELECT against the same backing table. GET returns the newest matching row. LIST returns the history with the newest first.

Compatibility

CREATE CHECKPOINT STORE is an EvolutionDB extension. It is not part of the SQL standard. It has no match in PostgreSQL. PostgreSQL has no checkpoint-store object. Its CHECKPOINT command is a different thing. That command flushes WAL and buffers.

EvolutionDB accepts only two clauses here. They are IF NOT EXISTS and the single-key WITH (retention = '…') option from the Synopsis. There is no TABLESPACE, owner, encoding, or other PostgreSQL CREATE … clause. The WITH clause looks like a normal storage-parameter list. But retention is the only key it knows. The engine stores it as descriptive metadata. It does not enforce it in this release.
