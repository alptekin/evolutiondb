SAVEPOINT

Create a new savepoint inside the current transaction. A savepoint marks a point you can roll back to later. You do not have to stop the whole transaction to do this.

Synopsis

SAVEPOINT savepoint_name

Description

SAVEPOINT sets a named marker inside the open transaction. After you set a savepoint, you have two choices. You can run ROLLBACK TO SAVEPOINT to undo every change made after the savepoint. The savepoint and the work done before it stay in place. You can also run RELEASE SAVEPOINT to drop the marker. Its work then becomes part of the transaction around it.

In EvolutionDB, the protocol adaptor reads transaction control statements. The SQL grammar parser does not. The adaptor keeps an undo log for each session. This log holds the changes made since the transaction began. When you create a savepoint, the adaptor saves the current length of that undo log. When you roll back to the savepoint, the adaptor replays the undo entries in reverse. It stops when the log is back at the saved length. So a savepoint is just a saved position in the undo stack. It is not a nested sub-transaction with its own transaction id.

You can only create a savepoint while a transaction is open. If no transaction is running, the command fails with SQLSTATE 25P01. The message is "SAVEPOINT can only be used in a transaction". A savepoint name is required. If you run SAVEPOINT with no name, it fails with SQLSTATE 42601.

You can create a savepoint with the same name as one that already exists in the transaction. In that case EvolutionDB overwrites the old one in place. It moves the marker to the current position in the undo log. This is not how the SQL standard and PostgreSQL work. They keep both savepoints and hide the older one. See Compatibility for more.

Savepoint names are compared without regard to case. They are cut to 127 characters. One transaction can hold up to 64 active savepoints. If you try to create more, the command fails with SQLSTATE 53000. The message is "too many savepoints".

Note. The keyword and the name are read by position. The adaptor reads the word SAVEPOINT. Then it takes the next token as the savepoint name. A token ends at a space, a semicolon, or the end of the statement. There is no quoting and no parenthesized option syntax.

Parameters

savepoint_name
The name for the new savepoint. This is required. The name is one token with no spaces. Later you can use it with ROLLBACK TO SAVEPOINT or RELEASE SAVEPOINT. The match does not care about case. The name is stored cut to 127 characters. If you reuse a name that already exists in the transaction, it overwrites that savepoint. It does not create a second one.

Examples

Set a savepoint. Then roll back to it to drop only the most recent change.

BEGIN;

INSERT INTO accounts VALUES (1, 'opening');

SAVEPOINT after_open;

INSERT INTO accounts VALUES (2, 'oops');

-- undo the second insert, keep the first
ROLLBACK TO SAVEPOINT after_open;

INSERT INTO accounts VALUES (3, 'correct');

COMMIT;
-- rows 1 and 3 are committed; row 2 is gone

Release a savepoint once you are sure its work is good. After the release, the marker is gone. The changes stay part of the transaction.

BEGIN;

SAVEPOINT sp1;

UPDATE inventory SET qty = qty - 1 WHERE id = 42;

RELEASE SAVEPOINT sp1;

COMMIT;

Reusing a savepoint name moves the marker forward. It does not stack a new one.

BEGIN;

INSERT INTO log VALUES ('a');
SAVEPOINT mark;

INSERT INTO log VALUES ('b');
-- re-pointing 'mark' here: a later ROLLBACK TO mark
-- will only undo work done after this point
SAVEPOINT mark;

INSERT INTO log VALUES ('c');
ROLLBACK TO SAVEPOINT mark;   -- undoes only 'c'

COMMIT;

Compatibility

EvolutionDB's SAVEPOINT follows the idea of the SQL standard's <savepoint statement>. For the common case of SAVEPOINT name, it works the same as the PostgreSQL form on the wire. The related commands ROLLBACK TO [SAVEPOINT] name and RELEASE [SAVEPOINT] name also work. The SAVEPOINT keyword is optional in them. This matches PostgreSQL.

Here are the differences to know about.

Duplicate names overwrite
The SQL standard and PostgreSQL allow many savepoints with the same name. The newest one hides the older ones until it is released. EvolutionDB does it another way. It overwrites a same-named savepoint in place. So a transaction never has more than one savepoint per name.
No nested sub-transactions
A savepoint is a position in the undo log for the session. It is not a nested transaction with its own id. There is no separate sub-transaction view. There is no per-savepoint error isolation. The only thing a savepoint can do is undo the recorded changes.
Fixed savepoint limit
A transaction can hold at most 64 active savepoints. If you go past this limit, the command fails with SQLSTATE 53000.
Plain identifier names only
A name is read as one token with no quotes and no spaces. The match does not care about case. Quoted or delimited identifiers are not supported. Parenthesized option syntax is not supported either.
Statement handled by the adaptor
The protocol adaptor reads transaction control statements. This includes SAVEPOINT. The SQL grammar does not read them. They must stand on their own. You cannot put them inside other SQL.
