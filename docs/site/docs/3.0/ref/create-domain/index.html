CREATE DOMAIN

Make a named data type that you can use again. It is built from a base type. It can also have NOT NULL and CHECK rules.

Synopsis

CREATE DOMAIN domain_name AS data_type
    [ DEFAULT default_expr ]
    [ NOT NULL ]
    [ CHECK ( expression ) ]

Description

CREATE DOMAIN creates a domain. A domain is a named type. It joins a base data_type with rules that you choose. EvolutionDB saves the domain in its system catalog. You can then use the domain by its name.

The domain does not get its own catalog entity. EvolutionDB stores it in the shared constraints B+ tree. This is under the system namespace, where table_id = 0. The constraint type is 'D'. The catalog row keeps three fields in one definition string. The string has the form "type_val|not_null|check_serialized". The three fields are the numeric type code of the base type, a 0/1 flag for NOT NULL, and the serialized check expression. The check expression is only there when a CHECK clause is given. The domain name must be unique in that system namespace. If you try to create a domain with a name that is already used, it fails with SQLSTATE 42710. This means the object is a duplicate.

The base type can be any type from EvolutionDB's data_type grammar. Some examples are INTEGER, BIGINT, DECIMAL(p,s), VARCHAR(n), CHAR(n), TEXT, BOOLEAN, DATE, TIMESTAMP, UUID, JSON, ENUM(...), and array types written type[].

Honest implementation note. The parser accepts two parts of the syntax. But they do little in this release. You should not rely on them.

  • DEFAULT default_expr is parsed but not stored. The grammar accepts the clause. But the value is thrown away. The catalog row keeps only the base type, the NOT NULL flag, and the CHECK expression. No default is saved with the domain.
  • Domains are recorded, not yet enforced as column types. A domain is written to the catalog. But the engine does not yet resolve a domain name when it is used as a column type in CREATE TABLE. So the domain's NOT NULL and CHECK rules are not applied to columns through the domain name.

Parameters

domain_name
The name of the domain to create. It must be unique in the system namespace. If the name is already used, you get domain "name" already exists with SQLSTATE 42710.
data_type
The base type for the domain. You give it after the required AS keyword. You can use any type from EvolutionDB's data_type grammar. This includes types with parameters like VARCHAR(n) and DECIMAL(p,s). It also includes array types written type[]. The type is stored as its internal numeric code, which is type_family × 10000 + size.
DEFAULT default_expr
The grammar accepts this. But it is not saved in this release. The expression is parsed and then thrown away. So it does not become part of the stored domain definition.
NOT NULL
Marks the domain so it does not accept null values. When you use it, the not-null flag is saved in the domain's catalog definition. You can use it together with a CHECK clause.
CHECK ( expression )
A boolean rule expression for the domain. When you give it, the expression is serialized and stored next to the base type in the domain's catalog row.

Examples

A simple domain on an integer base type.

CREATE DOMAIN postal_code AS INTEGER;

A text domain that does not allow nulls.

CREATE DOMAIN email_addr AS VARCHAR(255) NOT NULL;

A domain with a value check.

CREATE DOMAIN percent AS INTEGER
    CHECK (value >= 0 AND value <= 100);

A non-null domain with a check together.

CREATE DOMAIN positive_amount AS DECIMAL(12,2) NOT NULL
    CHECK (value > 0);

-- The DEFAULT clause parses. But the value is not stored with the domain.

CREATE DOMAIN country_code AS CHAR(2) DEFAULT 'US';

Compatibility

CREATE DOMAIN is part of the SQL standard. It is also in PostgreSQL. EvolutionDB supports only a small part of it on purpose.

  • EvolutionDB accepts DEFAULT, one NOT NULL, and one unnamed CHECK (expression). The standard and PostgreSQL allow more. They allow named constraints written CONSTRAINT name CHECK .... They allow many constraints. They allow NULL to mark a domain as nullable. They allow COLLATE. None of these are part of EvolutionDB's grammar.
  • PostgreSQL works in a different way here. EvolutionDB does not need or accept a schema-qualified domain name in this command. The domain is registered in one system namespace.
  • The syntax accepts DEFAULT. But in this release the default value is not kept. PostgreSQL stores and uses the default.
  • In PostgreSQL a domain becomes a real column type. It enforces its rules on every column that uses it. In this EvolutionDB release the domain is saved in the catalog. But it is not yet linked as a column type you can reference. So its rules are not enforced through the domain name.

Some code depends on default propagation. Some code depends on domain-typed columns. Treat both as not yet available. Use explicit column-level NOT NULL and CHECK rules in CREATE TABLE instead.
