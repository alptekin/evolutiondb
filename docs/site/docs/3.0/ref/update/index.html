UPDATE

UPDATE changes column values in the rows of a table that match a condition.

Synopsis

UPDATE [ LOW_PRIORITY ] [ IGNORE ] table [ AS alias ] [ , table [ AS alias ] ]
    SET [ table. ]column = expression [ , [ table. ]column = expression ] ...
    [ WHERE condition ]
    [ ORDER BY column [ ASC | DESC ] [ , ... ] ]
    [ LIMIT count [ OFFSET start ] ]
    [ LIMIT start , count ]
    [ RETURNING * | output_column [ , ... ] ]

The assignment operator in the SET list must be =. The parser rejects any other comparison operator in that spot. It reports "bad update assignment".

Description

UPDATE changes the values of the named columns. It does this for every row of table that matches the WHERE condition. Only the columns named in the SET clause change. All other columns keep their current values. The server reports the number of changed rows back to the client.

EvolutionDB applies each update without destroying the old data. This is the MVCC model. The engine does not overwrite a row in place. Instead it stamps the old version with the current transaction's xmax. This marks the old version as soft-deleted. The engine then writes the new version to a fresh heap location with a fresh xmin. It then points the primary-key B+ tree entry at the new RowID. The old version stays on the page. So other readers that hold an older snapshot still see the row before the update. The change can also be rolled back. The engine takes an exclusive row lock on each row before it changes the row. If the lock cannot be granted, the statement fails with a lock or deadlock error.

There are two ways to run an update. The WHERE-expression form scans the table. It collects the matching rows. Then it applies the update to each one. For a simple primary-key range condition like WHERE id <= 5000, the engine walks only the matching B+ tree keys instead. The single-key form has no WHERE clause. It finds one row directly by its primary key. The statement becomes a multi-table update when the FROM-style table list names more than one table. It also becomes a multi-table update when a join is present.

EvolutionDB checks the new values before it writes a row. It checks them against the column's declared type. It also checks the NOT NULL, UNIQUE, CHECK, and FOREIGN KEY constraints. It re-evaluates any STORED generated columns. It enforces row-level security USING and WITH CHECK policies. It fires BEFORE UPDATE and AFTER UPDATE triggers. A SET on a generated column is rejected. The engine enforces the configured ON UPDATE referential action on referencing child rows when a primary-key column changes. The action can be RESTRICT, NO ACTION, CASCADE, SET NULL, or SET DEFAULT. For system-versioned tables, the engine copies the old row to the history table. It sets the old row's valid_to to the current time. It also refreshes the live row's validity window automatically.

Parameters

LOW_PRIORITY
An optional update modifier that the grammar accepts. It is a MySQL-compatibility keyword. The parser reads it and records it. It does not change how EvolutionDB schedules work.
IGNORE
An optional update modifier that the grammar accepts. It is a MySQL-compatibility keyword. The parser reads it and records it as an option flag.
table [ AS alias ]
The target table whose rows you want to update. You can give an optional alias. You can list more than one table reference, separated by commas. Join syntax is also accepted. The statement is processed as a multi-table update when it references more than one table.
column
The name of a column in the target table. You can write it in the qualified form table.column. The qualified form is what drives multi-table assignments. The column cannot be a generated column. If you assign to one, the engine raises an error. The error says the column can only be updated to DEFAULT.
expression
The value to assign to the column. You write it as a SQL expression after =. The new value must match the column's declared type. The engine checks the type before it writes the row.
WHERE condition
An expression that returns a boolean result. Only rows where it is true are updated. In the single-key form you leave it out, and the engine finds the row by primary key. When you give a WHERE expression, every row that matches the condition is updated. A simple primary-key range or equality condition uses a B+ tree cursor walk instead of a full table scan.
ORDER BY column [ ASC | DESC ]
The grammar accepts this MySQL-style ordered update. It orders the candidate rows. It is most useful together with LIMIT.
LIMIT count [ OFFSET start ] / LIMIT start, count
Limits how many of the matched rows are actually updated. The engine slices the matched set by the OFFSET and LIMIT range first. Then it applies the updates. Rows outside the range are left unchanged.
RETURNING * | output_column [.. ]
Makes UPDATE return the affected rows. RETURNING * returns all columns. A column list returns just those columns. The returned values show the new row after the update.

Examples

This example changes one column on one row found by its primary key.

UPDATE employees SET salary = 75000 WHERE id = 42;

This example updates several columns at once. One new value comes from an expression.

UPDATE products
   SET price = price * 1.10, updated_by = 'admin'
 WHERE category = 'books';

This example uses a range condition over the primary key. It takes the B+ tree cursor fast path.

UPDATE orders SET status = 'archived' WHERE id <= 5000;

This example updates only the first few matching rows in order and returns the new values.

UPDATE tickets
   SET priority = 'high'
 WHERE assignee IS NULL
 ORDER BY created_at ASC
 LIMIT 10
RETURNING id, priority;

This example sets a column to NULL.

UPDATE accounts SET closed_at = NULL WHERE id = 7;

Compatibility

The core form UPDATE table SET col = expr [ , ... ] WHERE condition follows the SQL standard. It matches PostgreSQL closely. The optional RETURNING clause is a PostgreSQL extension. EvolutionDB supports it. It returns the column values after the update.

A few clauses are MySQL-style extensions. They are not part of the SQL standard or PostgreSQL. These are the LOW_PRIORITY and IGNORE modifiers before the table name. They also include the ORDER BY and LIMIT clauses on UPDATE. The LIMIT start, count two-argument form is MySQL syntax. The LIMIT count OFFSET start form follows PostgreSQL.

EvolutionDB's UPDATE differs from PostgreSQL in some ways. It does not implement PostgreSQL's UPDATE ... FROM from_list clause for joins. Instead, you write multi-table updates by listing tables and joins directly after the UPDATE keyword in a FROM-style table reference. You then use qualified SET table.column = ... assignments. This is the MySQL multi-table style. EvolutionDB also does not support WHERE CURRENT OF cursor targeting. It does not support table aliases used only for a FROM join. It does not support the standard SET (col1, col2) = (...) multi-column row-assignment form. You write assignments one column at a time. The assignment operator must be a plain =.
