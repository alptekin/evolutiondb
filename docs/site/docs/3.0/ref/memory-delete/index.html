MEMORY DELETE

Remove one key/value entry from a memory store. The entry is found by its namespace and key.

Synopsis

MEMORY DELETE FROM store_name
    WHERE NS = 'namespace' AND KEY = 'key'

Description

MEMORY DELETE removes exactly one entry from the memory store named store_name. Each entry has a unique identity. That identity is the pair of namespace and key. MEMORY DELETE targets the one row whose composite primary key matches the namespace and key you give.

Behind the scenes, a memory store is held in an ordinary table named mem_<store_name>. The two parts of the entry identity are joined into one composite primary-key value. The namespace and key are joined by a record-separator byte. MEMORY DELETE rewrites the request as DELETE FROM mem_<store_name> WHERE pk = '...'. It then sends it through the engine's standard delete path. So the operation works just like a hand-written DELETE. Row-level security policies on the backing table still apply. These are also called RLS policies. Foreign-key cascade actions still apply. MVCC visibility and triggers still apply too. This RLS pass-through makes MEMORY DELETE safe in multi-tenant deployments. A caller can only delete rows that its policy allows.

If the named store does not exist, the command fails with SQLSTATE 42704. This means the object is undefined. No data is changed. If the store exists but no entry matches the given namespace and key, the command still succeeds. It removes zero rows.

Both predicates are required. The grammar accepts only the fixed form WHERE NS = '...' AND KEY = '...'. The comparison operator must be =. The two predicates must be joined by AND. Both the namespace and key values are string literals. You cannot delete a whole namespace with MEMORY DELETE. You cannot match a key prefix. You cannot leave out either predicate.

Parameters

store_name
The name of the target memory store. You write it after FROM. The store must already exist. See CREATE MEMORY STORE. If it does not exist, the command raises SQLSTATE 42704.
NS = 'namespace'
A string literal. It gives the namespace part of the entry to delete. Together with the key, it forms the unique identity of the entry. An empty string '' is a valid namespace value.
KEY = 'key'
A string literal. It gives the key part of the entry to delete, inside the given namespace.

Examples

Delete one entry by namespace and key.

MEMORY DELETE FROM chat_memory
    WHERE NS = 'user:42' AND KEY = 'last_topic';

Delete an entry stored under the empty namespace.

MEMORY DELETE FROM kv_store
    WHERE NS = '' AND KEY = 'session_token';

Here is a common write-then-remove flow for one entry.

-- store an entry
MEMORY PUT INTO chat_memory
    VALUES ('user:42', 'last_topic', '{"text":"billing"}');

-- read it back
MEMORY GET FROM chat_memory WHERE NS = 'user:42' AND KEY = 'last_topic';

-- remove it
MEMORY DELETE FROM chat_memory WHERE NS = 'user:42' AND KEY = 'last_topic';

Compatibility

MEMORY DELETE is an EvolutionDB extension. It is not part of the SQL standard. There is no matching statement in PostgreSQL. It is a simpler way to write the standard DELETE statement. The request is turned into DELETE FROM mem_<store_name> WHERE pk = '...'. It runs through the ordinary delete engine. So it keeps the standard delete behavior. This includes RLS, foreign-key actions, MVCC, and triggers.

The MEMORY DELETE grammar is limited on purpose. A general-purpose DELETE is more flexible. MEMORY DELETE accepts only equality on both NS and KEY joined by AND. It supports no other WHERE shape. It supports no ORDER BY, LIMIT, or RETURNING clause. To delete by any other rule, work on the backing table with a standard DELETE statement.
