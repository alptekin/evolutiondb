CREATE JOB

CREATE JOB adds a scheduled job. The job runs a SQL statement on a cron-style timetable.

Synopsis

CREATE JOB [ IF EXISTS ] name
    ON SCHEDULE 'min hour dom mon dow'
    DO 'sql_statement'

Description

CREATE JOB adds a named scheduled job to the system catalog. A job joins a five-field cron expression with one SQL statement. The scheduler checks the current minute on the clock. When that minute matches the cron pattern, the job is marked as due.

You give both the cron expression and the SQL body as quoted string literals. Whitespace and special characters in them follow the lexer's normal quoting rules. They do not follow the SQL grammar. The outer quotes are removed before the values are stored in the catalog.

New jobs are enabled by default. Their last-run timestamp is null. You can turn jobs on and off with ALTER JOB ... ENABLE and ALTER JOB ... DISABLE. You can remove a job with DROP JOB. You can list jobs with SHOW JOBS. That command shows each job's name, cron expression, SQL body, enabled flag, and last_run_unix timestamp.

Execution status. In this release the scheduler does cron matching. It also handles persistence, the ENABLE and DISABLE state, and SHOW JOBS introspection. When a job's cron pattern matches the current minute, the scheduler updates the job's last_run_unix field. You can then check that a tick fired with SHOW JOBS. It also logs a line to stderr. Running the job's SQL body is not done yet. That part comes in a future release. Do not rely on the SQL in DO being run yet.

Parameters

IF EXISTS
Do not raise an error if a job with the same name is already registered. When the named job exists, the statement succeeds without a message. The existing job stays the same. The cron expression and SQL body in the new statement are ignored. Note the keyword spelling. EvolutionDB's grammar accepts the bare IF EXISTS form here. It does not use PostgreSQL's IF NOT EXISTS form.
name
The identifier of the job to create. It must be unique among scheduled jobs. Without IF EXISTS, creating a job whose name already exists raises an error. The SQLSTATE is 42710.
ON SCHEDULE 'min hour dom mon dow'
A five-field cron expression given as a quoted string. The fields are minute, hour, day-of-month, month, and day-of-week. Spaces or tabs separate the fields. Each field is either * or a single non-negative integer. The * matches any value. Ranges like 1-5, comma lists like 1,15,30, and step values like */5 are not supported.

Each field has a valid integer range. Minute is 0–59. Hour is 0–23. Day-of-month is 1–31. Month is 1–12. Day-of-week is 0–7, where both 0 and 7 mean Sunday. Matching is done in UTC. An expression that does not parse to exactly five valid fields raises an error. The SQLSTATE is 22023.
DO 'sql_statement'
The SQL statement to link with the job. You give it as a quoted string literal. It is stored in the catalog exactly as written. The DO body must not be empty.

Examples

This example creates a job that runs every day at 03:00 UTC.

CREATE JOB nightly_rollup
    ON SCHEDULE '0 3 * * *'
    DO 'INSERT INTO daily_totals SELECT today(), sum(amount) FROM orders';

This example creates a job that fires at the top of every hour.

CREATE JOB hourly_ping
    ON SCHEDULE '0 * * * *'
    DO 'UPDATE heartbeat SET ticked_at = now()';

This example creates a job only if no job with that name already exists.

CREATE JOB IF EXISTS weekly_cleanup
    ON SCHEDULE '30 4 * * 0'   -- 04:30 every Sunday (dow 0)
    DO 'DELETE FROM sessions WHERE expired = TRUE';

This example shows how to inspect, disable, re-enable, and remove a job.

SHOW JOBS;
ALTER JOB hourly_ping DISABLE;
ALTER JOB hourly_ping ENABLE;
DROP JOB [ IF EXISTS ] hourly_ping;

Compatibility

CREATE JOB is not part of the SQL standard. The SQL standard does not define scheduled jobs.

It is also not a PostgreSQL command. Core PostgreSQL has no built-in scheduler. The closest thing in PostgreSQL is the third-party pg_cron extension. Its cron.schedule('name', '0 3 * * *', 'SQL') function call uses the same five-field cron style. EvolutionDB offers this as native DDL instead of a function. Its cron evaluator is kept simple on purpose. These are the rules.

  • Exactly five fields. Each field is either * or a single integer. No ranges, comma lists, or step values. This is unlike pg_cron and standard crontab.
  • Day-of-week accepts 0 and 7 for Sunday. This follows the POSIX cron style.
  • Matching is done in UTC.
  • The IF EXISTS spelling here is different from PostgreSQL's usual IF NOT EXISTS guard on CREATE statements.
  • In this release the job's SQL body is matched and timestamped but not yet run. See the note under Description.
