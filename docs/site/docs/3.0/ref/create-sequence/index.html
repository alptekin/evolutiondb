CREATE SEQUENCE

Create a new sequence generator. A sequence is a named counter that is saved on disk. It gives you a series of BIGINT values that always go in one direction. You read these values with the nextval(), currval(), setval() and lastval() functions.

Synopsis

CREATE SEQUENCE [ IF NOT EXISTS ] name
    [ INCREMENT [ BY ] increment ]
    [ MINVALUE minvalue ]
    [ MAXVALUE maxvalue ]
    [ START [ WITH ] start ]
    [ CYCLE | NOT CYCLE ]

Description

CREATE SEQUENCE creates a new sequence-number generator. The sequence is saved as an object in the system catalog of the current schema. It stays on disk after a server restart.

Once the sequence exists, you work with it through the sequence functions. nextval(name) moves the sequence forward and returns the new value. The first call to nextval() on a brand new sequence returns the START value. At that point the sequence has not been used yet. Each later call adds increment to the current value. currval() returns the value you got most recently. lastval() returns the value of the sequence you used most recently in this session. setval() sets the current value to a number you choose.

If the given name already belongs to a sequence in the current schema, you get an error. To avoid this, use IF NOT EXISTS. Then you get a notice and no new sequence is created. Keep in mind that IF NOT EXISTS only checks if a sequence with that name exists. It does not check if the old definition matches the new one you asked for.

Each call to nextval() runs one at a time under an internal mutex. The new value is saved to the catalog right away. Because of this, two sessions never get the same value. The new value is saved before it is returned. So sequence values used inside a transaction are not rolled back if that transaction fails. This means gaps in the series are normal and expected.

When increment is positive, the sequence counts up. When it is negative, the sequence counts down. An ascending sequence may reach maxvalue. A descending sequence may reach minvalue. A sequence with CYCLE then wraps around. An ascending one goes back to minvalue. A descending one goes back to maxvalue. A sequence without cycling raises an error instead.

Parameters

IF NOT EXISTS
Do not raise an error if a sequence with the same name already exists in the current schema. You get a notice instead. The old sequence is left unchanged.
name
The name of the sequence to create. The sequence is created in the current schema. You cannot add a schema name to it in this form.
INCREMENT [ BY ] increment
The value added to the current sequence value on each call to nextval(). A positive value makes the sequence count up. A negative value makes it count down. The BY keyword is optional. The default is 1.
MINVALUE minvalue
The smallest value the sequence can produce. The default is 1. For a descending sequence, this is the wrap target when CYCLE is set. For a descending sequence without cycling, this is the floor where it raises an error.
MAXVALUE maxvalue
The largest value the sequence can produce. The default is 2147483647. This is 231−1. For an ascending sequence, this is the wrap target when CYCLE is set. For an ascending sequence without cycling, this is the ceiling where it raises an error.
START [ WITH ] start
The first value the sequence returns. This is the value you get from the first nextval() call. The WITH keyword is optional. If you leave it out, the start value is minvalue.
CYCLE / NOT CYCLE
CYCLE lets the sequence wrap around when it reaches a limit. An ascending sequence wraps at maxvalue. A descending sequence wraps at minvalue. NOT CYCLE is the default. With it, nextval() raises an error once a limit is reached.

Note. The clause order in the Synopsis is just a convention. It is not required. The grammar accepts the options in any order. If you give the same option twice, the later one wins. Sequence values are 64-bit signed integers. This is the BIGINT type. But the default MAXVALUE is 2147483647. Set MAXVALUE yourself if you need the full 64-bit range.

Examples

Create an ascending sequence that starts at 1 and goes up by 1. This uses all the defaults.

CREATE SEQUENCE order_id_seq;

This example reads values from the sequence.

SELECT nextval('order_id_seq');   -- 1  (the START value)
SELECT nextval('order_id_seq');   -- 2
SELECT currval('order_id_seq');   -- 2

Create a sequence that starts at 1000 and counts up by 10.

CREATE SEQUENCE invoice_seq
    START WITH 1000
    INCREMENT BY 10;

Create a sequence that has limits and cycles. It goes from 1 to 5, then back to 1.

CREATE SEQUENCE day_slot_seq
    MINVALUE 1
    MAXVALUE 5
    CYCLE;

Create a sequence that counts down.

CREATE SEQUENCE countdown_seq
    START 100
    INCREMENT -1
    MINVALUE 0;

Create the sequence only if it does not exist yet. If it already exists, skip it.

CREATE SEQUENCE IF NOT EXISTS session_seq;

Move a sequence to a new spot with setval(). After this, the next nextval() call returns 200.

SELECT setval('invoice_seq', 200);

Compatibility

CREATE SEQUENCE follows the spirit of the SQL standard. EvolutionDB has a few differences from it. They are listed below.

  • EvolutionDB supports the clauses INCREMENT [BY], MINVALUE, MAXVALUE, START [WITH] and CYCLE / NOT CYCLE. It also adds the extra clause IF NOT EXISTS.
  • The standard uses the keyword NO CYCLE. In EvolutionDB you write NOT CYCLE instead. The forms NO MINVALUE and NO MAXVALUE are not supported. Leave the clause out to use the default.
  • The grammar does not accept the PostgreSQL clauses AS data_type, CACHE, OWNED BY and TEMPORARY/UNLOGGED. Sequences are always saved BIGINT generators. Every nextval() writes straight to the catalog. There is no client-side cache.
  • The standard treats nextval/currval as expressions. Like PostgreSQL, EvolutionDB makes them functions. You pass the sequence name as a string. For example, nextval('seq').
  • The default MAXVALUE is 2147483647, not the 64-bit maximum. Set it yourself when you need a higher ceiling.
