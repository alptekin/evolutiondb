CREATE TABLE

Make a new base table in the current schema. You set its columns, their types, the column attributes, and the table constraints. You can also fill a new table from a query result.

Synopsis

CREATE [ TEMPORARY | GLOBAL TEMPORARY ] TABLE [ IF NOT EXISTS ] [ schema . ]table_name (
    { column_name data_type [ column_attribute [ ... ] ]
    | table_constraint }
    [ , ... ]
)
[ table_option [ ... ] ]
[ INHERITS ( parent_table ) ]

-- Create a table from a query result (CTAS):
CREATE [ TEMPORARY | GLOBAL TEMPORARY ] TABLE [ IF NOT EXISTS ] [ schema . ]table_name
    [ ( column_name [ , ... ] ) ]
    [ IGNORE | REPLACE ] [ AS ] select_statement

-- Create a range partition child bound to an existing parent:
CREATE [ TEMPORARY | GLOBAL TEMPORARY ] TABLE [ IF NOT EXISTS ] table_name
    PARTITION OF parent FOR VALUES FROM ( lo ) TO ( hi )

where column_attribute is one of:
    NOT NULL | NULL
    DEFAULT value
    DEFAULT { gen_random_uuid() | gen_random_uuid_v7() | snowflake_id() | CURRENT_TIMESTAMP | ( expr ) }
    AUTO_INCREMENT [ ( start [ , increment ] ) ]
    IDENTITY [ ( start [ , increment ] ) ]
    PRIMARY KEY | KEY
    UNIQUE [ KEY ]
    CHECK ( expr )
    COMMENT 'string'
    CONSTRAINT name { UNIQUE | PRIMARY KEY | CHECK ( expr ) }
    GENERATED ALWAYS AS ( expr ) [ STORED | VIRTUAL ]
    AS ( expr ) [ STORED | VIRTUAL ]

and table_constraint is one of:
    [ CONSTRAINT name ] PRIMARY KEY ( column [ , ... ] )
    [ CONSTRAINT name ] UNIQUE ( column [ , ... ] )
    [ CONSTRAINT name ] CHECK ( expr )
    [ CONSTRAINT name ] FOREIGN KEY ( column [ , ... ] )
        REFERENCES [ schema . ]ref_table ( ref_column [ , ... ] ) [ fk_action [ ... ] ]
    KEY ( column [ , ... ] ) | INDEX ( column [ , ... ] )
    FULLTEXT { INDEX | KEY } ( column [ , ... ] )

and fk_action is one of:
    ON DELETE { CASCADE | SET NULL | RESTRICT | SET DEFAULT | NO ACTION }
    ON UPDATE { CASCADE | SET NULL | RESTRICT | SET DEFAULT | NO ACTION }
    MATCH { FULL | SIMPLE | PARTIAL }
    [ NOT ] DEFERRABLE [ INITIALLY { DEFERRED | IMMEDIATE } ]

and table_option is one of:
    AUTO_INCREMENT [ = ] n
    ON COMMIT { DELETE ROWS | PRESERVE ROWS }
    SHARD BY HASH ( column ) SHARDS n
    SHARD BY RANGE ( column ) ( shard_def [ , ... ] )
    PARTITION BY RANGE ( column )
    WITH SYSTEM VERSIONING
    WITH ( ttl_column = 'column_name' )

Description

CREATE TABLE makes a new base table in the current schema of the current database. The table starts empty. The user who runs the statement owns the table. Each column gets a name, a data type, and an optional list of column attributes. Table constraints and table options can come after the column list.

EvolutionDB finds the current database and schema before it makes the table. You can write the table name as schema.table_name. Then the first name picks the target schema. If you do not write a schema, EvolutionDB uses the current schema of the session. See SET SCHEMA. The statement fails if the target schema does not exist.

Columns and storage use EvolutionDB's binary tuple format. Every record starts with a sentinel byte and the table id. After that comes a header, a null bitmap, and the column data. A type code holds both a type family and a size. For example, VARCHAR(50) is encoded as 130050. A table can have up to 256 columns.

The statement fails if a table with the same name already exists in the target schema. You get a "table already exists" error with SQLSTATE 42P07. To avoid this, add IF NOT EXISTS. Then EvolutionDB leaves the existing table as it is and the statement succeeds with no error.

Create from a query (CTAS). You can put a select_statement after the table name. Then the new table is filled from the query result. If you give a column list, those names are used. If you do not, EvolutionDB takes the column names and types from the SELECT output. IGNORE and REPLACE control what happens to rows that break a uniqueness rule during the bulk load.

Inheritance. INHERITS ( parent ) adds the parent table's columns at the front of the child's own columns. The parent must already exist. A child column cannot reuse the name of an inherited column.

Partition child. The PARTITION OF parent FOR VALUES FROM (lo) TO (hi) form makes a range-partition child. It is bound to an existing range-partitioned parent. It covers the half-open integer range [lo, hi).

Parameters

TEMPORARY / GLOBAL TEMPORARY
Marks the table as temporary. TEMPORARY makes a temporary table for the current session only. GLOBAL TEMPORARY makes a global temporary table. Use the ON COMMIT table option to control how long rows live when a transaction commits.
IF NOT EXISTS
Turns off the "table already exists" error. If a table with the same name already exists in the target schema, no new table is made and no error is raised. The grammar uses the IF NOT EXISTS clause for this. It does not check that the columns match the existing table.
schema.table_name / table_name
The name of the table to create. You can add a schema in front of it. A name with no schema is created in the current schema of the session.
column_name
The name of a column in the new table.
data_type
The data type of the column. The integer family has BIT, TINYINT, SMALLINT, MEDIUMINT, INT/INTEGER, and BIGINT. Each one can take an optional UNSIGNED/ZEROFILL modifier. The numeric types are REAL, DOUBLE, FLOAT, and DECIMAL(p,s). The date and time types are DATE, TIME, TIMESTAMP, DATETIME, and YEAR. The character types are CHAR(n), VARCHAR(n), TINYTEXT, TEXT, MEDIUMTEXT, and LONGTEXT. The binary types are BINARY(n), VARBINARY(n), TINYBLOB, BLOB, MEDIUMBLOB, and LONGBLOB. There are also ENUM(...), SET(...), BOOLEAN, UUID, JSON, VECTOR(dim), and array types written as type[]. Character and text types can take an optional CHAR SET 'name' / COLLATE 'name' suffix.
NOT NULL / NULL
Says if the column can hold NULL. NOT NULL blocks NULL values. NULL allows them. NULL is the default.
DEFAULT value
Gives a default value. EvolutionDB uses it when an INSERT leaves out the column. The value can be a string, an integer, a float, or a boolean literal. You can also use the function defaults gen_random_uuid(), gen_random_uuid_v7(), and snowflake_id(). The special value CURRENT_TIMESTAMP works too. You can also write an expression default as DEFAULT ( expr ).
AUTO_INCREMENT [ ( start [, increment ] ) ] / IDENTITY [ ( start [, increment ] ) ]
Gives the column increasing integer values on its own. With no arguments, the values start at 1 and go up by 1. You can set your own start value and an optional increment. IDENTITY is another name for the same feature.
PRIMARY KEY / KEY (column attribute)
Marks the column as the primary key of the table. When you use KEY as a column attribute, EvolutionDB treats it as PRIMARY KEY.
UNIQUE [ KEY ] (column attribute)
Makes the column hold a different value in each row.
CHECK ( expr )
Sets a check that each row must pass. You can write it on a single column or as a table constraint.
COMMENT 'string'
Adds a short note that describes the column.
GENERATED ALWAYS AS ( expr ) [ STORED | VIRTUAL ] / AS ( expr ) [ STORED | VIRTUAL ]
Defines a generated column. EvolutionDB works out its value from expr. STORED saves the computed value. VIRTUAL computes it each time you read the row. VIRTUAL is the default when you give neither keyword.
CONSTRAINT name..
Gives a name to a UNIQUE, PRIMARY KEY, CHECK, or FOREIGN KEY constraint. This works for a column constraint or a table constraint. A CHECK constraint with no name gets the auto name table_check_n.
PRIMARY KEY ( column [.. ] ) (table constraint)
Sets one or more columns as the primary key. It can use more than one column.
UNIQUE ( column [.. ] ) (table constraint)
Sets a uniqueness rule. It can cover one column or several columns.
FOREIGN KEY ( column [.. ] ) REFERENCES [ schema. ]ref_table ( ref_column [.. ] ) [ fk_action.. ]
Sets a foreign key. It links columns in this table to columns in ref_table. The other table can be in a different schema. You set the referential actions with the fk_action clauses below.
fk_action, ON DELETE / ON UPDATE { CASCADE | SET NULL | RESTRICT | SET DEFAULT | NO ACTION }
Picks what happens to the linked rows when the referenced row is deleted or updated.
fk_action, MATCH { FULL | SIMPLE | PARTIAL }
Sets how matching works for foreign keys that use more than one column.
fk_action, [ NOT ] DEFERRABLE [ INITIALLY { DEFERRED | IMMEDIATE } ]
Says if the constraint check can wait until the transaction commits. It also sets the first check timing.
KEY / INDEX ( column [.. ] )
Asks for an index on the listed columns inside the table definition. Here INDEX and KEY mean the same thing.
FULLTEXT { INDEX | KEY } ( column [.. ] )
Asks for a full-text index on the listed columns.
AUTO_INCREMENT [ = ] n (table option)
Sets the start value for the auto-increment values of the table.
ON COMMIT { DELETE ROWS | PRESERVE ROWS } (table option)
For temporary tables, this picks if rows are deleted or kept when the transaction commits.
SHARD BY HASH ( column ) SHARDS n / SHARD BY RANGE ( column ) (.. )
Spreads the table across n hash shards based on column. You can also spread it across set value ranges. You write each range shard as SHARD name VALUES LESS THAN { 'value' | MAXVALUE } ON NODE id.
PARTITION BY RANGE ( column ) (table option)
Sets the table as range-partitioned on column. You add child partitions with the PARTITION OF form.
WITH SYSTEM VERSIONING (table option)
Turns the table into a system-versioned table. EvolutionDB adds validity-period columns for you. It also links a table_history shadow table. That shadow table keeps old row versions.
WITH ( ttl_column = 'column_name' ) (table option)
Marks a TTL column. The auto-reclaim daemon removes rows when the value in this column is older than the current time. The named column must exist in the table.
INHERITS ( parent_table )
Takes the parent table's columns. EvolutionDB puts them at the front of the child's own columns. The parent must exist. Duplicate column names are not allowed.
PARTITION OF parent FOR VALUES FROM ( lo ) TO ( hi )
Makes a range-partition child of an existing partitioned parent. It covers the half-open integer range from lo up to hi. The range includes lo but not hi.
[ IGNORE | REPLACE ] [ AS ] select_statement
Fills the new table from the result of select_statement. This is CREATE TABLE AS. IGNORE skips rows that would conflict. REPLACE replaces them. The AS keyword is optional.

Examples

Make a simple table with a primary key, defaults, and a check.

CREATE TABLE users (
    id        INT PRIMARY KEY AUTO_INCREMENT,
    email     VARCHAR(255) NOT NULL UNIQUE,
    age       INT CHECK (age >= 0),
    status    VARCHAR(20) DEFAULT 'active',
    created   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Create only if the table is not there yet, and use a UUID default.

CREATE TABLE IF NOT EXISTS sessions (
    id    UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    token TEXT NOT NULL
);

This example uses a primary key on two columns and a named foreign key with referential actions.

CREATE TABLE order_items (
    order_id  INT,
    product_id INT,
    qty       INT NOT NULL DEFAULT 1,
    PRIMARY KEY (order_id, product_id),
    CONSTRAINT fk_order FOREIGN KEY (order_id)
        REFERENCES orders (id) ON DELETE CASCADE
);

This example adds a generated column and a vector column.

CREATE TABLE docs (
    id       BIGINT IDENTITY,
    title    VARCHAR(200),
    title_len INT GENERATED ALWAYS AS (length(title)) STORED,
    embedding VECTOR(768)
);

Make a table from a query result, and make a system-versioned table.

CREATE TABLE active_users AS
    SELECT id, email FROM users WHERE status = 'active';

CREATE TABLE accounts (
    id      INT PRIMARY KEY,
    balance DECIMAL(12,2)
) WITH SYSTEM VERSIONING;

Compatibility

The CREATE TABLE command in EvolutionDB mixes ideas from the SQL standard, PostgreSQL, and MySQL. It does not match any one of them exactly.

  • From the SQL standard and PostgreSQL. EvolutionDB supports column definitions, PRIMARY KEY, UNIQUE, CHECK, and FOREIGN KEY ... REFERENCES with ON DELETE/ON UPDATE actions and MATCH/DEFERRABLE clauses. It also supports GENERATED ALWAYS AS, INHERITS, IF NOT EXISTS, and CREATE TABLE AS.
  • From MySQL. These parts are MySQL-style. They are AUTO_INCREMENT, the KEY/INDEX and FULLTEXT declarations inside the column list, the COMMENT column attribute, and the UNSIGNED/ZEROFILL integer modifiers. The MySQL type names TINYINT, MEDIUMTEXT, LONGBLOB, ENUM, SET, DATETIME, and YEAR are also MySQL-style. So is the AS ( expr ) STORED|VIRTUAL short form for generated columns, and the table options written at the end.
  • EvolutionDB extras. These parts belong to EvolutionDB only. They are VECTOR(dim) columns, the gen_random_uuid_v7() and snowflake_id() default functions, WITH SYSTEM VERSIONING, and WITH (ttl_column = ...). The hash and range sharding with SHARD BY is also EvolutionDB only. So is the integer-range PARTITION OF ... FOR VALUES child form.
  • Differences and clauses that are not supported. Unlike PostgreSQL, there is no UNLOGGED, no OF type, no EXCLUDE constraints, and no tablespace clause. There is also no per-column COLLATE with an identifier. You write a quoted string after COLLATE instead. The grammar accepts the NULL column attribute. It does nothing on its own and only states that the column allows NULL.
