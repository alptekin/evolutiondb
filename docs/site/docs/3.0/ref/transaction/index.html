BEGIN / COMMIT / ROLLBACK

Open, commit, or undo a transaction with many statements. A group of changes either all take effect together or none of them do.

Synopsis

BEGIN
START TRANSACTION

COMMIT
END

ROLLBACK

SAVEPOINT savepoint_name
ROLLBACK TO [ SAVEPOINT ] savepoint_name
RELEASE [ SAVEPOINT ] savepoint_name

Description

In EvolutionDB each statement runs in its own transaction. This transaction is implicit and commits on its own. This happens unless you first open an explicit transaction block. BEGIN starts such a block. START TRANSACTION is a synonym for it. After that, the server records every data-modifying statement in a per-connection undo log. The block stays open until you finish it with COMMIT or ROLLBACK.

COMMIT saves all the changes made since BEGIN. The changes become durable. Other connections can now see them. Before the server confirms the commit, it does a few things. It flushes the transaction's dirty pages to the write-ahead log. It assigns a commit sequence number. It marks the transaction committed in the commit log. This commit log is called the CLOG. The server then discards the undo log. It also releases any locks the transaction held. END is an exact synonym for COMMIT.

ROLLBACK throws away the transaction. The server replays the undo log in reverse. This restores every inserted, updated, or deleted row to its earlier state. The server marks the transaction aborted in the CLOG. It then releases its locks. Under MVCC, a rollback of an UPDATE or DELETE does not rewrite data. Instead it clears the soft-delete marker xmax on the original tuple. Pre-MVCC rows use a re-insert fallback.

Each connection has its own transaction context. The undo log is thread-local. So transactions on different connections are independent. A connection may disconnect with a transaction still open. In that case the server rolls it back on its own.

Note. A statement inside the block may raise an error. Then the transaction enters an aborted state. While aborted, the server accepts only ROLLBACK, COMMIT, SAVEPOINT, and RELEASE. It rejects other statements until the block is closed. A COMMIT on an aborted transaction performs an implicit ROLLBACK. All work is discarded. This matches PostgreSQL.

Isolation level

You do not set the transaction isolation level on the BEGIN statement itself. EvolutionDB matches BEGIN and START TRANSACTION by prefix. So it accepts any words you type after the keyword, but it ignores them. To choose an isolation level, run SET [SESSION] TRANSACTION ISOLATION LEVEL first. Run it before you open the transaction. You can also run it before any query inside the transaction. The recognized levels are READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, and SERIALIZABLE. READ COMMITTED is the default.

At READ COMMITTED each statement takes a fresh snapshot. At REPEATABLE READ and SERIALIZABLE, BEGIN takes one snapshot. The whole transaction sees that same snapshot. A SERIALIZABLE transaction also registers with the conflict guard. This checks for write-skew. Such a transaction may fail at COMMIT with SQLSTATE 40001. The message is could not serialize access due to read/write dependencies among transactions. Then the server rolls back the transaction. The command reports ROLLBACK.

Savepoints

Inside an open transaction you may set named savepoints. SAVEPOINT name marks the current position in the undo log. ROLLBACK TO [SAVEPOINT] name replays the undo log back to that mark. It undoes everything done since the savepoint. It keeps earlier work and the transaction itself. It also clears the aborted state, so the transaction can continue. RELEASE [SAVEPOINT] name forgets the savepoint and all newer ones. It does not undo any work. If you define a savepoint with the name of an existing one, it overwrites the old one. Savepoint commands outside a transaction fail with SQLSTATE 25P01. A reference to a savepoint that does not exist fails with 3B001.

Parameters

BEGIN
Start an explicit transaction block on the current connection. It sets up a fresh undo log and savepoint stack. It also gives the transaction an MVCC transaction id. The command reports the tag BEGIN. A BEGIN while you are already inside a transaction logs a warning. Otherwise it does nothing. It does not start a nested transaction.
START TRANSACTION
An exact synonym for BEGIN. It behaves the same way.
COMMIT
End the current transaction and make all of its changes permanent. It flushes WAL. It records the commit in the CLOG with a commit sequence number. It releases locks. It discards the undo log. It delivers any pending notifications. It reports the tag COMMIT. Run outside a transaction, it does nothing and causes no harm.
END
An exact synonym for COMMIT.
ROLLBACK
End the current transaction and undo all changes made since BEGIN. It replays the undo log in reverse. It marks the transaction aborted. It releases locks. It reports the tag ROLLBACK. Run outside a transaction, it does nothing and causes no harm.
SAVEPOINT savepoint_name
Set a new savepoint at the current point inside the open transaction. Names are case-insensitive. If you reuse an existing name, it overwrites that savepoint.
ROLLBACK TO [ SAVEPOINT ] savepoint_name
Undo all changes made after the named savepoint was set. This does not end the transaction. The SAVEPOINT keyword is optional. The server discards any savepoints created after the target. It also clears the transaction's aborted flag if one is set.
RELEASE [ SAVEPOINT ] savepoint_name
Remove the named savepoint and all savepoints created after it. The work done since then is kept. The SAVEPOINT keyword is optional.

Examples

This example moves money between two accounts as one atomic step.

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

This example discards a set of changes instead of keeping them.

BEGIN;
DELETE FROM staging_rows;
-- changed your mind:
ROLLBACK;

This example uses a savepoint to undo only part of a transaction.

BEGIN;
INSERT INTO orders (id, total) VALUES (1, 50);
SAVEPOINT before_items;
INSERT INTO order_items (order_id, sku) VALUES (1, 'BAD-SKU');
ROLLBACK TO SAVEPOINT before_items;   -- keep the order, drop the bad item
COMMIT;

This example sets an isolation level before opening the transaction.

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
SELECT count(*) FROM inventory;
-- the same snapshot is seen for the whole transaction
SELECT sum(qty) FROM inventory;
COMMIT;

This example shows that END is interchangeable with COMMIT.

BEGIN;
UPDATE settings SET value = 'on' WHERE key = 'feature_x';
END;

Compatibility

EvolutionDB's transaction control follows the spirit of the SQL standard. It works well with PostgreSQL. It makes some changes to keep things simple on purpose.

  • The server accepts both BEGIN and the SQL-standard START TRANSACTION. They behave the same way. END works as a synonym for COMMIT. This is a PostgreSQL extension. The standard only defines COMMIT.
  • PostgreSQL allows optional noise words on these statements. These include WORK, TRANSACTION, and AND CHAIN. EvolutionDB does not parse them as separate clauses. It matches the leading keyword and ignores any text after it. So COMMIT WORK behaves the same as COMMIT. The extra words have no effect. AND CHAIN does not start a new transaction.
  • PostgreSQL lets you set transaction options on the BEGIN line. These include ISOLATION LEVEL ..., READ ONLY, READ WRITE, and DEFERRABLE. EvolutionDB does not read these from the BEGIN statement. Set the isolation level first with SET [SESSION] TRANSACTION ISOLATION LEVEL. There is no read-only or deferrable transaction mode.
  • A nested BEGIN does nothing. It emits a warning instead of an error. This matches PostgreSQL. A COMMIT or ROLLBACK outside a transaction does nothing and causes no harm.
  • SAVEPOINT, ROLLBACK TO [SAVEPOINT], and RELEASE [SAVEPOINT] work with PostgreSQL-compatible behavior. A reused savepoint name overwrites the old one. A ROLLBACK TO clears the aborted state.
  • A failed statement aborts the whole transaction. The server refuses later statements until you run ROLLBACK. A ROLLBACK TO a prior savepoint also works. This matches PostgreSQL's failed-transaction behavior. A COMMIT on an aborted transaction performs an implicit rollback.
  • Two-phase commit is available through the separate XA command family. This family is XA START/END/PREPARE/COMMIT/ROLLBACK/RECOVER. There is no PREPARE TRANSACTION statement.
