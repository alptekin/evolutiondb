B+ Tree

EvoSQL uses a page-based B+ tree called btree2 for all O(log n) key-value indexing. Every primary key index, secondary index, and system catalog tree is a B+ tree instance. Keys are variable-length strings up to 255 bytes. Values are RowID structs that point to heap records.

Overview

EvoSQL uses a page-based B+ tree called btree2 for all O(log n) key-value indexing. Every primary key index, secondary index, and system catalog tree is a B+ tree instance. Keys are variable-length strings up to 255 bytes. Values are RowID structs that point to heap records. All tree nodes are stored as 4KB pages managed by the Page Manager. All I/O passes through the Clock Sweep buffer pool.

The tree has these key properties.

  • Key format. Length-prefixed string. It uses 1 byte for length plus up to 255 bytes of data. The limit is BT2_MAX_KEY_LEN = 255.
  • Value format. A RowID { page_no: uint32_t, slot_idx: uint16_t }. This is 6 bytes.
  • Numeric-aware comparison. Pure numeric strings are compared by numeric value, so "2" < "10". Non-numeric keys fall back to lexicographic order.
  • Node capacity. About 4076 usable bytes per node. This is 4096 minus the 16-byte PageHeader minus the 4-byte NodeHeader.
  • Deletion strategy. Deletion is lazy. Deleted entries have their RowID set to {0, 0}. The key slot is reused on re-insert of the same key.

The BTree2 handle is a single struct that holds the root page number.

typedef struct {
    uint32_t root_page;   /* 0 = empty tree */
} BTree2;

Node Structure

Every B+ tree node occupies exactly one 4KB page. The first 16 bytes are the standard PageHeader. A 4-byte NodeHeader follows. The key/pointer data area comes after that. The page type field in PageHeader distinguishes internal nodes from leaf nodes. Internal nodes use PAGE_BTREE_INT = 2. Leaf nodes use PAGE_BTREE_LEAF = 3.

NodeHeader

typedef struct {
    uint16_t num_keys;   /* number of keys stored in this node */
    uint16_t flags;      /* reserved, must be 0 */
} NodeHeader;

The position within the page is fixed.

Offset 0:   PageHeader  (16 bytes)
Offset 16:  NodeHeader  (4 bytes)
Offset 20:  Key/pointer data area  (4076 bytes)

Leaf Node Layout

Leaf nodes store sorted key-RowID pairs. Each entry is a length-prefixed key followed by a 6-byte RowID. Leaf nodes are doubly linked through PageHeader.next_page and PageHeader.prev_page for sequential scans.

┌──────────────────────────────────────────────────────┐
│ PageHeader (16B)                                     │
│   page_type = PAGE_BTREE_LEAF                        │
│   page_id, next_page (→ right leaf), prev_page       │
├──────────────────────────────────────────────────────┤
│ NodeHeader (4B)                                      │
│   num_keys, flags                                    │
├──────────────────────────────────────────────────────┤
│ Entry 0: [klen:1B][key_data:klen B][RowID:6B]        │
│ Entry 1: [klen:1B][key_data:klen B][RowID:6B]        │
│ Entry 2: [klen:1B][key_data:klen B][RowID:6B]        │
│ ...                                                  │
│ (entries sorted by key)                              │
│                                                      │
│              ← free space →                          │
└──────────────────────────────────────────────────────┘

The entry overhead per key is 1 (length byte) + key_length + 6 (RowID). This equals 7 plus key_length bytes. For a typical 20-byte key, each entry is 27 bytes. That yields about 150 entries per leaf.

Internal Node Layout

Internal nodes store separator keys interleaved with child page pointers. With N keys there are N+1 child pointers. The first child pointer precedes the first key. Each later key is followed by the child pointer for keys greater than or equal to that separator.

┌──────────────────────────────────────────────────────┐
│ PageHeader (16B)                                     │
│   page_type = PAGE_BTREE_INT                         │
│   page_id                                            │
├──────────────────────────────────────────────────────┤
│ NodeHeader (4B)                                      │
│   num_keys, flags                                    │
├──────────────────────────────────────────────────────┤
│ child0 (4B)                                          │
│ [key0_len:1B][key0_data:N B]                         │
│ child1 (4B)                                          │
│ [key1_len:1B][key1_data:N B]                         │
│ child2 (4B)                                          │
│ ...                                                  │
│ childN (4B)                                          │
│                                                      │
│              ← free space →                          │
└──────────────────────────────────────────────────────┘

The entry overhead per key is 4 (child pointer) + 1 (length byte) + key_length bytes. There is also a trailing 4-byte child pointer at the end. For a typical 20-byte key, the fan-out is about 160 to 170 children per internal node. With this fan-out, a tree of depth 3 can index over 4 million keys.

Leaf Chain (Doubly Linked)

All leaf nodes are connected into a doubly linked list through PageHeader.next_page and PageHeader.prev_page. This enables efficient sequential and range scans. The engine does not need to traverse the tree from the root for each next-key lookup.

           ┌────────┐     ┌────────┐     ┌────────┐
  ... ←────│ Leaf A │←───→│ Leaf B │←───→│ Leaf C │────→ ...
           │ prev=? │     │prev= A │     │prev= B │
           │ next=B │     │next= C │     │ next=0 │
           └────────┘     └────────┘     └────────┘

Key Operations

bt2_create · Allocate Root Page

int bt2_create(BTree2 *tree);

This allocates a single page of type PAGE_BTREE_LEAF through pgm_alloc_page(). It initializes the page as an empty leaf node with num_keys = 0. It writes the page to disk and stores the page number in tree->root_page.

It returns 0 on success and -1 on allocation failure.

A tree with root_page == 0 is considered empty. bt2_insert() calls bt2_create() automatically if the root is 0, so explicit creation is optional.

bt2_insert · Insert Key/RowID

int bt2_insert(BTree2 *tree, const char *key, RowID rid);

This inserts a key-RowID mapping into the tree. The key must not exceed BT2_MAX_KEY_LEN, which is 255 bytes.

The return values are these.

  • 0. Success.
  • 1. Duplicate key. The key already exists with a non-deleted RowID.
  • -1. Error. This means an allocation failure or a key that is too long.

The algorithm runs in these steps.

  1. If the tree is empty with root_page == 0, create a root leaf automatically.
  2. Call insert_recursive() starting at the root page.
  3. The recursive function descends through internal nodes to the correct leaf.
  4. At the leaf, it does the following work.
    • Scan entries to find the sorted insertion position.
    • If the key already exists with a deleted RowID of {0, 0}, reuse the slot by overwriting the RowID. Return 0.
    • If the key exists with a live RowID, return 1 for duplicate.
    • If the entry fits in the leaf's free space, shift later entries right and write the entry. Return 0.
    • If the leaf is full, perform a leaf split. See Tree Splitting.
  5. If a split propagated up and the root itself split, allocate a new root internal node. The new root holds the separator key and two children. These are the old root and the new sibling.

bt2_search · Exact Key Lookup

int bt2_search(BTree2 *tree, const char *key, RowID *out_rid);

This searches for an exact key match. It descends from the root through internal nodes using find_leaf(). Then it performs a linear scan of the leaf's entries.

The return values are these.

  • 0. Found. The RowID is written to *out_rid.
  • -1. Not found. This also covers a key that exists but is marked deleted with RowID == {0, 0}.

The descent logic in find_leaf works as follows. At each internal node it scans separator keys from left to right. It follows child[i] where i is the index of the first separator key greater than the search key. If no separator is greater, it follows the rightmost child.

bt2_delete · Logical Delete

int bt2_delete(BTree2 *tree, const char *key);

This performs a lazy logical delete. It descends to the leaf that contains the key. It overwrites the entry's RowID with the sentinel value {0, 0}. The key bytes stay in the leaf. The slot is reused if the same key is re-inserted.

The return values are these.

  • 0. Success.
  • -1. The key was not found, or it was already deleted.

No merging or rebalancing is performed. This keeps the implementation simple. It avoids the complexity of B+ tree node merges. Over time, leaves may accumulate deleted entries. Re-inserts of the same key reclaim those slots.

bt2_update · Update RowID for Existing Key

int bt2_update(BTree2 *tree, const char *key, RowID new_rid);

This finds the leaf entry for the given key. It overwrites its RowID with new_rid. This is used when a heap record moves to a different page. For example, an UPDATE that changes record size moves the record. The index must then point to the record's updated location.

The return values are these.

  • 0. Success.
  • -1. The key was not found.
info
bt2_update() does not check whether the existing RowID is deleted. It unconditionally overwrites the RowID. This means it can also undelete an entry by writing a valid RowID over a {0, 0} sentinel.

bt2_destroy · Recursive Page Freeing

void bt2_destroy(BTree2 *tree);

This recursively frees every page in the tree by walking the tree depth-first. At each internal node it recurses into all children before freeing the node's own page through pgm_free_page(). Leaf pages are freed directly.

After completion, tree->root_page is set to 0. This is used by DROP TABLE and DROP INDEX to reclaim all storage.

Cursor API

The cursor API provides sequential iteration over the B+ tree's leaf chain. It supports full scans and prefix-based range scans.

typedef struct {
    uint32_t cur_page;    /* current leaf page (0 = exhausted) */
    uint16_t cur_idx;     /* current key index within leaf */
} BTree2Cursor;

bt2_cursor_first · Position at Smallest Key

int bt2_cursor_first(BTree2 *tree, BTree2Cursor *cur);

This descends to the leftmost leaf by always following child[0] at each internal node. It sets the cursor to the first non-deleted entry. If all entries in the first leaf are deleted, it follows next_page to find the first active entry.

It returns 0 on success. It returns -1 if the tree is empty or contains only deleted entries.

bt2_cursor_next · Advance to Next Entry

int bt2_cursor_next(BTree2Cursor *cur, char *key_out, RowID *rid_out);

This returns the current entry's key and RowID, then advances the cursor. The key is null-terminated and at most BT2_MAX_KEY_LEN + 1 bytes. It automatically skips deleted entries with RowID {0, 0}. It follows the leaf chain when the current leaf is exhausted.

It returns 0 on success. It returns -1 when no more entries remain.

The typical iteration pattern looks like this.

BTree2Cursor cur;
bt2_cursor_first(&tree, &cur);
char key[BT2_MAX_KEY_LEN + 1];
RowID rid;
while (bt2_cursor_next(&cur, key, &rid) == 0) {
    // process key, rid
}

bt2_cursor_seek · Position at Key or First Key >= Key

int bt2_cursor_seek(BTree2 *tree, const char *key, BTree2Cursor *cur);

This descends to the leaf that contains the given key. It uses the same find_leaf() logic as bt2_search. Then it scans the leaf to find the first entry whose key is greater than or equal to the search key. If all keys in the leaf are smaller, it follows next_page.

This is the primary mechanism for prefix-based range scans. The system catalog uses composite keys with zero-padded IDs such as "0000000001:tablename". A call to bt2_cursor_seek("0000000001:") positions the cursor at the first entry for table ID 1. Iteration continues with bt2_cursor_next() until the prefix no longer matches.

It returns 0 on success. It returns -1 if no key greater than or equal to the search key exists.

Tree Splitting

When a leaf or internal node runs out of space during insertion, the node is split into two siblings. A separator key is promoted to the parent. If the parent is also full, the split propagates upward recursively. If the root itself splits, a new root is created. This increases the tree height by one.

Leaf Split

  1. The insertion target leaf is full. Used bytes plus entry size is greater than 4076.
  2. A temporary oversized buffer is created. The new entry is inserted at its sorted position.
  3. The entries are divided at the midpoint with mid = total_keys / 2.
  4. Entries [0, mid) stay in the original page.
  5. Entries [mid, total) are copied to a newly allocated leaf page.
  6. The leaf chain is updated as follows.
    • New leaf's next_page = old leaf's former next_page.
    • New leaf's prev_page = old leaf's page_id.
    • Old leaf's next_page = new leaf's page_id.
    • If a leaf existed after the old leaf, its prev_page is updated to point to the new leaf.
  7. The separator key, which is the first key of the new leaf, is promoted to the parent.
BEFORE (leaf full, inserting "F"):

    ┌────────────────────────────────────────┐
    │ Leaf: [A][B][C][D][E][G][H]  (FULL)    │
    └────────────────────────────────────────┘

AFTER (split at midpoint, "E" promoted):

              separator = "E"
                   │
                   ▼
    ┌──── parent receives ("E", ptr→new leaf) ────┐
    │                                              │
    ▼                                              ▼
  ┌─────────────────┐    next    ┌─────────────────┐
  │ [A][B][C][D]    │──────────→│ [E][F][G][H]    │
  │ Old Leaf        │←──────────│ New Leaf        │
  └─────────────────┘    prev    └─────────────────┘

Internal Node Split

  1. The internal node is full. It cannot fit the new separator key plus child pointer.
  2. A temporary oversized buffer is created. The new separator is inserted at its correct position.
  3. The entries are divided at the midpoint. The middle key is not kept in either sibling. It is promoted to the parent.
  4. Keys [0, mid) and children [0, mid] stay in the original node.
  5. The key at index mid is sent up to the parent.
  6. Keys [mid+1, total) and children [mid+1, total] go to the new internal node.
BEFORE (internal node full, inserting separator "M"):

  ┌─────────────────────────────────────────────────┐
  │ Internal: [ptr0] D [ptr1] G [ptr2] J [ptr3]     │
  │           K [ptr4] P [ptr5] S [ptr6]  (FULL)     │
  └─────────────────────────────────────────────────┘

AFTER (split, "K" promoted to parent):

                    "K" promoted
                       │
                       ▼
           ┌──── parent receives ("K", ptr→right) ────┐
           │                                           │
           ▼                                           ▼
  ┌────────────────────────┐          ┌────────────────────────┐
  │ [ptr0] D [ptr1] G      │          │ [ptr3] M [ptr4] P      │
  │        [ptr2] J [ptr3] │          │        [ptr5] S [ptr6] │
  │ Left (original)        │          │ Right (new)            │
  └────────────────────────┘          └────────────────────────┘

Root Split

When the current root splits, bt2_insert() allocates a new internal node as the root. This new root has exactly one separator key. It has two children, which are the old root and its new sibling. It becomes the new tree->root_page. This is the only operation that increases the tree's height.

BEFORE:
         root (leaf or internal, now split)
              │
        ┌─────┴─────┐
    old_root     new_sibling

AFTER:
            new_root
           /    │    \
     child0   "sep"  child1
       │                │
   old_root        new_sibling

Sentinel RowID

The RowID value {page_no = 0, slot_idx = 0} is the universal deleted sentinel. Page 0 is always the file header and can never contain user data. This value is therefore guaranteed to be invalid as a record pointer.

The sentinel behaves like this across the API.

OperationBehavior
bt2_search()Returns -1 for not found if the matching entry has RowID {0, 0}
bt2_delete()Writes {0, 0} over the entry's RowID
bt2_insert()If a key exists with RowID {0, 0}, reuses the slot and writes the live RowID
bt2_cursor_next()Silently skips entries with RowID {0, 0}
bt2_cursor_first()Skips leading deleted entries and advances to the next leaf if needed

Key Comparison

The key_cmp() function implements numeric-aware comparison. It works in these steps.

  1. Attempt to parse both keys as floating-point numbers with strtod.
  2. If both parse successfully with no trailing characters, compare numerically.
  3. Otherwise, fall back to lexicographic comparison. This uses memcmp with length tie-breaking.

This ensures that numeric primary keys sort naturally, so "2" < "10" rather than "10" < "2". String keys retain standard dictionary order.

Max Key Length

The maximum key length is 255 bytes through BT2_MAX_KEY_LEN. This is constrained by the 1-byte length prefix used in the node entry format. bt2_insert() rejects keys longer than this limit with a return value of -1.

In practice, keys are much shorter.

  • Primary key indexes. The string representation of column values. This is typically 5 to 40 bytes.
  • System catalog composite keys. Zero-padded 10-digit IDs with a : separator. For example "0000000001:users" is 17 bytes.
  • Secondary index keys. The column value concatenated with the primary key for uniqueness.

Usage

The B+ tree is the sole indexing structure in EvoSQL. Every index is a BTree2 instance. This holds for user tables and for internal metadata.

System Catalog

The system catalog uses 8 B+ trees for all metadata. They are stored in FileHeader.catalog_roots[].

Catalog trees use composite keys with zero-padded IDs and : separators. bt2_cursor_seek() with a prefix like "0000000005:" enables efficient iteration over all entries that belong to a specific parent. For example it can iterate over all columns of table 5.

Primary Key Indexes

Every table with a primary key has an associated B+ tree. The key is the string representation of the primary key column values. The RowID points to the heap page and slot where the full record is stored. INSERT checks for duplicates through bt2_search() before inserting. SELECT ... WHERE pk = value uses bt2_search() for O(log n) point lookups. See Table API and DML.

Secondary Indexes

Secondary indexes created through CREATE INDEX are also B+ trees. Index.c populates the tree by scanning the table's heap pages with tapi_scan and inserting one entry per row.
