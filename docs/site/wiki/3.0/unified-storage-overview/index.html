Unified Storage Overview

EvolutionDB keeps all of its data in one binary file called evosql.db. This single file holds databases, schemas, tables, indexes, users, grants, and the system catalog. The file is split into fixed-size 4KB pages. Every disk read and write goes through a Clock Sweep buffer pool. All structural links inside the file are kept with B+ tree indexes and slotted pages.

Architecture Overview

All data lives in one file made of 4KB pages. Page 0 holds the file header. The rest of the file holds data pages, B+ tree nodes, and free pages.

evosql.db (single binary file, 4KB pages)
 Page 0: FileHeader (magic, free list head, catalog roots, ID counters)
 Page 1..N: Data pages, B+ tree nodes, free pages

The storage engine is built in layers. Each layer depends only on the layer below it.

Page Layout

Every page is exactly EVO_PAGE_SIZE bytes, which is 4096. The first 16 bytes of every page hold a common header.

typedef struct {
    uint8_t  page_type;    /* PAGE_FREE=0, PAGE_DATA=1, PAGE_BTREE_INT=2,
                              PAGE_BTREE_LEAF=3, PAGE_OVERFLOW=4 */
    uint8_t  flags;        /* reserved */
    uint16_t free_space;   /* bytes of free space on this page */
    uint32_t page_id;      /* self-identifying page number */
    uint32_t next_page;    /* free list chain or leaf chain */
    uint32_t prev_page;    /* B+ tree leaf backward pointer */
} PageHeader;              /* 16 bytes */

Usable space per page is 4096 - 16 = 4080 bytes.

Page 0 File Header

Page 0 is reserved for the FileHeader. It is never freed.

Free List

Free pages are chained through the next_page field in their PageHeader. When pgm_alloc_page() is called, the page manager does one of two things.

  1. If the free list is not empty, it pops the head page and initializes it with the requested type.
  2. If the free list is empty, it grows the file by 50% of the current size, with a minimum of 16 pages. It chains the new pages into the free list. Then it allocates.

When pgm_free_page() is called, the page is zeroed and pushed onto the front of the free list. Page 0 cannot be freed.

The initial file is 64 pages, which is 256KB. Pages 1 through 63 form the initial free list.

Buffer Pool

All page I/O flows through a Clock Sweep buffer pool in buffer_pool.h/c. It is modeled after PostgreSQL's buffer manager. The pool caches 4KB pages in memory. It uses write-back semantics. Dirty pages are flushed to disk only on eviction or on an explicit flush.

Key parameters.

ConstantValueDescription
BP_PAGE_SIZE4096Cache page granularity
BP_DEFAULT_PAGES256Default pool size (1MB)
BP_MAX_USAGE5Maximum usage count per page
BP_RING_SIZE8Sequential scan ring buffer slots

Clock Sweep algorithm. A clock hand rotates through all buffer slots. When a victim is needed, the pool checks each slot in turn.

  • If pin_count > 0, skip the page. It is in use.
  • If usage_count == 0, select the page as the victim. Flush it to disk if it is dirty, then evict it.
  • Otherwise, decrement usage_count and advance the hand.

Each cache hit increments usage_count, up to BP_MAX_USAGE. This gives frequently accessed pages more chances to survive eviction sweeps.

Anti-pollution ring buffer. Sequential scans use bp_read_seq() with a private BPRing. The ring has 8 slots and works round-robin. This stops large table scans from evicting hot pages from the main pool.

Core API.

void bp_init(int num_pages);           /* allocate pool */
int  bp_read(int fd, void *buf, size_t count, off_t offset);
int  bp_write(int fd, const void *buf, size_t count, off_t offset);
void bp_flush_all(void);               /* write all dirty pages to disk */
void bp_flush_fd(int fd);              /* flush dirty pages for one fd */
void bp_invalidate_fd(int fd);         /* flush + drop all pages for fd */

The pool uses an open-addressing hash table for fast lookup. The table is 4 times the pool size and uses linear probing. It is keyed by (fd, page_no) for O(1) page lookup. An internal mutex serializes all pool operations for thread safety.

Component Summary

Page Manager (page_mgr.h/c)

The page manager manages the global evosql.db file. It provides page allocation and deallocation, free list management, file growth, and access to file header fields. The header fields include the catalog root pages and the auto-increment counters. All reads and writes delegate to the buffer pool.

See Page Manager and Slotted Pages.

Slotted Pages (slotted.h/c)

Slotted pages store variable-length records within a single 4KB data page. The page layout is shown below.

[PageHeader 16B][SlottedHeader 6B][Slot Directory -->]
                   [ FREE SPACE ]
                [<-- Records packed from end of page]

Each record is identified by a RowID = {page_no, slot_idx}, which is 6 bytes. A RowID with page_no == 0 is the "deleted" sentinel. This works because page 0 is always the file header. The slot directory grows forward. Records grow backward. Deleted slots can be reused. slot_compact() defragments records to reclaim space from deletions.

Maximum record size is 4096 - 16 - 6 - 4 = 4070 bytes.

See Page Manager and Slotted Pages.

B+ Tree (btree2.h/c)

The B+ tree provides O(log n) key-to-RowID indexing. Each B+ tree node occupies one 4KB page. Leaf nodes are doubly linked for efficient sequential and range scans.

Key format. The key is a length-prefixed string. It uses 1 byte for the length followed by up to 255 bytes of data. Key comparison is numeric-aware. If both keys parse as numbers, they are compared as double. Otherwise lexicographic comparison is used.

Leaf entry. [key_len:1][key_data:N][RowID:6]

Internal entry. [child0:4][key0_len:1][key0:N][child1:4]...

Insertions handle splits automatically. They propagate separator keys up to the parent and create a new root when needed. Deletions use lazy marking. They set the RowID to {0, 0}.

Cursor operations traverse the leaf chain for ordered iteration. These are bt2_cursor_first, bt2_cursor_next, and bt2_cursor_seek.

See B+ Tree.

System Catalog (catalog_internal.h/c)

The system catalog stores all metadata in 8 B+ trees. There is one tree per catalog table.

0CAT_SYS_DATABASESdb_nameDatabase registry
1CAT_SYS_SCHEMASdb_id:schema_nameSchemas per database
2CAT_SYS_TABLESschema_id:table_nameTables per schema
3CAT_SYS_COLUMNStable_id:col_ordinalColumn definitions
4CAT_SYS_INDEXEStable_id:index_nameIndex metadata

Composite keys use zero-padded 10-digit IDs with a : separator. An example is "0000000001:tablename". Catalog records are stored as semicolon-delimited strings on slotted pages. The B+ tree entries point to those records. Root page numbers for all 8 trees are persisted in the file header's catalog_roots[] array.

See System Catalog.

Table API (table_api.h/c)

The Table API bridges the DML modules and the storage layers. It provides the following operations.

Table resolution
tapi_resolve() looks up a table by name using the catalog. It returns a TableDesc and arrays of ColumnDesc.
Heap operations
tapi_heap_insert(), tapi_heap_read(), tapi_heap_delete(), and tapi_heap_update() manage records on slotted pages. Pages are chained through next_page in the page header. New pages are allocated when existing ones are full.
PK key building
tapi_build_pk_key() extracts the primary key columns from a semicolon-delimited record. It builds the B+ tree key string. A single column gives "value". A composite key gives "val1|val2".
Table scan
tapi_scan_begin() and tapi_scan_next() iterate all records ordered by primary key. They use the PK B+ tree cursor. Each cursor step reads the RowID from the B+ tree. Then it fetches the full record from the heap page.
Cleanup
tapi_free_heap_pages() walks the heap chain and frees all pages. This is used by DROP TABLE.

See Table API and DML.

INSERT

SQL: INSERT INTO employees (id, name) VALUES (1, 'Alice');

 evoparser.y (parse + emit RPN)
      |
 Insert.c  InsertProcess()
      |
      +-- tapi_resolve("employees") --> TableDesc, ColumnDesc[]
      |
      +-- tapi_build_pk_key(cols, record) --> "1"
      |
      +-- bt2_search(pk_tree, "1") --> check for duplicate PK
      |
      +-- tapi_heap_insert(td, "1;Alice") --> RowID{page=5, slot=3}
      |       |
      |       +-- pgm_read_page(heap_page)    (via buffer pool)
      |       +-- slot_insert(page_buf, record)
      |       +-- pgm_write_page(heap_page)   (via buffer pool)
      |
      +-- bt2_insert(pk_tree, "1", RowID{5,3})
              |
              +-- find_leaf() --> descend internal nodes
              +-- write entry to leaf page (split if full)
              +-- pgm_write_page()            (via buffer pool)

SQL: SELECT * FROM employees;

 Select.c  SelectProcess()
      |
      +-- tapi_resolve("employees") --> TableDesc, ColumnDesc[]

      +-- tapi_scan_begin(td, &cursor) --> positions at first PK
      |
      +-- loop: tapi_scan_next(&cursor, pk_key, record_buf)
              |

Initialization

Server startup calls db_ensure_root() in DatabaseMgmt.c. This triggers the full initialization sequence.

db_ensure_root()
    |
    +-- mkdir("root/")                     create data directory
    |
    +-- pgm_init("root/evosql.db")         open or create the global file
    |       |
    |       +-- (if file exists) validate magic + page_size, load header
    |       +-- (if file absent) create file, write FileHeader to page 0,
    |           chain pages 1..63 into free list (256KB initial)
    |
    +-- cat_init()                         initialize system catalog
    |       |
    |       +-- (first run) create 8 B+ trees, store roots in file header,
    |       |   insert default database ("evosql") and schema ("default")
    |       +-- (subsequent runs) load tree roots from file header
    |
    +-- db_ensure_users()                  create admin user if absent
    |       |
    |       +-- cat_find_user("admin")
    |       +-- if not found: cat_create_user("admin", hashed_password)
    |
    +-- db_ensure_grants()                 create admin superuser grant
            |
            +-- cat_find_grant("admin", DATABASE, "*")
            +-- if not found: cat_create_grant("admin", DATABASE, "*", "ALL", 1)
warning
The buffer pool must be initialized before pgm_init(). The call is bp_init(). All page I/O depends on the buffer pool. The adaptor's server.c calls bp_init() during server startup before it invokes db_ensure_root().
