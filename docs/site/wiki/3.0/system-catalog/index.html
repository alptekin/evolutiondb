System Catalog

The system catalog stores all database metadata inside the unified storage file. Eight B+ trees map keys to records, and the records live as semicolon-delimited strings on slotted pages. This page describes the catalog trees, the descriptor structs, the key design, the serialization format, and the full catalog API.

Source files. evolution/db/catalog_internal.h, evolution/db/catalog_internal.c.

Tests. tests/test_catalog.c with 30 tests.

Overview

The system catalog stores all database metadata inside the unified storage file evosql.db. This metadata covers databases, schemas, tables, columns, indexes, constraints, user accounts, and privilege grants. There are no external metadata files. Every piece of metadata lives in one of eight B+ trees. The actual records are stored as semicolon-delimited strings on slotted pages. The B+ tree maps a key to a RowID. A RowID is a page number plus a slot index. The RowID points to the record on a slotted page.

                ┌─────────────────────────┐
                │      FileHeader         │
                │  (Page 0 of evosql.db)  │
                │                         │
                │  catalog_roots[0..7]    │  ← root page number of each B+ tree
                │  next_table_id          │
                │  next_schema_id         │
                │  next_db_id             │
                └────────┬────────────────┘
                         │
        ┌────────────────┼────────────────┐
        ▼                ▼                ▼
   B+ Tree 0        B+ Tree 1  ...  B+ Tree 7
  (databases)       (schemas)       (grants)
        │                │                │
        ▼                ▼                ▼
   Slotted Pages    Slotted Pages    Slotted Pages
   (records)        (records)        (records)

All catalog I/O flows through the buffer pool and the page manager. There is no direct file access. See the Unified Storage Overview and the Page Manager and Slotted Pages pages for the layers underneath.

The Eight Catalog Trees

Each catalog tree is identified by a CatalogID constant defined in page_mgr.h. The root page number of every tree is persisted in the FileHeader.catalog_roots[] array on page 0.

CAT_SYS_SCHEMAS1Schema registrydb_id:schema_name
CAT_SYS_TABLES2Table descriptorsschema_id:table_name
CAT_SYS_COLUMNS3Column definitionstable_id:col_ordinal

The CatalogID enum and the CATALOG_ROOT_SLOTS constant are defined in page_mgr.h. There are 16 slots and 8 are used.

Descriptor Structs

Each catalog tree has a matching C struct that represents one record. These structs are defined in catalog_internal.h.

DatabaseDesc

typedef struct {
    uint32_t db_id;
    char     db_name[CAT_MAX_NAME_LEN];   // 128
} DatabaseDesc;

The B+ tree key is the database name itself, for example "evosql". The record stores only the db_id. The name is recovered from the key.

SchemaDesc

typedef struct {
    uint32_t schema_id;
    uint32_t db_id;
    char     schema_name[CAT_MAX_NAME_LEN];
} SchemaDesc;

Key format is "%010u:%s". It is the zero-padded db_id, then a colon, then the schema name.

TableDesc

typedef struct {
    uint32_t table_id;
    uint32_t schema_id;
    char     table_name[CAT_MAX_NAME_LEN];
    uint32_t pk_root_page;      // root page of the table's primary-key B+ tree
    uint32_t heap_page;         // first slotted page for row storage (0 = lazy alloc)
    int      num_columns;
    int      pad_size;
    int      auto_inc_col;      // column ordinal, or -1 if none
    int      auto_inc_counter;
    int      auto_inc_step;
} TableDesc;

Key format is "%010u:%s". It is the zero-padded schema_id, then a colon, then the table name.

When a table is created, cat_create_table also creates a dedicated B+ tree for the table's primary index. It records its root page in pk_root_page. The heap_page field starts at 0. It is set on the first row insertion through cat_update_heap_page.

ColumnDesc

typedef struct {
    uint32_t table_id;
    int      col_ordinal;
    char     col_name[CAT_MAX_NAME_LEN];
    int      type_code;         // typeVal / 10000 = type family, % 10000 = size
    int      is_not_null;
    int      is_unique;
    int      is_pk;
    char     default_val[CAT_MAX_DEFAULT_LEN];  // 256
} ColumnDesc;

Key format is "%010u:%010d". It is the zero-padded table_id, then a colon, then the zero-padded col_ordinal. Both parts are numeric and zero-padded. This ensures columns are returned in ordinal order during prefix scans.

IndexDesc

typedef struct {
    uint32_t table_id;
    char     index_name[CAT_MAX_NAME_LEN];
    uint32_t root_page;
    char     col_list[CAT_MAX_NAME_LEN];    // comma-separated column names
    char     index_type;                     // 'P' primary, 'U' unique, 'N' normal
} IndexDesc;

Key format is "%010u:%s". It is the zero-padded table_id, then a colon, then the index name.

ConstraintDesc

typedef struct {
    uint32_t table_id;
    int      ordinal;
    char     constraint_type;   // 'C' check, 'F' foreign key, 'P' primary, 'U' unique
    char     definition[1024];
    uint32_t ref_table_id;      // for FK, 0 otherwise
    char     ref_columns[CAT_MAX_NAME_LEN];
} ConstraintDesc;

Key format is "%010u:%010d". It is the zero-padded table_id, then a colon, then the zero-padded ordinal. The ordinal is assigned from a global counter named g_constraint_ordinal. That counter increments monotonically across all tables.

UserDesc

typedef struct {
    char username[CAT_MAX_NAME_LEN];
    char password_hash[CAT_MAX_HASH_LEN];   // 512
} UserDesc;

The B+ tree key is the username. The record stores the PBKDF2-SHA256 password hash. The username is recovered from the key.

GrantDesc

typedef struct {
    char username[CAT_MAX_NAME_LEN];
    int  scope_type;            // 1=DATABASE, 2=SCHEMA, 3=TABLE
    char scope_name[CAT_MAX_NAME_LEN];
    char privileges[CAT_MAX_NAME_LEN];  // comma-separated or "ALL"
    int  grant_option;
} GrantDesc;

Key format is "%s:%d:%s". It is the username, then the scope type integer, then the scope name. This enables a prefix scan by "username:" to find all grants for a given user.

Composite Key Design

B+ tree keys are variable-length strings up to 255 bytes. The limit is CAT_MAX_KEY_LEN, which equals BT2_MAX_KEY_LEN. Composite keys use : as the separator. Numeric ID components are zero-padded to 10 digits so that lexicographic comparison preserves numeric ordering.

"0000000001:users"          schema 1, table "users"
"0000000003:0000000002"     table 3, column ordinal 2
"admin:1:evosql"            user "admin", scope DATABASE(1), database "evosql"

The B+ tree comparator performs lexicographic comparison on all keys. The keys contain : so they are treated as strings, not numbers. Zero-padding guarantees that "0000000002" sorts before "0000000010".

This encoding also enables efficient prefix scans. To list all columns of table 5, seek to "0000000005:" and iterate while keys still begin with that prefix. Every cat_list_* function uses this pattern. See the B+ Tree page for the cursor and comparator details.

Serialization Format

Records are stored as semicolon-delimited strings on slotted pages. This matches the project-wide convention for row storage. Each descriptor struct has a serialize_* and deserialize_* function pair.

For example, a TableDesc serializes to this string.

"1;1;users;42;0;3;256;-1;0;0"
 │ │ │     │  │ │ │   │  │ └─ auto_inc_step
 │ │ │     │  │ │ │   │  └─── auto_inc_counter
 │ │ │     │  │ │ │   └────── auto_inc_col (-1 = none)
 │ │ │     │  │ │ └────────── pad_size
 │ │ │     │  │ └──────────── num_columns
 │ │ │     │  └────────────── heap_page
 │ │ │     └───────────────── pk_root_page
 │ │ └─────────────────────── table_name
 │ └───────────────────────── schema_id
 └─────────────────────────── table_id

Field extraction uses the internal next_field() helper. It reads characters up to the next ; or the end of the string.

For some descriptors, the key carries data that the record omits.

  • DatabaseDesc. The record contains only db_id. The name comes from the B+ tree key.
  • UserDesc. The record contains only the password hash. The username comes from the key.

Maximum record size is CAT_MAX_RECORD_LEN, which is 2048 bytes.

ID Generation

Object IDs for databases, schemas, and tables are auto-incrementing integers stored in the FileHeader.

typedef struct {
    // ...
    uint32_t next_table_id;     // auto-increment for table IDs
    uint32_t next_schema_id;    // auto-increment for schema IDs
    uint32_t next_db_id;        // auto-increment for database IDs
    // ...
} FileHeader;

The pgm_next_id(int id_type) function atomically reads and increments the right counter.

id_typeCounterUsed by
0next_table_idcat_create_table
1next_schema_idcat_create_schema
2next_db_idcat_create_database

IDs start at 1. The counter initializes to 1 in a fresh file header. IDs are never reused after deletion.

Initialization and Bootstrap

Startup sequence

  1. pgm_init("evosql.db") opens or creates the storage file. It initializes the page manager and the buffer pool.
  2. cat_init() initializes the system catalog.

First run with an empty file

cat_init() detects that catalog_roots[CAT_SYS_DATABASES] is 0. This means there is no existing catalog. It then performs a full bootstrap.

  1. Creates 8 B+ trees through bt2_create().
  2. Stores each tree's root page number in FileHeader.catalog_roots[] through pgm_set_catalog_root().
  3. Inserts the default database "evosql". It is assigned db_id = 1.
  4. Inserts the default schema "default" under database 1. It is assigned schema_id = 1.
  5. Inserts the default user "admin" with an empty password hash. The server's authentication layer sets the real PBKDF2-SHA256 hash on first startup.

Subsequent runs

cat_init() loads the 8 root page numbers from FileHeader.catalog_roots[] into the in-memory BTree2 handles. Data pages for each catalog tree are lazily allocated on the first insert after restart. This costs at most 8 pages, which is 32 KB of overhead.

Shutdown

cat_shutdown() calls pgm_flush() to write all dirty buffer pool pages to disk. It then resets all in-memory state.

Convenience Three-Level Resolution

int cat_resolve_table(const char *db_name, const char *schema_name,
                      const char *table_name, TableDesc *out);

This performs a three-step lookup. First cat_find_database. Then cat_find_schema using db.db_id. Then cat_find_table using schema.schema_id. It returns 0 on success and -1 if any level fails. This is the primary entry point used by DML operations to resolve a table name to its descriptor. Those operations are SELECT, INSERT, UPDATE, and DELETE. See the Table API and DML page for how DML uses this descriptor.

Internal Data Storage Pattern

All catalog operations follow the same create, find, and drop pattern internally.

Create

1. Build composite B+ tree key from the input IDs and name.
2. Check for duplicates via bt2_search (return -1 if key exists).
3. Allocate an auto-increment ID via pgm_next_id().
4. Populate the descriptor struct.
5. Serialize the struct to a semicolon-delimited string.
6. Insert the string into a slotted page via cat_store_record() → returns RowID.
7. Insert (key → RowID) into the B+ tree via bt2_insert().

Find

1. Build key.
2. bt2_search(key) → RowID.
3. cat_read_record(RowID) → semicolon-delimited string.
4. Deserialize string into the descriptor struct.

List with prefix scan

1. Build prefix (e.g., "0000000005:").
2. bt2_cursor_seek(prefix) → position cursor at first matching key.
3. Loop: bt2_cursor_next() → (key, RowID).
4. Break when key no longer starts with the prefix.
5. Read and deserialize each record.

Drop

1. Build key.
2. bt2_search(key) → RowID.
3. cat_delete_record(RowID) marks slot as deleted on the slotted page.
4. bt2_delete(key) removes entry from the B+ tree.

Update with in-place or relocate

This applies to mutable fields such as auto_inc_counter, heap_page, and root_page.

1. bt2_search(key) → RowID.
2. Read record, deserialize, modify field, re-serialize.
3. Try slot_update() for in-place replacement on the same slotted page.
4. If the updated record does not fit in the same slot:
   a. Delete the record from the slotted page.
   b. Insert the record on a (possibly different) slotted page → new RowID.
   c. bt2_update(key, new_RowID) to point the B+ tree at the relocated record.

Slotted Page Management

Each of the 8 catalog trees keeps a current data page for new record insertions. The current page is held in g_cat_data_pages[CatalogID]. When cat_store_record is called, it does the following.

  1. If a current data page exists, it tries slot_insert on it.
  2. If the page is full, or no current page is set, it allocates a fresh PAGE_DATA page through pgm_alloc_page. It initializes the page with slot_page_init, inserts the record, and updates g_cat_data_pages.
info
After a server restart, all g_cat_data_pages entries reset to 0. The first insert into each catalog tree allocates a fresh data page. This can waste up to 8 pages, which is 32 KB, per restart. It avoids the complexity of scanning for partially-full data pages.
