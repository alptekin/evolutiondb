Table API and DML

The Table API in table_api.h and table_api.c bridges SQL operations and the storage engine. It wraps catalog resolution, heap page input and output, primary key management, and full table scans. The DML modules build INSERT, SELECT, UPDATE, DELETE, DROP, CREATE, and index operations on top of it.

All table data is stored in slotted pages for heap storage. Rows are indexed by B+ trees through the btree2 module. The system catalog tracks table metadata, column definitions, constraints, and index descriptors. Every DML module depends on the Table API. See also the unified storage overview, the page manager and slotted pages, and the B+ tree pages.

Table API

The Table API is a focused set of functions in table_api.h and table_api.c. It encapsulates catalog resolution, heap page input and output, primary key management, and full table scanning. Every DML module depends on it.

tapi_resolve, catalog resolution

int tapi_resolve(const char *name_or_path, TableDesc *td,
                 ColumnDesc *cols, int *ncols);

This function resolves a table name into a TableDesc structure and an optional ColumnDesc array. The name can be a plain name like "employees". It can also be a full path like "root/evosql/default/employees". The work happens in three steps.

  1. It strips directory prefixes and the .dat extension through tapi_basename().
  2. It calls cat_resolve_table(g_currentDatabase, g_currentSchema, name, td) to look up the table in the system catalog. The lookup uses the current database and schema context.
  3. If cols and ncols are non-NULL, it calls cat_find_columns(td->table_id, cols, CAT_MAX_COLUMNS) to fill the column descriptors.

It returns 0 on success. It returns -1 if the table does not exist.

tapi_pk_tree, B+ tree handle for the primary key

static inline BTree2 tapi_pk_tree(const TableDesc *td)
{
    BTree2 tree = { td->pk_root_page };
    return tree;
}

This function builds a BTree2 handle from the table's pk_root_page. It is a zero-cost inline. Every DML operation uses it to reach the primary key index.

tapi_build_pk_key and tapi_build_pk_key_from_vals, PK key composition

int tapi_build_pk_key(const ColumnDesc *cols, int ncols,
                      const char *record, char *key_out, int key_size);

int tapi_build_pk_key_from_vals(const ColumnDesc *cols, int ncols,
                                char **vals, int nvals,
                                char *key_out, int key_size);

These functions build a composite primary key string from record fields. They scan the ColumnDesc array for columns with is_pk == 1 and join their values.

  • Single PK. The key looks like "42".
  • Composite PK. The key looks like "42|alice". The parts are pipe-delimited.

If no column has is_pk set, the first column is used as a fallback. The _from_vals variant takes a pre-parsed value array. The plain variant parses a semicolon-delimited record string on its own.

tapi_get_pk_indices, PK column positions

int tapi_get_pk_indices(const ColumnDesc *cols, int ncols,
                        int *indices, int max);

This function returns the ordinal positions of PK columns within the schema. UPDATE and DELETE use it to detect when a SET clause changes a PK column. A change to a PK column requires B+ tree key updates.

Heap page operations

All record data lives in slotted pages. The pages are chained together through PageHeader.next_page. The Table API provides four heap operations.

tapi_heap_insert

RowID tapi_heap_insert(TableDesc *td, const char *record, uint16_t rec_len);
  1. It tries slot_insert() on the table's current heap_page.
  2. If the page is full, or if no heap page exists, it allocates a fresh PAGE_DATA page through pgm_alloc_page(). It initializes the page with slot_page_init(). It chains the page by setting new_page->next_page = td->heap_page.
  3. It updates td->heap_page and persists it through cat_update_heap_page().
  4. It returns the RowID of the inserted record. The RowID is a page number and a slot index. It returns {0, 0} on error.

tapi_heap_read

int tapi_heap_read(RowID rid, char *buf, int bufsize);

This function reads the page at rid.page_no. It then calls slot_read() for rid.slot_idx. It returns the record length, which is null-terminated, or -1 on error.

tapi_heap_update

RowID tapi_heap_update(TableDesc *td, RowID old_rid,
                       const char *new_record, uint16_t rec_len);

This function tries an in-place update through slot_update(). If the record does not fit in the existing slot, it falls back to delete-then-insert. The slot's free space is not enough in that case. It deletes the old slot. It then calls tapi_heap_insert() to place the record in a page with space. It returns the RowID, which may have changed.

tapi_heap_delete

int tapi_heap_delete(RowID rid);

This function reads the page. It calls slot_delete() to mark the slot as free. It writes the page back. It returns 0 on success.

Full table scan

int tapi_scan_begin(const TableDesc *td, TableScanCursor *cursor);
int tapi_scan_next(TableScanCursor *cursor,
                   char *pk_key_out, char *record_out, int rec_out_size);

Table scans iterate over the PK B+ tree, not the heap chain. This yields records in primary key order. tapi_scan_begin() opens a cursor at the first leaf entry through bt2_cursor_first(). Each call to tapi_scan_next() advances the cursor with bt2_cursor_next(). It retrieves the PK key and the RowID. It then fetches the record through tapi_heap_read().

The TableScanCursor wraps a BTree2Cursor and the BTree2 handle.

typedef struct {
    BTree2Cursor bt_cursor;
    BTree2       tree;
} TableScanCursor;

tapi_free_heap_pages

void tapi_free_heap_pages(const TableDesc *td);

This function walks the heap page chain. It starts at td->heap_page and follows next_page pointers. It calls pgm_free_page() on each page. DROP TABLE and bulk cleanup use it.

DML operations

CREATE TABLE (Create.c)

CreateTableProcess() orchestrates table creation through the catalog and storage layers.

  1. Catalog registration. It calls cat_create_table() to register the table in CAT_SYS_TABLES. This assigns a unique table_id.
  2. PK B+ tree creation. It calls bt2_create() to allocate the root page for the primary key index. The root page number is stored in the catalog through cat_update_pk_root().
  3. Column registration. It calls cat_create_columns() to store each column's metadata in CAT_SYS_COLUMNS. The metadata covers the name, type code, nullability, default value, uniqueness, and PK flag.
  4. Constraint registration. It registers CHECK constraints, UNIQUE constraints, and FOREIGN KEY constraints in CAT_SYS_CONSTRAINTS.
  5. AUTO_INCREMENT. If any column has AUTO_INCREMENT, the auto-increment column index, counter, and step are stored in the TableDesc and persisted in the catalog.

INSERT (Insert.c)

INSERT INTO employees (name, dept, salary) VALUES ('Alice', 'Eng', 90000);

Data flow.

Parser (RPN) --> InsertProcess()
                    |
                    v
            tapi_resolve(tblName)           -- catalog lookup
                    |
                    v
            [Column reordering]             -- if INSERT specifies column list
                    |
                    v
            [AUTO_INCREMENT]                -- generate value if needed
                    |
                    v
            [Validation pipeline]           -- types, NOT NULL, UNIQUE, CHECK
                    |
                    v
            [Unique index pre-check]        -- btree_search on secondary indexes
                    |
                    v
            store_single_row()
                |
                +-- tapi_build_pk_key()     -- compose PK from values
                +-- bt2_search()            -- duplicate PK check
                +-- g_tx_undo_callback()    -- record TX_OP_INSERT undo entry
                +-- tapi_heap_insert()      -- store record in slotted page
                +-- bt2_insert()            -- index PK in B+ tree
                    |
                    v
            [Secondary index update]        -- btree_insert() for each index
                    |
                    v
            [Persist AUTO_INCREMENT]        -- cat_update_auto_inc()

Key details.

  • Multi-row INSERT. g_insert contains all rows separated by ROW_SEP, which is \x02. Each row is processed on its own through the validation and storage pipeline.
  • Column reordering. When the INSERT gives a column list, such as INSERT INTO t (c2, c1) VALUES ..., reorder_row_for_column_map() maps the user-specified column order to the schema's physical column order. It fills missing columns with DEFAULT values or NULL.
  • Duplicate PK detection. store_single_row() calls bt2_search() before inserting. If the key exists, it returns 1 for duplicate without changing any data.
  • Root page tracking. After bt2_insert(), the function updates td->pk_root_page = pk_tree.root_page. A B+ tree split may have created a new root page.

SELECT (Select.c)

SELECT * FROM employees WHERE dept = 'Eng';

Data flow.

Parser (RPN) --> SelectAll() / SelectProcess()
                    |
                    v
            tapi_resolve(tblName)           -- catalog lookup
                    |
                    v
        +---[Access path decision]---+
        |                            |
   Full Scan                    PK Lookup
        |                            |
        v                            v
tapi_scan_begin()           bt2_search(pk_tree, key)
tapi_scan_next() loop            |
        |                        v
        v                  tapi_heap_read(rid)

        |                        v
        v                  [Single result]
[ORDER BY sort]
        |
        v

SelectAll() handles SELECT * with two paths.

  1. Simple scan. It opens a TableScanCursor through tapi_scan_begin(). It iterates with tapi_scan_next() and prints each record.
  2. ORDER BY. It collects all records into an array. It sorts them with qsort() using the specified column index g_sortColIndex and direction g_sortDesc. It then prints the sorted result.

SelectProcess() lives in the adaptor's query_executor.c. It handles full SQL SELECT with WHERE.

  • PK Lookup. When WHERE matches the primary key column with =, it performs a single bt2_search() plus tapi_heap_read().
  • Index Scan. When WHERE matches a secondary index column, it uses sec_idx_search() to find matching PKs. It then fetches each one through bt2_search() plus tapi_heap_read().
  • Sequential Scan. It falls back to tapi_scan_begin() and tapi_scan_next() over all rows. It evaluates the WHERE expression on each record through expr_evaluate().

UPDATE (Update.c)

UPDATE employees SET salary = 95000 WHERE name = 'Alice';

Data flow.

Parser (RPN) --> UpdateProcess()
                    |
                    v
            tapi_resolve(tblName)           -- catalog lookup
                    |
                    v
        [Identify matching rows]            -- WHERE scan or PK lookup
                    |
                    v
            ApplyUpdateToRow()
                |
                +-- bt2_search(pk_tree, pkKey)
                +-- tapi_heap_read(rid)     -- fetch current record
                +-- g_tx_undo_callback()    -- TX_OP_UPDATE with old data
                +-- [Modify fields]         -- apply SET column = value
                +-- [CHECK validation]      -- expr_evaluate on updated row
                +-- [Unique index check]    -- btree_search for conflicts
                +-- tapi_heap_update()      -- write modified record
                +-- bt2_update()            -- update PK tree if RowID moved
                +-- [Secondary idx update]  -- btree_delete old + btree_insert new

Key details.

  • In-place update versus relocated update. tapi_heap_update() first tries slot_update() in place. If the record grew and no longer fits, it deletes the old slot and inserts into a page with space. That returns a different RowID. When the RowID changes, bt2_update() updates the PK B+ tree's value pointer.
  • Secondary index maintenance. The SET clause may modify an indexed column. For each secondary index on a modified column, the function deletes the old composite key with btree_delete(path, old_value, pk). It then inserts the new one with btree_insert(path, new_value, pk).
  • CHECK constraint re-evaluation. After the field values change in memory, all CHECK constraints are re-evaluated against the updated row before it is persisted.

DELETE (Delete.c)

DELETE FROM employees WHERE dept = 'Eng';

Data flow.

                    |
                    v
        +---[WHERE present?]---+
        |                      |
   Expression WHERE       PK-based delete
        |                      |
        v                      v

[expr_evaluate loop]           |
[collect matching keys]        v

    for each match:       g_tx_undo_callback()
        |                      |

        +-- [sec idx cleanup]      v
        +-- tapi_heap_delete() tapi_heap_delete()
        +-- bt2_delete()       bt2_delete()

Two-phase WHERE delete.

  1. Phase 1, scan and collect. It iterates all records through tapi_scan_begin() and tapi_scan_next(). It evaluates the WHERE expression on each row. It collects matching PK keys into a dynamically allocated array called matchKeys. This avoids modifying the B+ tree while iterating it.
  2. Phase 2, delete collected. For each collected key, it looks up the RowID through bt2_search(). It reads the record for undo logging and index cleanup. It removes the row from secondary indexes. It then calls tapi_heap_delete() and bt2_delete().

DROP TABLE (Drop.c)

DROP TABLE employees;

Data flow.

Parser (RPN) --> DropTableProcess()
                    |
                    v
            tapi_resolve(tblName)           -- catalog lookup
                    |
                    v
            bt2_destroy(pk_tree)            -- free all PK B+ tree pages
                    |
                    v
            tapi_free_heap_pages()          -- walk heap chain, pgm_free_page each
                    |
                    v
            [Destroy secondary indexes]     -- cat_list_indexes + bt2_destroy each
                    |
                    v
            cat_drop_table(table_id)        -- remove from catalog

cat_drop_table() removes the table entry from the system catalog B+ trees. It also removes all associated column entries, all index entries, and all constraint entries. After DROP TABLE, all storage pages return to the free list. This covers PK tree nodes, heap data pages, and secondary index nodes.

TRUNCATE TABLE runs through TruncateTableProcess(). It preserves the table structure but deletes all data. It collects all PK keys through a B+ tree cursor scan. For each key it calls tapi_heap_delete() and bt2_delete(). The table's PK tree and heap pages stay allocated but empty.

Secondary indexes (Index.c)

Secondary indexes are B+ trees. Each key encodes both the indexed column value and the row's primary key. This ensures uniqueness at the B+ tree level.

Composite key format

"indexed_value" + \x1F + "pk_value"

The Unit Separator character is \x1F, ASCII 31. It is chosen for three reasons.

  • It sorts before all printable characters. This enables efficient prefix scans.
  • It never appears in user data.
  • A prefix scan on "indexed_value\x1F" finds all rows that match that indexed value.

For multi-column composite indexes, column values within the indexed portion are pipe-delimited. They look like "val1|val2\x1Fpk".

Core operations

sec_idx_search

int sec_idx_search(uint32_t root_page, const char *indexed_val,
                   char results[][256], int max_results);

This function performs a prefix scan on the secondary index B+ tree. It seeks to "indexed_val\x1F" through bt2_cursor_seek(). It then iterates forward with bt2_cursor_next() until the prefix no longer matches. It returns the PK portion of each matching key, which is the part after \x1F. A companion function sec_idx_search_rids() returns RowID values directly instead of PK strings.

sec_idx_insert

int sec_idx_insert(uint32_t *root_page, const char *indexed_val,
                   const char *pk, RowID heap_rid);

This function builds the composite key through sec_idx_make_key(). It inserts the key into the B+ tree with the heap RowID as the value. It updates *root_page if the tree's root splits.

sec_idx_delete

int sec_idx_delete(uint32_t *root_page, const char *indexed_val,
                   const char *pk);

This function builds the composite key and deletes it from the B+ tree. The PK component ensures only the exact row's entry is removed.

Legacy wrapper functions

The DML modules use wrapper functions. These encode index identity as a path string in the format "table_id:index_name:root_page".

  • btree_insert(path, key, pk). It parses the path, builds the composite key, calls bt2_insert(), and updates the catalog root page through cat_update_index_root() if it changed.
  • btree_search(path, key, results, max). It parses the path and delegates to sec_idx_search().
  • btree_delete(path, key, pk). It parses the path, builds the composite key, calls bt2_delete(), and updates the catalog if the root changed.

These wrappers give DML code a stable interface. They hide the composite key encoding and the catalog root-page tracking.

CREATE INDEX (CreateIndexProcess)

CREATE INDEX idx_dept ON employees(dept);
CREATE UNIQUE INDEX idx_email ON employees(email);
  1. Resolve table. It calls tapi_resolve() to verify the table exists and to retrieve column metadata.
  2. Verify columns. It checks that all specified columns exist in the schema. It supports composite indexes such as col1,col2.
  3. Check duplicates. It queries cat_list_indexes() to see if an index on the same columns already exists. It respects IF NOT EXISTS.
  4. Create B+ tree. It calls bt2_create() to allocate the root page for the index.
  5. Populate from existing data. It opens a TableScanCursor and iterates all rows. For each row it does the following.
    • It extracts the indexed column values from the record.
    • For UNIQUE indexes, it calls sec_idx_search() to check for duplicates. If a duplicate is found, it destroys the partially built tree and returns an error.
    • It builds the composite key indexed_value\x1Fpk_value and calls bt2_insert().
  6. Register in catalog. It calls cat_create_index() to store the index metadata. The metadata covers the name, root page, column list, and type 'U' or 'N'.

DROP INDEX (DropIndexProcess)

  1. It finds the index by name through cat_find_index_by_name().
  2. It rejects dropping PK indexes, where index_type == 'P'.
  3. It destroys the B+ tree through bt2_destroy().
  4. It removes the index from the catalog through cat_drop_index().

Transactions

EvoSQL implements single-connection ACID transactions. It uses an undo-log approach. The transaction subsystem lives in adaptor/transaction.h and adaptor/transaction.c.

Data structures

typedef struct UndoEntry {
    int   op_type;          /* TX_OP_INSERT (1) / TX_OP_UPDATE (2) / TX_OP_DELETE (3) */
    char *table;            /* table name */
    char *key;              /* primary key value */
    char *old_data;         /* previous record data (NULL for INSERT) */
    struct UndoEntry *next; /* linked list, stack order (newest first) */
} UndoEntry;

typedef struct {
    UndoEntry *head;        /* most recent entry (top of stack) */
    int        count;       /* number of entries */
} UndoLog;

The undo log is a singly-linked list used as a stack. Entries are pushed to the head. This way rollback processes them in reverse chronological order.

Session integration

Each client connection keeps a SessionCtx. It is defined in adaptor/query_executor.h.

typedef struct {
    char database[256];
    char schema[256];
    char username[256];
    int       in_transaction;   /* 1 if BEGIN was issued */
    int       tx_aborted;       /* 1 if error occurred in transaction */
    UndoLog  *undo_log;         /* undo entries for rollback */
} SessionCtx;

Transaction state transitions.

  • BEGIN. It sets in_transaction = 1 and allocates a fresh UndoLog.
  • COMMIT. It sets in_transaction = 0 and frees the UndoLog. Changes are now permanent.
  • ROLLBACK. It calls undo_log_rollback(), frees the log, and resets in_transaction = 0 and tx_aborted = 0.
  • Error in transaction. It sets tx_aborted = 1. Later queries are rejected with SQLSTATE 25P02. ROLLBACK is the exception.

DML hook mechanism

DML operations record undo entries through a function pointer callback. It is declared as g_tx_undo_callback in database.h.

void (*g_tx_undo_callback)(int op_type, const char *table,
                           const char *key, const char *data);

Before each query, safe_query_execute() in server.c checks whether the session is in a transaction.

  • In transaction. It sets g_tx_undo_callback = tx_undo_callback. It sets g_current_undo_log, which is thread-local, to the session's undo log.
  • Not in transaction. It sets both to NULL.

Each DML operation checks g_tx_undo_callback before it modifies data.

OperationUndo entry
INSERTTX_OP_INSERT, key = PK, old_data = NULL
UPDATETX_OP_UPDATE, key = PK, old_data = pre-update record
DELETETX_OP_DELETE, key = PK, old_data = deleted record

After the query completes, safe_query_execute() clears the callback and the thread-local undo log pointer.

Rollback

undo_log_rollback() traverses the undo log from head to tail. That is reverse chronological order. It applies the inverse operation for each entry.

Original opUndo action
TX_OP_INSERTbt2_search() to find the RowID, then tapi_heap_delete() and bt2_delete()
TX_OP_UPDATEbt2_search() to find the RowID, then tapi_heap_update() with the old data, then bt2_update() if the RowID moved
TX_OP_DELETEtapi_heap_insert() with the old data, then bt2_insert() to restore the PK index entry

Each undo action resolves the table through tapi_resolve() using the stored table name. This ensures it works with the correct catalog state.

Disconnect auto-rollback

Both protocol handlers check for open transactions when a client disconnects. The PG wire protocol handler is pg_handler.c. The EVO text protocol handler is evo_protocol.c. If session.in_transaction is true, the handler does the following.

  1. It allocates a temporary QueryContext. The per-query context is already freed.
  2. It restores the database and schema from the session.
  3. It calls undo_log_rollback().
  4. It frees the undo log and resets the session state.
info
This guarantees that uncommitted changes are never persisted when a client drops its connection.
