Page Manager and Slotted Pages

EvoSQL keeps all data in one global binary file made of fixed-size 4KB pages. The Page Manager owns this file and handles startup, page allocation, page free, and raw I/O. Slotted Pages store variable-length records inside a single data page using a slot directory layout.

The global file holds every database, table, index, and the system catalog. The Page Manager lives in page_mgr.h and page_mgr.c. It owns the file. It handles initialization, page allocation and free, and raw I/O. Slotted Pages live in slotted.h and slotted.c. They provide variable-length record storage inside individual data pages.

All disk I/O flows through the Clock Sweep buffer pool in buffer_pool.c. The page manager never calls read() or write() directly. It uses bp_read() and bp_write() instead.

Page Manager

File Structure

The global storage file is a sequence of 4096-byte pages. Page 0 is always the File Header. All remaining pages are available for allocation. When the server starts, pgm_init() opens the existing file or creates a fresh one. When it opens an existing file, it validates the magic number and the page size. When it creates a fresh file, it makes 64 pages, which is 256KB.

 Page 0          Page 1          Page 2             Page N-1
┌──────────┐   ┌──────────┐   ┌──────────┐       ┌──────────┐
│FileHeader│   │  Data /  │   │  Data /  │  ...  │  Data /  │
│ (4096 B) │   │  Free /  │   │  Free /  │       │  Free /  │
│          │   │  BTree   │   │  BTree   │       │  BTree   │
└──────────┘   └──────────┘   └──────────┘       └──────────┘

Common Page Header

Every page in the file begins with a 16-byte header.

typedef struct {
    uint8_t  page_type;    /* PageType enum                       */
    uint8_t  flags;        /* reserved, must be 0                 */
    uint16_t free_space;   /* bytes of free space on this page    */
    uint32_t page_id;      /* this page's own number (self-check) */
    uint32_t next_page;    /* free list chain or B+ leaf chain    */
    uint32_t prev_page;    /* B+ tree leaf backward pointer       */
} PageHeader;              /* 16 bytes                            */

The page_type field uses one of five values.

typedef enum {
    PAGE_FREE       = 0,   /* available for allocation           */
    PAGE_DATA       = 1,   /* slotted page for row storage       */
    PAGE_BTREE_INT  = 2,   /* B+ tree internal node              */
    PAGE_BTREE_LEAF = 3,   /* B+ tree leaf node                  */
    PAGE_OVERFLOW   = 4    /* overflow for large records          */
} PageType;

The next_page field has two jobs. On free pages it chains into the free list. On B+ tree leaf pages it links to the next leaf in key order. The prev_page field links B+ tree leaves backward.

File Header (Page 0)

Page 0 holds the global file header. It takes up an entire page. A reserved array pads it to fill exactly EVO_PAGE_SIZE bytes.

#define CATALOG_ROOT_SLOTS 16

typedef struct {
    uint32_t magic;                             /* EVO_MAGIC                          */
    uint16_t version;                           /* EVO_FILE_VERSION                   */
    uint16_t page_size;                         /* EVO_PAGE_SIZE (4096)               */
    uint32_t total_pages;                       /* total pages in file                */
    uint32_t free_list_head;                    /* first free page (0 = none)         */
    uint32_t catalog_roots[CATALOG_ROOT_SLOTS]; /* root pages for system catalog trees */
    uint32_t next_table_id;                     /* auto-increment counter             */
    uint32_t next_schema_id;                    /* auto-increment counter             */
    uint32_t next_db_id;                        /* auto-increment counter             */
    uint8_t  reserved[EVO_PAGE_SIZE - 92];      /* pad to full page                   */
} FileHeader;

The fixed fields sum to exactly 92 bytes. The math is 4 + 2 + 2 + 4 + 4 + (16 * 4) + 4 + 4 + 4 = 92. The reserved array takes the remaining bytes. That is 4096 - 92 = 4004 bytes.

Catalog root slots store the root page number of each system catalog B+ tree. The engine defines eight catalog IDs.

typedef enum {
    CAT_SYS_DATABASES   = 0,
    CAT_SYS_SCHEMAS     = 1,
    CAT_SYS_TABLES      = 2,
    CAT_SYS_COLUMNS     = 3,
    CAT_SYS_INDEXES     = 4,
    CAT_SYS_CONSTRAINTS = 5,
    CAT_SYS_USERS       = 6,
    CAT_SYS_GRANTS      = 7,
    CAT_MAX             = 8
} CatalogID;

Slots 8 through 15 are reserved for future use.

Auto-increment counters give globally unique IDs. They are next_table_id, next_schema_id, and next_db_id. They start at 1 in a fresh file. They are incremented atomically through pgm_next_id().

Initialization

pgm_init(filepath) is called once at server startup. It runs after the buffer pool has been initialized with bp_init(). It follows this logic.

  1. If the file already exists, open it and read page 0. Validate that magic == EVO_MAGIC and page_size == EVO_PAGE_SIZE. If validation fails, return an error.
  2. If the file does not exist (ENOENT), create it with O_RDWR | O_CREAT | O_TRUNC. Extend it to PGM_INITIAL_PAGES * EVO_PAGE_SIZE bytes through ftruncate(). Write the initial FileHeader to page 0. Then chain pages 1 through PGM_INITIAL_PAGES - 1 into the free list.
  3. If it is already initialized, this is a re-entrant call. Return immediately.

pgm_shutdown() flushes any dirty file header. It calls bp_flush_all() to write all dirty buffer pool pages to disk. Then it closes the file descriptor.

Page Allocation

pgm_alloc_page(page_type) pops a page from the head of the free list.

  1. If free_list_head == 0, the list is empty. Grow the file first.
  2. Read the page at free_list_head. Extract next_page from its header. This becomes the new free_list_head.
  3. Zero-fill the page buffer. Set page_type to the requested type. Write it back through the buffer pool.
  4. Flush the updated file header.
  5. Return the allocated page number. A return value of 0 indicates an error. Page 0 is never allocatable.

Page Deallocation

pgm_free_page(page_no) prepends the page to the free list.

  1. Refuse to free page 0. It is always the file header.
  2. Zero-fill the page. Set page_type = PAGE_FREE. Set next_page to the current free_list_head.
  3. Write the page through the buffer pool.
  4. Update free_list_head to point to this page.
  5. Flush the file header.

The free list is a singly-linked chain through PageHeader.next_page.

 FileHeader                 Page 7           Page 3           Page 12
┌────────────┐            ┌──────────┐     ┌──────────┐     ┌──────────┐
│free_list   │            │PAGE_FREE │     │PAGE_FREE │     │PAGE_FREE │
│ _head = 7 ─┼──────────>│next = 3 ─┼────>│next = 12─┼────>│next = 0  │
└────────────┘            └──────────┘     └──────────┘     └──────────┘
                                                             (end of list)

File Growth

When pgm_alloc_page() finds the free list empty, the internal grow_file() function extends the file.

  1. Compute the growth amount. It is 50% of the current total_pages. It is at least PGM_GROW_MIN, which is 16 pages.
  2. Call ftruncate() to extend the file to the new size.
  3. Chain all newly created pages into the free list through chain_free_pages(). Each page is zero-filled with page_type = PAGE_FREE and linked to the next. The last page in the batch points to the previous free_list_head.
  4. Update total_pages and free_list_head in the file header.

Raw Page I/O

int pgm_read_page(uint32_t page_no, void *buf);
int pgm_write_page(uint32_t page_no, const void *buf);

These are thin wrappers over bp_read() and bp_write(). They compute the byte offset as page_no * EVO_PAGE_SIZE. They transfer a full 4096-byte page through the buffer pool. They return 0 on success and -1 on error. The buffer must be at least EVO_PAGE_SIZE bytes.

pgm_flush() writes the file header if it is dirty. It then calls bp_flush_all() to force all cached pages to disk.

Catalog Root Management

uint32_t pgm_get_catalog_root(CatalogID id);
void     pgm_set_catalog_root(CatalogID id, uint32_t page_no);

These read and write entries in the catalog_roots[] array of the file header. A system catalog B+ tree is created for the first time. Its root page number is stored here. This lets the engine find it on later startups. Setting a root flushes the header right away. The system catalog itself is described on the System Catalog page.

Auto-Increment IDs

uint32_t pgm_next_id(int id_type);  /* 0 = table, 1 = schema, 2 = database */

This returns the current value and increments the counter atomically. The file header is flushed after each call. This makes the change durable.

Thread Safety

A single pthread_mutex_t named g_pgm_lock serializes all page manager operations. This covers allocation, free, catalog root access, and ID generation. Raw I/O does not acquire the page manager mutex. pgm_read_page and pgm_write_page rely on the buffer pool's own mutex for thread safety.

Global State

The page manager is a singleton. It maintains four pieces of state.

  • g_global_fd is the file descriptor for the global storage file.
  • g_header is the in-memory copy of the FileHeader, which is page 0.
  • g_header_dirty is a flag for unsaved header changes.
  • g_pgm_lock is the mutex for thread safety.

The file descriptor is exposed through pgm_get_fd(). This is for callers that need direct buffer pool access.

Slotted Pages

Slotted pages manage variable-length records inside a single 4KB page. The design follows a classic database textbook layout. A slot directory grows forward from the top of the page. Record data grows backward from the bottom. This gives stable record addressing through slot indices even as records are inserted and deleted.

Page Layout

Byte 0                                                    Byte 4095
┌──────────────┬───────────────┬────────┬────────┬─  ─┬──────────┐
│  PageHeader  │ SlottedHeader │ Slot 0 │ Slot 1 │ .. │          │
│   (16 B)     │    (6 B)      │ (4 B)  │ (4 B)  │    │          │
├──────────────┴───────────────┴────────┴────────┴─  ─┤          │
│                                                      │          │
│                     FREE SPACE                       │          │
│                                                      │          │
│         <── free_offset              free_end ──>    │          │
├──────────────────────────────────────────────────────┤          │
│              │ Record 1 │ Record 0 │                            │
│              │ (var)    │ (var)    │                            │
└──────────────┴──────────┴──────────┴────────────────────────────┘

  Slot directory grows FORWARD (toward higher offsets)  ──────>
  Record area grows BACKWARD (toward lower offsets)     <──────

When a slotted page is initialized through slot_page_init(), three values are set.

  • free_offset = PAGE_HEADER_SIZE + SLOTTED_HEADER_SIZE = byte 22, just past the slotted header.
  • free_end = EVO_PAGE_SIZE = byte 4096, the end of the page.
  • Free space = 4096 - 22 = 4074 bytes, before any slots are allocated.

The free space is the gap between free_offset and free_end. The free_offset marks the end of the slot directory. The free_end marks the start of the record area. As slots are added, free_offset advances forward by 4 bytes each. As records are placed, free_end retreats backward by the record size.

SlottedHeader

typedef struct {
    uint16_t num_slots;     /* total slot entries (including deleted) */
    uint16_t free_offset;   /* byte offset of first free byte after slot directory */
    uint16_t free_end;      /* byte offset where next record will be placed */
} SlottedHeader;            /* 6 bytes */
  • num_slots counts all slots ever created on this page. This includes slots marked as deleted. It only grows. It never shrinks.
  • free_offset tracks the end of the slot directory. It equals PAGE_HEADER_SIZE + SLOTTED_HEADER_SIZE + (num_slots * SLOT_ENTRY_SIZE) when no deleted slots have been reused.
  • free_end tracks the beginning of the record region. Records are placed just below this offset. Then free_end is decremented.

Slot Entry

typedef struct {
    uint16_t offset;   /* byte offset of record within page (0 = deleted) */
    uint16_t length;   /* record length in bytes */
} SlotEntry;           /* 4 bytes */

A slot with offset == 0 is treated as deleted or empty. This works because byte 0 is taken by the PageHeader. No valid record can start there.

Slot entries are stored one after another starting at byte 22. That is after the PageHeader and the SlottedHeader. Slot i is at byte offset 22 + (i * 4).

RowID

typedef struct {
    uint32_t page_no;
    uint16_t slot_idx;
} RowID;               /* 6 bytes */

A RowID uniquely identifies any record in the storage file. The pair of page number and slot index is stable across compaction. Slot indices are never renumbered. A RowID with page_no == 0 is the "deleted" sentinel. Page 0 is always the file header and never stores records.

Maximum Record Size

#define SLOT_MAX_RECORD  (EVO_PAGE_SIZE - PAGE_HEADER_SIZE - SLOTTED_HEADER_SIZE - SLOT_ENTRY_SIZE)

This evaluates to 4096 - 16 - 6 - 4 = 4070 bytes. A record of this size would use the entire page. That is one slot entry plus the record data. It leaves zero free space.

Record Operations

Insert (slot_insert)

int slot_insert(void *page_buf, const void *record, uint16_t rec_len);

It returns the slot index on success. It returns -1 if the record does not fit. The algorithm runs as follows.

  1. Reject zero-length records. Reject records larger than SLOT_MAX_RECORD.
  2. Scan the slot directory for a deleted slot. A deleted slot has offset == 0 and length == 0. If one is found, reuse it. No new slot overhead is needed.
  3. Calculate the space required. That is rec_len bytes for the record. Add 4 bytes for a new slot entry if a deleted slot is not being reused.
  4. If the contiguous free gap (free_end - free_offset) is too small, try slot_compact() to defragment the page. If it is still not enough, return -1.
  5. Place the record at free_end - rec_len and update free_end.
  6. Write the slot entry. It is either the reused slot or a newly appended entry in the directory.
  7. Update PageHeader.free_space.

Read (slot_read)

int slot_read(const void *page_buf, uint16_t slot_idx,
              void *out_buf, uint16_t out_size);

It returns the actual record length on success. It returns -1 if the slot index is out of range or the slot is deleted. It copies at most out_size bytes into out_buf. If the record is larger than out_size, it is truncated. The full length is still returned. The caller can then allocate a larger buffer and retry.

Delete (slot_delete)

int slot_delete(void *page_buf, uint16_t slot_idx);

It returns 0 on success. It returns -1 if the slot is already deleted or out of range.

  1. Check if the record being deleted is the physically last record. That is when se->offset == free_end. If so, advance free_end by se->length to reclaim the space right away.
  2. Mark the slot as deleted. Set offset = 0 and length = 0.
  3. Update PageHeader.free_space.

Records that are not physically last leave a gap. This is fragmentation. slot_compact() reclaims this space.

Update (slot_update)

int slot_update(void *page_buf, uint16_t slot_idx,
                const void *new_record, uint16_t new_len);

It returns 0 on success. It returns -1 if the record does not fit.

  • If new_len <= old_length, the record is overwritten in place. If it is shorter than the original, the tail bytes become wasted space. Compaction can reclaim them. The slot's length is updated to new_len.
  • If new_len > old_length, the function returns -1. The caller must do a delete-then-insert. This may land on a different page if the current page lacks space. The table API layer uses this overflow handling pattern. See Table API and DML.

Free Space Management

Contiguous Free Space

slot_free_space() returns the contiguous gap between the end of the slot directory and the start of the record area.

uint16_t slot_free_space(const void *page_buf);
// Returns: free_end - free_offset (or 0 if they've crossed)

This is the maximum amount of data that can be stored without compaction. Inserting a record consumes record bytes. It may also consume 4 bytes for a new slot entry.

Fragmentation

Fragmentation happens in two cases.

  • A record in the middle of the record area is deleted. This leaves a gap.
  • A record is updated to a shorter length. This leaves unused tail bytes.

These gaps reduce the effective capacity of the page. They do not reduce the value returned by slot_free_space(). That value only measures the contiguous gap. Compaction reclaims this space.

Compaction (slot_compact)

void slot_compact(void *page_buf);

It defragments the record area. It repacks all active records toward the end of the page and removes gaps.

  1. Allocate a temporary buffer.
  2. Walk all slots in order. For each active slot that is not deleted, copy its record to the temporary buffer. Pack records from the end of the page backward.
  3. Copy the compacted records back into the page buffer.
  4. Update each slot's offset to its new position.
  5. Update free_end to reflect the new start of the record area.
info
Compaction preserves slot indices. It never renumbers slots. It never removes deleted slot entries from the directory. This matters because external references point to specific slot indices. Those references include B+ tree leaf entries and RowIDs.

Compaction is triggered automatically by slot_insert(). This happens when the contiguous free gap is too small but the page has enough total free space once fragmented gaps are counted.

Active Record Count

uint16_t slot_active_count(const void *page_buf);

It returns the number of slots that are not deleted. It scans the directory and counts entries where offset != 0.

Caller Responsibilities

All slotted page functions work on an in-memory page buffer. They do not read from or write to disk. The caller must do two things.

  1. Read the page into a buffer through pgm_read_page() before operating on it.
  2. Call pgm_write_page() after modifying the page. This persists changes through the buffer pool.

This design gives callers full control over I/O batching. It also allows several modifications to a page before a single write.

Detailed Layout Example

Three records are inserted. R0 is 100 bytes. R1 is 50 bytes. R2 is 200 bytes. Then R1 is deleted.

Byte:  0          16   22   26   30   34                           3746  3846  3946 4096
       ┌──────────┬────┬────┬────┬────┬─────────────────────────────┬─────┬─────┬─────┐
       │PageHeader│Slot│ S0 │ S1 │ S2 │        FREE SPACE           │ R2  │(gap)│ R0  │
       │  (16 B)  │Hdr │off │DEL │off │                             │200B │ 50B │100B │
       │          │6 B │len │0,0 │len │                             │     │     │     │
       └──────────┴────┴────┴────┴────┴─────────────────────────────┴─────┴─────┴─────┘
                       │              ↑                             ↑
                       │         free_offset = 34            free_end = 3746
                       │
                       num_slots = 3

  S0: offset=3946, length=100    (points to R0)
  S1: offset=0,    length=0      (deleted)
  S2: offset=3746, length=200    (points to R2)

  The 50-byte gap at 3846-3896 is fragmented space from the deleted R1.
  It will be reclaimed when slot_compact() runs.

After compaction the page looks like this.

Byte:  0          16   22   26   30   34                                3796  3996 4096
       ┌──────────┬────┬────┬────┬────┬──────────────────────────────────┬─────┬─────┐
       │PageHeader│Slot│ S0 │ S1 │ S2 │           FREE SPACE             │ R2  │ R0  │
       │  (16 B)  │Hdr │off │DEL │off │                                  │200B │100B │
       │          │6 B │len │0,0 │len │                                  │     │     │
       └──────────┴────┴────┴────┴────┴──────────────────────────────────┴─────┴─────┘
                       │              ↑                                  ↑
                       │         free_offset = 34                 free_end = 3796
                       │
                       num_slots = 3 (unchanged)

  S0: offset=3996, length=100    (updated to new position)
  S1: offset=0,    length=0      (still deleted -- slot not removed)
  S2: offset=3796, length=200    (updated to new position)

  The 50-byte gap is gone. Free space increased from 3712 to 3762 bytes.
