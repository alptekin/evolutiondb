SELECT

Read rows from one or more tables, subqueries, or set-returning sources. You can also filter, group, order, and limit the rows.

Synopsis

SELECT [ ALL | DISTINCT | DISTINCTROW ] [ HIGH_PRIORITY ] [ STRAIGHT_JOIN ]
       [ SQL_SMALL_RESULT ] [ SQL_BIG_RESULT ] [ SQL_CALC_FOUND_ROWS ]
    { * | select_expr [ [ AS ] output_alias ] [, ...] }
    [ INTO new_table ]
    FROM from_item [, ...]
    [ FOR SYSTEM_TIME AS OF TRANSACTION xid ]
    [ WHERE condition ]
    [ GROUP BY expression [ ASC | DESC ] [, ...] [ WITH ROLLUP ] ]
    [ HAVING condition ]
    [ ORDER BY order_item [, ...] ]
    [ LIMIT { count | offset , count | count OFFSET offset } ]
    [ FOR UPDATE | FOR SHARE [ SKIP LOCKED ] ]
    [ INTO column [, ...] ]

where from_item is one of:

    table_name [ [ AS ] alias ] [ index_hint ]
    ONLY table_name [ [ AS ] alias ]
    schema.table_name [ [ AS ] alias ]
    ( select_stmt ) [ AS ] alias
    LATERAL ( select_stmt ) [ AS ] alias
    UNNEST ( array_expr ) [ [ AS ] alias [ ( col ) ] ]
    from_item [ INNER | CROSS ] JOIN from_item [ join_condition ]
    from_item STRAIGHT_JOIN from_item [ ON condition ]
    from_item { LEFT | RIGHT } [ OUTER ] JOIN from_item join_condition
    from_item NATURAL [ { LEFT | RIGHT } [ OUTER ] ] JOIN from_item

and join_condition is one of:

    ON condition
    USING ( column [, ...] )

and order_item is one of:

    column_name [ ASC | DESC ]
    table.column_name [ ASC | DESC ]
    column_position [ ASC | DESC ]
    vector_column <=> 'vector_literal' [ ASC | DESC ]

and index_hint is one of:

    { USE | IGNORE | FORCE } KEY [ FOR JOIN ] ( index [, ...] )

Description

SELECT reads rows from the database and returns them as a result set. EvolutionDB accepts two main shapes. Both are defined by the select_stmt rule in evoparser.y.

  1. A projection-only form, SELECT select_expr_list with no FROM. It evaluates the listed expressions once and returns a single row. This is how SELECT 1 and SELECT NOW() are handled.
  2. The full table query form. It adds FROM and the optional WHERE, GROUP BY, HAVING, ORDER BY, LIMIT, and row-locking clauses.

A table query is processed in this order.

  1. Each from_item in FROM is scanned. Then joins are applied. The leftmost item is the driving relation. The other items default to INNER JOIN unless you give an explicit join type.
  2. If WHERE is present, its condition is checked against each candidate row. Rows that are not true are dropped.
  3. If GROUP BY is present, the remaining rows are collapsed into groups. Aggregate functions in the select list are computed for each group. WITH ROLLUP adds super-aggregate summary rows.
  4. If HAVING is present, its condition filters the groups.
  5. The select list is evaluated to form the output rows. Then DISTINCT and DISTINCTROW remove duplicate rows.
  6. If ORDER BY is present, rows are sorted. Non-numeric values are compared as text. Values that parse cleanly as numbers are compared as numbers. See CompareRecords in Select.c.
  7. LIMIT and OFFSET limit how many rows are returned.
  8. If FOR UPDATE or FOR SHARE is present, the returned rows are locked.

The simple single-table path inside the engine works in a few steps. It finds the table in the catalog. It then does a primary-key lookup when a single equality on the PK is given, or a sequential table scan through the buffer pool. It decodes the binary tuple format. Finally it prints the matching rows.

Note. The engine has a built-in text renderer for the SELECT * path. It buffers rows in fixed-size storage so it can sort ORDER BY in memory. The normal way clients connect is through the dual-protocol server. In that case the adaptor's query executor handles result streaming and richer expression work.

Parameters

ALL / DISTINCT / DISTINCTROW
These control how duplicate rows are handled. ALL is the default and returns every row. DISTINCT removes duplicate result rows. DISTINCTROW means the same thing. If you give the same option twice, you get a parse error called "duplicate … option".
HIGH_PRIORITY, STRAIGHT_JOIN, SQL_SMALL_RESULT, SQL_BIG_RESULT, SQL_CALC_FOUND_ROWS
These are MySQL-compatibility select options. The grammar accepts them as leading keywords. They parse without error so MySQL-style statements still load. In this position, only DISTINCT and DISTINCTROW change the result.
* / select_expr [ [AS] output_alias ]
This is the select list. * returns all columns of the source. Each other item is an expression. It can be a column reference, a literal, or a call to an arithmetic, string, aggregate, or window function. You can rename it with AS alias. The AS keyword is optional.
from_item (FROM)
One or more sources, separated by commas. A comma makes an implicit cross or inner join. A source can be a table, written as table_name or schema.table_name. It can also be a subquery in parentheses with a name, a LATERAL subquery, an UNNEST(array_expr) set-returning source, a join in parentheses, or an explicit join. The join forms are shown below.
ONLY table_name
In a table source, ONLY limits the scan to the named parent table. It hides rows that would otherwise come from child tables that inherit from it.
JOIN clauses
EvolutionDB accepts these joins. They are INNER JOIN, CROSS JOIN, plain JOIN which is treated as inner, STRAIGHT_JOIN, LEFT [OUTER] JOIN, RIGHT [OUTER] JOIN, and NATURAL joins. A NATURAL join may also use LEFT or RIGHT OUTER. You give the join predicate with ON condition or USING ( column [, ...] ). Outer joins need an explicit condition.
index_hint
An optional per-table hint. It can be USE KEY, IGNORE KEY, or FORCE KEY. Each one takes an index list in parentheses. You can also scope it to joins with FOR JOIN.
FOR SYSTEM_TIME AS OF TRANSACTION xid
This is a time-travel qualifier. It runs the query against the database snapshot at the given transaction id xid instead of the current snapshot.
WHERE condition
A boolean expression. Only rows where it is true are passed on. It supports comparisons, AND/OR/NOT, IN (...) and IN (subquery), EXISTS (subquery), scalar and quantified subqueries, and vector-distance operators.
GROUP BY expression [ASC | DESC] [..] [WITH ROLLUP]
This groups rows that share the same values over the listed expressions. Aggregate functions are then computed for each group. An optional ASC or DESC can follow each grouping expression. WITH ROLLUP adds super-aggregate subtotal and grand-total rows.
HAVING condition
A boolean condition applied to grouped rows. It usually refers to aggregates. It filters whole groups after GROUP BY.
ORDER BY order_item [..]
A list of sort keys. Each item is a column name, a qualified table.column, a 1-based column position such as ORDER BY 2, or a vector nearest-neighbor expression column <=> 'vector_literal' that uses the cosine-distance operator <=>. Each key can be followed by ASC, the default, or DESC. In ORDER BY expression position only the <=> operator is allowed. Other comparison operators are rejected.
LIMIT { count | offset, count | count OFFSET offset }
This limits how many rows are returned. Three forms are accepted. They are LIMIT count, the MySQL two-argument form LIMIT offset, count, and LIMIT count OFFSET offset.
FOR UPDATE | FOR SHARE [ SKIP LOCKED ]
This locks the returned rows at row level. FOR UPDATE takes an exclusive row lock. FOR SHARE takes a shared lock. The optional SKIP LOCKED modifier skips rows that are already locked instead of waiting for them.
INTO column [..]
This is the trailing variable-target list form. It sends the selected values into the named targets. It is different from the table-creating SELECT ... INTO new_table FROM ... form described below.
SELECT.. INTO new_table FROM..
This is a PostgreSQL-style table-creating form. The query result is stored in a brand-new table new_table. Its columns are inferred from the select list. Inside the engine this uses the CREATE-TABLE-AS and inferred-schema path.

Examples

This example selects every column from a table.

SELECT * FROM employees;

This example returns specific columns with an output alias and filters the rows.

SELECT name, salary AS pay
FROM employees
WHERE dept_id = 10;

This example eliminates duplicates and sorts the result in descending order.

SELECT DISTINCT dept_id
FROM employees
ORDER BY dept_id DESC;

Aggregate per group, filter the groups, and keep only the top rows.

SELECT dept_id, COUNT(*) AS headcount, AVG(salary) AS avg_pay
FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 3
ORDER BY headcount DESC
LIMIT 5;

This example uses an inner join with an ON condition.

SELECT e.name, d.dept_name
FROM employees AS e
INNER JOIN departments AS d ON e.dept_id = d.id;

This example uses a left outer join with USING.

SELECT e.name, d.dept_name
FROM employees e
LEFT OUTER JOIN departments d USING (dept_id);

This example uses the MySQL-style two-argument LIMIT form (offset, count) for paging.

SELECT name FROM employees ORDER BY name LIMIT 20, 10;

This example runs a vector nearest-neighbor search using the cosine-distance operator.

SELECT id FROM documents
ORDER BY embedding <=> '[0.12, 0.84, 0.05]'
LIMIT 5;

This example runs a time-travel query against a past transaction snapshot.

SELECT * FROM accounts
FOR SYSTEM_TIME AS OF TRANSACTION 4200;

This example locks the matched rows for update. It skips rows that are already locked.

SELECT * FROM jobs
WHERE status = 'pending'
FOR UPDATE SKIP LOCKED;

This example materializes a query result into a new table using the PostgreSQL-style SELECT INTO form.

SELECT name, salary
INTO high_earners
FROM employees
WHERE salary > 100000;

This example expands an array into rows using UNNEST.

SELECT v FROM UNNEST(ARRAY[1, 2, 3]) AS t(v);

Compatibility

EvolutionDB's SELECT mixes SQL-standard, PostgreSQL, and MySQL spellings.

  • SQL standard / PostgreSQL. Several parts follow standard and PostgreSQL conventions. These are DISTINCT, the join family with INNER, CROSS, LEFT/RIGHT [OUTER], NATURAL, and ON/USING, then GROUP BY, HAVING, and ORDER BY, LATERAL subqueries, UNNEST, row locking with FOR UPDATE or FOR SHARE and SKIP LOCKED, the SELECT ... INTO new_table table-creating form, and the SQL:2011 temporal opener FOR SYSTEM_TIME AS OF. Note that the temporal clause takes TRANSACTION xid, a transaction id, instead of a timestamp.
  • MySQL extensions. Several parts are MySQL-style. These are DISTINCTROW, the leading options HIGH_PRIORITY, STRAIGHT_JOIN, SQL_SMALL_RESULT, SQL_BIG_RESULT, and SQL_CALC_FOUND_ROWS, the two-argument LIMIT offset, count form, WITH ROLLUP on GROUP BY, and the index hints USE/IGNORE/FORCE KEY. Several of the leading options are accepted for source compatibility but do not change the result.
  • EvolutionDB-specific. The vector-distance operator <=> means cosine distance. It is allowed in ORDER BY expression position for nearest-neighbor search. Other comparison operators are rejected there.
  • Not part of this command. The grammar's select_stmt rule does not define some clauses. These are WITH common table expressions, set operations such as UNION/INTERSECT/EXCEPT, WINDOW-clause named windows, FETCH FIRST, and OFFSET as a standalone clause. Only the LIMIT forms shown above are supported. Window functions appear as function expressions in the select list, not as a separate top-level clause.
