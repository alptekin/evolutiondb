GRANT

Give a user access rights on a database, schema, or table. You can also make a user a member of a role.

Synopsis

-- Grant privileges on a database object:
GRANT privilege [ , ... ]
    ON { DATABASE | SCHEMA | TABLE } object_name
    TO username
    [ WITH GRANT OPTION ]

-- Grant membership in a role (no ON clause):
GRANT role_name TO username

where privilege is one of:
    SELECT | INSERT | UPDATE | DELETE | CREATE | DROP
    ALL [ PRIVILEGES ]

Description

GRANT gives privileges to a user. EvolutionDB has two forms of this command. The forms differ by whether an ON clause is present.

Object privileges. The first form uses ON. It grants one or more privileges on a single database object to a named user. The object can be a DATABASE, a SCHEMA, or a TABLE. Each privilege maps to a SQL operation. These are SELECT, INSERT, UPDATE, DELETE, CREATE, and DROP. ALL is short for all six. You can also write it as ALL PRIVILEGES.

Role membership. The second form has no ON. It makes username a member of role_name. When EvolutionDB checks privileges, a user gets every privilege held by its roles. The user also keeps its own privileges.

Additive, merging semantics. You can grant privileges that overlap a grant that already exists for the same user and object. This does not replace the old grant. It merges with it. The new privilege bits are OR-ed into the bits that are already there. WITH GRANT OPTION is turned on if you ask for it. A re-grant never clears an existing grant option. A user's own grants and grants made to the special grantee PUBLIC are both additive. A user with no privileges still gains from any matching PUBLIC grant.

How privileges are resolved. EvolutionDB checks access with a waterfall. It starts at the most specific scope. First it looks for a matching grant at TABLE scope. Then it checks SCHEMA scope. Then it checks DATABASE scope. A DATABASE grant on the wildcard name * applies to every database. EvolutionDB then runs the same waterfall for PUBLIC. Last, it runs it for each role the user belongs to. If any level gives the requested privilege, access is allowed.

Who may run GRANT. The built-in admin user is a superuser. It may grant anything. It always holds WITH GRANT OPTION. A non-admin user may only pass on privileges it holds with the grant option on the same object. If not, the statement fails with "Permission denied". Only admin may grant role membership.

EvolutionDB comes with one grant already set up. The admin user holds ALL on DATABASE *. New databases have no default PUBLIC grants. So non-admin users start with no access. They get access only after something is granted to them. The grant can be direct, through PUBLIC, or through a role.

Parameters

privilege
The privilege to grant. It is one of SELECT, INSERT, UPDATE, DELETE, CREATE, or DROP. Privilege names are not case-sensitive. You can list several privileges. Separate them with commas. EvolutionDB quietly ignores any word it does not know. If the list ends up empty, the statement fails with "Invalid privilege specification".
ALL [ PRIVILEGES ]
Short for all six privileges. These are SELECT, INSERT, UPDATE, DELETE, CREATE, and DROP. The PRIVILEGES keyword is optional. EvolutionDB accepts it for SQL-standard compatibility. It has no extra effect.
ON DATABASE | SCHEMA | TABLE
Picks the scope the grant applies to. The scope is the kind of object. Exactly one of these keywords must follow ON. In the waterfall, TABLE is the most specific scope. DATABASE is the broadest.
object_name
The name of the object you grant on. You may write it bare, single-quoted, or double-quoted. For a DATABASE grant the name is a database name. You can also use the wildcard *, which means all databases. For finer scopes the name is stored as given. Checks build the lookup key from the session's database, schema, and table. At table scope this key is db.schema.table. So write the name to match the level you want.
TO username
The user that gets the privileges or role membership. You may write it bare or single-quoted. Use the special name PUBLIC to grant to every user. This grant is additive. For an object grant the target user does not need to exist. For a role grant the target must be a user that exists. It cannot be a role.
WITH GRANT OPTION
Lets the recipient grant the same privileges on the same object to other users. It is valid only in the object-privilege form. Without it, the recipient may use the privileges but not pass them on. A non-admin grantor must hold the grant option to give it.
role_name (role-membership form)
In the form with no ON, this is the name of an existing role. Its membership is given to username. The role must exist. The target must be a user. The user must not already be a member. If any of these is not true, the statement fails. Only admin may run this form.

Examples

This example gives a user read access on a table.

GRANT SELECT ON TABLE orders TO analyst;

This example gives several privileges at once and lets the recipient pass them on.

GRANT SELECT, INSERT, UPDATE ON TABLE orders TO app_user WITH GRANT OPTION;

This example gives every privilege on a schema to everyone.

GRANT ALL PRIVILEGES ON SCHEMA reporting TO PUBLIC;

Give create and drop rights across all databases. This uses the wildcard scope.

GRANT CREATE, DROP ON DATABASE '*' TO dba;

This example gives database-level read access to a single database.

GRANT SELECT ON DATABASE testdb TO readonly;

Grant role membership. There is no ON clause. Run this as admin.

GRANT reporting_role TO analyst;

This example shows how to look at the grants you made.

SHOW GRANTS FOR analyst;

Compatibility

EvolutionDB's GRANT follows the idea of the SQL standard and PostgreSQL. It is made simpler on purpose. A dedicated command interceptor handles the statement. The main SQL grammar does not.

  • Same as the standard and PostgreSQL. The privilege set SELECT, INSERT, UPDATE, DELETE matches PostgreSQL. So does the ALL [PRIVILEGES] short form. So does the TO target with the special grantee PUBLIC. So does WITH GRANT OPTION. So do the additive PUBLIC rules.
  • EvolutionDB specifics. CREATE and DROP are object privileges. They go in the same grant list as DML privileges. The object scope is one of exactly three keywords. These are DATABASE, SCHEMA, and TABLE. A DATABASE grant may use the wildcard name * to cover all databases. Privilege resolution uses a waterfall from most specific to least specific. The order is TABLE → SCHEMA → DATABASE. It also uses role inheritance. The admin user is an implicit superuser.
  • Differences and clauses that are not supported. There is no ON ALL TABLES IN SCHEMA. There are no column-level privileges. There are no privileges on sequences, functions, types, or foreign objects. There is no GRANTED BY clause. There is no WITH ADMIN OPTION for role grants. The grant object must be a single named object. Lists of objects are not accepted. Privilege checks use the username and roles. For object grants the target user does not need to exist first.
