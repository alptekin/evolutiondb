LISTEN / UNLISTEN / NOTIFY

Subscribe to a named channel, stop subscribing, and send messages on it.

Synopsis

LISTEN channel [ SELF ]

UNLISTEN { channel | * }

NOTIFY channel [ , 'payload' ]

Description

LISTEN signs up the current session to listen on the notification channel named channel. NOTIFY sends a notification to every session that is listening on the same channel. You can also send an optional payload string with it. UNLISTEN removes the current session from one channel. Use * to remove it from all channels.

The notification registry is kept only in server memory. It is shared between both front-end protocols. Connections that speak the PostgreSQL wire protocol use port 5433. Connections that speak the native EVO text protocol use port 9967. Both kinds of connection can listen and notify each other through the same channel names. When a connection closes, its listens are dropped automatically.

Channel names are matched without regard to case. They are folded to lower case for lookup. A channel name may be at most 63 bytes long. Longer names are rejected.

Transactional delivery. NOTIFY is commit-gated. A NOTIFY that runs inside an explicit transaction block is not delivered right away. Its channel and payload are queued on the session. The messages are sent only when the transaction commits. If the transaction rolls back, the queued notifications are dropped and nothing is sent. In auto-commit mode there is no open BEGIN. The statement commits on its own. So the notification is sent as soon as the NOTIFY statement finishes. LISTEN and UNLISTEN work differently. They take effect right away when the statement runs. They are not held back until commit.

Self-notification. By default a session does not receive the notifications it sends on a channel it listens to itself. You can opt in to same-session delivery. To do this, add the SELF keyword when you listen. Write it as LISTEN channel SELF. You can run LISTEN again for a channel the session already listens on. This just updates the self-delivery flag in place. It does not create a second registration.

Delivery format. On the PostgreSQL wire protocol a notification arrives as a NotificationResponse ('A') message. It carries the sending session id, the channel name, and the payload. On the EVO text protocol it arrives as an asynchronous NOTIFY <sender-session> <channel> <payload-length> line. The payload bytes follow that line.

Parameters

channel
The name of the notification channel. You give it as an identifier. It can be at most 63 bytes long. Matching is not case-sensitive. A name longer than the limit raises SQLSTATE 42622 (name too long). An otherwise invalid name raises the same error.
SELF
Valid only with LISTEN. It opts the current session in to receive its own NOTIFY messages on this channel. Without SELF, a session does not get its own notifications on a channel it listens to. This is the PostgreSQL default behavior. SELF is EvolutionDB's single-keyword way to say "deliver to self". There is no WITH (self = …) clause.
*
Valid only with UNLISTEN. It cancels every channel registration owned by the current session in one statement.
payload
An optional string literal sent with the notification. A comma separates it from the channel. It is delivered to every listener exactly as written. The payload can be at most 8000 bytes long. A longer payload raises SQLSTATE 22023 (payload string too long). When you do not give a payload, an empty payload is delivered.

Note. EvolutionDB accepts a quoted string literal as the NOTIFY payload. The NOTIFY statement does not let you pass the payload as any other expression. Sometimes you want to build a payload from query values. For that, use the built-in function evo_notify(channel, payload). It queues a notification through the same commit-gated path. The companion function pg_listening_channels() returns the channels the current session is listening on.

Examples

Set up a session to receive event notifications, then send to it.

LISTEN orders_changed;

NOTIFY orders_changed;
NOTIFY orders_changed, 'order 4711 shipped';

Receive your own notifications on a channel. You opt in with SELF.

LISTEN heartbeat SELF;
NOTIFY heartbeat, 'ping';   -- this session also receives 'ping'

Hold delivery until commit. The listener sees the notification only after COMMIT. It never sees it if the transaction is rolled back.

BEGIN;
NOTIFY orders_changed, 'pending';
COMMIT;            -- 'pending' is delivered here

Stop listening on one channel, then on everything.

UNLISTEN orders_changed;
UNLISTEN *;

Build a payload from query values with the helper function. Then check the current subscriptions.

SELECT evo_notify('orders_changed', 'order ' || id || ' shipped')
FROM orders WHERE id = 4711;

SELECT pg_listening_channels();

Compatibility

The SQL standard has no LISTEN, UNLISTEN, or NOTIFY command. These commands follow PostgreSQL.

These are the differences from PostgreSQL worth noting.

  • EvolutionDB delivers same-session notifications only when the listen used the LISTEN channel SELF form. PostgreSQL handles this at NOTIFY time and has no SELF keyword. SELF is an EvolutionDB extension. The default with no SELF matches PostgreSQL. By default neither one echoes a session's own notifications.
  • The NOTIFY payload must be a string literal. PostgreSQL also requires a string literal here. EvolutionDB adds the evo_notify() function for computed payloads.
  • EvolutionDB's payload limit is 8000 bytes. Channel names are limited to 63 bytes. These match PostgreSQL's limits.
  • Like PostgreSQL, notifications are commit-gated. Messages sent inside a transaction are delivered only on COMMIT. They are dropped on rollback.
  • The notification channel registry is in memory only. It is shared across both the PostgreSQL and EVO protocols. It does not survive a server restart. For subscriptions that persist and can be replayed, see CREATE SUBSCRIPTION.
