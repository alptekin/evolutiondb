DROP TABLE

Remove a table from the current database. This also removes its rows, indexes, constraints, and access policies.

Synopsis

DROP TABLE [ IF EXISTS ] table_name [ CASCADE | RESTRICT ]

Description

DROP TABLE removes one table and everything attached to it. EvolutionDB finds table_name in the system catalog. Then it removes the storage and metadata in one operation. The steps are below.

  • The primary-key B+ tree is destroyed.
  • All heap pages that hold the table's rows are freed back to the page manager.
  • Every secondary index is destroyed. This covers B+ tree indexes and hash and unique-hash indexes.
  • The catalog entry is dropped. This also removes the table's columns, indexes, and constraints.
  • The table's inheritance link is removed if the table was a child of another table.
  • Any row-level security policies on the table are removed.

You can name only one table per statement. EvolutionDB's grammar accepts a single table_name. It does not accept a comma-separated list. That form is only for TRUNCATE.

Two dependency checks run before any storage is touched.

  • Inheritance children. The drop is refused if the table has child tables that inherit from it. You must give CASCADE to allow it. With CASCADE, each child is dropped first, and so is each grandchild. The parent is dropped last.
  • Views. The drop is refused if any view in the current database uses the table by name. CASCADE does not override this check. You must drop the view that depends on the table first.

A successful DROP TABLE reports command(s) completed successfully!... A refused drop raises SQLSTATE 2BP01. This means that dependent objects still exist.

Parameters

IF EXISTS
Do not raise an error if the table does not exist. EvolutionDB reports success instead. Without this clause, dropping a missing table is an error. The message is table 'name' not found.
table_name
The name of the table to drop. The catalog finds the name in the current database and schema. Only one name is accepted.
CASCADE
Drop tables that inherit from this table first. This goes through the whole inheritance subtree. The named table is dropped after that. Note that CASCADE covers inheritance children only. It does not force the removal of dependent views. A view that uses the table still blocks the drop.
RESTRICT
Refuse to drop the table if any other object depends on it. This means an inheritance child. This is the default behavior when neither keyword is given.

Examples

This example drops a table.

DROP TABLE employees;

Drop a table only if it exists. This hides the error when the table is missing.

DROP TABLE IF EXISTS staging_import;

This example drops a parent table along with every table that inherits from it.

-- 'people' has child tables 'employees' and 'customers'
DROP TABLE people CASCADE;

Refuse to drop a table that still has children. This is the default behavior.

DROP TABLE people RESTRICT;
-- ERROR: cannot drop table "people" because other objects depend on it

Compatibility

DROP TABLE is defined by the SQL standard. EvolutionDB supports part of it. Its behavior is closest to PostgreSQL. The real differences are below.

  • Single table only. The SQL standard and PostgreSQL allow a comma-separated list of tables, such as DROP TABLE a, b, c. EvolutionDB's grammar accepts only one table_name per statement.
  • CASCADE scope. In PostgreSQL, CASCADE also drops dependent objects such as views and foreign-key constraints in other tables. In EvolutionDB, CASCADE follows the inheritance tree only. This means child tables. A view that uses the table still blocks the drop. You must remove it first.
  • RESTRICT is the default. This matches the SQL standard and PostgreSQL.
  • IF EXISTS matches PostgreSQL. A missing table is reported as success and not as an error.
  • EvolutionDB does not support the PostgreSQL-only DROP TABLE ... CASCADE CONSTRAINTS. It does not support schema-qualified multi-object drops. It also does not support the TEMPORARY keyword in this command.
