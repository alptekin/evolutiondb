15. Memory & Entity Stores

EvolutionDB gives you two built-in stores for agent memory. The first is the memory store. It is a namespaced JSON key/value table. Each row can hold an embedding, and you can run a semantic SEARCH over it. The second is the entity store. It keeps one summary row per key and counts how often each key is mentioned. Both are simple SQL statements. They compile down to ordinary tables, B+ tree primary keys, MVCC tuples, and the engine's existing INSERT/SELECT/UPDATE/DELETE code.

15.1. Overview

A memory store and an entity store are not new storage engines. Each one is a named catalog object bound to a single regular backing table. The MEMORY ... and ENTITY ... statements are shortcuts. They build the right tuple and send it through the same code path a hand-written DML statement would use. Everything the engine already does to a normal table still works the same way. That includes primary-key indexing, MVCC visibility, row-level security policies, foreign-key cascades, and triggers.

The two stores answer different questions.

Memory store
"Give me the value I filed under namespace + key." Or "find the rows whose embedding is closest to this query vector." It is a namespaced JSON KV store. Each row can also carry one VECTOR(N) column for semantic recall. See CREATE MEMORY STORE.
Entity store
"What do I know about this entity, and how often has it come up?" It is a per-key table. Each row holds a summary, an optional facts blob, an auto-incrementing mention_count, and a last_seen timestamp. See CREATE ENTITY STORE.

Relationship to PostgreSQL. These statements are EvolutionDB extensions. They are not part of the SQL standard and have no PostgreSQL equivalent. In PostgreSQL you would model the same data with your own table, the pgvector extension, and upsert logic in your application. EvolutionDB puts the table layout, the upsert behavior, and the distance ranking into single statements. The vector distance itself reuses the engine's native operators. Those are described in the Vectors chapter.

15.2. Creating a memory store

You create a memory store with CREATE MEMORY STORE. The optional WITH list says whether the store carries embeddings. It also says which distance metric SEARCH should use.

CREATE MEMORY STORE name
    [ WITH ( embedding_dim = N [, distance = 'cosine' ] ) ]
embedding_dim = N
The size of the per-row embedding. When N > 0, the backing table gets a VECTOR(N) column and the store supports MEMORY SEARCH. When you leave it out, the default is 0. In that case no embedding column is created at all. The store is then a plain JSON KV table. Running SEARCH against it is an error.
distance = '...'
The distance metric. It is saved in the catalog and used later by SEARCH. The accepted literals are listed in §15.5. The default is cosine. Any literal the engine does not recognize quietly falls back to cosine.
-- semantic store with 1536-d embeddings, L2 distance
CREATE MEMORY STORE agent_mem WITH (embedding_dim = 1536, distance = 'l2');

-- plain namespaced JSON KV store, no vectors
CREATE MEMORY STORE scratch;

The statement also supports CREATE MEMORY STORE IF NOT EXISTS name. If a store with that name already exists, this returns success. It does not raise 42710.

15.2.1. The backing table

When you create a memory store named foo, the engine creates a regular table named mem_foo in the current schema. It has the columns below. The embedding column is present only when embedding_dim > 0.

ColumnTypeNotes
mem_namespaceVARCHAR(255)NOT NULL
mem_keyVARCHAR(255)NOT NULL
pkVARCHAR(512)PRIMARY KEY, composite of namespace and key
mem_valueJSONnullable
embeddingVECTOR(N)only if embedding_dim > 0; nullable
created_atTIMESTAMPset at PUT time, UTC
ttl_atTIMESTAMPnullable; written NULL by every PUT (reserved)

The primary key is a single derived column. It is not a two-column composite. The namespace and the key are joined into one string, separated by the ASCII Record Separator byte 0x1E.

-- pk is: namespace + 0x1E + key
-- 0x1E avoids the engine's internal field/row separators (0x1F, 0x02)

The PK is an indexed VARCHAR. A point lookup by namespace and key is O(log n) through the B+ tree. The mem_namespace and mem_key columns also stay queryable on their own. You can use them in ordinary SQL against mem_foo.

15.3. Writing memories, MEMORY PUT

MEMORY PUT does an upsert by primary key. The full reference is at MEMORY PUT.

MEMORY PUT INTO name VALUES (ns, key, value [, embedding])

Three positional values are required. They are the namespace, the key, and the JSON value. A fourth optional value gives the embedding. The embedding only matters for a store created with embedding_dim > 0. Each value may be a quoted string literal or NULL.

-- KV-only write
MEMORY PUT INTO scratch
  VALUES ('user:42', 'lang', '{"locale":"tr-TR"}');

-- write with an embedding (store created WITH embedding_dim = 4)
MEMORY PUT INTO agent_mem
  VALUES ('user:42', 'fact:1',
          '{"text":"prefers dark mode"}',
          '[0.10, 0.22, 0.07, 0.91]');

The embedding literal is a list of floats in brackets, separated by commas. This is the same text form the engine accepts for VECTOR columns elsewhere. On a semantic store you can leave out the embedding or pass NULL. The row is still stored and you can still get it by key. It just will not give a meaningful SEARCH score.

Upsert is atomic and crash-safe. PUT does not delete and then insert. It inserts first. If there is a primary-key collision, the PK is pointed at the new tuple and the old one is dropped. New comes first, the swap comes last. A failed re-insert can never leave the row in a deleted state. The created_at timestamp is written again on every PUT. The value shows the most recent write, not the first one.

15.4. Reading by key, MEMORY GET

MEMORY GET is an exact point lookup on namespace + key. See MEMORY GET.

MEMORY GET FROM name WHERE NS = '<ns>' AND KEY = '<key>'

The grammar is fixed. Both the NS predicate and the KEY predicate are required. They are joined by AND. Inside, this compiles to a single-row SELECT ... WHERE pk = '<ns>0x1E<key>' LIMIT 1. The returned columns are the namespace, the key, the value, and created_at. The embedding is also returned when the store has one.

MEMORY GET FROM scratch WHERE NS = 'user:42' AND KEY = 'lang';

15.5. Semantic recall, MEMORY SEARCH

MEMORY SEARCH ranks every row in the store by its vector distance from a query embedding. The closest rows come first. See MEMORY SEARCH.

MEMORY SEARCH name USING VECTOR '<query>' [ LIMIT n ]

The query is a float list in brackets. It has the same form as a stored embedding. The statement returns (mem_namespace, mem_key, mem_value, score). The score is computed for each row using the distance function that was set for the store at CREATE time. The result is ordered by score ascending, so the nearest matches come first. If you leave out LIMIT, the engine sets no cap. Pass a value to bound the result set. It is best to always pass a small LIMIT.

MEMORY SEARCH agent_mem
  USING VECTOR '[0.11, 0.20, 0.08, 0.90]'
  LIMIT 5;

The distance metric is fixed when the store is created. See §15.2. It decides which native operator SEARCH uses for the score column.

distance literalInternal kindOperator used by SEARCH
'cosine' (default)MEMORY_DIST_COSINEcosine distance
'l2'MEMORY_DIST_L2Euclidean (L2) distance
'inner' / 'ip'MEMORY_DIST_INNERinner-product distance
'l1'MEMORY_DIST_L1Manhattan (L1) distance

SEARCH is a full scan, scored per row. It does not use an approximate vector index. The distance = ... metric set at creation time is what makes the ranking meaningful. Every row's distance is computed during the scan. Calling SEARCH against a store created without an embedding column raises error 42P01. A non-semantic store has no vector to compare. The ordering happens inside the engine using the score column's positional ordinal. This statement does not expose an ORDER BY <expression>.

15.6. Enumerating namespaces, MEMORY LIST NAMESPACES

To see which namespaces exist in a store, use MEMORY LIST NAMESPACES. It compiles to SELECT DISTINCT mem_namespace FROM mem_<name>. An optional prefix filter is added as a LIKE 'prefix%' predicate.

MEMORY LIST NAMESPACES IN name [ PREFIX '<prefix>' ]
MEMORY LIST NAMESPACES IN scratch;
MEMORY LIST NAMESPACES IN scratch PREFIX 'user:';

15.7. Removing memories, MEMORY DELETE

MEMORY DELETE removes the single row that matches a namespace and key. See MEMORY DELETE.

MEMORY DELETE FROM name WHERE NS = '<ns>' AND KEY = '<key>'

Like GET, both predicates are required. The statement runs through the engine's standard DELETE path against mem_<name>, filtered on the composite pk. Any row-level security policy, foreign-key cascade, MVCC version, or trigger on the backing table applies just as it would to a hand-written DELETE FROM mem_<name> WHERE pk = '...'. This makes MEMORY DELETE safe in a multi-tenant database where each tenant is fenced off by an RLS policy.

MEMORY DELETE FROM scratch WHERE NS = 'user:42' AND KEY = 'lang';

15.8. Dropping a memory store

DROP MEMORY STORE removes the catalog registration. It also removes the backing mem_<name> table.

DROP MEMORY STORE name [ CASCADE ]
DROP MEMORY STORE IF EXISTS name

IF EXISTS hides the not-found error 42704 and returns success. The backing table is dropped if it is present.

15.9. Entity stores

An entity store keeps one row per entity key. It tracks a rolling summary and counts how many times the entity has been mentioned. You create and drop it the same way as a memory store. It takes no WITH options. See CREATE ENTITY STORE.

CREATE ENTITY STORE name [ IF NOT EXISTS ]
DROP ENTITY STORE name [ IF EXISTS ]

When you create an entity store named foo, the engine builds a backing table named ent_foo. It has this fixed shape.

ColumnTypeNotes
entity_keyVARCHAR(255)PRIMARY KEY, NOT NULL
summaryVARCHAR(8000)nullable
factsVARCHAR(8000)nullable; a JSON-shaped string
mention_countINTNOT NULL; auto-incremented on each PUT
last_seenTIMESTAMPNOT NULL; advanced to the current UTC time on each PUT

15.9.1. ENTITY PUT, upsert with mention tracking

ENTITY PUT is an upsert keyed on entity_key. Unlike MEMORY PUT, it has behavior of its own. The first write inserts the row with mention_count = 1. Every later write to the same key updates the summary and facts, adds one to mention_count, and moves last_seen forward.

ENTITY PUT INTO name VALUES ('key', 'summary' [, 'facts'])
-- first mention: inserts with mention_count = 1
ENTITY PUT INTO people
  VALUES ('ada', 'Works on the storage layer');

-- second mention: updates summary, mention_count -> 2, last_seen bumped
ENTITY PUT INTO people
  VALUES ('ada', 'Leads the B+ tree rewrite',
          '{"team":"core","tz":"UTC+3"}');

The optional third value is the facts blob. When you leave it out, it is stored as NULL. The existence check is a direct primary-key probe of the backing table's B+ tree. The old mention_count is read from the existing tuple before it is incremented. The counter only goes up for each key.

15.9.2. ENTITY GET

ENTITY GET fetches a single entity row by key.

ENTITY GET FROM name WHERE KEY = 'k'
ENTITY GET FROM people WHERE KEY = 'ada';

It returns (entity_key, summary, facts, mention_count, last_seen) for the matching key. It returns one row.

15.9.3. ENTITY RANK

ENTITY RANK returns the store's rows ordered by mention_count descending. Ties are broken by last_seen descending. The most-discussed and most-recent entities come first.

ENTITY RANK FROM name [ LIMIT n ]
-- top 10 entities by how often they've been mentioned
ENTITY RANK FROM people LIMIT 10;

The engine sorts the result set internally. LIMIT bounds the output when you supply it.

15.10. Status & error codes

The store statements set SQLSTATE codes the same way as the rest of the engine. Below are the codes you will see most often with these stores.

SQLSTATERaised when
42601A required name is missing (e.g. CREATE MEMORY STORE with no store name).
42710Creating a store whose name already exists (without IF NOT EXISTS).
42704Referencing a store (PUT / GET / SEARCH / DELETE / RANK / DROP) that does not exist.
42P01MEMORY SEARCH against a store created without an embedding column.

15.11. Notes & limitations

  • SEARCH is exact, not indexed. MEMORY SEARCH scans and scores every row. It does not use an approximate-nearest-neighbour vector index. For large stores the cost grows with the number of rows.
  • The distance metric is fixed at creation. You cannot change it after CREATE MEMORY STORE. There is no ALTER for stores. To switch metrics, create the store again.
  • No ORDER BY <expr> surface. SEARCH sorts inside the engine by its computed score column using a positional ordinal. You cannot give your own ordering expression on the statement.
  • ttl_at is reserved. The memory backing table has a ttl_at column. The catalog descriptor has a ttl_days field. But MEMORY PUT always writes ttl_at as NULL. No automatic expiry pass is wired up. Treat TTL as not yet implemented.
  • Fixed predicate shapes. MEMORY GET and DELETE require exactly WHERE NS = '...' AND KEY = '...'. ENTITY GET requires exactly WHERE KEY = '...'. These are not general WHERE clauses. For other queries, run ordinary SQL against the backing mem_<name> or ent_<name> table directly.
  • Entity facts is a string, not a JSON column. It is stored as VARCHAR(8000) that holds a JSON-shaped string. The engine does not parse it or validate it as JSON.
  • Stores live in the current schema. The backing table is created in the schema from the current database and schema context. A store name is unique within the catalog's store registry.
  • Timestamps are UTC. created_at and last_seen are written as YYYY-MM-DD HH:MM:SS in UTC at write time.
