9. Functions & Operators

This chapter covers the operators, built-in functions, conditional expressions, type coercion, and NULL handling in EvolutionDB 3.0. These work in WHERE, SELECT, HAVING, SET, DEFAULT, and CHECK. They behave exactly as the engine's single expression evaluator runs them.

The Bison parser turns every scalar expression into an expression tree. One function evaluates that tree. The function is expr_evaluate, in evolution/db/expression.c. The tree has no separate type system. Each node is evaluated to a string. That string is its rendered value. The evaluator also returns a second flag. The flag says whether the value is NULL. A second routine, expr_evaluate_num, evaluates the same tree to a C double for arithmetic. This string-plus-NULL-flag model explains most of the type coercion and comparison behavior in this chapter.

Scope of this chapter. This chapter documents only the functions and operators that the expression evaluator really implements. It gives their real argument counts, return shapes, and edge behavior. Sometimes EvolutionDB borrows syntax from MySQL or PostgreSQL but runs it differently. Examples are integer-only %, a non-collating LIKE, and MySQL-style NULL-propagating CONCAT. The chapter calls out each such difference. Some functions are parsed but only matter in other clauses. These include aggregates resolved by the query executor, window functions, and the set-returning unnest in FROM. The chapter notes them where they touch the evaluator.

9.1. The Evaluation Model

A scalar expression is evaluated against one row. The row comes as two parallel arrays. One array holds the column names. The other holds the string column values. The evaluator resolves a column reference by name. The match is case-insensitive. It returns string literals as they are. For any arithmetic or numeric node it calls the numeric evaluator and renders the result back as text. Integer results print without a decimal point. Other numbers use %g formatting.

The string-valued design leads to two facts that come up again and again in this chapter.

Values carry no static type
At evaluation time, a column value, a literal, and a function result are all text. Each operation decides on its own whether to treat the operands as numbers or as text. It does this by looking at the operand strings. See Section 9.7.
NULL is a separate return channel
The evaluator returns 1 when it produced a value. It returns 0 to signal NULL. A NULL literal evaluates to NULL. So does a column whose stored value is the internal null marker. So does an unresolved column name. Some functions hold NULL in their output. They use the engine's null marker string.

9.2. Operators

9.2.1. Arithmetic and Bitwise Operators

Arithmetic runs as C double. The engine recognizes these operators.

OperatorOperationNotes
+Addition
-Subtraction. Also unary negation.
*Multiplication
/DivisionDivision by zero yields 0, not an error and not infinity.
%ModuloInteger modulo. Both operands are truncated to long long first. Modulo by zero yields 0.
<<   >>Bit shift left / rightOperands truncated to long long.
&   |   ^Bitwise AND / OR / XOROperands truncated to long long.
 
SELECT price * 1.1, qty % 12, flags & 4 FROM orders;

NULL in arithmetic. Arithmetic runs through the numeric evaluator. That evaluator treats an unresolved or non-numeric operand as 0.0. It does not propagate NULL. So NULL + 1 on the numeric path resolves to 1. A non-numeric string contributes its strtod value. Text with no leading number contributes 0. This differs from the SQL standard. In the standard, any NULL arithmetic operand makes the whole expression NULL.

9.2.2. Comparison Operators

The six comparison operators return a boolean. The boolean is rendered as the single character t or f.

=
Equal
<>
Not equal
<   >
Less than / greater than
<=   >=
Less than or equal / greater than or equal

A comparison first evaluates both sides. If either side is NULL, the result is NULL. The evaluator returns the null channel. This follows SQL three-valued logic. Otherwise the evaluator chooses how to compare. If both rendered values parse cleanly and fully as numbers, it compares them as numbers. Otherwise it compares them as byte strings with strcmp. See Section 9.7.

SELECT * FROM orders WHERE total >= 100 AND status <> 'cancelled';

The <=> operator. EvolutionDB parses <=> as a vector cosine-distance operator. See Section 9.6.7. When the operands are not vector literals, it falls back to a plain equality test. That test returns t or f. So SELECT 1 <=> 1 returns t. In the general case it is not MySQL's null-safe equals.

9.2.3. Logical Operators

AND, OR, NOT, and XOR work on boolean values. They use full SQL three-valued logic. A value counts as true if it is t, the word true in any case, or a non-zero number. Anything else is false. The truth tables for NULL follow the standard.

ExpressionResult
FALSE AND NULLf
TRUE AND NULLNULL
NULL AND NULLNULL
TRUE OR NULLt
FALSE OR NULLNULL
NOT NULLNULL
NULL XOR xNULL (XOR requires both operands non-null)

9.2.4. Pattern Matching: LIKE and REGEXP

LIKE and NOT LIKE support the two SQL wildcards. % matches any run of characters. It can also match none. _ matches exactly one character. An iterative backtracking matcher does the matching byte by byte.

SELECT name FROM users WHERE name LIKE 'A%' AND email NOT LIKE '%@test._';

If either the subject or the pattern is NULL, the result is NULL.

REGEXP and NOT REGEXP match against a POSIX extended regular expression. On a NULL operand the result is NULL. An invalid pattern that fails to compile evaluates to f. That means false. It does not raise an error.

Limitations. LIKE is byte-exact and case-sensitive. There is no ESCAPE clause and no ILIKE. So you cannot escape a literal % or _ in the pattern. On Linux and macOS, REGEXP uses the platform POSIX <regex.h>. It is compiled with REG_EXTENDED | REG_NOSUB. Windows (MinGW) builds lack POSIX regex. On those builds REGEXP always returns f.

9.2.5. NULL Predicates, BETWEEN, IN

IS NULL and IS NOT NULL test whether their operand is NULL. A column counts as null in two cases. The first is when its stored value is the engine's null marker. The second is when the named column is not in the row at all. A literal is never null. This holds for an integer, a float, a string, or a boolean.

BETWEEN, NOT BETWEEN, IN, and NOT IN are not their own node types. The parser desugars them into the operators above.

x BETWEEN a AND b
rewritten as (x >= a) AND (x <= b).
x NOT BETWEEN a AND b
rewritten as (x < a) OR (x > b).
x IN (a, b, c)
rewritten as (x = a) OR (x = b) OR (x = c). The parts go into a balanced OR tree. This keeps a long list shallow.
x NOT IN (a, b, c)
rewritten as (x <> a) AND (x <> b) AND (x <> c). The parts go into a balanced AND tree.

IN and NOT IN reduce to comparisons and logical operators. So they take the three-valued NULL behavior of those operators. An empty IN () list evaluates to false. An empty NOT IN () list evaluates to true.

9.3. Conditional Expressions

9.3.1. CASE

Both forms of CASE are supported. The searched form tests a boolean condition in each branch. The simple form compares one operand against each WHEN value. The engine desugars the simple form into the searched form. It turns each WHEN v into the comparison operand = v. Both forms share one evaluator.

-- searched CASE
SELECT CASE
         WHEN score >= 90 THEN 'A'
         WHEN score >= 80 THEN 'B'
         ELSE 'F'
       END FROM grades;

-- simple CASE (desugars to status = 'A', status = 'I', ...)
SELECT CASE status WHEN 'A' THEN 'active'
                   WHEN 'I' THEN 'inactive' END FROM accounts;

Branches are evaluated top to bottom. The first WHEN whose condition is true selects its THEN result. A branch whose condition is NULL counts as not matched. It is skipped. If no branch matches, the ELSE result is returned. If there is no ELSE, the result is NULL.

9.3.2. COALESCE, NULLIF, IFNULL, IF

COALESCE(a, b)
Returns a if it is non-null. Otherwise it returns b. If both are null, it returns NULL. The engine takes two arguments.
NULLIF(a, b)
Returns NULL when a and b are equal as strings. Otherwise it returns a.
IFNULL(a, b)
This is the MySQL form. It returns a if non-null. Otherwise it returns b.
IF(cond, t, f)
This is the MySQL three-argument conditional. It checks cond for truthiness. A value is true if it is t, true, 1, or any non-zero number. It returns the second or third argument by that test. A NULL condition takes the false branch.

9.3.3. IS TRUE / IS FALSE

x IS TRUE returns t only when x renders as t, true, or 1. Otherwise it returns f. A NULL operand also gives f. x IS FALSE returns t only when x renders as f, false, or 0.

9.4. String Functions

String functions evaluate their first argument. If that argument is NULL, they return NULL. A few functions are exceptions. They are noted below.

FunctionReturnsBehavior
UPPER(s) / LOWER(s)textASCII case fold per byte via toupper/tolower.
LENGTH(s)integerByte length of the string via strlen. This is not the character count.
SUBSTRING(s, pos [, len])textThe start position is 1-based. A position below 1 is clamped to 1. With no length, it returns the tail from pos.
TRIM([LEADING|TRAILING|BOTH] [chars] FROM s)textTrims any of the characters in chars from the chosen end or ends. The default is a single space.
CONCAT(a, b)textJoins two arguments. It is NULL-propagating in the MySQL style. If either argument is NULL, the result is NULL.
REPLACE(s, from, to)textReplaces all non-overlapping occurrences of from with to. An empty from returns s unchanged.
LEFT(s, n) / RIGHT(s, n)textFirst or last n bytes. n is clamped to [0, length].
LPAD(s, n, pad) / RPAD(s, n, pad)textPads to width n with the repeating pad string on the left or right. The default pad is a space.
REVERSE(s)textReverses the bytes of the string.
REPEAT(s, n)textJoins n copies of s.
INSTR(s, sub)integer1-based position of the first occurrence of sub in s. Returns 0 if not found.
LOCATE(sub, s)integerSame as INSTR with the arguments swapped.
TOKEN_LENGTH(s)integerRough word count from a whitespace split. It stands in for a tokenizer.
 
SELECT UPPER(LEFT(name, 3)), LENGTH(bio),
       REPLACE(slug, '-', '_'),
       SUBSTRING(sku, 2, 4)
FROM profiles;

Encoding caveat. All string functions work on bytes. LENGTH, SUBSTRING, LEFT/RIGHT, and REVERSE count and slice by byte. So a multi-byte UTF-8 character is several units. It can be split. UPPER/LOWER fold only ASCII letters. There is no separate CHAR_LENGTH. The evaluator's string buffers have a fixed size. It is often 512 bytes. So a function may truncate a very long value.

9.5. Mathematical Functions

Math functions evaluate their arguments as numbers. They render the result with %g. Some render as an integer, as noted. A NULL argument usually gives NULL.

FunctionResult

CEIL(x) / FLOOR(x)Round up or down to an integer value.
ROUND(x [, d])Round to d decimal places. The default is 0.
POWER(a, b)a raised to the power b.

LOG(x)Natural logarithm.
LOG10(x)Base-10 logarithm.
SIGN(x)The sign of x as an integer. It is 1, -1, or 0.
RAND()Pseudo-random double in [0, 1].

SELECT ROUND(price * 1.08, 2), CEIL(weight), SIGN(balance) FROM items;

Note. The MOD(a, b) function uses floating-point fmod. It returns NULL on a zero divisor. The % operator works differently. See Section 9.2.1. It truncates to integers and returns 0 on a zero divisor. SQRT, LOG, and LOG10 follow the C library on out-of-domain inputs. They may render as nan or -inf. They do not raise an error.

9.6. Other Built-in Functions

9.6.1. Date and Time

CURRENT_TIMESTAMP / NOW()
Current local date-time as YYYY-MM-DD HH:MM:SS.
CURRENT_DATE
Current local date as YYYY-MM-DD.
CURRENT_TIME
Current local time as HH:MM:SS.
YEAR / MONTH / DAY / HOUR / MINUTE / SECOND(dt)
Pull one field from a YYYY-MM-DD [HH:MM:SS] string. It parses the string with sscanf.
DATEDIFF(d1, d2)
Whole days between two dates, as d1 - d2. It uses mktime and difftime.
DATE_ADD(dt, n unit) / DATE_SUB(dt, n unit)
Add or subtract n of a unit. The known units are YEAR, MONTH, and HOUR. Any other unit is treated as DAY. The result is rendered as a date in YYYY-MM-DD form.
SELECT YEAR(created_at), DATEDIFF(CURRENT_DATE, signup_date) FROM users;

Note. Date functions work on the text YYYY-MM-DD form. They use the server's local time zone through the C library. There is no time-zone-aware TIMESTAMPTZ arithmetic here. DATE_ADD and DATE_SUB normalize through mktime. They emit a date even when you give an HOUR unit.

9.6.2. Identifier and Session Functions

gen_random_uuid()
Random RFC 4122 version-4 UUID. It is formatted as 8-4-4-4-12 lowercase hex.
gen_random_uuid_v7()
Time-ordered RFC 9562 version-7 UUID. It has a 48-bit millisecond timestamp prefix.
snowflake_id()
A 64-bit Snowflake-style id. It has a 41-bit millisecond timestamp, a 10-bit machine id, and a 12-bit per-millisecond sequence. The custom epoch is 2026-01-01 UTC. The machine id comes from $EVOSQL_MACHINE_ID or from a hash of the hostname.
last_insert_id()
The last value taken from a UUID, Snowflake, or auto-increment default. It is 0 if there is none.
CURRENT_USER
The session username from the active query context. It is unknown outside a session.
evo_current_xid()
The last-assigned MVCC transaction id.

You most often use these generator functions as column DEFAULTs. See CREATE TABLE.

9.6.3. Sequence Functions

Sequence values come from functions. These functions resolve against the catalog of the current schema. See CREATE SEQUENCE.

nextval(name)
Advance the sequence and return its new value. It also updates the session's lastval tracking.
currval(name)
The sequence's current value without advancing. It errors if the sequence does not exist. It also errors if it has not been called yet in this session.
setval(name, value [, is_called])
Set the sequence's current value. An is_called of false or 0 makes the next nextval return value itself.
lastval()
The value most recently made by nextval in this session. It errors if no sequence has been called yet.

9.6.4. JSON Functions and Operators

JSON is stored and worked on as text. The arrow operators and a MySQL-compatible function set are available.

JSON_VALID(j)

JSON_KEYS(j)

JSON_PRETTY(j)

JSON_REMOVE(j, path)

JSON_CONTAINS(target, candidate)

JSON_CONTAINS_PATH(j, mode, path)

Build an object from one key/value pair, or an array from up to three values. A value that is already valid JSON is inserted as is. Other values are quoted.
SELECT data->>'name', JSON_LENGTH(data->'tags')
FROM events
WHERE JSON_VALID(data) = 1 AND JSON_CONTAINS(data, '{"active":true}') = 1;

9.6.5. Array Constructors and Functions

Inside the engine, arrays are PostgreSQL-style brace text. An example is {e1,e2,NULL,...}. An element with a separator or whitespace is quoted.

ARRAY[e1, e2, ...]
Build an array literal. A NULL element is rendered as the unquoted token NULL.
arr[i]
1-based subscript. An out-of-bounds or malformed index gives NULL. This follows PostgreSQL semantics.
array_length(arr)
Element count.
x <op> ANY(arr)
True if the comparison holds for any element. If no element matches but the array holds a NULL, the result is NULL. Otherwise the result is false. Element comparison is numeric when both sides parse as numbers. Otherwise it is lexicographic.
unnest(arr)
In scalar context it returns the array text as is. True row-producing expansion needs a FROM clause. The join pipeline handles that, not this evaluator.

9.6.6. Full-Text Match

MATCH(col) AGAINST('query') tokenizes the row's column value and the query literal. It returns the count of query terms that appear in the row. You can use this count as a boolean in WHERE. You can also use it as a relevance score in ORDER BY. It is a simple any-term match. A positive count means a match. It is not a ranked full-text engine.

9.6.7. Vector Distance Functions and Operators

Vectors arrive as text. One form is bracketed, like [f1,f2,...] or {f1,f2,...}. Another form is a compact quantized form, like b64i8:<base64>. The quantized form is decoded back to floats. The table below shows the distance operators and their named equivalents.

OperatorDistance
<=>Cosine distance (1 - cosine similarity)
L2 operatorEuclidean distance
inner-product operatorNegative dot product
L1 operatorManhattan distance

If either operand is NULL, the result is NULL. If the two vectors have different dimensions, the result is also NULL. When the operands are not vector literals, the cosine operator falls back to a string-equality test. That test returns t or f. The other operators need vectors. They return NULL in that case. There are companion functions too. vector_dim(v) gives the element count. vector_norm(v) gives the L2 norm. vector_normalize(v) gives the unit vector as bracketed text. All three round-trip through the same parser. The approximate nearest-neighbor helpers hnsw_knn(...) and hnsw_filter_knn(...) return comma-separated primary-key lists. See CREATE INDEX.

9.7. Type Coercion and CAST

Values are untyped text during evaluation. Each operation decides coercion on its own. There is no declared expression type.

Comparisons
Both operands are compared as numbers only when each renders as a full, valid number. The whole string must be consumed by strtod and must be non-empty. If either side is not a clean number, the comparison falls back to a byte-wise strcmp. So '10' = 10 and '2' < '10' both behave as numbers. But '10x' < '9' compares as text.
Arithmetic
Operands are coerced with strtod. Text with no leading number contributes 0.
Boolean context
A value is true if it is t, true in any case, or a non-zero number. Inside CASE and IF, 1 is also taken as true.

You convert types on purpose with CAST(expr AS type). The target type's family code drives the conversion. The type encoding is described in Chapter 4. There, typeVal / 10000 is the family.

Target familyCAST behavior
Integer typesParse with strtol and render as a base-10 integer.
Float / double typesParse with strtod and render with %g.
BooleanA non-zero numeric value renders as true. Otherwise it renders as false.
VARCHAR / CHAR / TEXT / DATEThe string passes through unchanged. There is no reformatting.
 
SELECT CAST(price AS INT), CAST(active AS BOOLEAN) FROM items;

A CAST of a NULL value is NULL. For the family codes, see Chapter 4, Data Types. For where expressions can appear, see SELECT.

9.8. NULL Handling Summary

The treatment of NULL is not the same across operations. The table below gathers the rules from earlier sections and puts the differences side by side.

ContextNULL behavior
Comparison operators (=, <, …)Any NULL operand → result NULL. This is three-valued.
AND / OR / NOT / XORStandard three-valued logic. See Section 9.2.3.
LIKE / REGEXPAny NULL operand → NULL.
String functionsNULL input usually → NULL.
CONCATAny NULL argument → NULL. This is the MySQL style. It is not the PostgreSQL ignore-null style.
Most math functionsNULL input → NULL. MOD(a,0) → NULL.
Arithmetic operators (numeric path)A NULL or non-numeric operand is treated as 0. It does not propagate.
CASE branch with NULL conditionTreated as not matched. It falls through.
COALESCE / IFNULLReturn the first non-null argument.
Array subscript out of rangeNULL.
Vector distance, dimension mismatchNULL.

Note. Honest differences from the SQL standard. Three behaviors differ from a standard engine. They are worth remembering. (1) Division and modulo by zero return 0. They do not raise a division_by_zero error. (2) On the numeric path, arithmetic treats a NULL operand as 0. It does not propagate NULL. (3) CONCAT propagates NULL like MySQL. PostgreSQL's || and concat() ignore nulls. String functions work on bytes. Case folding is ASCII only. LIKE has no ESCAPE clause. Treat all of this as engine-specific. Test the edge cases yourself.

9.9. Expressions in CHECK Constraints

You can store part of the expression language in a table's metadata. This is for use in CHECK constraints. The expression is serialized to a compact reverse-Polish token stream. That is postfix form. It is re-parsed when the constraint is evaluated. The serializable node set has several parts. It includes column references. It includes the four literal kinds plus NULL. It includes the comparison, logical, and arithmetic operators. The logical operators are AND, OR, NOT, and XOR. It includes IS NULL and IS NOT NULL. It includes LIKE and NOT LIKE. It includes the functions UPPER, LOWER, LENGTH, TOKEN_LENGTH, CONCAT, and COALESCE. It includes the JSON arrow operators. It includes the session functions CURRENT_USER, CURRENT_TIMESTAMP, CURRENT_DATE, and CURRENT_TIME. Node types outside this set are dropped during serialization. So a CHECK built from unsupported functions will not round-trip faithfully. Keep CHECK expressions within the list above. For constraint syntax, see CREATE TABLE.

CREATE TABLE accounts (
    id       INT PRIMARY KEY,
    email    VARCHAR(120) CHECK (LENGTH(email) > 3),
    balance  DECIMAL(12,2) CHECK (balance >= 0),
    owner    VARCHAR(60) CHECK (owner = CURRENT_USER)
);

These expressions are used in several places. See SELECT for projection, WHERE, and HAVING. See Chapter 6, Data Manipulation for SET assignments, DEFAULT, and ON CONFLICT updates. See CREATE SEQUENCE for the sequence functions.
